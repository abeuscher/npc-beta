## What was built

### Phase 1 — Templates table and migration

- Created `templates` table with uuid PK, `type` discriminator (`page`/`content`), color/font/SCSS fields (nullable for inheritance), header/footer page FKs, `created_by` FK.
- Added nullable `template_id` FK to `pages` table.
- Built `Template` model with `resolved()` method (returns own value when set, falls back to default template for null fields), `scopeDefault`, `scopePage`, `scopeContent`, relationships to pages/users.
- Added `template_id` to Page model fillable + `template()` relationship.
- `TemplateFactory` for test support.

### Phase 2 — Content template presets

- `TemplateSeeder` creates the default page template from current site settings and `_header`/`_footer` system pages.
- Five content templates seeded: Contact Page, About Page, Event Landing Page, Blog Post, Blank — each with a `definition` JSONB array of `{handle, config, sort_order}` widget specs.
- Seeder registered in `DatabaseSeeder` after `SystemPageSeeder`.

### Phase 3 — Event landing page improvements (partial)

- Refactored `createLandingPageForEvent()` to hydrate widgets from the "Event Landing Page" content template's `definition` field, with hardcoded fallback if the template is missing.
- Added "Edit landing page" button to the event edit page header actions.
- Mandatory auto-creation via EventObserver was attempted but rolled back — Filament's transactional `CreateRecord` flow conflicts with creating cross-model side effects (Page + widgets + back-update) inside the observer. The manual "Create basic landing page" button remains as the working path.

### Phase 4 — Renderer integration

- `PageController` resolves the page's template (explicit `template_id` or default) and shares it with the layout via `View::share`.
- `public.blade.php` reads colors, fonts, header/footer, and SCSS from the template's `resolved()` method, with `SiteSetting` fallback for safety during transition.
- `ChromeRenderer::renderById()` added to render header/footer by page ID (from template) instead of hardcoded `_header`/`_footer` slugs.
- Template `custom_scss` appended to the runtime SCSS compilation block.

### Deployment fixes (committed to main)

- **Dockerfile fix:** Stage 1 (node-builder) was missing `tailwind.config.js`, `resources/views/`, and `app/Livewire/`. Tailwind's content scanner found zero classes and purged all utilities on production builds. Added the missing COPY directives.
- **Vendor JS bundling:** CalendarJS and ChartJS were static files in `public/vendor/` loaded via conditional `<script>` tags. Migrated both to Vite imports in `public.js` (matching the existing Swiper pattern). CalendarJS uses the ESM export (`calendar.export.js`) with explicit `window.calendarJs` assignment. ChartJS uses `chart.js/auto`. Removed `public/vendor/` directory and cleaned up widget type seeder `assets` entries. Calendar SCSS theme override kept in the runtime pipeline.
