# 182. Design System Editor — Typography

Build a Vue-based Typography editor as a new tab on the Design System Editor (admin, Tools group) with a reusable `font` inspector primitive, template-scoped storage, a three-bucket family inheritance model (heading / body / nav), live preview, and SCSS export. Deliberately experimental — we will iterate line-by-line on the UI after the scaffold lands.

---

## Design decisions (resolved before starting)

- **Vue control.** The Typography tab is a Vue component, not a Filament/Livewire form. Same architectural posture as the page builder. Mount inside the existing Design System Editor shell.
- **Template-scoped storage.** Typography lives on the `templates` table, not site settings. Per-template typography supports the "multiple theme systems" long-term direction.
- **Replace `heading_font` / `body_font` (migration).** Those columns are removed in this session. Existing values migrate into the new `buckets.heading_family` / `buckets.body_family`. The `nav` bucket is new and starts null.
- **Three-bucket family inheritance.** Heading / Body / Nav buckets each hold a single `font-family` value. Per-element `font.family` is nullable: null = inherit from the matching bucket.
  - Heading bucket → H1–H6.
  - Body bucket → P, UL li, OL li.
  - Nav bucket → nav widget only (no element row in the Typography tab today).
- **Font primitive scope.** The new `font` field type exposes: family, weight, size, line-height, letter-spacing, case transform. No margin / padding / border / bullet controls — those are separate controls on the element row where relevant.
- **Reuse existing four-side padding/margin picker.** Lift the Appearance Controls spacing picker (sessions 164–166) into a shared Vue primitive and drop it into element rows. Do not duplicate.
- **Elements in scope today:** H1, H2, H3, H4, H5, H6, P, UL li, OL li. Per-element controls are: font primitive + margin + padding + (for UL/OL) list-style-type + marker color.
- **Sample input editable.** Default value is "The quick brown fox jumps over the lazy dog." Editable inline — reflects in the live preview block.
- **Font families list.** Reuse the existing curated Google Fonts list already used by `heading_font` / `body_font`. Do not expand today.
- **Export: downloadable `.scss` file.** One file per template, plain selector blocks (`h1 { ... }`, `ul li { ... }`). No variables / mixins today. No import flow today.
- **Contextual styles (modals, form fields, etc.) deferred.** Today is text styles only.
- **Figma pipeline / external adapters.** No integration work today. Ensure the storage model is a clean JSON tree so future import adapters have a sane target.

## Phase 1 — `font` field type primitive

Build the reusable `font` inspector primitive. Standalone, testable in isolation, not yet wired to typography storage.

1. Schema side: register `font` as a config field type. Schema entries look like `['key' => 'heading_font', 'type' => 'font', ...]`.
2. Vue side: `resources/js/page-builder-vue/components/FontInput.vue` (or the correct shared-inspector path — see existing conventions). Controls:
   - Family (select from curated list).
   - Weight (select — 100/200/…/900, limited to what the family supports).
   - Size (number + unit picker: px / rem / em).
   - Line height (number, unitless).
   - Letter spacing (number + unit: em / px).
   - Case transform (select — none / uppercase / lowercase / capitalize / small-caps).
3. Each sub-control is nullable; emit `null` for unset rather than a default. The consumer (Typography tab) decides how to resolve inheritance.
4. Storage shape: JSON object `{ family, weight, size: {value, unit}, line_height, letter_spacing: {value, unit}, case }`. Null at any level is fine.

**Do not** bake bucket-inheritance logic into the primitive. Inheritance is a Typography-tab concern.

## Phase 2 — Storage model + migration

1. Add a `typography` JSONB column to `templates`. Shape:
   ```
   {
     "buckets": { "heading_family": "...", "body_family": "...", "nav_family": "..." },
     "elements": {
       "h1": { "font": {...}, "margin": {...}, "padding": {...} },
       "h2": { ... },
       ...
       "p":  { ... },
       "ul_li": { "font": {...}, "margin": {...}, "padding": {...}, "list_style_type": "disc",   "marker_color": null },
       "ol_li": { "font": {...}, "margin": {...}, "padding": {...}, "list_style_type": "decimal","marker_color": null }
     },
     "sample_text": "The quick brown fox jumps over the lazy dog."
   }
   ```
2. Migrate data: `heading_font` → `typography.buckets.heading_family`; `body_font` → `typography.buckets.body_family`. Drop both columns in the same migration.
3. Update `Template.php` model (casts, accessors if any, fillable).
4. Update `docs/schema/templates.md` to reflect the new column and the removals.
5. Add a service `App\Services\TypographyResolver` that takes a Template and returns a fully-resolved typography tree (per-element family = element override ?? bucket). This is the single source of truth for compile + preview.

## Phase 3 — Design System Editor: Typography tab (Vue)

1. Locate the Design System Editor admin page. Add a Typography tab. If the page is currently Filament-only (Buttons tab built in session 134), mount a Vue island inside it the same way the page builder mounts into Livewire.
2. Layout:
   - **Top row** — editable "Sample text" field (text input, defaults to the pangram). Persists to `typography.sample_text`.
   - **Bucket row** — three family pickers: Heading, Body, Nav. Family-only. Changes here cascade to any element whose `font.family` is null.
   - **Element rows** — one per element (H1, H2, H3, H4, H5, H6, P, UL li, OL li). Each row contains:
     - A **live preview** block rendering the sample text with the resolved styles.
     - The `font` primitive from Phase 1.
     - The lifted four-side padding + margin picker.
     - For UL li and OL li: list-style-type picker + marker color picker.
   - Inheritance affordance: when an element's `font.family` is null, the family control displays "inherited from Heading/Body" as a hint, with a one-click override. When overridden, a ↩ "reset to inherit" chip appears. Standard tri-state.
3. Pinia store: a single typography store scoped to the currently-edited Template. Debounced save to a new JSON endpoint (`POST /admin/templates/{template}/typography`). Pattern-match the page builder's save flow.
4. No inline validation gymnastics. Trust the resolver to fill missing values.

## Phase 4 — Compile & apply at render

1. Extend or add a compiler (`App\Services\TypographyCompiler`) that consumes `TypographyResolver`'s output and emits CSS: `h1 { font-family: ...; font-weight: ...; ... } h2 { ... } ul li { list-style-type: ...; ... }` etc.
2. Wire the compiled CSS into the public page render. Pattern-match how template `custom_scss` is injected today — inline `<style>` block in the page `<head>`, scoped under `.np-site` if the existing pattern requires it.
3. Admin-side: the Typography tab's live preview uses the same resolver + compiler so preview matches production output byte-for-byte (or as close as practical).
4. No build-server changes. Runtime compilation via the resolver + compiler is adequate; we can promote to build-time later if perf demands.

## Phase 5 — Export

1. Add a "Download SCSS" button to the Typography tab.
2. Route: `GET /admin/templates/{template}/typography/export.scss` — returns a `.scss` file with one selector block per element. Include a header comment with template name + export date.
3. No import today. Out of scope.

## Testing

- **Slow test groups to run this session:** none.
- **New tests expected:** yes.
  - Migration: data round-trip from `heading_font` / `body_font` into `typography.buckets.*` verified on existing seeded templates.
  - `TypographyResolver`: unit tests covering bucket inheritance — element with explicit family wins, element with null family falls back to bucket, null bucket falls back to reasonable default (document what that default is).

  - Export endpoint: returns `.scss` with correct MIME and filename, content matches compiler output.
