## What already exists (verified at the 356 close — build on this, don't rebuild)

### Item 1 — Image uploader (the two surfaces to converge)

Both image fields already call the **same** backend actions; the divergence is purely the **front-end field UI**.

- **Widget config image field** — `resources/js/page-builder-vue/components/fields/ImageUploadField.vue`: a bare file input + preview + remove `×` + (356) a "Browse library" button. Renders for `field.type === 'image'` (video fields share the file input but get no Browse). Calls `store.uploadImage` / `store.removeImage` / opens the browser via `store.openMediaBrowser({ kind: 'config', widgetId, key })`.
- **Appearance background image** — `resources/js/page-builder-vue/components/appearance/BackgroundPanel.vue` (image sub-panel): the **richer** model — Fit (cover/contain), Alignment (`NinePointAlignment`), a "use current page's header image" override, an upload-block / preview, and (356) a "Browse library" button. It is a **presentational** component: it emits `update` / `uploadImage` / `removeImage` / `browseLibrary`; `InspectorPanel.vue` handles them (`onWidgetUploadImage` / `onWidgetRemoveImage` / `onWidgetBrowseImage` → `store.openMediaBrowser({ kind: 'appearance', widgetId })`). `LayoutInspectorPanel.vue` reuses `BackgroundPanel` with `show-image=false` (layouts have no appearance image — leave that path alone).
- **Shared backend actions (do not touch)** — `resources/js/page-builder-vue/composables/useUploadActions.ts`: `uploadImage` / `removeImage` / `uploadAppearanceImage` / `removeAppearanceImage` / `useExistingMedia`. **Shared browser (do not touch)** — `MediaBrowserModal.vue` (356), opened via `store.openMediaBrowser(target)` for both `kind: 'config'` and `kind: 'appearance'`.

**The gap:** the *upload field UI* (preview + remove + file input + Browse button) is implemented twice — bare in `ImageUploadField`, embedded in `BackgroundPanel`'s image sub-panel. Converge on **one shared image-upload sub-component** modelled on the background version (the richer one), with the background-only controls (Fit / Alignment / use-page-header) kept conditional so the bare widget field doesn't grow them.

### Item 2 — Widget thumbnails (the generation pipeline)

- `scripts/generate-thumbnails.js` renders each widget and screenshots its `static.png`; `app/Http/Controllers/WidgetThumbnailController.php` + `app/Models/WidgetType.php` are the render/storage surface. See `docs/widget-development.md` for the documented regeneration flow.
- A widget renders its thumbnail from its **demo data**: the Definition's `demoSeeder()` (returns a `DemoSeeder::class` — e.g. `BoardMembers`, `LogoGarden`, `Carousel`, `BarChart`, `DonationForm`) and/or `demoConfig()` / `demoAppearanceConfig()` / `demoImages()` hooks (e.g. `Hero`, `TextBlock`, `Table`, `SocialSharing`, `EventRegistration`). **Most widgets have neither** — those that depend on data (listings, image-only widgets with no image, etc.) render blank.
- **The gap:** audit the widget set, identify which render blank, and give them representative demo data via the established hook (not a renderer special-case). Some widgets (portal/auth widgets, RandomDataGenerator, SetupChecklist) may be **blank-by-design** — `log()` those rather than forcing content.

### Item 3 — Demo re-enter affordance

- `/demo/enter` (`routes/web.php`, name `demo.enter`) → `app/Http/Controllers/DemoLoginController.php`: `abort_unless(isDemoMode(), 404)`, firstOrCreate the `demo@demo.local` user, `syncRoles(['demo'])`, `Auth::login`, redirect to the admin panel. `isDemoMode()` lives in `app/helpers.php`.
- The login page is Filament's default (`->login()` in `app/Providers/Filament/AdminPanelProvider.php`, which already registers many `PanelsRenderHook`s). **The gap:** once a demo user is logged out, the login page offers no way back in — add a **demo-mode-only** "Re-enter the demo" affordance (a link/button to `route('demo.enter')`) on the login page, via the appropriate auth/login render hook, shown only when `isDemoMode()`.

### Item 4 — Demo dashboard empty-state

- `resources/views/filament/widgets/dashboard-slot-grid.blade.php` (~line 24) renders *"No dashboard arrangement for your role — ask an admin to configure one in Tools → Dashboard View."*
- `database/seeders/DashboardViewSeeder.php` seeds a `DashboardView` for **`super_admin` only**, so the `demo` role resolves nothing → the generic message shows in the demo.
- **The gap:** seed a dashboard arrangement for the `demo` role so the demo shows a real dashboard. It must reach the **demo node** via the **demo baseline** (`demo:restore` / `demo:reset`), not only the dev `migrate:fresh --seed`. (Confirm the baseline path during the walk.)
