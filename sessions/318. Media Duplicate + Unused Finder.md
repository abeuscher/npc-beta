# 318. Media Duplicate + Unused Finder

Admin-tool build. A media-library scanner with two modes: duplicate-detection and unused-media. Outputs a triageable report; destructive operations are operator-confirmed.

## Drift correction (session-start finding — supersedes the original "media-id columns" framing)

The original draft of this prompt assumed the unused scan works by grepping JSON columns for `media_id` / `background_image_id` literals. **That is wrong for this codebase** and the rest of this prompt is written against the actual model. The session-start sweep ([ContentExporter.php](app/Services/ImportExport/ContentExporter.php), the 8 `HasMedia` owners, `PageWidget::registerMediaCollections`) established:

- **There are essentially no literal media-id columns.** Media is attached to an owner model through **Spatie collection ownership**: every `media` row carries `model_type` + `model_id` (the owner) and a `collection_name`. The owner reads it back by collection name (`getFirstMedia('post_header')`, `getFirstMedia("config_{$key}")`, `getFirstMedia('appearance_background_image')`). The config JSON does *not* store a media id.
- So "**referenced**" means: the owner row exists **and** the `collection_name` corresponds to a collection the owner *currently, actively reads*. A media row whose owner exists but whose collection the owner no longer reads (a renamed/removed widget field → a dead `config_oldkey`) is the genuinely-unused case nothing else catches.
- **`media-library:clean`** (scheduled daily, [app-reference.md § Media cleanup]) already deletes the **orphan-owner** case (owner row gone). The scan should *report* that case informationally — flagged as "the nightly sweeper handles these" — but the operator-facing delete targets the live-owner-dead-collection and dead-embedded-URL cases. *(Decision taken at the open gate: report orphan-owner inclusively rather than stay silent.)*

The reference model below is the authoritative inventory for the unused scan.

## Authoritative media-reference inventory

A media row is **referenced** if it matches any rule below. Anything in the `media` table matching none of them is an **unused candidate**.

**A. Spatie collection ownership — live collections per owner type.** For each owner, the set of collection names that owner currently reads:

| Owner (`model_type`) | Live collections |
|---|---|
| `Page` | `post_thumbnail`, `post_header`, `og_image` |
| `Event` | `event_thumbnail`, `event_header`, `event_og_image` |
| `Product` | `product_image` |
| `EmailTemplate` | `header_image` |
| `WidgetType` | `thumbnail`, `thumbnail_hover` |
| `PageWidget` | `appearance_background_image` **plus** `config_{key}` for every `image`/`video` field currently in the owning widget type's `config_schema` |
| `CollectionItem` | the collection-field-derived names (image/logo/video field keys) on the parent `Collection.fields`; SVG-logo collections like `logo` |
| `SampleImage` | per-category dynamic collections (treat all as live — fixture/library data) |

The two dynamic cases (`PageWidget` config fields, `CollectionItem` collection fields) are load-bearing: the "dead collection" condition is *owner exists but `collection_name` is not in the owner's current live set*.

**B. Embedded-by-URL in rich-text content.** Rich-text fields store sanitized HTML that may contain `<img src="/storage/…">` pointing at a media file by URL. These reference media **independently of collection ownership** — a media row referenced only via an embedded `<img>` must NOT be flagged unused. Surfaces holding rich-text:

- `PageWidget` config — `rich_text`-typed fields (e.g. Memos widget `body`).
- `CollectionItem.data` — `rich_text`-typed collection fields (sanitized in `CollectionItem::booted`).
- Page/post content rich-text.

**Resolution approach (refined in-session):** extract the canonical media path token `/storage/{id}/…` from the raw content and resolve the captured `{id}` to a `media` row. This is *safer* than DOM `<img src>`-only extraction because it also catches media referenced via `<a href>`, `srcset`, conversion paths, or `<video>`/`<source>` — an img-only parser would miss those and wrongly flag the media unused. The path generator is `DefaultPathGenerator` (`{id}/{file_name}`) on the public disk, so every in-app media URL carries its id in the first path segment; `inline-images` uploads ([InlineImageUploadController](app/Http/Controllers/Admin/InlineImageUploadController.php), owned by PageWidget/Event/EmailTemplate) are exactly this case and are kept alive only by this rule.

**C. Out-of-scope (documented in the tool narrative, NOT flagged unused):**

- Seeder fixtures / `SampleImage` library — intentionally unattached-looking.
- Orphan-owner media — reported informationally only; deletion is `media-library:clean`'s job.
- Media-reference *history* (what content used a media before it was swapped) — v1 doesn't track it.

## Phases

### Phase 1 — Surface inventory + scan engine scaffolding

Build a `MediaReferenceInventory` service that encodes rules A and B above. Shape: one method per reference rule, each returning the set of referenced media IDs. The scan engine takes the union. Reuse the export system's existing reference-walker logic ([ContentExporter::collectReferencedMedia / serializeWidgetMedia / serializePageMedia](app/Services/ImportExport/ContentExporter.php)) — it already enumerates owner→collection media; the new work is (a) the *liveness* check for dynamic collections and (b) the embedded-URL pass.

### Phase 2 — Unused-media scan

Union the referenced IDs from the inventory, diff against the full `media` table. Classify each unused candidate: `dead_collection` (owner live, collection dead), `orphan_owner` (owner gone — informational), `unreferenced` (no owner / no embed). Report row: media ID, filename, size, upload date, owner (or "none"), classification, preview.

### Phase 3 — Duplicate-media scan

Two passes, clustered output:

- **Content hash** — SHA-256 over the stored original bytes (authoritative; the "same file uploaded twice" case).
- **Filename + size match** — cheap secondary signal for rows the hash pass didn't already cluster.

Output: clusters of N>1 records, each with preview, inferred reason (hash vs name+size), and which records are referenced (so the operator keeps the referenced one).

### Phase 3b — Missing-file (404) scan *(added mid-session at user request)*

A third scan listing media **records whose file is gone from disk** — checked via `Storage::disk($media->disk)->exists($media->getPathRelativeToRoot())` (cross-disk safe). Each row is tagged **Broken (referenced)** — a live surface still points at it, so it's rendering a 404 now — or **Dead record** (stale row, nothing references it). Per-row action is the same confirmed delete (removes the dangling row + leftover conversions). This is the inverse of `media-library:clean`'s orphan-*conversion* sweep and is not otherwise covered.

### Phase 4 — Admin tool page UI

Tools-group page mirroring `SiteImportExportPage` (`manage_cms_settings` gate, breadcrumbs, header actions). Two header actions: Run Duplicate Scan / Run Unused Scan. Narrative section explaining each scan, the inventory (rules A/B), and the out-of-scope list (C). Report table with per-row triage controls.

### Phase 5 — Destructive-operation policy

Per-row operator-confirmed actions via Filament action modals:

- **Delete** — routes through the Spatie model `delete()` (unlinks file + conversions), matching `MediaLibraryPage`'s existing delete action. No bulk bypass.
- **Merge into** — for a duplicate cluster, re-point the duplicates' owners to a chosen canonical record, then delete the others. Confirmed per-merge.

No auto-delete. No path that bypasses per-row confirmation.

### Phase 6 — Tests + sample-data run + runbook

Feature tests: one fixture per inventory rule pinning it is recognized (false-negative protection — the load-bearing tests); embedded-URL reference prevents an unused flag; `dead_collection` classification fires when a widget field is removed; delete + merge require confirmation and merge rewires ownership. Run against seeded local data; capture output for the log. Add `docs/runbooks/media-finder.md`.

## Open questions to resolve at session start

- **Scan execution model.** Synchronous (loading spinner) v1; promote to queued only if the seeded sample data shows a timeout. *(Default: synchronous.)*
- **Duplicate-detection algorithm.** Content-hash + filename/size, two passes. *(Resolved: both.)*
- **Delete primitive.** Spatie model `delete()` (hard unlink, as `MediaLibraryPage` already does) — no separate soft-delete/archive flag in v1 unless the merge flow needs a reversible step. *(Lean: hard delete + confirmation, matching the existing media-page convention.)*
- **Rich-text parsing.** Extract the `/storage/{id}/…` path token from raw content and resolve `{id}` to media rows — a superset of img-only DOM extraction, chosen to minimize false-positive "unused" flags. Surfaces: `page_widgets.config`, `collection_items.data`, `events.description`, `events.meeting_details`, `email_templates.body`. *(Resolved.)*
- **Orphan-owner reporting.** Reported informationally, not operator-deletable in this tool. *(Resolved at open gate.)*

## Out of scope

- Bulk-deletion shortcuts that bypass per-row confirmation.
- Automatic merging of duplicates without operator review.
- Migration of media to a different storage backend (separate concern).
- Re-encoding / format conversion / image optimization (separate concern).
- Tracking media-reference history (which content used this media before it was changed) — out-of-scope for v1.
- Deleting orphan-owner media (that is `media-library:clean`'s job).
