## What was built

Demo-site build session **C** (reshaped + moved ahead of B at the 325 close). The "optional / no landing page" model is retired: **every event now gets a real landing page at creation**, so the mini-calendar and the (next-session) listing always have a resolvable URL to link to. The landing page is a normal page-builder `Page` linked via `events.landing_page_id` — fully editable afterward.

### Phase 1 — Creation flow + preset

- `EventResource::createLandingPageForEvent($event)` builds the LP. It is invoked from **every event-creation path** so no event is left page-less:
  - `CreateEvent::afterCreate()` (admin create form),
  - `EditEvent::afterSave()` (backfill — an event that somehow lacks an LP gets one on next save),
  - `RandomDataGenerator::generateEvents()` (generated/demo events),
  - the events importer progress page (imported events).
- **Registration-gating is automatic, not an operator toggle.** The original spec described an explicit Simple/Standard choice on the create form; as built, the registration card is seeded **only when the event needs it** — paid or capacity-capped, via `EventResource::eventNeedsRegistration()`. Owner-confirmed at close: the operator has full control of the page afterward, both directions are cheap to fix, so the automatic derivation is the final shape. No persisted Simple/Standard flag, no schema change.
- **The preset is a two-column layout** matching the owner's hand-built reference page:
  - One `page_layouts` row: `display: grid`, `columns: 2`, `grid_template_columns: 1fr 1fr`, `gap: 1.5rem`, `background_full_width: true`, framing padding **top 120px / bottom 300px**.
  - **Left column:** `event_image` (`image_source: header`).
  - **Right column:** `event_description`, then (paid/capped only) `event_registration`, then `social_sharing` — left-aligned, heading-less, 25px top padding.

### Phase 2 — Deletion protection

- `PageResource::guardLandingPageDeletion($page)` returns false when the page is referenced as an event's `landing_page_id`, wired into the Page list/delete action so an event's LP cannot be deleted out from under it (the existing block-with-counts shape; as built it lives on the `PageResource` delete action rather than `PageObserver`, but mirrors that guard pattern).

### Phase 3 — Relink + simplify + new widget

- **New `event_image` widget** (`app/Widgets/EventImage/`) — renders a chosen event's header or thumbnail image with fit / aspect-ratio / max-width controls and a fallback between the two sources. Registered in `WidgetServiceProvider`; a normal pickable widget (`events` / `media` categories).
- `SystemModelProjector::projectEvent` — removed the `/events`-index dead-end `url` fallback now that every event has an LP; `has_landing_page` simplifies accordingly.
- `EventMiniCalendar` — event rows + template relinked to the always-present LP URL; dead-end-era logic stripped; the inline expand kept as a rail-peek.
- `TemplateSeeder` — dropped the now-redundant "Event Landing Page" content template (the preset is built directly by `createLandingPageForEvent`).

### Tests

- `EventLandingPageTest` — preset shape, automatic registration-gating (free → no reg; paid/capped → reg), the no-op-when-already-set guard, the two-column structural layout (columns, framing padding, per-column widget order, share config), the deletion guard, and the create-form lifecycle. The `landingPageHandles()` helper now orders by `column_index` then `sort_order` so layout-column assertions are deterministic.
- New `EventImageTest`; updates to `EventTest`, `EventDuplicateTest`, `ImportEventsTest`, `TemplateTest`, `WidgetPickerSession119Test`.
