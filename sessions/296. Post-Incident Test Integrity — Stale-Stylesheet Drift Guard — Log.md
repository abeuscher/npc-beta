## What was built

### Phase 1 — The drift guard (centerpiece)

- **`AssetBuildService` made output-dir injectable (Open Question 2 → option (a)).** Constructor now `__construct(?string $outputDir = null, ?string $libsDir = null)`, defaulting to `public_path('build/widgets')` / `public_path('build/libs')`. Container auto-resolution (`app(AssetBuildService::class)`) is unaffected (params are optional/nullable). This is the single injection point both the pollution fix and the guard test ride. No call-site changes required.
- **Single source of truth for the hash.** Extracted `hashSources(array): string` (private) and `sourceHash(): string` (public); `build()` now calls `hashSources($sources)` instead of an inline `substr(md5(json_encode(...)),0,8)`, so `build()` and the guard can never disagree on derivation.
- **`bundleDrift(): ?string`** — returns `null` when the served bundle matches current source, else a human-readable reason. "Stale" is defined precisely: manifest absent while sources exist; manifest unreadable/corrupt; manifest declares no CSS bundle while sources exist; or the hash encoded in the served CSS filename ≠ recomputed `sourceHash()`. The method is scoped **by intent** to the build-server bundle (`collectSources()` → `manifest.json`); its docblock states that the incoming re-taxonomy's request-time inline per-template scheme overrides are out of scope by design (they are render-time composed, deliberately drift-proof).
- **The optional `built_at` heuristic was deliberately *not* added.** It is ambiguous: an identical re-save bumps a `SiteSetting.updated_at` without changing the bundle, which would report a false stale. The primary content-hash check is already robust to the re-taxonomy because `collectSources()` folds in the whole design group regardless of which keys it carries. The decision and the "if ever added, resolve `DESIGN_SETTINGS_GROUP` dynamically — never a hardcoded `button_styles`/`typography` list" requirement are captured in the method docblock and the `DESIGN_SETTINGS_GROUP = 'design'` class constant.
- **Optional CLI wrapper (Open Question 3).** `php artisan assets:check-drift` (`CheckAssetDriftCommand`) — thin, non-boundary, exits non-zero with the reason when stale. Cheap operator convenience for the post-deploy gap the s295 production notes flagged; auto-discovered by Laravel 11.
- **Two-direction regression test** — new `tests/Feature/AssetBundleDriftGuardTest.php` (4 tests): fresh immediately after a faked-HTTP build into a temp dir; stale after a design-group `SiteSetting` changes with no rebuild; stale when no manifest exists while sources do; stale when the manifest declares no CSS. Builds only into an injected temp dir; an unconditional `afterEach` asserts the real `public/build/widgets/manifest.json` is byte-identical before/after — proving zero real-FS pollution.

### Phase 2 — Stop the suite manufacturing the incident

Isolation mechanism for all build-writing tests: the injectable constructor (Phase 1) + a `try/finally` (or unconditional `afterEach`) that deletes the temp dir even when an assertion throws — the exact discipline the incident's tests violated. After this session no test writes or deletes anything under `public/build/**` or the real `manifest.json`.

- **Finding 1 — `BuildServerSettingsSession125Test`** ("prefers site settings…", "falls back to env config…"). Old: `app(AssetBuildService::class)->build()` wrote the real manifest + a `.test{}` stub, no cleanup, `$result` never asserted. New: build into an injected temp dir under `try/finally`; assert `$result->success === true` **and** the URL-selection behaviour. Now a fully-failed build cannot pass.
- **Finding 2 — `WidgetJsLibsSession138Test`** ("buildLibraryBundles produces files…", "manifest includes libs key…"). Old: wrote real `public/build/libs/*` + manifest, then `File::deleteDirectory(public_path('build/...'))` — deleting the real served dir, cleanup unguarded. New: build into an injected temp dir; assert lib files / manifest shape there; no `public/build/**` writes or deletes; `try/finally` cleanup.
- **Finding 3 — `DashboardSlotGridTest`** ("dashboard page loads…"). Old: read the real on-disk manifest; asserted the swiper markup only `if` the untracked file happened to exist (silent no-op on a clean checkout — the false-green class). New: the load test drops the real-FS branch entirely; a **new deterministic test** ("injects the swiper lib tag when the served manifest declares it") binds a stub `WidgetAssetResolver` (its `?string $manifestPath` ctor) at a temp manifest and asserts `data-widget-lib="swiper"` unconditionally. Split → +1 test.
- **Finding 4 — `tests/e2e/page-builder/full-width-matrix.spec.ts`** (editor solid-bg / gradient-bg). Old: asserted `getComputedStyle(...).backgroundColor/Image` — a stale `public/build` would still produce a green run. New: assert the element's **inline `style` attribute** (`el.style.backgroundColor` / `el.style.backgroundImage`). Confirmed via `LayoutRegion.vue` that the layout background is delivered by the Vue `:style="appearanceStyleString"` binding (render-time, bundle-independent — the deliberately drift-proof layer). Stale-bundle-proof by construction; comment states the bundle-only-by-intent boundary.
- **Finding 5 — `DesignSystemButtonsTest`** (kept; fixtures de-evoked only). `#ff0000`/`#ff5500`/`pill` → `#123456`/`rounded` in the two fixture-bearing tests and their assertions (`--btn-primary-radius` now `0.5em`, the `rounded` map value). `rounded` differs from the `slightly-rounded` default, so the round-trips still prove the override path. No elaboration/hardening (per Forward-awareness #3); the secondary-dark-defaults test was untouched (no evocative fixture).

### Phase 3 — Remediate the "proves nothing" cluster

Net 4 → 3.
- **Finding 8 — `CustomFieldTest`** "contact export includes custom field columns after standard columns". Old: asserted `CustomFieldDef` DB rows; a code comment conceded "action integration is manual-tested" — the name's claim was never exercised. New: invokes the real `exportContacts` action and asserts the CF column header is appended after the standard columns (`first_name` first, `Member Number` last) and its value lands in the last data cell. Count-neutral; name is now true.
- **Finding 9 — `TaxReceiptTest`** ("…correct total…", "…multi-fund breakdown"). Old: re-implemented `DonorsPage::buildBreakdown()` inline ("Simulate the buildBreakdown logic…") and asserted its own copy. - **Finding 10 — `PublishedAtBackfillTest` — DELETED (whole file, 6 tests).** Rationale (evidence-tied): confirmed at start that no live `published_at` backfill migration or command exists (`grep -rn "published_at" database/migrations app/Console` and `"published_at = created_at\|backfill"` across `database/`+`app/` non-test code both empty). The tests ran raw `DB::statement("UPDATE … SET published_at = created_at …")` inline and asserted the SQL did what the SQL says — they guarded nothing. Open Question 1 resolved to **delete**: no real missing feature surfaced (no legacy-null-row backfill artifact anywhere in the repo), so no scope expansion / surface.
- **Finding 11 — `WidgetDefaultsLintTest`** "detects a toggle-with-string-default as a lint failure" → renamed "the widget lint rejects a definition whose defaults omit a declared schema key". Old: asserted `is_bool('yes')` is false — re-implemented the rule, never invoked the linter (a broken linter passed). New: constructs a `WidgetDefinition` whose `defaults()` omit a key its `schema()` declares and asserts the **real** `WidgetDefinition::validate()` (the production lint entrypoint `WidgetRegistry::sync()` runs) throws the documented `RuntimeException`. No sibling already invoked `validate()` over a crafted bad def, so rewritten not deleted. The three `widget-lint`-grouped tests were untouched.
