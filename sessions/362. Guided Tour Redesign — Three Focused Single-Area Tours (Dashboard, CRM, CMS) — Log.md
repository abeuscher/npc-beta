## What was built

### Phase 1 — Retire the multi-page walkthrough + single-area engine

- **Retired** the session-338 multi-page product tour wholesale: `tours/product-tour.js` (the 11-step definition), the page-group/cross-page engine in `controller.js`, `state.js` (localStorage resume), the session-361 persistent control bar (+ its CSS), the role-gated showcase fallbacks (`TourRolesShowcasePage` + `TourImportShowcasePage` + views + `TourShowcasePagesTest` + the `@on-demand` screenshot spec + the committed import-showcase PNG), and every marker no new tour uses (`page.content`, `record.memberships`, `help.flyout`, `topbar.view-site`, the two `view-transactions` attributes). Owner's rule applied: delete anything not explicitly named in the reworked tour scripts.
- **Kept the durable 361 keepers untouched:** the 2FA e2e auth fix (`tests/e2e/helpers/db.ts` + `auth.ts`) and the eager `MembershipsRelationManager`.
- **New single-area engine** (`controller.js` rewrite): a tour is an ordered step list within one admin area; driver.js drives one segment per page. The **only cross-page surface is a one-shot sessionStorage launch flag** (consumed on read) — used by the conclusion-modal handoffs and the CRM tour's single contained hop. No URL-keyed page map, no resume; an abandoned tour just ends.
- **`anchors.js` additions** (primitives kept): a `navGroup` resolver (spotlight a whole sidebar group located by our own route URL), per-anchor `timeout` overrides for legitimately-conditional targets, and `ensureNavVisible` retries (a group toggle clicked before Alpine boots is dead — retry until it answers).
- **`AdminPanelProvider`** URL map rebuilt around the new anchor set (contacts/pages homes + the sidebar items the tours point at + the hero contact), still emitted per-key only when the viewer's role can reach the page — a missing target now degrades to a centered popover, never a 404 or a skipped beat.

### Phase 2 — The three tours + verification

- **Tour A — Dashboard Overview** (8 steps, stays on `/admin`): sidebar walk per the owner's `main-script.md` (CRM group, Memberships, Donations, Events, CMS group, Mailing Lists, help search) ending in a **conclusion modal with "Tour the CRM" / "Tour the CMS" handoff buttons**.
- **Tour B — CRM deep-dive** (8 steps): runs on the Contacts list; **one contained hop** — the user's own click on the spotlighted hero contact (or Next as the safety net) carries it into the record, where Notes (the record's ⋮ action menu), Custom Fields (the form section, marked via `Section::extraAttributes`), Custom Views, and Mailing Lists finish. Copy from `crm-script.md`; the click invitation is the one mechanical addition.
- **Tour C — CMS deep-dive** (7 steps, fully single-page): runs on the Pages list, pointing Widgets/Forms/Templates/Theming/Custom Snippets at their sidebar homes. Copy from `cms-script.md`.
- **Launchers:** the dashboard welcome widget carries "Tour the Product" (Tour A) plus direct **"Tour the CRM" / "Tour the CMS"** buttons (owner review request); `data-np-tour-start="<id>"` starts in place, `data-np-tour-goto="<id>"` navigates to the tour's home page and starts there via the launch flag.
- **Copy is the owner's three scripts verbatim** (`sessions/tourscripts/`, committed as the canonical source). A **storyboard artifact** (one live-captured frame per step + flagged copy notes: script typos, brand spellings, the "Customer Resource Manager" expansion, the added click sentence) was produced for owner rulings — **rulings deferred until external demo feedback lands**.

### Post-review fixes (owner smoke-test findings)

- **Next on the interactive step:** kept alongside the click invitation; a `navigateTo` URL-map key on the step makes Next navigate to the same destination, so an inattentive user is never stranded.
- **Hop delay killed — measured, not guessed:** baseline **9.1s** from navigation to popover on the record. Two causes fixed: (1) flag-launched resume had been moved to the full `load` event to dodge the Alpine sidebar race — reverted to `DOMContentLoaded`, with the race fixed at its source (`ensureNavVisible` retries); (2) the engine resolved every segment anchor serially before first paint, so one conditionally-absent marker billed its full timeout to the tour's appearance — now **only the entry anchor gates the drive**, the rest resolve in the background as recorded promises the advance path awaits per step (an `advancing` flag serializes Next). Re-measured: **1.5s**, ~0.5s of which is the deliberate settle-before-position gate (361 lesson, kept).
- **Mid-session deploy:** VERSION bumped to `0.362.01`, branch pushed, owner merged and deployed to the demo node for external feedback.

### Emergent — mailing-list filter bug scoped (owner-surfaced while walking the tour)

- Reproduced end-to-end on a clean e2e stack: the Simple Filters repeater declares **three same-named `value` components** (text / Yes-No select / tag picker) sharing one state path; the hidden inapplicable ones clobber the chosen value to `NULL` at save; `MailingListQueryBuilder` compiles `equals NULL` to `WHERE field IS NULL` → 0 matches; the refilled form shows the NULL. Likely inert since session 041 for **all** value-carrying simple filters. Scope verdict: **housekeeping-sized**; captured via `logbug` with the full diagnosis and made the anchor of session 363. Owner correction recorded: an "all opted-in contacts" list is a real use case — no default list exists, so a filtered list is currently the only way to express that segment.
