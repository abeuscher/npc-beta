## Session-specific deltas

- **The public marketing site is complete and lives on a different node.** The demo lives at `demo.nphelper.com` on its own droplet, isolated from the real `nonprofitcrm.com` site. The two never share data, never share a database, never share content. The demo node is the planning surface here.
- **The "demo page" itself is the auto-login arrival page.** It's the locked landing — a prospect arrives via `/demo/enter` on the marketing site, gets auto-logged into the demo instance, and lands on this page. They cannot edit it. The other demo pages on the demo node *are* editable, and the editing-from-the-back-end experience is the whole point of the demo.
- **Driver.js tour is coming next.** Soon after the demo pages land, a driver.js-based product tour walks prospects through the demo install. The plan must reserve the right integration hooks (a stable data-attribute scheme, slot positions in the demo pages the tour can anchor to, navigation behavior across pages). Tour build is not this session and not 325; it's downstream.
- **Portal pages — tour-only, included.** The product's portal feature isn't visible anywhere on the demo node today. Tour-only inclusion (read-only seeded portal-shape pages walked by driver.js, no live portal login) gives prospects a credible look at the portal without exposing it to abuse.
- **Calendar widget retirement.** The current `EventCalendar` widget is being **removed entirely** — not demoted to admin-only. The back-end calendar use case is deferred to scheduling work post-Beta. Sweep all references.
- **New simple calendar widget.** A vue-cal-wrapped widget for the events index mini-calendar (right rail, density indicators, click-to-scroll). The mount pattern (Vue runtime on the public side vs imperative per-instance) is an open question for the open gate.
- **Events listing widget gets an upgrade pass.** Add a list-mode preset (chronological day-grouped, the luma shape) alongside the existing Swiper-on layout. Plus a filter row (Event Type + Date Range), a featured-event hero option, and the art-fallback hooks the index spec calls for. Existing widget, additive changes.
- **Event LP work depends on what survives the spec cut.** The LP spec lists 10 sections including 6 registration-card variants, sponsors, hosts, volunteer signup, past-years gallery, practical info, and print stylesheet. Most of these will not be in v1. The 324 conversation prunes the LP to a credible v1 surface.
- **Per-page permission, not exception.** The locked landing page (and any tour-only portal pages) need a generalizable per-page permission mechanism — a new perm concept on pages that the demo role can't bypass for those specific pages. Settle the design in 324. The build session implements it.

## Open questions to resolve at session start

These are real decisions the owner takes a position on before any drafting happens. Bring a recommendation for each; settle and move on.

- **vue-cal mount pattern.** Three options: (a) stand up a small Vue 3 runtime on the public side and mount vue-cal as a regular component; (b) imperatively mount a vue-cal instance per calendar in a per-widget shim; (c) reject vue-cal, pick a vanilla-JS calendar lib (Cally, a hand-rolled mini-cal, etc.). Lean: (a) if we have any precedent for Vue on the public side, else (b); fall back to (c) only if the architectural cost reads as too high.
- **Portal pages — confirm tour-only scope.** Confirm: portal pages are seeded, read-only, walked by the driver.js tour; no actual portal-user login on the demo node. Lean: yes.
- **Per-page permission shape.** Three sketch-options to evaluate: (a) a `restricted_roles` column on `pages` listing roles barred from editing that page; (b) a separate `page_permissions` table joining pages to roles with an action verb (edit / publish / delete); (c) a flag column (e.g. `is_locked_from_demo: bool`) — concrete but the most exception-like, included for comparison. Lean: (a) — generalizable, simplest schema, room to grow. Owner explicitly prefers "not an exception" so (c) is the comparison case, not the recommendation.
- **Header/footer duplication mechanism.** The public-facing portion of the demo node (the locked landing page) carries a duplicated header + footer from the main public site and links back to it. Options: (a) point-in-time clone, frozen until manually refreshed; (b) imported with the demo:reset cycle from a versioned export; (c) live fetched from the main site at render (rejected — coupling). Lean: (b) — fits the existing demo:reset wiring, gets refreshes for free.
- **driver.js integration hook shape.** Reserve a data-attribute scheme for tour anchor points (e.g. `data-tour="events-index-list-row"`) so widget templates can declare them now; settle the convention so the tour build doesn't have to retroactively annotate every template. Lean: yes, settle the convention.

## Phases

### Phase 1 — Walk the specs, prune to v1

Walk `sessions/demo website/events-index-spec.md` and `sessions/demo website/event-lp-spec.md` section-by-section with the owner. For each item: **in v1 / deferred to v2 / dropped entirely**. The output is two annotated cut lists.

Likely high-friction conversations (raise explicitly): registration card variants (which of the 6 are in v1), sponsors model, hosts model, generated-tile fallback, past-years gallery, practical info, print stylesheet, mini-calendar density indicator behavior, member-pricing display, art-fallback chain (uploaded → event-type default → generated tile).

### Phase 2 — Settle the architecture pieces

Three pieces that aren't in the spec but block everything downstream:

- **Per-page permission design** — schema shape, where it surfaces in the admin UI, how it's enforced on the editor route, how the demo:reset wiring re-applies it, default values for existing pages (everyone can edit, opt-in to lock).
- **Calendar mount pattern** — confirm the open-gate decision and write down the implementation shape so 325 (or whichever build session takes the calendar widget) opens with no architectural surprise.
- **Driver.js anchor convention** — the data-attribute scheme, whether anchors live in widget templates or in page-level wrapping markup, how the tour authoring surface picks them up.

### Phase 3 — Multi-session plan

Sketch the work after 324 into right-sized sessions. Likely shape (subject to Phase 1 outcome):

- **Events module — calendar swap.** Retire `EventCalendar` widget. Build the vue-cal-wrapped calendar widget. One session.
- **Events module — listing widget upgrade.** Swiper toggle / list preset / filter row / featured-event hero / art-fallback hooks. One session.
- **Events module — landing page** (v1 cut of the LP spec). Possibly one session, possibly two depending on where the LP cut lands.
- **Per-page permission build + locked landing page.** The lockout mechanism + the demo landing page authored against it. One session.
- **Demo node assembly** — header/footer duplication wiring + demo:reset extension for the curated pages. Possibly merged with the locked-landing session above if both are small enough. One session.
- **Portal demo pages (tour-only).** Read-only seeded portal pages + the driver.js anchor convention applied to them. One session.

The exact session list, count, and order land at 324 close — driven by what Phase 1 cuts and what Phase 2 settles. Don't commit to a count this early in the doc.

### Phase 4 — Produce the deliverables

- Write `sessions/demo website/demo-site-plan.md` — captures Phases 1–3. The v1 cut of both specs, the per-page-permission design, the calendar mount pattern, the driver.js convention, the demo:reset wiring approach, the header/footer duplication approach, and the multi-session work plan.
- Draft `sessions/325. base-prompt.md` + `sessions/325. {title}.md` from the templates. The title and scope are whatever Phase 3 settles as the right first build session.
- Reference any local-only image assets (luma + SFMOMA references) by relative path from `sessions/demo website/scrap/` with a "(local-only, not committed)" annotation.
