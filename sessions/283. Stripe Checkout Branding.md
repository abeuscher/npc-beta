# 283. Stripe Checkout Branding

Implements consistent branding across the four Stripe Checkout flows (donation, event registration, product, membership). **First session of the Public Website Complete milestone** (Rule 12 in `release-plan.md`); execution-order position 33.

---

## Plan reference

This session executes `sessions/release-plan.md` § E4. Read that entry first — it is canonical for scope, success criterion, prerequisites, and artifact.

The E4 entry calls for: audit Stripe API constraints, then implement consistent branding across product / donation / event / membership checkouts using `custom_text`, statement descriptors, and line-item display fields as available.

---

## Session-specific deltas

Lifted at 282-close conversation (after the E4 entry was written):

- **Stripe Dashboard branding belongs in docs, not API.** Most of what an investor will visually associate with "branded checkout" — logo on the Stripe Checkout page, brand color, business display name, support contact info, Terms-of-Service URL, Privacy URL — is set in the Stripe Dashboard, not via API. This session does **not** ship those (we can't), but it **does** ship documentation telling the operator exactly what to set in the Dashboard, plus a new item in the Onboarding Checklist widget (E1) linking to Stripe Dashboard with the per-item list.
- **Common helper if needed.** Audit the four Stripe-session-creating controllers (`DonationCheckoutController`, `EventRegistrationController`, `ProductCheckoutController`, `MembershipCheckoutController`) at session start. If they diverge today on branding-field passing, lift a shared `StripeCheckoutBuilder` (or similar) service in this session. If they already share a path, extend that path instead.
- **Copy-string home.** `custom_text` values (terms-of-service link helper, submit-button helper text) live in `SiteSetting` with sensible defaults. Surface the fields on the natural settings page — likely `CmsSettingsPage` since they're customer-facing copy — gated by `manage_cms_settings`. Confirm placement at session start against existing convention.
- **Statement descriptors.** Configure `statement_descriptor` + `statement_descriptor_suffix` per session, sourced from the org name in SiteSetting. Stripe rules: 5–22 chars, alphanumeric + spaces, no special characters. Document the constraints in the help doc.
- **Tests: Pest only.** Skip Playwright — Stripe Checkout pages are third-party-hosted and not stable test targets. Pest tests assert each controller passes the configured fields to `Session::create` (mock the Stripe client per existing convention).

## Open questions to resolve at session start

- **Helper location and naming.** If the four controllers diverge today and a shared helper is warranted, confirm naming + namespace before lifting (e.g. `App\Services\Stripe\CheckoutBuilder`). If the controllers already share a path, no new helper.
- **Settings page placement.** `CmsSettingsPage` is the lean default; verify against `GeneralSettingsPage` / `MailSettingsPage` convention before adding the section.
- **Help-doc file name.** Proposed: `resources/docs/stripe-checkout-branding.md`. Confirm against existing naming convention.
- **Onboarding Checklist item placement.** Per the E1 implementation (closed at session 249), checklist items live in three bands (required-to-launch / required-for-features / optional). Stripe Dashboard branding is best filed under "required-for-features" since the feature *works* without branding configured, but it's not demo-ready until set. Confirm at session start.

## Out of scope

- Per-fund or per-event override of `custom_text` strings — default-only this session. Per-record overrides are a follow-on if real demand surfaces.
- Stripe Dashboard configuration itself — the operator runs the Dashboard config based on our doc. We don't automate it.
- Per-tier branding for memberships — all tiers in one membership Checkout session share the same branding fields.
- Branded transactional receipt emails — that's **C3b** (Auto tax receipt email), a separate entry post-milestone.
- Cross-product branding (e.g. unified brand color across CRM admin UI, public website, and Stripe Checkout) — that's the Design System Editor track, post-Beta.
