# 107. Paid Event Registration & Membership Checkout

Wire paid event registrations and paid membership signups through Stripe Checkout, creating transactions and triggering QB sync via the existing pipeline. Free events and complimentary memberships continue to work as they do today.

---

## Design decisions (resolved before starting)

- **Event checkout pattern:** follow the same Stripe Checkout Session pattern used by donations and products. The event registration form submits to a new `EventCheckoutController`, which creates a pending registration and redirects to Stripe. The webhook handler completes the registration on `checkout.session.completed`.
- **Membership checkout pattern:** similar flow. A `MembershipCheckoutController` creates a pending membership, redirects to Stripe, and the webhook completes it. For recurring tiers (monthly/annual), use Stripe Checkout in `subscription` mode.
- **Transaction creation:** both flows create a `Transaction` record on webhook success, with `subject_type` pointing to `EventRegistration` or `Membership` respectively. This feeds the existing QB sync pipeline — no new subject type keys needed in the account map for now (they fall through to `default`).
- **Existing schema:** `event_registrations.stripe_payment_intent_id` already exists. `memberships.amount_paid` exists. Add `stripe_session_id` to both tables if needed for webhook correlation.
- **Free path unchanged:** `EventController::register` continues to handle free events. The registration form conditionally submits to the checkout controller when `event.price > 0`. `SignupController::store` continues to handle free/complimentary tier signups.
- **No new account map keys this session.** Event and membership transactions use the `default` QB account. Per-type mapping for these can be added later if needed.

## Phase 1 — Paid Event Registration

### Migration (if needed)

Add `stripe_session_id` (nullable string) to `event_registrations` if not already present, for webhook correlation. The existing `stripe_payment_intent_id` column stores the PI after checkout completes.

### EventCheckoutController

New controller: `app/Http/Controllers/EventCheckoutController.php`

- `store(Request $request, string $slug)`: validates input, verifies event is published and not at capacity, verifies `price > 0`, creates a pending `EventRegistration` (status: `pending`), creates a Stripe Checkout Session in `payment` mode with `event_registration_id` in metadata, and redirects to Stripe.
- Uses `stripe_payment_method_types` from site settings (same as donations/products).
- Success/cancel URLs point back to the event landing page with query params.

### Event registration form update

The existing event registration widget/form currently POSTs directly to `EventController::register`. When the event has a price > 0, the form should instead submit to the new checkout controller. The free path remains unchanged.

### Webhook handler

Extend `handleCheckoutSessionCompleted` in `StripeWebhookController`:

- Detect `event_registration_id` in metadata.
- Find the pending registration, update status to `registered`, store `stripe_payment_intent_id`.
- Create a `Transaction` with `subject_type: EventRegistration`, `subject_id`, amount, direction `in`, status `completed`.
- Dispatch `SyncTransactionToQuickBooks`.

## Phase 2 — Paid Membership Signup

### Migration (if needed)

Add `stripe_session_id` (nullable string) to `memberships` for webhook correlation.

### MembershipCheckoutController

New controller: `app/Http/Controllers/MembershipCheckoutController.php`

- `store(Request $request)`: validates tier selection + contact/account info, verifies tier has a price, creates a pending `Membership`, creates a Stripe Checkout Session (payment mode for one_time/lifetime, subscription mode for monthly/annual), redirects to Stripe.
- Metadata includes `membership_id`.

### Signup flow update

When the selected membership tier has a `default_price > 0`, the signup form redirects through the checkout controller instead of completing immediately. Complimentary tiers (null/zero price) continue through `SignupController::store` unchanged.

### Webhook handler

Extend `handleCheckoutSessionCompleted`:

- Detect `membership_id` in metadata.
- Find the pending membership, update status to `active`, set `starts_on`, `amount_paid`.
- Create a `Transaction` with `subject_type: Membership`.
- Dispatch `SyncTransactionToQuickBooks`.

For subscription mode, also handle `invoice.payment_succeeded` to create renewal transactions.

## Out of scope

- Per-event or per-membership-tier QB account mapping (uses `default` account)
- Discount codes or promo pricing
- Partial payment / payment plans
- Refund handling for event cancellations (handled in Stripe dashboard)
- Membership renewal reminders (existing `renewal_notice_days` column reserved for future)
