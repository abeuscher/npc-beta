## Session-specific deltas

What the track doc defers to session time, plus what CB1 (366) settled:

- **Renders through CB1's reader.** The page consumes `App\Services\Billing\BillingStateReader` and the `BillingState` DTO (accessors: `isPresent()`, `asOf()`, `status()`, `plan()`, `nextInvoice()`, `billingContactEmail()`, `portalUrl()`, `reason()`, `graceEndsAt()`, `trial()`). It adds no read path, no cache, no migration. The document schema is the one authored in the contract at v2.6.0 (`docs/fleet-manager-agent-contract.md` § Client billing — suspension flag + billing-state document).
- **The page** (Filament, **Settings group → "Account"**), mirroring `FinanceSettingsPage` for the `canAccess()` gate + nav group + `getBreadcrumbs()`, but rendering **non-Eloquent** content (from the pushed document, not a record — infolist-from-array or a custom Blade view). Shows:
  - Plan name + price; **subscription status as a plain-English badge** (e.g. "Active" / "Payment problem — please update your card" / "Trial — N days left" / "Canceled") derived from `status()` / `trial()`.
  - **Next invoice** date + amount with its line items (subscription + any project-work hours); amounts are integer minor units + currency per the schema — format for display.
  - **Billing contact email, read-only.**
  - A **"Manage billing" button** linking to `portalUrl()` (Stripe's hosted portal). The only "Stripe" reference in the new code — a link label.
  - A **staleness footer**: "Billing information as of {relative time}" from `asOf()`, with a note that payments made just now can take up to a day to appear.
  - **Self-hides** from navigation + `canAccess()` false when no document is present (internal nodes / fresh installs).
- **Banners (grace/lock warning).** When the document's suspension state is past-due or in-grace: a prominent banner **on the account page** with the date the admin panel will lock and the portal link; plus a **slim version across the whole admin panel** so the person who can fix it sees it before the lock, not at it. (Grace framing = dunning → 14-day grace → lock, confirmed at CB1; reuse the wording.) The panel-wide banner is a render-hook injection (see `AdminPanelProvider`'s existing `renderHook` usage) reading through the same `BillingStateReader`.
- **Permission `manage_account`** — seeded through `PermissionSeeder` exactly like `manage_cms_settings` (a `canAccess()` override on the page; **no new role system, no migration**). Per open question 7, granted to **no shipped role** → super-admin-only unless a client grants it via a custom role. `docs/runbooks/permission-matrix.md` and `tests/Feature/PermissionMatrixTest.php` get the matching row, documenting the no-role grant as **intentional** (the deliberate version of a pattern the permission audit once flagged as an accident).
- **Convention-drift guard (§ Design decision 6), added to `tests/Feature/Infrastructure/ConventionDriftTest.php`:**
  1. **No CRM code outside the mapped donation surface** may reference a vendor-billing identifier (`stripe_billing`, `vendor_stripe`, `STRIPE_BILLING*` — the FM-side names, reserved and banned CRM-side).
  2. **The billing/account code** (`app/Services/Billing/*` + the new Account page) may not reference `services.stripe.*`, `StripeClient`, or any Stripe SDK class — it renders from the pushed document alone. The mapped donation surface (`StripeCheckoutService`, the checkout controllers, `StripeWebhookController`, the `services.stripe.*` namespace) is explicitly allowed and must not be caught.
- **Help doc** stub at `resources/docs/{handle}.md` for the new page, per the style rule.

- **New tests expected: yes.**
