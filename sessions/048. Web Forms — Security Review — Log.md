## Finding 1 — Regex ReDoS (High) — Fixed

**Confirmed.** `Form::fieldValidationRules()` had a `try/catch` as a secondary net at submission time, but no save-time validation. An admin could save a form field with a pattern like `(a+)+$`, which causes exponential backtracking on crafted input and can hang a PHP worker.

**Fix:** Added a `->rules(fn () => [...])` closure on the `validation_regex` TextInput in `FormResource::fieldRepeaterSchema()`. The closure:

1. Confirms the pattern compiles (`@preg_match` returning `false` → "This regex pattern is invalid.")
2. Tests the pattern against `str_repeat('a', 50) . '!'` with `pcre.backtrack_limit` temporarily set to 1 000. If `PREG_BACKTRACK_LIMIT_ERROR` is returned, the save is blocked with: "This regex pattern is too complex and may cause performance issues. Simplify the pattern."

**Note on Filament closure evaluation:** The first implementation used `->rules([function(...) {...}])` which triggered `BindingResolutionException` — Filament's container tried to inject `$attribute`/`$value`/`$fail` as dependencies. Fixed by wrapping in an outer Filament closure: `->rules(fn () => [function(...) {...}])`. Filament evaluates the outer closure and passes the returned array to Laravel's validator, which handles the inline validator closure correctly.

**Also fixed:** The `->visible()` condition on `validation_regex` and `validation_message` fields did not evaluate correctly on initial page load in a Repeater (Filament hydration timing issue). Changed both to `->hidden(fn () => $get('validation') !== 'custom_regex')` which evaluates reliably on initial render.

## Finding 2 — Hidden field value tampering (Medium) — Refuted

**Refuted.** `Form::fieldValidationRules()` skips hidden fields entirely, so they are absent from `$validated`. `array_intersect_key($validated, array_flip($fieldHandles))` therefore produces no hidden-field entries. The loop then unconditionally writes `default_value` from the form definition. A tampered POST value for a hidden field handle is discarded at the validation step and cannot reach the stored submission.

## Finding 3 — Submission flow test coverage — Done

Created `tests/Feature/FormSubmissionTest.php` with 10 test cases:

1. Valid submission stored with correct `form_id`, `data`, and `ip_address`
2. Honeypot filled → silent success, no submission stored
3. PII value (`4111111111111111` — Visa test PAN) → rejected, no submission stored
4. Inactive form → 404
5. Non-existent handle → 404
6. Required field missing → validation error
7. Hidden field `default_value` used even when attacker submits a different value (confirms Finding 2 refutation)
8. `form_type = contact` with email mapped → contact created with correct field values and note
9. Email collision → existing contact updated, note added, `contact_id` set on submission
10. Email collision, `mailing_list_opt_in` already true, form submitted without opt-in → opt-in unchanged

## Out-of-scope items folded in by user request

### Custom error message wiring (bug fix)

The `validation_message` field existed in the form builder since session 047 but was never wired up. Two fixes:

- **Server-side**: Added `Form::fieldValidationMessages()` which builds a Laravel messages array keyed `{handle}.regex`. Passed as second argument to `$request->validate($rules, $messages)` in `FormSubmissionController`.
- **Client-side**: Added `title="{{ $errMsg }}"` to inputs that have a `pattern` attribute in `public-form.blade.php`. Browsers show the title text alongside pattern-mismatch errors.

### Hint text field (new feature)

Added a "Hint text" `TextInput` to the form field repeater schema (hidden for `hidden`-type fields, `->columnSpanFull()`). On the public form, the hint renders as `<small>` below the input, before any validation error message, for all field types.

### JSON import round-trip (session 049 scope, folded in)

**Export button on Edit page**: Added a "Download JSON" header action to `EditForm.php` that mirrors the table action already present on the list view. Uses `fn () => $this->downloadJson()` to capture the page instance and stream the file.

**Import action on List page**: Added an "Import JSON" header action to `ListForms.php` with a modal containing a 20-row Textarea. Full strict validation before any record is created:

- Valid JSON parse
- Required top-level keys: `title` (non-empty string), `handle` (slug pattern), `fields` (array), `settings` (object)
- Per-field: `handle` (slug pattern), `type` (allowlisted), `label` (non-empty string), `validation` (allowlisted), `options` array structure if present
- ReDoS check on any `custom_regex` field (same backtrack-limit test as save-time check)
- `settings.form_type` allowlisted to `general` / `contact`
- All unknown keys stripped via `array_intersect_key` on both fields and settings
- Handle collision auto-suffixed (`-2`, `-3`, etc.)
- Imported forms saved as `is_active = false` — admin must review and activate
- On success: redirects to the new form's Edit page; notification states the final handle if it was changed

Validation errors are accumulated across all fields and shown in a single persistent danger notification as a bulleted list.
