# 146. Editor Architecture Review

Plan and document the migration of the page builder editor from Livewire to a Vue.js client-side architecture. This session produces the architectural plan, file inventory, and design decisions that sessions 147–151 will execute against.

---

## Design decisions (resolved before starting)

- **Framework:** Vue 3 (Composition API) with Pinia for state management.
- **Auth:** Session-based (Filament middleware). No token auth — the Vue app lives inside the authenticated admin page.
- **API style:** REST endpoints under Filament's auth middleware. These are the first internal consumers of the API surface planned for post-beta public access.
- **Preview rendering:** Server-rendered Blade templates, injected as HTML into the Vue canvas. Preview refresh is user-initiated — triggered on block selection change, mode switch, and explicit refresh action. Form field changes in the inspector do NOT trigger preview re-render. A dirty flag per widget signals unsaved preview state.
- **Widget templates:** Remain as Blade. Server renders them via the existing WidgetRenderer. The Vue editor manages the chrome around them, not the widget content itself.
- **Vue entry point:** Mounts inside `page-builder.blade.php` as a `<div id="page-builder-app">`. The blade file becomes a thin shell that passes bootstrap data (page ID, widget tree, widget type registry, collection/tag lookups) as a JSON blob. Filament chrome (sidebar, breadcrumbs, top bar) stays outside.
- **Livewire boundary:** Widget picker modal and save-as-template modal stay on Livewire/Filament. Vue dispatches browser events to open them; they dispatch events back with results.
- **Build pipeline:** Vue compiles via Vite alongside the existing Filament theme build. Public widget CSS/JS continues through the external build server — no change.

## Phase 2 — API endpoint design

Design the REST API endpoints the Vue editor will consume. Document in the migration plan:

### Widget CRUD
- `GET /admin/api/page-builder/{page}/widgets` — load full widget tree with nested children, config, style_config, query_config, and rendered preview HTML
- `POST /admin/api/page-builder/{page}/widgets` — create widget (accepts widget_type_id, parent_widget_id, column_index, sort_order)
- `PUT /admin/api/page-builder/widgets/{widget}` — update config, style_config, query_config, or label
- `DELETE /admin/api/page-builder/widgets/{widget}` — delete widget and children
- `POST /admin/api/page-builder/widgets/{widget}/copy` — duplicate widget and children
- `PUT /admin/api/page-builder/{page}/widgets/reorder` — batch reorder (accepts array of {id, parent_widget_id, column_index, sort_order})

### Preview rendering
- `GET /admin/api/page-builder/widgets/{widget}/preview` — render single widget preview HTML (called on selection change, mode switch, or explicit refresh)

### Lookup endpoints (read-only reference data for inspector dropdowns)
- `GET /admin/api/page-builder/collections` — list collections with types
- `GET /admin/api/page-builder/collections/{collection}/fields` — field definitions for data mapping
- `GET /admin/api/page-builder/tags` — tag list for query config
- `GET /admin/api/page-builder/pages` — page list for link pickers
- `GET /admin/api/page-builder/events` — event list for event widgets
- `GET /admin/api/page-builder/widget-types` — widget type registry with config schemas

Document request/response shapes for each endpoint. Base these on the data structures already used by the Livewire components — the API formalises what's already being passed around informally.

## Phase 3 — Vue application structure

Design the Vue component tree and Pinia store shape. Document in the migration plan:

### Pinia store (`useEditorStore`)
- `widgets` — flat map of widget objects keyed by ID
- `rootOrder` — ordered array of root-level widget IDs
- `selectedBlockId` — currently selected widget
- `editorMode` — 'edit' | 'handles'
- `dirtyWidgets` — set of widget IDs with unsaved preview state
- `widgetTypes` — registry of available widget types with config schemas
- `collections`, `tags`, `pages`, `events` — lookup data for inspector

### Vue components
- `PageBuilderApp.vue` — root, initialises store from bootstrap data
- `PreviewCanvas.vue` — left panel, renders widget preview regions with overlay system
- `PreviewRegion.vue` — single widget region: overlay div, selection handle, injected HTML
- `InspectorPanel.vue` — right panel, tabbed layout
- `InspectorTabs.vue` — dual stacked tab groups (Appearance/Layout + Content/Data Mapping)
- `InspectorField.vue` — dynamic field renderer (dispatches to specific field components)
- Field components: `TextField`, `SelectField`, `CheckboxField`, `ColorPickerField`, `RichTextField`, `ImageUploadField`, `SpacingControl`, `ButtonListField`
- `BlockList.vue` — widget tree list (for handle mode)
- `BlockCard.vue` — single block in the list

### Event bridge (Vue ↔ Livewire)
- Vue dispatches: `open-widget-picker`, `open-save-template-modal`
- Vue listens: `widget-type-selected`, `template-saved`
- Implementation: `window.dispatchEvent()` / `window.addEventListener()`
