## (A) Theme/Design export-import — design (canonical for this half)

Grounded to the actual code; mirrors the (B) draft's additive-envelope discipline.

- **Export.** Add `ContentExporter::exportDesign(): array` → `envelope(['design' => [...]])`, where `design` is the three design-group JSON `SiteSetting` rows captured **as stored**: `theme_colors`, `typography`, `button_styles`. Reuse the existing `envelope()` helper (`format_version` stays `1.1.0`, `exported_at`, `payload`). Additive only — `exportPages` / `exportTemplates` / `exportBundle` untouched. A combined "everything" export may carry `payload.design` alongside `payload.templates` / `payload.pages` / (Phase 2) `payload.media`.
- **Import.** Add a `ContentImporter` `payload.design` pass that runs **before** templates/pages (so a combined bundle establishes the theme, then templates/pages resolve against it). For each of the three rows: **deep-merge the imported value over the resolver defaults**, then persist — `theme_colors` over `ColorTokenResolver::defaults()`, `typography` over the `TypographyResolver` default shape, `button_styles` over its default shape. **Never sweep, replace wholesale, or zero unknown keys** — a key absent from the bundle keeps the target's current/default value (the session-295 em-rhythm-revert lesson; the concrete-values rule, `memory/feedback_concrete_values`). After the design pass, trigger the existing rebuild: `app(AssetBuildService::class)->build()` (the exact path `DesignSystemPage::saveColors()` / `::save()` already use; on failure, the same "rebuild failed — run `php artisan build:public`" persistent-notification fallback).
- **Surface.** An **"Export theme"** + **"Import theme"** action pair as header actions on `app/Filament/Pages/DesignSystemPage.php` (the Theme page), gated by that page's existing `canAccess` (`manage_cms_settings`). Export uses the same queued-job + stored-artifact + persistent-database-notification + gated-download pattern as (B) decision #5 (no `streamDownload`); a theme bundle is small but rides the identical mechanism for consistency. Import accepts the same zip-or-JSON input as `ImportBundleAction` and dispatches the same queued import job; the job's `ContentImporter::import()` runs the `payload.design` pass.
- **Reuse, do not fork.** No new exporter/importer class — `payload.design` is an additive pass exactly like the draft's `payload.media`. No new permission. No schema. No new page (action pair on the existing Theme page).
