# 154. Nav Widget & Footer

Split the existing `site_header` and `site_footer` widgets into smaller pieces ŌĆö a standalone `logo` widget and a standalone `nav` widget ŌĆö and re-seed the header/footer system pages to use the column layout system from session 153. This brings the page editor's column tooling to site chrome with no editor changes; header/footer are already system pages edited via the standard page builder, so the only work is on the widget side and the renderer side.

---

## Design decisions (resolved before starting)

- **Header/footer storage is unchanged.** They already live as system pages with slugs `_header` and `_footer`, edited via the standard `PageResource`/`EditPage` shell, referenced from `templates.header_page_id` / `templates.footer_page_id`, and rendered into `resources/views/layouts/public.blade.php` via `App\Services\Media\ChromeRenderer`. No new tables. No new editor. No new Filament pages.
- **The page builder already supports layouts on header/footer pages.** Session 153's column system applies to system pages with no changes needed in the editor ŌĆö it's the same tooling, the same store, the same components. The only thing missing is on the *render* side: `ChromeRenderer` was updated in session 153 to use `layout_id` but it does not yet load or render `PageLayout` records. Fixing that is part of this session.
- **Destructive migration.** This is a beta system with no production data. The migration deletes the `site_header` and `site_footer` widget types, wipes the widgets on `_header` and `_footer` pages, and re-seeds them with the new default content. No backward-compatibility shims, no data preservation.
- **The `nav` widget is a minimal extraction.** It is not redesigned in this session. It accepts a `nav_handle` config field, looks up the matching `NavigationMenu`, and outputs a lightly formatted block of links ŌĆö exactly what `site-header.blade.php` and `site-footer.blade.php` already do today. A full nav widget redesign (modes, dropdown, hamburger collapse, accordion, etc.) is its own future session and is **out of scope here**.
- **The `logo` widget is a single image + a text field, both wrapped in one anchor that links to `[URL]` (default `/`).** The image is left-aligned, the text sits to the right of it. When the widget sits at the root of a page or inside a 100%-width column, the contents anchor to the left and stay at their natural size (no stretching, no center-fill). This is plain CSS on the widget's outer wrapper, not a render-mode flag.
- **Footer copyright is a static value in a user-editable rich text field.** No `{year}` token replacement, no auto-current-year, no SiteSetting interpolation. The seeded default text is something like `<p>&copy; 2026 Your Organization. All rights reserved.</p>`; the user edits it from there.
- **Default header layout** (re-seeded by the migration): a single 2-column grid layout containing the logo widget in column 0 and the nav widget in column 1. Column widths default to something appropriate (e.g. `auto 1fr` or similar ŌĆö pick something sensible).
- **Default footer layout** (re-seeded by the migration): a single 2-column grid layout containing a `text_block` widget with the copyright HTML in column 0 and the nav widget in column 1.

## Phase 1 ŌĆö Fix `ChromeRenderer` to render layouts

**This is unfinished business from session 153.** The renderer currently loads only root widgets (`whereNull('layout_id')`) and ignores `PageLayout` records entirely. With the new default header containing a column layout, the chrome would render empty without this fix.

Update `App\Services\Media\ChromeRenderer::renderPage()` to:

1. Load root widgets (as today) AND load layouts on the same system page, ordered by `sort_order` and merged into a single page-flow list. Use the same merge pattern that `PageController::renderPage()` uses.
2. For each item in the merged flow, render either a widget block (existing path) or a layout block (new path that mirrors `PageController::renderLayoutBlock()` ŌĆö build the container CSS from `display`, `columns`, and `layout_config`; iterate the layout's child widgets grouped by `column_index`; emit a `<div class="page-layout" style="ŌĆ”">` wrapper with one `<div class="layout-column">` per slot).
3. Collect inline styles, scripts, and assets from layout child widgets the same way root widgets do.

There is no separate "render mode" for chrome layouts vs. page layouts ŌĆö they emit the same markup. Whatever wrapper class names `PageController` uses, `ChromeRenderer` should use too.

## Phase 2 ŌĆö Logo widget

### 2a. Blade template

Create `resources/views/widgets/logo.blade.php`:

- Reads `config['link_url']` (default `/`), `config['text']` (string, default empty), and `configMedia['logo']` (image media object)
- Outputs a single `<a>` element wrapping an `<img>` (when present) and a `<span>` (when text is present)
- Layout: image left, text right of image, both inline-aligned. When the widget sits at root or in a wider container, the anchor stays at its natural width and anchors to the left of its container ŌĆö `display: inline-flex` on the anchor and a `width: fit-content` (or equivalent) on the outer wrapper. No image stretching.
- Use modest, semantic class names (e.g. `widget-logo`, `widget-logo__img`, `widget-logo__text`).

### 2b. Widget type seed

Add a `logo` entry to `database/seeders/WidgetTypeSeeder.php`:

- `handle: 'logo'`
- `label: 'Logo'`
- `description: 'Site logo image with optional text and link target.'`
- `category: ['layout']`
- `render_mode: 'server'`
- `full_width: false`
- `config_schema:`
  - `logo` ŌĆö image, group `content`, label "Logo image"
  - `text` ŌĆö text, group `content`, label "Text beside logo"
  - `link_url` ŌĆö text (subtype `url`), group `content`, label "Link URL", default `/`
- `template: "@include('widgets.logo')"`

### 2c. CSS

Add the logo widget styles to the public widget SCSS pipeline (wherever new widget styles go ŌĆö find an existing widget's SCSS file as a reference).

## Phase 3 ŌĆö Nav widget

### 3a. Blade template

Create `resources/views/widgets/nav.blade.php` by extracting the nav-rendering portion of `site-header.blade.php`:

- Reads `config['nav_handle']` (default `primary`)
- Looks up `NavigationMenu::where('handle', $navHandle)->first()`
- Loads visible top-level items, ordered by `sort_order`, with their visible children
- Outputs a `<nav>` element containing anchors for each item, with the same dropdown handling and `aria-current` logic the existing `site-header.blade.php` has
- Keep the Alpine `x-data="{ open: false }"` and the hamburger toggle button. The nav widget is functionally identical to the right half of the current `site_header` ŌĆö same markup, same Alpine behavior, same class names where they apply.
- Do **not** redesign anything. Do not introduce new modes, configurable layouts, or behavior switches. The goal is "pull the nav out of `site_header` into its own widget so a future session can rebuild it."

## Phase 4 ŌĆö Migration: rebuild `_header` and `_footer`

Write a single migration that does the following, in order:

1. Find the `_header` and `_footer` system pages by slug.
2. Delete every `PageLayout` belonging to those pages (which cascades to delete all child widgets via the FK from session 153).
3. Delete every root `PageWidget` belonging to those pages.
4. Delete the `site_header` and `site_footer` widget type rows from `widget_types`.
5. **Re-seed `_header`:**
   - Create one `PageLayout` with `display: 'grid'`, `columns: 2`, `layout_config: { grid_template_columns: 'auto 1fr', gap: '1rem', align_items: 'center' }`, `sort_order: 0`.
   - Create one `PageWidget` of type `logo` in column 0, with default config (`link_url: '/'`, empty text, no logo image set ŌĆö the user uploads one in the editor).
   - Create one `PageWidget` of type `nav` in column 1, with `config: { nav_handle: 'primary' }`.
6. **Re-seed `_footer`:**
   - Create one `PageLayout` with `display: 'grid'`, `columns: 2`, `layout_config: { grid_template_columns: '1fr auto', gap: '1rem', align_items: 'center' }`, `sort_order: 0`.
   - Create one `PageWidget` of type `text_block` in column 0, with the seeded copyright HTML in whichever config field `text_block` uses (look at the existing seed to find the right key).
   - Create one `PageWidget` of type `nav` in column 1, with `config: { nav_handle: 'footer' }`.

The migration is one-way (no `down()` rollback beyond best-effort) ŌĆö this is a beta system with destructive intent.

After writing the migration, update `database/seeders/WidgetTypeSeeder.php` and any other seeder that references `site_header` / `site_footer` so a fresh seed produces the same end state. **Do not leave the `site_header` / `site_footer` entries in the seeder** ŌĆö they would be re-created by `migrate:fresh --seed`.

Run the migration via Docker after writing it.

## Out of scope

- **Nav widget redesign.** Modes, dropdown, hamburger collapse, stack collapse, accordion ŌĆö all deferred to a future dedicated nav session. This session only *extracts* the existing nav, it does not redesign it.
- **Header/footer-specific editor.** The standard page builder already edits these system pages.
- **Theme toggle.** The current `site_footer` includes an Alpine theme toggle. Drop it for now ŌĆö the user can re-add it as a future widget if needed. Do not block this session on it.
- **Sticky header behavior, mega menus, multi-level submenus beyond the existing one level, animations.**
- **Auto current-year copyright.** Static rich text only.
- **A "company name" widget separate from the logo.** The logo widget's `text` field handles that case.
