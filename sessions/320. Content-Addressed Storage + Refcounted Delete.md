# 320. Content-Addressed Storage + Refcounted Delete

Reclaim the disk wasted by duplicate media: store identical bytes once (content-addressed storage keyed on the `content_hash` column from 319), and refcount deletion so removing one owner's `media` row never deletes bytes another row still uses. This is the reclamation leg of the media trilogy (318 visibility → 319 prevention → **320 reclamation**).

## The problem this solves

A typical install runs ~3× its necessary size because the same logos and stock images are stored over and over, each multiplied again by responsive conversions. Session 319 stopped *new* duplicates from being created (upload-time dedup) but explicitly did **not** reclaim the disk already wasted, and even its "reuse existing" path still writes a fresh physical copy under the current `DefaultPathGenerator` (`{media.id}/{file_name}`). This session is the piece that makes identical bytes share one physical file and reclaims the existing waste.

The local dev library is currently 65 referenced `media` rows over **45 distinct files** — the gap (20 rows that are byte-duplicates of an existing file) is exactly what this session collapses on disk, with no visible change to the site.

## Phases

Sequence the work this way — land the storage model and prove references survive before touching deletion — because it's the order that's easiest to verify, not because a misstep is costly (the dev DB reseeds freely).

### Phase 1 — Content-addressed path generator

- Implement a custom `PathGenerator` (registered in `config/media-library.php`) that derives a media file's directory from its `content_hash` rather than its `media.id`, so two `media` rows with the same hash resolve to the same physical path. Resolve the **URL-stability** decision first (see Open questions) — the public URL convention (`/storage/{id}/…`) is depended on by `MediaReferenceInventory` Rule B and existing rich-text embeds.
- Set `moves_media_on_update` appropriately for the chosen generator.
- Confirm new uploads land at the content-addressed path and that a second upload of identical bytes resolves to the **same** file (no second physical write).

### Phase 2 — Relocation migration (collapse existing duplicates)

- Move every existing media file from its `{id}/` path to its content-addressed path, collapsing byte-duplicates onto one physical file. Idempotent, batched (chunk by id), resumable; a no-op for already-relocated files. Decide migration-time vs. a one-shot `media:relocate-cas` command against the library size and document it.
- If the URL-stability decision is "rewrite embedded URLs," do that rewrite here over the same rich-text surfaces `MediaReferenceInventory` scans; if it is "id-stable URLs," prove the decoupled `UrlGenerator` keeps every existing embed resolving.
- After running: re-run the full media classification (live / dead / orphan) and a duplicate scan to prove the relocation collapsed duplicates **without** orphaning or breaking any referenced file. Capture before/after disk + row/file counts for the log.

### Phase 3 — Refcounted deletion

- Implement a custom `file_remover_class` so deleting a `media` row removes the physical file **only when no other row references the same content-addressed path**; otherwise the row is deleted and the shared bytes stay. Decide computed-vs-stored refcount at the open gate.
- Verify the path the 318 Media Finder uses (per-row `delete()`) is now refcount-safe: deleting one duplicate member must not break the others.
- Update (or consciously keep) the 319 page-builder `use-existing-*` endpoints so reuse resolves to the shared file rather than writing a copy — row-per-owner is fine, byte-sharing is the requirement.

## Open questions to resolve at session start

- **URL stability (the crux).** Keep public URLs id-stable via a custom `UrlGenerator` decoupled from the content-addressed `PathGenerator`, or rewrite embedded `/storage/{id}/` URLs in the relocation migration? The first preserves every existing embed with no content rewrite but adds a URL-layer indirection; the second is a one-time content migration but must cover every rich-text surface `MediaReferenceInventory` knows about. Lean id-stable unless there's a reason not to; confirm.
- **Refcount mechanism.** Computed at delete time (count rows sharing the hash/path — no schema change) vs. a stored counter column (faster delete, must stay consistent on add/delete). Lean computed for v1 given the library sizes in play; confirm.
- **Conversions scope.** Dedupe originals only (simpler, still the bulk of the row-level win) or content-address the responsive/webp conversions too (the larger multiplier)? Decide and be explicit in the log.
- **Relocation shape.** Migration-time vs. one-shot command. `down()` realistically can't un-move files — that's fine, just document it; the project's reset is a reseed, not a migration rollback, and there's no production data to preserve.
- **Portability + backup interaction.** The 303 id-preserving media seed and the spatie backup/restore both touch the media tree by path, so check they still behave under the new layout and update the restore runbook if the on-disk layout changes. A docs/consistency check, not a gate.
