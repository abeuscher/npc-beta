## Finding register

Work through each finding below. For each one: confirm or refute the vulnerability, implement the fix if rated High, and write the disclosure entry.

### Finding 1 — Regex ReDoS (High)

**Description:** The custom regex validation field allows an admin to enter an arbitrary regex pattern. PHP's `preg_match` uses a backtracking engine. Certain patterns (e.g. `(a+)+$`, `(\w+\s?)*`) cause exponential backtracking on crafted input, hanging a PHP worker for seconds or longer. An attacker who can control submitted values against a form with a ReDoS-prone pattern can exhaust the server's worker pool.

**Fix:** Add save-time validation to `FormResource`. When a field's `validation` is `custom_regex` and `validation_regex` is non-empty, test the pattern before saving:

1. Check the pattern compiles without error using `preg_match` in a try/catch.
2. Test it against a known backtrack-triggering string (e.g. `str_repeat('a', 50) . '!'`) with PHP's `pcre.backtrack_limit` temporarily reduced (or simply run `preg_match` and check `preg_last_error() === PREG_BACKTRACK_LIMIT_ERROR` after). If it triggers the backtrack limit, reject the save with a validation error: "This regex pattern is too complex and may cause performance issues. Simplify the pattern."
3. Apply this check as a custom `->rules([])` closure on the `validation_regex` field in the Repeater, or as a `->afterStateUpdated()` validator, or in a `mutateFormDataBeforeSave()` override on `EditForm` and `CreateForm`.

The existing `try/catch` in `fieldValidationRules()` (submission time) is kept as a secondary safety net but is not sufficient on its own.

### Finding 2 — Hidden field value tampering (Medium)

**Description:** HTML hidden fields rendered by `public-form.blade.php` display the form's `default_value`. A user can alter the POST body (via browser devtools or curl) and submit any value for a hidden field handle — potentially bypassing intended defaults.

**Confirm or refute:** Review `FormSubmissionController@store`. The controller currently writes hidden field `default_value` from the form definition *after* building `$data` from submitted values. Determine whether the submitted value for a hidden field handle could overwrite the form definition's value in `$data`.

**Fix if confirmed:** Ensure hidden fields are excluded from the initial `$data` array, then written from the form definition only. Add a feature test that submits a tampered value for a hidden field and asserts the stored submission contains the form definition's `default_value`, not the submitted value.

---

### Finding 3 — Submission flow test coverage (gap, not a vulnerability)

Write a `FormSubmissionTest` feature test class covering:

- Valid submission is stored with correct `form_id`, `data`, and `ip_address`
- Honeypot field filled → silent success, no submission stored
- PII value in a field (`4111111111111111` — Visa test PAN) → rejected, no submission stored
- Inactive form → 404
- Non-existent handle → 404
- Required field missing → validation error returned
- Hidden field `default_value` used even when attacker submits a different value (see Finding 2)
- `form_type = contact` with email mapped → contact created with correct field values and a note
- Email collision → existing contact updated with non-empty submitted values, note added, `contact_id` set on submission
- Email collision → mailing_list_opt_in already true, form submitted without checking the box → opt-in status unchanged

## Part 2 — Security disclosure document

Create `docs/security-forms.md`. This is an internal developer reference, not a public-facing document. Write it in plain language with the following sections:

### What is protected

- CSRF — all form submissions require a valid Laravel CSRF token
- PII screening — submitted values are scanned for credit card PANs, SSNs, and ABA routing numbers before storage; rejections return a generic error
- Honeypot — a hidden field silently discards basic bot submissions
- Rate limiting — `throttle:10,1` (10 submissions per minute per IP) applied at the route level
- Input validation — field-level validation rules from the form definition are enforced server-side; client-side `required` and `pattern` attributes are a UX convenience only
- Regex safety — custom regex patterns are validated at save time; patterns that trigger backtrack limits are rejected before they can reach the submission handler
- XSS — all submitted values are stored as plain text and rendered through Blade's `{{ }}` escaping; no submitted value is rendered with `{!! !!}`
- Contact field writes are restricted to a defined allowlist; arbitrary column names from form definitions cannot reach the database

### Known limitations and accepted risks

**Honeypot predictability (Low).** The honeypot field is named `_hp` — a fixed, predictable name. Sophisticated bots that are explicitly programmed to avoid known honeypot patterns could bypass it. Mitigation: Cloudflare Turnstile (deferred). Accepted for now.

**Email collision identity abuse (Low).** A public form submission that matches an existing contact's email address updates that contact's record with submitted field values. A bad actor who knows another person's email could submit false address or phone data. This is a known limitation of any unauthenticated public form that writes to a contact record. Full mitigation requires authenticated member sessions (deferred to Member Portal). Accepted for now; the note written to the contact record on each submission provides an audit trail.

**Rate limiting behind a reverse proxy (Low).** The `throttle:10,1` middleware keys on `$request->ip()`. If the application is placed behind a reverse proxy (e.g. Cloudflare, nginx upstream), all requests will appear to come from the proxy's IP and the rate limit will be shared across all visitors. Fix: add Cloudflare's or the proxy's IP ranges to Laravel's `TrustProxies` middleware before placing the app behind a proxy. Not applicable to the current direct-served deployment.

**No CAPTCHA (Low).** Honeypot and rate limiting provide basic bot deterrence. Cloudflare Turnstile (challenge-free CAPTCHA) is deferred. Accepted for now.
