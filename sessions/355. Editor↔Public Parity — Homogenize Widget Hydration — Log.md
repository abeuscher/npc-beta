## What was built

The editor↔public parity stub's **Track 1**: the widget-owned JS hydration, previously two divergent copies that drifted, is now one shared routine both surfaces call. (Track 2 — clean Tailwind-v4 preflight layering — stays open, externally gated on Filament v4.)

### Phase 1 — The `hydrate(rootEl)` contract

- **Scoped cause confirmed against code.** Every interactive widget builds its own Swiper/Chart inside its Alpine `x-data` `init()` (e.g. `app/Widgets/BlogListing/script.js`, `BarChart/script.js`, `ProductCarousel/template.blade.php`), reading the `window.*` globals. So the genuinely shared, drift-prone primitive is the **Alpine init lifecycle**, not the libs themselves. The public path's "copy" is `Alpine.start()` (implicit, whole-document, one-shot); the editor's copy was `useLibraryLoader.reinitAlpine()` (explicit destroy + `initTree` per `v-html`-injected subtree) — the only literal second copy.
- **Boundary fixed precisely.** Widget-owned (libs present + correctly-layered CSS + Alpine/Swiper/Chart init) → single-sourced. Surface-owned (the editor's `requestAnimationFrame`/`nextTick` scheduling, selection/drag/inline-edit, `AdminPanelProvider`'s server-side lib emission, `public.js`'s lib bundling + component registration) → left per-surface, untouched.
- **Three open questions resolved (brought to the open gate, owner-approved):**
  - **Q1 (module home + import seam):** a lib-agnostic shared module operating over `window.Alpine/Swiper/Chart` — imports none of the libs, so neither build graph duplicates them. Confirmed by the build: it emitted a standalone `hydrate-*.js` chunk (1.8 kB) shared by both entries, with Swiper/Chart still bundled only into `public`.
  - **Q2 (bundled vs lazy libs):** kept **both** acquisition strategies, unified only the precondition. Public bundles + assigns the libs (no `libs` arg); the editor lazy-loads via `loadLibs(requiredLibs)`. `AdminPanelProvider`'s head emission stays the editor's surface-owned lib delivery — untouched.
  - **Q3 (first-init vs re-init):** one routine, internal fork on a `firstInit` flag. Public calls `hydrate(document, { firstInit: true })` (→ `Alpine.start()`); the editor calls `hydrate(scopeEl, { libs })` (→ the destroy/`initTree` dance). Component registration (`customSelect`, theme store) stays in `public.js`.

### Phase 2 — Shared routine + both-surface wiring

- **New `resources/js/shared/hydrate.ts`** — `loadLibs()` and the re-init dance (`reinitSubtree`) moved in verbatim, composed under `hydrate(rootEl, { firstInit?, libs? })`.
- **`resources/js/public.js`** — final `Alpine.start()` replaced with `hydrate(document, { firstInit: true })`; imports/`window.*` assignment/`alpine:init` registrations unchanged.
- **`resources/js/page-builder-vue/components/PreviewCanvas.vue`** — the three `loadLibs(...) + reinitAlpine(scopeEl)` call sites replaced with `hydrate(scopeEl, { libs })` (or `hydrate(scopeEl)` for the libs-already-loaded preset-switch path); the editor's rAF/`nextTick` scheduling kept (surface-owned).
- **Retired** `resources/js/page-builder-vue/composables/useLibraryLoader.ts` (the bespoke copy) — `git rm`'d; the two widget `script.js` comments that named `reinitAlpine` updated to `hydrate()`.
- **Build:** `npm run build` clean; `hydrate-*.js` shared chunk emitted, no lib duplication into the page-builder bundle.

### Mid-session owner-requested side-fix — nav mobile hamburger in the editor

- **Reported:** switching the page editor to the Mobile preset didn't show the hamburger. **Cause:** the nav widget hard-codes its mobile breakpoint as an inline render-time `<style> @media (max-width:768px)` in `app/Widgets/Nav/template.blade.php`; the editor's `@media`→`@container np-viewport` rewrite (`EditorCssTransformer`) runs only on the build-server bundle, never on inline template `<style>`, so the breakpoint evaluated against the (desktop-wide) browser window, not the simulated preset width. Not a hydration bug (the hamburger is CSS-only, checkbox+label) — unrelated to the Track-1 work.
- **Fix:** added a parallel `@container np-viewport (max-width: 768px)` block after the existing `@media` block — matches in the editor (the preview scope is the named container), inert on public (no `np-viewport` container there), with a `:checked` override to re-enable the mobile menu the desktop `@media` rule force-hides with `!important`. Owner verified visually; nav is the only widget template emitting inline `@media`.
