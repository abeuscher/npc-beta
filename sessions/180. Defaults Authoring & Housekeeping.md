# 180. Defaults Authoring & Housekeeping

Add a "defaults" authoring mechanism in the inspector (parallel to the existing draft-preset feature) so the designer can export the current widget instance state as a paste-ready `defaults(): array` method for a widget Definition file. Tighten the widget-file lint so the designer can paste-edit Definition files and catch mistakes fast. Plus two small bug fixes.

---

## Design decisions (resolved before starting)

- **The button lives next to the draft-preset button in the inspector**, not in a separate tab. Same inspector, adjacent affordance.
- **Output is a complete method body**, `public function defaults(): array { return [...]; }`, not just the array literal. Paste-ready.
- **Keys are emitted in schema declaration order**, so diffs against the existing `defaults()` in the widget file stay readable.
- **Values emitted are the resolved config** (the widget's declared defaults with the instance's overrides merged on top). That way the exported `defaults()` captures what the designer currently sees, not a sparse diff.
- **No database writes.** Unlike draft presets, defaults are not stored server-side — the endpoint is pure: widget instance in, PHP string out. The designer copies the string and pastes it into the Definition file by hand.
- **The lint command runs Pest test groups, not a separate framework.** Existing `WidgetDefinition::validate()`, `WidgetManifestTest`, and `WidgetRegistryTest` stay. The new lint command is a thin artisan wrapper that runs the relevant Pest groups and returns a clean pass/fail, so the designer can run `docker compose exec app php artisan widgets:lint` after every paste without remembering test flags.
- **Housekeeping items are genuinely small.** If either turns out to be non-trivial mid-session, pause and ask the user before expanding scope.

## Phase 1 — Defaults export endpoint + UI button

### Server side

- New controller action on the existing `PresetController` (or new `WidgetDefaultsController` — designer's call during the session): `POST /admin/api/widget-defaults/export` accepting `{ widget_id: uuid }`, returning `{ php: string }`.
- Permission gate: `update_page`, same as the draft-preset create action.
- Body:
  1. Load the `PageWidget` with its `widgetType`.
  2. Resolve its full config via `WidgetConfigResolver::resolvedDefaults($pw)` merged with `$pw->config ?? []` (instance overrides win — same layering as `WidgetConfigResolver::resolve()`).
  3. Walk the widget type's `config_schema` in declared order. For each key, pull the resolved value. Skip keys not in the resolved map (shouldn't happen if validate() passes, but be defensive).
  4. Emit PHP source using the same `buildPhpLiteral`-style escaping that the preset export already uses — reuse or lift the helper so the two exports share one formatter.
  5. Wrap in the method boilerplate:
     ```php
     public function defaults(): array
     {
         return [
             'key' => value,
             …
         ];
     }
     ```

### Vue / inspector side

- Add a button to the inspector's preset/export area. Label ideas: "Export defaults", "Copy as `defaults()`", "Save as defaults". Match the visual weight of the existing draft-preset button.
- Click handler calls a new `api.exportDefaults(widgetId)` → POSTs to the new endpoint → copies the returned `php` string to the clipboard → shows a transient "copied" affordance (same pattern as the existing preset clipboard export in `InspectorPresetGallery.vue`).
- If the existing `buildPhpLiteral` helper in the Vue store/component can be reused for display, do that. Otherwise the server's PHP string is authoritative.

### Tests

- Feature test: `POST /admin/api/widget-defaults/export` with a seeded widget returns a string containing `public function defaults(): array` and every schema key.
- Feature test: the emitted string round-trips — spawn a PHP `eval()` in the test (safe, trusted input) or regex-assert the shape — verify it's syntactically valid and the array contains the expected keys with the expected resolved values.
- Permission test: unauthenticated / missing permission returns 403.

## Phase 2 — Widget lint command

### What exists

- `WidgetDefinition::validate()` checks that every schema key has a matching entry in `defaults()`.
- `WidgetManifestTest` enforces preset shape (slug handle, non-empty label, config keys in schema and appearance-only, appearance_config is array).
- `WidgetRegistryTest` exercises `validate()`.
- No single command runs these.

### What to add

- New artisan command `widgets:lint` (file: `app/Console/Commands/WidgetsLintCommand.php`).
- Runs Pest groups `widget-lint` and prints a single-line pass/fail summary. On failure, prints the underlying Pest output.
- Tag existing relevant tests with `->group('widget-lint')` (in `WidgetManifestTest`, `WidgetRegistryTest`, and any others the designer wants included).
- Add new assertions to `WidgetManifestTest` (or a new `WidgetDefaultsLintTest`) that extend the existing coverage. Proposed checks — pick the subset that has real payoff, the rest are candidates for a follow-up:
  1. **Type alignment.** For each schema field, the corresponding `defaults()` value's PHP type matches the declared field type:
     - `toggle` → bool
     - `number` → int or float (or null for optional numbers)
     - `image` / `video` → null or int
     - `checkboxes` → array
     - everything else → string
  2. **Select options honoured.** If a field has `options: [...]` and a `default`, the default must be one of the option keys.
  3. **No orphan defaults.** Every key in `defaults()` must appear in `schema()`. (The existing validate() only checks the reverse direction.)
  4. **Preset config keys are appearance-only.** Already covered by `WidgetManifestTest` — confirm and leave alone.
- Document the command and its checks in `docs/widget-system.md` under the existing "CI validation" heading.

### Tests

- Deliberately break one widget in a temp test (add a toggle field with a string default) and assert the lint fails with a readable message.
- Run the full lint on the current widget set — everything should pass. If it doesn't, the offenders are pre-existing drift; fix them in-session or flag to the user as out-of-scope.

## Phase 3 — Housekeeping item: hero widget link override

**Symptom.** The hero widget's link colour override (set via the widget's `nav_link_color` / `nav_hover_color` config, or via appearance-config link styling — clarify at session start) no longer takes effect on the public site. Links inside the hero render with the default link colour instead.

**Likely cause.** CSS cascade order — a more specific rule from the public stylesheet (or a Tailwind-preflight-style reset) is winning over the inline style or widget-specific rule.

**Approach.**
1. Reproduce on the deployed preview or local dev. Inspect element to identify which rule is winning.
2. Raise specificity on the hero link rule OR move the inline style to use CSS custom properties set on the widget wrapper (`--link-color`) so the widget-level rule reads `color: var(--link-color, inherit)`.
3. Do not introduce `!important`. If the only path forward is `!important`, stop and ask.
4. Manual-test in the builder preview and on the public page.

**Tests.** Pure CSS change — no unit test expected unless the fix touches PHP.

## Phase 4 — Housekeeping item: phpunit.xml DB override is a no-op

**Symptom.** `artisan test` runs `RefreshDatabase` against the dev database (`nonprofitcrm`) instead of the test database (`nonprofitcrm_test`), wiping dev data on every test run. Discovered during the session 179 close — every `artisan test` run that session silently dropped and re-migrated the dev DB.

**Root cause.** `phpunit.xml` lines 26–31 set `DB_*` env values without `force="true"`. Without that attribute, phpunit's `<env>` tags are only applied when the variable isn't already set in the process environment. Since Laravel loads `.env` first (which has `DB_DATABASE=nonprofitcrm`), the phpunit override never takes effect.

**Fix.** Add `force="true"` to every `DB_*` line in `phpunit.xml`:

```xml
<env name="DB_CONNECTION" value="pgsql" force="true"/>
<env name="DB_HOST" value="postgres" force="true"/>
<env name="DB_PORT" value="5432" force="true"/>
<env name="DB_DATABASE" value="nonprofitcrm_test" force="true"/>
<env name="DB_USERNAME" value="postgres" force="true"/>
<env name="DB_PASSWORD" value="32feet" force="true"/>
```

**Verification.** After the fix:
1. Confirm dev DB has seeded data: `docker compose exec postgres psql -U postgres -d nonprofitcrm -c "SELECT count(*) FROM users;"` (should be > 0).
2. Run the fast test suite: `docker compose exec app php artisan test --exclude-group=slow`.
3. Re-check the dev DB user count — it should be unchanged.

**Consider also** adding `force="true"` to the other `<env>` tags in phpunit.xml (`MAIL_MAILER`, `QUEUE_CONNECTION`, `SESSION_DRIVER`, `CACHE_STORE`) for the same reason — if any are defined in `.env`, they'll leak into the test environment. The DB fix is critical; the rest are defense-in-depth and can be done in the same commit or deferred.

**Tests.** No new tests — the fix is a config change. The verification step IS the test.

## Phase 5 — Housekeeping item: color picker dropdown doesn't close

**Symptom.** In the widget inspector, the color picker's swatch-and-picker popover stays open after the user selects a color. It should close on selection, matching the pattern of other popovers in the inspector.

**Location.** The primitive lives under `resources/js/page-builder-vue/components/` — find the color-picker component (likely `ColorPicker.vue` or inside `appearance/`) and trace its open/close state.

**Approach.**
1. Identify where `open` / `isOpen` state is managed.
2. In the swatch-click / color-pick event handler, emit the selected value as it currently does AND set `open = false`.
3. Manual verify: swatch click closes, picker-panel click closes, clicking outside also closes (already works, don't break it).

**Tests.** Frontend-only — no Pest test. Verify by manual testing.
