## What was built

### Phase 1 — Shared-trait testids

- Added `data-testid="map-column-{n}"` to `Select::make($key)` inside `InteractsWithImportWizard::buildNamespacedMappingRow()` — inherited by events, donations, memberships, invoice-details wizards in one edit. Contacts already carried its own equivalent testid (session 195) and was untouched.
- Added `data-testid="import-duplicate-strategy"` to the Radio returned by `InteractsWithImportWizard::duplicateStrategyRadio()` — also inherited by all four non-contact wizards.

### Phase 2 — Per-wizard testids (PHP + blade)

PHP edits on `ImportEventsPage`, `ImportDonationsPage`, `ImportMembershipsPage`, `ImportInvoiceDetailsPage`:
- `data-testid="import-match-key"` on events' `event_match_key` Select (events is the only non-contact wizard with a second match key).
- `data-testid="import-contact-match-key"` on each wizard's `contact_match_key` Select.
- `data-testid="import-commit-button"` on each wizard's `submitAction(...)` Stage Import action.

Blade edits — four wizard views (`import-events.blade.php` etc.): added `data-testid="import-{events|donations|memberships|invoice-details}-wizard"` to each outer wrapper div.

Blade edits — four progress views (`import-events-progress.blade.php` etc.): added testids matching the session-195 contacts-progress pattern — `import-progress-phase-{rejected|awaiting|committing|done}`, `import-stat-{imported|updated|skipped|errors}`, `import-progress-commit-button`, `import-progress-cancel-button`, `import-progress-bar`, `import-save-mapping`, `import-go-to-review`. The `done` phase in each was wrapped in a single `data-testid="import-progress-phase-done"` div so the wizard helper can wait on it directly.

### Phase 3 — Fixture files

Created four new fixture folders under `tests/e2e/fixtures/`:
- `events/` — 3 events × 1 registration × 1 transaction; 9-column CSV; handcrafted.
- `donations/` — 5 donations; 6-column CSV (Email, Amount, Donation Date, External ID, Number, Payment State). Discovery during testing: the donations importer only creates a `Transaction` when either `donation:invoice_number` (the "Number" column) or `transaction:external_id` is mapped — added the "Number" column so transactions are created in the happy path.
- `memberships/` — 5 memberships tied to the seeded "Standard" tier; 6-column CSV.
- `invoice-details/` — 3 invoices × 2 line items (6 rows total); 9-column CSV with grouped `Invoice #` keys.

Each folder carries `happy-path.csv`, `update-second-pass.csv`, and `happy-path.expected.json`. All fixtures are PII-free (`example.com` emails, `555-` phone prefix, no SSN/CC shapes).

### Phase 4 — Wizard helper extensions

Extended `tests/e2e/helpers/wizard.ts`:
- `NonContactWizardOptions` type — extends `WizardOptions` with `contactMissingStrategy` and `extraColumnMappings`.
- One parametric private helper `driveNonContactHappyPath(page, key, opts)` driving any of the four non-contact wizards by key.
- Four thin public wrappers: `driveEventsHappyPath`, `driveDonationsHappyPath`, `driveMembershipsHappyPath`, `driveInvoiceDetailsHappyPath`.
- Internal helpers `openNonContactWizard`, `applyContactMissingStrategy` (sets `data[contact_missing_strategy]` radio), `applyExtraColumnMappings` (uses the shared `map-column-{n}` testid to manually map non-auto-guessed columns like `donation:external_id`), `advanceNonContactMapping` (four-step wizard advance with explicit duplicate-strategy radio check — same reliability pattern session 195 established for contacts).

### Phase 5 — DB helper extensions

Extended `tests/e2e/helpers/db.ts`:
- `findLatestImportSessionId(modelType: string = 'contact')` — added `modelType` argument with a backwards-compatible default so all prior call sites still work.
- Five import-session-scoped counters: `countEventsInSession`, `countEventRegistrationsInSession`, `countDonationsInSession`, `countMembershipsInSession`, `countTransactionsInSession` — all wrap a shared `countInSession(table, sessionId)` helper filtered on `import_session_id`. Isolation is essential because `TransactionSeeder` seeds 10 contacts/donations/memberships/transactions during every `migrate:fresh --seed`.
- Three external-id finders: `findEventByExternalId` (via `import_id_maps` join), `findDonationByExternalId` (direct `(import_source_id, external_id)`), `findMembershipByExternalId` (same pattern).
- `findTransactionByInvoiceNumber` — used by the invoice-details specs to assert line-item grouping.
- `findImportSourceIdByName` — used by update-strategy specs to look up the reused source between passes.
- `insertContactsForImport(emails)` — direct SQL insert of minimal contact rows. Events imports don't auto-create contacts (unlike donations/memberships/invoice-details), so the events specs seed their three contacts via this helper before driving the wizard.

### Phase 6 — Eight new Playwright specs

Under `tests/e2e/importer/`, matching the shape of the two session-195 contacts specs (`test.describe.configure({ mode: 'serial' })`, `beforeAll(async ({ browser }) => { await resetAndLogin(browser); })`, per-test `test.setTimeout(180_000)` or `300_000`):

- `events-happy-path.spec.ts` — pre-seeds 3 contacts, drives wizard, asserts 3 events + 3 registrations + 3 transactions scoped to the session, 0 staged updates, 0 errors, event titles visible on the events list page.
- `events-update-strategy.spec.ts` — seed first pass, approve, re-import with `update` strategy, asserts `updated` stat = 3, staged count = 3 (all match, all have title diffs), approves, asserts updated titles in the events list.
- `donations-happy-path.spec.ts` — drives wizard with `contactMissingStrategy=auto_create` and manual `External ID → donation:external_id` mapping, asserts 5 donations + 5 transactions in session.
- `donations-update-strategy.spec.ts` — seed + approve + update pass, asserts `updated` stat = 5 (all match), staged count = 5 (staged regardless of diff), amount before approval still old, amounts after approval match CSV.
- `memberships-happy-path.spec.ts` — 5 memberships + 5 auto-created contacts.
- `memberships-update-strategy.spec.ts` — 5 matched / 5 staged; notes-before-approval preserved; notes-after-approval match CSV.
- `invoice-details-happy-path.spec.ts` — 6 CSV rows collapse to 3 transactions grouped by invoice number, 3 contacts auto-created.
- `invoice-details-update-strategy.spec.ts` — updated line items on one invoice stage then apply; `line_items` JSONB amounts compared numerically (DB stores unformatted numeric strings).

## Deferred decisions

- **PII rejection spec** — upload CSV with detectable PII, assert rejection UI and downloadable report. Deferred from this session's scope; candidate for a dedicated follow-up session.
- **Duplicate-header review spec** — upload CSV with colliding headers, assert review step surfaces findings and ignore choices propagate through mapping.
- **Error-report spec** — upload CSV with malformed rows, assert error table and downloadable errored-rows CSV.
- **Invoice-details multi-row grouping edge cases** beyond basic fill-blanks-under-skip — deferred.
- **Random Data Generator integration** for fixtures — tracked under its own outline stub ("Random Data Generator — CSV Export for Import Testing"); handcrafted fixtures remain in place until that session lands and replaces them.
- **Staged-update diff filtering (product question, not test deferral):** the progress-page `build{Donation|Membership|…}StageAttrs` helpers stage all non-null incoming field values regardless of whether they differ from the existing record. As a result, `import-stat-updated` and `import_staged_updates.count` both report matches rather than diffs. Tests were adjusted to assert the actual behaviour (5 staged on a 5-row update pass, not 3). Whether this is desirable behaviour or should filter to diffs only is a product decision — not in scope for this session; flagging for a future importer-polish pass.
