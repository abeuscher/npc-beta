# 251. Fleet Manager Contract v2.1.0 — Log-Fetch Endpoint

Add a `/api/logs` endpoint to the Fleet Manager agent contract — additive within the v2 major (`v2.0.0` → `v2.1.0`) — so FM 013+ can fetch application logs from a CRM install without operator SSH. CRM-side companion to A2 (Fleet Manager — node operations parity) from `sessions/release-plan.md`. Boundary-touching: spec doc + CHANGELOG + Cross-Repo block + version constant all bump in the same cutover.

## Phase 0 — Premise verification (before Phase 1 sign-off)

Three load-bearing premises drive the range-parameter shape and the read-implementation shape. Verify all three at session start before anything else; the right answer to (1) collapses the range-parameter options.

1. **Log channel shape.** Read `config/logging.php`, `.env`, and `.env.example`. Confirm whether the install uses `LOG_CHANNEL=stack` with `LOG_STACK=single` (one ever-growing `storage/logs/laravel.log`) or `LOG_STACK=daily` (`storage/logs/laravel-YYYY-MM-DD.log` rolling daily).
   - **As of session 250:** `LOG_CHANNEL=stack`, `LOG_STACK=single`. The CRM writes one unbounded `laravel.log`. Range option B (`?date=YYYY-MM-DD`) cannot work without a channel migration. Confirm this hasn't changed before signing off Phase 1's range-parameter shape.
2. **Volume layout for `storage/logs/`.** Read `docker-compose.yml` and `docker-compose.prod.yml`. Confirm that the `app` and `worker` services share the `storage/logs/` path so that FM gets a unified view through `/api/logs` (the controller runs in the `app` container; queue-job logs land via the `worker` container).
   - **As of session 250:** local compose bind-mounts `.:/var/www/html` on both services; prod compose mounts the named volume `storage_data:/var/www/html/storage` on both. Logs are unified. Confirm this hasn't drifted before signing off the endpoint shape.
   - If a future compose layout breaks this assumption: either bind-mount a shared volume specifically for logs, document the gap explicitly in the spec, or defer the endpoint until log aggregation lands. Surface and decide.
3. **Concurrency posture on the shared file.** With `LOG_CHANNEL=stack`/`single` plus shared volume, both `app` (PHP-FPM) and `worker` (queue) processes append to the same `laravel.log`. PHP's stream-write at the OS append level is line-atomic for line-sized writes; concurrent readers (the new `/api/logs` controller) seeing a partial trailing line is possible but not catastrophic. Note in the spec, do not engineer around it.

If (1) is `daily`, range options reopen to include B. If (1) is `single` (today's posture), range options collapse to A (tail-only) or C (since-watermark on the single file). The recommendation below assumes today's posture.

## Design decisions (resolved before starting)

- **Auth.** Reuse the existing nginx-terminated mTLS gate. Apply the same per-location `if ($ssl_client_verify != "SUCCESS") { return 403; }` block on the new `/api/logs` location. No application-layer auth code. Same `fm-client.crt` trust — operators do not re-paste anything.
- **Throttle.** Same as `/api/health`: `throttle:60,1`. Note: nginx rejects no-cert/wrong-cert requests with 403/400 before they reach PHP, so the throttle never sees auth-failure storms. The cap is real protection against **authenticated polling-loop bugs** on the FM side, not auth-failure abuse. Apply at route registration in `routes/api.php`.
- **Path.** `/api/logs` (proposed). Sibling to `/api/health`. **Confirm at Phase 1.**
- **Range parameter shape — RESOLVE AT PHASE 1 KICKOFF (after Phase 0 confirms channel posture).** Three options:
  - **(A) `?lines=N` (tail of the single file).** Works against today's `single` channel. FM asks for the last N lines of `storage/logs/laravel.log`. Default N = 500, capped at 10K. Simplest.
  - **(B) `?date=YYYY-MM-DD&lines=N` (date + tail).** **Requires the `daily` channel.** As of session 250, the install runs `single` — picking B implies an additional sub-decision: flip `LOG_STACK=daily` in this session (operational change with side effects: existing `laravel.log` stays a static historical blob; new days roll daily; FM cutover must be coordinated). Or defer B until a future channel-migration session and ship A now.
  - **(C) `?since=ISO8601&lines=N` (timestamp watermark).** Works against today's `single` channel. FM gets all log entries since a watermark, capped at N lines. Implementation cost is highest — needs to parse line timestamps (Laravel's stack/single channel writes lines like `[2026-04-30 15:42:00] production.INFO: ...`, parseable but a real chunk of code; multi-line stack traces complicate the "where does an entry end" question).
  - **Recommended path: (A).** Today's channel posture, lowest implementation cost, smallest surface for bugs, smallest spec footprint. Date-based selection is a future v2.x bump that pairs with a channel-migration session if/when FM-side UX needs it.
- **Response format — RESOLVE AT PHASE 1 KICKOFF.** Two options:
  - **(I) `text/plain`** — body is the raw log lines, separated by `\n`. Smallest payload; FM displays as-is in a `<pre>` or terminal-shape pane. **Asymmetry note:** if I is picked for success, error envelopes (404/422/500) are still JSON. The asymmetry is sensible (success = raw bytes, errors = structured) but must be explicit in the spec.
  - **(II) `application/json`** — `{ "lines": ["...", "..."], "lines_returned": N, "lines_truncated": true|false, "source": "laravel.log" }`. Adds an envelope FM can display structurally (truncation hint, "fetch more" UI affordance, source identification for future multi-file shapes).
  - **Recommended path: (II).** The existing contract is JSON-shaped (per `/api/health`); a JSON envelope on `/api/logs` keeps consumer-side parsing uniform; the truncation hint is operationally useful (FM warns the operator they hit the cap). Errors and success share format; no asymmetry to document.
- **Size cap.** Default 500 lines, max 10K lines per request. Hard ceiling on bytes too — 1 MB body cap regardless of line count, to defend against pathological lines (full stack-trace dumps). Truncation rule: drop **older** lines when either cap fires (we want the tail). Set `lines_truncated: true` in the JSON envelope.
- **"Line" semantics — newline-delimited rows, not logical log entries.** A "line" in this endpoint means one `\n`-delimited row in the file. Laravel writes a stack trace across dozens of `\n`-delimited rows, so `?lines=500` during a heavy-error period might surface only 3 logical log entries. This is the cheap, well-defined cut; logical-entry parsing is out of scope. The spec doc must state this explicitly so FM-side UX assumptions don't drift.
- **Read implementation — tail-style backward read, not `file()`.** With `single` channel, `laravel.log` can grow to multi-GB by the time A2 ships in production. A naive `file($path)` to grab the last 500 lines OOMs on pathological logs. The controller must implement a backward read (seek from EOF, read in fixed-size chunks growing backward, accumulate lines until N is hit or the byte cap is hit). Keep the implementation small but explicit — this is the load-bearing operational concern at scale. Reference shape:
  - Open file, `fseek($fp, 0, SEEK_END)`, walk back in 8 KB chunks, split on `\n`, accumulate from tail, stop at N lines or the agreed byte cap.
  - **Long-line edge case.** A single log line can exceed 8 KB (SQL-error dumps with embedded queries; logged request payloads). If a chunk read contains zero `\n`, the implementation must keep walking backward — accumulating into a growing buffer — until a newline is found or BOF is reached. The naive loop ("read 8 KB, split on `\n`, take from the end") silently truncates a trailing partial line otherwise. One explicit unit/feature test should plant a >32 KB single line near EOF and verify the tail returns it intact.
- **Size cap on the encoded body, not the raw line content.** If Phase 1 picks format II (JSON), the envelope adds 10–15 % overhead from quoting + escaping + comma separators. Enforcing the 1 MB cap as `strlen($rawLines) > 1MB → truncate` produces an encoded response of 1.1 + MB. Two equivalent shapes — pick whichever is cleaner in the implementation:
  - **(a)** Cap the raw content at 850 KB (leaves headroom for envelope + escaping; conservative).
  - **(b)** Encode incrementally and truncate when the encoded byte length crosses 1 MB (precise but slightly more code).
  - Format I (text/plain) doesn't have this concern — raw bytes are the body.
- **Source path — register a `logs` disk in `config/filesystems.php`.** Add a new disk entry pointing at `storage/logs/` so:
  - The controller resolves the file path via `Storage::disk('logs')->path('laravel.log')` for the seekable `fopen()` backward-read shape.
  - Tests use `Storage::fake('logs')` to swap to a temp dir with automatic cleanup — no custom config keys, no ad-hoc path injection, follows the existing Laravel Storage idiom (the same idiom widget thumbnails and media library use).
  - **Do not** reuse `Storage::disk('local')` — it roots at `storage/app/`, not `storage/logs/`. If for any reason the disk-registration approach is rejected at session kickoff, the fallback is a config key under the existing `config/fleet.php` (e.g., `config('fleet.logs_source_path')`), **not** a new key under `config/logging.php` (which is Laravel's logging-driver config, not a contract-shape concern).
- **Security posture.** No PII surfaces in logs by current convention (the team has historically followed Laravel's "don't log secrets" discipline). The endpoint inherits that discipline — it does not re-redact. If a future audit surfaces a leak, redaction lands in a separate session, not retroactively in v2.1.
- **No log-write affordance.** Read-only endpoint. FM cannot post log entries to the CRM, cannot rotate, cannot delete. Only fetch.
- **Threshold-derivation deferral — RESOLVE AT PHASE 1 KICKOFF.** The other open boundary item (FM-raised question of whether `/api/health`'s `last_backup_at.status` should be FM-canonical or CRM-canonical) was explicitly deferred at v2.0.0. Two options:
  - **(α) Fold into v2.1.0.** Bump CONTRACT_VERSION once, settle both boundary items, give FM one absorption pass.
  - **(β) Keep deferred.** Land log-fetch at v2.1.0; threshold-derivation lands at a future v2.x. Tighter retrospective; cleaner version semantics.
  - **Recommended path: (β).** v2.1.0 stays narrowly the log-fetch addition; threshold-derivation is its own conversation when FM-side raises it again.
- **Spec doc + version bump.** `HealthController::CONTRACT_VERSION` bumps to `'2.1.0'`. The constant lives on `HealthController` despite covering both endpoints; there's no need to move it. Spec doc gets a new "Endpoints" section (renamed from singular "Endpoint") covering `/api/logs` alongside `/api/health`, plus a CHANGELOG entry for v2.1.0.

## Phase 2 — `LogsController`

- New controller at `app/Http/Controllers/Api/Fleet/LogsController.php`. Single action: `index(Request $request): JsonResponse` (or `Response` if format I is agreed at Phase 1).
- Read range parameters per Phase 1's agreed shape.
- Register a `logs` disk in `config/filesystems.php` pointed at `storage/logs/` (per Design decisions). Resolve the absolute file path via `Storage::disk('logs')->path('laravel.log')` for `fopen()`-style backward reads.
- **Backward-read tail implementation.** Open the file with `fopen($path, 'r')`, seek to EOF, read in 8 KB chunks walking backward, split on `\n`, accumulate lines from tail until N lines or the byte cap is reached.
  - **Long-line guard.** If an 8 KB chunk read contains zero `\n` characters, keep walking backward — growing the accumulator buffer — until a newline is found or BOF is reached. Do not silently truncate a trailing partial line.
  - **Byte-cap math.** If Phase 1 picks format II (JSON), enforce the 1 MB cap on the encoded JSON body, not on the raw line content. Either (a) budget raw content at ~850 KB and let the envelope take the rest, or (b) encode incrementally and stop when the encoded byte length crosses 1 MB. Pick whichever produces cleaner code; either is correct.
  - Do not use `file($path)` — the entire-file load OOMs on multi-GB logs.
- Return the agreed envelope shape. "Lines" in the response means newline-delimited rows from the file, not logical log entries (a single multi-line stack trace counts as N rows).
- Error envelopes (always JSON, even if Phase 1 picks I for success):
  - **404 with JSON envelope** if the log file does not exist (e.g., a fresh install before any request fired). FM distinguishes "no log yet" from "endpoint broken."
  - **422 with JSON envelope** for invalid parameters (negative `lines`, lines exceeding the cap, malformed date if Phase 1 picks B).
  - **500** is reserved for unexpected exceptions (file unreadable, etc.). Match `HealthController` exception-message discipline (class name only, no paths).

## Phase 5 — Tests

- **Feature test** at `tests/Feature/Api/Fleet/LogsControllerTest.php`:
  - **Happy path:** GET `/api/logs` (or whatever path is agreed) returns 200 + the agreed envelope shape with the last 500 lines of a fixture log file.
  - **Range parameter clipping:** requesting `lines=15000` returns 10K lines (the cap).
  - **Size cap:** with a synthetic 5 MB fixture log file, response body is ≤ 1 MB on the encoded body (not just on raw line content) and `lines_truncated: true`. Use a temp fixture path, not the real `storage/logs/laravel.log` — the test must not pollute the running app's logs.
  - **Backward-read correctness:** with a fixture log file containing 1000 numbered lines, `?lines=10` returns the last 10 (lines 991–1000). Verifies the seek-from-EOF read logic is correctly ordered.
  - **Long-line regression.** With a fixture file containing a single >32 KB log line near EOF (simulating a SQL-error dump or full request payload), `?lines=1` returns that line intact — not silently truncated. Guards the long-line edge case in the backward-read implementation.
  - **Missing file:** with no fixture log file present (a fresh `Storage::fake('logs')` with nothing planted), returns 404 with the agreed JSON envelope.
  - **Bad parameters:** negative `lines` returns 422; `lines` exceeding the cap returns 422 (or clips silently — pick one at Phase 1 and document).
  - **Throttle — fast assertion only.** Match the CRM 238 v1.0.0 retrospective pattern: source-grep the route registration to assert the route carries `throttle:60,1` middleware. Sixty-one sequential HTTP requests in a fast test pushes the suite past the 5s budget; a 60-call exercise belongs in `->group('slow')` if wanted at all (the source-grep covers the contract claim either way).
  - **Auth pretend (test-file header note):** the mTLS gate is at nginx, not the application. Pest tests cannot exercise the real gate (they don't go through nginx). Tests assert application behaviour assuming the request reached the controller. mTLS posture is verified manually with curl + the local `localhost.crt` against `https://localhost/api/logs`. Document this discipline at the test file's top.
- **Slow-group throttle exercise** *(optional)* at `tests/Feature/Api/Fleet/LogsControllerThrottleTest.php` with `->group('slow')`: exercises 60 successful + 1 throttled request to verify the cap fires. Skip unless the source-grep alone feels insufficient.
- **Contract version test** — extend whatever existing test asserts `CONTRACT_VERSION` shape to assert `'2.1.0'`. If no such test exists, add a one-liner.

**Fixture discipline.** Tests use `Storage::fake('logs')` to swap the registered `logs` disk to a temp directory with automatic cleanup — never write to the real `storage/logs/laravel.log`. The controller's `Storage::disk('logs')->path('laravel.log')` resolves to the faked path during tests; planted fixture content lands in the temp dir; teardown is automatic via Laravel's Storage idiom. No ad-hoc `tempnam()`, no manual `afterEach` cleanup, no custom config keys.
