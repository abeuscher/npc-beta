## Design decisions (resolved before starting)

- **Audit with small-fix in place.** Same rule as 271 — clearly safe fixes (dead imports, stale references, confirmed dead code, doc drift, single-line framework-alignment wins) are applied during the audit branch. Anything that needs strategic input is captured on W7 / W8 / Open Flags and left for session 273.
- **Flag large issues explicitly.** If the audit surfaces something bigger than a W7 / W8 entry — a subsystem-level divergence that needs real strategy — capture it as an Open Flag with enough context (risk, scope, proposed paths) and surface it to the user before continuing the sweep.
- **Lifecycle walk, not duplication grep.** For each subsystem, walk end-to-end: entry points → data flow → terminations. The 205 §W2 (CRM Importer) shape is the canonical model — reading the subsystem's seven-or-so structural pillars (registries, pages, services, models, blades, tests, paths through them) rather than scanning for repeated 10-line blocks.
- **Bug fixes are out of scope.** Catching a real bug during the audit triggers an Open Flag, not a fix — unless the fix is one line and obviously correct.
- **No subsystem-walk-driven schema changes.** Anything that wants a column / table / index goes on Open Flags for 274 to absorb during the squash.
- **Migration squash is not in this session.** The squash is session 274; it follows the 181 / 208 procedure verbatim.
- **Long runtime expected.** Five subsystem walks. Split into multiple iteration branches (`session-272/1`, `session-272/2`, …) if the audit cannot complete in one sitting.

## Workstream 4 — Fleet Manager API surface (deep pass — read-only)

**Four-endpoint divergence walk:**

- Error envelope shape across `/api/health`, `/api/logs`, `/api/backup/trigger`, `/api/backup/blob`. Tabulate each endpoint's 4xx + 5xx shapes; identify divergences. Flag for 273 if a unification is wanted (likely shape: an `EnvelopeBuilder` service or a base controller method).
- `contract_version` handling — 268 noted that `/api/backup/blob` 404 / 500 do *not* carry it. Confirm and decide: extend to all error responses, or document the asymmetry?
- `sanitise()` usage — only on BackupController? Should logs/health also sanitise messages?
- Throttle buckets — `60,1` for health/logs/blob, `6,1` for trigger. Document the per-bucket reasoning if not already in the spec doc.

**Structural-fix proposal — version-stamp drift prevention.**

Currently `HealthController::CONTRACT_VERSION = '2.3.0'` and `BackupController::CONTRACT_VERSION = '2.3.0'` and `docs/fleet-manager-agent-contract.md` `Contract Version: 2.3.0` are three independent sources of truth. Drift caught once at 268. Propose a structural fix:

- Option A: parse the spec doc's version line at boot (`config('fleet.contract_version')` derived from the markdown). Brittle, hidden coupling.
- Option B: a `ContractVersionTest` that asserts the controller constants match the spec doc (CI-enforced). Cheap, catches drift at PR time.
- Option C: extract `App\Http\Controllers\Api\Fleet\Concerns\HasContractVersion` trait that declares the constant; controllers `use` it. Reduces duplication but doesn't fix the doc/code drift.

Flag for 273 with the proposed paths.

**mTLS gate sanity:**

- Every endpoint's nginx location block carries `ssl_verify_client on;`?
- The throttle middleware is on every endpoint?
- The `Cache-Control: no-store` is consistent across endpoints?

Output: structured findings block + table of envelope shapes + structural-fix proposal. **No code changes** in W4 — the FM contract is sealed at v2.3.0; structural fixes land in 273.
