# 356. Media Picker — Browse or Upload in the Page Builder

When an author attaches an image to a widget today they can only **upload a new file** (then an optional post-upload dedup modal offers matches) — there is no way to **visually browse the site's existing images and pick one**. This session adds a media-picker modal to the page-builder image fields: a thumbnail grid of existing site images you can search and select, plus an upload affordance, in one surface. Not fancy — but it must show image thumbnails.

## What already exists (verified at 355 close — this is the foundation)

The CMS has substantial media infrastructure. The gap is a **browse-and-pick UI during authoring**, not storage or plumbing.

- **Storage / model:** Spatie laravel-medialibrary over **content-addressed storage** (`cas/<hash[0:2]>/<hash>/`) with byte-level dedup and refcounted deletion. Key services: `app/Services/Media/ContentAddressedPathGenerator.php`, `ContentAddressedFileRemover.php`, `MediaContentHasher.php`. The `media` table carries `content_hash`, `collection_name`, `model_type`/`model_id`, `mime_type`, `size`, conversions (`database/schema/pgsql-schema.sql`, media block). Disk: `public`. Config: `config/media-library.php`.
- **Attach-existing backend — ALREADY DONE.** `PageBuilderApiController::useExistingImage(widget, key, mediaId)` (route `POST widgets/{widget}/use-existing-image`) and `useExistingAppearanceImage(widget, mediaId)` (route `…/use-existing-appearance-image`) already attach an existing media row to a widget field by id and return `{ media_id, url }`. The page-builder API client (`resources/js/page-builder-vue/api.ts`) and the `useUploadActions.ts` composable already expose `useExistingImage` / `useExistingAppearanceImage`. **The pick-an-existing-image action is wired end-to-end — what's missing is the UI that lets you choose which one.**
- **Upload path:** `uploadImage` / `uploadAppearanceImage` (gated by `BlockDemoUploads`); SVG sanitised (`MediaSvgSanitizer`). Vue field: `resources/js/page-builder-vue/components/fields/ImageUploadField.vue` (current preview + remove + file input); appearance image lives in `InspectorPanel.vue` + `useUploadActions.ts`.
- **Post-upload dedup modal:** `resources/js/page-builder-vue/components/DedupPromptModal.vue` + `MediaDedupController::check()` (`POST media-dedup-check`, hash-keyed) + `MediaDedupService`. This is a **precedent** for a thumbnail-grid + select modal calling `useExistingImage` — but it only surfaces matches for a *just-uploaded* file's hash, not a general browse.
- **Existing admin surfaces:** **CMS → "Media Library"** (`app/Filament/Pages/MediaLibraryPage.php`) is a browse/search/delete/export *table* over all media (thumbnail column, owner, collection, conversions) — its thumbnail + owner query is a reusable reference. **Tools → "Media Finder"** (`MediaFinderPage.php`) scans for unused/duplicate/missing media. `MediaReferenceInventory` classifies live vs dead.
- **Render:** `resources/views/components/picture.blade.php` (responsive WebP + SVG fallback); conversions per model (`registerMediaConversions`, `ImageSizeProfile`).
- **Demo gate:** `BlockDemoUploads` 403s every *new-file* upload endpoint for the `demo` role; **reusing existing media is intentionally NOT gated** (no new file). The picker must honour this — browse/select stays open for demo, only the upload affordance is gated.

**Genuinely missing (this session builds exactly these two things):**
1. A **list endpoint** to enumerate browsable image media (none exists — the dedup check is hash-specific, not a general browse).
2. A **media-browser modal** (thumbnail grid + search + click-to-select + upload), wired into the widget image field and the appearance image field.

## Open questions to resolve at session start (bring a recommendation to the open gate)

1. **Dedup the browse list by `content_hash`?** The same image reused in five places is five `media` rows over one physical file. Recommend **showing each distinct image once** (group by `content_hash`) and attaching the chosen row's id — natural given CAS, and it keeps the grid clean.
2. **Browsable scope.** All `image/*` media, or exclude obviously-internal collections (e.g. `widget_types` `thumbnail`/`thumbnail_hover`, system site assets)? Recommend **author-content images only** — surface the exclude list at the gate.
3. **Modal home + relationship to `DedupPromptModal`.** New shared Vue modal in the page-builder app. Decide whether the post-upload dedup modal **folds into** the new browser (one surface: "you just uploaded — here are similar existing ones") or they coexist. Recommend a new component; evaluate subsuming the dedup modal rather than maintaining two grids.
4. **Video fields.** The image field type also accepts video (`mp4`/`webm`). Recommend **images-first**: the browser serves `type: image` fields this session; decide at the gate whether `type: video` fields get the browser now (filename/icon, no thumbnail) or in a follow-on.
5. **Thumbnail source.** Reuse an existing conversion (`webp` / `responsive-*`) vs a dedicated small thumb. Recommend reusing what `MediaLibraryPage`'s thumbnail column already uses, so no new conversion is introduced.

## Phases

Feature-style session against a green Playwright net. Split across iterations per Rule 11 if the endpoint contract proves load-bearing; otherwise land together.

### Phase 1 — Browsable-media list endpoint

A `GET` page-builder API endpoint returning browsable image media: `media_id`, thumbnail URL, filename, (optional) dimensions/size — deduped by `content_hash` (Q1), scoped per Q2, searchable by filename, paginated. Reuse `MediaLibraryPage`'s query + thumbnail approach and `MediaReferenceInventory` where useful. Auth via the existing page-builder API middleware. **Pest** for the contract.

### Phase 2 — Media-browser modal + field wiring

A Vue modal: thumbnail grid + search box + click-to-select (calls the existing `useExistingImage` / `useExistingAppearanceImage`) + an upload affordance (reuses `uploadImage`, demo-gated). Wire a "Browse library" entry point into `ImageUploadField.vue` and the appearance background-image field (`InspectorPanel.vue` / `useUploadActions.ts`). Keep the existing upload + remove behaviour intact. `npm run build` after.

### Phase 3 — Verify the browse-and-pick flow

Playwright on the isolated e2e page-builder stack: open the picker from a widget image field → see thumbnails → select → the widget's image updates and persists; the upload path still works; the `demo` role can browse + select but the upload affordance is gated. Build + `docker cp` the bundle into `nonprofitcrm_e2e_app` (image-baked named volume).

## Out of scope (fence hard)

- **Metadata on media** — `alt_text` / `caption` / `title` columns + editing. The natural companion (the owner's "nice-to-have at some point"), but it needs a migration and a metadata-edit surface — **defer to a follow-on.** Alt text stays per-use this session.
- **Folders / tags / albums** — no user-defined organisation; collections stay system-driven.
- **Upgrading the `MediaLibraryPage`** itself (grid view, promoting it to a Resource) — leave it as-is; this session adds a *picker*, not a library rebuild.
- **Inline rich-text (Quill) image picker** — a different surface (editor toolbar, not a widget field). Note as a follow-on; don't build it here.
- **Video thumbnail generation.**
- **Schema / migrations** — none expected (the browse reads existing media). **Fleet Manager surface** — untouched; non-boundary, CRM stays **v2.4.0**.

## Drift signals (stop and surface)

- The dedup-by-hash grouping (Q1) can't cleanly map back to a single attachable `media_id` for a chosen image — surface before forcing it.
- Wiring the browser into the appearance image field forces a change to the shared upload composable that risks the existing upload/remove behaviour — surface before committing.
- The browsable-scope exclude list (Q2) turns out to need per-collection knowledge the model doesn't expose — surface rather than guessing what's "internal."

## Testing

- **Pest:** the new list endpoint — auth, returns image media only (non-image excluded), dedupes by `content_hash`, search filter narrows results, the `demo` role can still browse (no upload involved). Follow existing page-builder API controller test conventions.
- **Playwright:** the browse-and-pick flow + upload-still-works + demo-gate, on the isolated e2e stack. Run Pest and Playwright **sequentially** (never parallel — they collide on DB/FPM). Use the canonical `npm run test:e2e` runner (it sets the compose-targeting env so `resetDatabase()` targets the e2e stack, **not** dev — a bare `playwright test` with only `APP_URL` will `migrate:fresh --seed` your dev DB; learned at 355).
- **Build:** `npm run build` after any `resources/js/**` change, then `docker cp public/build/. nonprofitcrm_e2e_app:/var/www/html/public/build/` (the e2e build volume is image-baked).
