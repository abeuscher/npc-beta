### Phase 3 — `HealthController::checkLastBackupAt()` rewrite + CONTRACT_VERSION bump

- `app/Http/Controllers/Api/Fleet/HealthController.php` — `CONTRACT_VERSION` constant `'1.1.0' → '1.2.0'`.
- `checkLastBackupAt()` body fully rewritten: file-missing → `unknown` with `'no successful backup yet'` message; file-empty → `unknown` with `'last-backup-at file is empty'`; unparseable timestamp → `unknown` with `'last-backup-at file unparseable'`; recent (<24h) → `green` with the ISO timestamp normalized through Carbon; 24–36h → `yellow`; >36h → `red`. Going through the Storage facade keeps controller and listener using the same abstraction, makes `Storage::fake('local')` work seamlessly in tests, and removes path-string coupling. Surfaced in the manual-test announcement at the time.

## Production deployment notes

Setting `SPACES_KEY` / `SPACES_SECRET` / `SPACES_BUCKET` / `SPACES_REGION` / `SPACES_ENDPOINT` plus `BACKUP_DISKS=spaces` on the production droplet's `/opt/nonprofitcrm/.env` is required before the next deploy. Per-install bucket setup procedure documented in `docs/app-reference.md` Backups section. No migration needed.

**Important caveat — scheduler runner gap.** The 242 base prompt's deployment one-liner stated "the daily scheduler runs `backup:run` against the configured bucket; first successful run flips `last_backup_at.status` from `unknown` to `green`." That is **not accurate today.** The `worker` service in both `docker-compose.yml` and `docker-compose.prod.yml` runs `php artisan queue:work` only — no `schedule:work`, no host-cron-driven `schedule:run`. The 242 wiring registers `backup:clean` and `backup:run` in `bootstrap/app.php`'s `withSchedule()` block (visible via `php artisan schedule:list`), but they will not fire automatically until a scheduler runner is added. Manual `docker exec nonprofitcrm_app php artisan backup:run` works today and is the supported mechanism until the runner gap is closed. Listed as a carry-forward in `sessions/tracks/fleet-manager-agent.md` and surfaced in `docs/app-reference.md` Backups → "Scheduler runner — known gap" subsection.

After deploy, even without the runner: contract v1.2.0 is live (controller code emits `'1.2.0'` regardless of whether any backup has run); `last_backup_at.status` will report `unknown` with `'no successful backup yet'` until the operator runs `backup:run` manually for the first time. FM-side `StatusInterpreter` is at v1.1.0; FM-side semantics treat `unknown` as yellow-tier, so the existing alerting behavior is unchanged from v1.1.0 production state.

After running `backup:run` manually for the first time on production: confirm a fresh blob lands in the configured Spaces bucket via the DO console, confirm `storage/app/private/fleet/last-backup-at` exists with a recent ISO timestamp, hit `/api/health` with the bearer token and confirm `last_backup_at.status === 'green'` with `value` matching the file contents and `threshold === [24, 36]`.

## Deferred decisions

- **Scheduler runner on the worker container.** Named carry-forward; needs its own session (sidecar variant of the worker, or host-level cron). Out of scope by the 242 prompt's "do not implement scheduler-runner infrastructure in this session" rule. Surfaces operationally as soon as a production install wants automated daily backups.
- **`BackupHasFailed` event listener.** Out of scope by design — failure is observed via timestamp aging into yellow then red on the agent endpoint. Active failure-alerting (email, Slack) revisited if FM-side observation proves insufficient.
- **Backup-restore tooling.** No `php artisan backup:restore` command; restoration is the documented manual procedure in `docs/app-reference.md`. Verified end-to-end during 242 manual testing. Tooling deferred — restoration is high-stakes and benefits from operator pause.
- **Per-install retention configuration beyond the 14-day default.** Lands when a client has a different requirement.
- **Multi-region backup redundancy** (DO Spaces in NYC3 + Backblaze B2 in EU, etc.). Future infrastructure.
- **Backup integrity verification** (download blob, parse, confirm structure). Future infrastructure.
- **`Storage::fake('public')` audit of tests.** Some tests may write real media to the public disk during execution; surfaces as orphan accumulation alongside the `migrate:fresh` issue. Lands more naturally as a Test Audit slice item than a Housekeeping item.
- **Backup pipeline tweaks queued for 243** (Housekeeping Batch 1, Phase 2): flip `database_dump_compressor` to `\Spatie\DbDumper\Compressors\GzipCompressor::class` (~10× dump compression at trivial CPU cost); set `relative_path = base_path()` so backup zips store media at `storage/app/public/...` instead of `var/www/html/storage/app/public/...`. Surfaced during 242 manual-test pass.
- **Sitemap content-type expansion** queued for 243 audit. The audit is the deliverable; fixing requires a separate scope decision about which content types belong in the sitemap.
