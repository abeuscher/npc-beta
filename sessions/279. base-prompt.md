## Style rules

The style rules in `sessions/template-base-prompt.md` apply. Session-specific:

- **Iteration commit messages describe the deliverable** — e.g. "Iteration /1 — quantity column + controller rework + webhook fan-out + per-tier withSum" or "Iteration /2 — quantity-spinner widget + Playwright + admin tickets column".
- **Quantity defaults to 1, never null.** A registration row always represents at least one ticket; `quantity = 0` is not a valid state (a cancellation flips `status`, not `quantity`).
- **One row per (buyer, tier) pair within a purchase.** Mixed-tier purchases produce multiple rows sharing a `stripe_session_id` (paid) or sharing a `created_at` window + email (free). The "buyer" abstraction lives implicitly across these rows; no separate `event_orders` table.
- **The webhook's stripe_session_id-based grouping is the canonical paid-path linkage.** Free-path mixed-tier purchases produce N rows without a Stripe linkage; the rows share `email` and an instant `created_at` — sufficient for the admin "View Registrants" grouping but not a strong relational join. If this becomes load-bearing later, the post-1.0 "Event Registration Depth" stub picks it up.
- **No docstrings, comments, or type annotations on code you did not write.**
- **Simple, correct, well-considered solutions over fast or clever ones.**
