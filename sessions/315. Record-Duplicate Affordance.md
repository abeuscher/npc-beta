## Session-specific deltas

- **First-resource-sets-pattern rule.** The first iteration picks whichever target has the cleanest existing duplicate-style affordance to mirror (or the simplest model if no precedent). The user signs off on the resulting pattern before the remaining three resources port to it.
- **Pattern source.** The codebase may already have duplicate or copy actions on other resources (e.g. content templates, widget instances via the page builder); inventory at session start to find the closest matching pattern.
- **Per-resource semantics.** The walkthrough at session start resolves what carries forward on duplicate. See *Open questions* below.

## Open questions to resolve at session start

These need explicit user sign-off before any code lands. Each is per-resource because the answers may legitimately differ across the four targets:

- **Slug / identifying name** — almost certainly modified (append `-copy`, or `Copy of …`, or open the duplicate in edit mode with an editable suggested slug). Confirm the convention.
- **Timestamps** — `created_at` resets to now? `updated_at` resets to now? Are there domain-specific timestamps on Events (`starts_at`, `ends_at`) that should reset, carry forward, or be cleared?
- **`created_by` / author** — current user (duplicator), or carry forward the original author?
- **Publish state** — duplicate lands as `draft` regardless of source state? Or carries forward the source's publish state?
- **Relationships** — `tags`, `categories`, `taggables`: re-use the same target rows (almost certainly), not duplicate the related rows themselves.
- **Embedded media** — re-use the same media records (almost certainly), not duplicate the media files.
- **Page widgets / layouts** (Pages and Posts only) — these are owned children of the page. Duplicate the children too, re-pointing `owner_id` to the new page? Yes by definition or the duplicate would be hollow.
- **Product variants / Event ticket tiers** — same question as widgets/layouts; same default answer (duplicate the children).

## Phases

Five iterations expected:

### Phase 1 — Pattern-lift + first resource

Inventory existing duplicate-style actions; identify the cleanest pattern. Implement Duplicate on the first resource (recommend Pages or Posts — the page builder's existing copy-widget code is the most likely source pattern). Both surfaces (list-level + record-page header). Land tests covering: the new record's column shape vs. the source, child-row handling, slug uniqueness, redirect/route to the new edit view.

### Phase 2 — Port to second resource

Apply the pattern lifted in Phase 1 to the second resource. Note any per-resource deviations that surfaced during port.

### Phase 3 — Port to third resource

Same shape as Phase 2.

### Phase 4 — Port to fourth resource

Same shape as Phase 2. By this point any pattern weaknesses have been amortized.

### Phase 5 — Polish + bug-fix sweep (if needed)

If the four ports surfaced consistency issues (e.g. button labels, action ordering, confirmation modal text varying across resources), one polish iteration normalizes them.

## Out of scope

- Duplicate semantics for resources outside the four named targets (Donations, Memberships, Notes, Invoices, etc.).
- Bulk-duplicate (e.g. "duplicate 5 selected records") — single-record duplicate only at both list and record surfaces.
- Cross-resource duplicate (e.g. "duplicate this Page as a Post").
- Versioning or revision history — duplicate creates a new independent record, not a snapshot.
- Schema changes — duplicate operates over existing tables.
