# 176. Designer Preset Drafts & Export

Second session of the Stage 5 preset sub-arc. Session 175 shipped the Preset Inspector tab with code-authored presets and replace-semantic apply. This session closes the designer-feedback loop: a designer can iterate on a widget's appearance live in the builder, save the current look as a draft preset from inside the inspector, optionally rename/describe it, and export it as a PHP array literal ready to paste into the widget's `presets()` method. The intent is that user-authored presets are *input to the code*, not a parallel runtime source — the draft pool lives in the DB so the designer can iterate without touching code, but promotion to a shipped preset is a deliberate copy-paste into the definition class.

---

## Design decisions (resolved before starting)

- **New `widget_presets` table, not a widget-level JSON blob.** Drafts live in a dedicated DB table scoped by `widget_type_id`. Columns: `id` (uuid), `widget_type_id` (uuid FK → `widget_types`, cascade on delete), `handle` (string, slug, unique per widget_type_id), `label` (string), `description` (text, nullable), `config` (jsonb, default `{}`), `appearance_config` (jsonb, default `{}`), `timestamps`. No FK to `pages` or `page_widgets` — a draft belongs to a widget type, not an instance.
- **Drafts are global per widget type, not per site/user.** A single install has one pool of drafts per widget type. No per-user ownership, no per-template scoping. The draft pool is small by design (10s of entries, not 1000s) and the designer workflow assumes one editor at a time in practice.
- **Appearance-group rule enforced server-side.** Every DB-authored preset's `config` keys must correspond to schema fields whose `group` is `appearance`. Validation runs on save (reject with 422) and is re-asserted by the existing `WidgetManifestTest` preset-shape suite — extend the test to cover DB rows as well as code-declared presets.
- **Save action captures live instance state.** The "Save current appearance as preset" action reads the selected widget instance's current `config` (filtered to appearance-group keys only) and full `appearance_config`, and writes those to a new `widget_presets` row. The instance itself is not mutated.
- **Auto-naming — no naming required at save time.** New drafts get an auto-generated `handle` (`draft-N` where N is the next unused integer for that widget_type_id) and `label` (`"Draft N"`). Description is blank. Users rename later if they want. The rationale is explicit: the designer is iterating visually, not coming up with names — names are added when the preset is exported or promoted.
- **Gallery merges code + DB, code first.** The inspector Presets tab gallery continues to render the synthetic "Blank" card, followed by code-authored presets from the widget's `presets()` method, followed by DB drafts. DB drafts get a small "Draft" badge on the card to distinguish them from shipped presets.
- **Draft cards expose rename / delete / export actions.** Code-authored preset cards remain read-only (apply only). Draft cards gain an actions menu with: Rename (inline label + description edit), Delete, and Export. No "duplicate" action — apply + save-current is the duplicate path.
- **Export = copy PHP array literal to clipboard.** The Export action generates a PHP array literal (pretty-printed, 4-space indented) representing the preset's `handle`, `label`, `description`, `config`, `appearance_config`, and copies it to the clipboard via the browser's `navigator.clipboard.writeText`. The user then pastes it into the widget's `presets()` method in the definition class. No file download, no multi-preset export this session. A small toast confirms the copy.
- **Apply semantics unchanged.** Applying a draft card is identical to applying a code-authored preset: overlay `preset.config` onto existing `config`, replace `appearance_config` wholesale, route through the debounced save path. The frontend does not care whether the preset source is code or DB.
- **Promotion is manual paste-then-delete.** There is no "promote draft to code" automation. The designer exports, you (the developer) paste into the definition, run tests, and delete the draft from the gallery. Automation of this handoff is explicitly out of scope.
- **No Filament resource for `widget_presets`.** All draft CRUD happens inside the inspector Presets tab. No separate admin list/edit pages, no bulk actions, no navigation entry. The draft pool is a builder-surface feature, not a top-level admin concept.
- **Permissions reuse the page-builder gate.** Creating, updating, or deleting a draft requires the same permission as editing a page (`update_page`). No new permission names.

## Phase 2 — Backend: save / update / delete / list

Add routes under the existing admin-api group (reuse the page-builder API controller or split into `PresetController` — implementer's call, flag the choice in the log):

- `POST   /admin/api/widget-presets` — body: `widget_type_id`, `widget_id` (the source instance to snapshot). Server reads the instance's `config` (filtered to appearance-group schema keys) + `appearance_config`, generates an auto handle/label, writes the row. Returns the preset.
- `PATCH  /admin/api/widget-presets/{preset}` — body: `label`, `description`, `handle`. Server re-validates handle uniqueness + slug shape. No edits to `config`/`appearance_config` via this endpoint.
- `DELETE /admin/api/widget-presets/{preset}` — removes the row.

Every endpoint gates on `auth()->user()?->can('update_page')`.

Server-side validation rejects a save whose resulting `config` contains any key not found under an appearance-group schema field on the associated widget type. Error message names the offending key.

The bootstrap payload's `widget_types` entries already include `presets` (from session 175, sourced from the registry). Extend that entry to include a second array, `draft_presets`, containing the DB rows for that widget type (shape: `handle`, `label`, `description`, `config`, `appearance_config`, plus a server-supplied `id` and `is_draft: true`). Code presets stay under `presets` and are unchanged.

Add a backend feature test asserting:

- Save captures appearance-group keys only (content-group keys are stripped or the request is rejected — pick one and document the choice; I'd lean reject-with-422).
- Save rejects mutations to a widget type's schema that the draft would now violate (don't worry about schema drift — this is a one-line coverage point, not a full migration story).
- List endpoint (or bootstrap) returns drafts alongside code presets.
- Delete removes the row.
- Update renames and validates handle uniqueness.

## Phase 3 — Frontend: save button, gallery merge, actions menu

1. **`InspectorPresetGallery.vue`** — extend the gallery to render a "Save current appearance as preset" button above the card stack (or below — implementer's visual judgment, but it must sit within the tab, not in the header). Clicking it calls a new store action `saveDraftPreset(widgetId)` that POSTs to the new endpoint and appends the returned preset to the gallery.
2. **Gallery merge logic.** The gallery now iterates: synthetic Blank → `widgetType.presets` (code) → `widgetType.draft_presets` (DB). Each card renders the same layout; DB cards get a small "Draft" pill in the top-right corner of the label row.
3. **Draft card actions.** Add an actions affordance (a kebab menu or a small toolbar revealed on hover — implementer's call) exposing Rename, Delete, Export. Each routes through a corresponding store action:
   - `renameDraftPreset(presetId, { label, description, handle })` → PATCH
   - `deleteDraftPreset(presetId)` → DELETE
   - `exportDraftPreset(presetId)` → generates the PHP literal and writes to clipboard; no server call.
4. **Rename UX.** Inline editing is fine (click-to-edit label; a small description textarea below). Commit on blur / Enter. Revert on Escape. Validation errors surface as an inline message under the field.
5. **Export clipboard format.** The PHP literal must be self-contained and paste-ready. Example shape:

   ```php
   [
       'handle'            => 'dark-hero',
       'label'             => 'Dark Hero',
       'description'       => 'Short sentence.',
       'config'            => [
           'text_position' => 'center-center',
           'fullscreen'    => true,
       ],
       'appearance_config' => [
           'background' => ['color' => '#111111'],
           'text'       => ['color' => '#ffffff'],
       ],
   ],
   ```

   Include the trailing comma so the user pastes it directly into a `presets(): array` return list without editing.

6. **Types.** Add a `WidgetDraftPreset` type (or extend `WidgetPreset` with an optional `id` + `is_draft` flag). Update the `WidgetType` shape to include `draft_presets: WidgetDraftPreset[]`.

## Security checklist

Before closing:

- [ ] All four new endpoints gate on `update_page` permission and reject unauthenticated requests.
- [ ] Save validates that `config` keys resolve to appearance-group schema fields on the widget type; content-group keys are rejected (422) with an explicit error message.
- [ ] CSRF protection is enforced by the existing admin-api middleware pipeline — no new exemptions.
- [ ] Handle uniqueness check is server-side, not trusted to the client.
- [ ] Export writes to clipboard via `navigator.clipboard.writeText` with no server round-trip; the PHP literal contains no credentials, IDs, or session data — only the preset fields.
- [ ] Preset data stays admin-only: `draft_presets` is emitted by the authenticated page-builder bootstrap, not by any public endpoint.

---

## Out of scope

- **Automated promotion to code.** The workflow is export-then-paste; no CLI or admin action writes to `app/Widgets/{PascalName}/{PascalName}Definition.php`.
- **Per-site or per-template preset pools.** Single global pool per widget type.
- **Per-user draft ownership.** Any admin with `update_page` can see, edit, and delete any draft.
- **Preset import** (JSON or otherwise). Import is out of scope; the code path is the canonical authoring surface.
- **Multi-preset export.** One card = one clipboard write. Bulk export of all drafts for a widget type can be a future session.
- **Preset tagging or categorisation.** A flat list per widget type is fine at this volume.
- **Preset thumbnails.** Deferred to session 177.
- **Conflict detection between draft handles and code handles.** The gallery tolerates two cards with the same handle (one code, one DB) — they are visually distinguished by the Draft badge. A handle collision only matters at paste time; CI will reject a duplicate on promotion.
- **Versioning / draft history.** No edit history, no restore. Overwrite in place.
