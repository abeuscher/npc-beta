## What was built

Lifted-gap session resolving **G23** (form submissions don't email the site owner). Scope expanded twice mid-session on user direction; the as-built is materially different from the originally-scoped fixed-template shape. The journey, in order:

### Phase 1 — Notifications, first as a fixed template (superseded same session)

- `app/Mail/FormSubmissionMailable.php`, a `FormSubmissionObserver` (registered via `#[ObservedBy]` on `FormSubmission`), a `site_owner_email` SiteSetting on the Mail settings page, a `Form.settings.notifications` array, a FormResource repeater, and a fixed Blade template `mail/forms/submission-default.blade.php`. Wired the seeded `contact-page` form. 11 Pest cases incl. the three required security controls. Fast suite 2432 / 0. Committed as `e9c8555`.

### Phase 2 — Reworked onto the editable System Email surface (user-directed)

- User asked whether the email is operator-editable. It wasn't. Reworked: deleted the fixed Blade template; added a `form_submission` handle to `EmailTemplate::defaults()` + `DatabaseSeeder`; `FormSubmissionMailable` now renders via `EmailTemplate::forHandle('form_submission')` → `resolveWrapper()` → `mail.system-email`, mirroring `PortalFormCollision`. Submission data enters **only** through an `e()`-escaped `{{submission}}` block token (mirrors the `{{donations}}` pattern); `{{form_title}}` token; subject rendered from a token set that **excludes** submission data so a misconfigured subject can't leak submitted content into a header. Per-form config narrowed to routing only (`to` + `include_submission_data`); subject/body now live in the editable System Email (Settings → System Emails → "Form submission"). `EmailTemplateResource` token hint added.

### Phase 3 — `contact_email` reuse + graceful mail-failure handling (user-directed)

- Removed the new `site_owner_email` SiteSetting entirely (reverted all MailSettingsPage additions); the recipient token is now **`{{contact_email}}`**, resolved against the existing CMS Contact Email SiteSetting (Settings → CMS). Allowlist resolver unchanged in shape (single allowed token, literal-or-reject, single-email validation).
- Graceful mail-failure handling: `app/Exceptions/FormNotificationDeliveryException.php`; the observer wraps the send, `report()`s full detail server-side (no client leak), and throws the dedicated exception; `FormSubmissionController` catches it around `FormSubmission::create()` and returns the form's normal inline error ("could not be sent due to a server error"). The submission row is still persisted; the page never breaks; nothing leaks. The admin "Send test email" action hardened the same way (clean Filament danger notification, no stack trace) — that was the setup path the user actually hit with a miskeyed Resend key.
- Footer line removed from the `form_submission` template (both defaults + the local DB row) — `{{form_title}}` was rendering literally because the wrapper does not token-interpolate `footer_reason`. Logged the root cause; queued the real fix as a post-Beta roadmap stub.

### Phase 4 — FM pre-Beta pseudo-versioning (user-folded FM request)

- `VERSION` file at repo root = `0.291.1`. `deploy.yml` reads + format-validates it (`^0\.X\.Y$`, build fails if missing/malformed), bakes it as the `APP_VERSION` build-arg, tags the GHCR app/web images with that exact string **and** moving `latest`, and **hard-fails the build if that version tag already exists** (`docker manifest inspect` guard) — mechanically enforcing immutability. The runtime side already existed (Dockerfile bakes `APP_VERSION` → `/var/cache/app/VERSION`; `config/fleet.php` reads it; `/api/health` returns it) — no code change there. `docs/fleet-manager-agent-contract.md` `version`-field description + examples corrected (it falsely said "seven-character git SHA"); **contract version intentionally NOT bumped** (no shape/enum change — semantics firmed + doc corrected; consistent with the user's "purely additive, no contract restructure" framing and the 291 base prompt's "contract stays v2.3.0"). `tests/Feature/Infrastructure/AppVersionStampTest.php` locks the whole chain as a fast-suite invariant.

### The five scoping decisions, as resolved

1. **Dispatch:** sync (mirrors existing Mailables; no queue runner). As recommended.
2. **Recipient resolution:** allowlist token + literal, one recipient/entry — but the token is **`{{contact_email}}`** (reused the existing CMS setting; the recommended new `site_owner_email` key was added then removed on user direction).
3. **Template surface:** **not** a single fixed built-in (the recommended default) — reworked to the **editable `form_submission` System Email**, operator-editable like donation receipts.
4. **Spam/volume controls:** none beyond the existing gates. As recommended; residual accepted (security-forms Finding 4).
5. **Demo sequencing:** resolved by sequencing — and the planning audit during 292-prompt drafting found the "Demo builds on the notification surface" framing is **superseded**: Demo (292) is form-less / no-email per the s289-close architecture, so it does NOT consume this surface. This session's notification consumer is the Contact page only.

### Final shapes

- **`Form.settings.notifications`:** `[{ "to": "{{contact_email}}" | literal, "include_submission_data": bool }]`. `subject`/`template` keys dropped (now owned by the editable System Email; legacy keys ignored if present — backward-compatible).
- **Dispatch:** observer (`#[ObservedBy]`), synchronous, one mail per entry, graceful-failure-wrapped.
- **Template:** editable `form_submission` `EmailTemplate` (subject `New submission: {{form_title}}`, body the intro line + `{{submission}}` table), branded wrapper.
- **FormResource UI:** "Email Notifications" section, `to` + include-data toggle, points operators at Settings → System Emails for wording.
- **`contact-page` config:** one entry, `{{contact_email}}`, include data true. `FormSeeder` switched to `updateOrCreate` for that form so re-seed applies it.

End-to-end verified twice via the log mailer (real CSRF'd `/forms/contact-page` submit): correct recipient from the token, correct subject, branded body, escaped field table, no `Cc`/`Bcc`/`Reply-To` leakage.

## Security considerations

The notification surface is the session's main security axis and was treated as such. Upheld and verified by Pest: the **recipient invariant** (submission data can never redirect/add/CRLF-inject a recipient or any header — recipients come only from operator config); the **subject invariant** (subject rendered from a token set excluding submission data, so even a misconfigured `{{submission}}` in the subject can't leak submitted content into a header); **escaping** (submission values enter only through the `e()`-escaped `{{submission}}` token; no `{!! !!}` on submission data); the **allowlist token resolver** (unknown/malformed tokens resolve to nothing, send nothing, leak no SiteSetting value). Graceful mail-failure handling logs full detail server-side via `report()` and never surfaces transport/exception detail to the visitor or the operator UI. `docs/security-forms.md` Finding register rows 4–9 and the planned-291 section were moved from "pre-registered / planned" to as-shipped dispositions ("document as we go" discipline). The shared-account auto-login surface for Demo is pre-flagged as 292's load-bearing security work in the 292 prompt.

## Production deployment notes

- **Re-seed required on deploy targets:** `php artisan db:seed --class=FormSeeder` (applies the `contact-page` notifications config via `updateOrCreate`) and the `DatabaseSeeder` email-templates block creates the `form_submission` System Email (it also self-creates on first notification send via `forHandle`).
- **Set the recipient:** Settings → CMS → Contact Email must hold a real address on each instance (it is the `{{contact_email}}` target).
- **Real delivery needs a real transport:** with the `log` mailer the notification is only written to the log. Configuring Resend is an env decision, out of this session's scope.
- **Versioning discipline (new, enforced):** every merge to main must bump the repo-root `VERSION` to a new `0.X.Y`; the deploy pipeline hard-fails a duplicate/missing version rather than overwriting. 291 ships `0.291.1`; 292 bumps to `0.292.x` (carried into the 292 base prompt).
