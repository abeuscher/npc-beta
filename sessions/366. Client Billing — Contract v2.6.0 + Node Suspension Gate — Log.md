### The suspension flag + enforcement middleware

- **`fleet.suspension.state`** added to `config/fleet.php` (`env('SUSPENSION_STATE', 'none')`), beside the existing agent config — same env-derived grain as demo mode.
- **`App\Services\Billing\SuspensionState`** — a string-backed enum (`None` / `AdminLocked` / `SiteOff`) with `resolve()` (fail-safe: absent/empty → `None` silently; unrecognized → `None` **plus a logged warning**, so a typo in a pushed key never bricks a paying client's admin) and `current()` (reads config). Both the middleware and the health subcheck resolve through it, so the enforced state and the reported state can never disagree.
- **`App\Http\Middleware\EnforceSuspensionState`** — one class, registered on two surfaces via a `:admin` / `:public` middleware parameter:
  - **`:admin`** in the Filament panel's **base** `->middleware([...])` stack (first, so a locked node short-circuits before session/cookie work). Verified via `route:list` that this one registration covers the discovered panel pages, the login, **and** the in-panel API `->routes()` groups (page-builder / theme / dev-tools) — resolving the base-prompt's flagged wiring question favourably; no per-group registration needed.
  - **`:public`** prepended to the **`web`** middleware group in `bootstrap/app.php` (public site + member portal).
  - It is deliberately **not** on the `api` group, so all five FM `/api/*` contract endpoints stay reachable under every state.
  - Behaviour: `admin_locked` → **403** suspension notice on the admin surface (public site, donation/event/membership checkout, member portal, backups, scheduler, FM `/api/*` untouched); `site_off` → **503** maintenance notice on public + admin surfaces (FM `/api/*` untouched); `none` / absent → pass-through. The gate locks correctly with no document present.
- **Two self-contained branded notice views** (`resources/views/suspension/admin-locked.blade.php`, `site-off.blade.php`), following the `errors/layout` defensiveness (read brand/site-name defensively, do not extend the public layout). The admin-locked view switches copy on the reason code (`delinquent` / `trial_expired` / `canceled` / `manual`, generic default) and renders the Stripe-hosted portal link + billing contact only when the document carries them. Stable `data-suspension-notice` attributes for test hooks.

### The billing-state reader (display half)

- **`App\Services\Billing\BillingStateReader`** — reads the FM-pushed display-only JSON at `storage/app/private/fleet/billing-state.json` (the `local` disk's fleet-metadata dir). Returns a **null-object** (`BillingState::absent()`) when the file is missing, unreadable, not valid JSON, a non-object, or carries an unsupported `schema_version` (the last three log a warning). Registered as a **container singleton** in `AppServiceProvider` so the first read is memoized per request (both the lock screen and the subcheck read it within one request).
- **`App\Services\Billing\BillingState`** — a read-only DTO / null-object with typed accessors for the full documented schema (`asOf`, `status`, `plan`, `nextInvoice`, `billingContactEmail`, `portalUrl`, `reason`, `suspensionState`, `graceEndsAt`, `trial`, `isPresent`). CB2's Account page consumes the same object; this session wires only the lock-screen copy (`reason`/`portalUrl`/`billingContactEmail`) and the subcheck (`asOf`).
- Confirmed the path stays outside the backup include set (`config/backup.php` `source.files.include` = `storage/app/public` only) and is untouched by `demo:restore` — per-node billing metadata can never ride a backup blob onto another node.

## Security considerations

The one new hard gate rides the same env-derived, code-level grain as demo mode (`isDemoMode()`), and is orthogonal to it. It **fails open toward access** by design: an absent or unrecognized flag resolves to `none`, so a misconfigured push can never lock out a paying client's admin (the warning log surfaces the typo instead). The stays-up invariants under each state are pinned by the middleware matrix test — under `admin_locked` the public site, donation/event/membership checkout, the member portal, and all five FM `/api/*` endpoints remain reachable; under `site_off` the FM `/api/*` endpoints remain reachable — so suspension can never take down a nonprofit's donation page or a monitored node's recovery surface. The billing-state document is treated as **display-only data**: nothing it returns influences enforcement, routes, or config, and the reader hard-validates its schema version and fails to a null-object. No vendor-Stripe credential, config key, SDK, or webhook exists CRM-side; billing flows FM→node only, and the node gains no new inbound surface (the document arrives over FM's existing SSH provisioning channel, not an HTTP endpoint). The subcheck stays within the contract's counts-only / no-PII wire discipline (a state string + a timestamp).
