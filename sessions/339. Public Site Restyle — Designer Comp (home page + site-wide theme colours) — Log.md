### Phase 3 — Home page sweep (`home` page builder appearance — exportable)

Per-Band (`page_layouts.appearance_config`) background changes:

| Band | bg old → new |
|---|---|
| Band 3 — What it Does (heading) | `#e5e7eb` → `#e9f3fa` |
| Band 4 — What it Does (4-cell grid) | `#e5e7eb` → `#e9f3fa` |
| Band 6 — Pricing | `#e5e7eb` → `#e9f3fa` |

| Band 8 — Try it (final CTA) | `#000000` → `#252b33` |
| Band 2 (Small Data) / Band 7 (What it doesn't do) | unchanged per comp (`#ffffff` / `#e5e7eb`) |

Spacing (the column-padding fix — owner identified the layer): **every Band's vertical padding halved** (50→25, hero 50/100→25/50) — the comp's section rhythm lives on the **column/layout** padding, not the widget; the page had been double-padding (band + widget), running ~20 % taller than the comp. All home **widget** vertical padding zeroed so the band owns the section spacing.

Band 4 grid: `grid_auto_rows` `1fr` → `auto` (cells were forced to equal/tall height, stacking unevenly on mobile; now size to content).

Hero (Band 1): the **hero widget swapped to a `text_block`** so its content sits flush to the content container — the hero widget self-wraps a *second* `.site-container`, which double-contained it inside the column and inset the heading ~53px. Content + CTAs preserved.

Bottom CTA: **split into two dark Bands** — a new `Band 8a` dark heading band carrying the orange **"Try it free!"** heading, above `Band 8` (white subtitle + both CTAs). Two text blocks can't share one grid column (`.widget--text_block { height: 100% }` makes each fill the column → overflow), so the heading got its own band. Subtitle copy adopted from the comp ("First month is free, then $50/month for the next 6 months").

## Production deployment notes

The restyle is **not in this commit** — it is local DB state (the `theme_colors` + `button_styles` design settings and the `home` page's `page_layouts`/`page_widgets`). To ship it:

1. Owner does a **full-site export** (carries pages incl. layouts/widgets/appearance/copy **and** the `design` block = `theme_colors` + `button_styles` + typography) and imports it on the target.
2. On the target, a **`build:public`** (or deploy) must run after import to recompile the colour/button settings into the live CSS bundle — verify the rendered site, not just the settings.

The committed changes (planning docs, prompts, the `demo.json` move + `DemoLoginTest` path, the doc-reference cleanup) need nothing beyond standard `migrate` (and there are no migrations).
