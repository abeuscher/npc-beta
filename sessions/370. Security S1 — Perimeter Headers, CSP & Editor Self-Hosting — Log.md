# Session 370 — Security S1 — Perimeter Headers, CSP & Editor Self-Hosting — Log

**Date:** 2026-07-17
**Branch:** session-370/1

The first session of the Security Hardening track (Gate 3 of the launch plan). Stood up the app's missing perimeter — a security-headers layer with an enforced Content-Security-Policy, self-hosted the admin editor off its CDN, enforced production-hardening defaults in code, and folded in two small guards. A follow-on (same branch) made the CSP host allow-list editable from the product admin instead of `.env`.

---

## What was built

### Phase 1 — Perimeter security headers + CSP

- **`App\Http\Middleware\SecurityHeaders`** (new) sets the header baseline on every web/admin response: `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Referrer-Policy`, `Permissions-Policy` (deny-by-default), HSTS (secure requests only), and a CSP. Registered on the `web` group (`bootstrap/app.php`) and the admin panel base stack (`AdminPanelProvider`). The `api` group is deliberately excluded, with a defensive early-return on `api/*`, so the five Fleet Manager `/api/*` endpoints and their mTLS gates are byte-untouched.
- **CSP, two speeds** (`config/security.php` drives it, env-flippable per surface):
  - **Public/portal: enforced.** `default-src 'self'`; `script-src 'self' 'nonce-…'` (a per-request nonce, **no `'unsafe-inline'`**); `style-src 'self' 'unsafe-inline' https://fonts.googleapis.com` (inline styles are pervasive in the widget appearance layer — accepted v1 trade-off); `img-src 'self' data: blob:` + the media origin (derived from the configured media disk so prod DigitalOcean Spaces works without a hand-set convention); `font-src` for Google Fonts; `frame-src` for the YouTube/Vimeo/Google-Maps embed widgets; `object-src 'none'`, `base-uri 'self'`, `form-action 'self'`, `frame-ancestors 'none'`.
  - **Admin panel: Report-Only.** Same policy plus `'unsafe-eval'` (Alpine needs it). Ships non-enforcing because Filament/Alpine/Livewire enforcement is unproven; flip via `SECURITY_CSP_ADMIN_REPORT_ONLY=false` once validated. **Recorded as a Gate-3 residual.**
  - The nonce is generated in the middleware via `Vite::useCspNonce()` before the view renders; `@vite` and the four legitimate inline framework scripts (public `window.__site`, portal `inlineScripts`, the two password-mismatch bootstraps) carry `nonce="{{ Vite::cspNonce() }}"`. Verified on the running stack that the header nonce matches the rendered script nonce.
- **nginx** (`docker/nginx/default.conf` + `prod.conf`): the static header set added to the static-asset `location` only (files served straight from disk bypass PHP), so no header is duplicated on PHP responses and the mTLS/`/api` blocks are untouched (additive diff, zero lines changed in those blocks). HSTS added at the TLS edge in `prod.conf` only. CSP is intentionally never set in nginx (it needs the per-request nonce).

### Phase 1 — Self-hosted admin editor (Quill)

- Quill pinned at **2.0.3** in `package.json` and bundled into `resources/js/admin.js` (imported + `window.Quill = Quill`, snow CSS imported), retiring the `cdn.jsdelivr.net` `<script>`/`<link>` render hook in `AdminPanelProvider`. The `.ql-editor` min-height rule moved to `public/css/admin.css` (loads after the bundled quill.snow.css, preserving cascade order). Verified the admin page no longer references the CDN and the bundle carries Quill + the global; owner confirmed the editor works.

### Phase 1 — Enforced prod-hardening defaults

- `config/session.php`: `SESSION_SECURE_COOKIE` now defaults **on** for every environment except `local`/`dev`/`testing` (so an HTTPS production/demo node is secure by default; still env-overridable).
- `AppServiceProvider::boot`: force-disables `app.debug` on any `production` node even if a copied `.env` left it true — a bad `.env` can no longer leak stack traces with credentials.
- `.env.example`: security comments on `APP_DEBUG` + `SESSION_SECURE_COOKIE`.

### Phase 1 — Two folded guards

- **Stripe live-key guard** (`App\Support\StripeMode::isLive()` — `sk_live_`/`rk_live_`): the Random Data Generator refuses to generate synthetic data (donations/transactions) when a live Stripe key is configured — the form hides (widget template) and the `store()` endpoint refuses (controller backstop). **Wipe stays available** so leftover scrub data can still be cleared off a live install. Resolves the long-standing 245-era "release blocker" stub.
- **Demo-role seeding gate**: `PermissionSeeder`'s `demo`-role block is wrapped in `isDemoMode()`, so a production node carries no vestigial `demo` role (mirrors the demo-user gate already in `DatabaseSeeder`). Three existing tests that acquire the role in non-demo mode were adjusted to seed it under demo mode (`DemoRoleLockdownTest` seeds-then-restores `testing` env so CSRF-bypass still applies to its POST cases; `DemoLoginTest`'s `enterDemoMode()` re-seeds; `DashboardViewSeederTest`/`PageBuilderMediaListTest` per-block). Resolves the `[0.367.02]` inbox item.

### Phase 2 — Admin-editable CSP host allow-list (same-branch follow-on)

- The per-node analytics-host allow-list is now editable in **CMS Settings → Site → "Allowed External Hosts (Content Security Policy)"** (six fields: Scripts, Connections, Images, Frames, Styles, Fonts), instead of only `.env`. Stored as `csp_*_extra` SiteSettings, merged onto the env floor in `AppServiceProvider::boot` (piggybacks the existing single `SiteSetting::all()` query — no added per-request cost), read by the middleware from config.
- **`App\Rules\ValidCspHostList`** (new) constrains input to named `https://host` / `*.sub.host` entries and rejects the policy-weakening broadeners — bare `*`, quoted keywords (`'unsafe-inline'`/`'unsafe-eval'`/`'self'`), and scheme-only sources (`https:`/`data:`/`blob:`). The field can only widen to specific hosts, never punch a hole. The `.env` vars remain an optional fleet-wide floor. Verified end-to-end on the running stack (an admin-set host appears in the live `script-src`).

## Security considerations

This session *is* security work; the whole change is on the security axis. What was reviewed and decided:

- **The Fleet Manager contract surface was treated as untouchable.** The middleware skips `api/*`; the nginx headers live only in the static-asset location; the mTLS `if ($ssl_client_verify …)` blocks and the `/api/*` response envelopes are byte-identical (confirmed by an additive-only nginx diff and a test asserting `/api/health` carries no security headers). Contract stays **v2.6.0**.
- **CSP is a second layer, not the only one.** The public `script-src` is strict (nonce, no `'unsafe-inline'`); the accepted weakening is `style-src 'unsafe-inline'` (widget inline styles) — documented, not silent. The XSS *first* layer (save-time sanitization) is S3's audit; this session did not touch `HtmlSanitizer` or any write path.
- **The admin CSP is Report-Only** — an honest residual, not enforcement. It is on the Gate-3 findings register for a later promotion once Filament compatibility is validated in a browser.
- **The operator code-snippet feature vs. a strict `script-src`.** External-host analytics (GTM/GA) is restricted by the enforced public policy. Rather than weaken the policy, the friction moved to a validated per-node allow-list (admin UI + env floor) — the standing "security over usability, friction on the UX layer" rule. The allow-list is gated behind `manage_cms_settings`, the same trust level that already permits arbitrary `<script>` in the snippet fields, so it adds no new capability; validation prevents it from introducing a wildcard or unsafe keyword.
- **Supply chain narrowed:** the admin editor no longer executes third-party CDN script inside authenticated PII sessions.

## Production deployment notes

- **Headers, the self-hosted editor, and the settings page reach a node only on its next deploy.** The marketing and demo droplets get them at the next deploy cycle.
- **Check analytics after deploy.** Any node that currently runs Google Tag Manager / Analytics via the CMS code-snippet fields will have those external hosts blocked by the enforced public CSP until they're added under **CMS Settings → Site → Allowed External Hosts** (or the `SECURITY_CSP_*_EXTRA` env floor). Pre-launch there are no client installs relying on this, but the marketing node should be checked.
- **HSTS** is served by `prod.conf` at the TLS edge; nodes on plain HTTP (dev/e2e) never receive it.
- Standard `migrate` otherwise; nothing else special.
