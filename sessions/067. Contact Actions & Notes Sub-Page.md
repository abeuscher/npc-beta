# 067. Contact Actions & Notes Sub-Page

Two focused improvements to the contact edit page:

1. **Ellipsis menu consolidation** — all infrequent admin actions on a contact record are gathered into the ⋯ header menu. Portal-related controls currently buried in the form body are moved up here.
2. **Notes sub-page** — the NotesRelationManager is removed from the contact edit page (it causes layout jank as notes load) and replaced with a dedicated sub-page at `/contacts/{id}/notes`, accessible via a header action button.

---

## Architectural decisions

- **Portal access is independent of membership.** Granting portal access is a separate admin act from "Promote to Member." A member may never need portal access; a contact may need portal access before they become a member.
- **Ellipsis menu is the home for all infrequent contact actions.** Primary buttons (Save, Delete) remain in the header. Everything else goes in the ⋯ group. The portal section of the contact form becomes read-only status info only — controls move out of it.
- **Two portal invite paths:** (a) Send invite email — creates the portal account, sends the existing portal invite/verification email; user verifies and sets a password on first login. (b) Admin bypass — creates the portal account and marks email as verified immediately, no email sent. Both paths use the existing portal email and auth infrastructure.
- **Notes sub-page** is a standard Filament resource page within ContactResource (not a separate Resource). It renders a paginated, sortable table of the contact's notes. The relation manager is removed; the sub-page is the only notes UI on a contact from this session forward.
- **No new tables or migrations.** All changes are UI/routing only.

## Phase 1 — Ellipsis menu consolidation

**File:** `app/Filament/Resources/ContactResource/Pages/EditContact.php`

The existing `ActionGroup` already contains **Promote to Member**. Add the following actions to the same group:

### Grant Portal Access

- **Label:** `Grant Portal Access`
- **Hidden when:** `$this->record->portalAccount !== null`
- **Opens a modal** with:
  - **Portal email** (TextInput, email, required) — pre-filled with `$this->record->email` if set
  - **Send invite email** (Toggle, default true) — when on, sends the portal invite/verification email after account creation
- **On submit:**
  1. Create a `PortalAccount` record:
     - `contact_id` → `$this->record->id`
     - `email` → modal value
     - `password` → `Hash::make(Str::random(32))` (temp, discarded on first password set)
     - `is_active` → `true`
     - `email_verified_at` → `null` if send invite is on, `now()` if admin bypass
  2. If **Send invite email** is on: dispatch whatever existing portal email verification/invite mechanism is already in place (inspect the codebase to find it — do not invent a new one)
  3. Success notification: `Portal access granted.` (or `Portal access granted — invite sent.` when email was dispatched)

### Suspend portal access

- **Label:** `Suspend portal access`
- **Hidden when:** `$this->record->portalAccount === null || !$this->record->portalAccount->is_active`
- **Requires confirmation**
- **On confirm:** `$this->record->portalAccount->update(['is_active' => false])`
- Success notification: `Portal access suspended.`

### Restore portal access

- **Label:** `Restore portal access`
- **Hidden when:** `$this->record->portalAccount === null || $this->record->portalAccount->is_active`
- **Requires confirmation**
- **On confirm:** `$this->record->portalAccount->update(['is_active' => true])`
- Success notification: `Portal access restored.`

### Mark email verified

- **Label:** `Mark email verified`
- **Hidden when:** `$this->record->portalAccount === null || $this->record->portalAccount->email_verified_at !== null`
- **Requires confirmation**
- **On confirm:** `$this->record->portalAccount->update(['email_verified_at' => now()])`
- Success notification: `Email marked as verified.`

## Phase 2 — Clean up portal form section

**File:** `app/Filament/Resources/ContactResource.php`

The portal section of the contact form currently contains `Actions` components for Suspend, Restore, and Verify email. Remove those three `Action` components from the form section — the controls now live in the ellipsis menu. Leave the read-only `Placeholder` fields (Status, Portal Email, Account Created) intact so the form still shows current portal status at a glance.

## Phase 3 — Notes sub-page

**File (new):** `app/Filament/Resources/ContactResource/Pages/ContactNotes.php`

A `ListRecords`-style custom page (extend `Filament\Resources\Pages\Page` directly — not ListRecords, since this is not the resource's primary model) that:

- Is registered in `ContactResource::getPages()` under the key `'notes'` with route `/{record}/notes`
- Receives the contact record via the route parameter
- Renders a table of the contact's notes (from `$contact->notes()`) — columns: Note body (truncated to 100 chars), Author name, Occurred at (sortable, default desc)
- Has a **Create note** header action (inline modal form matching the existing `NotesRelationManager` form: body textarea, occurred_at datetime picker, hidden author_id)
- Has per-row Edit and Delete actions
- Has `getBreadcrumbs()` returning `[ContactResource::getUrl('edit', ['record' => $record]) => $record->display_name, 'Notes']`
- Has a **← Back to contact** header action (secondary, no icon needed) that redirects to `ContactResource::getUrl('edit', ['record' => $record])`

**File:** `app/Filament/Resources/ContactResource.php`

- Remove `NotesRelationManager` from `getRelations()`
- Register the new notes page in `getPages()`

**File:** `app/Filament/Resources/ContactResource/Pages/EditContact.php`

- Add a **Notes** header action (secondary button, icon `heroicon-o-document-text`) that links to the new notes sub-page URL. Place it to the left of the Delete button (i.e., first in the array).
