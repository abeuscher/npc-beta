# 235. Widget Primitive ŌĆö Phase 5e (Permissions in the Contract Layer)

Lift the per-arm permission gate from `resolveNote` / `resolveMembershipOne` / `resolveDonationList` into a declared `requiredPermission` field on `DataContract`, enforced uniformly at the resolver dispatch boundary. Closes the "Permissions in the contract layer" pending paradigm question that the track doc has carried since the start of Phase 5; brings permissions in line with the other declarative axes of the contract (source, fields, filters, cardinality).

---

## Design decisions (resolved before starting)

### Decision 1 ŌĆö Field shape on `DataContract`

Add a constructor parameter:

```php
public readonly ?string $requiredPermission = null,
```

Nullable, optional, default `null`. `null` is the **declared** "no permission gate" state ŌĆö public-facing contracts (BlogListing, EventsListing, Carousel, etc.) pass nothing or explicit `null`; gated contracts pass a Spatie permission name string.

Consistency with existing `DataContract` field shapes (`?string $model`, `?string $resourceHandle`, `?ContentType $contentType`, `?QuerySettings $querySettings` are all already optional-typed-with-null-default). The concrete-values rule applies to JSON-config field maps, not to constructor parameters.

### Decision 2 ŌĆö Enforcement point

Top of `ContractResolver::resolve()`, **before** the source-arm dispatch. The check runs once per contract per resolve call, regardless of source. When the gate fails, return the cardinality-appropriate empty shape:

- `CARDINALITY_MANY` ŌåÆ `['items' => []]`
- `CARDINALITY_ONE` ŌåÆ `['item' => null]`

The empty-shape determination already lives on the contract (`$contract->cardinality`) ŌĆö no new lookup needed.

### Decision 3 ŌĆö Auth check shape

`auth()->user()?->can($contract->requiredPermission)` ŌĆö identical to today's per-arm gates. Fail-closed when no user is authenticated (`?->can(...)` returns null ŌåÆ falsy ŌåÆ deny). Single-permission boolean check; **instance-bound `Gate::denies('view', $model)` checks are deliberately out of scope** (no current arm uses one; forward hook if a later arm needs it).

### Decision 4 ŌĆö Migration plan for the three existing inline gates

For each of `resolveNote` / `resolveMembershipOne` / `resolveDonationList`:

1. Delete the first-3-lines `if (! auth()->user()?->can(...))` block from the method body.
2. The arm's logic continues unchanged from there (ambient gate, scope, query, projector).

The corresponding widget-definition `dataContract()` method gains `requiredPermission: 'view_note'` (or `'view_membership'` / `'view_donation'`) on the `DataContract` constructor call.

### Decision 5 ŌĆö What stays per-arm

**Ambient-type gating** ŌĆö `! $record instanceof Contact` (membership, donation), `$record === null` (note) ŌĆö is a separate axis from permissions and stays inline at the head of each arm. A potential `requiredAmbientType: Contact::class` declaration is a forward hook, not in scope here.

### Decision 6 ŌĆö Source coverage

`requiredPermission` enforcement runs for all four source types (`SOURCE_PAGE_CONTEXT`, `SOURCE_RECORD_CONTEXT`, `SOURCE_SYSTEM_MODEL`, `SOURCE_WIDGET_CONTENT_TYPE`). The current consumers are all `SOURCE_SYSTEM_MODEL`; centralizing at dispatch level means future contracts on other sources inherit the mechanism for free.

## Phase 1 ŌĆö `requiredPermission` field + resolver enforcement

1. **`app/WidgetPrimitive/DataContract.php`** ŌĆö add `?string $requiredPermission = null` constructor parameter, with a docblock entry per the existing style. Declared after `$cardinality` and before `$formatHints` ŌĆö places permission alongside the structural axes.

2. **`app/WidgetPrimitive/ContractResolver.php`** ŌĆö at the top of `resolve()`'s per-contract loop body, before the `match` on `$contract->source`:

   ```php
   if ($contract->requiredPermission !== null && ! auth()->user()?->can($contract->requiredPermission)) {
       $results[$i] = $contract->cardinality === DataContract::CARDINALITY_ONE
           ? ['item' => null]
           : ['items' => []];
       continue;
   }
   ```

   No other resolver changes at this layer.

3. **Tests:**

   - `tests/Unit/WidgetPrimitive/ContractResolverPermissionGateTest.php` ŌĆö new file. Covers the centralized enforcement directly:
     - Returns `['items' => []]` for a `CARDINALITY_MANY` contract when user lacks the declared permission.
     - Returns `['item' => null]` for a `CARDINALITY_ONE` contract when user lacks the declared permission.
     - Returns expected results for a contract whose user **has** the declared permission (gate passes; arm runs normally).
     - Returns expected results for a contract with `requiredPermission: null` (no gate; arm runs normally).
     - Fail-closed when no user is authenticated (guest) and `requiredPermission` is set.

## Out of scope

- **Instance-bound permission checks** (`Gate::denies('view', $model)`). All three current arms use single-permission boolean checks; an instance-bound declaration grammar (e.g., `requiredAbility: ['view', $modelKeyFromAmbient]`) is a forward hook for a later session if a future arm needs it.
- **Ambient-type gate centralization** (`! $record instanceof Contact`). Separate axis from permissions; stays per-arm. Could become its own phase if/when more arms accumulate the same "must-be-Contact" predicate.
- **PageContext / WidgetContentType source contracts adopting `requiredPermission`.** The mechanism supports them (Decision 6) but no current public-facing widget needs it ŌĆö their public-by-construction stance stays intact. New gated contracts on these sources land as their concrete features arrive.
- **Spatie Policy classes.** This session keeps using `auth()->user()?->can('view_X')` (string-named permission). Wiring contracts to model policies (`$user->can('view', $donation)`) is the instance-bound forward hook above.
