# 161. Appearance Controls for Widgets — Docs Refresh & Shared Primitives

Phase 1 of the Appearance Controls project. Two pieces of groundwork land in this session:

1. Refresh the stale widget documentation (`resources/docs/widget-types.md` and `resources/docs/widget-development.md`) to reflect the Vue inspector architecture delivered in sessions 147–151. The current docs still describe a Livewire/Alpine inspector with `wire:model.live` bindings and a `PageBuilderInspector` Livewire class — none of that is true anymore.
2. Build the three shared primitive components that every upcoming Appearance panel depends on: a 9-point alignment toggle, a new color picker that reads the theme palette, and a gradient picker. Document the API for each so later sessions (and future widget authors) can consume them cleanly.

This session is groundwork only. **No Appearance panels are built and no widget config fields change.** Background, Text, and Section Layout panels land in session 163. Schema migration and widget dedupe land in session 162.

## Design decisions (resolved before starting)

### Scope of the docs refresh

`resources/docs/widget-types.md` and `resources/docs/widget-development.md` both describe the inspector as a Livewire/Alpine component with `wire:model.live` bindings, a `PageBuilderInspector` Livewire class, and the Advanced (Spacing) panel wired via that class. All of that is stale. The inspector migrated to Vue over sessions 147–151 and `app/Livewire/PageBuilderInspector.php` has been deleted.

**Rewrite both docs to reflect the current architecture:**

- The Vue app lives under `resources/js/page-builder-vue/` with Pinia state and a REST persistence layer (`PUT /widgets/{id}` with a 350ms debounce).
- The inspector is a Vue island mounted inside `resources/views/livewire/page-builder.blade.php`. The only remaining role of `app/Livewire/PageBuilder.php` is the add-widget modal, the save-as-template modal, and bootstrap data generation.
- Two-tab structure inside the widget inspector: **Content** and **Appearance**, driven by `InspectorTabs.vue`.
- `WidgetAppearanceControls.vue` exists today with full-width + background color + text color (these will be reorganised into the universal Background / Text / Layout panels in session 163, but the doc as written in this session reflects today's reality).
- `SpacingControl.vue` exists today with the all-sides shorthand + four individual padding/margin inputs. (Retired in session 163 — but this doc only describes today.)
- Field-type → component map lives in `InspectorField.vue` (11 user-facing types: text, textarea, richtext, number, toggle, color, select, image, video, url, buttons, plus `checkboxes` and `notice` as internal-only).
- `ColorPickerField.vue` is the current color picker — it is **replaced in this session** by the new ColorPicker primitive (see Phase 4).
- Widget composition (Blade templates, SCSS partials, `WidgetTypeSeeder`, `WidgetRenderer`, `AssetBuildService`, demo seeders, the `$pageContext` API) is unchanged from what those docs already describe and stays accurate. The render pipeline section of `widget-development.md` does not need a rewrite.

**Do not document the universal Appearance layer in this session.** The Background / Text / Section Layout panels are delivered in session 163, and a second documentation pass at the end of session 164 adds them. This session only removes lies and documents the three new primitives.

### Primitive 1 — NinePointAlignment

- New component at `resources/js/page-builder-vue/components/primitives/NinePointAlignment.vue`.
- Visual: a 3×3 grid of selectable points. Implement as a single inline SVG with nine target shapes (circles or squares) inside a bounding box. Compact — roughly 3rem square so it fits next to other inspector fields.
- **Value shape:** a string. Use these nine names: `top-left`, `top-center`, `top-right`, `middle-left`, `center`, `middle-right`, `bottom-left`, `bottom-center`, `bottom-right`. They map cleanly to CSS `background-position` values and to flex `align-items` / `justify-content` combinations.
- **Props:** `modelValue` (string, current selection, default `center`), `disabled` (bool — gray out with tooltip when set), `label` (optional string above the control).
- **Emits:** `update:modelValue`.
- **Keyboard:** arrow keys move the selection by one cell; Enter confirms (no-op for the bound value, but keeps focus consistent). Focus ring follows the selected point.
- **Accessibility:** the SVG is wrapped in a focusable element with an `aria-label` describing the current selection. Each point has a small invisible label for AT.

### Primitive 2 — ColorPicker (replaces ColorPickerField)

- New component at `resources/js/page-builder-vue/components/primitives/ColorPicker.vue`.
- **Replaces** the existing `resources/js/page-builder-vue/components/fields/ColorPickerField.vue`. The old file is deleted at the end of this session and all consumers are repointed.

**Shape:** a dropdown control. Collapsed state shows the current color swatch and hex value (or a "no color" indicator). Clicking opens a popover containing:

- **Theme palette row** — one swatch per palette color sourced from the active `Template`. Each swatch is labeled on hover. Clicking a swatch sets the stored value to that color's hex. Storage is hex, not a token reference (token-based storage is post-beta and requires a render-time resolver — out of scope here).
- **Reset / no-color** — a button with a ban icon. Clears the stored value to empty string.
- **Freeform picker disclosure** — a chevron that reveals the existing native HTML5 color wheel input plus a hex text input. Hidden by default; only users who need a non-palette color see it.
- **User swatches row** — the existing `SiteSetting` `editor_color_swatches` system continues to work. Display these alongside the theme palette but visually separated by a thin rule so users understand the difference between "theme colors" and "my saved colors".

**Props:**

- `modelValue` (string hex or empty)
- `label` (optional)
- `icon` (optional Vue component slot — this is how session 164's text-color variant gets a "T" icon without forking the primitive)

**Emits:** `update:modelValue`.

**Bootstrap data extension:** add `theme_palette` to the `BootstrapData` payload returned by `app/Livewire/PageBuilder.php::getBootstrapData()`. Source it from the active template via the existing `Template::resolved()` method (or whatever method exposes the resolved palette colors today — Phase 1 audit confirms). Shape: an array of `{ key, label, value }` objects, one per palette field. Mirror the addition in the TypeScript `BootstrapData` interface and expose it as a reactive getter on the Pinia editor store (`themePalette`).

### Primitive 3 — GradientPicker

- New component at `resources/js/page-builder-vue/components/primitives/GradientPicker.vue`.
- **Shape:** a panel that expands inline when the user clicks the current-gradient preview swatch. Layout inside the expanded panel: preview on the right, controls on the left.

**Controls (left side):**

- **Eight preset swatches.** Hard-coded in this session — choose eight sensible starting gradients (a mix of light pastels, dark inversions, and accent pairs covering common page builder use cases). The preset list is a `const` at the top of the component for easy future edits. Clicking a preset writes the full gradient value into the model.
- **Gradient 1 editor:**
  - Color stop "from" (using the new ColorPicker primitive)
  - Color stop "to" (using the new ColorPicker primitive)
  - Type selector: `linear` or `radial`
  - Optional angle input for `linear` (degrees, default 180)
  - CSS value override text input — when non-empty, takes precedence over the structured fields
- **+Add second gradient** button — enables a Gradient 2 editor with the same fields. The two gradients render as stacked CSS background layers (`background: gradient2, gradient1;`). Removable via a small × on Gradient 2.

**Value shape:**

```
{
  gradients: [
    { type: 'linear' | 'radial', from: '#hex', to: '#hex', angle?: 180, css_override?: '' },
    ...
  ]
}
```

When the array is empty or the value is null, the gradient is considered absent.

**Props:** `modelValue` (the gradient object or null), `label` (optional). **Emits:** `update:modelValue`.

**Composition helpers (built in this session, no consumer yet — that's intentional):**

- `resources/js/page-builder-vue/helpers/gradient.ts` exporting `composeGradientCss(value): string` — turns the value object into a valid CSS `background-image` string for client-side rendering and the editor preview.
- `app/Services/GradientComposer.php` with a matching `compose(array $value): string` method for the public renderer (which is PHP/Blade). Also a `blank(): string` helper that returns an empty string when the value is missing.

Both helpers are in place so session 163 can plug them in directly without re-deriving the composition logic.

## Security checklist

Before closing:

- [ ] Theme palette values exposed via bootstrap data are non-sensitive template colors only — no credentials, no tokens.
- [ ] `GradientComposer::compose()` sanitizes all inputs. The CSS override path refuses anything that isn't a recognised CSS gradient expression with hex colors. The structured path refuses non-hex colors, non-integer angles, and unknown gradient types.
- [ ] No raw user input is rendered into the editor preview without sanitization.
- [ ] No new routes, permissions, or database writes introduced.
