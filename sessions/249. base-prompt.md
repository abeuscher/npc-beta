## Process rules for every session

- **Before writing any code**, read every file you intend to modify.
- **Before implementing any integration**, ask the user what existing system handles it. The widget links to existing admin surfaces — do not build new admin pages for items that already have one. If an item lacks an admin surface (e.g., mail "from" address may live only in `.env`), surface and decide at session time whether the checklist item describes env-var status or whether a small admin surface lands as part of this session.
- **Pause and ask** any time a decision point arises that is not covered by the agreed prompt.
- **Surface architectural choices before going deep.** "Should I use X or Y here" is a question for the user, not something to commit to and unwind later. This includes data model shape, new abstractions, framework patterns that will propagate, and any decision that's expensive to reverse. Cheap and local — just decide.
- **If any external service is unavailable**, stop and ask — do not attempt to troubleshoot.
- **Run migrations via Docker** after writing them: `docker compose exec app php artisan migrate`. (One migration expected this session — adding `installation_completed_at` to `site_settings`.)
- **Update `docs/schema/`** after writing the migration — bump `docs/schema/site_settings.md` to reflect the new column.
- **If the PostgreSQL container becomes unresponsive**, ask the user to restart it.
- **When implementation is complete**, run the fast test suite: `docker compose exec app php -d memory_limit=2G vendor/bin/pest --exclude-group=slow`. Baseline at 248 close: 1734 fast tests / 9226 assertions. Expect the count to grow — new tests for checklist status detection, widget render, reset action.
- **Front-end builds — when to run which:**
  - The per-widget outline CSS change touches `resources/scss/**` — run `npm run build` after the SCSS edit.
  - No widget asset changes expected; `build:public` not needed unless something unexpected surfaces.
  - The widget itself is a Filament admin widget (Livewire), not a public-site widget — its blade template is rendered at request time, no build step.
- **Cross-repo coordination — Fleet Manager agent contract.** This session is **not** boundary-touching. The standing cross-repo flag is dormant: this session does not modify any file in `app/Http/Controllers/Api/Fleet/*`, `docker/nginx/{default,prod}.conf`'s mTLS sections, the response schema, or the VERSION pipeline. If during the session a change **would** touch any of those, stop and ask — the contract surface is locked at v2.0.0 until the next deliberately boundary-touching session.
- **Verify objective outcomes yourself; pull the user in for judgment.** When you can observe the result directly, do — don't ask the user to run it and report back. Close the loop yourself for CLI output, file contents, exit codes, logs, and anything Playwright can verify. Console errors, network responses, data round-trips — same idea. **Playwright is a verification mechanism, not just an existing test suite** — the widget renders in the browser, so a quick spec written for the moment to verify "checklist appears with N items, item-X status pill is green, deep-link routes to admin page Y" is a real verification path. Reach for it without being told. If the success criterion is objective, the user doesn't need to be in the loop. Pull them in for judgment on visual design (the per-widget outline shape), UX feel (does the checklist read as actionable?), and any architectural choice that wasn't explicit in the prompt.
- **Pause for manual testing only when human judgment is required.** The widget's rendering is partly visual — the per-widget outline change especially needs a human eye. After implementation, announce that visual testing is ready, then stop and wait. Do not suggest closing the session.
