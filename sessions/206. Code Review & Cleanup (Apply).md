## Design decisions (resolved before starting)

- **Every row below is a candidate, not a mandate.** Walk through the open flags and the W6 / W7 tables with the user at the start of the session and agree on the subset to apply. Items not chosen stay on the tables for a future cleanup session or are explicitly marked won't-fix.
- **Refactors that change public API shape** (JSON responses, serialized map shape in `ImportSource.*_field_map` columns, enum string values persisted on `import_sessions.model_type`) must be called out before starting so the user can weigh churn against payoff.
- **Refactors that change database storage shape are out of scope** — this is a refactor session, not a data migration session. The migration squash session (207) handles schema consolidation separately. If Flag A resolves to path (b) it gets its own carve-out below.
- **Tests for behaviour-changing extractions:** any iteration that introduces a new seam (service, enum, observer, base class) gets test coverage added or updated. Pure structural extractions with no behaviour change rely on the existing suite; note that explicitly in the commit message so the reviewer understands the thin-test-change intent.
- **Iteration strategy:** group related refactors into one iteration branch each (`session-206/1`, `session-206/2`, …) so the user can deploy, test, and merge in slices. Do not bundle unrelated extractions.

## Open flags from 205 (decide before implementation)

Walk through these first. Several are forks in the road — the iteration plan depends on the outcome.

### Flag A — LayoutInspectorPanel architectural divergence

`resources/js/page-builder-vue/components/LayoutInspectorPanel.vue` is a bespoke one-pane inspector with its own Background/Padding/Margin controls, while widget inspectors use the 6-panel shell (`InspectorPanel.vue` + `BackgroundPanel.vue` / `TextPanel.vue` / `SectionLayoutPanel.vue`). Layout appearance lives on `layout_config`; widget appearance on `appearance_config`. Consequences: gradients, background images, and the unified spacing grid are not reachable on a column layout; the two inspector trees drift independently.

Three possible paths:

- **(a) Widen the three universal appearance panels to take a polymorphic target (widget or layout).** `BackgroundPanel` / `TextPanel` / `SectionLayoutPanel` stop taking `widget: Widget` and start taking `target: Widget | Layout`. The store gains `updateLocalAppearanceConfig(targetId, path, value)` that branches on type. Dual contract in Vue; no DB migration.
- **(b) Give layouts a parallel `appearance_config` column.** Migration adds `page_layouts.appearance_config jsonb NOT NULL DEFAULT '{}'`; `InspectorPanel.vue` stops early-returning on `selectedLayout`; existing panels work unchanged. Requires a data migration to move the handful of `layout_config.background_color` / padding / margin values already in use to the new column. Single contract in Vue; schema change negotiated with the 207 squash schedule.
- **(c) Defer to a dedicated session.** The divergence is older than 205 and the scope (especially under (a)) is substantial. If 206's other work is already large, carve Flag A out as its own session.

Decision required: (a) / (b) / (c). Path (b) is the simpler Vue change but touches schema; path (a) preserves the schema but complicates the store and panel contract; path (c) is the conservative call.

### Flag B — ImportContactsPage convergence with the namespaced pattern

`app/Filament/Pages/ImportContactsPage.php` is ~976 LOC and predates the `InteractsWithImportWizard` trait. A retrofit would require: rewriting `processUploadedFile` to call the trait's `processUploadedFileNamespaced`, renaming the `__custom__` sentinel to `__custom_contact__` in DB-persisted maps on every `ImportSource`, converting the flat field grammar (`email`) to namespaced keys (`contact:email`), rewriting collision resolution as a trait hook, updating ~100 Pest tests and 4 Playwright specs.

**Decision:** leave as is for 206. This is a dedicated session's worth of work with high regression risk. Note in the 206 log that this was audit-confirmed out of scope; reevaluate in a future housekeeping pass.

### Flag C — Notes audit-trail-via-import gap

`ImporterPage::approve()` (lines 337-371) creates an activity-note audit entry only when the staged update's subject is a Contact. Notes staged updates apply via `fill()->save()` with no audit trail. Compounded by the fact that `Note` has no observer — Contact, Donation, Event, Membership, Page, Purchase all have observers; Note does not — so note CUD never appears in `activity_logs`.

Two paths:

- **(a) Defer to the stubbed "Notes Permissions & Permissions Audit" session** (already queued in `session-outlines.md`). That session is the natural home for both the `NoteObserver` addition and the approve-time audit question because both touch Note lifecycle observability. Drop W7 #2 from 206's docket.
- **(b) Include in 206 as a small iteration adding a `NoteObserver` that logs `created` / `updated` / `deleted` events on the notable subject.** Quick and low-risk (mirrors `ContactObserver`). The approve-time audit note question stays deferred to the notes-perm session regardless.

Decision required: (a) or (b). Default recommendation: (a) — keeps 206 focused and gives the notes-perm session a coherent scope.

### Flag D — Staged-update per-update ownership

Any user with `review_imports` can approve / rollback / delete an entire import session, implicitly acting on every staged update in it regardless of who created it. By design — sessions are the unit of review — but the absence of per-update ownership deserves an explicit decision-and-document.

**Decision:** docs-only update. Add a paragraph to `resources/docs/importer.md` (Review Queue section) and to `docs/schema/import_staged_updates.md` stating "staged updates are acted on at the session level; reviewers with `review_imports` permission can approve / rollback / delete the full session and every staged update inside it, regardless of who authored them." No code change.

### Flag E — Missing per-type importer help docs

`import-donations.md`, `import-events.md`, `import-memberships.md`, `import-invoice-details.md`, `import-history.md` do not exist. Shape question: one combined doc vs per-type stubs vs extending the existing `import-contacts.md` frontmatter.

**Decision:** extend `resources/docs/import-contacts.md` as the umbrella importer doc — add the five missing routes to its `routes:` frontmatter array and add a per-type section inside the doc body for each of the five types that are currently missing. Rename the doc title from "Import Contacts" to "Data Imports" (or similar) so it reads as the cross-cutting reference. Update `resources/docs/importer.md` to point at the umbrella doc rather than only linking out to Import Contacts. Content-writing iteration; no code changes.

### Flag F — SCSS export permission granularity

`ThemeTypographyController::export()` gates on `manage_cms_settings` — the same permission as its `update()`. Semantic concern: theme SCSS export is a theme operation, not a CMS-content operation.

**Decision:** leave as is, no new permission. `manage_cms_settings` is the de-facto theme permission today and adding a dedicated `edit_theme_scss` would require seeder + role-matrix updates with no practical benefit. Document the intent in a one-line comment on the controller method.

### Flag G — Playwright pre-seed contacts are email-only

Surfaced during 205's Playwright sweep: `tests/e2e/helpers/db.ts:192-203` (`insertContactsForImport(emails: string[])`) bulk-inserts `(id, email, source, created_at, updated_at)` rows with every other column NULL. Introduced in session 196 as a speed optimization replacing a wizard drive-through. Callers today: `events-happy-path.spec.ts:28`, `events-update-strategy.spec.ts:26`, `notes-happy-path.spec.ts:36`. Every contact pre-seeded this way lacks the prefix, first/last names, phone, address, city, state, postal_code the factory and `FakeCsvComposer::contactRow()` would otherwise produce — breaking the "consistent partial-fill that simulates real data collection" intent.

**Decision:** include as a 206 iteration. Replace `insertContactsForImport(emails: string[])` with `insertContactsFromCsv(csvPath: string)` — parse the generated `contacts.csv` (the same CSV `FakeCsvComposer::composeContacts()` already emits with the realistic partial-fill probability mix) and bulk-insert each row with all columns. Update the three callers. Scope: ~50 LOC in test fixtures; no production code, no schema, no migrations. Run `npm run test:e2e` after the change to confirm the importer match flow still resolves against the enriched rows.
