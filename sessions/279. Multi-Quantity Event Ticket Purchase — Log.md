## What was built

### Iteration /1 — Schema + controllers + webhook + per-tier withSum (045498f, on session-279/1)

Promoted the 278 tier-canonical registration shape from "one ticket per submission" into "N tickets across one or more tiers per submission." Schema (A) chosen at 278-close: one row per `(buyer, tier)` pair, each carrying its own quantity; per-tier capacity aggregates by `sum(quantity)`, not `count(rows)`.

- **Migration `2026_05_11_120000_add_quantity_to_event_registrations.php`** — `event_registrations.quantity smallint default 1`. The 25 existing rows in dev DB were backfilled to 1 by the column default at migrate time (verified via SELECT — 25 rows, 25 sum, 0 nulls).
- **Model + factory:** `EventRegistration::$fillable` adds `quantity`; `$casts` adds `quantity => 'integer'`. `EventRegistrationFactory::definition()` sets `quantity => 1` explicitly for clarity.
- **`TicketTier::isAtCapacity()`** — switched `count()` for `sum('quantity')` over the same status filter.
- **Resolver + projector:** `ContractResolver::resolveEventOne()` swapped `withCount(['registrations as registered_count' => …])` for `withSum(['registrations as registered_count' => …], 'quantity')`. `SystemModelProjector::projectTiers()` exposes a new `remaining_capacity` field (capacity − registered sum, or null when unlimited).
- **New service `App\Services\EventRegistrationQuantities`** — resolves the request's `quantities` map into a normalized list of `(tier, quantity)` lines, validates per-tier remaining capacity, computes `totalCents` and `isPaid`, and assembles Stripe `line_items` for the multi-line-item session. One service, four caller controllers.
- **Controllers (4):** `EventController::register`, `EventCheckoutController::store`, `Portal\EventRegistrationController::store`, `Portal\EventCheckoutController::store` all switched from `ticket_tier_id` single-value form input to the `quantities` map shape. Pre-beta — no backward-compat shim retained for the old shape.
- **Webhook fan-out:** `StripeWebhookController::handleEventRegistrationCheckout` now finds all pending registrations sharing the `stripe_session_id` and promotes them in one pass, recording one `Transaction` at the order total against the first row. Metadata discriminator switched from `event_registration_id` (single id) to `event_registration_checkout: '1'` (boolean flag); the webhook uses the Stripe session id directly to find rows.
- **Schema doc** — `docs/schema/event_registrations.md` updated with the new `quantity` column.
- **Tests:** new `tests/Feature/MultiQuantityRegistrationTest.php` (14 tests) covering the column persistence, per-tier `withSum` aggregation, free-path multi-tier rows, mixed-price routing, webhook fan-out, capacity bound, 999 hard ceiling. `TicketTierPublicPickerTest` rewritten for the `quantities` shape (+1 test for all-zero rejection). `PaidCheckoutTest` / `EventTest` / `FinancialSourceWritePathTest` retargeted from `ticket_tier_id` to `quantities`.

Iteration /1 also shipped the **widget UX** (originally framed in the prompt as "iteration /2") in the same commit because the widget template change is tightly coupled to the controller's input shape and splitting them would have left the system mid-broken between commits. The combined iteration /1 includes:

- **Widget template `app/Widgets/EventRegistration/template.blade.php`** rewritten with native `<input type="number">` per tier. Three rendering modes: 0 tiers (no quantity UI, single submit), 1 tier (one spinner defaulted to 1 — one-click registration), 2+ tiers (one spinner per tier defaulted to 0, plus a live subtotal beneath the spinners that sums each tier's `quantity × price` as the buyer adjusts). ### Iteration /2 — Bugfix: merge checkout controllers into entry controllers (525b157, on session-279/1)

User deployed iteration /1 to a real Stripe environment and reported "checkout page does not exist" coming from the registration widget on paid-tier purchases.

Diagnosis: `EventController::register` returned `redirect()->action([EventCheckoutController::class, 'store'], ['slug' => $slug])->withInput()` — a 302 Found. Browser follows 302 with GET. The `events.checkout` route is POST-only → 404. The bug was introduced at session 278 when the cross-controller dispatch pattern landed; the existing Pest test asserts the redirect *exists* but does not follow it, so the broken integration went uncaught.

Considered options:
- **(A) 307 the redirect** (preserve method + body). Smallest diff. Unusual flow.
- **(B) Merge the controller pairs** (inline Stripe assembly into entry controllers, delete the checkout controllers + routes). Cleaner architecture.
- **(C) JS-swap form action.** Brittle.

User chose (B). Implementation:

- Inlined the Stripe assembly into `EventController::register` and `Portal\EventRegistrationController::store`. The entry controller is now the single registration endpoint; it decides "land free" vs "create pending + Stripe session" inline.
- Deleted `app/Http/Controllers/EventCheckoutController.php` and `app/Http/Controllers/Portal/EventCheckoutController.php`.
- Removed the `events.checkout` and `portal.events.checkout` POST routes from `routes/web.php`.
- Tests retargeted: `PaidCheckoutTest` / `TicketTierPublicPickerTest` / `FinancialSourceWritePathTest` switched POSTs from `events.checkout` → `events.register` (and the portal equivalents). Two obsolete tests removed (free-tier-via-checkout-rejected, $0-only-via-checkout-rejected — both made no sense after the merge). Two regression-guard tests added to `MultiQuantityRegistrationTest`: "events.checkout and portal.events.checkout routes no longer exist" and "paid registration does not internally redirect to a removed events.checkout endpoint."
- Pre-beta — no backward-compat shim retained for the dropped routes.

Note: legacy `resources/views/events/show.blade.php` and `index.blade.php` reference `events.show` (route never existed) and `$event->price` (column dropped at 278) — they were already dead code; left for a separate cleanup pass.

## Security considerations

The dropped routes (`events.checkout` and `portal.events.checkout`) were the only callers exposing `EventCheckoutController`'s validation surface; nothing else routed to them. Removing them strictly narrows the public attack surface.

The `event_registration_checkout` metadata discriminator (replacing `event_registration_id`) is webhook-only; it does not influence which row gets promoted (the `stripe_session_id` lookup does). The shift away from "trust an id in metadata, look it up directly" toward "look up by the Stripe session id and verify pending status" is incrementally safer — Stripe's signature verification already gates the webhook entry, but the change removes a degree of metadata-trust.

Stripe accepts `unit_amount: 0` for `price_data` line items (mixed free + paid purchases include the free tier as a $0 line). The session's `amount_total` must be > 0 for `mode=payment`, which is naturally the case when any chosen tier has price > 0. No new Stripe-side validation gap.

The widget's per-tier spinner capping at `max=remaining-capacity` is a UX affordance, not a server-side guarantee — server re-validates against actual current registrations on submit, which is the canonical gate.
