## What was built

### Phase 1 — Install Swiper & frontend build

- Installed Swiper via npm. Imported only the required modules (Core, Navigation, Pagination, Autoplay, EffectFade) in `resources/js/public.js`.
- Imported corresponding Swiper CSS modules in `resources/scss/public.scss`.
- Vite build succeeds. JS gzipped: ~43 KB, CSS gzipped: ~14 KB.

### Phase 2 — Static picture widget

- New `image` widget type in `WidgetTypeSeeder` with config fields: `image` (image upload), `alt_text` (text), `object_fit` (select: cover/contain), `link_url` (url).
- Widget template at `resources/views/widgets/image.blade.php` renders the `<x-picture>` component from session 088. Optional link wrapper. Renders nothing if no image uploaded.
- Added static `options` support to `PageBuilderInspector::resolveSelectOptions()` so select fields can define inline options (not just `options_from` dynamic sources).

### Phase 3 — Carousel widget type

- New `carousel` widget type in `WidgetTypeSeeder` with primary config fields (`collection_handle`, `image_field`, `caption_template`, `object_fit`) and advanced fields behind a collapsible toggle (`autoplay`, `interval`, `loop`, `pagination`, `navigation`, `slides_per_view`, `effect`, `speed`, `link_color`, `text_color`).
- Generic inspector enhancement: config schema fields with `advanced: true` render inside a collapsible "Carousel Settings" section. Field rendering extracted to `resources/views/livewire/partials/inspector-field.blade.php` partial.
- Added `collections` data source to `PageBuilderDataSources` for the collection picker.
- Added `collection_fields:image` dynamic data source with `depends_on` support to `PageBuilderInspector` so the `image_field` select populates based on the selected collection's image-type fields.
- Changed select fields to `wire:model.live` for immediate server roundtrips (required for dependent field resolution).
- Added collection data resolution to `PageController::renderWidget()` — widgets declaring collections get resolved data passed as `$collectionData`.

### Phase 4 — Carousel template & rendering

- Widget template at `resources/views/widgets/carousel.blade.php` with Swiper initialization via Alpine.js `x-init`.
- Collection items rendered as Swiper slides with `<x-picture>` for images and token-replaced captions as overlays.
- Enhanced `WidgetDataResolver::resolveCustom()` to eager-load media on collection items and include a `_media` key mapping image field keys to Spatie Media objects.
- Carousel CSS in `_layout.scss`: 16:9 aspect ratio container via padding-bottom hack, absolutely positioned Swiper fills the box, caption overlay with gradient.
- ResizeObserver fallback in init to handle edge case where container has no dimensions at Alpine mount time.

### Phase 5 — Seed & test data

- `CarouselDemoSeeder` creates a `carousel-demo` collection with 4 sample items (title, description, image fields). Images uploaded manually during testing.
- Both widget types registered via `WidgetTypeSeeder`.

### CSS layers infrastructure

- Introduced `@layer base, libraries;` in `public.scss`. Pico CSS in the `base` layer, Swiper CSS in the `libraries` layer, custom styles unlayered (highest priority). This prevents third-party CSS collisions — Swiper styles automatically win over Pico without per-element overrides.
- Required switching from `@use` to `@import` for layered imports due to SCSS's restriction that `@use` must precede all other rules.

### Session prompts for Tailwind migration

- Wrote session prompts and base prompts for sessions 090 and 091 to migrate the public frontend from Pico CSS to Tailwind CSS, resolving the systemic CSS collision problem discovered during carousel development.

## Deferred decisions

- Pico CSS replacement with Tailwind CSS (sessions 090–091) — systemic collision between Pico's classless element styles and third-party library CSS (Swiper buttons, images, SVGs). CSS layers mitigate but don't fully resolve. Tailwind chosen for long-term compatibility.
- Swiper navigation arrow styling — currently affected by Pico button resets. Will be resolved by Tailwind migration.
- SCSS scoping strategy for widget CSS — `@layer` is the current solution. `@import` deprecation in Dart Sass is a known concern but not yet breaking.
