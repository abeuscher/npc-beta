# Fleet Manager — Requirements & Product Description

**Status:** Draft v2
**Author:** Al
**Context:** Companion operational tool for NonprofitCRM
**Last updated:** 2026-04-28 (session 237 close — added CRM-side starting state, two-repo coordination protocol, local development scheme, pre-Beta position; resolved several open questions to leans)

## Purpose

Fleet Manager is an internal operational tool for monitoring NonprofitCRM installations deployed across client droplets. Its role is to give a single operator (me) confidence that every client's system is healthy, backed up, and reachable — without requiring me to check each one manually.

It is not a product. It is not client-facing. It is not multi-tenant. It exists to prevent disasters and surface problems before clients find them.

## Scope

### In scope for v1

- Central registry of client installations
- Scheduled health polling of each install
- Dashboard showing current state across the fleet
- Historical health data with tiered retention
- Email alerts on state transitions
- Per-client operational notes
- Backup freshness tracking (reported by clients, read by Fleet Manager)
- SSL certificate expiry monitoring
- Version tracking per install
- A public system status page exposing aggregate uptime

### Explicitly out of scope for v1

- Remote action on client droplets (upgrades, restarts, SSH orchestration)
- Centralized log aggregation
- Performance metrics and graphing (CPU, memory, response times)
- Multi-user roles, permissions, audit logs
- Orchestrating or performing backups (the CRM does this; Fleet Manager observes)
- Direct verification of backup file integrity in object storage
- Client-facing views of any kind beyond the public status page

## Architecture Overview

### Deployment shape

- Standalone Laravel application on its own DigitalOcean droplet
- PostgreSQL for persistence
- Redis for queue backend
- Own failure domain — does not share a droplet with bundle server, thumbnailer, or any client install
- Same operational patterns as the CRM (nginx, deploy scripts, PHP version) to minimize cognitive overhead

### Stack (initial — mirror CRM versions to reduce cognitive overhead)

- **PHP:** `^8.2` (matches CRM)
- **Framework:** Laravel `^11`
- **Admin UI:** Filament `^3.2`
- **Database:** PostgreSQL 16 (16-alpine container image, matching CRM)
- **Cache / Queue:** Redis 7 (7-alpine container image, matching CRM); `predis/predis ^2.2` client
- **Queue worker / monitoring:** Laravel queue worker container (mirrors CRM's `worker` service); `laravel/horizon ^5` for queue dashboard
- **Scheduler:** Laravel scheduler (`schedule:run` in cron, run from inside the app container)
- **Auth:** Standard Laravel auth, single user to start; 2FA via Filament's built-in path if it lands cleanly, otherwise deferred
- **Mail:** `resend/resend-php ^1.1` (matches CRM)
- **HTTP client (poll outbound):** Laravel HTTP facade (Guzzle under the hood)
- **Container shape:** Docker compose with `app` (PHP-FPM), `nginx`, `postgres`, `redis`, `worker` services — same shape as CRM's `docker-compose.yml`. Container names prefixed `fleetmanager_` to mirror CRM's `nonprofitcrm_` convention.

### Communication model

- **Polling, not push.** Fleet Manager initiates all health checks on a schedule.
- Each client install exposes an authenticated health endpoint at `/api/health`.
- Auth is **bearer token in `Authorization` header**. Per-client API key, generated by the CRM operator (manual env-var on first deploy for v1; automation is post-v1).
- No inbound connections from clients to Fleet Manager in v1.

## CRM-side starting state

Important context for the Fleet Manager build — the CRM today does not have most of the surface FM expects to find:

- **No health endpoint exists.** Phase 1 of the Fleet Manager Agent track (CRM-side) is greenfield: a new `/api/health` route, bearer-token auth middleware, subcheck implementations, and version reporting all land in that session.
- **No backup mechanism exists.** Phase 2 is fully greenfield. `spatie/laravel-backup` is not installed; no `pg_dump` script exists; no media backup runs. The agent endpoint reports `last_backup_at` as `null` initially and Fleet Manager treats null as "unknown — don't alarm" until the backup pipeline lands.
- **No VERSION file exists.** Phase 1 either writes one at deploy time or reads `git rev-parse HEAD` cached at boot. The decision lands in the Phase 1 session.
- **CRM repo:** [`https://github.com/abeuscher/npc-beta`](https://github.com/abeuscher/npc-beta), primary branch `main`. The Fleet Manager repo is a separate repository — to be created.
- **CRM container shape (deploy server):** `nonprofitcrm_app` (PHP-FPM, where the agent endpoint runs), `nonprofitcrm_postgres` (PostgreSQL 16-alpine), `nonprofitcrm_redis` (Redis 7-alpine), `nonprofitcrm_worker` (queue worker), `nonprofitcrm_nginx`. App path: `/opt/nonprofitcrm`. Compose: `/opt/nonprofitcrm/docker-compose.prod.yml`.
- **CRM internal network:** Docker bridge named `nonprofitcrm` (project-prefixed at runtime — likely `nonprofitcrm_nonprofitcrm` after Compose normalization). Nginx publishes port 80 to the host; the app container is reachable internally only.
- **CRM transactional email:** `resend/resend-php`. Resend account is **shared with FM during dev and pre-launch** (single account, lower op cost). When client onboarding starts and the CRM's Resend becomes a passthrough expense, FM moves to its own Resend account for clean failure-domain isolation.

## Data Model (initial sketch)

Names and exact fields to be refined in the FM-side planning session, but the baseline entities:

### Clients (install registry)
- Display name, slug
- Primary domain, droplet IP/hostname
- API key for health polling (encrypted at rest)
- Current deployed version
- Install date, contact info
- Free-text notes field (timestamped entries)
- Active / archived status

### Health checks
- Foreign key to client
- Timestamp
- Overall status (green / yellow / red)
- Subchecks — app responsive, database reachable, Redis reachable, disk usage, backup freshness, SSL expiry
- Raw response payload

### Incidents
- Derived from health check transitions
- Start time, end time, duration
- Client, severity, summary
- Used to drive alerting and the status page

### Alerts
- Sent notifications log
- For audit and to prevent duplicate paging during ongoing incidents

## Polling & Health Checks

### Cadence
- Scheduler dispatches a "poll all active clients" job every minute (tunable)
- Each client poll is its own queued job so a slow client doesn't block others
- Failed polls retry with backoff; N consecutive failures = red

### What gets checked
Collected from the client's authenticated health endpoint:
- Application responding to HTTP
- Database connection live (`DB::connection()->getPdo()`)
- Redis connection live (`Redis::ping()`)
- Disk usage percentage on droplet (`disk_free_space('/') / disk_total_space('/')`)
- Timestamp of last successful backup (read from a known location written by the backup pipeline; `null` until Phase 2 lands)
- SSL certificate expiry date — **checked externally by Fleet Manager**, not reported by the CRM (more honest about reachability; handles the case where the CRM cannot read its own cert)
- Current application version (from VERSION file written at deploy time, fallback to git SHA)
- Any client-defined custom checks (future)

### Status interpretation
- **Green:** all subchecks passing
- **Yellow:** a subcheck is in a warning threshold (e.g., disk >80%, SSL <30 days, backup age 24–36h)
- **Red:** a subcheck is failing (e.g., DB unreachable, backup >36h, SSL <7 days, poll itself failed N times)

Thresholds configurable per-client to accommodate unusual situations.

## Alerting

v1 is deliberately simple:

- Any transition to red triggers an email alert
- Any transition from red back to green triggers a resolution email
- One alert per incident — do not re-send while the incident is ongoing
- Email to the operator address; no SMS, Pushover, or other channels in v1

The system is designed to scale alert sophistication later (quiet hours, severity routing, multiple channels) but none of that is built now.

## Data Retention

Health check history follows tiered retention:

- **Full resolution for 30 days:** every individual health check preserved. Supports detailed forensics of recent incidents.
- **Monthly rollups for 3 years:** after 30 days, individual checks are aggregated into a single monthly summary per client (uptime percentage, incident count, notable events). Individual records purged.
- **Permanent:** incident records (not individual checks), client registry, notes.

Implementation detail deferred to the FM-side planning session, but the shape matters now: Fleet Manager must be able to answer "what happened last Tuesday" for 30 days and "how was Client X doing in Q2 last year" indefinitely.

## Public Status Page

A read-only status page exposed at a public URL showing aggregate fleet health — not per-client detail unless individual clients opt in.

Logically lives within Fleet Manager since it has the data. Implementation deferred. Domain choice deferred — production domain stuff is unsettled at planning time; dev uses a subdomain placeholder; production lands when the broader product domain shape is decided.

Open question for later: whether individual clients appear by name on a public page, or only aggregate metrics. Leaning aggregate-only for v1.

## Security Posture

- Single operator account; standard Laravel auth with a strong password and 2FA if Filament supports it cleanly
- Per-client API keys stored encrypted at rest
- Fleet Manager can read status from clients but cannot take action on them in v1
- Compromise of Fleet Manager should not compromise client data — it holds no database credentials, no SSH keys, no backups
- HTTPS-only, standard hardening

## Capsize Testing Before Launch

Before declaring v1 ready:

- Destroy a client install and confirm Fleet Manager alerts correctly
- Simulate database-down, Redis-down, disk-full, and backup-stale states and confirm each is detected
- Attempt to hit a client's health endpoint without a valid API key and confirm rejection
- Attempt to compromise Fleet Manager itself and confirm blast radius is limited

## Two-Repo Coordination Protocol

The Fleet Manager build runs in a separate repository from the CRM. Two parallel agentic workstreams operating across two repos can quietly diverge on shared interfaces, because each repo's session prompts and roadmap are read in isolation. Without explicit coordination artifacts, the agent contract drifts and incidents emerge weeks later. This section names the protocol that prevents drift.

### Spec doc — the single source of truth

- **Location:** CRM repo at [`docs/fleet-manager-agent-contract.md`](../docs/fleet-manager-agent-contract.md). The CRM emits the contract; the FM repo consumes it. Locality of policy-to-surface — the side that defines the surface owns the doc.
- **Contents:** endpoint path, auth handshake, request schema, response schema (subcheck names + value shapes), version negotiation, error envelope, CHANGELOG.
- **Versioned:** the doc carries a top-of-file `Contract Version: X.Y.Z` field (semver). Bumps are explicit and CHANGELOG-recorded.
- **Initial state:** v0.0.0 stub at session 237 close, naming what the file will become. First real version (v1.0.0) lands in the FM Agent Phase 1 session when the actual contract is authored.

### Cross-repo access pattern

- **FM repo accesses the spec doc via WebFetch** against the CRM repo's `main`-branch raw URL: `https://raw.githubusercontent.com/abeuscher/npc-beta/main/docs/fleet-manager-agent-contract.md`
- The FM repo also keeps a local cached copy (e.g., at `docs/imported/fleet-manager-agent-contract.md`) as a session bootstrap baseline. On every boundary-touching session in the FM repo, the agent fetches the canonical version, diffs against the local copy, and flags drift if found.
- Submodule was considered and rejected — the two products are not coupled tightly enough to justify the release-friction submodules introduce.
- Manual copy-and-sync was considered and rejected — drift is inevitable without a verification step.

### Cross-Repo status block

Both repos carry a "Cross-Repo: Fleet Manager / CRM" subsection at the top of `sessions/session-outlines.md` (just below the Active tracks block). The block states:

- Current Contract Version
- Last boundary-touching session in *this* repo
- Last boundary-touching session in the *other* repo
- Pending boundary changes (planned but not yet shipped)
- Pointer to the spec doc

The block is updated as part of any session that touches the boundary, in either repo.

### Boundary-touching session marker

- **In the CRM repo:** session titles starting with `Fleet Manager Agent — *` are boundary-touching. The base prompt template enforces a "read the agent-contract spec doc before doing any work" pre-implementation step on these sessions.
- **In the FM repo:** session titles starting with `CRM Integration — *` are boundary-touching. Same enforcement.
- Each marked session's close-gate updates the agent-contract spec doc (if anything changed), bumps the Contract Version (if anything changed), and updates the Cross-Repo block in *both* repos before the next boundary-touching session in either.

### Prompt template flag (passive cross-cutting check)

A standing rule in **every** session base prompt (both repos), regardless of session title:

> If your work modifies any file or surface that participates in the Fleet Manager agent contract — the `/api/health` endpoint, its auth middleware, its response schema, the VERSION file at deploy time, anything in `app/Http/Controllers/Api/Fleet/*` (CRM-side) or any HTTP client / contract validator (FM-side) — pause, update `docs/fleet-manager-agent-contract.md`, bump its Contract Version, and update the Cross-Repo block in `sessions/session-outlines.md` before continuing.

The rule is in place from session 237 forward; it is dormant until the agent surface exists, and self-activates the moment it does.

### Version-bump checklist

When the Contract Version bumps:

1. Update `docs/fleet-manager-agent-contract.md` (spec body + CHANGELOG section + Contract Version field).
2. Update the Cross-Repo block in this repo's `sessions/session-outlines.md` (current version + last boundary-touching session in this repo).
3. Update the Cross-Repo block in the *other* repo's `sessions/session-outlines.md` before that repo runs another boundary-touching session.
4. The boundary-touching session's log explicitly names the version bump.

### First-write asymmetry

The CRM emits the contract; the FM repo consumes it. So:
- The spec doc lives in the CRM repo.
- The FM repo treats the spec doc as a **read-only dependency**.
- All boundary-defining changes originate from CRM-side sessions; FM-side sessions react.

If the FM repo ever needs to express a contract requirement (e.g., "FM needs a new subcheck the CRM doesn't currently expose"), the request lands as a CRM-side issue — the CRM authors the contract change, then FM consumes the new version.

## Local Development Scheme

To make the two-repo workstream practical day-to-day, both apps need to run locally and reach each other.

### Layout

- **CRM** runs in its own Docker compose stack at `~/nonprofitcrm/`. Already configured: nginx publishes port 80 on the host, postgres on 5432, redis on 6379. App container is reachable internally on the `nonprofitcrm` bridge network only.
- **Fleet Manager** runs in its own Docker compose stack at a sibling path (e.g., `~/fleetmanager/`). Mirrors CRM's compose shape: `app`, `nginx`, `postgres`, `redis`, `worker` services. FM's nginx publishes port `8001` on the host (CRM has 80; FM avoids the collision). FM's postgres publishes 5433, redis 6380.

### Inter-stack reachability — host-bridge pattern (lean)

- FM's app/worker containers reach the CRM at `http://host.docker.internal/api/health` (the CRM's nginx is on host port 80).
- WSL2 Linux requires an explicit `extra_hosts` directive on the FM service definitions:
  ```yaml
  services:
    app:
      extra_hosts:
        - "host.docker.internal:host-gateway"
    worker:
      extra_hosts:
        - "host.docker.internal:host-gateway"
  ```
- The CRM does not need any change. The FM compose owns the wiring.

### Alternative — shared external Docker network

If the host-bridge pattern proves fragile (e.g., port conflicts, network changes, WSL2 Docker quirks), the alternative is to make both stacks join the same Docker network:

- CRM's `nonprofitcrm` network is declared `external: true` (CRM-side compose change required).
- FM's compose declares `nonprofitcrm` as an external network and joins it.
- FM containers reach the CRM at `http://nonprofitcrm_nginx/` (DNS-resolvable inside the shared bridge).

The host-bridge pattern is preferred because it requires zero CRM-side change. The shared-network pattern is the fallback.

### Bearer token for local dev

- The CRM generates a test API key on first run (an artisan command, e.g., `php artisan fleet:agent-key:generate`, written to `.env` as `FLEET_MANAGER_AGENT_KEY`).
- The FM operator copies the key into FM's `.env` as `CRM_AGENT_API_KEY` (or per-client, into FM's clients table at registry time).
- Manual copy is fine for dev. Automation (e.g., a bootstrap pairing flow) is post-v1.

### Smoke-test wiring

- FM ships with `bin/check-crm-local.sh` (or the Laravel equivalent — `php artisan fleet:smoke-test http://host.docker.internal/api/health`) that hits the CRM's local agent endpoint with the bearer token and prints the response. First thing the operator runs after setting up both stacks side by side.

## Pre-Beta Position

Both Fleet Manager Agent phases sit in **pre-Beta-1 scope** in the CRM roadmap (`sessions/session-outlines.md`):

- **Phase 1 — CRM-Side MVP + Two-Repo Coordination Protocol.** Lands the CRM-side health endpoint and the protocol artifacts named above. Unblocked as of session 237's close (the Widget Primitive track was the prior big architectural beat; it's substantially complete).
- **Phase 2 — Backup Pipeline.** Greenfield CRM-side backup (pg_dump + media → DigitalOcean Spaces, scheduler cadence, success-record location). Lands when a second client install is in flight or when Fleet Manager v1 is approaching production launch and `last_backup_at = null` becomes operationally annoying. Not blocking on Phase 1 of FM itself — FM v1 ships with `last_backup_at = null` interpreted as "unknown."

The Fleet Manager repo build runs in parallel with these CRM-side phases. The two workstreams stay decoupled except at the contract boundary, which the protocol above governs.

## Resolved leans (open to revisit in the FM-side Phase 1 planning session)

These were Discussion Points in v1 of this spec. The lean is named so both workstreams have a starting point; the planning session can override.

1. **Auth scheme for the agent endpoint.** Bearer token in `Authorization` header. Simpler than signed-request; meets v1 needs.
2. **Version reporting mechanism.** `APP_VERSION` Docker build-arg sourced from `${GITHUB_SHA::7}` in GitHub Actions (`.github/workflows/deploy.yml`); written to `/var/cache/app/VERSION` inside the image (outside the local bind-mounted tree, so local dev's `.:/var/www/html` mount doesn't shadow it). Read at boot by Laravel into a cached config value. Local dev defaults the build-arg to `dev`. `docker-compose.prod.yml` pulls pre-built images from GHCR — no change needed there; the version is baked into the image at CI build time. Local `docker-compose.yml` wires the build-arg in its `build:` block (also defaults to `dev`).
3. **SSL expiry check location.** Fleet Manager checks externally. More honest about reachability than CRM-side self-check.
4. **API key bootstrapping.** Manual env-var on first deploy. Automation deferred.
5. **Spec doc location.** CRM repo at `docs/fleet-manager-agent-contract.md`. CRM-emits, FM-consumes.
6. **Local dev inter-stack reachability.** Host-bridge via `host.docker.internal` from FM into CRM's host-published port 80. Shared-network as fallback.
7. **Resend account separation.** Shared during dev / pre-launch; FM separates onto its own Resend account when client onboarding makes the CRM's Resend a passthrough expense.

## Open questions for the FM-side Phase 1 planning session

These genuinely depend on specifics resolved best with code in front of us:

1. **Backup implementation details.** How the CRM actually performs backups — `pg_dump` invocation, compression, encryption at rest, object storage credentials model, retention policy on the client side, how success is recorded for Fleet Manager to observe. The storage target (DO Spaces) is decided but the mechanics are not. Belongs to FM Agent Phase 2.
2. **Retention implementation.** How the FM-side tiered retention is actually enforced — scheduled job cadence, rollup query shape, whether rollups happen in Postgres directly or in application code. FM-side concern only.
3. **Database schema specifics.** Field names, indexes, whether `health_checks` is partitioned by time given retention patterns, how rolled-up data is stored. FM-side concern only.

## v1 Success Criteria

Fleet Manager v1 is done when:

- Every client install is in the registry and being polled
- The dashboard shows current state at a glance
- I get email alerts within a few minutes of a real problem
- Historical data is queryable for at least 30 days
- I have personally tested the alerting by breaking things on purpose
- The public status page is live

## Deferred to Later Versions

Captured here so they don't get lost but don't distract from v1:

- Upgrade orchestration (push a new CRM version to one or all clients)
- Automated backup restoration testing against a scratch environment
- Backup integrity verification (read the file from object storage, check structure)
- Cross-provider backup redundancy (Backblaze B2 as secondary)
- Performance metrics and graphing
- Multi-channel alerting (SMS, Pushover, etc.)
- Alert sophistication (quiet hours, severity, grouping)
- Per-client opt-in to public status page with individual naming
- Additional operators with role separation
