# 038. MailChimp Webhook Debugging

Debug and fix the MailChimp unsubscribe webhook. The sync (CRM → MailChimp) works correctly. The webhook (MailChimp → CRM) is returning HTTP 500 for every real request from MailChimp's servers, while the same request simulated via curl from the production server returns 200 OK.

**There are no consequences to failure here. The production server has no real users and all contact data is test data.**

---

## What was built in session 037

- `MailChimpService` — syncs a mailing list's contactable members to a MailChimp audience via the batch API, tagging each member with the list name
- `MailChimpWebhookController` — receives unsubscribe events from MailChimp and sets `mailing_list_opt_in = false` on the matching contact
- `MailingListMembersWidget` — table widget on the mailing list edit page showing current members
- MailChimp credentials (API key, server prefix, audience ID, webhook path, webhook secret) stored in the admin Mail Settings page (DB-backed, same pattern as Resend)
- Webhook route excluded from CSRF at `/webhooks/*` in `bootstrap/app.php`

## What works

- **Sync**: hitting "Sync to MailChimp" on a mailing list pushes contacts to MailChimp correctly. Verified via MailChimp batch API response.
- **Webhook GET** (MailChimp validation ping): returns 200 OK correctly.
- **Webhook POST via curl from the server**: returns 200 OK and updates the contact record. Tested with the exact MailChimp payload format including all fields (`type`, `data` as JSON string, `fired_at`, `merges`, etc.) and `User-Agent: MailChimp`.
- **SSL**: cert is valid, nginx serves HTTPS correctly, MailChimp reaches the endpoint.

---

## What fails

Every real POST from MailChimp's servers returns **HTTP 500** with a **6628-byte response body** (consistent size every time). MailChimp retries the failed webhook multiple times in rapid succession.

### Nginx log pattern
```
3.22.143.129 - - "POST /webhooks/mailchimp?secret=... HTTP/1.1" 500 6628 "MailChimp"
3.134.222.173 - - "POST /webhooks/mailchimp?secret=... HTTP/1.1" 500 6628 "MailChimp"
```
MailChimp IPs observed: `3.19.249.238`, `3.134.222.173`, `3.22.143.129` (all AWS us-east-2).

### What is NOT the cause (ruled out)

- **Secret mismatch** — nginx log shows the correct secret in the URL; a wrong secret would return 403, not 500
- **CSRF** — excluded at `bootstrap/app.php` for `/webhooks/*`; wrong CSRF would return 419
- **Data format** — MailChimp sends `data` as a JSON string; the controller correctly uses `json_decode($request->input('data', '{}'), true)`. Curl simulation with identical format returns 200.
- **Server state / cold start** — 500s occur consistently across multiple deploys and restarts, not just at startup
- **Concurrency** — the very first MailChimp request also fails; it's not just retry storms overwhelming PHP-FPM
- **Laravel error log** — no entries appear in `storage/logs/laravel.log` for any of the 500s, even with `APP_DEBUG=true`
- **PHP-FPM error log** — `docker logs nonprofitcrm_app 2>&1 | grep -v GET | grep -v POST | grep -v HEAD` shows only FPM startup notices, no PHP errors

### The key mystery

The 500 is not being logged by either Laravel or PHP-FPM. This suggests either:
1. The error happens before Laravel's error handler initialises (e.g. in a service provider or autoloader)
2. Something at the nginx/FastCGI layer is generating the 500 before PHP even runs
3. There is a difference between MailChimp's actual request and the curl simulation that hasn't been identified yet

## Suggested debugging paths

1. **Trigger a fresh MailChimp unsubscribe and check the log** — if the `Log::error('MailChimp webhook received')` entry appears, Laravel is reached and the error is in the controller logic. If not, the 500 is pre-Laravel.

2. **If pre-Laravel**: check nginx error log (`docker logs nonprofitcrm_nginx 2>&1 | grep -i error`) and PHP-FPM config (`docker exec nonprofitcrm_app cat /usr/local/etc/php-fpm.d/www.conf`). Consider whether a FastCGI parameter or upstream timeout is causing nginx to return a synthetic 500.

3. **Capture the actual 500 response body**: use a proxy or temporary nginx `mirror` directive to capture MailChimp's request as-is and replay it, or temporarily log the full response body somewhere accessible.

4. **Compare headers precisely**: MailChimp may be sending a header that curl isn't — e.g. `Accept-Encoding: gzip`, `Expect: 100-continue`, or a non-standard `Content-Type` variant. Use tcpdump on port 9000 (PHP-FPM, unencrypted) to capture MailChimp's actual FastCGI params vs curl's.

```bash
tcpdump -i any -A -n 'port 9000' 2>&1 | head -200
```
Run this, then trigger a MailChimp unsubscribe, then do the same for a curl simulation — compare the output.
