# 232. Widget Primitive — Phase 5d-2 (Membership Status — Single-Row Record-Detail Widget)

Land the second concrete record-detail widget: **Membership Status for the current contact**. Single-row contract (cardinality: 'one'), bound to the ambient Contact via the record-detail Slot. Mirror of 5d-1's shape — contract design + projector + per-arm permission gate + test-lock — with the single-row DTO pattern from 4g instead of the list pattern. Lands the second piece of evidence for the per-arm `Gate::denies` posture; full Phase 5e architecture stays carved out.

---

## Design decisions (resolved before starting)

- **`SOURCE_SYSTEM_MODEL` with `model: 'membership'`, `cardinality: 'one'`.** Membership is an Eloquent model; the contract source is `SOURCE_SYSTEM_MODEL`, the resolver gains a `membership` arm dispatched from the cardinality-one branch, the projector gains a `projectMembership()` method. Mirror of 4g's EventDescription/EventRegistration retrofit — same single-row DTO shape (`['item' => row | null]`), same `cardinality: 'one'` declaration.
- **Ambient record id flows into the resolver via `SlotContext`.** The widget's contract declares `model: 'membership'`, `fields: [...]`, `cardinality: 'one'` — but **does not** declare a per-instance contact filter. The resolver, when running a `membership` contract under a record-detail slot, reads the ambient record from `SlotContext::ambient()` and applies `where('contact_id', $contact->getKey())->where('status', 'active')->orderBy('starts_on', 'desc')->first()` automatically. **Same shape as the note arm in 5d-1.** The model already has `Contact::activeMembership()` with this exact query — the resolver inlines the equivalent rather than calling the model helper, so the DTO derivation stays in the projector pipeline rather than crossing through model methods.
- **Permission gate at the resolver, fail-closed.** Before the resolver dispatches a `membership` contract, it checks `auth()->user()?->can('view_membership')`. If false, the resolver short-circuits and returns `['item' => null]` — the widget renders the empty state. This is the second use of the per-arm gate pattern locked in 5d-1; the third widget (Recent Donations at 5d-3) makes three evidence points, at which point the case for centralizing into Phase 5e becomes evaluable. Until then, per-arm.
- **Projector field-naming follows the 226 concept-named convention.** Display fields are named after concepts the template will read, not after raw column names. Initial whitelist:
  - `membership_tier_name` (from `tier->name`; empty string when tier is missing)
  - `membership_billing_interval` (raw enum: `monthly` / `annual` / `one_time` / `lifetime`)
  - `membership_status` (raw enum: `active` here by definition since the resolver filters)
  - `membership_starts_on` (formatted via `DateFormat::format($m->starts_on, DateFormat::MEDIUM_DATE)`; empty string when null)
  - `membership_expires_on` (formatted same way; empty string when null — lifetime tiers are explicitly null per schema)
  - `membership_amount_paid` (raw decimal string from the column, e.g. `"50.00"`; empty string when null)
  - `membership_id` (uuid, for any future deep-link affordance)
- **Cardinality: one.** Single-row DTO. The contract declares `cardinality: 'one'`. The DTO shape is `['item' => array<string, mixed> | null]`. Empty state: "No active membership."
- **No projector-side label assembly in this session.** EventDescription's 4g pattern derived `event_time` and `event_location` from multiple raw columns inside the projector (template-side assembly retired). For Membership Status the equivalent would be a derived `membership_label` like "Annual Member" combining tier name + billing interval, plus a `membership_renewal` like "Renews May 2026" or "Lifetime." **Defer this.** Ship raw fields only; if the template's branching turns out to want assembly, add a follow-up session for it. The 226 pattern proves the affordance is available when needed; don't pre-design here.
- **No widget assets (CSS/JS).** Server-rendered Blade widget that uses only the existing public typography, mirror of RecentNotes. No `assets()` declaration, no SCSS partial. `npm run build` not required (no Vue source touched). `build:public` not required (no widget assets declared).
- **Seed Membership Status into `contact_overview`.** The seeded primary View on Contact already carries Recent Notes (from 5d-1). 5d-2 appends Membership Status as the second widget on the same View, sort_order 1. **The sub-nav threshold cleanup landed in 231/2 means this does not trigger sub-nav** — both widgets render in EditContact's footer alongside the form, no chrome added. The seeder's "widgets already exist" early-return semantic from 5d-1 needs adjusting: it currently short-circuits if *any* widget exists on the View, which would skip seeding Membership Status on installs that already have Recent Notes. Change: the seeder checks per-handle, not per-row-count — Recent Notes seeded → check; Membership Status seeded → check; if missing, seed.
- **Test-lock mirrors 5d-1.** Fail-closed field whitelist (a contract requesting `notes`, `stripe_session_id`, `external_id`, `custom_fields` returns empty strings for those keys — even though `notes` and `custom_fields` are real columns on the model, they're not in the projector's map). Resolver-is-only-path query-count pin (rendering the widget hits the resolver exactly once, plus one tier eager-load). Ambient-record scoping pin (widget configured under contact A renders A's membership, not B's). Permission-gate pin (cms_editor without `view_membership` gets `['item' => null]`). Single-row-shape pin (`['item' => row | null]`, never `['items' => ...]`).
- **Active-only by design.** The resolver filters to `where('status', 'active')`. Pending, expired, cancelled memberships do not appear. If the contact has no active membership (zero rows match), the resolver returns `['item' => null]` and the widget renders the empty state. Showing the latest expired membership as "your last membership" is explicitly out of scope — that's a different semantic (history) and would shift the cardinality and template branches.

## Phase 1 — Resolver `membership` arm + projector + ambient-record scoping

1. Update `app/WidgetPrimitive/ContractResolver.php`:
   - Add a `'membership'` case to `resolveSystemModel()`'s **cardinality-one** match (the single-row branch). Mirror shape of `'event' => $this->resolveEventOne(...)`. The match's cardinality-many branch does not gain a `'membership'` arm — list-shaped membership reads are not in scope this session.
   - New private `resolveMembershipOne(DataContract $contract, array &$cache, SlotContext $context): array` method. Permission gate: `if (! auth()->user()?->can('view_membership')) { return ['item' => null]; }`. Reads ambient record from `$context->ambient()`; if not a `RecordDetailAmbientContext` or the record is not a `Contact` instance, return `['item' => null]`. Otherwise: `Membership::query()->where('contact_id', $contact->getKey())->where('status', 'active')->orderBy('starts_on', 'desc')->with('tier')->first()`. Cache key: `'membership:one:' . $contact->getKey()`.
   - Pass through `$this->systemModelProjector->projectOne($contract, $cache[$key])` to convert the model into the single-row DTO. The projector's existing fail-closed field filtering applies.
2. Update `app/WidgetPrimitive/Projectors/SystemModelProjector.php`:
   - Add `'membership'` to `projectorFor()`'s match. Shape: `'membership' => fn (Membership $m) => $this->projectMembership($m)` (matching the existing `'note' => fn (Note $n) => ...` shape).
   - New private `projectMembership(Membership $membership): array` method. Returns the field map per the design decisions above. Use `$membership->tier?->name ?? ''` for `membership_tier_name`. Use `$membership->tier?->billing_interval ?? ''` for `membership_billing_interval`. Cast `amount_paid` to string (it's a decimal:2 cast already, so `(string) $membership->amount_paid` produces `"50.00"` shape). Format dates via `DateFormat::format()` with the model's date attribute (no manual Carbon parsing required — the model already casts `starts_on` and `expires_on` to date).
3. **Tests** at `tests/Unit/WidgetPrimitive/ContractResolverMembershipTest.php` and `tests/Unit/WidgetPrimitive/Projectors/SystemModelProjectorMembershipTest.php`:
   - Resolver returns `['item' => null]` for a user without `view_membership` (cms_editor case from 5d-1's role layout — confirm via PermissionSeeder grep before writing the test).
   - Resolver returns the active membership only — does not return pending / expired / cancelled.
   - Resolver returns `['item' => null]` when the contact has no active membership.
   - Resolver returns the membership attached to the ambient contact only — Contact A's membership does not leak into Contact B's render.
   - Resolver eager-loads the tier (no N+1 — exactly one `memberships` SELECT plus one `membership_tiers` SELECT, no third query for the tier).
   - Resolver returns `['item' => null]` when the ambient slot is not record-detail (PageAmbientContext) or the record is not a Contact.
   - Projector returns only declared fields; `notes`, `stripe_session_id`, `external_id`, `custom_fields` are undeclared and return empty strings even when present in the source row.
   - Projector formats `starts_on` and `expires_on` via DateFormat::MEDIUM_DATE.
   - Projector returns empty strings for null tier (defensive — schema allows tier_id NULL on tier deletion).
   - Projector returns empty string for null `expires_on` (lifetime memberships).

## Phase 2 — `MembershipStatus` widget (definition + Blade template)

1. Create `app/Widgets/MembershipStatus/MembershipStatusDefinition.php`:
   - `handle: 'membership_status'`
   - `label: 'Membership Status'`
   - `description: 'Displays the contact\'s active membership tier, status, and key dates.'`
   - `category: ['admin']`
   - `allowedSlots(): ['record_detail_sidebar']`
   - `acceptedSources(): [Source::HUMAN, Source::IMPORT]` (memberships are imported from CSVs in addition to being human-authored)
   - `schema(): []` — no per-instance config in this first cut. Future work might add a `show_amount: bool` toggle for compliance use-cases; defer.
   - `defaults(): []`
   - `dataContract(array $config): DataContract` — `SOURCE_SYSTEM_MODEL`, `model: 'membership'`, `cardinality: 'one'`, `fields: ['membership_id', 'membership_tier_name', 'membership_billing_interval', 'membership_status', 'membership_starts_on', 'membership_expires_on', 'membership_amount_paid']`, `filters: []` (the resolver derives the filter from the ambient record).
2. Create `app/Widgets/MembershipStatus/template.blade.php`:
   - Reads `$widgetData['item']`.
   - When null: empty state "No active membership."
   - When present: heading "Membership Status", followed by tier name, billing interval, starts_on, and (conditionally) expires_on with a "Lifetime" inline label when expires_on is empty AND billing_interval is `lifetime`.
   - amount_paid is rendered when non-empty, prefixed with `$` for display. (No locale-aware currency formatting in this session — the existing app emits raw decimals in similar admin contexts.)
3. Register in `app/Providers/WidgetServiceProvider.php` alongside other widgets.
4. Run `db:seed --class=WidgetTypeSeeder` to register the new `widget_types` row.
5. **Tests** at `tests/Unit/Widgets/MembershipStatusDefinitionTest.php`:
   - allowedSlots returns `['record_detail_sidebar']`
   - acceptedSources returns `[Source::HUMAN, Source::IMPORT]`
   - dataContract shape (cardinality: 'one', model: 'membership', fields list, empty filters)
   - schema is empty (no config knobs in this first cut)

## Phase 3 — Seeder update

1. Update `database/seeders/RecordDetailViewSeeder.php`:
   - Refactor `seedContactOverview()` to seed widgets per-handle rather than via the count-based early-return. Each widget (Recent Notes, Membership Status) has its own `firstOrCreate`-equivalent guard: "widget with this widget_type_id already attached to this View? skip; else attach."
   - Membership Status seeds at `sort_order: 1` (Recent Notes stays at 0).
   - **Idempotency invariant preserved**: existing installs that already have Recent Notes attached gain Membership Status on next seed; installs with neither get both; installs with both get nothing changed; installs that have manually reordered or removed widgets via the admin UI keep their customizations (the seeder only adds missing handles, never removes or reorders).
2. Run `db:seed --class=RecordDetailViewSeeder` to confirm idempotency on the dev install (existing Recent Notes stays, Membership Status appends).
3. **Test** at `tests/Feature/RecordDetailViewSeederTest.php`:
   - Update the existing "attaches the recent_notes widget" test to assert both widgets are attached on fresh seed.
   - Add a test asserting the seeder is idempotent across re-runs (run twice, assert exactly one of each widget on `contact_overview`).
   - Add a test asserting the seeder adds Membership Status to an install that already has Recent Notes attached but not Membership Status (simulates the migration path for installs that ran 5d-1's seeder).

## Out of scope

- **Full Phase 5e (perm-in-contract architecture).** This session lands a literal `Gate::denies('view_membership')` check at the resolver's `membership` arm. Generalizing to "every contract declares required perms; resolver gates uniformly" stays carved out for a dedicated Phase 5e session, evaluated at 5d-3 close based on three widgets of evidence.
- **Membership history.** This widget shows the active membership only. A separate "Membership History" list-shaped widget covering pending/expired/cancelled rows is a possible 5d-4+ widget; not in this session.
- **Projector-assembled labels** (`membership_label`, `membership_renewal`). The 226 affordance is available; this session ships raw fields only. Add as a follow-up if the template's branching gets unwieldy.
- **`show_amount` config knob.** Compliance use-cases may want to hide the dollar amount from non-finance roles. Defer until requested — out-of-the-box `view_membership` already gates the entire panel.
- **Widget help system integration.** Track-doc carry-forward. Lands when 5d arc reaches its third concrete widget (5d-3) and the rollup gap becomes visible.
- **Recent Donations.** Phase 5d-3 — the natural forcing function for either a third per-arm gate or a Phase 5e architecture session.
- **`DashboardConfig` retrofit to `DashboardView` implementing `IsView`.** Cheap follow-up post-5d, scheduled ad-hoc.
