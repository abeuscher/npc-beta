### Workstream 2 — CRM Importer deep structural pass

Lifecycle walked end-to-end across all six content types (contacts / events / donations / memberships / invoice details / notes) and the two shared traits (`InteractsWithImportWizard`, `InteractsWithImportProgress`). Per-type pages follow a consistent Upload → Review → Map → Preview → runImport → Progress shape with the divergent exception of `ImportContactsPage.php` (976 LOC), which predates the trait and uses the flat (non-namespaced) field grammar, a separate `__custom__` sentinel, and its own collision-resolution UI. Aggregator registries share near-identical `CONTACT_MATCH_KEYS` / `groupedOptions()` / `flatFields()` / `split()` shapes across five files. Approve / rollback / delete actions on `ImporterPage` dispatch on `model_type` with 60-LOC inline branches per operation per type.

Findings captured on W6 (#1–#6) and W7 (#1) tables; Contacts convergence captured as Open Flag B.

### Workstream 3 — Typography, theme & images subsystem

Typography call sites clean — `TypographyResolver::load()` / `TypographyCompiler::compileScoped()` invoked only from intended sites (public layout, page-builder preview scope, design-system bootstrap, SCSS export controller). `ImageSizeProfile::configuredBreakpoints()` consistent across every upload and render path (all models route through `photo()` / `thumbnail()` factory methods). `SampleImage` categories consistent with widget references (spot-check during audit initially flagged `CATEGORY_PRODUCT_PHOTOS` as unused, but verification showed active use by `DemoDataService` and `DashboardDebugGeneratorWidget` — false positive). SCSS export route gated on `manage_cms_settings` (same permission as the Vue typography editor's write endpoint); demo seeder integration sound.

### Workstream 4 — Notes structured interactions subsystem

`Note::$fillable` covers every structured column (`type`, `subject`, `status`, `body`, `occurred_at`, `follow_up_at`, `outcome`, `duration_minutes`, `meta`, plus `import_source_id` / `import_session_id` / `external_id`). Defaults applied at DB level (`type = 'note'`, `status = 'completed'`) — no model events / observer needed for correctness. `ContactNotes::getTimeline()` projects every structured field and interleaves activity-log entries correctly. Notes importer paths follow the namespaced pattern with the staged-update / approve / rollback lifecycle intact. Edit-by-creator gate confirmed absent (deferred to Notes Permissions session per 204's out-of-scope decision).

One gap surfaced: `Note` has no observer (Contact, Donation, Event, Membership, Page, Purchase all do). Note CUD does not appear in `activity_logs`. Compounded by `ImporterPage::approve()` creating an audit-trail Note only when the staged-update subject is a Contact — Note subjects get `fill()->save()` with zero trace. Captured as **W7 #2** and **Open Flag C**.

### Workstream 6 — Permission and security audit

All checks clean except one architectural note.

- `PageBuilderApiController::update()` schema-key filtering (session 179 Flag C) intact: `config` filtered against widget schema, `appearance_config` against `['background', 'text', 'layout']`, `query_config` against declared collection handles.
- Importer permission separation consistent across all 11 pages: wizards and progress pages gate on `import_data`; `ImporterPage` approve/rollback/delete actions gate on `review_imports` with three-layer defense (hidden + requiresConfirmation + abort_unless inside closure).
- Typography SCSS export route gates on `manage_cms_settings` (same as update endpoint). No semantic confusion in practice but flagged as **Open Flag F** for documentation.
- `WidgetThumbnailController` regex `^(static|preset-[a-z0-9-]+)\.png$` unchanged; widget folder derived from registry not URL.
- All portal routes added since 179 correctly scope to the authenticated portal contact.
- `PresetController` — all three public methods (`store` / `update` / `destroy`) gated on `update_page`.

Architectural note: no per-update ownership check on `ImportStagedUpdate`. A user with `review_imports` can approve / rollback / delete every staged update in a session regardless of who authored it. By design (sessions are the unit of review) but captured as **Open Flag D** for an explicit docs-update decision.

## Open Flags

- **A — LayoutInspectorPanel architectural divergence.** Three paths: (a) widen the universal appearance panels with a polymorphic target (widget or layout); (b) add `page_layouts.appearance_config jsonb` and reuse the existing panels (requires a data migration); (c) defer to a dedicated session. Decision required at the 206 walkthrough.
- **B — ImportContactsPage convergence with the namespaced pattern.** Retrofit is substantial (~976 LOC page, ~100 tests, 4 Playwright specs, sentinel rename in DB-persisted maps). Confirmed out of scope for 206; reevaluate in a future housekeeping pass.
- **C — Notes audit-trail-via-import gap.** Approve-time staged-update audit is created only for Contact subjects; Note subjects get `fill()->save()` with no trail. Plus no `NoteObserver`. Two paths: defer both to the stubbed Notes Permissions session, or land `NoteObserver` in 206 and defer only the approve-time question. Decision at the 206 walkthrough.
- **D — Staged-update per-update ownership.** No per-update author check on approve / rollback / delete — session is the unit of review. Decision: docs-only clarification in 206.
- **E — Missing per-type importer help docs.** Shape question: umbrella doc vs per-type stubs vs extending `import-contacts.md`'s frontmatter `routes:` array. Recommendation: umbrella-doc approach, content-writing iteration within 206.
- **F — SCSS export permission granularity.** `ThemeTypographyController::export()` gates on `manage_cms_settings` (same as its update). Recommendation: leave as is, one-line docstring comment.
- **G — Playwright pre-seed contacts are email-only.** `tests/e2e/helpers/db.ts:192-203` inserts only `(id, email, source, ...)`. Fix: switch to reading the generated `contacts.csv` and bulk-inserting full rows. ~50 LOC in test fixtures, no production code. Scheduled as a 206 iteration.
