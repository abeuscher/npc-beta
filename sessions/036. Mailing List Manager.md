# 036. Mailing List Manager

Build a mailing list system that lets administrators define named audience lists using filter rules against the contacts table. Lists are dynamic — they evaluate at query time against current contact data. The system has two filter modes: a **simple** mode with a repeater UI covering standard fields, and an **advanced** mode exposing a raw `WHERE` clause textarea for power users. Advanced mode is a privileged feature — off by default for all roles, manually granted.

This system produces filtered contact sets. It does not send email. MailChimp and future integrations will consume these lists.

---

## Background & decisions

- Mailing lists are definitions, not static rosters. A list's `contacts()` method re-evaluates the filter every time it is called.
- Simple filters support standard contact columns and tags. Custom fields (stored as JSON) are accessible via the advanced raw `WHERE` clause.
- AND/OR conjunction is a single top-level toggle per list — all simple rules use the same conjunction. Advanced and simple rules are combined under that same conjunction.
- The advanced filter executes against a dedicated read-only Postgres connection to minimise risk. A blocklist of dangerous SQL keywords and a 5-second statement timeout provide defence-in-depth.
- The `use_advanced_list_filters` permission is seeded off for all roles and must be explicitly granted by a super_admin. It does not appear in the standard role editor — only super_admin can grant it.

| `tags` | relationship | uses `whereHas` / `whereDoesntHave` |

### Operators

| Operator key | SQL behaviour |
|-------------|--------------|
| `equals` | `= ?` |
| `not_equals` | `!= ?` |
| `contains` | `ILIKE '%value%'` |
| `not_contains` | `NOT ILIKE '%value%'` |
| `includes` | `whereHas('tags', fn => where('name', value))` |
| `not_includes` | `whereDoesntHave('tags', fn => where('name', value))` |
| `is_empty` | `whereNull` + `orWhere('' =)` |
| `is_not_empty` | `whereNotNull` + `where(!= '')` |

Apply rules using `when($list->conjunction === 'and', fn => $q->where(...), fn => $q->orWhere(...))` — use `where(function($q) {...})` closures to group correctly.

### Advanced filter

If `$list->raw_where` is set:

1. Validate: reject if the clause contains any of these keywords (case-insensitive): `DROP`, `DELETE`, `UPDATE`, `INSERT`, `TRUNCATE`, `ALTER`, `CREATE`, `GRANT`, `REVOKE`, `EXECUTE`, `CALL`, `--`, `/*`.
2. Execute against the `pgsql_readonly` connection:
   ```php
   DB::connection('pgsql_readonly')->transaction(function () use ($list) {
       DB::connection('pgsql_readonly')->statement("SET LOCAL statement_timeout = '5000'");
       return DB::connection('pgsql_readonly')->select(
           "SELECT id FROM contacts WHERE ({$list->raw_where}) LIMIT 50000"
       );
   });
   ```
3. Extract the returned IDs and apply `Contact::query()->whereIn('id', $ids)` — combined with simple filter results using the list's conjunction.
4. If the advanced query throws, re-throw as a user-readable `\RuntimeException` with the Postgres error message.

## Read-only Postgres connection

### `.env` variables (add to `.env.example` too)

```
DB_READONLY_USERNAME=nonprofitcrm_readonly
DB_READONLY_PASSWORD=
```

### `config/database.php`

Add a `pgsql_readonly` connection that inherits from `pgsql` but uses the readonly credentials:

```php
'pgsql_readonly' => array_merge(config('database.connections.pgsql'), [
    'username' => env('DB_READONLY_USERNAME', env('DB_USERNAME')),
    'password' => env('DB_READONLY_PASSWORD', env('DB_PASSWORD')),
]),
```

### Creating the Postgres user

This cannot be automated in a migration (requires superuser). Document in a **Setup note** in the help article:

```sql
CREATE USER nonprofitcrm_readonly WITH PASSWORD 'choose-a-password';
GRANT CONNECT ON DATABASE nonprofitcrm TO nonprofitcrm_readonly;
GRANT USAGE ON SCHEMA public TO nonprofitcrm_readonly;
GRANT SELECT ON contacts TO nonprofitcrm_readonly;
```

If `DB_READONLY_USERNAME` / `DB_READONLY_PASSWORD` are not set in `.env`, the `pgsql_readonly` connection falls back to the main credentials — advanced mode still works but without the extra isolation layer. Document this fallback behaviour clearly in the help article.

## `MailingListResource`

`app/Filament/Resources/MailingListResource.php`

- Navigation: group `CRM`, label `Mailing Lists`, icon `heroicon-o-user-group`, sort `6`

### List columns

- `name` — label `List`
- `conjunction` — badge, formatted as `ALL rules` / `ANY rule`
- `is_active` — toggle column
- Live contact count — `TextColumn` using a `getStateUsing` closure that calls `$record->contacts()->count()`

### Form

**Section "List"** (columns: 2):
- `TextInput::make('name')` — required
- `Select::make('conjunction')` — options `['and' => 'ALL rules must match (AND)', 'or' => 'ANY rule must match (OR)']`, required, default `and`
- `Textarea::make('description')` — nullable, `->columnSpanFull()`
- `Toggle::make('is_active')` — default `true`

**Section "Filters"**:
- `Repeater::make('filters')` — relationship repeater, `->relationship('filters')`, `->orderField('sort_order')`
  - `Select::make('field')` — options: the simple filter field list (see above), required
  - `Select::make('operator')` — options: all operators from the operator table, required
  - `TextInput::make('value')` — nullable (not required — `is_empty` / `is_not_empty` need no value)
- `Placeholder` showing current contact count (re-evaluated on load): *"X contacts match this list's filters."*

**Section "Advanced Filter"** — visible only when `auth()->user()->can('use_advanced_list_filters')`, collapsed by default:
- `Placeholder` warning: *"⚠ Advanced mode. Write a raw PostgreSQL WHERE clause applied to the contacts table. This bypasses the standard filter UI. Queries time out after 5 seconds. See the help article for the table schema and examples."*
- `Textarea::make('raw_where')` — label `WHERE clause`, nullable, `->extraAttributes(['style' => 'font-family: monospace'])`, `->columnSpanFull()`

### Header actions

- **Export CSV** — `Action::make('export')`, icon `heroicon-o-arrow-down-tray`. Calls `MailingListQueryBuilder::build($this->record)`, streams a CSV with columns: `first_name`, `last_name`, `email`, `phone`, `city`, `state`, `postal_code`, `tags` (comma-joined tag names). Filename: `list-{$record->name}-{date}.csv`.

### Pages

- `index`, `create`, `edit` — standard pages

## Out of scope

- Unsubscribe / opt-out mechanics (handled by MailChimp)
- Sending email directly from a mailing list
- Scheduled or automated list evaluation
- JOINs to other tables in the advanced filter (contacts table only for v1)
- Custom field filters in the simple repeater UI (use advanced mode for custom fields)
- GIN index on `custom_fields` (noted as a future performance optimisation for large installations)
