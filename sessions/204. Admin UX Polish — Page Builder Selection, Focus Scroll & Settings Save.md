# 204. Admin UX Polish — Page Builder Selection, Focus Scroll & Settings Save

Batched admin-side UX polish surfaced during designer review of the page builder, admin nav, settings pages, importer surfaces, and the Notes Timeline. Every item is an independent behaviour/chrome fix; no schema changes, no new widgets, no public-site impact.

---

## Design decisions (resolved before starting)

- **Per-section Save commits only that section's fields.** A Save button adjacent to a section saves that section. Save-whole-form-anchor-at-section is surprising; rejected. Implementation: each per-section action filters `$this->form->getState()` to the Section's field names and persists only those keys. Validation still runs on the whole form; a per-section save that fails its own validation errors in place.
- **Inspector tab persistence is conversation-local.** Pinia store holds the active-tab index / key; switching widgets does not reset it. First widget selected after page open lands on the `Content` tab (or whichever tab is index 0 at that point — verify during implementation).
- **Focus scroll behaviour — two tiers.** "Scroll to the middle of the viewport on selection" is **must-have** and ships. "Cannot scroll the focused widget off screen" (lock/clamp) is **stretch**; if wheel/touch event interception turns adversarial (inertial scrolling, nested-scroll in column widgets, mobile), pause, surface the trade-off, and ship without the clamp. The tall-widget exception (widget becomes the scroll container) is only relevant if the clamp is kept.
- **Notes Timeline expanded/collapsed toggle is conversation-local.** Livewire property (`$viewMode = 'collapsed' | 'expanded'`, default `collapsed`) threaded through the page; toggle button in the top-right header. No per-user persistence yet — the preference can graduate to a user-level setting later if it comes up.
- **Importer Preview button fix: wire up content, do not rename.** The action already has `modalContent` callbacks for events / financial / contacts / notes; the surfaced complaint is that the Contacts path only shows a count. Verify during implementation; if the issue is actually "the contacts blade renders only a count" then fix that blade. Renaming is a fallback only if wiring real content proves out of scope.
- **FieldRegistry leak audit is mechanical.** For every `*FieldRegistry` that derives fields from `$model->getFillable()`, ensure `$excluded` lists `import_source_id`, `import_session_id`, and (where applicable) `external_id`. `external_id` is allowed to leak only if the registry explicitly re-adds it at the end (pattern used by `ContactFieldRegistry`).

## Phase 1 — Page builder: default selection, add-widget selection, sticky inspector tab

Scope: `resources/js/page-builder-vue/` only — Pinia store + two components.

- On page-builder mount, select the top widget automatically if any widgets exist. Inspector panel populates; no empty-state placeholder.
- After **Add Widget**, set the newly-added widget as selected.
- After **Add Columns** (column-layout primitive), set the new column container as selected.
- Inspector tab state lives on the Pinia store. On widget-select change, **do not** reset the active tab. First-time selection (no prior tab state in this session) lands on the `Content` tab.
- Exact tab set: verify against `InspectorPanel.vue`. Expected tabs (subject to confirmation): Content, Margin & Padding, Appearance, Advanced.

**Acceptance:** open any page edit view → top widget is highlighted and inspector shows its content tab; click Add Widget → new widget becomes selected; click Widget A's Margin & Padding tab → click Widget B → inspector shows Widget B on Margin & Padding.

## Phase 2 — Page builder: focus scroll behaviour

Scope: `PreviewCanvas.vue` + a small scroll helper (new utility file under `resources/js/page-builder-vue/utils/` is acceptable).

- **Must-have:** when a widget is selected (by click or by store state change from Phase 1), smoothly scroll the page so the widget's vertical centre is at the viewport centre. Respect `prefers-reduced-motion` — jump instantly when reduced motion is requested.
- **Stretch — scroll clamp:** once a widget is focused, the viewport cannot scroll past it on wheel / touch. Implement by listening to `wheel` / `touchmove` on the canvas container and short-circuiting events that would move the focused widget out of view.
- **Stretch — tall widget exception:** if the focused widget is taller than the viewport, the clamp behaves as a scroll *container* for that widget only — up-scroll locks at the widget top, down-scroll locks at the widget bottom; the rest of the page is inaccessible while focus holds.
- **Escape hatch:** if the clamp fight turns into a tar pit (inertial scroll conflicts, nested-scroll with the Column widget, flaky on mobile), stop, report the trade-off, and merge just the must-have.

**Acceptance:** select a short widget → page scrolls so widget is vertically centred. (Stretch acceptance only if clamp ships.)

## Phase 3 — Admin left nav: expand-into-view

Scope: tiny — probably a single Alpine listener on `.fi-sidebar-group` toggle events, attached via `resources/css/filament/admin/theme.css` or a render hook.

- When a collapsible nav group is clicked open and its newly-revealed items would extend below the viewport fold, scroll the sidebar so the full group is visible. If scrolling the full group into view would hide the header itself, prioritise keeping the header visible and show as many child items as fit.
- No animation needed beyond the browser's smooth-scroll default.

**Acceptance:** collapse the CRM group at the bottom of the sidebar, scroll so it's near the fold, click to expand → the items become visible instead of disappearing off-screen.

## Phase 4 — Settings pages: per-section Save buttons

Scope: `app/Filament/Pages/Settings/*Page.php` — audit first, then apply consistently.

- Identify every settings page that renders multiple `Forms\Components\Section`s on one form. Likely candidates per `docs/app-reference.md`: `GeneralSettingsPage`, `FinanceSettingsPage`, `CmsSettingsPage`, `MailSettingsPage`.
- For each multi-section page, add a `Forms\Components\Actions` row **at the end of each Section** with a single `Save [Section Name]` action. Clicking runs validation on the whole form (Filament default) but persists only the Section's field keys.
- Mechanism: Action's `->action(fn () => $this->saveSection([...]))`. Implement `saveSection(array $keys)` on the page (or a shared trait `InteractsWithSectionedSettings`) that filters `$this->form->getState()` to the declared keys, updates the same settings sink the page's existing `save()` method uses, and fires a success toast scoped to the section.
- The bottom-of-page global Save button **stays** — per-section is additive, not a replacement.

**Acceptance:** open General Settings → scroll to the second section → change a field → click that section's Save → toast confirms, other sections' pending changes are NOT persisted, page stays on its current scroll position.

## Phase 5 — Importer: FieldRegistry leak + Preview button content

Two small independent fixes.

### Phase 5a — FieldRegistry `$excluded` audit

Walk every `app/Importers/*FieldRegistry.php`:

- For each that derives fields from `$model->getFillable()` (pattern: `ContactFieldRegistry` style), ensure `$excluded` lists `import_source_id`, `import_session_id`.
- `external_id` is exempt from this rule: some registries (like `ContactFieldRegistry`) intentionally re-add it at the end as a synthetic field. Leave those alone. Do exclude `external_id` from models whose registries hand-curate fields (`DonationFieldRegistry` etc. that don't auto-derive) — no action needed there.
- Regenerate the corresponding template headers and confirm the leaked `*_session_id` columns disappear from `CsvTemplateService::stream()` output.

### Phase 5b — Importer Preview modal content

`app/Filament/Pages/ImporterPage.php` — the `preview` action's `modalContent`:

- Confirm the actual complaint by reading the contacts branch (`filament.pages.import-review-preview` blade).
- If the blade renders only a row count: enrich to include the first 20 contacts (name + email) and first 20 staged updates (existing value vs proposed value), mirroring the financial / notes previews.
- If the blade already renders rows and the complaint applies to a different model type's preview: fix that one to match.

**Acceptance:** after a contacts import session reaches reviewing state, clicking Preview shows a usable list of imported contacts and any staged updates — not just `N rows imported`.

## Phase 6 — Notes Timeline: expanded / collapsed toggle

Scope: `ContactNotes.php` + `contact-notes.blade.php`.

- Add `public string $viewMode = 'collapsed';` to `ContactNotes`. Allowed values: `'collapsed'`, `'expanded'`.
- Add a `toggleViewMode()` action wired to a top-right button in the header actions row. Icon: `heroicon-o-arrows-pointing-out` when collapsed (invites expand), `heroicon-o-arrows-pointing-in` when expanded.
- Thread `$viewMode` into the Blade via the existing projection surface.
- **Collapsed card (new, default):** single line per note. Left: type icon-badge + subject (or stripped-body preview truncated to ~60 chars). Right: status pill (only if status ≠ `completed`) + relative timestamp + follow-up / overdue pill. No body, no outcome, no `meta` disclosure.
- **Expanded card (existing):** the full session-198 treatment — body HTML, outcome italic-muted, Source-fields `<details>`, inline Edit / Delete buttons.
- Toggle affects all notes in the list uniformly. Activity rows (the `activity_logs` stream interleaved in the timeline) are unaffected — they remain as single-line entries either way.
- Persistence: Livewire property only. Default on every page load is `collapsed`.

**Acceptance:** open a contact with 20+ notes → list renders as 20 one-liners → click the top-right expand toggle → all 20 notes swap to the full-card treatment → toggle again → back to one-liners.
