# 026. Deployment — Get Live on a Public Server

First deployment to a real public server. The goal is to surface environment, configuration, and architecture problems that only appear outside Docker — not to build new features. A working production install is the only deliverable.

---

## Architecture decisions

- **Hosting**: DigitalOcean Droplet (Ubuntu 22.04, Docker pre-installed via DO marketplace)
- **Image registry**: GitHub Container Registry (GHCR) — free, integrated with the repo
- **Deployment trigger**: GitHub Actions on push to `main`
- **Deployment method**: Build Docker images in CI → push to GHCR → SSH to droplet → pull images → `docker compose up -d` → run migrations
- **Web server**: Keep existing Nginx (no SSL for this session — no domain, raw IP only)
- **Database**: PostgreSQL in Docker on the same droplet, named volume for persistence
- **Two images per build**: `nonprofitcrm-app` (PHP-FPM) and `nonprofitcrm-web` (Nginx + static assets) — see Phase 1

## Phase 1 — Dockerfile: multi-stage production build

The current Dockerfile has two problems for production:
1. It doesn't build frontend assets (`npm run build` is missing)
2. The nginx container has no way to access `public/` unless the code is bind-mounted

**Solution**: Convert to a 3-stage multi-stage Dockerfile that produces two build targets:

**Stage 1 — `node-builder`**: Install Node dependencies and run `npm run build` to compile Vite assets into `public/build/`.

**Stage 2 — `app`** (final PHP-FPM image): Start from `php:8.4-fpm`, install system deps and PHP extensions (same as today), run `composer install --no-dev`, copy application code, copy the built `public/build/` from stage 1, set permissions on `storage/` and `bootstrap/cache/`. This is the `nonprofitcrm-app` image.

**Stage 3 — `web`** (final Nginx image): Start from `nginx:alpine`, copy `docker/nginx/default.conf` into place, copy `public/` from stage 2. This is the `nonprofitcrm-web` image. Nginx gets the static assets baked in; it proxies PHP requests to the `app` container as it does today.

Build commands (used in CI):
```
docker build --target app -t ghcr.io/OWNER/nonprofitcrm-app:TAG .
docker build --target web -t ghcr.io/OWNER/nonprofitcrm-web:TAG .
```

## Phase 2 — Production Compose file

Create `docker-compose.prod.yml`. Key differences from `docker-compose.yml`:

- `app` service: uses `ghcr.io/OWNER/nonprofitcrm-app:${IMAGE_TAG}` — no bind volume mount for code. Named volumes for `storage/` and `bootstrap/cache/` only (these must persist across deploys).
- `nginx` service: uses `ghcr.io/OWNER/nonprofitcrm-web:${IMAGE_TAG}` — no bind volume mount. Exposes port 80 only.
- `worker` service: uses the same `app` image with `command: php artisan queue:work ...` — no bind mounts.
- `postgres` service: keep as-is but remove the host port exposure (`5432:5432` → not exposed). DB is internal only.
- `redis` service: remove host port exposure (`6379:6379` → not exposed).
- All services: `env_file: .env`

The `IMAGE_TAG` variable will be set to the git SHA by the GitHub Action.

## Phase 3 — GitHub Actions workflow

Create `.github/workflows/deploy.yml`.

**Trigger**: `push` to `main` branch.

**Steps**:
1. Checkout code
2. Log in to GHCR using `GITHUB_TOKEN`
3. Set `IMAGE_TAG` to the short git SHA (`github.sha` truncated to 7 chars)
4. Build and push both images (`nonprofitcrm-app:$TAG` and `nonprofitcrm-web:$TAG`) — also tag both as `latest`
5. SSH into the droplet (using stored GitHub Secret: `DEPLOY_SSH_KEY`, `DEPLOY_HOST`, `DEPLOY_USER`)
6. On the droplet:
   - `cd /opt/nonprofitcrm`
   - Pull new images from GHCR: `docker compose -f docker-compose.prod.yml pull`
   - `IMAGE_TAG=$TAG docker compose -f docker-compose.prod.yml up -d --remove-orphans`
   - `docker compose -f docker-compose.prod.yml exec -T app php artisan migrate --force`
   - `docker compose -f docker-compose.prod.yml exec -T app php artisan storage:link`
   - Prune old images: `docker image prune -f`

Use `appleboy/ssh-action` for the SSH step.

## Phase 4 — Droplet setup (one-time manual steps)

Write a setup checklist at `docs/deployment/droplet-setup.md` covering:

1. Create Ubuntu 22.04 Droplet on DigitalOcean with Docker marketplace image (comes with Docker + Compose pre-installed). Minimum 2GB RAM / 1 vCPU.
2. Add your SSH public key during creation.
3. SSH in as root, create a deploy user: `useradd -m -s /bin/bash deploy && usermod -aG docker deploy`
4. Generate a deploy SSH keypair (`ssh-keygen -t ed25519 -C "github-deploy"`), add the public key to `~deploy/.ssh/authorized_keys`, store the private key as GitHub Secret `DEPLOY_SSH_KEY`.
5. Store the droplet IP as GitHub Secret `DEPLOY_HOST`, and `deploy` as `DEPLOY_USER`.
6. On the droplet as root: `mkdir -p /opt/nonprofitcrm && chown deploy:deploy /opt/nonprofitcrm`
7. As the deploy user, copy `docker-compose.prod.yml` to `/opt/nonprofitcrm/`
8. Create `/opt/nonprofitcrm/.env` from `.env.example`. Set at minimum:
   - `APP_ENV=production`
   - `APP_DEBUG=false`
   - `APP_URL=http://DROPLET_IP`
   - `APP_KEY=` (generate with `php artisan key:generate --show` locally)
   - `DB_PASSWORD=` (strong random password)
   - `ADMIN_EMAIL=` / `ADMIN_PASSWORD=` (for first seed)
9. Log in to GHCR on the droplet using a GitHub Personal Access Token (read:packages scope).

## Phase 6 — First deploy

1. Ensure the droplet is set up per Phase 4.
2. Merge this branch to `main` (or push directly).
3. Watch the GitHub Actions run.
4. On success, visit `http://DROPLET_IP` and confirm the app loads.
5. Run `php artisan db:seed` once manually on the server if needed to create the admin user.
