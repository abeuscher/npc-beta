## What was built

### Phase 1 — Audit

- Read `WidgetType` model, `WidgetTypeSeeder` (745 lines), `widget_types.md`, `AppServiceProvider`, `bootstrap/providers.php`, `WidgetRenderer`, `WidgetDataResolver`, `ChromeWidgetTest`, `NavigationWidgetTest`, `widget-conventions.md`.
- Confirmed nav seeder entry lives at seeder lines 379–421 with keys mapping 1:1 to `widget_types` columns.
- Confirmed Laravel 11 provider registration pattern via `bootstrap/providers.php` rather than `config/app.php`.
- Confirmed existing `widget-conventions.md` still current; no changes needed.

### Phase 2 — WidgetDefinition base class

- Created `app/Widgets/Contracts/WidgetDefinition.php`. Abstract class with required methods (`handle`, `label`, `description`, `schema`, `defaults`, `template`) and defaulted optional methods (`category`, `collections`, `assets`, `fullWidth`, `defaultOpen`, `allowedPageTypes`, `renderMode`, `requiredConfig`).
- `toRow()` assembles the array shape `WidgetType::updateOrCreate()` consumes.
- `validate()` throws `RuntimeException` when `defaults()` omits any schema key.

### Phase 3 — WidgetRegistry service

- Created `app/Services/WidgetRegistry.php` with `register()`, `all()`, `find()`, `sync()`. `sync()` iterates definitions, calls `validate()`, then `WidgetType::updateOrCreate(['handle' => $def->handle()], $def->toRow())`.
- Created `app/Providers/WidgetServiceProvider.php`. `register()` binds the singleton; `boot()` registers `NavDefinition`.
- Added `App\Providers\WidgetServiceProvider::class` to `bootstrap/providers.php`.

### Phase 4 — NavDefinition

- Created `app/Widgets/Nav/NavDefinition.php` porting the seeder entry one-to-one: 19 schema fields including colors, alignment picker, gradient, drop menu grouping, mobile animation, and `_drop_label` heading.
- `defaults()` declared for every schema key (including image/null fields and the heading pseudo-field) to satisfy `validate()`.
- `requiredConfig()` preserved as `['keys' => ['navigation_menu_id'], 'message' => …]`.

### Phase 5 — Seeder integration

- Removed the `['handle' => 'nav']` `updateOrCreate` block from `WidgetTypeSeeder::run()`.
- Added `app(\App\Services\WidgetRegistry::class)->sync()` at the end of `run()`.

### Phase 6 — Runtime equivalence verified

- `ChromeWidgetTest` ("seeds logo and nav widget types") and all `NavigationWidgetTest` cases pass unchanged.
- New regression test `seeder-sourced nav row matches registry-sourced nav row` compares all `toRow()` fields to the seeded row (using canonicalised comparison for `config_schema` since Postgres JSONB does not preserve object-key order).

### Phase 7 — Documentation

- Created `docs/widget-system.md` describing the definition-class pattern, the registry, the coexistence story, and how to add a new widget under the new system.
- `docs/widget-conventions.md` left unchanged — still accurate for styling/markup conventions.
