## What was built

Executes **CB2** of the Client Billing & Account track — the node's read-only account window. **Non-boundary:** consumes contract v2.6.0 (authored at CB1 / 366) without changing it; no `/api/*` surface, no response-schema change, no `CONTRACT_VERSION` bump. Depends on CB1 only, so it ran in the CRM lane in parallel with Fleet Manager's own chain. **No vendor-Stripe credential, config key, or SDK call CRM-side** — the page renders a pushed JSON document and links out to Stripe's hosted portal; that link label is the only "Stripe" in the new code.

### Phase 1 — The "My Account" page

- **`App\Filament\Pages\Settings\AccountPage`** (Settings group, nav label "Account", heading "My Account", route `filament.admin.pages.account-page`). Renders **non-Eloquent** content: `mount()` reads CB1's `BillingStateReader` once and bakes a display-ready view-model into a public array; the Blade view is dumb. No second read path, no cache, no migration.
- **Fields, all from the pushed document:** plan name + price; a plain-English **status badge** derived from `status()` / `trial()` — `active` → "Active" (success), `past_due` / `unpaid` → "Payment problem — please update your card" (danger), `trialing` → "Trial — N days left" (warning, computed from `trial.ends_at`), `canceled` → "Canceled" (gray), unknown → title-cased fallback; **next invoice** date + amount + line items; **read-only billing contact**; a **"Manage billing"** button to `portal_url`; a **staleness footer** ("Billing information as of {relative time}", plus the note that a payment made just now can take up to a day to appear).
- **Money formatting:** the schema carries integer **minor units** + a lowercase ISO currency. A local symbol map (`usd → $`, eur, gbp, cad, aud, nzd) formats to `$49.00`, falling back to `49.00 XXX` for unmapped codes — deliberately deterministic (no `intl`/locale dependence) so tests don't drift. Invoice amounts inherit the plan's currency (the schema carries currency only on `plan`).
- **Self-hides.** `canAccess()` requires **both** the `manage_account` ability **and** a present billing-state document, so on internal nodes / fresh installs the page is absent from navigation (Filament skips nav registration when `canAccess()` is false) and returns **403** on direct URL — for everyone, super-admin included. Additive by construction, exactly as CB1 is inert until FM pushes. `shouldRegisterNavigation()` is overridden to follow `canAccess()` explicitly.

### Phase 2 — The two banners

- **`BillingState::needsBillingAttention()`** — one derived, display-only accessor added to CB1's DTO so the page and the panel-wide banner share a single source of truth for "is this a pre-lock delinquency": `status` in `{past_due, unpaid}` **or** a grace clock (`suspension.grace_ends`) is set. The null-object returns `false`, so an absent/unusable document is never "attention."
- **On-page prominent banner** — under past-due/grace, an amber card naming the date admin access will be paused (from `grace_ends`, formatted long) and linking to the portal, with the CB1 framing reused: the public site, donations, and member portal keep running; only the back office is affected.
- **Slim panel-wide banner** — a `PanelsRenderHook::PAGE_START` injection in `AdminPanelProvider` (verified: the hook renders inside `<x-filament-panels::page>`, which resource list/edit pages also use, so it covers every admin screen). Gated on `can('manage_account')` — the person who can fix it sees it wherever they are, and billing status is **not** leaked to non-billing roles. Suppressed on the Account page itself via the render hook's `$scopes` (Filament passes `[static::class]`), since the prominent banner is already there. Reads the same per-request `BillingStateReader` singleton; no Stripe reference.

### Phase 3 — Permission, guard, docs

- **`manage_account`** seeded in `PermissionSeeder` (a row, **no migration** — mirrors `manage_cms_settings`) and **granted to no shipped role**, making it super-admin-only via the existing `Gate::before` bypass unless a client adds it to a custom role. This is the **deliberate** version of the unassigned-permission shape the session-280 permission audit flagged as an *accident* for `manage_dashboard_config` / `manage_record_detail_views`; `docs/runbooks/permission-matrix.md` gains the grant-table row, the Settings-pages row, and a per-page note saying so in as many words.
- **Convention-drift guard — two new cases** in `tests/Feature/Infrastructure/ConventionDriftTest.php`, per track § Design decision 6:
  1. **No CRM code may reference a vendor-billing identifier.** Scans `app/` + `config/` for `stripe_billing` / `vendor_stripe` (case-insensitive, covering `STRIPE_BILLING*`) — the Fleet-Manager-side names, reserved and banned CRM-side. Scanning source and config only means the test's own banned-token list (in `tests/`) never self-trips.
  2. **The billing/account code may not reference the Stripe SDK or config.** Scans `app/Services/Billing/*.php` + `AccountPage.php` for `services.stripe` / `StripeClient` / the `Stripe\` namespace. The mapped **donation** surface (`StripeCheckoutService`, the checkout controllers, `StripeWebhookController`, the `services.stripe.*` boot block) is outside the scanned set by construction — and a **positive control** asserts `StripeCheckoutService` *does* match the banned pattern, so the guard can never silently degrade into a test that catches nothing.
- **Help doc** `resources/docs/account.md` (category `settings`, route `filament.admin.pages.account-page`); `help:sync` accepted it cleanly.
- En passant: `AccountPage::getBreadcrumbs()` uses `self::getUrl()` per the established page convention (a second commit — the first draft used `url()->current()`).

## Security considerations

The session's security surface is **who can see billing state**, and the answer is deliberately narrow. `manage_account` is granted to no shipped role, so only a super-admin reaches the page unless a client explicitly delegates it. The slim panel-wide banner carries the same `manage_account` check rather than rendering for every admin user — a delinquency notice is a business fact about the organization, and there was no reason to surface it to a volunteer coordinator or blogger. The page also self-hides entirely where no document exists, so nodes that were never commercial expose no new surface at all.

The page is **read-only by construction**: there is no local billing form, no node→FM write channel, and every edit affordance is a hand-off to Stripe's hosted portal (per track open question 5, confirmed at the gate). The `portal_url` rendered into an `<a href>` originates from a document Fleet Manager writes over its trusted SSH provisioning channel — not from user input — and Blade's auto-escaping covers attribute injection regardless; the link carries `rel="noopener noreferrer"` with `target="_blank"`.

The standing "two Stripes stay separate" property is now **mechanically enforced** rather than merely documented, and the guard is written so it fails loudly if the billing code ever reaches for a vendor credential *and* if the guard itself stops matching the donation surface it is supposed to leave alone. Enforcement (the `SUSPENSION_STATE` env flag and its middleware) was not touched; nothing on this page can influence what the node locks.

## Production deployment notes

Two steps beyond the standard `migrate`, both idempotent:

- **`php artisan db:seed --class=Database\Seeders\PermissionSeeder`** — creates the `manage_account` permission row. Without it, a super-admin still reaches the page (the `Gate::before` bypass short-circuits the check), but the ability cannot be granted to a custom role.
- **`php artisan help:sync`** — publishes the new `account` help article.

The Account page remains **invisible on every node** until Fleet Manager pushes a billing-state document to `storage/app/private/fleet/billing-state.json` (FM-side session **FM-B2**). That is the designed behavior, not a defect — smoke-testing before FM-B2 means hand-placing a fixture document at that path.

## Deferred decisions

- **CB3 stays its own session** — the demo-conversion cleanup command (node half, ≤0.5 session, pairs with FM-B4) was explicitly **not folded** into CB2 at the open gate; CB2 was already a full page + permission + banners + guards. Drafted as session **368**.
- **CB3's safety gate** — surfaced in the 368 prompt, not resolved here: blanket-unlocking pages is a no-op on a fresh install (`pages.locked` defaults false) but would silently unlock an established client's deliberately-locked pages. The 368 open gate must choose an evidence gate, a `--force`, or unconditional, and order the command's two operations so a failed run converges on retry.
- **CB3's inverted guard** — recorded prominently in the 368 prompt: `demo:reset` / `demo:restore` refuse to run unless `isDemoMode()`, but the cleanup command runs *after* FM flips `APP_ENV` to production, so it must carry no such guard.
- **Track-doc open questions 2 / 3 / 4 / 8** — trial policy, hourly-work logging shape, payment default, overall slotting — remain owner-facing and gate FM-side work, not CRM. Questions 1 + 6 were answered at the 366 gate; **5 + 7 were confirmed at this session's gate** (read-only account page / no local edit forms; `manage_account` granted to no shipped role).
- **Housekeeping capture buffer** — one item folded into `sessions/housekeeping-inbox.md`: form spam / CAPTCHA. Digested at the close gate and **reframed**: honeypot fields already ship on custom forms, event registration, membership checkout, and portal signup, alongside per-route rate limits — so spam is arriving *despite* a first line of defense, making a user-visible challenge the last rung of a ladder (timing checks, tighter throttles, content heuristics) rather than the first. Needs its own sizing pass; not folded into 368.
