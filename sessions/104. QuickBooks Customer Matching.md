# 104. QuickBooks Customer Matching

Link CRM contacts to QuickBooks Customer records so that synced Sales Receipts and Refund Receipts carry donor/purchaser attribution, making the QuickBooks ledger useful for per-person reporting and reconciliation.

---

## Design decisions (resolved before starting)

- **Matching strategy:** match by email address. When syncing a transaction that has a `contact_id`, look up the contact's email against QuickBooks Customers. If a match is found, reuse it. If not, create a new QB Customer from the CRM contact's name and email. Store the returned QB Customer ID on the CRM contact record for future lookups.
- **No-contact fallback:** transactions without a `contact_id`, or contacts without an email address, sync exactly as they do today — no `CustomerRef` on the receipt. This is not an error.
- **Cache on contact:** once a contact is matched or created in QB, store the QB Customer ID on `contacts.quickbooks_customer_id`. Subsequent transactions for the same contact skip the QB lookup entirely.
- **Contact updates:** if a contact's name or email changes in the CRM after the initial QB Customer was created, do NOT automatically update the QB Customer record. Accounting records should not shift retroactively. If this becomes a user need, it can be added later as an explicit "push update to QB" action.
- **Duplicate handling:** if the QB email query returns multiple customers, use the first active match. Log a warning but do not error.
- **Display name format:** QB Customer `DisplayName` = `"{FirstName} {LastName}"`. If both are blank, use the email address.

## Phase 1 — Schema & QuickBooksClient Extension

### Migration

Add to the `contacts` table:

- `quickbooks_customer_id` — nullable string. Stores the QuickBooks Customer ID once matched or created.

Update `docs/schema.md` after writing the migration.

### QuickBooksClient — Customer methods

Add to `QuickBooksClient`:

- `findOrCreateCustomer(Contact $contact): ?string` — takes a CRM contact, queries QB for a Customer with a matching email. If found, returns the QB Customer ID. If not found, creates a QB Customer with `DisplayName`, `PrimaryEmailAddr`, `GivenName`, and `FamilyName` from the contact, and returns the new ID. Returns null if the contact has no email. On creation, stores the returned ID in `contacts.quickbooks_customer_id`. On match, also stores the ID for caching.
- Handle the QB API's `DisplayName` uniqueness constraint — if creation fails due to a duplicate `DisplayName`, append the contact's email in parentheses and retry once.

### Sync job update

Modify `SyncTransactionToQuickBooks`:

- Before creating the Sales Receipt or Refund Receipt, check if the transaction has a `contact_id`.
- If yes, check `contact.quickbooks_customer_id`. If already set, use it. If not, call `findOrCreateCustomer()` to resolve it.
- Pass the QB Customer ID as `CustomerRef` on the Sales Receipt or Refund Receipt.
- If the contact has no email or `findOrCreateCustomer` returns null, proceed without a `CustomerRef` (current behaviour).

## Out of scope

- Automatic QB Customer update when CRM contact name/email changes (deferred — accounting records should not shift)
- Manual "link to QB Customer" action on the contact page
- Batch linking of existing contacts to QB Customers
- QB Customer deletion or unlinking
- Syncing address or phone to QB Customer record
