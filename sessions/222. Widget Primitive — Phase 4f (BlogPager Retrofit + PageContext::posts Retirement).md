# 222. Widget Primitive — Phase 4f (BlogPager Retrofit + PageContext::posts Retirement)

Retrofit BlogPager onto the contract layer and retire its sole upstream dependency — `PageContext::posts()` — in the same session. After this lands, the only remaining widgets walking direct Eloquent are the Event single-row pair (EventDescription, EventRegistration) and the Product pair (ProductCarousel, ProductDisplay), and `PageContext` retains only single-row helpers (`event(slug)`, `product(slug)`, `form(handle)`) plus the `pages()` directory list (still consumed elsewhere — verify).

## Design decisions (resolved before starting)

- **BlogPager is the first non-list-shaped post-consumer to retrofit.** Its template walks the full post list to find prev/next of the current page. The contract pattern handles this cleanly — declare `SOURCE_SYSTEM_MODEL`/`'post'` with no `limit`, fetch the same set BlogListing fetches (published posts ordered by COALESCE(published_at, created_at) DESC), let the template scan the items array for the current page's id and return adjacent items.
- **No `querySettings()` declaration.** BlogPager has no panel — there's no admin-meaningful "limit" or "order_by" for prev/next navigation. Its `dataContract()` returns a contract with `querySettings: null` (the parent default). The QuerySettings panel does not render for BlogPager. Verified by the existing 221 test pattern: `query_settings === null` in the per-widget JSON.
- **Field whitelist** (the canonical contract): `['id', 'title', 'slug', 'url', 'published_at_label', 'image', 'author_name']`. The template tokens reference `{{title}}`, `{{url}}`, `{{date}}`, `{{author}}`, `{{image}}` — these become `item.title`, `item.url`, `item.published_at_label`, `item.author_name`, and a derived `<img>` HTML string built from `item.image` (template-side composition, parallel to BlogListing/EventsListing's `price_badge` shape). `id` is on the whitelist because the template needs it to find the current-page index.
- **Add `author_name` to the post projector.** `SystemModelProjector::projectPost()` currently emits 8 fields. Add a 9th: `author_name` derived from `$post->author?->name ?? ''`. Per-field fail-closed: contracts that don't declare `author_name` (BlogListing, etc.) don't receive it. The User model is **not** exposed; only the name string crosses the contract boundary.
- **Eager-load `author` on the post query.** `ContractResolver::resolvePost()`'s `with('media')` becomes `with(['media', 'author'])`. Adds one query (`select * from "users" where ...`) to every post-contract render — bumps BlogListing's existing query-count pin by one. Update `BlogListingContractRetrofitTest::it renders BlogListing through the contract resolver only` to assert the one extra `users` select.
- **`PageContext::posts()` retires this session.** BlogPager is the sole production caller (verified mid-session 221 via grep). Mirror the 217 `upcomingEvents()` retirement: after BlogPager is retrofitted, delete `posts(?int $limit = null)` (lines 29–40) and the `?Collection $postsCache = null` property (line 16). Update `resources/docs/widget-types.md` — three prose blocks reference the helper (lines 94, 135, 153).
- **`PageContext::pages()` stays.** Different method (returns non-post pages, keyed by `pagesCache`). Confirm zero callers in `app/Widgets/*` first; if also unused, retire it too. If used elsewhere, leave alone.
- **`SOURCE_SYSTEM_MODEL` is the right primitive, not a new "navigation" arm.** A "neighbor" or "navigation" source would let the resolver do the prev/next index search server-side — tempting but premature. The blast radius would be a new source constant on `DataContract`, a new projector method, and a new resolver arm, all for one widget. Do the simple thing: contract returns the posts list, template walks it. If a second neighbor-shaped widget appears, formalize the source then.
- **Contract-locked widgets do not declare `collections()`.** System rule established in 218; BlogPager does not currently declare `collections()` (verified) — no removal needed. No `WidgetTypeSeeder` re-run required either (no row-shape change for `widget_types.collections`).

## Phase 2 — Projector + resolver extension

1. Extend `SystemModelProjector::projectPost()` to emit `author_name` from `$post->author?->name ?? ''`. Imports stay the same — `User` is not referenced (the relation is accessed via the dynamic `author` accessor on `Page`).
2. Extend `ContractResolver::resolvePost()`'s eager-load list from `with('media')` to `with(['media', 'author'])`. Cache key shape unchanged.
3. Verify `Page::author()` is a `belongsTo(User::class, 'author_id')` relation. If absent, this is a real architectural gap — stop and ask before manufacturing the relation.

## Phase 3 — BlogPager contract + template rewrite

1. Add `dataContract(array $config): ?DataContract` to `BlogPagerDefinition`. Source `SOURCE_SYSTEM_MODEL`, model `'post'`, fields `['id', 'title', 'slug', 'url', 'published_at_label', 'image', 'author_name']`, no `filters` (the existing post-contract default order — COALESCE — is what BlogPager needs). No `querySettings()` declaration.
2. Rewrite `BlogPager/template.blade.php`:
   - Drop the `$pageContext->posts()` call. Read `$widgetData['items'] ?? []`.
   - Drop the `$post->getFirstMediaUrl(...)` / `$post->title` / `$post->author?->name` / `$post->published_at?->format(...)` reads. Replace with `$item['image']`, `$item['title']`, `$item['author_name']`, `$item['published_at_label']`.
   - The current-page identification logic stays in the template — search the items array for the current page's id. Use `$pageContext->currentPage->id` to find the index; the item shape uses `$item['id']`.
   - The `$resolveTokens` closure stays in the template — it now operates on a DTO row, not a model.
3. Verify the rendered output matches the pre-retrofit shape via manual comparison or a Blade-render byte-equivalence test.
