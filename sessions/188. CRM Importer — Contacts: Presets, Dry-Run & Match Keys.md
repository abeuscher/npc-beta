# 188. CRM Importer — Contacts: Presets, Dry-Run & Match Keys

Extend the existing Tools → Importer (contacts pipeline) so it survives a real nonprofit's data. Four changes: mapping presets that persist on the `ImportSource`, a true pre-commit dry-run with halt-on-error, a configurable single-column match key, and clearer collision-option labels.

---

## Background

The importer today is a 4-step wizard (source → upload → map → preview-5-rows) backed by `ImportContactsPage` + `ImportProgressPage` + `ImporterPage` (review/approve/rollback) + `ImportHistoryPage`. It handles: custom-field creation inline, external-ID matching via `ImportIdMap`, email matching, staged-updates for existing contacts, rollback of the whole batch, per-contact notes linking to the import session, and PII pre-scan.

What's missing for a real-world import:

1. **Mappings don't persist.** `ImportSource` is a name + notes. Every re-upload re-maps. The three hardcoded presets (generic / wild_apricot / bloomerang) are heuristic guesses at upload time, not saved choices.
2. **No true dry-run.** The import starts writing rows as soon as the user clicks through preview. Errors are captured but the import continues and commits anyway. There's no "preview the whole run, then decide".
3. **Match key is hardcoded.** Duplicate detection runs against external_id (if a column was mapped to it) + email. A user with member IDs in a custom field can't declare that as the unique key.
4. **Collision labels are ambiguous.** "Skip duplicates" vs "update existing" under-describes what happens when imported values are blank or when the user actually wants a fresh duplicate.

Nothing about the CMS `ContentImporter` is touched this session — that's a backup/restore format, different concern.

## Design decisions (resolved before starting)

- **Presets live on `ImportSource`.** Extend the table (and model) with `field_map` (jsonb), `custom_field_map` (jsonb), `match_key` (string — the field-key the user picked as the unique column), and `match_key_column` (string — the original CSV column header that maps to `match_key`, kept for reference/display). No separate `ImportPreset` table — a source IS a preset.
- **Dry-run via DB transaction.** The dry-run runs the real row processor inside `DB::beginTransaction()` and rolls back after counting. Two reasons: (1) the existing row processor already handles custom field writes, staged updates, note creation — duplicating it in a "fake" path would drift; (2) large-import perf is explicitly out of scope for this session — we'll see how it holds and tune later if needed.
- **Dry-run is mandatory.** The user no longer has a "preview 5 rows, then commit" path. After mapping, the dry-run always runs, produces a report, and only then does the user get a commit button. No bypass.
- **Halt-on-error is the default.** If any row errored during dry-run, the commit button is replaced by two options: "Proceed, omitting the N errored rows" or "Cancel and fix the CSV". No silent errors in the final run.
- **Match key: single column, user-picked.** Any mapped field can be the match key — email, external_id, phone, a custom field. Composite keys out of scope. If the user doesn't pick one, default is external_id if mapped, else email. "None" is not allowed — every import must have a match key; without one, the importer can't tell an update from a duplicate.
- **Collision options, clearer labels.**
  - **Skip** — if a row matches an existing contact via the match key, leave the existing contact alone and move on.
  - **Update** — if matched, imported non-blank values overwrite existing fields. Blank imported cells are ignored (existing data is preserved).
  - **Create duplicate anyway** — ignore the match key for this run; every row becomes a new contact. Shown with an inline warning; intended for edge cases.
  - The old "Replace existing completely" semantics are dropped — no one actually means it.
- **Timeline notes reference the source.** Notes created on import already include `session_label` in the body. Extend them to also store `import_source_id` (or its name) so the contact timeline can cross-reference the source preset. Keep the note's body text human-readable and include the source name.
- **Real-data testing stays local.** WCG exports live in `local/import-fixtures/wcg/` (gitignored). Tests use anonymised fixtures under `tests/Fixtures/import/contacts/`. Never commit real data.

## Phase 1 — `ImportSource` as preset

- Migration: add `field_map` (jsonb, default `{}`), `custom_field_map` (jsonb, default `{}`), `match_key` (string, nullable), `match_key_column` (string, nullable) to `import_sources`.
- Update `ImportSource` model casts + fillable.
- Update `docs/schema/import_sources.md` to reflect the new shape.
- At mapping step: if the selected source has a saved `field_map`, pre-populate the column-to-field dropdowns from it. The user can override per run; overrides don't mutate the source unless they explicitly save.
- After a successful commit, offer "Save this mapping to the source" — updates the source's `field_map` + `custom_field_map` + `match_key`. Next run using the same source starts from this shape.
- Built-in presets (generic / wild_apricot / bloomerang) become seed data: on install, seed three `ImportSource` rows with preset names and pre-populated maps. Admin can clone them for a new org.

---

## Phase 2 — Match-key configuration

- In the mapping step, add a "Match contacts by" select. Options: every field the user has mapped (dynamically). Default to external_id if mapped, else email.
- `ImportProgressPage::processRow()` generalises: instead of hardcoded external-id-then-email branching, match against `$matchKey` field. External-id-via-`ImportIdMap` remains a special path when `matchKey === 'external_id'`; otherwise a straight `Contact::where($matchKey, $value)->first()` lookup.
- If two existing contacts share the same match key value, the row errors with "Ambiguous match on <matchKey> = <value>" — don't guess.
- Save the match key choice to the source if the user saves the mapping.

---

## Phase 3 — Dry-run with halt-on-error

- New `tick()` mode on `ImportProgressPage`: `dryRun` (default) and `commit`. Dry-run wraps row processing in `DB::transaction()` and throws at the end to force rollback; counts stay in memory.
- After dry-run completes:
  - Show summary: "Would create X / update Y / skip Z / error N".
  - If N > 0: show error table — row number, first-name/email/relevant-identifying-fields from the source row, error message. Let the admin download the errored rows as CSV (for upstream fixing).
  - Buttons: "Commit (errored rows will be omitted)" | "Cancel — I'll fix the CSV".
  - If N === 0: single "Commit" button.
- Commit path re-runs the same pipeline (no transaction this time), skipping the row numbers that errored in dry-run. Errors at commit stage should be rare since dry-run already caught them, but if they occur, they halt with a loud error and the admin gets the partial-commit situation visible in the history.

---

## Phase 4 — Collision-option relabel

- Replace the current radio ("skip existing" / "update existing") with three options using the labels above:
  - Skip
  - Update
  - Create duplicate anyway *(warning: may create multiple records for the same person)*
- The radio appears on the mapping step once the match key is chosen (Create duplicate ignores match key — it's the "I know what I'm doing" path).
- Internally, map to the existing strategy values (`skip` / `update` / new value `duplicate`); `ImportProgressPage` processes `duplicate` by creating the Contact without any match lookup.

---

## Phase 5 — Timeline notes carry source reference

- Update note bodies created in `ImportProgressPage` (new + staged-update) and `ImporterPage` (approval + rollback) to include the source name: e.g. "Imported from <Source Name> (session: <label>)".
- Add `import_source_id` as a nullable column on `notes` OR stash it in a structured field on the note body — whichever is least invasive. Preferably a proper FK so the timeline can click through to the source.
  - Decision during implementation: if `notes` is already generic/polymorphic and adding a column would touch a lot of surface, store the source id in a JSONB `context` column (add if missing). If a clean column addition is cheap, prefer that.
- Surface on the contact detail timeline: the import note becomes a clickable link to the source's history page.

## Testing

- **Slow test groups this session:** none expected.
- **New tests expected:** yes.
  - `ImportSource` persists field_map, custom_field_map, match_key across runs.
  - Dry-run rolls back all writes (contact count unchanged, custom fields unchanged, notes unchanged).
  - Dry-run reports accurate create/update/skip/error counts against a fixture with known shape.
  - Match key: pick email → matches by email; pick custom field "member_id" → matches by custom field; two existing rows sharing same match-key value → row errors with "Ambiguous match".
  - Commit after dry-run with errored rows: those rows are omitted; others commit successfully; row numbers align with the dry-run report.
  - Collision: "Create duplicate" bypasses match-key lookup — creates two contacts with same email.
  - Timeline note on imported contact references `ImportSource.name` in its body and links back to the source.
- Anonymised fixture CSVs under `tests/Fixtures/import/contacts/` covering: happy path, duplicates, malformed rows, custom field values, ambiguous match keys.
- Run `docker compose exec app php artisan test --exclude-group=slow` before close.
- No JS changes expected; no `build:public` needed.

## Out of scope

- Events / registrations import (session 189).
- Donations / memberships / financial import (session 190).
- Composite match keys (e.g. last_name + postal_code). Single column only.
- AI-assisted column mapping suggestions (future).
- Undo of approved imports (rollback today only works for pending/reviewing batches — that's unchanged).
- Performance tuning for very large imports (we'll stress-test after this session and tune based on what shows up).
- Background queue / job-based processing (Livewire polling stays).
