# 328. Per-Page Permission — Locking the Demo Site

Add a per-page **edit lock** — a `locked` flag on pages plus an `edit_locked_pages` permission — enforced in `PagePolicy` and the page-builder save API, and use it to protect the demo site's pages from the shared `demo` account. Plus close the one header/footer gap the page lock can't reach by trimming the demo role's `navigation_menu` access to view-only.

## What the demo actually is (architecture, clarified)

`nphelper.com` is the published marketing-site "demo" page; it carries a button that logs the visitor into **`demo.nphelper.com`** — a separate instance. The **demo site** is the set of sample pages on that instance (no home page — just sample pages demonstrating the product + widgets, on spoofed "Acme-style" data). The demo experience is "you're an admin managing that sample company." The `demo` role already has CRM CRUD (contacts/events/donations/etc.) intentionally. **What this session prevents:** a `demo`-logged-in user *editing the sample site's page content* — an inappropriate access point on a shared, daily-wiped instance.

---

## Design decision — `locked` flag + permission (NOT `restricted_roles`, NOT a status value)

- **Store a `locked` boolean on `pages`** (concrete, default `false`) — the access-control axis, **orthogonal to `status`**.
- **Why not a `status` value:** `Page::scopePublished()` is `where('status', 'published')`, and ~13 places key **public visibility** off it. A locked demo page is still a *live, public* page — visitors see it; the demo user just can't edit it. A "locked" status enum value would drop the page out of every `published()` query and make it vanish from the site. So keep `status = 'published'` and a **separate** `locked` flag; **display** the combination as "Published & Locked."
- **Why not `restricted_roles` (the §2a deny-list):** per-page-per-role granularity is more than the demo needs and doesn't match the app's RBAC. A binary lock gated by a **permission** is simpler and consistent with how this codebase does authorization.

## What ships

- **`locked` column** on `pages` — boolean, default `false`. Update `docs/schema/pages.md` in the same pass.
- **`edit_locked_pages` permission** — granted to admin + super_admin, **not** `demo`. (super_admin also bypasses everything via the existing `Gate::before`.)
- **Enforcement in two places, one rule** — `locked && ! user->can('edit_locked_pages')` ⇒ deny:
  - `PagePolicy::update()` (covers the Filament `PageResource`/`PostResource` edit + delete actions — posts are the same `Page` model), and
  - the **page-builder save API** (`routes/admin-api.php` `pages/{owner}/...` via `ResolvePageBuilderOwner` → `PageBuilderApiController`) — the real edit surface, which does **not** check the page policy per-page today. This is the one genuinely new bit of wiring.
- **Lock toggle on the `PageResource` form**, visible only to holders of `edit_locked_pages`, surfacing the state as **"Published & Locked."**
- **`demo:reset` locks the demo site's content pages** — sets `locked = true` on the demo's pages/posts/header+footer system pages (all `Page` model), re-applied idempotently each reset so it survives the daily wipe (guarded on `isDemoMode()`).
- **Trim the `demo` role's `navigation_menu` to view-only** in `PermissionSeeder` (revoke create/update/delete). The header/footer *link structure* lives in `NavigationMenu`/`NavigationItem` — a different model the page lock can't reach — and the demo has no need to edit the site nav. One-line gap-closer; update `DemoRoleLockdownTest`.

## What this covers (and what it deliberately doesn't)

The `locked` flag is a clean fit for everything that is a **Page**: pages, posts (`type='post'`), and the header/footer **system pages** (chrome is rendered from those pages' widgets). The `navigation_menu` trim closes the header/footer *links*. Deliberately **out of scope** for D (noted for the pen-test pass, not built here): `form` and `collection` edits that surface in public widgets (same "site content" category — decide view-only vs lock later); and the **demo upload surface** (any new-image upload by the demo role) — that's **session 329**, a separate axis. CRM/event record edits stay allowed (the demo's purpose; daily wipe is the mitigation).

## Phases

### Phase 1 — Column + permission + enforcement
Migration for `locked` (default `false`). Seed `edit_locked_pages` (admin + super_admin). Add the deny rule to `PagePolicy::update()` **and** wire the per-page policy check into the page-builder save routes (the Filament guard alone is insufficient — the builder is the real edit surface).

### Phase 2 — Admin control
Permission-gated lock toggle on `PageResource`, surfaced as "Published & Locked."

### Phase 3 — Demo protection
`demo:reset` locks the demo site's content pages, idempotently re-applied. Trim demo `navigation_menu` to view-only in `PermissionSeeder`. Confirm a `demo` user is blocked from editing a locked page (Filament + builder) and from editing nav menus, while admin/super_admin are not.

## Out of scope

- **`restricted_roles` per-page-per-role deny-list** — superseded by the `locked` flag + permission (see Design decision).
- **A `locked` *status* enum value** — breaks `published()` visibility; locked pages must stay `status='published'`.
- **`form` / `collection` public-widget edit lockdown** — pen-test follow-up, not D.
- **Demo upload lockdown** — session 329.
- **Demo-node assembly (E)** — header/footer duplication, curated showcase pages, the Faker/`--no-dev` blocker.
