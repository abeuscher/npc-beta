# 244. Pre-Beta Release Plan — Rehearsal Vetting & Success Criteria

A meta-session that converts `sessions/release-plan-outline.md` (working notes — every candidate is a *maybe*) into the actual pre-Beta-1 release gate: a vetted working set of rehearsals, a per-scenario success criterion, an execution order, and a roadmap update that lifts the selected rehearsals into named follow-up sessions while marking the un-selected ones as post-1.0 hardening.

This session does **not execute** any rehearsal. Each selected rehearsal becomes its own subsequent session.

---

## Design decisions (resolved before starting)

- **Outline is the seed; plan is the deliverable.** The session ends with `sessions/release-plan-outline.md` replaced by `sessions/release-plan.md` (the canonical, vetted, sequenced gate) — same path conceptually, distinct contents and meaning. The plan doc is the single source of truth that subsequent rehearsal sessions read at start.
- **Working set is finite.** Pick a manageable subset, not every candidate. Bias toward scenarios that produce reusable artifacts (runbooks, sizing docs, onboarding playbooks) — those double as sales credibility against incumbents per the outline's own framing.
- **Success criterion is required before a rehearsal is in.** "It worked" is not a gate. Every selected rehearsal carries a specific, measurable outcome before it's committed to the plan. If a candidate is genuinely valuable but resists a measurable success criterion, that is itself a finding — flag it and either iterate the criterion in-session or move the candidate to post-1.0.
- **Existing stubs may be consumed wholesale.** Several pre-Beta-1 stubs in `session-outlines.md` already overlap rehearsal candidates (Notes Permissions audit ↔ Workflow `Permission audit`; Org Model Overhaul ↔ Onboarding `Custom fields + public collection ingestion`; Event Ticket Tiers ↔ Workflow `Event with everything`). Per-pair design call: does the rehearsal *replace* the existing stub (the rehearsal becomes the canonical instance of that work), *consume* the existing stub (the rehearsal needs the stub's work as a prerequisite — keep the stub, sequence it before the rehearsal), or *coexist* with the existing stub (different ground; both remain).
- **Sequencing matters.** Order of execution is part of the deliverable when scenarios share fixtures or build on each other. Capsize drills logically follow Onboarding (so there's something to capsize); Scale rehearsals can run in parallel with Workflow (different fixtures); Compatibility rehearsals run last (against a finalized surface).
- **The 244 conversation may surface new candidates** the outline did not capture. Add them with the same vetting discipline. The 244 deliverable is not "vet this list as written" — it is "produce the working set, drawing from this list and any conversation-surfaced additions."
- **Non-boundary-touching by design.** No code edits, no migrations, no Fleet Manager contract surface change. Output is planning artifacts in `sessions/`.
- **Investment-seeking gate (post-244).** After 244 produces the working set and execution order, the user may pick a *subset* of those rehearsals + pre-release requirements to form an **investment-seeking gate** — a smaller, earlier bar that must be met before pitching investors, distinct from the full Beta-1 release gate. The 244 deliverable does not pre-pick the subset (that is its own future conversation) — but it **structures `release-plan.md` to support the future subset selection**: each entry in the working set carries an explicit slot (e.g., a `gate:` line per entry, or a tier label `[release]` vs `[release, investment]`) so when the time comes the subset is a markup choice, not a structural rewrite. The mental model: investment-seeking is a gate-on-the-gate — a credibility threshold lower than full release-readiness but high enough to demonstrate the product is real.

## Phase 1 — Per-candidate vetting

For each candidate in the outline (and any new ones surfaced in conversation), produce a verdict:

- **In** — promoted to the working set. Carries a written success criterion, an estimated time cost, and an explicit mapping to either a new rehearsal session or an existing pre-Beta-1 stub.
- **Merged into existing stub** — the existing stub's scope already covers this rehearsal; the stub remains as the canonical work item, possibly with its scope tightened to name the rehearsal explicitly.
- **Out (post-1.0)** — the candidate is real but doesn't meet the bar for the Beta-1 gate. Recorded with a one-line "why not now" so future-us doesn't re-litigate.

Run the four vetting criteria from the outline against each candidate:

1. How much confidence does passing it actually deliver?
2. How much real-world failure does it prevent?
3. Time cost to set up and run.
4. Whether it produces a reusable artifact (runbook, sizing doc, onboarding playbook) or just a green check.

Bias toward scenarios that produce artifacts.

---

## Phase 2 — Sequencing

Order the working set into an execution sequence. Concerns:

- **Fixture sharing.** Onboarding rehearsals may produce data that Workflow rehearsals consume; running Onboarding first amortizes the data-prep cost.
- **Capsize follows Onboarding.** Wiping a database is meaningful only when there's data worth losing. The DB wipe + backup recovery drill needs realistic-shape data first.
- **Scale runs orthogonally.** Synthetic-data scale rehearsals don't share fixtures with Workflow rehearsals; can run in parallel (different sessions, no ordering constraint).
- **Compatibility runs last.** Browser bingo and accessibility passes go against the surface as it'll ship — better to run them after Workflow and Onboarding have shaken out structural changes.

The sequence is part of the deliverable. Each rehearsal session in the eventual chain reads `release-plan.md` to know its order and prerequisites.

---

## Phase 3 — Roadmap update

Update `sessions/session-outlines.md`:

- Remove or rewrite any existing pre-Beta-1 stub that the vetting consumed wholesale.
- Add a new `### Rehearsal Sessions` subsection inside the Beta-1 scope block. One stub per inbound rehearsal, in the agreed execution order. Each stub points into the corresponding subsection of `release-plan.md` rather than duplicating its contents.
- Mark the un-selected (post-1.0) candidates appropriately — if any post-1.0 stub is worth carrying forward, add it to the post-Beta-1 section of the roadmap; otherwise it lives only in the 244 log's "out (post-1.0)" register.

---

## Phase 4 — Replace the outline with the plan

Move `sessions/release-plan-outline.md` to `sessions/release-plan.md` (in the same directory). The new plan doc replaces the old contents with:

- Purpose paragraph (lifted from the outline, refined).
- Working set: one section per inbound rehearsal **and pre-release requirement**, with title, success criterion (the explicit measurable outcome), prerequisites, estimated time cost, artifact produced, and a `gate:` line that holds either `release` (in the full Beta-1 gate only) or `release, investment` (also part of the future investment-seeking subset). 244 leaves every entry as `release` — the investment subset is selected in a follow-up conversation; the slot exists so that selection is a markup change.
- Execution order: numbered list with rationale.
- Out-of-gate register: candidates explicitly *not* in the working set, with one-line "why not now."
- Pointer to the corresponding stubs in `session-outlines.md`.

The plan doc is the canonical pre-Beta-1 release gate. Subsequent rehearsal sessions read it at start and update it on completion (per-rehearsal completion lands a checkmark; the gate is closed when all checkmarks are present).
