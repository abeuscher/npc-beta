## Security considerations

Planning-only — no code changed; the design itself was reviewed on the security axis. The plan adds **no inbound surface anywhere**: FM stays outbound-only (polling), the node gains no endpoint, no vendor-Stripe credential, and no webhook; card data never touches owner infrastructure (Stripe-hosted pages only); the restricted key is scoped to the billing object families and lives in FM's existing off-filesystem-key vault. The suspension gate deliberately reuses the demo-mode enforcement grain (env-derived flag + code-level hard gate) rather than introducing a new trust path, and the new health subcheck carries a state string + timestamp only — no email, no amounts — inside the contract's existing no-PII wire discipline. The standing privacy boundary (FM is never built to read node data) is preserved: billing flows FM→node exclusively. Two-Stripes co-mingling gets a mechanical convention-test guard in the build sessions.
