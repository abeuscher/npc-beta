# Session 114 — Demo Role & Read-Only Enforcement — Log

**Date:** 2026-04-02

## What was built

### Phase 1 — Demo role & helpers

- Created `demo` role in `PermissionSeeder` with 45 view-only permissions (`view_any_*` and `view_*` for all 21 resources, plus `view_any_form_submission`, `view_form_submission`, `view_any_member`).
- Added `isDemoMode()` global helper in `app/helpers.php` — checks `app()->environment('demo')`.
- Added `isDemo()` convenience method on the User model.
- Registered `app/helpers.php` in `composer.json` autoload files array.

### Phase 2 — Permission layer fixes (benefits all roles)

- Created 5 missing policies: `EventPolicy`, `FormPolicy`, `ProductPolicy`, `MailingListPolicy`, `FormSubmissionPolicy`. Registered all in `AuthServiceProvider`.
- Added hidden + `abort_unless` permission checks to 20+ custom actions across: `EditContact` (promote_to_member, portal access actions), `ListContacts` (merge, dismiss duplicates), `ContactNotes` (create/edit/delete note), `EditEvent` (cancel, send reminders, delete registrants, create landing page), `EditPage` (save as template, edit snippets), `EditUser` (invite, resend, revoke), `ProductResource` / `FundResource` / `FormResource` / `MembershipTierResource` (toggle_archive), `TemplateResource` (setDefault), `UserResource` (invite).
- Fixed `canDelete()` on `EventResource`, `ProductResource`, `FundResource` to check permissions alongside business logic.
- Gated `DetachAction` on `OrganizationResource` `ContactsRelationManager` by `update_organization` permission.

### Phase 3 — Page-level enforcement

- Routing settings section hidden in demo mode on `GeneralSettingsPage`. Server-side block prevents saving routing values when `isDemoMode()`.
- Users list scoped to exclude demo-role and super_admin accounts when the viewer has the demo role.
- Settings pages, import pages, Horizon, and DonorsPage already blocked by existing `canAccess()` checks — no changes needed.

### Phase 4 — Livewire method hardening

- Added `assertCanEdit()` (`abort_unless` checking `update_page`) to all write methods in `PageBuilder` (7 methods), `PageBuilderBlock` (4 methods), `PageBuilderInspector` (4 methods), and `EditPageTemplate` (7 methods).

### Phase 5 — Infrastructure gating

- Forced `mail.default` to `log` when `isDemoMode()` in `AppServiceProvider::boot()`, after the SiteSetting config load.
- Verified Horizon is already blocked (super_admin only via `Horizon::auth`).
- Verified debug generator widget is already gated by `APP_DEBUG_TOOLS` env var. Added `! auth()->user()?->isDemo()` check to hide it from demo users.

### Read-only edit view

- Created `ReadOnlyAwareEditRecord` base class that allows users with `view` (but not `update`) permission to access edit pages in read-only mode: form fields disabled, save/delete buttons hidden. The `save()` method still enforces `update` permission server-side.
- Updated all 25 edit pages to extend `ReadOnlyAwareEditRecord`.
- Added `getReadOnlyHeaderActions()` to `EditContact` (view transactions, timeline links), `EditEvent` (landing page, registrants, tickets links), and `EditPage` (public URL link) to preserve navigation actions in read-only mode.
- Added `recordUrl` to 8 resource tables (Contact, Page, Post, Event, Product, Form, NavigationMenu, Tag) so demo users can click rows to reach read-only edit views.

### Additional lockdowns

- Form JSON import action: hidden and server-side blocked for users without `create_form`.
- Form JSON download: hidden for users without `update_form` (both table and edit page header).
- Media library delete: requires `update_page` (was `view_any_page`), with server-side `abort_unless`.
- Widget manager delete: gated by `delete_widget_type` permission.
