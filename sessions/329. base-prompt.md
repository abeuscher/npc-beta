The whole surface funnels through **two chokepoints**, so this is a contained gate + a UX layer + a lockdown-test extension, not a sprawl.

## Process rules for every session

Base-prompt process rules apply in full. Session-specific notes:

- **Gate server-side at the chokepoints; the field-disable is UX, not the protection.** A demo user must be *unable* to upload even if the form field were re-enabled client-side. The endpoint gates are the security boundary.
- **Block new uploads, not existing-media reuse.** `useExistingImage` / `useExistingAppearanceImage` and any media-library *picker* introduce no new file — leave them working so the demo can still set images from the seeded library.
- **Scope the gate to the `demo` role, not all of demo mode.** The instance runs in demo mode, but an admin/super_admin maintaining the node must still upload. Gate on the user holding the `demo` role (super_admin never holds it).
- **Document what's already covered.** The media pipeline already MIME-validates and re-encodes; this session adds the *who-can-upload* gate. The log should be clear about the layered posture.
