# 265. Donation Credits — Soft-Credit Layer

Add `donation_credits` polymorphic junction supporting soft-credit attribution — a single donation attributed to multiple parties (e.g., the giving Org plus the human who triggered it; matching-gift configurations where credit can exceed 100%). Surface an inline soft-credit repeater on the Donation admin form and a soft-credit display on the Contact + Organization donations panels.

This is the soft-credit half of `release-plan.md` § B1b, paired with the structural half that shipped at 264. **B1b's checkmark drops at this session's close.**

---

## Design decisions (resolved before starting)

- **Single junction with polymorphic target.** Per user sign-off at 265 prompt-time:
  ```
  donation_credits
    id                 bigint pk
    donation_id        bigint fk -> donations on delete cascade
    attributable_type  varchar
    attributable_id    bigint
    credit_pct         decimal(5, 2)
    credit_role        text nullable    (free-text — operator-curated)
    created_at, updated_at
  ```
  Indexes: `(donation_id)`, `(attributable_type, attributable_id)`. **No unique constraint** on `(donation_id, attributable_type, attributable_id)` — operationally useful to allow the same recipient to receive credit at multiple roles (rare but legitimate; e.g., "Triggered by" + "Honour of" attributed to the same Contact on one gift).
- **`credit_pct` allows values exceeding 100.** Matching-gift configurations: a $1000 gift that triggers a 1:1 match could carry a 200% soft-credit attribution to the original donor (covers both the gift and the matched portion). v0 does NOT enforce a sum-ceiling at the application layer; operator discipline owns the invariant.
- **`attributable` polymorphic target — Contact OR Organization** in v0. Future targets (Affiliation rows, Campaigns, etc.) lift if a real driver appears.
- **`credit_role` is free-text, nullable.** Mirrors `affiliations.role` from 264. Examples: "Honour of," "In memory of," "Triggered by," "Match recipient." Operator-curated suggestions surface across donation_credits via the Filament Select's `searchable()` if convention exists.
- **Default `credit_pct` at form layer = 100.** Common case ("this gift is fully soft-credited to this person"). Operator overrides for percentage-split or matching-gift cases.
- **Donation source-of-truth stays unchanged.** `donations.contact_id` (and the post-264 affiliation-derived org binding via `Contact->primaryAffiliation`) remains the authoritative "who paid" record. Soft-credits are an attribution layer, not a payment layer.
- **No importer touch in v0.** The Donations importer doesn't gain a soft-credit column. Lift to a follow-on session if a real driver appears (likely paired with a real-world data scrub that actually carries soft-credit columns — none surfaced at B2).
- **No acknowledgment-loop integration in v0.** Year-end statements and tax-receipt emails continue to address `donations.contact_id` only. Soft-credit recipients do NOT automatically receive year-end statements. C4 territory if/when it lifts.
- **Soft-credit display on Contact donations panel — combined view.** The existing "Donations" panel on Contact lists direct donations; this session extends it to also list soft-credits received by the contact, with a "(soft-credit, X%)" tag on the relevant rows. Single chronological list (sort by `donation.donated_at`); clearer than two separate sections. Confirm at session start; lean combined.
- **Soft-credit display on Organization donations panel — same shape.** If Organization has a donations panel post-264, mirror the Contact-side combined view. If it doesn't have a donations panel yet, surface a separate small "Soft-credits received" panel rather than expanding scope to build a full Org donations panel.
- **Out of scope:** soft-credit reporting / aggregation surfaces, soft-credit on Donation list-view column, soft-credit-aware acknowledgment, importer support, Org-Org soft-credit chains.

## Phase 1 — Schema migration + Eloquent

**Goal:** `donation_credits` table live; polymorphic relations wired in both directions.

**Steps:**

1. **New migration** named e.g. `add_donation_credits_table`:
   - `CREATE TABLE donation_credits` per the shape above.
   - Indexes per the shape.
   - Standard `down()` drops the table.

2. **`DonationCredit` model** at `app/Models/DonationCredit.php`:
   - `belongsTo(Donation::class)`.
   - `morphTo('attributable')`.
   - `$fillable = ['donation_id', 'attributable_type', 'attributable_id', 'credit_pct', 'credit_role']`.
   - `$casts = ['credit_pct' => 'decimal:2']`.

3. **`Donation` model:**
   - `hasMany(DonationCredit::class, 'donation_id')` named `softCredits()`.

4. **`Contact` model:**
   - `morphMany(DonationCredit::class, 'attributable')` named `softCreditsReceived()`.

5. **`Organization` model:**
   - `morphMany(DonationCredit::class, 'attributable')` named `softCreditsReceived()`.

6. **Morph-map registration** — if the verify-at-start step found `Relation::enforceMorphMap([...])` in use, register `'contact' => Contact::class` + `'organization' => Organization::class` per the codebase's existing naming convention. If not in use, default class-name persistence is fine for v0; do not introduce a morph map this session.

**Pass criterion:** `php artisan migrate` clean; `Donation::with('softCredits.attributable')->first()` resolves both Contact and Organization targets cleanly; `migrate:status` clean; reverse-direction `Contact::with('softCreditsReceived.donation')` resolves.

## Phase 2 — Donation admin form — soft-credit repeater

**Goal:** soft-credit editor lives inline on Donation admin form, mirroring the affiliations-repeater shape from 264 with the polymorphic-target swap.

**Steps:**

1. Add a `Repeater` named `softCredits` with the relationship binding (`->relationship('softCredits')`). Schema per row:
   - **Polymorphic target Select** — `Filament\Forms\Components\MorphToSelect::make('attributable')` with `->types([MorphToSelect\Type::make(Contact::class)->titleAttribute('full_name'), MorphToSelect\Type::make(Organization::class)->titleAttribute('name')])->searchable()`. Required. If the verify-at-start confirmed `MorphToSelect` is awkward in this codebase's Filament version, fall back to a `Select::make('attributable_type')` (Contact / Organization) + dependent `Select::make('attributable_id')` shape — read the Filament docs reference site if needed and pick the cleaner option.
   - `TextInput::make('credit_pct')` — numeric, default 100, suffix `%`. Allows >100 (no max-rule). `->numeric()->default(100)`.
   - `TextInput::make('credit_role')` — free-text, nullable. Filament `->datalist()` with operator-curated suggestions sourced from `DonationCredit::query()->whereNotNull('credit_role')->distinct()->pluck('credit_role')` if the convention exists; plain TextInput if not.
2. Repeater UX:
   - Default collapsed; row preview format: "{Recipient name} — {credit_pct}%". Use `->itemLabel()`.
   - Reorderable: false.
   - "Add soft-credit" button below the repeater, default Filament shape.
3. Form-level validation: `credit_pct >= 0`. No upper bound. **No sum-of-credits validation against the donation amount or 100%** — operator owns the invariant.

**Pass criterion:** Donation admin form renders the repeater; saving rows persists `donation_credits` with the right polymorphic type/id; both Contact and Organization targets work; >100% credit values save without complaint; `credit_role` saves as free-text.

## Phase 3 — Contact / Organization soft-credit display

**Goal:** the existing Contact donations panel surfaces soft-credits received; the same for Org-side if such a panel exists, otherwise a separate Org-side soft-credits panel.

**Steps:**

1. **Locate the Contact donations panel** — likely in `app/Filament/Resources/ContactResource/Pages/...` or a partial referenced by ViewContact. The verify-at-start step produces the canonical site.
2. **Extend the data source** to include `softCreditsReceived()` (eager-loaded via `with('softCreditsReceived.donation')`).

   - Soft-credit rows tagged with "(soft-credit, X%)" — small inline indicator next to the amount.
   - Direct-donation rows render the amount as-is.

4. **Org-side parity:**

**Pass criterion:** seeded contact with one direct donation + one soft-credit shows both rows in the panel, soft-credit row tagged; the Org case mirrors per its panel's existence post-264.

## Security checklist

Before closing:

- [ ] **Soft-credit attribution does not leak donor identity across contacts.** A soft-credit on Contact A's panel showing "donation made by Contact B" should respect existing donor-privacy semantics (whatever they are; check whether the Donation admin already gates donor identity behind permission). If donor identity is gated, the soft-credit display inherits the gate; otherwise no new gate is introduced. Verified by reading the Donation resource's permission posture before authoring the panel extension.
- [ ] **No new env vars introduced.**
- [ ] **No new privileged endpoints.** All soft-credit surfaces are inside the admin authentication scope.
- [ ] **Polymorphic morph-map registered if the codebase uses one.** Verify-at-start determines this; if morph-map is used, Contact + Organization land in the map. If not in use, default class-name persistence is acceptable for v0.
- [ ] **`credit_pct` accepts values >100** — explicitly tested; not a security issue, but a contract decision easy to regress.
- [ ] **No sum-validation introduced** — operator owns the invariant per design decision; do not add a validator that checks against donation amount or 100%.

## Out of scope

- **Sum-of-soft-credits validation against donation amount.** v0 lets the sum exceed donation amount (matching-gift cases) and lets the sum be less than 100% (partial attribution). Operator discipline owns the invariant.
- **Soft-credit-aware acknowledgment / year-end statements.** C4 territory.
- **Donations importer soft-credit support.** No real-world data driver yet; lift if surfaced.
- **Soft-credit reporting / aggregations** ("total soft-credited to Contact X this year"). Defer until a real reporting driver appears.
- **Soft-credit on Donation list-view column.** Defer until a UX request surfaces.
- **Org-Org soft-credit chains** (e.g., a fiscal-sponsor org receiving soft-credit on a sponsored project's gift). Org-Org relationships are out-of-scope at B1b entirely.
- **Promoting Organization to a full donations panel.** If Org doesn't have one post-264, this session adds only a soft-credits panel; the broader Org-donations-panel question lifts to a follow-on session if a driver appears.
