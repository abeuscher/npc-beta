## What was built

Behaviour-preserving legibility split of the two >1k-line site-bundle files into a
slim orchestrator + per-domain collaborators, along their mirror seams. No feature,
schema, or contract change — the existing round-trip suite is the net. The owner
chose a **clean split** (genuine injected collaborator classes) over a light
trait/file-move, on both sides.

Both classes are resolved via `app(...)` and callers touch only their public
surface (`export*` / `import` / `analyze`, the `FORMAT_VERSION` + SiteSettings
policy constants), so the public surface and class names are preserved; only the
internals were reorganised. Collaborators live in `app/Services/ImportExport/Export/`
and `app/Services/ImportExport/Import/`, mirroring the flat single-responsibility
layout of `app/Services/Media/`. Container auto-resolves the whole graph (no
bindings needed).

**Handling the importer's shared per-call state** (the structural reason the
importer is harder than the stateless exporter): the zip media-root + `archiveFile`
resolution and the `replaceDuplicatePages` flag, previously held on `$this` for the
duration of one `import()` call, are now threaded explicitly — a fresh
`BundleMediaArchive($mediaRoot)` is constructed per call and passed to the
collaborators that rewire media; `replaceDuplicatePages` is a method parameter on
`PageImporter`. The author fallback stays lazy (resolved only in create branches,
so a media-/design-only import on a userless install still doesn't throw).

The 345/2 `EventRegistration::withoutEvents()` wrap (imported registrations are
historical data — suppress the confirmation-email/contact-auto-create observer)
travels intact with `importEvent` into `EventImporter`.

### Shared policy lift

The SiteSettings export/import allow/deny policy (allow-list + deny prefixes/regex/
named + `isDenied()`) moved from `ContentImporter` to a shared
`App\Services\ImportExport\SiteSettingsBundlePolicy` (118 LOC), consumed by both the
export `SiteSettingsSerializer` and the import `SiteSettingsImporter`. This removes
the prior cross-class coupling where the exporter reached into
`ContentImporter::SITE_SETTINGS_ALLOW_LIST`. No external callers referenced those
constants, so the move is internal-only.

### Minor cleanups (behaviour-preserving)

- Dropped an unused `$itemsById` local that was assigned but never read in the old
  `serializeNavigationMenu`.
- `serializePage` now passes the already-loaded `$page` to the widget-tree
  serializer instead of re-finding it by id — identical output, one fewer query.

## Security considerations

No security surface changed. The SiteSettings allow/deny boundary is byte-identical
after the lift to `SiteSettingsBundlePolicy` (lists copied verbatim; `isDenied()`
logic unchanged) and is still enforced on **both** sides (exporter pre-filter +
importer inbound re-check — belt-and-suspenders preserved). The rich-text
`HtmlSanitizer` sanitisation at the widget-config and SiteSettings import seams,
the media path-traversal guards, and the `encrypted`-type hard-skip all moved
verbatim into their collaborators. No new trust boundary; no new external input
path. This was a pure relocation — the round-trip + sanitisation test suites are
the proof.

## Production deployment notes

None beyond standard `migrate` for this split itself.

**Context — the 345/2 deploy-brick fix shipped during this session.** Mid-session
the FM-side surfaced a production incident: the 0.341.01 → 0.345.06 upgrade bricked
a live node (and the CRM auto-deploy server) with `Class
"Barryvdh\Debugbar\ServiceProvider" not found` — a stale package-discovery manifest
on the persistent `bootstrap/cache` volume (session 342 removed laravel-debugbar)
shadowed the new image's manifest, fataling at framework bootstrap (web + artisan).
Fixed on a separate branch (**`session-345/2`**, VERSION **0.345.07**, merged to
main and deployed by the owner): `docker/entrypoint.sh` now clears
`packages.php`/`services.php` before launch (load-bearing, image-baked, self-applies
on the delivering upgrade, covers app + worker); a baked `HEALTHCHECK` runs a new
`app:health-check` artisan command (boots the framework + DB ping) so `compose up
--wait` catches a non-booting image at `docker_up` instead of false-greening into
migrate, letting FM's existing auto-rollback handle it; the `bootstrap_cache`
volume was dropped from prod compose; and `BootstrapCacheVolumeTest` guards both
against regression. Not the FM `/api/health` contract endpoint — `HealthController`
untouched, contract stays v2.3.0. That branch was merged into `session-346/1` to
keep this branch current, so the 346 close commit sits on top of it.
