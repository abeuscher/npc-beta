# 242. Fleet Manager Agent — Phase 2 (Backup Pipeline)

Land the greenfield CRM-side backup mechanism (none exists today). Install `spatie/laravel-backup`, configure for `pg_dump` + media library, target a per-install DigitalOcean Spaces bucket, write each successful backup's timestamp to a known location, and flip `HealthController::checkLastBackupAt()` from the v1.1.0 hardcoded `unknown` to threshold-driven (`green` < 24h, `yellow` 24–36h, `red` > 36h, `unknown` if no successful run yet). Bump the agent contract `v1.1.0 → v1.2.0` (additive — `last_backup_at.value` becomes ISO 8601 timestamp, `threshold` becomes `[24, 36]`).

---

## Design decisions (resolved before starting)

- **Backup engine: `spatie/laravel-backup`.** Battle-tested; same author family as `spatie/laravel-medialibrary` and `spatie/laravel-permission` already in use. Handles `pg_dump`, tarballing, multi-disk upload, retention pruning, and emits `BackupWasSuccessful` / `BackupHasFailed` events out of the box. Rolling our own reproduces it badly.
- **Lives in `require`, not `require-dev`.** Backups run in production. Adds about ~7 transitive packages; no runtime cost when the scheduler isn't firing.
- **Storage target: DigitalOcean Spaces, one bucket per install.** Decided during 241 close. DO Spaces is pooled-storage-priced ($5/month for first 250 GiB across all buckets, up to 100 buckets per subscription), so isolation wins on simplicity at zero marginal cost. Each install gets its own bucket (e.g., `nonprofitcrm-{install-handle}-backups`) and its own scoped Spaces access key. A leaked credential reaches one bucket, not the fleet. Documented per-install setup procedure in `docs/app-reference.md`.
- **Encryption at rest: server-side via Spaces SSE.** Default for Spaces. Sufficient for v1; client-side encryption was the original planning-spec lean but introduces key-management complexity (rotation, recovery, key-loss = data-loss) that does not pay back at one or two installs. Revisit if a client demands end-to-end.
- **Success-record location: `storage/app/fleet/last-backup-at`.** Flat file holding an ISO 8601 timestamp. Symmetric with the VERSION-file pattern. Lives in Laravel's storage tree (already a persistent volume in production, bind-mounted in local dev), readable by both the app container and the worker container without extra wiring. Atomic write via `Storage::disk('local')->put('fleet/last-backup-at', $iso)` — Laravel handles temp-file + rename under the hood. No Redis dependency, no DB row, no schema work.
- **Cadence: daily.** Tunable via the spatie config. The scheduler runs `backup:run` and `backup:clean` once per day each.
- **Retention: 14 days.** `keepAllBackupsForDays = 7`, `keepDailyBackupsForDays = 14`, `deleteOldestBackupsWhenUsingMoreMegabytesThan = 5000` — spatie's tunable knobs. 14 days lets us see drift across a typical reporting window without unbounded growth.
- **Threshold: `green` < 24h, `yellow` 24–36h, `red` > 36h.** Pinned by the planning spec and Phase 2 forward plan in the track doc.
- **Local dev backups run to local disk only.** `BACKUP_DISKS` env var defaults to `local` in `.env.example`; production sets it to `spaces`. This way `php artisan backup:run` works in local dev without requiring Spaces credentials, and the production-only path is a single env-var flip. Multi-disk targets (e.g., `local,spaces` for redundancy) remain available but unused at v1.
- **Tests do not hit real Spaces.** `Storage::fake()` for any test that touches a disk. Real-Spaces verification is the manual test step.
- **No Filament UI for backup-job status, no admin-side backup-history table.** Backup state is observed via `last_backup_at` on the agent endpoint. Operator-side observation is Fleet Manager's job; this session only emits the surface FM polls.
- **No backup-restore tooling in this session.** Restoration is a `pg_restore` + tarball-extract operation the operator performs manually against a downloaded blob. spatie/laravel-backup does not ship restore tooling; that is correct for a v1 backup pipeline (restore is a high-stakes manual operation, not a click-button feature).
- **Boundary-touching by design.** Contract spec doc bumps `1.1.0 → 1.2.0`. `unknown` semantic refines: was "no pipeline at all" (v1.1.0), now "pipeline exists but no successful run yet" (v1.2.0). FM-side already treats `unknown` as yellow-tier — the existing alerting behavior is unchanged. The CHANGELOG documents the semantic refinement explicitly.

## Phase 2 — Backup-success listener (writes the timestamp file)

1. **Create `app/Listeners/RecordBackupSuccess.php`.** A simple invokable listener:
   ```php
   <?php

   namespace App\Listeners;

   use Illuminate\Support\Facades\Storage;
   use Spatie\Backup\Events\BackupWasSuccessful;

   final class RecordBackupSuccess
   {
       public function handle(BackupWasSuccessful $event): void
       {
           Storage::disk('local')->put('fleet/last-backup-at', now()->toIso8601String());
       }
   }
   ```
   No constructor injection (pure-static-call shape). The `local` disk is Laravel's default `storage/app` disk, so the file lands at `storage/app/fleet/last-backup-at`.

2. **Register the listener.** Laravel 11 favors event-discovery via `EventServiceProvider` or auto-discovery from listener method signatures. Confirm the project's posture by reading `app/Providers/AppServiceProvider.php` and `bootstrap/providers.php`. If auto-discovery is on (the default in Laravel 11), the listener registers itself by virtue of the `BackupWasSuccessful` type-hint — no further wiring needed. If explicit registration is in use, add the mapping to whichever provider holds it.

3. **Optional: handle `BackupHasFailed`.** Out of scope by design. Failed backups produce no timestamp update, so `last_backup_at` ages naturally and the agent endpoint's threshold logic flips to yellow then red. That is the intended observation path.

## Phase 3 — `HealthController::checkLastBackupAt()` rewrite

Replace the current body (which returns hardcoded `unknown` / null / `'backup pipeline not yet implemented'`) with threshold-driven logic.

```php
private function checkLastBackupAt(): array
{
    $threshold = [24, 36]; // hours: green < 24, yellow 24-36, red > 36

    $path = storage_path('app/fleet/last-backup-at');

    if (! is_readable($path)) {
        return [
            'status'    => 'unknown',
            'value'     => null,
            'threshold' => $threshold,
            'message'   => 'no successful backup yet',
        ];
    }

    $iso = trim((string) @file_get_contents($path));
    if ($iso === '') {
        return [
            'status'    => 'unknown',
            'value'     => null,
            'threshold' => $threshold,
            'message'   => 'last-backup-at file is empty',
        ];
    }

    try {
        $timestamp = \Illuminate\Support\Carbon::parse($iso);
    } catch (\Throwable $e) {
        return [
            'status'    => 'unknown',
            'value'     => null,
            'threshold' => $threshold,
            'message'   => 'last-backup-at file unparseable',
        ];
    }

    $hours = $timestamp->diffInHours(now(), absolute: true);
    $status = match (true) {
        $hours > $threshold[1] => 'red',
        $hours >= $threshold[0] => 'yellow',
        default => 'green',
    };

    return [
        'status'    => $status,
        'value'     => $timestamp->toIso8601String(),
        'threshold' => $threshold,
        'message'   => null,
    ];
}
```

The threshold values are hard-coded as a pair (no env-var override at v1; Fleet Manager observes the same boundaries across all installs). The `value` is the ISO timestamp the file holds, normalized through Carbon (so an off-format file still emits a clean ISO string in the response). The `unknown` cases are explicit — file-missing, file-empty, file-unparseable — each with a different `message` so operator debugging is straightforward.

Bump the constant: `CONTRACT_VERSION = '1.1.0'` → `'1.2.0'`.

The worst-of derivation in `index()` does **not** change. `unknown` continues to rank as yellow-tier exactly as v1.1.0 set it.

## Phase 8 — Documentation

Update `docs/app-reference.md`. Add a new section after the existing "Build server — public asset pipeline" section, before "Dev tooling — widget thumbnails":

```
## Backups

CRM data is backed up nightly to a per-install DigitalOcean Spaces bucket. The pipeline is `spatie/laravel-backup` invoked by the Laravel scheduler.

### What gets backed up

- The `nonprofitcrm` Postgres database (via `pg_dump`).
- The Spatie media library tree under `storage/app/public`.

### Per-install bucket setup procedure

Each install gets its own DigitalOcean Spaces bucket and its own scoped access key. Setup steps for a new install:

1. **Create a Space** (DO console → Spaces → Create) with a name like `nonprofitcrm-{install-handle}-backups`. Region should match the droplet's region for lowest latency.
2. **Generate a bucket-scoped Spaces access key** (DO console → API → Spaces Keys → Generate New Key, with the bucket scoped to the new bucket and permission `Read/Write/Delete`).
3. **Set production env vars** in the install's `/opt/nonprofitcrm/.env`:
   - `SPACES_KEY=<scoped key>`
   - `SPACES_SECRET=<scoped secret>`
   - `SPACES_BUCKET=<bucket name from step 1>`
   - `SPACES_REGION=<region slug, e.g. nyc3>`
   - `SPACES_ENDPOINT=https://<region>.digitaloceanspaces.com`
   - `BACKUP_DISKS=spaces`
4. **Restart the worker container** so the scheduler picks up the new env. The first scheduled `backup:run` (next 01:30 UTC) writes the first blob to the bucket and updates `storage/app/fleet/last-backup-at`.
5. **Verify** by hitting `/api/health` with the bearer token after the first backup runs — `last_backup_at.status` should be `green` with a recent ISO timestamp.

### Local dev posture

Local dev defaults to `BACKUP_DISKS=local`, which writes backups to `storage/app/backups/`. No Spaces credentials are required to run `php artisan backup:run` locally for testing. Each manual run also writes `storage/app/fleet/last-backup-at`, so local `/api/health` responses reflect a green `last_backup_at` once a manual run has happened.

### Restoration (manual)

spatie/laravel-backup does not ship restore tooling — restoration is a manual operator operation:

1. Download the most recent backup blob from the install's Spaces bucket.
2. Extract: `tar -xzf <backup>.zip` (it's a zip despite the .tar.gz convention spatie uses internally — confirm at restore time).
3. Restore database: `pg_restore --dbname=postgresql://... --clean --if-exists <dump>.sql`.
4. Restore media: copy the extracted `storage/app/public` tree onto the target install's storage volume.

This is intentionally manual at v1 — backup restoration is high-stakes and benefits from the operator pausing to verify each step rather than a click-button shortcut.
```

The section is positioned between the build-server pipeline and the widget-thumbnail tooling because backups are infrastructure-adjacent and operator-facing.
