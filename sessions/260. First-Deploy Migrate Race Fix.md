# 260. First-Deploy Migrate Race Fix

Add a postgres healthcheck to `docker-compose.prod.yml` and switch both deploy workflows to `docker compose up -d --wait` so first-deploy `migrate --force` runs against a postgres container that's actually ready, eliminating the documented manual-migrate workaround in the install runbook.

---

## Design decisions (resolved before starting)

These need to be confirmed at session start during the read-through.

- **Healthcheck shape — `pg_isready -U $DB_USERNAME -d $DB_DATABASE`.** Standard postgres-image health probe. Available out of the box (`pg_isready` ships with the postgres binaries inside `postgres:16-alpine`); no extra deps. Returns 0 only when postgres is actually accepting connections — not just when the process has started. Verifies the actual root cause of the first-install race (postgres process up but data-dir init still in flight). Confirm before going deep.

- **Healthcheck scope — postgres only.** The documented race is specifically postgres-side (data-dir init on first volume mount). Redis race is theoretically possible (`redis:7-alpine` first-time RDB snapshot init) but hasn't surfaced in any session log; the deploy workflow doesn't immediately touch redis after `up -d`. Adding a redis healthcheck is out of scope unless the user agrees otherwise at session start.

- **`--wait` scope — both workflows.** Both [`.github/workflows/deploy.yml`](../.github/workflows/deploy.yml) and [`.github/workflows/deploy-demo.yml`](../.github/workflows/deploy-demo.yml) have the same `up -d --remove-orphans` → `migrate --force` shape. Both pick up `--wait` in this session.

- **`depends_on` enrichment.** Today's `app` service has bare `depends_on: [postgres, redis]` (start-order only). With a postgres healthcheck in place, the `app` service can opt into `depends_on: postgres: condition: service_healthy` so `docker compose up` itself blocks the app from starting until postgres is healthy. This is in addition to `--wait`, not instead of — `--wait` blocks the workflow until services are healthy; `condition: service_healthy` blocks app's startup ordering inside the compose graph. Both belt and suspenders. Apply to `app` and `worker` (both wait on postgres).

- **Verification surface — operator's call.** Existing droplets are past the first-install race; the only meaningful test is a fresh first-install. Two options:
  - **(a) Fresh test droplet** — provision a one-shot droplet on DO, run the install runbook against it, confirm `migrate --force` lands tables on first attempt without manual `migrate:fresh`. ~15 min provisioning + ~10 min walkthrough. Clean, throwaway. Preferred if operator has time.
  - **(b) Demo droplet teardown + redeploy** — `docker compose down -v` on the demo droplet (DESTRUCTIVE — drops the demo DB and named volumes), then redeploy via push to `release/public-demo`. Faster but wipes demo data; coordinate explicitly first.
  Default lean: (a) unless operator pre-decides (b).

- **Out of scope this session:**
  - Redis healthcheck (per scope above).
  - Switching nginx conf to a directory bind-mount (separate carry-forward, related but orthogonal).
  - Workflow migrate-step retry logic (the healthcheck makes retry unnecessary).
  - Any application-level changes (no Pest tests added; no model changes; no controller changes).
  - Fleet Manager contract changes.
  - The dev-environment `docker-compose.yml` (this session is prod-shape only — local dev doesn't have the race because `docker compose up -d` waits implicitly when the volumes already exist).

## Phase 2 — Implementation

**Goal:** Add the healthcheck and the `--wait` flag.

**Steps:**
1. Edit [`docker-compose.prod.yml`](../docker-compose.prod.yml):
   - Add `healthcheck:` block to the `postgres` service with `pg_isready` shape.
   - Enrich `app.depends_on` from `[postgres, redis]` to the long-form with `condition: service_healthy` on postgres.
   - Same enrichment on `worker.depends_on`.
2. Edit [`.github/workflows/deploy.yml`](../.github/workflows/deploy.yml):
   - Flip `docker compose -f docker-compose.prod.yml up -d --remove-orphans` → `docker compose -f docker-compose.prod.yml up -d --wait --remove-orphans`.
3. Edit [`.github/workflows/deploy-demo.yml`](../.github/workflows/deploy-demo.yml):
   - Same flip.
4. Local sanity check:
   - `docker compose -f docker-compose.prod.yml down -v` (wipes local volumes — only do this on local dev).
   - `docker compose -f docker-compose.prod.yml up -d --wait` — confirm it blocks until postgres reports healthy.
   - `docker compose -f docker-compose.prod.yml exec -T app php artisan migrate --force` — confirm tables land first-attempt.
   - Tear down again to leave local dev clean.

**Pass criterion:** the local sanity check succeeds. Workflows pass YAML lint (`gh workflow view deploy.yml` or local `yamllint`). No application-level test regression.
