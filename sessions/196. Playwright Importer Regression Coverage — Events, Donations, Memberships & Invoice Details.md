# 196. Playwright Importer Regression Coverage тАФ Events, Donations, Memberships & Invoice Details

Extend the Playwright harness from session 195 with eight new specs тАФ one happy-path and one update-strategy spec per non-contact importer тАФ giving end-to-end UI regression coverage of the session-194 update-strategy work across all five importers.

---

## Design decisions (resolved before starting)

- **Scope: happy-path + update-strategy per importer, only.** Eight specs total. PII-rejection, duplicate-header-review, error-report, and invoice-details multi-row-grouping specs are out of scope for this session and will land in a follow-up.
- **Fixtures: handcrafted per importer, PII-free, small.** Five to ten rows each. Realistic-but-fake data (no client CSVs). The upcoming "Random Data Generator тАФ CSV Export" session will replace these with generator output; for now they live under `tests/e2e/fixtures/{importer}/` alongside their `expected.json`.
- **Testid naming: shared where the UI is shared.** The `map-column-{n}` testid moves into the shared `buildNamespacedMappingRow()` helper so all four non-contact wizards inherit it in one place. The two match-key dropdowns on each non-contact wizard use `import-match-key` (primary) and `import-contact-match-key` (contact lookup).
- **Duplicate-strategy radio: always explicitly checked.** Session 195 discovered that Filament's `->default('skip')` is not reliably applied through the wizard advance helper; the spec always calls `.check()` on the chosen strategy radio. Same pattern here.
- **Existing source selection: native `<select>`.** Uses `selectOption({ label })` тАФ same helper as contacts update-strategy spec.
- **Each spec file resets the DB + re-logs in via `resetAndLogin(browser)` in `beforeAll`.** Session 195 established this pattern because `migrate:fresh --seed` wipes server-side sessions; non-negotiable.
- **DB assertions count within the import session.** `TransactionSeeder` seeds 10 contacts, donations, memberships, and transactions during every `migrate:fresh --seed`. All counts must filter by `import_session_id` to isolate what the spec imported vs what was seeded.
- **One spec = one scenario.** No combined happy+update in a single spec. Keeps failure attribution clean and matches session 195 convention.
