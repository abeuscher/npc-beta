## Session-specific deltas

- **Two pieces of work, one session:** (1) retire `EventCalendar` completely; (2) build the new mini-calendar widget. They share the same surface (the events module's calendar story), so they land together.
- **The retirement is total, not a demotion.** `EventCalendar` is removed — not made admin-only. The back-end calendar use case (scheduling, recurring events) is deferred to post-Beta scheduling work. Sweep every reference (see `demo-site-plan.md` §4).
- **The new widget is server-rendered, vanilla, no Vue.** Carbon does all date math; a small `script.js` does only scroll-to-day + month-nav. vue-cal was evaluated and rejected at 324 — do not reintroduce a Vue runtime on the public side.
- **This widget is the events-index right-rail component** from the index spec: month grid, event-density indicators (dot/fill, not titles), click-a-day → scroll to that day's list section, month-nav arrows, desktop-only (hides on narrow viewports). It is **navigation, not filtering** — picking a day does not collapse the list.
- **Lay down the `data-tour` anchor** (`demo-site-plan.md` §2c) — e.g. `data-tour="events-index.mini-calendar"`. No tour code this session.

## Open questions to resolve at session start

- **widget_types removal mechanism.** Remove the `event_calendar` row via a migration or a seeder edit? And do existing demo/dev databases need a data migration to drop already-placed `event_calendar` widget instances (so they don't render against a missing definition)? Lean: a small migration that removes the `widget_types` row and any orphaned placed instances, so `migrate` cleans existing databases; re-verify the seeder no longer recreates it.
- **Month-nav mechanism.** Endpoint-fetch (a tiny server route returns the requested month's Carbon-computed grid) vs pre-render a ±1-month window toggled client-side. Lean: pre-rendered ±1-month window for v1 (no new route, no JS date math, covers the near-term-clustered demo events); endpoint-fetch only if arbitrary deep navigation is wanted. The spec says "do not auto-load events from other months on this view," so a narrow window is consistent.
- **Widget handle / label / placement.** Confirm the new widget's handle (e.g. `event_mini_calendar`), label, category (`events`), and that it's placed in the index right-rail column (a column, not a modification of `EventsListing`).

## Phases

### Phase 1 — Retire EventCalendar

Sweep the full surface from `demo-site-plan.md` §4: the widget folder, the `WidgetServiceProvider` registration, the `jcalendar` lib in `AssetBuildService`, the `public.js` imports, the `widget_types` seeder row, any `DemoBaselineSeeder`/seeded page placing it, and the six referencing test files. Mirror the existing widget pattern (`EventsListing` / `ThisWeeksEvents`): a `WidgetDefinition` (handle, label, schema, defaults, data contract for the events query, assets), a Blade template rendering the Carbon month grid with density indicators, and a small vanilla `script.js` for scroll-to-day + month-nav. Add the `data-tour` anchor. Register it in `WidgetServiceProvider` and the `widget_types` seeder.
