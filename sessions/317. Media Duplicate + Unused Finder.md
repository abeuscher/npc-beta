# 317. Media Duplicate + Unused Finder

Admin-tool build. A media-library scanner with two modes: duplicate-detection and unused-media. Outputs a triageable report; destructive operations are operator-confirmed.

---

## Plan reference

This session executes `sessions/release-plan.md` § E20 (execution-order position 46). The plan entry is canonical for scope and milestone-block positioning. This prompt covers session-shape detail: the surface inventory, the rich-text parsing approach, the destructive-operation policy.

---

## Session-specific deltas

- **Scan surface coverage is load-bearing.** False positives on the "unused" scan (real references the scan misses) erode operator trust fast. Build the surface inventory deliberately — every place media gets referenced needs to be either in the scan or in an explicit "out-of-scope" list shipped with the tool.
- **Rich-text content is the hardest surface.** Quill JSON in `posts.content`, `pages.content`, widget rich-text config fields, and possibly other JSON columns may embed media references in `<img src>` attributes. The scan needs to parse the content (not just regex-grep) to extract references reliably.
- **Two scans, one page.** Both modes live on the same admin tool page with separate header actions. The duplicate scan is straightforward (hash-by-content or filename-similarity); the unused scan is the harder one.

---

## Phases

### Phase 1 — Surface inventory + scan engine scaffolding

Before any scan code: grep the codebase for every place a media reference can land. Build a list. Common shapes:

- Direct columns referencing media (`logo_media_id` on settings, hero `background_image_id`, etc.).
- JSON columns carrying media references inside config blobs (widget configs, page templates, layout appearance configs).
- Rich-text JSON columns embedding `<img>` tags or attachment shortcodes.
- Seeder fixtures (probably out-of-scope for "unused" but worth documenting).

Output: a `MediaReferenceInventory` class (or service) that enumerates every reference shape. Each shape has a method that returns a Collection of referenced media IDs given a row. The scan engine iterates over the inventory.

### Phase 2 — Unused-media scan

Given the inventory: walk every row that could reference media, collect the union of referenced media IDs, diff against the media library. Anything in the library not in the union is "unreferenced" (unused candidate). Produce the report row including: media ID, filename, file size, upload date, any guessed-context (e.g. "uploaded in same session as Page X"), preview.

### Phase 3 — Duplicate-media scan

Identify duplicates within the library. Options to evaluate at session start:

- **Content hash** — most accurate; requires reading each file (consider performance on large libraries).
- **Filename similarity + size match** — cheap; catches the common "uploaded the same file twice with different names" case.
- **Both, two passes** — most useful.

Output: clusters of N>1 media records that look like duplicates, with a preview and the inferred reason for the cluster.

### Phase 4 — Admin tool page UI

Tools-group page mirroring the existing admin-tool pattern (`SiteImportExportPage` is a recent reference). Header actions to run each scan. Narrative section explaining what each scan does + what's in the inventory + what's out-of-scope. Report rendering with per-row triage controls.

### Phase 5 — Destructive-operation policy

Per-row "delete" and "merge into" actions on the report. Both operator-confirmed via Filament action modals. Deletions are soft-delete by default (use spatie-media-library's soft-delete if available, or a custom "archived" flag). "Merge into" re-points all the duplicate cluster's references to a chosen canonical record, then archives the others.

### Phase 6 — Tests + sample-data run

Feature tests pinning: scan output shape; surface inventory completeness (every shape in the inventory is exercised by a test fixture); destructive operations require confirmation; merge-into rewires references correctly. Run the tool against the local sample data and capture the output for the log.

## Open questions to resolve at session start

- **Scan execution model.** Run synchronously on the page (loading spinner) or dispatch as a queued job and pick up the result? For a large media library the synchronous path may time out; the queued path is more work to build. Default leaning: synchronous v1; promote to queued if the local sample data shows a timeout.
- **Duplicate-detection algorithm.** Content-hash, filename-similarity, both? Default leaning: both, two passes.
- **Soft-delete vs. archive flag.** Does spatie-media-library expose a soft-delete primitive that integrates with its own URL resolution? If so, use it; if not, build a custom archive flag (simpler).
- **Rich-text-content parsing.** Use the Quill Delta parser (if the project already has one) or extract via DOM-parsing of the rendered HTML? Default: whichever the existing rich-text infrastructure already uses.
- **Out-of-scope surfaces.** Seeder fixtures, archived data, anything else explicitly excluded from the "unused" scan. List them in the tool's narrative so operators understand what the scan does and doesn't cover.

---

## Out of scope

- Bulk-deletion shortcuts that bypass per-row confirmation.
- Automatic merging of duplicates without operator review.
- Migration of media to a different storage backend (separate concern).
- Re-encoding / format conversion / image optimization (separate concern).
- Tracking media-reference history (which content used this media before it was changed) — out-of-scope for v1.
