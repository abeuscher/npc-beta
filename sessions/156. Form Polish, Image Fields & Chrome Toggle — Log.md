## What was built

### Phase 1 — Audit

Read-only investigation across the trait, both models, `SeoMetaGenerator`, the Hero seeder/blade, the Vue inspector, the Filament topbar render hooks, the existing user-menu full-width toggle, and every reference to `og_image_path`. Surfaced four findings that materially shifted scope, all resolved with the user before code changes:

- **Hero nav-color fields were never removed.** They still exist at [WidgetTypeSeeder.php:660-661](database/seeders/WidgetTypeSeeder.php#L660-L661) and have been there since session 122. The full chain (PageController/PostController/PagePreviewController → `View::share` → `public.blade.php` → `--nav-link-color` CSS custom properties → `_layout.scss`) is intact. The actual bug was a Vue inspector regression: [InspectorField.vue](resources/js/page-builder-vue/components/InspectorField.vue) treated `shown_when` / `hidden_when` as objects, but the documented contract (and the seeder data) is a string. `Object.entries('overlap_nav')` produced character-index pairs that never matched config keys, silently hiding the fields. Phase 2 became a one-file Vue fix.
- **The site-default OG image fallback already existed**, but in [public.blade.php:39-45](resources/views/layouts/public.blade.php#L39-L45) — meaning the meta tag was correct but the JSON-LD generated inside `SeoMetaGenerator` never picked it up. Confirmed with the user that the duplicated layout-level fallback would be removed once `SeoMetaGenerator` owned the chain.
- **Page-type thumbnail collection naming.** Confirmed the cleanest call: share the existing `post_thumbnail` collection across both Page-type and Post-type rather than introducing a new `page_thumbnail` collection.
- **Full-screen toggle wiring conflict.** The existing user-menu toggle is server-side (PHP session + POST round trip + page reload) while the sidebar collapse is client-side (Filament Alpine state). User chose the consolidation path: delete the user-menu item, the controller, the session-key render hook, and the route entirely; replace with a single client-side topbar button as the canonical control.

### Phase 2 — Vue inspector `shown_when` / `hidden_when` fix

- [InspectorField.vue:43-60](resources/js/page-builder-vue/components/InspectorField.vue#L43-L60) — replaced the object iteration with a string-contract check matching the documented behaviour: `shown_when: 'key'` means "field is visible when `config[key]` is truthy". Same for `hidden_when` (inverted).
- This restores all three previously-hidden fields: Hero `nav_link_color`, Hero `nav_hover_color`, and the Data Table `name_field` (and the Hero `min_height` `hidden_when` rule).
- No seeder, blade, or controller changes were needed — the rest of the chain was already correct.

- Post and Event use a 2-column author with an empty `Group` spacer in place of the missing template column to keep row 1 at exactly 12 columns.
- [CmsFormFields.php](app/Forms/Fieldsets/CmsFormFields.php) refactored from section helpers (`pageName`, `settings`, `tags`) into individual field helpers (`titleField`, `slugField`, `authorField`, `statusField`, `publishedAtField`). The status field keeps its `live()` + `afterStateUpdated` behaviour for auto-setting publish date.
- [TagSelect.php](app/Forms/Components/TagSelect.php) gains `select($type)` and `creator($type)` static methods so the tag selector and create-tag input can sit in independent grid columns. The existing `make($type)` method is preserved unchanged for ContactResource, ContentCollectionResource, and CollectionResource which still use the bundled form.

### Phase 7 — Full-screen toggle (consolidated single control)

- New Blade component [admin-fullscreen-toggle.blade.php](resources/views/components/admin-fullscreen-toggle.blade.php) — Alpine button with eye / eye-slash heroicons. State persisted to `localStorage['np-fullscreen']`. Click handler:
  - Adds/removes `np-fullscreen` class on `<html>`.
  - Calls `$store.sidebar.close()` / `$store.sidebar.open()` to drive Filament's existing sidebar Alpine store in lockstep.
  - Writes localStorage so the state survives reloads.
- Registered via `PanelsRenderHook::USER_MENU_BEFORE` in [AdminPanelProvider.php](app/Providers/Filament/AdminPanelProvider.php) — places the icon immediately to the left of the user menu.
- Inline `<script>` injected at `panels::head.end` reads localStorage and sets the `np-fullscreen` class on `<html>` before first paint, avoiding layout flash on page load.
- New CSS rule in [public/css/admin.css](public/css/admin.css): `html.np-fullscreen .fi-main { max-width: 100% !important; }`.
- **Removed:** the user-menu `layout-toggle` `MenuItem`, `App\Http\Controllers\Admin\ToggleFullWidthController`, the `/admin/toggle-full-width` route registration, the `MenuItem` import from AdminPanelProvider, and the old session-key `panels::head.end` render hook. Single source of truth for fullscreen state.

### Page builder dark mode (added mid-session at user request)

- New `Page builder dark mode (editor chrome only)` section in [public/css/admin.css](public/css/admin.css) — ~80 selectors targeting the Vue editor's chrome classes (toolbar, viewport bar, add-block dropdown, preview canvas wrapper, inspector panel/header/tabs/field labels, all field input components, color picker, image upload, button list, spacing/query/appearance controls, layout inspector, block list, apply-changes bar, confirm-delete modal).
- Triggered by Filament's existing `html.dark` class — no new toggle, no new state. Specificity (`html.dark .foo` = 0,2,1) beats Vue's scoped selectors (`.foo[data-v-x]` = 0,2,0) by tag specificity, so no `!important` is needed.
- The preview content (`.widget-preview-scope.np-site` and descendants) is deliberately untouched. The rendered widgets always show the public site's light styling regardless of admin theme — what editors see in the preview matches what visitors see.
- Apply Changes selectors corrected after first pass (the initial dark mode rules used `.apply-changes-button` but the actual class is `.apply-changes`).
