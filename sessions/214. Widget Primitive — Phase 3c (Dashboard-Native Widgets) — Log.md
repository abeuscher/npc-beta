## What was built

### Phase 1 — MemosDefinition widget + memos collection + `publicSurface` resolver fix

- `app/Widgets/Memos/MemosDefinition.php` — `handle: 'memos'`, `allowedSlots(): ['dashboard_grid']`, `acceptedSources(): [Source::HUMAN]` (explicit for grep). `dataContract(array $config)` returns a `SOURCE_WIDGET_CONTENT_TYPE` contract: version `1.0.0`, fields `['title', 'body', 'posted_at']`, `ContentType` handle `memos.entry` with fields `title/body/posted_at` and `accepts: [Source::HUMAN]`, `resourceHandle: 'memos'`, `filters: ['limit' => int $config['limit'] ?? 5]`.
- `app/Widgets/Memos/template.blade.php` — simple list of memos with per-memo title, formatted `posted_at` (via `Carbon::parse`), and raw body (HTML from rich_text). Empty state: "No memos posted yet — visit Collection Manager ▸ Memos to add the first."
- `database/seeders/MemosCollectionSeeder.php` — idempotent `Collection::firstOrCreate(['handle' => 'memos'], …)`. `name: 'Memos'`, `source_type: 'custom'`, `is_public: false`, `is_active: true`, `accepted_sources: [Source::HUMAN]`, `fields: title (text, required) / body (rich_text, required) / posted_at (date, optional)`. Wired into `DatabaseSeeder::run()` after system collections.
- **`publicSurface` resolver fix** — the session's load-bearing architectural move. Adding a `bool $publicSurface = true` constructor field to `App\WidgetPrimitive\SlotContext`. `DashboardGridSlot::ambientContext()` now constructs its context with `publicSurface: false`. `ContractResolver::resolveWidgetContentType()` reads the flag: when `publicSurface === true`, the collection query applies `is_public = true AND is_active = true` (existing behavior); when `publicSurface === false`, it drops the `is_public` clause and only requires `is_active = true`. Cache key namespaced by scope (`collection:public:<handle>` vs `collection:any:<handle>`) so a public-surface widget and an admin-surface widget that both read the same handle don't cross-contaminate. This is what unlocks the admin-only Memos collection rendering on the dashboard without ever being exposed to public surfaces.
- **`SOURCE_WIDGET_CONTENT_TYPE` limit filter** — `ContractResolver::resolveWidgetContentType()` now honors `$contract->filters['limit']` by applying `->take(N)` to the per-handle cached collection in memory. Preserves the per-handle batching cache (two widgets reading the same collection with different limits both hit the DB once).
- `app/Providers/WidgetServiceProvider.php` — `MemosDefinition` registered alongside the other widgets.
- Tests:
  - `tests/Unit/Widgets/MemosDefinitionTest.php` — 5 tests: allowedSlots, acceptedSources, contract shape, config-driven limit, limit default.
  - `tests/Feature/MemosCollectionSeederTest.php` — 3 tests: seeder produces admin-only human-only policy, expected field keys, idempotency.
  - `tests/Feature/MemosDashboardRenderTest.php` — 4 tests: renders items on `dashboard_grid` despite `is_public=false`, returns empty for the same contract on a public-surface slot, honors `limit` filter, renders the empty-state copy.
  - `tests/Unit/WidgetPrimitive/Slots/DashboardGridSlotTest.php` — added a test pinning `ambientContext()->publicSurface === false`.

### Phase 2 — QuickActionsDefinition widget (closed-set internal-URL registry)

- `app/Widgets/QuickActions/QuickActionsDefinition.php` — `handle: 'quick_actions'`, `allowedSlots(): ['dashboard_grid']`, `acceptedSources(): [Source::HUMAN]`, `dataContract(array $config): null` (static config; no read-side contract). `schema()` exposes a single `actions` field of type `checkboxes` whose options come from the registry. `defaults(): ['actions' => ['new_contact', 'new_event', 'new_post']]`.
- Public static `actionRegistry()` returning closed-set map:
  - `new_contact` → `ContactResource::getUrl('create')`
  - `new_event` → `EventResource::getUrl('create')`
  - `new_post` → `PostResource::getUrl('create')`
  - `new_form` → `FormResource::getUrl('create')`
  - **`new_donation` dropped** — mid-session discovery: `DonationResource::canCreate()` returns false and the resource does not register a `create` page. Donations enter via the payment system, never by admin form. Keeping `new_donation` in the registry would have 404'd.
- `template.blade.php` — selected actions render as a vertical list of links; URLs resolved lazily from the registry closures at render time (no persisted URLs). Unknown action keys are silently skipped. Empty selection renders "No actions selected."
- `app/Providers/WidgetServiceProvider.php` — registered.
- Tests:
  - `tests/Unit/Widgets/QuickActionsDefinitionTest.php` — 6 tests: allowedSlots, acceptedSources, `dataContract` returns null, single-field schema, defaults, registry entry shape (label/url/icon keys, `url` is a Closure — no string concat).
  - `tests/Feature/QuickActionsDashboardRenderTest.php` — 4 tests: renders anchor to Contact create URL, empty-state copy on empty selection, every registry URL begins with the app's admin path (open-redirect guard), unknown action keys do not break render.

### Phase 3 — ThisWeeksEventsDefinition + `ContractResolver` Event support

- `app/Widgets/ThisWeeksEvents/ThisWeeksEventsDefinition.php` — `handle: 'this_weeks_events'`, `allowedSlots(): ['dashboard_grid']`, `acceptedSources(): [Source::HUMAN]`. Schema: one `days_ahead` field, number, default 7, advanced. `dataContract(array $config)` returns a `SOURCE_SYSTEM_MODEL` contract: version `1.0.0`, `model: 'event'`, `fields: ['id', 'title', 'slug', 'starts_at', 'ends_at', 'address_line_1', 'city', 'state', 'meeting_label']`, `filters: ['date_range' => ['from' => 'now', 'to' => '+N days'], 'order_by' => 'starts_at asc']`.
- **`location` field drop, address/meeting fields exposed instead.** Session prompt initially listed `location` — the Event model has no such column. Instead, the per-field fail-closed whitelist exposes `address_line_1`, `city`, `state`, `meeting_label` and the widget template composes the displayed location string (address or meeting label) without the projector doing derivation. Keeps the fail-closed discipline honest — no synthetic derived fields in the resolver.
- `template.blade.php` — list of events with formatted `starts_at` (via `Carbon::parse`), title, composed location line. Empty-state copy parameterized by `days_ahead`.
- **`ContractResolver` extension** — `resolveSystemModel()` now dispatches on `$contract->model`: `'post'` → existing `resolvePost()` helper, `'event'` → new `resolveEvent()` helper, anything else → `['items' => []]` (fail-closed on unknown model handles). The `resolveEvent()` helper applies `Event::published()`, parses `filters.date_range.from/to` via `Carbon::parse`, `whereBetween('starts_at', …)` when both bounds present (or one-sided `>=` / `<=`), applies a whitelisted `order_by` (allowed columns: `starts_at`, `ends_at`, `created_at`; defaults to `starts_at asc`), honors `limit`, caches on `'event:' . sha1(serialize($contract->filters))`.
- **`SystemModelProjector` extension** — `project()` now dispatches on `$contract->model` with a per-model projector closure. New `projectEvent()` returns a fixed map of fields: `id`, `title`, `slug`, `starts_at` (ISO8601), `ends_at` (ISO8601), `address_line_1`, `city`, `state`, `meeting_label`. Contract's `$fields` are looked up against this map — any field not in the map returns empty string (fail-closed). A contract requesting `internal_notes` or `price` gets empty strings.
- `app/Providers/WidgetServiceProvider.php` — registered.
- Tests:
  - `tests/Unit/Widgets/ThisWeeksEventsDefinitionTest.php` — 5 tests: allowedSlots, acceptedSources, contract shape, date_range translation from config, days_ahead default.
  - `tests/Feature/ThisWeeksEventsDashboardRenderTest.php` — 5 tests: renders events in window and omits those outside, renders empty-state, resolver returns only whitelisted fields (`internal_notes`, `price`, `description`, `author_id` all fail-closed), date_range clips, unknown system-model handle returns empty items.

## Deferred decisions

- **Dashboard widget arrangement UI + storage model for admin-arranged dashboards.** Drafted as session 215 (Phase 3d). Shape agreed during 214 close: reuse `page_widgets` with a new `DashboardConfig` polymorphic owner (honors the migration doc's locked-in "polymorphic owner discipline"); per-role ownership (not per-user); reuse the Vue page builder via a new Livewire shell with a `mode: 'dashboard'` field on `BootstrapData` — the first concrete step of Phase 6 convergence arriving naturally rather than as a later rewrite.
- **Memo composer widget.** Capability check during 214 close confirmed it is architecturally in range today: `DataSink::write(CollectionItem::class, Source::HUMAN, [...])` with the memos collection accepting HUMAN universally satisfies the security invariant without requiring a write-contract primitive. Not scheduled; can land as a standalone session after 215 if the Collection Manager → Memos authoring flow proves friction-laden.
- **`order_by` filter on `SOURCE_WIDGET_CONTENT_TYPE`.** Memos currently sort by `sort_order` (inherited from CollectionItem). The MemosDefinition contract does not declare `order_by`; ordering a JSONB field like `posted_at` would require either a generated column or a runtime projection. Left for the session that surfaces a concrete need.
- **`new_donation` quick action.** Dropped from the registry when `DonationResource::canCreate()` was discovered to be false. If donations ever gain an admin-authored creation path, reinstating the key is a one-line change.
- **Phase 4 widget retrofit.** Retrofitting the remaining ~20 widgets in `app/Widgets/*` onto declared contracts remains pending per the migration doc.
- **Per-field write contracts.** Still 2.0 per migration doc. Source policy remains the write-side gate.
- **Pre-existing `SeoTest` OOM mid full-suite run.** Unchanged from 212/213. `SeoTest` in isolation runs clean. Flagged for the eventual test-suite audit session.
