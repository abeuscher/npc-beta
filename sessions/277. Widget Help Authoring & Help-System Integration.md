## Session-specific deltas

The design questions from the stub resolved as follows, ratified at 276 close:

- **Where widget help lives:** as markdown files in `resources/docs/widget-{handle}.md`, with `standalone: true` frontmatter so they don't bind to any Filament route. They flow through the same `help:sync` pipeline as every other doc, get indexed in `help_articles`, and become searchable automatically.
- **How widget help surfaces:** through the existing standalone-help-article page (`HelpArticlePage`) — *not* through a new per-widget `?` affordance in the page-builder inspector. The page builder's existing `?` icon opens `cms-pages.md` (already wired); that doc gains a "Widget Help" callout near the top linking to a curated catalog.
- **Rollup story:** a single standalone `widget-catalog.md` doc renders a sortable HTML table of every widget that has a help doc. Hand-curated — when a new widget gets a doc, the author appends a catalog row. Decoupled from `widget_types` on purpose: avoids ghost rows for widgets without docs and keeps the catalog a curated index, not a generated mirror.

**Why this shape is the looseness target:** no new admin pages, no new help-system primitives, no per-widget config-schema introspection. Everything is markdown flowing through the existing `help:sync` → `help_articles` → search pipeline. The catalog is one new markdown file with a sortable table. The widget docs are 5 new markdown files. The page-builder help gets one paragraph added.

**First batch of five widgets to validate the pattern:**

- `bar_chart` — many `chart_config` knobs (axes, colours, data sources, label rules)
- `event_calendar` — filter rules, status/type-display config, view-mode toggles
- `donation_form` — amount-tier configuration, designation pickers, recurring-vs-once shape
- `web_form` — form-binding, submission-handling, success-page redirect
- `event_registration` — tier picking (pre-C2 the today-shape), capacity, attendee flow

Per the stub's "concrete examples to scope" line.

## Implementation shape

Three deliverables, all in `resources/docs/` plus a one-paragraph edit to `cms-pages.md`:

### 1. `widget-catalog.md`

Standalone help article. Frontmatter:

```yaml
---
title: Widget Catalog
description: A sortable index of every widget that has detailed configuration help.
tags: [widget, page-builder, catalog, cms]
category: cms
standalone: true
---
```

Body: a short intro paragraph, then a sortable HTML table. Columns: **Label** · **Category** · **Description** · **Help Doc** (link to the widget's article). Sorting is client-side Alpine — load with default sort by Label A→Z, click a header to re-sort. Five rows in this session; designed to grow.

If Alpine sorting turns out to be more code than warranted, fall back to a static HTML table with a manual sort order — search still works for "find the widget I want," and the table is short enough to scan visually.

### 2. Five widget help docs

Per the batch above. Each follows the same frontmatter shape:

```yaml
---
title: Bar Chart Widget
description: Configuring data sources, axes, colours, and chart-display options.
tags: [widget, page-builder, bar-chart, charts, cms]
category: cms
standalone: true
---
```

Body covers: what the widget does, the inspector tabs (Content / Appearance), each significant config field with its effect, common configuration patterns, and any gotchas surfaced by the codebase (e.g. for `bar_chart`, the dependency on a Collection with numeric fields; for `event_calendar`, the date-filter behaviour vs. event status).

Grounding content: read each widget's `WidgetDefinition::config_schema()` and the `widget_types` row to be sure the doc matches the actual inspector. This is the kind of drift the new templates ask us to fix-and-note rather than ask-and-wait.

### 3. `cms-pages.md` callout

Add a short section near the top — between "Pages" intro and "Page List" — with a callout-style block:

> **Widget Help.** For per-widget configuration help, see the [Widget Catalog](/admin/help/widget-catalog).

That's it. One paragraph, no other changes to `cms-pages.md`.

## Out of scope

- **Auto-generating the catalog from widget-*.md files at `help:sync` time.** Tempting but breaks the "everything is just markdown" mental model. Revisit only if the curated catalog starts going stale in practice.
- **Per-widget `?` affordances in the page-builder inspector.** Adds UI surface area for low marginal value once the catalog is one click away. Out of scope per the looseness principle.
- **Per-field tooltips inside the inspector.** Same reason. Help system is for help; inspector is for editing.
- **Authoring help docs for the other 33 widgets.** Five validate the pattern; the rest land incrementally as their complexity warrants.
- **Building a content-pipeline or auto-discovery tool** (per the stub's explicit out-of-scope line).
- **Search-ranking tweaks.** `HelpSearch` works the way it works; if widget docs surface at acceptable rank, leave it alone. If they don't, that's a separate session's worth of search-tuning work.
