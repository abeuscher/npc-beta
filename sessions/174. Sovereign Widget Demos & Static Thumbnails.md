# 174. Sovereign Widget Demos & Static Thumbnails

Stage 4.5 of the Sovereign Widget arc, phase one. Move the existing widget demo seeders into their owning widget folders, stand up a dev-only demo route that renders any widget in isolation at a fixed viewport, and produce static PNG thumbnails for a representative set of six widgets using a Playwright script that runs on the host (outside Docker). This session delivers the authoring surface and static-capture pipeline. Animated capture and the ffmpeg pipeline are deferred to a follow-up session.

---

## Design decisions (resolved before starting)

- **`DemoDataService` is out of the thumbnail path.** It stays where it is today (wired into the builder preview as a fallback) — removing it is a separate cleanup — but no new code in this session touches or depends on it. Thumbnails render widgets from real data (seeded collections) or from widget defaults, full stop.
- **Widget sovereignty extends to demo data.** The five existing demo seeders relocate from `database/seeders/` into their owning widget folders at `app/Widgets/{PascalName}/DemoSeeder.php` with namespace `App\Widgets\{PascalName}`. No central demo-data service, no parallel manifest, no shared registry — each widget that needs seed data ships its own.
- **`ProductDemoSeeder` is deleted, not relocated.** It creates domain `Product` models rather than collection items and serves two widgets (`ProductCarousel`, `ProductDisplay`). That's app-domain seeding pretending to be widget data, which is the opposite of sovereignty. When there are zero products, `ProductCarousel` and `ProductDisplay` render their disconnected / bugged state — that's honest and correct per the render-state rule. Delete `database/seeders/ProductDemoSeeder.php` and every reference to it; do not create a `DemoSeeder` in either product widget folder. If either widget ever needs curated demo content, the user decides then.
- **Widget render states are honest.** A widget has two legitimate states: functional (requirements met, real output) and disconnected (requirements unmet, widget's own warning). Thumbnails target the functional state. Widgets without seeder-backed data render from `defaults()` alone — whatever they produce is what their thumbnail shows.
- **No new `thumbnailMetadata()` method.** The outline's Stage 4.5 note proposed per-widget `thumbnailMetadata()` (viewport, interaction key, duration). Skip it. Static PNGs use one standard viewport (`800 × 500`). Animated capture is deferred; when it lands, the per-widget interaction signal will be designed then.
- **No new `demo.blade.php` in each widget.** The dev route renders each widget through the existing `WidgetRenderer` pipeline — same path the builder preview uses. No duplicate demo templates, no parallel authoring surface.
- **Dev routes gated to non-production.** `routes/dev.php` is conditionally required from `routes/web.php` only when `! App::environment('production')`. A `DevRoutesMiddleware` on the route group enforces the same check at request time as belt-and-braces. No auth required on the routes themselves — they're local-only already.
- **Playwright runs on the host, outside Docker.** The script is standalone Node, invoked from WSL2 against `http://localhost` (or whatever the app domain is). Chromium is not installed in the app container. Playwright is added as a host-level dev dependency, not a project `npm install`.
- **Script output lands inside the widget folder.** Static thumbnails write to `app/Widgets/{PascalName}/thumbnails/static.png`. Thumbnails are committed to the repo — they are part of the widget package, not build artifacts. Future animated outputs will sit beside them as `animated.mp4` / `animated.webp`.
- **Phase-1 widget set (6 widgets):** `text_block`, `hero`, `video_embed`, `logo_garden`, `event_calendar`, `donation_form`. Coverage rationale:
  - `text_block`, `hero` — defaults-only, no seeding needed.
  - `video_embed` — defaults-only, renders an iframe (representative of third-party embed widgets).
  - `logo_garden` — has a collection-item demo seeder already, exercises the seeder-relocation path and Spatie media resolution.
  - `event_calendar` — needs real `Event` models; exercises the "needs a small fixture" path.
  - `donation_form` — needs a `Fund` + `Campaign`; exercises the domain-model fixture path a second time with different requirements.
- **Fixtures for widgets without seeders.** `event_calendar` and `donation_form` need minimal seeded fixtures to render functionally. Treat their fixtures the same way as the relocated seeders: ship them at `app/Widgets/EventCalendar/DemoSeeder.php` and `app/Widgets/DonationForm/DemoSeeder.php`. Final count after this session: 6 widget-level demo seeders (4 relocated, 2 new, 1 deleted).
- **No Artisan wrapper command this session.** The workflow doc proposes `php artisan widgets:thumbnails` as a wrapper over the Node script. Skip it — running `node scripts/generate-thumbnails.js` directly is fine for dev use. Revisit if the direct invocation proves inconvenient.
- **Widget registration of demo seeders.** Each widget definition that has a demo seeder gets an optional `demoSeeder(): ?string` method on the base class returning the fully-qualified seeder class name (or `null`, the default). The dev-route controller calls this to decide whether to run the seeder before rendering. The Dashboard debug-generator `seedWidgetCollections()` method is rewritten to iterate the registry and call every widget's `demoSeeder()`, eliminating the hard-coded list.

## Phase 3 — Write two new demo seeders for widgets that need domain-model fixtures

- **`app/Widgets/EventCalendar/DemoSeeder.php`** — creates 3 published upcoming events at known dates (e.g. next Tuesday, next Friday, two weeks out). Idempotent (`updateOrCreate` on a stable slug prefix like `demo-event-N`). Enough for the calendar widget to render a populated month.

Both seeders follow the existing pattern (`Illuminate\Database\Seeder` subclass, `updateOrCreate` writes, no cleanup) — see any of the relocated seeders as a reference.

## Phase 4 — Dev-only demo route + minimal layout

- **`routes/dev.php`** — new file. One route: `GET /dev/widgets/{handle}` → `WidgetDemoController@show`.
- **`routes/web.php`** — conditionally require `dev.php` at the bottom:
  ```php
  if (! App::environment('production')) {
      require base_path('routes/dev.php');
  }
  ```
- **`app/Http/Middleware/DevRoutesMiddleware.php`** — aborts with 404 when `App::environment() === 'production'`. Applied to the route group in `dev.php` as a second layer.
- **`app/Http/Controllers/Dev/WidgetDemoController.php`** — `show(string $handle)`:
  1. Resolve definition via `WidgetRegistry::find($handle)`; 404 if missing.
  2. If `$def->demoSeeder()` is non-null, run it via `Artisan::call('db:seed', ['--class' => $def->demoSeeder(), '--force' => true])` (idempotent by design).
  3. Build a transient `PageWidget` instance (unsaved) with widget type + `defaults()` as config. If the widget has a `collections` slot and its seeder created a collection with a predictable handle, set `config['collection_handle']` accordingly.
  4. Render via `WidgetRenderer::render($pw)` and return the minimal layout with the rendered HTML inside.
- **`resources/views/dev/widget-demo.blade.php`** — minimal shell: DOCTYPE, `<head>` with the same widget CSS/JS bundle the public site uses (read `public/build/widgets/manifest.json` exactly as `layouts/public.blade.php` does), fixed `body` dimensions of `800px × 500px`, no nav/header/footer, no site-container wrapper. The widget is the only thing on the page. Wait — use the same `x-picture` component / asset-manifest conventions the public layout uses so the widget renders identically to production.

**Widget-to-collection-handle mapping** for widgets with seeders:

| Widget | Seeded collection handle (config key) |
|--------|---------------------------------------|
| `logo_garden` | `logo-garden-demo` (existing) |
| `carousel` | `carousel-demo` (existing) |
| `bar_chart` | `chart-demo` (existing) |
| `board_members` | `board-members-demo` (existing) |
| `event_calendar` | N/A — reads events directly |
| `donation_form` | N/A — reads funds/campaigns directly |

The controller only needs to set `collection_handle` for the four collection-based widgets. For the other two, defaults alone — plus the seeded fixtures — are enough.

## Phase 5 — Playwright capture script (host-side)

- **`scripts/generate-thumbnails.js`** — standalone Node script. CLI flags: `--all` (default), `--widget=handle`, `--base-url=http://localhost` (default; can override for an alternate dev server address).
- Script reads the widget list by calling a new Artisan command `widgets:manifest-json` (next bullet) via `docker compose exec app php artisan widgets:manifest-json`, parses the JSON, iterates.
- **`app/Console/Commands/WidgetsManifestJsonCommand.php`** — signature `widgets:manifest-json`. Outputs `app(WidgetRegistry::class)->manifests()` as JSON to stdout. One-liner body. The Playwright script is the only consumer in this session.
- For each widget:
  1. Launch Chromium at `viewport: { width: 800, height: 500 }`.
  2. `page.goto(baseUrl + '/dev/widgets/' + handle)`.
  3. `page.waitForLoadState('networkidle')`.
  4. `page.screenshot({ path: 'app/Widgets/{Folder}/thumbnails/static.png', type: 'png' })`.
  5. Log success/failure per widget.
- The script fails loudly if the dev route returns a non-200. Missing thumbnails are better than silently-broken ones.
- **Installation note in the prompt:** Playwright is installed on the host (WSL2) with `npm install --global playwright` + `npx playwright install chromium`. Not added to the project's `package.json`. Document this in `docs/widget-system.md`.
