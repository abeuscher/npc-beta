## What was built

### Phase 1 — Lifecycle foundation: handle §A4 refactor + §A14 MutationObserver

The store's active-editor handle was extended from the 305 stub `{ quill, widgetId, path }` to the full spec §A4 shape `{ quill, hostEl, widgetId, configPath, getRect }`. `useInlineEdit.activateRich` publishes the extended record on rich-text activation; plain-text fields remain non-publishing per §I4. The §A14 MutationObserver was added: while the handle is non-null, the preview-html container's direct children are observed for `childList` mutations with `subtree: false` — narrow enough to ignore Quill's own per-keystroke DOM churn, broad enough to detect a wholesale `v-html` replacement of the field's subtree. If the active host element leaves the document, the handle is cleared synchronously within the MutationObserver's microtask, structurally guaranteeing the toolbar can never be left bound to a torn-down editor.

### Phase 2 — ColorPicker arrow-key grid navigation (spec §6.1 / §K18)

`ColorPicker.vue` got 2-D arrow-key grid navigation across its swatch groups using a roving-tabindex pattern per group (Theme colors, My swatches). Arrow Left/Right step linearly; Arrow Up/Down find the swatch on the next row whose centre-X is closest, computed from live `getBoundingClientRect` so the layout adapts to flex-wrap reflow without hardcoded column counts. Enter and Space apply the focused swatch; Tab moves between groups; existing click behaviour and `panelOnly` mode unchanged.

### Phase 3 — `InlineFormatToolbar.vue` — the centrepiece

A fresh component, ~900 lines including scoped CSS, mounted once at the page-builder Vue app root via `<Teleport to="body">` (spec §A1/§A2/§C15) and never destroyed at runtime. All 18 controls in the spec's eight groups, Lucide glyphs from `lucide-vue-next`. Format-state reflection (active / mixed / inactive / disabled) on `selection-change` and `text-change` (the spec's `editor-change` aggregate was unrolled into the two granular events to avoid re-entering Quill's selection update during recompute). Selection preservation via `mousedown.preventDefault` on every control. The bar holds no direct Quill reference; every operation reads through `handle.value.quill` at call time, satisfying §A7 / §M3. Position tracking anchored to the field's bounding rect with viewport-top pinning (§C8), off-screen hide (§C9/§C10), and canvas-edge clamping (§C4/§C5). Re-target transitions per §B9 / §B10. Keyboard contract per §K — `role="toolbar"`, `aria-pressed`, roving-tabindex, Alt+F10 entry, Cmd/Ctrl+K opens link popover. `prefers-reduced-motion: reduce` disables fades.

Popovers (text-style menu, colour, highlight, link, overflow) all teleport to body alongside the bar and use a single shared anchoring helper with flip-above-when-no-room behaviour. The link popover supports both the URL field and a page-picker combobox (typeahead over `store.pages` with `url` resolved by 305 bootstrap), the open-in-new-tab checkbox writes `target="_blank" rel="noopener noreferrer"` (305's sanitiser allow-list keeps it on save). The colour and highlight popovers reuse `ColorPicker.vue` in `panelOnly` mode unchanged. The heroicon button reuses the existing `heroicon-picker.js` popover from session 304. Image upload uses the existing `store.inlineImageUploadUrl` endpoint with an inline spinner during upload and a red toast on failure.

### Phase 4 — Two-row layout (in-session UX adjustment)

After the user reviewed the natural-width single-row bar, the decision was made to default the bar to a two-row layout (max-width 480px + always flex-wrap) which fits better against the canvas. The spec's responsive collapse-ladder logic (§J Steps 1–4 + overflow menu) is parked in code with `collapseStep` pinned to 0; the wrap form keeps all 18 controls visible. The collapse-ladder math is retained in version control for a future amend if the wrap form proves too tall.

### Phase 5 — Bug fixes surfaced during manual UX testing

Eight distinct bugs emerged during the user's walkthrough and were each diagnosed and fixed before close. In order:

- **Initial `getSelection()` race**. The toolbar's first synchronous read of Quill's selection in `setupForHandle` threw inside `selection.update()` on a freshly mounted editor whose native anchor node wasn't yet established. Fix: focus the editor BEFORE publishing the handle so by the time the watcher fires, the selection is settled; plus a try/catch wrap on the format-state read so any future race-class equivalent degrades gracefully.

- **Format buttons doing nothing ("outbound reactivity but no inbound")**. The toolbar correctly reflected selection format but pressing Bold/Italic/etc. had no visible effect. Diagnosed via Playwright + console capture: every Quill format API (`formatText`, `formatLine`, `format`, `removeFormat`) internally calls `selection.update()` → `getRange()` → `normalizedToRange()`, which maps the native browser range's anchor nodes back into Quill blots. If the native cursor has drifted outside the editor's DOM — which happens whenever the user clicks a button (even with `mousedown.preventDefault` in some browser builds) — `scroll.find` returns null and the chain throws "Cannot read properties of null (reading 'offset')". `withQuill`'s try/catch swallowed the throw silently; the format never applied. Fix: `clearNativeSelection()` (i.e. `window.getSelection().removeAllRanges()`) immediately before every Quill format call. With no native range to normalise, Quill's update sees null, sets `lastRange = null` cleanly, and the format API proceeds against the explicit `(index, length)` we pass. The format handlers were also restructured to use `formatText`/`formatLine` on the explicit saved range instead of `format()` (which reads the live selection) — collapsed-selection inline marks still use `format()` because the pending-format-for-next-character semantics only exist there.

- **Right-click activates inline edit**. The activate handler in `useInlineEdit` listened on `pointerdown` which fires on right-click too — so right-clicking anywhere in an inline-editable field activated the editor and parked the toolbar at the top of the screen. Fix: gate both `activate` and `onDocPointerDown` on `PointerEvent.button === 0` (left-button only). Touch reports button=0 too, so this remains correct under touch.

- **Popover triggers don't toggle closed**. Clicking the text-style trigger opened the menu but clicking it again didn't close it (the handler always set the open state without checking). Added a `togglePopover` helper for text-style and overflow; ad-hoc toggle conditionals on link/color/highlight triggers.

- **Toolbar bar persists in fixed position after click-outside / right-click / scroll**. The headline bug — bar visibly stuck after user left edit mode. Diagnosed via Playwright with traces inserted into `useInlineEdit.teardownActive` and `store.clearActiveInlineEditor`: the equality check `cur.quill === quill` in `clearActiveInlineEditor` was always returning false. Root cause: Vue 3 auto-wraps any object assigned to a ref in a reactive Proxy. `activeInlineEditor.value = { quill, ... }` stored a proxied payload, so `cur.quill` returned a Proxy wrapping the raw Quill instance — never `===` the raw Quill the caller passed in. The handle was never cleared on teardown. Fix: `markRaw(payload)` before storing. Vue leaves it alone, identity is preserved, the equality check works.

- **Bullet lists vanish when clicking into a field containing them**. `useInlineEdit.activateRich` seeded the editor by assigning to `quill.root.innerHTML` directly. That bypasses Quill v2's blot parser — a saved `<ul><li>...</li></ul>` is visually rendered by the browser but Quill v2 doesn't recognise that as its list format (it uses `<ol><li data-list="bullet">`). On the next normalisation pass — typically the moment focus enters the editor — Quill strips the unknown markup and the list disappears. Fix: parse the seed HTML through `quill.clipboard.convert({ html: raw })` and apply via `setContents(delta, 'silent')`.

- **Ctrl+Z wipes all editor content**. Same root cause as the bullet-list bug, different symptom. With direct-innerHTML seeding Quill's internal Delta is EMPTY; the seeded content is on screen but Quill doesn't know about it. Every user keystroke is recorded as a delta against the empty base. Ctrl+Z replays those deltas in reverse until reaching that empty base — wiping all the content the user thought they were editing. The same clipboard-parse seed fix establishes the initial Delta correctly; `quill.history.clear()` after seeding ensures the user can't undo below the seeded state. The silent source keeps the seed out of the text-change stream so no stray save/refresh cycle fires.

- **Popovers (link URL field, colour swatches) cause editor teardown**. The bar's popovers teleport to `<body>`, so a click inside them sits OUTSIDE both the preview HTML and the bar's own DOM subtree. `useInlineEdit.onDocPointerDown` interpreted that as an outside click and tore down the editor. Fix: extend the early-return check to skip targets inside `[data-inline-toolbar-popover]` (spec §B12 — the bar and its popovers are never an exit gesture).

### Phase 6 — Comprehensive Playwright coverage (used as the iteration loop, not just as final coverage)

A standalone Playwright script at `/tmp/diag-suite.mjs` exercises every toolbar control plus the dismissal gestures (Escape, click-outside, right-click, scroll-then-click-outside). 20 cases. The script was the diagnostic vehicle for most of the Phase 5 bugs above — each fix was developed against a failing case, verified against the case, and then verified against the whole suite for regression. Final state: 20/20 passing, zero console errors. A parked Playwright spec at `tests/e2e/page-builder/inline-formatting-toolbar.spec.ts` covers the same surface as `test.describe.skip` per the parallel e2e-stabilisation arc — re-enable when the broader e2e suite is un-parked.

## Security considerations

The toolbar adds no new persistence path; every change still flows through Quill's HTML buffer → existing inline-edit raw-write contract → existing PageWidget recursive sanitiser. The 305 sanitiser allow-list (`target`/`rel` on `<a>`, strict closed-set values) is what makes the link popover's "Open in new tab" persistable; no new attribute or tag was admitted. The `clipboard.convert` parse path used for seeding is Quill's own — it doesn't bypass the sanitiser, it just converts HTML to Delta before Quill renders it; the persisted HTML still flows through `PageWidget::saving`'s sanitiser. The page-picker combobox reads `store.pages[].url` which is server-computed at bootstrap, not client-typed, so it can't be coerced into a hostile URL via the picker path. CRM contract remains v2.3.0; no boundary surface touched.

## Drift

Substantial and worth stating plainly.

The base-prompt's scope was the toolbar build itself + the §A4 handle refactor + §A14 observer + the §6.1 ColorPicker keyboard nav. All shipped. But the manual UX testing pause discovered **eight distinct bugs** — most of them in the existing 305-and-earlier inline-edit substrate, not in the new toolbar. The Quill v2 / native-browser-selection interaction model in particular was more fragile than the spec assumed: every Quill format API re-enters selection.update and can throw on a stale native range, the focus path on a freshly mounted editor throws on a null anchor, Vue's reactive-proxy wrapping breaks identity checks on instances stored in refs, direct-innerHTML seeding bypasses both the parser and the undo stack. Each was a real bug the user would have hit in real use; each got diagnosed and fixed against a Playwright reproduction.

This means we landed work originally scoped for session 307's "polish + parity" pass — specifically the stability fixes that would have been the natural close-out of the inline-editing arc. 307's remaining scope is narrower as a result: builder↔public render-parity tests + eligibility-gate widening (DonationForm heading, EventsListing static heading, others). The architectural foundation for 307 is now load-bearing-tested via the 20-case Playwright suite.

The two-row layout is also a divergence from the spec's §4.2 (38px fixed height) and §J (collapse-ladder). The owner decided in-session that the wrap form was a better on-page fit than the collapse-ladder; the spec is the contract, but the spec author asked for the change and confirmed it after seeing the result, so the change is in. The spec document is not amended — recording the divergence here.

The `clipboard.convert` + `setContents` + `history.clear` seed fix should also be applied to the Inspector `RichTextField.vue` (same bug class, different surface). Deferred to housekeeping per the deferred-decisions list above.
