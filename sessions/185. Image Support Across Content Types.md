# 185. Image Support Across Content Types

Extend the sample image library (built in session 184) to cover real model records created by the dev dashboard generator, and wire image display through the blog and event widget families. Resolve the open question of how image-as-content widgets (Hero, Image, Product Display) get a baseline image for Playwright thumbnail capture.

---

## Design decisions (resolved before starting)

- **Pool is the single source of demo imagery** across all four consumer paths: (a) widget config fields via `DemoDataService`, (b) widget demo collections via `SampleImageLibrary` (already wired in 184), (c) real model records created by the dashboard generator (new this session), (d) preset/thumbnail baselines for image-as-content widgets (new this session).
- **Real records get real `Media` rows.** When the dashboard generator creates a Product / Event / Post, it must attach a pool image to the appropriate media collection on that model — not just set a URL on a config field. This is what the widgets read when fetching published content.
- **Blog widgets render thumbnails.** BlogListing and BlogPager currently ignore any image on the Post model. Both render a thumbnail when one exists. The blog page template header (applied to single-post pages) also renders the post's image as a header above the title.
- **Event widgets are grouped with blog.** EventsListing and EventCalendar get the same treatment as BlogListing/BlogPager. The event detail page header renders the event's image.
- **Pages get a featured image.** Pages currently have no featured-image support at the model level. This session adds it — the model gains a `featured_image` media collection, the page editor exposes it in Filament, and page templates that have a header can render it. The dashboard generator populates it from the `still-photos` pool when generating pages.
- **Image-as-content presets get a baseline from the pool.** Any widget whose `presets()` list includes a variant where an image is intrinsic content (Hero background, Image widget source, Product Display featured image) resolves its baseline by drawing from the appropriate pool category at thumbnail-capture time. Widgets declare the category they want; the dev demo controller resolves a URL from the pool.
- **No new migrations beyond what's needed** for the page featured image. Existing Product / Event / Post media collections stay as-is.
- **ProductCarousel verification.** After product generation gains real image attachment, verify the ProductCarousel widget renders seeded product images end-to-end. No code change expected unless a bug surfaces.

## Phase 1 — Real model media attachment in the dashboard generator

Update `App\Filament\Widgets\DashboardDebugGeneratorWidget` to attach a pool image to each generated record:

- **`generateProducts()`** — after `Product::factory()->create()`, attach one image from `SampleImage::CATEGORY_PRODUCT_PHOTOS` to the `product_image` media collection.
- **`generateEvents()`** — after each `Event` creation, attach one image from `SampleImage::CATEGORY_STILL_PHOTOS` to the event's image media collection (confirm the collection name on the `Event` model — add one if none exists and the downstream widgets need it).
- **`generateBlogPosts()` / whatever creates Post records** — attach one image from `CATEGORY_STILL_PHOTOS` to the post's featured image media collection. Confirm or add the collection on the `Post` model.
- **`generateMembers()`** — attach one image from `CATEGORY_PORTRAITS` to each generated Contact's portrait media collection. Confirm or add the collection on the `Contact` model.
- **Page generation** (if the dashboard generator creates pages, or a new `generatePages()` method if not) — attach one image from `CATEGORY_STILL_PHOTOS` to the page's new `featured_image` media collection (added in Phase 2).

Each attachment should be guarded so an empty pool is a no-op, not an error.

- Expose it in the Page Filament resource (edit form, appropriate placement — likely under a "Media" section or near the title).
- Update page templates that render a header region to show the featured image above the title when present. No-op when absent.
- Update `docs/schema/pages.md` only if a column is added (media rows don't require schema doc changes; confirm no migration is needed beyond the Spatie `media` table already existing).

## Phase 3 — Blog widget image display

- **BlogListing widget** — render the post's featured image as a thumbnail for each listed post. Config-level toggle if needed; otherwise always show when the post has an image.
- **BlogPager widget** — same treatment: render the post's featured image alongside the prev/next navigation cards.
- **Blog post page template header** — single-post pages render the featured image as a header above the title. This is the template that wraps `/{blog_prefix}/{slug}` content.

All three paths read from the same `Post` media collection; all degrade gracefully when no image is attached.

---

## Phase 4 — Event widget image display (grouped with blog)

Parallel to Phase 3:

- **EventsListing widget** — render the event's image as a thumbnail for each listed event.
- **EventCalendar widget** — decide placement (likely in the event detail tooltip / modal / row) and render when available.
- **Event detail page header** — single-event pages render the event's image as a header above the title.

## Phase 6 — Image-as-content preset baselines

Define the mechanism for widgets whose presets intrinsically require an image (Hero background variant, Image widget, Product Display featured image):

- A widget definition declares the pool category it wants for thumbnail capture — e.g. `demoImageCategory(): ?string` on the definition base class, returning `null` by default.
- The dev demo controller ([app/Http/Controllers/Dev/WidgetDemoController.php](app/Http/Controllers/Dev/WidgetDemoController.php)) calls `SampleImageLibrary::randomUrl($category)` when a preset is rendered and injects the URL into the config at the expected key (Hero → `background_image`, Image → `image_src`, etc.).
- Confirm this flows through to the Playwright thumbnail script without extra changes (the script hits the dev route; if the route returns a valid rendered widget, the screenshot captures it).
- Document the hook in `resources/docs/widget-development.md` (new subsection under the sample image library section from 184).

Widgets to enable in this pass: Hero (background image variant), Image widget, Product Display. Any other widgets that fail thumbnail capture due to missing image content should be included.
