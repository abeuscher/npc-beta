# 349. Table Widget

A new Page-Builder widget called **Table** that produces a semantic HTML `<table>` from operator-authored rows and columns — rich-text inline-editable cells, a Google-Docs-style grid picker for dimensions, header-row + per-column alignment + border-style affordances, and mobile-friendly horizontal overflow.

## The template is PricingChart — clone it

`app/Widgets/PricingChart/` already solves every hard part of this widget. Read it first; the Table widget is the same pattern emitting a `<table>`:

- **Count-driven dimensions.** `column_count` / `attribute_row_count` are `select` fields; the repeater that holds the content carries `'inspector' => false` so **cells are edited inline on the canvas, not in the Inspector**. Lowering a count **hides, does not delete** ("their content is kept, not deleted").
- **Inline-eligible rich-text cells.** PricingChart's `attribute_rows.value` is a `richtext` field rendered through the inline-editing toolbar (the component refactored in 347) over the Quill + `HtmlSanitizer` pipeline. Table cells are the same shape.
- **Standard surfaces.** Definition (`schema()` + `defaults()` + render hooks) + `template.blade.php` + `styles.scss` + a `WidgetTypeSeeder` append + a thumbnail. No bespoke admin form beyond the config schema — *except* the one custom field below.

Mirror PricingChart's structure exactly; deviate only where a table genuinely differs (header row, `<thead>`/`<tbody>`, per-column alignment, border style).

## Session-specific deltas (vs the 311 spec)

1. **Grid-picker dimension control (the UX upgrade — the image the owner attached).** Instead of the 311 spec's two separate count pickers, dimensions are set with a **Google-Docs-style grid picker**: a small grid where hovering highlights an *N×M* region, the live label reads "N × M", and clicking commits those dimensions. It is a **custom Inspector field** (a new `resources/js/page-builder-vue/components/fields/GridPickerField.vue`, registered by a new `field.type` in `InspectorField.vue`'s `componentMap`) — bounded, mirroring the 12 existing field components. Dimension changes are **non-destructive on shrink**, exactly like PricingChart's count fields.
   - **It is an Inspector field, NOT an insert-time creation modal.** The widget is added with sensible defaults (2×3) and resized via the picker in the Inspector; cells fill inline on the canvas. A true Google-Docs "pick size on insert" flow would be a new widget-instantiation interaction (pattern-invention) and is **out of scope** — see Out of scope.
2. **Inline-eligibility from day one** (311 decision option (a)). The cell shape is a direct copy of an already-eligible widget, so ship eligible rather than queueing a one-line follow-on. Mirror PricingChart's inline-eligibility wiring (the per-widget roster + its pinning regression test, the `useInlineEdit` / `PreviewRegion` eligibility path).

## Open questions to resolve at session start (settle, don't drift)

1. **Grid-picker data model.** A field component v-models one config key, but the picker sets *two* dimensions. Pick early: (a) a composite key (e.g. `dimensions: {rows, columns}`) the Definition/template read; or (b) the picker drives two sibling count keys via the config-path update mechanism. Either is fine — choose the one that keeps the row-content repeater reconcile (pad on grow / trim-from-right on shrink) simplest, and note it in the log.
2. **Grid-picker max + reconcile.** Cap the picker at a sane max (the 311 spec capped columns at 2–8; pick a rows cap too, e.g. 1–10) with the existing "shrink hides, does not delete" semantics. A confirmation when a shrink would drop **non-empty** cells is desirable but not required for v1 — match whatever PricingChart does on count-down (it hides silently); if PricingChart warns, mirror it.

## Public render (per the 311 spec, unchanged)

- Semantic `<table>` wrapped in `.np-table-widget` with `overflow-x: auto; max-width: 100%` (narrow viewports scroll, not overflow).
- Header-row toggle → row 0 as `<thead>`/`<th>`; otherwise all rows `<tbody>`/`<td>`.
- Per-**column** alignment via `text-align` on the cells.
- Border style `none | outer | all` via a single wrapper utility class (`np-table-widget--border-{style}`) — SCSS owns the visual; no inline borders.
- Cell rich text rendered through `{!! HtmlSanitizer::sanitize($cell['content']) !!}` — the same path every other rich-text cell uses.
