# 158. Inspector Panel Reactivity

Make the page builder preview update automatically as the user edits config in the inspector, with a delay-gated blur + cascading spinner indicator while a refresh is in flight. Also: style changes (padding/margin) apply locally with instant feedback and save to the server in the background, no preview round-trip required. The Apply Changes button stays unchanged.

This is a follow-on to Session 157, which fixed the underlying reactivity bug that made Apply Changes not work at all. Now that the preview pipeline is sound, this session removes the need to click Apply on every edit.

---

## Design decisions (resolved before starting)

### The Apply Changes button stays untouched

- Do not delete it. Do not change its enabled/disabled logic. Do not change its position or styling. Leave it exactly as it is.
- Rationale: the user may remove it later, may not. Treat it as a "skip the debounce wait" affordance — the close-elevator-door button. It must continue to work in parallel with the new auto-refresh path.
- After this session, clicking Apply Changes should still flush pending saves and force an immediate `refreshPreview`, exactly as today.

### Server-rendered preview, no template port

- Widget templates remain Blade. There is no work in this session to port templates to Vue or to render previews client-side. Reactivity is achieved by chaining `refreshPreview` calls (which fetch a freshly rendered preview from the server) after debounced saves settle.
- The one exception is `style_config` (padding / margin), which is **not** part of the rendered widget HTML — it lives on the wrapper div applied outside the widget render. That is the only thing that gets a local-instant apply path in this session.

### Debounce window: 350ms

- Tighten `flushDebouncedSave` from the current 500ms to 350ms. The auto-refresh chains off the save, so the user sees the preview update ~350ms + server render time after their last keystroke.
- 350ms is fast enough to feel responsive on a normal connection without firing a request on every keystroke during typing bursts.
- If the audit phase finds 500ms is referenced anywhere outside `flushDebouncedSave`, leave those alone unless they're directly part of the same debounce flow.

### Indicator timing — delay-show gate at 250ms, spinner at 500ms

Two-stage cascading reveal driven by per-widget timers:

| Stage | Threshold | What appears |
|---|---|---|
| 0 | 0–250ms | Nothing. The widget renders fast enough that no indicator is needed. |
| 1 | 250ms+ | `backdrop-filter: blur(3px)` on the existing `.preview-region__overlay`, with a 100ms fade-in. |
| 2 | 500ms+ | A small CSS-only spinner element appears, centered over the blurred widget, also with a 100ms fade-in. |

When the refresh settles (success, error, or abort), all stage classes clear and the blur + spinner fade out.

For continuous edits (color picker drag, slider scrubbing) the user keeps triggering new refreshes while one is in flight. The indicator must **not** flicker between requests — it should stay continuously on across the burst. This is achieved with a per-widget refresh **counter** (not a boolean): increment on entry, decrement on settle, indicator-active when count > 0. The 250ms / 500ms timers only start on the 0→1 transition; subsequent refreshes during the same window do not restart them.

The 500ms spinner threshold is tunable. Start there; if mid-speed widgets feel like they need the spinner sooner, lower it in a follow-up.

### Spinner: CSS-only, no asset

- A single `<div>` element with a CSS animation (border ring rotating, or two arcs — whatever reads cleanly at 16–20px). No SVG, no icon library, no asset.
- Color: matches the existing editor chrome accent (probably `--c-primary-600` or the same indigo used in the selection outline at [PreviewRegion.vue:75](resources/js/page-builder-vue/components/PreviewRegion.vue#L75)).
- Sized so it sits comfortably in the center of even small widgets without overflowing.
- Element lives inside `.preview-region__overlay` so it stacks above the blur and the selection outline. Visible only when the spinner stage is active.

### Blur only — no tint

- `backdrop-filter: blur(3px)` is the entire visual treatment for stage 1. No color overlay, no opacity wash, no border change.
- Rationale: the spinner provides the "active" cue at stage 2; the blur alone is enough at stage 1 without making the widget look broken or disabled.
- If 3px feels too soft or too harsh in practice, tune within `2px–4px`. Anything heavier looks like the widget crashed.

### Style config: local-instant apply with background save

- `updateLocalStyleConfig` (padding, margin, etc.) currently mutates `w.style_config` and queues a debounced save. The save fires server-side, then the user has to wait for a preview refresh to see the new spacing.
- New behaviour: apply the style change to the **wrapper element on the preview DOM directly** (instant local feedback, no server round-trip), and save to the server in the background via the existing debounced save path. **No `refreshPreview` call** for style_config — the server preview HTML hasn't changed because the wrapper inline style is applied outside the rendered widget HTML.
- Implementation depends on what the audit phase finds about how `style_config` currently flows into the preview. Two likely paths:
  - **(a)** The server-rendered preview HTML already includes the wrapper div with the inline style baked in. If so: read and rewrite the relevant `style` attribute on the rendered preview DOM via a small Vue effect that watches `widget.style_config`. The next time a config change triggers `refreshPreview`, the server will return fresh HTML with the same style baked in, and the local override seamlessly hands off.
  - **(b)** The Vue PreviewRegion already wraps the rendered widget HTML in its own wrapper div, and the wrapper takes its style from `widget.style_config` reactively. If so: there is nothing to do for instant apply except confirm reactivity is working.
- Audit phase decides which path applies. Whichever is the case, the user-visible outcome is the same: changing a padding value updates the preview spacing instantly, with no blur and no spinner.
- The background save still happens. The server still gets the new style_config. Other clients editing the same page (if any) still see the change on their next preview refresh.

### Image uploads trigger auto-refresh

- After `uploadImage` or `removeImage` settles, automatically call `refreshPreview` on the affected widget. No need for the user to click Apply Changes after dropping in a new image.
- The blur + spinner indicator applies normally during this refresh — image uploads are typically the slowest preview refresh case (the server has to generate conversions and resolve media), so the spinner is most likely to appear here. That is the right behaviour.

### Query config triggers auto-refresh

- `updateLocalQueryConfig` (collection limits, ordering, tag filters) already debounces a save. After the save settles, automatically chain `refreshPreview` exactly like config changes do.
- Query config changes always need a server round-trip — they affect what data the widget pulls.

### Failure state: persistent error badge

- If a `refreshPreview` fetch errors (not just aborts), show a persistent visible badge on the affected widget's preview region: small, top-right corner of the overlay, red, with the error message text. The badge stays visible until the next successful refresh on that widget, then auto-clears.
- The user is in beta. The user wants to see when something fails and what failed. Silent failures or generic "preview failed" tooltips are out.
- The badge text is the actual error message from the response (or "Network error" if the request never reached the server). Truncate if longer than ~80 characters with a `title` attribute holding the full text.
- Aborted requests (AbortController-canceled because a newer request superseded them) are **not** errors. They are silent. Only true failures show the badge.

### Request cancellation via AbortController

- `refreshPreview` must abort any in-flight request for the same widget before starting a new one. Without this, a slow preview from edit N can land after edit N+1 and overwrite the newer HTML with a stale render.
- Per-widget AbortController stored in a `Map<string, AbortController>` in the store. Cleared on settle.
- The audit phase confirms `api.ts` uses `fetch` (so AbortController is native). If it uses something else, surface that before implementing.

## Phase 2 — Store: refresh tracking, cancellation, timer-gated indicator state

Working in `resources/js/page-builder-vue/stores/editor.ts`:

- Add a per-widget refresh counter: `const refreshCounts = ref<Record<string, number>>({})`. Increment on entry to `refreshPreview`, decrement on settle (success, error, **and** abort).
- Add a per-widget AbortController map: `const abortControllers = new Map<string, AbortController>()`. Before starting a new refresh for a widget, abort the previous one and clear the entry. Store the new controller, pass `signal` to `api.getPreview`.
- Add per-widget timer state for the cascading indicator stages:
  - `const blurTimers = new Map<string, ReturnType<typeof setTimeout>>()`
  - `const spinnerTimers = new Map<string, ReturnType<typeof setTimeout>>()`
  - `const indicatorStage = ref<Record<string, 0 | 1 | 2>>({})`
  - On the 0→1 transition of the refresh counter, set timers for 250ms (stage 1) and 500ms (stage 2). On settle back to 0, clear both timers and reset stage to 0.
- Add a per-widget error state: `const previewErrors = ref<Record<string, string | null>>({})`. Set on a true error (not abort). Clear on the next successful refresh for the same widget.
- Expose getters: `widgetRefreshing(id)`, `widgetIndicatorStage(id)`, `widgetPreviewError(id)`. These drive the PreviewRegion's class bindings and badge.

The existing `refreshPreview` becomes the single funnel for all of this — preserve its current behaviour (flush pending saves, fetch preview, update HTML in place per session 157's fix) and wrap it with the counter / abort / timer / error logic.

## Phase 3 — Store: chain auto-refresh after debounced saves and image operations

- **Tighten `flushDebouncedSave` from 500ms to 350ms.**
- After each per-widget save in `flushDebouncedSave` settles successfully, call `refreshPreview(widgetId)`. Only chain on success, not on save failure (a failed save means the server has stale config, so refreshing the preview would just show the stale state — let the next user edit re-trigger).
- After `uploadImage` settles successfully, call `refreshPreview(widgetId)`.
- After `removeImage` settles successfully, call `refreshPreview(widgetId)`.
- After `updateLocalQueryConfig`'s eventual server save settles, call `refreshPreview(widgetId)`. (Note: `updateLocalQueryConfig` also flows through `flushDebouncedSave`, so this may already be covered by the change above — confirm during audit and don't double-call.)
- **Do not** chain `refreshPreview` after `updateLocalStyleConfig` server saves. Style config gets the local-instant apply path in Phase 4 instead.
- **Do not** change `updateLocalConfig` for label-only edits (where `key === null`). Labels don't affect the rendered preview; chaining a refresh would be wasteful.

## Phase 5 — PreviewRegion: blur, spinner, error badge, aria-busy

In `resources/js/page-builder-vue/components/PreviewRegion.vue`:

- Bind a `preview-region--refreshing-blur` class when `store.widgetIndicatorStage(widget.id) >= 1`.
- Bind a `preview-region--refreshing-spinner` class when `store.widgetIndicatorStage(widget.id) >= 2`.

- Add an error badge child element inside `.preview-region__overlay`, top-right corner, visible only when `store.widgetPreviewError(widget.id)` is non-null. - Bind `aria-busy="true"` on `.preview-region` when stage >= 1.
- Selection outline (`.preview-region--selected`) and hover outline must remain crisp and visible through the blur — they are on the overlay element itself, above the backdrop-filter, so this should work without extra effort. Verify visually.

## Phase 7 — Manual testing matrix

There is no JS test infrastructure in this project (Vue test coverage is captured as a separate post-Beta 1 outline item from session 157). All testing is manual.

Run through every case below in the browser. Each case should pass before moving on.

### Indicator timing

- **Fast text widget** (text block, hero heading): edit a config text field, type a single character. Preview updates within ~400–500ms. **No blur, no spinner.**
- **Mid-speed widget** (anything with a single DB query — event single, product display): edit config. Preview updates with **blur visible, no spinner**.
- **Slow widget** (product carousel, board members, anything with `collection()` calls): edit config. Preview updates with **blur first, then spinner appearing ~250ms later**, then both fade out together when the render lands.
- **Continuous input** (color picker drag, slider scrub): drag for ~3 seconds. The blur/spinner indicator stays continuously on across the entire drag — **no flicker between requests**. Final value lands ~350ms after the drag stops.

### Style config local apply

- **Padding/margin edit on any widget**: change a padding value. The preview spacing updates **instantly**, with no blur and no spinner. Network panel shows a save request firing in the background ~350ms later (no preview request).
- **Style + config combo**: change a padding value, immediately change a config text field. The padding stays applied locally; the config change triggers a preview refresh that returns HTML that should also reflect the new padding (because the server has the saved value by then). **No visual jump on the padding when the new HTML lands.**

### Image uploads

- **Upload a new image**: drop a file into an image field. Preview refreshes automatically, with the blur + spinner indicator visible during the refresh. New image appears in the preview when the refresh completes.
- **Remove an image**: click remove. Preview refreshes automatically. Image disappears.

### Apply Changes button

- **Click Apply on a clean widget** (nothing pending): button is disabled (current behaviour, unchanged).
- **Click Apply during a debounce window**: pending save flushes immediately, refresh fires immediately, preview updates without waiting for the 350ms debounce. Same behaviour as today.

### Failure mode

- **Throttle network in DevTools to "Slow 3G"**: edit config. Blur appears at 250ms, spinner appears at 500ms, eventually the render lands and clears.
- **Force a 500 error from the preview endpoint** (temporarily break the controller, or simulate via DevTools network override): edit config. Refresh starts, fails. **Persistent red error badge** appears in the top-right corner of the affected widget's preview region with the error message. Make another successful edit. Badge clears after the next successful refresh.
- **Rapid edits during a slow refresh**: with Slow 3G still throttled, type rapidly into a config field. The first few in-flight requests should be aborted. No stale HTML should land. Final keystroke's render should win.

### Layout interaction

- **Edit a config field on a widget inside a column layout**: only the affected child widget shows the indicator. The layout container itself does not blur. Other children in the same layout are unaffected.
- **Drag-reorder widgets**: still works. No regression from the new wiring.

### Apply Changes button regression

- Confirm the button still appears, still has the same enabled/disabled logic, still flushes pending saves and forces a refresh on click. **It must not have been touched.**

## Out of scope

- **Removing the Apply Changes button.** Explicit user direction: leave it untouched. Future session may remove it.
- **Porting widget templates from Blade to Vue** for true client-side rendering. This is a months-long lift, would double widget maintenance, and is not needed to achieve the user-visible reactivity goal of this session.
- **Adding JS test infrastructure** (vitest / @vue/test-utils / @pinia/testing). This is captured as a separate post-Beta 1 outline item from session 157.
- **Tuning performance** further than the chosen 350ms debounce / 250ms blur gate / 500ms spinner gate. These are starting values; revisit only if manual testing reveals a specific problem.
- **Layout-level indicator** on the layout container itself. Layouts are structural; only their child widgets show the indicator.
- **Optimistic config rendering** for non-style fields (e.g., live-rendering a config text change in the preview before the server returns). Server round-trip is the contract for everything except style_config.
- **Multi-user editing conflict handling** (two editors changing the same widget). Out of scope for this session and for the project at large in the current beta scope.

## Acceptance criteria

- Editing any non-style config field in the inspector causes the preview to update automatically ~350ms after the last edit, with no need to click Apply Changes.
- For fast widgets (<250ms server render), no indicator appears at all — the update feels instant.
- For mid-speed widgets (250–500ms), the blur appears briefly with no spinner.
- For slow widgets (>500ms), the blur appears at 250ms and the spinner at 500ms, both fading in/out smoothly.
- During continuous input (color picker drag), the indicator stays continuously visible without flicker between in-flight requests.
- Editing a padding or margin value updates the preview spacing instantly, with no blur, no spinner, and no preview round-trip. The server save still fires in the background.
- Image upload and removal trigger automatic preview refresh.
- Failed preview refreshes show a persistent red error badge on the affected widget's preview region. The badge displays the actual error message and clears on the next successful refresh.
- Aborted requests (superseded by a newer edit) are silent — no error badge, no console noise beyond what's already there.
- The Apply Changes button is unchanged in appearance, position, and behaviour. It still works as a "skip the debounce wait" affordance.
- The selection outline and hover outline on widgets remain crisp and visible during refresh (not blurred).
- `aria-busy="true"` is set on the preview region during refresh for screen reader feedback.
- `npm run build` succeeds. Fast PHP test suite passes with no new failures.
