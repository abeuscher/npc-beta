# 075. Donations — Foundation

Build one-off and recurring donation support backed by Stripe Checkout and Stripe Billing, surfaced via a page widget, with contact lookup/create and activity logging following the same patterns established in session 074.

---

## Design decisions (resolved before starting)

**What a donation is:**
A donation is a voluntary financial contribution from a supporter. One-off donations are single payments. Recurring donations are Stripe Billing subscriptions — one subscription record per supporter/amount/frequency, with a transaction written on each billing cycle.

**Data model:**
A `donations` table records every donation commitment (one-off or recurring). Each completed payment is recorded as a `Transaction` (the polymorphic table from session 073) with `subject_type = 'App\Models\Donation'` and `subject_id = donation.id`. Read the existing `Transaction` model and the polymorphic design from session 073 before writing anything.

**Anonymous donors:**
No portal authentication required. On webhook completion, read `customer_details.email` (and `customer_details.name` if available) from the Stripe event. Look up an existing contact by email; create one if none exists. Attach the donation to that contact. This is identical to the products pattern — reuse the same contact lookup/create logic.

**Amount handling:**
Amounts are passed from the frontend to the checkout controller and validated server-side against a permitted set (configurable predefined options plus a "custom" path). For this session, the predefined amounts are hardcoded: $10, $25, $50, $100, and custom. The widget renders these as buttons. Amount is never taken as a raw arbitrary input — the controller enforces a minimum of $1 and a maximum of $10,000.

**Stripe Checkout modes:**
- One-off donations use `mode: 'payment'` with inline `price_data` (no Stripe Price object required).
- Recurring donations use `mode: 'subscription'`. A Stripe Price object is required; the controller creates it inline using `price_data` within the subscription line item.

**Metadata convention:**
Checkout session metadata carries `{ donation_id: '...' }`. The webhook uses this to locate the `Donation` record and create the `Transaction`.

**Recurring webhook events:**
- `checkout.session.completed` — marks the donation `active`, creates the first `Transaction`.
- `invoice.payment_succeeded` — creates a subsequent `Transaction` on each billing cycle (idempotency on `stripe_id`).
- `invoice.payment_failed` — marks the donation `past_due`, logs a failed transaction.

**Activity log:**
On donation created (status transitions to `active`): log `'donated'` to the contact's activity log. On recurring payment received after the first: log `'donation_received'`. On failure: log `'donation_failed'`. Follow the `ActivityLogger::log($contact, $event, $description)` pattern exactly as in session 074.

**Stripe customer:**
For recurring donations, a Stripe Customer is required. On checkout, pass `customer_creation: 'always'` and store the returned `stripe_customer_id` on the `Donation` record. For one-off donations this field is left null.

**Admin collection:**
Donations are not a system collection — they are not surfaced in the Collection Manager. The admin resource (DonationResource) lives under the Finance navigation group.

## Phase 1 — Data model and admin resource

1. Migration: `donations` table
   - `id` (uuid)
   - `contact_id` (FK → contacts, nullOnDelete)
   - `type` (string: `one_off` / `recurring`)
   - `amount` (decimal 10,2)
   - `currency` (string, default `'usd'`)
   - `frequency` (string, nullable: `monthly` / `annual` — null for one-off)
   - `status` (string: `pending` / `active` / `cancelled` / `past_due`)
   - `stripe_subscription_id` (string, nullable)
   - `stripe_customer_id` (string, nullable)
   - `started_at` (timestamp, nullable)
   - `ended_at` (timestamp, nullable)
   - timestamps

2. `Donation` model: `HasUuids`, `HasFactory`; `contact()` BelongsTo; `transactions()` morphMany via the polymorphic Transaction relation; casts: `amount` decimal:2, `started_at`/`ended_at` datetime.

3. `DonationObserver`: on `updated`, when `status` transitions to `active` and `contact_id` is set, log `'donated'` to activity. When `status` transitions to `past_due`, log `'donation_failed'`. Register in `AppServiceProvider`.

4. `DonationResource` in Filament under navigationGroup `Finance`:
   - Table: contact name (via relation), type badge, amount (money), frequency, status badge, started_at.
   - Form: read-only — this resource is view/audit only. Admin does not create donations manually.
   - Relation manager: Transactions (read-only — stripe_id, amount, status, occurred_at).

5. Update `docs/schema.md`.

## Phase 2 — Checkout controller and routes

1. `DonationCheckoutController@store`:
   - Validates: `amount` (numeric, min:1, max:10000), `type` (`one_off` or `recurring`), `frequency` (required if recurring: `monthly` or `annual`).
   - Creates a `Donation` record with `status = 'pending'`.
   - Builds a Stripe Checkout session:
     - `mode`: `'payment'` for one-off, `'subscription'` for recurring.
     - For one-off: use inline `price_data` with `unit_amount` and `currency`.
     - For recurring: use `price_data` with a `recurring` block (`interval: 'month'` or `'year'`).
     - `customer_creation: 'always'` for recurring; omit for one-off.
     - `metadata`: `{ donation_id: donation.id }`.
     - `success_url` / `cancel_url`: from HTTP `Referer`, with `?donation=success` or `?donation=cancelled` appended.
   - Returns a JSON redirect to the Stripe Checkout URL.

2. Route: `POST /{donations_prefix}/checkout` → `DonationCheckoutController@store`, throttle 20/1. No auth required.

3. Add `donations_prefix` to `AppServiceProvider` config (same pattern as `products_prefix`), defaulting to `'donate'`. Add it to the Routing section of General Settings (mount, form field, save).

4. Update `docs/schema.md` if needed.

## Phase 3 — Webhook handling

Extend `StripeWebhookController` to handle donation events. Read the existing controller carefully before modifying.

1. In `handleCheckoutSessionCompleted`: after the existing `product_price_id` check, add a check for `metadata.donation_id`. If present, call a new private `handleDonationCheckout()` method.

2. `handleDonationCheckout($event)`:
   - Idempotency: check `Transaction::where('stripe_id', $sessionId)->exists()` — return if duplicate.
   - Find the `Donation` by `metadata.donation_id`. If not found, log a warning and return.
   - Contact lookup/create from `customer_details.email` and `customer_details.name` (same logic as `handleProductPurchase` — extract to a shared private method if both paths can reuse it cleanly; otherwise duplicate with a comment).
   - Update the `Donation`: `contact_id`, `stripe_customer_id` (from event, if present), `status = 'active'`, `started_at = now()`.
   - Create a `Transaction`: `subject_type = 'App\Models\Donation'`, `subject_id`, `type = 'payment'`, `amount` (from event, cents → dollars), `currency`, `direction = 'in'`, `status = 'completed'`, `stripe_id = $sessionId`, `occurred_at = now()`.

3. Register `invoice.payment_succeeded` handler `handleInvoicePaymentSucceeded($event)`:
   - Extract subscription ID from `$event->data->object->subscription`.
   - Find `Donation` by `stripe_subscription_id`. If not found, return (belongs to a different system).
   - Idempotency on `stripe_id`.
   - Create a `Transaction` (same shape as above).
   - If this is not the first invoice (check `$event->data->object->billing_reason !== 'subscription_create'`), log `'donation_received'` to the contact's activity log.

4. Register `invoice.payment_failed` handler `handleInvoicePaymentFailed($event)`:
   - Find `Donation` by `stripe_subscription_id`.
   - Update `status = 'past_due'` (triggers the observer log).
   - Create a `Transaction` with `status = 'failed'`.

## Phase 4 — Donation widget

Insert a new `WidgetType` with handle `donation_form` via `WidgetTypeSeeder` (or direct insert following the existing pattern):
- `label`: `Donation Form`
- `render_mode`: `server`
- `config_schema`: a text field for a custom heading (label: `Heading`, key: `heading`, optional).

Blade template `resources/views/widgets/donation-form.blade.php`:
- Renders the heading (if set) above the form.
- Amount buttons: $10, $25, $50, $100, plus a "Custom amount" text input revealed on click.
- Frequency toggle: One-time / Monthly / Annual (defaults to One-time).
- Single "Donate" button submits via JS (`fetch`) to `POST /{donations_prefix}/checkout`, then redirects to the returned Stripe URL.
- Reads `?donation=success` query param to show a confirmation message.
- Reads `?donation=cancelled` to show a cancellation message.
- Uses `@csrf` — pass the token in the JS fetch headers.

## Security checklist

Before closing:

- [ ] Amount validated server-side: minimum $1, maximum $10,000 — never taken raw from form input.
- [ ] Contact identity on donations derived from Stripe's verified `customer_details.email`, not from user-submitted data.
- [ ] Webhook signature verified before any processing (existing controller behaviour — confirm it covers new event types).
- [ ] Idempotency check on `stripe_id` before creating any Transaction.
- [ ] Stripe secret key never exposed in responses or logs.
- [ ] Recurring subscription ID stored only after Stripe confirms it — never pre-assigned.
