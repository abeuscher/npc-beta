# 143. Codebase Hygiene Audit

Codebase-wide hygiene pass covering areas not scoped to the builder overhaul. Five workstreams: Docker security, help system completeness, stale code comments, dependency footprint, and debug output cleanup.

---

## Design decisions (resolved before starting)

- This is an audit-and-fix session. No new features.
- Docker findings are presented as a table — not fixed until discussed (some may be endemic to the stack or intentional).
- Unused package removal requires discussion before acting — present findings first.
- Debug output removal (console.log, dd(), dump(), var_dump(), Log::debug() left from development) is safe to fix in place.
- TODO/FIXME comments are surfaced as a table — not resolved, just catalogued so the user can triage.

---

## Before writing any code

Read `docs/app-reference.md` for the full view-to-file mapping. This session covers the entire codebase, not just the builder.

---

## Workstream 1 — Docker security audit

Scrutinize the Docker setup from a security perspective. Review:

- `docker-compose.yml` (local) and `docker-compose.prod.yml` (deploy server at `/opt/nonprofitcrm/`)
- `Dockerfile` and any related build files
- Container networking — are any ports exposed that shouldn't be?
- Volume mounts — are any host paths mounted that create risk?
- User permissions — does the app container run as root or a restricted user?
- Secrets management — are credentials passed via environment variables, build args, or baked into images?
- Image provenance — are base images pinned to specific versions or floating on `latest`?
- Redis and PostgreSQL containers — are they accessible from outside the Docker network?
- Any `.env` handling that could leak secrets into image layers or logs

Surface all findings even if they are endemic to the Docker/Laravel stack. Present as a table: Finding, Severity (critical/medium/low), Location, Recommendation.

---

## Workstream 2 — Help system completeness audit

Audit the help system to verify there is a help page for every admin view. Cross-reference the full view list in `docs/app-reference.md` against help docs in `resources/docs/`.

For each admin resource and page:
- Check if a corresponding help doc exists in `resources/docs/`
- Check if the help doc's `routes:` array covers all routes for that view (index, create, edit)
- Check if the help doc content is still accurate for the current UI

Present findings as a table: View, Help Doc Status (exists/missing/stale), Notes.

Fix confirmed gaps in place (create stubs with frontmatter for missing docs, update stale route arrays).

---

## Workstream 3 — TODO/FIXME code comment sweep

Grep the entire codebase for `TODO`, `FIXME`, `HACK`, `XXX`, `WORKAROUND`, and similar markers in code comments (PHP, Blade, JS, SCSS, CSS). Exclude `vendor/`, `node_modules/`, `storage/`, and `public/build/`.

Present all findings as a table: File, Line, Marker, Comment text.

Do not resolve any of them — the user will triage.

---

## Workstream 4 — Dependency footprint audit

Evaluate the size of the codebase and look for opportunities to reduce the overall footprint without negatively affecting functionality.

- **Composer packages:** Run `composer show` and cross-reference against actual usage. Flag any package that appears unused (no `use` statement, no config reference, no artisan command usage). Pay attention to packages that may be used only by other packages (transitive dependencies) — don't flag those.
- **NPM packages:** Run `npm ls --depth=0` and cross-reference against `resources/js/`, Vite config, and build server usage. Flag unused packages.
- **Dead routes:** Check `routes/web.php` and `routes/api.php` for routes pointing to controllers or methods that no longer exist.
- **Dead config:** Check `config/` for config keys that reference services or features no longer in use.

Present findings as a table: Package/Item, Type (composer/npm/route/config), Evidence of usage or non-usage, Recommendation. Do not remove anything until discussed.

---

## Workstream 5 — Debug output cleanup

Grep the codebase for debugging output that was left behind during development:

- **JavaScript:** `console.log(`, `console.warn(`, `console.error(`, `console.debug(`, `debugger` statements. Exclude intentional error handling in catch blocks (e.g., `console.warn('Failed to load widget lib CSS:', lib)` in the preview loader is intentional).
- **PHP:** `dd(`, `dump(`, `var_dump(`, `print_r(` used outside of test files. `Log::debug(` calls that appear to be development leftovers rather than intentional production logging.
- **Blade:** `{{ dd(`, `{{ dump(`, `@dump(` directives.

Fix confirmed debug output in place. Flag uncertain cases (could be intentional logging) for discussion.
