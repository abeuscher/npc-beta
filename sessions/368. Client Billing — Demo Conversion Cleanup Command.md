# 368. Client Billing — Demo Conversion Cleanup Command

Build the one-shot, idempotent artisan command Fleet Manager runs on a node immediately after converting a personalized demo into a paying client — removing the shared demo login account and unlocking the pages the demo reset had locked, so the converted node is indistinguishable from a fresh production install.

## Session-specific deltas

What the track doc defers to session time, plus what the code actually looks like today:

- **Where it sits in the conversion sequence.** Fleet Manager's conversion action runs a fixed order: confirm plan/price/billing contact → create the Stripe customer + subscription → remove the demo reset cron → **flip `APP_ENV` demo→production via the config-push channel (containers recreate)** → **[this command]** → push the initial billing-state document → write the audit chain. It is invoked over FM's **existing SSH channel** (`docker compose exec app php artisan …`), not over HTTP. The node gains no endpoint.

- **The inverted guard — the single most important fact in this session.** `DemoResetCommand` and `DemoRestoreCommand` both open with `if (! isDemoMode()) { … return self::FAILURE; }` — they refuse to run outside demo mode, so a misconfigured cron can never wipe a production database. **This command is the mirror image:** by the time it runs, step 4 has already flipped `APP_ENV` to `production`, so `isDemoMode()` is **false**. It must carry **no `isDemoMode()` guard**. Copying the demo commands' opening stanza would make it refuse to run at the only moment it exists for.

- **What it removes: exactly one account.** `DemoLoginController` (`abort_unless(isDemoMode(), 404)`) `firstOrCreate`s the shared account **`demo@demo.local`** and `syncRoles(['demo'])`. That user row is what gets deleted. Once `APP_ENV` is production the `/demo/enter` route 404s, so the account cannot reappear — deletion is final and safe.

- **What it unlocks: the blanket page lock.** `DemoResetCommand` does `Page::query()->update(['locked' => true])` on every reset (and `DemoRestoreCommand` re-asserts it after a blob restore) so the shared demo account can't edit the sample site. `pages.locked` **defaults to `false`** (`docs/schema/pages.md`) and **no non-demo seeder ever sets it true** — so on a converted demo node, clearing the flag across all pages restores exactly the fresh-install default. `locked` is enforced in `PagePolicy::update()` + the page-builder save API, gated by the `edit_locked_pages` permission.

- **The `demo` role stays.** It is seeded vocabulary in `PermissionSeeder`; with `/demo/enter` 404ing and the account gone, it grants nothing to nobody. Removing the *account*, not the *role*, is the scope (confirm at the gate).

- **Exit codes are the interface.** FM reads them over SSH. `SUCCESS` on both the did-work path and the nothing-to-do path (idempotent re-run); `FAILURE` only on a genuine error. Emit a terse line per action taken so the SSH transcript lands usefully in FM's audit ledger.

## Open questions to resolve at session start

1. **The safety gate (the one real design decision).** Blanket-unlocking every page is a harmless no-op on a fresh install, but **not** on an established client node — a real CMS editor holding `edit_locked_pages` may have deliberately locked pages, and this command would silently unlock them. What stops that if the command is ever pointed at the wrong node? Candidates: (a) gate the unlock on evidence the node *was* a demo — the presence of the `demo@demo.local` account; (b) require an explicit `--force` or interactive confirmation; (c) unconditional, relying on FM never mis-invoking it. **Come with a recommendation**, and be explicit about how the chosen gate interacts with idempotence: an evidence gate makes a second run a no-op (probably right), but also means a run that deleted the account and *then* failed before unlocking cannot be completed by re-running. Consider ordering the two operations so a retry always converges (e.g. unlock first, delete last).

2. **Confirm the inverted guard** — no `isDemoMode()` check on this command. (Reading-list item 6 in the base prompt.)

3. **Confirm the `demo` role survives** — the account is removed, the seeded role row stays.

4. **Command name** — `demo:convert-cleanup` / `demo:cleanup` / `conversion:cleanup`. Lean: the `demo:` prefix, because that's where an operator scanning `artisan list` will look, even though it deliberately does not share the demo-mode guard. Confirm, don't relitigate.

## Testing

- **Slow test groups to run this session:** none beyond the fast suite.
- **New tests expected: yes.**
  - **Happy path:** with a `demo@demo.local` account present and every page locked, the command removes the account and leaves `pages.locked = false` on all pages; exits `SUCCESS`.
  - **Idempotence:** a second immediate run exits `SUCCESS` and changes nothing further.
  - **Runs in production mode:** the command executes with `APP_ENV=production` (i.e. `isDemoMode()` false) — the case that would fail outright if the demo commands' guard were copied. Pin this explicitly; it is the session's central regression risk.
  - **Non-destructive:** other users, contacts, pages' content, and any other model are untouched — only the demo account row is gone and only `locked` changed.
  - **The safety gate** resolved at question 1 — a test that the command does *not* unlock pages on a node showing no evidence of having been a demo (if the evidence gate is chosen), or whatever the confirmed gate implies.
  - **Exit codes:** `SUCCESS` on the nothing-to-do path; `FAILURE` only on genuine error.
