# 301. Schemes, Template Page-Shell & Chrome, Shared Resolver (Theme/Template Re-Taxonomy 3 of 3–4)

Define a tiny, contrast-vetted scheme set (start: **Default + Inverse**) that a Template *selects* for its content region only, formalise the header/footer slot + "No header / No footer" model, and route every per-template deviation through a **single shared resolver** that delivers request-time inline custom properties (never the bundle) so the page-builder preview is faithful by construction — restoring the deliberate per-page deviation 297 removed. This is the arc's design-completion session; the risk (300, widget consumption) is behind us.

## Session-specific deltas

- **The design is fixed in the stub; this session implements it, it does not re-design it.** Schemes select among the **fixed** `--np-color-*` token set (297, `docs/theme-color-tokens.md`) — no new tokens, no palette change, no per-token page-author editor.
- **Edge-#1 rule resolves every ambiguous case.** *Appearance is always a Theme token, including chrome appearance.* The Template never edits primitives — it makes **structural** choices (header/footer present? which?) and **selects vetted wholes** (a scheme; supply its own chrome). Same principle in both halves (colour *and* chrome): deviate by selecting a coherent unit, never by fiddling individual values.
- **Schemes recolour the content region only — chrome composes, never bleeds.** The standard header/footer always render in their own vetted colours; a dark content scheme does **not** restyle them. A dark landing that wants a dark header *supplies its own* via the slot model.
- **Delivery is the 295 lesson encoded.** Theme defaults stay bundle-compiled + `.np-site`-scoped + 296-guarded. Per-template scheme overrides are **request-time inline custom properties on a page wrapper, resolved from the Template record, never compiled into the bundle.** `AssetBuildService` already documents that these are out of the 296 guard by design — honour that; do not extend the guard.
- **One resolver, two call sites.** A single shared resolver (Template record → inline `--np-*` string) is called by *both* the public layout and the page-builder preview renderer with no divergent branch. That discipline is the only guarantee of preview fidelity and the reason `.np-site` exists.
- **Concrete-values rule applies to the new resolver.** A fresh, unconfigured install must render correctly with no seed row (Default scheme = the 297 token defaults). Stored selections deep-merge over defaults; never sweep stored config.
- **Export/import carry-through is a correctness obligation, not a new feature.** Templates already round-trip through `ContentExporter::serializeTemplate` / `ContentImporter` (versioned envelope). The new `templates` columns (`scheme`, `no_header`, `no_footer`) **must** be carried through both, additively, or template export/import silently drops a page's deviation config — a round-trip regression. This is the *only* export/import work in 301: extend the existing serializer + importer + a round-trip test. The standalone portable-theme feature (the three `design` JSON rows) + media bytes is a **separate** combined post-301 session — the **Theme + Media Portability** stub in `session-outlines.md`, paired with `sessions/media-portability-prompt-draft.md` — explicitly **out of scope here**.

## Open questions to resolve at session start

1. **Scheme vocabulary & storage.** Ship a **fixed Default + Inverse** pair first, or build the power-user named-scheme config surface (a `theme_schemes` SiteSetting mirroring `theme_colors`) now? Recommendation: **fixed Default + Inverse this session**, structured so a named-scheme config layer is an additive follow-up — the page/template author only ever *selects*, so the selector contract is stable either way. Confirm.
2. **Template selector + suppression schema shape.** A `templates.scheme` column (enum/string, default `'default'`) plus two booleans (`no_header`, `no_footer`) — versus reusing the existing nullable `header_page_id`/`footer_page_id` semantics for suppression. Recommendation: explicit `scheme` + explicit `no_header`/`no_footer` flags (a null page-id means "inherit the theme header," not "no header" — distinct states, concrete-values rule). Confirm the column shape before the migration; update `docs/schema/`.
3. **The shared-resolver seam.** Where the one resolver lives and how the public layout and the page-builder preview renderer both call it with no divergence. Identify both call sites before designing; the resolver returns the inline `--np-*` override string for the wrapper, given a Template record.
4. **Inverse scheme derivation (human-judgment).** Is Inverse a mechanical swap (bg↔text/heading, surface darker) or an independently contrast-vetted token assignment? It must be WCAG-legible against the fixed token set. Batch the proposed Inverse values and surface them with multi-surface captures — do not silently pick.
5. **Chrome compose-not-bleed confirmation.** Confirm the rule that a content scheme never recolours the standard chrome, and that "No header/No footer" wins even when a page-id is set. These are the only genuinely new behaviours; the rest is formalising existing inheritance.

## Phases

### Phase 1 — Scheme model + the single shared resolver

Define the Default + Inverse scheme assignments over the fixed `--np-color-*` tokens (Default = the 297 token defaults; Inverse = the contrast-vetted dark content assignment, surfaced for confirmation). Build the **single shared resolver**: Template record → request-time inline `--np-*` custom-property string for the content-region wrapper. Wire it into **both** the public layout and the page-builder preview renderer (no divergent branch). Concrete-values: no Template selection → Default → byte-identical to today. **Not compiled into the bundle**; 296 guard untouched.

### Phase 2 — Per-template scheme selector + chrome page-shell/suppression

`templates` migration (scheme selector + `no_header`/`no_footer`) per the confirmed shape; update `docs/schema/`. Templates Filament resource: a **scheme select** (not a colour editor) + the two suppression checkboxes. Formalise header/footer slot resolution: empty → theme header/footer · filled → template's own · suppressed → none (wins even if filled). Reframe "the default template's header" as "the theme header" in the UI/labels. **Extend `ContentExporter::serializeTemplate` + `ContentImporter`** to carry the three new columns additively (correctness obligation — see deltas; not a new export feature).

### Phase 3 — Verify (objective, scheme-switched) + drift + handoff

Fast suite green incl. the new scheme/selector/parity tests and the unchanged no-hardcoded-colour + 296 guards. Real-browser computed-value verification across **three surfaces**, scheme-switched: public (content region follows the selected scheme; chrome holds its vetted colours), admin (no `.np-site` → no leak), page-builder preview (scheme-faithful, identical to public — flip a template's scheme and confirm public and preview move together). Before/after multi-surface captures. Surface the Inverse-vetting judgment batch. If split per Rule 11, hand off the remaining batch.
