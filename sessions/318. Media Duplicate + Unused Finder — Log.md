**Date:** 2026-05-22
**Branch:** session-318/1

## What was built

An operator-facing **Media Finder** tool (Tools group, `manage_cms_settings`-gated) with three read-only on-demand scans and confirmed per-row deletion. The session also corrected a load-bearing factual error in its own prompt before building, and — at the user's direction during the build — added a third scan and reframed the larger media-cleanup work into a sequenced plan (318 visibility → 319 prevention → 320 reclamation).

### Mid-session prompt correction (drift)

The original prompt assumed the unused scan works by grepping JSON columns for `media_id` / `background_image_id` literals. That is wrong for this codebase: there are essentially no media-id columns — media is attached to owner models through **Spatie collection ownership** (`model_type` + `model_id` + `collection_name`), read back by collection name (`getFirstMedia('post_header')`, `getFirstMedia("config_{key}")`, …). The prompt was rewritten in place (at user request, for connectivity safety) to the actual reference model before any code was written.

### Phase 1 — `MediaReferenceInventory` (the surface inventory)

`app/Services/Media/MediaReferenceInventory.php` — encodes what "referenced" means:

- **Rule A — live collection ownership.** Fixed live-collection sets per owner type (Page `post_thumbnail`/`post_header`/`og_image`; Event `event_*`; Product `product_image`; EmailTemplate `header_image`; WidgetType `thumbnail`/`thumbnail_hover`). PageWidget is dynamic: `appearance_background_image` + `config_{key}` for each image/video field currently in the widget type's `config_schema`. CollectionItem and SampleImage have consumer-driven collection names, so any media owned by a live instance is treated as referenced (conservative — avoids false-positive "unused" flags).
- **Rule B — embedded `/storage/{id}/` URLs in rich-text.** Surfaces scanned: `page_widgets.config`, `collection_items.data`, `events.description`, `events.meeting_details`, `email_templates.body`. Refined in-session: instead of DOM `<img src>`-only extraction, the inventory pulls the media id from the canonical `/storage/{id}/` path token in raw content — a superset that also catches `<a href>`, `srcset`, `<video>`/`<source>`, and conversion paths, so it cannot wrongly flag those as unused. (`inline-images`, owned by PageWidget/Event/EmailTemplate via `InlineImageUploadController`, are kept alive only by this rule.)

`classify(Media)` returns `live` / `dead_collection` / `orphan_owner`.

### Phase 2/3 — `MediaFinderService` (the scans)

`app/Services/Media/MediaFinderService.php`:

- **Unused scan** — every media the inventory does not classify `live`, with classification + display metadata.
- **Duplicate scan** — pass 1 clusters by SHA-256 of stored bytes (`identical_content`); pass 2 clusters the *remaining* rows by `file_name|size` (`same_name_size`). Each member tagged with whether it is currently referenced (so the operator keeps the in-use one).
- **Missing-file (404) scan** *(added mid-session at user request)* — media rows whose file is gone from disk, checked via `Storage::disk($media->disk)->exists($media->getPathRelativeToRoot())` (cross-disk safe). Tagged `Broken (referenced)` (a live surface points at it now) vs `Dead record`.

### Phase 4/5 — `MediaFinderPage` + delete

`app/Filament/Pages/MediaFinderPage.php` + `resources/views/filament/pages/media-finder.blade.php`. Three header actions (one per scan); results render in a narrative-headed blade with per-row confirmed **delete** via the `mountAction('deleteMedia', …)` pattern (mirrors `RecordTimelinePage`). Deletes route through Spatie's model `delete()`; deleted ids are filtered out of the in-memory result sets so the UI updates without re-scanning. No bulk delete, no auto-delete.

### Phase 6 — tests + docs

- `tests/Feature/MediaFinderTest.php` — 16 tests: one per inventory rule (live config collection, appearance background, fixed-collection owner, dead collection, embedded-URL keep-alive, storage-id extraction from href/srcset, orphan owner), duplicate clustering (identical-content, same-name-size, referenced-member tagging), missing-file (surfaced, broken-referenced flag, dead-record delete), page gate, and the two delete paths.
- `docs/runbooks/media-finder.md` (operator runbook + safety model) and `resources/docs/media-finder.md` (help-doc stub, route registered).

### Merge: deliberately not built

At the open gate we agreed on delete + merge. Building delete confirmed merge has no safe meaning in this reference model: duplicate files are each owned by a *distinct* owner+collection (the same logo uploaded to two widgets), and Spatie single-file collections own exactly one row each — so "merge two owned copies into one canonical" can't consolidate without breaking an owner. The only well-defined merge is embedded-URL duplicates, and seed data has zero. Surfaced to the user, who agreed to skip merge in v1 and pursue the real fix (below) as follow-on sessions.

### Sample-data run (seeded local DB)

178 media → **8 unused** (all `orphan_owner`), **18 duplicate clusters** (all `identical_content`, sizes 2–21 — heavy sample-image reuse across seeded widgets), **0 missing files**, **0 embedded references** in seed content.

### Bug found and fixed

The duplicate scan's name+size pass excluded *every hashed* row instead of only rows already in a hash *cluster* — so it never fired for normal files. Fixed; pinned by the `same_name_size` test.

## Security considerations

The page is gated behind `manage_cms_settings` (matching `SiteImportExportPage`); both the header scan actions and the delete action re-check the permission with `abort_unless`. Deletion is per-row and operator-confirmed via a Filament modal — no bulk path, no automatic deletion. The scans are read-only. The embedded-URL rule reads stored rich-text content but only to extract numeric media ids from `/storage/{id}/` tokens; no content is rendered or executed. Orphan-owner deletion is intentionally left to the existing nightly `media-library:clean` sweeper and only *reported* here.

## Deferred decisions

- **Merge** — skipped in v1 (no safe meaning under Spatie's ownership model; see above). If real installs show embedded-URL duplicates, revisit.
- **Disk reclamation** — the finder is visibility-only. The actual size win requires content-addressed storage + refcounted delete; sequenced as **session 320** (folds in the existing "Media Path Generator — UUID Shape" stub).
- **Upload-time dedup prevention** — the root-cause fix for the compounding re-upload problem; sequenced as **session 319** (content_hash column + warn / use-existing / replace-existing across all upload surfaces, with a most-referenced-first disambiguation list when more than one asset matches). Both 319 prompts drafted at this close.
- **Scan execution model** — synchronous v1 confirmed (178-file local run was instant); promote to queued only if a real library times out.
