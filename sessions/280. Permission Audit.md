## Session-specific deltas

### Persona-vs-role mismatch — resolve at session start

The plan-entry's four personas (volunteer / board-read-only / staff-admin / public-visitor) do not map cleanly to the shipped role vocabulary (`super_admin`, `cms_editor`, `crm_editor`, `event_manager`, `volunteer_coordinator`, `treasurer`, `blogger`, `developer` — see `database/seeders/PermissionSeeder.php`). Three options for resolution; pick one at session start before walking:

- **(a) Walk shipped roles, cross-reference personas.** The matrix's columns are the 8 shipped roles + "unauthenticated"; the persona names from the plan entry land as a header annotation indicating which shipped role(s) best fulfill each persona. Closes the success criterion via cross-reference rather than literal role-name alignment.
- **(b) Walk personas, treat as role-groupings.** Map "volunteer" → `volunteer_coordinator` (closest); "board-read-only" → no current role, audit shows this as an unfilled persona; "staff-admin" → `super_admin` or a combined `cms_editor` + `crm_editor` projection; "public-visitor" → unauthenticated. Matrix columns are the four personas. More compact but loses fidelity for the 4 roles outside the persona set (`event_manager`, `treasurer`, `blogger`, `developer`).
- **(c) Both.** The matrix has 8 role columns; a separate persona-mapping table maps each persona to its closest shipped role(s). Most thorough; biggest doc.

Lean toward **(a)** — preserves audit fidelity, satisfies the success criterion via the persona annotation, doesn't bloat the matrix. **(c)** if the matrix produces surprises that warrant the cross-walk. Confirm at session start.

### Audit-style absorption rule (release-plan Rule 2)

Findings that fit in a small in-session fix (e.g. a missing `canViewAny()` callback that should mirror an established pattern) get fixed inline. Findings that exceed in-session-fix capacity (e.g. a controller doesn't enforce a gate at all and needs a full middleware addition + tests + downstream review) lift to a follow-on entry and stay in the matrix as `deferred to 32b` or `open flag for NNN+1`.

Apply the standing rule `feedback_audit_findings_default_to_fix.md` — surface inclusively in the matrix; let the apply walkthrough (close gate or 32b session) decide won't-fix. Do not filter on the audit side.

## Open questions to resolve at session start

- **Persona-vs-role mapping shape** — (a) / (b) / (c) above. Lean (a).
- **In-session-fix capacity ceiling.** Numeric guess: about 3–5 small fixes feel sane in one session; beyond that, defer. Confirm or adjust based on early walk findings.
- **Test shape for the matrix.** Either codify it as a Pest test (`it allows role X to view resource Y` × N — explicit and slow but executable) or leave the matrix as documentation only and rely on existing per-resource auth tests. Lean codified (Pest), but the cell count may make this 60+ tests and not all of them need spec-level coverage. Pick a "spec-worthy threshold" at session start (e.g. only codify cells where the audit found and fixed a gap).
- **Direct-URL reachability.** Filament's `canViewAny()` hides the resource from the nav and the list pages; it does NOT automatically gate `/admin/{resource}/{id}/edit` against deep links. The audit verifies both layers — what's the simplest probe? Pest feature tests visiting the URL as each role and asserting 403 / 200 is the canonical check.

## Phases

### Phase 1 — Persona-vs-role mapping + matrix scaffold

- Pick the matrix shape (per the open question above).
- Create `docs/runbooks/permission-matrix.md` with the header table (columns = roles or personas, rows = surfaces) and the persona-mapping section.
- Stub-row every Filament resource and page (so we have a checklist to walk). Mark every row as "not yet walked."
- Land the doc skeleton as a small commit so the walk has a tangible artifact growing in-flight.

### Phase 2 — Walk Filament resources

For each resource under `app/Filament/Resources/*.php`:

- Read the resource. Identify which `canViewAny()` / `canCreate()` / `canEdit()` / `canDelete()` callbacks exist (or are absent, which means "default permissive").
- For each role × this resource: determine UI-gate status (gated / open / missing-but-implied).
- Probe direct-URL reachability for at least 2 representative roles (likely `cms_editor` and `volunteer_coordinator` — chosen for being non-super-admin with distinct permission scopes). For the resource's `/edit` route, a Pest feature test logged in as that role, asserting 403 if the role shouldn't have edit, 200 if it should.
- Fill the matrix row. Note any findings as `deferred to 32b` or `open flag` or fix-inline-this-iteration.
- Iterate over the 27 resources. Walk likely splits across 2–3 iterations given the size.

### Phase 3 — Walk Filament pages and non-Filament controllers

- Pages under `app/Filament/Pages/*.php` — each has a `canAccess()` callback. Same shape as resources.
- Sub-pages under `app/Filament/Resources/{Resource}/Pages/` — e.g. `ViewRegistrations.php` (session 278), `EditEvent.php`, etc.
- Non-Filament admin-shaped controllers under `app/Http/Controllers/` — anything that's reachable from admin context and isn't a public-facing endpoint. Auth middleware coverage is the gate here.

### Phase 4 — Codify the matrix as a Pest spec (judgment)

Per the "test shape" open question — if the spec-worthy threshold is "only codify fixed cells," write a single Pest test file (e.g. `tests/Feature/PermissionMatrixTest.php`) that asserts the fixes hold. If "codify the full matrix," the test file becomes the executable spec for the doc.

The user's standing test discipline applies (per `memory/feedback_verify_objective_outcomes.md`): Pest feature tests visiting admin URLs as each role with `actingAs(...)` and asserting status codes is a verification mechanism that closes the loop without pulling the user in.

### Phase 5 — In-session fixes

For each finding that fits the small-fix definition: edit the resource / page / controller to add the missing `canViewAny()` / `canAccess()` callback or the missing middleware. Test it. Update the matrix cell to `fixed in-session 280/N`.

## Out of scope

- **Concurrent admin editing** (the 2nd of C3's three concerns) — splits to 32b. Specifically: two admin sessions editing the same contact / page simultaneously, last-write-wins vs warning UX, no data corruption.
- **Accidental public exposure** (the 3rd of C3's three concerns) — splits to 32b. Specifically: sensitive-field-toggle gating, public-content indicators on records and widgets.
- **Adding new roles.** The shipped 8 are what's audited. If the audit finds a gap that a new role would cleanly fill (e.g. the missing "board-read-only" persona), surface as an open flag for design conversation, not as in-session work.
- **Restructuring `PermissionSeeder.php`.** The seeder's vocabulary is the audit reference. Edits to it during in-session fixes are scoped to adding permissions an existing role should have (or removing ones it shouldn't); not refactoring the structure.
- **Portal-user permissions.** Portal users are contacts, not admins. The portal-auth middleware (`['portal.auth', 'verified:portal.verification.notice']`) is the gate; this audit verifies it's in place but doesn't redesign the portal permission model.
- **Stripe / billing operations gates.** Treasurer-role-specific work (e.g. finance settings page, transaction admin) is in the audit, but operational concerns like "treasurer should be able to refund but not see donor home addresses" are part of the data-classification work that splits to 32b (accidental exposure).
