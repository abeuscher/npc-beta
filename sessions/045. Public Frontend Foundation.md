# 045. Public Frontend Foundation

Establish the complete public-facing frontend build pipeline. Vite is not currently configured at the project root — this session builds it from scratch. Pico CSS is introduced as the base styling framework via SCSS. The existing public layout and any ad-hoc public CSS are discarded and rewritten cleanly. The output is a compiled `public.css` and `public.js` artifact served to all public pages, with a server-side inline `<style>` block bridging stored site settings (brand color, fonts) to Pico's CSS custom properties without a build step.

---

## Before writing any code

Read the following files:

- `resources/views/` — locate the public Blade layout file and any public-facing view files. Understand the current structure before discarding it.
- `app/Settings/GeneralSettings.php` — understand what branding/appearance fields currently exist (site name, logo, colors if any)
- `config/settings.php` — understand how settings are stored and cast
- `routes/web.php` — understand the public route structure
- `resources/css/` and `resources/js/` — inventory anything currently there
- `package.json` — confirm whether npm/Vite is already configured at root level. If the file does not exist at the project root, it needs to be created from scratch.

## Part 1 — npm and Vite setup

### If `package.json` does not exist at the project root

Create it. Required dependencies:

```json
{
  "private": true,
  "type": "module",
  "scripts": {
    "dev": "vite",
    "build": "vite build"
  },
  "devDependencies": {
    "laravel-vite-plugin": "^1.0",
    "vite": "^5.0",
    "sass": "^1.70",
    "autoprefixer": "^10.4",
    "cssnano": "^7.0"
  },
  "dependencies": {
    "@picocss/pico": "^2.0",
    "alpinejs": "^3.0"
  }
}
```

### `vite.config.js`

Create at project root:

```js
import { defineConfig } from 'vite'
import laravel from 'laravel-vite-plugin'

export default defineConfig({
  plugins: [
    laravel({
      input: [
        'resources/scss/public.scss',
        'resources/js/public.js',
      ],
      refresh: true,
    }),
  ],
})
```

### `postcss.config.js`

Create at project root:

```js
export default {
  plugins: {
    autoprefixer: {},
    cssnano: { preset: 'default' },
  },
}
```

Run `npm install` inside the app Docker container after creating these files:

```bash
docker compose exec app npm install
```

## Part 2 — SCSS structure

Create the following files. **Do not deviate from this structure** — it is the foundation that session 046 builds on.

### `resources/scss/_theme.scss`

This is the user-editable file. Session 046 will expose it in the admin SCSS editor. Pre-populate it with Pico variable overrides and a clear comment block:

```scss
// ─────────────────────────────────────────────────────────────────────────────
// Theme — Pico CSS variable overrides
//
// This file is exposed in the admin SCSS editor (CMS › Site Theme).
// Define Pico variable overrides here. You may also add custom SCSS.
// Changes here require a build (npm run build) to take effect on the public site.
// Brand color and font choices set in CMS › Site Theme › Appearance are applied
// at runtime via inline CSS custom properties and do not require a build.
// ─────────────────────────────────────────────────────────────────────────────

// Uncomment and adjust to override Pico defaults:
// $semantic-root-element: ":root";
// $breakpoints: (...);
// $spacing-block: 1rem;

// Custom SCSS (acts on Pico markup or any public-facing HTML):
```

### `resources/scss/public.scss`

The compiled entry point. This file is **not** user-editable:

```scss
// Import theme overrides before Pico so variables take effect
@use "theme";

// Pico CSS — classless semantic theme
@use "../../node_modules/@picocss/pico/scss/pico";

// Public layout utilities
@use "layout";
```

### `resources/scss/_layout.scss`

A minimal layout file for structural rules that are not part of Pico:

```scss
// ─────────────────────────────────────────────────────────────────────────────
// Layout — structural rules not covered by Pico
// ─────────────────────────────────────────────────────────────────────────────

// Page-type body classes — reserved for per-page-type overrides
// body.page-type-event { ... }
// body.page-type-post  { ... }
```

## Part 5 — Public Blade layout rewrite

Locate the existing public layout file (likely `resources/views/layouts/public.blade.php` or similar). **Discard its contents entirely** and rewrite it.

The new layout must:

1. Use `@vite(['resources/scss/public.scss', 'resources/js/public.js'])` to load the compiled assets.

2. Render an inline `<style>` block in `<head>` that bridges site settings to Pico CSS custom properties. The block is only rendered if at least one setting has a value:

```blade
@php
    $settings = app(\App\Settings\GeneralSettings::class);
    $cssVars  = [];

    if ($settings->primary_color) {
        $cssVars[] = "--pico-primary: {$settings->primary_color}";
    }
    if ($settings->heading_font) {
        $cssVars[] = "--pico-font-family-heading: {$settings->heading_font}";
    }
    if ($settings->body_font) {
        $cssVars[] = "--pico-font-family-sans-serif: {$settings->body_font}";
    }
@endphp

@if ($cssVars)
<style>:root { {{ implode('; ', $cssVars) }}; }</style>
@endif
```

3. Add a `body` class combining the page type and slug:
   - Pages: `page-{slug}` and `page-type-{type}` (e.g. `page-about`, `page-type-default`)
   - Posts: `post-{slug}`
   - Events: `event-{slug}`
   - Fallback: `page-unknown`

   The layout is used by multiple view types. Pass the body class string from each view via a `$bodyClass` variable, with a sensible default. Each public route's view is responsible for setting it.

4. Include the site navigation (existing header/nav component) and footer.

5. Structure the main content area using Pico-compatible semantic HTML (`<main>`, `<article>` where appropriate).

6. Include the `<meta charset>`, `<meta name="viewport">`, and `<title>` as before. Pull the site name from GeneralSettings.
