# 200. Admin CSS Consolidation ‚Äî Tightening Pass

Reduce drift in `public/css/admin.css` without changing its architectural role. Three specific problems, all scoped to how the file is organised:

1. **Vue dark-mode rules live in the wrong file.** ~200 lines of `html.dark .editor-*` / `.inspector-*` / `.color-picker__*` / etc. currently sit in `admin.css`, far from the Vue components they paint. They get migrated into each component's `<style scoped>` block (or `inspector.css` for shared primitives) alongside the existing light styles.
2. **Raw RGB values bypass the `--np-control-*` token vocabulary** that already exists at the bottom of the file. Rules that use hardcoded `rgb(156 163 175)` / `rgb(209 213 219)` get switched to the matching token where one fits.
3. **The Trix toolbar block uses `!important` on every property.** Specificity gets fixed so the flags can come off.

The file stays plain CSS, stays loaded via the render hook, and stays at `public/css/admin.css`. **No architectural change.** Filament's Vite theme entry is out of scope ‚Äî that's a separate, larger session.

---

## Design decisions (resolved before starting)

- **Scope boundary.** Only the three problems above. The help-system Blade chrome (~250 lines), the Quill toolbar block, and the Filament `fi-*` ring overrides all stay where they are, untouched except where they use raw RGB that has a token equivalent.
- **Dark-mode colocation rule.** For each Vue component whose dark rules currently live in `admin.css`, the dark rules move into that component's `<style scoped>` block, prefixed with `:global(html.dark)` (or `html.dark &` syntax if the Vue scoped sass/postcss pipeline supports it ‚Äî verify before committing to a form). If the component is one of the shared-primitive classes in `inspector.css`, the dark rules move into `inspector.css` alongside the light rules. No Vue file gets a new `<style>` block added for this ‚Äî only files that already have one.
- **Token-substitution rule.** Only substitute where the existing token's value is an **exact match** for the raw RGB. No re-colouring. No introducing new tokens. If a rule uses a shade that isn't in the token set, it keeps its raw RGB. The goal is to stop manually translating Tailwind shades, not to redesign the palette.
- **`!important` removal rule.** For the Trix toolbar block: try to remove `!important` by raising selector specificity (e.g. `.fi-fo-rich-editor-toolbar .fi-fo-rich-editor-toolbar-btn` instead of `.fi-fo-rich-editor-toolbar-btn`). If that fails against a current Filament build, leave the `!important` in place and note it in the session log. Do not strip `!important` without confirming the resulting cascade still wins.
- **No visual regressions.** The net result must be pixel-identical in both light and dark modes. Every migrated block gets a manual before/after check.
- **No new systems.** No SCSS, no PostCSS migration, no Tailwind `@layer` usage, no moving anything into the Vite theme entry. If a migration step requires any of those, stop and ask.

## Phase 1 ‚Äî Vue dark-mode migration

The following blocks in `admin.css` all target Vue page-builder components and must move out. Work one component at a time. For each:

1. Identify the owning `.vue` file (the selector names map 1:1 to BEM-style component classes).
2. Read its `<style scoped>` block so the light-mode rules are understood.
3. Move the corresponding `html.dark .foo` rules from `admin.css` into the scoped block, rewritten to work inside Vue's scoped-style cascade (`:global(html.dark) .foo { ... }` or equivalent for this project's tooling ‚Äî confirm from an existing scoped block if any component already handles dark mode inline).
4. Remove the migrated lines from `admin.css`.
5. Build and manually verify the component in both themes before moving on.

Migration list (from `admin.css` lines 78‚Äì258 roughly):

- **`.vue-editor`** (root chrome rules ‚Äî background, border, color).
- **`.editor-toolbar__*`** ‚Üí `EditorToolbar.vue`.
- **`.preview-canvas__*`** ‚Üí `PreviewCanvas.vue`.
- **`.add-block-dropdown__*`** ‚Üí `AddBlockDropdown.vue` (find the component with that class).
- **`.inspector-panel__*`** ‚Üí `InspectorPanel.vue`.
- **`.inspector-header*`** and **`.inspector-header__*`** ‚Üí `InspectorHeader.vue`.
- **`.inspector-tabs*`** ‚Üí `InspectorTabs.vue` (or wherever the tabs live inside InspectorPanel).
- **`.inspector-label`, `.inspector-section-title`, `.inspector-control`, `.inspector-hint`, `.inspector-toggle-label`, `.inspector-checkbox`** ‚Üí `inspector.css` (these are the shared primitives). Add dark-mode variants there, keyed off `html.dark` at the file level.
- **`.bg-panel__*`** ‚Üí `BackgroundPanel.vue`.
- **`.inspector-notice`** ‚Üí `NoticeField.vue` (from the known Vue fields list).
- **`.color-picker__*`** ‚Üí `ColorPicker.vue`.
- **`.gradient-picker__*`** ‚Üí `GradientPicker.vue`.
- **`.nine-point__*`** ‚Üí `NinePointAlignment.vue`.

- **`.spacing-control`, `.query-settings`, `.widget-appearance`, `.layout-inspector`, `.block-list*`** ‚Üí respective components (grep to find each).
- **`.apply-changes`** ‚Üí `ApplyChangesButton.vue` (or similar).

At the end of Phase 1, `admin.css` should be shorter by the migrated block range, and `html.dark .editor-*` / `.inspector-*` / etc. should no longer appear in `admin.css` (exceptions noted in the log).

## Phase 2 ‚Äî Token substitution

Sweep the remainder of `admin.css` (after Phase 1 migration) and substitute raw RGB values where a `--np-control-*` token is an exact match. The token values are:

```
--np-control-bar-bg:        #eceff3  /  rgb(17 24 39)    (dark)
--np-control-group-bg:      #fafbfc  /  rgb(31 41 55)    (dark)
--np-control-border:        #d1d5db  /  rgb(55 65 81)    (dark)
--np-control-chip-border:   #d1d5db  /  rgb(75 85 99)    (dark)
--np-control-border-hover:  #9ca3af  /  rgb(99 102 241)  (dark)
--np-control-border-active: #6b7280  /  rgb(129 140 248) (dark)
--np-control-hover-tint:    rgba(17, 24, 39, 0.06)
--np-control-active-bg:     #e5e7eb  /  rgb(55 65 81)    (dark)
```

Focus areas:

- **Filament `fi-*` ring overrides** (lines ~272‚Äì305). `rgb(156 163 175)` (gray-400) is currently used for section / input / fieldset / table outlines. That's NOT an exact match for `--np-control-border` (`#d1d5db` = gray-300), so leave these alone unless the user agrees to shift them. Flag the mismatch in the log as a design question ‚Äî do we want form outlines tightened from gray-400 to `--np-control-border` (gray-300), or loosened the other way?
- **Table row dividers** (`rgb(209 213 219)` / gray-300) ‚Äî this IS an exact match for `--np-control-border`. Swap.
- **Trix toolbar block** ‚Äî `rgb(209 213 219)` (gray-300) separator and button border ‚Üí `--np-control-border`. `rgb(229 231 235)` (gray-200) hover ‚Üí no exact token match, leave as raw RGB or note for future token addition.
- **Help system chrome** ‚Äî out of scope. Help has its own colour language (`#9ca3af`, `#e5e7eb`, `#f3f4f6`, etc.) and doesn't consume control tokens. Leave alone.

Do not introduce new `--np-*` tokens in this session. If a shade needs one, note it in the log for a future pass.

## Phase 3 ‚Äî Trix `!important` removal

The Trix toolbar block (lines ~333‚Äì380) uses `!important` on every declaration. For each rule:

1. Try raising specificity by prepending a parent selector (e.g. `.fi-fo-rich-editor .fi-fo-rich-editor-toolbar-btn`).
2. Remove the `!important` flag.
3. Manually verify the button still paints correctly on a page with a rich-editor field (the Event description editor, or any Quill-field page).

If specificity alone doesn't win against Filament's utility classes, leave the `!important` in place and document the failure in the log. Do not use more aggressive hacks (e.g. `html body .fi-...`) to force the cascade.
