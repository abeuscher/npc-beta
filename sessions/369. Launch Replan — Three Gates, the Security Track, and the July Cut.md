## The three launch gates (proposed; confirm wording at the Open Gate)

- **Gate 1 — Looks professional to paid traffic.** The public site and demo render correctly and feel credible on the browsers and devices real donors and prospect EDs actually use, and the public money flows don't misbehave on bad connections. This is the *public-facing slice* of the old compatibility work — not the full QA program.
- **Gate 2 — Safe to hold one real client's money and PII.** Donations work end to end and are provably acknowledged (tax receipt), events sell tickets (core feature — comp/$0 path included), the data can't be lost (backup/restore proven on production-shape infra), and the legal register is live (Privacy + ToS — Stripe requires both URLs).
- **Gate 3 — The owner believes it's secure.** The self-driven hardening pass, built from the security surface map's top-5 ranking plus the parked security flags and the security-flavored items already in the plan (accidental-exposure audit, portal-scoping E2E, permission-matrix E2E). Internal review is the release bar; external help is a bonus, not a dependency.

## Session-specific deltas

What this session must actually decide, beyond mechanically sorting:

- **Size the security track for real.** The old plan carries `── SECURITY AUDIT READY ──` as unsized post-Beta intent. This session turns the surface map's top-5 into named sessions with scopes and an order. A starting decomposition to react against, not to rubber-stamp — the session may merge/split/extend it:
  1. **Perimeter + headers session** — security-headers middleware (CSP first among them) + nginx enforcement; self-host the Quill editor assets (CSP-blocking + supply-chain); `SESSION_SECURE_COOKIE` and friends get enforced defaults, not `.env` conventions.
  2. **Second-lock session** — app-layer auth gate on the five fleet mTLS endpoints, worst-first (`/api/backup/blob`, `/api/admin/recover`); resolve the raw-log-tail PII tension on `/api/logs`. *(Boundary-touching — contract bump; plan for it.)*
  3. **XSS containment session** — audit every rich-text write path funnels through `HtmlSanitizer` (importer, seeder, API paths especially); the member-page widget-scoping check the map left open; CSV formula-injection on export.
  4. **Adversarial pass session(s)** — attacker-goal-driven probing of the public write surface and the two E2E security suites (portal scoping F2, permission matrix F3) folded in here rather than standing alone at the old positions.
- **Propose the disposition of all 20 items** — every item lands in exactly one bucket (*gate-1/2/3*, *deferred-until-customer*, *cut*, *register*) with a one-line rationale. Contested calls to bring to the checkpoint explicitly rather than bury: A3 (four nodes vs the minimum to launch), A4 (backup drill — Gate 2, but right-sized), D1 (1000× scale rehearsal for a contacts-and-dollars first client?), D2 (what's the Gate-1 slice vs the rest), C3a/#32b (admin accountability/concurrency — pre- or post-customer?), G2/D4/T1 (internal QA hygiene under a deadline), E13/E14 (help docs, licensing — cheap but are they gating?), F1 (donation E2E depth — Gate 2 or Gate 3?), C4/C5's QuickBooks legs.
- **Sequencing constraints that survive the re-sort:** the old plan's "integration retest runs last among rehearsals" rule; C3b (auto tax receipt) before the donation loop rehearsal; C3c ($0 tickets) before the event rehearsal; the Privacy/ToS register item is external-blocking (counsel) and needs to start now regardless of session order.
- **Fit the schedule to the budget.** Sum the ratified plan; if it exceeds ~30–40 sessions including a contingency margin, cut deeper — the budget wins, not the list. State the assumed pace (owner: 3–4 sessions/day, ~10 working days) and mark the earliest credible "ads on" date.
- **Say where FM-repo work fits.** Not scheduled here (separate repo), but the plan should state what the launch does/doesn't need from FM — e.g. billing pushes (FM-B2) are not launch-gating because the first client can be billed by hand in Stripe.
