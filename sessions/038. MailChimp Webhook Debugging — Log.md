# Session 038 — MailChimp Webhook Debugging — Log

## What was fixed

The MailChimp unsubscribe webhook was returning HTTP 500 for every real request from MailChimp's servers, while a curl simulation from the production server returned 200 OK.

Two bugs were found stacked on top of each other:

### Bug 1 — Log file permissions (masking bug 2)

`storage/logs/laravel.log` was owned by `root` because artisan commands in the container run as root. PHP-FPM runs as `www-data` and could not write to the file. The controller's `Log::error()` call caused Monolog's `StreamHandler::write()` to throw, which became the 500 response. No log entries were produced for any web request, making the real error invisible.

**Fix:** `docker exec nonprofitcrm_app chown www-data:www-data storage/logs/laravel.log`

This must be re-applied any time artisan commands write to the log and change its ownership. A permanent fix (chown in the container startup script or Dockerfile) is deferred.

### Bug 2 — Wrong assumption about MailChimp's payload format

The controller assumed MailChimp sends `data` as a JSON string and called `json_decode($request->input('data', '{}'), true)`. MailChimp actually sends the data as nested form fields (`data[email]=...&data[id]=...`), which PHP parses into an array. Calling `json_decode()` on an array throws a `TypeError` in PHP 8.

The curl simulation used `data={"email":"..."}` (a JSON string), which worked — masking the format mismatch.

**Fix:** Check whether `data` is already an array before attempting to decode it:

```php
$rawData = $request->input('data', []);
$data    = is_array($rawData) ? $rawData : (json_decode($rawData, true) ?? []);
```

## Diagnostic path

The key steps that led to the fix:

1. Identified that the 500 response was ~200KB — the Ignition debug page, not a minimal error — confirming Laravel was running and generating the error
2. Identified that the Ignition page's CSS referenced `Monolog\Handler\StreamHandler:164` (the `write` method) — the logger itself was throwing
3. Checked log file ownership: `root root` with `644` permissions — `www-data` cannot write
4. Fixed ownership → log entries immediately appeared, revealing the `json_decode` TypeError
5. Fixed the controller to handle both array and string formats

## What was learned

The session 037 assumption that "MailChimp sends `data` as a JSON string" was incorrect. For any future third-party webhook integration, the actual documented payload structure from the vendor must be obtained and provided before implementation begins. Notes added to the Stripe Webhooks and QuickBooks Integration stubs in session-outlines.md.
