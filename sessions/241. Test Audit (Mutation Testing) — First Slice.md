# 241. Test Audit (Mutation Testing) — First Slice

Install Infection (mutation testing) against this codebase, run it on a bounded first slice (the Widget Primitive contract-resolution layer — `app/WidgetPrimitive/{ContractResolver,DataContract,Source}.php` and the eleven `*ContractRetrofitTest.php` files), produce a findings report keyed by test file (dead / redundant / load-bearing / surviving mutants), pause for user review, apply approved deletions and consolidations, and ship a workflow doc at `docs/testing/mutation-audits.md` so the next mutation pass — in a fresh context — does not have to rediscover the toolchain. The point is twofold: (1) introduce an external signal for which tests carry weight, distinct from "does Pest pass," and (2) leave behind a repeatable mechanism for periodic future audits.

---

## Design decisions (resolved before starting)

- **Slice = Widget Primitive contract-resolution layer.** Production code: `app/WidgetPrimitive/ContractResolver.php`, `DataContract.php`, `Source.php`. Tests: the eleven `tests/Feature/*ContractRetrofitTest.php` files plus any unit tests Phase 1's discovery pass identifies as exercising those classes. Rationale: cohesive, recently consolidated (Widget Primitive track substantially closed at session 237 — test surface is curated, not session-old drift), pure-PHP deterministic logic (no DB writes, no HTTP, no async — Infection runs fast on this), large enough to teach but small enough to ship in one session. Importer, page-builder save flow, and finance modules are deliberately out of scope for this first slice — they have edge-case-heavy branches that would dominate the report and dilute the signal.
- **Coverage driver = PCOV, opt-in only.** PCOV is ~10× faster than Xdebug for line coverage. The Dockerfile gains a PCOV install but the extension is not loaded in the default test profile (enabling it slows the suite measurably). The workflow doc names the env-var or `php -d` flag invocation pattern that activates it for Infection runs only.
- **Redundancy threshold for "consolidation candidate" = N≥2.** A test is flagged redundant if every mutant it catches is also caught by 2 or more other tests. Strict enough to flag genuine duplication; loose enough to keep the second test as belt-and-suspenders for load-bearing logic. The user adjusts the threshold during Phase 2 review if it produces too much (or too little) signal.
- **Single session, with a hard pause after Phase 2.** Not a 205/206-style audit-then-apply pair. The pause point is the load-bearing session moment: Claude surfaces the findings table and stops; the user reviews and approves which rows to act on; only then does Phase 3 begin. This is enforced in the base prompt's process-rules section.
- **Two-commit branch shape.** First commit: Infection install + PCOV-in-Dockerfile + `infection.json5` + workflow doc (lands even if Phase 3 produces nothing). Second commit: Phase 3 deletions and consolidations, if any. Splitting means the infrastructure is reusable independently of any single audit's findings.
- **Override path for tests kept against mutation evidence.** When the user keeps a test that Infection's report flagged as dead or redundant — typically because it guards a known historical regression — Phase 3 adds a one-line `// guards: <regression description>` comment above the test body. This prevents the next mutation pass from re-flagging it. The `// guards:` token is searchable.
- **New tests for surviving mutants are out of scope.** If Infection surfaces a real bug via a surviving mutant, file it as a follow-up against the "Test Suite Audit — Cost, Coverage, and Shape" stub in the roadmap. Do not write the missing test in this session — the deliverable here is **fewer, sharper tests**, not net-new coverage.
- **Failure-mode plan — what if Phase 3 finds nothing?** That is a real outcome and a successful one. The session still ships Phase 1 (Infection installed and configured), Phase 2 (the report itself, archived in the session log), and Phase 4 (the workflow doc). The slice is now known to be well-tested, and the next mutation pass — against a different slice — can run from a fresh context using the documented workflow. Do not invent deletions to justify the session.
- **Workflow doc location = `docs/testing/mutation-audits.md`.** The `docs/testing/` directory does not yet exist; this session creates it. Sets the precedent for future testing-methodology docs (Playwright wiring, fast/slow boundary policy, integration-test discipline).
- **Not a boundary-touching session.** The Fleet Manager agent contract surface (`app/Http/Controllers/Api/Fleet/*`, the auth middleware, the response schema, `config/fleet.php`, the VERSION pipeline) is locked at v1.1.0 from session 240 and is not modified. If a Phase 3 deletion would touch any of those, stop and ask.

## Phase 1 — Install and configure Infection

Add Infection as a dev dependency, install PCOV in the app image, configure Infection against the slice, and confirm a smoke-test run produces a usable report.

**Toolchain install:**

1. **Composer dev dependency.** `docker compose exec app composer require --dev infection/infection`. Pin to the latest stable that supports PHP 8.2+ (currently `^0.27` or `^0.29` depending on what resolves cleanly against the existing dev deps — Pest 2 + Mockery 1.6 + the rest of the lockfile). If Composer reports a conflict, stop and surface it before retrying.

2. **PCOV in the Dockerfile.** Add PCOV install + extension activation behind an opt-in build arg or env-var pattern. Concretely, in the `app` build stage:
   - Add `pecl install pcov` (or `apk add php82-pecl-pcov` if the base image is alpine-php — match the existing extension-install style in the file).
   - Do **not** add a default `docker-php-ext-enable pcov` line — instead, leave the extension installed-but-not-loaded, and document the activation pattern in the workflow doc (`php -d extension=pcov -d pcov.enabled=1 vendor/bin/infection ...` or similar). Loading PCOV by default would slow every Pest run.
   - Rebuild the local image (`docker compose build app`) to pick up the new extension. Confirm the rebuild does not break the existing test suite — run `docker compose exec app php -d memory_limit=2G vendor/bin/pest --exclude-group=slow` and verify 1678 cases still pass.

3. **`infection.json5` config at repo root.** Configure for the slice:
   - `source.directories`: `["app/WidgetPrimitive/"]` (full directory — Infection handles file-level filtering via `--filter` at runtime if needed).
   - `tests.directories`: defaults — Infection auto-discovers via the existing `phpunit.xml`/`pest.xml` config.
   - `phpUnit.configDir`: point at the project root if needed.
   - `logs`: emit both XML (`infection.xml`) and HTML (`infection.html`) — XML is what Phase 2 parses, HTML is for human spot-checks. Optional: `text` log for terminal-friendly summary.
   - `testFramework`: `pest` (Infection 0.27+ supports Pest natively; older versions need PHPUnit-compatible test running).
   - `mutators`: start with `@default` — Infection's curated default set. Custom mutator selection is post-v1 of the workflow.
   - `timeout`: per-mutant default (10s). If the slice has any test that legitimately runs longer, bump in the Phase 1 smoke-test step.

4. **`.gitignore` excludes.** Add `infection.xml`, `infection.html`, `infection-log.txt`, `.infection-cache/`, and any other Infection-emitted artifacts at repo root. The session ships the **config** and the **doc**, not the report — every mutation run produces a fresh report.

**Smoke-test run.** Run Infection against a tiny target — e.g., `vendor/bin/infection --filter=ContractResolver.php --threads=1` against a single source file with PCOV enabled. Confirm:

- The run completes without crashing.
- The XML report is produced and parseable.
- At least one mutant is generated (i.e., Infection's mutator engine is finding instrumentable code).
- The fast Pest suite still passes after the smoke-test (Infection should not leave the codebase in a mutated state).

If the smoke-test fails, stop and surface the error before proceeding to Phase 2. Common failure modes: PCOV not actually loaded (Infection falls back to "no coverage available"), Pest discovery mismatch, test framework version conflict.

**Phase 1 completion criteria:** Infection installed, PCOV available behind opt-in, `infection.json5` checked in, `.gitignore` updated, smoke-test passes, fast Pest suite still 1678 green. **Commit at this point** as the first of the two-commit shape — even if Phase 2 produces nothing actionable, the toolchain is now reusable.

## Phase 2 — Run the substantive Infection pass and analyse the report

Run Infection against the full slice (`app/WidgetPrimitive/ContractResolver.php`, `DataContract.php`, `Source.php`) and build the findings table from the XML report.

**Run command** (record the exact invocation in the workflow doc):

```bash
docker compose exec app php -d extension=pcov -d pcov.enabled=1 \
    vendor/bin/infection \
    --threads=4 \
    --min-msi=0 \
    --min-covered-msi=0 \
    --no-progress
```

Threads ≥ 2 parallelizes mutant runs. `--min-msi=0` and `--min-covered-msi=0` disable the failure-on-low-score gates — this run is exploratory, not a CI gate.

**Build the findings table from `infection.xml`.** The XML schema names every mutation (location, mutator, status: `killed` / `escaped` / `timed_out` / `not_covered`), the test(s) that killed it (when killed), and the resulting mutation score (MSI — Mutation Score Indicator).

Group the findings into four buckets, keyed by test file:

1. **Dead** — a test in the slice catches **zero** mutants across the entire run. Deletion candidate. Each row in this bucket needs a one-line rationale before the user reviews ("this test asserts X but X is also asserted by Y/Z; Infection confirms it adds no mutation kills beyond Y/Z").

2. **Redundant** — every mutant this test catches is also caught by ≥2 other tests in the slice. Consolidation candidate. Per the redundancy threshold above; the user can adjust during review.

3. **Load-bearing** — this test catches at least one mutant that **no other test catches**. Keep, no question. Worth surfacing in the report so the user can see what these tests are guarding.

4. **Surviving mutants** — code paths nothing in the slice catches. Note these as a forward queue (they go into the session log under "Deferred decisions" → forwarded to the Test Suite Audit stub in the roadmap). **Do not write tests for these in 241.**

**Surface the table to the user before making any changes.** This is the hard pause point. The format that reads cleanly:

```
## Findings — Widget Primitive slice

Mutation score: NN.N% (XX killed / YY total mutants, ZZ surviving)
Slice: app/WidgetPrimitive/{ContractResolver,DataContract,Source}.php
Run wall-clock: M minutes

### Dead (N tests)
- tests/Feature/AaaContractRetrofitTest.php :: test name — rationale
- ...

### Redundant (M tests)
- tests/Feature/BbbContractRetrofitTest.php :: test name — caught only by [list]
- ...

### Load-bearing (P tests)
- tests/Feature/CccContractRetrofitTest.php :: test name — uniquely catches [mutator @ line]
- ...

### Surviving mutants (Q paths)
- ContractResolver.php:42 — TrueValue mutator survived; no test exercises this branch
- ...
```

**STOP after surfacing the table.** Do not begin Phase 3 deletions until the user has explicitly approved which rows to act on. The user may approve all, some, or none.

## Phase 3 — Apply approved deletions and consolidations

For each row the user approves:

- **Delete** — remove the test. Re-run the fast Pest suite (`docker compose exec app php -d memory_limit=2G vendor/bin/pest --exclude-group=slow`) and confirm it stays green. The case count drops by exactly one per deletion.
- **Consolidate** — delete the redundant test, optionally adding an assertion to a surviving sibling if the redundant test was asserting something useful that the sibling does not yet cover. Re-run the fast Pest suite; case count drops by one per consolidation, assertion count may stay the same or rise by a small amount.
- **Override / keep** — for tests the user wants kept despite mutation evidence, add a one-line `// guards: <regression description>` comment above the test body (Pest's `it(...)` block). Search-ready via `grep -rn "// guards:" tests/`. Do not delete.

After every change, run the fast suite. If it goes red, stop and investigate before continuing.

**Phase 3 completion criteria:** all approved changes applied, fast Pest suite green, final case count recorded for the session log. Commit as the second of the two-commit shape. If the user approved zero rows, Phase 3 is a no-op and only the Phase 1 commit ships.

## Phase 4 — Write the workflow doc

Create `docs/testing/mutation-audits.md`. The doc covers:

- **Why mutation testing.** Three sentences max — Pest passes prove "the assertions I wrote pass against the code I wrote"; mutation testing probes whether the assertions actually constrain the code; together they answer "are the tests doing useful work."
- **The slice-first discipline.** Why we don't run Infection across the whole suite (slow + unreadable). How to pick a slice (cohesive module, deterministic logic, recently consolidated, not importer-heavy).
- **The toolchain commands.** Exact invocations:
  - `composer require --dev infection/infection` (already done — but here for the next operator who hits a fresh checkout where the lockfile got wiped).
  - The Phase 2 run command verbatim (including the PCOV opt-in flags).
  - How to read `infection.xml` and `infection.html`.
- **The decision rule.** Dead / redundant (N≥2) / load-bearing / surviving — the four buckets and what each one means for action.
- **The override-comment convention.** `// guards: <regression description>` — when to use it, how to find existing ones (`grep`).
- **Suggested cadence.** One slice per quarter, or after any session that adds substantial test code (>50 cases or a new track-level surface). Mention as a `/schedule` candidate offer at session close, not a hard rule.
- **Slice ideas for future passes.** A short queue: AppearanceStyleComposer, Importer FieldMapper, Page builder save flow, Filament resource action visibility, the Fleet Manager controller (now substantial enough to mutation-test post v1.1.0).
- **What the doc explicitly is not.** A CI gate. A whole-suite tool. A replacement for human judgment on test shape.

Keep it under 200 lines. The doc is a workflow reference, not a treatise.
