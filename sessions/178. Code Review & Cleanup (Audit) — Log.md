### Workstream 2 — Widget sovereignty invariants

All 30 widgets under `app/Widgets/` pass the sovereignty invariants:

- Each has `{Name}Definition.php` extending `WidgetDefinition` and a `template.blade.php`.
- All are registered in `app/Providers/WidgetServiceProvider.php:54-83`.
- `requiredConfig()` keys verified to exist in `schema()` for every widget that declares them (BarChart, BoardMembers, Carousel, EventDescription, EventRegistration, LogoGarden, MapEmbed, Nav, VideoEmbed, WebForm).
- `defaults()` coverage enforced at runtime by the base class `validate()` method (called from `WidgetRegistry::sync()`).
- Presets (BarChart, Hero, TextBlock) conform to the session-175 rule that appearance-layer keys live under `appearance_config` and content-layer keys under `config`.
- All widgets with non-empty `collections()` ship a `demoSeeder()`.

### Workstream 3 — Preset & thumbnail pipeline consistency

- `WidgetType::forPicker()` and `PageBuilderApiController::widgetTypes()` produce an identical preset/thumbnail shape — both delegate to `WidgetType::resolvePresetThumbnails()` and `WidgetType::resolveStaticThumbnail()`. The API response is a strict superset (adds `assets`, `full_width`, `default_open`).
- Both resolvers derive the widget folder from the registry (`Str::beforeLast(class_basename($def), 'Definition')`), not from the URL handle. No untrusted-input → filesystem risk.
- `WidgetThumbnailController::show()` regex `^(static|preset-[a-z0-9-]+)\.png$` matches the filename convention produced by `scripts/generate-thumbnails.js` (`static.png`, `preset-{handle}.png`).
- Cache-buster `?v={filemtime}` applied consistently by both resolvers.
- Preset-apply semantics match across the Vue store (`applyPreset` in [resources/js/page-builder-vue/stores/editor.ts:711-725](resources/js/page-builder-vue/stores/editor.ts#L711-L725)) and the dev preset route (`WidgetDemoController::showPreset`): overlay `config`, replace `appearance_config`.

### Workstream 4 — Dead code and orphaned files

### Workstream 5 — Permission & security audit

- Every public method on `PageBuilderApiController` (31 methods) gates on `view_page` or `update_page` as appropriate.
- `PresetController::store` / `update` / `destroy` all gate on `update_page`.
- `WidgetThumbnailController` regex rejects traversal; folder resolved from the registry not the URL.
- `DevRoutesMiddleware` aborts 404 in production and wraps both `/dev/widgets/{handle}` and `/dev/widgets/{handle}/presets/{presetHandle}`.
- All six portal-authenticated data-read paths are correctly scoped to the authenticated `contact_id` (or equivalent). `WidgetDataResolver` reads only public/published collections — no portal-specific filtering needed.
- **Flag:** `PageBuilderApiController::update()` validates `config` / `appearance_config` / `query_config` only as `array`, not against the widget's schema keys. Same issue as session 141's `PageBuilderInspector::updated()` flag, and tolerated on the same basis (admin-only permission; extra keys ignored by templates). Deferred to session 179.
