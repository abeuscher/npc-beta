# 171. Defaults Binding & Sovereign Rendering

Stage 2 of the Sovereign Widget arc. Make `WidgetDefinition::defaults()` the runtime source of truth for default values, and introduce a `WidgetConfigResolver` that composes defaults → theme overrides (stub) → instance config into a fully-resolved config. Templates stop defending against missing keys with `$config['x'] ?? ''`. Instance configs become sparse — only user overrides are stored. The nav widget is the proof-of-concept; other widgets continue working unchanged via a coexistence path.

---

## Design decisions (resolved before starting)

- **Defaults source of truth.** For widgets that have a `WidgetDefinition` in the registry, `defaults()` wins. For widgets still declared only in `WidgetTypeSeeder`, fall back to the existing `WidgetType::getDefaultConfig()` (which reads from `config_schema`). No widget is forced to migrate this session.
- **Resolver shape.** `App\Services\WidgetConfigResolver` is a stateless service with one public method: `resolve(PageWidget $pw): array`. It composes in this order — `defaults()` → theme overrides (stub: empty array for now, feature-flagged hook point) → instance `$pw->config`. Later values overwrite earlier ones, per-key.
- **Theme overrides are a stub.** A private method `themeOverrides(PageWidget $pw): array` returns `[]` in Stage 2. Document the seam so Stage N can wire in template/theme-level defaults without touching widget templates again.
- **Sparse instance configs.** When a `PageWidget` is saved via the builder/inspector, we strip keys whose values are equal to the resolved default (from `defaults()`). Only overrides persist. Don't rewrite existing rows — new sparsity applies only on save. Existing fat rows continue working because the resolver still composes correctly from them.
- **Nav template migration.** Port `resources/views/widgets/nav.blade.php` to assume `$config` is fully resolved. Remove `?? ''` / `?? 'default'` fallbacks. Leave other widget templates alone.
- **Renderer integration.** `WidgetRenderer::render()` calls the resolver before passing `$config` to Blade. For widgets without a definition, the resolver still runs but composes from `$widgetType->getDefaultConfig()` (coexistence branch). This means every widget benefits from theme-override stub wiring without per-widget changes.
- **Inspector — resolved vs override state.** The inspector displays which fields are overridden vs inheriting the default. Minimal UI: a small indicator per field (dot, "inherited" label, or similar — simplest acceptable affordance). A "reset to default" action per field is in scope; a bulk reset is out of scope.
- **No DB migration.** `widget_types.config_schema` still carries `default` keys (consumed by the inspector for field-level rendering). `page_widgets.config` remains JSONB but now stores sparse data going forward.
- **Demo data / preview unchanged.** `DemoDataService` keeps working. The resolver applies to real renders and admin previews equally.

## Phase 2 — WidgetConfigResolver

Create `app/Services/WidgetConfigResolver.php`:

- `resolve(PageWidget $pw): array` — returns the merged config array.
- Internally:
  1. Look up the widget's `WidgetDefinition` via `WidgetRegistry::find($handle)`.
  2. If found, start from `$def->defaults()`. Otherwise, start from `$pw->widgetType->getDefaultConfig()`.
  3. Apply `themeOverrides($pw)` — stub returning `[]` for now. Leave a clear extension point (typed docblock).
  4. Apply `$pw->config ?? []` on top.
  5. Return the merged array.

- `hasOverride(PageWidget $pw, string $key): bool` — returns true when the instance config contains `$key` and its value differs from the resolved default (defaults + theme overrides). Used by the inspector.

- `defaultFor(PageWidget $pw, string $key): mixed` — returns the value the resolver would use if the instance did not override `$key`. Used by "reset to default".

Register as a singleton in `WidgetServiceProvider::register()` if that simplifies injection, otherwise leave as a plain new-able service. Keep it stateless either way.

- Pass the resolved `$config` to Blade.
- `configMedia` and richtext inline-image processing continue to operate on the resolved config.

Confirm no other consumer of `$pw->config` needs the resolver (inspector is the likely other site — see Phase 5).

---

## Phase 4 — Sparse Save

Update the page-builder save path (`PageBuilder` / `PageBuilderBlock` / wherever instance config is persisted). On save:

1. Compute the resolved default for each key in the incoming config.
2. Strip keys whose submitted value equals the resolved default.
3. Persist only the remaining keys to `page_widgets.config`.

Edge cases:

- Image / file fields that reference media are not stripped when equal to `null`-default — keep them if the user explicitly set them. Equality is strict (`===`) for non-null defaults; null-default fields with null values get stripped.
- Pre-existing fat rows are not migrated in this session. They will be naturally slimmed the next time a user saves.

## Phase 5 — Nav Template Simplification

Port `resources/views/widgets/nav.blade.php` to trust `$config`:

- Replace `$config['x'] ?? 'fallback'` with `$config['x']`.
- Remove null-coalesce chains where the resolver guarantees a value.
- Keep conditional rendering based on semantic values (e.g. `branding_type === 'none'`).

Run the nav regression tests after this change.

## Phase 6 — Inspector UI

In `PageBuilderInspector` / its blade:

- For each field, show a small visual indicator when the field is overridden (`WidgetConfigResolver::hasOverride`). Minimal: a dot, an "inherited" / "custom" label, or equivalent. Pick the simplest affordance that reads at a glance.
- Add a per-field "reset to default" action that clears the key from the instance config (equivalent to `$pw->config = array_diff_key($pw->config, [$key => null])`).
- No bulk reset this session.
