# 245. Random Data Generator as Dashboard Widget

Promote the existing `APP_DEBUG_TOOLS`-gated debug data generator into a first-class super-admin-gated Filament dashboard widget. A1 in `sessions/release-plan.md` — first session in pre-Beta-1 release plan execution order. Foundation for downstream rehearsal sessions that need to seed realistic synthetic data quickly.

---

## Success criterion *(from `release-plan.md` § A1; do not relitigate)*

A super-admin-gated Filament dashboard widget that:

- Generates contacts / donations / events / registrations / memberships in configurable counts.
- Tags every generated row with `source = 'generated'` so teardown queries are clean.
- Respects custom-field types when seeding contact records.
- Has a per-action confirmation step preventing accidental clicks.

The existing `APP_DEBUG_TOOLS`-flagged debug generator is retired in this session.

---

## Design decisions (resolved before starting)

- **Widget shape, not page.** The deliverable is a Filament dashboard widget, not a new admin page. Breadcrumbs and route stubs do not apply. Lives on the admin dashboard via Filament's widget registration mechanism.
- **Permission gate.** Super-admin only. Use the existing super-admin permission predicate (read what's already in the codebase before inventing one). Lower roles do not see the widget at all — not "see it greyed out," but "do not render."
- **`source = 'generated'` tag.** Every record this widget produces — across all five record types — sets `source = 'generated'`. The `source` column exists on all four financial tables per session 233 (`donations`, `memberships`, `event_registrations`, `transactions`). For `contacts` and `events`, confirm at session start whether the column exists; if not, surface it as a finding rather than silently adding a migration in this session (see "Out of scope").
- **Confirmation step.** Each generation action runs through a confirmation modal showing what will be created (counts per record type) before any DB write. Single click + accept; no chained confirmations.
- **Custom-field type respect.** When seeding contacts, the generator inspects the active custom-field definitions and produces type-appropriate values: text → string, number → numeric, date → date, boolean → bool, select → one of the declared options. Rich-text custom fields (per E3 in the plan) are not yet a type at session 245; if E3 has shipped by the time A1 runs, include it.
- **Retirement of the old generator.** The `APP_DEBUG_TOOLS` env var stays in `.env.example` for any future debug-only surfaces, but the existing data-generator surface gated on it is removed in this session. The widget replaces it entirely; do not leave both paths active.

## Phase 1 — Permission gate, widget shell, and form

Build the Filament dashboard widget with its permission gate and the form surface for configuring counts. No generation logic yet — the form just collects the config and the confirmation modal renders.

- New widget class under `app/Filament/Widgets/` (or wherever existing dashboard widgets live — confirm in the codebase). Registers via Filament's standard widget mechanism.
- Permission gate via `canView()` (or whatever Filament's current pattern is) — returns false unless the authenticated user is a super admin.
- Form fields: integer count input per record type (contacts, donations, events, registrations, memberships). Defaults to zero across the board so an empty submit does nothing.
- Submit triggers a confirmation modal showing: "Generate N contacts, M donations, …" and offers Confirm / Cancel.

---

## Phase 2 — Generator wiring per record type

Implement the actual generation logic, one method per record type. Each method:

- Sets `source = 'generated'` on every row it produces.
- Uses Faker / the existing seeder helpers where possible — do not re-implement what already exists in `database/seeders/`.
- Respects custom-field types for contacts.
- Wires foreign keys correctly (donations need a contact, registrations need an event + contact, memberships need a contact, events stand alone).
- Writes the matching `Transaction` row for paid donations and active memberships (per session 233's ledger-asymmetry resolution — paid+active memberships write transactions; this session's generator must follow the same discipline).

Run all generation inside a single DB transaction per submit so a partial failure rolls back cleanly.

---

## Phase 3 — Retire the old debug generator + tests

- Remove the `APP_DEBUG_TOOLS`-gated debug-generator surface (the conditional rendering on the dashboard, the underlying generation code if it's not reused). Leave the `APP_DEBUG_TOOLS` env var itself in place — it may be used elsewhere; check before removing.
- Tests:
  - Permission gate — non-super-admin user does not see the widget.
  - Submit-with-zeros — no rows created, no error.
  - Submit-with-counts — correct number of rows created across each type, all tagged `source = 'generated'`.
  - Custom-field type respect — seed a custom field per type (text / number / date / boolean / select) and assert the generator produces type-appropriate values.
  - Foreign-key integrity — generated registrations point at generated events; generated donations point at generated contacts.
  - Transaction-row matching — paid donations + active memberships produce matching `Transaction` rows with `source = 'generated'`.
  - Transaction-rollback — simulate a mid-run failure, assert nothing persisted.

## Out of scope

- **Adding a `source` column to `contacts` or `events` if they don't have one.** If the column is missing, surface it as a finding and stop — adding it touches schema doc + migration discipline that should land as its own small session, not folded silently into A1. The discovery itself is fine; the migration is not.
- **A "delete all generated records" sweep tool.** That is its own stub if it surfaces a need (per the existing roadmap entry). The `source = 'generated'` tag is the seam future tooling leans on.
- **Generating realistic relationships** (e.g., having generated contacts donate to generated events with realistic clustering). The generator produces independent random rows; rehearsals that need correlated data build their own fixtures on top.
- **Importer-shaped generators** (CSV-import simulations). The Onboarding rehearsal cluster (B2) brings its own messy-CSV fixture; this generator is for direct DB seeding only.
- **Updating the Page-level rehearsal stubs to reflect new fixture availability.** That happens organically when each rehearsal's session runs.
