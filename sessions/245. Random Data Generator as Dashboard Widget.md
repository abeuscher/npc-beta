# 245. Random Data Generator as Dashboard Widget

Promote the existing `APP_DEBUG_TOOLS`-gated debug data generator into a first-class super-admin-gated **contract widget** that lives in the `dashboard_grid` slot. A1 in `sessions/release-plan.md` — first session in pre-Beta-1 release plan execution order. Foundation for downstream rehearsal sessions that need to seed realistic synthetic data quickly.

The widget owns generation **and** teardown of its own scrub data. Generation produces source-tagged rows; teardown wipes them cleanly via a source-scoped delete. A new `Source::SCRUB_DATA` value carries the tag, and a new inheritance trait makes the source infectious downward through FK relationships and locked once set — so any row created in relationship to scrub data is itself scrub data, and a single source-scoped wipe removes the entire scrub subgraph without touching real data.

The widget exercises the **classic form + POST + redirect-with-flash** pattern already in use by `PortalContactEdit` (and the other portal-side contract widgets), proving that the existing widget-contract surface supports admin-interactive widgets without architectural change. The "dashboard widgets are display-only" reading was an accident of which dashboard widgets happened to ship first; the contract has been admin-interactive-ready since `PortalContactEdit` landed.

---

## Success criterion *(revised from `release-plan.md` § A1; re-sign-off on close)*

A super-admin-gated contract widget in the `dashboard_grid` slot that:

- **Generates** contacts / donations / events / registrations / memberships in configurable counts.
- **Tags** every generated row (where a `source` column exists) with a new `Source::SCRUB_DATA` value, distinguishable from `DEMO` (preview/thumbnail data) and from real-source values (`HUMAN`, `IMPORT`, `STRIPE_WEBHOOK`).
- **Inherits scrub downward**: any row created with a FK to a scrub-data parent is itself force-tagged scrub_data; rows already tagged scrub_data cannot have their source mutated.
- **Wipes** all scrub data on demand, in FK-safe order, removing the entire scrub subgraph and leaving real data untouched. Replaces the old generator's "wipe all of kind" indiscriminate teardown.
- **Respects custom-field types** when seeding contact records.
- Has **per-action confirmation** (generate AND wipe) preventing accidental clicks.

The existing `APP_DEBUG_TOOLS`-flagged debug generator and its `wipe()` method are retired in this session.

The release-plan.md § A1 wording (*"a Filament dashboard widget that generates …"*) gets corrected at close-gate Phase 2 to reflect both the contract-widget architecture and the generation-AND-teardown deliverable. This is the agreed Rule-6 carve-out for this session.

## Design decisions — ratified

1. **One-piece architecture: contract widget alone.** The widget renders an Alpine-driven generation form and an Alpine-driven teardown form on the same dashboard surface. Each form's submit triggers an Alpine confirmation modal showing the totals; modal confirm posts to a dedicated admin route. No separate Filament Tools page is created. Pattern reference: `app/Widgets/PortalContactEdit/template.blade.php` (multiple forms on one widget surface, CSRF + flash + error display) — the cleanest existing precedent for an interactive contract widget.

2. **`Source::SCRUB_DATA` as a new source value.** Lives parallel to `Source::DEMO` in `app/WidgetPrimitive/Source.php`. Semantic split:
   - `DEMO` = throwaway preview data the system writes for itself (Playwright thumbnail captures, widget previews). **Stays out of financial models** — preserves session 213's security invariant.
   - `SCRUB_DATA` = synthetic data the operator generates on a real install for rehearsals, scale tests, recovery drills. **Included** on financial models — that's the whole point of the generator.

3. **Scrub inheritance is policy, not convention.** A new trait `EnforcesScrubInheritance` enforces:
   - **Rule 1 — infectious downward:** on `creating`, if any declared parent FK points to a row with `source = 'scrub_data'`, this row's source is force-set to `'scrub_data'`, overriding whatever the caller passed.
   - **Rule 2 — sticky:** on `updating`, if `source` was `'scrub_data'` and is being changed, throw `ScrubSourceLocked` (a clear named exception with the offending model/id).
   - **Rule 3 — multi-parent, safe-rule precedence:** if a row has multiple parent FKs and any parent is scrub, the row is scrub. Errs on the side of teardown completeness over leaked rehearsal-flow rows.

   Trait is applied to: `Contact`, `Donation`, `EventRegistration`, `Membership`, `Transaction`. Each model declares a static array of parent FKs the trait inspects, supporting both regular FKs (`contact_id` → Contact) and polymorphic parents (Transaction's `subject_type/subject_id`).

4. **The 213 invariant is preserved AND a parallel positive invariant is added.** Session 213's `it locks in the Donation/DEMO security invariant` test (asserting `Donation::ACCEPTED_SOURCES` does **not** include `DEMO`) stays unchanged. A new parallel positive test asserts the five models DO accept `SCRUB_DATA`. Both invariants in one test file so the policy story reads in one place.

5. **Permission gate is `User::isSuperAdmin()`, not a `->can(...)` predicate.** `AuthServiceProvider::Gate::before` returns true for super_admin against any ability, which makes `->can()` brittle as a super-admin gate. Direct `isSuperAdmin()` checks at three layers — defense in depth: widget template wrap, route middleware, controller method body.

6. **`events` table has no `source` column — surfaces as a finding, not a fix.** Generated events go in untagged in this session. The wipe service deletes related rows (registrations, transactions, etc.) via the inheritance trait + source-scoped query, but events themselves persist after a wipe. This is a known incompleteness, captured in the close-gate deferred-decisions list. Adding `source` to `events` belongs in its own micro-session per the original prompt's out-of-scope rule.

7. **Confirmation steps use Alpine inline modals, not server round-trips.** Both forms use the same modal pattern — submit-button click opens a modal showing the action summary (totals for generate; per-table scrub row counts for wipe) with Confirm / Cancel. Confirm submits the underlying form. No two-step preview-then-confirm POST flow.

8. **`Page::ACCEPTED_SOURCES` also gains `SCRUB_DATA`.** Page already has a source column and is in the same family as the other source-bearing models. The existing debug generator's blog-post path uses `Source::DEMO`; future blog/page generation can move to `SCRUB_DATA` consistently. The inheritance trait is **not** applied to Page in this session — Page sits as a root-source row (no source-bearing parents) and gets the trait only when a page generator surfaces. Documented in deferred decisions.

9. **Edge models (notes, tags, taggables, activity logs) are out of scope for inheritance.** Inheritance is initially only on the five generator-touched models. The wipe service uses FK queries to clean up notes/tags/taggables attached to deleted scrub primary rows, not source-tag enforcement. Future expansion to cover more content types is acknowledged in the post-Beta-1 service-worker-sweep stub now in `session-outlines.md`.

10. **Stripe test-mode awareness is out of scope here, blocker stub added to outlines.** Per the architecture conversation: a production install configured against a real (non-test) Stripe account could see an operator generate scrub donations that create real Stripe activity. New release-blocking stub added to `session-outlines.md` to detect Stripe test mode at runtime and either refuse-to-render or warning-gate the generator widget. Not 245 work; lands as its own session before Beta-1.

11. **Activity-log + Media-Library orphan sweep is out of scope here, post-Beta-1 stub added to outlines.** Scrub-data wipe will leave Spatie ActivityLog entries and Media Library rows pointing at deleted scrub records. Benign noise; accumulates over time. Future scheduled cleanup is a service-worker concern that pairs with the existing Housekeeping Batch 2 `media-library:clean` work. Documented in `session-outlines.md`.

12. **Retire the old generator.** Delete `app/Filament/Widgets/DashboardDebugGeneratorWidget.php` and `resources/views/filament/widgets/dashboard-debug-generator-widget.blade.php`. Confirm `APP_DEBUG_TOOLS` is/is-not referenced elsewhere; remove from `.env.example` only if no other consumer remains.

## Phase 2 — `EnforcesScrubInheritance` trait

The behavioral half of the scrub policy. Enforces rules 1 (infectious downward), 2 (sticky), and 3 (multi-parent any-scrub-wins).

- New file `app/WidgetPrimitive/EnforcesScrubInheritance.php`. Trait shape:
  - Static `bootEnforcesScrubInheritance()` registers `creating` and `updating` listeners.
  - `creating` listener: if `$model->source !== Source::SCRUB_DATA`, walk `$model::scrubInheritsFrom()` (per-model declaration), look up each declared parent. If any parent has `source === 'scrub_data'`, force `$model->source = Source::SCRUB_DATA`. The any-scrub-wins precedence (rule 3) is implicit: any single match flips the row.
  - `updating` listener: if `$model->getOriginal('source') === 'scrub_data'` AND `$model->source !== 'scrub_data'`, throw `App\WidgetPrimitive\Exceptions\ScrubSourceLocked` with model class + key + attempted-new-source in the message.
  - Per-model declaration via static method `scrubInheritsFrom(): array`. Supports two shapes:
    - Regular FK: `['contact_id' => Contact::class]` — looks up parent via `Contact::find($model->contact_id)`.
    - Polymorphic: `[['type' => 'subject_type', 'id' => 'subject_id']]` — resolves type via the value of `subject_type` (a model class name), then `find($model->subject_id)`.
- New file `app/WidgetPrimitive/Exceptions/ScrubSourceLocked.php` — extends `LogicException` (mirroring `SourceRejectedException`'s shape).
- Apply trait to five models, with declarations:
  - `Contact` — `scrubInheritsFrom(): []` (no source-bearing parent; trait still active for rule-2 source-locking).
  - `Donation` — `['contact_id' => Contact::class]`.
  - `EventRegistration` — `['contact_id' => Contact::class]` (event_id intentionally skipped — events have no source column).
  - `Membership` — `['contact_id' => Contact::class]`.
  - `Transaction` — `[['type' => 'subject_type', 'id' => 'subject_id'], 'contact_id' => Contact::class]` — both parent slots inspected.
- Tests (`tests/Unit/WidgetPrimitive/EnforcesScrubInheritanceTest.php` — new file):
  - **Rule 1, single FK** — create a scrub Contact, then create a Donation with `source = 'human'` referencing it; assert Donation's persisted source is `scrub_data`.
  - **Rule 1, polymorphic parent** — create scrub Donation, create Transaction with `subject_type = Donation::class` + `subject_id = scrub_donation->id` and `source = 'stripe_webhook'`; assert Transaction's persisted source is `scrub_data`.
  - **No-op when no parent is scrub** — real Contact + new Donation with `source = 'import'`; assert Donation's source remains `import`.
  - **No-op when source already scrub** — caller explicitly sets `source = 'scrub_data'`; trait's parent lookup is skipped (verified by counting queries or by passing a parent that doesn't exist — wouldn't be looked up).
  - **Rule 3, multi-parent precedence** — Transaction with scrub subject + real contact; assert source forced to scrub. Reverse: real subject + scrub contact; assert scrub still wins.
  - **Rule 2 — update locked** — create a scrub Donation, attempt to update its source to `import`; assert `ScrubSourceLocked` thrown, persisted row unchanged.
  - **Rule 2 — other updates allowed on scrub rows** — update a scrub Donation's `amount`; assert succeeds, source still scrub.
  - **Polymorphic resolution with non-existent subject** — Transaction with `subject_type = Donation::class` + invalid `subject_id`; assert no force (parent lookup returns null, no inheritance fires), source remains whatever was passed.

## Phase 3 — Generation service

Pure backend. Independently testable; no Filament / Livewire / widget dependencies.

- New file `app/Services/RandomDataGenerator.php` with one public orchestrator: `generate(array $counts): array`. Runs all per-type generation inside `DB::transaction(...)`. Returns a per-type count summary on success. Throws on any per-type failure; transaction rolls back cleanly.
- One protected method per record type: `generateContacts($n)`, `generateDonations($n)`, `generateEvents($n)`, `generateRegistrations($n)`, `generateMemberships($n)`.
- Each method:
  - Sets `source = Source::SCRUB_DATA` on every row it produces (events excluded — no column).
  - Uses Faker / existing factories where possible. No re-implementation of what already exists in `database/factories/`.
  - Wires foreign keys correctly: donations → contact, registrations → event + contact, memberships → contact, transactions → polymorphic subject + contact.
  - For contacts: respects custom-field types when seeding `custom_fields`. Inspects `custom_field_defs` rows where `model_type = 'contact'`; produces type-appropriate values (text → fake string, number → numeric, date → date, boolean → bool, select → one of declared options). If E3 (rich-text custom fields) has shipped by 245 runtime, include `rich_text → fake paragraph`.
  - For paid donations and active memberships: writes the matching `Transaction` row per session 233's ledger discipline. The Transaction's `source` will be force-set to `scrub_data` by the inheritance trait via its scrub-tagged parent — this is the trait's first real use beyond its own tests.
- Re-assert the super-admin gate at the service entry point (one-line check; throws `AuthorizationException` if missing). Defense in depth.
- Where `HasSourcePolicy` applies (Contact, Donation, etc.) and the model accepts the source via Phase 1's ACCEPTED_SOURCES change, prefer routing through `app(DataSink::class)->write(...)`. For models without the trait or where direct `Model::create(['source' => ...])` reads cleaner with surrounding factory state, that's fine — policy enforcement happens at the model-acceptance layer either way.

## Phase 5 — Wipe service

The teardown half of the generator service. Lives on the same `RandomDataGenerator` service class.

- `wipe(): array` — runs all source-scoped deletes inside `DB::transaction(...)`. Returns per-table count summary of rows actually deleted.
- Delete order (FK-safe, executed in this sequence inside the transaction):
  1. `Transaction::where('source', 'scrub_data')->forceDelete()` — first (subjects point at Donations/Memberships).
  2. `EventRegistration::where('source', 'scrub_data')->forceDelete()` — second (FK to events; events themselves untouched).
  3. `Donation::where('source', 'scrub_data')->forceDelete()` — third.
  4. `Membership::where('source', 'scrub_data')->forceDelete()` — fourth.
  5. `Contact::where('source', 'scrub_data')->withTrashed()->forceDelete()` — last (parents of all others; `withTrashed` to handle soft-deleted scrub contacts; `forceDelete` to bypass the soft-delete column).
- `scrubCounts(): array` — returns per-table counts of currently-tagged scrub rows. Used by the widget's wipe modal.
- Re-assert super-admin at the service entry point.
- Tests:
  - **Generate-then-wipe round-trip** — generate N of each, wipe, assert all five tables back to pre-generation row counts; assert real-data rows seeded into the test setup are still present.
  - **Wipe with no scrub data** — call wipe on a clean install; assert zero deletions, no error.
  - **Wipe inside transaction with simulated mid-failure** — mock one of the per-table delete calls to throw; assert all other deletions rolled back, no rows lost.
  - **Inheritance + wipe interaction** — create a scrub Contact, then create a Donation with `source = 'human'` (forced to scrub by the trait), then create a Transaction with `source = 'stripe_webhook'` against that Donation (forced to scrub). Wipe. Assert all three deleted.
  - **`scrubCounts()` accuracy** — generate known counts, assert returned summary matches.
- Out of scope (consequence of decision 9): notes / tags / taggables / activity logs are not deleted by the wipe service. The resulting orphans are documented in design decision 11's outline stub.
