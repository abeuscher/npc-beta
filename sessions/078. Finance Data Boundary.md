# 078. Finance Data Boundary

Enforce a clean separation of concerns between Finance and the rest of the admin: financial amounts and Stripe links are removed from CRM and CMS views, and the Transactions table becomes the single place where financial detail and Stripe links live. Navigation affordances (buttons/links) replace inline financial data everywhere outside Finance.

---

## Design decisions (resolved before starting)

**No backwards compatibility needed:**
This system has no live users and no PII. All data can be wiped in minutes. Do not plan for migration of existing data, do not add shims or fallbacks, and do not preserve old views "just in case." Remove code and views that no longer fit — dead code is worse than no code.

**The boundary rule:**
Financial amounts (dollar values) and Stripe links must only appear inside the Finance navigation group. Outside Finance, a view may show counts ("5 donations", "3 purchases") but never amounts. Navigation buttons that take the admin *into* a Finance view are fine; rendering financial data inline is not.

**Stripe links belong in Transactions only:**
Every transaction row that has a Stripe ID gets a direct link to the Stripe dashboard object. This link appears in the Transactions table and nowhere else in the admin. Remove any Stripe links found outside that table.

**Navigation affordance pattern:**
From a contact record, product page, or any non-Finance view: a button labelled "View in Finance →" (or similar) navigates to the relevant Finance view pre-filtered. The Finance view is responsible for rendering amounts. This is a link/redirect — not an embedded panel.

**Transactions table as the financial hub:**
Confirm that the Transactions table is reachable from Finance navigation, filterable by subject (contact, product, donation), and shows Stripe links per row. If any of this is missing, add it as part of this session.

---

## Phase 1 — Audit

Before writing any code, do a full audit:

1. Grep for `amount`, `total`, `price`, `stripe_` rendered in any Blade view outside `resources/views/filament/` Finance pages (Donors, Transactions). List every hit.
2. Identify all places in the admin where financial data is rendered outside the Finance section — specifically: contact record panels, product resource views, any relation managers that show purchase or donation amounts.
3. Identify all Stripe links or `stripe_` URL constructions outside the Transactions view.
4. Report findings before writing any changes.

---

## Phase 2 — Remove financial data from non-Finance views

For each finding from Phase 1:

1. **Contact record** — remove any inline donation totals, purchase amounts, or financial panels. Replace with a count badge and a "View donations in Finance →" button that links to the Donors page filtered to that contact, and/or a "View transactions →" button to Transactions filtered to that contact.
2. **Product resource** — remove the inline purchases panel (amounts, statuses). Replace with a "View purchases →" button linking to Transactions filtered to that product.
3. Any other hits from the audit — apply the same pattern: remove amount rendering, add navigation affordance.

Delete relation managers, view components, or partial views that exist solely to render financial data outside Finance. Do not leave them in place commented out.

---

## Phase 3 — Transactions table as financial hub

1. Confirm every transaction row with a `stripe_session_id`, `stripe_subscription_id`, or similar Stripe identifier has a visible link to the corresponding Stripe dashboard object.
2. Confirm the Transactions table is filterable by contact and by subject type (donation, purchase, membership).
3. Confirm Transactions is in the Finance navigation group.
4. If any of the above is missing, add it. If Stripe links exist elsewhere in the admin, remove them and ensure Transactions is the canonical place.

---

## Security checklist

Before closing:

- [ ] No `amount` or financial total is rendered in any Blade view outside the Finance section.
- [ ] No Stripe dashboard links appear outside the Transactions table.
- [ ] Navigation affordances (buttons linking into Finance) do not leak financial data in their labels or tooltips — counts only, no amounts.
- [ ] Removed code is fully deleted, not commented out.

---

## Out of scope

- Changing the Transactions data model or schema.
- Adding new Finance pages beyond what is needed to complete the hub role of Transactions.
- Portal-facing financial views (separate concern, separate session).
