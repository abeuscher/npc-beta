## Stub reference

Not a `release-plan.md` entry. An inbox-driven housekeeping batch from `sessions/housekeeping-inbox.md`. The items it consumes (verify each against current code at session start ‚Äî staleness guard):

- **`[0.324.01]` (‚≠ê) Chrome drops per-layout padding/appearance ‚Äî stale duplicate renderer.** `ChromeRenderer::renderLayoutBlock()` reads flat legacy keys (`padding_*`, `background_color`) from `layout_config`; layout appearance is actually nested in `appearance_config.layout.{padding,margin,border,background}` (what the builder writes, what `PageBlockRenderer::renderLayoutBlock` ‚Üí `AppearanceStyleComposer::composeForLayout` reads). So column-layout padding/background set on a header/footer shows in the builder and on page bodies but vanishes on public chrome.
- **`[0.326.01]` Footer columns don't stack on mobile.** Confirmed at s339: `ChromeRenderer` never emits the `data-collapse-mobile` attribute, so the `@container ‚Ä¶ .layout-grid[data-collapse-mobile="true"]` collapse rule (`_layout.scss`) never fires for chrome. (`PageBlockRenderer` emits it, default true ‚Äî that's the parity target.)
- **`[0.317.01]` Nav landmark semantics.** `resources/views/components/site-header.blade.php` has `<nav role="group">`; the `role="group"` overrides the navigation landmark. Drop it.
- **`[0.316.01]` Unquoted `url()` breaks background-image for filenames with parens/spaces.** `AppearanceStyleComposer` emits `background-image:url(...)` unquoted (line ~331), so `photo (1).jpg` produces invalid CSS and the background silently fails to paint. Quote/escape it (check the chrome path's `buildInlineStyle` too).
- **`[0.317.01]` `.sr-only` not defined in the public bundle.** Widgets use `<span class="sr-only">` but no `.sr-only` rule ships in `resources/scss/` partials, so visually-hidden text may render visible. Add the standard utility.
- **`[0.324.01]` Hero left-aligned text auto-padded inside the container.** The hero widget wraps content in a second `.site-container`, so left-aligned hero text insets instead of sitting flush. 339 worked around this on the home page by swapping the hero to a text block; this session finishes the *widget* fix so any hero left-flushes. **Closes Page Composition Fidelity gap G2.**

## The work

Six items, one theme (public render / chrome parity). Group the chrome ones into one iteration if it reads cleanly; the small fixes can ride the same branch.

### 1‚Äì2. Chrome renderer parity (the core)

In `app/Services/Media/ChromeRenderer.php::renderLayoutBlock()`:

- **Emit `data-collapse-mobile`** on the `.layout-grid` div, honoring the layout's `collapse_mobile` (default true) ‚Äî mirror `PageBlockRenderer` line ~103-104 exactly. This makes header/footer columns collapse on mobile via the existing `_layout.scss` rule. *(No new toggle ‚Äî chrome starts honoring the one page bodies already use.)*
- **Replace the flat-key appearance reads** (`padding_*`, `background_color`, and the bespoke `layout_config` full-width reads) with `app(AppearanceStyleComposer::class)->composeForLayout($layout)` + `resolveFullWidthForLayout($layout)`, bringing chrome to full parity with the page-body path. Per-*widget* padding in chrome already reads nested keys correctly (`buildInlineStyle`) ‚Äî only per-*layout* appearance is affected.
- **Regression test** (required): assert that a chrome layout with a nested `appearance_config.layout.padding` + a `collapse_mobile` setting renders with both the `data-collapse-mobile` attribute and the nested padding in the HTML.

### 3. Nav landmark

Remove `role="group"` from `<nav>` in `resources/views/components/site-header.blade.php` (the `<nav>` element already provides the navigation landmark).

### 4. Unquoted `url()` background-image

Quote/escape the URL in `AppearanceStyleComposer` (the `background-image:url(...)` emission) and check the chrome `buildInlineStyle` path for the same. A filename with spaces/parens must produce valid CSS.

### 5. `.sr-only` utility

Add the standard visually-hidden utility to the public SCSS bundle (`resources/scss/` ‚Äî the partial that fits the existing structure).

### 6. Hero left-flush finish

Make the hero widget's left-aligned content sit flush to the container (the `hero--pos-*-left` variants) ‚Äî the ~few-line CSS finish in `resources/scss/_custom.scss` of what 339 worked around. Verify a left-positioned hero now aligns with the container content edge. Record G2 closed in the Page Composition Fidelity gap backlog at close.
