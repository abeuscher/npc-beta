## What was built

### Phase 1 — `BorderInput` primitive

- New Vue component `resources/js/page-builder-vue/components/primitives/BorderInput.vue`. Single row: an SVG 4-edge toggle box, a compact color picker swatch handle, a width number input, a radius number input. Mirrors the layout/idioms of `NinePointAlignment` and `SpacingInput`. Emits the full border value object atomically.
- SVG edges use `vector-effect: non-scaling-stroke` so off/on stroke widths render at exact px regardless of the viewBox scale.
- Hit lines are separate from visible lines — `stroke-width: 14` (viewBox units, ≈9px rendered) with `stroke-linecap: butt` so neighboring edges don't fight over corner pixels.

### Phase 2 — Inspector mount + concrete defaults

- `SectionLayoutPanel.vue` mounts `BorderInput` below the Padding/Margin sections. Updates emit `update('border', value)` — the parent's `onWidgetLayoutUpdate` prefixes with `layout.`, so the border state lives at `appearance_config.layout.border` and is set as a single atomic write.
- `WidgetDefinition::defaultAppearanceConfig()` now ships the concrete `border` block under `layout` (all four sides `false`, `width: 0`, `color: '#000000'`, `radius: 0`).
- Three widgets that fully override `defaultAppearanceConfig()` got the same block added so every widget's default stays concrete: `TextBlockDefinition`, `PricingChartDefinition`, `CarouselDefinition`.

### Phase 3 — Render-surface parity via one shared emitter

- `AppearanceStyleComposer::composeBorderProps(array $border): array` — single static helper that emits `border-{side}:Wpx solid #color` per enabled side, `border-radius:Rpx` independently when radius > 0, and `box-sizing:border-box` whenever any side is on (the public site has no global `box-sizing: border-box` reset, so this is load-bearing — without it an enabled border would grow the outer box).
- `AppearanceStyleComposer::compose()` consumes the helper after the padding/margin loops. `PageBlockRenderer` (public site) and `WidgetPreviewRenderer` (builder preview) both already delegate to `compose()`, so they pick the border up for free.
- `ChromeRenderer::buildInlineStyle()` (system / chrome pages) calls the same helper — one source of truth for the border across all three surfaces.

### Phase 4 — Tests (Pest)

- `AppearanceStyleComposerTest` — eight new border cases: default no-op, single side on, all sides on, radius alone, malformed hex drops the color token, width/radius int coercion + negative clamp, the helper's pure-array shape, and an empty-config zero-return guard.
- `ChromeWidgetTest` — one new parity assertion: a system page with a bordered widget carries exactly what `composeBorderProps` emits for the same config (asserts on each declaration, plus negative checks for the disabled sides).
- `TextWidgetBorderRenderTest` (new file) — three cases: a standalone bordered Text widget block carries the inline border style; a Text widget inside a column layout receives the border via `renderLayoutBlock`; a default-off Text widget emits no border style.
- `WidgetBorderDefaultTest` (new file) — two concrete-values guards: the base `WidgetDefinition::defaultAppearanceConfig()` carries the all-off block, and every registered widget's `defaultAppearanceConfig()` carries a complete, valid border block.

### Phase 5 — Standing Playwright spec

- `tests/e2e/text-widget-border.spec.ts` — creates a Text widget on the seeded home page with a 4-sided 2px `#336699` border + 4px radius, visits `/`, asserts computed `border-{side}-width: 2px`, `border-top-style: solid`, `border-top-color: rgb(51, 102, 153)`, `border-top-left-radius: 4px`, `box-sizing: border-box`. The widget is cleaned up in `afterAll`. Inline appearance styles are bundle-independent so this guard is CI-viable in the isolated e2e stack.

### Phase 6 — Visual + UX polish (iterated with the owner mid-session)

A series of fixes the design pass surfaced:

- **Color picker layout fix.** The compact `ColorPicker` rendered its popover inside the row column when opened, stretching the flex row and pushing Width / Radius out of view. Replaced with a small swatch button as the handle and a `panel-only` `ColorPicker` rendered **below** the row — the established `BackgroundPanel` pattern.
- **Vertical alignment.** Set both the swatch button and the number inputs to an explicit `height: 2rem` so the three labeled columns share an identical column height and bottom-align cleanly under `align-items: flex-end`.
- **The "Sides" label.** Without a label above the 4-edge box it read as decoration and was being mistaken for an unrelated graphic — directly causing the "width isn't being assigned" report (no side was ever toggled on, so the emitter correctly emitted nothing for width). Wrapped the SVG box in a labeled column ("Sides") matching the Color / Width / Radius columns; the affordance is now self-evident.
- **Contrast bump.** Off edges at `#ccc` / 2px stroke, on edges at `#000` / 3px stroke (1px thicker as well as much darker). The box was sized to `calc(2rem - 2px)` so the thicker on-stroke doesn't make the box visually overflow vs the swatch.
- **Sharp corners** on the sides-box rect (`rx`/`ry` removed) so the off-edges form a clean square outline rather than a rounded one.
- **Hit-area widening.** Hit-line `stroke-width` bumped from 8 to 14 (viewBox units) → ~9px rendered grab zone per edge; `stroke-linecap` swapped from round to butt so neighboring edges don't overlap into the corners (a click near a corner has no ambiguity about which side it toggles).
- **Color-picker stillness pattern.** Two pre-existing UX problems with the shared `ColorPicker` panel-only mode: (1) opening near the bottom of the inspector pane rendered the popover off-screen — the click looked like a no-op; (2) closing collapsed its height and the inspector jumped. Landed a small parent-side pattern in both `BorderInput` and `BackgroundPanel`: on open, `nextTick → scrollIntoView({ block: 'nearest' })`; on close, hold the measured popover height as an invisible spacer so subsequent open/close cycles are dead still. `BorderInput` keeps the max-ever measured height (single picker, content grows); `BackgroundPanel` keeps the last opened panel's height (color / gradient / image have different intrinsic sizes, so max would cause a downward jump when closing a shorter panel after a taller one). `BackgroundPanel`'s wrapper uses `display: flow-root` to prevent margin collapse so the measurement includes the popover's `margin-top`.

## Security considerations

None beyond standard convention. The border emitter only consumes the operator's own `appearance_config.layout.border` value; the hex color is validated against the same `HEX_PATTERN` the existing background/text color path uses, and a malformed hex drops the color token (falls through to `currentColor`) rather than emitting an unsafe declaration. Width / radius are int-coerced and clamped to non-negative. No new admin surface, no new permission, no new model.

## Deferred decisions

- **Per-side width / color** — out of v1 by explicit plan-entry scope. The value model has one shared `width` and one shared `color` for whichever sides are on; per-side variation lands later if asked for.
- **Per-corner radius** — out of v1; the value model is one shared radius across all four corners.
- **Non-solid styles (dashed / dotted)** — out of v1; the emitter hard-codes `solid`.
- **In-widget `border` field-type registration** — designed-for as a future bind target (the primitive is reusable), not built. The `alignment`-field precedent is the model when it lands.
- **Borders beyond the outer box** (interior rules, column-to-column rules, borders on elements inside a widget) — per-widget / per-context follow-ups, not a v1 universal.
- **Color picker discoverability fix** — the scrollIntoView + reserved-height pattern landed at the parent (`BorderInput`, `BackgroundPanel`). A more universal fix (e.g., teleporting the panel-only `ColorPicker` to a floating layer so it escapes scroll-container clipping entirely) was discussed and deferred; not a regression here.
