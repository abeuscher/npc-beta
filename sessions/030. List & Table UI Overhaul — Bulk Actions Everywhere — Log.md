# 030. List & Table UI Overhaul — Bulk Actions Everywhere — Log

## What was built

### Bulk actions on all resource list tables
Added `EditAction`, `DeleteAction`, and `DeleteBulkAction` (wrapped in a `BulkActionGroup`) to every Filament resource list table that was missing them:

- `OrganizationResource`
- `MembershipResource`
- `CampaignResource`
- `DonationResource`
- `TransactionResource`
- `FundResource`
- `NoteResource`
- `WidgetTypeResource`
- `HouseholdResource`
- `TagResource`
- `CmsTagResource`
- `ContactResource`
- `EventResource`
- `NavigationItemResource`
- `PageResource`

Added `DeleteBulkAction` only to `RoleResource`, which already had row actions but had an empty `bulkActions([])`.

The reference implementation throughout is `CustomFieldDefResource`. `ContentCollectionResource` was intentionally left untouched (read-only CMS editor view).

### PostResource restored
Discovered during session that `PostResource` had been deleted in session 026 when blog posts were migrated to the `pages` table, leaving posts with no admin management UI. Created a new `PostResource` (and its three page classes) that:

- Points at the `Page` model scoped to `type = 'post'`
- Appears in the CMS nav group as "Blog Posts" at sort order 2
- Has the same form as `PageResource` (title, slug, page builder, publication, SEO, custom fields) with `type` hidden and defaulting to `post`
- `CreatePost` auto-generates the slug with the blog prefix (`config('site.blog_prefix')`) prepended, matching the pattern enforced by `PageObserver` for type changes
- Includes the full Edit / Delete / BulkDelete action pattern

### Files created
- `app/Filament/Resources/PostResource.php`
- `app/Filament/Resources/PostResource/Pages/ListPosts.php`
- `app/Filament/Resources/PostResource/Pages/CreatePost.php`
- `app/Filament/Resources/PostResource/Pages/EditPost.php`

## Deferred decisions

**Bulk actions always visible:** The user asked whether the "Bulk Actions" dropdown could be permanently visible rather than sliding in on row selection. In Filament v3 this is not configurable without publishing and modifying vendor views or using fragile CSS overrides. Deferred — accepted as standard Filament behaviour for now.

## Notes
The PostResource bug was caught two sessions late (should have been caught in session 026). The fix is clean and complete; no data issues as posts were already correctly stored in the pages table throughout.
