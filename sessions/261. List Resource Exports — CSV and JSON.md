# 261. List Resource Exports — CSV and JSON

Add operator-facing **Export CSV** and **Export JSON** admin actions to every list resource that holds migration-relevant data, mirroring the existing Contacts CSV exporter pattern at [`ListContacts.php:177-225`](../app/Filament/Resources/ContactResource/Pages/ListContacts.php#L177). Adds JSON to the Contacts list resource for parity. Both formats stream against the current filtered+sorted table query; both include custom-field columns where the model supports them. CSV flattens CF to one column per `CustomFieldDef`. JSON nests CF as a sub-object on each row.

This is **B2b** from `release-plan.md` Track B, expanded at session 260 close to include JSON in scope and to lift XLSX to a follow-on (Out-of-Gate register).

---

## Design decisions (resolved before starting)

These are the user-agreed shape from session 260 close. Confirm at session start during the read-through.

- **Resource list — 10 list pages total:**
  - `ListContacts` (gets new JSON action; keeps existing CSV action)
  - `ListOrganizations`
  - `ListDonations`
  - `ListEvents`
  - `ViewRegistrations` (event-scoped — exports the currently-viewed event's registrations only; not a top-level resource)
  - `ListMemberships`
  - `ListTransactions`
  - `ListFunds`
  - `ListCampaigns`
  - `ListNotes`

- **Standard column shape per resource:** mirror each corresponding `*ImportFieldRegistry::flatFields()` output (Organization / Donation / Event / Membership / Invoice / Note registries). Sentinel keys (`__org_contact__`, `__custom_organization__`, etc.) are import-only constructs and **do NOT appear as exporter columns.** For resources without a dedicated `*ImportFieldRegistry` (Fund, Campaign, EventRegistration, Transaction): inventory the model's fillable + relevant FK columns and declare the export shape inline at session-start sign-off.

- **Custom-field flattening:**
  - **CSV:** one column per `CustomFieldDef` row (matching the existing Contacts exporter shape exactly). Header is the CF's `label`; value is `$model->custom_fields[$def->handle] ?? ''`.
  - **JSON:** nested as a sub-object on each row: `{"first_name": "...", ..., "custom_fields": {"donor_tier": "gold", "anniversary": "2024-06-12"}}`. Empty CFs are omitted from the nested object (not present rather than `null`-or-empty-string).
  - Resources without a `custom_fields` column (Fund, Campaign, Note): no CF flattening at all; the row shape is just the standard columns.

- **JSON output container:** array of objects (`[{...}, {...}, ...]`), not JSON-Lines. Streamed via `streamDownload` with `Content-Type: application/json`. Sub-5K-row exports fit in memory comfortably.

- **Date serialization in JSON:** ISO 8601 strings (e.g. `2024-06-12T15:30:00Z` for datetime, `2024-06-12` for date-only fields). Mirrors Laravel's default Carbon serialization.

- **Action placement:** both actions live in each page's **ellipsis ActionGroup**. If a page doesn't have one today, add one. Target: every ListXxx page has an ellipsis menu containing at least the two new export actions, plus any existing actions that already live in an ActionGroup on that page.

- **Permission gate:** match each resource's existing `view_any_<resource>` permission. Same pattern as the Contacts exporter.

- **Filename / content-type:**
  - CSV: `<resource>-YYYY-MM-DD.csv`, `Content-Type: text/csv`
  - JSON: `<resource>-YYYY-MM-DD.json`, `Content-Type: application/json`
  - For `ViewRegistrations`: `event-<id>-registrations-YYYY-MM-DD.{csv,json}` (event-scoped naming).

- **Architectural shape:** shared `App\Services\ListExportService` (or equivalent — trait or service, decide at session start). Inputs: filtered+sorted query, column spec, format flag (`'csv'` / `'json'`), filename, optional CF model-key. Output: `Symfony\Component\HttpFoundation\StreamedResponse`. Each ListXxx page calls into it. The CF flattening + format-switching + streaming logic lives in one place. The existing Contacts CSV exporter refactors to call into the same service.

- **Custom-field model-key strings:** the 6 CF-having models need confirmed `CustomFieldDef::forModel('<key>')` keys. Confirmed today: `contact`, `event`. Need verification at session start: `organization`, `donation`, `membership`, `transaction`, `event_registration`. If a key isn't established yet (the CF def rows simply don't exist in current usage), define it consistently — singular-noun snake_case is the established convention. Land any necessary `CustomFieldDef` registrations or factory-side fixtures.

- **Out of scope:**
  - **XLSX format** — stubbed in `release-plan.md` Out-of-Gate register at 260 close. Will lift in a follow-on session (B2b' — XLSX format add). The shared writer service this session lays down should be designed to accommodate XLSX as a third format flag without restructuring.
  - **JSON-Lines (`.jsonl`)** — lifted only if a future bulk-export consumer requires it. Default array shape covers the operator-facing 90% case.
  - **Bulk-action exports** (per-row checkbox-driven export of selected rows). Operator filters via the existing table filters; no per-row selection needed.
  - **Importer-side adjustments.** The importer mapper services from session 259 already recognize the standard column shape (since exporter columns mirror `flatFields()`). No importer changes expected.
  - **Schema changes.** No new columns, no new migrations.
  - **Help-system widgets / dashboard surfaces.** Pure list-page admin actions only.

## Phase 2 — Build the shared writer service

**Goal:** A single `App\Services\ListExportService` (or chosen shape) that 10 ListXxx pages call into.

**Steps:**

1. Create `app/Services/ListExportService.php` with a `stream()` method:
   ```
   public function stream(
       Builder $query,
       array $columnSpec,        // [['key' => 'first_name', 'header' => 'First Name', 'transform' => null], ...]
       string $format,           // 'csv' or 'json'
       string $filename,         // 'organizations-2026-05-04.csv'
       ?string $cfModelKey = null,  // 'organization', 'event', null for non-CF resources
   ): StreamedResponse
   ```
2. CSV branch — mirrors the existing `ListContacts::exportContacts` shape: `fputcsv` row by row; CF columns flatten one-per-CFDef; iterate via `$query->each()` for memory.
3. JSON branch — streams the canonical-array shape: `[`, then comma-separated objects, then `]`. CF nested as `custom_fields` sub-object. ISO 8601 dates via Carbon's default cast. Skip empty CFs (don't emit the key if value is empty/null) per the design decision.
4. Refactor the existing [`ListContacts::exportContacts`](../app/Filament/Resources/ContactResource/Pages/ListContacts.php#L177) to call into the service. Behavior must remain bit-for-bit identical for the CSV path; this is a no-op refactor on the user-visible side.
5. Add a unit test set at `tests/Feature/ListExportServiceTest.php` covering: CSV header row + data round-trip; JSON shape + CF nesting; ISO 8601 date format; empty-CF omission in JSON; permission-gate behavior is per-page and lives in the page-level tests.

**Pass criterion:** Existing Contacts CSV behavior unchanged (no test regression on the existing ContactResource tests). The unit-test set on `ListExportServiceTest` covers the format-switching logic.

## Phase 3 — Wire actions into each ListXxx page

**Goal:** All 10 list pages get the ellipsis menu with both export actions.

Each action: appropriate label / icon / permission gate / call into `ListExportService::stream()` with that page's filtered+sorted query, the page's column spec (declared as a static method on the resource or page, depending on architectural-shape decision), the format flag, the filename, and the CF model-key (or `null` for non-CF resources).
4. For `ViewRegistrations.php`: column spec includes the parent event ID + standard registration columns; filename includes the event ID; the query is already event-scoped via the page's existing `getTableQuery` override.

**Pass criterion:** Manual smoke test on each of the 10 list pages confirms the ellipsis menu shows both export actions and clicking each downloads the right shape.

## Phase 4 — Tests

**Goal:** +20 Pest tests (one CSV + one JSON per resource × 10 resources).

**Per-resource test pattern:**

1. **CSV test:** seed 2-3 rows via factory; visit the list page as an authorized user; trigger the export action; assert the response status / content-type / filename; parse the CSV; assert the header row matches the expected column list; assert at least one data row's values round-trip; for CF-having resources, assert CF columns appear and the values match.
2. **JSON test:** same setup; trigger the JSON export; assert response shape; parse the JSON; assert it's a top-level array; assert each element is an object with the expected standard keys; for CF-having resources, assert `custom_fields` is a nested object on each row with the expected handle keys; assert empty CFs are omitted (not present as null/empty); assert ISO 8601 date format.
3. **Permission-gate test (one per resource, can be parametrized):** unauthenticated user, role-mismatch user → 403 / hidden / denied.

**Pass criterion:** 2018 baseline + 20 new = 2038 fast tests passing. Slow group unchanged at 15.

## Security checklist

Before closing:

- [ ] Permission gate enforced at controller layer (not just UI hidden) on every export action — anonymous users + role-mismatched users get a 403, not a redirect-with-empty-export.
- [ ] No PII leakage in JSON: confirm sensitive fields (e.g. internal-only contact notes, admin-only audit columns) don't appear in the export shape unless they're in the importer's `flatFields()` output (which is the canonical shape they round-trip through).
- [ ] Custom-field flattening in CSV doesn't expose unintended fields — CFs scoped to one model don't leak into another's export via shared `CustomFieldDef::forModel` lookups.
- [ ] EventRegistration export is event-scoped — a user viewing event A's registrations cannot URL-fish their way to event B's registrations export.
- [ ] Filter+sort state in the URL is honored on the server side — exports run against the filtered query, not the full unfiltered table (the existing `getFilteredSortedTableQuery()` covers this; verify no per-page override breaks it).
