## What was built

### Phase 1 вЂ” `demoContext()` auth-context rendering (iteration 1)

- Added `demoContext(): ?array` to `app/Widgets/Contracts/WidgetDefinition.php` (default `null`) вЂ” a new member of the `demo*()` hook family declaring `['guard', 'seeder', 'login']` for widgets that need an authenticated user to render.
- `app/Http/Controllers/Dev/WidgetDemoController.php` wraps the render (both `show()` and `showPreset()`) in a `withDemoContext()` helper: when a widget declares a context, run its seeder, resolve the guard's user provider by the `login` identifier, set the member on the guard for the render, and forget it in a `finally`. **Adaptation:** used request-scoped `setUser()`/`forgetUser()` rather than `login()`/`logout()` вЂ” the isolated dev render is a single request and shouldn't mutate session state, which also sidesteps the "needs session/middleware machinery" drift risk entirely.
- New `database/seeders/DemoPortalMemberSeeder.php` вЂ” one idempotent stand-in member: a Contact + a PortalAccount (keyed on `ACCOUNT_EMAIL`) + one EventRegistration against the existing `DemoEventSeeder` event (`demo-community-gala`). `event_registrations.name`/`email` are NOT NULL, so both are set.
- Wired the four portal widgets вЂ” `PortalAccountDashboard`, `PortalContactEdit`, `PortalEventRegistrations`, `PortalChangePassword` вЂ” to `demoContext()`. Regenerated their thumbnails from the byte-identical 2350-byte blank to honest renders (greeting + email; the address form; a real registration row; the password form).
- Three render-path tests in `tests/Feature/WidgetDemoRouteTest.php`: the authenticated portal render is non-blank; the event-registration row resolves through the stand-in member; the portal guard is left guest after the render.

## Security considerations

The only security-relevant surface touched is the **dev-route `demoContext()` render**, which authenticates a seeded stand-in portal member for the thumbnail capture. It is gated to non-production by `DevRoutesMiddleware` (aborts 404 in production), request-scoped via `setUser()`/`forgetUser()` (no session is written, no auth state persists past the render, restoration happens in a `finally`), and the member is dev/demo data (`DemoPortalMemberSeeder`). Portal contact-scoping, the `portal` guard configuration, and the public/admin auth paths are unchanged. Nothing here ships to a production auth path.
