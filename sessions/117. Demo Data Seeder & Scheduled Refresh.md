# 117. Demo Data Seeder & Scheduled Refresh

Build a dedicated demo seeder that populates every content area of the CRM with realistic sample data, and a scheduled command that wipes and reseeds CRM data hourly to keep the demo instance clean. Marketing site pages are never touched.

---

## Design decisions (resolved before starting)

- The demo seeder is a separate seeder class (`DemoSeeder`) — distinct from the existing `DatabaseSeeder` used for development.
- Financial data (donations, purchases, transactions) **is** included in the demo seeder. The constraint against generating financial data in session 112 applied to the dev debug generator; the demo instance needs financial records visible to demonstrate that part of the product.
- Marketing site pages (type `default`, `system`) and navigation menus are **never** wiped. They are managed by the CMS export/import tool (session 116) and considered precious.
- Blog posts (type `post`), events, contacts, members, organizations, products, donations, purchases, transactions, tags, forms, form submissions, and event registrations are all CRM sample data — wiped and reseeded.
- The scheduled refresh runs hourly, combined with the demo account cleanup from session 115. The single `demo:cleanup` command (or a new `demo:refresh` wrapper) handles both user expiry and data reset.
- The seeder uses the existing factory system and the improved generators from session 112 where applicable.
- Collection records are seeded as part of the demo seeder if they don't already exist (content collections like "Featured Events" that widgets reference).
- Custom field definitions are seeded for the demo to showcase the custom fields feature.

---

## Phase 1 — Demo seeder

Create `database/seeders/DemoSeeder.php` that populates:

1. **Contacts** — 50–100 contacts with varied data (some with organizations, some with custom field values, some with tags).
2. **Organizations** — 10–20 organizations, some linked to contacts.
3. **Members** — 15–25 contacts promoted to members with active memberships across different tiers. Include portal accounts.
4. **Events** — 15–20 events with varied prices, capacities, and dates (mix of past, upcoming, and far-future). Free events get registrations. Use the improved generator logic from session 112.
5. **Products** — 8–12 products with prices. Mix of active and archived.
6. **Donations** — 30–50 donations across varied contacts, funds, and dates. Include corresponding transactions.
7. **Purchases** — 15–25 purchases across varied contacts and products. Include corresponding transactions.
8. **Blog posts** — 8–12 posts with text_block and blog_pager widgets. Use the improved generator logic from session 112.
9. **Tags** — 10–15 tags applied to various contacts and events.
10. **Forms** — 2–3 sample forms (a general contact form, a volunteer interest form). Include a handful of sample submissions.
11. **Custom field definitions** — 3–5 definitions (text, select, boolean) with values populated on some contacts.
12. **Mailing lists** — 2–3 lists with filter rules that match seeded data.

Use the `SampleLibrary` data source for realistic names and titles where the factories already support it.

---

## Phase 2 — Wipe command

Create (or extend) an artisan command `demo:refresh` that:

1. **Wipes CRM sample data** — deletes all records from: `event_registrations`, `events`, `donations`, `purchases`, `transactions`, `memberships`, `portal_accounts`, `contacts`, `organizations`, `products`, `product_prices`, `form_submissions`, `taggables`, `tags`, `mailing_list_filters`, `mailing_lists`, `custom_field_defs`, `notes`, `activity_logs`. Also deletes blog posts (`pages` where `type = 'post'`) and their cascading `page_widgets`.
2. **Preserves:** pages of type `default`, `system`, `event`, `member`; navigation menus and items; templates; widget types; site settings; user accounts (those are handled by `demo:cleanup`); funds (reference data); roles and permissions.
3. **Re-runs `DemoSeeder`** after wiping.
4. **Logs** the wipe and reseed to `storage/logs`.

---

## Phase 3 — Scheduled refresh

1. **Register `demo:refresh`** in the scheduler to run hourly (on the hour, or offset from `demo:cleanup` to avoid overlap).
2. **Ensure the command is only registered when `APP_ENV=demo`** — it must not appear in the scheduler on production or local instances.
3. **Verify** that a demo user currently logged in and browsing the admin panel is not broken by the refresh (their session remains valid, they just see fresh data on next page load).
