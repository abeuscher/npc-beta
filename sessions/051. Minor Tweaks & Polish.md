# 051. Minor Tweaks & Polish

A focused polish session. No new features or architectural changes. Work through each workstream in order, reading relevant files before touching them.

---

## Workstream 1 — Event Edit Form Layout

Read `app/Filament/Resources/EventResource.php` in full before making any changes.

1. **Description overflow** — the description field falls outside its container at the bottom of the event edit form. Identify and fix the layout issue.
2. **Field order** — move the title field above the description field.
3. **Tags into Settings** — the tag block should move into the Settings section, matching the pattern used on the contact edit form.
4. **Dates above Settings** — on the event record view/edit page, the dates panel should appear above the Settings section.
5. **Date(s) label** — the dates panel heading should read "Date" when there is exactly one date and "Dates" when there are two or more.
6. **Location Info label** — rename the address fieldset label from its current value to "Location Info".

---

## Workstream 2 — Upload Field UX Investigation

The file upload field in the importer does not enable the Next button until the field turns green (upload complete), but there is no loading indicator during the gap between file selection and completion. This causes a confusing pause with no feedback.

Investigate what Livewire and Alpine lifecycle events are available on the file upload component. Determine whether a loading state (spinner or progress indicator on the Next button, or a status label below the field) can be wired in without fighting the component's internals.

- If a clean approach exists: implement it.
- If it requires overriding Filament internals or is otherwise fragile: document the findings in the session log and defer.

Do not guess. Read the relevant component files and Filament's upload handling before deciding.

## Workstream 3 — Contacts List Action Placement

On the contacts resource list page, move the "Import contacts" and "Export CSV" header actions to the end of the action bar (after any other existing actions). Read `app/Filament/Resources/ContactResource.php` before changing anything.

## Workstream 4 — Delete Confirmation Copy

Filament's default delete confirmation gives no context about what is being deleted. Update the confirmation message on all bulk delete actions and single-record delete actions across all resources to include the record count and type — e.g. "You are about to permanently delete 3 contacts. This cannot be undone."

Read how the existing delete actions are defined before deciding on the cleanest approach. If Filament provides a standard way to override confirmation text globally or per-resource, prefer that over one-off overrides.

## Workstream 5 — CMS Tags on Records

Wire the tag system into the page, post, and event edit forms. The tag field should use the same multi-select with create-on-confirm behaviour as the contact tag field. Tags should be filtered to the appropriate type (`page`, `post`, `event` respectively).

Read the contact edit form tag field implementation first and replicate the pattern exactly. The tag manager screen is not touched.

## Workstream 6 — Custom Header / Footer Test Files

The site supports custom header and footer template overrides but there are no test files to exercise this feature. Create minimal test template files at the appropriate paths (read the existing header/footer implementation to determine where custom overrides are expected). The files should contain clearly labelled placeholder HTML — enough to confirm the override is working, not a finished design.

## Workstream 7 — Debug Data Generator Dashboard Widget

Add an ENV-gated dashboard widget that allows quick generation and deletion of test records. This widget must **only appear** when `APP_DEBUG_TOOLS=true` is set in `.env`. It must not be visible in any other environment.

The widget should support:
- A record type selector (contacts, events, donations — start with these three)
- A quantity input
- A "Generate" button that creates that many seeded records of the selected type using the existing factories
- A "Wipe all" button that hard-deletes all records of the selected type (with a confirmation step)

Scope is intentionally narrow. This is a development aid, not a feature. It will be removed before the product goes live.

Do not create factories that don't already exist. If a factory is missing for a selected type, omit that type from the selector for now.

## Workstream 8 — Default Role Set

The system ships with only `super_admin` and `staff` as seeded roles. Add the following default roles to the roles seeder with sensible permission sets:

- **CRM Editor** — full access to contacts, organizations, households, notes, tags, memberships, donations. No access to finance settings, user management, or import approval.
- **Event Manager** — full access to events and registrations. Read-only access to contacts.
- **Volunteer Coordinator** — full access to contacts, tags, notes. Read-only access to events. No finance access.
- **Treasurer** — full access to donations, funds, campaigns, transactions. Read-only access to contacts. No CRM editing.
- **Blogger** — full access to pages, posts, collections, navigation. No CRM or finance access.

Read the existing seeder and `RoleResource::permissionAreas()` before writing anything. Map each role to the permissions that already exist — do not create new permissions. If a clean mapping is not possible for any role, note it in the log and use the nearest available permissions.

## Workstream 9 — Newsletter Signup Base Form

Add a seeded base form for newsletter signups alongside the existing seeded forms. Read the existing form seeder and any other seeded forms before writing this one.

The form should contain:
- First name (text, optional)
- Last name (text, optional)
- Email (text, required)
- A required checkbox whose label defaults to "Sign up for our newsletter" — the label must be editable in the form builder by the admin, like any other field label

On submission the form should set `mailing_list_opt_in = true` on the created or updated contact record. Use the existing `contact_field` mapping mechanism to achieve this — do not add custom submission logic unless the existing mechanism genuinely cannot handle it.

The form handle should be `newsletter-signup`. The form type should be `contact`.
