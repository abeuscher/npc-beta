# 333. Page Builder ↔ Public — Widget Parity (Behavior + Layout)

Drive the page-builder editor preview to true parity with the public render — in **behavior and layout**, not just content style — by fixing the three editor↔public fidelity bugs that survived the 332 container-query rollback, then standing up a per-widget parity harness (modeled on session 308) that enforces editor==public going forward.

---

## Model + stub reference

Not a `release-plan.md` entry. Driven by the 332 walk-back conversation's **"c" deliverable** and the Mobile Public-Site Readiness stub in `sessions/session-outlines.md` (this session is **promoted ahead of Mobile Nav**, which becomes 334+).

The reusable model is **session 308 — "Page Builder ↔ Public View — Visual Parity (Content Inside Widgets)"** (`sessions/archived/308. *`): a Playwright harness walking the canvas DOM and public DOM in lockstep, diffing computed styles, with a locked exclusion list and the complementary HTML-layer guard (`tests/Feature/BuilderPublicRenderParityTest.php`, session 307). **This session reuses that skeleton and widens the comparison axis from content computed-style to behavior + layout.**

---

## The contract this session defends

**WYSIWYG.** The page-builder preview must match the public render in **appearance and behavior**. An editor that renders a carousel differently from (or more broken than) the public site is "what you see is *not* what you get" — unsellable. "Matches in HTML output" (the 307 guard) is necessary but not sufficient: the carousel must actually initialize, the pager must render, and counts/columns must agree.

A helpful consequence of the 332 rollback: widgets now render their **configured count at every width** (no `@container` collapse), so editor↔public count parity is **width-independent** — both surfaces should show the same configured count regardless of the editor's simulated preview width. The residual divergences are the three bugs below.

## The three bugs (all reproduced live in 332; none introduced by 330/331)

### C — Editor Swiper load race *(priority — the WYSIWYG-killer)*

The page-builder preview boots widget JS by injecting `preview_html` via `v-html` and re-initializing Alpine (`reinitAlpine`), while loading widget libraries via `loadLibs(store.requiredLibs)` ([PreviewCanvas.vue](../resources/js/page-builder-vue/components/PreviewCanvas.vue), [useLibraryLoader.ts](../resources/js/page-builder-vue/composables/useLibraryLoader.ts)). When the widget's Alpine init wins the race against the Swiper `<script>` load, `window.Swiper` / `window.SwiperModules` are undefined; the widget's `buildOpts()` reads `window.SwiperModules.Navigation` **unguarded** (`init()` guards `window.Swiper` but the `x-effect`-driven `rebuildSlides()` does not), throws, and the carousel never initializes — slides stack, zero pager bullets. **Confirmed intermittent** (Swiper absent on one editor load, present on the next). Affects all four Swiper widgets in the editor (BlogListing, EventsListing, LogoGarden, ProductCarousel — all declare `libs: ['swiper']`). Public is unaffected. Loader last changed at 325/148 → predates 330/331.

**Fix candidates (settle at Open Gate, smallest-that-works):** await `loadLibs` before `reinitAlpine` on the initial tree load (not only the `preview_html` watch); guard the widget JS's `window.SwiperModules` access; or both. The acceptance test is determinism: Swiper initializes in the editor on every load, not most loads.

### B — Uneven grid tracks

`.content-card` grid items have no `min-width: 0`, so `grid-template-columns: repeat(N, 1fr)` distorts to content min-width when the grid is narrow (measured `172px / 207px / 159px` for three "equal" `1fr` tracks). Fix: `min-width: 0` on the grid item so N columns render as N *even* columns. Affects [BlogListing/styles.scss](../app/Widgets/BlogListing/styles.scss) and [EventsListing/styles.scss](../app/Widgets/EventsListing/styles.scss). This honors the operator's count cleanly (per 332's preserve-user-control principle) rather than rendering it raggedly.

### E — Possible layout-column drop *(bounded attempt, then report)*

A layout column was lost at some point — the Welcome-page layout is labeled "3 Column Layout" but `page_layouts.columns = 2`; an early measurement saw three content columns, a later one saw two. The current state is **consistent and renders fine** (logo_garden at column_index 0, blog_listing at 1, no stranded widget). Owner hypothesis: **changing a widget's `columns` config inside the middle column of a 3-column layout knocked out a layout column** — i.e. a widget-`columns` write bleeding into the layout's `columns`. **Owner guidance:** make **one** bounded repro attempt (set up a 3-column layout, listing widget in the middle, change its `columns`, watch the layout's column count) — if it fires, fix the state-bleed; if it does not, **report and stop**. Do not chase.

## Phases

### Phase 1 — Reproduce + root-cause (no big change)
Re-confirm C / B / E against the real browser (live local site for public, owner-creds editor for the back end, standalone read-only scripts — never the DB-resetting specs). Pin C's exact race ordering and decide the smallest fix. Decide whether E reproduces. Output: a one-paragraph cause + fix-plan per bug, surfaced at the Open Gate.

### Phase 2 — Fix
- **C:** make Swiper-in-editor deterministic (the chosen approach). Verify by driving the editor repeatedly — pager + carousel present every load.
- **B:** `min-width: 0` on `.content-card`; verify even tracks at narrow width in both surfaces.
- **E:** if reproduced, fix the widget-`columns`→layout-`columns` bleed; else report.

### Phase 3 — Standing parity harness
Extend 308's `tests/e2e/page-builder/widget-visual-parity.spec.ts` (or a sibling) to add, per widget, **behavior + layout** parity on top of 308's content computed-style diff:
- Carousel/JS init parity: same `.swiper-initialized` state and same pagination-bullet count in editor and public.
- Layout/count parity: same rendered column count / cells-per-row for the same widget config.
Runs against the isolated e2e stack on every CI push, non-gating like the rest of the Playwright suite.

### Phase 4 — Lock the exclusion set
Encode surviving intentional divergences (editing affordances, caret/selection, the zoom transform, the documented 307 HTML-only diffs) as an explicit exclusion list; the harness fails on anything outside it. Disposition the won't-fix set with the owner before close.
