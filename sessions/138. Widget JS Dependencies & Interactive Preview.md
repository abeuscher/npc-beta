# 138. Widget JS Dependencies & Interactive Preview

Load widget JS dependencies in the admin page builder so widgets with client-side behavior (Swiper carousels, Chart.js charts, jCalendar) render interactively in the edit-mode preview. Add responsive viewport simulation to the preview.

---

## Design decisions (resolved before starting)

- **Per-widget dependency declaration.** Each widget type declares its external JS library dependencies in the `assets` JSONB column under a new `libs` key ‚Äî an array of library identifiers (e.g. `["swiper", "chart.js"]`). The system reads this at runtime; no hardcoded widget-to-library mapping.
- **Build server produces per-library bundles.** The build server compiles a self-contained JS bundle for each library that registers the expected globals (`window.Swiper`, `window.SwiperModules`, `window.Chart`, `window.calendarJs`). These bundles do NOT call `Alpine.start()` and do not conflict with Filament's Alpine instance. This avoids CDN dependencies and version drift.
- **Dynamic loader in the admin.** When a widget is selected and its preview renders, a JS loader reads the widget type's `libs` array, checks which globals are already present, and loads missing library bundles on demand. Libraries persist for the page session ‚Äî no unloading needed.
- **Alpine.js is already available** via Filament. Widgets that only use Alpine `x-data` (donation form, site header, social sharing, map embed) already work in preview. No special handling needed.
- **Responsive preview uses the existing zoom mechanism.** The `viewportW` variable from session 136 already controls the simulated viewport width. A toggle switches between preset widths (1920, 1024, 768, 375) and recalculates the zoom factor.
- **Undo/redo is out of scope.** Deferred to post-beta.

## Phase 1 ‚Äî Library dependency declarations

Add a `libs` key to the `assets` JSONB for each widget type that needs external JS. Update the seeder and the live DB.

The `libs` array uses string identifiers. Each identifier maps to a build server output bundle. Example:

```json
{ "libs": ["swiper"], "scss": ["resources/scss/widgets/_carousel.scss"] }
```

Audit every widget type's template and JS to determine which ones need `libs` entries. Check blade templates in `resources/views/widgets/` for `<script>` tags referencing `window.Swiper`, `new Chart()`, `new calendarJs()`, etc. Also check the `js` and `code` columns in the seeder for inline JS that references globals.

---

## Phase 2 ‚Äî Build server per-library bundles

Extend the build server to produce individual library bundles alongside the existing combined `public-widgets-{hash}.js`.

Each library bundle:
- Is a self-contained JS file that registers the expected globals
- Does NOT import or start Alpine.js
- Is written to `public/build/libs/{name}.js` (e.g. `public/build/libs/swiper.js`)
- Is produced by the `build:public` artisan command as part of the normal build

The build server needs a new endpoint or an extension to the existing endpoint that accepts per-library source and returns compiled output. Alternatively, use the existing endpoint with a separate payload per library.

Update `AssetBuildService` to:
1. Compile the per-library bundles
2. Write them to `public/build/libs/`
3. Include them in the manifest so the admin loader knows the file paths

---

## Phase 3 ‚Äî Admin JS loader

Build a dynamic script loader that runs in the admin page builder:

1. When a widget preview renders, read the widget type's `libs` array from the block data (already available in the Livewire component's `$block` array via `widget_type_config_schema` ‚Äî add `widget_type_assets` similarly)
2. For each declared lib, check if the global already exists (e.g. `window.Swiper`)
3. If missing, dynamically inject a `<script>` tag pointing to `/build/libs/{name}.js`
4. Wait for the script to load before allowing the preview HTML to initialize
5. Once loaded, the library persists ‚Äî subsequent widgets using the same lib skip the load

Expose the `libs` data: add `widget_type_assets` to the `$block` array in `PageBuilderBlock::loadBlock()` so the blade template can read it.

---

## Phase 4 ‚Äî Responsive preview toggle

Add viewport width presets to the preview container:

- Desktop (1920px) ‚Äî default
- Tablet (1024px)
- Mobile (375px)

UI: a small segmented control or icon toggle above or beside the preview, inside the block card. Changing the preset updates `viewportW` and recalculates `zoomFactor`. The toggle is only visible when a widget is selected.
