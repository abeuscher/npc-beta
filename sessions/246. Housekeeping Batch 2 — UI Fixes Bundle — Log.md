### Phase 1 — Blog Post Pager — page context not resolving correctly

Bug surfaced at session 245 UAT: the `blog_pager` widget on individual blog post detail pages rendered in preview mode (literal `{{item.title}}` template strings) instead of resolving prev/next post navigation.

**Root cause.** `WidgetRenderer::render` reads `app(PageContext::class)` from the container ([WidgetRenderer.php:62](app/Services/WidgetRenderer.php#L62)). `PostController::renderPage` (and `PageController::renderPage`) `View::share('pageContext', $pageContext)` but do not `app()->instance(PageContext::class, $pageContext)` — so the container resolves a fresh `PageContext` with `currentPage = null` in production. The BlogPager template is the only widget template that reads `$pageContext->currentPage` directly (other widgets get page context via the `SOURCE_PAGE_CONTEXT` contract resolution path which uses `$tokenPage` correctly), so the bug only manifested there. The template's `if ($pageContext->currentPage && $pageContext->currentPage->type === 'post')` check short-circuited to false, falling through to the `$isPreview = !$hasPosts` branch that renders literal template strings.

**Fix.** Reconstruct `$pageContext` in `WidgetRenderer::render` after computing `$tokenPage` (the resolved current/owner page). One-line change at [WidgetRenderer.php:69](app/Services/WidgetRenderer.php#L69) plus a comment block naming the production-vs-test parity issue. Renderer-level fix (not controller-level) because four entry points hit `WidgetRenderer::render` — `PageBlockRenderer`, `WidgetDemoController` (dev), `WidgetPreviewRenderer` (admin preview), and `ChromeRenderer`. Single seam covers all of them.

**Regression test.** New `tests/Feature/BlogPagerEndToEndTest.php` walks `PostController::show` via the actual HTTP route (no `app()->instance(PageContext::class, ...)` binding) and asserts:
- The pager renders real anchor links for prev/next neighbors (not `{{item.title}}` template strings).
- URLs match `url('/' . $blogPrefix . '/' . $slug)` for each neighbor.
- Both prev and next render when host is mid-list; only next renders when host is oldest.

The existing `BlogPagerContractRetrofitTest` passing tests already covered the contract-resolution layer (with explicit `app()->instance(...)` binding). The new test covers the previously-untested production HTTP path.

### Phase 2 — Text Widget Vertical Alignment

Per-widget control (option a in the prompt's design call), narrow scope. Generalized version (option b — universal Appearance layer) captured as a new post-Beta-1 stub.

**Schema.** Added `vertical_align` to [TextBlockDefinition::schema()](app/Widgets/TextBlock/TextBlockDefinition.php) as a `select` field with options Top / Middle / Bottom, default `middle` (preserves current behavior). Added matching default to `defaults()`. Group: `appearance` (matches existing pattern from SocialSharing/Hero alignment fields).

**Template.** [TextBlock template.blade.php](app/Widgets/TextBlock/template.blade.php) emits `<div class="widget-text-block widget-text-block--vertical-{{ top|middle|bottom }}">`. The template defends against unknown values via an `in_array` allowlist (falls back to `middle`).

**SCSS.** New rules in [resources/scss/_custom.scss](resources/scss/_custom.scss). The `.widget--text_block` wrapper gets `height: 100%` so it stretches to its parent's row height in column-layout grid context; the `.widget-text-block` template wrapper becomes a column-direction flex container at `height: 100%` with `justify-content` mapped from the alignment value. Visible only when the parent row has determinate height taller than the rich-text content (the user's "text block next to a taller image" scenario). At root level the wrapper has natural height and the flex layout is a no-op — no behavioral regression.

**Render test.** New `tests/Feature/TextBlockVerticalAlignTest.php`:
- Default value renders `widget-text-block--vertical-middle` class.
- Each of `top` / `middle` / `bottom` renders the matching modifier class.
- Unknown value falls back to `middle` (defensive allowlist test).

**Roadmap stub.** Added `### Universal Appearance — Vertical Alignment` to [session-outlines.md](sessions/session-outlines.md) under the CMS & Page Builder Post-Beta 1 section, immediately after Hero Full-Bleed Promotion to Universal Layer (both stubs are about lifting per-widget concerns into the universal Appearance layer). Lift earlier than post-Beta-1 if a second widget surfaces the same friction.

### Phase 3 — Heroicon Picker in Quill Editor

**Confirmed at session start:** Quill v2 (CDN-loaded by AdminPanelProvider — the prompt was correct; the 243 close-gate log mis-named it 1.x). 324 outline heroicons available server-side via the `blade-ui-kit/blade-heroicons` Composer package (no npm install needed).

**Server endpoint.** New [HeroiconController](app/Http/Controllers/Admin/HeroiconController.php) — single index method that reads `vendor/blade-ui-kit/blade-heroicons/resources/svg/o-*.svg`, maps each filename to `{name, svg}`, caches the resulting array under `admin:heroicons:outline:v1` for 24 hours, returns JSON. Route registered in [AdminPanelProvider.php:60-62](app/Providers/Filament/AdminPanelProvider.php#L60-L62) with the `Filament\Http\Middleware\Authenticate` middleware (mirrors the inline-image-upload pattern).

**Quill blot.** New [resources/js/admin/heroicon-blot.js](resources/js/admin/heroicon-blot.js) — extracted into a shared module so both the Filament admin Quill and the page-builder Vue Quill consume the same registration. Defines `HeroiconBlot extends Embed` with `tagName: 'span'`, `className: 'ql-heroicon'`, `data-heroicon` attribute carrying the icon name, inner `<svg>` carrying the visual. Round-trip via DOM: Quill rebuilds the document model from `root.innerHTML` on load, finds matching `<span class="ql-heroicon">` elements, and rewraps them — the blot's `static value(node)` reads back the name + SVG. Idempotent registration via a module-level flag.

**Picker UI.** New [resources/js/admin/heroicon-picker.js](resources/js/admin/heroicon-picker.js) — a vanilla-JS popover with search input + 7-column icon grid. Lazy-fetches the manifest once per page session, caches in module scope. Anchored to the toolbar button via `position: fixed` (so Filament's overflow chrome can't clip it). Closes on outside click or Escape. Filters live on input; first 240 matches render. The fetch URL is configurable via `setHeroiconsUrl(url)` so different bootstrap contexts (admin Quill, page-builder Quill) can share the same picker module.

**Filament admin Quill wiring.** [resources/js/admin/quill-editor.js](resources/js/admin/quill-editor.js) — adds `'heroicon'` to the toolbar container, registers a `heroicon` handler that opens the picker and inserts via `quill.insertEmbed(range.index, 'heroicon', icon)`. Toolbar button SVG (heroicon outline `face-smile`) injected into the empty `.ql-heroicon` button after Quill init. URL passed via `data-heroicons-url` attribute on the Quill blade view ([quill-editor.blade.php](resources/views/forms/components/quill-editor.blade.php)) — same pattern as the existing `data-upload-url` for inline images.

**Page-builder Vue Quill wiring.** Surfaced during 246 manual testing — the user noticed the heroicon picker was only available in Filament admin Quill (event description, blog post body, email templates) but not in the page-builder Vue inspector's Quill (TextBlock, Hero, ThreeBuckets, BlogPager templates). Lifted in scope:
- [RichTextField.vue](resources/js/page-builder-vue/components/fields/RichTextField.vue) imports the shared blot + picker modules, registers the blot at `onMounted`, conditionally adds `'heroicon'` to the toolbar (only when `store.heroiconsUrl` is set), and wires a `handleHeroiconInsert` handler that mirrors the admin Quill behavior.
- URL plumbed through bootstrap data: [PageBuilder.php:378](app/Livewire/PageBuilder.php#L378) emits `'heroicons_url' => '/' . $adminPath . '/heroicons'`. The two sibling Livewire builders ([DashboardBuilder.php](app/Livewire/DashboardBuilder.php), [RecordDetailViewBuilder.php](app/Livewire/RecordDetailViewBuilder.php)) emit the same URL — so any richtext field rendered in those contexts (Memos in dashboard, future record-detail richtext widgets) also gets the picker.
- [types.ts](resources/js/page-builder-vue/types.ts) BootstrapData gains optional `heroicons_url`. [editor.ts store](resources/js/page-builder-vue/stores/editor.ts) gains a `heroiconsUrl` ref populated from bootstrap, exposed to RichTextField.

**Inline icon styling.** Added to BOTH [public/css/admin.css](public/css/admin.css) (admin editor preview) AND [resources/scss/_custom.scss](resources/scss/_custom.scss) (public render). Inline icon: 1em square, `vertical-align: -0.125em`, `currentColor` inheritance via the heroicon outline `stroke="currentColor"` convention. Picker UI styles live in admin.css only (popover + grid + search; not needed on public).

**Help-doc stub.** New [resources/docs/quill-heroicon-picker.md](resources/docs/quill-heroicon-picker.md) — frontmatter only. Body deferred to E13 (Help docs body content) per the help-docs roadmap.

**Visual smoke-tested.** User confirmed the picker works in admin Filament Quill (event description manual test) and in the page-builder Vue Quill (TextBlock manual test). Round-trip persistence confirmed (insert → save → reopen).

### In-session lift — Demo seeder leak bug captured

Surfaced during 246 manual testing: the events list contained three rogue events ("Bedrock Community Dig", "Moose Lodge Annual Meeting", "Annual Anvil Drop Gala", slugs `demo-event-1/2/3`, source `human`). Traced to [EventCalendar/DemoSeeder.php](app/Widgets/EventCalendar/DemoSeeder.php) writing fixture rows on the "Seed Widget Collections" admin action without tagging them with `Source::DEMO` (or `Source::SCRUB_DATA`) — so they default to `Source::HUMAN` and escape the Random Data Generator's wipe (which filters by `source = scrub_data`). Same shape applies to LogoGarden, Carousel, BoardMembers, BarChart, DonationForm — every widget whose `WidgetDefinition::demoSeeder()` returns a class. The leaked rows were deleted from the live DB during diagnosis. Per user instruction, the seeder fix itself is **out of scope for 246** — captured as a new Housekeeping Batch 2 bullet in [session-outlines.md](sessions/session-outlines.md) so it travels into a future Batch 2 sub-session. The note also flags an apparent flaky `RefreshDatabase` rollback under the parent `seedWidgetCollections — runs each widget definition that declares a demoSeeder` Pest test — rows confirmed in the DB with timestamps matching the full-suite run, despite the test passing in isolation cleanly. Investigation slot belongs in the same fix.
