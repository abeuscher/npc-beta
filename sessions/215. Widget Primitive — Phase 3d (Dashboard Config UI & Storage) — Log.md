## What was built

### Phase 1 — Schema, model, seeder, render-path swap

- **Migration** `2026_04_24_022229_create_dashboard_configs_table` — `dashboard_configs` with `id` (uuid PK), `role_id` (unsignedBigInteger, FK→`roles.id` cascade on delete, **unique** — one config per role), `label` (nullable string), timestamps. Schema doc at `docs/schema/dashboard_configs.md`, indexed in `docs/schema/README.md`.
- `App\Models\DashboardConfig` — `HasUuids`, `HasFactory`, `$fillable = ['role_id', 'label']`, `role()` → `belongsTo(Spatie\Permission\Models\Role::class)`, `widgets()` → `morphMany(PageWidget::class, 'owner')` (reusing the existing polymorphism — no parallel `dashboard_widgets` table). Static helper `forUser(?User $user): ?self` resolves the user's roles by ascending `roles.id` and returns the config for the first-matching role, or null.
- `Database\Seeders\DashboardConfigSeeder` — idempotent. `firstOrCreate` a `DashboardConfig` for `super_admin`; attaches three `PageWidget` rows (Memos `limit=5`, Quick Actions `actions=['new_contact','new_event','new_post']`, This Week's Events `days_ahead=7`) only on first creation. Never clobbers an existing arrangement — the "config has widgets" check bails early. No-op when `super_admin` role is absent. Wired into `DatabaseSeeder::run()` after `MemosCollectionSeeder`.
- `app/Filament/Widgets/DashboardSlotGridWidget::widgets()` — three hardcoded entries replaced with `DashboardConfig::forUser(auth()->user())?->widgets()->where('is_active', true)->with('widgetType')->orderBy('sort_order')->get()->all()` (or `[]`). The Filament blade view (`resources/views/filament/widgets/dashboard-slot-grid.blade.php`) gained an `@if ($hasArrangement)` branch: empty state reads *"No dashboard arrangement for your role — ask an admin to configure one in Settings → Dashboard Settings."*
- Tests:
  - `tests/Unit/Models/DashboardConfigTest.php` — 6 tests covering `morphMany` relation, `role()` belongs-to, `forUser()` for sole role, null user, user-with-no-roles, user-with-no-config-for-role, and multi-role-picks-lowest-id.
  - `tests/Feature/DashboardConfigSeederTest.php` — 5 tests: happy-path seed, idempotency, "never clobbers existing arrangement," no-op when super_admin absent, and defaults match the session-214 hardcoded configs byte-for-byte.
  - `tests/Feature/DashboardSlotGridTest.php` — existing file rewritten. 6 tests, including the pre-existing three plus: `widgets()` returns `[]` when acting user has no role, `widgets()` returns the super_admin config's three widgets in sort order, empty-state copy renders when config is absent. Livewire mount query count recorded: 7 (was 6 in 214; the +1 is `DashboardConfig::forUser` lookup).

### Phase 2 — `DashboardBuilderApiController` + routes

- `app/Http/Controllers/Admin/DashboardBuilderApiController.php` — every endpoint gated by `manage_dashboard_config`; widget-keyed endpoints additionally assert `$widget->owner_type === DashboardConfig::class` AND `$widget->owner_id === $dashboardConfig->id` (IDOR guard at the owner-scoped route boundary). Endpoints:
  - `GET configs/{dashboardConfig}/widgets` — tree (`items` + `required_libs`).
  - `POST configs/{dashboardConfig}/widgets` — create. Rejects (422) if the widget handle's `allowedSlots()` does not include `dashboard_grid` — **first enforcement of `allowedSlots()` at the create boundary.**
  - `PUT configs/{dashboardConfig}/widgets/{widget}` — update. Filters `appearance_config` to the keys declared by `DashboardGridSlot::layoutConstraints()['allowed_appearance_fields']` (currently `['background', 'text']`). Writes to `layout` or other disallowed keys are silently stripped, not stored.
  - `DELETE configs/{dashboardConfig}/widgets/{widget}` — delete.
  - `PUT configs/{dashboardConfig}/widgets/reorder` — batch sort-order update scoped to the config's widgets.
  - `GET configs/{dashboardConfig}/widgets/{widget}/preview` — preview rendered with `slotHandle: 'dashboard_grid'`.
  - `GET configs/{dashboardConfig}/widget-types` — filtered to handles whose definition's `allowedSlots()` includes `dashboard_grid`.
  - `POST/DELETE configs/{dashboardConfig}/widgets/{widget}/appearance-image` — parity with the page-builder controller's appearance-image endpoints.
- `routes/admin-dashboard-api.php` — new route file mirroring the page-builder shape but owner-scoped only (no widget-keyed routes outside the config prefix, no layout/nesting routes — dashboard is flat). Registered under `/admin/api/dashboard-builder` in `AdminPanelProvider::panel()`.
- Supporting extensions:
  - `WidgetType::forPicker(?string $pageType, ?string $slotHandle = null)` — the `$slotHandle` filter is the mechanism that makes the picker slot-aware.
  - `WidgetPreviewRenderer::render(PageWidget $pw, string $slotHandle = 'page_builder_canvas')` — slot handle plumbed through to `WidgetRenderer::render()`.
  - `WidgetPreviewResource::withSlotHandle()` — resource-level preview-slot plumbing to mirror `withRequiredHandles()`.
- `tests/Feature/DashboardBuilderApiControllerTest.php` — 11 tests covering: 403 for users without the permission, 200 for super_admin, allowedSlots rejection on create, `dashboard_grid`-allowed happy path on create, IDOR rejection for a widget belonging to a different config, IDOR rejection for a widget belonging to a Page (not a DashboardConfig), appearance-key whitelist stripping, reorder happy path, delete happy path, widget-types filter returns exactly the three dashboard-allowed handles.

## Deferred decisions

- **Per-user dashboards.** Locked decision stays per-role. If demand surfaces, add a `user_dashboard_overrides` join table or an ownership field on `DashboardConfig`; not blocking.
- **Multi-role users: union / override / precedence beyond "first role by ascending `roles.id`".** If the simple rule proves painful (e.g. when a user has both `developer` and `super_admin` roles and the wrong dashboard surfaces), add an explicit `default_dashboard_role_id` on `users`. Not needed today.
- **Per-widget width/height sizing on the dashboard.** `DashboardGridSlot::layoutConstraints()['dimensions']` remains declarative. List-ordered arrangement is the sole mechanism. Implementing `dimensions` is a separate session, paired with builder-side preview grid rendering (see below).
- **Builder-side dashboard grid preview.** Confirmed during manual testing that the builder stacks widgets vertically while the actual dashboard tiles them at 2-across via `.np-dashboard-slot-grid`'s CSS Grid. This mismatch is expected at present and out of scope; the session prompt explicitly descoped per-widget sizing. Tied to the `dimensions` implementation whenever that lands.
- **Presets tab in dashboard mode.** Hidden entirely rather than filtered. If designer-driven presets become useful for dashboard widgets, the `createDraftPreset` / `applyPreset` flows would need a dashboard-aware equivalent; trivial in principle but not needed yet.
- **Config lookup caching.** The `DashboardConfig::forUser` call adds one query per dashboard mount (6 → 7). If this becomes a hotspot, memoize at the request level. Not today.
- **Full-suite OOM mid-run.** Unchanged pre-existing condition tracked since session 212. `SeoTest` passes in isolation (22/22). Flagged for the eventual test-suite audit session; not this track's responsibility.
- **Phase 4 widget retrofit — rollout sequence.** Drafted into session 216: BlogListing first as the canary (its contract landed in 209, much of the retrofit plumbing was already done during Phase 3). Pattern validation before committing to the remaining ~19 widgets.
