# 236. Widget Primitive — DashboardConfig → DashboardView Retrofit

Lift the dashboard's per-role widget-composition model into the unified View vocabulary established by Phase 5: rename `App\Models\DashboardConfig` to `App\WidgetPrimitive\Views\DashboardView` (mirror of `RecordDetailView`'s namespace), rename the `dashboard_configs` table to `dashboard_views`, and have the model implement the `IsView` interface that has carried its name in the docblock since Phase 5b. Render and admin behavior are byte-equivalent — the three existing dashboard-native widgets (Memos, Quick Actions, This Week's Events) and the `DashboardSlotGridWidget` host continue to work end-to-end through the same paths after the rename, providing live verification.

---

## Design decisions (resolved before starting)

### Decision 1 — Rename, not Eloquent overlay

Two paths were considered:

- **Overlay**: keep the table named `dashboard_configs`; move only the model class to `App\WidgetPrimitive\Views\DashboardView` with a `protected $table = 'dashboard_configs'` override. Saves the migration.
- **Rename** (chosen): rename both model class AND table. Update polymorphic `page_widgets.owner_type` rows in the same migration. Symmetric with `record_detail_views`.

The track doc's "a+" decision says DashboardView "speaks the View vocabulary natively from its introduction." Native means model name and table name both align with the View vocabulary. The migration cost is small (one `Schema::rename` + one `DB::table` update) and the symmetry pays off every time someone reads `database/migrations/` or `docs/schema/` looking for the View pattern. Overlay would leave `dashboard_configs.md` sitting next to `record_detail_views.md` in `docs/schema/` and would require future readers to know that table-name and concept-name diverge. Not worth it.

### Decision 2 — Keep `role_id` as the unique key; do not introduce a `handle` column

`RecordDetailView` is keyed by `(record_type, handle)` because each record type can carry multiple Views (Templates/Themes is the canonical multi-View case). The dashboard's natural key is `role_id` — one DashboardView per role, no multi-View affordance and no roadmap need for one. The `IsView::handle()` method returns a derived value: the role's name slug (`'super-admin'`, `'crm-editor'`, etc.). No new column on the table.

```php
public function handle(): string
{
    return str($this->role?->name ?? 'unknown')->slug()->toString();
}
```

`handle()` is read by callers that want a stable string identifier; the `role_id` FK remains the canonical lookup. `forUser()` keeps its current shape and stays the primary access seam.

### Decision 3 — `slotHandle()` returns the literal `'dashboard_grid'`

Identical pattern to `RecordDetailView::slotHandle()` returning `'record_detail_sidebar'`. No column needed; the constant is part of the type.

### Decision 4 — `widgets()` shape: keep the MorphMany relation and rename the IsView accessor

`RecordDetailView` has both a `pageWidgets(): MorphMany` relation method AND a `widgets(): array` IsView interface method that wraps the relation with eager-loading + ordering. `DashboardConfig` today has a single `widgets(): MorphMany` relation. To preserve callers that depend on the relation shape (e.g., `DashboardSlotGridWidget` calling `$config->widgets()->where(...)`) AND satisfy the IsView interface, rename the relation method to `pageWidgets()` (matching `RecordDetailView`) and add a new `widgets(): array` returning the eager-loaded, ordered collection. Update the four caller files accordingly.

### Decision 5 — `layoutConfig()` returns `[]`

Dashboards do not carry layout config today — they're a flat `dashboard_grid` slot, not a column-stackable canvas like record-detail. The interface method satisfies the contract by returning an empty array. Adding a `layout_config` JSONB column is deferred — there is no current consumer that would read it.

### Decision 6 — `ViewRegistry` integration is out of scope

`ViewRegistry` today carries `forRecordType()` and `findByHandle()` — both shaped against `RecordDetailView`'s `(record_type, handle)` key. DashboardView's per-role keying doesn't fit that shape, and the existing `DashboardConfig::forUser()` static covers every current call site. Pushing DashboardView into a generalized registry is a forward hook, not this session's scope. The `IsView` interface alone is enough to make the type a citizen of the View vocabulary; registry membership is a separate axis.

### Decision 7 — Seeder rename + DatabaseSeeder reference

`DashboardConfigSeeder` renames to `DashboardViewSeeder`. The class name is the only thing that consumes the FQN — its body changes only `App\Models\DashboardConfig` → `App\WidgetPrimitive\Views\DashboardView` and any prose. `DatabaseSeeder` (and the e2e helper `tests/e2e/helpers/db.ts` if it references the seeder class) update accordingly.

## Phase 1 — Migration: rename table + backfill polymorphic owner_type

Single new migration. Filename pattern: `YYYY_MM_DD_HHMMSS_rename_dashboard_configs_to_dashboard_views.php`.

```php
Schema::rename('dashboard_configs', 'dashboard_views');

DB::table('page_widgets')
    ->where('owner_type', 'App\\Models\\DashboardConfig')
    ->update(['owner_type' => 'App\\WidgetPrimitive\\Views\\DashboardView']);

DB::table('page_layouts')
    ->where('owner_type', 'App\\Models\\DashboardConfig')
    ->update(['owner_type' => 'App\\WidgetPrimitive\\Views\\DashboardView']);
```

`page_layouts` is included for parity even though dashboards do not use column layouts today — if any layout row was ever attached, the rename catches it. The `down()` method reverses both the table rename and both backfills.

**Run:** `docker compose exec app php artisan migrate`. **Update schema docs:** rename `docs/schema/dashboard_configs.md` to `docs/schema/dashboard_views.md`, edit the title and any in-doc references, and update `docs/schema/README.md`'s table index.

## Phase 2 — Move + rename the model

Delete `app/Models/DashboardConfig.php`. Create `app/WidgetPrimitive/Views/DashboardView.php`.

```php
namespace App\WidgetPrimitive\Views;

use App\Models\PageWidget;
use App\Models\Role;
use App\Models\User;
use App\WidgetPrimitive\IsView;
use Illuminate\Database\Eloquent\Concerns\HasUuids;
use Illuminate\Database\Eloquent\Factories\HasFactory;
use Illuminate\Database\Eloquent\Model;
use Illuminate\Database\Eloquent\Relations\BelongsTo;
use Illuminate\Database\Eloquent\Relations\MorphMany;

class DashboardView extends Model implements IsView
{
    use HasFactory, HasUuids;

    protected $fillable = ['role_id', 'label'];

    public function role(): BelongsTo { return $this->belongsTo(Role::class); }

    public function pageWidgets(): MorphMany
    {
        return $this->morphMany(PageWidget::class, 'owner');
    }

    public function handle(): string
    {
        return str($this->role?->name ?? 'unknown')->slug()->toString();
    }

    public function slotHandle(): string { return 'dashboard_grid'; }

    public function widgets(): array
    {
        return $this->pageWidgets()
            ->where('is_active', true)
            ->with('widgetType')
            ->orderBy('sort_order')
            ->get()
            ->all();
    }

    public function layoutConfig(): array { return []; }

    public static function forUser(?User $user): ?self
    {
        if (! $user) return null;
        $role = $user->roles()->orderBy('id')->first();
        if (! $role) return null;
        return static::where('role_id', $role->id)->first();
    }
}
```

The factory class (if any) renames in step with the model — `DashboardConfigFactory` → `DashboardViewFactory`, namespaced accordingly, with the `$model` property updated.

## Out of scope

- **`ViewRegistry` integration for DashboardView.** Per Decision 6 — the registry is shaped against `RecordDetailView`'s `(record_type, handle)` key; per-role DashboardView doesn't fit. Forward hook.
- **`layout_config` JSONB column on dashboards.** No current consumer; deferred until something genuinely needs it.
- **Multi-DashboardView per role.** No use case; `role_id` stays `unique`. If the dashboard ever grows multiple authoring surfaces (e.g., default vs. custom-tab), a `(role_id, handle)` composite-unique migration follows then.
- **Refactoring `DashboardSlotGridWidget` to consume the `IsView` interface generically.** That would be the start of Phase 6 (page-builder convergence). This session keeps the existing host code; only the model shape changes.
