## Design resolved at 280-close (do not re-litigate)

The 281 session-prep conversation at 280-close resolved the full design. Reproduced here so the session executes against a fixed target:

- **Pessimistic lock acquired on edit-page mount.** Lock key = (user_id, lockable_type, lockable_id, session_id).
- **Optimistic version check at save** as the safety-net layer for any race where two editors got the page open. Second save shows "this changed while you were editing" and forces a reload.
- **Super-admin override anytime**, including against other super-admins.
- **Non-super-admin takeover after 5+ minutes since the last save** on the locked record.
- **Idle definition:** time since the last save on the locked record (deterministic, simple — not heartbeat-driven for v1).
- **`beforeunload` + `sendBeacon` JS hook** to release the lock proactively on tab close. Confirmed for v1 scope.
- **No active force-close** of the displaced editor's window — they get a save-time warning instead.
- **List-view "locked by X" indicator** across resources. Filament table column shared via the lock trait.
- **Centralized polymorphic lock table + shared trait/service.** Resources opt in via a flag. New resources get locking for free.
- **Portal users out of scope** — locks live in the admin panel only.
- **Cancel button releases the lock immediately.** Tab close releases via `beforeunload`. Browser crash / OS kill releases via idle expiry.

## Open questions to resolve at session start

- **Which resources opt into locking on day 1?** The polymorphic table supports all admin-editable resources; the question is which surfaces get the trait turned on and the list-view indicator wired up in this session. Recommendation: start with the resources named in the release-plan (contacts, pages) plus the high-edit-traffic surfaces (posts, events). The rest opt in over time as the trait is well-bedded. Decide at session start.
- **Lock-expiry sweep mechanism.** The 5-min idle is enforced by either (a) a scheduled job that sweeps expired rows, or (b) on-acquire check that drops any expired lock the new acquirer overlaps. Option (b) avoids the scheduler-runner gap and keeps the lock table self-cleaning under traffic. Recommendation: on-acquire check, with a much-longer scheduled-sweep (e.g. daily) as a safety net once the scheduler runner is in place.
- **Session-id handling.** Including `session_id` in the lock key means a user's two browser windows on the same record each acquire their own lock (and block each other). Excluding `session_id` means a user's two windows share the lock (both can edit; last save wins for that user). Recommendation: exclude `session_id` from the key for v1 — two-window same-user is a self-inflicted edge case, and excluding it simplifies the takeover UX. Decide at session start.
