## Workstream Results

### WS1 — Edit page field audit
**Clean.** All new models fully covered by their edit forms:
- `FormResource` — title, handle, description, is_active, fields (repeater), settings (sub-fields) all present
- `FormSubmissionsRelationManager` — view-only infolist; no edit form needed (immutable records)
- `ImportStagedUpdate` — no admin UI; internal import use only; correct
- `SiteThemePage` — all 12 SiteSetting keys covered by form fields

### WS2 — DB schema doc accuracy
**Clean.** `forms`, `form_submissions`, and `import_staged_updates` accurately documented in `docs/schema.md`. Schema doc updated to session 049 with `deleted_at` columns and FK index notes added for forms and form_submissions.

### WS3 — Permission gate audit
**Two gaps fixed:**

1. `form` was missing from `RoleResource::permissionAreas()['cms']` — form CRUD permissions existed in the DB but could not be assigned to any custom role through the UI. Added.

2. Six permissions were seeded but absent from `RoleResource::standalonePermissions()` — `import_data`, `review_imports`, `edit_theme_scss`, `view_any_form_submission`, `view_form_submission`, `delete_form_submission`. Non-super-admin roles had no way to receive these through the UI. All six added with labels and descriptions.

Gate table for new resources/pages — all checked and correct:
- `FormResource.canViewAny()` → `view_any_form` ✓
- `SiteThemePage.canAccess()` → `view_any_page` OR `edit_theme_scss` ✓
- Import pages → `import_data` / `review_imports` ✓
- `super_admin` covered by `Gate::before` bypass; no explicit permissions needed ✓

### WS4 — Soft deletes consistency
**Decisions made and implemented:**
- `Form` → **soft delete added** — form definition + submission history; accidental hard delete loses association
- `FormSubmission` → **soft delete added** — audit records; but deletion action swapped to force-delete only (GDPR compliance — no restore path)
- `ImportStagedUpdate` → **hard delete kept** — transient data; created for import review, no recovery value

Migrations: `2026_03_21_000002` and `2026_03_21_000003`.

### WS5 — Fillable vs. casts consistency
**Clean.** All three new models correct:
- `Form`: fields→array, settings→array, is_active→boolean ✓
- `FormSubmission`: data→array, created_at→datetime, UPDATED_AT=null ✓
- `ImportStagedUpdate`: attributes→array, tag_ids→array ✓

### WS6 — Foreign key index audit
**One gap fixed.** `form_submissions.form_id` and `form_submissions.contact_id` lacked explicit indexes (PostgreSQL does not auto-index FK columns). Migration `2026_03_21_000001` adds both. `import_staged_updates` already had explicit indexes. `docs/schema.md` updated.

### WS7 — Orphaned documents and help coverage
**Clean.** All routes in every `resources/docs/*.md` file verified against `php artisan route:list`. No orphaned routes. No admin pages or resources added since session 042 are missing a help doc. `forms.md` and `public-theme.md` route arrays both correct.

### WS8 — Factory coverage
**Two factories created:**
- `FormFactory` — minimal factory with sensible settings defaults
- `FormSubmissionFactory` — references `FormFactory`; contact_id nullable

Both models gained `HasFactory` trait.

### WS9 — Frontend build system audit
**Clean.**
- `/public/build` covered by `.gitignore` ✓; no compiled assets tracked
- `public.scss`: three `@use` imports, all resolved, no orphaned imports ✓
- `_theme.scss`: one active variable (`$spacing-block`), commented examples — clean ✓
- `_layout.scss`: all selectors correspond to active public template elements ✓
- `public.js`: Alpine store + start — no orphaned imports or listeners ✓
- `npm run build` inside container: exits 0, no warnings, 1.5 s ✓

### WS10 — Per-resource soft-delete trash UI (added during session)
**Implemented.** Discussed and agreed during session. Native Filament trash pattern applied across all soft-deleted resources:

- `TrashedFilter` added to every soft-deleted resource list (Contact, Organization, Household, Membership, Donation, Campaign, Note, Page, Post, Collection, Form)
- `RestoreAction` + `ForceDeleteAction` + bulk variants added to every resource, gated to the existing `delete_{resource}` permission
- `FormSubmissionsRelationManager` — `DeleteAction` replaced with `ForceDeleteAction` (permanent, no restore); existing `delete_form_submission` permission check preserved

**Privacy principle recorded**: product is not a marketing tool; always err toward removing data and respecting privacy; erasure paths should be prominent and restoration paths absent for PII-bearing records.
