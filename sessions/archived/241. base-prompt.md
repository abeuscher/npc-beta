## Process rules for every session

- **Before writing any code**, read every file you intend to modify.
- **Before implementing any integration** (email, file storage, authentication, payments, queues, or any other infrastructure), ask the user what existing system handles it. This project has accumulated significant infrastructure — assume something already exists before building new. Do not implement from scratch until you have confirmed no existing system covers the use case.
- **Pause and ask** any time a decision point arises that is not covered by the agreed prompt.
- **If any external service is unavailable**, stop and ask — do not attempt to troubleshoot.
- **No migrations expected** this session — pure tooling + test edits + a doc. If a migration becomes necessary, stop and ask (it would mean scope has wandered).
- **If the PostgreSQL container becomes unresponsive** (500/exec errors), ask the user to restart it rather than retrying.
- **The fast Pest suite must stay green throughout**. Run `docker compose exec app php -d memory_limit=2G vendor/bin/pest --exclude-group=slow` between every commit. Baseline at 240 close: 1678 fast cases passing. The 2G memory override avoids the pre-existing mid-suite OOM fatal documented since session 212. Phase 3 deletions reduce this number — that is expected. Phase 1 and 2 do not touch tests at all and the fast suite must produce exactly 1678 after Phase 1.
- **Hard pause after Phase 2.** Once Infection has produced a usable report and you have built the dead/redundant/load-bearing/surviving-mutants table, stop and surface it. Do not begin Phase 3 deletions until the user has reviewed the table and approved which rows to act on. This is the load-bearing session moment — without it, 241 devolves into "Claude deletes things and we hope they were right."
- **Front-end builds:** none expected.
- **Tests for new behaviour:** **NO new tests in this session, by design.** This is an audit; deletions and consolidations are the deliverable. If Infection surfaces a real bug via a surviving mutant, file it as a follow-up and do not fix it in-session. The "Test Suite Audit" stub in the roadmap is the natural home for any surviving-mutants triage that emerges.
- **Override-comment discipline.** When the user keeps a test against mutation evidence (e.g., it guards a known historical regression), add a one-line `// guards: <regression description>` comment above the test body. This prevents the next mutation pass from re-flagging it. The `// guards:` token is the searchable marker.
- **Branch shape — two commits.** First commit: Infection install + PCOV-in-Dockerfile + `infection.json5` + workflow doc + Phase 2 report findings (committed even if Phase 3 lands no deletions — the infrastructure is reusable independently of any single audit's findings). Second commit: deletions and consolidations from Phase 3, if any. If Phase 3 lands nothing, only the first commit ships.
- **Coverage-driver opt-in.** PCOV gets installed in the Dockerfile but is NOT enabled by default in the test profile. Enabling it for a normal Pest run slows the suite measurably. The Phase 1 invocation pattern (in the workflow doc) names the env-var or `php -d` flag that turns it on for Infection runs only.
- **Cross-repo coordination — Fleet Manager agent contract.** This session is **not** boundary-touching. The standing cross-repo flag is dormant: this session does not modify any file in `app/Http/Controllers/Api/Fleet/*`, `app/Http/Middleware/AuthenticateFleetManagerAgent.php`, `config/fleet.php`, the response schema, or the VERSION pipeline. If during Phase 3 a deletion or consolidation **would** touch any of those, stop and ask — the contract surface is locked at v1.1.0 until the next deliberately boundary-touching session.
- **Pause for manual testing.** Phase 1 ends with a smoke-test Infection run against a tiny target (one or two methods of `ContractResolver` against the matching test file) — this proves the toolchain works end-to-end. Announce the smoke-test result and the discovered slice scope, then proceed to Phase 2 (the substantive Infection run). Phase 2 ends with the report — pause there per the rule above. After Phase 3 (if it lands deletions), announce the final fast-suite count and the deletions made. Do not suggest closing the session or take any further action.
