# Session 167 — Page Builder as Main Edit View — Log

**Date:** 2026-04-12
**Branch:** session-167/1

## What was built

### Phase 1 — Audit

Read all 15+ files in scope. Identified the cleanest architecture: keep `EditPage` as an `EditRecord` (for Filament route/record resolution) but override the view to render only the Livewire `PageBuilder` component. Confirmed `EventResource` also uses the `HasPageBuilderForm` trait, requiring backward-compatible refactoring.

### Phase 2 — New "Edit Page Details" / "Edit Post Details" sub-pages

- Split `HasPageBuilderForm` trait into three methods: `metadataFormSchema()` (metadata only), `pageBuilderSection()` (builder only), and `pageBuilderFormSchema()` (combined, kept for backward compat with `EventResource`).
- Created `EditPageDetails.php` at `app/Filament/Resources/PageResource/Pages/` — extends `ReadOnlyAwareEditRecord`, renders the full metadata form (title, slug, status, template, page type, images, custom fields, SEO).
- Created `EditPostDetails.php` at `app/Filament/Resources/PostResource/Pages/` — same pattern for posts.
- Registered `/{record}/details` routes in both `PageResource` and `PostResource`.
- Updated `PageResource::form()` and `PostResource::form()` to use `metadataFormSchema()` (metadata only — no page builder section).
- Header actions on details pages: "Edit Page/Post" button (returns to builder), ellipsis menu with export, save-as-template, snippets (pages only), and delete.
- Breadcrumbs: `Pages > Edit Page > Page Details` / `Blog Posts > Edit Post > Post Details`.

### Phase 3 — Restructured EditPage/EditPost as builder-primary views

- Rewrote `EditPage.php` — extends `EditRecord` with a custom Blade view (`edit-page.blade.php`) that renders only the Livewire `page-builder` component. Empty form schema, no form actions.
- Created `resources/views/filament/resources/page-resource/pages/edit-page.blade.php`.
- Same pattern for `EditPost.php` with its own Blade view.
- Header action: "Edit Page/Post Details" button (cog icon) linking to the details sub-page.

### Phase 4 — Builder header redesign

- Added 6 new fields to `PageBuilder::getBootstrapData()`: `page_title`, `page_author`, `page_status`, `page_url`, `page_tags`, `details_url`.
- Updated `BootstrapData` interface in `types.ts` and Pinia store (`editor.ts`) to expose these fields.
- Redesigned `EditorToolbar.vue`: Row 1 = title (bold) + author + status (italic). Row 2 = public URL (linked, dimmed if draft) + tag badges.
- Redesigned `PreviewCanvas.vue`: removed `+ Add Block` dropdown from viewport bar. Added `+ Widget` and `+ Columns` dropdown buttons below the preview canvas with block count right-aligned.
- Removed `ApplyChangesButton.vue` entirely. Moved "Save as Template" to `App.vue` footer (bottom-right of the builder view, below the grid layout).
- Removed `margin-top` from `.vue-editor` since builder is now the full page content.

### Phase 5 — Cleanup

- Deleted `ApplyChangesButton.vue` (unused).
- Updated two existing tests in `TemplateManagerTest.php` that referenced actions on the old `EditPage` — now point to `EditPageDetails`.
- Created help docs: `resources/docs/page-details.md` (routed to details page) and `resources/docs/post-details.md`. Updated `cms-pages.md` and `posts.md` to focus on the builder view with cross-references to the new details docs.
- CSS tweaks: `editor-toolbar` margin-bottom:0, `preview-canvas__container` padding-top:1.5rem, `inspector-panel` margin-top:2rem.
