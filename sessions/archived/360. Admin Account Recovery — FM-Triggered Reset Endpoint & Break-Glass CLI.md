## The agreed design (from the 359 close conversation)

The shape settled with the owner — bring it to the Open Gate to confirm, then build:

- **FM-control-panel-triggered, not CLI-only.** The owner rejects the "CLI is more secure" framing: SSH-only just relocates trust to whoever holds the key and makes the riskiest operation (resetting an admin's auth on prod) a hand-typed `tinker` with no confirmation and no log. The reset logic lives in the CRM as a small **service**, exposed via a **gated, mTLS, audit-logged node endpoint** FM calls — and the same core stays runnable as a **local artisan break-glass** if a node is ever cut off from FM.
- **Identity verification is out-of-band and operator-mediated — no secret in the app or FM.** A per-admin recovery **PIN lives in the operator's own external vault (Bitwarden-style)**, operator-issued/pre-shared at onboarding ("telephone password" model). When a locked-out admin contacts the operator, the operator checks the PIN against their vault and *then* triggers the reset. The app never stores, hashes, or verifies a PIN; the endpoint rides the existing **"trust the connection"** mTLS model — if the request arrived through the FM mTLS gate, the operator has already done the human verification.
- **Conscious tradeoff (accepted):** vault-stored PIN makes the identity check a *procedure the operator follows*, not a *control the system enforces/attests*. The audit log can record "operator reset 2FA for user Y," not "…after PIN verification." Fine for the current solo-operator, person-to-person trust model; a hashed in-app PIN the node verifies is the heavier path for the day untrusted support staff exist — explicitly **not** this session.
- **Layers below the existing recovery codes.** 359 ships Fortify's 8 single-use recovery codes (self-serve: lost device, kept codes → admin recovers themselves, no operator). This operator-triggered reset is the *last* resort, for lost device **and** codes.

## Open questions to resolve at session start

These propagate — settle them at the Open Gate, don't commit-and-unwind.

1. **Reset scope (the load-bearing one).** Two readings of "recovery":
   - **(a) 2FA-only** — the endpoint/command clears `two_factor_secret` / `two_factor_recovery_codes` / `two_factor_confirmed_at`, so the admin re-enrolls on next login (password still required and assumed known).
   - **(b) Full credential recovery** — also reset/set a temporary password and/or create-or-promote a `super_admin`, closing the **original** s304 gap (which is about a lost *password*, not just 2FA). The two share the exact same seam (the `User` record, the mTLS endpoint, the audit log, the break-glass command), so doing both now closes s304 in one coordinated session rather than leaving a second follow-up.
   - **Recommended lean: (b), shaped as an explicit action set** on one endpoint (`reset_2fa`, `reset_password` — composable), so FM can ask for exactly what's needed and the seam is built once. Confirm with the owner.
2. **Endpoint + envelope shape.** Mirror `/api/backup/trigger`: `POST /api/admin/recover` (or `/api/admin/reset-2fa` if scope (a)), identifying the target by `email`, returning **`200` always with a `status` envelope** (success/failed in the body, never relying on HTTP status alone), with the same error-sanitisation discipline. Confirm the path + payload.
3. **Auth model.** Confirm the new endpoint **rides the existing nginx mTLS gate** ("trust the connection," no app-layer auth) and joins the per-location strict-gate list in `docker/nginx/{prod,default}.conf`. This is the established pattern — just ratify it.
4. **Protected-admin handling.** `User::isProtected()` (the oldest super-admin) is hidden from the UI delete-guard — but it is the *likeliest* lockout victim, so the recovery path **must be able to reset it**. Reset ≠ delete; confirm the endpoint/command deliberately bypasses the delete-guard for the reset operation.
5. **Audit sink.** `spatie/laravel-activitylog` is already a dependency (it logs CRM-record activity) — decide whether the reset audit rides it, a dedicated table, or a structured log channel, and whether the audit summary returns to FM in the response envelope.

## Phases

### Phase 1 — Reset service + break-glass CLI
A reusable service that performs the agreed reset action(s) on a target `User` (clear 2FA state; per scope decision, set a temp password / promote super-admin), able to operate on the protected admin. A local artisan command wrapping it (`admin:recover {email} …` or similar), **hard-gated to refuse in demo mode** (mirror `demo:reset`/`demo:restore`). Audit every invocation.

### Phase 2 — mTLS node endpoint
A new Fleet API controller (`app/Http/Controllers/Api/Fleet/…`) calling the same service, registered in `routes/api.php` with its own throttle bucket, returning the `/api/backup/trigger`-style `status` envelope. Add its `location` block to the nginx per-location mTLS gates (`prod.conf` + `default.conf`). No app-layer auth — trust the connection.

## Out of scope

- **The Fleet Manager control-panel UI** and the FM-side call to this endpoint — separate FM-repo work, coordinated via the contract. 360 delivers the node endpoint the FM UI consumes.
- **Any in-app PIN storage / verification.** The recovery PIN lives in the operator's external vault; the app holds no recovery secret. (The heavier hashed-in-app-PIN model is explicitly deferred.)

- **The post-Beta Security Audit Ready milestone work** — this is the table-stakes operational backstop for 359's 2FA, not the broader hardening pass.

## Drift signals (stop and surface)

- The mutating admin endpoint can't cleanly ride the mTLS "trust the connection" model (e.g., something demands per-operator identity in the app) — surface before bending the established auth pattern.
- Resetting the protected super-admin collides with the `isProtected()` guard in a way that's load-bearing elsewhere (not just the UI delete action) — surface.
- The chosen reset scope (b) drags in password-policy / notification surface bigger than a single session — surface; fall back to (a) + a documented follow-up.
- `activitylog`'s table/shape doesn't fit an operator-action audit (it's record-centric) — surface before forcing it; a dedicated audit row may be cleaner.
