# Session 157 — Page Builder Preview Reactivity Bug Fix — Log

**Date:** 2026-04-10
**Branch:** session-157/1

---

## What was built

### Phase 1 — Diagnosis

- Reproduced the reported symptom: clicking **Apply Changes** in the Vue page builder inspector did nothing; the preview only updated after a full page reload.
- Traced the bug to [resources/js/page-builder-vue/stores/editor.ts](resources/js/page-builder-vue/stores/editor.ts) `refreshPreview()`. The action was replacing the widget entry with a brand-new spread object: `widgets.value[id] = { ...widgets.value[id], preview_html: res.html }`. After the replacement, `widgets.value[id]` is a fresh reactive proxy, but the **old** proxy is still referenced from `pageItems`, which is what `<draggable :list="store.pageItems">` in [PreviewCanvas.vue](resources/js/page-builder-vue/components/PreviewCanvas.vue) iterates and passes to `PreviewRegion`. Result: the v-html binding in [PreviewRegion.vue:47](resources/js/page-builder-vue/components/PreviewRegion.vue#L47) was bound to the stale proxy, so the new HTML never reached the DOM.
- Confirmed regression introduction in **Session 153 (Column Layout System)**, which added `pageItems` as an independent ref and switched the draggable from `store.rootWidgets` (a `computed` derived from `widgets.value` on every read) to `store.pageItems` (populated once in `populateFromItems`). Before that change, the spread-replace pattern had been silently working because `rootWidgets` always re-read the live map.
- Audited the rest of `editor.ts` for the same anti-pattern. Found three more instances that were either latent or working around the bug manually.

### Phase 2 — Fix

In [resources/js/page-builder-vue/stores/editor.ts](resources/js/page-builder-vue/stores/editor.ts):

- `refreshPreview` now mutates in place: `widgets.value[id].preview_html = res.html`. This is the user-visible fix.
- `updateWidget` now uses `Object.assign(widgets.value[id], updated)` instead of spread-replace. Latent bug — the symptom was previously masked because `updateLocalConfig` had already mutated the proxy locally before the server round-trip, but the same proxy-divergence would have surfaced under different code paths.
- `updateLayout` now uses `Object.assign(layouts.value[id], updated)` and the manual `pageItems[idx]` re-sync block was deleted. Same fix shape — the manual sync was a workaround for the same anti-pattern, made unnecessary by mutating in place (because Vue's `reactive()` cache means `layouts.value[id]` and the entry in `pageItems` are the **same** proxy when the underlying target is shared from `populateFromItems`).
- `updateLocalLayout` had the same workaround block (manual `pageItems[idx]` re-sync after field-by-field local mutations). Removed; the in-place mutations on `l = layouts.value[layoutId]` now propagate automatically.

Other spread patterns in the store (`updateLocalConfig`, `uploadImage`, `removeImage`, `updateLocalStyleConfig`, `updateLocalQueryConfig`) were verified safe — they replace **sub-properties** of the proxy (`w.config = { ... }`), not the proxy itself, so reactivity tracks the property mutation correctly. Their safety depends on `widgets.value[id]` never being replaced — fixing the four bugs above preserves that invariant.

### Phase 3 — Outline update for follow-up testing
