# 165. Appearance Controls — Text & Section Layout Panels

Phase 3b of the Appearance Controls project. Builds the remaining two universal Appearance panels (Text and Section Layout) in the Vue inspector, retires the temporary `WidgetAppearanceControls.vue` and `SpacingControl.vue` components, and lands regression smoke tests for every widget that lost or renamed config fields in session 162.

This session is the second of a two-iteration split of the original 164 outline. Session 164 delivered the Background panel, the `AppearanceStyleComposer` extraction, and the gradient alpha extension. This session completes the universal Appearance layer for Beta 1.

## Design decisions (resolved before starting)

### Storage shape — `appearance_config.text` and `appearance_config.layout`

Both keys already exist in the schema doc shape. Session 164 wired the `text.color` read in the renderer via `AppearanceStyleComposer`. Session 162 wrote the initial `layout.full_width`, `layout.padding.{top,right,bottom,left}`, and `layout.margin.{top,right,bottom,left}` keys via the temporary `SpacingControl.vue`. This session adds **no new keys** — it builds the proper UI surfaces that write the existing keys, and retires the temporary surfaces that wrote them before.

```
text: {
  color: '#hex'
}

layout: {
  full_width: bool,
  padding: { top, right, bottom, left },  // integer pixels
  margin:  { top, right, bottom, left }   // integer pixels
}
```

### Text panel UI — `TextPanel.vue`

New file at `resources/js/page-builder-vue/components/appearance/TextPanel.vue`. Single section, single control:

- `ColorPicker` bound to `appearance_config.text.color`. Use the existing primitive's `icon` slot to pass a "T" glyph (the primitive already supports a named slot for the trigger icon — confirmed in session 164's audit at [ColorPicker.vue:111](resources/js/page-builder-vue/components/primitives/ColorPicker.vue#L111)). The "T" glyph should be a simple inline SVG, not an icon-library import.
- A small helper-text line under the control: "Inline color styling inside rich text fields overrides this value."

The panel header reads "Text", with a subtle divider above it. It mounts as the second child of the Appearance tab, directly after `BackgroundPanel` (which session 164 placed at the top).

The `TextPanel` does **not** add any new appearance keys. The `text.color` key was already wired into the renderer in session 164's `AppearanceStyleComposer`; this session only adds the inspector surface that writes it.

### Section Layout panel UI — `SectionLayoutPanel.vue`

New file at `resources/js/page-builder-vue/components/appearance/SectionLayoutPanel.vue`. Three sections:

**Width**

- A toggle bound to `appearance_config.layout.full_width`. When the widget has `layout_id !== null` (i.e., the widget lives inside a column slot), the toggle is **disabled** (grayed out + tooltip "The parent column controls width for column widgets"). The Background panel's full-width toggle and this Section Layout panel's full-width toggle bind to the same store path — the duplicate UI affordance is intentional for discoverability. Both sites use `store.updateLocalAppearanceConfig(widget.id, 'layout.full_width', value)`.

**Padding**

- An "All sides" number input that, when changed, writes the same integer to all four sides via four `updateLocalAppearanceConfig` calls.
- Four individual number inputs for `top`, `right`, `bottom`, `left`.
- The all-sides input shows `mixed` as a placeholder when the four sides are not equal. When all four are equal and set, the all-sides input shows that value.
- All values are integer pixel counts. Empty strings are valid (means "no value, no inline style emitted") and the renderer continues to skip empty values.

**Margin**

- Same shape as Padding — an all-sides input plus four per-side inputs.

The all-sides shorthand logic exists today in `SpacingControl.vue` (via `paddingAll`/`marginAll` computeds and `setPaddingAll`/`setMarginAll` setters at [SpacingControl.vue:16-68](resources/js/page-builder-vue/components/SpacingControl.vue#L16-L68)). **Move that logic into `SectionLayoutPanel.vue` verbatim** — same computeds, same setters, same field labels. If the same shorthand pattern ends up duplicated between the padding and margin sub-sections inside the new panel, factor it into a small inline helper inside the same file. Do not extract a shared composable for it — three call sites do not justify the abstraction.

The panel header reads "Section Layout", with a subtle divider above it. It mounts as the third child of the Appearance tab, after `TextPanel`.

### Retire `WidgetAppearanceControls.vue`

The file at [resources/js/page-builder-vue/components/WidgetAppearanceControls.vue](resources/js/page-builder-vue/components/WidgetAppearanceControls.vue) was a session-162 temporary panel housing a full-width toggle, a background color picker, and a text color picker. Once `BackgroundPanel`, `TextPanel`, and `SectionLayoutPanel` are mounted in session 164/165, every control on `WidgetAppearanceControls` is duplicated by the new panels in nicer form.

**Delete the file.** Remove the import from [InspectorPanel.vue](resources/js/page-builder-vue/components/InspectorPanel.vue) and remove the `<WidgetAppearanceControls :widget="widget" />` element from the appearance tab template. No backwards-compat shim, no `// removed` comment.

The `schemaKeys`/`showFullWidth`/`showBgColor`/`showTextColor` conditional logic that the temporary panel used to suppress its controls when a widget had its own config field for those keys is **not** carried forward. After the session 162 dedupe sweep, no widget has its own `background_color`, `text_color`, or `full_width` config field — every widget routes through the universal Appearance layer for those values. The conditional is dead code and should not be reproduced in the new panels.

### Retire `SpacingControl.vue`

The file at [resources/js/page-builder-vue/components/SpacingControl.vue](resources/js/page-builder-vue/components/SpacingControl.vue) was a session-153/162 temporary spacing input that lived in the appearance tab. `SectionLayoutPanel`'s Padding and Margin sections supersede it.

**Delete the file.** Remove the import from `InspectorPanel.vue` and remove the `<SpacingControl :widget="widget" />` element from the appearance tab template. No shim.

### Inspector tab structure after retirements

After both retirements and the three new panel mounts, the appearance tab's body is:

```vue
<div v-show="activeTab === 'appearance'" class="inspector-panel__tab-content">
  <BackgroundPanel :widget="widget" />
  <TextPanel :widget="widget" />
  <SectionLayoutPanel :widget="widget" />
  <InspectorFieldGroup :fields="appearanceFields" :widget="widget" />
  <QuerySettings :widget="widget" />
</div>
```

`InspectorFieldGroup` continues to render any per-widget appearance-group fields that survive the universal layer (e.g., `hero.nav_link_color` and `hero.nav_hover_color`, which were preserved in session 162 because they relate to chrome interaction in full-bleed mode, not background or text styling). `QuerySettings` is unchanged.

### No new appearance keys

This session adds **zero** new keys to `appearance_config`. The schema-doc shape and the `AppearanceStyleComposer` are unchanged. Every change is UI plumbing or test coverage.

## Phase 3 — Regression smoke tests

For each of the eight widgets in the regression list above, write or extend Pest tests under `tests/Feature/` to lock in:

1. **Surviving-keys-only smoke**: render the widget through `WidgetRenderer::render()` with a `config` containing only the surviving keys. 2. **Universal appearance layer integration**: create a `PageWidget` with the same surviving config, plus an `appearance_config` setting `background.color` and `text.color` to specific hex values. Render the widget through the public renderer (or the page-widgets blade component) and assert the outer widget wrapper's inline `style` attribute contains `background-color: #...` and `color: #...` matching the values set.

Group the tests by widget.
