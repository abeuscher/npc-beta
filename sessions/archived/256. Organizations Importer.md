# 256. Organizations Importer

Add a top-level CSV importer for Organizations so real-shape Wild Apricot / Neon exports can land Org records with full metadata (type / website / email / phone / address / custom fields / tags / notes) before B2 runs the onboarding rehearsal against the same fixture. Without this stub, B2's "Migration in" sub-scenario has no clean admission path — the Org-as-source sentinels added at B1a only auto-create Org stubs by name.

This session is the **B1c stub** in the release plan, slotted between B1a (closed at 255) and B2 (next).

---

## Design decisions (resolved before starting)

- **Importer shape — namespaced single-bucket.** Mirror `ImportMembershipsPage` / `ImportMembershipsProgressPage` (uses `InteractsWithImportWizard` + `InteractsWithImportProgress` traits). The `OrganizationImportFieldRegistry` aggregates exactly one bucket — `organization` — since Orgs are top-level records. **No contact-match bucket.** The contacts importer's reverse case (Contact creates Org stub by name) is the established direction; this importer does not need to find or create contacts.

- **Dedup match-key — name (default), email (alternate), external_id (alternate).** Match-key dropdown on the Map Columns step lets the operator pick which column drives uniqueness. Default is `organization:name` because that's the field every CSV will have. The match logic uses `LOWER(TRIM(value))` parity with the Org-as-source sentinels, so an Org auto-created by `__org_contact__` at "ACME" will be recognised as the same row when the Org importer runs against "acme" or "  ACME  ".

- **Duplicate strategy — `skip` / `update` / `error`** on the existing `duplicateStrategyRadio` (no-default radio per session 254 UX). `update` follows the fill-blanks-only semantic that the invoice-details importer uses for Transactions: the importer never overwrites a non-blank existing value, only enriches blanks. This means an Org stub auto-created by an earlier `__org_contact__` import (just `name`) gets enriched with type / website / email / address by a follow-up Org importer run — without disturbing manual edits.

- **Schema additions — five new columns on `organizations`.** The session prompt's earlier scoping called this out:
  - `source` (string, NOT NULL, default `'human'`, `Organization::ACCEPTED_SOURCES = [HUMAN, IMPORT, SCRUB_DATA]`).
  - `custom_fields` (jsonb, nullable).
  - `import_source_id` (uuid, nullable, FK→`import_sources`, nullOnDelete).
  - `import_session_id` (uuid, nullable, FK→`import_sessions`, nullOnDelete).
  - `external_id` (string, nullable).
  - Indexes: `(source)` and `(import_source_id, external_id)` for fast upsert lookup.

  Brings Organization into the same source-policy / import-trace discipline as the rest of the peer-record family. Necessary for: (a) re-runs of the same Org CSV deduping against the prior import-source's `external_id`, (b) source-scoped wipe of scrub-source Orgs working correctly, (c) `EnforcesScrubInheritance` flowing scrub source from a Contact's organization auto-create.

- **`Organization::ACCEPTED_SOURCES`** — `[HUMAN, IMPORT, SCRUB_DATA]`. No `STRIPE_WEBHOOK` (Org records aren't created by payment events). The Stripe webhook path may set `donations.organization_id` later, but the Org row itself is HUMAN / IMPORT / SCRUB_DATA only.

- **`Organization::scrubInheritsFrom`** — declared as **empty array** (or trait method returns `[]`). Organization sits at the **top** of the source-policy graph for the Org sub-tree. There is no parent record from which Org inherits scrub. (Donations / Memberships inherit from `contact_id`. Org has no analogous parent.) The Random Data Generator widget that auto-creates scrub Orgs sets `source = SCRUB_DATA` directly on the Org row.

- **Three new sentinels — `__custom_organization__`, `__tag_organization__`, `__note_organization__`.** Each follows the established parallel-edit discipline:
  - **`__custom_organization__`** — column gets demoted to a custom-field entry on `organizations.custom_fields`. UX: same custom-field sub-form as the other importers (label / handle / field type, with rich_text supported per session 250).
  - **`__tag_organization__`** — column value (or delimited list) creates Tag rows with `type = 'organization'` and morphs them onto the Org. UX: same `tag_delimiter_{$n}` sub-form as `__tag_contact__` / `__tag_event__`. Add `'organization'` to the existing canonical Tag types.
  - **`__note_organization__`** — column value creates a Note record with `notable_type = Organization::class`. UX: same `note_split_{$n}` sub-form as `__note_contact__`, with `none` / `date_prefix` / `regex` split options. Note records flow through the timeline page lifted at session 255.

- **Match-key entries on the registry.** `OrganizationFieldRegistry::CONTACT_MATCH_KEYS` is irrelevant (no contact match). The match-key dropdown for *this importer* lists `organization:name`, `organization:email`, and `organization:external_id` — the three viable dedup columns.

- **`ImportSource` saved-mapping triplet.** Add three columns to `import_sources` (mirroring the existing per-importer triplets):
  - `organizations_field_map` (jsonb)
  - `organizations_custom_field_map` (jsonb)
  - `organizations_match_key` (string)

- **`ImportModelType::Organization`** — add the enum value if missing. Used by `ImportSession::model_type` and `ImportLog::model_type`.

- **Admin nav.** Register the importer card on `ImporterPage` (the Tools group index). The card carries the same shape as the other importer cards.

- **Coexistence with Org-as-source sentinels.** Documented at session 255 close — both orderings work:
  - Org importer first → Orgs land with full metadata. Contact / Donation / Membership / Event / Invoice imports then *match* by name (`__org_*__` strategy = `auto_create` still works; the match path is hit, no new orgs created).
  - Member / Donation / etc. importer first → Orgs auto-created by name only via sentinels. Org importer follow-up enriches via `update` strategy.

  No code change needed for coexistence; the existing sentinel logic in `InteractsWithImportProgress::resolveOrganizationByName` already does the case-insensitive trim match.

- **Out of scope this session.** Notes-importer Org sentinel (the standalone Notes importer at `ImportNotesPage` — that page is Contact-only by design; Org notes are imported via the new `__note_organization__` sentinel inside the Org importer itself). Field-level permissions on the importer (covered at C1 / C3). Bulk-edit of imported orgs (out of B1c scope; standard Filament resource handles this).

## Phase 4 — Importer pages

Two new files mirroring the Memberships shape:

- `app/Filament/Pages/ImportOrganizationsPage.php` — uses `InteractsWithImportWizard`. Wizard with Upload → Review → Map Columns → Preview & Confirm steps. Match-key dropdown defaults to `organization:name`.
- `app/Filament/Pages/ImportOrganizationsProgressPage.php` — uses `InteractsWithImportProgress`. `processOneRow()` builds Org attrs from the namespace map, resolves an existing Org by the chosen match key (case-insensitive `LOWER(TRIM(...))` for name/email; exact for external_id), then either creates a new Org (with `source = Source::IMPORT`) or applies the duplicate strategy.

Tag and note handling reuses the trait's existing `applyPerRowTags` / `applyPerRowNotes` patterns, polymorphic-keyed against the new Org row.

## Phase 7 — Tests

- **Migration shape tests** — assert the five new columns + indexes on `organizations`, the three new columns on `import_sources`. Standard Pest pattern (mirror `AddOrganizationIdToTransactionalTablesMigrationTest`).
- **Round-trip tests** — for each major path:
  - Simple CSV with name + type + website + email → Org created with all fields, `source = 'import'`.
  - CSV with `__custom_organization__` mapping → custom field lands in `organizations.custom_fields` jsonb.
  - CSV with `__tag_organization__` mapping → Tag row(s) created with `type = 'organization'`, morphed onto the Org.
  - CSV with `__note_organization__` mapping → Note record created with `notable_type = Organization::class`.
- **Dedup-strategy tests** — three cases per match-key (name / email / external_id):
  - `skip` → second run with same key is a no-op, no row created or updated.
  - `update` → second run with new non-blank values for blank columns enriches the existing row; existing non-blank values are not overwritten.
  - `error` → second run raises an error; no row created.
- **Coexistence test** — Contact import auto-creates Org stub via `__org_contact__` (name only), Org importer follow-up `update` strategy enriches the same row with full metadata. Confirm same `id`, no duplicate row.
- **`EnforcesScrubInheritance` test** — Org row created by importer carries `source = Source::IMPORT`. Org row created by Random Data Generator (or directly via `Organization::create(['source' => SCRUB_DATA])`) carries `source = SCRUB_DATA`.

## Out of scope

- **Contact-match in the Organizations importer** — Orgs are top-level; no "match a contact for each org row" path. Out of scope (and out of any future scope — this is structural).
- **Org-related-records seeded by the importer** — the importer creates Organization rows, plus optional tags / notes / custom-fields directly. It does not create Donations, Memberships, EventRegistrations, or Transactions on behalf of the Org. Those records flow through their own importers.
- **Bulk-edit of imported orgs** — standard Filament resource handles this; no special importer-side affordance.
- **Field-level permission gates on the importer** — covered at C1 (Notes Permissions) / C3 (Permission audit).
- **`__note_contact__` sentinel inside the Org importer** — Notes inside the Org importer are Org-typed by definition; if a CSV has a column that's actually about a contact, that's an export/source-data mapping decision, not an importer feature.
- **Updating the standalone Notes importer to support Organizations as notable** — that page (`ImportNotesPage`) stays Contact-only. Org notes flow through the `__note_organization__` sentinel inside the Org importer itself. Out of scope unless B2 surfaces a real-world fixture that demands a standalone "import these notes onto these orgs" path.
- **Migrating existing Orgs** — there's no production data; the new columns are nullable / defaulted, so existing Org rows survive the migration without a data step. No backfill needed.
