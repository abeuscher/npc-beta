# 221. Widget Primitive — Phase 4 Follower (Query Settings in the Contract Layer)

Route the page-builder's QuerySettings panel through the contract layer instead of the legacy slot-keyed `WidgetDataResolver` path. After this session, every list-shaped widget — `SOURCE_SYSTEM_MODEL` and `SOURCE_WIDGET_CONTENT_TYPE` — exposes the same five filter knobs (`limit`, `order_by`, `direction`, `include_tags`, `exclude_tags`) uniformly via the contract resolver, the `WidgetRenderer::render()` line 46 legacy loop and `WidgetDataResolver::resolveCustom` are deleted, the slot-strip mechanism in `PageBuilderApiController::filterQueryConfig` is deleted, the QuerySettings panel's chevron disclosure is removed, and the misnamed "Carousel Settings" chevron heading on `advanced: true` config fields is replaced with a flat inline rendering.

---

## Design decisions (resolved before starting)

- **Five knobs uniformly across every list-shaped widget.** No per-widget exception list. The five knobs are `limit`, `order_by`, `direction`, `include_tags`, `exclude_tags`. List-shaped contracts are `SOURCE_SYSTEM_MODEL` (BlogListing, EventsListing, ThisWeeksEvents) and `SOURCE_WIDGET_CONTENT_TYPE` (Carousel, BarChart, BoardMembers, LogoGarden, Memos). `SOURCE_PAGE_CONTEXT` widgets (TextBlock, QuickActions) do not get the panel — they aren't queries. Single-row widgets that aren't yet retrofitted (EventDescription, ProductDisplay, EventRegistration) remain out-of-scope until their retrofit lands.
- **`order_by` field options come from the source.** SWCT widgets surface user-mapped Collection fields (the contract's content type fields, excluding image fields) plus three system columns: `sort_order`, `created_at`, `updated_at`. SYSTEM_MODEL/post surfaces `published_at`, `created_at`, `updated_at`, `title`. SYSTEM_MODEL/event surfaces `starts_at`, `ends_at`, `created_at`, `title`. **No model UUIDs (`id`) and no internal flags (`is_published`, `is_active`, `is_public`) ever appear in the dropdown.** The lists are static per source/model — no auto-discovery against the column schema.
- **`order_by` validation is double-gated.** The Vue panel only shows allowed options (UX gate). The resolver re-validates the user-supplied value against the same allowlist (security gate). An unknown value falls back to the source's default order without erroring.
- **Tag relations are uniform across every taggable model.** Page, Event, CollectionItem, Organization, and Contact all use `morphToMany(Tag::class, 'taggable')` named `tags()`. One implementation works for every source-arm: `whereHas('tags', fn ($q) => $q->whereIn('slug', $tagSlugs))` for `include_tags`, `whereDoesntHave('tags', ...)` for `exclude_tags`. The `cmsTags` reference at [WidgetDataResolver.php:70](app/Services/WidgetDataResolver.php#L70) and [:74](app/Services/WidgetDataResolver.php#L74) is stale — that relation was deleted in session 040 — and goes away when `resolveCustom` is deleted.
- **Precedence between widget defaults and user-supplied knobs.** For knobs in the contract's declared honored set, user-supplied values win. For knobs NOT in the honored set, the contract's defaults are immutable (e.g. EventsListing's `date_range: from now` — date_range is not in the universal knob set, stays as a hard-coded default in the contract; the user cannot override it via `query_config`). The contract's `filters` array carries widget defaults; user `query_config` merges on top of it for honored keys only.
- **Honored-knob declaration on the contract.** `DataContract` gains a small `querySettings(): ?QuerySettings` accessor or equivalent — exact API shape a Phase 2 decision (recommend a new `QuerySettings` value object on `App\WidgetPrimitive` carrying `hasPanel: bool` and `orderByOptions: array<string, string>` — the second is a `value => label` map). Contracts with no panel return null; contracts with a panel return the per-source-arm declaration. The widget-type API serializes this into `query_settings` on the per-widget JSON; the Vue store reads it.
- **`query_config` keying flips from slot-keyed to flat.** Today: `query_config[$slot][$key]` (e.g. `{ slides: { limit: 4 } }`). Tomorrow: `query_config[$key]` (e.g. `{ limit: 4 }`). One-time data migration flattens existing rows: take the first non-empty inner object per row and lift its keys to the top level; if multiple inner objects, log a warning and pick the union (later keys win). Empty rows stay empty. Both dev DB and the deploy server need the migration.
- **Carousel-Settings chevron cleanup is minimum scope.** The misnamed `"Carousel Settings"` heading at [InspectorFieldGroup.vue:99](resources/js/page-builder-vue/components/InspectorFieldGroup.vue#L99) is **not** a system-derived "carousel-shape widget" panel — it is a generic "advanced fields" accordion fed by the `advanced: true` flag on individual config-schema fields. This session removes the chevron disclosure and renders advanced fields inline. The heading is renamed to `"Advanced"` (or removed entirely if visual hierarchy is preserved by an `<hr>` rule — call it during implementation). Building a real "carousel-shape" abstraction is out of scope.
- **No-collapse extends to QuerySettings.** Both inspector chevron toggles (`QuerySettings.vue` and `InspectorFieldGroup.vue`) are removed in the same session. Per `feedback_no_collapse_ui`.
- **Vue panel render gate.** The QuerySettings panel renders if and only if `widget.query_settings?.has_panel === true`. The legacy `widget.widget_type_collections.length > 0` gate is removed. The TypeScript type `Widget.widget_type_collections` is deleted (vestigial).
- **Deletions.** This session deletes:
  - `WidgetDataResolver::resolveCustom` (private static method) and any of its private helpers (`getCollectionFieldKeys`, `getCollectionImageFieldKeys`) that have no remaining consumers after the deletion.
  - The legacy `foreach ($widgetType->collections ?? [])` loop in `WidgetRenderer::render()` (lines 44–51) and the `$collectionData` template variable construction. `$collectionData` is no longer passed into the Blade template.
  - `PageBuilderApiController::filterQueryConfig` and its callsite at line 128.
  - The `<button class="query-settings__toggle">` chevron block in `QuerySettings.vue` (lines 39–58) and the `open` ref.
  - The `inspector-field-group__accordion` chevron in `InspectorFieldGroup.vue` (lines 82–110), replaced with inline rendering of `advancedFields` under an `"Advanced"` heading or `<hr>`.
  - The `widget_type_collections` field on `Widget` in `types.ts`. The API stops surfacing it; any remaining template references are cleaned up.
- **Frontend build expected.** `npm run build` runs at end of implementation. No `build:public` expected (no widget asset files touched).
- **`resolveProducts` stays.** `WidgetDataResolver::resolveProducts` is still called directly by `ProductCarousel/template.blade.php`. That widget hasn't been retrofitted yet — `resolveProducts` retires when ProductCarousel lands its retrofit (next session or a near-future session). Out of scope here.

## Phase 2 — Contract-layer surface

1. Add a `App\WidgetPrimitive\QuerySettings` value object (new file, ~20 lines): `readonly bool $hasPanel; readonly array $orderByOptions; readonly bool $supportsTags;`. The third flag exists because while the universal knob set includes tag filters, the design allows future widgets to opt out (a contract can declare `supportsTags: false` if the model isn't taggable — preserves the per-source flexibility cheaply). For this session every list-shaped contract sets `supportsTags: true` (every model is taggable).
2. Add `querySettings(): ?QuerySettings` to `App\Widgets\Contracts\WidgetDefinition` returning `null` by default. List-shaped widget definitions override.
3. For each list-shaped widget, override `querySettings()` to return a `QuerySettings` whose `orderByOptions` are the source-appropriate list:
   - **SWCT widgets** (Carousel, BarChart, BoardMembers, LogoGarden, Memos): build dynamically from the contract's content-type fields (text-typed only, no image fields) plus `['sort_order' => 'Sort order', 'created_at' => 'Created', 'updated_at' => 'Updated']`. Per-instance dynamic — same shape as `dataContract()`'s field union.
   - **BlogListing**: static `['published_at' => 'Published date', 'created_at' => 'Created', 'updated_at' => 'Updated', 'title' => 'Title']`.
   - **EventsListing, ThisWeeksEvents**: static `['starts_at' => 'Start date', 'ends_at' => 'End date', 'created_at' => 'Created', 'title' => 'Title']`.
4. Add `App\WidgetPrimitive\DataContract::orderByDefault(): string` returning a sane source-specific default (`'sort_order'` for SWCT, `'published_at'` for post, `'starts_at'` for event). Read by the resolver when `query_config` doesn't supply one.
5. Extend `ContractResolver::resolveWidgetContentType()`:
   - Accept user `order_by` from `$contract->filters['order_by']` after merging. Validate against the contract's `querySettings()->orderByOptions` keys; fall back to default on miss.
   - Accept `direction` from filters; coerce to `asc`/`desc`.
   - Apply tag filters via `tags()` relation: `$query->whereHas('tags', fn ($q) => $q->whereIn('slug', $includeTags))` and `whereDoesntHave('tags', ...)` for excludes.
   - Preserve the per-handle batching cache key — but note that the cache key now must include the filter shape (or we serialize-hash the relevant filters into the key). Two BoardMembers widgets pointing at the same Collection but with different `order_by` cannot share a cached query.
6. Extend `ContractResolver::resolvePost()` and `resolveEvent()`:
   - Accept user `order_by` and `direction` from filters; validate against the per-source allowlist (which mirrors `querySettings()->orderByOptions`).
   - Apply tag filters via `tags()` on Page (post type) and Event.
   - Same cache-key adjustment.
7. **Critical**: do **not** allow user `query_config` to override `date_range` on Event contracts. The merge step in step 8 is a one-way merge for honored knobs only.
8. In `WidgetRenderer::render()`, before the `$contract` is passed to the resolver, merge user `query_config` into a copy of `$contract->filters`:
   ```php
   $userFilters = is_array($pw->query_config) ? $pw->query_config : [];
   $honoredKeys = ['limit', 'order_by', 'direction', 'include_tags', 'exclude_tags'];
   $userFilters = array_intersect_key($userFilters, array_flip($honoredKeys));
   $mergedFilters = array_merge($contract->filters, $userFilters);
   $contract = new DataContract(/* same args, filters: $mergedFilters */);
   ```
   This is the precedence rule: honored knobs allow user override; non-honored knobs (date_range) are immutable contract defaults.

## Out of scope

- **The 5 remaining widget retrofits** (BlogPager, EventDescription, EventRegistration, ProductDisplay, ProductCarousel). The Phase 4 retrofit campaign continues in 222+.
- **Building a real "carousel-shape" abstraction.** The user observed during scoping that QuerySettings and a hypothetical CarouselSettings panel could be parallel system-derived panels. There is no abstract "carousel-shape widget" concept in the codebase today (Carousel is one widget; LogoGarden has carousel-as-one-of-four-modes; ProductCarousel is unrelated infrastructure). Designing this is bigger than this session and shouldn't ride alongside the query-settings architecture work. The minimum scope (chevron removal, heading rename) is what this session does.
- **`date_range` as a user-configurable knob.** EventsListing and ThisWeeksEvents lock `date_range` to identity-defining defaults (`from now`, this week). Exposing it to admins requires UX design (date pickers, named presets) and isn't in the universal five-knob set agreed during scoping. Future session.
- **System-model `published_status` filter.** Public surfaces always filter to `published`; not user-configurable.
- **Per-tenant or per-page custom orderable-field configuration.** The allowlists are static per source/model.
- **`WidgetDataResolver::resolveProducts` retirement.** Still called by `ProductCarousel/template.blade.php`. Retires when ProductCarousel is retrofitted.
- **Per-field demo-data hints.** The "widget-owns-demo-data" track surfaced in 218 is a separate session.
