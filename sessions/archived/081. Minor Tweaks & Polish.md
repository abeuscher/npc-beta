# 081. Minor Tweaks & Polish

A collection of accumulated minor UI fixes, label corrections, and small feature additions, plus a set of design discussions to resolve at the end of the session.

---

## Design decisions (resolved before starting)

- "Donors" nav label and page title renamed to "Giving Summary" (agreed session 080).
- Portal access section in contact record should default to expanded, not collapsed.

---

## Phase 1 — UI Bug Fixes

Fix the following visual defects:

1. **Block editor delete button** — the confirm/delete button renders white on white. Change the button to the `outlined` variant so it is visible.
2. **Blank blue buttons in email templates** — locate the email button styles and fix the rendering bug that produces blank/invisible blue buttons.

---

## Phase 2 — Admin Navigation & Quick Links

1. **"View public site" link in admin header** — add a small button to the right of the company name/logo in the admin header that opens the public site in a new tab. Small and unobtrusive.
2. **"Giving Summary" rename** — rename the "Donors" label in the Finance navigation menu and on the Donors page/resource title to "Giving Summary".

---

## Phase 3 — Contact & Donations Table Enhancements

1. **Uncollapse portal access fields** — in the contact record edit form, change the portal access section's default state to expanded.
2. **View contact button on donations** — add a row-level "View contact" action to the donations table that navigates to the associated contact record.

---

## Phase 4 — User Management: Password Confirmation

In the admin edit user form:

1. Add a second "Confirm new password" field. If both password fields are filled but don't match, show a validation error before saving. If both are empty, leave the password unchanged (existing behaviour).
2. If a password generator approach is agreed during the discussion phase, implement it here.

---

## Phase 5 — API Key Reset Pattern: Resend & Mailchimp

The Stripe integration (session 073) implemented a high-friction key rotation flow with warnings. Apply the same pattern to the Resend and Mailchimp API key fields in their respective settings pages.

---

## Discussion items (resolve after all code phases pass tests)

Work through each of the following with the user:

**A. Edit Transaction View** — the view reportedly shows fields that may not exist and it is unclear whether an edit view should exist at all. Walk through what is currently shown, agree on whether to keep, remove, or simplify it, and plan any follow-up work.

**B. Mailing list field policy** — decide which contact fields may and may not be used as mailing list targeting criteria. Output: written decisions to stub into a future session.

**C. Content type deletability** — walk through every content type in the system. For each: is it deletable? Is there already a delete path? Do batch-delete controls exist where they shouldn't? Output: written decisions to stub into a future session.

**D. Secure password generator** — decide whether to include a generate-password button alongside the confirm field added in Phase 4. If yes, agree on approach (client-side JS, copy-to-clipboard). Implement if trivial; stub if not.

**E. API key pattern recognition & test key warning** — two proposed features: (1) form validation that recognises API key format patterns (e.g. Stripe `sk_test_` prefix); (2) a warning surfaced when test-mode keys are detected in a production context. Agree scope, decide where warnings surface, stub a session if it warrants one.

**F. Test/sandbox data system** — the user wants a mode where a single controllable test record can be acted on without touching real data. Discuss the concept creatively. Output: a concept description to stub as a future session.

**G. Default sample record for test email sends** — a user-editable singleton that pre-fills the system email preview wizard with representative sample data. Discuss scope, persistence model, and whether this is part of item F or stands alone.
