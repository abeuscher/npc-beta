# 230. Widget Primitive ‚Äî Phase 5c.5 (Admin UI for Authoring per-Record-Type View Sets)

Land the admin UI for adding / removing / reordering / configuring widgets inside `record_detail_views` rows. The Vue page-builder mounts in a new `record_detail` editor mode (mirror of the `dashboard` mode shipped in 215); a per-View API controller scoped to the View as polymorphic owner; a Filament resource for managing View metadata (record_type, handle, label, sort_order); a permission seeder. **No new schema** ‚Äî `record_detail_views` exists from 5c.

---

## Design decisions (resolved before starting)

- **Filament Resource, not a single Settings page.** Views are many per record_type (potentially many per install over time), so a flat `RecordDetailViewResource` with list/create/edit/delete is the right shape. Mirrors how Resources work for any "user manages a table of these things" use case. Deviates from session 215's `DashboardSettingsPage` (a custom Page) because dashboards are 1-per-role and Views are N-per-record-type.
- **The Edit page mounts the page-builder beneath the form.** A single Edit page, no sub-tabs. Form (record_type / handle / label / sort_order) at top; page-builder (Vue, in `record_detail` mode) below. The form saves via Filament's standard form save; the page-builder saves widget changes through the dedicated API controller. Mirrors how `EditPageTemplate` post-5c lays out General-fields-then-builder.
- **Vue editor mode: `record_detail`.** Add to `EditorMode` union alongside `'page' | 'dashboard'`. The mode transition is parallel to dashboard mode's: hide "Save as Template" footer button, show `View: {label} ({record_type})` in the toolbar instead of page title/author/status, filter inspector tabs through `RecordDetailSidebarSlot::layoutConstraints()['allowed_appearance_fields']` (currently `['background', 'text']` ‚Äî same as dashboard).
- **Column layouts ARE allowed.** Unlike dashboard mode (`column_stackable: false` on `DashboardGridSlot`), `RecordDetailSidebarSlot::layoutConstraints()` declares `column_stackable: true`. The Vue `+ Columns` dropdown stays visible in `record_detail` mode. The slot's other constraints (`full_width_allowed: false`, `dimensions: null`) are also respected.
- **Permission: `manage_record_detail_views`.** Mirror of `manage_dashboard_config`. super_admin bypasses via the existing `Gate::before` rule. Other roles require explicit grant. Added to `PermissionSeeder`.
- **Chrome Views are excluded from the Resource.** `page_template_header` and `page_template_footer` (and any future Views whose handle begins with `page_template_`, bound to `App\Models\Template`) are page-builder anchors that mount against the Template's `header_page_id` / `footer_page_id`. Authoring widgets directly on those rows would silently no-op (the chrome rendering pulls widgets from the linked Page, not from the View). The Resource's list/create/edit pages filter these out via a query scope; the seeded chrome rows still exist but are invisible to the authoring UI. Captured here as a session-local exclusion rule; if/when chrome Views grow widget-set semantics they can be unhidden via the same scope.
- **`record_type` Select options come from a hardcoded list.** The dropdown on the create/edit form lists `App\Models\Contact` only at first ship. Adding a new record_type is a code edit (one line in the Resource's `getRecordTypeOptions()` static method) when a new concrete widget arc demands it. No dynamic discovery ‚Äî keeps the surface auditable. Documented as a known carry-forward in the close-out log.
- **Slug uniqueness enforced at form + DB.** The form rule on `handle` requires unique-per-record_type (Filament `unique('record_detail_views', 'handle', modifyRuleUsing: ...)`). The DB-level unique index on `(record_type, handle)` from 5c is the safety net.
- **No widget contract changes.** No schema changes. The session is purely additive admin-side plumbing.
- **No `EditContact` change.** The seeded `contact_overview` View (from 5c's seeder) becomes the first View an admin can configure widgets on. Its current widget (the placeholder) stays ‚Äî admins can add more or remove the placeholder via the new authoring UI. Test-lock asserts the placeholder still renders against `EditContact` after the API is in place.
- **`pageWidgets` cascade on View delete.** When a View row is deleted, its polymorphic `page_widgets` rows are deleted via a model `deleting` observer (not DB-level FK ‚Äî `page_widgets.owner_id` has no FK constraint). Mirror of how `DashboardConfig` deletion would behave if it were ever deleted (today it's role-cascade; the page_widgets cleanup pattern is documented in 215's schema doc).

## Phase 1 ‚Äî `RecordDetailViewBuilderApiController` + routes

1. Create `app/Http/Controllers/Admin/RecordDetailViewBuilderApiController.php` ‚Äî mirror of `DashboardBuilderApiController`. Every endpoint gated by `manage_record_detail_views`. Widget-keyed endpoints assert IDOR boundary (`$widget->owner_type === RecordDetailView::class && $widget->owner_id === $view->id`). Endpoints:
   - `GET views/{view}/widgets` ‚Äî tree (`items` + `required_libs`)
   - `POST views/{view}/widgets` ‚Äî create. Rejects (422) if the widget handle's `allowedSlots()` does not include `record_detail_sidebar`
   - `PUT views/{view}/widgets/{widget}` ‚Äî update. Filters `appearance_config` to keys declared by `RecordDetailSidebarSlot::layoutConstraints()['allowed_appearance_fields']` (currently `['background', 'text']`)
   - `DELETE views/{view}/widgets/{widget}` ‚Äî delete
   - `PUT views/{view}/widgets/reorder` ‚Äî batch sort-order update scoped to the View's widgets
   - `GET views/{view}/widgets/{widget}/preview` ‚Äî preview rendered with `slotHandle: 'record_detail_sidebar'`
   - `GET views/{view}/widget-types` ‚Äî filtered to handles whose definition's `allowedSlots()` includes `record_detail_sidebar` via `WidgetType::forPicker(null, 'record_detail_sidebar')`
   - `POST/DELETE views/{view}/widgets/{widget}/appearance-image` ‚Äî parity with the page-builder controller's appearance-image endpoints
2. Create `routes/admin-record-detail-view-api.php` ‚Äî mirror of `routes/admin-dashboard-api.php`. Owner-scoped routes only (no widget-keyed routes outside the view prefix, no layout/nesting routes ‚Äî record-detail does have column layouts, so layout routes ARE needed; bring them across from `admin-dashboard-api.php` if absent there, or from `routes/admin-page-builder-api.php` if dashboards don't currently expose layout routes).
3. Register the route file in `app/Providers/Filament/AdminPanelProvider::panel()` under `/admin/api/record-detail-view-builder`.

## Phase 2 ‚Äî Vue page-builder `record_detail` mode

1. Update `resources/js/page-builder-vue/types.ts` ‚Äî extend `EditorMode` union to `'page' | 'dashboard' | 'record_detail'`.
2. Update `resources/js/page-builder-vue/stores/editor.ts` ‚Äî the existing `mode`, `allowedAppearanceFields`, `allowedWidgetHandles`, and `roleLabel` refs from session 215 are reused. Add a parallel `viewLabel` and `recordTypeLabel` ref populated from bootstrap when mode is `record_detail`. (Or: rename `roleLabel` to a generic `ownerLabel` and populate per-mode. Decision call during the session ‚Äî `viewLabel` keeps the diff smaller.)
3. Update `InspectorPanel.vue` ‚Äî same tab-filtering pattern as dashboard mode. `Presets` tab hidden in `record_detail` mode (presets are page-mode-only for now). Bottom-pane tabs filtered through `allowedAppearanceFields`. The `record_detail` slot's `allowed_appearance_fields = ['background', 'text']` matches dashboard exactly.
4. Update `PreviewCanvas.vue` ‚Äî the `+ Columns` dropdown stays visible in `record_detail` mode (because `column_stackable: true`). The current `v-if="store.mode !== 'dashboard'"` works as-is.
5. Update `EditorToolbar.vue` ‚Äî when `mode === 'record_detail'`, show `"View: {label} ({record_type})"`. Read `viewLabel` and `recordTypeLabel` from the store.
6. Update `App.vue` ‚Äî hide the `Save as Template` footer button when `mode === 'record_detail'` (parallel to the existing dashboard-mode hide).

## Phase 3 ‚Äî `RecordDetailViewBuilder` Livewire shell

1. Create `app/Livewire/RecordDetailViewBuilder.php` ‚Äî mirror of `DashboardBuilder`. Owns a `RecordDetailView` rather than a `DashboardConfig`. Mounts with 403-fail-closed gate on `manage_record_detail_views`. `getBootstrapData()` emits the `record_detail` shape:
   - `mode: 'record_detail'`
   - `owner_id: $viewId`
   - `owner_type: 'record_detail_view'`
   - `api_base_url: /admin/api/record-detail-view-builder/views/{viewId}`
   - `api_lookup_url` equal to `api_base_url`
   - `widget_types` pre-filtered to `record_detail_sidebar` via `WidgetType::forPicker(null, 'record_detail_sidebar')`
   - `allowed_appearance_fields` read from `RecordDetailSidebarSlot::layoutConstraints()`
   - `view_label` and `record_type_label` from the owning `RecordDetailView` row
2. Create `resources/views/livewire/record-detail-view-builder.blade.php` ‚Äî mirror of `dashboard-builder.blade.php`. Mounts the Vue app via `data-page-builder-app` with `data-bootstrap` JSON.
3. `createBlock(widgetTypeId)` ‚Äî Livewire handler for the widget-picker modal. Same allowedSlots check as the API controller.

## Phase 4 ‚Äî `RecordDetailViewResource` Filament resource

1. Create `app/Filament/Resources/RecordDetailViewResource.php`:
   - Navigation group: `Tools` (alongside Widget Manager)
   - Permission: `manage_record_detail_views` on `canViewAny` / `canCreate` / `canEdit` / `canDelete`
   - Form: `record_type` Select, `handle` TextInput, `label` TextInput, `sort_order` integer
   - Form-level validation: `(record_type, handle)` unique
   - Static `getRecordTypeOptions()` returning the hardcoded record-type list (`App\Models\Contact => 'Contact'`)
   - Table: columns for record_type, handle, label, sort_order, widget count (count of polymorphic page_widgets), updated_at
   - Filter: by record_type
   - Default sort: record_type asc, sort_order asc
   - **Query scope: exclude chrome Views** ‚Äî `RecordDetailView::query()->whereNot(fn ($q) => $q->where('record_type', Template::class)->where('handle', 'like', 'page_template_%'))` applied via `getEloquentQuery()` override.
2. Create the three resource pages:
   - `app/Filament/Resources/RecordDetailViewResource/Pages/ListRecordDetailViews.php`
   - `app/Filament/Resources/RecordDetailViewResource/Pages/CreateRecordDetailView.php`
   - `app/Filament/Resources/RecordDetailViewResource/Pages/EditRecordDetailView.php` ‚Äî extends `EditRecord` (no custom base), renders the form + the `@livewire('record-detail-view-builder', ['viewId' => $this->record->id])` mount beneath
3. Create `resources/views/filament/resources/record-detail-view-resource/pages/edit-record-detail-view.blade.php` ‚Äî page Blade with form section + builder section
4. Create help doc stub at `resources/docs/record-detail-views.md` with frontmatter and route registration per the standard pattern. Run `php artisan help:sync` after.
