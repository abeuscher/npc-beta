# 225. Date Handling — Centralized Vocabulary + Schema Audit

Land `App\Support\DateFormat` as the canonical date-format vocabulary, migrate every presentation-layer date format call onto it, and add `published_at` to the `events` and `products` tables (with backfill + observer wiring) to fix 221's QuerySettings `events` order_by allowlist, which currently falls back to `created_at` because the column doesn't exist.

This session is **infrastructure, not user-facing capability.** It does not deliver per-widget date-format selection, token-modifier syntax (`{{date|long}}`), or site-setting-driven format overrides. Those are user stories that future sessions can deliver cheaply *because* this session lands the vocabulary they will pick from. The framing is "pay one prerequisite session now so successor sessions are single-concern."

The Phase 4 schema fix (`published_at` on events and products) is the one user-visible improvement: the events QuerySettings panel's "newest first" sort starts ordering by *publication time* instead of *admin-row-creation time*.

---

## Design decisions (resolved before starting)

- **`DateFormat` is a final class with public string constants and a static `format()` helper, not a backed enum.** Constants grep cleanly, inline cheaply, and don't require `->value` everywhere. Enums-with-methods feel heavy for a closed-set lookup table. The class lives at `app/Support/DateFormat.php` (`App\Support\DateFormat`).
- **Helper signature:** `DateFormat::format(?Carbon $date, string $format): string`. Null-safe — null `$date` returns `''`. Handles the `TIME_SMART` conditional branch internally (the only format that needs a runtime check). Every other format is a passthrough to `$date->format($const)`.
- **9 named formats.** The 14 distinct format strings in production today collapse to 9 once author-taste inconsistencies (uppercase `PM` vs lowercase `pm`, full vs abbreviated month at the same precision) are resolved. The collapsed set:
  - `LONG_DATE` → `'F j, Y'` → "April 25, 2026"
  - `MEDIUM_DATE` → `'M j, Y'` → "Apr 25, 2026"
  - `LONG_DATETIME` → `'F j, Y \a\t g:i a'` → "April 25, 2026 at 5:00 pm"
  - `MEDIUM_DATETIME` → `'M j, Y g:i a'` → "Apr 25, 2026 5:00 pm"
  - `EVENT_FULL` → `'D, F j, Y \a\t g:i a'` → "Sat, April 25, 2026 at 5:00 pm"
  - `EVENT_COMPACT` → `'D M j · g:i a'` → "Sat Apr 25 · 5:00 pm"
  - `TIME_OF_DAY` → `'g:i a'` → "5:00 pm"
  - `TIME_SMART` (conditional) → `'ga'` if `$date->minute === 0` else `'g:ia'` → "5pm" / "5:30pm"
  - `EVENT_LIST_DATE` → `'F jS'` → "April 25th"
- **Lowercase `pm` is canonical.** 5 of 8 datetime callsites today use lowercase. Pick lowercase. The 3 uppercase callsites (TransactionResource QB-synced tooltip, FinanceSettingsPage cookie expiry, InteractsWithImportWizard default label) flip to lowercase under migration. **Document the visible casing change in the log** — it's a deliberate consistency move, not an accident.
- **`MEDIUM_DATETIME_TZ` is not a vocabulary entry.** One callsite (`FinanceSettingsPage` cookie expiry) appends `' T'` for timezone. Inline the timezone tail at that callsite as `DateFormat::format(...) . ' ' . $date->format('T')`. Doesn't earn vocabulary status for one consumer.
- **Time-zone semantics out of scope.** Every format renders in the app's `app.timezone` (set from `site_settings.timezone`). Per-event timezones, per-user timezone preferences, etc. — not in scope.
- **Locale-aware formatting explicitly deferred.** The vocabulary makes a future locale swap cheap (one constant body change moves the whole app); no hooks are exposed this session.
- **Date-display user controls explicitly deferred.** Per-widget format selection, token modifiers, and site-setting "default date display" are downstream stories that this vocabulary unblocks. Not delivered here.
- **Phase 4's `published_at` is a real timestamp column, not a virtual attribute.** Backfill from `created_at` for already-published rows (`status = 'published'`). Observer wiring sets it on first transition to `status = 'published'` if currently null. Mirror is "a fresh `Page` admin'd today gets `created_at = now()`, `published_at = now()` because the admin sets `published_at` via Filament's date picker." Pages and events/products differ here: pages have an admin-controlled `published_at` form field; events/products do not, so observer auto-set is the cleanest path. **A future session can add a Filament form field for explicit override** if user-driven publish scheduling becomes a need — out of scope here.
- **Backfill is idempotent and runs in the migration's `up()` method.** `UPDATE events SET published_at = created_at WHERE status = 'published' AND published_at IS NULL;` Same for products. Idempotent because the `WHERE published_at IS NULL` clause guards against re-running.
- **`ProductObserver` does not exist today.** Create it at `app/Observers/ProductObserver.php`. Register via `#[ObservedBy(ProductObserver::class)]` attribute on `Product` (mirror of `Event`'s registration). Single behaviour this session: set `published_at = now()` on first transition to `status = 'published' && !is_archived`. Future product lifecycle hooks can join the same observer.
- **Update QuerySettings `orderByOptions` allowlists.** EventsListing, ThisWeeksEvents (shares the events shape), and ProductCarousel all gain `'published_at' => 'Published'`. BlogListing already has it. The post-arm allowlist is unchanged.
- **Machine date formats are explicitly out of scope.** `Y-m-d`, `Y-m-d H:i:s`, `Ymd-His` (filenames), `Y-m-d` for QuickBooks API — these are wire formats, not presentation. Different concern. Not migrated.
- **`deleted_at` symmetry deferred.** Per the stub. Events and products keep using `status='cancelled'` and `is_archived` as tombstones.

## Phase 1 — `DateFormat` vocabulary

1. Create `app/Support/DateFormat.php`. Final class. Public constants for the 9 named formats. Static `format(?Carbon $date, string $format): string` helper.
2. The helper:
   - If `$date === null`, return `''`.
   - If `$format === self::TIME_SMART`, return `$date->format($date->minute === 0 ? 'ga' : 'g:ia')`.
   - Otherwise, return `$date->format($format)`.
3. Note: `TIME_SMART`'s constant value can be a sentinel string (e.g. `'__time_smart__'`) since it's never passed through to `Carbon::format` directly. Pick a sentinel that won't collide with any legitimate format string.
4. Unit test `tests/Unit/Support/DateFormatTest.php`:
   - One test per named format. Use a fixed Carbon instance (`Carbon::parse('2026-04-25 17:00:00')` — Saturday 5:00 PM) to assert exact output strings.
   - One test for `TIME_SMART` with both halves: `17:00` → `'5pm'`; `17:30` → `'5:30pm'`.
   - One test for null-safety: `DateFormat::format(null, DateFormat::LONG_DATE)` → `''`.

## Phase 4 — Schema migration: `published_at` on events and products

1. Write migration `database/migrations/YYYY_MM_DD_HHMMSS_add_published_at_to_events_and_products.php`:
   - `Schema::table('events', fn (Blueprint $t) => $t->timestamp('published_at')->nullable()->after('status'));`
   - Same for `products`.
   - After both `Schema::table()` calls, run the backfill:
     ```php
     DB::statement("UPDATE events SET published_at = created_at WHERE status = 'published' AND published_at IS NULL");
     DB::statement("UPDATE products SET published_at = created_at WHERE status = 'published' AND is_archived = false AND published_at IS NULL");
     ```
   - `down()`: drop both columns.
2. Run the migration via Docker: `docker compose exec app php artisan migrate`.
3. Update `docs/schema/events.md` — add the `published_at` row to the column table with notes on observer-driven set behaviour.
4. Update `docs/schema/products.md` — same.
5. `app/Models/Event.php` — add `'published_at' => 'datetime'` to `$casts` (Eloquent infers it, but explicit is preferred).
6. `app/Models/Product.php` — same.
7. `app/Observers/EventObserver.php` — extend with:
   - In `creating`: if `status === 'published'` and `published_at` is null, set `published_at = now()`.
   - In `updating`: if `wasChanged('status')` is *about to be* true and the new status is `'published'` and `published_at` is null, set `published_at = now()`. (Use the model's dirty attribute API — `isDirty('status')` + `getOriginal('status')` pattern.)
8. Create `app/Observers/ProductObserver.php`. Single method (`creating` + `updating`) covering both transitions: status changes to `'published'` AND `is_archived === false`. Register via `#[ObservedBy(ProductObserver::class)]` attribute on `Product` (mirror of `#[ObservedBy(EventObserver::class)]` on `Event`).
9. `app/Widgets/EventsListing/EventsListingDefinition.php::querySettings` — add `'published_at' => 'Published'` to `orderByOptions`.
10. `app/Widgets/ThisWeeksEvents/ThisWeeksEventsDefinition.php::querySettings` — same. (Shares the events allowlist; verify by reading the file.)
11. `app/Widgets/ProductCarousel/ProductCarouselDefinition.php::querySettings` — same addition.

## Security checklist

Before closing:

- [ ] `DateFormat::format()` accepts only typed inputs (`?Carbon`, `string`) — no user-controlled format string ever reaches `Carbon::format`. The format parameter is constrained at every callsite to one of the public constants on the `DateFormat` class.
- [ ] Backfill SQL is idempotent (`WHERE published_at IS NULL` guards re-runs). Verified by running the migration twice on a non-fresh DB and asserting no row mutation.
- [ ] Observer auto-set respects existing `published_at` values (only sets when null on transition to published) — never overrides an admin-set value. Asserted by `EventPublishedAtObserverTest` / `ProductPublishedAtObserverTest`'s "existing value preserved" subtest.

## Out of scope

- **Per-widget date-format selection.** Inspector-driven "Long / Medium / Smart" select on each widget instance. Successor session, not scheduled.
- **Token modifier syntax** (`{{date|long}}`, `{{date|smart}}`). Successor session, not scheduled.
- **Site-setting "default date display."** Cheapest extension, but explicitly deferred. Successor session, not scheduled.
- **Locale-aware formatting** (24-hour time, European date order, language-specific month/day names). The vocabulary makes this future swap cheap; no hooks exposed this session.
- **Per-event or per-user timezone overrides.** Out of scope. App uses `app.timezone` globally.
- **`deleted_at` symmetry on events and products.** Per the stub — `status='cancelled'` and `is_archived` keep being the tombstones. Migrating off them is a bigger refactor than this session covers.
- **Machine date formats** (`Y-m-d` for CSV/QB, `Ymd-His` for filenames, single-component reads like `format('A')` for AM/PM extraction). Different concern from presentation; not touched.
- **Membership `starts_on` / `expires_on`.** `date` columns (not timestamps), intentionally — membership periods are calendar days, not points in time. Not migrated.
- **Memos `posted_at`** as a real column. Lives as a JSONB key inside `collection_items.data` (the SWCT JSONB-key sort path 221 added). Not a candidate for promotion.
- **Filament form field for `published_at` on events / products.** A future session can expose admin-controlled publish scheduling (mirror of pages); this session only wires observer-driven auto-set on first publish.
