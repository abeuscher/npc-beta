# 087. Inspector Panel & Shared Page Builder Form

Refactor all page-builder edit pages to share a single form layout via a trait, reshape that layout to surface SEO and Custom Fields, give the page builder full horizontal width, and replace inline block editing with a fixed inspector panel so that widget controls are accessible at any nesting depth.

---

## Design decisions (resolved before starting)

- A new trait `App\Traits\HasPageBuilderForm` defines the shared form layout. Each resource (`PageResource`, `PostResource`, `EventResource`) adopts the trait and injects its unique fields into designated slots. No layout logic is duplicated across resources.
- The edit page layout for all three resources becomes:
  - **Row 1:** Title / type / slug section (1 column) beside Settings — Author, Published, Publish Date / Tags (2 columns).
  - **Row 2:** Custom Fields section — full 3 columns, collapsed by default, hidden when no custom fields exist for the model type.
  - **Row 3:** SEO section — full 3 columns, collapsed by default, always visible.
  - **Row 4:** Page Builder section — full 3 columns.
- `PageBuilder` becomes a two-column layout: structural block list (left, ~55% width) and inspector panel (right, ~45% width).
- `PageBuilderBlock` is stripped to a structural handle only: drag grip, label, type badge, select trigger, ellipsis menu. No inline body panel.
- Clicking a block (or its label area) sets `$selectedBlockId` on `PageBuilder` and loads `PageBuilderInspector` in the right panel.
- `PageBuilderInspector` is a new Livewire component. It owns all config editing, query settings, and the style_config spacing panel. Persistence logic mirrors what `PageBuilderBlock` previously handled.
- The inspector panel uses `position: sticky` so it remains visible while scrolling a long block list.
- `config_schema` gains a `select` field type. A `select` field definition includes an `options_from` key (string) naming a registered data source. The inspector resolves named sources at mount time. Built-in sources: `events` (published Event records, keyed by slug, labelled by title), `products` (published Product records, keyed by slug, labelled by name), `forms` (active Form records, keyed by handle, labelled by name). Existing `text` fields used for `event_slug`, `product_slug`, and `form_handle` in the seeder are converted to `select` with the appropriate `options_from` value.
- Column widget slot panels remain in the structural view as layout panels. They show compact child block handles. Clicking a child selects it in the inspector.
- The "Add Block" modal remains on `PageBuilder` (root level) and on the column block's slot panels (child level) — no change to that flow.
- Child blocks inside column slots continue to dispatch scoped events (`child-*-requested`) as built in session 086.
- The inspector shows a neutral "Select a block to edit" placeholder when nothing is selected.

## Phase 1 — Shared form trait

Create `app/Traits/HasPageBuilderForm.php`. The trait provides:

- `pageBuilderFormSchema(array $titleSectionFields, array $uniqueFields = []): array` — a static method that assembles the full 3-column form layout. Parameters:
  - `$titleSectionFields` — the fields to render in the left title column (title input, type selector/display, slug, public URL, and any resource-specific extras).
  - `$uniqueFields` — additional resource-specific fields appended to the title column below the common fields (e.g. event date fields, post excerpt).
- The method constructs the Settings section with Author, Published toggle, Publish Date, and the TagSelect component, arranged as: Row 1 = Author · Published · Date, Row 2 = Tags.
- Custom Fields section: full-width, collapsed, hidden when `CustomFieldDef::forModel($modelType)->doesntExist()`. The model type string is passed as a parameter.
- SEO section: full-width, collapsed, always visible.
- Page Builder section: full-width, `Livewire::make(PageBuilder::class)`, hidden on create.

Read `PageResource.php`, `PostResource.php`, and `EventResource.php` fully before writing the trait. Map every field in each resource to its slot in the new layout.

## Phase 2 — Form reshape

Update `PageResource`, `PostResource`, and `EventResource` to call the trait's method. Each resource passes its unique title-section fields and the correct model type string for custom fields.

After updating each resource, verify that all existing fields are present and in the correct position. No fields may be dropped.

`EventResource` has fields that do not appear in the other two (start/end dates, capacity, registration mode, etc.). Read its form definition in full before deciding how to place those fields — they likely belong in the title column or as a second `Group` within the left section.

## Phase 3 — `config_schema` select type and data sources

Add `select` to the set of supported `config_schema` field types. A select field definition:

```php
['key' => 'event_slug', 'type' => 'select', 'label' => 'Event', 'options_from' => 'events']
```

Add a `PageBuilderDataSources` class (or equivalent) in `app/Services/` that resolves named sources to `[value => label]` option arrays:

- `events` — `Event::published()->orderBy('starts_at')->get(['slug', 'title'])` → keyed by slug
- `products` — `Product::published()->orderBy('name')->get(['slug', 'name'])` → keyed by slug
- `forms` — `Form::where('is_active', true)->orderBy('name')->get(['handle', 'name'])` → keyed by handle

Update `WidgetTypeSeeder` to convert the following fields from `text` to `select`:
- `event_description.event_slug` → `options_from: events`
- `event_registration.event_slug` → `options_from: events`
- `product_display.product_slug` → `options_from: products`
- `web_form.form_handle` → `options_from: forms`

Re-run the seeder after updating. Existing saved `config` values (slugs/handles) remain valid — no migration needed.

## Phase 4 — PageBuilderInspector component

Create `app/Livewire/PageBuilderInspector.php` and its blade `resources/views/livewire/page-builder-inspector.blade.php`.

The inspector receives `$blockId` as a prop. On mount it loads the block's full data (same fields as the current `PageBuilderBlock::loadBlock()`) plus all data-source option arrays needed for any `select` fields in the widget's `config_schema`.

The blade renders:
- Block label (editable inline, same pattern as the current block header edit-in-place).
- All `config_schema` fields — same field types as the current block body, plus `select` (rendered as a `<select>` element wired to `block.config.{key}`).
- Query settings collapsible section (same as current, if the widget has collections).
- Advanced spacing panel (style_config, same as current advanced panel).

Persistence via `updated()` and `updateConfig()` — same logic currently in `PageBuilderBlock`, moved here verbatim.

When `$blockId` is empty, the blade renders the neutral placeholder: "Select a block to edit its settings."

## Phase 5 — PageBuilder two-column layout and block selection

Add `public string $selectedBlockId = ''` to `PageBuilder`.

Add `selectBlock(string $blockId): void` — sets `$selectedBlockId = $blockId`.

Update `PageBuilder::render()` to pass `$selectedBlockId` to the view.

Update `resources/views/livewire/page-builder.blade.php`:
- Wrap the existing layout in a two-column flex container: left column (structural list + Add Block header) and right column (inspector).
- Right column renders `@livewire('page-builder-inspector', ['blockId' => $selectedBlockId], key('inspector-' . $selectedBlockId))`.
- The inspector column uses `position: sticky; top: 1rem` and a `max-height` with `overflow-y: auto` so it stays in view and scrolls independently of the block list.

## Phase 6 — PageBuilderBlock stripped to structural handle

Remove the block body panel (`<div x-show="open" ...>`) from `page-builder-block.blade.php` entirely.

Remove the advanced panel toggle and the advanced panel itself — those controls now live in the inspector.

The block header becomes the full component. The label area (currently a collapse/expand toggle) becomes the block select trigger: clicking it calls `$wire.$parent.selectBlock('{{ $blockId }}')` and adds a visual selected state (highlighted border or background) when `$selectedBlockId === $blockId`.

Keep: drag handle, label with edit-in-place, type badge, ellipsis menu (add above/below, copy, move up/down, delete).

Remove from `PageBuilderBlock.php`: `updated()`, `updateConfig()`, all properties and methods related to `style_config` editing, the advanced panel toggle state. The structural properties (`blockId`, `isFirst`, `isLast`, `isRequired`, `parentBlockId`, `parentColumnIndex`, child slot methods) all stay.

The column widget body (slot panels) stays in the block body but is now the *only* body content rendered by `PageBuilderBlock`. The `x-show="open"` wrapper remains for column widgets only — the slot panels need to be collapsible.

## Security checklist

Before closing:

- [ ] Inspector data source queries return only published/active records — no draft events, inactive forms, or unpublished products are offered in dropdowns.
- [ ] `selectBlock()` validates that the requested `$blockId` belongs to a widget on `$this->pageId` — a block from another page cannot be loaded into the inspector.
- [ ] The inspector's `updated()` method validates `$blockId` belongs to `$this->pageId` before writing to the database.
- [ ] The `select` field type renders values via `htmlspecialchars` / Blade's `{{ }}` — no raw option values injected into HTML attributes.
