# 135. Widget Data Contract & Demo Data

Foundation session for the page builder overhaul. Every widget type becomes a complete, self-describing contract: its config schema declares field groups for the future properties panel, and a new demo data service can generate plausible sample content for any widget on the fly — enabling meaningful previews before real content is bound.

---

## Design decisions (resolved before starting)

- **Two config groups, not three.** Each field in `config_schema` gets a `group` key: `content` or `appearance`. Content = what data the widget works with and how it's structured. Appearance = everything about how it renders, including whether optional elements are visible. The properties panel (session 137) will further subdivide appearance into collapsible sections (Layout, Styling, etc.) — that's a UI concern, not a schema concern.
- **Rule for the boundary:** visibility toggles (show/hide date, show/hide CTA) are `appearance`. Data format or data source decisions are `content`.
- **Demo data is generated on the fly, not stored.** The widget stores its schema (with `type` and optional `subtype` on each field). A `DemoDataService` reads the schema and fills it with plausible values. Nothing is persisted — demo data is always in sync with the current schema.
- **The demo data service is a first-class app service**, not a dev tool. It lives in `app/Services/DemoDataService.php` and is available in all environments. The existing `DashboardDebugGeneratorWidget` should be refactored to use this service for its data generation rather than duplicating logic. The debug widget remains gated behind `APP_DEBUG_TOOLS`, but the underlying service it calls is production-safe.
- **Field type + subtype resolution.** The demo data generator uses a two-level map: field `type` (richtext, text, number, etc.) and optional `subtype` that narrows the output shape. If no subtype exists, the generator writes the default for that type. Subtypes are added over time as we encounter fields where the type default would look obviously wrong in a preview.

## Phase 1 — Config schema field grouping

Add a `group` key to every field definition in every widget type's `config_schema`. Values: `content` or `appearance`.

**Write a migration** that reads all widget type records and backfills the `group` key on each field. Use these rules as defaults:

- Fields that control *what data* the widget displays → `content` (e.g. `content`, `collection`, `url`, `title`, `ctas`, `email`, `fund_id`, `event_id`, `form_id`, query config like `limit`, `order_by`)
- Fields that control *how it looks or what's visible* → `appearance` (e.g. `overlay_opacity`, `text_position`, `fullscreen`, `show_date`, `alignment`, `columns`, `min_height`, `background_color`, `text_color`, `scroll_indicator`, `overlap_nav`)

After the migration, update the seeder so fresh installs also have the `group` key. Verify by spot-checking several widget types after running.

## Phase 2 — Field subtype system

Add an optional `subtype` key to config schema field definitions. This narrows the demo data generator's output for that field. The subtype is only needed when the field type's default output would look obviously wrong in a preview.

**Initial subtype map** (expand as needed during backfill):

| Type | Subtype | Output shape |
|------|---------|-------------|
| `richtext` | *(none / default)* | 1-2 paragraphs of lorem ipsum |
| `richtext` | `article` | 3-5 paragraphs with an h2 heading, a bullet list, inline formatting |
| `text` | *(none / default)* | Short phrase (2-5 words) |
| `text` | `title` | Sentence-case phrase (4-8 words) |
| `text` | `email` | `jane.doe@example.org` pattern |
| `text` | `url` | `https://example.com/page` |
| `text` | `phone` | `(555) 867-5309` pattern |
| `text` | `state` | Two-letter US state abbreviation |
| `text` | `zip` | 5-digit or 9-digit zip |
| `text` | `city` | City name |
| `text` | `address` | Street address |
| `text` | `person_name` | Full name |
| `number` | *(none / default)* | Random integer in a reasonable range |
| `number` | `currency` | Dollar amount (two decimals) |
| `color` | *(none)* | Hex color value |
| `select` | *(none)* | First option from the field's `options` list |
| `toggle` | *(none)* | `true` |
| `image` | *(none)* | Placeholder image reference (or null — handled by the preview shell) |
| `buttons` | *(none)* | Array of 1-2 sample CTAs with labels and `#` URLs |
| `url` | *(none)* | `https://example.com` |

Backfill subtypes onto existing widget config schemas where the default would produce wrong-looking output. Not every field needs a subtype — only add them where the type default is visibly implausible for that field's purpose.

## Phase 3 — DemoDataService

Create `app/Services/DemoDataService.php`. This is the single service responsible for generating plausible sample content.

**Core method:** `generateForWidget(WidgetType $widgetType): array` — reads the widget's `config_schema`, generates a value for each field based on `type` + optional `subtype`, returns a config array in the same shape the widget template expects.

**For collection-bound widgets:** The service also generates sample collection data matching the shape `WidgetDataResolver` produces. The four source types need coverage:

- `events` → array of event-shaped records (title, slug, starts_at, ends_at, is_virtual, is_free, url, thumbnail_url)
- `blog_posts` → array of post-shaped records (title, slug, published_at, thumbnail_url)
- `products` → array of product-shaped records (name, slug, description, capacity, available, image_url, prices)
- `custom` → read the collection's field schema, generate sample values per field using the same type/subtype map

**For collection items:** use Laravel's `fake()` helper (Faker) where it provides good built-in generators (names, addresses, dates, emails). For structured content (richtext), assemble from building blocks: sentences, paragraphs, headings.

**Implementation guidance:**
- The type/subtype map should be a clean, declarative structure (array or match expression) — easy to extend when new subtypes are needed.
- The service must not persist anything. All generated data is ephemeral.
- Image fields should return `null` for now — the preview shell (session 136) will handle placeholder rendering.

---

## Phase 4 — Refactor debug generator onto DemoDataService

Refactor `DashboardDebugGeneratorWidget` to use `DemoDataService` for its data generation where possible. The debug widget still handles persistence (creating real DB records) — but the *shape and content* of the generated data should come from the service.

The goal: one place that knows what plausible sample data looks like. The debug widget calls the service to get the data shape, then persists it. The preview system calls the same service but uses the data ephemerally.

Where the debug widget generates data that doesn't map cleanly to the service (e.g. creating event registrations with contact linkage, portal accounts with passwords), keep that logic in the debug widget — don't force it into the service. The service generates *content-shaped data*; the debug widget handles *relationship wiring and persistence*.
