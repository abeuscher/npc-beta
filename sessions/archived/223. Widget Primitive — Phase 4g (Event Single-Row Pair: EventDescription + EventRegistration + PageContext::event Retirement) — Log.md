### Phase 2 — Contract layer extension

- `app/WidgetPrimitive/DataContract.php`: added `CARDINALITY_MANY = 'many'` and `CARDINALITY_ONE = 'one'` constants and a new `public readonly string $cardinality = self::CARDINALITY_MANY` constructor field. Default `'many'` preserves byte-equivalence for every existing list-shaped contract — no list-shaped widget required any change.
- `app/WidgetPrimitive/Projectors/SystemModelProjector.php`:
  - Refactored the per-model dispatch out of `project()` into a new private `projectorFor(DataContract): ?callable` helper so both `project()` (list) and `projectOne()` (single-row) can share it.
  - Added `projectOne(DataContract, ?Model): array{item: array|null}` returning `['item' => null]` when the model is absent or the per-model dispatch returns null; otherwise field-filtering through `$contract->fields` exactly as `project()` does.
  - Extended `projectEvent()` from 15 to 24 fields. New: `description` (raw HTML string), `is_in_person`, `is_virtual`, `is_at_capacity`, `registration_mode`, `mailing_list_opt_in_enabled`, `external_registration_url`, `price`, `status`. The `status` addition was not in the prompt's "8 new fields" prose but is required by EventRegistration's whitelist (which includes `status` for the cancelled-event branch). Implementation note: `is_at_capacity` derives from `$event->capacity !== null && (int) ($event->getAttribute('registered_count') ?? 0) >= (int) $event->capacity` — null-capacity returns false, matching `Event::isAtCapacity()`'s behaviour. The `registered_count` attribute is the `withCount`-aliased count from the resolver.
- `app/WidgetPrimitive/ContractResolver.php`:
  - `resolveSystemModel()` rewritten to dispatch on `cardinality` first. Cardinality `ONE` dispatches to `resolveEventOne()` (or returns `['item' => null]` for unknown models). Cardinality `MANY` keeps the existing `resolvePost`/`resolveEvent` arms.
  - New private `resolveEventOne(DataContract, &$cache)`:
    - Reads `$slug = (string) ($contract->filters['slug'] ?? '')`. Empty-slug early-return with `['item' => null]` and zero DB hits.
    - Cache key `'event:one:' . $slug`. Two single-row Event widgets on one event landing page share one query.
    - Query: `Event::published()->where('slug', $slug)->withCount(['registrations as registered_count' => fn ($q) => $q->whereIn('status', ['pending','registered','waitlisted','attended'])])->with(['media', 'landingPage'])->first()`.
    - Returns `$this->systemModelProjector->projectOne($contract, $cache[$key])`.

### Notable findings

- **`status` field was missing from the prompt's "8 new fields" enumeration but required by the EventRegistration whitelist.** The prompt's prose listed 8 new projector fields; the canonical EventRegistration whitelist (also in the prompt) explicitly included `status` — needed for the cancelled-event branch (`$event->status === 'cancelled'`). Resolved by emitting `status` from the projector. Final projector field count: 24 (15 existing + 9 new).
- **`Blade::render()` does not inject `$errors`.** Pre-existing template behaviour: EventRegistration's `@elseif ($regOpen)` branch references `$errors->has('register')`, which is normally provided by `Illuminate\View\Middleware\ShareErrorsFromSession`. When tests render a widget via `WidgetRenderer::render()` (which calls `Blade::render()` directly, not the framework's view-render lifecycle), `$errors` is undefined. The simplest sidestep was to set `registration_mode = 'closed'` in Test B so the template short-circuits before reaching `$errors`. Carry-forward note for the Product retrofit in 224: ProductDisplay's at-capacity branch references `$errors->has('waitlist')` and the open-checkout branch references `$errors->has('checkout')` — same caveat applies, and 224's prompt prefers binding a `ViewErrorBag` via `view()->share('errors', new ViewErrorBag())` in `beforeEach` since both branches reference `$errors` and the gerrymander route is harder.
- **`DB::enableQueryLog()` does not clear the existing log.** Test B initially failed because the log carried over queries from an earlier render block. `DB::flushQueryLog()` is the explicit clear. Carry-forward note for any future test that captures multiple query-log windows in the same test method.
- **Eager-load query-skip caveat (carried from 217 / verified again here).** EventDescription Test B creates a real `Page` row and sets `landing_page_id` on the published event so the `pages where "id" in` eager-load query fires. Without that, Laravel's belongsTo eager-load skips the secondary query when all parent FKs are null and the assertion fails. Same fix shape as 217's EventsListing Test B.
- **`PageContext` after this session.** Retains: `currentPage` property, `currentUser` property, `collection()` method (+ `collectionCache`), `product()` method (+ `productCache`), `form()` method (+ `formCache`). Three of those (`collection`, `product`, plus the entire `WidgetDataResolver` shell that backs them) retire in 224. `form()` survives 224 — the Forms widget has not retrofitted and has no scheduled session.
