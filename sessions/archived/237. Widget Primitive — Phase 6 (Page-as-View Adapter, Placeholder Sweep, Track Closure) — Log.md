### Phase 1 — Page-as-View collision discovery + deliberate retirement

The original prompt scoped Page-as-View as "implement the small `IsView` adapter on `App\Models\Page`." Reading `app/Models/Page.php:91-94` surfaced the collision the prompt explicitly anticipated: `Page::widgets()` already returns a `MorphMany` query builder, consumed by ~30 call sites (PageController, PostController, ChromeRenderer, SeoMetaGenerator, ContentImporter, PageObserver, multiple seeders, EventResource, DashboardDebugGeneratorWidget). The `IsView` interface requires `widgets(): array`. Adopting the interface would force renaming `Page::widgets()` → `Page::pageWidgets()` across all call sites for a capability with no current consumer.

User chose **option B** (formally retire Page-as-View as an elective hook) over option A (rename + adapt) and option C (rename the interface method). Reasoning recorded as feedback memory `feedback_page_is_not_a_view.md`: Page is functionally and security-distinct from DashboardView/RecordDetailView (slug-routed, soft-deleted, has its own publishing/SEO/media/custom-fields stack, has `HasSourcePolicy` with `ACCEPTED_SOURCES`); flattening it into the View family was never the architectural goal even when forced.

No code change to `App\Models\Page`. The `IsView` interface stays in place; if a future feature genuinely needs unified iteration across all widget-mounting surfaces, the rename + adapter is a single small session.
