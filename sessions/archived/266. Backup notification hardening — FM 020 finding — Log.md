## Findings register

- **Listener-chain ordering — diagnosis confirmed.** `php artisan event:list` showed spatie's notification closure (`vendor/spatie/laravel-backup/src/Notifications/EventHandler.php:24`) registers ahead of `App\Listeners\RecordBackupSuccess@handle` for `Spatie\Backup\Events\BackupWasSuccessful`. A throw in spatie's mail path does kneecap the listener — the prompt's diagnosis was correct, and dropping the success-channel mail removes the throw surface entirely.
- **Spatie's `Config` is a lazy-scoped singleton, not direct `config()` reads.** Spatie's mail-notification path does NOT read `config('backup.notifications.mail.*')` directly. It reads `app(Spatie\Backup\Config\Config::class)->notifications->mail->...`, where `Config` is bound via `BackupServiceProvider::packageRegistered()` as `app->scoped(Config::class, fn() => Config::fromArray(config('backup')))`. Because the binding is a closure-resolver (not eager), the Config object is built only on first resolution — which happens at notification dispatch time, well after `AppServiceProvider::boot()`. So runtime `config([...])` overrides DO propagate. Phase 3 test #3 carries a `forgetInstance` + `app(SpatieBackupConfig::class)` assertion to prove the bridge reaches spatie's actual read path, not just the `config()` proxy.
- **SiteSetting rows present but empty in dev DB.** All three keys (`mail_from_address`, `mail_from_name`, `contact_email`) exist post-seed but with empty string values, so the `?->value ?: 'default'` fallback resolves to defaults under untouched dev DB. Test #3 must explicitly seed values to be meaningful — used `SiteSetting::updateOrCreate` to handle either new-test-DB or pre-seeded state.
- **Demo-mode ride-along confirmed.** `BaseNotification::via()` returns the configured channel array; spatie sends through Laravel's standard Notification system, which uses Laravel's default mailer. Existing `mail.default = log` short-circuit covers spatie's mail in demo mode without additional override.
- **Re-invoking `(new AppServiceProvider(app()))->boot()` in tests is non-disruptive enough.** The boot block calls a few `Model::observe()` registrations and `*::configureUsing()` callbacks alongside the `config([...])` overrides. Re-invocation re-registers observers (idempotent in practice) and appends additional configurer closures (slightly additive, but doesn't break anything within a single test run). The pattern is fine for the test surface; if a future session needs heavier re-binding, a dedicated SiteSetting-bridge helper extracted out of `boot()` would be cleaner.
- **A1e lifted at 266 close.** A1e (Fleet Manager Contract v2.3.0 — Backup Blob Download Endpoint) was lifted into the release-plan working set during 266 close prep, sourced from the FM 020 follow-on planning thread. Position 22 in the execution order, immediately after A1d'. CRM-side prerequisite for FM 021 + FM 022's restore-to-fresh-node primitive; A2(c) success-criterion enabler. A2's success-criterion narrative updated to reflect the dependency.
