## Open questions to resolve at session start

- **G3 variant naming.** `secondary-dark` reads as a pair with `secondary` and scales into the eventual per-variant night-mode treatment. `ghost-on-dark` reads as a one-off. The variant becomes a permanent option in `button_styles` SiteSetting and shows up in DesignSystemPage UI — pick the name that scales for the future direction.
- **Typography target values for G13/G14.** Spec calls for H1 ≈ 3.5rem; H2/H3 deltas should be set proportionally so the heading scale feels balanced against H1 (a 0.5rem gap between H2 and H3 is the existing scale and reads cramped — consider widening). Heading margin-bottoms: H1 1.5rem, H2 1.25rem, H3 1rem, `p` 1rem are the proposed defaults; confirm before pushing or pick alternatives.
- **G2 CTA schema shape on TextBlock.** Mirror Hero's `ctas` field exactly (text / URL / style with primary/secondary/text options), or restrict to a smaller set per the text-widget's lighter-weight intent? Recommendation: mirror Hero's full shape — operators benefit from consistency across widgets that both carry CTAs.

## Phases

### Phase 1 — Image widget audit

Run before any other change. Confirm the Image widget renders cleanly at 4:3, 1:1, and 4:5. Source: `app/Widgets/Image/ImageDefinition.php` + `template.blade.php` + `styles.scss`. If all three render cleanly, log the audit finding in the gap report and move on. If a ratio is broken, surface to the user with a one-line proposed fix scope — the user redirects to either expand 285 to absorb the image fix or defer.

### Phase 2 — G1: Layout-level appearance_config round-trip

- Extend `App\Services\ImportExport\ContentExporter::serializeLayout` to emit `appearance_config`.
- Extend `App\Services\ImportExport\ContentImporter::hydrateLayout` to pass `appearance_config` through on layout creation.
- Add a fast Pest test: create a page with a column layout carrying a non-trivial appearance_config (background color + padding), export it via `ContentExporter::exportPages`, re-import the resulting bundle via `ContentImporter::import`, assert the rebuilt layout's appearance_config matches the original.
- Verify by re-importing PMW1's cleaned home.json with a temporary `appearance_config` test value on one of its layouts; round-trip and confirm the value survives.

### Phase 3 — G2: Text widget CTAs

- Add a `ctas` schema field to `App\Widgets\TextBlock\TextBlockDefinition::schema()` — type `buttons`, with the same per-button fields Hero exposes (text, URL, style with primary/secondary/text options).
- Update `app/Widgets/TextBlock/template.blade.php` to render the CTAs below the rich-text content using `@include('widget-shared.buttons', ...)`. CTA alignment derives from the existing `vertical_align` knob OR a new sibling field (decide at session start; lean toward a new field for cleaner control).
- Run `npm run build` so the SCSS / Vue page-builder picks up the schema change. Run `php artisan build:public` so the public widget bundle picks up any new template CSS.
- Add a fast Pest test: a TextBlock with one or more CTAs renders the button HTML and respects the style selection.

### Phase 4 — G3 (interim): Ghost-on-dark button variant

- Add a sixth variant to `App\Filament\Pages\DesignSystemPage::defaultButtonStyles()` — white outline, transparent fill, white text, lightens-on-hover. Variant name per the open question above.
- Update `DesignSystemPage::buttonFormSchema()` to render the new variant's section in the Theme page UI.
- Add the new variant slug to the per-CTA `style` options on the Hero widget schema (and on Text widget if Phase 3's CTA shape uses the same options list — likely yes).
- Save the new variant defaults via DesignSystemPage's save action, which triggers `AssetBuildService::build()` and refreshes the public CSS bundle.
- Add a fast Pest test asserting the new variant slug is a valid `button_styles` SiteSetting option and renders the expected hex colors.

### Phase 5 — G13/G14: Typography tuning

- Push the new typography values via the Theme page at `Admin → CMS → Theme → Text Styles`. Target deltas (confirm at session start):
  - H1 size → ~3.5rem (56px)
  - H1 margin-bottom → 1.5rem
  - H2 margin-bottom → 1.25rem
  - H3 margin-bottom → 1rem
  - `p` margin-bottom → 1rem
- Confirm via tinker or DB read that the `typography` SiteSetting JSON reflects the new values.
- Surface a visual confirmation to the user — load the current home page in a browser, scan visual hierarchy against the four reference images. The user judges whether the deltas hit or need adjustment.
