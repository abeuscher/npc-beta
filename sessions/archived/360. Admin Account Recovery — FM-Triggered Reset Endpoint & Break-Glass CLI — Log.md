## What was built

The CRM half of operator-mediated admin account recovery — promoting the session-304 "admin lockout has no recovery" flag, made urgent by session 359's mandatory admin TOTP 2FA (a correct password alone no longer gets a locked-out admin in). **Boundary-touching:** adds a fifth Fleet Manager node endpoint; CRM contract bumped **v2.4.0 → v2.5.0** (additive). The FM control-panel UI that calls the endpoint is FM-repo work and is out of scope here.

Scope settled at the Open Gate (recommended leans, owner-confirmed): **(b)** full credential recovery shaped as two composable actions — `reset_2fa` + `reset_password` — **not** super-admin creation/promotion and **not** force-change-on-next-login (the lockout victim already exists; the temp password stands until the admin changes it). Audit rides the app's existing activity log (decision flipped from the gate's activitylog recommendation once a deeper bug surfaced — see Security considerations).

### Phase 1 — Reset service + break-glass CLI

- `app/Services/AdminAccountRecovery.php` — a reusable service with composable actions `reset_2fa` (clears the `two_factor_*` columns via Fortify's `DisableTwoFactorAuthentication` action — stays on the same primitives 359 used) and `reset_password` (sets a node-generated temp password via `Str::password(20, symbols:false)`, hashed by the model's `hashed` cast). Rejects an empty/unrecognised action set (`InvalidArgumentException`). Operates on **any** admin including the protected oldest super-admin — reset ≠ delete, so `User::isProtected()` (the UI delete-guard) is deliberately not consulted; the protected admin is the likeliest lockout victim. Returns an `AdminRecoveryResult` DTO (`app/Services/AdminRecoveryResult.php`) carrying the actions applied, the one-time temp password (or null), the target, and the timestamp — so callers build their output from the in-memory result, not by re-reading the audit log.
- `app/Console/Commands/AdminRecoverCommand.php` — `admin:recover {email} {--reset-2fa} {--reset-password}`, the break-glass path for a node cut off from FM. **Hard-gated to refuse in demo mode** (`isDemoMode()`, mirroring `demo:reset`/`demo:restore`'s gate direction) so a destructive auth reset can never run on the shared demo node. Errors on no-action / no-matching-user; prints the temp password once for the operator to relay.

### Phase 2 — mTLS node endpoint

- `app/Http/Controllers/Api/Fleet/RecoveryController.php` — `POST /api/admin/recover`, calling the same service with `path = endpoint`. Mirrors `BackupController`'s discipline: **always 200** on application-level conditions with the outcome in a `status` envelope (`{contract_version, status, email, actions_applied, temporary_password, recovered_at, message}`), `BackupController`-style `sanitise()` on the exception path. Validation failure, no-matching-admin, and unexpected exceptions all return a `200 failed` envelope (never rely on HTTP status alone). Uses the shared `HasContractVersion` trait.
- `routes/api.php` — the route joins the FM group behind `throttle:6,1` (tight, matching `/api/backup/trigger` — admin recovery is sensitive and should never see burst traffic).
- `docker/nginx/default.conf` + `docker/nginx/prod.conf` — a `location = /api/admin/recover` strict mTLS gate (`if ($ssl_client_verify != "SUCCESS") { return 403; }`) added to both, and the gate-comment lists updated (four → five endpoints). nginx recreated via `up -d --force-recreate nginx` (single-file bind-mount). **No app-layer auth** — "trust the connection": identity is verified out-of-band against the operator's external-vault recovery PIN before the request is made; no recovery secret is stored, hashed, or verified in the CRM or FM.

## Security considerations

This session *is* a security/operational control, so the review is the work. The endpoint mutates an admin's authentication with **no app-layer identity check** by design — it rides the existing nginx-terminated mTLS "trust the connection" gate (the established pattern for all FM endpoints), and the operator verifies identity out-of-band against an external-vault PIN before triggering. **No recovery secret is stored, hashed, or verified anywhere** (no PIN column, no hash, no verify endpoint) — the conscious procedure-not-control tradeoff, accepted for the solo-operator trust model; the audit records "operator reset … for user Y," not "…after PIN verification." The one deliberate exception to the contract's "no user secrets in any response" rule is the one-time `temporary_password`, which transits **once** over the mTLS-encrypted channel and is never stored in plaintext (only the hash); it is documented as such in the contract and flagged for FM-side show-once/never-log handling. The break-glass CLI hard-gates to refuse in demo mode so a destructive reset can't reach the shared demo node. The protected super-admin is intentionally recoverable (reset ≠ delete). Every reset is audited (who-by-path/what/when) through the app's activity log.

**Audit-sink decision — flipped mid-build on a discovered bug.** The Open Gate recommended Spatie's `activitylog`. Implementation surfaced that the app does **not** use Spatie's `activity()` helper at all — it has its own `App\Services\ActivityLogger` → `App\Models\ActivityLog` → table `activity_logs` (custom `actor_*`/`subject_*`/`meta` columns, written by every CRM observer). Spatie's `config/activitylog.php` is vestigial **and** its `table_name` default was the missing-`s` `activity_log` (a non-existent table) — a latent landmine. Per the owner's direction ("fix the bug, don't work around it"), the config default was corrected to `activity_logs`, and the recovery audit rides the app's real `ActivityLogger` (subject = the target `User`, event `admin_recovery`, meta `{actions, path}`) — so a reset also surfaces in that user's activity timeline. No dedicated audit table, no migration. Actor resolves to `system` under both the endpoint and CLI (no authenticated guard), correctly reflecting "app sees no identity." The FM `/api/*` surface is otherwise the only boundary touched; the contract bump is additive and forward-compatible with v2.4.0 consumers.

## Deferred decisions

- **Super-admin creation/promotion and force-change-on-next-login** — deliberately omitted (owner-confirmed). The lockout victim already exists, so resetting their auth is the need; creating a new super-admin from nothing is a distinct, riskier capability, and force-change needs a new column + middleware. Revisit only if a "no admin exists at all" case or a forced-rotation policy is wanted.
- **Hashed in-app PIN the node verifies** — the heavier identity-as-control path for when untrusted support staff exist; explicitly not this session.
- **FM control-panel UI + the FM-side call** — FM-repo work, coordinated via the v2.5.0 contract (pending-absorption entry recorded in the Cross-Repo block).
- **`sessions/tracks/fleet-manager-agent.md` status snapshot is stale** (reads v2.1.0 / session 253 — the 263/268/353 contract bumps never refreshed it either; the canonical cross-repo status lives in `session-outlines.md`, which is current).
