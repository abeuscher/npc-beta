## Session-specific deltas

Constraints surfaced since the prior stabilization that bear on this session:

- **A002/2 is a deliberate design decision.** It retired (a) per-instance inline widget CSS/JS emission in `WidgetRenderer::render` and the matching `<style>` / `<script>` blocks in `resources/views/layouts/public.blade.php`, and (b) the dead `$widgetAssets` collection plumbing through `WidgetRenderer::collectAssets`, `PageBlockRenderer::renderLayoutBlock`, `ChromeRenderer::renderLayoutBlock`, `PageController`, `PostController`, and `ChromeRenderer::render*`. The new contract is: widget CSS/JS ships once via the build-server bundle (`public/build/widgets/manifest.json` → the bundle `<link>` and `<script>` in the public layout). **Reverting A002 is not on the table.** If a failing spec asserts on the retired inline emissions, the spec is wrong against the new contract and updates to match — the brief's decision-tree (a). If a failing spec asserts on the bundle-only contract and the bundle is missing for the same reason the prior stabilization session diagnosed (no build server in CI), the spec is structurally CI-dishonest and follows the brief's decision-tree (b) — the same disposition the prior session applied to widget-color-tokens / template-scheme PUBLIC + PREVIEW painted assertions.

## Phases

### Phase 1 — Evidence: establish the read channel, gather the actual digest

No product or test fixes in this phase. The deliverable is a single CI run that publishes a compact, readable result digest the agent can pull via MCP.

- Branch off `main` as `session-A003/2`. (`session-A003/1` is this prep iteration; A003's actual work begins on `/2`.)
- Mirror the methodology from the prior `sessions/playwright-test-fix.md` stabilization's run-#1: temporarily scope the workflow to Playwright-only (`if: false` on `fast` / `slow` / `tests`; drop `playwright`'s `needs: slow`), add an `on: push: branches: ["**", "!ci-logs/**"]` exclusion, capture Playwright's JSON reporter + tee stdout, and publish a compact digest (failing spec list with first error lines, per-spec timings, slowest-N for signature detection) plus any in-container probes (e.g., `storage/logs/laravel.log` tail, `public/build` state, `public/build/widgets/manifest.json` presence) to a `ci-logs/playwright` branch the agent reads via `mcp__github__get_file_contents`. All of this is **temporary plumbing**; mark each piece with a `DIAGNOSTIC (temporary, A003)` comment so the handback strip is mechanical.
- Push and read the digest. Do not propose fixes from inference before the digest lands.

### Phase 2 — Classify each residual

For each failing spec the digest names, decide which decision-tree branch from the open-questions list applies. Evidence first; reasoning recorded inline in the session log (one bullet per failing spec is enough). Ambiguous cases — including any that look architecturally significant (a real product regression in A002's render path; a contract change that affects a non-trivial number of specs; anything that would expand scope beyond stabilization) — get surfaced to the user via `AskUserQuestion` with the diagnosis + recommendation, **not** silently resolved.

### Phase 3 — Apply fixes

One commit per coherent class of fix. For each:

- **Spec adjusted to the new product contract** (decision-tree a): minimal edit; the new assertion captures what the contract actually delivers; reference the post-A002 contract in the test comment so a future reader sees the *why*.
- **Spec deleted as CI-dishonest** (decision-tree b): remove the assertion (or the whole test if its existence depended on the deleted assertion); clean up any imports/helpers that go orphaned; add a `[test-integrity]` line to `sessions/housekeeping-inbox.md` summarising what was removed and why, with the coverage-tradeoff statement (consistent with the prior session's pattern). Do not "skip" as a default — only `describe.skip` when the test still has value but is provably environment-blocked (the `ticket-tier-picker.spec.ts` precedent), and even then prefer deletion-with-housekeeping-note unless the user steers otherwise.
- **Product fix** (decision-tree c): only when evidence points to a real regression in product code (not in A002's retired surfaces). Keep the fix minimal and surgical; matching tests if the fix changes behaviour. **Surface the architectural choice before going deep** — a product fix is non-trivial scope for a stabilization session and the user should bless it.
- **Surfaced to the user** (decision-tree d): clear PM-level summary + recommendation + the option(s) you'd take if redirected.

Verify with one CI run. If it goes green (rc=0, no `unexpected`; retry-recovered `flaky` is acceptable per the project's standard `retries: 1` model and the precedent set by the prior stabilization session), proceed to Phase 4.

- **Reverting A002/2.** The render-path consolidation is a deliberate design decision; if a spec asserts on retired inline emissions, the spec updates.
- **Adding a build server to the e2e stack.** The project's recorded posture is "no external build pipeline in CI"; if a failing spec requires the widget bundle, treat it as CI-dishonest (decision-tree b), not as a license to add `bundleserver` infra.
- **Re-quarantining the suite.** Adding entries to a `QUARANTINED_SPECS` list (or its equivalent) to chase green is forbidden — the prior session's permanent decision was to delete that pattern.
- **Reworking the un-parked-but-non-gating posture of the Playwright job.** It runs after `slow`, non-gating, as designed. Promoting it into `tests.needs` is a separate, deliberate decision and not in scope here.
- **Generalised flake-tuning.** Per-spec `waitForTimeout` extensions / retry bumps are last-resort fixes; evidence must show variance the retry model can't absorb before tuning, and even then the prior session's no-whack-a-mole rule applies.
- **Numbered-session work.** Sessions 305–312 are on the numbered cadence and out of A003's scope.
