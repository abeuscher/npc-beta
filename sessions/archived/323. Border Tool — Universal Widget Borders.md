## Design (settled in the session-322 close discussion)

The owner walked the design at 322 close. The decisions, so this session doesn't relitigate them:

- **One value, one control, several bind targets.** The border *value* and the *control* are universal; only the mount point changes. v1 binds it in the appearance panel (all widgets). Layout/column and in-widget field-type bindings come later.
- **The control is a definitive shape**, like the alignment tool — not a row of checkboxes. A 4-edge box where each side toggles its border independently (lit edge = on), plus a plain **width** number, a single **radius** number, and the app's standard color picker. **No slider, no min/max** on the numbers (owner simplified this explicitly). Solid only.
- **Value model (all concrete, per the project's concrete-values rule):**
  ```
  border: { top: bool, right: bool, bottom: bool, left: bool, width: int(px), color: '#hex', radius: int(px) }
  ```
  "No border" = all four sides `false` — a concrete state, not absence. One width + one color + one radius shared by whatever sides are on (radius is a single value for all four corners in v1).
- **Universal-first is the simpler path, not the harder one** (owner course-corrected here). Margin/padding already live in the shared appearance layer and apply to all widgets with no per-widget code; border belongs in exactly that layer and is the same effort. Scoping it to one widget would *add* complexity (artificial gating), not reduce it.
- **Default-off makes universal shipping safe.** The concrete default is all-sides-off, so lighting the control up on every widget changes nothing visually until an operator turns a side on — no regression on existing widgets.

## What v1 builds

1. **`BorderInput` primitive** (`resources/js/page-builder-vue/components/primitives/`) — the 4-edge toggle box + width number + radius number + reused `ColorPicker`. Mirror the structure/idioms of `NinePointAlignment.vue` and `SpacingInput.vue`. Emits/consumes the concrete border value object.
2. **Mount it in the appearance inspector** alongside padding/margin/background (`components/appearance/SectionLayoutPanel.vue`), so it surfaces for every widget.
3. **Concrete default** in `WidgetDefinition::defaultAppearanceConfig()` — add the `border` block (all sides false, width 0, radius 0, a default color).
4. **Server-side emitter** in `AppearanceStyleComposer::compose()` — emit `border-{side}`, `border-radius`, color/width from the concrete value, only for enabled sides.
5. **Render-surface parity** — the same border output in `ChromeRenderer::buildInlineStyle` and `WidgetPreviewRenderer` (or refactor chrome to delegate to the composer — decide at open gate). The builder preview, the public page, and chrome must agree.
6. **Verify on the Text widget** — turn borders on for Text widgets standalone and **inside layouts**, confirm correct computed render in all three surfaces. Confirm `box-sizing: border-box` so a border doesn't shift layout.

## Open questions to resolve at session start

- **Parity approach:** refactor `ChromeRenderer::buildInlineStyle` to delegate to `AppearanceStyleComposer` (one emitter, less drift risk) vs keep them parallel and add border to each. Lean: delegate if the refactor is contained; otherwise parallel with a shared private helper.
- **Standing Playwright guard:** a computed-`border-*` assertion on a bordered Text widget on a public page. Unlike widget-bundle CSS, inline appearance styles are emitted by the renderer regardless of the e2e stack's no-build posture, so this *is* CI-viable. Lean: yes, one small spec.
- **Radius control affordance:** a single number field (all corners) — confirm the UI doesn't imply per-corner.

## Out of scope (v1)

- Borders *beyond* the outer box: interior / column-to-column rules, borders on elements *inside* a widget. These are per-widget / per-context follow-ups, handled as we reach each widget — **not** v1.
- The in-widget `border` field-type registration (the `alignment`-field precedent) — design the primitive to allow it later, but don't build that bind path in v1 unless it falls out for free.
- Per-side width/color, per-corner radius, and non-solid styles (dashed/dotted).
- Any Fleet Manager / boundary surface (v2.3.0 unchanged).
