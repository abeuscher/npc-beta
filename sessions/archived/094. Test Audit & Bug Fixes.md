# 094. Test Audit & Bug Fixes

Audit the existing test suite for correctness, relevance, and coverage gaps. Fix a page builder inspector bug. After implementation, discuss and codify testing policy for the project.

---

## Design decisions (resolved before starting)

- This is an audit session — the primary output is knowledge and decisions, not new tests.
- New tests may be written to replace broken or dead tests, but expanding coverage is a separate future effort.
- The testing policy discussion happens after the audit and bug fix, when context is fresh.

---

## Phase 1 — Test audit

Read every test file in `tests/`. For each file, answer:

1. **Does it match current infrastructure?** — Do the models, controllers, routes, and relationships it tests still exist and work the way the test assumes?
2. **Is it still testing active behaviour?** — Has the feature been removed, replaced, or substantially refactored since the test was written?
3. **Is it a false-confidence test?** — Does it assert something meaningful, or does it just check for a 200 response without verifying content?
4. **Is it isolated?** — Does it depend on execution order or shared state from another test?
5. **Is it slow?** — Flag any test that consistently takes more than 5 seconds.

Produce a written summary organised by test file. For each file: status (keep / update / remove), issues found, and recommended action.

### Deliverable

A concise audit report. Not a new document — include the findings in the session log.

---

## Phase 2 — Slow test separation

Split slow tests (>5s) into a separate group so they can be run independently. Establish two Pest groups:

- **Default** — fast tests, expected to complete in under 60 seconds. Run at the end of every session.
- **Slow** — integration or browser-hitting tests that take longer. Run only when relevant features are modified.

Verify both groups pass independently.

---

## Phase 3 — Bug fix: inspector stays open after widget deletion

**Bug:** When a widget is deleted from the page builder while its inspector panel is open, the inspector panel remains visible with the deleted widget's controls. It should close automatically when the widget it references is removed.

Read the relevant page builder and inspector Livewire components before implementing. The fix should detect that the inspected widget no longer exists and close the panel.

---

## Phase 4 — Testing policy discussion

After the audit and fixes are complete, discuss with the user:

1. **When should tests be added?** — Define criteria for when a session's work warrants new tests. Not every session needs tests — what's the threshold?
2. **Fast vs slow classification** — How to decide which tier a new test belongs in.
3. **When to run each tier** — When does the fast suite run? When does the slow suite run?

### Deliverable

Update `sessions/template-base-prompt.md` and `sessions/template-session-prompt.md` with the agreed testing criteria. The updated templates should include:

- Guidance on when to add tests during a session.
- How to classify tests as fast or slow.
- When to run each test tier (e.g. fast tests every session, slow tests only when touching related features).

## Out of scope

- Writing new tests for uncovered features (audit identifies gaps; a future session fills them).
- Refactoring test infrastructure (test base classes, factories, helpers).
- Performance optimisation of individual tests beyond the fast/slow split.
