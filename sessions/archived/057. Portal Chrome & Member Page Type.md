# 057. Portal Chrome & Member Page Type

This session makes the member portal feel like a distinct, branded space and gives member pages a proper home in the page builder. It does not build the member dashboard or profile — those arrive in a later session.

---

## Architecture decisions (agreed pre-session)

- **Separate portal layout.** Portal views extend `layouts/portal.blade.php`, not `layouts.public`. They share the same compiled CSS (`public.scss`) so no new build dependency, but the portal header is visually distinct from the public nav.
- **Portal prefix is a SiteSetting.** Stored as `portal_prefix` (default `members`). Changing it may break bookmarked URLs — that is acceptable and documented. Redirect handling is a future concern.
- **Member page type.** Pages of type `member` are served via the portal layout and are only accessible to authenticated, verified portal users. The slug prefix is driven by `portal_prefix`.
- **Scope of this session.** Layout, member page type, portal prefix setting, and `account.blade.php` converted to the portal layout. No member profiles, no dashboard, no widgets yet.

---

## Phase 1 — Portal layout

Create `resources/views/layouts/portal.blade.php`. It must:

- Use the same `@vite(['resources/scss/public.scss', ...])` as the public layout — no new asset bundle.
- Include a portal header distinct from the public site nav. The header must show:
  - A "Member Portal" label (or the site name + "Member Area" — whichever is less noisy)
  - The authenticated member's first name: `auth('portal')->user()->contact->first_name`
  - A logout button (POST to `portal.logout`)
- Use `@yield('content')` for page body.
- Not include the public site nav or footer. Portal pages are a separate context.

The visual differentiation can be minimal — a different header background using Pico's secondary colour, or a simple CSS class on `<body>`. Do not over-engineer the styling. The goal is: when a member logs in they know they are somewhere different.

---

## Phase 2 — Portal prefix SiteSetting

Add `portal_prefix` to `SiteSetting`. Default value: `members`.

Add a **Routing** section to `GeneralSettingsPage` with a single field for now:

| Field | Key | Default | Notes |
|-------|-----|---------|-------|
| Member portal prefix | `portal_prefix` | `members` | alphaDash, required |

Helper text: "The URL prefix for the member portal. Example: 'members' → /members/login. Changing this will break existing bookmarked links."

---

## Phase 3 — Member page type

Add `member` to the page `type` enum in a new migration (alter the enum to include `member` alongside `default`, `post`, `event`).

Update `PageResource`:
- The `type` select includes `member` as an option labelled "Member Page".
- When type is `member`, the slug field is prefixed with `portal_prefix/` and that prefix is locked (read-only). Follow the same pattern used for the `event` prefix lock if one exists; otherwise implement it now for both types.
- Type is locked on edit when the current value is `event`, `post`, or `member` — these are system-managed types. Only `default` ↔ `member` is switchable (but locked once saved as `member` — check with user if this is correct before implementing).

Update `PageController` so pages of type `member` require `auth:portal` and `verified:portal.verification.notice`. Unauthenticated visitors hitting a member page URL are redirected to `/login`.

---

## Phase 4 — Update `/account` to use portal layout

Convert `resources/views/portal/account.blade.php` to extend `layouts.portal` instead of `layouts.public`. Remove the logout button from the account view body — it is now in the portal header.

Update the `/account` route if any middleware changes are needed to match the portal layout's auth requirements.

---

## Phase 5 — Help doc stub

Create `resources/docs/portal-pages.md` following the frontmatter convention of existing docs. Register `portal.account` in the `routes:` array.

## Security checklist

Before closing:
- [ ] Member pages are inaccessible to unauthenticated visitors (redirect to `/login`)
- [ ] Member pages are inaccessible to authenticated but unverified portal users (redirect to `/email/verify`)
- [ ] The portal header renders the authenticated user's first name from `auth('portal')->user()->contact->first_name` only — no other contact data exposed in the layout
- [ ] The portal layout does not include any public nav links that could expose organisation structure to members unintentionally
