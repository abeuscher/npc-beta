# 264. Affiliations Junction — Schema, Migration, Admin UI

Lift the Contact↔Organization relationship from a single `organization_id` FK to a junction-table model supporting multi-employer / multi-role affiliations. Drop `contacts.organization_id` entirely. Add Org admin gap-fill columns (industry / EIN). Land the affiliation editor as a repeater on the Contact admin form and an Affiliated Contacts panel on the Org show page. Defer the soft-credit layer (`donation_credits`) to session 265.

This is the structural half of `release-plan.md` § B1b. Soft-credit half ships at 265. B1b's checkmark drops at 265 close.

---

## Design decisions (resolved before starting)

- **Drop `contacts.organization_id` entirely.** No cached pointer to the primary affiliation, no backwards-compat shim. Reads that need "the contact's primary org" walk through `affiliations()->where('is_primary', true)`. Rationale: pre-Beta-1, no live data, no integrity-cost from a clean schema. Per user sign-off at 264 prompt-time.
- **`affiliations` table shape:**
  ```
  id              bigint pk
  contact_id      bigint fk -> contacts on delete cascade
  organization_id bigint fk -> organizations on delete cascade
  role            text nullable    (free-text — operator-curated)
  start_date      date nullable
  end_date        date nullable
  is_primary      bool default false
  created_at, updated_at
  ```
  Indexes: `(contact_id)`, `(organization_id)`. Partial unique index: `UNIQUE(contact_id) WHERE is_primary = true` (one primary per contact). FK delete cascade is the right call: deleting a contact or org removes the affiliation rows (not orphan them); the deletion-guard at the admin layer surfaces affiliated-contact counts before letting the operator force-delete.
- **No unique constraint on `(contact_id, organization_id)` or `(contact_id, organization_id, role)`.** Operationally useful to allow a contact to hold multiple roles at the same org (Board member + Donor liaison) or a current role plus a historical one (start_date / end_date semantics). Operator discipline owns dup avoidance; v0 doesn't enforce.
- **`role` is free-text, nullable.** v0 has no enum. The admin form Select uses `searchable()->createOptionUsing()` so operator-curated values surface as suggestions across Contacts. No migration of existing role values needed (none exist yet).
- **`is_primary` invariant double-enforced.** Partial unique index (DB layer) + model boot-hook clearing prior primaries on save (write-time invariant maintenance). Either alone is sufficient for the v0 invariant; both is belt-and-suspenders.
- **Deletion semantics — affiliations cascade with parent.** Deleting a contact removes their affiliations. Deleting an org removes affiliations with that org. The deletion-guard at the admin layer surfaces the count ("This organization has N affiliated contacts; force-delete will remove the affiliations") before the cascade fires. No ON DELETE SET NULL — affiliations are bound to both sides.
- **No Org-Org affiliations.** Org parent / subsidiary / fiscal-sponsor relationships stay deferred per `release-plan.md` § B1b out-of-scope. No `org_affiliations` table; no `organizations.parent_organization_id`.
- **Org admin gap-fill — industry + EIN.** Both nullable text columns on `organizations`. Surfaces in the Org admin form alongside type / email / address. Migration is single-purpose; no ancillary cleanup.
- **Importer touch — Contacts importer only.** The Contacts importer's `organization` column-mapping currently writes `Contact.organization_id`. Reroute to create an `affiliations` row with `is_primary = true`. The four B1a Org-as-source sentinels (`__org_donor__` / `__org_member__` / `__org_sponsor__` / `__org_invoice_party__`) stay routed to their transactional FKs — they represent transactional involvement, not affiliation.
- **Org-side panel — promoted in this session.** The existing ellipsis-menu "View affiliated contacts →" link gets promoted to a proper panel on the Org show page (mirroring the Events Sponsored panel from 255). Cheap to implement; lives alongside the structural change rather than carrying forward to a polish session.

## Phase 1 — Schema migration + data migration + Eloquent

**Goal:** `affiliations` table live; `contacts.organization_id` dropped; existing rows migrated; Org gap-fill columns added; Eloquent relations wired.

**Steps:**

1. **New migration** named e.g. `add_affiliations_drop_contact_organization_id_add_org_gaps`:
   - `CREATE TABLE affiliations` per the shape above.
   - `CREATE UNIQUE INDEX affiliations_one_primary_per_contact ON affiliations(contact_id) WHERE is_primary = true`.
   - `ALTER TABLE organizations ADD COLUMN industry text NULL, ADD COLUMN ein text NULL`.
   - **Data migration step (in the same migration's `up()`):** `INSERT INTO affiliations (contact_id, organization_id, is_primary, created_at, updated_at) SELECT id, organization_id, true, NOW(), NOW() FROM contacts WHERE organization_id IS NOT NULL`.
   - `ALTER TABLE contacts DROP COLUMN organization_id`.
   - `down()` drops the table + columns; does NOT attempt to restore `contacts.organization_id`. Per user direction.

2. **`Affiliation` model** at `app/Models/Affiliation.php`:
   - `belongsTo(Contact::class)`, `belongsTo(Organization::class)`.
   - `$fillable = ['contact_id', 'organization_id', 'role', 'start_date', 'end_date', 'is_primary']`.
   - `$casts = ['start_date' => 'date', 'end_date' => 'date', 'is_primary' => 'boolean']`.
   - Boot hook on `saving`: when `is_primary = true` and the row is being saved, mark all other affiliations for the same `contact_id` as `is_primary = false` BEFORE the save lands. Order matters — clear-then-save, not save-then-clear, so the partial unique index doesn't trip during the transition.
   - Apply `EnforcesScrubInheritance` if the verify-at-start grep confirms the trait is used by sibling models in this section of the schema (Affiliation inherits scrub-source from the contact it binds; if the trait is conventional for relationship rows, apply it).

3. **`Contact` model:**
   - Remove `organization()` belongsTo, remove `organization_id` from `$fillable` / `$casts`.
   - Add `affiliations()` hasMany.
   - Add `organizations()` belongsToMany through affiliations (read-side convenience).
   - Add `primaryAffiliation()` accessor returning the `is_primary = true` row or null. Use `hasOne(Affiliation::class)->where('is_primary', true)` for query-side eager-loadability.
   - Add `primaryOrganization()` convenience: `$this->primaryAffiliation?->organization`.

4. **`Organization` model:**
   - Refactor `contacts()` from hasMany (via the dropped FK) to belongsToMany through affiliations. Read-side compatibility — every existing caller of `$org->contacts` continues working but now walks through the junction.
   - Add `affiliations()` hasMany.
   - Add `industry` + `ein` to `$fillable`.

**Pass criterion:** `php artisan migrate` clean against a database with seeded contacts+orgs; affiliations rows present where `contacts.organization_id` was set; `contacts.organization_id` column no longer exists; `migrate:status` clean. Existing tests that asserted against the old relation FAIL at this stage — they get fixed in Phase 6.

## Phase 3 — Contact admin form — affiliations repeater

**Goal:** affiliations editor lives inline on Contact admin form, replacing the standalone Org Select.

**Steps:**

1. Remove the existing `organization_id` Select from the Contact admin form schema.
2. Add a `Repeater` named `affiliations` with the Affiliation relationship binding (`->relationship('affiliations')`). Schema per row:
   - `Select::make('organization_id')` — searchable, with `relationship('organization', 'name')`. Required. `createOptionForm()` if Org-creation-from-here is conventional in the codebase; defer if not.
   - `TextInput::make('role')` — free-text, nullable. If Filament's `->datalist()` is available and the codebase uses it elsewhere, source suggestions from `Affiliation::query()->whereNotNull('role')->distinct()->pluck('role')`. If not, plain TextInput is fine for v0.
   - `DatePicker::make('start_date')`, `DatePicker::make('end_date')` — both optional.
   - `Toggle::make('is_primary')` — default false.
3. Repeater UX:
   - Default collapsed; row preview format: "{Organization name} — {role or 'No role'}". Use `->itemLabel()` for the preview.
   - Reorderable: false (chronological order from start_date carries more semantic weight than manual order; promoting reorder is a v1 question).
   - "Add affiliation" button below the repeater, default Filament shape.
4. Form-level invariant: when saving with multiple `is_primary = true` rows (operator confusion or copy-paste error), the model's boot hook handles the cleanup deterministically. The form does not enforce single-primary at the UI layer — keeps the form simple, lets the model do the work.

**Pass criterion:** Contact admin form renders the repeater; saving a new affiliation persists the row; toggling primary clears prior primaries; manual UI test against a `migrate:fresh --seed` database passes.

**Goal:** the Contacts importer's `organization` column-mapping creates affiliation rows, idempotent under re-import.

**Steps:**

1. Locate the contact-create / contact-update site where the column-map walk currently lands `organization_id` on the Contact (likely `ImportProgressPage::processOneRow` and/or `ImportContactsJob`; the verify-at-start grep produces the canonical list).

## Phase 6 — Test updates

**Goal:** existing tests that asserted against `Contact.organization_id` / `Contact->organization` updated; new tests cover affiliations behavior; data-migration covered.

**Steps:**

1. **Mechanical existing-test updates** — every test fixture or assertion using `Contact::factory()->create(['organization_id' => $org->id])` migrates to creating an Affiliation row instead. Recommend a `Contact` factory state `->withPrimaryAffiliation($org)` that handles the boilerplate (creates the contact, then creates an affiliation with `is_primary = true`).
2. **Data migration test.** The migration runs against the post-migration schema in tests, so a literal "insert with `organization_id` then run migration" test is awkward. Pragmatic shape:
   - Test the migration's data-transformation logic in isolation: extract the SELECT-INSERT step into a static method on the migration class; call it from a test against a manually-constructed pre-migration shape.
   - OR: a simpler smoke test that asserts post-migration, an affiliations row exists for any contact whose pre-migration organization_id was set — exercised by seeding the test database with a known fixture-shape. Pick whichever fits the codebase's existing migration-test conventions.
3. **`Affiliation` model tests** — relationship loads (both directions), primary-toggle invariant (set primary on row B; row A's primary should be cleared on save), partial-unique-index enforcement (raw INSERT of a second `is_primary = true` row for the same contact raises `QueryException` — bypasses the model's boot hook).
4. **Contact admin form Livewire test** — render form, fill repeater with two affiliations, submit, assert two rows persist; toggle primary on second row, save, assert first row's primary is cleared.
5. **Importer test** — covered in Phase 5 pass criterion; lift into `tests/Feature/Importers/...`.
6. **Org show-page panel test** — render show page with seeded affiliations, assert panel displays them with the right names + roles + primary indicators.
7. **Deletion-guard test** — confirm the new affiliated-contacts count surfaces in the message; force-delete on an org with affiliations cleanly removes the rows via cascade.

**Pass criterion:** fast suite green at +15 to +25 above the 2092 baseline.

## Security checklist

Before closing:

- [ ] **Data migration is forward-only.** No partial-state risk — the column drop happens in the same transaction as the data copy. If the migration fails, the database is left in a known-bad state recoverable via `migrate:fresh`. Per user direction; acceptable pre-Beta-1.
- [ ] **FK cascade is the canonical deletion semantics.** The application-layer deletion-guard surfaces the count *before* cascade fires; once the operator confirms force-delete, the FK takes care of cleanup.
- [ ] **`is_primary` invariant is double-enforced.** Partial unique index (DB layer) + boot-hook clearing prior primaries (model layer).
- [ ] **No new env vars introduced.**
- [ ] **No new privileged endpoints introduced.** All affiliation surfaces are inside the admin authentication scope; no public route gains affiliation-read capability.
- [ ] **Importer rerouting preserves source-policy semantics.** The Affiliation row created by the Contacts importer inherits scrub-source from the contact (or the trait posture confirmed at verify-at-start); the four B1a Org-as-source sentinels remain unchanged.
