# 215. Widget Primitive ‚Äî Phase 3d (Dashboard Config UI & Storage)

Land per-role admin dashboard arrangement. Introduce a `DashboardConfig` owner that reuses `page_widgets` polymorphically (no parallel `dashboard_widgets` table), and reuse the existing Vue page builder in a `dashboard` mode as the config UI ‚Äî the first concrete step of Phase 6 convergence arriving naturally rather than as a future rewrite.

---

## Design decisions (resolved before starting)

- **Reuse `page_widgets`, not a new `dashboard_widgets` table.** The migration doc's "Polymorphic owner discipline" bullet locks this in ‚Äî anything widget-shaped uses the existing tables with new owner types. A cheap `dashboard_configs` indirection is the polymorphic owner; `page_widgets.owner_type = App\Models\DashboardConfig`. Nulls on `layout_id` and column-stacking fields are expected and fine on dashboard rows. No parallel pool of widget-shaped storage.
- **Per-role ownership.** One `DashboardConfig` row per Spatie `Role`. Not per-user (rejected in 214 close ‚Äî no compelling reason for it). Not global. When a user has multiple roles, display the first role's dashboard config by role assignment order; document this clearly, and surface the resolved role in the admin UI so it's not a mystery. This is a deliberately simple rule ‚Äî revisit if it proves painful.
- **Reuse the Vue page builder, not a parallel Filament settings page.** Building a Filament-native config UI would introduce exactly the duplicated config-surface pattern the primitive is trying to eliminate (*"Every configurable surface in the app is configured the same way"* ‚Äî premise doc). Phase 6 convergence starts here, naturally, rather than as a separate later rewrite. Scope is bounded: new Livewire shell + a `mode` field threaded through `BootstrapData` + slot-aware filtering in picker/inspector + a sibling API controller that owns a `DashboardConfig` instead of a `Page`.
- **Picker enforces `allowedSlots()` for the first time.** Today the field is declarative. In `dashboard` mode, the picker filters to widgets whose `allowedSlots()` includes `'dashboard_grid'`. Page-builder mode behavior is unchanged (no filtering ‚Äî canvas is the default and every widget opts in). The change is mode-local; no widget declaration changes.
- **Inspector honors `DashboardGridSlot::layoutConstraints()['allowed_appearance_fields']`.** Only `background` and `text` appearance fields are editable in dashboard mode. Layout fields (padding, margin, full_width) are hidden. Enforced both client-side (inspector) and server-side (API controller rejects writes to disallowed appearance keys).
- **Slot handle stays `dashboard_grid`.** No new slot introduced. `WidgetRenderer::render(..., 'dashboard_grid')` continues to be the rendering path.
- **Fresh-install default.** A `DashboardConfigSeeder` seeds the `super_admin` role's dashboard with the three widgets the hardcoded `widgets()` in 214 ships (Memos + Quick Actions + This Week's Events), in the same order and with the same default configs. Other roles start with empty dashboards until an admin arranges one; the dashboard surfaces an empty-state that says so.
- **Authorization.** A new `manage_dashboard_config` permission, granted to `super_admin` by default. Non-super-admins cannot reach the Dashboard Settings page or hit the config API; asserted by feature tests.
- **No new slot for dashboard resizing.** `layoutConstraints()['dimensions']` remains declarative ‚Äî this session does not implement per-widget width/height sizing. List-ordered arrangement only.
- **No new widget definitions this session.** Memos, Quick Actions, and This Week's Events are the starting eligible set; more widgets can be made eligible by adding `'dashboard_grid'` to their `allowedSlots()` in a later session as the dashboard set grows.

## Phase 1 ‚Äî Schema, model, seeder, render path swap

Schema + Eloquent:

- **Migration** `create_dashboard_configs_table` ‚Äî columns:
  - `id` (uuid, PK)
  - `role_id` (unsignedBigInteger, FK ‚Üí `roles.id`, cascade on delete, **unique** ‚Äî one config per role)
  - `label` (string, nullable ‚Äî for future per-role naming; leave null at seed time)
  - `created_at` / `updated_at`
- `App\Models\DashboardConfig`:
  - `HasUuids`, `HasFactory`.
  - `$fillable = ['role_id', 'label']`.
  - `role()` ‚Üí `belongsTo(\Spatie\Permission\Models\Role::class)`.
  - `widgets()` ‚Üí `morphMany(PageWidget::class, 'owner')` (reusing the existing polymorphism on `page_widgets`).
  - Static helper: `DashboardConfig::forUser(User $user): ?self` ‚Äî resolves the user's first Spatie role, returns its config (or null).
- `app/Filament/Widgets/DashboardSlotGridWidget::widgets()` ‚Äî replace the three hardcoded entries with `DashboardConfig::forUser(auth()->user())?->widgets()->with('widgetType')->orderBy('sort_order')->get()`. Hydrate each returned `PageWidget` unchanged (it already has a `widget_type_id` and `config`). Empty-state handled by the blade: if the list is empty, render a helpful "No dashboard arrangement for your role ‚Äî ask an admin to configure one in Settings ‚Üí Dashboard Settings."

Seeder:

- `DashboardConfigSeeder` (new file):
  - Idempotent. For `super_admin` role, `firstOrCreate` a `DashboardConfig`.
  - If the config is freshly created (no widgets yet), attach three `PageWidget` rows polymorphically: Memos (`limit=5`), Quick Actions (`actions=['new_contact','new_event','new_post']`), This Week's Events (`days_ahead=7`), in that order.
  - Wire into `DatabaseSeeder::run()` **after** `WidgetTypeSeeder` (widget types must exist) and after the role creation loop.
- `MemosCollectionSeeder` remains as-is ‚Äî the memos collection is independent of the dashboard config.

Tests:

- Unit: `DashboardConfigTest` ‚Äî the model, `widgets()` relation, `forUser()` resolution (user with one role, user with no role, user with multiple roles picks the first-assigned).
- Feature: `DashboardConfigSeederTest` ‚Äî the seeder is idempotent, produces three widgets for super_admin on first run, does not duplicate on repeat, is a no-op when the config already has widgets (never clobbers admin arrangement).
- Feature: `DashboardSlotGridTest` ‚Äî update the existing test to run the seeder in `beforeEach` and assert the three widgets render through `DashboardConfig`.

## Phase 2 ‚Äî `DashboardBuilderApiController` (mirrors `PageBuilderApiController`)

`app/Http/Controllers/Admin/DashboardBuilderApiController.php` (new):

- Scopes every operation to a specific `DashboardConfig` (owner). Routes receive `{dashboardConfig}` as a bound parameter.
- Middleware: `auth` + a custom authorization check (`$user->can('manage_dashboard_config')` for every endpoint). Fail-closed 403 otherwise.
- Endpoints (mirror the shape of `PageBuilderApiController`, minus page-only concerns like layouts/nesting):
  - `GET /admin/dashboard-config/{dashboardConfig}/bootstrap` ‚Äî returns `BootstrapData` JSON: the widget-type manifest **filtered to widgets whose `allowedSlots()` includes `dashboard_grid`**, the current config's widgets, the slot's `layoutConstraints`, the current user's role name (for context display).
  - `POST /admin/dashboard-config/{dashboardConfig}/widgets` ‚Äî create a `PageWidget` owned by this config. Reject if the requested widget handle's `allowedSlots()` does not include `dashboard_grid`.
  - `PATCH /admin/dashboard-config/{dashboardConfig}/widgets/{pageWidget}` ‚Äî update a widget's `config` / `appearance_config`. Reject writes to `appearance_config` keys outside `DashboardGridSlot::layoutConstraints()['allowed_appearance_fields']` (currently `background`, `text`). Reject updates to a widget that does not belong to this config (IDOR guard).
  - `DELETE /admin/dashboard-config/{dashboardConfig}/widgets/{pageWidget}` ‚Äî delete; same ownership guard.
  - `POST /admin/dashboard-config/{dashboardConfig}/widgets/reorder` ‚Äî batch sort-order update.
- No layout / nesting endpoints. Dashboard is flat, single-column conceptually (layout is the grid in the blade).

Routes:

- Register in `routes/web.php` (or wherever `PageBuilderApiController` is registered ‚Äî follow existing convention) with the admin middleware group. Match the existing URL convention.

Tests:

- Feature: `DashboardBuilderApiControllerTest` ‚Äî each endpoint hit by a super_admin (happy path) and by a `cms_editor` without the permission (403); a cross-config write attempt fails 404/403; a widget-handle that doesn't allow `dashboard_grid` is rejected on create with a clear error; an `appearance_config.layout.padding` write is rejected in dashboard mode.

## Phase 3 ‚Äî Vue dashboard-mode reuse

`resources/js/page-builder-vue/types.ts` (read first):

- Add `mode: 'page' | 'dashboard'` (or `'page_builder' | 'dashboard'`, match naming to existing conventions) to `BootstrapData`.
- Add slot-specific fields as needed: `allowedAppearanceFields: string[]`, `allowedWidgetHandles: string[]` (or compute from manifest + `allowedSlots`).

`resources/js/page-builder-vue/stores/editor.ts`:

- Read `bootstrap.mode`; switch the `ApiClient` base URL accordingly (page-builder endpoints vs dashboard endpoints). Both clients share the same HTTP verbs and payload shapes ‚Äî no duplicated client code beyond URL prefix.
- In dashboard mode, skip page-only refs (`pageAuthor`, `pageTitle`, etc.) or initialize them to empty. The store's widget CRUD operations stay shape-identical.

`resources/js/page-builder-vue/api.ts`:

- Factor endpoint paths behind a `mode` toggle (or introduce a second client factory `createDashboardApiClient` that wraps the same axios shape with a different URL prefix).

Picker:

- `resources/js/page-builder-vue/components/WidgetPicker.vue` (locate the actual filename) ‚Äî when `bootstrap.mode === 'dashboard'`, filter visible widgets to `bootstrap.allowedWidgetHandles`. Preserve category grouping.

Inspector:

- `resources/js/page-builder-vue/components/InspectorPanel.vue` ‚Äî when `bootstrap.mode === 'dashboard'`, hide appearance tabs/sections for keys not in `bootstrap.allowedAppearanceFields`. Specifically, hide the Layout panel entirely (padding / margin / full_width), and hide appearance keys beyond `background` + `text`.

Preview canvas:

- `PreviewCanvas.vue` (or the root rendering component) ‚Äî when in dashboard mode, render widgets with `slotHandle: 'dashboard_grid'` passed into any preview-render API call. The backend preview renderer already accepts a `$slotHandle` via `WidgetRenderer::render(..., 'dashboard_grid')` ‚Äî confirm the path.

No changes to the core Pinia state shape ‚Äî only mode-aware conditionals.

Tests:

- No Vue unit tests exist in the repo (the project has not adopted a Vue test runner). Vue coverage is indirect via feature tests that hit the controller and via e2e. If the Vue surface grows later and a Vitest/Cypress investment is warranted, that's a separate session. For this session, coverage via:
  - Feature tests that POST through the dashboard API.
  - A Playwright e2e spec that loads the dashboard settings page, adds a widget, reorders, and deletes. Lightweight; must clean up via `afterAll`.

## Phase 4 ‚Äî Filament "Dashboard Settings" admin page

`app/Filament/Pages/DashboardSettingsPage.php` (new):

- Under the Settings navigation group (`$navigationGroup = 'Settings'`), routed at `/admin/dashboard-settings`.
- Permission gate: `public static function canAccess(): bool { return auth()->user()?->can('manage_dashboard_config') ?? false; }`.
- Role selector: a dropdown listing all roles, defaulting to the admin's own role. Changing the role swaps the mounted Livewire shell's `dashboardConfigId`. If the chosen role has no config yet, the page shows a "Create dashboard for {role}" button that `firstOrCreate`s a `DashboardConfig` row before mounting the builder.
- Mounts `DashboardBuilder` Livewire shell via the page's blade view.
- `getBreadcrumbs()` following the established pattern: `['Settings', 'Dashboard Settings']`.
- Help doc stub at `resources/docs/dashboard-settings.md` with the standard frontmatter.

`app/Livewire/DashboardBuilder.php` (new):

- Mirrors `app/Livewire/PageBuilder.php` structure. Owns a `DashboardConfig` id and exposes it as the Vue bootstrap target.
- Emits the same `data-bootstrap` JSON blob the page builder does, with `mode: 'dashboard'` and the per-slot allowlists.
- Blade view at `resources/views/livewire/dashboard-builder.blade.php` ‚Äî mirrors `page-builder.blade.php`.

Permission seeding:

- `PermissionSeeder` (or wherever permissions are created) ‚Äî add `manage_dashboard_config` and grant it to `super_admin`.

Tests:

- Feature: `DashboardSettingsPageTest` ‚Äî super_admin can load, cms_editor cannot (403/redirect), breadcrumbs correct, role selector renders, creating a config for a role without one works.

## Out of scope

- **Per-user dashboards.** Decided: per-role only. A per-user layer, if ever needed, is a follow-on.
- **Multi-role users: union/override/precedence.** Decided: first-role-assigned wins. If that proves painful, revisit in a later session with an explicit "default dashboard role" field on the user.
- **Dashboard widget resizing (width / height).** `layoutConstraints()['dimensions']` stays declarative. List-ordered arrangement only.
- **Column stacking on the dashboard.** `column_stackable: false` in the slot's constraints remains enforced; no nested-widget support this session.
- **Migrating the public page builder to the same API controller shape.** That's the full Phase 6 convergence ‚Äî follow-on session. This session only introduces the dashboard sibling.
- **Memos composer widget.** The 214 close conversation surfaced this as an interesting dogfood target; it is a separate session, orthogonal to dashboard config.
- **Record-detail sidebar slot.** Still Phase 5 territory, unchanged.
- **Email-block slot.** Still deferred.
- **Phase 4 widget retrofit.** Still pending.
