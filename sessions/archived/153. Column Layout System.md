# 153. Column Layout System

Replace the column widget with a first-class layout primitive. Columns are layout containers, not widgets — this distinction drives the data model, interaction design, and rendering pipeline. This session introduces the `page_layouts` table, migrates existing column widget data, builds the column interaction model in the Vue editor, and implements the CSS layout inspector controls.

**Reference document:** `docs/editor-migration-plan.md` — read this file in full. Note that session 153's description in that document is outdated — this prompt supersedes it.

**Prior sessions:** Sessions 147–152 built the full Vue editor stack including the preview canvas, inspector, drag-and-drop reordering, and config requirements system. The quadrant-based widget overlay model (grab handle, action menu, click-to-select zones) was introduced in session 152.

## Design decisions (resolved before starting)

- **Columns are not widgets.** They do not live in `page_widgets`. They have their own table (`page_layouts`) and their own model. The `column_widget` widget type is removed.
- **New table: `page_layouts`.** Stores layout containers with their CSS properties. Widgets reference layouts via `layout_id` + `column_index` on `page_widgets`.
- **Adding columns.** Columns are added from the same "+ Add Block" button/location as widgets. The widget picker includes a "Layout" category (or similar separator) with column presets (2-column, 3-column, etc.). Once created, a layout occupies a position in the page flow sequence alongside root widgets.
- **Page flow model.** The page is an ordered sequence of items, where each item is either a root widget or a layout. Both have a `sort_order` in the page flow. Root widgets use `page_widgets.sort_order` with `layout_id IS NULL`. Layouts use `page_layouts.sort_order`. The editor merges both into a single ordered list for display.
- **The column handle.** Each layout has a handle: a 50–100px tall, 100%-wide toolbar-style bar. Its bottom edge sits exactly at the top boundary of the layout's content area. It extends upward from that seam, floating above and overlaying whatever sits above it. It is not in page flow. It becomes visible when the cursor enters the handle's bounding box. No proximity trigger below the seam — this eliminates collision with widget action menus on widgets sitting above a column. Hovering and clicking the handle selects the column layout. An arrow/selector affordance on the left side of the layout, outside the widget area, provides a secondary way to select the column.
- **Widget hit zones.** Each widget's transparent overlay is divided into zones: upper-left quadrant is the grab handle (click-drag to reorder), upper-right quadrant is the action menu (hover reveals toolbar), everywhere else is click-to-select. The action menu wins z-index conflicts with the column handle.
- **Drag and drop across boundaries.** Dragging a widget into a column drops it into the column cell it lands in. Dragging a widget out of a column returns it to full-width page flow. Dragging between columns moves the widget to the target cell. All drag operations use the existing `store.reorderWidgets()` action with updated parent/column data.
- **Column deletion.** Deleting a column layout warns the user that contained widgets will be deleted. The user can drag widgets out of the column first if they want to keep them. No "move to main list" prompt — the drag-out affordance is sufficient.
- **Empty column cells.** A layout with widgets in only some cells renders empty cells as blank slots. No auto-collapse or fill.
- **Column inspector: CSS layout controls.** The inspector for a selected column exposes CSS properties directly, with brief explanatory copy. CSS property names are visible to the user — no abstraction behind proprietary labels. The goal is to put users on rails while teaching them CSS layout.
  - **Top-level toggle:** `display: flex` | `display: grid`
  - **Flex controls:** number of columns, `justify-content`, `align-items`, `gap`, `flex-wrap`, per-column `flex-basis`/width
  - **Grid controls:** `grid-template-columns` (dropdown with common presets: `1fr 1fr`, `2fr 1fr`, `3fr 1fr`, `1fr 1fr 1fr`, etc. plus manual entry), `gap`, `grid-auto-rows`, `align-items`, `justify-items`
- **Data migration.** Existing `column_widget` entries in `page_widgets` are migrated to `page_layouts`. Their children are updated to reference the new layout. The `column_widget` widget type is deleted from `widget_types`.

## Phase 1 — Data model

### 1a. Create `page_layouts` table

| Column | Type | Nullable | Notes |
|---|---|---|---|
| id | uuid | no | PK |
| page_id | uuid | no | FK→pages, cascade |
| label | string | yes | User-visible name |
| display | string | no | default: `'grid'`; values: `flex`, `grid` |
| columns | integer | no | default: 2; number of column slots |
| layout_config | jsonb | no | default: `{}`; CSS properties: `grid_template_columns`, `gap`, `flex_wrap`, `justify_content`, `align_items`, `justify_items`, `grid_auto_rows`, per-column flex-basis array, etc. |
| sort_order | integer | no | default: 0; position in the merged page flow |
| created_at | timestamp | no | |
| updated_at | timestamp | no | |

### 1b. Add `layout_id` to `page_widgets`

Add `layout_id` UUID column (nullable, FK→page_layouts.id, cascade on delete) to `page_widgets`. This replaces `parent_widget_id` for layout membership. Keep `column_index` — it still indicates which column slot.

### 1c. Migration: move existing column widget data

Write a migration that:
1. Creates the `page_layouts` table
2. Adds `layout_id` to `page_widgets`
3. For each `page_widget` where `widget_type.handle = 'column_widget'`:
   - Creates a `page_layout` with the column widget's `page_id`, `sort_order`, `label`, and config mapped to `layout_config`
   - Updates all child widgets (`parent_widget_id = column_widget.id`) to set `layout_id` to the new layout's ID
   - Deletes the column widget row from `page_widgets`
4. Deletes the `column_widget` row from `widget_types`
5. Drops `parent_widget_id` from `page_widgets` (after all data is migrated)

**Run the migration via Docker after writing it.**

### 1d. Create `PageLayout` model

New Eloquent model at `app/Models/PageLayout.php`:
- UUID primary key
- `page()` belongsTo relationship
- `widgets()` hasMany relationship (page_widgets where layout_id = this.id)
- `$casts`: `layout_config` as array

### 1e. Update `PageWidget` model

- Add `layout_id` to fillable
- Add `layout()` belongsTo relationship
- Remove `parent_widget_id` references (relationship, fillable, etc.)

### 1f. Update schema docs

- Create `docs/schema/page_layouts.md`
- Update `docs/schema/page_widgets.md` — remove `parent_widget_id`, add `layout_id`
- Update `docs/schema/README.md` — add `page_layouts`
- Update `docs/schema/widget_types.md` if the column_widget removal changes anything

### 2b. Layout CRUD endpoints

Add to the existing route group under `/admin/api/page-builder`:

- `POST /{page}/layouts` — create a layout (columns, display, label)
- `PUT /layouts/{layout}` — update layout config
- `DELETE /layouts/{layout}` — delete layout and cascade-delete contained widgets
- `PUT /{page}/reorder` — update the existing reorder endpoint to handle mixed widget/layout reordering, including moving widgets into/out of layouts

### 2c. Update `PageBuilder::getBootstrapData()`

Bootstrap data must return the merged page flow (widgets + layouts interleaved). Match the API response shape so the store can consume either.

### 2d. Update public page rendering

`PageController` and the public layout must render layouts as CSS grid/flex containers with their children, instead of rendering a `column_widget` widget type. The layout's CSS properties come from `layout_config`.

### 2e. Update `WidgetRenderer` / preview rendering

Remove column widget rendering logic from `WidgetRenderer`. Add layout rendering — a layout renders as a CSS container div with its children rendered inside column-indexed slots.

### 3a. TypeScript types

Add to `types.ts`:

```typescript
interface PageLayout {
  id: string
  page_id: string
  label: string
  display: 'flex' | 'grid'
  columns: number
  layout_config: Record<string, any>
  sort_order: number
  slots: Record<number, Widget[]>
```

- Add layout CRUD actions: `createLayout`, `updateLayout`, `deleteLayout`
- Update `reorderWidgets` to handle mixed reordering

## Phase 4 — Vue editor: preview & interaction

### 4a. Layout regions in PreviewCanvas

`PreviewCanvas` renders the merged `pageItems` list. For layout items, render a `LayoutRegion.vue` component (new) instead of `PreviewRegion`.

`LayoutRegion.vue`:
- Renders the column handle (50–100px toolbar bar above the content area, overlaying content above)
- Handle is visible on hover within its bounding box
- Left-side arrow/selector affordance for selecting the layout
- Clicking the handle or the arrow selects the layout (opens inspector)
- Renders column slots as a CSS grid/flex container matching `layout_config`
- Each slot contains `PreviewRegion` components for its widgets
- Empty slots render as visible blank areas

### 4b. Widget hit zones (from session 152)

Each `PreviewRegion` overlay is divided into zones:
- Upper-left quadrant: grab handle (cursor: grab, drag to reorder)
- Upper-right quadrant: action menu (hover reveals toolbar with add above/below, copy, move up/down, delete)
- Rest of widget body: click-to-select

### 4c. Cross-boundary drag and drop

Extend the `vuedraggable` setup from session 152:
- Widgets can be dragged from root level into a column slot
- Widgets can be dragged out of a column slot to root level
- Widgets can be dragged between column slots (same or different layout)
- Use SortableJS `group` option to enable cross-list dragging between the root list and each column slot
- On drag end, determine the new parent (root or layout + column_index) and call the reorder API

### 4d. Layout deletion

When deleting a layout, show a confirmation: "Deleting this column layout will also delete all widgets inside it. Drag widgets out first to keep them." Delete button, Cancel button. On confirm, call `store.deleteLayout(id)`.

## Phase 5 — Column inspector

### 5a. `LayoutInspectorPanel.vue`

New component rendered by `InspectorPanel.vue` when a layout is selected (instead of widget inspector fields).

Structure:
- **Header:** layout label (editable, same pattern as widget label) + delete button
- **Display toggle:** `display: flex` | `display: grid` — two-button toggle. Brief explanatory text: "Controls how child widgets are arranged within this layout."
- **Columns count:** number input, min 1, max 12

### 5b. Grid controls (shown when `display: grid`)

- **`grid-template-columns`:** dropdown with presets + manual entry
  - Presets: `1fr 1fr`, `2fr 1fr`, `1fr 2fr`, `3fr 1fr`, `1fr 3fr`, `1fr 1fr 1fr`, `1fr 2fr 1fr`, etc.
  - Manual mode: text input for custom values
  - Brief label: "Defines the width of each column."
- **`gap`:** text input with common presets dropdown (0, 0.5rem, 1rem, 1.5rem, 2rem)
  - Brief label: "Space between columns and rows."
- **`grid-auto-rows`:** text input (e.g. `auto`, `minmax(100px, auto)`, `1fr`)
  - Brief label: "Default height for rows not explicitly sized."
- **`align-items`:** select — `stretch`, `start`, `center`, `end`, `baseline`
  - Brief label: "Vertical alignment of items within their grid cell."
- **`justify-items`:** select — `stretch`, `start`, `center`, `end`
  - Brief label: "Horizontal alignment of items within their grid cell."

### 5c. Flex controls (shown when `display: flex`)

- **Number of columns:** number input (drives how many flex children / conceptual columns)
- **`justify-content`:** select — `flex-start`, `flex-end`, `center`, `space-between`, `space-around`, `space-evenly`
  - Brief label: "How items are distributed along the main axis."
- **`align-items`:** select — `stretch`, `flex-start`, `flex-end`, `center`, `baseline`
  - Brief label: "How items are aligned on the cross axis."
- **`gap`:** text input with presets (same as grid)
  - Brief label: "Space between flex items."
- **`flex-wrap`:** toggle — `nowrap` | `wrap`
  - Brief label: "Whether items wrap to a new line when they overflow."
- **Per-column flex-basis:** for each column slot, a text input for `flex-basis` / width (e.g. `50%`, `200px`, `1fr`-style values)
  - Brief label: "Base width of each column before flex grow/shrink."

### 5d. Live preview

Changes to layout config update the preview in real time. The `LayoutRegion` component applies `layout_config` as inline CSS on the container div. Debounced save to the API (same pattern as widget config saves).

## Out of scope

- Nested layouts (layouts inside layouts)
- Layout templates / presets saved for reuse
- Animation or transition controls on layouts
- Per-widget override of column placement (e.g. `grid-column: span 2`)
- Responsive breakpoint controls for layouts
