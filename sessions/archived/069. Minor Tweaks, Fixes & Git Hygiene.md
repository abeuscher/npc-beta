## Item 2 — MemberResource (read-only Members lens)

Add a read-only `MemberResource` that provides a focused view of contacts who are active members. This follows the same pattern as `PostResource`, which scopes `Page` to `type = 'post'`.

**Model:** `Contact` (same model as ContactResource — no new model).

**Scoping:** The resource query must be scoped to contacts where `isMember()` returns true. Confirm the exact scope by reading the `Contact` model's `isMember()` method before writing the resource.

**Resource behaviour:**
- List view: relevant columns — name, email, membership tier, membership status, member since. Searchable by name and email. Filterable by membership tier and status.
- No create action — members are created by promoting a contact (session 066 flow).
- No edit form — the edit action (and row click) redirects to the corresponding `ContactResource` edit page.
- Delete, restore, force-delete actions follow ContactResource's rules.
- Navigation: place in the CRM group, after Contacts, with an appropriate icon (e.g. `heroicon-o-user-group`).

**Permission:**
- Add a `view_any_member` permission.
- Grant it automatically to any role that already holds `view_any_contact` — do this in a migration or seeder, reading how existing permission grants are handled in the codebase before writing anything.
- `canViewAny()` checks `view_any_member`.

**Help doc:** Create a stub at `resources/docs/members.md` following the existing frontmatter convention.

## Item 6 — CI test pipeline

Add a GitHub Actions workflow that runs the test suite on every push to any branch and on every pull request targeting `main`. The workflow should:

1. Check out the code.
2. Set up PHP (match the version used in the project — read `composer.json` or `docker-compose.yml` to confirm).
3. Install Composer dependencies (`composer install --no-interaction --prefer-dist`).
4. Copy `.env.example` to `.env` and run `php artisan key:generate`.
5. Run `php artisan test`.

Place the workflow file at `.github/workflows/tests.yml`. Read the project's existing `.env.example` and `docker-compose.yml` before writing the workflow to ensure the PHP version and any required environment variables are correct.

If the test suite requires a database, configure a PostgreSQL service in the workflow using the same version as the project's Docker config. Use an in-memory SQLite database as a fallback only if that is already how the test suite is configured.
