# 224. Widget Primitive — Phase 4h (Product Pair Retrofit + `WidgetDataResolver` Shell Retirement) — Phase 4 Close

Retrofit the Product pair (ProductCarousel + ProductDisplay) onto the contract layer, retire the entire `WidgetDataResolver` shell along with its remaining `PageContext` consumers (`collection()`, `product()`), and close Phase 4 of the Widget Primitive track. After this lands, every retrofit candidate in `app/Widgets/*` flows through `ContractResolver`; `PageContext` retains only `currentPage`, `currentUser`, and `form()` (the Forms widget has not retrofitted and has no scheduled session).

This is the **first session that introduces both list-shaped and single-row contracts for the same model in one beat**, the **first nested DTO projection** (`prices` as an array of price-row sub-DTOs inside each product row), and the **first new `'product'` arm in both `ContractResolver` and `SystemModelProjector`** since the system began with `post` and `event`. Get the nested-DTO shape right — every Phase 5 record-detail-sidebar widget that wants a parent record + a list of children will adopt it.

## Design decisions (resolved before starting)

- **Two cardinalities for `'product'`, both in this session.** ProductCarousel = `CARDINALITY_MANY` with filters `['order_by' => 'sort_order asc']` plus optional `limit` from the widget's `limit` config field (lifted into the contract's filters at `dataContract($config)` time). ProductDisplay = `CARDINALITY_ONE` with filters `['slug' => $config['product_slug']]`. Resolver dispatches by cardinality first (mirror of 223's `resolveSystemModel()` shape), then by model.
- **ProductCarousel declares `querySettings()` for consistency with every other list-shaped widget** (BlogListing, EventsListing, ThisWeeksEvents, Carousel, BoardMembers, LogoGarden, BarChart, Memos all carry one). `hasPanel: true`, `orderByOptions: ['sort_order' => 'Sort order', 'name' => 'Name (A–Z)', 'created_at' => 'Created', 'updated_at' => 'Updated']`, `supportsTags: false` (the Product model has no `tags()` relation today — Tag's morphMap covers Page, Event, CollectionItem, Organization, Contact only; adding products as a sixth taggable model is a separate scope). The QuerySettings panel renders in the inspector with the four order_by options and the limit/direction knobs; the existing `limit` schema field remains as a per-instance default that the contract's filters carry by default, but the panel's `limit` knob takes precedence at render time via `WidgetRenderer::mergeUserQueryConfig()`.
- **Nested DTO shape: `prices` as an array of fixed-shape sub-rows.** Each `prices[]` row carries `['id', 'label', 'amount']` — the minimum the templates read. Sub-row shape is **fixed by the projector**; the contract language is not extended to support nested-fields declaration this session. Rationale: the templates' read surface for nested rows is small and stable; introducing a recursive contract grammar (e.g. `fields: ['prices' => ['id', 'label', 'amount']]`) is a generalization without a current consumer. **Defer** the nested-fields contract grammar until a second nested case demands it. The flat `fields` whitelist gates `prices` as a single boolean ("does this contract receive prices at all?"); when included, every emit carries the full sub-row shape.
- **Aggregate-derived `is_at_capacity` via `withCount` baked into the resolver.** Mirror of 223's pattern for `Event::isAtCapacity()`. The resolver's `resolveProduct()` and `resolveProductOne()` both add `withCount(['purchases as active_purchases_count' => fn ($q) => $q->where('status', 'active')])` to the base query. The projector compares `$product->capacity > 0 && (int) ($product->active_purchases_count ?? 0) >= (int) $product->capacity` (matching `Product::isAtCapacity()`'s behaviour, with a zero-capacity guard returning `false` to avoid divide-by-zero semantics — the existing model method has no such guard but the contract-side guard is defensive against demo data with `capacity = 0`).
- **Per-request slug-keyed cache for single-row.** Cache key `'product:one:' . $slug`. Mirror of `'event:one:' . $slug`. Two ProductDisplay widgets pointing at the same product on one page hit one query. Per-filter-shape cache for list-shaped: `'product:list:' . sha1(serialize($contract->filters))` (mirror of `resolveEvent()`).
- **No `Product::published()` scope today.** The Product model uses inline `where('status', 'published')->where('is_archived', false)` in `WidgetDataResolver::resolveProducts`. The new resolver arms inline the same pair. **Out of scope**: adding a `published()` scope to `Product`. (Could be done as a one-line addition for cleanliness — leave the call as a Phase 4 close-cleanup decision the user can take or defer.)
- **Field whitelists**:
  - **ProductCarousel** (list-shaped, 6 fields): `['id', 'name', 'description', 'image_url', 'prices', 'capacity']`. `id` is consumed by the `prices` form's hidden `product_price_id` (actually that's `$price['id']` — but `id` is included for parity with future widgets that may key off the parent product). `capacity` is read by no template branch today but is included as a defensive whitelist for any future "X spots remaining" UI; **on reflection**, drop `capacity` from the ProductCarousel whitelist. Final whitelist: `['id', 'name', 'description', 'image_url', 'prices']` — 5 fields.
  - **ProductDisplay** (single-row, 6 fields): `['id', 'name', 'description', 'is_at_capacity', 'prices', 'slug']`. `slug` is on the whitelist for parity with EventRegistration's slug inclusion; the template does not currently key off it but route-helper future proofing matters. **On reflection**, drop `slug` — the template doesn't read it, and the YAGNI call mirrors EventDescription (which omitted `slug`). Final whitelist: `['id', 'name', 'description', 'is_at_capacity', 'prices']` — 5 fields.
  - The two whitelists overlap (both declare `id`, `name`, `description`, `prices`); per-field fail-closed means each widget receives only what it declared. The projector emits the superset.
- **Projector field superset.** `SystemModelProjector::projectProduct(Product)` emits 8 fields: `id`, `name`, `slug`, `description`, `image_url`, `capacity`, `is_at_capacity`, `prices`. `image_url` derived via `getFirstMediaUrl('product_image', 'webp') ?: getFirstMediaUrl('product_image')` (parallel to existing event/post `image` derivations). `prices` derived via `$product->prices->map(...)` returning the fixed-shape sub-rows. `is_at_capacity` derived from the `withCount`-aliased `active_purchases_count`.
- **`description` carries plain text, not raw HTML.** ProductCarousel renders `{{ $product['description'] }}` (escaped); ProductDisplay renders `{{ $product->description }}` (escaped). Both treat `description` as plain text. The projector emits the raw string; templates render with `{{ }}` escaping. **No `InlineImageRenderer` involvement** (unlike EventDescription, which renders raw HTML).
- **Conditional template-side state stays template-side.** ProductDisplay reads `request()->query('checkout')`, `session('waitlist_success')`, `$errors->has('waitlist')`, `$errors->has('checkout')`, `old('email')`, `old('name')`, route helpers. All session/request state. The contract layer does not project these. Templates continue to access them directly.
- **`PageContext::collection()`, `product()`, and their caches retire this session.** `collection()`'s only caller is internal (the cache-write line). `product()`'s only callers are EventDescription (already retired) — wait, no — `product()`'s only caller is ProductDisplay (retiring this session). Re-verify both via repo-wide grep mid-session before deletion (mirror of 217/222/223 patterns). `form()` and `formCache` STAY.
- **`WidgetDataResolver` retires entirely.** The class file deletes. Its three remaining callers:
  - `ProductCarousel/template.blade.php` (retiring this session via the contract retrofit).
  - `PageContext::collection()` (retiring this session in the same beat).
  - `tests/Feature/WidgetDataResolverTest.php` (3 tests for `resolve()` / private-collection / non-existent-handle / `blog_posts` dispatch — file deletes alongside).
- **Sibling test fallout in `ProductCarouselTest.php`.** 3 tests directly call `WidgetDataResolver::resolveProducts(...)` (lines 26–80) and 1 test renders the template via `view('widgets::ProductCarousel.template', [...])` passing the legacy `$collectionData` shape (line 154). The 3 resolver tests delete; the template-render test migrates to `WidgetRenderer::render($pw)` (mirror of EventsListing's session-217 sibling-test refactor pattern). The 2 controller tests (`success_page` validation) and the 2 seeder tests (config schema + total widget count) stay unchanged.
- **`product_carousel`'s `widget_type_collections` API field is already empty.** Asserted in `ProductCarouselTest::it seeder creates product_carousel...`. The 221 `WidgetTypeSeeder` re-run already synced this. No `widget_types.collections` change needed. **No `WidgetTypeSeeder` re-run required this session** — neither Product widget changes its `collections()` declaration (both already `[]`).
- **Slug uniqueness gates the single-row resolver.** `products.slug` has UNIQUE constraint (verified in `database/schema/pgsql-schema.sql`). `Product::where('slug', $slug)->first()` returns at most one row.
- **No `Spatie\MediaLibrary\HasMedia` cast on `prices` rows.** `ProductPrice` is not a media-bearing model. The `prices` sub-DTO carries scalars only.
- **Contract-locked widgets do not declare `collections()`.** System rule from 218; both Product widgets currently declare `collections()` returning `[]` (verified — they don't override the parent default). No removal needed.

## Phase 2 — Contract layer extension

1. Extend `SystemModelProjector` with `projectProduct(Product)`:
   - Field map: `id`, `name`, `slug`, `description` (plain string), `image_url` (Spatie `product_image` URL with webp preference, parallel to event's `image` derivation), `capacity` (int or null), `is_at_capacity` (bool, derived from `withCount`-aliased `active_purchases_count`), `prices` (array of `['id' => ..., 'label' => ..., 'amount' => ...]` sub-rows from `$product->prices->map(...)`).
   - Wire into `projectorFor()` via a new `'product' => fn (Product $p) => $this->projectProduct($p)` arm. The dispatch added in 223 already returns `null` for unknown models; the new arm fits cleanly.
2. Extend `ContractResolver` with both cardinalities of `'product'`:
   - **List-shaped** `resolveProduct(DataContract, &$cache)`:
     - Cache key `'product:list:' . sha1(serialize($contract->filters))`.
     - Query: `Product::where('status', 'published')->where('is_archived', false)->withCount(['purchases as active_purchases_count' => fn ($q) => $q->where('status', 'active')])->with(['media', 'prices'])`.
     - `[$col, $dir] = $this->resolveOrderBy($contract);` and apply `$query->orderBy($col, $dir)` (mirror of `resolveEvent`). The contract carries `'order_by' => 'sort_order asc'` as its default; user `query_config.order_by` is re-validated against `querySettings->orderByOptions` by the helper, falling back to `sort_order` (the source-arm default) on any unknown column.
     - `$this->applyTagFilters($query, $contract);` — no-op when `supportsTags: false`, which is the case for Product. Call it anyway for parity with the other list-shaped arms; if products become taggable in a future session, the call site is already in place.
     - If `$contract->filters['limit']` is set, apply `limit()`.
     - `$cache[$key] = $query->get();`
     - Return `$this->systemModelProjector->project($contract, $cache[$key])`.
   - **Single-row** `resolveProductOne(DataContract, &$cache)`:
     - Read `$slug = (string) ($contract->filters['slug'] ?? '');`. If empty, return `['item' => null]` without DB hit (mirror of `resolveEventOne`).
     - Cache key `'product:one:' . $slug`.
     - Query: `Product::where('status', 'published')->where('is_archived', false)->where('slug', $slug)->withCount(['purchases as active_purchases_count' => fn ($q) => $q->where('status', 'active')])->with(['media', 'prices'])->first()`.
     - Return `$this->systemModelProjector->projectOne($contract, $cache[$key])`.
   - Wire both into `resolveSystemModel()`'s cardinality dispatch (the if/match shape from 223). Both `'product'` arms join the existing `'event'` and `'post'` arms.
3. The contract's `resolve()` outer return shape is unchanged: list-shaped contracts get `['items' => [...]]`, single-row contracts get `['item' => row | null]`. The new `'product'` arm follows the same pattern.
