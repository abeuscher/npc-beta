# 197. Quill Toolbar Visual Language — Reference-Image Pilot

Restyle the Quill editor toolbar to match a supplied reference image, producing light + dark variants — pilot for the reference-image → visual-language workflow that will feed the widget-tightening sessions.

---

## Design decisions (resolved before starting)

- **Scope: visual language only.** No change to Quill's button set in this session. No new modules (Cut/Copy/Paste, anchor, YouTube, special char as seen in the reference are *not* added — reference is style-only). Existing buttons stay.
- **Reference image:** `/mnt/e/Clients/nonprofitcrm/test_import_data/wysiwyg toolbar.jpg`. Two-row compact toolbar, light-grey chrome, light borders, grouped with small gaps, slim dropdown chips for Format/Styles. Target is the **visual feel** — grouping, button chrome, spacing, hover/active states — not pixel match.
- **Two surfaces, one CSS.** Quill v2 + `quill.snow.css` is loaded globally via CDN from `AdminPanelProvider.php` on every admin page. The Filament form field `QuillEditor` and the page-builder Vue `RichTextField.vue` both instantiate the same global Quill. One set of CSS overrides styles both.
- **Light + dark variants required.** Existing dark overrides sit in `public/css/admin.css` lines 844–862. New styles live alongside them — same file, dedicated section — unless a better home surfaces during the session.
- **Iteration workflow: Playwright screenshot loop.** Claude writes styles, drives a Playwright script to a page with a Quill instance (Event edit, Email Template edit, or a page-builder widget with a richtext field), captures a screenshot, compares to reference, iterates. User reviews screenshots in-chat and steers.
- **Pilot-to-pattern: deferred.** If the resulting styles feel like they generalise to other toolbars in the admin (widget inspector controls, page-builder sidebar chrome, etc.), flag candidates for extraction into shared SCSS tokens — but do not extract yet. The user wants to see the Quill outcome first before committing to a site-wide design language. Claude re-raises this question explicitly when the Quill styling lands.
- **Pure CSS change.** No Quill config edits, no behavioural changes, no JS.

## Phase 1 — Reconnaissance: baseline screenshots

Before any styling, capture the current state of the Quill toolbar on at least two surfaces (one Filament form field, one page-builder richtext widget), in both light and dark admin modes. Save them to a temp directory — they're the "before" comparison points for every subsequent iteration.

Use the Playwright harness from session 195/196 for this. A throwaway script (not a committed spec) that navigates, waits for the toolbar, screenshots, and exits is fine.

Also open the reference image via the Read tool at the start of the session and keep it in working context.

---

## Phase 2 — Target CSS surface

Decide where the new styles live:

- **Option A:** Append a `/* ───────── Quill toolbar visual language (session 197) ───────── */` section to `public/css/admin.css` below the existing dark overrides. Keeps all admin-side chrome in one file.
- **Option B:** Create a new `public/css/quill-toolbar.css` (or similar) and register it alongside the Quill CDN snippet in `AdminPanelProvider.php`. Cleaner separation; easier to extract later.

Decision happens in-session — read both candidate files and propose before committing. Default to Option A unless the styles grow large enough that separation is clearly better.

---

## Phase 3 — Light-mode styling pass

Iterate reference → change → screenshot → compare. Focus areas, roughly in order of visual impact:

1. **Toolbar chrome.** Background colour of the `.ql-toolbar.ql-snow` strip. Border radius (top corners of the editor). Subtle separator between toolbar and content area.
2. **Button chrome.** `.ql-toolbar button` — background, border, border-radius, padding, hover/active states. Target: the soft-rounded, lightly bordered button look in the reference.
3. **Group separators.** Quill renders each `[…]` array as a `<span class="ql-formats">` with right-margin. Tune that margin, optionally add a subtle vertical rule between groups to match the reference's "group chip" feel.
4. **Dropdown chips.** `.ql-picker` (used by the header/align/color/background dropdowns). Match the Format/Styles chip shape in the reference — slightly wider, small caret, matching chrome.
5. **Icon weight.** Quill icons are SVG `stroke` paths. Verify stroke weight reads comfortably at the button size; adjust `ql-stroke` width if needed.
6. **Two-row wrap behaviour.** Reference is explicitly two-row. Current config produces a single row that wraps when the container narrows. Don't hard-force two rows — let natural wrap handle it — but verify the wrapped state looks intentional.

Each iteration: write change → screenshot via Playwright → compare → adjust. Expect 3–6 rounds.

---

## Phase 4 — Dark-mode variant

The current dark overrides (`admin.css` 844–862) are inherited colours on top of the snow theme. Rework them to match the light-mode chrome in dark-mode equivalents:

- Dark toolbar background with the same visual hierarchy (slight contrast vs the editor area below it).
- Button chrome adapted — borders, hover, active.
- Dropdown chips in dark form.
- Icon colours (`ql-stroke`, `ql-fill`, `ql-picker` text) remain readable against the new chrome.

Same screenshot loop, in dark mode (toggle admin theme). Keep parity with the light-mode work — a visible dark/light A-B comparison should feel like the same control family.

---

## Phase 5 — Cross-surface verification

With both modes styled, verify on all Quill surfaces:

- Filament form field on `EventResource` (event description + meeting details).
- Filament form field on `EmailTemplateResource`.
- Filament form field on `GeneralSettingsPage`.
- Vue `RichTextField.vue` inside the page builder (any widget with a `richtext` field in its schema — e.g. Text Block).

Each should pick up the styling automatically since all share the global Quill CSS. If any surface looks off — specificity conflict, Vue-scoped styles overriding, etc. — fix the root cause rather than duplicating rules.

---

## Phase 6 — Pilot-to-pattern review

With Quill styled in both modes, re-raise the question to the user:

- Do the resulting styles (button chrome, group gap, hover/active transitions, dropdown chip shape) feel like they could be extracted into shared SCSS variables / a `_toolbar-chrome.scss` partial that other admin toolbars could import?
- If yes: propose a follow-up session for the extraction — don't do it in-session.
- If no: close the session as Quill-scoped, and the widget-tightening sessions start from scratch each time without a shared vocabulary.

This is the one open design question the session carries into close.

## Out of scope

- **Button-set changes.** Cut/Copy/Paste, anchor, YouTube embed, special-character, flag/bookmark buttons from the reference are *not* added this session. Deferred — will be considered in a follow-up if the user wants them.
- **Quill module additions** (custom blots, custom toolbar handlers beyond the existing image upload).
- **Other toolbars** — page-builder inspector, widget control bars, custom-field editor toolbars — all deferred to future sessions (widget tightening or dedicated toolbar-homogenisation sessions).
- **Public-site Quill output styling** (`.ql-align-*` content rules in `resources/scss/_custom.scss`) — only admin-side *toolbar chrome* is in scope; content rendering is unchanged.
- **Pilot-to-pattern extraction** — flagged for a follow-up session if warranted, not executed here.
- **Widget Toolset Tightening — Per-Widget Pass** proper (the `session-outlines.md` stub under Housekeeping & Review). That work starts *after* this pilot lands and the design direction is agreed.
