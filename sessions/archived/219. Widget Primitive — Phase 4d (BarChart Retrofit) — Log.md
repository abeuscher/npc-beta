### Notable findings

- **Stale seeder discipline gap surfaced via user manual testing.** Mid-session, the user reported that the QuerySettings panel still rendered for BarChart (and its `limit:4` had no effect). Investigation revealed: `widget_types.collections` for `bar_chart` was still `["data"]` in the dev DB even after `BarChartDefinition::collections()` was removed in code ŌĆö `WidgetTypeSeeder` had not been re-run. Same issue retroactively affected `carousel` (still `["slides"]` from session 218). Pest's `RefreshDatabase` re-seeds in `beforeEach`, but the dev DB does not get re-seeded automatically. Fix landed: `docker compose exec app php artisan db:seed --class=WidgetTypeSeeder --force`. Both `bar_chart` and `carousel` `collections` columns synced to `[]`. Carry-forward discipline added to the 220 base prompt: re-seed after any retrofit-lock that removes definition methods affecting `widget_types` row shape.
- **Architectural surfacing: query settings on contract-locked widgets is a real gap, not a UX accident.** The user pushed back on the panel-disappearance framing: query knobs (`limit`, `order_by`, `direction`, `include_tags`, `exclude_tags`) are about *how many / which order / which subset* of rows, orthogonal to the contract pattern's *which fields* security boundary. There is no architectural reason `ContractResolver` can't honor user-supplied `query_config`. `limit` is already supported in `$contract->filters` on every source. The retirement direction is: merge `query_config` into `$contract->filters` at the renderer's contract resolution call, extend `resolveWidgetContentType` to honor `direction`/`include_tags`/`exclude_tags`, flip the Vue panel render gate from `widget_type_collections.length > 0` to "this widget has a list-shaped contract." Queued in `session-outlines.md` under **"Phase 4 follower ŌĆö Query settings in the contract layer"** with the keying-convention question, the precedence question (some widget defaults are load-bearing semantics ŌĆö e.g. EventsListing's `date_range: from now`), the strip-mechanism retirement, and the **no-collapse UX rule** for the rebuilt panel (current `<button class="query-settings__toggle">` disclosure is to be removed; fields display directly). Not in scope for 219 or 220.
- **Pre-implementation prompt correction caught a `_media` key mistake.** Initial reading of the session prompt had Test A asserting row keys `['label', 'value', '_media']`, but `WidgetContentTypeProjector::projectRow` only emits `_media` when the contract's content type declares image fields (BarChart has none). Caught and corrected in the prompt before any code was written; Test A asserts `['label', 'value']` only, matching projector behavior.
- **Slot-strip-test whack-a-mole stopped this session.** 218's swap pattern (CarouselŌåÆBarChart) would have continued in 219 (BarChartŌåÆBoardMembers/LogoGarden) and then in 220, only to retire the tests entirely once every widget is locked. The user's "we don't need test coverage on features we are expiring across the board" framing landed the better outcome: delete Test 1, rewrite Test 2 against a contract-locked widget. No further sibling-test slot-strip retargeting expected in any future Phase 4 session.
