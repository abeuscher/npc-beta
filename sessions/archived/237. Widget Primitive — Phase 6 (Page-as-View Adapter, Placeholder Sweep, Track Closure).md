## Design decisions (resolved before starting)

### Decision 1 — Implement Page-as-View now (elective hook → committed)

The track doc carries Page-as-View as a forward hook ("could be never") with a forcing function described as "search across every widget-mounting surface" — which has not arisen. This session **converts the elective into committed work** anyway, not because the forcing function arrived but because leaving an open elective on a closing track is itself unsatisfying. The implementation is cheap (a small adapter), preserves Page's existing public-facing surface unchanged (no schema change, no code-path changes for public rendering), and definitively closes the question. Future capabilities that *do* want "walk every widget surface" inherit the iteration spanning Page+Dashboard+RecordDetail uniformly without further work.

If during implementation it turns out the adapter cannot be cheap (e.g., `widgets()` on Page is materially different from the sibling impls in a way that requires schema work or refactoring), stop and reassess — the alternative is to formally retire Page-as-View as a hook with a closure note rather than implement it badly.

### Decision 2 — Page's `IsView` adapter is read-only and shape-mirroring

`App\Models\Page` already has the storage shape an `IsView` impl needs — root `page_widgets` are owned via the existing `morphMany` relation, the page-builder canvas slot is the natural `slotHandle()`, the slug is a stable handle, and `layout_config`-style data lives on `page_layouts` per layout (not on Page itself). The adapter wraps existing accessors:

```php
public function handle(): string { return $this->slug; }
public function slotHandle(): string { return 'page_builder_canvas'; }
public function widgets(): array
{
    return $this->pageWidgets()
        ->whereNull('layout_id')
        ->where('is_active', true)
        ->with('widgetType')
        ->orderBy('sort_order')
        ->get()
        ->all();
}
public function layoutConfig(): array { return []; }
```

Two notes on the body:

- The existing relation method on `Page` is conventionally `pageWidgets()` already (per the polymorphic-owner discipline). Verify before writing — if the existing method is named differently (e.g. `widgets()` like pre-236 DashboardConfig), the adapter implementation has to navigate that without renaming the existing relation, since callers across the page builder depend on the current name. The session's body is "implement IsView," not "rename Page's widgets relation."
- `widgets()` returns root-only (`whereNull('layout_id')`). Layout-nested widgets reach the renderer via their parent layout, not via the IsView accessor. This mirrors how DashboardView and RecordDetailView's `widgets()` work today (both are flat lists of root widgets).
- `layoutConfig()` returns `[]` because Page doesn't carry View-level layout config — column layouts are per-`PageLayout` row, not per-View. Same shape as DashboardView's `layoutConfig()` returning `[]`.

### Decision 3 — Placeholder widget removal is full retirement

`RecordDetailPlaceholderDefinition` was introduced in 5b as a smoke-test widget that confirmed the record-detail ambient flowed. Phase 5d-1 (Recent Notes) replaced its role on `contact_overview` with a real widget, and 5d-2 / 5d-3 added two more. The placeholder is no longer attached to any seeded View, but the widget *type* remains registered (visible in the widget picker if a user filtered by `record_detail_sidebar`). Sweep:

- Delete `app/Widgets/RecordDetailPlaceholder/` (definition + template).
- Drop the registration line from `app/Providers/WidgetServiceProvider.php`.
- Re-run `WidgetTypeSeeder` — its existing semantics handle orphan removal (verify; if it doesn't, add a one-line migration that deletes `widget_types` where `handle = 'record_detail_placeholder'`). The fast Pest suite running with `RefreshDatabase` will exercise the re-seeded registry on every test, so coverage falls out for free.
- Delete the placeholder's two test files (`tests/Unit/Widgets/RecordDetailPlaceholderDefinitionTest.php` and `tests/Feature/RecordDetailPlaceholderRenderTest.php`).
- Edit `tests/Feature/EditContactRecordDetailViewTest.php` — the case that asserted the placeholder rendered against `EditContact` after Phase 5b can drop. The other cases in that file (footer-widgets registration, no-record empty render, redirect-when-unauth) stay.
- Decrement any widget-count assertions that bumped *up* in 5b — `tests/Feature/ProductCarouselTest.php` and `tests/Feature/WidgetPickerSession119Test.php` per the 5b log. Verify with grep before editing.

The Phase Retrospectives section in `sessions/tracks/widget-primitive.md` carries 6 phases of compressed history (Phases 1, Refinements, 2, 3, 4, 5a, 5b, 5c/5c.5, 5d, 5e). The phase-expiry compression rule was applied at every phase close already, so each entry is already 3–8 lines. The closure pass collapses the entire section into a single milestone-closure block:

- One line each for the load-bearing decisions that survived (source-as-capability for `SOURCE_PAGE_CONTEXT`, the "a+" View framing now closed by Page-as-View, fail-closed read discipline, polymorphic-owner storage, sources as a closed set).
- Carry-forwards: Forms widget retrofit (no scheduled session; would retire `PageContext::form()` and `formCache`), `PageContext` full retirement (post-Forms-retrofit), the per-record-type `RecordContextTokens::TOKENS` expansion deferred since 5b, and the `PageContextTokens` namespace migration from `App\Services` to `App\WidgetPrimitive` (pre-existing inconsistency, mechanical cleanup whenever convenient).

### Decision 5 — Out of scope (named explicitly so they don't drift in)

- **Permission rename `manage_dashboard_config` → `manage_dashboard_view`.** High blast radius (Spatie permissions, role assignments, the `Gate::before` admin bypass logic, the `manage_dashboard_config` literal in tests + middleware). Low product value. Carry forward as a small follow-up if anyone notices the inconsistency at admin-UX review time.
- **Vue concept-name string updates** (`mode: 'dashboard'`, `'owner_type' => 'dashboard_config'` in `DashboardBuilder::getBootstrapData()`, the `ownerType: 'dashboard_config'` JS event payload). All three are decorative — Vue's handler reads only `ownerId`. Leaving them as-is preserves byte-equivalence on the 236 retrofit and keeps this session's diff focused.
- **Three Builder API controllers consolidation** (`PageBuilderApiController`, `DashboardBuilderApiController`, `RecordDetailViewBuilderApiController`). Each has a different ownership shape (slug-routed Page, role-keyed DashboardView, FQCN+handle-keyed RecordDetailView). A shared base class would lose nuance without a clear win.
- **`PageContext` full retirement.** Survives for `currentPage`, `currentUser`, `form()`, `formCache`. Forms widget hasn't retrofitted; nothing forces removal.
- **The route URI segment `{dashboardConfig}` rename to `{dashboardView}` and the corresponding controller parameter rename** (carried over from 236). URLs are byte-equivalent; rename them when convenient, not in this closure session.
