# 084. Widget Render Context

Build a `PageContext` service class that supplies structured, scoped, lazily-resolved data to all widget templates — replacing the ad-hoc variable threading currently done inside `PageController::renderPage()` and resolving the `Blade::render()` variable scoping bug that prevents context-aware widgets from functioning.

---

## Background

`PageController::renderPage()` calls `Blade::render($template, $data)` to render each server-side widget. `Blade::render()` compiles the template string as an anonymous component internally — `$data` is bound to the component's property bag and is never extracted as PHP local variables. Any `@include` inside the template calls `get_defined_vars()` to gather the current scope, finds nothing there, and receives no variables. This is why the blog pager widget cannot see `$currentPage` despite the controller passing it.

The fix is `View::share()`: values shared this way are available in every view and every `@include` for the remainder of the request, regardless of how the render was initiated. The broader architectural fix is a `PageContext` class that makes all widget-accessible data available through a single, typed, lazy object rather than an assortment of ad-hoc variables threaded through `Blade::render()`.

---

## Design decisions (resolved before starting)

- The context class is `App\Services\PageContext`.
- It is exposed in widget templates as `$pageContext` via `View::share()`.
- `View::share()` is called once at the start of `renderPage()`, before the widget loop.
- All data streams on `PageContext` are lazy — each query runs only when first accessed — and memoized for the remainder of the request.
- `$config` continues to be passed per-widget as before. It is instance configuration, not context.
- `PageContext` exposes only public/published data. It must never surface admin-only, private, or unpublished records.
- The portal user accessor (`currentUser`) returns the authenticated portal user only — it is not settable from outside the class.
- The existing `WidgetDataResolver` service remains in place. `PageContext::collection()` delegates to it.
- Session 085 will migrate all remaining widgets to use `$pageContext`. This session updates three pilot widgets only.

## Phase 1 — Build PageContext

Create `App\Services\PageContext` with the following interface:

**Properties (eager, set at construction):**
- `currentPage` — the `Page` model being rendered. `null` outside a page render context.
- `currentUser` — the authenticated portal user (`Auth::guard('portal')->user()`), or `null`.

**Lazy stream methods (memoized on first call):**
- `posts(int $limit = null): Collection` — published pages of type `post`, ordered by `COALESCE(published_at, created_at) DESC`. Applies `$limit` if provided.
- `upcomingEvents(int $limit = null): Collection` — published events where `starts_at >= now()`, ordered ascending. Applies `$limit` if provided.
- `collection(string $handle, int $limit = null): array` — delegates to `WidgetDataResolver::resolve($handle, ['limit' => $limit])`. Returns an empty array for unknown or non-public handles.

**Lazy record methods (per-slug cache):**
- `event(string $slug): ?Event` — returns the published `Event` with the given slug, or `null`. Results are cached per slug.
- `product(string $slug): ?Product` — returns the published `Product` with the given slug, or `null`. Results are cached per slug.

---

## Phase 2 — Refactor PageController

In `PageController::renderPage()`:

- Construct a `PageContext` with the current page at the start of the method, before the widget loop.
- Call `View::share('pageContext', $pageContext)` immediately after construction.
- Remove the per-widget `$collectionData`, `$eventData`, and `$productData` resolution blocks from the loop — this logic is now handled lazily inside `PageContext` when a widget requests it.
- The `Blade::render()` call passes only `['config' => $config]`. The `$pageContext` variable reaches widget templates via `View::share()`.

## Phase 3 — Pilot widget updates

**`text_block`** — no context needed. Verify it renders correctly after the controller refactor. No template changes required; this is a regression check only.

**`blog_pager`** — replace `$currentPage` with `$pageContext->currentPage` throughout. Remove the debug output block added during session 083 diagnosis.

**`blog_listing`** — update to call `$pageContext->posts()` (with any configured limit from `$config`) rather than resolving posts through whatever mechanism it currently uses. Confirm the seeder template and live record are consistent.

After updating each widget: reseed the `WidgetTypeSeeder` to push any template changes to the live DB record.

## Security checklist

Before closing:

- [ ] `PageContext` streams return only published/public records — no draft, private, or admin-only data is reachable from a widget template.
- [ ] `currentUser` is read from the portal guard — it cannot be spoofed or overridden by widget config.
- [ ] `PageContext` has no setter methods — it is constructed once and immutable after that.
- [ ] No widget template can reach outside `$pageContext` and `$config` to access unscoped model data.
