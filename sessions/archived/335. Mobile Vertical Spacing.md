# 335. Mobile Vertical Spacing — Reduce Section Whitespace

Shrink the public site's **vertical** section spacing on tablet and mobile so the operator-set padding/margins stop reading as far too tall on a phone — by emitting those values as scalable custom properties and scaling them down with one tunable ratio, leaving room for the eventual per-breakpoint editor controls.

## The problem (confirm in a real browser first)

On the live mobile public site there is **too much vertical whitespace**: the section padding/margins set in the appearance controls (tuned for desktop) are not reduced at narrow widths, so phones get desktop-sized gaps that look wrong. Drive the site at tablet (~768) and mobile (~375) widths and confirm/measure this before changing anything.

## The mechanism

Today `AppearanceStyleComposer` writes section padding/margin as **literal inline px** (`padding-top:80px`) — and inline always beats any stylesheet rule, which is exactly why nothing can compress them. The fix (mirroring the existing `--np-link-color` pattern in the same composer):

1. **Emit** the vertical spacing as inline **custom properties** (`--np-pad-top`, `--np-pad-bottom`, `--np-mar-top`, `--np-mar-bottom`) instead of literal `padding-*`/`margin-*`.
2. **Consume** them in a **host-layer** rule (`resources/scss/**`) so the property, not an inline declaration, sets the box's vertical padding/margin.
3. **Scale** them with cascading viewport `@media` steps (host CSS): at `≤768px` multiply by the ratio, at `≤576px` multiply again — e.g. `padding-top: calc(var(--np-pad-top) * 0.8)` then `* 0.64`. The multiplier is **single-sourced** (one Sass var / one place) so it's tunable by eye.

Horizontal padding/margin (`left`/`right`) are **left as-is** — only the vertical axis scales.

This is **host CSS keyed off the viewport** — the brief's sanctioned signal for section spacing — so it does not touch widget-interior SCSS and does not collide with the `WidgetStyleBoundaryConsumptionTest` widget-`@media` ban or the 332 container-query rollback. It only *scales* the operator's own values; it never changes layout, counts, or columns.

## Open questions to resolve at session start

1. **Preserving "concrete `0` = no override."** Today the composer emits padding/margin **only when non-zero**; a `0` means "no override — let the widget's intrinsic SCSS spacing apply." A naive host rule `padding-top: var(--np-pad-top)` would force unset widgets to `0` and wipe their intrinsic padding. Decide the approach and note the tradeoff:
   - **(a)** Only emit + scale the **explicitly-set** vertical values; widgets with no override keep their intrinsic SCSS spacing untouched (host rule must apply only where the custom property is present).
   - **(b)** Always emit a **concrete resolved value** per the concrete-values rule, so the host rule + scaling apply uniformly (requires resolving each widget's effective default).
   Lean (a) for a minimal, low-blast-radius stopgap unless (b) is clearly cleaner.
2. **Ratio + breakpoints.** Owner's guess: **tablet (`$bp-md` 768) ×0.8, mobile (`$bp-sm` 576) ×0.8 again (≈0.64 of desktop)**. Confirm the steps and the breakpoint mapping; keep the multiplier single-sourced and tunable.
3. **Surfaces in lockstep.** Confirm the same change is applied to **widgets** (`compose`), **column layouts** (`composeForLayout`), and **chrome** (`ChromeRenderer`) so header/footer/page sections all scale consistently.
