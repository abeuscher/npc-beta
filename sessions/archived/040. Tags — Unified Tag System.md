# 040. Tags — Unified Tag System

The app currently has two separate, partially-built tag models: `Tag` (for contacts/organisations, stores `name` and `color`) and `CmsTag` (for collection items, stores `name` and `slug`). Neither is wired into any content type form. This session consolidates them into a single model, wires tag pickers into all four main content types, and replaces both admin resources with a unified Tag Manager under Tools.

---

## Goals

1. **Consolidate** — migrate both existing systems into a single `tags` table with a `type` discriminator and a `slug`. Remove `CmsTag`, `CmsTagResource`, and the `cms_tags`/`cms_taggables` tables.
2. **Wire tag pickers** — add a tag multi-select with create-on-confirm to the Contact, Page, Post, and Event edit forms, each scoped to its own tag type.
3. **Tag Manager** — replace `TagResource` with a unified resource in the Tools navigation group covering all tag types.
4. **Help doc** — update or create `resources/docs/tags.md`.

## Phase 1 — Consolidation migration

### Schema changes to `tags`

Add two columns to the existing `tags` table:
- `slug` — string, unique, generated by Spatie Sluggable from `name`
- `type` — string, not null, default `'contact'` — enum values: `contact`, `page`, `post`, `event`, `collection`

Backfill `slug` for any existing `tags` rows using `Str::slug(name)`.

### Migrate `cms_tags` data

In the same migration (or a separate seeder-style migration), copy all rows from `cms_tags` into `tags` with `type = 'collection'`, generating a `slug` for each. Then copy all rows from `cms_taggables` into `taggables`, mapping `cms_tag_id` → the new `tag_id` and `taggable_type` / `taggable_id` across.

### Drop old tables

Drop `cms_taggables` then `cms_tags` after the data is moved.

### Update `Tag` model

- Add `HasSlug` (Spatie Sluggable, already used in the project)
- Add `slug` and `type` to `$fillable`
- Add `getSlugOptions()` generating from `name`
- Add relationships: `pages()`, `posts()`, `events()`, `collectionItems()` using `morphedByMany` through the `taggables` pivot — matching the existing `contacts()` and `organizations()` pattern
- Remove `CmsTag` model entirely

### Update Contact and Organization models

Verify that `tags()` is already defined as `morphToMany(Tag::class, 'taggable')`. Add it if missing. Do the same for Page and Event models.

## Phase 2 — Tag pickers on content type forms

Add a tag picker field to each content type's Filament resource form. Use the same pattern for all four:

```php
Forms\Components\Select::make('tags')
    ->multiple()
    ->relationship('tags', 'name', fn ($query) => $query->where('type', 'contact'))
    ->createOptionForm([
        Forms\Components\TextInput::make('name')->required(),
    ])
    ->createOptionUsing(function (array $data) {
        return \App\Models\Tag::create([
            'name' => $data['name'],
            'type' => 'contact',
        ])->id;
    })
    ->searchable()
    ->preload()
```

Apply with the appropriate `type` value:

| Content type | Form resource | `type` value |
|---|---|---|
| Contact | `ContactResource` | `contact` |
| Page (non-post) | `PageResource` | `page` |
| Post (blog) | `PostResource` | `post` |
| Event | `EventResource` | `event` |

Place the tag field in a natural position within each form — after the main content fields, before any advanced/status fields. Follow the existing field placement conventions in each resource.

## Phase 3 — Unified Tag Manager

Replace the existing `TagResource` with an updated version:

- **Navigation group**: Tools
- **Navigation label**: Tags
- **Table columns**: name, slug, type (shown as a badge), count of attached records
- **Filters**: filter by type
- **Edit**: allow changing `name` only (slug auto-updates via Spatie Sluggable)
- **Delete**: Filament's standard delete action. The `taggables` pivot must have `onDelete('cascade')` on `tag_id` so deletion removes all pivot rows automatically. Verify this is set in the migration; add a migration to correct it if not.
- **Create**: allow creating a tag with name and type
- Remove `CmsTagResource` and its Pages entirely

---

## Migrations needed

1. Add `slug` and `type` to `tags` table; backfill `slug`
2. Migrate `cms_tags` rows into `tags` (type=collection), migrate `cms_taggables` rows into `taggables`, drop old tables
3. If `taggables.tag_id` foreign key does not already cascade on delete, add a migration to drop and re-add it with `onDelete('cascade')`

Run after each migration: `docker compose exec app php artisan migrate`
