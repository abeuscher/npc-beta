## What was built

### Phase 1 ‚Äî Simplified column control on the pricing widget (shipped)

The deferred structural repeater control from session 304 is replaced by a far simpler model the user agreed to mid-session: a "Number of columns" dropdown (1‚Äì10), a "Feature rows per column" dropdown (0‚Äì12), and a single "Emphasized column" selector. Each defaults to **`auto`** ‚Äî meaning "fit the existing column / row count" ‚Äî so every pre-existing pricing chart renders byte-identically with no migration. The deployed marketing pricing page render was verified byte-equivalent (3 columns, column 3 emphasised, 5 row-slots per column, identical visible content). Lowering a count hides extras without deleting their data (collapse-and-keep, the user's explicit preference). The `columns` repeater stays in the schema with a new `inspector: false` flag ‚Äî its data path through the save allow-list and recursive richtext sanitiser is unchanged, only its Inspector form rendering is suppressed. The Inspector form acquires that `inspector: false` filter for any future structural-but-not-form fields.

### Phase 2 ‚Äî Inspector cleanup for inline-editable widgets (shipped)

The duplicate text editor in the right-hand panel for the four inline-editable widgets (TextBlock, Hero, ThreeBuckets, PricingChart) is gone. The Inspector now hides `text`/`richtext` content-group fields for any widget whose `inlineEditable()` is true ‚Äî prose is edited on the page, structural settings stay in the panel. The Inspector's `RichTextField` component is unchanged and continues to serve non-inline-editable widgets. The one straggler that wasn't yet inline-annotated (PricingChart's `eyebrow_label`) was annotated so the hide-rule is exact rather than fuzzy.

### Phase 3 ‚Äî Public-vs-builder render conflation split (shipped; latent session-304 condition)

Surfaced during investigation: `WidgetRenderer::render()`'s `$slotHandle` parameter defaulted to `'page_builder_canvas'`, and `PageBlockRenderer` (the public-site render path) was calling `render($pw)` with no slot ‚Äî meaning the public site was rendering through the page-builder canvas branch. The single `$isCanvas` flag was overloaded with two meanings: "this render needs page-context token resolution" (true for both public and builder) and "this is the editable builder, emit editing markers" (only builder). Split into two distinct flags: token resolution remains slot-keyed (needed for both surfaces); inline-editing is now an explicit `bool $inlineEditing = false` parameter, opted in only by `WidgetPreviewRenderer`. Visible effect: the public site stops emitting the dormant `data-config-placeholder` attributes and stops rendering empty wrappers around blank fields; visible content is unchanged. Net side benefit for the upcoming builder/public parity check: the public output is now genuinely just public output, not "public plus leaked builder annotations."

### Phase 4 ‚Äî On-page formatting toolbar (first cut built and discarded)

An initial Vue formatting toolbar was built over the Quill API per the session prompt ‚Äî bold/italic/underline/strike, headings, lists, quote, link with popover, font, clear formatting; floating above the active editor; one-shared-bar / active-editor-handle pattern; selection-preserved buttons. Two failure modes surfaced under manual UX testing: (a) the toolbar disappeared as soon as the editor lost focus, even briefly, before the user could click any control, and (b) when it did appear it parked in the top-left corner with no functionality. Diagnosis: the toolbar was binding to a Quill instance whose DOM had been replaced by an out-of-band preview refresh, so its positional measurement returned (0,0) and its commands operated on a detached editor. Surface symptoms (wrong icons, wrong text-style dropdown shape, missing formatting reflection, ambiguous look-and-feel) on top of that bug made it clear the visual/behaviour layer was under-specified. The candidate was discarded; the user produced a full visual + behaviour design spec ‚Äî saved to **`docs/inline-formatting-toolbar-spec.md`** ‚Äî which is now the implementation contract for the rebuild. The deleted first-cut component, its App.vue mount, the speculative `.ql-font-*` SCSS (font-family is out per spec `M8`), and the unused `inlineToolbarHoldingFocus` flag have all been stripped so the next session starts on a tidy foundation. What was kept from the first-cut effort because it aligns with the spec: useInlineEdit's outside-pointerdown teardown (matches spec `B4`), reconcile-skip on within-widget field-switching (matches `B10`), and the store's active-editor handle stub (extended next session to the full spec `A4` shape).

### Phase 5 ‚Äî Foundational companion changes from the spec (shipped)

Four of the five companion changes the spec lists in ¬ß6 are landed on this branch so the next session opens to a tidy foundation:

- **HTML sanitiser (spec ¬ß6.2):** the `<a>` allow-list extended to permit `target` (strict allow of `_blank`/`_self`) and `rel` (strict allow of token-set `noopener`/`noreferrer`, with duplicates collapsed and disallowed tokens dropped). Eight new Pest tests cover the new behaviour; one pre-existing test that asserted the old strip-on-unknown rule was updated to pick a genuinely-unknown attribute. Without this, the spec's "Open in new tab" link option could not persist ‚Äî the attributes were silently stripped on save.
- **`refreshPreview` suppression centralised (spec ¬ß6.5, new ‚Äî added on review):** `useRefreshPreview` now short-circuits at the entry of `refreshPreview(widgetId)` when its target matches `store.inlineActiveWidgetId`. Before, only the save-echo path checked the flag; other refresh paths (per-region `needsConfig` watcher, direct programmatic calls) didn't, leaving a latent route for the active widget's preview HTML to be swapped out from under a live editor. Centralised here so every refresh path automatically respects it; the legitimate reconciling refresh that fires after teardown is unaffected because `endInlineEdit` clears the flag before calling refresh. Makes the spec's `A13` (primary defence against wholesale HTML replacement) literally true.
- **Bootstrap data additions (spec ¬ß6.3):** the page-builder bootstrap payload now carries `theme_heading_family` and `theme_body_family` (resolved via `TypographyResolver`, falling back through bucket ‚Üí element family ‚Üí `DEFAULT_FAMILY`) for the upcoming text-style menu's per-row WYSIWYG preview; and each `PageRef` in the `pages` array now carries a resolved `url` so the link popover's page picker can populate the URL field on selection. New types in `types.ts`; new store refs `themeHeadingFamily` / `themeBodyFamily` populated from bootstrap with safe defaults.
- **Lucide icon delivery (spec ¬ß6.4):** `lucide-vue-next` installed as a runtime dependency; all 18 toolbar icons named in the spec resolve as tree-shakeable imports.

Deferred to the next session (per the inline-arc structure): the ColorPicker arrow-key grid navigation (¬ß6.1), the active-editor handle refactor to the full `A4` shape, the MutationObserver fallback (`A14`), and the toolbar component rebuild itself.

## Security considerations

The sanitiser change is the only security-axis touchpoint, and it's intentionally strict: `target` and `rel` are now permitted on `<a>`, but the allow-list is closed ‚Äî `target` accepts only `_blank` or `_self` (anything else, including `javascript:` payloads, is stripped); `rel` accepts only tokens from the closed set `{noopener, noreferrer}` (disallowed tokens like `nofollow` are filtered out, duplicates collapsed). Eight new Pest tests pin this. The public-vs-builder render conflation split is the other change that touches output shape, but it removes attributes from public output rather than adding any ‚Äî strictly a reduction in attack surface. Token substitution, the inline-edit raw-write contract from session 304, and the recursive richtext sanitiser are all unchanged. CRM contract remains v2.3.0; no boundary surface touched.
