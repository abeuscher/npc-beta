# 252. CI Test Suite Cascade — Root-Cause & Fix

Find and fix the structural test-isolation bug that produces 86 cascading unique-constraint violations in CI. **Out-of-gate** (not a release-plan working-set entry); lifted ahead of E2 because feature work cannot ship green tests on top of a broken suite.

---

## What we know going in

### The data

User-supplied CI output at [`sessions/test-failures.txt`](test-failures.txt) — 2202 lines. Summary: **86 failed, 1724 passed**. Run completed in 308.37s, exit code 2. **Zero memory signatures** (no `Allowed memory size`, no `Fatal error`, no `exhausted` mentions across the entire file) — confirms session 251's `memory_limit = 256M → 1G` bump did its job for the OOM, but the cascade has a different root cause.

### The pattern

86 failures, **all unique-constraint violations**, distributed across three different tables:

| Table | Constraint | Failing tests |
|---|---|---|
| `site_settings` | `site_settings_key_unique` (`key = blog_prefix`) | `BlogPagerEndToEndTest`, `PostTest`, `TemplateTest`, others |
| `navigation_menus` | `navigation_menus_handle_unique` (`handle = primary`) | `ChromeWidgetTest` |
| `help_articles` | `help_articles_slug_unique` (`slug = contacts` / `donations`) | `HelpArticlePageTest`, `HelpArticleServiceTest`, `HelpSearchTest` |

All three keys are seeded by `database/seeders/DatabaseSeeder` (or its callees — `BasePageSeeder` for `navigation_menus`; `Artisan::call('help:sync')` for `help_articles`). All three seeders use `firstOrCreate` (idempotent — they don't fail when called twice). **The failing tests insert via `create()` (not `firstOrCreate`)** — e.g., [`tests/Feature/BlogPagerEndToEndTest.php:13`](tests/Feature/BlogPagerEndToEndTest.php#L13).

### The contradiction

With `RefreshDatabase` + no autoseed in `phpunit.xml` + no `$seed = true` on the test classes, the DB at the start of each test should be empty (post-`migrate:fresh`, no rows). The failing tests' `create()` calls should succeed because the row doesn't exist. But CI reports the row already exists. **Something is seeding the DB outside the per-test transaction boundary.**

### Five candidate hypotheses (none confirmed, all worth ruling out)

1. **A test uses `DatabaseMigrations` instead of `RefreshDatabase` and runs `db:seed`.** `DatabaseMigrations` doesn't wrap in a transaction; seeded rows persist past test end.
2. **A test explicitly commits inside its transaction** (e.g., `DB::commit()`, an artisan command opening its own connection, an event listener firing a job that opens a separate connection).
3. **CI's `php artisan migrate --force` is doing more than expected** (a service provider's `boot()` writes on first connection; a migration with a data step inserts; a custom command in `bootstrap/app.php`'s `withSchedule()` runs at boot).
4. **PostgreSQL connection-handling quirk** — CI's services-network postgres differs from local docker-compose postgres in connection pooling, statement-cache shape, or `pg_settings` defaults. RefreshDatabase's transaction assumption may break under that.
5. **Pest test ordering in CI vs local differs**, exposing an order-dependent leak that local doesn't trigger. The first failure in CI is `BlogPagerEndToEndTest` — alphabetically very early; what runs *before* it that pollutes the DB?

The investigation phase (Phase 1) rules each in or out methodically.

## Phase 0 — Reproduce locally

Before any code change, reproduce the failure under conditions that match CI:

1. **Match CI's invocation exactly.** CI runs `php artisan test` (no exclude flags, no memory overrides, against `nonprofitcrm_test` DB on localhost-shape Postgres). The local equivalent is `docker compose exec app php artisan test` — same command, runs against `nonprofitcrm_test` (configured via `phpunit.xml`). **Do not use `vendor/bin/pest --exclude-group=slow`**; that's session 251's local pattern and it didn't surface the cascade.
2. **Verify it reproduces.** Expect ~86 failures with the documented pattern. If it reproduces locally, proceed to Phase 1. If it doesn't reproduce locally, the asymmetry is the next investigation thread — local-vs-CI environment difference is itself the bug, and Phase 1 pivots to identifying that difference.
3. **Capture a baseline.** Note the failure count and the first three failing test names; the fix should drop failures to zero against the same invocation.

## Phase 1 — Identify the root cause

Methodically rule each hypothesis in or out. Stop the moment one is confirmed.

### Hypothesis 1: a non-RefreshDatabase test runs `db:seed`

- `grep -rn "DatabaseMigrations\|use Illuminate\\\\Foundation\\\\Testing\\\\DatabaseMigrations\|->artisan('db:seed'\|Artisan::call('db:seed'" tests/`
- For every match, confirm the file's `uses(...)` line includes `RefreshDatabase::class`. If any test uses `DatabaseMigrations` *and* calls `db:seed`, that test is the polluter.
- If one is found, confirm it's the polluter by **excluding it from the test run** and re-running CI's invocation. If failures drop to zero, hypothesis confirmed.

### Hypothesis 2: a test explicitly commits

- `grep -rn "DB::commit\|DB::beginTransaction\|->withoutTransactions\|RefreshDatabaseAlways\|RefreshDatabaseState" tests/ app/`
- Audit any matches in test code. A `DB::commit()` inside a test, an artisan command that opens its own connection, or a job that runs synchronously and writes through a fresh connection would all break RefreshDatabase's wrap.
- If found, confirm by isolating + excluding.

### Hypothesis 3: something seeds at boot

- `grep -rn "DatabaseSeeder\|->seed(\|::seed(\|firstOrCreate.*blog_prefix\|firstOrCreate.*navigation_menus\|help:sync" app/ bootstrap/ database/`
- Check `bootstrap/app.php`'s `withSchedule()` and any `withMiddleware()` / `withProviders()` blocks for hooks that run on test boot.
- Check `app/Providers/*.php` `boot()` methods for any **writes** to `site_settings`, `navigation_menus`, or `help_articles`. (`AppServiceProvider::boot()` reads but does not write — verify.)
- Check every migration in `database/migrations/` for data-insert steps. A migration with a `DB::table(...)->insert(...)` would seed at `migrate --force` time, before any test runs, and the row would persist (migrations run outside RefreshDatabase's transaction wrap).
- **Strong candidate: the help articles.** `Artisan::call('help:sync')` at `DatabaseSeeder.php:167` — if `help:sync` reads from disk and writes via `firstOrCreate`, it's idempotent. But if it runs from somewhere *other* than DatabaseSeeder (e.g., a `bootstrap/app.php` schedule, a console command kernel autodiscovery, a service-provider hook), it could fire at test boot. Verify whether `help:sync` runs anywhere outside DatabaseSeeder.

### Hypothesis 4: Postgres connection quirk

- This is hardest to rule out without a CI reproduction. Lean on Phase 0's local reproduction first; if it reproduces locally, the quirk is environmental-independent and we can rule out CI-specific behavior.
- If it does *not* reproduce locally but reproduces in CI, the next investigation thread is connection-shape diff. Compare `phpunit.xml`'s `DB_*` env block against CI's `tests.yml` env block. Look for `DB_PERSISTENT`, `PDO::ATTR_PERSISTENT`, or any session-level config that might preserve state across tests.

### Hypothesis 5: test-order-dependent leak

- If Phase 0 reproduces locally, run `docker compose exec app php artisan test --order-by=random --random-order-seed=<N>` with several seeds. If failures shift between seeds, ordering matters and the polluter is probabilistic.
- If failures are stable across seeds, ordering is not the variable — back to hypothesis 1–3.

### Reporting findings

When the root cause is identified, **stop and surface to the user before applying the fix**. The fix shape determines scope:

- **One-line trait change** (e.g., add `RefreshDatabase` to one test file): apply directly, proceed to Phase 2.
- **Multiple-test rewrite**: surface scope; the user may want to split into two sessions.
- **Bootstrap rewrite**: surface scope; this likely deserves design discussion before implementation.
- **Migration data-step extraction**: surface; could touch deploy story.

## Phase 2 — Apply the fix

Based on Phase 1's finding. Three likely shapes:

- **Convert a `DatabaseMigrations` test to `RefreshDatabase`.** One-line `use ... RefreshDatabase;` change + `uses(..., RefreshDatabase::class);` in the Pest header. Lowest risk.
- **Change a `create()` to `firstOrCreate()` in a test's `beforeEach`.** Defensive — works around the polluter rather than fixing it. Acceptable only if Phase 1 confirms the polluter is in **production code** (a migration data step, a service-provider boot write) that we can't easily move; otherwise fix the polluter directly.
- **Move a data-insert step out of a migration into a seeder.** If a migration is doing `DB::insert(...)` for a row that should only exist under `--seed`, the migration is wrong. Move the insert to `DatabaseSeeder` (or a dedicated seeder it calls).

**Whichever shape the fix takes:** apply minimally. No test-coverage improvements, no test refactoring, no "while I'm here" cleanup. The goal is **green CI with the smallest possible diff**.
