# 208. Migration Squash & Code Optimization

Collapse the migrations written since the session-181 squash into a fresh schema baseline, and ride alongside it with a broad pre-beta cleanup pass — perf hot-spots surfaced by 206 / 207's refactors, help-docs coverage audit, Playwright importer-spec teardown, and any loose ends that have accumulated since 181.

---

## Design decisions (resolved before starting)

- **Migration squash follows the session-181 procedure exactly.** `schema:dump` → delete all current migration files → verify `migrate:fresh --seed` → update `docs/schema/README.md` squash-note block. No schema changes during the squash itself — schema changes this session happen as **new** migrations written during the optimization phase, committed before the squash runs, and absorbed into the dump alongside everything else.
- **Squash cut-off is today's `database/schema/pgsql-schema.sql` state plus whatever new migrations this session commits.** No edits to historical migrations, no reordering — reordering has no effect on a squash (the dump erases sequence) and editing historical files is wasted effort when the files are about to be deleted.
- **Pre-beta license applies throughout.** No backward-compat shims, no deprecation periods. When the optimization phase identifies a column, method, route, or config file that isn't pulling its weight, cut it. The squash absorbs the cut-migration into the baseline; the old shape disappears from history.
- **Optimization phase is inventory-first.** Produce a categorised table of candidates, present to user, user picks what to apply. Mirrors the session-181 Phase 2 pattern for JS/CSS extraction. Cap: if more than ~10 candidates surface, present the list and let the user narrow before applying. The rest become deferred stubs.
- **Help-docs coverage is an audit, not a content-writing session.** For each admin view (Filament Resources and Pages) lacking a corresponding doc under `resources/docs/`, create a frontmatter stub and register the route(s) in the doc's `routes:` array. Actual body content can stay as a one-liner placeholder — the goal is coverage and routability, not documentation quality.
- **Playwright teardown audit covers all importer specs.** Housekeeping stub calls out events specifically; audit the full importer spec set (contacts / events / donations / memberships / invoice-details / notes / contacts-update-strategy etc.) and add `afterAll` teardown wherever rows persist beyond the feature being exercised. Use the 207 layout-inspector spec as the reference pattern.
- **Iteration strategy.** All phases sequence within a single branch `session-208/1`. Inventory + user picks happen first (no code changes yet). Then each pick is its own logical commit. The squash itself is the final commit before test + manual-test announcement. If scope balloons mid-session the user may bless a `/2` carve-out — but the squash must still land in this session, not be deferred.

## Phase 1 — Inventory

**No code changes in this phase.** Produce four inventories and present them to the user.

### 1a. Migration candidates

List every file under `database/migrations/`. Expected count is ~28 as of session 207's close (count may shift if additional work landed between sessions). For each, capture in a table:

| File | Date | What it does | Post-squash notes (if any) |
|------|------|-------------|---------------------------|

Expected columns of interest during the squash audit: migrations whose `up()` method has a data-phase (backfills, row transforms) that would be orphaned post-squash, because the schema dump captures schema only — not migrated-in data. For each such migration, confirm the corresponding seeder produces the new shape directly, so the fresh-install path doesn't need the data phase. The 207 `add_appearance_config_to_page_layouts` migration is a known example — its data phase is already superseded by the updated `SystemPageSeeder` from 207.

### 1b. Code-optimization candidates

Read the session 206 log, the session 207 log, and `sessions/session-outlines.md` Housekeeping stub. Produce a table of perf-relevant candidates, categorised:

- **N+1 / query-count regressions** — particularly look at the importer-services refactor (206 W6 #2 `ImportSessionActions`, W6 #3 `ImportSessionPreview`) and the polymorphic owner scopes used by the page-builder's `buildTree` path. Trace what eager-loading is declared vs what's accessed.
- **Oversized payloads** — `PageBuilderApiController::buildTree` / `formatWidgetWithPreview` / `formatLayout` are the hot path; after 207 added `appearance_config` to the layout payload these deserve a look. Also check the bootstrap-data shape on `PageBuilder::getBootstrapData()` — anything the Vue app doesn't actually read can come out.
- **Unused indexes / columns** — scan the schema dump for columns with no `Grep` hits outside models and migrations, and for indexes that aren't referenced by any query the app actually runs.
- **Dead code paths** — deprecated methods, unused traits, orphan config files. **Known starting point: the root `tailwind.config.js` flagged as apparent orphan in the session 181 log** (session 181 verified it but didn't delete). Verify it's genuinely unused this time, then cut it. Similar sweep across `resources/js/` and `resources/scss/` for any entry-points that aren't referenced in `vite.config.js`.
- **API-surface leftovers** — post-207 `api.ts` refactor: confirm no module-level API exports remain. Post-206 importer-services refactor: any static methods on models that only exist to delegate to a now-extracted service can go.

This is opportunistic. If 206 / 207 didn't surface real perf concerns (they may not have — both were refactors with byte-equivalent outputs), be honest about that in the table rather than forcing findings.

### 1c. Help-docs coverage

Walk every admin view (Filament Resource + Filament Page classes). For each, check whether `resources/docs/[handle].md` exists with frontmatter registering the view's route. Produce:

| Admin view | Current help doc | Status |
|------------|------------------|--------|

Status values:
- **Complete** — doc exists, routes registered, body content present
- **Stub** — doc exists with frontmatter only
- **Missing** — no doc file at all
- **Routes unregistered** — doc exists but its `routes:` array doesn't cover this view

For any **Missing** or **Routes unregistered** row, Phase 3 creates or updates the doc. Body content is not required — a one-liner placeholder is fine. The goal is that the help-panel affordance in the admin UI resolves to something rather than 404'ing.

### 1d. Playwright importer-spec teardown audit

For each spec file under `tests/e2e/importer/`, inventory what rows the spec creates during its run that persist past `afterAll`. Produce:

| Spec | Rows persisted | Natural output? | Teardown needed |
|------|---------------|-----------------|-----------------|

"Natural output" = the feature under test produces these rows on purpose (e.g. contacts from a happy-path contacts-importer spec). Those stay. "Out-of-band" rows — anything the spec created as scaffolding or incidentally as a side-effect — get deleted in `afterAll` using the pattern from `tests/e2e/page-builder/layout-inspector.spec.ts`.

## Phase 3 — Apply picks

For each approved pick:

- **Schema changes** (drop column, rename column, drop table, add index) go as new migration files in `database/migrations/`. Name them `2026_MM_DD_HHMMSS_description.php`. Run `docker compose exec app php artisan migrate` after writing. Update the relevant `docs/schema/*.md` table file.
- **Code cleanups** (delete unused method, rename service, remove config file) are straight code changes. If the change affects public test surfaces, add / update tests.
- **Help-doc creations** go as new files under `resources/docs/` with frontmatter matching the pattern of existing docs. Route array populated, body content may be a placeholder.
- **Playwright teardown additions** add `afterAll` hooks to the relevant spec files. Verify by running the spec in isolation — `npm run test:e2e -- tests/e2e/importer/FOO.spec.ts` — then confirming the admin UI is clean afterward.

Commit each pick (or logical group) separately so regressions are bisectable. Run the fast test suite after each group.

---

## Phase 4 — Migration squash

Only runs after Phase 3 is complete and all new migrations from the optimization phase have landed.

1. Confirm dev DB is clean: `docker compose exec app php artisan migrate:status` — every migration should be `Ran`.
2. `docker compose exec app php artisan schema:dump` — regenerate `database/schema/pgsql-schema.sql`.
3. Delete every migration file currently under `database/migrations/` — all of them, including any written this session in Phase 3.
4. `docker compose exec app php artisan migrate:fresh --seed`. Confirm:
   - No errors.
   - Seed cycle completes cleanly.
   - Row counts look sane (spot-check users, widget_types, templates, pages, contacts).
5. Update the squash-note block at the top of `docs/schema/README.md`:
   - "Last updated" → today's date.
   - Squash-note date and session number → 208 / today's date.
   - Keep wording otherwise consistent with the existing note.

Pause and announce the squash is complete before moving to Phase 5.

## Testing

- **Slow test groups to run this session:** none by default.
- **New tests expected:** yes, but scoped. Any optimization pick that changes observable behaviour needs coverage (e.g. a dropped column must be proved unused by a test that previously read it being adjusted or deleted; a query-count reduction should ideally have a `DB::getQueryLog()`-style assertion). Pure deletions of already-unused code don't need new tests.
- **Playwright:** required — `npm run test:e2e` must pass before pausing for manual test. The `migrate:fresh --seed` in each spec's `beforeAll` validates the squash end-to-end by construction; any seed breakage will surface here.
- **Widget asset rebuild:** not expected.

## Out of scope

- Feature additions of any kind
- Refactors that don't net-reduce LOC, reduce query count, or tighten the API surface
- Any work the 207 log explicitly deferred to other named stubs (Notes Permissions; Template Editor Record Sub-Navigation; Column Widget UI Improvements)
- Production deploy of the squash — the user handles deploy timing and the one-time schema replay against the prod DB (if any prod DB exists at close time)
- Inline JS / CSS extraction beyond anything that surfaces incidentally during the optimization pass — session 181 already did the comprehensive extraction; this is a follow-up, not a redo
- Widget SCSS/CSS/JS changes unless a cleanup pick touches them directly
- Migration edits or reordering — both are no-ops under squash
