## Process rules for every session

Base-prompt process rules apply in full (`sessions/template-base-prompt.md`). Session-specific notes:

- **Don't oversell.** Every feature the tour claims must be live and demoable. (Confirmed in 337-close planning: Stripe checkout *and* QuickBooks sync are both real, so the Financial step is honest.) If a step would point at a placeholder, cut the step or reword it.
- **Concise beats complete.** The win is a tour a prospect finishes, not a manual. ~1 point per step; 2 only for a heavy feature. Hard cap 12 steps; target ~9. Point-and-invite ("click into a contact"), don't narrate.
- **The tour lives in the admin tab.** The public-site step works by clicking the page's **"View Site"** button — the live site opens in a **new tab** and the tour stays running on the admin tab underneath. **No cross-tab / front-end↔back-end tour continuation** (owner-decided: a neat trick, not worth the cost).
- **Pin your own selectors.** Anchor every step on a `data-tour` attribute *we* inject (via render hooks / widget templates), never on a Filament-generated class or DOM-shape that can churn across Filament upgrades. This is the standing 249 mitigation.
- **Front-end build required.** driver.js + the tour module compile through `resources/js/admin.js` → `npm run build`. Pest does not catch JS/TS errors.
