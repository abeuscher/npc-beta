# 079. Debug Generator ‚Äî Donations, Products & Purchases

Extend the existing `DashboardDebugGeneratorWidget` to support generating products (with prices), purchases, and realistic donations with tax year control and fund/contact linkage, so the Donors page, tax receipt flow, and purchase history can be tested with minimal manual setup.

---

## Design decisions (resolved before starting)

**Environment assumption:**
The widget is only shown when `APP_DEBUG_TOOLS=true`. All generated data is understood to be scrub data. This tool exists solely in environments with no real data ‚Äî it is expected to be used to freely abuse and stress-test the system. Erratic behaviour from wipe operations is acceptable and expected.

**Record linkage ‚Äî existing records only:**
Donations and purchases always link to **existing** contacts, funds, and products. The generator never creates supporting records (contacts, funds, products) as a side effect of generating donations or purchases. If the required records do not exist, the widget returns a feedback message telling the user what to generate first. Recommended workflow: generate contacts ‚Üí generate products ‚Üí generate donations and/or purchases.

**Donations ‚Äî tax year control:**
Add a "Tax Year" input to the widget (numeric, default current year). Generated donations spread `started_at` randomly across that year. If active funds exist in the database, pick one at random per donation; otherwise `fund_id` is null.

**Donations ‚Äî Transaction row:**
Each generated donation also creates one Transaction row (`subject_type = App\Models\Donation`, `subject_id = donation.id`, `contact_id`, `type = payment`, `direction = in`, `status = completed`, `stripe_id = null`, `occurred_at = donation.started_at`). This mirrors what the Stripe webhook produces in production and keeps the Transactions list coherent.

**Purchases ‚Äî contact and product source:**
Use existing contacts; fail with feedback if none exist. Use existing active products that have at least one price; fail with feedback if none exist.

**Purchases ‚Äî Transaction row:**
Each generated purchase also creates one Transaction row (`subject_type = App\Models\Purchase`, `subject_id = purchase.id`, `contact_id`, `type = payment`, `direction = in`, `status = completed`, `stripe_id = null`, `occurred_at = purchase.occurred_at`). Same rationale as donations.

**Products generation:**
Generates standalone products with 1‚Äì3 prices each. Purchases and Transaction records are not created alongside them ‚Äî the admin generates purchases separately.

**Stripe IDs:**
All `stripe_id` / `stripe_session_id` / `stripe_subscription_id` / `stripe_customer_id` fields are left null on all generated records. This is understood and accepted ‚Äî Stripe links in the UI will simply be absent or non-functional.

**Wipe behaviour:**
- Wipe "products" deletes all products and their prices. Associated purchases and transactions are not cascade-wiped (erratic behaviour is acceptable).
- Wipe "purchases" deletes all purchases and their linked Transaction rows.
- Wipe "donations" deletes all donations and their linked Transaction rows.

## Phase 1 ‚Äî New factories

1. Write `ProductFactory` ‚Äî realistic name, slug derived from name, status `active`, random capacity (10‚Äì200), sequential `sort_order`.
2. Write `ProductPriceFactory` ‚Äî label (e.g. "General Admission", "Early Bird"), amount ($10‚Äì$500), `sort_order`.
3. Write `PurchaseFactory` ‚Äî `status` randomised between `active` and `completed`, `occurred_at` within the last year, `amount_paid` matching the linked price. Requires `product_id`, `product_price_id`, and `contact_id` to be provided by the caller ‚Äî the factory does not create these automatically.
4. Improve `DonationFactory` ‚Äî add `started_at` defaulting to a random date in the current year.

No state methods beyond defaults are needed ‚Äî the widget passes overrides directly.

---

## Phase 2 ‚Äî Widget expansion

1. Add `taxYear` public property (int, default `now()->year`) and a "Tax Year" number input in the Blade view (shown for all types, meaningful only for donations).
2. Add `products` and `purchases` to the type dropdown in the Blade view.
3. Add `generateProducts(int $count)` ‚Äî creates N products, each with 1‚Äì3 prices.
4. Add `generatePurchases(int $count)` ‚Äî resolves existing contacts and product/price records, creates N purchase records each with a linked Transaction row. Returns feedback and aborts if contacts or qualifying products are missing.
5. Replace the existing `'donations'` plain factory call with `generateDonations(int $count)` ‚Äî applies year-spread, fund linkage (existing active funds or null), and contact linkage (existing contacts or feedback + abort). Each donation is accompanied by a linked Transaction row.
6. Extend `wipe()` to handle `products` (delete prices then products), `purchases` (delete linked transactions then purchases), and update `donations` wipe to also delete linked Transaction rows.
7. Update the `match` in `generate()` to dispatch all five types. Update the feedback message.

## Out of scope

- Populating the tax year dropdown from existing donation data (plain number input is sufficient).
- Generating event registrations or memberships alongside purchases.
- Cascading purchase/transaction deletion when wiping contacts or products.
- Armageddon button (single-click full system wipe) ‚Äî deferred to a future session.
