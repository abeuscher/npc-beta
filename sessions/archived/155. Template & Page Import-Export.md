# 155. Template & Page Import-Export

Build a backup/restore mechanism for the marketing site so the user can export pages, blog posts, content templates, and the active page template to a JSON file, then re-import them after a `migrate:fresh --seed` cycle and keep working without losing content. This is not a multi-tenant content marketplace — it is a one-install round-trip tool that survives database wipes during ongoing development.

---

## Design decisions (resolved before starting)

- **Framing.** This is a backup/restore mechanism for the user's own working set of content, not a portable cross-install content marketplace. The primary use case is: build the marketing site → export → wipe the DB with `migrate:fresh --seed` → re-import → keep working. Cross-install transfer is allowed but not optimised — most cross-install hard cases (mismatched collections, mismatched widget types, mismatched templates) should fail gracefully into unset state rather than presenting a mapping UI or raising an on-screen error.

- **One file format, multiple entry points.** Single JSON envelope:
  ```json
  {
    "format_version": "1.0.0",
    "exported_at": "2026-04-09T17:00:00Z",
    "source_install_id": "<some stable id>",
    "payload": {
      "templates": [ ... ],
      "pages":     [ ... ]
    }
  }
  ```
  Same shape used for single-record export and bulk export — single-record exports just have one entry in the relevant array. Templates are imported before pages so pages can resolve their `template_id`. `format_version` is a semver string. **Compatibility rule:** the importer accepts any bundle whose **major** version matches the importer's own current major version. A higher minor or patch is allowed (minor changes are additive / compatible). A different major version is rejected with a clear error and no DB writes. This session writes `1.0.0` and reads any `1.x.y`.

- **Pages and Blog Posts share the same code path.** Both are `Page` records (the latter with `type = 'post'`). The Resource UIs are separate (`PageResource`, `PostResource`) but the export/import service is one. Bulk actions appear on both list pages and the import upload accepts a single page, a single post, or a mixed bundle.

- **Templates included in the bundle.** Both content templates (`type = 'content'`, the reusable widget stacks) AND page templates (`type = 'page'`, the chrome — colours, fonts, SCSS, header/footer system pages) round-trip. The default page template is the load-bearing case for the user's `migrate:fresh` workflow: re-importing after a wipe must restore the user's customised header chrome, footer chrome, and brand colours, not leave them at seeder defaults. A page template export includes its associated `_header` and `_footer` system pages as nested page entries in the bundle.

- **Slug collisions on page import = overwrite.** This is the deliberate happy path for the round-trip workflow. If a page with the same slug exists on import, its widgets and layouts are wiped and replaced with the imported ones. The `Page` record itself is updated in-place where possible (preserving its `id`, `author_id`, `created_at`, etc.) to avoid breaking foreign key references. If you must recreate the row, document why in the implementation.

- **Collection handle mismatches on import = graceful fallback.** If a widget references a collection that does not exist on the destination install, the importer leaves that widget's `collection_handle` config field empty. The widget renders its empty/needs-config state on the public site; no error, no on-screen prompt. The import log records a warning. This applies similarly to:
  - **Missing template_id** — fall back to the default page template, log a warning.
  - **Missing widget_type handle** — skip the widget entirely, log a warning. (Should be rare; widget types are seeded, not user-created.)

- **No widget data mapping UI in this session.** Deferred to a future cross-install session along with the auto-match feature mentioned in the original outline.

- **Media handling = Tier 1 (file rewiring, no bundling).** Each media reference in a widget config is exported as a small descriptor recording at minimum the file's `file_name`, `disk`, and disk-relative `path` (verify Spatie's exact field names in Phase 1). Bytes are NOT bundled in the export file. On import, the importer:
  1. Looks for an existing file at that path on the destination disk.
  2. If found, registers it as a new Spatie media row attached to the imported widget (use `addMediaFromDisk` or equivalent — verify the exact API in Phase 1).
  3. Patches the widget's config to reference the new media id.
  4. If not found, leaves the widget's media reference null and writes a warning to the import log. The widget renders its empty state on the public site, not a crash.

  This works for the user's primary workflow because `storage/app/public/` survives `migrate:fresh` (only the DB rows are rebuilt). Tier 2 (bundling file bytes into the export) is a future session and will add to the same envelope, not replace it.

- **Export storage = both browser download (primary) and server-side storage (secondary).** Default Export action downloads the JSON to the admin's browser. A secondary "Save to server" option (or a separate server-side export action) writes to `storage/app/exports/`. A small "Saved Exports" management UI lists files in that directory and offers in-place re-import or delete. This is what makes the round-trip frictionless for the user — they don't have to babysit the file between `migrate:fresh` cycles.

- **Layouts are part of the page widget tree.** The export must encode column layouts (`PageLayout` records from session 153) and their child widgets. Phase 1 must verify whether `PageWidget::serializeStack()` already handles layouts — it was written before layouts existed, then session 153 added layouts and may or may not have updated it. If `serializeStack` does not handle layouts, this session must extend it (or add a parallel exporter that does).

- **No nav linking on import.** Imported pages do not create or update `navigation_items` rows. The user wires nav by hand after import. (Acknowledged limitation; a future session can add it.)

- **No cross-version migration.** Format is versioned but this session writes `format_version: "1.0.0"` and reads any `1.x.y`. Importing any non-`1.x.y` bundle errors out with a clear message and no DB writes. Future sessions that introduce incompatible changes bump the major to `2.0.0`; future sessions that add fields without breaking older importers bump the minor or patch.

## Phase 2 — File format and serializer service

Build the export side as a pure read-only service with no DB writes. Suggested layout, adjust if Phase 1 surfaces existing services to extend instead:

- `App\Services\ImportExport\ContentExporter` — top-level service.
  - `exportPages(array $pageIds): array` — returns the bundle envelope
  - `exportTemplates(array $templateIds): array` — same
  - `exportBundle(array $pageIds, array $templateIds): array` — combined
  - Internally calls helpers for page serialisation, template serialisation, layout serialisation, widget serialisation, and media reference serialisation.
- `App\Services\ImportExport\ImportLog` — collects warnings and notices during a serialise/import operation. Returned to the caller for display in the UI and for writing to a log file. Has methods like `warning(string $message, array $context = [])`, `info(...)`, plus a `toArray()` for serialising.

The serializer must handle:

- Page metadata: title, slug, type, template_id (export as the template's *name*, not its UUID — the importer resolves by name on the destination), status, published_at, meta fields, head/body snippets, custom_fields.
- Root widgets via the existing `PageWidget::serializeStack()` (extend it in this phase if Phase 1 shows it doesn't handle layouts).
- Page layouts and their child widgets, grouped by `column_index`, in their own `slots` structure.
- Widget config — recurse into media references and emit `{file_name, disk, path}` for each (use Spatie's actual field names — verify in Phase 1).
- Template metadata: name, type, definition (for content templates), all chrome fields (for page templates), nested header and footer system pages (for page templates only).

No new tables. No new migrations. The export is computed on demand from existing records.

## Phase 3 — Single-record export/import via Resource actions

UI:
- "Export" header action on `EditPage`, `EditPost`, and the existing `EditPageTemplate` page (the one we touched in session 154). Each downloads a single-record bundle.
- "Import" page-level action on `ListPages`, `ListPosts`, and the Templates list (whatever it's called — confirm in Phase 1). Opens a file upload modal. Accepts a single-record bundle OR a multi-record bundle (the importer treats single-record as a bundle of one).

Backend:
- `App\Services\ImportExport\ContentImporter` — top-level service.
  - `import(array $bundle, ImportLog $log): void` — accepts a parsed bundle envelope, validates `format_version`, walks templates first then pages, applies the rules from the design decisions section.
  - `findOrCreatePageBySlug(string $slug, array $pageData): Page` — overwrite-on-collision behaviour.
  - Skips media rewiring in this phase if it makes Phase 6 cleaner — Phase 6 wires it in. (Or do it inline here and skip Phase 6 — implementer's call, but explain in the log.)

Test exit criterion for this phase: a page with text widgets, column layouts, and one media reference round-trips on the same install with no data loss. Re-importing the same file overwrites the existing page cleanly.

## Phase 4 — Bulk export/import

UI:
- Bulk action "Export selected" on `ListPages` and `ListPosts`. Selecting N records and clicking the action downloads a single bundle file containing all selected records.
- Single "Import bundle" action on the same list pages — same upload modal as Phase 3, accepts both single-record and bulk bundles. (No separate UI from Phase 3's import — it's the same modal.)

Backend:
- `ContentExporter::exportPages(array $pageIds)` already handles arbitrary lists from Phase 2.
- `ContentImporter::import(...)` already handles bundles with multiple page entries.
- Order of operations inside import: templates first, then pages. Within each group, follow the order they appear in the bundle so the user has a predictable result.

## Phase 5 — Page template (chrome) export/import

The page template — colours, fonts, SCSS, header_page_id, footer_page_id — is the load-bearing case for the user's `migrate:fresh` round-trip. Without this phase, re-importing pages after a wipe gives back the seeder default chrome, not whatever the user customised.

- "Export" action on `EditPageTemplate` (added in Phase 3 if not already done). The exported template includes its associated `_header` and `_footer` system pages as nested page entries in the bundle's `pages[]` array.
- "Import" handles a template entry by:
  1. Finding the template by name on the destination, or creating it if missing.
  2. Updating its colour/font/SCSS fields from the bundle.
  3. Importing its nested header/footer system pages via the same page-import path (with overwrite-on-slug for `_header` / `_footer`).
  4. Re-linking `header_page_id` / `footer_page_id` to the now-imported pages.

Test exit criterion: customised colours, fonts, SCSS, AND header/footer widgets round-trip across `migrate:fresh && import`.

## Phase 6 — Media reference rewiring (Tier 1)

For each media reference in an imported widget's config, the importer:

1. Reads the descriptor `{file_name, disk, path}`.
2. Looks up the file at that path on the destination disk via Laravel's `Storage` facade.
3. If found, registers it with Spatie as a new media row attached to the imported widget. Use `addMediaFromDisk` (or whichever API Phase 1 confirmed). Patch the widget's config to reference the new media id.
4. If not found, write a warning to the `ImportLog` and leave the widget's media reference null.

The widget renders its empty / needs-config state when the media reference is null — confirm this in Phase 1 by reading at least one widget that uses media (e.g. logo or hero) so you know what the empty state looks like.

Test exit criterion: a page with a logo widget round-trips on the same install — the logo image still renders after re-import.

## Phase 7 — Server-side exports directory and "Saved Exports" UI

Make the round-trip frictionless by letting the user save exports to the server, not just to their browser:

- Filesystem location: `storage/app/exports/` (verify the disk; this should be the local `app` disk, NOT `public`, so the files aren't web-accessible).
- Naming convention: `{type}-{timestamp}-{slug-or-name}.json` (e.g. `pages-20260409-170000-marketing-bundle.json`).
- Each Export action gets a "Save to server" toggle (or a secondary "Save to server" button) in addition to the browser download.
- New tool page or section: "Saved Exports" — likely under the Tools group in admin nav. Lists files in `storage/app/exports/` with: filename, file size, created_at, "Re-import" button, "Download" button, "Delete" button.
- Re-import reads the file from disk and runs it through the same `ContentImporter::import(...)` path used by the upload form.

This is what closes the loop on the user's workflow: export → migrate:fresh → re-import without ever touching a download folder.

## Phase 8 — Tests

Cover the round-trip cases that matter for the user's actual workflow:

- **Round-trip: page with text widgets only** — export, delete, re-import, assert rendered HTML matches.
- **Round-trip: page with a column layout** — assert layout structure and slot contents are preserved.
- **Round-trip: page with a media reference** — assert the media file is rewired and the widget renders the image after re-import.
- **Round-trip: bulk page bundle** — export N pages, wipe, re-import, assert all N exist with correct content.
- **Round-trip: page template with chrome** — export the default page template (with its `_header` and `_footer` system pages), `migrate:fresh --seed`, re-import, assert the chrome is restored to the customised state, not the seeder state.
- **Slug collision: re-importing the same page overwrites** — assert the existing page id is preserved and widgets are replaced (not duplicated).
- **Missing collection handle on import** — bundle references collection `does_not_exist`, importer leaves config field empty, no exception, warning is in the import log.
- **Missing template_id on import** — bundle references template `Some Custom Template` that doesn't exist, importer falls back to default, warning is in the import log.
- **Missing media file on import** — bundle references media file at path `not/on/disk.jpg`, importer leaves media null, warning is in the import log.
- **Invalid format_version on import** — bundle has `format_version: "2.0.0"`, importer errors out with a clear message, no partial state written. Bundle has `format_version: "1.5.3"`, importer accepts it (minor/patch within the supported major).
- **Server-side exports flow** — write a file to `storage/app/exports/`, list it via the Saved Exports page, re-import, delete.
- **Authorization** — import/export endpoints require `update_page` (for pages/posts) and `update_template` (for templates). Unauthenticated and unauthorised users get 403.

Pest, RefreshDatabase, factory-based setup, no slow group unless something genuinely takes >5s per test (the bulk round-trip test might — measure first).

## Security checklist

Before closing:

- [ ] Import/export endpoints require `update_page` (for pages/posts/system pages) and `update_template` (for templates). Reject unauthenticated and unauthorised users with 403.
- [ ] File upload validates: file type is JSON, file size has a sensible cap (e.g. 50 MB), parsed JSON validates against the envelope schema before any DB writes happen.
- [ ] No SQL injection from imported strings — use Eloquent / query builder, never raw SQL with imported values.
- [ ] No XSS from imported content — widgets that render rich text already escape via Blade; verify the import path doesn't introduce a new bypass.
- [ ] Imported `head_snippet` and `body_snippet` are stored as-is (these are intentionally raw HTML) but only writeable by users with `update_page`.
- [ ] Server-side exports directory (`storage/app/exports/`) is on a non-public disk and not directly URL-accessible.
- [ ] Filenames in the Saved Exports UI are sanitised before display (no HTML injection from a malicious filename written to disk).
- [ ] Re-import does not allow path traversal — never trust paths in the bundle to read files outside `storage/app/public/` (the media disk).
- [ ] The importer's media-rewiring step refuses to read files outside the configured media disk root.

## Out of scope

- **Widget data mapping UI with auto-match** — deferred to a future cross-install session. This session handles missing collections by leaving the config field empty, not by prompting the user to map.
- **Bundling file bytes into the export** — Tier 2. Future session.
- **Cross-install transfer beyond same-install round-trip** — works incidentally to the extent that paths and handles happen to line up, but is not a design goal.
- **Cross-version format migration** — format is versioned but only version 1 is supported.
- **Navigation linking on import** — imported pages do not create or update `navigation_items`. User wires nav by hand.
- **Server-to-server pull / push** — no external API surface.
- **Diff or merge UI** — overwrite is the only collision strategy.
- **Importing into non-system pages from a content template** — content templates already work as-is for new-page-from-template; this session does not change that flow.
- **Onboarding UX, scheduled exports, export retention policies** — all future.

## Acceptance criteria

- Selecting one or more pages on `ListPages`, clicking "Export selected", and getting a single JSON file that re-imports cleanly on the same install.
- Same for `ListPosts`.
- Single-record export/import works from `EditPage`, `EditPost`, and `EditPageTemplate`.
- A page with column layouts round-trips with the layouts intact.
- A page with media references round-trips with the images still rendering on the public site (Tier 1 — files survive `migrate:fresh` because they live in `storage/app/public/`).
- The default page template round-trips with its customised header/footer/chrome intact across `migrate:fresh && import`.
- Server-side exports directory exists, the Saved Exports UI lists files and supports re-import / delete.
- Missing collections, templates, and media files all fall back gracefully without throwing or breaking the import — warnings go into an import log returned to the user.
- Slug collisions overwrite the existing record without duplicating it.
- Fast test suite passes.
- `npm run build` succeeds.
