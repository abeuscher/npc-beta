## What was built

### Item 1 — Image uploader homogenization (commit `f7a4ffd`)

- New `resources/js/page-builder-vue/components/primitives/ImageUploadControl.vue` — the shared image-upload core (click-to-replace preview / dashed upload-block when empty / remove / Browse library). Purely presentational: emits `upload`/`remove`/`browse`; the parent owns the store actions and the uploading display state.
- `appearance/BackgroundPanel.vue` consumes it; the background-only controls (Fit / Alignment / use-page-header) stay in `BackgroundPanel`. The now-unused `idPrefix` prop was dropped (id-based file-input lookup → a component ref) at both call sites (`InspectorPanel.vue`, `LayoutInspectorPanel.vue`).
- `fields/ImageUploadField.vue` routes **image** fields through the shared control; **video** fields keep their bare file input (no Browse), unchanged.
- No change to `useUploadActions.ts`, the backend upload/use-existing/remove routes, or `MediaBrowserModal.vue`. Behaviour-preserving at the user level; the widget image field's DOM converged on the background model (a structural change, owner-approved).
- `media-picker.spec.ts` selectors updated to the shared-control classes (`image-upload-control__*`). Playwright net green: **media-picker 2/2, full-width-matrix 16/16**.

### Item 2 — Widget thumbnail audit (commit `8e57dda`)

Gave widgets that rendered blank picker thumbnails representative demo data via the established `demo*()` hooks (no renderer special-casing), then regenerated their `static.png`. **11 widgets fixed:**

- **Cheap wins:** `Logo` (`demoImages` + a demo-URL template fallback mirroring the Image widget); `WebForm` (`demoConfig` → seeded `newsletter-signup` form); `MapEmbed` (`demoConfig` → a Google Maps place URL); `LogoGarden` (`demoConfig` maps the seeded collection's `image_field`/`name_field` — the dev render only set `collection_handle`, so media never resolved).
- **New seeders:** `EventDescription` / `EventImage` / `EventRegistration` share a new `database/seeders/DemoEventSeeder.php` (one published event + header image) wired via `demoSeeder()`/`demoConfig()`; `ProductDisplay` (`app/Widgets/ProductDisplay/DemoSeeder.php` — a product with two prices + image); `Nav` (`app/Widgets/Nav/DemoSeeder.php` — a fixed-id menu with five links).
- **Stale-only (regeneration alone):** `EventMiniCalendar`, `Table`.
- `WidgetManifestTest`'s pinned `demoSeeder` allowlist updated to include the five new seeder-backed widgets.
- **Left blank, by category:** the 4 portal widgets (`PortalAccountDashboard`, `PortalContactEdit`, `PortalEventRegistrations`, `PortalChangePassword`) render against `auth('portal')->user()`, which the isolated dev render can't supply — **deferred to session 358's `demoContext()` hook.** `RandomDataGenerator` and `SetupChecklist` are dev/admin tools with no standalone preview — **blank-by-design.**
- Out-of-scope thumbnail churn from the audit regen was reverted; the committed diff is the 11 fixed widgets only.

## Security considerations

None beyond standard convention. The demo re-enter affordance is gated on `isDemoMode()` (`app()->environment('demo')`) and points at the existing `demo.enter` route, whose controller already `abort_unless(isDemoMode(), 404)` and throttles per-IP — no new auth surface. The demo dashboard seeds only product-feel widgets the `demo` role already has permission to view (donations/notes confirmed). The widget demo seeders run only via the dev thumbnail-render route and the demo baseline, never in the production seed path. The image-uploader refactor left the backend upload/remove routes untouched.
