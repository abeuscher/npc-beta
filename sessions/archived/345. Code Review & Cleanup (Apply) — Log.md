### /1 — Security + demo batch

- **Flag 344-A (SVG sanitizer).** New `App\Services\Media\MediaSvgSanitizer` runs on `MediaHasBeenAddedEvent` *before* `MediaContentHasher` — so the strip (script / foreignObject / event-handlers / `javascript:` URIs, via the previously-orphaned `SvgSanitizer`) happens pre-hash and the content-addressed relocation sees the cleaned bytes (no CAS re-hash). This is the single media-add seam that covers every upload path (builder API controllers, Filament `SpatieMediaLibraryFileUpload`, `Media::copy()`, importer seeding) — which a route middleware would have missed (Filament uploads bypass the API routes). Unparseable `.svg` → inert empty SVG. Corrected the false "SVGs run through SvgSanitizer" claim in the archived 329 record (log + prompt + base-prompt) — it had zero callers until now.
- **Flag 344-D (demo:restore lock).** `demo:restore` now `Page::query()->update(['locked' => true])` after restoring (mirrors `demo:reset`), so an unlocked-authoring blob can't ship an editable demo.
- **Flag 344-E (BlockDemoUploads).** Added to the dashboard-builder and record-detail-view-builder `uploadAppearanceImage` routes (the single-chokepoint gap; not exploitable today — the demo role lacks the gating permissions).
- Tests: `MediaSvgSanitizerTest` (neutralize/reject), a **slow-grouped** `demo:restore` round-trip asserting the post-restore lock (real backup→wipe→psql-restore), and a `BlockDemoUploads` middleware-attachment assertion in `DemoRoleLockdownTest`.

## Security considerations

This session *closed* the one real security surface the 344 audit surfaced: **Flag 344-A** (raw SVG upload → stored-XSS via direct URL, plus a false documented protection in the 329 record). `MediaSvgSanitizer` now neutralizes every stored SVG at the single media-add seam, before the content hash is computed, covering all upload paths; the false 329 claim is annotated with a correction pointing to 344/345. Two defence-in-depth picks also landed: `demo:restore` now locks pages (344-D), and `BlockDemoUploads` covers the two previously-uncovered builder upload routes (344-E) — neither was exploitable, both restore an invariant. The EventRegistration email-suppression fix (344-C) removes an unintended outbound-email side-effect on imports (no real PII, but historical imports should not email registrants). The convention-drift test adds a standing gate against the inert-dead-code and permission-gate-uniformity classes. No new security surface introduced; the `withoutEvents()` wraps suppress only the side-effect observer (verified UUID assignment + mail suppression via tinker and the parity suite).
