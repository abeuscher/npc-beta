## What was built

### Phase 1 — `DataContract::formatHints` field

- [app/WidgetPrimitive/DataContract.php](app/WidgetPrimitive/DataContract.php): added `public readonly array $formatHints = []` constructor field. Sibling axis to `querySettings`. Per-field validation responsibility lives in the consumer (today: `SystemModelProjector`); the contract itself enforces nothing on hint values.
- [app/Services/WidgetRenderer.php:173-184](app/Services/WidgetRenderer.php#L173-L184): the `mergeUserQueryConfig()` contract-rebuild path now threads `formatHints` through the new `DataContract` it constructs. Without this, any list-shaped widget with both `querySettings.has_panel` and `formatHints` (EventsListing, ThisWeeksEvents, BlogListing, BlogPager) would silently lose its date_format selection whenever the user touched the QuerySettings panel.

### Phase 2 — `DateFormat` extensions

- [app/Support/DateFormat.php](app/Support/DateFormat.php): new `EVENT_TILE_DATE = 'D, M j'` constant ("Sat, May 2"). Fills the dow-prefixed compact-date gap that `EVENT_FULL` (year + time) and `LONG_DATE` / `MEDIUM_DATE` (no dow prefix) didn't cover for tile presentation.
- New static helpers `eventDateOptions()` and `postDateOptions()` returning curated `[constant => human-label]` maps. The same map is consumed by widget schemas as `options`, by the projector as the allowlist gate, and by the unit tests as the canonical-equality assertion target.

### Phase 3 — `SystemModelProjector` refactor

- [app/WidgetPrimitive/Projectors/SystemModelProjector.php](app/WidgetPrimitive/Projectors/SystemModelProjector.php): `projectorFor()` threads `$contract->formatHints` into per-model closures.
- `projectEvent()` retired `starts_at_label` and `ends_at_label`; emits new `event_date` (hint-validated against `eventDateOptions()`, fallback `EVENT_TILE_DATE`), `event_time` (assembled by new private `buildEventTime()` helper), and `event_location` (assembled by new private `buildEventLocation()` helper — venue-mode-aware label that ports the prior EventDescription template's in-person / virtual / both / empty branching into the projector).
- `projectPost()` retired `published_at_label`; emits `post_date` (hint-validated against `postDateOptions()`, fallback `LONG_DATE`).
- New private `resolveFormatHint()` helper centralises the allowlist gate: known hint values pass through, unknown values fall back to the per-field default. Mirror of session 221's `orderByOptions` allowlist.
- **Multi-day event branching removed.** `buildEventTime()` joins start and end with an em-dash regardless of whether they fall on the same day. Multi-day support is not in the product; the structural decision was solving a problem we don't have.

### Notable findings

- **`mergeUserQueryConfig()` had to thread `formatHints` through the contract rebuild.** Discovered while writing Phase 1 — this code path constructs a fresh `DataContract` with merged user filters whenever a list-shaped widget with `querySettings.has_panel` carries non-empty `query_config`. Forgetting to include `formatHints` in that copy would silently drop the user's `date_format` selection the moment they touched the QuerySettings panel. Captured as a one-line fix; documented here as a carry-forward note for any future sibling axis on the contract — every per-instance contract field has to ride the rebuild.
- **`cardinality` is missing from the same `mergeUserQueryConfig()` rebuild but doesn't currently break.** The rebuild sits behind a `$contract->querySettings === null || ! $contract->querySettings->hasPanel` early-return; every cardinality-`ONE` contract today (EventDescription, EventRegistration, ProductDisplay) has no querySettings and exits early. Latent fragility — if a future single-row contract ever gains a panel, the rebuild would silently drop the `cardinality` and re-default to `MANY`. Not fixed this session (out of scope), but flagged.
- **Map labels for `LONG_DATE` and `MEDIUM_DATE` carry the same illustrative example** (`'Long (May 2, 2026)'` vs `'Medium (May 2, 2026)'`) because May has a three-letter abbreviation that matches its full form. Followed the prompt verbatim. The `(Long ...)` / `(Medium ...)` prefix differentiates them in the UI; user can pick a different example date if the duplicated illustration becomes confusing.
- **EventDescription's template went from 51 lines to 20.** The 14-line `@php` block was the largest template-side imperative in the entire widget catalogue; pulling it back into the projector (via the new `buildEventTime()` and `buildEventLocation()` helpers) was the load-bearing architectural win this session. The "templates are dumb token renderers" rule now holds across all five contract-locked widgets in scope.
- **`script.js` files were unanticipated touch points.** The session prompt said `build:public` ran "once at end-of-session as a safety check" — but the BlogListing and EventsListing client-side search filters reference projector field names directly (`item.starts_at_label.toLowerCase().includes(q)`), so the field rename forced a real build. Hash flipped, manifest regenerated. Carry-forward note: any future field-rename session should grep widget JS bundles, not just templates and definitions.

## Production deployment notes

- Standard `migrate` covers everything; no migrations this session.
- **`build:public` must run on deploy** — the BlogListing and EventsListing widget JS bundles changed when their search-filter field references were renamed. Stale `public/build/widgets/manifest.json` would silently break client-side search on those widgets.
- **Existing widget instances with custom `content_template` strings referencing retired tokens render those tokens as empty strings.** Specifically: any pre-existing `page_widgets.config.content_template` value containing `{{item.starts_at_label}}`, `{{item.ends_at_label}}`, or `{{item.published_at_label}}` will silently render empty for those tokens until the user updates the template. Manual repair paths: (a) reset the template to the new default via the inspector, or (b) hand-edit the JSONB column. Pre-Beta 1, the user is the only consumer; no data migration was written because token rewrites against custom templates risk false positives.

## Deferred decisions

- **`time_format` hint.** Only `date_format` lands this session; `event_time` always uses `TIME_SMART`. A second hint can land in a follow-on if user feedback supports the choice.
- **Site-setting "default date format."** Cheapest extension off the hint mechanism, but separate UX concerns (where in Settings? per-widget-type defaults? tenant-wide?). Defer.
- **Multi-day event support.** Out of product scope. If it ever returns, the structural branching lives in the projector — never the template.
- **Memos `posted_at` formatting.** Different code path (JSONB key on collection items). Template-side `Carbon::parse + DateFormat::format` remains. Bringing it into the pattern would require either a SWCT-side hint mechanism or a per-content-type display field — out of scope for this session.
- **PortalEventRegistrations.** Direct-query, not contract-locked. Migrating would require a portal-data contract — its own architectural beat.
- **EventCalendar.** JS-driven (jcalendar lib); date formatting happens client-side. Out of scope.
- **Token modifier syntax** (`{{item.event_date|long}}`). Different ergonomic — useful for power users but doesn't surface in widget settings. Deferred.
- **`event_location` migration of ThisWeeksEvents's location composition.** ThisWeeksEvents's template still composes `address_line_1`/`city`/`state`/`meeting_label` inline (its logic differs slightly from EventDescription's — meeting_label fallback). Left as-is per the session prompt's out-of-scope list. Fold-in if uniformity becomes valuable.
- **`mergeUserQueryConfig()` `cardinality` threading.** Latent fragility flagged in Notable findings above. Not fixed this session; trivial when a future single-row contract first gains a query panel.
- **Map-label illustrative-example collision (`May 2, 2026` for both `LONG_DATE` and `MEDIUM_DATE`).** Cosmetic. One-line cleanup if it surfaces in UI testing.
- **Ninth consecutive session using `-d memory_limit=2G`** to sidestep the latent mid-suite OOM (Spatie GD image processing). Whether to bake into project tooling remains a separate question.
