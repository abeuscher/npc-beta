# 258. Onboarding Rehearsal Cluster (B2)

Exercise the full onboarding flow end-to-end against a shared adversarial fixture set, measure manual-cleanup percentages and export performance, and produce the **Onboarding Playbook** at `docs/runbooks/onboarding-playbook.md` — a sales-credibility document covering the four sub-scenarios + the edge-case register the rehearsal surfaces.

This is a **feature-style rehearsal** per Rule 1+2 of `release-plan.md`. **Findings get logged and lifted, not fixed in-session.** The base prompt covers the discipline; the only writes that land in 258 are the playbook + minor doc updates + possibly a small Pest assertion or two. Importer behavior changes belong in follow-on sessions.

---

## Design decisions (resolved before starting)

- **Execution mechanism: Playwright-steered, observation-only.** Claude drives each rehearsal through the actual admin UI via Playwright rather than predicting UI behavior by reading code. **No spec files persist after this session** — Playwright is a feedback mechanism for B2, not a durable regression artifact. (The Track-F on-demand category, introduced at session 256, exists for that purpose; B2 does not author within it.) Rationale: code-aware UI prediction misses procedural surfaces (focus management, validation-message wording, modal sequencing, console errors during chunked import ticks); direct observation is cheaper than authoring durable specs and produces materially better playbook content. **Hard rule:** if a Playwright snippet is created during the session, it lives only in scratch state — no commits to `tests/e2e/`.

- **Honest framing of "manual cleanup %".** Claude is system-savvy, not operator-naïve. The measured % reflects "what a code-aware reviewer would call manual cleanup" — useful as a lower bound, not a true operator measurement. The playbook reports it as such. The honest naïve-operator measurement is the planned user-testing pass (see `session-outlines.md` § User-Testing Plans — Beta 1 Scope). B2 explicitly does NOT claim to be that pass.

- **Blind evaluation phase.** After all rehearsals run (with Claude having full knowledge of inputs to set up correctly), a dedicated Phase 6 collects each importer × shape's outcome report, strips fixture identifiers, shuffles order, and asks Claude to evaluate each anonymously. Catches the expectation-bias failure mode where Claude rationalizes outcomes because it knows what a fixture was *supposed* to do. Whole-session anonymization would be theater (Claude can re-derive shape from cell content in seconds); a scoped blind eval at outcome-report level is real signal.

- **Shared messy fixture is *generated*, not hand-crafted.** With G1 shipped, the rehearsal uses generated fixtures across all seven importers as its primary test input rather than a single hand-crafted Wild Apricot scrub. The two existing real-customer scrubs continue to ride along as a sanity check (per the standing motivation: real data has stopped finding new bugs but it's still a useful smoke test). A `messy` shape × all seven importers + the two real scrubs = the full Migration-in input set.
- **10K-record install construction.** A1's Random Data Generator widget plants the bulk of the install (contacts, donations, events, registrations, memberships). G1 stress fixtures are NOT replayed through the actual importer for this purpose — that's an importer-stress test, distinct from a "10K-record export" test. Use A1.
- **Custom-field-with-lookup feasibility.** Verify-at-start question. Today's `CustomFieldDef.options` is a static array; pulling allowed values from another model/collection probably means **no, not supported**. If confirmed, sub-scenario 4 is a finding ("custom-field-with-lookup not supported in v1; planned post-Beta-1 if a customer asks"). If a path exists I missed, follow it through.
- **Notes importer in scope.** The Notes importer shipped pre-256 and is part of the operator's intake toolkit. Migration-in covers it alongside the other six. If time pressure forces a split (Rule 11), Notes can move to the Custom-fields half.
- **The four sub-scenarios are not equally weighted.** Migration-in is the heaviest (multi-importer dry-run + commit + approval flow). Migration-out is the simplest (run an export, time it, count rows). Custom fields + public collection is medium. Custom-field-with-lookup is light (likely one finding, no rehearsal proper).
- **Out of scope this session:**
  - Any importer behavior fixes — surfaced findings get lifted, not patched. Specifically: events `status` case-normalization (lifted from 257), control-char/malformed-email/oversized-cell permissiveness (lifted from 257), any new findings during 258. None get fixed in 258.
  - Salesforce / NPSP preset support (post-Beta-1, tracked in `session-outlines.md` § Test-Data Generator — Salesforce Export Shape Support).
  - B1b's affiliations junction — explicitly post-B2 follow-up; B2 surfaces what shape it should take.
  - Any boundary changes to the Fleet Manager agent contract.

## Phase 2 — Migration-in rehearsal (sub-scenario 1)

**Goal:** prove an operator can move messy real-shape data into a fresh CRM install with <5% manual cleanup post-import. Measure the cleanup percentage; log every edge case + its resolution.

**Execution: Playwright-steered.** Drive the importer UI through Playwright (page.goto + page.fill + page.click + screenshots/DOM checks). Observe what an operator sees — focus states, validation messages, modal flows, console output, chunked-tick progress. Do **not** persist any spec files; this is observation-only.

**Steps:**
1. Generate the fixture matrix from Phase 1's plan (`import-fixtures:generate --shape=messy --seed=...` for all seven importers).
2. For each importer, drive through Playwright: upload → map columns → review pii/dry-run → approve → verify outcomes. Capture: screenshots at each step, console errors observed, time-to-complete, any UI affordance that wasn't where an operator would expect it.
3. Run the same flow against a real-customer Wild Apricot scrub for sanity comparison.
4. Tally: rows imported / rows requiring manual cleanup / rows skipped / rows errored. Compute manual-cleanup % per importer + overall.
5. Log every edge case encountered: what failed, what the operator did to resolve it, whether it was a one-off or a pattern.

**Outcome report (feeds Phase 6 blind evaluation):** for each importer × shape, produce a flat record with: row tallies, top 3 skip/error reasons, manual cleanup steps required, time-to-complete (Playwright-measured). Strip fixture identifier when handing to Phase 6.

**Pass criterion:** manual cleanup <5% across the fixture matrix. If a specific importer drives the percentage up, that's a finding (not a fix).

**Findings register entries (already known from 257 — do not re-discover):**
- Events `status` case sensitivity → lift as new entry under `release-plan.md`'s Out-of-Gate register or as a small standalone session candidate.
- Importer permissive about control chars / oversized cells / malformed emails → playbook entry: "validate cells before import; the importer will accept obviously-bad input today."
- `hide_pending_imports` global scope → playbook entry: "imported contacts are gated behind reviewer approval until the import session is approved. Non-reviewers will not see them in the Contacts list. This is intentional. Approve the session to release."

## Phase 3 — Migration-out rehearsal (sub-scenario 2)

**Goal:** prove a 10K-record install exports cleanly in <5 minutes with no data loss.

**Execution: Playwright-steered for the operator-facing parts** (clicking Export buttons, observing download initiation, watching the Filament toast feedback). DB count comparisons happen via direct DB queries — those are objective and don't need UI observation.

**Steps:**
1. Plant a 10K-record install via A1's Random Data Generator widget. Document the exact widget settings used (per-record-type counts).
2. Verify the install plants the expected counts: contacts ≈ 10K, donations ≈ correct ratio, events / registrations / memberships per A1's defaults.
3. For each list page that exposes an "Export CSV" action: drive the click via Playwright; observe what the operator sees during export (loading state? blocking modal? streaming download? error feedback if it fails?). Capture the downloaded file.
4. Time each export end-to-end (click → download complete). Sum the totals.
5. Verify no data loss: row counts match (DB count == CSV row count − 1 header). 5-table spot-check: pick 5 random rows from each export, verify field values match the underlying DB row.
6. Open one export in a spreadsheet — does the structure feel sane to a non-developer operator?

**Outcome report (feeds Phase 6 blind evaluation):** per export action, capture: time-to-complete, row-count match (yes/no), spot-check match (5/5, 4/5, etc.), any unexpected fields/columns, any operator-visible failure modes during export.

**Pass criterion:** total export time <5 min for the 10K-record install. Zero discrepancies between DB and CSV row counts.

**Findings to watch for:**
- Missing exporters on resources where one would be expected (Organizations / Donations / Memberships / etc.) — log as findings, don't build them.
- Custom-field columns appearing as raw JSON vs flattened — note for the playbook.
- Date format inconsistencies between import and export — note (importer accepts mixed; exporter probably emits one format).

## Phase 4 — Custom fields + public collection (sub-scenario 3)

**Goal:** prove custom fields on Contacts + Events plus a public event-photos collection ingest cleanly and the public collection page renders.

**Execution: Playwright-steered.** Custom-field definition setup, the collection admin, and the public-side rendering are all UI-driven; observe each through Playwright (no spec persistence). The public collection page render in step 6 is the single most operator-facing moment of this sub-scenario — capture screenshots + DOM presence + console state.

**Steps:**
1. Create custom fields on the Contacts model (one of each supported type — text, number, date, boolean, select, rich_text — pick a sensible distribution). Drive through CustomFieldDef admin via Playwright.
2. Create custom fields on the Events model (similar distribution; events supports rich-text custom fields per E3).
3. Generate a Contacts fixture with the new custom-field columns mapped through `import-fixtures:generate --shape=messy --importer=contacts ...`. Run the import via Playwright. Verify custom fields land on the Contact records.
4. Generate an Events fixture similarly. Run the import via Playwright. Verify custom fields land on Event records.
5. Set up a public event-photos collection via the Collection / ContentCollection admin pages. Attach photos to events.
6. Hit the public URL via Playwright. Capture: page loads (HTTP 200), expected events visible in DOM, photo attachments rendered, no console errors, layout feels coherent at desktop + mobile viewport widths.

**Outcome report (feeds Phase 6 blind evaluation):** custom-field round-trip success rate per type, public collection page render outcome (renders cleanly / renders with issues / fails to render), operator-perceived "would-I-ship-this" call.

**Pass criterion:** all custom fields ingest without loss; public collection page renders cleanly with the imported data visible.

**Findings to watch for:**
- Rich-text custom-field cells imported with HTML — verify they render correctly on the public side (E3 ships QuillEditor on admin; the public-side render path uses `{!! !!}`).
- Event registration custom fields (separate from event custom fields) — covered by E3 but worth a sanity check if the rehearsal has time.

## Phase 5 — Custom-field-with-lookup edge case (sub-scenario 4)

**Goal:** determine whether the platform supports a custom field that pulls allowed values from another collection or another model's column. If yes, document the path. If no, document as a finding with a planned post-Beta-1 fix.

**Steps:**
1. Per Phase 1's feasibility decision: if not supported, log the finding + the rationale. If a path exists, exercise it.
2. If the answer is "no" (likely): document in the playbook as "custom-field-with-lookup is not supported in v1. Workaround: use a `select` field with a static options list and update the list when the source changes. Tracked for post-Beta-1 if a customer asks."

**Pass criterion:** the question has a definitive answer (works / works with caveats / not supported), and the playbook reflects it.

This phase is intentionally light — most of the value is the documented answer, not the rehearsal itself.

## Phase 6 — Blind evaluation

**Goal:** catch the expectation-bias failure mode where Claude rationalizes outcomes because it knows what each fixture was *supposed* to do. Whole-session anonymization would be theater (Claude can re-derive shape from cell content immediately); blind evaluation at the outcome-report level is real signal.

**Steps:**
1. Collect every outcome report produced by Phases 2–5 into a flat list. Each report is the per-importer × per-shape record from the rehearsal.
2. Strip fixture identifiers (importer name, shape, seed, source preset, encoding). Keep: row tallies, top reasons, manual-cleanup steps, time-to-complete, operator-visible findings.
3. Shuffle order. Assign anonymous Case A / B / C / D / etc.
4. For each Case, evaluate independently and record the assessment **before moving to the next**:
   - Would I sign this off as a successful import?
   - What concerns would I raise to the operator?
   - Estimate manual-cleanup % from the report alone.
   - Confidence: high / medium / low.
5. Once all Cases evaluated: pause. The user reveals the key (which Case maps to which fixture).
6. Reconcile:
   - Where did blind evaluation match the expected disposition? (Confirms the rehearsal's findings are robust.)
   - Where did blind evaluation diverge? Two failure modes worth distinguishing:
     - **Over-permissive:** the fixture intentionally planted an issue and Claude's knowing-the-shape evaluation rationalized it; blind evaluation caught it.
     - **Over-strict:** Claude's blind read flagged something the manifest considers acceptable.
7. Document divergences. They feed both the playbook's edge-case register AND the lifted-findings list.

**Pass criterion:** blind evaluation completes for every Case with an assessment recorded *before* the key is revealed. Reconciliation produces zero, one, or many findings — any number is informative; the goal is honest evaluation, not zero-divergence.

**Discipline:** if the blind eval feels too easy ("obviously this is the messy one"), that's likely Claude reading shape clues from the *content* of the report (e.g., a `cf_type_mismatch` corruption_kind in the report literally names the shape). If that happens, it's a signal that report sanitization needs to go further next time, not a reason to abandon the phase.

## Phase 7 — Onboarding Playbook

**Goal:** write `docs/runbooks/onboarding-playbook.md` — a sales-credibility document covering the full intake flow including edge-case handling.

**Sections** (target shape):
1. **What this playbook covers** — one paragraph: who reads it (operator + sales), when it's used (pre-sale demo + first-customer onboarding), what success looks like.
2. **The intake flow** — operator's perspective: receiving customer data, choosing importers, configuring matchers, dry-running, approving, releasing pending records.
3. **Migration in: the seven-importer flow** — Contacts → Organizations → events/donations/memberships/etc. in dependency order. Why this order. What each importer does. What the operator sees (from Playwright observation in Phase 2).
4. **Edge-case register** — every finding from 257 + everything 258 surfaces (rehearsal phases AND blind-evaluation reconciliation). Each entry: symptom, reason, operator-side resolution. This is the heart of the playbook's sales-credibility value.
5. **Migration out** — the export surfaces, what they cover, performance expectations on a 10K-record install. Limits.
6. **Custom fields** — what's supported (six types per session 250), how they round-trip through import/export, the lookup-value caveat.
7. **Approval workflow** — `hide_pending_imports` semantics, the reviewer permission, what "approve the session" actually does.
8. **First-customer checklist** — distilled action list from the rehearsal: "before pulling the trigger, do A / B / C."
9. **Methodology + honest framing** — one paragraph stating: this rehearsal was Claude-driven via Playwright with full system knowledge; the manual-cleanup % is a system-savvy lower bound, not a naïve-operator measurement. The honest naïve-operator pass is the planned user-testing program (link to `session-outlines.md` § User-Testing Plans). Blind-evaluation results from Phase 6 are summarized here too, including any divergences between knowing-the-shape and blind reads.
10. **Known gaps + planned work** — the finding lifts. Each entry links to a release-plan entry or out-of-gate register entry.

The playbook is **prose-first**, not a bulleted-list dump. Sales-credibility means it reads as written by someone who has done the migration. Cite specific affordances, specific numbers (manual-cleanup % from Phase 2, export time from Phase 3, blind-eval reconciliation from Phase 6), specific failure modes.

## Phase 8 — Manual verification

After implementation:

1. Read the playbook end-to-end. Does it read as a credible operator document, not a developer scratchpad?
2. Verify every finding has a destination — playbook entry, out-of-gate register entry, or new release-plan entry.
3. Hit the public collection page in a browser. Does it actually render the imported data?
4. Run the fast-suite + slow-group tests to confirm no regression from any minor doc-adjacent code touches.
5. Open one of the 10K-record exports in a spreadsheet — final sanity check.

Pull the user in for visual judgment on:
- Whether the messy import flow *feels* tractable for a real operator (vs. just "passes the <5% bar")
- Whether the playbook's tone matches "sales-credibility" rather than internal-dev-doc
- Whether the manual-cleanup % is honest (it should reflect what an operator actually has to do, not what a developer with full DB access can fix in five minutes)

Stop and wait for user feedback. Do not suggest closing.
