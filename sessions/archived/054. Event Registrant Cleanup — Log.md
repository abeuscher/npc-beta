## Workstream 1 — Delete Registrant Contacts Action

### What was built

Added a **Delete Registrant Contacts** header action to `app/Filament/Resources/EventResource/Pages/EditEvent.php`.

**Visibility logic:** The action is shown when the event is cancelled (`status === 'cancelled'`) OR when the event has at least one date and no upcoming dates (`eventDates()->exists()` and `eventDates()->where('starts_at', '>=', now())->doesntExist()`).

**Contact identification:** A contact is eligible for deletion if:
- Their `id` appears in `event_registrations.contact_id` for this event
- `contacts.source = 'web_form'`
- No other event registration records link them to a different event
- No membership records
- No donation records

Implemented using `whereNotExists` subqueries against `event_registrations` (other events), `memberships`, and `donations`.

**Confirmation modal:** Uses `->requiresConfirmation()` with a `->modalDescription()` closure that runs the identification query at modal-open time and presents the computed contact count, registration count, and a fixed explanatory message.

**On confirm:**
- Iterates eligible contacts calling `$contact->delete()` (soft delete)
- Deletes all `event_registration` rows for the event regardless of eligibility
- Sets `events.registrants_deleted_at = now()` to mark the cleanup
- Redirects to the same edit page to force a fresh render
- Shows a summary notification: "X contacts removed, Y registration records deleted."

### Relation manager update

`EventRegistrationsRelationManager` now checks `$event->registrants_deleted_at`. When set, the table renders with no columns or actions, displaying instead: heading "Event registrants deleted" / description "All registrant contacts and registration records for this event have been removed."

### Migration

`2026_03_21_200002_add_registrants_deleted_at_to_events_table.php` — adds nullable `registrants_deleted_at` timestamp to `events`.

## Out-of-scope fixes

### Event landing page 404 bug

**Root cause:** `PageResource::getEloquentQuery()` excludes pages of type `event` and `post` to keep the CMS pages list clean. Filament uses this same method to resolve records when loading edit routes (`resolveRecordRouteBinding`). Redirecting to `PageResource::edit` for an event landing page therefore silently returned a 404.

**Fix:** Added `resolveRecordRouteBinding()` override to `PageResource` that queries `Page::where('id', $key)` directly, bypassing the type filter for record lookup only. The list table is unaffected.

### Registrants deleted — page reload

After the delete action completes, `$this->redirect(static::getResource()::getUrl('edit', ['record' => $event]))` forces a full page reload so the Registrations tab immediately reflects the deleted state.
