# 142. Inline JS/CSS Extraction & Migration Squash

Extract large inline JavaScript blocks from page builder blade templates into registered Alpine.js components. Squash migration history. The goal is navigability: the blade files should read as declarative markup with named component references, not walls of procedural JS.

---

## Design decisions (resolved before starting)

- **Alpine.data() pattern.** Each extracted block becomes an `Alpine.data('componentName', ...)` registration in a dedicated JS file. Blade templates reference them as `x-data="componentName"` or `x-data="componentName($wire, ...)"` when they need Livewire entanglement.
- **File organisation.** Extracted components live in `resources/js/page-builder/` as individual files (one component per file), with an `index.js` barrel that registers all of them. This barrel is imported by the admin Vite entrypoint.
- **No behaviour changes.** Every extraction must be a pure refactor — identical runtime behaviour before and after. If a block's logic is tangled with blade-rendered PHP variables, pass those as parameters to the Alpine.data factory function.
- **Inline CSS in blade files** should move to the Filament custom theme SCSS or a scoped CSS file, not remain as `style=""` attributes on elements (unless the style is truly dynamic and data-driven).
- **Migration squash** follows the procedure from session 062: `schema:dump`, delete superseded migrations, verify `migrate:fresh --seed`.

---

## Phase 3 — Extract components

For each approved extraction:

1. Create `resources/js/page-builder/{component-name}.js` exporting an Alpine.data factory function
2. Move the inline JS into the factory, converting any blade-rendered PHP values (`@js(...)`, `@entangle(...)`) into parameters passed from the blade
3. Replace the inline `x-data="{...}"` block in the blade with `x-data="componentName(...params)"` 
4. Test that the component works identically — no regressions in tab switching, spacing controls, widget filtering, preview behaviour, etc.

Commit after each component extraction (or logical group) so regressions are easy to bisect.

---

## Phase 4 — Inline CSS audit

Scan the page builder blade files for inline `style=""` attributes and `<style>` blocks. For each:

- If the style is static (not data-driven): move to SCSS/CSS
- If the style is dynamic (bound to Alpine or Livewire state): leave as inline but document why
- If the style duplicates something already in the Filament theme: remove and use the existing class

Present findings before making changes.
