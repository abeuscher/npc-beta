# 202. Importer CSV Generator & Playwright Edge-Case Coverage

Ship an artisan command that emits a self-consistent bundle of fake-but-realistic import CSVs across all five content types, a separate seeder command for demo `import_source` rows bound to the canonical headers, and four Playwright edge-case specs (PII rejection / duplicate-header review / error-report / invoice-details multi-row grouping) that consume the generator's output.

This session pairs two previously separate stubs — "Random Data Generator — CSV Export for Import Testing" and "Playwright Importer — Deferred Edge-Case Specs" — because they feed each other: the edge-case specs need varied, realistic fixtures, and the generator's shape can be validated in real time by the specs consuming it.

---

## Design decisions (resolved before starting)

- **Generator is pure file I/O.** Writes CSVs to a configurable `--out` dir (default `storage/app/fake-csvs/`). No DB writes, no `import_source` side-effects, no fixtures auto-populated. Runnable on a fresh checkout without a DB up. Keeps the command reusable outside tests — stress testing, offline demos, help-doc screenshotting.
- **Cross-file coherence within a single run.** One invocation produces a self-consistent fixture set. `donations.csv`, `memberships.csv`, and `invoice_details.csv` reference contact emails from the `contacts.csv` written in the same run. `invoice_details.csv` rows group into parent invoices by a stable `invoice_number` key. This matches the real-world shape of competitor exports and gives the update-strategy specs meaningful relationship data to exercise.
- **Grouped output for invoices.** `invoice_details.csv` emits parent invoices with 1–5 line items per invoice. Every line-item row in the same group shares `invoice_number` and parent-invoice fields. The edge-case spec for conflicting parent fields does NOT come from the generator — the generator emits *clean* groups; the spec mutates a clean group to introduce the conflict.
- **Grouped output for events.** `events.csv` is structurally one-row-per-registration (plus rows for events with no registrations). The generator emits distinct events; each event is represented by 1–5 rows that all share Event columns and vary in Registration + Transaction columns. Events with zero registrations get a single row with blank Registration/Transaction columns. Roughly 40% of events are free (blank Transaction columns); the other ~60% carry populated Transaction fields. Registration `Contact Email` values reference emails from the same run's `contacts.csv`.
- **Happy-path only from the generator.** Corrupted CSVs for PII / duplicate-header / error-report are produced in Playwright test setup by reading a clean generated CSV and mutating it before upload. This keeps the generator focused, and keeps each corruption's intent next to the spec that consumes it.
- **Demo `import_source` seeding lives in a separate command.** `seed:fake-import-sources` creates the five `import_sources` rows bound to the canonical headers. The Playwright fixture helper invokes it once per suite so specs land on the saved-mapping path without re-mapping each run. Generator and seeder do not know about each other at the code level — the seeder just knows the canonical header list.
- **Headers — source of truth is `CsvTemplateService`.** The generator reads its header list from `CsvTemplateService` (or an equivalent interface if the service exposes this per-type). If the service's headers drift from the generator's internal key set, the command fails fast with a clear diff rather than silently emitting headers that won't round-trip through the importer. Same invariant for the seeder's field map.
- **Row counts are ranges, randomised per run.** The specific ranges below ensure no two test runs have identical row counts — so specs can't accidentally depend on magic numbers. Use a `--seed=N` flag for reproducibility when investigating a failing run.

### Row-count ranges

| Content type | File | Range |
|---|---|---|
| Contacts | `contacts.csv` | 200–250 rows |
| Events (total rows — registrations + blank-registration rows) | `events.csv` | 20–40 rows |
| Distinct events within `events.csv` | *(within events.csv)* | 8–15 |
| Rows per event | *(within events.csv)* | 1–5 per group |
| Donations | `donations.csv` | 300–500 rows |
| Memberships | `memberships.csv` | 50–100 rows |
| Invoice details (total line-item rows) | `invoice_details.csv` | 100–300 rows |
| Line items per parent invoice | *(within invoice_details.csv)* | 1–5 per group |

## Phase 1 — Generator command

Build `csv:generate-fake-imports` (or whatever the existing debug generator's naming convention suggests — match it). Signature: `{--out=storage/app/fake-csvs} {--seed=}`.

Requirements:

1. **Reuse existing synthetic-value logic.** The debug generator from sessions 079 / 184 already produces plausible names / emails / amounts / dates for DB seeding. Hoist the row-composition helpers into a shared service (`App\Services\FakeCsvComposer` or similar) if needed, and have both the DB-seeding path and the new CSV-writing path call into it. Do not duplicate synthetic-value generation.
2. **Header source of truth.** Read headers from `CsvTemplateService` per content type. Fail fast if the generator's internal key map is missing any header the service declares (or has extras the service doesn't).
3. **Cross-file references.** Write `contacts.csv` first. Keep the emitted contact list in memory. When writing donations / memberships / invoice_details / events (Registration `Contact Email` column), pick `contact_email` values from that list at random, biased so some contacts have multiple related records and some have none (realistic distribution).
4. **Invoice grouping.** When writing `invoice_details.csv`, determine the total row count first, then partition rows into groups of 1–5 line items, each group sharing a stable `invoice_number` and parent-invoice fields.
5. **Event grouping.** When writing `events.csv`, pick the distinct-event count (8–15), then assign each event a row count (1–5). All rows in an event group share Event columns; Registration columns vary per row. Events flagged as "free" emit blank Registration + Transaction columns on all their rows; "paid" events emit populated Registration fields and consistent Transaction fields within the group (one payment per registration). An event with `rowCount=1` and zero registrations emits a single row with blank Registration and Transaction columns — event metadata only.
6. **`--seed` flag.** Pin the PRNG so failing runs are reproducible. Default is random.
7. **Idempotency on output dir.** If the out dir already contains prior-run CSVs, overwrite them (don't append, don't error out). Log the files written.
8. **Pest test.** Ship `tests/Feature/FakeImportCsvGeneratorTest.php` asserting: (a) each output CSV exists after command run, (b) headers match `CsvTemplateService` exactly for each content type, (c) row counts fall in the declared ranges, (d) cross-file references resolve (every non-blank `contact_email` in donations/memberships/invoice_details/events-registrations matches an email in the contacts.csv from the same run), (e) invoice grouping invariants hold (every `invoice_number` appears 1–5 times, all rows in the same group share their parent-invoice field values), (f) event grouping invariants hold (every event external-id / title appears 1–5 times, all rows in the same event group share Event columns). Fast suite.

## Phase 2 — Demo import source seeder

Build `seed:fake-import-sources {--force}`.

Requirements:

1. **Idempotent.** Seeds five `import_sources` rows (one per content type: contacts, events, donations, memberships, invoice_details). On re-run without `--force`, leaves existing rows untouched. With `--force`, replaces them.
2. **Field maps are identity.** Each source's saved field map maps each canonical CSV header directly to the obvious destination field on the target model. No clever guesses — just 1:1.
3. **Source labels** make it clear these are demo / fake fixtures (e.g. "Demo Fake Data — Contacts", "Demo Fake Data — Donations", etc.). The Playwright specs select by label.
4. **Pest test.** Ship `tests/Feature/SeedFakeImportSourcesTest.php` asserting: (a) command creates the five rows on first run, (b) second run without `--force` is a no-op, (c) `--force` replaces existing rows with fresh field maps.

No migration in this session. The `import_sources` table already exists (session 188).

## Phase 3 — Playwright edge-case specs

Four new spec files under `tests/playwright/importer/`, following session 195/196 conventions.

Each spec starts from a shared fixture helper that:

1. Runs `csv:generate-fake-imports` against a temp dir (or uses the default `storage/app/fake-csvs/` seeded once per spec run).
2. Runs `seed:fake-import-sources --force` so the mapping step is pre-populated.
3. Provides a small utility for reading a clean generated CSV, mutating it, and writing to a temp file ready for upload.

### Spec A — PII rejection

- Read clean `contacts.csv`. Inject a new column `ssn` and populate with synthetic 9-digit values for all rows (or a recognised PII trigger per whatever the detector pattern actually is — verify at the start of implementation). Write to temp file.
- Upload via the contacts importer UI.
- Assert: rejection UI appears; PII violations table lists the offending column(s) and sample values; downloadable violations report exists and contains the expected rows.

### Spec B — Duplicate-header review

- Read clean `contacts.csv`. Duplicate one header (simplest: write the same header twice — e.g. two `email` columns). Leave row data intact (the dup column gets the same data).
- Upload.
- Assert: review step surfaces the duplicate-header finding; selecting "ignore the duplicate" (or equivalent choice) propagates through the mapping step so the import proceeds on the remaining single column.

### Spec C — Error report

- Read clean `donations.csv`. Pick 5 rows; corrupt each differently:
  - Bad date (e.g. `2026-13-45`).
  - Non-numeric amount (e.g. `twenty dollars`).
  - Blank required field (e.g. empty `contact_email` where a contact ref is required).
  - An unrecognised fund (if the donations importer validates fund names against the DB).
  - A fifth variant to exercise any remaining validator branch the above four don't cover — pick based on what the donations importer actually checks.
- Upload.
- Assert: error table renders with exactly the five corrupted rows, each row's error reason is specific (not generic "row failed"), and the downloadable errored-rows CSV contains those five rows with original row-number context.

### Spec D — Invoice-details multi-row grouping under update strategy

Two assertions in one spec, both against the update strategy path:

1. **Line-item ordering is preserved.** Upload a clean generated `invoice_details.csv`, run under update strategy. For at least one parent invoice with ≥3 line items, assert the post-import persisted line-items are in the same order as the CSV rows within that group.
2. **Conflicting parent-invoice fields surface a warning (or are resolved by a documented precedence rule).** Read a clean `invoice_details.csv`, pick one parent-invoice group, and mutate a parent-field value on ONE of the group's rows so the rows disagree on that field. Upload under update strategy. Assert the UI surfaces the conflict and the resolution matches documented behaviour (either a warning in the review step, or a deterministic first-wins / last-wins rule — whichever the existing code actually does).

**Scope note:** the invoice-details spec is narrow — ordering preservation + one parent-field conflict shape. Broader scenarios (partial matches, composite-key edge cases, orphaned line items) are explicitly deferred.

### Shared helper conventions

- Keep per-spec mutation helpers in the spec file itself (≤20 lines each) so a reader can see the corruption intent inline. The shared fixture helper owns "run generator, run seeder, load clean CSV" only — never corruption.
- Do not commit the generator's output CSVs. Playwright generates them fresh on each run. If the generator's output dir is not already `.gitignore`d, add it.

## Out of scope

- **The `buildStageAttrs` matches-vs-diffs product question** flagged in session 196. Deferred as a product call, not a test gap.
- **Event registrations as a separate CSV.** Registrations are folded into `events.csv` alongside event and transaction columns (matching `CsvTemplateService::eventHeaders()` shape), not emitted as a separate file.
- **Stress-test row volumes.** Ranges above are demo-scale. A future session can add `--stress` or `--multiplier` flags.
- **Non-English / locale-specific data.** Faker default locale only.
- **Corruption helpers inside the generator itself.** Corruption lives in Playwright by design.
- **Additional importer source presets** (Blackbaud, Little Green Light, etc.). That's the separate "Importer Source Presets — Competitor Coverage" stub.
- **Broader invoice-grouping edge cases** beyond ordering + one parent-field conflict: partial matches, composite-key edges, orphaned line items. Deferred.
- **Rewriting session 196's happy-path specs.** They stand as-is. This session adds only.
