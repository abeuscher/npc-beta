# 042. Codebase Audit — Fields, Schema, Permissions & Help Coverage

A housekeeping session with no new features. The goal is to find and resolve everything that has quietly drifted out of alignment across the codebase — data model drift, permission gaps, missing help articles, and structural inconsistencies — before the next round of feature work begins.

---

## Before writing any code

Read the following files:

- Every file in `app/Filament/Resources/` — all resource edit forms
- Every file in `app/Filament/Pages/` — all custom admin pages
- Every file in `app/Models/` — all models, their `$fillable`, `$casts`, and soft delete status
- Every migration in `database/migrations/` — note nullable, index, and cascade behaviour
- Every file in `resources/docs/` — all help articles and their registered routes

Take notes as you read. The output of this session is corrections, not features.

---

## Workstream 1 — Edit page field audit

For every Filament resource and admin page that has an edit form, compare the fields shown in the form against:

1. The database columns for that model (derived from migrations)
2. The model's `$fillable` array

**For each discrepancy found**, make one of two decisions:

- **Deliberate omission** — the field is system-managed, read-only by design, or intentionally hidden. Document the reason in a comment in the resource file (`// system-managed: set by X`).
- **Gap to fix** — Flag the problem and report back so a decision can be made about each instance. With the exception of the fields mentioned specifically below.

Known issues to resolve before the audit surface them:

- The plain-text `notes` column on `contacts` coexists with the morphMany `Note` system. The column should be removed from `Contact::$fillable` (and the migration-level column evaluated for dropping) since the Note relationship is the correct editing surface.
- `email_secondary` and `phone_secondary` are in `Contact::$fillable` and the database schema but were removed from the edit form. Decision: drop the columns via migration and remove from `$fillable`, or reinstate them in the form. Document the outcome.

Produce a brief written summary of every field decision made — field name, model, decision (omission or fix), and reason.

## Workstream 2 — DB schema doc

Create `docs/schema.md` at the repo root. This is a developer-facing reference, not a user-facing help article.

Structure: one section per model/table, with a simple markdown table listing:

| Column | Type | Nullable | Notes |
|---|---|---|---|

Cover every table in the database. Order sections alphabetically by table name. Include a one-line description of each table above its column table.

**Establish a rule**: updating `docs/schema.md` is a required step in every future session that includes a migration. Add this to the process rules section of `sessions/base-prompt.md`.

---

## Workstream 3 — Permission gate audit

Every admin page and Filament resource must have an intentional access check. Some pages built before the permission system matured may have none.

Produce a table of every admin page and resource listing:

| Page / Resource | Gate method | Permission checked |
|---|---|---|

For any page with no gate, add one. Use `canAccess()` for custom pages, and the appropriate Filament policy method for resources. If a page should be accessible to all authenticated admin users with no specific permission, document that explicitly rather than leaving it implicit.

---

## Workstream 4 — Soft deletes consistency

Produce a table of every model listing whether it uses `SoftDeletes` or not, and a one-line justification. Correct any that are clearly wrong (e.g. a model that should survive accidental deletion but hard-deletes). Do not change models where hard-delete is the right behaviour — just document it.

---

## Workstream 5 — Fillable vs. casts consistency

Every column in `$fillable` that is a date, boolean, or JSON type must also appear in `$casts`. Scan all models and add any missing cast entries. Do not change behaviour — this is a correctness and consistency fix only.

---

## Workstream 6 — Foreign key index audit

Every `foreignId` and `foreignUuid` column should have a database index. Review all migrations. For any FK column that lacks an explicit index (and is not covered by a unique constraint on that column), write a new migration to add the missing index.

---

## Workstream 7 — Orphaned documents and help coverage

**Orphaned documents**: scan `resources/docs/` for help articles whose `routes:` frontmatter lists a route name that no longer exists in the application. Flag and remove or update them.

**Missing help articles**: for every distinct admin view in the application (resource list, resource edit, custom page), check whether a help article exists with that route registered. If not, create a stub: copy the frontmatter structure from an existing doc, set the title, register the route, and write a single placeholder line as the body. Body copy to be written in a future session.

---

## Workstream 8 — Factory coverage

List every Eloquent model in `app/Models/`. For each one, note whether a factory exists in `database/factories/`. For any model that participates in feature tests (or reasonably should) but has no factory, create a minimal factory.

---

## Out of scope

- Data retention and cascade delete policy (separate session already scheduled)
- New features of any kind
- Refactoring code that works correctly
- Updating help article body copy (stubs are sufficient)
