## Style rules

- Follow existing code conventions exactly — the existing `csv:generate-fake-imports` command + `CsvTemplateService` are the closest references for file-emission shape; `app/Services/Import/` is the namespace conventions.
- Do not add features, refactor, or improve anything beyond what the session specifies.
- Prefer editing existing files to creating new ones — the new generator, the per-importer fixture builders, the four shape transforms, the manifest writer, the `FixtureRunner` service, and the Pest runner are unavoidable additions; everything else is in-place edits.
- No docstrings, comments, or type annotations on code you did not write.
- Simple, correct, well-considered solutions over fast or clever ones.
- **Custom-field variance discipline** — all four shapes (clean / messy / corrupt / pii / stress) include custom-field columns where the importer supports them. The user surfaced this explicitly at 256 close: every system the project sells against (Wild Apricot / Neon / Bloomerang / Salesforce / etc.) has custom fields, so test coverage that ignores them isn't credible. Per-importer fixture builders declare which custom-field types they emit; transforms apply variance to those columns alongside the canonical ones.
- **Manifest-as-source-of-truth discipline** — every CSV the generator emits has a sibling `.expected.json`. A CSV without a manifest is a regression noise generator, not a test input. If implementation pressure tempts dropping manifests for one shape, raise it; don't drop silently.
