# 156. Form Polish, Image Fields & Chrome Toggle

A grab-bag session that fixes two real bugs (Hero widget nav colors, OG image fallback to site default), reshuffles the Edit form layouts on Page/Event/Post into a single unified header fieldset, adds a new "Images" section with Spatie media collections that replace the legacy `og_image_path` string column, adds a max-width to the inspector panel, and adds an admin topbar full-screen toggle button. Mostly cosmetic but two of the items are user-facing bugs and one is a schema migration.

---

## Design decisions (resolved before starting)

### OG image storage replaces the legacy string column

- **Drop `pages.og_image_path`** in a migration. The current column is a string holding a URL or disk path. Replace with a Spatie media collection (`og_image`) on the `Page` model.
- Pre-beta — there is no production data to preserve. The migration is one-way for the forward path; the `down()` method should re-create the column for emergency rollback.
- After the migration, [SeoMetaGenerator::forPage()](app/Services/SeoMetaGenerator.php#L17) must read from the new media collection instead of the dropped string column.

### OG image fallback bug — site default is never consulted

- [CmsSettingsPage.php:243](app/Filament/Pages/Settings/CmsSettingsPage.php#L243) provides a `site_default_og_image` setting. Today it is stored but never read.
- [SeoMetaGenerator::forPage()](app/Services/SeoMetaGenerator.php#L25-L27) chains `og_image_path → extractFirstImage($page) → ''`. The site default is missing from the chain.
- Fix: chain `SiteSetting::get('site_default_og_image')` after `extractFirstImage()` and before the empty string. Combine with the storage migration above so the same method picks up both changes in one place.

### Image fields per resource

| Resource | Thumbnail | Header | OG Image |
|---|---|---|---|
| **Page** (default type) | new | — | new |
| **Post** (post type) | existing (`post_thumbnail`) | existing (`post_header`) | new (third, distinct field) |
| **Event** | new | new | new |

- Page and Post share the `Page` model, so collection naming needs care. Phase 1 audit must read the existing `Page::registerMediaCollections()` and decide whether the Page-type thumb shares `post_thumbnail` or has its own collection name. Same call for OG image — the cleanest design is one shared `og_image` collection on the `Page` model that both types use, since it serves the same OG metadata purpose for both.
- Event is a separate model. Its three new collections live on the `Event` model with names that don't collide.
- These fields are **storage-only** for now. Wiring them into widgets, public templates, or rendering surfaces is a future session. The exception is OG image, which is rendered into the meta tag in the same session.

### Hero widget nav color override fields — restore

- The Hero widget previously had `nav_link_color` and `nav_hover_color` fields in its `config_schema` that took effect when the hero was in full-bleed mode (where the site nav overlays the hero rather than sitting above it). These were removed at some point and need to be restored.
- Phase 1 audit: find the Hero widget's seeder definition, find the field shape from a previous version (git history if needed), and find where the hero blade template would consume them.

### Edit form layout — single unified fieldset

- Today, Edit Page / Edit Event / Edit Post each have multiple sections at the top of the form (Page Name, Templates, Settings, Tags). These collapse into one unlabeled, full-width fieldset with two rows.
- **Edit Page row 1:** Page Name (3) | Template (3) | Author (2) | Status (2) | Publish date (2) = 12 columns
- **Edit Page row 2:** Slug (3) | Tags (6) | Create tag (3) = 12 columns
- **Edit Event row 1:** Page Name (3) | Author (3) | (blank 2) | Status (2) | Publish date (2) = 12 columns
- **Edit Event row 2:** Slug (3) | Tags (6) | Create tag (3) = 12 columns
- **Edit Post row 1:** Page Name (3) | Author (3) | (blank 2) | Status (2) | Publish date (2) = 12 columns
- **Edit Post row 2:** Slug (3) | Tags (6) | Create tag (3) = 12 columns
- The shared trait `App\Traits\HasPageBuilderForm` drives all three forms — most of the change happens in one file.

### Images section — new collapsed fieldset

- Above the SEO section in each form. Full width. Collapsed by default. Section heading: "Images".
- Houses the new media-upload fields per the table above.
- Same position and visual treatment on all three resources.

### Inspector panel max-width

- CSS-only change. Add a `max-width` to the inspector panel container so additional horizontal space flows to the preview panel rather than widening the inspector indefinitely.
- No state or behaviour change.

### Full-screen toggle button in admin topbar

- Eye icon (heroicons has `eye` and `eye-slash` — pick whichever pair makes the toggle state obvious).
- **Position:** in the Filament admin topbar, immediately to the left of the user-menu icon.
- **Behaviour:** toggles two existing pieces of state at once — the left navigation collapse, AND the content area's full-width mode (currently a toggle in the user menu).
- **Supersedes** the user-menu full-width toggle. The user-menu toggle can stay in the menu, but the topbar button is the canonical control. If the two ever conflict, the topbar button wins.
- Implementation likely uses a Filament render hook (`PanelsRenderHook::TOPBAR_END` or equivalent — confirm in Phase 1) and a small Blade component with Alpine state.

### Out of scope (explicitly)

- **Theme palette / colour token UX** — deferred to a post-beta session. The user wants to discuss the design with their designer first and look at solutions from other products before committing. Current behavior stays.
- **Bulk image uploader** — confirmed not needed for this session. The Media Library page is browse + delete only and the per-field `SpatieMediaLibraryFileUpload` covers the current use cases. A general bulk uploader is a larger feature deferred to a future session.
- **Wiring the new Page/Event thumbnail/header into widgets or templates** — these are storage-only for now. The display surfaces are a future session.
- **Builder-mode auto-collapse sidebar** — the existing post-beta "Builder Chrome & Layer Explorer" stub covers that. The new full-screen toggle is admin-wide and unrelated.

## Phase 6 — Inspector panel max-width

- Add a CSS rule to the inspector panel container that caps its width.
- The exact value is a judgment call — start with something that feels right against the existing inspector content and adjust if it looks cramped. Reasonable starting point: `28rem` to `32rem`.
- The rule should only apply on viewport widths where the layout is wide enough to make the cap meaningful (probably anything above the Filament tablet breakpoint).
- Verify by editing a page with a few widgets and resizing the browser — the inspector should stop growing past the cap and the preview panel should absorb the extra width.

## Acceptance criteria

- Hero widget exposes `nav_link_color` and `nav_hover_color` fields again, and they take effect when the hero is in full-bleed mode.
- An admin who has uploaded a default OG image in CMS Settings sees that image in the `og:image` meta tag for any page that does not have its own OG image and contains no image widgets.
- `pages.og_image_path` column no longer exists in the schema; all existing tests pass after the migration.
- Edit Page, Edit Event, and Edit Post each render with the new single unified fieldset (4 sections collapsed into 1).
- An "Images" section appears above SEO on all three Edit forms, collapsed by default, with the expected fields per the design table.
- The inspector panel does not grow past its configured max-width — extra horizontal space flows to the preview panel.
- An eye icon button appears in the admin topbar to the left of the user menu. Clicking it toggles full-screen mode (collapses left nav + content full-width). Clicking again restores normal layout. The user-menu full-width toggle still exists but the topbar button is the canonical control.
- Fast test suite passes. `npm run build` succeeds.
