## What was built

### Phase 1 ‚Äî Per-source projectors

New namespace `app/WidgetPrimitive/Projectors/`. Each projector owns one source type's shape; `ContractResolver` becomes a thin dispatcher that keeps source routing and the per-call fetch-dedup cache but delegates row/scalar projection.

- `PageContextProjector.php` ‚Äî takes a contract + `?Page`, returns the scalar DTO. Wraps `PageContextTokens`.
- `SystemModelProjector.php` ‚Äî takes a contract + pre-fetched Eloquent collection, returns `['items' => [...]]`. Owns the `projectPost` helper (moved from `ContractResolver`).
- `WidgetContentTypeProjector.php` ‚Äî takes a contract + pre-fetched `CollectionItem` set, returns `['items' => [...]]` with `_media` when the content type declares image fields. Exposes `projectRow(contract, data, media)` as a public method so fallback data (Phase 2) goes through the same per-row projection as live data.
- `ContractResolver` refactored: injects the three projectors, keeps the `match` on `$contract->source` and the per-call cache, delegates everything field-shaped.
- Three new unit-test files under `tests/Unit/WidgetPrimitive/Projectors/` ‚Äî 7 tests total, one fail-closed assertion per projector plus edge cases.

### Phase 2 ‚Äî Fallback semantics moved into the resolver

- `ContractResolver::resolve()` grew a third optional parameter: `array $fallback = []`.
- For `SOURCE_WIDGET_CONTENT_TYPE` contracts that resolve to an empty item-set, the resolver looks up fallback rows ‚Äî matched by `resourceHandle` first, then the first non-empty entry in `$fallback` for back-compat with the renderer's slot-name keying (e.g. `'slides'`) ‚Äî and hands them to `WidgetContentTypeProjector::projectFallback()` so they run through the same field-projection pass as live items.
- `WidgetRenderer` no longer merges `$collectionData[$collSlot]` into `$widgetData['items']` post-resolve. It passes `$fallbackCollectionData` straight through to `resolve()`. The per-slot fallback block in `WidgetRenderer::render()` is gone.
- The Carousel fallback integration test in `tests/Feature/WidgetPrimitiveIntegrationTest.php` now asserts a `secret_internal: 'leak-attempt'` value in fallback data is stripped from the rendered HTML ‚Äî proof that fallback respects declared fields.

### Phase 3 ‚Äî `PageContextTokens` as a container singleton with per-page memoization

- `PageContextTokens::values()` memoizes computed token maps keyed by `$page->id` (with `spl_object_id` fallback for unsaved pages). Twelve widgets on one page compute the token map once, not twelve times.
- Bound as a singleton in `WidgetServiceProvider::register()`, matching the pattern of `WidgetRegistry` and `WidgetConfigResolver`. Every widget on a request shares the cache.
- Three new tests in `tests/Unit/PageContextTokensTest.php`: memoization-within-instance (a mid-render page mutation doesn't disturb the cached values), distinct-page computation, container-singleton identity.
- `PageBuilderApiTest > buildTree eager-loads owner` query count dropped from 30 ‚Üí 28. Threshold tightened from `< 32` to `< 29`.
- The `{{` short-circuit in `WidgetRenderer::configHasTokens()` stays as belt-and-braces ‚Äî memoization is the load-bearing defence.

### Phase 4 ‚Äî Richtext-consumer pattern documented (then superseded)

Initial implementation: added a docblock note on `WidgetDefinition::dataContract()` naming the "declare the full `PageContextTokens::TOKENS` union" pattern, and a bullet under "Design decisions locked in" in `widget-primitive-migration.md` capturing the same observation.

Superseded mid-session ‚Äî see the next entry.

### Source-level capability for `SOURCE_PAGE_CONTEXT` (post-Phase-4 amendment)

User-raised challenge after Phase 4 landed: the "declare the union" pattern was being treated as a permanent category observation when it didn't need to be. `PageContextTokens::TOKENS` is six scalars of public page metadata (`title`, `date`, `excerpt`, `author`, `starts_at`, `location`) ‚Äî not a novel's worth of sensitive data. Per-widget declaration granularity was never meaningful for a bounded set of public metadata.

Amendment, agreed mid-session, applied before close:

- `DataContract::$fields` default changed to `[]` (was required).
- `PageContextProjector::project()` ‚Äî when `$contract->fields === []`, returns the full `PageContextTokens::TOKENS` map. Explicit-fields behaviour still honoured for the narrower case.
- `TextBlockDefinition::dataContract()` ‚Äî dropped the `fields: PageContextTokens::TOKENS` argument and the now-unused `PageContextTokens` import. Contract declaration collapsed to three lines.
- `WidgetDefinition::dataContract()` docblock rewritten: capability grain varies by source. `SOURCE_PAGE_CONTEXT` treats the source itself as the boundary. `SOURCE_SYSTEM_MODEL` and `SOURCE_WIDGET_CONTENT_TYPE` keep per-field declaration because they can expose walkable relationships and arbitrary model columns.
- `widget-primitive-migration.md` bullet rewritten to match.
- Two new tests added to `PageContextProjectorTest`: empty-fields-with-page returns the full token map; empty-fields-with-null-page returns empty strings for every token.
- Existing `ContractResolverTest` indexed-DTO test updated ‚Äî the third contract (previously `fields: []` expecting empty DTO) now omits `fields` entirely and expects the full token map.

Effect: the "declare the union" footnote no longer needs to exist. Adding a new token (e.g. a future `{{github}}`) is a single grep-visible edit to `PageContextTokens::TOKENS` and its `values()` method ‚Äî no sweep across widget definitions.

### Out-of-scope architectural review

Manual testing surfaced two pre-existing issues on Text Block ‚Äî full-width toggle not visibly applying, and Quill alignment classes (`ql-align-*`) not rendering outside `.hero-content`. Both verified as pre-Session-210 behaviour (last touched in session 207). Not in scope for the contract-refinements mandate; flagged for a future housekeeping pass.

A broader architectural check was done at close: the widget-primitive track (contract + slot + resolver) and the sovereign-widget track (directory, manifest, registry, presets, thumbnails ‚Äî sessions 170‚Äì177) are not two tracks but two views of the same move. The contract is what makes a sovereign-distributable widget trustworthy; the sovereign shape is what makes the contract's unit-of-distribution legible. Tensions named for future phases: contract versioning (unsolved, more acute for distributed widgets), appearance outside the contract (visually consequential for installable widgets), mutation still via Filament, slot taxonomy curated by core. None block; all should solidify as Phase 2 onward land. Observation recorded in this log; no architectural doc amendment beyond what Phase 4's bullet already captures.
