## Process rules for every session

Base-prompt process rules (`sessions/template-base-prompt.md`) apply in full. Session-specific notes:

- **296 is a hard prerequisite.** Do not start 297 against a tree where 296 has not landed. Confirm the drift guard exists and is green at start; it must still be green at close. 297 is exactly the "large CSS system built on the build pipeline" that 296 exists to protect — treat the guard as the objective oracle that the bundle matches saved settings.
- **Verify in a real browser, computed values, three surfaces.** Public (`<body class="np-site">` → tokens resolve), Filament admin (no `.np-site` → tokens do **not** leak), page-builder preview (carries `.np-site` → tokens resolve faithfully). Never assert "it works" from the emitted CSS string or `curl` — that was the session-295 failure, and it is now a rule, not a guideline. A run-on-demand computed-value probe (the 295 pattern) is the right tool; promote to an in-suite spec only if an objective outcome warrants it.
- **Delivery mirrors the 295 correction exactly.** Theme color defaults are emitted as `.np-site`-scoped custom properties into the public bundle via `AssetBuildService::collectSources()`, the same path and scoping as the now-corrected typography/buttons (`TypographyCompiler::compileScoped(['.np-site'])` is the pattern). SCSS `$color-*` are demoted to build-time fallbacks (`var(--np-color-x, #{$color-x})`) — alias, do not break, existing `--color-primary` / `--btn-*` consumers.
- **Scope fences — do not cross them:**
  - **No widget consumption migration.** 297 *defines and delivers* the token contract and publishes the widget-dev contract doc. Auditing/migrating the ~38 widgets' SCSS to consume the tokens is **session 298** (the arc's risk session). Touching widget SCSS here is scope drift.
  - **No per-template override / schemes / chrome.** Relocating colors off `templates` removes per-template color *for the interim*; the deliberate-deviation mechanism (theme-defined schemes, the page-shell/chrome model) is **session 299**. Do not build a per-template override path here. The interim loss of per-template color is acceptable and is called out in the session prompt's open questions — confirm, don't silently re-add it.
  - **No public dark mode** (decided: amateur-illegibility footgun; admin dark mode is Filament's separate concern). The palette is single light mode.
  - **`custom_scss` per-template stays untouched** — it remains the bespoke power-user valve.
- **The relocation is a data-preservation operation.** The default template's stored color values must land **byte-faithfully** as the Theme defaults (this is the G1/285-shape concern carried from session 295 — a lost/mutated configured color is silent data loss). Write that regression guard first. Non-default templates' divergent colors: capture/log them before the column drop (they have no Theme home until 299 schemes; surface the count, do not silently discard without a recorded decision).
- **The token list is the agreed contract — do not re-derive it.** The canonical tier-1/tier-2 set is fixed in the session prompt and the arc stub. Naming is `--np-color-*`. This is formalization of existing partial machinery, not a greenfield palette design.
- **Run the fast suite when implementation is complete:** `docker compose exec app php artisan test --exclude-group=slow`. Front-end build per the base-prompt build-decision rule (`resources/scss/**` → `npm run build`; the bundle path → `build:public`). Then verify the three surfaces in a real browser and confirm the drift guard green.
