## Architectural calls (resolved at session start; not relitigated at apply time)

This iteration is the **third in the A001 arc**, branching off main after the user-merged A001/1 (PR #10 — four-entity extension to events / products / navigation / collections) and A001/2 (PR #11 — VERSION hotfix). Two architectural calls were resolved in conversation at the start of /3, not relitigated:

1. **Manifest shape: top-level on existing envelope, not multi-file zip.** Two architectures were on the table — Shape A (manifest as a declarative block alongside `payload`, single-file JSON or single-zip-wrapping-JSON) and Shape B (manifest + sidecar JSON files inside a zip directory tree). Shape A chosen because it's additive on the existing envelope (309/310/A001 work doesn't need to be reshaped), preserves the check-into-git-friendliness of single-file JSON bundles, and the "visible structure" advantage of Shape B is mostly mooted because operators interact through the UI, not by inspecting the bundle directly.
2. **Scope: manifest + SiteSettings together, not forms / widget_presets / unified rollup.** Three phasing options were on the table; Option 2 was selected — manifest infrastructure ships with its first new consumer (SiteSettings) so the design gets real-world exercise immediately. Forms and widget_presets bolt on later as identical-pattern additions once manifest plumbing is in place. The unified-rollup UI is deferred to a separate UI-shaped session (out of A001's hard-no scope: `app/Filament/Pages/`).

Two security-design calls also resolved:

3. **Manifest is documentation + integrity + provenance, NOT a security gate.** A bundle could declare any policy_hints it wants; the importer treats them as UX hints only. The deny-list lives in importer code, never in the bundle. Count mismatches between manifest and payload are logged as warnings, never used to reject the import — a malicious bundle could claim "0 pages, all clean" while smuggling 50, so manifest-count alone is not load-bearing. Real security stays on the importer's defensive checks at write time.
4. **SiteSettings `import_site_settings` opt defaults to FALSE.** Same posture 309 established for `merge_design` — SiteSettings carry brand / identity values an operator may not want overwritten by an inbound bundle, so opt-in is required. The export side has no such default-off restriction; exports are operator-driven by their nature.

## What was built

### Phase 1 — Manifest layer (commit `a5d0c767`)

- **`envelope()` extended** with `?string $sourceAction = null, array $policyHints = []` parameters. New top-level `manifest` block stamped alongside `payload`. All 11 `exportX()` methods updated to pass their own name as source action. `exportSite()` delegates to `exportBundle()` via a new `?string $sourceActionOverride` parameter so the manifest correctly stamps `'exportSite'` rather than `'exportBundle'`.
- **`buildManifest()`** assembles `{sections: [{key, count}], policy_hints: object, exported_with: string, exporter_version: string}` from the payload. `policy_hints` cast to `(object)` for stable JSON serialization shape (empty hints → `{}` not `[]`).
- **`verifyManifest()`** on the importer cross-checks declared section counts against payload reality; returns a list of human-readable warnings (e.g. `"Manifest section 'pages' declared 99 items, payload has 1."`). Stray payload keys not declared in manifest are tolerated (forward-compat for new section types); only count mismatches are flagged.
- **`analyze()`** return shape extended with `'manifest' => array|null` (the full bundle manifest passed through verbatim) and `'manifest_warnings' => string[]` (the verifier output). Old bundles without a manifest return `null` + `[]`.
- **`import()`** runs the verifier ahead of the `DB::transaction()` and logs each warning into `ImportLog`. Mismatches are warning-level only — the import proceeds. Backwards-compatible: bundles without a manifest skip the check entirely.
- **5 new Pest tests** covering: manifest stamping on every exporter, `exportSite` source-action override, analyze surfaces manifest + mismatch warnings, import logs warnings to `ImportLog`, backwards-compat import of pre-manifest bundles.

### Phase 2 — SiteSettings exporter + importer + tests (commit `33a799a6`)

- **Security constants on `ContentImporter`** (the security boundary):
  - `SITE_SETTINGS_ALLOW_LIST` — explicit list of ~30 visual / SEO / routing / publishing-default keys. New keys land here on review only; additions don't auto-flow into bundles.
  - `SITE_SETTINGS_DENY_PREFIXES` — `stripe_`, `qb_`, `quickbooks_`, `mailchimp_`, `mail_`, `resend_`, `build_server_`.
  - `SITE_SETTINGS_DENY_REGEX` — `/(_api_key|_secret|_token|_password|_webhook)/i` tail-guard for keys we haven't named explicitly.
  - `SITE_SETTINGS_DENY_NAMED` — `installation_completed_at`, `horizon_enabled` (install-state flags, not secrets but not portable either).
  - `isSiteSettingKeyDenied(string $key): bool` — single helper used by both exporter (pre-emptive filter) and importer (defensive re-check).
- **`ContentExporter::exportSiteSettings()`** reads `whereIn('key', ContentImporter::SITE_SETTINGS_ALLOW_LIST)`, drops anything `isSiteSettingKeyDenied()` returns true for (defence-in-depth in case allow + deny overlap), drops `type === 'encrypted'` rows. Emits `payload.site_settings: {key: {value, type, group}}` and stamps `policy_hints: {allow_list: [...]}` into the manifest for transparency (the bundle is self-documenting about what slice it carries).
- **`ContentImporter::importSiteSettings()`** runs each inbound key through `isSiteSettingKeyDenied()` regardless of whether it's in the allow-list (the importer never trusts the bundle), hard-skips `type === 'encrypted'` entries with a warning, sanitises values for keys in `SiteSetting::RICH_TEXT_KEYS` via `HtmlSanitizer::sanitize()` (same allow-list boundary `SiteSetting::set()` uses), and writes via `updateOrCreate` + `Cache::forget` matching the model's `set()` convention.
- **`import_site_settings` opt-flag**, **default FALSE** (same posture as `merge_design` per the security design call). Skip-path logs `info` into ImportLog.
- **SiteSettings runs first in the import loop**, ahead of design / templates / pages, so any downstream pass that reads `SiteSetting::get()` sees the imported values.
- **Analyzer extension** — new `site_settings: {keys: string[], count: int, blocked_keys: string[]}` key. `keys` are the inbound keys that would be imported; `blocked_keys` are the inbound keys the deny-list would reject. UI can surface "this bundle contains 28 keys (3 blocked: stripe_secret_key, mail_password, …)" without operator having to crack open the JSON.
- **7 new Pest tests** covering: positive round-trip of the curated slice across six representative keys; secret-leak guard (seeds 9 secret-shaped rows + asserts none appear in the bundle + asserts no sentinel string leaks anywhere via `json_encode`); importer-side re-block of secret-shaped keys in a tampered bundle (with `data-testid`-style sentinel values to prove they didn't land); encrypted-type skip even when not deny-listed; default-OFF opt-flag respects existing values; analyze surfaces both allow + deny lists; allow-list snapshot lands in `policy_hints` for transparency.

## Security considerations

The SiteSettings security model is **belt + suspenders by design** — neither half of the import/export pipeline trusts the other:

- **Exporter side:** allow-list is the *primary* gate (anything not on it is silently dropped). Deny-list is the *secondary* gate applied to whatever the allow-list lets through, catching accidental overlaps. Encrypted-type rows are a *tertiary* gate that catches any row marked sensitive at the model level. All three must pass.
- **Importer side:** `isSiteSettingKeyDenied()` is re-applied to every inbound key regardless of whether it's in the allow-list, because the importer can't assume the bundle came from a trusted exporter. Encrypted-type entries are re-rejected on the inbound side too. A tampered bundle that smuggles `stripe_secret_key` in `payload.site_settings` produces a warning + skip — no DB write.
- **Manifest is NOT a security gate.** The manifest carries `policy_hints` including `allow_list`, but the importer never trusts these — they're for UI display and audit only. Defensive checks always re-derive policy from importer-side constants.
- **No new attack surface beyond what 309 established.** Bundle envelope validation, zip-slip / zip-bomb guards, `format_version` major-bump rejection, and the existing collision policies all carry through unchanged.
- **Rich-text sanitisation at the import seam.** SiteSettings has three RICH_TEXT_KEYS (`dashboard_welcome`, `system_page_content_reset_password`, `system_page_content_email_verify`). Direct `updateOrCreate` bypasses `SiteSetting::set()`'s sanitiser, so `importSiteSettings()` calls `HtmlSanitizer::sanitize()` explicitly for those keys — matching the model's through-the-set-method behaviour.

## Deferred decisions / carry-forward

- **Forms (custom form definitions)** — identified during planning as the next content-shaped section to add. Same pattern as A001 phases; defer to a follow-up session.
- **Widget presets** — identified during planning. Depends on whether the operator's install actually uses them (vs seeded-only). Defer.
- **Custom field defs** — CRM-shaped schema config; needs a CRM-vs-CMS scope call. Defer.
- **Unified rollup UI extension.** `SiteImportExportPage` (310) currently runs `exportSite()` which enumerates pages + templates + optional design + media. To carry the new SiteSettings section through the rollup, the UI would need a new toggle ("Include site settings?") + `exportSite()` would need to invoke `exportSiteSettings()` and merge its `payload.site_settings` into the rollup envelope. Both are mechanical; deferred to a separate UI-shaped session per A001's hard-no scope (`app/Filament/Pages/`).
- **Per-section integrity hashes in manifest.** Discussed but not implemented for v1. The current manifest carries `count` per section; a future addition could carry `sha256` per section so import can detect partial edits. Additive; no envelope churn needed.
- **Manifest schema versioning.** Currently the manifest carries `exporter_version` (the bundle's `FORMAT_VERSION`). If the manifest's own shape evolves independently of the format_version, a separate `manifest_version` field can be added. Not needed at v1.
