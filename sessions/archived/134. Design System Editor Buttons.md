# 134. Design System Editor: Buttons

This session launches the **Design System Editor** — a tabbed admin page that will grow over time to cover typography, spacing, and other site-wide style controls. This session builds the framework page and the first functional tab: Buttons.

---

## Design decisions (resolved before starting)

- **The Design System Editor lives under Tools** in the admin navigation. CMS is too crowded and will be reorganised in a future taxonomy session — Tools is the interim home.
- **Five built-in button variants** ship with the system: primary, secondary, text, destructive, link. Users customise these but cannot create new variants in this session. The architecture should not preclude user-created variants later (variant definitions stored as a JSON array on a site setting, compiled into `.btn--{handle}` rules).
- **The user controls appearance, the system controls structure.** Users configure colors, radius, weight, hover, borders. The system manages padding, sizing, overflow, display, transitions, and font family.
- **Contextual size modifiers** (`.btn--sm`, `.btn--lg`) are system-managed, applied by widget templates. They are never exposed in the editor.
- **This is the beginning of a design system editor**, not a button-only tool. The page framework and tab pattern must accommodate future tabs cleanly.

## Phase 2 — Button variant editor

The Buttons tab displays the five built-in variants in an editable form. Each variant gets a collapsible section with the following controls:

**Per-variant user controls:**

- **Border-radius**: category picker — sharp (`0`), slightly rounded (`0.25em`), rounded (`0.5em`), pill (`999px`). Em-based values so they scale with contextual font size.
- **Background color**: hex color input (for primary, destructive; transparent for secondary, text, link)
- **Text color**: hex color input
- **Border color**: hex color input (relevant for secondary; hidden for variants with no border)
- **Border width**: select — 0, 1px, 2px
- **Hover behavior**: preset picker — darken, lighten, opacity. Not custom CSS.
- **Font weight**: select — normal (400), semibold (600), bold (700)
- **Text transform**: select — none, uppercase

**Storage:** JSON structure on a site setting key (e.g. `button_styles`). Each variant is keyed by handle with its style overrides. Example structure:

```json
{
  "primary": {
    "border_radius": "rounded",
    "bg_color": "#0172ad",
    "text_color": "#ffffff",
    "border_color": null,
    "border_width": "0",
    "hover": "darken",
    "font_weight": "600",
    "text_transform": "none"
  },
  "secondary": { ... },
  "text": { ... },
  "destructive": { ... },
  "link": { ... }
}
```

**Defaults:** If no custom settings exist, the editor pre-fills with values matching the current hardcoded SCSS. Saving writes the setting; the build pipeline reads it.

Each variant section should include a **live preview strip** showing the button in its current configured state (rendered inline, not in an iframe — just styled `<span>` or `<button>` elements reflecting the current form values via Alpine.js).

## Phase 3 — Icon buttons & form-append buttons

**Icon buttons** — a separate section below the variant editor:

- Icon size relative to button text size (select: match text, slightly larger, 1.5x)
- Icon placement: left of text, right of text, icon-only
- Mobile collapse option: a toggle that causes icon+text buttons to collapse to icon-only below `$bp-md`
- `.btn--icon-only`: square aspect ratio, centered icon

**Form-append buttons** — a separate section:

- Buttons that attach to the beginning or end of form inputs (search bars, filter fields, URL inputs)
- Inherit border-radius from adjacent input on the joined side, apply button variant radius on the outer side
- Height matches input height automatically
- Variant picker (which button variant to use for form-append by default)

## Phase 4 — SCSS compilation wiring

Wire the saved button settings into the public CSS build pipeline:

- When button settings are saved, generate CSS custom properties and/or `.btn--{variant}` rule overrides
- The build server compiles these into the public widget CSS bundle
- Update `resources/scss/_buttons.scss` to read from CSS custom properties where appropriate, falling back to current hardcoded defaults when no settings exist
- Run `php artisan build:public` after saving to recompile

**Button base rules** (enforced in SCSS, not user-configurable):

- Default overflow: `white-space: nowrap; overflow: hidden; text-overflow: ellipsis; max-width: 100%;`
- `.btn--wrap`: allows two-line text (`white-space: normal; text-align: center;`) — applied by widget templates
- `.btn--block-mobile`: full-width below `$bp-md` — applied by widget templates contextually
- Focus ring on all variants (accessibility)
- Consistent disabled state across all variants

**Auto external link icon:** Update the shared button component (`resources/views/widgets/components/buttons.blade.php`) to detect outbound URLs (not matching the site domain) and append an external link icon automatically.
