# 086. Column Widget & Widget Picker UX

Build a column layout widget with unlimited child nesting and CSS grid presets, add a universal `style_config` layer for per-widget padding and margin applied by the renderer, and surface those controls in every PageBuilderBlock behind a show/hide advanced panel.

---

## Design decisions (resolved before starting)

- `page_widgets` gains three columns: `parent_widget_id` (uuid nullable, self-referential FK, set-null on parent delete), `column_index` (smallint unsigned nullable, default null), `style_config` (jsonb, default {}).
- `style_config` is universal — padding (top/right/bottom/left) and margin (top/right/bottom/left), stored as individual pixel-value keys, applied as inline styles on every widget's wrapper div by the renderer. Not specific to column widgets.
- Column widget uses CSS grid only. No display mode selector. Proportions are defined by a `grid_template_columns` string. Four presets: "Equal halves" (`1fr 1fr`), "Equal thirds" (`1fr 1fr 1fr`), "2/3 + 1/3" (`2fr 1fr`), "1/3 + 2/3" (`1fr 2fr`). Selecting a preset writes to `grid_template_columns`; the text input always overrides.
- No row controls of any kind — row height is determined by content.
- Nesting is unlimited. No depth limit in code or UI.
- Drag-and-drop at root level continues to work on collapsed column containers. No drag-and-drop within column slots — up/down buttons are the fallback for child widget reordering.
- The advanced controls panel (style_config) is hidden in every PageBuilderBlock behind a small toggle button. `x-show` / hidden div pattern — no lazy loading.
- Padding/margin control: one "all sides" shorthand input at the top sets all four individual T/R/B/L values when changed. Individual inputs remain independently editable. When all four match, shorthand shows that value; when they differ, shorthand placeholder reads "mixed".
- Column widget blocks default to `open` in the page builder (`default_open: true` in seeder).
- `resources/docs/widget-types.md` receives a full architecture section immediately after this session is built. The file will be split in a future session — write everything into it now.

Add to `page_widgets`:
- `parent_widget_id` — uuid nullable, FK to `page_widgets.id`, `onDelete('set null')`
- `column_index` — smallint unsigned nullable, default null
- `style_config` — jsonb, default {}

Run migration via Docker. Update `docs/schema.md`.

## Phase 2 — PageWidget model

Add `parent_widget_id`, `column_index`, `style_config` to `$fillable`. Cast `style_config` as array. Add `parent()` (belongsTo self) and `children()` (hasMany self, ordered by `column_index` then `sort_order`) relationships.

## Phase 3 — Rendering pipeline update

Before writing anything, read the PageController and page view template to understand the current widget rendering path.

Changes needed:
- Load root widgets (`parent_widget_id = null`) for the page, with children eager-loaded (ordered by `column_index`, `sort_order`).
- Apply `style_config` padding/margin as inline styles on each widget's wrapper div. This is universal — every widget receives this treatment regardless of type.
- The column widget template receives its child PageWidget records grouped by `column_index`. Determine the right mechanism for passing child data into the template (via the controller, PageContext, or a variable injected at render time) before implementing.

## Phase 4 — Column widget type

Add `column_widget` to `WidgetTypeSeeder`:

- label: `Column Layout`
- render_mode: `server`
- collections: `[]`
- default_open: `true`
- config_schema: `num_columns` (number, "Number of columns"), `grid_template_columns` (text, "Column widths (e.g. 1fr 1fr)")

Create `resources/views/widgets/column-widget.blade.php`. The template renders a `<div style="display:grid; grid-template-columns:{{ $config['grid_template_columns'] ?? '1fr 1fr' }}">` and iterates child widgets grouped by `column_index`, rendering each child's output into its column. Before implementing, read how the existing rendering pipeline produces widget output — use the same mechanism for child rendering.

Run `db:seed --class=WidgetTypeSeeder` via Docker after adding the entry.

---

## Phase 5 — PageBuilderBlock: style_config panel and advanced toggle

- Update `loadBlock()` to include `style_config` from the PageWidget record.
- Update `PageBuilder::loadBlocks()` to include `style_config` in the block map.
- Add a small toggle button to the block header (visually separate from the label and ellipsis menu) that shows/hides the advanced panel via `x-show`.
- Advanced panel contains the padding/margin Alpine component: shorthand input + four individual T/R/B/L inputs for each property. Shorthand drives the individuals; individuals remain editable. See design decision above for shorthand/mixed behaviour.
- Wire `style_config` persistence on change: update `page_widgets.style_config` the same way `config` is already persisted.

---

## Phase 6 — Column widget admin UI

When `$block['widget_type_handle'] === 'column_widget'`, the PageBuilderBlock body renders a column-specific UI:

- Preset picker: four buttons (Equal halves, Equal thirds, 2/3 + 1/3, 1/3 + 2/3) that write the corresponding `grid_template_columns` string to `block.config.grid_template_columns` via Livewire.
- `grid_template_columns` text input (wired to `block.config.grid_template_columns`).
- `num_columns` number input (wired to `block.config.num_columns`), controlling how many column slot panels appear.
- N column slot panels (N = `block.config.num_columns ?? 2`), each showing:
  - A column label ("Column 1", "Column 2", …)
  - The child PageBuilderBlock list for that slot (`parent_widget_id = this block's id`, `column_index = N`)
  - An "Add Block" button scoped to that slot

`PageBuilder::openAddModal()` and `createBlock()` must accept optional `parentWidgetId` and `columnIndex` parameters. When set, `createBlock()` writes `parent_widget_id` and `column_index` on the new PageWidget rather than creating a root-level block.

Child block up/down reordering operates among siblings within the same slot (same `parent_widget_id` and `column_index`). Ensure the existing move-up/move-down logic respects this scope.

## Phase 7 — Docs

Add a section to `resources/docs/widget-types.md` covering:

- The column widget: purpose, configuration, grid presets, how columns are assigned to children.
- The `parent_widget_id` / `column_index` schema and nesting model, including unlimited depth behaviour.
- The `style_config` layer: what fields it holds, how the renderer applies it universally, and that it is available on all widget types.
- The advanced controls panel: how to reveal it, what it controls.
- Operational notes (e.g. deleting a column widget sets `parent_widget_id` to null on its children — they become orphaned root-level widgets rather than being deleted).

## Security checklist

Before closing:

- [ ] Column widget template does not render raw HTML from `style_config` — all values are applied as measured CSS properties only.
- [ ] `parent_widget_id` is validated on child widget creation to belong to a widget on the same page — a widget on page A cannot be parented to a widget on page B.
- [ ] The Add Block flow for column slots applies the same widget-type permission checks as the root-level Add Block flow.

---

## Out of scope

- Drag-and-drop reordering within column slots (up/down buttons are the fallback)
- Flex column layout variant (future session)
- `grid-template-rows` or any row controls
- Moving a child widget out of a column slot to root level via the UI
- Persisting per-instance expanded/collapsed state
- Splitting `widget-types.md` into multiple files (future housekeeping)
