## What was built

Two public-site mobile-nav defects, both diagnosed live in a real browser at phone viewports before any code changed. The first task was establishing **which of two nav surfaces actually renders**: the homepage chrome is a **CMS chrome page** (rendered via `ChromeRenderer`), so the live header/footer are the **CMS Nav widget** (`app/Widgets/Nav`) — `widget-nav--horizontal` in the header, `widget-nav--columns` in the footer. The hardcoded `components.site-header`/`site-footer` Blade chrome is **dead on this install** and was correctly left untouched.

### Hamburger — rebuilt as a CSS-only morphing drop-down

**Original break (root-caused in browser):** the hamburger toggled Alpine state fine, but the menu never appeared. `.widget-nav__mobile` carried a stylesheet `display: none`, and Alpine's `x-show` only clears the *inline* display — so the stylesheet rule kept winning and the menu stayed `display: none` (links rendered 0×0). Compounding it, the nav widget lives in the header's narrow `auto 1fr` grid cell (~155px on a 390px screen), so even when shown the menu was crammed into the right ~40%.

**Iteration (recorded because the dead-ends are instructive):**
- First attempt — remove the base `display: none` and make the menu a `position: fixed` full-screen overlay with a separate close-X and a body/`html` scroll-lock. **Rejected by the owner**: full-screen "looks simpler but is actually more complicated." It dragged in a scrollbar/X **hop** (the overlay's `inset:0`/`right:0` shifted when the page scrollbar was removed), a **double-gutter** scrollbar (`scrollbar-gutter: stable` reserved an empty gutter beside the overlay's own scrollbar), a **separate close button** mispositioned vs the hamburger, and a covered hamburger whose morph animation never showed.

**Final solution** (the owner's pattern — morph, don't swap; reference example they provided):
- **Three siblings**, no JS: a visually-hidden `<input type="checkbox" class="widget-nav__toggle">` holds open/closed state; the hamburger and the menu are siblings driven by plain `:checked ~` selectors.
- **One hamburger that morphs to the X in place** — the `<label class="widget-nav__hamburger">` stays put (verified x=323,y=24 closed *and* open) and its bars rotate to an X via `:checked`. **No separate close button.**
- **Absolute drop, anchored full-width below the header.** Removed `position: relative` from `.widget-nav` so the drop anchors to the nearest containing block — the full-width `.page-layout` (it carries `container-type: inline-size`) — and `top: 100%; left: 0; right: 0` drops it full width (x=0, w=390) instead of trapped in the nav cell. **No `position: fixed`, no `html`/`body` scroll-lock**: the nav is never fixed or sticky, so the drop lives in the document and scrolls with the page; a tall menu just extends the page's own scroll. The whole scroll-lock/scrollbar/hop class of problems disappears because nothing leaves the document.
- Desktop dropdowns stay on Alpine (they anchor to `.widget-nav__item`'s own `position: relative`, unaffected by the nav-root change). Sub-menu expanders are nested `:checked` checkboxes (mobile path fully JS-free). `aria-controls` on the checkbox names the menu it operates.

### Footer columns — left-aligned via the existing alignment control

**Break:** the `widget-nav--columns` footer read off-kilter — centered, wrapping raggedly on mobile.
- A viewport `@media (max-width:768px){ flex-direction: column }` collapse was tried and **reverted**: it put a viewport `@media` in **widget** SCSS, which the `WidgetStyleBoundaryConsumptionTest` guard bans and which runs against the 332 "no width-based widget collapse" principle.
- A hardcode to `justify-content: flex-start` was tried and **reverted** in favor of keeping the operator's alignment control. The widget already maps the alignment config left/center/right → `flex-start`/`center`/`flex-end` via `--nav-justify`, so the fix is config-side: set the footer Nav widget's alignment to **left** (→ `flex-start`). `.widget-nav__columns` keeps `justify-content: var(--nav-justify, flex-start)`.

### Standing guard

`tests/Feature/NavMobileCollapseTest.php` (design group; `DESIGN_GROUP_FILES` 22 → 23): a build-pipeline-independent source-level guard on `app/Widgets/Nav/styles.scss` — asserts the CSS-only reveal (`.widget-nav__toggle:checked ~ .widget-nav__mobile { display: block }`) and that the menu is `position: absolute`, **not** `fixed` (locking the session's "in-document drop, not a fixed overlay" decision so the rejected pattern can't silently return).
