## Design decisions (resolved before starting)

- **Default value fix:** The current Livewire inspector does not populate form fields with defaults from the widget type's `config_schema`. Fields appear blank even when a default is defined. The Vue inspector must fix this: when rendering a field, if `widget.config[field.key]` is `undefined` or absent, use `field.default` from the schema. This applies to all field types.
- **Debounced saves:** Config changes are saved via the existing `PUT /admin/api/page-builder/widgets/{id}` endpoint. Saves are debounced — accumulate changes for 500ms of inactivity before sending. The widget is marked dirty in the store after save (preview not auto-refreshed; user clicks "Apply Changes").
- **Select options resolution:** Select fields with `options_from` need their options resolved. Use the existing `GET /admin/api/page-builder/data-sources/{source}` endpoint. For `collection_fields:*` sources, use the `GET /admin/api/page-builder/collections/{handle}/fields` endpoint filtered by the `depends_on` field's current value. Options are resolved on mount and re-resolved when the dependency field changes.
- **Richtext (Quill) integration:** Port the Quill setup from `resources/js/page-builder/richtext-editor.js`. Quill is loaded from CDN (already in the page via Filament). The Vue component initialises Quill in `onMounted`, syncs content changes to the store, and handles inline image upload using the existing `inline_image_upload_url` from bootstrap data.
- **Conditional visibility:** Fields support `shown_when` and `hidden_when` properties that reference other config keys. The Vue field wrapper evaluates these against the current widget config to show/hide fields reactively.
- **Layout groups:** Fields with the same non-semantic `group` value (not "content" or "appearance") are rendered in a 2-column grid, matching the existing Blade `inspector-field-group` behaviour.
- **Advanced fields accordion:** Fields with `advanced: true` are grouped into a collapsible "Carousel Settings" accordion below the primary fields, matching the existing Blade behaviour.
- **Inspector placement:** The Vue inspector renders inside `App.vue` in the right pane (same grid layout as the Livewire editor: `2fr 1fr`). It appears in both edit and handles modes. The Livewire inspector continues to work above in the Livewire section — both are visible during this transition.
- **Label editing:** The inspector header includes inline label editing (click pencil icon to edit, Enter/OK to save, Escape/Cancel to discard). Label saves go through the same debounced `updateWidget` action.

## Phase 1 — Inspector shell components

### 1a. `InspectorPanel.vue`

Create `resources/js/page-builder-vue/components/InspectorPanel.vue`:

- Shows "Select a block to edit its settings." placeholder when no block is selected
- When a block is selected: renders `InspectorHeader`, `InspectorTabs`, tab content, and `ApplyChangesButton`
- Sticky positioning and scroll behaviour matching the existing inspector (the parent container in `App.vue` handles `position: sticky`)

### 1b. `InspectorHeader.vue`

Create `resources/js/page-builder-vue/components/InspectorHeader.vue`:

- Props: `widget` (the selected Widget object)
- Displays the widget type label as a small uppercase badge
- Inline label editor: display mode shows label with pencil icon; edit mode shows input with OK/Cancel buttons
- Label save calls `store.updateWidget(id, { label })` — goes through the same debounced save path

### 1c. `InspectorTabs.vue`

Create `resources/js/page-builder-vue/components/InspectorTabs.vue`:

- Two tabs: "Content" and "Appearance"
- Active tab state managed locally (default: "content")
- Visual style matches the existing Blade tab bar (rounded-top tabs with border treatment)
- Emits `update:activeTab` or uses v-model for the parent to know which tab is active

## Phase 2 — Field infrastructure

### 2a. `InspectorFieldGroup.vue`

Create `resources/js/page-builder-vue/components/InspectorFieldGroup.vue`:

- Props: `fields` (array of FieldDef), `widget` (Widget)
- Splits fields into primary (no `advanced` flag) and advanced (`advanced: true`)
- Renders primary fields with layout grouping: consecutive fields with the same non-semantic `group` value render in a 2-column grid
- Renders advanced fields inside a collapsible accordion labelled "Carousel Settings"
- Each field rendered via `InspectorField`

### 2b. `InspectorField.vue`

Create `resources/js/page-builder-vue/components/InspectorField.vue`:

- Props: `field` (FieldDef), `widget` (Widget), `modelValue` (the current config value for this field)
- Emits: `update:modelValue`
- Renders the field label
- Handles conditional visibility: evaluates `field.shown_when` and `field.hidden_when` against `widget.config` to show/hide
- Dispatches to the correct typed field component based on `field.type`
- **Default value handling:** if `modelValue` is `undefined` and `field.default` is defined, emit the default value on mount so the parent populates it

### 2c. Field type mapping

| `field.type` | Component | Notes |
|---|---|---|
| `text` | `TextField.vue` | Standard text input |
| `url` | `TextField.vue` | Same component, `type="url"` |
| `textarea` | `TextareaField.vue` | 4-row textarea |
| `number` | `NumberField.vue` | Number input |
| `select` | `SelectField.vue` | Dropdown with option resolution |
| `toggle` | `ToggleField.vue` | Checkbox with "Enabled" label |
| `checkboxes` | `CheckboxesField.vue` | Multi-select checkbox grid |
| `notice` | `NoticeField.vue` | Read-only info/warning display |
| `richtext` | `RichTextField.vue` | Quill editor |
| `image` | Fallback message | "Image fields available in next update" |
| `video` | Fallback message | "Video fields available in next update" |
| `color` | Fallback message | "Color fields available in next update" |
| `buttons` | Fallback message | "Button fields available in next update" |

Unsupported types show a small muted notice rather than breaking.

## Phase 3 — Field components

### 3a. `TextField.vue`

- Props: `field`, `modelValue`
- Input type derived from `field.type` (`text` or `url`)
- Emits `update:modelValue` on input (parent debounces the save)

### 3b. `TextareaField.vue`

- Props: `field`, `modelValue`
- 4-row textarea
- Emits `update:modelValue` on input

### 3c. `NumberField.vue`

- Props: `field`, `modelValue`
- Number input with `min` from `field.min` if present
- Emits `update:modelValue` on input

### 3d. `SelectField.vue`

- Props: `field`, `widget`, `modelValue`
- If `field.options` is present, use those directly
- If `field.options_from` is present, resolve via `GET /admin/api/page-builder/data-sources/{source}` on mount
- For `collection_fields:*` sources: resolve via `GET /admin/api/page-builder/collections/{handle}/fields` using the value of the `depends_on` config field; re-resolve when that dependency changes
- First option: "— Select —" with empty value
- Emits `update:modelValue` on change

### 3e. `ToggleField.vue`

- Props: `field`, `modelValue`
- Checkbox input with "Enabled" label
- Emits `update:modelValue` on change (boolean)

### 3f. `CheckboxesField.vue`

- Props: `field`, `modelValue` (array of selected values)
- Renders checkbox grid with `field.columns` (default 2) columns
- Options from `field.options`
- Emits `update:modelValue` with the updated array on toggle

### 3g. `NoticeField.vue`

- Props: `field`
- Read-only display: info (blue) or warning (amber) variant based on `field.variant`
- Renders `field.content` as HTML (matching the existing Blade `{!! $content !!}` pattern)
- No model value — purely informational

### 3h. `RichTextField.vue`

- Props: `field`, `widget`, `modelValue`
- Initialises Quill in `onMounted` with the same toolbar config as `richtext-editor.js`
- Inline image upload handler: uses the `inline_image_upload_url` from bootstrap data (store it in the Pinia store or read from the mount element's data attribute)
- Syncs Quill content changes to `update:modelValue` (debounce the emit to avoid excessive updates)
- Updates Quill content when `modelValue` changes externally (guard against echo loops)
- Destroys Quill instance in `onUnmounted`

## Phase 4 — Store changes and wiring

### 4a. Update `stores/editor.ts`

- Add a debounced save mechanism: when a config field changes, accumulate changes and call `updateWidget` after 500ms of inactivity
- Add `updateLocalConfig(widgetId, key, value)` action that updates the widget's config in the store immediately (for reactive UI) and queues the debounced API save
- Track `inlineImageUploadUrl` from bootstrap data (needed by RichTextField)

### 4b. Update `App.vue`

- In edit mode: render `PreviewCanvas` (left, 2fr) and `InspectorPanel` (right, 1fr) in a grid layout
- In handles mode: render `BlockListPoc` (left, 2fr) and `InspectorPanel` (right, 1fr) in a grid layout
- The inspector is shared between both modes

### 4c. `ApplyChangesButton.vue`

Create `resources/js/page-builder-vue/components/ApplyChangesButton.vue`:

- Sticky at the bottom of the inspector scroll area
- Calls `store.refreshPreview(store.selectedBlockId)` on click
- Disabled state when no block is selected or no dirty widgets

## Acceptance criteria

- Inspector panel renders when a block is selected (in both edit and handles modes)
- "Select a block" placeholder shown when nothing is selected
- Widget type badge and inline label editor work in the header
- Content and Appearance tabs display correct fields from the widget's `config_schema`

- Advanced fields render in a collapsible accordion
- Richtext field initialises Quill with toolbar and inline image upload
- Config changes persist to the database via debounced API calls
- Widget is marked dirty after config change
- "Apply Changes" triggers preview refresh for the selected widget
- Unsupported field types (image, video, color, buttons) show a graceful fallback message
- Existing Livewire editor and inspector above the Vue section continue to work
