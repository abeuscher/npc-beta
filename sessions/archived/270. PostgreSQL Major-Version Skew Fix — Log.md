## What was built

Emergency unblocker for FM 021. The app container's `pg_dump` (resolved to client 17 from Debian Trixie's `postgresql-client` meta-package) was producing dumps containing the PostgreSQL 17-only `transaction_timeout` directive against the project's `postgres:16-alpine` server. Result: the source CRM literally couldn't restore from its own backups, and FM 021's `fleet:restore-from-blob` command died at the load step.

Fix pinned both ends explicitly and aligned them on PostgreSQL 17:

- [Dockerfile](Dockerfile) — `postgresql-client` (unpinned meta-package) → `postgresql-client-17` with a coupling comment naming the compose-file dependency.
- [docker-compose.yml](docker-compose.yml) and [docker-compose.prod.yml](docker-compose.prod.yml) — `postgres:16-alpine` → `postgres:17-alpine` with matching coupling comments.
- New test: [tests/Feature/Infrastructure/PostgresVersionSkewTest.php](tests/Feature/Infrastructure/PostgresVersionSkewTest.php) asserts `pg_dump --version` major matches `SHOW server_version` major. Skips cleanly if either is unreachable. Catches future drift mechanically.
- Bumped the queued housekeeping cluster's session numbering 270/271/272/273 → 271/272/273/274 since this session lifted into the 270 slot.

## In-session findings + remediations

- **The original upgrade runbook drafted mid-session was wrong for the deploy environment.** It assumed local docker-compose semantics on the deploy droplet, used generic container references, and recommended `docker compose down -v` — which on the prod compose file would have wiped not just `postgres_data` but also `storage_data` (the spatie media tree + backup blobs at `storage/app/private/`), `bootstrap_cache`, `redis_data`, and `widget_build`. The runbook also missed that the GitHub deploy workflow uses `docker compose up -d --wait --remove-orphans`, which would hang indefinitely on a PostgreSQL 17 binary trying to start on a PostgreSQL 16 data directory. I had not consulted [docs/app-reference.md](docs/app-reference.md) or [.github/workflows/deploy.yml](.github/workflows/deploy.yml) before writing it. User correctly called this out. The runbook was deleted at close in favour of the destructive shortest-path procedure, which is now adequately covered by the existing "Need to wipe and start over" section of [docs/deployment/digital-ocean-droplet-install.md](docs/deployment/digital-ocean-droplet-install.md).

- **User-confirmed posture: pre-Beta, no live data, prefer destructive shortest path over data preservation.** Worth pinning as a memory — the careful-data-preservation framing wastes time and produces friction that doesn't pay for itself at this stage.

- **Deploy-server cred mismatch surfaced post-deploy.** After the user wiped + redeployed the prod droplet, admin login failed despite `/opt/nonprofitcrm/.env` showing the correct values. Root cause was the env-propagation gotcha already documented in [docs/app-reference.md](docs/app-reference.md): `docker-compose.prod.yml` wires `.env` via `env_file:` (read at container start, not mounted live). The seeder's [admin-block](database/seeders/DatabaseSeeder.php#L36-L53) reads `ADMIN_EMAIL` / `ADMIN_PASSWORD` via `env()` and silently no-ops when either is empty. Fix was `docker compose -f docker-compose.prod.yml restart app worker` followed by `docker exec nonprofitcrm_app php artisan migrate:fresh --seed --force`. The doc was right; the trap is real.

- **The PostgreSQL 17 dump round-trip was verified end-to-end locally before push.** Production deploy was carried out by the user wiping + redeploying. The deploy attempt initially threw an error (cause: env-propagation as above), remediated via restart + reseed.

## Production deployment notes

**Both droplets** (prod `167.172.141.225`, public-demo `147.182.214.147`) needed a destructive wipe + redeploy because the existing PostgreSQL 16 data directory is not readable by the PostgreSQL 17 binary. The shortest-path sequence used:

```bash
ssh root@<droplet> 'cd /opt/nonprofitcrm && docker compose -f docker-compose.prod.yml down -v'
# merge to main (or fast-forward release/public-demo + push) → deploy workflow runs
ssh root@<droplet> 'docker exec nonprofitcrm_app php artisan migrate:fresh --seed --force'
```

If `.env` has been edited since last container start, add `docker compose -f docker-compose.prod.yml restart app worker` between the workflow completion and the seed step — otherwise the admin-block of the seeder reads stale env values and the new admin user either has the wrong password or doesn't get created at all.

This procedure is destructive and acceptable only because the project carries no live data at this stage. Future major-version bumps after Beta-1 launch will need a real dump-and-restore procedure; defer that until the situation calls for it.

## Findings register

- **Env propagation gotcha** is real and bites. The doc was right; the trap surfaces specifically when (a) you bring up the stack, (b) edit `.env`, (c) run a seeder or any artisan command that reads `env()`. The artisan command sees the container's start-time environment, not the file. Restart the relevant containers first.
- **`docker compose down -v`** removes every named volume in the compose file. On this project's prod compose that's five volumes (postgres_data, redis_data, storage_data, bootstrap_cache, widget_build). For a real-data install this would be catastrophic; for the current pre-Beta posture it's the cleanest path. Worth flagging structurally if/when real data lands.
- **Deploy workflow uses `up -d --wait`.** Healthcheck timeouts will fail the deploy. Any future schema-breaking infrastructure change must verify the postgres healthcheck passes on the post-deploy state.
