## What was built

### Phase 1 — Structural unification: polymorphic widget ownership

Collapsed the two storage shapes (pages own widget rows, templates own JSONB definitions) into a single polymorphic model so both kinds of stack share one editor, one API, and one data shape. The session prompt's original "template-aware API" approach was replaced on open with this structural unification after discussing the design with the user; we're pre-beta and change is cheap.

- **Migration** `2026_04_15_120000_polymorphic_widget_owner.php`:
  - Added `owner_type` + `owner_id` columns (indexed) to `page_widgets` and `page_layouts`.
  - Backfilled page-owned rows (`owner_type='App\Models\Page'`, `owner_id=page_id`).
  - Hydrated every existing content `Template.definition` JSONB into polymorphic rows owned by the template.
  - Dropped `page_id` from both tables and `definition` from `templates`.
- **Models:**
  - `PageWidget` + `PageLayout` gained `owner()` morphTo and `forOwner(Model $owner)` scope.
  - `Page` and `Template` both gained `widgets()` and `layouts()` morphMany relations. `Page::pageWidgets()` renamed to `Page::widgets()`.
  - Added `PageWidget::copyOwnedStack(Model $source, Model $target)` — a generalisation of the old `copyBetweenPages()`, now handling Page→Page, Page→Template, Template→Page copies including attached spatie media. Old `serializeStack()` and `copyBetweenPages()` removed.
  - Added `PageWidget::resolveAppearance(array $stored, ?string $handle)` — returns `defaultAppearanceConfig()` from the widget's definition when the stored appearance is empty. Wired into every production widget-creation path so freshly hydrated widgets carry their widget type's defaults (150px padding on text blocks, etc.).
- **Controller + routes:**
  - `PageBuilderApiController` generalised to accept a polymorphic owner. Routes split into `pages/{owner}` and `templates/{owner}` prefixes behind a new `ResolvePageBuilderOwner` middleware that resolves the `{owner}` param to the correct model based on the URL segment.
  - Widget-keyed routes (`PUT widgets/{widget}`, etc.) unchanged — they infer owner from `$widget->owner`.
- **Livewire bootstrap:** `PageBuilder` component now takes `ownerId` + `ownerType`; legacy `pageId` prop still works (resolves to `ownerType='page'`). Bootstrap data exposes `owner_id` / `owner_type` and splits `api_base_url` (owner-scoped) from `api_lookup_url` (widget-types / collections / etc.).
- **Vue API client:** `api.ts` now configures both `baseUrl` (owner-scoped) and `lookupUrl`. Owner-scoped calls (get widgets, create widget, reorder, create layout) use base; everything else (widget-keyed CRUD, preview, lookups, presets, swatches, image upload) uses lookup. Store exposes `ownerId`; Vue components emit `ownerId` in custom events; Blade handlers match on `ownerId`.
- **Cascade:** added `deleting` observer on `Page` (respecting `isForceDeleting`) and `booted::deleting` on `Template` to tear down owned widget + layout rows, replacing the previous DB-level FK cascade.
- **Every query updated.** Roughly 109 references across 41 files: controllers, services, seeders, import/export, tests. `$page->pageWidgets()` → `$page->widgets()`, `PageWidget::where('page_id', …)` → `$page->widgets()` or `PageWidget::forOwner($page)`, `PageWidget::inSlot($pageId, …)` → `PageWidget::inSlot($owner, …)`.
- **Import/export:**
  - `ContentExporter` now serialises content-template widget trees as `widgets` (polymorphic rows) instead of the removed `definition` JSONB.
  - `ContentImporter` accepts either the new `widgets` key or the legacy `definition` key for backwards-compatible bundles; hydrates into owned rows via generalised `hydrateRootWidgetForOwner` / `hydrateLayoutForOwner` methods.
- **Seeders:** `TemplateSeeder` now creates template widgets as polymorphic rows instead of writing a JSONB definition. Wires `default_content_template_post` site setting to the seeded Blog Post template on fresh install (only if not already configured).

### Phase 2 — Editor UI for content templates

- `EditContentTemplate` replaced with a real editor: the existing Details form (name, description) sits at the top; below it, the full Vue page-builder mounts with the Template as polymorphic owner. Designers can now add, configure, reorder, and remove widgets directly on a content template with zero special-case code in the Vue app.
- New blade: `resources/views/filament/resources/template-resource/pages/edit-content-template.blade.php`.

### Phase 3 — Creation-time selector + per-type defaults

- Removed the blog-post exclusion — posts now expose the `content_template_id` selector on creation, identical to pages.
- Select options: "None (blank)" + every content template. The select's **initial value** is pre-populated with the per-type default (`default_content_template_default` for pages, `default_content_template_post` for posts) resolved from `site_settings`. User sees the template name that will be used — no "— Default —" indirection.
- `CreatePage::afterCreate()` and `CreatePost::afterCreate()` simplified to a single lookup-by-UUID branch: UUID → `copyOwnedStack`, 'none' → do nothing. No server-side default resolution needed (the form pre-populated it).
- Event-landing-page creation (`EventResource::createLandingPageForEvent`) rewritten to use `copyOwnedStack` from the "Event Landing Page" template; hardcoded fallback kept for installs where the template is missing or empty.
- New CMS Settings section "Default Content Templates" with per-type selects (pages / blog posts / events), persisting to `site_settings`.

### Phase 4 — Polish captured during the session

- **Widget picker visibility in template mode.** Template editors now show every widget in the picker regardless of `allowed_page_types` restrictions. Blog pager, portal widgets, etc. are available in template mode because templates are page-type-agnostic — the restriction only fires when editing an actual Page owner.
- **Blog pager token support.** Added `prev_template` and `next_template` configurable richtext fields with `{{title}}`, `{{url}}`, `{{date}}`, `{{author}}`, `{{image}}` tokens. Matches the pattern used by `blog_listing` / `events_listing`. Template-editor preview renders tokens literally so the designer can see and edit the format; public render substitutes from real prev/next post data. Fixes the "invisible widget in template editor because no prev/next data" issue.
- **Default appearance restoration.** Every production widget-creation path (`copyOwnedStack`, `CreatePost` hardcoded fallback, `EventResource` fallback, `TemplateSeeder`, `BasePageSeeder`, `PortalPageSeeder`, `DashboardDebugGeneratorWidget`) now calls `PageWidget::resolveAppearance()` so newly created widgets carry their widget type's `defaultAppearanceConfig()` (padding, colors, etc.). Previously only the Livewire `+ Widget` picker applied these — so template-seeded and hardcoded-fallback widgets came up with empty appearance.
- **Token substitution on blog posts.** `PostController::renderPage()` had its own inline widget-rendering logic that called `Blade::render($widgetType->template, ['config' => $config])` directly — bypassing `WidgetRenderer` where token substitution lives. Refactored `PostController` to use the shared `PageBlockRenderer` (same path `PageController` uses), so `{{title}}` / `{{author}}` / etc. now substitute on public blog posts. Behavior now consistent between pages and posts.
- **Layer explorer dropdown.** The block-count text at the bottom-right of the page-builder is now a clickable dropdown listing every widget and layout in page order (layouts show nested children indented). Clicking an item selects it in the editor. Intended for troubleshooting "editor says N blocks but I only see M" situations — quickly reveals collapsed or hidden widgets.
- **Template list columns tidied.** Dropped the `widget_count` column (read the now-removed `definition` JSONB — always showed 0 post-migration) and the `is_default` icon column (only meaningful for page templates; noisy on the content tab). Set-as-Default action on the row menu still works for page templates.

## Schema changes

**`page_widgets`:**
- Added `owner_type` (string, not null), `owner_id` (uuid, not null), indexed `(owner_type, owner_id)`.
- Dropped `page_id` and its FK.

**`page_layouts`:**
- Added `owner_type` (string, not null), `owner_id` (uuid, not null), indexed `(owner_type, owner_id)`.
- Dropped `page_id` and its FK.

**`templates`:**
- Dropped `definition` (jsonb). Widget stacks now live as rows in `page_widgets` / `page_layouts` with `owner_type='App\Models\Template'`.

**`site_settings`:** no column change. New keys consumed: `default_content_template_default`, `default_content_template_post`, `default_content_template_event` (all string type, empty = no default).

**Docs updated:** `docs/schema/page_widgets.md`, `docs/schema/page_layouts.md`, `docs/schema/templates.md`, `docs/schema/site_settings.md`.

## Production deployment notes

- Run `docker exec nonprofitcrm_app php artisan migrate` to apply `2026_04_15_120000_polymorphic_widget_owner.php`. The migration is **irreversible** (one-way structural) and handles backfill inline — existing page widgets keep their owner as the page; existing content-template definitions are hydrated into template-owned rows; the old columns are dropped. No manual data fixup needed.
- Run `docker exec nonprofitcrm_app php artisan db:seed --class=TemplateSeeder` to pick up the Blog Post template's newly-seeded `blog_pager` widget and wire `default_content_template_post` to it (only if not already configured — admin choices are preserved).
- Run `docker exec nonprofitcrm_app php artisan db:seed --class=WidgetTypeSeeder` to sync the blog_pager widget's new `prev_template` / `next_template` config schema fields.
- No `build:public` needed — no widget SCSS/CSS/JS touched.
- `npm run build` output committed (Vue store + api + components changed).

## Deferred decisions

- **Table rename deferred.** `page_widgets` / `page_layouts` are misnomers now that they hold template-owned rows too. Renaming was explicitly out of scope — polymorphism is the load-bearing change, the rename can land in a follow-up when the naming pain shows up in practice.
- **Notes source-reference for importer timeline.** Flagged in session 188 planning as a decision point (FK column on `notes` vs JSONB `context` field). Not relevant to this session; deferred to 188.
- **Orphaned widget rows.** Diagnosis during UAT turned up a handful of `PageWidget` rows whose `owner_id` points to a Page that no longer exists (pre-dating this session). Not cleaned up — harmless since nothing renders them. Could be swept in a later housekeeping pass.
