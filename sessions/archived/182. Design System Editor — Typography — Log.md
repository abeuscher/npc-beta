## What was built

### Phase 1 — `font` field type primitive

- `resources/js/page-builder-vue/components/primitives/FontInput.vue` — reusable Vue primitive exposing family / weight / size+unit / line-height / letter-spacing+unit / case. Every control renders a concrete value (no "— unset —" states); the family field has a "Use {parent}" button that copies a parent bucket's family into the element.

### Phase 2 — Storage + migration

- Decision shift from the original spec, resolved mid-session with the user: typography is site-wide under the new **theme** concept rather than per-template. Storage is `SiteSetting('typography')` — same row shape as `button_styles` (session 134), not a JSONB column on `templates`.
- `database/migrations/2026_04_14_100000_add_typography_and_drop_template_fonts.php` — seeds `SiteSetting('typography')` from the default template's `heading_font` / `body_font`, then drops those columns from `templates`.
- `app/Services/TypographyResolver.php` — concrete-value defaults for every field (`'Inter', system-ui, sans-serif`, sensible per-element weights/sizes/line-heights, `0em` letter-spacing, `'none'` case, `0` spacing). `load()` merges storage over defaults but skips explicit-null overrides so the UI never shows empty controls.
- `Template` model, `ContentExporter` / `ContentImporter`, `TemplateSeeder`, `EditPageTemplate` (form + `clearAppearance`), and the template edit blade all updated to drop the font columns.
- `docs/schema/templates.md` and `docs/schema/site_settings.md` refreshed; `docs/app-reference.md` shows Theme under CMS.

### Phase 3 — Typography Vue island

- Nav moved at the user's request: `Tools → Design System` became `CMS → Theme` above Templates. `DesignSystemPage` class kept; `$navigationGroup`, `$navigationLabel`, `$navigationSort`, `$title`, and breadcrumbs updated. Templates bumped from sort 7 to 8. Default active tab is now `text-styles`, with Buttons as the secondary tab.
- `resources/js/theme-editor/` — new Vue app entry (`main.ts`), Pinia store (`stores/typography.ts`), types (`types.ts`), and the main `TypographyPanel.vue`. Mounts via `[data-theme-editor-app]` inside the Design System page blade; `vite.config.js` updated to register the entry point. A `MutationObserver` in `main.ts` handles late-mounting when the Livewire tab switch reveals the mount element, so no reload is needed.
- Element rows are collapsible drawers: each row is a button header showing the label + a one-line inline preview (real `<ul>` with `list-style-position: inside` so discs/numbers render). Click to expand; inside is the FontInput, two SpacingInputs (margin + padding), and for lists a list-style-type + marker-color picker.
- Bucket cascade: changing the Heading or Body family sweeps all elements currently matching the old bucket value to the new one, so users can re-theme wholesale without losing per-element overrides.

### Phase 4 — Compile & apply at render

- `app/Services/TypographyCompiler.php` — emits concrete CSS rules for each element using `:not(nav ...)` exclusions so headers/footers inherit theme typography but the nav widget's internal `<ul>` markup does not.
- `resources/views/layouts/public.blade.php` updated: bucket families feed `--font-family-heading` / `--font-family-body` CSS vars, Google Fonts `<link>` derived from `TypographyCompiler::googleFontsUsed()`, compiled typography injected as an inline `<style>` in `<head>`.
- `resources/scss/_base.scss` deconflicted: its `.np-site h1..h6` rules (font-family/weight/line-height/size) were overriding per-element typography by specificity and only read the bucket CSS var. Removed. Typography is now the sole source of truth for heading styles on the public site.
- Curated font catalog lives on `TypographyResolver::fontCatalog()` (lifted from the previous per-template `getFontOptions()` on `EditPageTemplate`). Dropped the "System UI" entry at the user's request so every catalog entry corresponds to a font we actually load.

### Phase 5 — Export + return-to navigation

- `app/Http/Controllers/Admin/ThemeTypographyController.php` — two endpoints: `POST api/theme/typography` (JSON save, validated shape, normalised against defaults) and `GET design-system/typography/export.scss` (download of compiled rules with a header comment). Both gated by `manage_cms_settings` (matches Buttons on the same page — align with the theme-wide page permission rather than `edit_theme_scss`).
- Return-to flow: `DesignSystemPage` reads a sanitised `?return_to=` query param and renders a "← Back to editor" link at the top. `RichTextField.vue` adds a discreet "Edit site styles →" link above each Quill editor carrying the current URL, powered by `theme_editor_url` now in the page builder bootstrap data. `$activeTab` is `#[Url]`-bound so the Typography tab opens directly.

### Shared primitives

## Production deployment notes

Standard `migrate`. The migration reads the default template's current font values, writes them into `SiteSetting('typography').buckets.heading_family` / `body_family`, then drops the columns. No manual steps beyond that. If a deploy host has non-default templates with custom font values, those are not migrated — under the new theme model, those would have been template-level overrides and now collapse into the single site-wide theme. Flag to operators if any customer site had diverging fonts on non-default templates (expected to be rare in current deployments).

## Deferred decisions

- **Page builder preview + Quill editor do not yet consume typography CSS.** The injection only happens in `public.blade.php`, so in-editor previews still show admin-default fonts. Scheduled into session 183.
- **Theme editor UI polish.** The session 182 UI is an experimental scaffold. Session 183 phase 3 covers line-by-line polish based on user notes captured at session start.
- **Template Colors refactor.** Under the new theme/template split, `primary_color` is clearly theme-level while `header_bg_color` / `footer_bg_color` / `nav_*_color` are ambiguous. Deferred to a dedicated Colors session to avoid shoehorning mid-Typography.
- **Marker color nullable.** `ul_li` / `ol_li` `marker_color` stays nullable (meaning "inherit text color") rather than being forced concrete. Low-priority UX decision; can revisit if the "no unset fields" principle needs tightening for list controls.
- **Buckets can be null.** Unlike element fields, bucket family values remain nullable (`— unset —`) because buckets are presets/shortcuts, not CSS outputs. Changing a bucket cascades to matching elements; clearing a bucket does not wipe element values.
