10. **Verify-at-start questions** the session prompt names but the implementation should resolve before going deep:
    - **Audit for existing internal-scroll widgets.** Walk `resources/scss/widgets/**` and `app/Widgets/**` looking for any widget with `overflow: auto/scroll` or rendered DOM that scrolls natively (e.g. a tall fixed-height carousel with internal pan). If none today, the nested-scroll concern may be a forward-looking guard rather than a current bug — adjust scope accordingly. Surface findings before committing to defensive event-walking logic.
    - **Drag-drop interaction.** When a new widget is dragged from the library into the canvas, the canvas auto-scrolls during drag. The clamp must **not** engage during drag (else it blocks the drag-into-edge-to-scroll pattern) and must engage after drop / when `selectedItemId` settles. Inspect the drag-drop wiring in `PreviewCanvas.vue` and `App.vue` to confirm: when does `selectedItemId` actually change relative to drag lifecycle? If selection lands only at drop completion, the clamp's selection-watch hooks naturally; if mid-drag selection is possible, the clamp needs a drag-state guard.
    - **Release signal.** What un-clamps? Today: clicking the canvas background de-selects (via the existing `clearSelection` path); pressing Escape doesn't (verify). Confirm the existing release mechanism + ensure the clamp listener tears down when `selectedItemId` becomes null. The Escape-key shortcut may want lifting in-session if it's a small, obvious add — surface and decide.
    - **Keyboard scrolling.** Arrow keys / PageUp/PageDown / Space / Home / End all scroll the document by default when focus is on a non-input element. Decide whether keyboard scrolling is in scope. **Recommendation:** include — same clamp math applies via `keydown` preventDefault, low marginal cost, otherwise the lock is incomplete on a real interaction surface. Surface the recommendation; pivot only if scope inflates unexpectedly.
    - **Inertial-scroll behaviour on Mac trackpad.** The browser fires multiple wheel events with momentum after the user lifts. preventDefault on each clamps the scroll but may produce visual jitter at the boundary (the browser still tries to render the would-be scroll). Test on a real trackpad if available; if jitter is unacceptable, the fallback is debouncing or dampening the listener — but committing to a more complex mechanism only if the simpler shape ships visibly broken.
    - **Tall-widget bounds — viewport vs. canvas-pane.** Tall here means "taller than the visible canvas area." The viewport is `window.innerHeight`; the canvas pane (`paneEl`) is its own scroll container with its own `clientHeight`. The clamp math uses the canvas pane's `clientHeight`, not the window's, since the canvas pane scrolls independently of the document. Confirm the canvas pane scrolls (i.e. `paneEl.scrollTop` is a real thing, not document-level). Audit the SCSS to confirm it's `overflow-y: auto` or similar.
