# 177. Preset Thumbnails

Third session of the Stage 5 preset sub-arc. Session 175 shipped the Presets tab; session 176 added DB-backed designer drafts and PHP-literal export. This session plugs per-preset static PNG thumbnails into the same capture pipeline that ships widget thumbnails (session 174), and wires the inspector preset cards — which currently render an empty placeholder box above the label — to display those thumbnails when present. The path convention `app/Widgets/{PascalName}/thumbnails/preset-{handle}.png` was reserved in session 174 and documented in 175; this session fills it in end-to-end.

---

## Design decisions (resolved before starting)

- **Code-authored presets only.** Only presets declared in a widget's `presets()` method get a captured thumbnail committed to the repo. DB drafts (`widget_presets` table) are out of scope — they change too fast to be worth a committed artefact, and the repo is not the right home for per-install designer scratch data. Draft cards in the inspector continue to render the empty placeholder.
- **Reuse the existing pipeline.** The capture stack stays: `widgets:manifest-json` → host-side Playwright script (`scripts/generate-thumbnails.js`) → `/dev/widgets/{handle}/...` dev route → PNG written to `app/Widgets/{PascalName}/thumbnails/`. No new pipeline is introduced. The dev route gains a preset variant; the script gains a per-preset capture loop; the output filename is already defined.
- **New dev route for preset variants.** Add `GET /dev/widgets/{handle}/presets/{presetHandle}` alongside the existing `GET /dev/widgets/{handle}`. Same middleware gate (`DevRoutesMiddleware`), same 404-in-production behaviour. The controller starts from the same demo-config baseline as the plain widget route (so the preset's appearance reads against sensible content), then applies the preset using the runtime apply semantics: `preset.config` is **overlaid** onto the demo `config`, and `preset.appearance_config` **replaces** the demo `appearance_config` wholesale. This mirrors what the inspector does at apply-time, so the captured image is faithful to what a designer will see when they click the card.
- **Viewport and output format stay the same.** 800 × 500, PNG, `thumbnails/preset-{handle}.png`. No retina variant, no WebP, no gif/video — parity with `static.png`. Animated thumbnails remain deferred (see session-outlines Stage 4.5 Phase 2).
- **Serve thumbnails via a public controller, not a symlink.** Add a lightweight `GET /widget-thumbnails/{handle}/{file}.png` controller that streams the file from `app/Widgets/{PascalName}/thumbnails/` with a validated filename. No auth — these are static visual assets, not sensitive. Filename validation must reject path traversal (`..`, slashes, non-PNG extensions). This is the same decision that would eventually serve `static.png` to the Stage 6 widget browser, so scope it general-purpose, not preset-specific. Filenames the controller accepts: `static.png` and `preset-{slug}.png`.
- **Bootstrap payload resolves thumbnail URLs server-side.** Each code-authored preset in the `widget_types[*].presets[]` payload gains a `thumbnail` key — either the public URL string (if the file exists on disk) or `null`. The frontend does not probe for files; the server is authoritative. DB drafts in `widget_types[*].draft_presets[]` do **not** get this key added this session.
- **Card rendering is a no-op fallback.** The inspector preset card template reads `preset.thumbnail`. If present, render an `<img>` in the existing thumbnail slot; if null/missing, render the current empty box. Blank card and draft cards keep the empty box. No new card variant.
- **No manifest change.** The widget manifest emitted by `widgets:manifest-json` does not grow a `thumbnail` field per preset — thumbnail resolution is a bootstrap-time concern (serves the inspector), not a manifest concern (the manifest is for the registry / browser, which consumes file paths directly and computes URLs itself in Stage 6). Keeping the manifest pure preserves its role as a code-derived artefact.
- **Capture-on-CI is out of scope.** Generation remains a local host-side command run by a developer. No GitHub Action, no pre-commit hook. Existing `static.png` files are committed by hand today; preset PNGs follow the same discipline.

## Phase 1 — Dev route for preset variants

1. Add a method `showPreset(string $handle, string $presetHandle)` to `App\Http\Controllers\Dev\WidgetDemoController`. Resolve the widget definition via `WidgetRegistry`, 404 on miss. Look up the preset on the definition by matching `preset['handle'] === $presetHandle` in `$def->presets()`; 404 on miss.
2. Reuse the existing `show()` logic to build the baseline `PageWidget` instance (demo seeder run, `defaults() + demoConfig()` for config, seeded collection handle for collection-backed widgets, `demoAppearanceConfig()` for appearance). Then apply the preset:
   - `$config = array_merge($config, $preset['config'] ?? []);` (overlay)
   - `$appearanceConfig = $preset['appearance_config'] ?? [];` (replace)
3. Reuse `resources/views/dev/widget-demo.blade.php` — no new template. The page must render at 800 × 500 with a white background, same as the plain widget demo route.
4. Register the route in `routes/dev.php` inside the existing `DevRoutesMiddleware` group. Name it `dev.widgets.preset`.
5. Factor out shared rendering into a private helper on the controller (e.g. `renderWidget(PageWidget $pw)`) so `show()` and `showPreset()` don't duplicate the renderer/composer/view calls.

## Phase 2 — Thumbnail capture script extension

1. Extend `scripts/generate-thumbnails.js`:
   - Keep the existing widget-capture loop (`static.png`) unchanged.
   - After each widget is captured, iterate `manifest[handle].presets ?? []` and hit `${baseUrl}/dev/widgets/${handle}/presets/${preset.handle}`. Save to `app/Widgets/{PascalFolder}/thumbnails/preset-${preset.handle}.png`.
   - Add a `--preset=<handle>` filter (scoped by the existing `--widget=<handle>` flag when both are passed) so a developer can regenerate a single preset without re-capturing everything.
   - Failure of a single preset must not abort the script — log `FAIL` and continue, same as the current per-widget failure handling. Exit 1 if any failure occurred.
2. The manifest already carries `presets` for each widget (session 173 plumbing). Do not modify `widgets:manifest-json` or `WidgetRegistry::manifests()`.
3. No code changes to `DemoSeeder` invocations — the dev controller already handles seeding once per request.

## Phase 3 — Thumbnail serving controller

1. Create `App\Http\Controllers\WidgetThumbnailController` with a single method `show(string $handle, string $file)`. Behaviour:
   - Validate `$file` against a strict regex: `^(static|preset-[a-z0-9-]+)\.png$`. Reject anything else with 404.
   - Map `$handle` to its PascalFolder using the same normalisation as the Node script (`handle.split('_').map(capitalize).join('')`), guarded server-side by deriving the folder from `WidgetRegistry::find($handle)` → `class_basename()` → strip `Definition` suffix. 404 if the widget is unknown.
   - Stream the file from `app/Widgets/{Folder}/thumbnails/{file}` with `Content-Type: image/png`, a 1-hour `Cache-Control: public, max-age=3600` header, and a 404 if the file doesn't exist.
2. Register the route in `routes/web.php` (not `admin-api.php` — this is public): `GET /widget-thumbnails/{handle}/{file}`, name `widget-thumbnails.show`. Public, no auth.
3. Feature tests:
   - Returns PNG bytes with the right content-type for an existing file.
   - Rejects filenames containing `..`, slashes, or a non-PNG extension with 404.
   - Rejects unknown widget handles with 404.
   - Returns 404 when the widget exists but the requested file is not on disk.

## Security checklist

Before closing:

- [ ] Dev preset route is gated by `DevRoutesMiddleware` and returns 404 in production — same behaviour as the existing widget demo route.
- [ ] `WidgetThumbnailController` validates the filename against a strict regex and rejects traversal attempts (`..`, slashes, non-PNG extensions) with 404, not 200.
- [ ] The PascalFolder is derived from `WidgetRegistry::find($handle)` rather than re-implementing the handle-to-folder transform in the controller, so an unknown widget handle cannot synthesize a filesystem path.
- [ ] Thumbnail URLs returned in the bootstrap payload route through the named controller — not raw `file://` paths or direct `public_path()` URLs.
- [ ] Cache header on the controller is `public, max-age=3600` — no aggressive caching that would prevent a designer from seeing an updated thumbnail after regeneration.

---

## Out of scope

- **DB draft thumbnails.** No capture, no resolution, no rendering — draft cards keep the empty placeholder this session.
- **Animated / video thumbnails.** Deferred per session-outlines Stage 4.5 Phase 2.
- **Widget `static.png` plumbing into the inspector or admin.** The thumbnail serving controller will accept `static.png` requests (it's cheap to generalise), but no inspector or widget-picker UI consumes `static.png` yet — that is Stage 6 work.
- **Preset thumbnails for any widget other than Hero.** Other widgets get preset thumbnails when their preset libraries land in Stage 5d+.
- **Capture in CI.** Generation stays a local developer command.
- **Retina / 2x variants.** Parity with `static.png`.
- **Thumbnail regeneration UI.** No admin button that reruns the script; the script remains host-side.
- **Cache busting via hashed filenames.** The 1-hour `Cache-Control: max-age` is the whole caching story; thumbnails rarely change between sessions.
