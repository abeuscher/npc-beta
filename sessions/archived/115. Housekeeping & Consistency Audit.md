# 115. Housekeeping & Consistency Audit

Infrastructure and consistency pass across three independent workstreams: schema documentation restructure, project file inventory, and admin field input standardisation.

---

## Design decisions (resolved before starting)

- The three workstreams are independent and can be completed in any order.
- Schema split is a documentation-only change — no migrations, no code changes.
- The field input audit produces fixes, not a separate document. Where an inconsistency is found, fix it in place. If a fix would be disruptive (e.g. changing a field type that affects stored data), flag it for discussion instead of changing it.

---

## Phase 1 — Schema doc restructure

Split `docs/schema.md` into a `docs/schema/` directory with one Markdown file per table, named after the table (e.g. `docs/schema/contacts.md`, `docs/schema/events.md`).

- Each file contains the existing content for that table — column list, types, constraints, notes — copied verbatim from the current `docs/schema.md`.
- Add a `docs/schema/README.md` that lists all table files alphabetically as a clickable index.
- Delete the original `docs/schema.md` after the split is complete.
- Update any references to `docs/schema.md` elsewhere in the project (session templates, CLAUDE.md, etc.) to point to `docs/schema/` instead.

---

## Phase 2 — File inventory & orphan audit

Walk the project tree and produce a written inventory at `sessions/115. File Inventory.md`. The inventory is a working audit document — it exists so the user can review it for anything suspicious that might not be obvious from code analysis alone. It can be archived or deleted after the session.

For each file or directory, write a one-line note on what it is and whether it appears to be in active use. Most files are identifiable by filename alone — only open a file when its purpose is genuinely ambiguous. Skip `vendor/`, `node_modules/`, and other dependency directories entirely.

After producing the inventory, review it for orphaned files — dead code, unused views, leftover artifacts, stale config. For each candidate:
- Verify it is truly unused (grep for references, check route registrations, check Blade includes, etc.).
- Delete confirmed orphans. Flag anything uncertain for discussion rather than deleting.

---

## Phase 3 — Admin field input audit

Audit every field input type used across the Filament admin panel. The goal is consistency: the same logical control should use the same implementation everywhere it appears.

Scope:
- **Image/file uploads** — identify every place an image or file is uploaded in the admin. Confirm all use the same pattern (Spatie MediaLibrary via Filament's SpatieMediaLibraryFileUpload, or whichever pattern is established). Flag any that use a different mechanism.
- **Rich text editors** — confirm all rich text fields use QuillEditor consistently. Flag any that use Filament's default RichEditor, Textarea with HTML, or other alternatives.
- **Date/time pickers** — confirm consistent usage of DatePicker, DateTimePicker, and the custom time picker pattern (hour/minute/ampm selects used in EventResource). Flag any date/time inputs that use a different approach.
- **Colour pickers** — if any exist, confirm they use the same component.
- **Select inputs** — no need to audit every select, but flag any case where a searchable select should be used but isn't (e.g. contact pickers on large datasets).
- **Toggle vs Checkbox** — confirm consistent usage. Filament convention is Toggle for boolean settings, Checkbox for opt-in lists. Flag deviations.

For each inconsistency found:
1. If the fix is safe (swapping a component type, no data migration needed), fix it in place.
2. If the fix could affect stored data or behaviour, describe the issue and stop — do not change it without discussion.
