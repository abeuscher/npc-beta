## What was built

### Iteration /1 — Stripe Checkout branding across all five checkout call sites

**Audit finding at session start.** The shared helper question the prompt anticipated was already resolved: every Stripe Checkout call site (DonationCheckoutController, ProductCheckoutController, MembershipCheckoutController, public EventController::register, portal Portal\EventRegistrationController) routes through `App\Services\StripeCheckoutService::createSession`. Five call sites, not four — events split into public + portal entry points at session 279 — but a single shared service. Extended that service rather than introducing a new builder. Decision logged via the work-log adaptation rule; not surfaced as a question because the answer was obvious from surrounding code.

**API constraints surfaced and discussed with user before going deep.** Stripe's `custom_text` slots accept plain text + limited Markdown (`**bold**`, `*italic*`, `[link](url)`); no HTML; max 1200 chars per slot. Statement descriptors are payment-mode only — subscriptions inherit from the Stripe Account default, a Dashboard-only setting. ToS acceptance only renders when a Dashboard ToS URL is configured. User confirmed scope: settings live on `CmsSettingsPage` (not Finance), per-flow `submit_type` is correct ('donate' / 'pay' / inherited for subscriptions), per-record image first then per-flow defaults (four image fields total).

**StripeCheckoutService extension.**

- Split `createSession` into `createSession` (calls Stripe) + `buildParams` (pure, returns the params array; the testable surface).
- New `submitType: ?string` parameter, payment-mode only.
- `buildParams` applies `custom_text.{submit, after_submit, terms_of_service_acceptance}` from SiteSetting; `payment_intent_data.statement_descriptor[_suffix]` (payment mode only); `consent_collection.terms_of_service: 'required'` (gated on operator confirmation that ToS URL is set in Dashboard); and `submit_type` (payment mode only). Subscription-mode flows have these fields suppressed where Stripe rejects them.
- `defaultImageUrl(string $flow)` static helper resolves per-flow default image URLs from SiteSetting via `Storage::disk('public')->url($path)`. Flows: `donation`, `event`, `product`, `membership`.

**Per-flow wiring across all five call sites.**

- DonationCheckoutController: `submitType: 'donate'` for one-off; subscription path omits. Default donation image attached to line items.
- ProductCheckoutController: `submitType: 'pay'`. Per-product `product_image` first, fallback to default product image.
- MembershipCheckoutController: `submitType: 'pay'` for one-off; subscription path omits. Default membership image attached.
- EventController (public) + Portal\EventRegistrationController (portal): `submitType: 'pay'`. Per-event `event_thumbnail` first, fallback to default event image. `EventRegistrationQuantities::stripeLineItems` extended with optional `$imageUrl` parameter applied uniformly to every tier line.
- All five switched from `new StripeCheckoutService()` to `app(StripeCheckoutService::class)` so tests can spy via container instance binding.

New "Stripe Checkout — Branding" section, ten new SiteSetting keys behind it: `stripe_dashboard_branding_confirmed` (toggle for Onboarding Checklist completion), `stripe_checkout_submit_text`, `stripe_checkout_after_submit_text`, `stripe_tos_url_configured` (toggle gating ToS-acceptance copy), `stripe_checkout_terms_acceptance_text`, `stripe_statement_descriptor` (with regex validation: 5–22 chars, alphanumeric + spaces only), `stripe_statement_descriptor_suffix`, and four image-upload paths (`stripe_default_donation_image`, `stripe_default_event_image`, `stripe_default_product_image`, `stripe_default_membership_image`). ## Security considerations

API-half and Operator-UI changes were both reviewed. The new SiteSetting keys are operator-configurable plain-text strings (copy + descriptors) plus image uploads going to the public disk — no secrets, no encrypted-type fields. CmsSettingsPage gating remains on `manage_cms_settings` which was already established. `stripe_tos_url_configured` deliberately gates the `consent_collection.terms_of_service: 'required'` field; flipping it on without configuring the URL in Stripe Dashboard would cause every Checkout session to fail — that risk is surfaced in the toggle's helper text and the help doc.

The Markdown shape Stripe accepts in `custom_text` (limited Markdown only — no HTML) is enforced by Stripe server-side, not by our client, so an operator who pastes raw HTML will see Stripe strip it; not a security concern, just a UX one (covered by the helper-text format note).

PMW track lift is documentation only. No code, no permissions surface, no schema. None.
