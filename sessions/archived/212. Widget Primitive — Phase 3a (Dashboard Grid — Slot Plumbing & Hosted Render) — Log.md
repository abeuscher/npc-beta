- `app/WidgetPrimitive/Slots/DashboardGridSlot.php` — replaced the Phase 2 `RuntimeException('Slot ambient context not yet wired — lands with Phase 3')` with `public function ambientContext(): SlotContext` returning `new SlotContext(app(PageContext::class), null)`. ### Phase 2 — `WidgetRenderer::render()` slot-handle parameter

- `app/Services/WidgetRenderer.php` — added trailing `string $slotHandle = 'page_builder_canvas'` parameter to `render()`. Branches:
  - **Canvas path** (`$slotHandle === 'page_builder_canvas'`, the default): unchanged — computes `$tokenPage` from `$pageContext->currentPage` + owner page, calls `ambientContext($pageContext, $tokenPage)`, runs the full token-substitution loop over richtext/text config fields.
  - **Dashboard path** (`$slotHandle === 'dashboard_grid'`): skips `$tokenPage` computation, calls the slot's zero-argument `ambientContext()`, and **skips the token-substitution loop entirely** — richtext fields render their raw template. Consistent with "no current page" and the dashboard slot's layout-constraints declaration. `PageContextProjector::project()` still populates `$widgetData` with empty-string tokens when a `SOURCE_PAGE_CONTEXT` contract runs on the dashboard, but nothing consumes it.
- Byte-equivalence for existing callers verified: `WidgetPrimitiveIntegrationTest`, `PageBuilderApiTest`, `WidgetRegistryTest`, `WidgetConfigResolverTest`, `WidgetDataResolverTest`, and `PageContextTokensTest` all stayed green through the refactor (146 assertions across the widget-adjacent surface).

### Asset-loading fix — the interim and the paradigm-consistent version

Manual browser testing surfaced a Livewire morph / Alpine race: per-widget lib bundles (`/build/libs/swiper.js` et al.) were not loading when the carousel's `x-data` `init()` fired, causing `Cannot read properties of undefined (reading 'Navigation')`. Two-stage fix:

- **Interim fix** (kept the bug's full shape visible before generalising): added a `panels::head.end` render hook in `AdminPanelProvider` that emits `<link>`+`<script>` tags for every manifest `libs` entry in the admin `<head>`. Synchronous load before any Livewire widget mount, Alpine finds `window.Swiper` / `window.SwiperModules` / `window.Chart` etc. defined on first init. Confirmed working in the browser.
- **Paradigm-consistent fix**: extracted `App\Services\WidgetAssetResolver` as a container-bound singleton. Surface:
  - `manifest(): array` — memoized read from `public/build/widgets/manifest.json`, empty-array fallback.
  - `widgetCss(): ?string` / `widgetJs(): ?string` — prefixed URLs or null.
  - `allLibs(): array` — every manifest `libs` entry keyed by handle.
  - `libs(array $handles): array` — resolve specified handles, silently drop unknowns (fail-closed at the asset boundary, mirroring the read-side contract posture).
  - `libsForWidgets(iterable $items): array` — walk `WidgetType`/`PageWidget` instances, collect `assets['libs']` handles, dedupe, resolve. API support for per-slot scoping when it lands.
  - Optional `$manifestPath` constructor argument so tests can target a temp file without touching the real manifest.
- `AdminPanelProvider`'s four previously-duplicated manifest reads (the `__widgetLibs` JSON blob, the widget CSS bundle, the widget JS bundle, and the new lib-bundle hook) now all call the resolver. Collapsed four inline `json_decode(@file_get_contents(...))` reads into resolver method calls.
- `WidgetServiceProvider::register()` gained `$this->app->singleton(WidgetAssetResolver::class, fn () => new WidgetAssetResolver());` alongside the other widget singletons.
- New unit test `tests/Unit/Services/WidgetAssetResolverTest.php` — 8 tests, 18 assertions, covering missing-manifest fallback, memoization, URL prefixing, null handling, `allLibs`, fail-closed `libs`, `libsForWidgets` dedupe, and the `PageWidget`-via-relation overload.

The admin still eagerly loads all manifest libs (breadth, not a paradigm violation). Narrower per-page scoping can flip the call site from `allLibs()` to `libsForWidgets($pageWidgets)` when a page-level required-libs registry lands — the resolver's API already supports that.
