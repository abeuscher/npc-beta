# 267. Full-Width Architecture Enforcement — Background and Content Split

Make widget compliance with the page-layout's full-width contract structural (E10), and split the single `full_width` toggle into two — `background_full_width` and `content_full_width` — applied to both the widget Appearance layer (`page_widgets.appearance_config.layout`) and the column-layout primitive (`page_layouts.layout_config`). The split enables the common design pattern of "full-width background band with content constrained to the container width."

Working-set placement: lifts **E10** at position 22. The original stub framed E10 as enforcement-only (three candidate fix shapes: layout-aware wrapper / contract-level composer guard / per-template CSS audit). The two-knob split was lifted into this session at 266 close because the schema/render surfaces overlap — splitting `full_width` would otherwise immediately revisit the same rendering pipeline E10 hardens.

---

## Background — why folded

The single `full_width` toggle covers two structurally distinct effects in the same boolean: whether the **background** extends edge-to-edge and whether the **content** does. For most widgets this is fine — both want the same answer — but the common design pattern of "colored band across the page with text centered in the middle" requires them to diverge. Today operators get there by nesting a non-full-width widget inside a full-width column, which is unnecessarily complicated for a use case that should be one toggle.

Splitting the knob is a schema + render-pipeline change that touches the same places E10's structural-enforcement work does (`AppearanceStyleComposer`, `PageBlockRenderer`, the page-widgets blade, the two inspector panels). Doing them separately would be wasted work — the second pass would re-touch every site the first pass touches.

---

## Design decisions (resolved before starting)

- **Two booleans, both always visible, conditional grayed-out.** Both `background_full_width` and `content_full_width` toggles render in the inspector. When `content_full_width` is on, the `background_full_width` toggle is grayed out with a tooltip explaining the behavior — background follows content automatically when content is full-width. The grayed state is a UI-layer constraint; the underlying boolean retains whatever value it had. The 4th state (content bleeds, background constrained) is not representable in the UI.
- **Two-boolean naming.** `background_full_width` / `content_full_width`. Storage matches per-table convention:
  - widgets: `page_widgets.appearance_config.layout.{background_full_width, content_full_width}`
  - columns: `page_layouts.layout_config.{background_full_width, content_full_width}`
  - per-type defaults: `widget_types.{background_full_width, content_full_width}` (or JSON-keyed equivalent — TBD verify-at-start).
- **Migration preserves current behavior.** Existing rows with `full_width: true` get rewritten to `{background_full_width: true, content_full_width: true}`. Existing rows with `full_width: false` get `{background_full_width: false, content_full_width: false}`. Absent/unset rows get both `false`. No surprise visual changes post-migration; operators see exactly what they saw pre-deploy.
- **Render pipeline converges to one source of truth.** The current three full-width read paths in [`AppearanceStyleComposer.php`](../app/Services/AppearanceStyleComposer.php) (lines 75-76) and [`PageBlockRenderer.php`](../app/Services/PageBlockRenderer.php) (lines 28-29, 38, 118) collapse to a single resolution helper that returns `{background_full_width, content_full_width}` — per-instance overrides per-type default, false-false fallback. This is the structural-enforcement half of the original E10 ask.
- **Bypass-audit-driven enforcement shape.** The original E10 stub named three candidate fix shapes; the right one depends on what the audit finds. Surface the audit summary to the user before committing — different bypass classes (CSS escapes vs. wrapper escapes vs. ignoring the appearance layer entirely) call for different fixes.
- **No new full-bleed primitives.** "Hero Full-Bleed Promotion" stays in the post-Beta-1 stub. This session works with the existing widget set.

## Phase 1 — Schema migration

**Goal:** Both new keys land on every relevant row, with concrete values preserving current behavior. Widget types' per-type defaults split in parallel.

**Steps:**

1. New migration `database/migrations/YYYY_MM_DD_HHMMSS_split_full_width_into_background_and_content.php`.
2. **Widgets path** (`page_widgets.appearance_config`): walk every row; for the `appearance_config.layout.full_width` JSON key (boolean, may be absent), rewrite to `{background_full_width: <existing>, content_full_width: <existing>}`. Absent → both `false`.
3. **Columns path** (`page_layouts.layout_config`): walk every row; for `layout_config.full_width`, rewrite to the same two-key shape.
4. **Per-type defaults path** (`widget_types`): per verify-at-start finding, either column-replace migration (drop `full_width` column, add two new columns) or JSON-key rewrite. Preserve existing per-type values to both new keys exactly.
5. **Down migration:** reverses by collapsing back. If both keys are equal (the migration's output for all existing rows), restore as `full_width: <equal-value>`. If they diverge (any rows authored post-migration with `(background: true, content: false)` — the new pattern), the down path collapses to `full_width: content_full_width` since content-width is the load-bearing user-facing knob. Document the lossy down-path in the migration comments.
6. Update `docs/schema/page_widgets.md`, `docs/schema/page_layouts.md`, and `docs/schema/widget_types.md` to reflect the new key shape.
7. Run migration: `docker compose exec app php artisan migrate`.

## Phase 2 — Render pipeline (structural enforcement + two-knob emission)

**Goal:** Convergence on one resolution path; outer/inner wrapper emission supporting all three meaningful states; bypass-audit fixes.

**Steps:**

1. **Read-path convergence.** The current three read sites in `AppearanceStyleComposer.php:75-76`, `PageBlockRenderer.php:28-29`, and `PageBlockRenderer.php:118` collapse to a single helper returning `{background_full_width, content_full_width}` from the canonical resolution chain (per-instance → per-type → false-false). The composer's `compose()` return shape changes from `{inline_style, is_full_width}` to `{inline_style, background_full_width, content_full_width}`; the renderer threads both through.
2. **Wrapper emission.** [`page-widgets.blade.php`](../resources/views/components/page-widgets.blade.php) gains an inner content wrapper. The behavior matrix:
   - `(bg: false, content: false)` — outer wrapper boxed (container max-width); inner unchanged.
   - `(bg: true, content: true)` — outer wrapper extends edge-to-edge; inner inherits.
   - `(bg: true, content: false)` — outer wrapper extends edge-to-edge with the background style; inner wrapper applies the container max-width (the new design pattern).
   - `(bg: false, content: true)` — UI-layer prevents this state; if encountered (legacy/seed/test), treat as `(bg: true, content: true)` per the gray-out semantics.
3. **Bypass audit + fix.** Per the verify-at-start audit, fix the widgets currently bypassing the layout's instruction. Choose the load-bearing fix shape from the original E10 candidates (layout-aware wrapper / composer contract guard / per-template cleanup) based on what the audit finds; surface the audit + chosen shape to the user before going deep.
4. **CSS / SCSS update.** Inner content-wrapper rule inherits the existing public-site `.container` max-width token. Outer background-wrapper applies the background color/gradient/image edge-to-edge. Verify against the existing container token rather than introducing a new one.

## Phase 3 — Admin UI

**Goal:** Both inspector panels gain the second toggle with the conditional gray-out behavior. Uniform UX across both surfaces.

**Steps:**

1. **Widget Appearance panel** ([`SectionLayoutPanel.vue`](../resources/js/page-builder-vue/components/appearance/SectionLayoutPanel.vue)):
   - Existing toggle reads/writes `content_full_width` (renamed from `full_width`).
   - New toggle below: `background_full_width`. Reuse the existing `fullWidthDisabled` + `fullWidthDisabledReason` props mechanism — when `content_full_width` is true, `background_full_width` toggle is `disabled` with tooltip "Background follows content when content is full-width."
   - Default state (both off) renders as "boxed."
2. **Column-layout inspector** ([`LayoutInspectorPanel.vue`](../resources/js/page-builder-vue/components/LayoutInspectorPanel.vue)):
   - Same parallel UI: existing toggle becomes `content_full_width`, new toggle next to it for `background_full_width`, same conditional gray-out + tooltip wording. Match the widget Appearance panel's UX exactly.
3. **TypeScript types** ([`types.ts`](../resources/js/page-builder-vue/types.ts)):
   - Update widget appearance type at line 18 — replace `full_width?: boolean` with `background_full_width?: boolean; content_full_width?: boolean`.
   - Update layout type at line 107 — same replacement on `full_width: boolean`.
4. **Labels** (inline-direct per "no collapse to summary panels" feedback):
   - Content toggle: "Content fills page width"
   - Background toggle: "Background fills page width"
   - Background tooltip when grayed: "Background fills page width automatically when content is set to fill page width."

## Phase 4 — Tests

**New test files / additions:**

1. **Migration test** at `tests/Feature/SplitFullWidthMigrationTest.php` — assert (a) post-migration JSON shape for both `page_widgets` and `page_layouts`; (b) value preservation (rows with `full_width: true` now have both new keys true; rows with false now both false; absent rows now both false); (c) per-type defaults split correctly; (d) down migration round-trips matched-pair rows correctly and the lossy-collapse rule for diverged rows.
2. **`AppearanceStyleComposer` test additions** — three new tests covering (a) instance override → per-type default → false-false fallback resolution chain; (b) two-knob output structure on the composer's return value; (c) the converged read-path returns consistent results regardless of which entry point invokes it.
3. **`PageBlockRenderer` test additions** — render-side wrapper assertions: (a) `(false, false)` → both wrappers boxed; (b) `(true, false)` → outer extends, inner boxed (the new pattern); (c) `(true, true)` → both extend; (d) per-type default fallback when no instance override is set.
4. **Bypass-audit regression tests** — one test per widget that needed an in-session-absorbed fix, asserting the widget honors the layout instruction structurally post-fix (no per-template CSS escape).
5. **Vue component tests** for `SectionLayoutPanel.vue` and `LayoutInspectorPanel.vue` — the conditional gray-out behavior on the background toggle when content is full-width, and the tooltip text on the disabled state.
6. **Playwright spec** at `tests/e2e/page-builder-full-width-split.spec.ts` — render a page with a single widget (or column-layout band) in three configurations (boxed / bg-bleed / full-bleed) and assert the rendered DOM has the expected wrapper structure + that the inner content-width matches the `.container` token in the bg-bleed case.
