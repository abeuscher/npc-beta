# A002. Public-Bundle Consolidation — One CSS, One JS

> **Agentic session.** Removal-driven cleanup pass on the public-render asset pipeline. Two parallel inline-emission paths shadow the bundled `<link>` + `<script>` the layout already emits via `public/build/widgets/manifest.json`; one dead-code collection rides along to no consumer. Retire all three. Pest regression tests on the rendered `<head>` shape are the binary acceptance signal.

One sentence: the public layout should ship exactly **one bundled `<link rel="stylesheet">`, one bundled `<script>`**, and a small set of intentional inline blocks (font-family `:root` vars, optional `<style>` for template custom SCSS pending the Phase 3 call, raw operator head/body snippets) — nothing else.

---

## Why this is agent-friendly

- The bundle pipeline already exists and works (`AssetBuildService::build()` writes `public/build/widgets/manifest.json`; the layout reads + emits the hashed asset URLs at `resources/views/layouts/public.blade.php:163,248`). The session is **deletion of redundant paths**, not addition of new shape.
- Three independent fixes, sequenceable in any order. No mid-session UX decisions.
- The hard-no list keeps widget code, SCSS, and unrelated rendering-pipeline services out of scope. The fixes live in three specific service files + two controllers + the public layout.
- Acceptance is binary: regression Pest test asserts the rendered `<head>` of a public page contains the expected ≤ N inline blocks and no per-widget-instance CSS/JS leaks. Either the test passes or it doesn't.

---

## Diagnosis (load-bearing — read before any code)

The public-render pipeline has three redundant paths that ship widget CSS/JS:

**Path 1 — `AssetBuildService::build()` (the intended path).** Walks every `WidgetType.assets[]` declared file (under `app/Widgets/<Handle>/*.scss/.css/.js`) and every `widget_types.css` / `widget_types.js` DB column, sends them to the remote build server, writes a hashed bundle to `public/build/widgets/<hash>.{css,js}` and a manifest at `public/build/widgets/manifest.json`. The public layout reads the manifest and emits one `<link>` + one `<script>`. **This works correctly.**

**Path 2 — per-instance inline emission via `WidgetRenderer::render()`.** Lines 161-174 (server render-mode) and 172-174 (client render-mode) concat `widgetType->css` and `widgetType->js` into `$styles` and `$scripts` accumulators returned from `render()`. `PageBlockRenderer::renderLayoutBlock()` and `ChromeRenderer::renderLayoutBlock()` accumulate those into `$inlineStyles` / `$inlineScripts` reference parameters passed through from `PageController` / `PostController`. The public layout dumps them inline at `resources/views/layouts/public.blade.php:184-190` (`<style>$__allInlineStyles</style>`) and `:252-258` (`<script>$__allInlineScripts</script>`).

  - **Currently a no-op for seeded widgets** (the `WidgetTypeSeeder` doesn't populate `widget_types.css`/`js`; widgets declare assets via files in `app/Widgets/<Handle>/`).
  - **NOT a no-op when operators use `WidgetTypeResource`** — the Filament admin (`app/Filament/Resources/WidgetTypeResource.php:187,212`) exposes `css` and `js` textareas. Any operator-customised widget gets double-emitted: once in the bundle (correct), once inline per widget instance on every page render (the bug). A page with five `text_block` instances would emit the custom CSS five times inline plus once in the bundle.
  - Latent footgun; ships content out of the bundle the moment anyone touches those textareas.

**Path 3 — dead `$widgetAssets` collection.** `WidgetRenderer::collectAssets()` (lines 220-238) populates a `['css' => [], 'js' => [], 'scss' => []]` deduplicated path list. Called from `PageBlockRenderer::renderLayoutBlock()` line 97, `ChromeRenderer::renderLayoutBlock()` line 201, `PageController` line 104, `PostController` (equivalent). Passed to the view via `compact('blocks', 'inlineStyles', 'inlineScripts', 'widgetAssets')` in both controllers. **No view consumes `$widgetAssets`.** Dead plumbing — ~20 lines across the four files.

Plus one legitimate inline emitter that stays (out of scope for this session):

- **Per-widget-instance `appearance_config` → `style="..."` attribute** on wrapper divs (set by the admin's spacing/colour editors via `ChromeRenderer::buildInlineStyle()`). Instance-specific by design — cannot be bundled (different per widget instance). Bloats HTML but is correct. Leave it.

## Scope, per phase

### Phase 1 — Retire dead `$widgetAssets` collection

**File changes:**

- `app/Services/WidgetRenderer.php` — delete `collectAssets()` method (lines ~220-238 inclusive).
- `app/Services/PageBlockRenderer.php` — remove `array &$widgetAssets` from `renderLayoutBlock()` signature; remove the `WidgetRenderer::collectAssets($pw->widgetType, $widgetAssets)` call.
- `app/Services/Media/ChromeRenderer.php` — mirror change on its `renderLayoutBlock()` signature + drop the `collectAssets` call.
- `app/Http/Controllers/PageController.php` — remove the `$widgetAssets = ['css' => [], 'js' => [], 'scss' => []]` local + the per-block `collectAssets` call + the `&$widgetAssets` arg passed to `renderLayoutBlock()` + the `'widgetAssets'` key from `compact()` in `view(...)`.
- `app/Http/Controllers/PostController.php` — mirror change.

**Test:** a one-shot Pest that grep-asserts the four signature changes (no `&$widgetAssets` parameter remains, no `collectAssets` method or call remains). Cheap regression guard.

### Phase 2 — Retire per-instance inline widget CSS/JS emission

**File changes:**

- `app/Services/WidgetRenderer.php` — in `render()`, remove the four `if ($widgetType->css) { $styles .= "\n" . $widgetType->css; }` / `if ($widgetType->js) { $scripts .= "\n" . $widgetType->js; }` blocks (both render-mode branches). The function still returns `['html' => ..., 'styles' => $styles, 'scripts' => $scripts]` but `$styles` / `$scripts` will only ever carry content explicitly produced by the template render (which is empty in current widget code — verify by grep).
- `app/Services/PageBlockRenderer.php` — keep `string &$inlineStyles, string &$inlineScripts` reference params for now (still wired up in callers; `WidgetRenderer::render` just no longer feeds them widget-type-level content). If those accumulators end up always-empty after Phase 2, remove them in a follow-up; do not expand scope here.
- `app/Services/Media/ChromeRenderer.php` — mirror change.
- `resources/views/layouts/public.blade.php` — remove the `<style>$__allInlineStyles</style>` block (lines ~188-190) and the `<script>$__allInlineScripts</script>` block (lines ~256-258). The `@php` blocks computing `$__allInlineStyles` / `$__allInlineScripts` go too.

**Test — the close-gate regression:**

`tests/Feature/PublicLayoutBundleConsolidationTest.php` — new file. Seed a Page with two widgets where the widget types have non-empty `widget_types.css` / `widget_types.js` rows in the DB (mimicking an operator-customised widget). GET the public page URL. Assert on the response HTML:

- Exactly one `<link rel="stylesheet" href="/build/widgets/...">` for the bundled widget CSS.
- Exactly one `<script src="/build/widgets/...">` for the bundled widget JS.
- The widget-type CSS string does NOT appear inline in any `<style>` block.
- The widget-type JS string does NOT appear inline in any `<script>` block (other than the bundled `<script src=...>`).

Two assertions: bundle present, inline absent. Belt + suspenders.

### Phase 3 — Template `custom_scss` to bundle (open question — judgment call at session start)

**The call to make:** template `custom_scss` is per-template, scoped naturally to pages that use that template. Today it compiles inline per request at `public.blade.php:178-180` (1-hour cache). To move it into the bundle, it would need a scope so it only applies to pages on that template — a body class like `page-using-template-{handle}` is the natural shape.

**Two options:**

- **Option A — bundle the SCSS.** `AssetBuildService::build()` walks `Template::whereNotNull('custom_scss')->get()`, wraps each template's SCSS in `body.page-using-template-{handle} { ... }`, concatenates into the public bundle. Public layout drops the inline `<style>` block. Page render adds the body class. Bigger change, more moving parts.
- **Option B — leave inline.** Keep the inline `<style>` block. Note the "moves to build server post-beta" comment in code is still accurate; defer for a numeric session.

**Recommendation:** the agent should default to **Option B** unless Phase 1 + Phase 2 land in under three iterations and there's headroom — Option A introduces a body-class contract that ripples through the layout and possibly the page-builder canvas. Scope expands quickly. The bundle consolidation goal is *mostly* met after Phases 1 + 2; the template-SCSS inline block is a single bounded artifact, not a per-instance leak.

If Option A is chosen, the new files / changes:

- `app/Services/AssetBuildService.php` — extend the `theme/public.scss` concat block (around line 470-475 in the current file) to also walk templates and emit a `// ── template custom SCSS (from DB) ──` section. Each template's SCSS wrapped with the body-class scope.
- `resources/views/layouts/public.blade.php` — remove the `@php $__templateScss = ...; $__compiledCss = ...; @endphp` + `<style>$__compiledCss</style>` block. Add the body-class to the `<body>` tag (`class="np-site page-using-template-{{ $__tpl?->handle }} ..."`).
- One new test asserting a template's SCSS appears in the built bundle scoped under its body class.

If Option B is chosen, no file changes in Phase 3; note in the session log that this is deferred and why.
