# 278. Event Ticket Tiers

Promote event ticket pricing from a single `events.price` field into a tiered structure: `TicketTier` rows owned by an event, an `event_registrations.ticket_tier_id` FK, admin form repeater, public registration tier picker, "General"-tier backfill for existing priced events, and retroactive linkage of the events-importer's `ticket_type` / `ticket_fee` fields to Tier rows.

## Session-specific deltas

The plan entry resolves the "what" (model + FK + admin repeater + public picker + data migration + importer back-link). This session pins the few remaining design choices:

- **Tier shape.** `ticket_tiers` has `id`, `event_id` (FK), `name`, `price` (decimal), `capacity` (nullable int), `sort_order` (int), timestamps. No `currency` column — inherited site-wide. No `description` column — keep tier rows small; the operator describes tier differences in event copy.
- **Where the price lives after this session.** Two viable shapes:
  - **(A) Tier-canonical:** drop `events.price`. Every paid event has at least one tier; price comes from the chosen tier. Free events have zero tiers and check `ticket_tiers()->count() === 0` as the "free" predicate.
  - **(B) Event-canonical-fallback:** keep `events.price` as the single-tier-default price; when tiers exist, they win. `is_free` stays computed off `price === 0 && ticket_tiers()->count() === 0`.
  - **Decide at session start.** (A) is cleaner long-term but expands scope (every read site of `events.price` migrates). (B) is the smaller change but leaves two sources of truth. Lean toward (A) given the pre-beta-license latitude and the "concrete values" rule; confirm before cutting.
- **Capacity model.** Two viable shapes:
  - **(A) Per-tier capacity only:** `events.capacity` drops. Total capacity = sum of tier capacities; "registration full" check is per-tier. A nullable `capacity` on a tier means unlimited.
  - **(B) Event-level capacity + optional per-tier subcaps:** `events.capacity` stays. Tier capacity is an optional subcap inside the event's overall capacity.
  - **Decide at session start.** Lean toward (A) for the same reasons as price.
- **Backfill rule.** For every event with `events.price > 0` at migrate time, create one tier `{name: 'General', price: events.price, capacity: events.capacity, sort_order: 0}`. Events with `price === 0` and `capacity === null` (truly free + unlimited) get no tier created. Events with `price === 0` but `capacity > 0` (free but capped) — judgment call, see the open question below.
- **Retroactive importer linkage.** Session 189's events importer carries `ticket_type` + `ticket_fee` on imported registrations. After the migration runs, walk every `event_registrations` row that has a non-null `ticket_type`:
  - If a tier on the registration's event has a name matching `ticket_type` case-insensitively, link to it.
  - Otherwise, create a new tier on that event with `{name: ticket_type, price: ticket_fee, capacity: null, sort_order: <next>}` and link.
  - Track the match-vs-back-fill ratio in the session log.

## Open questions to resolve at session start

- **Price-shape decision (A) vs (B) above.** Recommend (A); confirm before implementation.
- **Capacity-shape decision (A) vs (B) above.** Recommend (A); confirm before implementation.
- **Free-but-capped events.** Events with `price === 0` and `capacity > 0` — do they get a "General" tier with `price: 0` and `capacity: <event capacity>`, or stay tier-less with `events.capacity` retained as a fallback? If price-shape resolution is (A), the free-but-capped event needs a tier to retain its capacity, so a `{name: 'General', price: 0, capacity: <events.capacity>}` row is the natural backfill. Confirm.
- **What about cancelled / past events at backfill time?** Same backfill rule applies (a past event still has historical registrations whose tier needs to resolve), or skip past events? Lean toward applying uniformly — past registrations need a tier reference for the importer linkage to work.

## Implementation shape

Three (possibly four) iterations:

### Iteration /1 — Schema + backfill + retroactive importer linkage

- Migration `create_ticket_tiers_table` — id, event_id, name, price, capacity, sort_order, timestamps, FK constraint on event_id with cascade-on-event-delete.
- Migration `add_ticket_tier_id_to_event_registrations` — nullable FK, set-null-on-tier-delete.
- Data migration (same `up()` as the table create, or a separate dated migration — judgment call): backfill per the rules above; walk `event_registrations` for the retroactive importer linkage; resolve every existing registration to a tier or leave `ticket_tier_id` null where genuinely free.
- If price-shape (A) lands: drop `events.price` in the same migration block. If (B): leave it.
- If capacity-shape (A) lands: drop `events.capacity` in the same migration block. If (B): leave it.
- `TicketTier` model (`HasUuids` for parity with other CRM-side models if applicable, or `bigInteger` per existing event-related convention — match `event_registrations`).
- `Event::ticketTiers()` relation + helper methods: `isFree()`, `lowestTierPrice()`, `availableTierCount()`.
- `EventRegistration::ticketTier()` relation.
- Update `docs/schema/`: new `ticket_tiers.md`, modify `event_registrations.md` and `events.md` as appropriate, update `docs/schema/README.md`.
- Tests: model factory for `TicketTier`, the backfill assertion (priced event → one General tier; tier-less for genuinely free), the retroactive importer linkage assertion (matching name → reuse; new name → create).

### Iteration /2 — Admin tier repeater

- `EventResource` form gets a Filament repeater on tiers: name (required), price (required, decimal), capacity (optional, int), sort order (managed via Filament's drag-handle).
- Validation: tier name unique within event; price ≥ 0; capacity > 0 when set.
- The form persists tiers via repeater's standard sync.
- Test: a feature test creating an event with two tiers via the admin form and asserting tier rows land correctly.

### Iteration /3 — Public registration flow tier picker + widget integration

- `EventRegistrationDefinition`'s template gains tier selection — when the event has > 1 tier, render a tier picker (radio buttons or select; UX choice at impl time). When the event has exactly 1 tier (the backfilled "General" or operator-created), render hidden + auto-select. When the event has 0 tiers, "free" path as today.
- `EventController::register` (or `EventCheckoutController::store`) reads `ticket_tier_id` from the request, validates membership in the event's tiers, persists on the `EventRegistration` row.
- Stripe Checkout's line-item amount comes from the chosen tier's price (paid path).
- Capacity check (registration-time, per-tier): "this tier is full" instead of "this event is full." If all tiers full, "registration full."
- Playwright spec covering: free event (no tier UI, submit lands registration), single-tier paid event (auto-select, Stripe redirect), multi-tier paid event (visible picker, Stripe redirect with the chosen tier's amount).
- Update `resources/docs/widget-event-registration.md`: drop the "ticket tiers are not on this widget today" gotcha; add a short tier-aware section. Re-run `help:sync`.

### Iteration /4 — *(judgment, only if needed)*

If price-shape (A) requires touching every read site of `events.price`, the convergence sweep may warrant its own iteration. If the affected list is small (importer, widget, maybe one report), fold into /3.

## Out of scope

- **Member-tier-aware pricing** (member discount on a per-tier basis). The Memberships → discount story is a separate ask; tiers ship without that hook this session.
- **Capacity-waitlist on tier-full.** When a tier fills, the public flow shows "this tier is full." No waitlist queue. Matches today's event-level behaviour.
- **Group registration on a single submission.** "Buy 3 General tickets in one go" — not in scope. Each registration is one tier, one attendee.
- **Per-tier registration-window controls** (e.g. early-bird tier auto-expires at a date). Defer — operator can manually remove the tier when the window closes.
- **Tier description / image / "what's included" copy.** Tier rows stay small (name + price + capacity). Operator describes tier differences in event body copy via the page builder.
- **Multi-currency tiers.** Currency inherited from site setting; one currency per install.
