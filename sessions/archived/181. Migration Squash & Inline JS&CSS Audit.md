# 181. Migration Squash & Inline JS/CSS Audit

Squash migration history since the session-142 squash. Audit inline JavaScript and CSS in admin Blade templates for extraction into proper Alpine components and SCSS — carried forward from session 142's companion pass, this time covering anything added (or regressed) in the 40-session interval since. The squash is the load-bearing deliverable; the JS/CSS audit is the "while we're in here" pass, scoped tightly so it does not swallow the session.

---

## Design decisions (resolved before starting)

- **Migration squash follows the session-142 procedure exactly.** `schema:dump`, delete all superseded migration files, verify `migrate:fresh --seed`, update `docs/schema/README.md` squash note. No schema changes during the squash itself.
- **Squash cut-off is today's `database/schema/pgsql-schema.sql` state.** Any migration files that currently exist alongside the dump get collapsed into a fresh dump. No reordering, no edits to historical migrations.
- **Alpine.data() pattern for any extractions.** Per session 142: extracted inline `x-data` blocks become `Alpine.data('componentName', factory)` registrations under `resources/js/page-builder/` with an `index.js` barrel. Blade sites become `x-data="componentName(...params)"`.
- **No behaviour changes during extraction.** Any block that cannot be moved without changing runtime behaviour is flagged and deferred, not rewritten.
- **Inline CSS that is data-driven stays inline.** Only static style blocks or `style=""` attributes that duplicate existing SCSS get lifted.
- **The JS/CSS audit is time-boxed.** If inventory reveals more than ~5 extraction candidates, present the list and let the user decide which subset to do this session. The rest become a stub for a follow-up session.
- **Production deploy coordination.** The squash must be deployed with `migrate:fresh` or manual `schema.sql` replay on the production server. The user handles production; the session only delivers the dump + cleaned migration state.

## Phase 1 — Migration squash

Follow the session-142 / session-062 procedure:

1. Confirm dev DB is in a known-good state: `docker compose exec app php artisan migrate:status` — every migration should be `Ran`.
2. Run `docker compose exec app php artisan schema:dump` to regenerate `database/schema/pgsql-schema.sql`.
3. Delete every migration file that is now represented by the dump (i.e. every file currently under `database/migrations/`).
4. Run `docker compose exec app php artisan migrate:fresh --seed`. Confirm:
   - No errors.
   - The full seed cycle completes.
   - Row counts look sane (users, widget_types, templates, pages, etc. all non-zero).
5. Run the fast test suite: `docker compose exec app php artisan test --exclude-group=slow`. All tests must pass.
6. Update the squash note at the top of `docs/schema/README.md`:
   - Change "Last updated" to today's date.
   - Change the squash-note date and session number to this session.
   - Keep the wording otherwise consistent with the existing note.

Pause and announce the squash is complete before moving to Phase 2.

## Phase 2 — Inline JS inventory

Produce a table of every inline `x-data` block longer than ~5 lines, and every inline `<script>` block of any length, in admin Blade files (Filament resources, Livewire components, admin layouts — **not** widget templates under `app/Widgets/*/template.blade.php`, which are public-site-rendered and have their own build pipeline).

Format:

| File | Line | Proposed component name | Lines | What it does | Dependencies (PHP vars, $wire, $entangle) |
|------|------|------------------------|-------|-------------|-------------------------------------------|

Known starting points to check:

1. **Page-builder Livewire bootstrap** — `resources/views/livewire/page-builder.blade.php` still has the Vue app mount + Livewire↔Vue bridge. Check whether any of that has ossified into inline JS that could move to a module.
2. **Filament form blocks** — any custom form components under `resources/views/filament/forms/` or resource page views that emit inline JS.
3. **Settings pages** — `resources/views/filament/pages/settings/` — check for inline handlers.

Present the table to the user. The user picks which subset to extract this session. Anything not picked becomes a deferred stub (see Phase 4).

---

## Phase 3 — Extract approved components

For each approved block:

1. Create `resources/js/page-builder/{component-name}.js` (or `resources/js/admin/{component-name}.js` if the component is not page-builder-specific) exporting an `Alpine.data` factory.
2. Move the inline logic in, converting any `@js(...)` / `@entangle(...)` / Blade-rendered variables into factory parameters.
3. Register the factory in the appropriate `index.js` barrel.
4. Replace the inline block in the Blade with `x-data="componentName(...params)"`.
5. Run `npm run build` and confirm success.
6. Manually verify the affected page still works — the component is a pure refactor, any behaviour change is a regression.
7. Commit each extraction (or logical group of extractions) separately so regressions are bisectable.

---

## Phase 4 — Inline CSS audit

Scan the same set of Blade files for:

- Inline `<style>` blocks.
- `style=""` attributes.

Classify each hit:

- **Static + duplicate** — same style already exists in SCSS or Filament theme. Remove the inline copy and use the existing class.
- **Static + novel** — static styling that isn't yet in SCSS. Move to the relevant SCSS partial (`resources/scss/` for public styles, or the Filament theme SCSS for admin).
- **Dynamic / data-driven** — bound to Alpine or Livewire state, or composed from Blade-rendered values. Leave inline.

Present findings before making changes. Small changes (1–2 class extractions) can be done inline; larger cleanups should be confirmed with the user first.

Deferred items (anything not picked in Phase 2 or Phase 3) get written into a new stub session in `sessions/session-outlines.md` under "Housekeeping & Review — Beta 1 Scope" so they are not forgotten.

## Phase 5 — Final verification

1. `docker compose exec app php artisan migrate:fresh --seed` — clean completion.
2. `docker compose exec app php artisan test --exclude-group=slow` — all pass.
3. `npm run build` — success.
4. If any widget SCSS/CSS/JS was touched (unlikely in this session): `docker compose exec app php artisan build:public`.
5. Manual smoke test of the page builder and any admin page whose inline JS was extracted.

Announce that testing is ready. Then stop.
