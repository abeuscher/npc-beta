# 061. System Page Type ā€” Infrastructure and Migration

Establishes the `system` page type with all governing rules, then migrates existing portal routes to CMS-backed system pages. Previously planned as two sessions (061 Infrastructure + 062 Migration); combined into one because the boundary between them dissolved during implementation.

---

## Architecture decisions (agreed pre-session)

- `system` is a first-class page type alongside `default`, `post`, `event`, and `member`.
- System pages: type field locked read-only in the editor; slug completely locked (no editing of any segment); not deletable through any UI or bulk action; not offered in the New Page type selector.
- Required widgets are declared by a static definition on the widget type, keyed by page slug. Widget cards for required widgets on system pages render without a remove button. Widget types that appear in any required-widget definition are pinned and cannot be deleted from the Widget Types admin.
- `system_prefix` SiteSetting added to the Routing fieldset in General Settings. Default: `system` ā€” system pages live at `/system/login`, `/system/account`, etc. on a fresh install. Users who want different namespacing can change the prefix; prefix changes migrate all system page slugs with the same safeguards already in place for other prefix types.
- Editor heading for system pages: "Edit System Page".
- No create flow ā€” system pages are seeded only. The type is excluded from the New Page type selector entirely.
- Every publicly viewable page falls under one of the five types. No public-facing route renders outside the CMS (with two technical exceptions noted below).

**Pages migrated to full CMS system pages** (GET route renders through the page builder):

| current route | slug | required widget |
|---|---|---|
| `GET /system/login` | `system/login` | `portal_login` |
| `GET /system/signup` | `system/signup` | `portal_signup` |
| `GET /system/forgot-password` | `system/forgot-password` | `portal_forgot_password` |
| `GET /system/account` | `system/account` | `portal_account_dashboard` |

**Exception pages** ā€” settings-based rich text, Blade views continue to render:

| route | reason | settings key |
|---|---|---|
| `GET /system/reset-password/{token}` | Token in URL path makes slug-based CMS lookup impractical | `system_page_content_reset_password` |
| `GET /system/email/verify` | Verification notice ā€” grouped with reset-password for consistency | `system_page_content_email_verify` |

**Carved out entirely** (action endpoints, not pages):

| route | reason |
|---|---|
| `GET /system/email/verify/{id}/{hash}` | Signed redirect URL ā€” no rendered view, not a page |

**POST routes stay at root** (no prefix) for form action simplicity ā€” no impact on the portal's POST handlers.

Add `system_prefix` to the site settings seeder:

| key | value | group | type |
|-----|-------|-------|------|
| `system_prefix` | `system` | `routing` | `string` |

## Phase 2 ā€” Routing fieldset

Add a `system_prefix` field to the Routing fieldset in General Settings alongside `blog_prefix`, `events_prefix`, and `portal_prefix`.

Apply the same validation, slug migration logic, and warning UI already in place for the other prefix fields. When the prefix changes, all pages with `type = 'system'` have their slugs updated in bulk.

Also add `system_prefix` to `config/site.php` following the same IIFE/try-catch pattern as `blog_prefix` and `events_prefix`.

Update all portal auth GET routes to be prefixed by `system_prefix`:

```php
$systemPrefix = config('site.system_prefix', 'system');
$systemBase   = $systemPrefix ? "/{$systemPrefix}" : '';

Route::get("{$systemBase}/login",                    ...)->name('portal.login');
Route::get("{$systemBase}/signup",                   ...)->name('portal.signup');
Route::get("{$systemBase}/forgot-password",          ...)->name('portal.password.request');
Route::get("{$systemBase}/forgot-password/sent",     ...)->name('portal.password.sent');
Route::get("{$systemBase}/reset-password/{token}",   ...)->name('portal.password.reset');
Route::get("{$systemBase}/email/verify",             ...)->name('portal.verification.notice');
Route::get("{$systemBase}/email/verify/{id}/{hash}", ...)->name('portal.verification.verify');
Route::get("{$systemBase}/account",                  ...)->name('portal.account');
// ... sub-routes
```

POST routes stay at root: `/login`, `/signup`, `/logout`, `/forgot-password`, `/reset-password`.

---

## Phase 3 ā€” Editor UI

When `type = 'system'`, apply the following to the page editor:

- **Type field**: read-only Placeholder ā€” same pattern as `event`, `post`, `member`.
- **Slug field**: completely locked. Display the full current slug as static text. No editable segment, no prefix display ā€” just the full path shown as a read-only value.
- **Editor heading**: "Edit System Page".
- **Delete action**: suppressed ā€” no delete button in the editor toolbar, excluded from bulk delete on the pages list.
- **Pages list**: add a `type` BadgeColumn between slug and is_published; add a SelectFilter for type pre-selected to `['default']` so system/event/member pages are hidden by default.

---

## Phase 4 ā€” Required widgets and seeding

Add `portal_forgot_password` and `portal_account_dashboard` widget types to the widget seeder. (`portal_login` and `portal_signup` were added in session 058 ā€” verify present.)

Seed the four system pages in `DatabaseSeeder` using `firstOrCreate`:
- `type = 'system'`
- `is_published = true`
- Slug honoring `system_prefix` at seed time
- A seeded `PageWidget` record for the required widget, `sort_order = 1`, `is_active = true`

Add a static required-widget definition to the widget type system:

```php
protected static array $requiredFor = [
    'system/login'            => ['portal_login'],
    'system/signup'           => ['portal_signup'],
    'system/forgot-password'  => ['portal_forgot_password'],
    'system/account'          => ['portal_account_dashboard'],
];
```

In the page builder: hide the remove button when a widget's handle appears in `requiredForPage($page->slug)`. Suppress the delete action for pinned widget types in the Widget Types admin.

---

## Phase 5 ā€” Wire GET routes to CMS rendering

For each of the four migrated pages, update the GET route handler to look up the system page by slug and render it through the page builder. Keep named routes intact (`portal.login`, `portal.signup`, etc.).

POST routes are unchanged ā€” they stay as controller handlers.

---

## Phase 6 ā€” Settings rich text for exception pages

Add a "System Page Content" section to General Settings with two rich text fields:

| label | settings key | used by |
|---|---|---|
| Password reset page content | `system_page_content_reset_password` | `ResetPasswordController::show()` |
| Email verification notice content | `system_page_content_email_verify` | `EmailVerificationController::notice()` |

The Blade views for these two routes read the relevant SiteSetting and render it as HTML.
