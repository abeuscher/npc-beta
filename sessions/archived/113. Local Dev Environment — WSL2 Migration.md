# 113. Local Dev Environment — WSL2 Migration

Migrate the local development environment from a Windows bind-mount Docker setup to WSL2-native filesystem for dramatically better I/O performance, container responsiveness, and volume reliability. Wipe and rebuild the database from scratch to verify the new setup end-to-end. Deploy to production afterward to confirm the migration introduced no regressions.

---

## Design decisions (resolved before starting)

- The project files move to the WSL2 Linux filesystem (e.g. `~/nonprofitcrm` inside the WSL distro). The Windows path (`e:\Clients\nonprofitcrm`) becomes a secondary access point via `\\wsl$\` for IDE use, or the IDE connects to WSL directly.
- Docker Desktop's WSL2 backend is used — containers run inside WSL2 natively.
- `docker-compose.yml` volume paths are updated to point to the Linux filesystem.
- The database is wiped (`migrate:fresh --seed`) as part of verification — this is intentional to ensure no setup detail is missed in the new paradigm.
- A deploy to the production server is performed after local verification to confirm nothing broke.
- This session is infrastructure-only — no application code changes beyond configuration and documentation.

---

## Phase 1 — WSL2 filesystem setup

1. **Verify WSL2 is installed and configured** — confirm the distro, Docker Desktop WSL2 integration, and available disk space.
2. **Clone or copy the project** into the WSL2 native filesystem (e.g. `~/nonprofitcrm`).
3. **Update `docker-compose.yml`** volume paths to reference the Linux filesystem.
4. **Verify `.env`** — confirm all paths, database credentials, and service references are correct for the new location.
5. **Bring up containers** — `docker compose up -d` and verify all five containers start cleanly (app, nginx, postgres, redis, worker).

---

## Phase 2 — Database wipe & verification

1. **Run `migrate:fresh --seed`** — full database rebuild from migrations and seeders.
2. **Run the full fast test suite** — `php artisan test --exclude-group=slow`. All tests must pass.
3. **Manual smoke check** — confirm the admin panel loads, a public page renders, and the queue processes a job.

---

## Phase 3 — Production deploy verification

1. **Push the branch to origin** and merge to main (user handles this).
2. **Deploy to production** — user runs the deploy. Verify the app boots, migrations run, and the site is live.
3. **Confirm no regressions** — admin panel, public site, queue worker all functional on production.

---
