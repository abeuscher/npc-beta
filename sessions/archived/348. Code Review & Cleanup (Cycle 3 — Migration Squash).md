## Design decisions (resolved before starting)

- **Squash recipe = the 274 shape verbatim.** `schema:dump` → delete all current migration files → verify `migrate:fresh --seed` identity → update the squash-note block in `docs/schema/README.md`. No edits to historical migrations, no reordering (the dump erases sequence; editing files about to be deleted is wasted effort).
- **Cut-off = today's `database/schema/pgsql-schema.sql` plus whatever new migrations land this session.** As of the 347 close there are **13 migrations** under `database/migrations/` (the May-10 → May-30 set accumulated since the May-9 / 274 baseline). The squash absorbs all of them.
- **Cycle 3 is the compressed 3-session shape, so this squash is LIGHTER than 274.** The audit (344) and apply (345) already did the code-optimization / help-docs / Playwright-teardown / findings work that 274 carried as a four-part inventory. **Do not re-run those inventories.** This session is the migration squash + a short scan for leftover schema picks from 345–347 + the Phase-5 buffer. Be honest if no leftover picks surface — that's the expected outcome.
- **Pre-beta license applies.** No backward-compat shims. If a leftover pick wants to drop a column/table/index, cut it as a new migration that the squash then absorbs.
- **Phase 5 is a buffer, not bundled into the squash commit.** Cycle 2's squash needed a follow-on commit to delete an obsolete migration **test** file (3 tests). Keep Phase 5 separate so squash fallout (an orphaned migration test, a seeder that assumed a now-deleted migration) lands as its own commit, not inside the squash.

## Phase 1 — Inventory (no code changes)

### 1a. Migration inventory + data-phase audit

List all 13 files under `database/migrations/`. For each, capture: file, date, what it does, and **whether `up()` has a data phase** (backfill / row transform / file relocation). The schema dump captures **schema only** — not migrated-in data — so every data-phase migration must be confirmed redundant on the fresh-install path before deletion: the **seeders must produce the end shape directly**.

Known data-phase migrations to verify (confirm the seeder / factory / DemoSeeder emits the post-migration shape, so `migrate:fresh --seed` reproduces it without the migration):

- `2026_05_10_120000_convert_collection_items_trix_to_quill.php` — Trix→Quill content transform on `collection_items`. Verify seeders emit Quill HTML directly.
- `2026_05_10_130000_decode_url_tokens_in_richtext_config.php` — URL-token decode in rich-text config. Verify seeded widget config carries decoded URLs.
- `2026_05_10_160000_create_ticket_tiers.php` — table create **+ backfill** (General tier per priced/capped event, relink registrations, drop `events.price`/`events.capacity`). Verify the event seeders create tiers directly and no seed path writes the dropped columns.
- `2026_05_17_120000_relocate_template_colors_to_theme.php` — colour relocation. Verify the template/theme seeders emit the relocated shape.
- `2026_05_22_120000_add_content_hash_to_media.php` + `2026_05_22_130000_relocate_media_to_content_addressed_storage.php` — content-hash backfill + **filesystem** relocation of media to content-addressed storage. The most involved: verify the media/DemoSeeder writes content-addressed media directly (schema dump won't carry the moved files; a fresh install reseeds media, not relocates it).

The remaining files (`add_search_weight`/`add_parent_slug` to help_articles, `add_quantity` to event_registrations, `add_scheme_and_chrome_suppression` to templates, `create_notifications_table`, `add_sold_out` to events, `add_locked` to pages) are pure schema additions/creates with no data phase — they absorb cleanly.

This is the 208/274 pre-flight: confirm `migrate:fresh --seed` produces the same end state the data-phase migrations produce against an existing DB, **before** any are deleted.

## Phase 4 — Migration squash

Runs only after Phase 3 is complete and every new migration has landed.

1. `docker compose exec app php artisan migrate:status` — every migration `Ran`.
2. `docker compose exec app php artisan schema:dump` — regenerate `database/schema/pgsql-schema.sql`.
3. Delete **every** migration file under `database/migrations/` — all of them, including any written this session.
4. `docker compose exec app php artisan migrate:fresh --seed`. Confirm: no errors; seed cycle clean; row counts sane (spot-check users, widget_types, templates, pages, contacts, organizations, affiliations, ticket_tiers, notifications, media — the post-274 additions are worth explicit verification).
5. Update the squash-note block at the top of `docs/schema/README.md`: "Last updated" → today; squash-note session/date → **348 / today**; keep wording otherwise consistent with the existing note.

Pause and announce the squash is complete before Phase 5.

2. `./dev test` (full fast Pest suite) — all pass. Baseline coming in is **2816 / 0** from the 346 close (347 was Vue-only — no Pest delta). **Squash fallout candidate:** if a migration **test** file references a now-deleted migration, it breaks here — delete it as a **separate** Phase-5 commit (Cycle 2's squash precedent), and log the test-count delta with reasoning.

The 347 standing `inline-formatting-toolbar.spec.ts` (7 cases) + the inline-editing + design-parity specs are the relevant net. **Run Pest and Playwright sequentially — never in parallel** (they collide on DB/FPM).
5. Manual smoke: admin panel loads, page builder mounts, one public page renders.

Announce testing is ready, then stop.
