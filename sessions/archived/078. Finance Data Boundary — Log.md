## What was built

### Phase 1 — Audit

- Full audit of all financial data rendered outside the Finance navigation group.
- Violations found: `PurchasesRelationManager` (amount + Stripe link on CMS Product page); `MembershipsRelationManager` (amount_paid column and form field on CRM Contact page); `contact-notes.blade.php` (Stripe checkout session link in the activity timeline).
- TransactionResource gaps found: no Stripe link on rows, no contact filter, no subject-type filter, purchases not tracked as transactions at all.
- Public-facing views (`donation-form.blade.php`, `product-display.blade.php`) confirmed acceptable — amounts are required there.
- `ProductPricesRelationManager` confirmed acceptable — product price tier setup is configuration, not financial reporting.

### Phase 2 — Remove financial data from non-Finance views

- Deleted `PurchasesRelationManager.php`; removed from `ProductResource::getRelations()`.
- Added "View transactions →" header action on `EditProduct`, pre-filtered by product ID.
- Removed `amount_paid` column from `MembershipsRelationManager` table.
- Removed `amount_paid` form field from `MembershipsRelationManager` edit form.
- Removed Stripe checkout session link from `contact-notes.blade.php` activity timeline.
- Added "View transactions →" header action on `EditContact`, pre-filtered by `contact_id`.

### Phase 3 — Transactions table as financial hub

- Added `contact_id` (nullable UUID FK → contacts, nullOnDelete) to `transactions` via migration; backfilled from donations for all existing rows.
- Updated `Transaction` model: `contact_id` added to fillable, `contact()` BelongsTo relation added.
- Updated all five `Transaction::create` calls in `StripeWebhookController` to populate `contact_id`.
- Made product purchases create a `Transaction` row alongside the `Purchase` record — previously only donations created transactions. This was a data model gap identified during the session.
- Updated `TransactionResource`: added contact name column; added Stripe link row action; added contact SelectFilter (using new `contact_id`), product SelectFilter (subquery through purchases), and subject-type SelectFilter.
- Fixed `stripe_id` to store the payment intent ID (`pi_...`) instead of the checkout session ID (`cs_...`) for checkout-based transactions — `cs_...` IDs have no direct Stripe dashboard page and produce a silent redirect to the dashboard root. Idempotency check in `handleDonationCheckout` updated to match.
- Added `cs_...` URL routing in the Stripe link action to cover any legacy rows still holding a session ID.
- Moved `DashboardDebugGeneratorWidget` above `DashboardTaxonomyWidget` (sort 9 vs 10) to make test data generation easier during development.

## Production deployment notes

Run `php artisan migrate` after pulling. The migration backfills `contact_id` automatically from existing donation records — no manual data work required.

Any existing transaction rows holding a `cs_...` stripe_id will show a Stripe link that silently redirects to the dashboard root rather than the specific payment. Since there is no live data, a reseed clears this.

## Deferred decisions

- Subscription donation checkout (`stripe.subscription` set, `payment_intent` null) creates a transaction with null `stripe_id`. The subsequent `invoice.payment_succeeded` event creates the canonical transaction with the invoice ID, which may result in two transaction rows for the first billing cycle of a recurring donation. Not a blocker for now; noted for a future Finance cleanup session.
