# 189. CRM Importer — Events, Registrations & Transaction Linking

Extend the session 188 importer to a second content type — event data. The distinguishing work is that each row in a real-world events export is denormalized across **three-to-four entities at once**: an event, a contact (looked up, not created), a registration, and optionally a transaction. The session builds a separate wizard (`ImportEventsPage`) and row processor (`ImportEventsProgressPage`) that handle the multi-entity resolution, plus introduces `Transaction ID` as the universal "external identifier for payments" dedupe primitive that session 190 will also key off.

---

## Design decisions (resolved before starting)

- **Denormalized single-CSV shape accepted.** Real-world events exports (Wild Apricot, Bloomerang, Little Green Light, etc.) ship one CSV where each row is a registration plus denormalized event and contact fields. We accept that shape natively rather than asking users to split it upstream.
- **Contacts must pre-exist.** This session does NOT auto-create contacts from the events sheet. A row whose contact can't be resolved via the chosen match key errors out with a clear message ("Contact not found — import contacts first, or re-check the Contact Match Key column"). This is consistent with the one-hour-import philosophy: import contacts first, then events.
- **Events match by external ID only.** Match key is `Event External ID`, resolved via `ImportIdMap` (same pattern as contact `external_id`). If the source has no event ID, the user splits and re-uploads with an ID column — no composite keys in this session.
- **Transaction ID is the universal payment external key.** Renaming the conceptual "Invoice number" to `Transaction ID` in UI labels and mapping dropdowns. Maps to whatever the source calls it — invoice number, payment reference, confirmation code, Stripe PaymentIntent, etc. Backed by `ImportIdMap` keyed on `(import_source_id, model_type='transaction', source_id=transactionId)`.
- **Transaction creation is opt-out, not opt-in.** When the user maps financial fields (amount, payment state, transaction_id) on an events import, Transactions are created/upserted by default. Deduped via `(import_source_id, transaction_id)` so the same invoice showing up in multiple sheets (Events CSV + future Invoice Details CSV in session 190) never duplicates — the second import enriches the first.
- **Duplicate-then-diverge.** `ImportEventsPage` and `ImportEventsProgressPage` are siblings of the contacts versions, not subclasses. Session 191's polish pass will extract a shared base once there are two or three concrete implementations to inform the abstraction.
- **Ticket tiers are out of scope.** Multi-tier pricing goes into its own pre-Beta 1 session (stubbed in `session-outlines.md`). For now, `ticket_type` + `ticket_fee` live as flat fields on `event_registrations`.
- **Financial fields live on `event_registrations` AND on `Transaction`.** Not a duplicate — different concerns:
  - `event_registrations.ticket_type`, `event_registrations.ticket_fee` — denormalized snapshot of what was registered for (survives even if the user never imports Transaction data).
  - `event_registrations.transaction_id` — FK to the Transaction row when one exists.
  - `transactions.amount`, `transactions.payment_state`, etc. — authoritative ledger.
- **Local-only datafiles stay in `local/import-fixtures/wcg/`** (gitignored, same as 188). Anonymised fixtures for tests live under `tests/Fixtures/import/events/`.

## Phase 1 — Data model

Migrations:
- `event_registrations` gains: `ticket_type` (string, nullable), `ticket_fee` (decimal, nullable), `transaction_id` (uuid, nullable, FK→transactions nullOnDelete), `payment_state` (string, nullable — denormalized snapshot; authoritative copy lives on the Transaction when present).
- `Transaction` model audit — confirm it has `external_id` (or equivalent) for cross-system matching. If absent or named differently, add `external_id` (string, nullable) with an index on `(import_source_id, external_id)` for fast upsert lookup. Keep existing fields as-is; only add what's missing.
- `ImportIdMap` — zero schema changes. The existing `(import_source_id, model_type, source_id)` uniqueness already generalises. Start writing `model_type='event'` and `model_type='transaction'` rows.

Update `docs/schema/event_registrations.md` and `docs/schema/transactions.md` (create if missing).

## Phase 2 — Field registries

- `EventFieldRegistry` — mirrors `ContactFieldRegistry`. Lists Event's importable fields (title, slug, starts_at, ends_at, location, description, etc.), derived from `Event::$fillable` with excludes + type overrides.
- `RegistrationFieldRegistry` — ticket_type, ticket_fee, status, checked_in, registered_at, transaction_id, payment_state.
- Mapping Select groups these via Filament's grouped-options syntax — **Event fields**, **Registration fields**, **Contact match** (email, external_id, phone — just the fields usable as a contact lookup key), **Transaction fields** (transaction_id, amount, payment_state, payment_type, online_offline, paid_at), **— Create as custom field —**, **— Import as tag / note / link —** (reuse the relational-destination primitives from 188 where relevant — tags especially).
- Skip-list additions for Wild Apricot noise headers specific to events exports ("Shown to public", "Linked registration", etc. — audit against the fixture and add to the appropriate preset in `FieldMapper::SKIPPED_HEADERS` if necessary).

## Phase 3 — `ImportEventsPage` wizard

Duplicate-then-diverge from `ImportContactsPage`. Three steps:

1. **Upload** — session label, source (same `ImportSource` dropdown, seeded rows already include Generic/Wild Apricot/Bloomerang), tags (applied to newly-created Events), auto-create-custom-fields toggle, file upload. Source presets save/restore the events-specific `field_map` + `custom_field_map` + `match_key` separately from the contacts mapping — same `ImportSource` row carries both, keyed on content type.
2. **Map Columns** — same UI primitives (top nav, per-column dropdown + __custom__/__tag__/__note__ subforms, duplicate-column resolution, "Match contacts by" select). New: a separate "Match events by" select (defaults to external_id if mapped, else errors — no fallback this session).
3. **Preview & Confirm** — 5-row preview, dry-run summary preview.

`ImportSource` preset storage needs a small extension to hold per-content-type mappings (contacts vs events vs donations). Cleanest: add a `content_type` scope to `field_map` / `custom_field_map` / `match_key` — either via a keyed structure (`field_map: { contacts: {...}, events: {...} }`) or three parallel columns (`events_field_map`, `events_match_key`, etc.). Decide during implementation based on which is less invasive.

## Phase 4 — `ImportEventsProgressPage` row processor

Mirrors `ImportProgressPage` (mount auto-runs dry-run, `runCommit()` commits, wire:poll tick). Per-row pipeline:

1. **Resolve Event.** Look up by external_id via `ImportIdMap(model_type='event')`. If found, reuse. If not, create from mapped event fields (title, dates, etc.) and write a new `ImportIdMap` row.
2. **Resolve Contact.** Same pattern — `ImportIdMap(model_type='contact')` for external IDs, email lookup as the user's chosen contact match key. If unresolved → row errors ("Contact not found: <email or ext id>"). Error gets stored, row skipped on commit (same pattern as 188's ambiguous-match errors).
3. **Create Registration.** `EventRegistration::create([...contact + event + ticket_type + ticket_fee + status + registered_at...])`. Emit a timeline note on the contact ("Registered for <event title> (session: <label>)") carrying `import_source_id` same as 188 contact notes.
4. **Upsert Transaction (when financial fields are mapped).** On `(import_source_id, transaction_id)`. If existing Transaction found, Update strategy (fill in blanks, never overwrite populated fields). If not, create. Link `event_registrations.transaction_id` to the result.
5. **Apply relational destinations.** Tags, notes, organization — same primitives as 188. Tags on the registration (new pivot? or on the contact? decide: probably on the registration — "this registrant was VIP for this event"). Notes on the contact.

Dry-run report extends with **Events preview** (would create X events / match Y existing), **Registrations preview** (N new registrations), **Transactions preview** (M new transactions / K matched existing).

## Testing

- **Slow test groups this session:** none expected.
- **New tests expected:** yes.
  - Events: external ID match reuses the existing Event; no match creates a new one.
  - Contacts: unresolved contact rows error with a clear message, stay in `skipRowNumbers`, don't create registrations.
  - Registrations: each committed row produces exactly one `EventRegistration` linked to the right contact and event.
  - Transaction upsert: running the same events CSV twice doesn't duplicate Transactions (dedupe on transaction_id works); a second sheet with richer fields for the same transaction_id updates (doesn't overwrite populated fields).
  - Financial linking: `event_registrations.transaction_id` resolves to the right Transaction; nullable when no financial fields were mapped.
  - Dry-run rollback: all writes (events, registrations, transactions, id_maps) roll back cleanly.
  - Source preset save/restore: saving an events import mapping persists as events-scoped; contact preset on the same source is untouched.
- Anonymised fixture CSVs under `tests/Fixtures/import/events/` — happy path, denormalized with mixed pre-existing and new events, unresolved contact, malformed dates, duplicate-Transaction across two runs.
- Run `docker compose exec app php artisan test --exclude-group=slow` before close.
