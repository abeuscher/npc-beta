# 171. Defaults Binding & Sovereign Rendering

Stage 2 of the Sovereign Widget arc. Make `WidgetDefinition::defaults()` the runtime source of truth for default values, and introduce a `WidgetConfigResolver` that composes defaults â†’ theme overrides (stub) â†’ instance config into a fully-resolved config. Templates stop defending against missing keys with `$config['x'] ?? ''`. Instance configs become sparse â€” only user overrides are stored. The nav widget is the proof-of-concept; other widgets continue working unchanged via a coexistence path.

> **Architecture note.** The page builder is a **Livewire shell + Vue island**. The Livewire shell (`App\Livewire\PageBuilder`) provides bootstrap data and owns two modals (add-widget, save-as-template); everything interactive â€” canvas, inspector, field editing, drag/drop â€” is Vue 3 + Pinia, persisting through the REST API at `App\Http\Controllers\Admin\PageBuilderApiController`. The legacy per-block and inspector Livewire classes were removed in sessions 146â€“152 and **do not exist**. This prompt names the current files.
>
> The core invariant this session must preserve: **the inspector's displayed value for any field must always equal the value the renderer uses for that field.** If the renderer paints nav links blue, the inspector's link-color field reads blue â€” never blank, never a different shade. Achieving this architecturally is the reason the resolver exists.

## Design decisions (resolved before starting)

- **Defaults source of truth.** For widgets that have a `WidgetDefinition` in the registry, `defaults()` wins. For widgets still declared only in `WidgetTypeSeeder`, fall back to `WidgetType::getDefaultConfig()` (which reads from `config_schema`). No widget is forced to migrate this session.
- **Resolver shape.** `App\Services\WidgetConfigResolver` is a stateless service with one public method: `resolve(PageWidget $pw): array`. It composes in this order â€” `defaults()` â†’ theme overrides (stub: empty array for now, feature-flagged hook point) â†’ instance `$pw->config`. Later values overwrite earlier ones, per-key.
- **Theme overrides are a stub.** A private method `themeOverrides(PageWidget $pw): array` returns `[]` in Stage 2. Document the seam so Stage N can wire in template/theme-level defaults without touching widget templates again.
- **Sparse instance configs â€” creation and save.** New widgets are created with `config = []` (sparse from birth). On save, keys whose submitted value equals the resolved default are stripped before persisting. This keeps the design self-consistent and lets future changes to `defaults()` propagate to untouched fields on existing widgets. Pre-existing fat rows are not rewritten; they continue working because the resolver composes correctly from them.
- **Nav template migration.** Port `resources/views/widgets/nav.blade.php` to assume `$config` is fully resolved. Remove `?? ''` / `?? 'default'` fallbacks. Leave other widget templates alone.
- **Renderer integration.** `WidgetRenderer::render()` calls the resolver before passing `$config` to Blade. For widgets without a definition, the resolver still runs but composes from `$widgetType->getDefaultConfig()` (coexistence branch). Every widget benefits from theme-override stub wiring without per-widget changes.
- **Inspector â€” resolved defaults on the wire.** The API payload for each widget includes a new `resolved_defaults` field: a map of schema-key â†’ the value the resolver would use if the instance did not override that key. The Vue inspector displays `widget.config[key] ?? widget.resolved_defaults[key]` â€” **`field.default` from the schema is no longer consulted at display time**. This is what makes the "inspector === renderer" invariant architectural rather than coincidental.
- **Override indicator and reset-to-default.** The inspector shows a small per-field indicator (dot or "inherited" label â€” simplest affordance that reads at a glance) when the field is overridden. A per-field "reset to default" action clears the key from the instance config. No bulk reset this session.
- **Resolver registered as a singleton.** In `WidgetServiceProvider::register()`, so both `app(WidgetConfigResolver::class)` and constructor injection work uniformly.
- **`removeImage` null-write is allowed to be stripped.** If `null` is the resolved default for an image field, sparse-save strips the key. Functionally identical â€” the resolver re-adds `null` â€” and consistent with the sparse-save rule.
- **No DB migration.** `widget_types.config_schema` still carries `default` keys (consumed by the seeder and by widgets that don't yet have a definition). `page_widgets.config` remains JSONB but now stores sparse data going forward.
- **Demo data / preview unchanged.** `DemoDataService` keeps working. The resolver applies to real renders and admin previews equally.

## Phase 2 â€” WidgetConfigResolver

Create `app/Services/WidgetConfigResolver.php`:

- `resolve(PageWidget $pw): array` â€” returns the merged config array.
- Internally:
  1. Look up the widget's `WidgetDefinition` via `WidgetRegistry::find($handle)`.
  2. If found, start from `$def->defaults()`. Otherwise, start from `$pw->widgetType->getDefaultConfig()`.
  3. Apply `themeOverrides($pw)` â€” stub returning `[]` for now. Leave a clear extension point (typed docblock).
  4. Apply `$pw->config ?? []` on top.
  5. Return the merged array.

- `hasOverride(PageWidget $pw, string $key): bool` â€” returns true when the instance config contains `$key` and its value differs from the resolved default (defaults + theme overrides). Used by the inspector payload.

- `defaultFor(PageWidget $pw, string $key): mixed` â€” returns the value the resolver would use if the instance did not override `$key`. Used to build `resolved_defaults`.

- `resolvedDefaults(PageWidget $pw): array` â€” convenience: returns the full `defaults + themeOverrides` map (without the instance layer). Used by the API payload builder.

Register as a singleton in `WidgetServiceProvider::register()`.

## Phase 4 â€” Sparse Save & Inspector Payload

### 4a. Sparse creation

Update both creation paths to seed `config => []`:

- `App\Http\Controllers\Admin\PageBuilderApiController::store()`
- `App\Livewire\PageBuilder::createBlock()`

### 4b. Sparse save

Update `PageBuilderApiController::update()`:

1. When the request includes a `config` array, compute the resolved defaults for each key.
2. Strip keys whose submitted value equals the resolved default (strict `===` comparison).
3. Persist only the remaining keys to `page_widgets.config`.

Edge cases:

- Image / file fields that reference media are not stripped when equal to `null`-default â€” but only because media upload writes a non-null media ID. `removeImage` writes `null`; if `null` is the resolved default, the key is stripped. This is acceptable â€” the resolver re-adds `null` at render time.
- Pre-existing fat rows are not migrated in this session. They will be naturally slimmed the next time a user saves.

### 4c. `resolved_defaults` in the API payload

Extend `PageBuilderApiController::formatWidget()` and `formatWidgetWithPreview()` to include a `resolved_defaults` key â€” the output of `WidgetConfigResolver::resolvedDefaults($pw)`. Also extend `App\Livewire\PageBuilder::getBootstrapData()` widget entries to include the same key, so the initial page load carries it.
