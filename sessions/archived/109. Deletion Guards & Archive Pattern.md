# 109. Deletion Guards & Archive Pattern

Implement deletion safety guards across all content types and introduce an "archived" status for models that need a "retired but historically referenced" state. This is the user-facing safety net — preventing accidental data loss and giving admins a clear retire-without-delete workflow.

---

## Design decisions (resolved before starting)

- Each content type gets a deletion guard appropriate to its risk level: hard block, soft warning, or no guard.
- The archive pattern is a new status value, not a soft delete. Archived records are hidden from default list views and excluded from new-association dropdowns, but all history is preserved.
- Donations and Email Templates are never deletable — no UI for delete should exist.
- No schema changes for deletion guards — these are policy checks in Filament resources. Archive status requires adding an `is_archived` column (or equivalent) to applicable tables.

## Phase 1 — Deletion guards

Implement deletion protection for each content type. Read each resource file before modifying it.

### Hard blocks (prevent deletion entirely)

- **Donation** — not deletable. Remove any delete action from the resource if present. Financial record; Stripe is source of truth.
- **Email Template** — not deletable. Already enforced — verify and confirm.

### Conditional blocks (prevent deletion when dependencies exist)

- **Event** — block deletion if any registrations exist. Allow for drafts with no registrations.
- **Event Registration (paid)** — block deletion with a warning directing admin to issue the refund in Stripe first. Allow after acknowledgement. Do not automate the refund.
- **Membership Tier** — block deletion if any active memberships reference it.
- **Fund** — block deletion if any donations reference it.
- **Product** — block deletion if any purchases reference it.
- **Widget Type** — block deletion if any page widgets reference it.
- **Form** — warn if submissions exist. Allow deletion after confirmation.
- **Collection** — warn if items exist. Allow deletion after confirmation.

### Warning-only (allow deletion with a clear warning)

- **Navigation Menu** — general warning on delete: "This menu may be in use on your site." No dependency check — too unpredictable given widget-level nav use.
- **Page (system/portal)** — audit existing page-type protection and plug any gaps. Add a loud second-tier warning for system pages: "Deleting this page may render the member portal unusable."
- **Contact** — soft-delete in place. Hard-delete restricted to super_admin only. Verify this is enforced.

## Phase 2 — Archive pattern

Introduce an explicit "archived" state for models where "retired but historically referenced" is a real operational state. For each model:

1. Add an `is_archived` boolean column (default false) via migration.
2. Add an "Archive" / "Unarchive" toggle action in the Filament resource.
3. Default list view excludes archived records (add a filter to show them).
4. Archived records cannot be selected for new associations (exclude from relationship dropdowns).
5. Update `docs/schema.md` for each migration.

### Models to archive

- **Product** — "no longer for sale"
- **Membership Tier** — "closed to new members"
- **Fund** — "no longer accepting donations"
- **Form** — "retired"

### Models to evaluate

Evaluate **Mailing List** and **Collection** against the same criteria during this session. Implement if appropriate; document the decision if not.
