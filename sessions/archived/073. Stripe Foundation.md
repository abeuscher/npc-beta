# 073. Stripe Foundation

Build the infrastructure that all Stripe-connected features depend on: split and encrypted API key storage, a verified webhook receiver, and the core transaction recording pattern. No payment initiation is built in this session — inbound webhooks only.

---

## Core security principle

**This application is deposit-only. It never initiates a refund, payout, balance transfer, or any outbound financial transaction through Stripe or any other service.** All financial reversals are handled directly in the Stripe dashboard by authorised staff. This constraint is enforced at the API key scope level, not just in application code. It must never be loosened without explicit design review.

---

## Design decisions (resolved before starting)

**API key split:**
- The existing single `stripe_api_key` SiteSettings entry is replaced with three entries: `stripe_publishable_key`, `stripe_secret_key`, `stripe_webhook_secret`.
- `stripe_publishable_key` is stored plaintext — it is a public key by design.
- `stripe_secret_key` and `stripe_webhook_secret` are stored encrypted using `Crypt::encryptString()` / `Crypt::decryptString()` (Laravel's AES-256-CBC, keyed from `APP_KEY`).
- Stripe keys are removed from `.env` entirely. The DB is the single source of truth. `AppServiceProvider` is updated to load all three keys and decrypt the secret ones.
- The Settings admin UI uses password-style inputs for the two secret fields — they never echo the stored value back to the browser.

**Restricted API key scope:**
The Stripe restricted key used by this application must be scoped to the minimum required permissions. At the end of this session, document in the session log exactly which Stripe permissions the key needs. Permissions that must be **absent**: Refunds (write), Payouts (write), Balance transfers (write), Customer payment methods (write beyond what Checkout requires). The goal is that a stolen key cannot move money out of the account.

**Transaction table — polymorphic subject:**
The existing `transactions.donation_id` FK is replaced with a polymorphic pair: `subject_type` (string, nullable) and `subject_id` (string, nullable). This allows one table to cover donations, event registrations, memberships, and future payment types without adding a nullable FK column for each. Write a migration. Update `docs/schema.md`. Update the `Transaction` model.

**Subjects store their own Stripe ID:**
Each payable model stores its own `stripe_payment_intent_id`. `event_registrations` already has this column. `donations` will have one added in a later session when online donation is built. The webhook handler looks up the subject by the payment intent ID embedded in the Checkout session metadata (see Phase 3).

**Webhook events handled in this session:**
- `checkout.session.completed` — primary success path; record a completed transaction
- `payment_intent.payment_failed` — record a failed transaction for visibility
- All other event types — return HTTP 200 immediately without processing; do not error

**Idempotency:**
Before recording any transaction, check whether a `transactions` row with the same `stripe_id` already exists. If it does, return 200 without writing. Stripe may deliver the same event more than once.

**No raw payload storage:**
Stripe webhook payloads can contain billing details and other PII not needed by this system. Do not store raw payloads. Extract only the fields needed (event type, payment intent ID, amount, currency, status) and discard the rest.

**Metadata convention (established here, consumed in future sessions):**
When a Checkout session is created (next session), it will include a `metadata` object: `{subject_type: 'App\\Models\\EventRegistration', subject_id: 'uuid'}`. The webhook handler reads this metadata to link the transaction to the correct subject. The convention is established and documented here so future sessions have a clear contract to implement against.

## Phase 1 — API key management

Read the current SiteSettings admin page and `AppServiceProvider` before making any changes.

1. Write a migration that:
   - Renames the `stripe_api_key` settings row to `stripe_secret_key` (update the `key` column value).
   - Inserts two new rows: `stripe_publishable_key` (type: string) and `stripe_webhook_secret` (type: string).
   - Sets `type = 'encrypted'` on both `stripe_secret_key` and `stripe_webhook_secret`.

2. Add encrypted read/write support to `SiteSetting`: when `type = 'encrypted'`, the model encrypts on set and decrypts on get. Existing non-encrypted rows are unaffected.

3. Update `AppServiceProvider` to load `stripe_publishable_key`, `stripe_secret_key` (decrypted), and `stripe_webhook_secret` (decrypted) into `config('services.stripe.*')`.

4. Create a new Filament Settings page — `FinancialSettingsPage` — following the same pattern as existing settings pages. Gate it behind a dedicated `manage_financial_settings` permission. Read the existing role/permission setup before deciding the best way to assign this permission to the `developer` and `treasurer` roles. The page must not appear in navigation for users without this permission.

5. The Financial Settings page exposes fields for all four keys. The three secret fields (`stripe_secret_key`, `stripe_webhook_secret`, `quickbooks_api_key`) use `->password()->dehydrated(fn ($state) => filled($state))` — never echoed, only written when a new value is submitted. The publishable key is a standard text input. Remove the Stripe and QuickBooks key fields from General Settings.

6. Remove Stripe and QuickBooks keys from `.env` — the DB is the single source of truth.

Update `docs/schema.md` for any schema changes.

## Phase 3 — Webhook receiver

1. Create `App\Http\Controllers\StripeWebhookController` with a single `handle(Request $request)` method:
   - Retrieve the raw request body and `Stripe-Signature` header.
   - Verify the signature using the `stripe_webhook_secret` from config and the Stripe PHP SDK's `\Stripe\Webhook::constructEvent()`. Return HTTP 400 on failure.
   - Dispatch to a private handler method based on `$event->type`. Unknown types return HTTP 200 immediately.
   - The entire handler is wrapped in try/catch — any unhandled exception returns HTTP 200 to prevent Stripe retry storms. Log the exception internally.

2. Implement `handleCheckoutSessionCompleted` and `handlePaymentIntentFailed`:
   - Extract: payment intent ID, amount (convert from cents to dollars), currency, status.
   - Extract `subject_type` and `subject_id` from `$event->data->object->metadata`.
   - Idempotency check: if `Transaction::where('stripe_id', $intentId)->exists()`, return without writing.
   - Create a `Transaction` record: `subject_type`, `subject_id`, `type = 'payment'`, `amount`, `direction = 'in'`, `status` (completed / failed), `stripe_id`, `occurred_at = now()`.

3. Register the route in `routes/web.php` (or `api.php` — confirm which is appropriate given CSRF exemption needs):
   - `POST /stripe/webhook` → `StripeWebhookController@handle`
   - Exempt from CSRF middleware.

4. Add the Stripe PHP SDK if not already present: `composer require stripe/stripe-php`.

## Security checklist

Before closing:

- [ ] Webhook signature verified on every request before any processing — unsigned requests return 400.
- [ ] Stripe secret key and webhook secret are encrypted at rest; never stored or logged in plaintext.
- [ ] No card data, billing address, or other payment instrument PII from the Stripe payload is stored.
- [ ] The application holds no Stripe permission that allows it to initiate a refund, payout, or balance transfer.
- [ ] Idempotency check prevents duplicate transaction records on repeated webhook delivery.
- [ ] Webhook endpoint is CSRF-exempt and does not require authentication, but is verified by signature.
- [ ] Session log documents the exact Stripe API key permission set required.
