# 059. Password Reset

Adds self-service password reset for portal accounts. This is the last security gap from session 056.

---

## Architecture decisions (agreed pre-session)

- Uses Laravel's built-in password broker system, extended to work with the `portal` guard and `PortalAccount` model.
- A dedicated `portal_password_reset_tokens` table (separate from the admin `password_reset_tokens` table — the two user populations must never share a token store).
- The reset email is editable in the system email editor (same `EmailTemplate` pattern as other system emails).

---

## Phase 1 — Database

Add a `portal_password_reset_tokens` table. Same structure as Laravel's default `password_reset_tokens`:

| column | type | notes |
|--------|------|-------|
| `email` | string | PK |
| `token` | string | bcrypt-hashed token |
| `created_at` | timestamp, nullable | |

Add a `portal` password broker to `config/auth.php`:

```php
'passwords' => [
    'users' => [...],
    'portal_accounts' => [
        'provider' => 'portal_accounts',
        'table'    => 'portal_password_reset_tokens',
        'expire'   => 60,
        'throttle' => 60,
    ],
],
```

---

## Phase 2 — Routes

| method | URI | name | notes |
|--------|-----|------|-------|
| GET | `/forgot-password` | `portal.password.request` | "enter your email" form |
| POST | `/forgot-password` | `portal.password.email` | sends reset link |
| GET | `/reset-password/{token}` | `portal.password.reset` | reset form |
| POST | `/reset-password` | `portal.password.update` | processes new password |

---

## Phase 3 — Controllers

### ForgotPasswordController

`GET /forgot-password` — render the "enter your email" form. Redirect to `/account` if already authenticated and verified.

`POST /forgot-password` — validate email (required, email). Look up `PortalAccount` by email. Whether or not the account exists, respond with the same message: "If an account with that email exists, we've sent a reset link." This prevents email enumeration. If the account exists, send the reset email via the `portal_accounts` password broker.

### ResetPasswordController

`GET /reset-password/{token}` — render the password reset form (token + email + new password + confirm).

`POST /reset-password` — validate, attempt `Password::broker('portal_accounts')->reset(...)`. On success: redirect to `/login` with a flash "Your password has been reset. Please log in." On failure: redirect back with error.

---

## Phase 4 — Email

A new mailable or notification wired to the `portal_accounts` broker. The reset email is editable in the system email editor via a `portal_password_reset` handle in `EmailTemplate`. Available tokens: `{{first_name}}`, `{{reset_url}}`.

Default subject: `Reset your password`
Default body: `<p>Hi {{first_name}},</p><p>Click the link below to reset your password. This link expires in 60 minutes.</p><p><a href="{{reset_url}}">Reset password</a></p><p>If you did not request a password reset, you can safely ignore this email.</p>`

---

## Phase 5 — Views

All views extend `layouts.public` (not the portal layout — the user is unauthenticated at this point).

- `portal/forgot-password.blade.php`
- `portal/reset-password.blade.php`

Add a "Forgot password?" link to `portal/login.blade.php` pointing to `portal.password.request`.

---

## Security checklist

Before closing:
- [ ] Forgot password response is identical whether or not the email exists
- [ ] Reset tokens are stored hashed
- [ ] Reset links expire in 60 minutes
- [ ] After a successful reset the token is consumed and cannot be reused
- [ ] The `portal_password_reset_tokens` table is entirely separate from `password_reset_tokens`

---

## Out of scope

- Admin UI for forcing a portal account password reset
- "Change password" form on the portal account page (member self-service — later session)
