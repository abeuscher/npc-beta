# 056. Secure Public Signup Flows

This session builds the authentication foundation for the public member portal. It covers only signup, login, logout, and email verification ‚Äî no portal UI, no member-specific profile fields. Those arrive in session 057.

---

## Architecture decisions (agreed pre-session)

- **Custom auth guard** against a `portal_accounts` table. Do not use Laravel Breeze or Fortify ‚Äî they assume the `User` model and would require overriding most of their logic anyway. Use Laravel's guard/provider extension points directly.
- **`portal_accounts`** is the credential table. It has a `contact_id` FK ‚Äî it is not a replacement for contacts and does not merge concerns with the admin `users` table.
- **Members and volunteers are contacts with portal access.** A contact with a `portal_account` record has portal credentials. Type-specific data (member tier, volunteer skills, etc.) lives in separate profile tables (`member_profiles`, `volunteer_profiles`) added in later sessions ‚Äî not on `contacts`.
- **Security rule ‚Äî no PII leakage.** Portal pages must never expose data belonging to anyone other than the authenticated portal user. This is a hard architectural rule, not a nice-to-have. Every route, query, and view in the portal must be scoped to `auth('portal')->user()->contact`. No list pages, no search, no "find another member" surfaces unless those are explicitly designed and reviewed in a future session.

## Phase 1 ‚Äî Database

### `portal_accounts` table

| column | type | notes |
|--------|------|-------|
| `id` | bigint PK | |
| `contact_id` | bigint FK ‚Üí contacts | nullable; null means not yet linked (should not persist ‚Äî always resolve at signup) |
| `email` | string 255, unique | copied from contact at signup; used as login identifier |
| `password` | string | bcrypt hash |
| `email_verified_at` | timestamp, nullable | null = unverified |
| `remember_token` | string 100, nullable | |
| `created_at` / `updated_at` | timestamps | |

Create `app/Models/PortalAccount.php`. It must implement `Illuminate\Contracts\Auth\Authenticatable` (use the `Authenticatable` trait) and `Illuminate\Contracts\Auth\MustVerifyEmail` (implement the interface methods). It belongs to a `Contact` via `contact_id`.

## Phase 2 ‚Äî Auth Guard

### Config

Add a `portal` guard and a `portal` provider to `config/auth.php`:

```php
'guards' => [
    // ... existing web/admin guards ...
    'portal' => [
        'driver'   => 'session',
        'provider' => 'portal_accounts',
    ],
],

'providers' => [
    // ... existing providers ...
    'portal_accounts' => [
        'driver' => 'eloquent',
        'model'  => App\Models\PortalAccount::class,
    ],
],
```

### Middleware

Register a `portal.auth` middleware alias in `bootstrap/app.php` (or `app/Http/Kernel.php` depending on Laravel version) that redirects unauthenticated requests to `/login`. Verify email using the standard `verified` middleware wired to the portal guard.

## Phase 3 ‚Äî Routes

Add a `routes/portal.php` file (or append to `routes/web.php` under a named group ‚Äî check how existing public routes are structured and follow that pattern). Routes:

| method | URI | name | notes |
|--------|-----|------|-------|
| GET | `/signup` | `portal.signup` | signup form |
| POST | `/signup` | `portal.signup.post` | process signup |
| GET | `/email/verify` | `portal.verification.notice` | "check your email" page |
| GET | `/email/verify/{id}/{hash}` | `portal.verification.verify` | signed verify link |
| GET | `/login` | `portal.login` | login form |
| POST | `/login` | `portal.login.post` | process login |
| POST | `/logout` | `portal.logout` | logout |
| GET | `/account` | `portal.account` | placeholder portal home (just a "welcome, [name]" view for now) |

The `/account` route requires `auth:portal` and `verified` middleware.

## Phase 4 ‚Äî Controllers & Logic

### SignupController

`GET /signup` ‚Äî render the signup form.

`POST /signup` ‚Äî validate, then:

1. Check whether a `portal_account` already exists for this email. If yes, redirect back with error "An account with this email already exists. Please log in."
2. Look up a `Contact` by email.
   - **If found:** update `first_name` and `last_name` on the contact if they were empty; leave all other contact fields untouched. Link the new `portal_account` to this contact.
   - **If not found:** create a new `Contact` with `first_name`, `last_name`, `email`, and `source = 'member_signup'`.
3. Create the `portal_account` record with a hashed password and `email_verified_at = null`.
4. Send a verification email (signed URL to `portal.verification.verify`).
5. Log in the user via `Auth::guard('portal')->login($account)`.
6. Redirect to `portal.verification.notice`.

Validation rules:
- `first_name` ‚Äî required, max 100
- `last_name` ‚Äî required, max 100
- `email` ‚Äî required, email, max 255
- `password` ‚Äî required, confirmed, min 12

### LoginController

`GET /login` ‚Äî render login form. Redirect to `/account` if already authenticated via portal guard.

`POST /login` ‚Äî attempt `Auth::guard('portal')->attempt(['email' => ..., 'password' => ...], $remember)`. On success redirect to `/account`. On failure redirect back with error "Invalid email or password."

`POST /logout` ‚Äî `Auth::guard('portal')->logout()`, invalidate session, redirect to `/login`.

### EmailVerificationController

`GET /email/verify` ‚Äî render "check your email" notice. Redirect to `/account` if already verified.

`GET /email/verify/{id}/{hash}` ‚Äî validate the signed URL, mark `email_verified_at = now()`, redirect to `/account` with a success flash.

## Phase 5 ‚Äî Views

All portal views go in `resources/views/portal/`. They use the existing public layout (`layouts.public`) or a minimal wrapper ‚Äî whichever is lighter. No new layout file unless the public layout is genuinely incompatible.

Views needed:
- `portal/signup.blade.php`
- `portal/login.blade.php`
- `portal/verify-email.blade.php`
- `portal/account.blade.php` ‚Äî just a heading "Welcome, [first name]" and a logout button for now

Forms use the standard public form styling already in place (no Filament, no extra JS framework). Honeypot + timing fields on signup (same pattern as event registration).

## Security checklist

Before closing:
- [ ] Every portal route that reads contact data is scoped strictly to `auth('portal')->user()->contact_id`
- [ ] The verification URL uses `URL::temporarySignedRoute` (or equivalent) ‚Äî it expires and cannot be replayed
- [ ] Passwords are hashed with `bcrypt` ‚Äî never stored plain
- [ ] The login form does not reveal whether an email address exists in the system (same error message for wrong email vs wrong password)
- [ ] Honeypot + timing check on signup form
