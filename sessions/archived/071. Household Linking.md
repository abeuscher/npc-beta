# 071. Household Linking

Replace the existing `households` table with a self-referential household model on contacts, add admin household assignment to the contact edit form, and add a household display and address-change prompt to the member portal.

No backwards compatibility required. Drop the existing `households` table and `HouseholdResource` and start clean.

---

## Design decisions (resolved before starting)

**Data model:**
- `contacts.household_id` becomes a self-referential FK ‚Üí `contacts.id`.
- A contact where `household_id = contact.id` is the household head (or is solo ‚ÄĒ same thing).
- A contact where `household_id = some_other_contact.id` is a member of that contact's household.
- Household name is always computed: `"[head contact's last_name] Household"`. Never stored.
- There is no separate `households` table.

**Admin assignment:**
- An admin sets `contact.household_id` on the contact edit form.
- When a contact is assigned to a household (i.e. their `household_id` is changed to another contact's ID), their address fields are automatically updated to match the head contact's address. This keeps addresses in sync at the moment of assignment.

**Portal:**
- The member portal account page displays the member's household name if they are in one (i.e. `household_id ‚Č† contact.id`). Read-only.
- When a non-head member updates their address in the portal, they are prompted: "Update just my address (this will remove you from the household)" or "Update the household address for everyone." No address change is saved until the member chooses.
- When the head contact updates their address, it cascades automatically to all contacts whose `household_id = head.id`. No prompt ‚ÄĒ the head's address is definitionally the household address.

**Member-to-member invite flow:** Out of scope. Household assignment is admin-only for now.

**No staff approval step.** Admin assignment is direct.

## Phase 1 ‚ÄĒ Schema

Write a migration that:
1. Drops the `households` table (with cascade ‚ÄĒ contacts currently reference it).
2. Adds `household_id` (UUID, nullable, FK ‚Üí `contacts.id`, `nullOnDelete`) to the `contacts` table.
3. Sets `household_id = id` for all existing contacts (each becomes their own head).

Update `docs/schema.md` to remove `households` and update the `contacts` table entry.

---

## Phase 2 ‚ÄĒ Model and admin

**Contact model:**
- Add the `household_id` to `$fillable`.
- Add `head()` belongsTo relationship: `Contact::belongsTo(Contact::class, 'household_id')`.
- Add `householdMembers()` hasMany relationship: contacts whose `household_id = this->id`.
- Add a computed `householdName()` method: returns `"{$this->head->last_name} Household"` if `household_id ‚Č† id`, otherwise `"{$this->last_name} Household"`.

**Drop HouseholdResource** and its pages/relation managers. Remove it from navigation.

**ContactResource:**
- Add a Household section to the contact edit form. Use a searchable `Select` that lists contacts by full name. The selected contact becomes this contact's household head (`household_id`). Pre-populate with the contact's current head.
- When the household_id is changed via the form, after save, automatically copy the head contact's address fields to this contact.
- Read how the existing ContactResource edit form is structured before modifying it.

---

## Phase 3 ‚ÄĒ Portal

**Account page household display:**
- Read the existing portal account page before touching it.
- If the authenticated member's `household_id ‚Č† contact.id`, display a read-only line showing the household name (e.g. "Smith Household"). If they are solo (`household_id = contact.id`), show nothing.

**Address change prompt:**
- Read the existing portal address update controller/route before touching it.
- When a non-head member submits an address change, intercept and return a view (or the same page with a confirmation state) that presents the two choices:
  - "Update just my address" ‚ÄĒ sets `contact.household_id = contact.id` and saves the new address.
  - "Update the household address for everyone" ‚ÄĒ updates the head contact's address AND cascades to all contacts whose `household_id = head.id`.
- When the HEAD member submits an address change, save their address and cascade to all members automatically. No prompt.

## Security checklist

- [ ] Portal address update is scoped to the authenticated portal user's own `contact_id` ‚ÄĒ no cross-contact address changes.
- [ ] Household display on the account page only exposes the household name (derived from head's last name) ‚ÄĒ no other PII about household members is exposed.
- [ ] Admin household assignment is gated by the existing contact edit permission.
