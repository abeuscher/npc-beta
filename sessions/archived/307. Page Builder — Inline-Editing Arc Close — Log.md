## What was built

### Phase 1 — Standing builder↔public render-parity guard

`tests/Feature/BuilderPublicRenderParityTest.php` in the `design` Pest group asserts that `WidgetRenderer::render($pw, ..., inlineEditing: true)` (the page-builder canvas branch) is appearance-equivalent to `WidgetRenderer::render($pw, ..., inlineEditing: false)` (the public render path) for the full inline-eligible widget set, modulo a documented expected-diff set: the `data-config-placeholder="..."` attribute (builder-only emission), empty inline-prose wrappers around blank non-`always` fields (builder-only), and the EventCalendar per-render random `id="cal-XXXXXXXX"` minted by `Str::random(8)`. Both branches use the `page_builder_canvas` slot (which is what production's public render path uses too — the 305 conflation split moved editing-only behaviour to a separate `$inlineEditing` parameter, not to the slot). A normalisation function strips the three documented diffs from both branches symmetrically; anything remaining must be byte-identical between builder and public.

16 cases shipped: TextBlock filled + blank, Hero, ThreeBuckets (one filled / two empty exercising non-`always` wrappers), PricingChart populated (the canonical 304 stress test) + partially-empty (exercises `always` subgrid tracks), each of the nine widgets widened in Phase 2, and a sanity assertion that the `data-config-placeholder` attribute actually exists on builder but not on public — proving the harness is doing real work, not vacuously passing.

`DesignGroupIntegrityTest`'s pinned list updated 19 → 20 (the new parity file added). The 296 `AssetBundleDriftGuardTest` (served-bundle-layer drift) and this new render-layer guard are complementary by design — both stay in the `design` group, both pinned in the integrity list, both run as part of `./dev test:design` ≈21s scoped loop.

### Phase 2 — Widened the code-declared eligibility gate to 9 additional widgets

The four inline-eligible widgets at 306 close (TextBlock, Hero, ThreeBuckets, PricingChart) grew to 13. Each widening edit was: (a) `inlineEditable(): bool { return true; }` on the widget's Definition class, (b) the heading display-prose node migrated from a literal `<h2 ...>` (or `@if ($heading) <h2 ...>` for the never-annotated set) to `@include('widget-shared.inline-prose', [...])` with the correct `tag` / `class` / `key` / `type` / `label`. The Tier-B exempt set (`pricing_chart.gap`, `three_buckets.gap`, `image.max_width`, `image.alt_text`, `board_members.image_aspect_ratio`, `nav.branding_text`, `bar_chart.x_label`, `bar_chart.y_label`) stayed un-annotated throughout — the 304 Tier-B guard test stayed green across every edit.

Widgets widened:

- **DonationForm** — heading (migrated from the dormant 137-era literal `data-config-key` annotation to the `inline-prose` partial).
- **EventsListing** — heading (same migration).
- **BarChart** — heading (new annotation; `x_label` / `y_label` remain Tier-B-excluded).
- **BlogListing** — heading (new annotation; the `content_template` data-driven `{{item.*}}` template remains un-annotated by design).
- **BoardMembers** — heading (new annotation; `image_aspect_ratio` remains Tier-B-excluded).
- **EventCalendar** — heading (new annotation).
- **MapEmbed** — heading (new annotation; `map_input` is `textarea` subtype, not display prose).
- **ProductCarousel** — heading (new annotation; `success_page` is URL config).
- **SocialSharing** — heading (new annotation; documented dual-purpose coupling — the heading also feeds the share-link text on Bluesky / Mastodon / email subject — analogous to the Logo.text/alt coupling the 304 log already accepted; flagged for the user to confirm-or-revert).

The 304 inline-editability test (`InlineEditingFoundationSession304Test`) was updated to reflect the new ground-truth: 13 inline-editable widgets, and only `image` + `nav` explicitly excluded (both for Tier-B couplings the 304 prompt enumerated and which haven't changed). The expanded eligibility test runs as a guard against accidental regression of either set.

For widgets whose outer container is gated on the heading or on data (BarChart returns early without items; ProductCarousel and MapEmbed are gated on configured input; BoardMembers' outer block requires `count($members) > 0 || $heading`), the in-Inspector path remains the way to set an initial heading on an empty widget — once the heading is non-empty (or the data exists), inline editing of the heading lights up on the canvas. The parity test for the two data-gated widgets (BarChart, ProductCarousel) passes trivially because both branches early-return without items; the visible heading round-trip is covered at the Playwright level once the visual-parity work in session 308 lands.

## Deferred decisions

- **SocialSharing heading dual-purpose coupling.** The heading is now inline-editable and also seeds the share-link text on Bluesky / Mastodon / email subject. This is analogous to the Logo.text/alt coupling the 304 log explicitly accepted ("editing the visible node updates alt — consistent, acceptable"). Flagged here for the user to confirm-or-revert.
- **The data-gated parity cases** (BarChart, ProductCarousel) pass trivially because both branches early-return without items. The visible heading round-trip for these widgets needs a Playwright case once the e2e suite is exercised on this surface — picked up naturally by session 308's visual-parity harness, which loads real pages and exercises actual rendered DOM.
- **BoardMembers / data-gated widgets in general** — when the heading is empty and the data is empty, the outer block doesn't render at all on either side. Inline editing of an empty heading isn't possible until either the heading is set (via Inspector) or data exists. Future extension could extend the outer condition to also fire on `$inlineEditing`, but that would introduce a builder-vs-public diff that the parity test's normalisation would have to handle. Not pursued in this session — V1 behaviour is: set initial heading in Inspector, then inline-edit thereafter.
- **The `widget-visual-parity.spec.ts` Playwright spec** drafted as session 308's centrepiece — sits at the rendered-DOM / computed-style layer below the HTML-layer parity guard this session ships. Sessions 296 (served-bundle freshness) + 307 (HTML parity) + 308 (visual / computed-style parity) form the three-layer coverage stack.

## Drift

The base prompt asked for confirmation of the widening list at session start. Rather than block on a multiple-choice question (per `feedback_no_multiple_choice_questions`), the session ran a fresh per-widget read across the 36 non-eligible widgets, surfaced the Tier-A-clear residue as a 9-widget proposal in plain prose, flagged the one judgment call (SocialSharing's heading / share-text coupling), and proceeded. The original parity test used `slotHandle: 'page_main'` for the public branch (mirroring the 304 test's pattern, which works for PricingChart because it has no data contract). That broke for widgets with data contracts (BlogListing, EventsListing, BoardMembers, etc.) because `SlotRegistry::find('page_main')` returns null — `page_main` isn't a registered slot. The 305 conflation split moved editing-only behaviour to `$inlineEditing`, but kept `page_builder_canvas` as the default slot for both branches. The test was updated to use `page_builder_canvas` for both — which is what production actually does (`PageBlockRenderer::render` calls `WidgetRenderer::render($pw)` with no slot arg, defaulting to `page_builder_canvas`).

The EventCalendar `Str::random(8)` per-render id was an unexpected divergence at first surface; added to the parity normaliser's documented diff set with a one-line rationale.

The base-prompt note that 307 inherits "the e2e suite is PARKED" is historical (true at session start). The Playwright suite was un-parked at 2026-05-19 (PRs #7 + #8). Stale memory entries (`project_playwright_ci_cost`, `project_e2e_suite_systemic_scss_cause`) were updated to reflect resolution.
