## Workstream 1 — Template system artifact audit

Review the template system implementation for leftover artifacts from the Site Theme migration:

- **SiteSetting references**: grep for any remaining reads or writes of the retired keys (`public_primary_color`, `public_heading_font`, `public_body_font`, `header_bg_color`, `footer_bg_color`, `nav_link_color`, `nav_hover_color`, `nav_active_color`, `theme_scss`). Exclude session logs and docs. Any live-code reference is a bug.
- **SiteThemePage references**: grep for `SiteThemePage` in live code (exclude session logs). The class was deleted — any remaining import or reference will cause errors.
- **Template fallback correctness**: verify that `Template::resolved()` is used consistently wherever template values are read. Check that pages with `template_id = null` correctly fall back to the default template in the public layout.
- **System page slug collisions**: the custom header/footer feature creates system pages with slugs like `_header_{id}` and `_footer_{id}`. Verify these slugs cannot collide with user-created pages (the underscore prefix exclusion in the public router should handle this — confirm).

---

## Workstream 2 — Widget copy / serialization consistency

Three places now serialize or copy widget stacks:

1. `PageBuilder::serializeWidgetStack()` — Save as Template from the page builder
2. `EditPage::serializeWidgetStack()` — Save Block Layout as Template from the edit page menu
3. `CreatePage::hydrateWidgets()` — Hydrate widgets from a content template definition
4. `ListTemplates::copyWidgets()` / `EditPageTemplate::copyWidgets()` — Copy widgets for "New From Default" and custom header/footer creation

Audit these for:

- **Field coverage**: do all copy/serialize paths handle the same set of PageWidget fields? Specifically check: `label`, `config`, `query_config`, `style_config`, `column_index`, `sort_order`, `is_active`. Flag any path that drops a field.
- **Nested widget handling**: do all paths correctly handle column widget children (recursive copy)?
- **Code duplication**: identify the repeated patterns. If the same logic appears in 3+ places, flag it for extraction into a shared method on the Template or PageWidget model.

---

## Workstream 3 — Edit page field audit (template-related)

For every Filament resource and admin page that now interacts with templates, verify:

- `PageResource` form — `template_id` and `content_template_id` fields: correct options query, correct defaults, correct visibility (create vs edit).
- `TemplateResource` table — columns, actions, tab filtering. Verify edit links route to the correct edit page based on type.
- `EditPageTemplate` form — all color/font fields map to the correct database columns. Verify the form saves correctly for both default and non-default templates.
- `EditContentTemplate` form — name and description save correctly.

## Workstream 4 — Permission gate audit

Verify template-related access control:

| Surface | Expected gate |
|---|---|
| Template list page | `view_any_page` OR `edit_theme_scss` |
| Edit Content Template | Same as list |
| Edit Page Template | Same as list |
| Create Page Template | Same as list |
| SCSS tab (edit page template) | `edit_theme_scss` |
| Header/Footer tabs (edit page template) | `edit_site_chrome` |
| "Save as Template" (page builder) | Inherits page access — no additional gate needed |
| "Save Block Layout as Template" (edit page menu) | Inherits page access — no additional gate needed |

If any gate is missing or incorrect, fix it.
