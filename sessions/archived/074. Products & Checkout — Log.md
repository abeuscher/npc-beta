## What was built

### Phase 1 — Data model and admin UI

- Migrations: `products`, `product_prices`, `purchases`, `waitlist_entries` tables with UUID PKs and appropriate FKs
- Models: `Product`, `ProductPrice`, `Purchase`, `WaitlistEntry` with relations and casts
- `PurchaseObserver`: logs `purchased` and `cancelled` to contact activity log on create/update
- `ProductResource` in Filament (CMS group): name, slug (auto-generated), description, capacity, status, sort_order; relation managers for price tiers, purchases, and waitlist entries
- `products` registered as a system collection handle; `WidgetDataResolver` updated with `resolveProducts()`
- `products_prefix` added to routing settings (later removed — see patch notes)
- `docs/schema.md` updated

### Phase 2 — Stripe price sync

- `ProductPriceObserver`: on save, creates a Stripe Price for paid tiers; archives and recreates on amount change; skips sync for zero-amount tiers. Uses `updateQuietly()` to prevent observer loops.

### Phase 3 — Checkout and waitlist routes

- `ProductCheckoutController`: validates `product_price_id`, checks capacity server-side, creates Stripe Checkout session (paid tiers use `stripe_price_id`; free tiers use inline `price_data` with `unit_amount: 0`), metadata carries `product_price_id`, success/cancel URLs from Referer
- `ProductWaitlistController`: contact lookup/create by email, dedup check, creates `WaitlistEntry`, logs `waitlisted` to activity
- `StripeWebhookController` extended: detects `metadata.product_price_id` in `checkout.session.completed`, routes to `handleProductPurchase()` — idempotency on `stripe_session_id`, contact lookup/create, creates `Purchase`
- Routes registered: `POST /products/checkout` and `POST /products/waitlist` (throttled)

### Phase 4 — Product widget

- `product_display` widget type registered via `WidgetTypeSeeder`
- Blade template: renders product name/description, price tier checkout forms or waitlist form at capacity, confirmation/cancellation messages from query params
- `PageController` updated to inject product data when `config.product_slug` is set on a widget

### Patch 002 — Price tier UI fixes

- Added `CreateAction` header button to `ProductPricesRelationManager` (was missing — no way to add tiers)
- Removed `stripe_price_id` from form and table (internal field, no user value)
- Arranged label/amount/sort_order in single row with 5/5/2 column proportions (12-column grid)

### Patch 003 — Products prefix removal, Stripe link, activity log meta

- Removed `products_prefix` from General Settings UI, `AppServiceProvider`, `DatabaseSeeder`, and routes — product POST routes hardcoded to `/products/*` since products are widget-only with no public page routes
- Blade template action URLs switched to `route()` helpers
- Added `meta` JSON column to `activity_logs` (migration `2026_03_25_000005`)
- `ActivityLogger::log()` updated to accept optional `array $meta` parameter
- `PurchaseObserver` passes `stripe_session_id` in meta when logging `purchased`
- Contact timeline (`ContactNotes` + Blade) updated to carry and render meta; shows "View in Stripe" link (new tab, nofollow) when `stripe_session_id` present; auto-detects test vs live from session ID prefix
- `PurchasesRelationManager` updated: "View in Stripe" action button linking to Stripe dashboard checkout session; hidden when no session ID

## Schema changes

- `products`: uuid PK, name, slug (unique), description, capacity, status, sort_order, timestamps
- `product_prices`: uuid PK, product_id FK (cascade), label, amount decimal(10,2), stripe_price_id nullable, sort_order, timestamps
- `purchases`: uuid PK, product_id FK, product_price_id FK, contact_id FK (nullOnDelete), stripe_session_id nullable, amount_paid decimal(10,2), status, occurred_at, timestamps
- `waitlist_entries`: uuid PK, product_id FK, contact_id FK (nullOnDelete), status, timestamps
- `activity_logs`: added `meta` JSON nullable column
- `docs/schema.md` updated for all of the above

## Production deployment notes

- Run `php artisan migrate` (purchases table and activity_logs meta column)
- Run `php artisan db:seed --class=WidgetTypeSeeder` to register `product_display`
- Run `php artisan db:seed --class=PermissionSeeder` to add product permissions
- Stripe keys must be entered in Finance Settings before testing
- HTTP Basic Auth (`APP_ENV=public-dev`) must be disabled or webhook path exempted for Stripe webhooks to reach the app — Stripe cannot pass HTTP Basic Auth challenges

## Stripe API key permissions required

- Checkout Sessions: Write
- Prices: Write
- Products: Read (auto-created via `product_data`)

## Deferred decisions

- Stripe link on Purchases tab links to checkout session URL; for paid transactions a direct payment intent link would be more useful — deferred, requires storing `stripe_payment_intent_id` on Purchase (noted for help doc session)
- Product name/label changes do not update the Stripe Price label — Stripe Prices are immutable; stale label on Stripe-hosted checkout page is an accepted limitation for now
