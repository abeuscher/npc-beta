## Phases

### Phase 1 — Verify the lockdown holds (the headline)

This is a public, internet-facing auto-login, so confirm the deny-list is real, not just declared. Exercise a `demo`-role user against each walled surface — **settings, user/role management, API keys, the import tools, export, Stripe-touching actions, the scrub-data wipe, Media Finder delete** — and assert it bounces (403 / hidden / unreachable). Land a **standing guard** (Pest preferred; Playwright if the wall is UI-gated) so a future permission change can't silently widen the demo. Decide the test shape at the open gate.

### Phase 2 — Tune the demo-role width

The user wants the interior surface *as wide as safely possible*, and the egress firewall (FM-side) backstops the blast radius. Decide whether to widen beyond the current allow-list — e.g. let demo users create events/donations to show those flows — given outbound is locked closed on the droplet so Stripe/email calls fail closed regardless. Keep it an allow-list; widen deliberately, not by switching to a deny-list.

### Phase 3 — Baseline + reset

- Build a **`demo:reset`** console command that returns the node to a curated, known-good baseline (idempotent; re-runnable on a cron). Lean: a local reseed (no egress), not FM `RestoreFromBlob` — confirm at the open gate.
- Curate the **baseline content** a prospect sees on first entry: the public site + enough sample CRM data to feel real (a `demo` seeder). This is partly a content/design call — surface the shape to the user.

### Phase 4 — Local dress-rehearsal

Run the app under `APP_ENV=demo` locally and walk the full path: public `/demo` → click CTA → `/demo/enter` auto-login → confirm the locked-down panel (denied surfaces really denied), confirm mail goes to `log`, run `demo:reset` and confirm a clean baseline. Restore the `local` env afterward. This is the readiness check before the user provisions the live droplet.
