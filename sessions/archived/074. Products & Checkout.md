# 074. Products & Checkout

Build a lightweight product catalogue with fixed inventory, price tiers, and a waitlist — backed by Stripe Checkout. Products are named entitlements (plots, slots, packages, etc.) offered in finite quantity. No shipping, no SKUs, no physical fulfilment. Payment is handled by Stripe; the app records who bought what.

---

## Design decisions (resolved before starting)

**What a product is:**
A named thing an organisation offers in finite quantity. Has a `slug` used as a unique handle throughout the system. Examples: garden plots, parking spaces, naming rights packages, CSA shares.

**Price tiers:**
One product can have multiple price tiers. Each tier has a label and an amount. All tiers draw from the same inventory pool. A zero-amount tier follows the same Stripe Checkout flow as a paid tier — Stripe supports $0 sessions, and keeping one code path is simpler.

**Purchases:**
A `purchases` table records completed transactions. A purchase is created only on `checkout.session.completed` from the Stripe webhook. No pre-purchase state, no inventory hold during an active Checkout session. Any theoretical double-booking race is acceptable at this scale and handled manually.

**Anonymous checkout:**
No portal authentication is required to purchase or join the waitlist. On `checkout.session.completed`, the webhook reads `customer_details.email` from the Stripe event. If a contact with that email exists, the purchase is attached to them. If not, a new contact is created using the email (and name from `customer_details.name` if available), and the purchase is attached to that new contact.

**Stripe metadata:**
Checkout session metadata carries `product_price_id` only. The webhook uses this plus the Stripe-verified customer email to create the purchase record.

**Waitlist:**
When a product's active purchase count reaches its capacity, the widget shows a waitlist form instead of buy buttons. The form collects email and optional name. The same contact lookup/create logic applies. Waitlist processing is entirely manual.

**Product display:**
Products are displayed via a page widget. The widget is configured with a product slug and receives the product's public data object. No dedicated page route is created for products — they surface only through widgets placed on pages.

**Activity log:**
Purchases are written to the contact's activity log under event `'purchased'`. Waitlist entries under `'waitlisted'`.

**Admin collection:**
Products are registered as a system collection (handle: `products`) so they appear in the Collection Manager under Tools, following the same pattern as events and blog posts.

## Phase 1 — Data model and admin UI

1. Migrations:
   - `products`: `id` (uuid), `name`, `slug` (unique string), `description` (text, nullable), `capacity` (integer), `status` (string: draft/published), `sort_order` (integer, default 0), timestamps
   - `product_prices`: `id` (uuid), `product_id` (FK→products, cascade), `label` (string), `amount` (decimal 10,2), `stripe_price_id` (string, nullable), `sort_order` (integer, default 0), timestamps
   - `purchases`: `id` (uuid), `product_id` (FK→products), `product_price_id` (FK→product_prices), `contact_id` (FK→contacts, nullOnDelete), `stripe_session_id` (string, nullable), `amount_paid` (decimal 10,2), `status` (string: active/cancelled), `occurred_at` (timestamp), timestamps
   - `waitlist_entries`: `id` (uuid), `product_id` (FK→products), `contact_id` (FK→contacts, nullOnDelete), `status` (string: waiting/notified/converted/cancelled), timestamps

2. Create `ProductResource` in Filament: fields: name, slug (auto-generated from name, editable), description, capacity, status, sort_order. Relation managers for: price tiers (inline, sorted by sort_order), purchases (read-only: contact, price tier, amount_paid, occurred_at, status), waitlist (contact, status, created_at).

3. Register `products` as a system collection handle in `Collection` model alongside `events` and `blog_posts`. Add query logic to `WidgetDataResolver`.

4. Add `PurchaseObserver`: on `created`, log `'purchased'` to the contact's activity log. On `cancelled`, log `'cancelled'`.

5. Add `products_prefix` to the routing section of general settings (same pattern as other prefix settings).

6. Update `docs/schema.md`.

## Phase 2 — Stripe price sync

1. Add a `ProductPriceObserver`: on save, if `amount > 0` and `stripe_price_id` is empty, create the Stripe Price and store the returned ID. If the amount changes on an existing price, archive the old Stripe Price and create a new one (Stripe prices are immutable). If `amount = 0`, `stripe_price_id` stays null.
2. Read the existing Stripe integration (`AppServiceProvider`, config) before writing any API calls.

## Phase 3 — Checkout and waitlist action routes

Two action routes only — no product display routes:

1. `POST /{products_prefix}/checkout` — creates a Stripe Checkout session.
   - Takes `product_price_id` from request body.
   - Reads price and product from the database — never from form input.
   - Checks capacity server-side. If full, return error.
   - Metadata: `{ product_price_id: '...' }`
   - `success_url` / `cancel_url`: use the HTTP `Referer` to return the user to the originating page, with `?checkout=success` or `?checkout=cancelled` appended.
   - No authentication required.

2. `POST /{products_prefix}/waitlist` — joins the waitlist.
   - Takes `product_id`, `email`, `name` (optional).
   - Validates product is at or over capacity.
   - Look up contact by email; create if none found.
   - Create `waitlist_entries` row if contact is not already on the list for this product.
   - No authentication required.

3. Update `StripeWebhookController` to handle `checkout.session.completed` for product purchases:
   - Detect via presence of `metadata.product_price_id`.
   - Read `customer_details.email` (and `customer_details.name`) from the event.
   - Look up contact by email; create if none found.
   - Create `Purchase` record: status `active`, `occurred_at` = now, `stripe_session_id` and `amount_paid` from event.

## Phase 4 — Product widget

Insert a new `WidgetType` with handle `product_display`:
- `label`: `Product`
- `render_mode`: `server`
- `config_schema`: select field "Product" populated from published products (value = slug, label = name)
- Blade template renders:
  - Product name and description
  - If capacity available: each price tier with label, amount, and a checkout form (POST to checkout route with `product_price_id`)
  - If at capacity: waitlist form (email + optional name, POST to waitlist route)
  - Reads `?checkout=success` / `?checkout=cancelled` query param to show confirmation or cancellation message

## Security checklist

Before closing:

- [ ] Checkout session created server-side — price is read from the database by ID, never taken from form input.
- [ ] Capacity check performed server-side before Checkout session creation.
- [ ] Contact identity on purchases is derived from Stripe's verified `customer_details.email`, never from user-submitted data.
- [ ] Waitlist entries cannot be duplicated for the same contact/product combination.
- [ ] Stripe webhook signature is verified before processing.
- [ ] Stripe secret key is never exposed in responses or logs.

---

## Out of scope

- Portal authentication for purchases or waitlist
- "Already purchased" detection for anonymous visitors
- Automated waitlist notification and slot-holding
- Product expiry / term-based allocations
- Shipping, fulfilment, or physical inventory tracking
- Admin-initiated purchases without payment
- Stripe product/price archival UI
