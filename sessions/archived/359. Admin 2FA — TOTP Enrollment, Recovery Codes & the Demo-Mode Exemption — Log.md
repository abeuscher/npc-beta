## What was built

Mandatory TOTP two-factor authentication on the Filament admin panel, end to end — executing `release-plan.md` § **A5**. Built on Laravel Fortify's 2FA *primitives* (the encrypted column shape, TOTP provider, recovery-code generation) with the enrollment and challenge flow driven through Filament ourselves; Fortify's own routes/views are suppressed.

### Phase 1 — 2FA backend + schema

- Added `laravel/fortify` (^1.37) + `bacon/bacon-qr-code` (^3.1). Called `Fortify::ignoreRoutes()` in `AppServiceProvider::register()` so no `/user/*` or two-factor web routes/views mount (they would collide with Filament's stock `->login()`); the package's classes stay available. No `config/fortify.php` published — the merged package defaults are correct and routes are killed regardless.
- Migration `2026_06_12_000001_add_two_factor_columns_to_users_table` — `two_factor_secret` (text, nullable), `two_factor_recovery_codes` (text, nullable), `two_factor_confirmed_at` (timestamp, nullable). Run via Docker. First migration since the 348 squash.
- `User` uses `Laravel\Fortify\TwoFactorAuthenticatable`; added a `hasConfirmedTwoFactor()` helper (the gate reads it) and overrode `twoFactorQrCodeUrl()` so the authenticator labels the entry with the configured brand (cascade: `admin_brand_name` site setting → `site.name` → `app.name`) and the user email, rather than the raw `APP_NAME`.
- `docs/schema/users.md` updated for the three new columns.

### Phase 2 — Enrollment + challenge wired into Filament

- `App\Filament\Pages\TwoFactorSetup` (a `SimplePage`): QR code + manual-entry key + recovery codes shown once, confirmed with a live TOTP code before `two_factor_confirmed_at` is set (Fortify's `EnableTwoFactorAuthentication` / `ConfirmTwoFactorAuthentication` actions). Marks the session passed and redirects to the panel.
- `App\Filament\Pages\TwoFactorChallenge` (a `SimplePage`, rate-limited via `WithRateLimiting`): accepts a TOTP code or one single-use recovery code (consumed via `replaceRecoveryCode`), then marks the session passed.
- Both routes registered manually in the panel's `->routes()` closure — `SimplePage` isn't auto-routable — behind `Authenticate` but **outside** the panel `authMiddleware`, so the enforcement gate can never trap them in a redirect loop. Canonical `filament.admin.pages.two-factor-{setup,challenge}` names; blade views under `resources/views/filament/pages/`.

### Phase 3 — Enforcement gate + demo exemption + FM-path guard

- `App\Http\Middleware\EnsureTwoFactorAuthenticated`, added to the panel `authMiddleware` after `Authenticate` (so it runs only on authenticated panel routes): un-enrolled → enrollment; enrolled but session not yet cleared → challenge; otherwise pass. Session flag `two_factor_passed`. The 2FA pages and the logout route are allow-listed.
- **Demo exemption (hard requirement):** `isDemoMode()` (`APP_ENV=demo`) short-circuits the gate entirely, so the single-button `/demo/enter` auto-login stays one click and the demo user accrues no 2FA state.
- **Test-context posture:** the gate bypasses the `testing` environment by default (so the existing `actingAs($admin)` suite stays green); the dedicated 2FA tests opt in via a static `enableInTesting()`, reset in `afterEach`.
- FM `/api/*` path left untouched (it carries no session gate); a test asserts it stays reachable with the gate active.

### Phase 4 — Help doc + tests + suite-green

- Help doc `resources/docs/two-factor-authentication.md` (standalone, category settings) + a pointer added to the Users doc; `help:sync`'d (it runs in `DatabaseSeeder` + both deploy workflows, so prod/test pick it up).
- `tests/Feature/TwoFactorAuthenticationTest.php` — 13 cases: testing-bypass default; gate redirects (un-enrolled → enrollment, enrolled-unchallenged → challenge, cleared → panel); enrollment confirm valid/invalid; QR issuer labelling; challenge TOTP valid/invalid, recovery-code single-use + already-consumed; the demo exemption; the FM-path guard.
- Admin theme rebuilt (`npm run build`) so the new blade utility classes (the recovery-codes box, grid) land in the compiled Filament theme CSS.

## Security considerations

This session *is* a security control, so the review is the work itself. 2FA is uniformly mandatory for all real panel users; the gate sits in the panel `authMiddleware` (covers every entry point — no per-page `canAccess()` leakage). Secret and recovery codes are stored encrypted (Fortify's `Crypt`-based encryption), never as plaintext. Recovery codes are single-use (replaced on consumption). The two exemptions were designed to be non-leaky: the demo bypass keys on `isDemoMode()` (`APP_ENV=demo`), which a real install never matches, so it cannot weaken production 2FA, and the demo user is verified to carry no 2FA secret; the testing bypass keys on `app()->environment('testing')` and is opt-out-by-default, liftable only in-process by the dedicated tests, so it cannot reach production. The FM `/api/*` path is stateless and structurally cannot hit the session gate; a guard test holds it reachable. **Surfaced, not built:** mandatory 2FA widens the admin-lockout surface — a lost device *and* lost recovery codes locks an admin out with no in-app reset (by design); the operator-mediated recovery feature is scoped for session 360 (the session-304 "admin lockout has no recovery" flag, now urgent).

## Production deployment notes

Standard `migrate` runs the new migration. After deploy, **every existing admin is forced through one-time 2FA enrollment on next login** and needs an authenticator app (Google Authenticator, 1Password, Authy, …); the QR labels with the configured brand name. No seeder or data step. Two things to know: (1) there is **no in-app 2FA reset** — a locked-out admin currently requires node CLI (`tinker` to null the `two_factor_*` columns) until the session-360 recovery feature lands; (2) the local-dev and production environments are **not** exempt (only demo and the test suite are), so local admins enroll too.

## Deferred decisions

- **Operator-mediated admin account recovery → session 360.** An FM-control-panel-triggered reset (mTLS node endpoint) + a local break-glass artisan command + an audit trail; identity verified out-of-band against an operator-held external-vault PIN, with no recovery secret stored in the app or FM. Closes the CRM half of the session-304 "admin lockout has no recovery" flag, which this session's mandatory 2FA made urgent. Open scope question carried to 360: 2FA-reset only vs. full credential recovery (also password / super-admin). Both 360 prompts drafted and approved.
