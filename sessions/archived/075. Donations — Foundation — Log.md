**Date:** 2026-03-25

## What was built

### Donations schema & model

- Migration replacing old donations table: dropped campaign/fund/method/reference/anonymous/notes/soft-delete fields; added `type`, `currency`, `frequency`, `status`, `stripe_subscription_id`, `stripe_customer_id`, `started_at`, `ended_at`; re-added `contact_id` FK with nullOnDelete
- `Donation` model rewritten: new fillable, casts, `contact()` BelongsTo, `transactions()` morphMany, SoftDeletes removed
- `DonationObserver`: logs `donated` activity on contact when status transitions to `active`; logs `donation_failed` when status transitions to `past_due`
- `DonationFactory` rewritten to match new schema

### Admin resource

- `DonationResource`: view/audit only — `canCreate`, `canEdit`, `canDelete` all return false
- Table columns: contact name, type badge, amount, frequency, status badge, started_at, Stripe dashboard link
- `ViewDonation` page with breadcrumbs
- `TransactionsRelationManager`: read-only; Stripe link routes `in_` prefixed IDs to `/invoices/`, others to `/checkout/sessions/`

### Donation checkout controller

- `DonationCheckoutController`: validates amount (1–10,000), type, frequency; creates pending Donation; builds Stripe Checkout session in payment or subscription mode (`customer_creation: always`); deletes pending Donation on Stripe error; returns `{url}` JSON

### Webhook handling

- `StripeWebhookController` extended with `invoice.payment_succeeded` and `invoice.payment_failed` event handlers
- `handleCheckoutSessionCompleted` now checks `metadata.donation_id` first, then falls through to product purchase logic
- `handleDonationCheckout`: idempotency on session ID; findOrCreateContact; updates donation (contact_id, stripe_customer_id, stripe_subscription_id, status=active, started_at); creates Transaction
- `handleInvoicePaymentSucceeded`: finds donation by stripe_subscription_id; idempotency; creates Transaction; logs `donation_received` for subsequent billing cycles
- `handleInvoicePaymentFailed`: sets donation to `past_due`; creates failed Transaction; triggers observer
- `findOrCreateContact()` extracted as a shared method used by both product and donation handlers

### Donation form widget

- `donation_form` widget type seeded: heading, preset amounts (CSV), show_monthly toggle, show_annual toggle, success_page slug
- `donation-form.blade.php`: Alpine.js fetch submission; configurable preset amounts defaulting to 10/25/50/100; optional frequency section (hidden if both toggles off); amount and frequency buttons use `outline` class; Donate button is solid/primary; success/cancelled states driven by `?donation=` query param

### Site settings & routing

- `donations_prefix` setting added to General Settings (Routing section); config registered in `AppServiceProvider`
- `POST /{donations_prefix}/checkout` route registered with `throttle:20,1`
- `docs/schema.md` updated; `resources/docs/donations.md` rewritten; `resources/docs/stripe-setup.md` created

### Carry-forward from session 074

- Simplified Stripe dashboard URLs in `PurchasesRelationManager` and `contact-notes.blade.php` (removed test/live prefix detection)

### Session planning

- Session 076 prompt and base prompt written (`sessions/076. Tax Receipts.md`, `sessions/076. base-prompt.md`)
- Pledge Tracking removed from roadmap (belongs in QuickBooks, not the CRM)
- `session-outlines.md` updated: session 075 completed, Tax Receipts stub scoped, Code Housekeeping Notes section added

### Patches

- **patch-001** — `PublicDevAuth` decoupled from `APP_ENV`: HTTP basic auth now triggers solely on presence of `PUBLIC_DEV_USER` / `PUBLIC_DEV_PASS` env vars, allowing production server to reach Stripe without auth wall blocking migrations
- **patch-002** — Optional `success_page` slug on donation form widget: admin enters a page slug; validated `exists:pages,slug` server-side; Stripe success URL redirects to that page; invalid slugs return 422; cancel always returns to the form page

## Security checklist — verified

- [x] Amount validated server-side (min $1, max $10,000) — client value never trusted
- [x] No card data stored — all payment data remains in Stripe
- [x] Webhook signature verified via Stripe SDK before any processing
- [x] Webhook handlers are idempotent — duplicate events do not create duplicate records
- [x] Pending Donation record deleted on Stripe session creation failure — no orphaned rows
- [x] `success_page` validated `exists:pages,slug` — no open-redirect possible
- [x] Donation checkout route is public but throttled (20/min); no authenticated session required to donate
