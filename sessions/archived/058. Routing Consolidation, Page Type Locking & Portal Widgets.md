# 058. Routing Consolidation, Page Type Locking & Portal Widgets

Three related workstreams that clean up loose ends from the portal and CMS foundations, and add the first self-service portal widgets to the page builder.

---

## Architecture decisions (agreed pre-session)

- **Routing prefixes are user-configurable.** Changing a prefix may break external links and event invites — this is acceptable and documented. Redirect handling is a future concern (to be explored as a follow-on when an org actually needs it).
- **Admin path is NOT in this group.** The Filament admin path is configured via `ADMIN_PATH` in `.env` (added session 056). It is not stored in `SiteSettings` because Filament builds its routes at boot time and cannot read DB values. This is a deployment-level concern.
- **Page type is announced prominently.** The edit form header for every page resource must tell the user what kind of page they are editing. Type is locked after creation for `event`, `post`, and `member` pages.
- **Slug prefix is locked for system-typed pages.** A page of type `event`, `post`, or `member` has its slug prefix set by the routing system — the user may only edit the slug segment after the prefix.

---

## Phase 1 — Routing prefix consolidation

Move the `blog_prefix` field out of `CmsSettingsPage` and into a new **Routing** section in `GeneralSettingsPage`, alongside `portal_prefix` (added in session 057) and a new `events_prefix` field.

| Field | Key | Default | Currently in |
|-------|-----|---------|-------------|
| Blog prefix | `blog_prefix` | `news` | CMS Settings |
| Events prefix | `events_prefix` | `events` | `config/site.php` only |
| Member portal prefix | `portal_prefix` | `members` | General Settings (session 057) |

After moving `blog_prefix`: remove it from `CmsSettingsPage`. Update `config/site.php` to read `events_prefix` from `SiteSetting` with the existing config value as fallback (same pattern as `blog_prefix`).

Helper text on the section: "Changing these prefixes will break existing bookmarked links and any external links published elsewhere. Old URLs will return 404 unless redirects are configured."

## Phase 2 — Page type locking and editor labeling

### Slug prefix locking

In `PageResource`, the slug field must show the system-assigned prefix as locked (read-only) text when the page type carries one. The user may only edit the slug segment after the prefix. The prefix must update live if the global prefix setting changes — read from `SiteSetting` at render time, not hardcoded.

Affected types and their prefixes:

| Type | Prefix source |
|------|--------------|
| `event` | `events_prefix` SiteSetting |
| `post` | `blog_prefix` SiteSetting |
| `member` | `portal_prefix` SiteSetting |
| `default` | none — slug is fully editable |

### Type locking on edit

When editing a page, the type select must be read-only (disabled / displayed as plain text) if the type is `event`, `post`, or `member`. Only pages of type `default` can have their type changed after creation.

Show a helper note on locked types explaining why: "Page type cannot be changed after creation for event, post, and member pages."

### Editor header labeling

The edit page heading (Filament's `$title` or a custom heading component) must clearly state what type of page is being edited:

| Type | Heading |
|------|---------|
| `default` | Edit Page |
| `post` | Edit Post |
| `event` | Edit Event Landing Page |
| `member` | Edit Member Page |

This applies to both `EditPage` and `CreatePage` forms (for Create, use "New Post", "New Event Landing Page", etc.).

## Phase 3 — Signup and login widgets

Add two new widget types to the page builder:

### `portal_signup` widget

Renders the member signup form inline on any page. Posts to `portal.signup.post`. On successful submission (or duplicate-account silent pass) the user is redirected to `portal.verification.notice` as normal.

The form must include the same honeypot and timing fields as the standalone `/signup` route.

The widget has no configurable settings — it is a fixed form.

### `portal_login` widget

Renders the member login form inline on any page. Posts to `portal.login.post`. On success, redirects to `portal.account` (or the intended destination via `redirect()->intended()`).

The widget has no configurable settings.

Both widgets should be registered in the `WidgetType` seeder and follow the existing widget registration pattern.
