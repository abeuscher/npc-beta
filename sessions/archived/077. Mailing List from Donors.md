# 077. Mailing List from Donors

Wire up the "Create Mailing List" button on the Donors page to create a real mailing list from the current donor filter set, augmenting the mailing list system as needed to support donation-based criteria.

---

## Design decisions (resolved before starting)

**Open discussion at session start — tax receipt audit trail:**
Before writing any code, discuss with the user the question of re-send prevention and activity logging for tax receipts. The current system allows unlimited sends (each creates a new `donation_receipts` row). Decide: should the admin be warned before re-sending? Should each send be written to the activity log? Agree the policy before implementing anything.

**Mailing list architecture:**
The existing mailing list system uses `MailingListFilter` rows evaluated by `MailingListQueryBuilder` and `MailingListFieldRegistry`. Read all three before designing the approach. The donor criteria (year + minimum total) require aggregate queries against the `donations` table, which the current filter system may not support. Determine whether to: (a) add a new filter field type that supports donation aggregates, or (b) create the mailing list using `raw_where` with a generated SQL fragment. Choose the cleanest option that fits the existing architecture — do not add complexity unless necessary.

**What the created list contains:**
The mailing list is created from the contacts currently visible in the Donors table (respecting the active year and threshold filters). It is a snapshot — a static list definition, not a live query that updates automatically. After creation the admin can edit it in the Mailing Lists resource like any other list.

**List name:**
Default to `Donors — {year} — ${threshold}+` (e.g. `Donors — 2025 — $250+`). Editable in the modal before creating.

---

## Phase 1 — Understand and extend the mailing list filter system

1. Read `MailingListFieldRegistry`, `MailingListQueryBuilder`, and `MailingListFilter` model fully before writing anything.
2. Determine whether donation aggregate filters (year + minimum total) can be expressed as a standard filter row, or whether `raw_where` is the appropriate path.
3. If extending `MailingListFieldRegistry`: add a `donation_total` field with an appropriate operator set (e.g. `>=`, `<=`, `=`) and a `donation_year` field. Update `MailingListQueryBuilder` to handle these fields with the correct aggregate logic.
4. If using `raw_where`: generate the SQL fragment server-side from the year and threshold values — never pass user input raw.

---

## Phase 2 — Wire up the modal and create the list

1. Replace the stub action on the "Create Mailing List" button in `DonorsPage` with a working implementation.
2. The modal form: list name (editable, defaulted), tax year (dropdown, matches Donors page options), minimum total (money field, default $250). Pre-fill from the current page filter state.
3. On submit: create a `MailingList` record and the appropriate filter rows (or `raw_where`) representing the donor criteria. Redirect the admin to the new list's edit page in the Mailing Lists resource.
4. Update help docs for both `tax-receipts.md` (Donors page) and `mailing-lists.md`.

## Security checklist

Before closing:

- [ ] SQL fragment (if using `raw_where`) is generated entirely server-side — no user input passed to raw SQL.
- [ ] Mailing list creation is admin-only — no public route.
- [ ] Created list is scoped to contacts — no cross-model data leakage.

## Out of scope

- Live/dynamic donor lists that re-evaluate on every send (static snapshot only).
- Paging through individual donors in the modal preview.
- Any sending from this session — the list is created, not sent.
