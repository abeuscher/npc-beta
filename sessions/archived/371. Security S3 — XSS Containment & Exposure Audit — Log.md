### Write-path XSS containment

- **The one live bypass â€” event-registration rich-text custom fields.** The event importer and its Filament progress page create `EventRegistration` rows inside `EventRegistration::withoutEvents()` (to suppress the observer's confirmation-email + contact-auto-create), which collaterally suppressed the `SanitisesRichTextCustomFields` `saving()` hook â€” so a rich-text custom field on an event registration imported **unsanitized** (attacker-influenceable via an uploaded import file, when such a custom-field def exists). Fixed by exposing the trait's funnel as a reusable static helper **`EventRegistration::sanitizeRichTextCustomFields()`** and calling it explicitly at both import sites before the `withoutEvents()` create. `app/Models/Concerns/SanitisesRichTextCustomFields.php`, `app/Services/ImportExport/Import/EventImporter.php`, `app/Filament/Pages/ImportEventsProgressPage.php`.
- **Structural weak spot â€” SiteSetting.** `SiteSetting` was the one rich-text model that sanitized only inside its static `set()` method (no mutator, no hook), so `create` / `update` / `firstOrCreate` (the seeder path) bypassed it. Added a `booted()` `saving()` backstop that sanitizes the `RICH_TEXT_KEYS` values on every write path (encrypted-type rows skipped â€” `set()` sanitizes those before encrypting). `app/Models/SiteSetting.php`.

### Export formula-injection

- **`ListExportService` (donations / transactions / memberships â†’ CSV + XLSX)** had zero neutralization â€” a stored value beginning `=`/`+`/`-`/`@` executes when the file is re-opened in Excel/Sheets. Added a single `neutralizeFormula()` helper applied uniformly at the CSV row-build, both XLSX cell loops, and the headers. Technique (owner-confirmed): prefix the offending cell with a leading apostrophe (the spreadsheet text-escape, stripped on display). **Refinement:** plain numeric values (`is_numeric`, e.g. a negative amount `-50.00`) are left untouched so they stay numeric â€” a number can never be a formula; only genuine formula payloads (`-2+3+cmd`, `=â€¦`, `@â€¦`) are escaped. JSON export is out of scope (not a spreadsheet re-open surface). `app/Services/ListExportService.php`.

### Public-exposure gate (#32c, Path A)

The audit reframed this: **no sensitive CRM data can reach a public page today** â€” the donor/note/membership widgets are admin-record-sidebar-only, permission-gated, and fail closed to empty on a public/member canvas; contact addresses have no public widget path at all; CRM entities are architecturally excluded from the collection system. So #32c became one hardening + warnings + docs, not a per-field lock:

- **Closed the one write-time back door.** The page-builder add-widget API (`PageBuilderApiController::store`) validated only `widget_type_id exists`, unlike the dashboard/record-detail builders which re-check `allowedSlots`. A crafted POST could attach an admin/record-context widget to a public page (it would render blank, but the door should be shut). Added the `allowedSlots` server-side check â†’ 422. `app/Http/Controllers/Admin/PageBuilderApiController.php`.
- **Warning gates (not hard blocks) on the human-judgment publish surfaces.** Reactive warning on page `status` â†’ `published` (shared `CmsFormFields::statusField`, so it propagates to all CMS resources; suppressed for portal-gated `member`/`system` pages) and on the collection `is_public` toggle (escalates the existing warning copy to a visible warning when enabled). `app/Forms/Fieldsets/CmsFormFields.php`, `app/Filament/Resources/CollectionResource.php`.

## Security considerations

This session *was* the security review â€” the whole point was to enumerate and close the first XSS layer's gaps and the accidental-exposure surface. Reviewed: every rich-text write path against the model-mutator convention (importers, seeders, the admin JSON API, bulk operations); the list-export family for formula injection; the member-page render path for cross-member data reads; and every "make public" surface. Fixed the two funnel gaps (event-registration `withoutEvents`, SiteSetting non-`set()`), neutralized the export surface, closed the widget-add slot hole, and added the publish warnings. The sanitizer itself was not re-implemented (rated solid at 368; the audit targeted the paths *into* it, per the integration-seams-first rule).

**Residuals surfaced and consciously accepted by the owner** (on the Gate-3 findings register â€” none a live attacker path): (1) help-article content rendered unsanitized, but populated from repo markdown via the HelpSync command (developer-authored, not a user write path); (2) page `head_snippet`/`body_snippet` raw by design (they exist to hold `<script>`, e.g. analytics; admin-only); (3) email-template scalar `{{tokens}}` substituted without escaping â€” traced in full at the owner's request: no token carries a contact custom field (each mailable builds a fixed hardcoded token list; the two HTML-block tokens `{{submission}}`/`{{donations}}` `e()`-escape user values), and the scalar tokens are plain-text name/title fields going into an email (clients don't execute script), so the custom-HTML-field â†’ email-body path the owner probed is closed on two independent grounds; (4) WidgetPreset has no independent sanitizer, but its config is derived server-side from already-sanitized widget config and re-sanitized on apply.

The CSP backstop from S1 (public enforced, admin Report-Only) makes S3's residual-risk statement honest but did not lower the bar â€” the audit treated the admin panel as if the CSP weren't there, since admin CSP is not yet enforcing.
