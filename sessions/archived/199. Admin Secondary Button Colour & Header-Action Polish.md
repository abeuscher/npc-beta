# 199. Admin Secondary Button Colour & Header-Action Polish

Introduce an admin-panel secondary colour (site setting + Filament palette registration) so white-bg/black-outline header actions across the admin get a consistent tinted style. Relocate the `view_transactions` header action on Contact and Product edit pages into the ellipsis overflow. Optional admin-form style tweaks at the end of the session.

---

## Design decisions (resolved before starting)

- **Admin-only scope.** The public site has its own colour system (themes + templates). This session does not touch it. The new setting sits alongside `admin_primary_color` under General Settings → Admin section.
- **Site-settings key:** `admin_secondary_color`. Default value: `#73bbbb`. Read/write pattern mirrors `admin_primary_color` exactly.
- **Filament palette registration.** In `AdminPanelProvider::panel()`, register the new value as a named colour under a new palette key (working name: `secondary`) via `Color::hex($secondaryColor)`, alongside the existing `primary` entry. Buttons then opt in via `->color('secondary')`.
- **Header-action swap.** Every admin `Actions\Action::make(...)` currently using `->color('gray')` as its outer-chrome style is a candidate. This session takes a **deliberate audit pass** — each match gets a yes/no based on whether it is a primary affordance (move to `secondary`) or a true grey/muted utility (leave as `gray`). Do not mass-replace without reviewing.
- **Action relocation pattern.** "View transactions" on Contact and Product edit pages moves from the main header action list into the ellipsis/overflow menu next to Delete. Same label, same URL, same permission gate — only the parent container changes. Apply to *both* the read-only and the editable header action arrays on `EditContact` (identical relocation in both methods).
- **No new `secondary` button component.** This is purely Filament action colour plumbing; no new Blade component, no class-name invention.
- **Admin form tweaks are discretionary.** The session prompt may gain a Phase 3 list of specific form-style adjustments once the user is in front of the UI. Keep the phase small — pixel polish, not layout surgery.

## Phase 1 — Secondary colour setting

### Settings page

- Add a new `Forms\Components\ColorPicker::make('admin_secondary_color')` to the `Admin` section of `GeneralSettingsPage.php`, directly below the existing primary-colour picker.
  - Label: `Secondary colour`
  - Helper text: `A muted accent for secondary header actions in the admin panel. Default: #73bbbb`
- Extend `mount()` to seed `admin_secondary_color` from `SiteSetting::get('admin_secondary_color', '#73bbbb')`.
- Extend the save action to write `SiteSetting::set('admin_secondary_color', $data['admin_secondary_color'] ?? '#73bbbb')`.

### Panel provider

- In `AdminPanelProvider::panel()`, load `$secondaryColor = SiteSetting::get('admin_secondary_color', '#73bbbb')` inside the same `try/catch` as the primary.
- Register it in `->colors([...])` alongside the primary:
  ```php
  ->colors([
      'primary'   => Color::hex($primaryColor),
      'secondary' => Color::hex($secondaryColor),
  ])
  ```
- In `app/Providers/AppServiceProvider.php`, share `site.admin_secondary_color` alongside `site.admin_primary_color` so any admin Blade that needs the raw hex has it.

### No migration

Site settings are JSON-keyed — no schema change needed.

## Phase 2 — Action audit + swap

### Relocate `view_transactions`

- **`EditContact.php`** — in both `getReadOnlyHeaderActions()` and `getHeaderActions()`, remove the standalone `view_transactions` action and add it to an `Actions\ActionGroup` (ellipsis overflow) that sits after the Delete action. If `Delete` is currently implicit via Filament's default header actions, wrap it in an explicit overflow group. Confirm the ordering: primary action(s), Delete, then ellipsis group containing `view_transactions` and any other overflow items.
- **`EditProduct.php`** — same relocation.
- Preserve the permission gate (`view_any_transaction`), label (`View transactions →`), and URL (the `tableFilters[contact_id]` / `tableFilters[product_id]` query string).

### Audit `->color('gray')` on admin header actions

Run a grep for `->color('gray')` under `app/Filament/`. For each header-action match, decide:

- **Swap to `->color('secondary')`**: this is a secondary affordance the user is expected to notice but not act on first (e.g. "View timeline", "Download JSON", "Back to contact", "Show activity log" — the white-bg buttons the user called out).
- **Keep as `->color('gray')`**: genuinely muted utility (filter-toggle ellipsis groups, read-only metadata links, disabled-looking auxiliary toggles).

Build a short decision list in the session log of which actions moved and which stayed. Prefer leaving things alone over a blanket swap.

### Known swap candidates (from session 198 investigation)

Confirm and include in the audit:

- `ContactResource/Pages/EditContact.php` → `notes` / "Timeline" action.
- `ContactResource/Pages/ContactNotes.php` → `back_to_contact` action (the `← Back to contact` grey button).
- `FormResource/Pages/EditForm.php` → `download_json` action.
- `FormResource.php` (table row action) → `download_json` — **leave as `gray`** (table rows are not header-primary affordances).
- Other resource edit pages with a `gray` header action (grep will list them).
