## Design decisions (resolved before starting)

- **Audit with small-fix in place.** Clearly safe fixes ‚Äî dead imports, stale references, confirmed dead code, doc drift, obvious trivial duplications, single-line framework-alignment wins (e.g., `preserveKeys = true` on a `JsonResource`) ‚Äî are applied during the audit branch. Anything that needs strategic input (architectural calls, cross-cutting refactors, retire-or-repurpose decisions) is captured on the W7 / W8 tables and left for session 272.
- **Flag large issues explicitly.** If the audit surfaces something bigger than a W7 / W8 entry ‚Äî a subsystem-level divergence that needs real strategy ‚Äî capture it as an Open Flag with enough context (risk, scope, proposed paths) and surface it to the user before continuing the sweep. Do not decide architecture during the audit.
- **W11 + W12 are mechanical.** File-length and inline-code-in-markup are quantitative grep-and-count workstreams. The output is a list, not a judgment. The apply session walks the lists with the user and picks targets.
- **No subsystem deep walks.** Page Builder Vue, the importer, Organizations, Fleet Manager API, and rich-text surfaces are **deferred to 271**. If a foundational workstream (W1 / W5 / W6 / W9) surfaces a subsystem-shaped finding, route it to a 271 pre-loaded-finding rather than chasing it here.
- **Pre-loaded findings: none.** This is the first audit session in the cycle. 271's pre-loaded-findings block fills at this session's close; 272's open flags + table picks fill at 271's close.
- **Migration squash is not in this session.** The squash is session 273; it follows the 181 / 208 procedure verbatim. Different risk profile (schema vs code), distinct checklist.
- **Bug fixes are out of scope.** Catching a real bug during the audit triggers an Open Flag, not a fix ‚Äî unless the fix is one line and obviously correct.
- **Long runtime expected.** Scope is wide. Split into multiple iteration branches (`session-270/1`, `session-270/2`, ‚Ä¶) if the audit cannot complete in one sitting. Each iteration ships a self-contained slice plus the safe fixes found in it.

## Workstream 12 ‚Äî Inline-code-in-markup grep *(new)*

Pattern lineage: session 142 (Inline JS&CSS Extraction & Migration Squash) and session 181 (Migration Squash & Inline JS&CSS Audit). The standing rule says inline `<script>` / `<style>` blocks in blades should be extracted into named asset files referenced via `vite.config.js` or widget definition `assets`. This workstream verifies the rule still holds and surfaces any drift.

**Scope:**

- **Inline `<script>` blocks** in blade templates, with a configurable trivial-line threshold (default: ‚â•5 lines triggers a finding; one-liners that genuinely belong inline ‚Äî `@vite` directives, JSON `<script type="application/json">` data blocks for blade‚ÜíJS handoff, etc. ‚Äî get carve-outs).
- **Inline `<style>` blocks** in blade templates, same threshold.
- **Inline JS event handlers** (`onclick="‚Ä¶"`, `onchange="‚Ä¶"`, `onsubmit="‚Ä¶"`) when they contain non-trivial logic. Single-line `onclick="this.form.submit()"` ‚Äî fine. Multi-statement handlers ‚Äî flag.
- **Multi-line CSS strings** in PHP / TS that build up a `style` attribute through string concatenation. The composed-style pipeline (`AppearanceStyleComposer`) is the legitimate path ‚Äî flag any *bypass* of it.
- **Multi-line HTML strings** in PHP that should be a blade partial.
- **Widget-template inline assets** ‚Äî `app/Widgets/*/template.blade.php` files with non-trivial inline `<script>` or `<style>` blocks that should live in the widget's `js` / `css` asset declaration (per the sovereign-widget pattern).

**Output format:**

| # | File:lines | Block size (LOC) | Block kind | Suggested target |
|---|---|---|---|---|
| 1 | `resources/views/filament/pages/foo.blade.php:42-78` | 36 | `<script>` | `resources/js/admin/foo-page.js` (new file, register in `vite.config.js`) |
| 2 | `app/Widgets/Carousel/template.blade.php:15-45` | 30 | `<script>` | `app/Widgets/Carousel/script.js` + assets declaration on `CarouselDefinition` |

**No fixes during this workstream.** Targets land in 272 (apply); some may be picked up opportunistically by 273 (squash optimization phase).
