## Process rules for every session

Base-prompt process rules apply in full. Session-specific notes:

- **This is a security feature — protect server-side, not in the UI.** The lock must be enforced in `PagePolicy::update()` **and** on the page-builder save routes (the real edit surface). The Filament toggle is UX for *setting* the value; it is not the protection. Per the security-over-usability posture, friction belongs in the UX layer, never the protection layer.
- **`locked` is a concrete boolean, orthogonal to `status`.** Defaults to `false`; a locked page stays `status='published'` (so `published()` visibility is unaffected) and is displayed as "Published & Locked." Do **not** make "locked" a status value.
- **Do not overload `type = 'system'`.** That protects infrastructure pages by *type* in the Filament UI (slug lock, delete hidden). A content page locked from a *role* is a distinct concern — keep them separate.
- **Run the migration via Docker** and **update `docs/schema/pages.md`** in the same pass.
- **`demo:reset` re-applies the lock every reset** so it survives the daily wipe — verify the re-apply is idempotent and guarded on `isDemoMode()`.
