### Phase 2 — `LogsController` + `logs` disk

- New `logs` disk registered in `config/filesystems.php` — local driver rooted at `storage_path('logs')`. Tests use `Storage::fake('logs')` with automatic cleanup via Laravel's Storage idiom. Did **not** reuse `Storage::disk('local')` (which roots at `storage/app/private`, not `storage/logs`).
- New controller at `app/Http/Controllers/Api/Fleet/LogsController.php`. Single `index(Request $request)` action.
  - Range parameter validation: `?lines=N` must be a positive integer expressible as `^\d+$`; non-positive, non-numeric, fractional, or negative values return **422** with `{"error":"invalid_lines","message":"..."}`. Values above 10000 clip silently to 10000.
  - File-existence check returns **404** with `{"error":"log_not_found","message":"log file does not exist"}`.
  - **Backward-read tail implementation.** Opens file, seeks to EOF, strips a single trailing newline so `explode()` doesn't create a phantom empty line. Walks backward in 8 KB chunks, accumulating buffer + remainder across chunk boundaries. When a chunk read contains zero `\n` characters (i.e., a single line longer than 8 KB), the implementation keeps walking backward and growing the accumulator until a newline is found or BOF is reached — guards the long-line edge case.
  - **Byte cap (~850 KB raw)** on accumulated line content; under JSON encoding overhead this stays under the 1 MB encoded body ceiling. When either cap fires, **older** lines are dropped and `lines_truncated` is set to `true`.
  - Filesystem exceptions are caught and returned as **500** with the exception class name only (matching `HealthController` discipline) — no internal paths, no stack traces.

### CI memory-limit fix (separate stream of work, mid-session)

- User dropped `sessions/test-failures.txt` containing CI output: 86 failed / 1707 passed, two specific failures shown both citing `UniqueConstraintViolationException` on `site_settings_key_unique` for `blog_prefix`.
- Diagnosis traced the 86 failures to a single root cause: **PHP `memory_limit = 256M`** in `docker/php/local.ini`. The full Pest suite (1800+ tests in one process accumulating Filament + Spatie + Pest object overhead) OOMs at 256M mid-run. When PHP fatal-errors mid-test, the running test's `RefreshDatabase` transaction can't tear down cleanly, and subsequent tests that try to `INSERT INTO site_settings` for `blog_prefix` (many tests share that beforeEach pattern) cascade with unique-constraint violations. The two traces in the file were samples of the cascade, not the root cause.
- Why the fast suite passed locally on session-251/1 but CI failed: I ran `vendor/bin/pest --exclude-group=slow` with `-d memory_limit=2G` (Pest honours the flag when invoked directly). CI runs `php artisan test` which spawns Pest as a subprocess and does **not** propagate `-d` flags — so CI gets 256M even when the parent sets the override.
- Fix: `docker/php/local.ini` `memory_limit = 256M` → `memory_limit = 1G`. Both containers (`app`, `worker`) recreated to pick up the bind-mount change. Confirmed with `php -r 'echo ini_get("memory_limit");'` returning `1G`. Re-ran the previously-cascading subset (`BlogPagerEndToEndTest`, `PostTest`, `TemplateTest`, `BuildServerSettingsSession125Test`) under `php artisan test` — 31/31 pass.
- Fix lands on session-251/1 alongside the log-fetch endpoint work. The 1G limit covers local dev too — when someone runs the full suite outside CI, they no longer hit OOM. 256M was thin for a Filament + Spatie + Pest app of this size in 2026; the bump is appropriate, not just a CI-specific patch.
