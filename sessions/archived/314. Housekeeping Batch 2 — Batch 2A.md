# 314. Housekeeping Batch 2 — Batch 2A

First of two expected batches absorbing the public-website-blocking subset of `sessions/housekeeping-inbox.md`. Walks the inbox at session start, picks the Batch 2A subset, applies iteration-sliced.

## Plan reference

This session executes `sessions/release-plan.md` § E12 (execution-order position 42). The plan entry is canonical for scope and milestone-block positioning. This prompt is the session-shape delta — the plan entry doesn't enumerate inbox items because the inbox accretes between sessions; the walkthrough at session start does the picking.

## Inbox walkthrough (session start)

Open `sessions/housekeeping-inbox.md` with the user. For each item under "## Inbox":

- **`[post-milestone]`-tagged** → skip (deferred past the Public Website Complete milestone).
- **Small + standalone** → candidate for Batch 2A.
- **Pairs with another inbox item by file or surface** → bundle the pair into one iteration; both candidates for 2A together or both deferred to 2B together.
- **Larger than a single iteration** → defer to Batch 2B or flag for promotion to its own session if the size has grown.

Output of the walkthrough: an ordered list of N iterations for Batch 2A (each one inbox item, or a small bundled pair), and a leftover list that carries to Batch 2B.

---

## Iteration discipline

- **One iteration per inbox item** by default. Each iteration is its own commit on `session-314/N`. Bundle only when items share a file or surface; the bundling rationale lands in the commit message.
- **Six-iteration ceiling.** Past 6 iterations the next iteration is the carve-out signal — stop and let leftovers move to Batch 2B regardless of how much remains in the inbox.
- **Tests follow the item.** Small bug-fixes that don't change observable behavior at a tested seam rely on the existing suite; behavior-changing items add coverage per the standard test-for-new-behavior rule.
- **Commit message names the inbox item.** Use the inbox bullet's leading clause as the commit subject so a future audit can trace the close to the originating item.

---

## Open questions to resolve at session start

- **Subset size.** How many items does the user want in Batch 2A? Default leaning: as many as fit cleanly under the six-iteration ceiling, leaving budget for the leftover items in 2B.
- **Bundling rules.** Are there pairs of items the user wants explicitly bundled (e.g. the text-toolbar `title` attributes paired with the columns-alignment labels because both touch operator-facing affordances)? Or kept strictly separate? Default: bundle only when files / surfaces overlap.
- **Organization importer helper.** Sits in the inbox as a "low-risk, scope-fenced" add. Goes in Batch 2A if there's iteration budget; otherwise 2B.

---

## Out of scope

- Items tagged `[post-milestone]` in the inbox.
- Items the walkthrough flags as having grown past housekeeping size — those get pulled out to be promoted to their own session via a release-plan entry, not absorbed here.
- Anything outside the inbox. New items surfaced mid-session go into the inbox, not into 314.

---

## Testing

- **Fast Pest after every iteration.** Each iteration should leave the suite green.
- **New tests:** per the standard rule — behavior changes get coverage, byte-equivalent or trivial fixes rely on the existing suite. Commit message notes which.
- **Playwright:** isolated stack via `npm run test:e2e` only if an iteration touches a frontend surface that has Playwright coverage. CI runs the full Playwright set on every push regardless.
