# 117. Permissions Framework & Coverage Audit

Establish a principled permission framework, then audit and fix every admin-accessible feature for correct authorization coverage. Builds on the findings from session 116 (framework alignment audit) and session 114 (permissions audit).

## Design decisions (resolved before starting)

- The guiding principles below are the decision-making lens for every audit finding. When in doubt, apply the principles rather than inventing ad-hoc solutions.
- The `developer` role is added this session. All other existing roles (super_admin, cms_editor, crm_editor, event_manager, volunteer_coordinator, treasurer, blogger, demo) are retained.
- `super_admin` remains the only role with the Gate::before bypass. Every other role — including `developer` — is governed by explicit permissions.

---

## Guiding principles

These are the standing rules for how permissions work in this app. They apply to this session and all future sessions that touch authorization.

1. **Total coverage.** Every action with CRUD consequences on application data must be gateable. No admin feature should be accessible without an authorization check.
2. **Useful granularity.** Permissions must be fine enough to serve standard nonprofit staff roles, but no finer. The ability to assign multiple roles and create custom roles provides flexibility — the system does not need to anticipate every possible org chart.
3. **Two tiers of permission.**
   - **Tier 1 — Resource CRUD.** The standard `view_any / view / create / update / delete` grid, one row per resource. Every model-backed Filament resource gets these. They appear in the permission grid on the role editor.
   - **Tier 2 — Standalone capabilities.** Cross-cutting operations, elevated-risk features, or sub-features within a resource that warrant independent control. These appear as checkboxes in the Advanced section of the role editor. A feature gets a standalone permission when it (a) doesn't map to a single model's CRUD lifecycle, (b) carries elevated risk relative to its parent resource, or (c) controls a sub-feature that not all resource users should access.
4. **Roles are buckets.** Each seeded role represents a real job function. When a new permission is created, it is assigned to the role(s) whose job function would expect to use it. New roles are created only when no existing role fits.
5. **No `isSuperAdmin()` as a permission substitute.** Super-admin-only gates are reserved for true system administration (role management, the Gate::before bypass itself). Every other feature should use a Tier 1 or Tier 2 permission so it can be delegated without granting super_admin.
6. **Policies are the single source of truth.** Filament resources should rely on their registered policy for authorization. Inline `canViewAny()` / `canCreate()` / `canEdit()` / `canDelete()` overrides that duplicate what the policy already checks should be removed. Overrides are justified only when the resource adds business logic the policy cannot express (e.g., checking related records before delete, using a different model than the policy covers).

## Phase 2 — Developer role creation

Add a `developer` role to `PermissionSeeder` with these permissions:

- **Full CMS access:** all page, post, collection, collection_item, navigation_menu, form, template, widget_type permissions
- **Full theme/code access:** edit_theme_scss, edit_site_chrome, edit_page_snippets
- **Full settings access:** manage_routing_prefixes, manage_financial_settings (read QuickBooks/Stripe config, not just finance data)
- **Import access:** import_data, review_imports
- **Read-only CRM:** view_any/view for contact, organization, membership, note, tag, event, mailing_list
- **Read-only Finance:** view_any/view for donation, transaction, fund, campaign
- **Form submissions:** view_any_form_submission, view_form_submission, delete_form_submission
- **Advanced features:** use_advanced_list_filters
- **No user management.** No `user` permissions. Cannot create, edit, or delete admin users or roles.

Present the role for confirmation before seeding.

## Phase 3 — Convert `isSuperAdmin()` gates to permissions

For each resource or page currently gated by `isSuperAdmin()`, decide whether to:

- **Convert** to a Tier 1 or Tier 2 permission (most cases)
- **Keep** as super_admin-only (only for true system administration)

Known candidates from session 116:

| Resource / Page | Current gate | Proposed action |

| MembershipTierResource | `isSuperAdmin()` | Convert to standalone `manage_membership_tiers` permission |
| CmsSettingsPage | `isSuperAdmin()` | Convert to standalone `manage_cms_settings` permission |
| MailSettingsPage | `isSuperAdmin()` | Convert to standalone `manage_mail_settings` permission |

Present each conversion for confirmation. Then implement: create permissions, update policies or `canAccess()` methods, update seeder with role assignments.

## Phase 4 — Remove redundant resource overrides

From session 116 findings: 16 resources have `canViewAny()` overrides that duplicate their registered policy. Per Principle 6, remove these redundant overrides. Keep overrides only where justified:

- **Keep:** PostResource (checks `view_any_post`, policy would check `view_any_page`), MemberResource (checks `view_any_member`, policy would check `view_any_contact`), resources with no policy
- **Remove:** ContactResource, EventResource, PageResource, ProductResource, FundResource, FormResource, UserResource, WidgetTypeResource, TransactionResource, DonationResource, NoteResource, OrganizationResource, CampaignResource, MembershipResource, MailingListResource, TagResource, ContentCollectionResource

Also remove redundant `.hidden()` / `.visible()` calls identified in session 116 findings 4–9.
