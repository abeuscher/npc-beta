## What was built

### Phase 1 — Read-through, audit, sign-off

- Catalogued the per-importer floor (existing `guessDestination()` match-statement entries) across the five namespaced importers (Events / Donations / Memberships / Invoice Details / Notes). Confirmed the floor count per page and observed the existing match-statements were preset-blind (no `presetFromSourceName` consultation in any page's `guessDestination`).
- Confirmed the integration seam at [`InteractsWithImportWizard::processUploadedFileNamespaced()`](app/Filament/Pages/Concerns/InteractsWithImportWizard.php#L766) — single `$this->guessDestination($normalized)` call, with `FieldMapper::isSkipped` and `FieldMapper::presetFromSourceName` already wired adjacent. The lift kept the seam, changed the implementation.
- **Architectural recommendation surfaced and accepted:** option (b) — per-importer mapper services parallel to the existing `FieldMapper` for Contacts. Rationale: mirrors the existing pattern symmetrically (`FieldMapper.php` is the template; 5 new files at `app/Services/Import/<Type>FieldMapper.php`); each importer's mapping confined to a single class; `guessDestination()` becomes a 3-line delegation.
- **Preset coverage scope accepted:** `generic` everywhere (universal floor); `wild_apricot` and `bloomerang` populated where the source actually exports the record type. Per-importer defaults: events / donations / memberships — populate WA + Bloomerang; invoice details — WA populated, Bloomerang stays empty (Bloomerang does not expose line-item invoice detail in the same shape); notes — populate both.
- **`SOURCE_SKIPPED_HEADERS` parity:** do not pre-populate. Add only when a known source-system export carries a sensitive header for that record type.
- **Sentinel-pattern parity:** flagged as out-of-scope, lifted to Out-of-Gate register at close (see § Phase 5b).

### Phase 3 — Per-importer mapper service skeletons

Five new service classes created at `app/Services/Import/`, each mirroring `FieldMapper.php`'s shape (`map()` / `presets()` / `presetMap()` / `genericMap()` / `wildApricotMap()` / `bloomerangMap()`):

- [`EventFieldMapper.php`](app/Services/Import/EventFieldMapper.php)
- [`DonationFieldMapper.php`](app/Services/Import/DonationFieldMapper.php)
- [`MembershipFieldMapper.php`](app/Services/Import/MembershipFieldMapper.php)
- [`InvoiceFieldMapper.php`](app/Services/Import/InvoiceFieldMapper.php)
- [`NoteFieldMapper.php`](app/Services/Import/NoteFieldMapper.php)

Each `map()` accepts `?string $preset = null` (defaults to `generic` when null) and returns the namespaced destination key (`event:title`, `contact:email`, `__note_contact__`, etc.) or null. Output is identical-shape to the per-page match-statement returns.

Wiring changes:
- [`InteractsWithImportWizard.php:766`](app/Filament/Pages/Concerns/InteractsWithImportWizard.php#L766) — `$guess = $this->guessDestination($normalized)` → `$this->guessDestination($normalized, $sourcePreset)`. The trait's `$sourcePreset` (already computed via `FieldMapper::presetFromSourceName`) now reaches the mapper.
- Each of the five page classes' `guessDestination()` signature: `(string $normalizedHeader): ?string` → `(string $normalizedHeader, ?string $preset = null): ?string`.
- Each page's match-statement body: replaced with a 1-line delegation `return (new <Type>FieldMapper())->map($normalizedHeader, $preset);`.

Floor migration: every header→destination pair the existing match-statements recognized lands in the new `genericMap()` for each importer. 30 new test cases (6 per importer × 5 importers) cover the floor under `generic` preset, plus null-preset fallback, normalization, and presets-list assertions. Floor preserved.

### Phase 5c — Pre-existing bug fix (surfaced during user testing)

User hit a `TypeError: fgetcsv(): Argument #1 ($stream) must be of type resource, string given` at [`ImportInvoiceDetailsProgressPage.php:369`](app/Filament/Pages/ImportInvoiceDetailsProgressPage.php#L369) during invoice-details import testing with the corrupt fixture + auto-create-CFs on. Investigation confirmed pre-existing variable-scope bug, not introduced by 259's changes.

**Root cause:** `runCommit()` opens a file handle at line 360 (`$handle = fopen($fullPath, 'r')`), then iterates rows via `while (($row = fgetcsv($handle)) !== false)`. Inside that loop, a `foreach (... as $handle => $val)` at line 404 was using `$handle` as the foreach key, **shadowing the outer file resource at the same lexical scope.** After the first row that produced custom fields, `$handle` became a string (the last CF-handle key); the next iteration's `fgetcsv($handle)` blew up. Bug only surfaced when a row produced custom fields AND there were more rows after — which is exactly the auto-create-CFs + multi-row scenario.

**Fix:** rename the inner foreach key from `$handle` to `$cfHandle` ([line 404](app/Filament/Pages/ImportInvoiceDetailsProgressPage.php#L404)). Three lines.

**Why other progress pages are fine:** they use `$handle` for CF keys at function-level scope (inside `processOneRow` or similar), separate from the file-handle scope. Only `ImportInvoiceDetailsProgressPage::runCommit()` had both variables in the same lexical scope. The dry-run path in this same file is also safe because its foreach is inside `processOneRow`.

**Regression test** at [`ImportSession194Test.php`](tests/Feature/ImportSession194Test.php) — imports 2 invoice rows with a `Project` column mapped via `__custom_invoice__` + a `custom_field_map` entry. Asserts `errorCount === 0` and `processed === 2`. Without the fix, the test fails with the TypeError.
