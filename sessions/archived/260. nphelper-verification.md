## 2. UFW security hardening (drop Docker API ports, allow web traffic, fix SSH LIMIT)

Paste this block as one unit:

```bash
ss -tlnp | grep -E ':237[56] ' || echo "(nothing listening on 2375/2376 — good)"
ufw delete allow 2375/tcp
ufw delete allow 2376/tcp
ufw allow 80/tcp
ufw allow 443/tcp
ufw reload
ufw status numbered
```

Look at the numbered output. Find the line that says `22/tcp                     LIMIT IN    Anywhere` and note its `[N]` number. Then:

```bash
ufw delete <N>
```

It will print the rule and prompt `Proceed with operation (y|n)?` — answer `y`.

Then:

```bash
ufw allow 22/tcp
ufw reload
ufw status verbose
```

**Expected:** `22/tcp` shows `ALLOW IN` (not `LIMIT IN`) for both v4 and v6, plus `80/tcp` and `443/tcp` both `ALLOW IN`.

## 10. (Back on the droplet SSH session) — manual deploy with `--wait`

This is the load-bearing step. The whole session is set up to make this command succeed first-attempt.

```bash
cd /opt/nonprofitcrm
docker compose -f docker-compose.prod.yml pull
```

**Expected:** Pulls 4 images from GHCR (app, web, postgres, redis). Takes 1-3 minutes. Should end without errors.

Then **the actual race-fix moment of truth**:

```bash
docker compose -f docker-compose.prod.yml up -d --wait --remove-orphans
```

**Expected output (this is the new behavior):**
- Lines like `Container nonprofitcrm_postgres  Creating` / `Started` / `Waiting` / `Healthy`
- `--wait` blocks for ~10-30 seconds (until postgres reports healthy on first init)
- Eventually: all containers report `Started` or `Healthy`, command exits 0

**The bug it's fixing was:** without `--wait`, this returned in ~2 seconds with postgres still init'ing, and the next command (`migrate`) ran against a not-yet-ready postgres and silently no-op'd.

Now run the migration — **first attempt, no `migrate:fresh` workaround**:

```bash
docker compose -f docker-compose.prod.yml exec -T app php artisan migrate --force
```

**Expected (the success criterion):** A list of migration names being executed, e.g.:

```
Migration table created successfully.
Migrating: 0001_01_01_000000_create_users_table
Migrated:  0001_01_01_000000_create_users_table (X.XX ms)
Migrating: 0001_01_01_000001_create_cache_table
Migrated:  0001_01_01_000001_create_cache_table
... [many lines like this]
```

**The bug looked like:** `migrate --force` exited 0 with no migrations listed, and then `migrate:status` afterward said "Migration table not found." If you see anything like that, **the fix didn't work — stop and paste the output back.**
