# 195. Playwright E2E Harness & Contacts Importer Coverage

Stand up a project-local Playwright end-to-end test harness and prove it out with two specs covering the contacts importer — happy-path commit and re-import-with-update-strategy — replacing ~30 minutes of manual click-through per iteration with an automated browser run.

---

## Design decisions (resolved before starting)

- **Dev-only test runner.** Playwright never runs on git / CI / deploy server. It is a local-machine tool for the developer. `npm run test:e2e` from the WSL2 host only. No GitHub Actions wiring, no Docker-containerized browser.
- **Project-local Playwright install.** `@playwright/test` added to `package.json` devDependencies (pinned). The existing global install used by `scripts/generate-thumbnails.js` stays as-is; that script is out of scope.
- **DB reset per spec file, not per test.** Playwright's `globalSetup` (or a per-file `beforeAll`) invokes `docker compose exec -T app php artisan migrate:fresh --seed` to reset state. Machine-time acceptable; dev-time is the priority.
- **Auth via the actual login form.** One login at the start of a run, `storageState` saved to a gitignored file, reused across specs. No test-only auth-bypass route. Every step is user-experience-accurate — the login form can harbour bugs too.
- **Existing dev admin used.** `ADMIN_EMAIL` + `ADMIN_PASSWORD` already live in `.env` for local dev. Playwright reads them via `dotenv`. No second admin created. Values stay out of test code.
- **Assertion strategy: mostly UI, DB helper for spot checks.** Most assertions drive through the admin UI (navigate to the Contacts list, assert expected rows visible). A small DB helper (`pg` client in Node, connects to exposed Postgres port 5432) is available for cases where UI navigation is tedious. UI first, DB second — never DB only.
- **`data-testid` naming convention:** `{area}-{thing}` or `{area}-{thing}-{index}`. Examples: `import-contacts-wizard`, `import-commit-button`, `import-progress-bar`, `map-column-{n}`, `import-staged-approve`. Lowercase, hyphens, stable across Filament/Livewire rerenders. No reliance on Filament-generated IDs.
- **Traces + video + screenshots on failure.** Enabled by default in `playwright.config.ts`. All outputs gitignored. `npx playwright show-trace` is how failures are diagnosed.
- **Stress tag infrastructure (no specs yet).** The config supports a `@stress` tag for long-running tests (future 10k-row imports). Default `npm run test:e2e` filters them out; `npm run test:e2e:stress` runs only `@stress`-tagged specs. Writing stress specs is deferred to a later session.
- **Fixture CSVs are handcrafted and small.** Committed under `tests/e2e/fixtures/`. PII-free. 5–10 rows each. Realistic fixtures come later from the Random Data Generator CSV Export session.
- **Scope fence: contacts only.** Events, donations, memberships, and invoice details are out of scope for this session — they land in session 196 once the harness is proven on contacts.

## Phase 1 — Project setup

Add `@playwright/test` and `dotenv` and `pg` to `package.json` devDependencies (pinned to current stable). Do not touch any other deps.

Create `playwright.config.ts` at the project root with:

- `testDir: './tests/e2e'`.
- `use.baseURL: process.env.APP_URL ?? 'http://localhost'` (dotenv loads `.env`).
- `use.storageState: 'tests/e2e/.auth/admin.json'` (gitignored path for logged-in session).
- Chromium project, headless by default, `--headed` / `--ui` available via CLI.
- `use.trace: 'retain-on-failure'`, `use.video: 'retain-on-failure'`, `use.screenshot: 'only-on-failure'`.
- `reporter: [['list'], ['html', { open: 'never' }]]`.
- `grepInvert: /@stress/` at the project level (default excludes stress tests).
- A second project entry `stress` that sets `grep: /@stress/` and extends timeouts significantly (e.g. `timeout: 600_000`).
- `globalSetup: './tests/e2e/global-setup.ts'` (handles DB reset + admin login + storageState capture).

npm scripts in `package.json`:

- `"test:e2e": "playwright test"` — runs the default project (excludes @stress).
- `"test:e2e:ui": "playwright test --ui"` — interactive mode.
- `"test:e2e:headed": "playwright test --headed"` — watch the browser drive.
- `"test:e2e:stress": "playwright test --project=stress"` — stress-only runs.
- `"test:e2e:show-report": "playwright show-report"` — open the last HTML report.

`.gitignore` additions: `tests/e2e/.auth/`, `test-results/`, `playwright-report/`, `blob-report/`.

`tests/e2e/README.md`: short doc covering how to run, where traces/videos live, `data-testid` naming convention, DB reset behaviour, and a one-paragraph "how to add a new spec" checklist.

## Phase 2 — Global setup: DB reset + auth

`tests/e2e/global-setup.ts`:

1. Invokes `docker compose exec -T app php artisan migrate:fresh --seed` via `child_process`. Fails the run loudly on non-zero exit.
2. Launches a Chromium context, navigates to the login page, submits the form with `process.env.ADMIN_EMAIL` + `process.env.ADMIN_PASSWORD`, waits for the admin dashboard to render, saves `storageState` to `tests/e2e/.auth/admin.json`.

The global setup runs once per `playwright test` invocation, so all specs in a run share a freshly-seeded DB + a logged-in session. Specs that need a clean state mid-run can call a helper `resetDatabase()` (same artisan command, wrapped in a function exported from `tests/e2e/helpers/db.ts`).

## Phase 3 — DB helper (`tests/e2e/helpers/db.ts`)

Small typed helper using the `pg` client. Connects to `localhost:5432` (Docker exposes Postgres on the host). Reads `DB_DATABASE` / `DB_USERNAME` / `DB_PASSWORD` from `.env` via dotenv.

Exports:

- `countContacts(): Promise<number>` — returns `SELECT COUNT(*) FROM contacts`.
- `countStagedUpdatesForSession(sessionId): Promise<number>`.
- `findContactByEmail(email): Promise<Record|null>`.
- `resetDatabase(): Promise<void>` — runs `migrate:fresh --seed` via `child_process`.

Kept minimal: only the queries the two specs actually need. Expand in future sessions.

## Phase 6 — Spec: contacts happy-path

`tests/e2e/importer/contacts-happy-path.spec.ts`:

1. `beforeAll`: reset DB.
2. Navigate to `/admin/import-contacts`.
3. Fill session-label, enter new source name, upload `happy-path.csv`, click Next through Upload → Review → Map Columns → Preview.
4. At Map Columns, verify each CSV header is mapped to its expected contact field (snapshot via `data-testid` selectors).
5. Match key defaults to `email`; duplicate strategy defaults to `skip`. Do not change them.
6. At Preview, verify the preview table shows 5 rows.
7. Click Stage Import.
8. On progress page: assert phase is `awaiting`, dry-run stats show 5 would-import, 0 errors.
9. Click Commit.
10. Wait for phase `done` (timeout generous, ~30s).
11. Assert done-state stats: 5 imported.
12. Navigate to the Contacts list; assert 5 new rows visible with the expected emails.
13. Spot-check via DB helper: `countContacts()` = 5 (plus whatever seeded contacts exist — agree baseline at implementation time).
14. Assert a staged-update count of 0 (no matches on first import).

---

## Phase 7 — Spec: contacts update-strategy

`tests/e2e/importer/contacts-update-strategy.spec.ts`:

1. `beforeAll`: reset DB, run the happy-path import to populate 5 contacts (via the UI, not direct-DB, so we test the real commit path — a small helper function that drives the wizard is OK).
2. Navigate to `/admin/import-contacts` again.
3. Upload `update-second-pass.csv`, select `update` duplicate strategy at Map Columns, commit.
4. On done page: assert stats show 5 staged (updated counter).
5. Navigate to Importer landing page.
6. Find the session row, click Preview — assert modal shows the 3 staged changes.
7. Close modal, click Approve on the row.
8. Confirm the modal, submit.
9. Navigate to the Contacts list; assert the updated phone numbers are visible on the matching rows.
10. Spot-check via DB helper: staged-update count for that session is 0 after approval; `findContactByEmail()` on an updated row returns the new phone.
