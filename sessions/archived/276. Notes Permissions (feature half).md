# 276. Notes Permissions (feature half)

Build out the finer-grained permission gates around the structured-interactions surface that landed at session 198. Today's `update_note` / `delete_note` trio is too coarse — any user with `update_note` can edit any note regardless of authorship, see internal-only material if such a flag existed, and change a note's subtype with no audit. C1 shipping the permission gates clears the path for C3 (audit half).

This session is the carve-out from the original "Notes Permissions & Permissions Audit" stub at session 244. C3 will audit the gates this session establishes.

---

## Design decisions (resolved before starting)

- **Three concrete pieces — implement all three.** Per the C1 stub:
  1. **Finer-grained gates around structured fields** — at minimum, gating who can edit a note authored by someone else.
  2. **`edit-only-by-creator` opt-in tenant setting** — `auth()->user()->id === notes.author_id` required to edit when the SiteSetting toggle is on.
  3. **Manager override permission** — a separate permission (e.g. `edit_others_note`) that bypasses (2) regardless of the SiteSetting state.
- **Storage shape for the tenant setting:** a single `SiteSetting` row, key `notes_edit_only_by_creator`, value `'1'`/`'0'`, type `'boolean'`. No migration — `SiteSetting::set()` creates the row on first write per the existing `SiteSetting::set` pattern. Default off (preserve existing behaviour for any install that doesn't change the setting).
- **New permission added:** `edit_others_note`. Granted to `super_admin` and `admin` by default in the seeder; `content_editor` and any other role does not get it. Pattern mirrors the existing `note` resource permissions in `PermissionSeeder.php`.
- **Policy logic shape:**
  ```
  NotePolicy::update($user, $note):
      // Existing capability gate stays as the outer guard.
      if (! $user->can('update_note')) return false;

      // If edit-only-by-creator is off, fall through to today's behaviour.
      if (! SiteSetting::get('notes_edit_only_by_creator', false)) return true;

      // When the toggle is on: must be the author OR have the manager override.
      return $user->id === $note->author_id || $user->can('edit_others_note');
  ```
- **Delete gating:** mirrors update — same `notes_edit_only_by_creator` toggle gates `NotePolicy::delete`. No separate `delete_others_note` permission; reuse `edit_others_note` (managers who can edit-anyone's-notes can also delete-anyone's-notes — the alternative is a 4-permission grid that's overkill for this scope). Document this choice in the log.
- **`type` (subtype) change gate:** out of scope for 276. The original stub mentioned "who can change the subtype of an existing note" but the cleanest shape is a separate permission (`change_note_type`) that's only meaningful if downstream consumers care about subtype-stability. No downstream consumer cares today; lift to C3's audit findings if an actual need surfaces.
- **`is_internal` flag:** out of scope for 276. The C1 stub mentioned visibility gates around an internal-only flag, but the schema has no such column today. Adding a column for a flag with no consumers is premature. Lift to a follow-on if a workflow surfaces (likely C5 or C6, where membership/event coordinator roles need to author manager-only notes).
- **`direction` and `participants` columns:** out of scope. The session 244 stub anticipated these as part of the structured-interactions landing, but the actual session 198 implementation routes that data through `meta` (jsonb) and `notable_type` polymorphism. No separate columns needed.
- **Iteration strategy.** Three iterations on `session-276/1`:
  - `/1` — `edit_others_note` permission (added to seeder + assigned to super_admin + admin). PermissionSeeder edit + companion test asserting the permission exists and is assigned correctly post-seed.
  - `/2` — Extended `NotePolicy::update` + `NotePolicy::delete` with the SiteSetting-driven edit-only-by-creator gate + manager override. Companion tests covering all four cells of the matrix (toggle off / toggle on × author / non-author).
  - `/3` — `notes_edit_only_by_creator` SiteSetting form field on `GeneralSettingsPage` (under the existing CRM section if there is one, or a new "Notes" section). Help-doc update if the admin-page help doc covers the section. Playwright spec exercising the policy gate in the admin UI (operator A creates note, operator B sees Edit greyed / clicking errors). Final fast Pest + Playwright pass.

## Phase 2 — `NotePolicy::update` and `NotePolicy::delete` gating (`session-276/1`)

Extend the existing policy methods. Both `update($user, $note)` and `delete($user, $note)` get the same three-step shape: capability check → SiteSetting read → author-equals-user OR manager-override.

### Implementation shape

```php
public function update(User $user, Note $note): bool
{
    if (! $user->can('update_note')) {
        return false;
    }

    if (! \App\Models\SiteSetting::get('notes_edit_only_by_creator', false)) {
        return true;
    }

    return $user->id === $note->author_id || $user->can('edit_others_note');
}
```

Symmetric shape for `delete`.

### Test surface

A 4-cell matrix per gated method (8 tests total minimum):

| Toggle | Actor | Expected |
|--------|-------|----------|
| off    | author | allow |
| off    | non-author with `update_note`   | allow |
| on     | author | allow |
| on     | non-author with `update_note` only | **deny** |
| on     | non-author with `update_note` + `edit_others_note` | allow |

Plus a baseline test that `update_note` capability still gates the outer level (no `update_note` → always deny, regardless of toggle / authorship).

Run fast Pest. Commit.

## Phase 3 — Tenant setting form field + Playwright + manual smoke (`session-276/1`)

### 3a. SiteSetting form field

Add a Toggle component on `GeneralSettingsPage` for the `notes_edit_only_by_creator` key. Site location to be picked at impl time — likely a new "Notes" section, or an existing "CRM" section if one fits. Field shape:

```php
Forms\Components\Toggle::make('notes_edit_only_by_creator')
    ->label('Restrict note edits to author')
    ->helperText('When on, users can only edit / delete notes they authored. Users with the "Edit any note" permission bypass this restriction.')
    ->default(false),
```

The save action must call `SiteSetting::set('notes_edit_only_by_creator', $state)` with proper boolean coercion (the existing pattern handles type='boolean' rows; verify or add the type on first write).

### 3b. Playwright spec

New `tests/e2e/notes/notes-permissions.spec.ts`:

- Reset DB. Create two operator users — `op_a` and `op_b` — both with `update_note` and `delete_note` but neither with `edit_others_note`.
- `op_a` creates a Note via the admin UI (or seed it via DB helper).
- Toggle `notes_edit_only_by_creator = on` (via DB helper or admin UI).
- Log in as `op_b`. Navigate to the Note's edit page.
- Assert the Edit affordance is either hidden or the page returns 403 / shows a permission-denied notification.
- Toggle off. `op_b` can now edit.
- Restore `op_b` to no `edit_others_note`. Toggle on. Grant `edit_others_note` to `op_b`. Assert `op_b` can edit (manager override).

DB helpers needed:
- `findUserIdByEmail(email)` (likely already exists; add if not)
- `findNoteIdBySubject(subject)` or similar
- `setSiteSetting(key, value)`
- `assignPermissionToUser(userId, permissionName)` and `revokePermissionFromUser(...)`

### 3c. Manual smoke test

Open admin → Settings → General → Notes section (or wherever the toggle lands). Toggle on. Author a note as super_admin, then log in as a non-super-admin role with `update_note` only. Confirm the note is read-only. Confirm a manager-override role (super_admin) can still edit.

Run fast Pest. Run `npm run build` if any front-end source touched (unlikely — Filament Toggle is server-rendered). Run the new Playwright spec.

Announce manual testing is ready. Then stop.
