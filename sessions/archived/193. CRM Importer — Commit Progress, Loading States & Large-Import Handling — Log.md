## What was built

### Phase 1 — Loading-state pattern audit

- Canonical pattern adopted: `wire:click` + `wire:loading.attr="disabled"` + `wire:target="method"` on the button, with two sibling `<span>`s — one `wire:loading.remove` carrying the default label, one `wire:loading` carrying the spinner + busy label. Documented as the project's standard based on the existing scss-editor precedent.
- Filament already injects a global top-of-page `wire:loading.delay` pulse bar via `AdminPanelProvider` render hook; per-button indicators are additive to that, not a replacement.
- Added two anonymous Blade components: `resources/views/components/loading-spinner.blade.php` (pure SVG, `animate-spin`, accepts a `class` prop) and `resources/views/components/loading-link.blade.php` (Alpine-driven `<a>` wrapper with modifier-key-aware click handler and `x-cloak`-safe swap).

### Phase 2 — Per-button loading indicators on the five progress blades

- Applied the two-span pattern to every `wire:click` button across `import-progress.blade.php`, `import-events-progress.blade.php`, `import-donations-progress.blade.php`, `import-memberships-progress.blade.php`, `import-invoice-details-progress.blade.php`:
  - `runCommit` — "Commit" / "Starting commit…"
  - `cancel` — "Cancel" / "Cancelling…"
  - `saveMapping` — "Save mapping" / "Saving…"
  - `downloadPiiErrors`, `downloadErrors` — "Download…" / "Preparing…"
- Filament's built-in table actions (Approve / Rollback / Preview / Delete on `ImporterPage`) have loading built into the confirmation-modal submit button — intentionally left untouched.

### Phase 3 — Navigation loading indicators on heavy-page links

- `<x-loading-link>` swaps its icon slot for a spinner on click and dims the anchor while the browser navigates. Honours modifier keys (⌘/Ctrl/Shift/Alt) so "open in new tab" never triggers the spinner.
- Replaced plain `<a>` links on the five progress-page `done` phase and on the importer landing page's "View Import History" button.

### Phase 4 — `ImporterPage` render performance

- Measured baseline: **5 queries** on initial render with 252 seeded ImportSessions — already well under concerning.
- Applied two small defensive optimisations to `ImporterPage::getBlockedTypes()`: memoize the result on a `$blockedTypesCache` property (so repeat calls within a render free), and push `DISTINCT` down to the database (`->select('model_type')->distinct()`) instead of `->pluck()->unique()` in PHP.
- Added a regression test asserting the page renders in under 15 queries even with hundreds of historical sessions.

### Phase 5 — 1k-row commit transition fix

- Confirmed server-side state machine is clean at 1000 rows via a new `->group('slow')` end-to-end Pest test — the commit reaches the `done` phase correctly.
- Root-cause theory for the intermittent Chrome error at the commit → done transition: the final `tick()` both processes the last heavy chunk (~200 rows × DB inserts) **and** flips the Livewire component to `phase='done'`, which triggers a DOM morph of the whole progress page. Heavy response + DOM morph on the same request is the most likely trigger for the observed morph error.
- Fix: added `public bool $finalizing = false;` to `InteractsWithImportProgress` and split the transition. When the last chunk is processed, the tick sets `$finalizing = true` but leaves phase on `committing`. The next poll (essentially empty of row work) fires the actual transition to `phase='done'` and `finaliseSession()`. Decouples the heavy response from the phase-transition DOM morph.
- Replaced the awkward `$this->done` mid-loop flag inside `tick()` with a clean local `$eofReached` variable; `$this->done` is now only ever set by the finalizing branch (intended terminal state) or the "file not found" error branch.

### Bonus — Map Columns step wizard hang (pre-existing session 192 regression)

- During UAT, discovered a 15s+ Input Delay when the Upload → Review → Map wizard transition rendered the Map step. Reproduced consistently on clean main.
- Bisected the regression via three git checkouts (5f59b01 clean → 432a8cb clean → 97d927c hangs), isolating it to session 192's iteration-2 commit. Within that commit, further narrowed by applying iter-2 changes one at a time on top of iter-1 (via a scratch `bisect-iter2-B` branch).
- Culprit: the "N rows detected" `Placeholder` added to `getReviewStepSchema()` in session 192/2 called `countCsvRows()` on every schema evaluation. Filament re-evaluates all wizard steps' schema closures on every Livewire request while the wizard is mounted, so `countCsvRows()` ran repeatedly for large client CSVs — each call blocking the Livewire response long enough to surface as client-side Input Delay in INP.
- Fix: removed the `$rowCount` lookup and the `review_row_count` `Placeholder` block from `InteractsWithImportWizard::getReviewStepSchema()`. The `countCsvRows()` method itself is preserved — it's still called once (not per-render) by `createSessionAndLog()` to persist `row_count` on the `ImportSession` record at commit time.
