## What was built

### Phase 1 — `Source` primitive

- `app/WidgetPrimitive/Source.php` — final class (private constructor, no instances). Six public-const strings: `HUMAN = 'human'`, `DEMO = 'demo'`, `IMPORT = 'import'`, `GOOGLE_DOCS = 'google_docs'`, `LLM_SYNTHESIS = 'llm_synthesis'`, `STRIPE_WEBHOOK = 'stripe_webhook'`. Public `KNOWN` array enumerates all six. Static `isKnown(string $source): bool`.
- `tests/Unit/WidgetPrimitive/SourceTest.php` — 4 tests, 16 assertions: every constant present, `KNOWN` enumerates all, `isKnown()` accepts every declared value, rejects empty/unknown/case-mismatched strings.

### Phase 2 — `HasSourcePolicy` trait + system-model declarations

- `app/WidgetPrimitive/HasSourcePolicy.php` — trait exposing **instance method** `acceptsSource(string $source): bool`. Universal `Source::HUMAN` pass first; unknown sources rejected; missing `ACCEPTED_SOURCES` constant fails closed (only HUMAN passes); otherwise `in_array($source, static::ACCEPTED_SOURCES, true)`. Trait method is instance-based by deliberate design — `Collection` and `CollectionItem` need per-row policy that reads `$this->accepted_sources` / `$this->collection`, which a static method cannot express.
- `app/Models/Page.php` — `HasSourcePolicy` trait applied. `ACCEPTED_SOURCES = [Source::DEMO, Source::GOOGLE_DOCS, Source::LLM_SYNTHESIS]`. Pages accept blog-post ingestion and LLM authoring beyond the universal HUMAN.
- `app/Models/Contact.php` — trait applied. `ACCEPTED_SOURCES = [Source::IMPORT, Source::DEMO]`. Imports and dev fixtures.
- `app/Models/Donation.php` — trait applied. `ACCEPTED_SOURCES = [Source::IMPORT, Source::STRIPE_WEBHOOK]`. **No `Source::DEMO`** — sharpest security win of the session, asserted by a dedicated test.
- `tests/Unit/WidgetPrimitive/HasSourcePolicyTest.php` — 8 tests, 28 assertions. Two fixture classes (one without `ACCEPTED_SOURCES`, one with `[DEMO]`) exercise the trait directly. The three real models each get a dedicated policy assertion. The Donation/DEMO invariant is locked in by an explicit `in_array(Source::DEMO, Donation::ACCEPTED_SOURCES, true)` assertion that returns false.

### Phase 3 — `DataSink` service + `SourceRejectedException`

- `app/WidgetPrimitive/Exceptions/SourceRejectedException.php` — extends `RuntimeException`, named static factory `for(string $modelClass, string $source)`.
- `app/WidgetPrimitive/DataSink.php` — final class. `write(string $modelClass, string $source, array $payload): Model` with three guards in order:
  1. `Source::isKnown($source)` → `InvalidArgumentException` with the unknown source named.
  2. `class_uses_recursive($modelClass)` contains `HasSourcePolicy` → `LogicException` if absent.
  3. `(new $modelClass)->forceFill($payload)->acceptsSource($source)` → `SourceRejectedException` if false.
  On pass: `$modelClass::create($payload)` and return. The `forceFill` prototype is what enables `CollectionItem` to delegate to its parent collection — the prototype carries `collection_id` from the payload, and accessing `$prototype->collection` triggers Laravel's lazy-load path.
- `app/Providers/WidgetServiceProvider.php` — `DataSink` registered as a singleton alongside `WidgetRegistry`, `SlotRegistry`, `WidgetAssetResolver`, etc.
- `tests/Unit/WidgetPrimitive/DataSinkTest.php` — 7 tests, 11 assertions. Covers Contact accepting IMPORT and HUMAN, Contact rejecting GOOGLE_DOCS, unknown-source `InvalidArgumentException`, target-without-trait `LogicException` (uses `Event::class` since Event isn't policied yet), Donation rejecting DEMO with the row count asserted at zero, and Donation accepting IMPORT.

- `app/Models/Collection.php`:

  - **Instance** `acceptsSource(string $source): bool` override — universal HUMAN pass, then read `$this->accepted_sources` (cast handles the JSONB → array). Per-row policy, not per-class.

  - **Instance** `acceptsSource(string $source): bool` override delegates to `$this->collection?->acceptsSource($source) === true`. Universal HUMAN pass first; unknown source rejected. The null-safe `?->` returns false when the parent collection cannot be resolved, which is fail-closed by construction.
- `app/Filament/Resources/CollectionResource.php` — added a `CheckboxList` field labelled "This collection accepts data from:" in the Collection Details section, full-width below the Active toggle. Options drawn from `Source::KNOWN` minus `Source::HUMAN` (HUMAN is implicit and not selectable). `formatStateUsing` strips HUMAN from the loaded state on edit; `dehydrateStateUsing` re-injects HUMAN into the saved array on every persist.
- `docs/schema/collections.md` — updated with the new column entry, including the gating note pointing at `DataSink::write` and `CollectionItem::acceptsSource`.
- `tests/Feature/CollectionAcceptedSourcesTest.php` — 10 tests, 28 assertions. Defaults-to-`["human"]` on `Collection::create` (with `->refresh()` to read the DB-applied default), DB-default backfill via raw insert, per-row read of `accepted_sources` on the model, CheckboxList renders with HUMAN hidden, create-form-with-no-checks produces `["human"]`, create-form-with-DEMO-checked produces `["human","demo"]`, CollectionItem write rejects/accepts based on parent collection, HUMAN universal pass for items, edit-form hydrates without HUMAN visible.

### Phase 6 — Demo generator migration

- `app/Services/DemoDataService.php` — **no migration needed.** Confirmed in-memory only — every method returns arrays for widget previews; nothing writes to the database. Reading the entire surface confirmed this; documented here as the conclusion of the audit.
- `app/Filament/Widgets/DashboardDebugGeneratorWidget.php`:
  - `generateContacts()` — replaced `Contact::factory()->count($count)->create()` with a loop calling `app(DataSink::class)->write(Contact::class, Source::DEMO, $factory->raw())`. Preserves the optional `withGmailBase` factory state.

  - `generateBlogPosts()` — replaced `Page::create([...])` with `$sink->write(Page::class, Source::DEMO, [...])`. PageWidget writes via `$page->widgets()->create(...)` left untouched.
  - `generateEvents()` and `generateProducts()` — untouched. Event, EventRegistration, Product, ProductPrice are not yet under `HasSourcePolicy`, so direct factory writes remain in place per the session prompt's scope.
- Four widget demo seeders updated to declare `accepted_sources = [Source::HUMAN, Source::DEMO]` on their seeded collections:
  - `app/Widgets/Carousel/DemoSeeder.php`
  - `app/Widgets/LogoGarden/DemoSeeder.php`
  - `app/Widgets/BoardMembers/DemoSeeder.php`
  - `app/Widgets/BarChart/DemoSeeder.php`

The seeders' `CollectionItem::updateOrCreate` write paths were intentionally **not** routed through `DataSink::write` — `updateOrCreate` is upsert-shaped and `DataSink::write` is create-only. The collection-level `accepted_sources` declaration is what carries the policy intent through this surface; if a future session migrates seeders to a sink-aware upsert path, the policy is already in place.
- `tests/Feature/DebugGeneratorSourcePolicyTest.php` — 4 tests, 6 assertions. Contacts/members/blog-posts paths each generate the expected count through the sink. The fourth test asserts the security invariant directly: `DataSink::write(Donation::class, Source::DEMO, [...])` throws `SourceRejectedException` and writes zero rows. (No DebugGenerator code path actually writes Donations — the test asserts the sink would refuse if one did.)

## Deferred decisions

- **Importer routing through `DataSink` with `Source::IMPORT`.** The primitive is ready; the importer code at `app/Filament/Pages/ImportDonationsProgressPage.php:511` (and other importer paths) still calls `Donation::create($payload)` directly. Migrate when the importer track next opens.
- **Stripe webhook handler tagging with `Source::STRIPE_WEBHOOK`.** The constant is declared for future use only — no Stripe code path consumes it yet.
- **Google Docs ingestion adapter / source tagging.** Per the deferred-track outline.
- **LLM-synthesis adapter / source tagging.** Per the deferred-track outline.
- **System-model `HasSourcePolicy` application beyond Page / Contact / Donation.** Event, Transaction, Organization, Membership, Purchase, EventRegistration, PortalAccount, and other write targets stay un-policied this session — they land as the relevant write paths come up. `DashboardDebugGeneratorWidget` continues to write directly to these tables; the surface is documented here so the next pass knows where to look.
- **Per-field source gating.** The full write-contracts story remains 2.0.
- **Audit log of source writes.** Throws are observable in standard exception logs; no dedicated source-write log written.
- **`updateOrCreate` paths through `DataSink`.** Four widget demo seeders use `CollectionItem::updateOrCreate` rather than routing through the sink, because `DataSink::write` is create-only. A sink-aware upsert API can land later if the seeder surface needs tightening; for now, the parent collection's `accepted_sources` carries the policy intent.
- **`TransactionSeeder` base-seed Donation writes.** Lines 44 and 53 use `Donation::factory()->create([...])` directly. Synthetic but not "demo" — base-seed fixtures for dev/test environments. Out of scope per the session prompt; will need a source convention (`Source::SEED`? `Source::FIXTURE`?) when the next non-demo synthetic-data session opens.
- **Pre-existing `SeoTest` OOM mid full-suite run.** Unchanged from 212. `SeoTest` in isolation runs clean (22 passed). Flagged for the eventual "Test Suite Audit — Cost, Coverage, and Shape" session.
