# 332. @layer Cascade Isolation

Roll all host CSS into CSS cascade layers in one coordinated pass, so widget interiors win over host base rules by layer order rather than selector specificity — making the widget styling boundary robust and retiring the specificity workarounds the 330 audit catalogued.

---

## Stub reference

This is the closing session of the **Widget Styling Contract arc** (source spec `sessions/styles-rework-spec.md`, **"Cascade isolation — `@layer`"** + Phase D). It is a track-doc step on the **Widget Autonomy** track, not a `release-plan.md` entry. The 330 audit's **Matrix 4** is the canonical inventory of the cascade collisions this session resolves.

## Why now / why its own session

The 330 audit's headline was that the widget styling boundary is mostly clean *except* for cascade: widget precedence over the host relies on (a) source order — the widget bundle is concatenated last — for equal-specificity ties, and (b) deeper host base selectors winning where they out-specify a widget rule. That second case is fragile: it holds only because we author every widget and control the bundle order. A third-party widget (the arc's destination) writing a deeper selector, or loaded in a different order, breaks unpredictably.

`@layer` fixes this structurally: put host reset/base in a low layer and widget interiors in a high `widgets` layer, and author precedence is guaranteed by **layer order, independent of selector depth**. But it is the single highest-blast-radius change in the arc, because **any unlayered rule beats every layered rule** — so it is only safe as a coordinated all-host rollout, which is exactly why 330 and 331 deferred it to its own session with its own verification gate.

---

## Open questions to resolve at session start

1. **Layer order.** Proposed: `@layer reset, base, components, widgets;` — host reset lowest, then base typography/elements, then host components (buttons/forms/controls/grid/layout), then widget interiors highest. Confirm the names and the placement of each host source against them.
2. **Where the runtime typography + colour emit sits.** The `TypographyCompiler` `.np-site` rules and any rule-based colour emit must land in a **low** layer so a widget's own heading/text rules can override — this is what retires the EventMiniCalendar / EventsListing workarounds. Confirm.
3. **Operator `custom_scss`.** Layered (and in which layer) or left **unlayered** so it wins over everything? Operators likely expect their own CSS to beat the theme; unlayered-wins is the intuitive default, but it must be a deliberate call (an unlayered operator rule beating a layered widget rule is the *intended* exception, not a leak).
4. **Verification mechanism.** How the before/after compiled-CSS comparison is captured, and which public pages carry the visual gate.
5. **Standing guard.** Does a `@layer`-discipline Pest test (every host CSS source emits into a declared layer; nothing unlayered escapes) join the `design` group?

## Phases

### Phase 1 — Design + inventory (no irreversible change)

- Settle the layer order and the custom-SCSS slot (the Open Gate decisions).
- Inventory **every** source of host CSS that reaches the public bundle: each static partial (`_base`, `_layout`, `_grid`, `_forms`, `_controls`, `_buttons`, `_icons`, `_media`, `_custom`; `_variables` is vars-only), `theme/public.scss`, the `TypographyCompiler` emit, the colour emit, per-template `custom_scss`, and the widget SCSS bundle. Produce the source→layer assignment table — this is the artifact that proves nothing is missed.
- Capture the **before** compiled CSS (the baseline for the diff).

### Phase 2 — Roll out (the coordinated pass)

- Declare the layer order once, in the right place in the concatenation, so every consumer sees the same `@layer` statement first.
- Wrap each inventoried source's rules in its assigned layer. The mechanism (Sass `@layer` blocks in the partials vs. wrapping at concatenation time in `AssetBuildService` vs. wrapping the runtime emit in PHP) is part of Phase 1's design — pick the one that covers the runtime emitters too, since those are the easy misses.
- **Revert the specificity workarounds** as the rollout's positive proof: EventMiniCalendar / EventsListing `display: block` rows back to natural list rendering; EventsListing `flex: 0 0 14rem` back to `width: auto`. If they still render correctly, the layering took.

### Phase 3 — Verify

- **Compiled-CSS before/after diff** — confirm the change is structural (layer wrappers + the reverted workarounds), not a restyle.
- **Visual gate** — representative public pages at desktop + mobile widths; the output must not move.
- **Tests** — fast suite green; the consumption gate + `DesignGroupIntegrityTest` green; the new `@layer`-discipline guard (if adopted) green.

## Out of scope

- **No appearance changes.** A visible difference means the rollout is wrong, not improving. Resist restyling while layering.
- **No new mobile rules.** The Mobile Collapse Ruleset and its viewport-spacing primitive are session 333+'s work — this session lands the *layer structure* those rules will later be authored into, nothing more.
- **No widget-interior changes** beyond reverting the two named workarounds.
- **No breakpoint single-sourcing** — that's a precursor task for the mobile push, not this session.
