# 214. Widget Primitive ŌĆö Phase 3c (Dashboard-Native Widgets)

Author three dashboard-native widgets ŌĆö **Memos**, **Quick Actions**, **This Week's Events** ŌĆö that exercise all three contract sources, prove the dashboard-grid slot end-to-end, and land the first user-visible widget content on the admin dashboard. Re-enable `DashboardSlotGridWidget::widgets()` with these three definitions and remove the legacy Filament-native quick-actions widget once the new one is rendering.

This session is the first one to author widgets purpose-built for a non-page-builder slot, the first to consume the `accepted_sources` paradigm landed in 213, and the first to extend `ContractResolver` beyond `'post'` for `SOURCE_SYSTEM_MODEL`.

---

## Design decisions (resolved before starting ŌĆö do not relitigate)

- **Three widgets, one per contract source.** Memos ŌåÆ `SOURCE_WIDGET_CONTENT_TYPE`. Quick Actions ŌåÆ no contract (`dataContract()` returns `null`, pure static config). This Week's Events ŌåÆ `SOURCE_SYSTEM_MODEL` on Event. The combination is the proof point ŌĆö each widget tests a different read path through the resolver.
- **Each widget overrides `allowedSlots()` to `['dashboard_grid']` only.** None of these are page-builder widgets. `allowedSlots()` is declarative ŌĆö there is no enforcement in the picker today, but the declaration is correct and grep-visible.
- **Each widget keeps `acceptedSources()` at the `[Source::HUMAN]` default.** None of them write data; the declaration is for completeness and future write paths.
- **Memos collection is regular-seeded, not demo-seeded.** A `memos` collection with `accepted_sources = [Source::HUMAN]` is part of every install. Population happens through the existing Collection Manager ŌåÆ Memos ŌåÆ items relation manager. **No dedicated memos admin UI in this session** ŌĆö that is friction the user accepted to keep scope honest. A streamlined memo composer can land later if the friction matters.
- **Memos collection is `is_public = false`.** Items render on the admin dashboard, never on the public site. If `WidgetDataResolver` (or `ContractResolver` for `SOURCE_WIDGET_CONTENT_TYPE`) currently filters by `is_public`, this session must verify whether dashboard-rendered widgets bypass that filter or whether `is_public` semantics need to be revisited. The verification is part of Phase 1 ŌĆö pause and ask if the resolver's behaviour conflicts with the "admin-only memos" requirement.
- **Quick Actions config is a list of action keys.** Each key resolves at render time to a URL/label/icon via a small static registry inside the `QuickActionsDefinition` class. The widget's `schema` exposes the action list as a `checkboxes`-typed field with the registry's keys as options. The `DashboardSlotGridWidget::widgets()` registration hardcodes a default selection (e.g., `['new_contact', 'new_event', 'new_post']`) ŌĆö admin choice is deferred to the dashboard config UI session.
- **Action registry is internal-routes-only.** Every URL the registry resolves to comes from a Filament `Resource::getUrl()` call ŌĆö no user-supplied URLs, no external destinations. Closes the obvious XSS / open-redirect surface up front.
- **`Event` is added to `ContractResolver`'s `SOURCE_SYSTEM_MODEL` switch.** Fields exposed are exactly the ones the Events widget needs: `id`, `title`, `slug`, `starts_at`, `ends_at`, `location` (verify this column exists on `Event` during read; drop it from the field list if not). Per-field fail-closed remains in force ŌĆö a contract that requests an undeclared field gets `null`.
- **The legacy Filament-native quick-actions widget is removed** from the admin chrome, but only **after** the new `QuickActionsDefinition` is rendering correctly on the dashboard. Removal is a single edit to `AdminPanelProvider::widgets()` (or wherever the legacy widget is registered). Locate the file first; it has not been verified yet.
- **No new admin config UI for dashboard arrangement** in this session. The dashboard config UI / storage model for admin-arranged dashboards is the Phase 3d / 3e session, plausibly 215.
- **No new schema columns.** This session adds a regular-seeded collection row but no migrations.

## Phase 1 ŌĆö `MemosDefinition` widget + `memos` collection seed

`app/Widgets/Memos/`:

- `MemosDefinition.php`:
  - `handle(): 'memos'`, `label(): 'Memos'`, `description()`, `category(): ['dashboard']` (or whichever category fits ŌĆö verify what categories the picker recognises).
  - `allowedSlots(): ['dashboard_grid']`.
  - `acceptedSources(): [Source::HUMAN]` (default ŌĆö explicit anyway for grep clarity).
  - `schema()`: a small config surface ŌĆö at minimum `limit` (default 5) for how many memos to show.
  - `defaults()`: matches schema keys.
  - `dataContract(array $config): ?DataContract`:
    - `version: '1.0.0'`, `source: SOURCE_WIDGET_CONTENT_TYPE`.
    - `contentType` is a `ContentType` with `handle: 'memos.entry'`, `fields: [{key: title, type: text}, {key: body, type: rich_text}, {key: posted_at, type: date}]`, `accepts: [Source::HUMAN]`.
    - `fields: ['title', 'body', 'posted_at']` (per-field fail-closed declaration).
    - `resourceHandle: 'memos'`.
    - `filters: ['limit' => $config['limit'] ?? 5, 'order_by' => 'posted_at desc']` (or whatever filter shape the resolver supports ŌĆö verify).
- `template.blade.php`: simple list. Each memo: title, formatted posted_at, body (rendered as HTML). Empty state: "No memos posted yet ŌĆö visit Collection Manager ŌåÆ Memos to add the first."
- Optional `DemoSeeder.php`: a few example memos in the `memos` collection for development. Idempotent. Required only if the empty state is going to confuse manual testing ŌĆö judgement call, write it if convenient.

`database/seeders/MemosCollectionSeeder.php` (new file):
- Idempotent `Collection::firstOrCreate(['handle' => 'memos'], [...])`.
- `name: 'Memos'`, `description`, `source_type: 'custom'`, `is_public: false`, `is_active: true`, `accepted_sources: [Source::HUMAN]`.
- `fields`: title (text, required), body (rich_text, required), posted_at (date, optional).
- Wire into `DatabaseSeeder::run()` so production installs get it.

`app/Providers/WidgetServiceProvider.php`:
- Register `new MemosDefinition()` alongside the others.

Tests:
- Unit: `MemosDefinition::dataContract($config)` returns the expected shape ŌĆö version, source, contentType handle and fields, resourceHandle.
- Feature: after running `MemosCollectionSeeder`, the `memos` collection exists with `accepted_sources = ['human']`.
- Feature: a memo written through the Collection Manager UI surfaces in the rendered Memos widget output.

## Phase 2 ŌĆö `QuickActionsDefinition` widget

`app/Widgets/QuickActions/`:

- `QuickActionsDefinition.php`:
  - `handle(): 'quick_actions'`, `label(): 'Quick Actions'`, `description()`, `category(): ['dashboard']`.
  - `allowedSlots(): ['dashboard_grid']`.
  - `acceptedSources(): [Source::HUMAN]`.
  - `schema()`: one field ŌĆö `actions` of type `checkboxes`, options drawn from `static::actionRegistry()` keys.
  - `defaults(): ['actions' => ['new_contact', 'new_event', 'new_post']]` (sensible default selection).
  - `dataContract(array $config): ?DataContract { return null; }` ŌĆö no read-side data needed.
  - Private static `actionRegistry()` returning an array keyed by action key, each value an array of `['label' => ..., 'url' => Closure that returns Filament URL, 'icon' => 'heroicon-o-...']`. Suggested initial keys:
    - `new_contact` ŌåÆ ContactResource::getUrl('create')
    - `new_event` ŌåÆ EventResource::getUrl('create')
    - `new_post` ŌåÆ PostResource::getUrl('create')
    - `new_donation` ŌåÆ DonationResource::getUrl('create')
    - `new_form` ŌåÆ FormResource::getUrl('create')
- `template.blade.php`: render the selected actions as a vertical list of admin-styled link buttons. Use existing admin button styling ŌĆö no new CSS unless absolutely required. If new CSS is needed, hand-edit `public/css/admin.css` (no build) per the base-prompt rule.

`app/Providers/WidgetServiceProvider.php`:
- Register `new QuickActionsDefinition()`.

Tests:
- Unit: `QuickActionsDefinition::dataContract([])` returns null. Schema has a single `actions` field. Defaults match.
- Feature: rendering the widget with `['actions' => ['new_contact']]` config produces an `<a>` tag pointing at the ContactResource create URL.
- Feature: rendering with `['actions' => []]` produces a non-broken empty state.

## Phase 3 ŌĆö `ThisWeeksEventsDefinition` widget + `ContractResolver` Event support

`app/WidgetPrimitive/ContractResolver.php`:
- Extend the `SOURCE_SYSTEM_MODEL` switch to handle `model: 'event'`.
- Field whitelist for events: `id`, `title`, `slug`, `starts_at`, `ends_at`, `location` (verify column).
- Filter support: `'date_range'` filter (e.g., `'date_range' => ['from' => 'now', 'to' => '+7 days']`) that translates to `whereBetween('starts_at', ...)`. If the resolver's filter language doesn't naturally express this, add the smallest extension that does ŌĆö do not refactor the filter system.
- Fail-closed: a request for an undeclared field returns null. Add a unit test asserting a contract that asks for `internal_notes` (or any non-whitelisted column) gets nothing back.

`app/Widgets/ThisWeeksEvents/`:

- `ThisWeeksEventsDefinition.php`:
  - `handle(): 'this_weeks_events'`, `label(): "This Week's Events"`, `description()`, `category(): ['dashboard']`.
  - `allowedSlots(): ['dashboard_grid']`.
  - `acceptedSources(): [Source::HUMAN]`.
  - `schema()`: one field ŌĆö `days_ahead` (number, default 7, advanced).
  - `defaults(): ['days_ahead' => 7]`.
  - `dataContract(array $config): ?DataContract`:
    - `version: '1.0.0'`, `source: SOURCE_SYSTEM_MODEL`, `model: 'event'`.
    - `fields: ['id', 'title', 'slug', 'starts_at', 'ends_at', 'location']`.
    - `filters: ['date_range' => ['from' => 'now', 'to' => '+' . ($config['days_ahead'] ?? 7) . ' days'], 'order_by' => 'starts_at asc']`.
- `template.blade.php`: list, no images. Each event: formatted date, title, location (if present). Empty state: "No events in the next 7 days."

`app/Providers/WidgetServiceProvider.php`:
- Register `new ThisWeeksEventsDefinition()`.

Tests:
- Unit: `ContractResolverTest` ŌĆö Event model resolves, only whitelisted fields populate, undeclared field requests return null, date-range filter works.
- Unit: `ThisWeeksEventsDefinition::dataContract($config)` returns the expected shape, including a config-driven `days_ahead`.
- Feature: with three events seeded ŌĆö one tomorrow, one in 6 days, one in 30 days ŌĆö the widget renders the first two and not the third.
