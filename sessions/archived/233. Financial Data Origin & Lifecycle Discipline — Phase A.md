# 233. Financial Data Origin & Lifecycle Discipline — Phase A (`source` column + write-path discipline)

Foundational data-discipline session. Adds a queryable **`source`** column to the four financial parent tables (`donations`, `memberships`, `event_registrations`, `transactions`), backfills it from existing predicates, applies `HasSourcePolicy` discipline uniformly across the four models, and updates every write path (Stripe webhook, public checkout controllers, four import progress pages, manual admin entry) to set `source` on creation. Resolves the membership-import ledger asymmetry surfaced during the 233 prompt-drafting audit. **No UI changes, no read-side changes, no front-end builds** — purely write-path-and-data-layer.

This is Phase A of the **Financial Data Origin & Lifecycle Discipline** stub in `session-outlines.md`. Phase B (admin-action gating generalization) follows when an action surface that needs it is imminent. Phase C (QuickBooks sync + email-trigger origin-awareness) is deferred until those features arrive.

## Design decisions to confirm before starting

This session has more decision points than a typical phase-arc session. Confirm each with the user before writing code. Defaults below reflect the conclusions reached during 233 prompt-drafting; flag if anything reads wrong on second look.

### Decision 1 — `ACCEPTED_SOURCES` per model

| Model | Proposed `ACCEPTED_SOURCES` | Rationale |
|---|---|---|
| `Donation` | `[IMPORT, STRIPE_WEBHOOK]` *(already declared; unchanged)* | Admins cannot hand-write donations. Every donation either flowed through a Stripe checkout or was imported. |
| `Membership` | `[HUMAN, IMPORT, STRIPE_WEBHOOK]` | Admins can create memberships via the *Promote to Member* action on Contact. Importers carry historical memberships. Stripe membership checkout creates them via webhook. |
| `EventRegistration` | `[HUMAN, IMPORT, STRIPE_WEBHOOK]` | Public visitors register themselves (HUMAN). Importers carry historical attendance. Paid event registration goes through Stripe. |
| `Transaction` | `[HUMAN, IMPORT, STRIPE_WEBHOOK]` | Admin can create manual transactions (grants, paper checks, adjustments). Importers carry historical ledger. Stripe webhook records every payment. |

### Decision 2 — Backfill rules

The migration's `up()` includes a backfill UPDATE per table. The `down()` simply drops the column.

| Table | Backfill predicate (in evaluation order) | Source assigned |
|---|---|---|
| `donations` | `import_source_id IS NOT NULL` | `import` |
| `donations` | `stripe_subscription_id IS NOT NULL` | `stripe_webhook` |
| `donations` | (remainder) | `stripe_webhook` *(Donation does not accept HUMAN; legacy rows pre-import are all Stripe-driven)* |
| `memberships` | `import_source_id IS NOT NULL` | `import` |
| `memberships` | `stripe_session_id IS NOT NULL OR stripe_subscription_id IS NOT NULL` | `stripe_webhook` |
| `memberships` | (remainder) | `human` |
| `event_registrations` | `import_source_id IS NOT NULL` | `import` |
| `event_registrations` | `stripe_session_id IS NOT NULL` *(verify column name in step 6 — may be `stripe_payment_intent_id` or similar)* | `stripe_webhook` |
| `event_registrations` | (remainder) | `human` |
| `transactions` | `import_source_id IS NOT NULL` | `import` |
| `transactions` | `stripe_id IS NOT NULL` | `stripe_webhook` |
| `transactions` | (remainder) | `human` |

**Order matters:** the IMPORT predicate runs first because some imported rows may also carry Stripe identifiers (e.g. an imported recurring donation that originally lived in another Stripe-driven CRM). Treating those as IMPORT is correct because the row's *current* lifecycle in this system isn't Stripe-driven — we're not the entity Stripe is calling webhooks for.

### Decision 3 — Public-checkout-flow source attribution

When a public visitor clicks "Donate" / "Register" / "Buy Membership" and `DonationCheckoutController::store` (or the corresponding event/membership controller) creates the row **before** the Stripe webhook fires, what is `source`?

**Locked: `Source::STRIPE_WEBHOOK`.** Reasoning: the row will carry a Stripe identifier (`stripe_subscription_id` or `stripe_session_id` or `stripe_customer_id`), its lifecycle (pending → active or pending → cancelled) is driven exclusively by Stripe webhooks, and action gating downstream (refund/cancel/retry) belongs to the Stripe path. Putting these rows in the HUMAN bucket would conflate them with admin-created memberships and manual transactions, which have a fundamentally different downstream action surface.

The alternative reading — "the row was created by a controller responding to a human click, so source=HUMAN" — is rejected because origin tracks the **lifecycle owner**, not the creator-actor. Stripe owns the lifecycle for these rows.

**Free-event registration paths** (where there's no Stripe at all — `EventController::register` and `EventRegistrationController::create`) get `Source::HUMAN`. The visitor is the lifecycle owner; there's no payment system to defer to.

### Decision 4 — Membership-import ledger asymmetry

Today donation imports write `Donation` + `Transaction`; event imports write `EventRegistration` + `Transaction`; membership imports write `Membership` only. **This session resolves the asymmetry by adding conditional Transaction writes to `ImportMembershipsProgressPage`.**

Specifically: when an imported membership row carries a non-zero `amount_paid` AND its status is `active` or `expired` (i.e., money actually moved at some point), write a corresponding `Transaction` with `source = Source::IMPORT`, `subject_type = 'App\Models\Membership'`, `subject_id = $membership->id`, amount and date copied from the membership row. Comp / lifetime / unpaid memberships skip the transaction write — same shape as the donation-import conditional pattern.

This matters because: (a) reconciliation against QuickBooks expects every paid membership to appear in the transactions ledger, (b) lifetime giving aggregates that walk transactions otherwise undercount imported member contributions, (c) the symmetric pattern (one row per parent + conditional Transaction child when paid) is easier to reason about than three different importer behaviours.

## Phase 1 — Schema migration + backfill + schema docs

1. Create one migration `database/migrations/YYYY_MM_DD_HHMMSS_add_source_to_financial_tables.php` covering all four tables in a single up()/down() pair. Per-table:
   - `up()`: `$table->string('source')->nullable()->after('status')` (nullable initially so the backfill can run, then a follow-up `change()` to NOT NULL after backfill — OR simpler, set with default during column creation and run the backfill UPDATE afterwards). Plus `$table->index('source')` for the reconciliation query path.
   - Backfill: raw `DB::statement` UPDATEs per Decision 2's order-mattering predicates.
   - After backfill, ALTER COLUMN to NOT NULL (`$table->string('source')->nullable(false)->change()`). The DataSink + HasSourcePolicy boundary will enforce non-null on writes going forward.
   - `down()`: drop the column.
2. Run `docker compose exec app php artisan migrate` against the dev install.
3. **Pause for user inspection** before running the migration. Surface the exact UPDATE SQL that will run; user reviews before commit. This is data-criticality paranoia warranted by the affected tables.
4. Update schema docs:
   - `docs/schema/donations.md` — add `source` row to the column table.
   - `docs/schema/memberships.md` — same.
   - `docs/schema/event_registrations.md` — same.
   - `docs/schema/transactions.md` — same.
5. **Tests** at `tests/Feature/Migrations/AddSourceToFinancialTablesTest.php` (or follow the existing migration-test convention if there is one — verify in step 6 of base prompt):
   - Column exists on all four tables with correct nullable=false constraint after migration.
   - Index on `source` exists per table.
   - Pre-existing rows backfilled correctly per the predicate matrix.
   - Down-migration drops the column cleanly.

## Phase 5 — Membership-import ledger fix

Resolves Decision 4. Add a conditional Transaction creation step to `ImportMembershipsProgressPage` that mirrors the donation-import pattern.

1. Add a new private method `createTransactionForMembership(Membership $membership, array $attrs): ?Transaction` that:
   - Returns null if `$membership->amount_paid` is null or zero.
   - Returns null if `$membership->status` is not in `['active', 'expired']` (don't write transactions for pending/cancelled).
   - Otherwise builds a Transaction payload with `subject_type = Membership::class`, `subject_id = $membership->id`, `amount = $membership->amount_paid`, `currency = 'usd'` (or pull from a column if memberships carry currency — verify in step 6), `status = 'succeeded'` (or whatever the canonical Transaction status is — verify), `import_source_id`, `import_session_id`, `external_id`, `occurred_at = $membership->starts_on ?? now()`, `source = Source::IMPORT`.
   - Calls `Transaction::create($payload)`.
2. Wire the call into the existing membership-row-creation flow at line ~505 (or wherever `Membership::create($payload)` is, adjacent to it).
3. **Tests**:
   - `ImportMembershipsTransactionLedgerTest.php` — three cases: paid+active membership creates matching Transaction; comp/$0 membership creates Membership only; pending membership creates Membership only.

## Out of scope

- **Phase B — Admin gating generalization.** Lifting the `! $record->stripe_id` action-visibility pattern from `TransactionResource` into a shared trait and applying it to Donation/Membership Resources. Not in this session because there are no Refund/Cancel/Sync actions on Donation or Membership today; the gating is unneeded until those actions exist. Phase B lands when an action surface that needs it is imminent.
- **Phase C — QuickBooks sync + email-trigger origin-awareness.** Deferred until those features arrive. Currently no QB sync feature exists beyond the placeholder `qb_sync` action on Transaction; no admin-driven bulk email actions exist that would pull imported rows into audiences accidentally.
- **DataSink-as-mandatory-boundary.** This session adds `source` to fillable + payload at every write site, but does not require all writes to flow through `DataSink::write()`. Centralizing financial writes through DataSink is its own future cleanup; the persisted column is the prerequisite, not the consequence.
- **DEMO source on financial models.** `Source::DEMO` is not added to any financial model's `ACCEPTED_SOURCES` in this session. The dashboard debug generator is reworked to use `Source::HUMAN` per the Phase-6 sub-decision; if DEMO becomes load-bearing later (e.g., for a stronger demo-vs-real separation in marketing-prospect installs) it can be added then.
- **Source for non-financial models.** Contact, Page, Collection, etc. have their own origin questions, but they're outside the financial-data-criticality scope of this session. If `import_source_id` discriminators are insufficient elsewhere, lift in a separate session.
- **Read-side projector / widget changes.** Recent Donations (5d-3) ships in session 234 and may or may not project a `donation_origin` field — design call at 234 prompt-drafting time.
- **`event_registrations` schema verification.** Step 6 of the base prompt's read-list explicitly calls out verifying the Stripe-identifier column name on `event_registrations` (named `stripe_session_id` in this prompt's matrix but may differ — confirm before writing the backfill SQL).
- **Membership currency field.** Phase 5's transaction-creation logic assumes `currency = 'usd'` since memberships don't currently carry a currency column. If multi-currency support is added later, this needs revisiting.
