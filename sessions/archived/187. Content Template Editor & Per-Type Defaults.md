# 187. Content Template Editor & Per-Type Defaults

Unify widget-stack storage so pages and templates share one shape, give content templates a real editor (the existing Vue page builder, pointed at a template owner), let any page type pick a template at creation, and wire a per-page-type default so new pages/posts/events hydrate their starter widgets from a template instead of hardcoded fallbacks.

---

## Background

Most of the plumbing already exists:

- Page widget stacks live in `page_widgets` + `page_layouts` rows, edited by the Vue page builder via `PageBuilderApiController`.
- `templates.definition` (JSONB) stored a serialised copy of a widget stack — produced by `PageWidget::serializeStack()`, consumed by `CreatePage::hydrateWidgets()`.
- "Save as Template" (in `app/Livewire/PageBuilder.php`) already serialises a live page's stack.
- Token substitution (`{{title}}`, `{{author}}`, `{{date}}`, etc.) already runs at render time against the widget's owning page.

What was missing, and what surfaced during session-open design discussion:

1. **Two storage shapes for one concept.** Pages store their stacks as rows; templates stored theirs as a JSONB blob. The split forced synthetic widget IDs, blocked per-widget image uploads on templates, and required a parallel controller for the template editor. Since the project is pre-beta with no production users, we're fixing the split rather than building around it.
2. **No editor for a content template's widget stack.** `EditContentTemplate.php` is a bare name + description form.
3. **Blog posts can't pick a template at creation.** Currently excluded.
4. **No per-page-type default.** Each creation form requires manual selection or falls through to hardcoded widgets.

The user still has total control after creation. A template is a *starting point*, not a contract — once widgets are on the page, they're owned by the page and can be added, removed, or reconfigured freely. Editing a template later does **not** retroactively touch pages already hydrated from it.

---

## Design decisions (resolved before starting)

- **Unified storage via polymorphic ownership.** `page_widgets` and `page_layouts` gain `owner_type` + `owner_id` columns; existing `page_id` rows become `owner_type = 'page'`. Templates stop storing a JSONB `definition`; their widget stack is just rows with `owner_type = 'template'`. One controller, one stack shape.
- **Backfill on migration.** Every existing content Template's `definition` JSONB is hydrated into rows owned by that Template at migration time. The `definition` column is then dropped.
- **Table rename deferred.** `page_widgets` is a misnomer once it holds template widgets, but renaming is deferred — polymorphism is the important change, the rename is cosmetic and can happen in a follow-up session.
- **`hydrateWidgets()` becomes a row-copy.** It becomes a generalised `copyOwnedStack($sourceOwner, $targetOwner)` — a thin extension of the existing `PageWidget::copyBetweenPages()`. Media attached via spatie copies with the widget row.
- **"Save as Template" becomes a row-copy.** It no longer serialises to JSON. It creates a Template and copies the page's rows to it.
- **Template scope stays flat.** No `applicable_to` field, no per-page-type filtering of the template dropdown. The full list shows at every page-type's creation form. (Tagging / scoping deferred — can be added later if the list grows unwieldy.)
- **Per-page-type defaults live on `site_settings`.** One setting per page type: `default_content_template_page`, `default_content_template_post`, `default_content_template_event`, etc. No new table. Empty string = no default (user starts blank unless they pick one).
- **Default applies only when the user does not make a selection.** "None" must be a first-class choice on the creation form. Selecting it produces a blank page regardless of the default.
- **No re-apply on template edit.** Editing a content template never touches pages that already hydrated from it.
- **Blog posts get the selector.** Remove the current exclusion on `post` type; the template dropdown + default lookup applies to `type = post` identically to other page types.
- **Tokens for template widgets.** Already wired (session 186). Template widgets render with tokens literal in the template editor (no owning page) and resolve against the page once hydrated.
- **Image upload in template editor.** Works natively — template widget rows are real `PageWidget` rows, so spatie media attaches the same way it does for pages. On hydration, media records copy to the destination widget.

## Phase 1 — Structural unification: polymorphic owner on `page_widgets` + `page_layouts`

Collapse the two storage shapes into one.

### Phase 1a — Migration

- Add `owner_type` (string) and `owner_id` (uuid) to `page_widgets` and `page_layouts`. Index on `(owner_type, owner_id)`.
- Backfill: for every existing row, set `owner_type = 'page'`, `owner_id = page_id`.
- For each existing content template with a non-empty `definition`, instantiate rows owned by the template (same logic as `hydrateWidgets()`, but targeting an `owner_type = 'template'` owner).
- Drop `page_id` from `page_widgets` and `page_layouts`.
- Drop `definition` from `templates`.
- Update `docs/schema/page_widgets.md`, `docs/schema/page_layouts.md`, and `docs/schema/templates.md`.

### Phase 1b — Models, scopes, queries

- `PageWidget` and `PageLayout` gain `owner()` morphTo relationship.
- Replace every `where('page_id', …)` in the codebase with an owner-scoped equivalent. Add a scope like `forOwner(Model $owner)` to keep call sites readable.
- `Page` and `Template` both get `widgets()` and `layouts()` morphMany relationships.
- `PageWidget::copyBetweenPages()` generalises to `PageWidget::copyOwnedStack(Model $source, Model $target)` — handles Page→Page, Page→Template, Template→Page. Existing `copyBetweenPages()` becomes a thin wrapper for call-site compatibility or gets replaced outright (project is pre-beta).
- `PageWidget::serializeStack()` can be removed — no longer used. (Confirm no other callers before deleting.)

### Phase 1c — Controller + bootstrap

- `PageBuilderApiController` takes a polymorphic owner instead of `Page $page`. Introduce a route param `{owner}` bound via a resolver that looks up by UUID across Pages and Templates (or two parallel route groups — pick whichever is cleaner).
- `app/Livewire/PageBuilder.php` accepts an owner (page or template), builds bootstrap data from `$owner->widgets` / `$owner->layouts`. Most of `getBootstrapData()` stops caring which owner it's on.
- "Save as Template" (`PageBuilder::saveAsTemplate`) creates a Template and calls `copyOwnedStack($page, $template)` — drops the `definition` write.
- Template-only UX concerns in bootstrap (no tags, no details-url, no page author, etc.) fall back to empty/null cleanly.

## Phase 2 — Editor UI for content templates

Replace the bare form in `EditContentTemplate.php` with a real editor.

- Add a tab structure on `EditContentTemplate` (match `EditPageTemplate`): "Details" (name, description) + "Widgets" (embeds the Livewire page-builder with the Template as owner).
- The Vue app doesn't need any new mode — it points at the polymorphic API and renders the owner's widget rows, the same way it does for pages.
- "Save as Template" from a page continues to work — now a row-copy instead of a JSON export.

---

## Phase 3 — Creation-time template selection + per-type defaults

- Remove the post-type exclusion. The `content_template_id` selector appears on every page-type's create form, defaulted to the per-type default, with "None" as a selectable option.
- Add per-page-type default settings in `site_settings`: `default_content_template_{type}` for each page type. Surface these in an admin settings page (CMS Settings is the natural home).
- `CreatePage::afterCreate()` (and `CreatePost`, `CreateEvent` equivalents) looks up the default when the form's `content_template_id` is unset, and calls `copyOwnedStack($template, $newPage)`. If the user picked "None," copy nothing.
- Leave existing hardcoded page-type fallback widgets in place *only* until a default template is configured — once a default exists for a page type, it supersedes the hardcoded fallback.

## Testing

- **Slow test groups this session:** none.
- **New tests expected:** yes.
  - Polymorphic owner: PageWidget/PageLayout can be owned by Page or Template; morphTo resolves correctly.
  - `copyOwnedStack()`: Page→Template and Template→Page both produce the expected row set, including layouts and nested widgets. Media attached to source rows is copied to destination rows.
  - Migration backfill: pre-existing pages keep their widgets; pre-existing templates with a `definition` end up with equivalent rows (test via a seeded fixture, re-running the migration on a fresh DB).
  - "Save as Template" from a page creates a Template with a matching widget stack.
  - Creation-time hydration from a template produces the expected owned rows on a new page/post/event.
  - Per-page-type default fires only when the creation form leaves `content_template_id` unset; "None" suppresses it.
  - Post-type can pick a template (no exclusion).
  - Editing a template does not mutate pages that were previously hydrated from it.
- Run `docker compose exec app php artisan test --exclude-group=slow` before close.
- Run `npm run build` if any JS changes land (unlikely this session — Vue store unchanged).
- Run `docker compose exec app php artisan build:public` if any widget SCSS/CSS/JS changes land (unlikely).
