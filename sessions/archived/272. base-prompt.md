## Process rules for every session

The base-prompt rules (process, close gate, style) in `sessions/template-base-prompt.md` apply. Session-specific notes:

- **No schema changes; no Composer changes; no migrations.** Same rule as 271 — safe-fix-in-place is the only code change permitted, and only for confirmed-trivial cases. Anything bigger goes on W7 / W8 / Open Flags for session 273.
- **No agent contract change.** No `docs/fleet-manager-agent-contract.md` edit, no version bump. The W4 walk is read-only.
- **Pause and ask** if a workstream surfaces something architecturally bigger than a "safe in-place fix" — those are W7 / W8 / Open Flags entries, not in-session decisions.
- **Bug fixes are out of scope.** Catching a real bug during the audit triggers an Open Flag, not a fix — unless the fix is one line and obviously correct.
- **Long runtime expected.** Five subsystem walks plus W7/W8 table merging. Split into multiple iteration branches (`session-272/1`, `session-272/2`, …) if the audit cannot complete in one sitting. Each iteration ships a self-contained slice plus the safe fixes found in it.
- **Tests for new behaviour.** Pure structural extractions or rename-only safe fixes don't need new tests. Anything that *adds* a new seam (won't happen in this audit per scope) would need coverage.
- **When implementation is complete**, run the fast test suite: `docker compose exec app php artisan test --exclude-group=slow`. Target: **271 close baseline ± a small delta**. Slow group: unchanged expected.
- **Front-end builds — only if a safe-fix touches front-end source.** `npm run build` if any `resources/js/**` or `resources/scss/**` change lands; `build:public` if any widget asset surface changes. Most of this session's safe fixes will be PHP / blade — front-end touches should be rare.
- **Manual verification not expected.** Audit output is text (W7/W8 tables, Open Flags block, per-workstream findings); the user reviews the log directly.
