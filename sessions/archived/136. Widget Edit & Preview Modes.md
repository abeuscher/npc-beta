# 136. Widget Edit & Preview Modes

Introduce a two-mode editing experience in the page builder: an **edit mode** where a single widget gets focus and renders live with public styles, and a **preview mode** that shows the full saved page in a read-only iframe with widget selection handles. This replaces the original iframe-per-widget preview plan with a simpler, save-then-render architecture.

---

## Design decisions (resolved before starting)

- **Two modes, not live sync.** Edit mode focuses one widget at a time with live Livewire re-rendering. Preview mode renders the full saved page — no bidirectional config syncing, no postMessage protocol. Exiting edit mode triggers a save; preview always shows persisted state.
- **Edit mode renders the real widget.** The focused widget is rendered via `WidgetRenderer::render()` with the public CSS bundle loaded alongside Filament's admin styles. This is the WYSIWYG surface. Neighboring widgets show as blurred placeholders so they don't need data.
- **Preview mode uses an iframe.** The saved page loads in a lightweight admin-authenticated iframe route — public CSS, modern-normalize, no Filament styles. Widget handles (click targets for entering edit mode) overlay each widget region. This is read-only.
- **CSS `zoom` for focus scaling, not `transform: scale()`.** `zoom` reflows content correctly (text wraps at the right width). `transform: scale()` just visually scales without reflowing, which would misrepresent how text and grids actually render. `zoom` is now supported in all major browsers.
- **Save can trigger a CSS build.** If a widget's CSS/SCSS changed, the save path can invoke `build:public`. Preview mode always loads fresh bundles.
- **Placeholder images via LoremFlickr.** `DemoDataService` returns LoremFlickr URLs instead of `null` for image fields (e.g. `https://loremflickr.com/800/600/nature`). Keywords derived from widget/field context. No API key needed.
- **Widget picker is unchanged.** The picker uses its existing static thumbnail/hover thumbnail fields. No live rendering in the picker.

## Phase 1 — Zoom/focus CSS proof of concept

Prove out the focus/blur/zoom approach before building the full feature. This phase is exploratory — if edge cases emerge that change the approach, stop and discuss.

**Build a prototype that:**
- When a widget block is clicked (selected) in the page builder, it enters "focused" state
- Focused widget: rendered at a comfortable editing zoom level using CSS `zoom`, centered in the available space
- Sibling widgets: visually recede — `filter: blur(4px)` + reduced opacity, non-interactive (`pointer-events: none`)
- Column widget children: verify that focusing a child widget inside a column slot works correctly — the parent column and sibling slots should blur, the focused child should zoom

**Test with:** hero (full-width), text block (simple), column widget with 2+ children (nesting), carousel (collection-bound).

**If `zoom` causes layout problems** (grid blowouts, overflow issues), document what broke and fall back to `transform: scale()` with a fixed-width container that preserves the widget's natural width. Do not proceed to Phase 2 until the zoom approach is validated or an alternative is agreed.

## Phase 2 — Edit mode: live widget rendering

Replace the current block representation (collapsed/expanded form) with a live-rendered widget when in focused/edit state.

- When a block enters focus, call `WidgetRenderer::render()` server-side and inject the output into the block's DOM area
- Load the public CSS bundle (`public/build/widgets/manifest.json` → CSS file) in the admin page alongside Filament styles. Scope it if needed to avoid conflicts (e.g. wrap rendered widget output in a `.widget-preview-scope` container and prefix public styles, or rely on the existing class-based scoping in widget CSS)
- Config changes in the inspector panel trigger Livewire re-render of the focused widget — same mechanism as today, but now the rendered output updates visually
- For collection-bound widgets with no bound collection, use `DemoDataService::generateCollectionData()` as fallback data
- Unfocusing (clicking away, clicking another block, or pressing Escape) exits edit mode for that widget

**CSS isolation strategy:** The public widget styles use `.widget--{handle}` class namespacing. Filament uses its own class conventions. Test for conflicts on: form elements, headings, links, buttons, grid/flex layouts. If conflicts are severe, wrap the rendered widget in a `<div>` with `all: initial` or use a shadow DOM boundary — but try without isolation first.

## Phase 3 — Preview mode: full-page iframe

Build a preview route and integrate it as the builder's "preview mode."

**Preview route:**
- Admin-authenticated GET route: `/admin/preview/{page}` 
- Renders the full page using the public layout and `WidgetRenderer`, same as the public site but with an admin-only overlay
- Each widget region gets a transparent click-target overlay with the widget's label, keyed to its `page_widget.id`
- Clicking a widget handle sends a message to the parent frame (or navigates) to enter edit mode for that widget
- No Filament CSS — this is a clean public-site render

**Builder integration:**
- Add an "Edit / Preview" toggle to the page builder toolbar
- Preview mode: the main content area becomes an iframe loading the preview route
- Edit mode: the current block list with the focused-widget rendering from Phase 2
- Toggling from edit → preview triggers a save (Livewire `save()` call), then loads/reloads the iframe
- Toggling from preview → edit closes the iframe, returns to the block list, optionally focuses the widget that was clicked in the preview

## Phase 4 — LoremFlickr placeholder images

Update `DemoDataService` to return LoremFlickr placeholder URLs for image fields instead of `null`.

**Implementation:**
- Image fields: return a LoremFlickr URL with dimensions appropriate to context (e.g. `800x600` for hero backgrounds, `400x400` for board member photos, `600x400` for general content images)
- Use keywords derived from the widget handle or field key where sensible: `nature` for hero backgrounds, `portrait` for member/people images, `logo` for logo garden, `office` for generic content
- For collection image fields, use the collection handle or a generic keyword
- Video fields: remain `null` (no placeholder video service)
- Update existing tests that assert `null` for image fields
