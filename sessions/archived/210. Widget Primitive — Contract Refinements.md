# 210. Widget Primitive — Contract Refinements

Resolve the four refinements named in session 209's findings — fallback-data semantics, page-context batching, projection extraction, and the richtext-consumer case — before Phase 2 (slot taxonomy) locks the contract shape across the remaining widgets.

This session is the gap between Phase 1 (prototype, session 209) and Phase 2 (slot taxonomy) of the plan in `sessions/widget-primitive-migration.md`. Scope is deliberately narrow: no new widgets migrate, no slot taxonomy starts. The only goal is to clean the contract surface so the next phase inherits a honest shape.

---

## Design decisions (resolved before starting — do not relitigate)

Settled in session 209 findings and scoping for this session:

- **Verdict from 209 was "proceed with refinements."** This session is cleanup, not a do-over. The three migrated widgets (Blog Listing, Text Block, Carousel) keep their contract declarations unchanged at the widget level; refinements are internal to the primitive.
- **No contract-API breakage in scope.** `DataContract`, `ContentType`, and `SlotContext` keep their current public shapes. `ContractResolver::resolve()`'s signature grows one optional parameter (fallback map). That's the only external-facing change.
- **Projector extraction is one class per source type, not per widget.** `PageContextProjector`, `SystemModelProjector`, `WidgetContentTypeProjector`. Each takes a contract + source-specific input and returns the DTO. `ContractResolver` becomes a thin dispatcher. Source-type-level keeps the count bounded; per-widget projectors explode with the widget count.
- **Page-context batching is singleton memoization on `PageContextTokens`.** Bind `PageContextTokens` as a container singleton; add per-page-id memoization to `values()`. Simplest path that eliminates the regression without restructuring the resolver's request lifecycle. The `{{` short-circuit added in 209 stays as belt-and-braces.
- **Richtext-consumer case is named, not re-architected.** Text Block's "declare the full `PageContextTokens::TOKENS` union" pattern is the correct shape for richtext consumers under the current contract. The finding records this as a category observation; this session documents it in the `WidgetDefinition::dataContract()` base-method docstring and in the migration doc. No new `SOURCE_*` constant, no split.
- **Fallback-data merge moves into `ContractResolver`.** `WidgetRenderer` currently pulls from `$collectionData` into `$widgetData['items']` when the contract is empty. The resolver gets an optional `array $fallback = []` parameter; for `SOURCE_WIDGET_CONTENT_TYPE` contracts that resolve to an empty item-set, the resolver merges the fallback *through the same field-projection pass* so fallback data respects the contract's declared fields.
- **Pre-beta license carries over.** No backward-compat shims. If the resolver's signature changes, every call site gets updated.

## Phase 1 — Extract projectors

`ContractResolver` today has one method per source type (`resolvePageContext`, `resolveSystemModel`, `resolveWidgetContentType`) with inline projection logic (e.g. `projectPost` as a private method). Move each to its own class.

- New namespace: `app/WidgetPrimitive/Projectors/`.
- `PageContextProjector` — wraps `PageContextTokens`; takes contract + `?Page` → scalar DTO.
- `SystemModelProjector` — takes contract + pre-fetched Eloquent collection → `['items' => [...]]`. The post-projection (`projectPost`) method moves here.
- `WidgetContentTypeProjector` — takes contract + pre-fetched `CollectionItem` set → `['items' => [...]]` with `_media` where the content type declares image fields.
- `ContractResolver` keeps its source-routing match expression, per-call fetch-dedup cache, and fallback-merge glue (Phase 2 below). It delegates projection to the injected projectors.

Tests: existing `ContractResolverTest` passes unchanged (black-box behaviour preserved). Add at least one direct test per projector demonstrating fail-closed field scoping in isolation.

## Phase 2 — Fallback semantics into the resolver

Move the `$collectionData[$collSlot] → $widgetData['items']` merge out of `WidgetRenderer` and into `ContractResolver`.

- `ContractResolver::resolve(array $contracts, SlotContext $context, array $fallback = []): array`.
- For `SOURCE_WIDGET_CONTENT_TYPE` contracts: if the projector returns empty items and `$fallback[$contract->resourceHandle]` (or a slot-name key) exists, merge the fallback rows through the same `WidgetContentTypeProjector::projectRow()` pass so fallback data respects declared fields.
- `WidgetRenderer` passes `$fallbackCollectionData` (already a parameter) straight through to `resolve()`. The existing per-contract-slot fallback lookup goes away.
- Shape for the fallback-key convention: match by `resourceHandle` first, fall back to `$widgetType->collections[0]` for backward compat with the current `$fallbackCollectionData['slides']` keying. Document the choice in the findings-addendum for this session.

Tests: update the Carousel fallback integration test from session 209 to confirm the fallback row is projected (e.g. an unexpected field in the fallback data does not leak into the DTO).

## Phase 3 — Page-context batching via singleton

Eliminate the per-widget `PageContextTokens::values()` cost on pages with many widgets declaring `SOURCE_PAGE_CONTEXT` contracts.

- Bind `PageContextTokens` as a singleton in `AppServiceProvider` (or wherever the current container bindings live).
- Add a per-page-id memoization cache inside `PageContextTokens::values()`. Key by `$page->id`. First call computes and caches; subsequent calls return the cached map.
- The `{{` short-circuit in `WidgetRenderer` stays — defence in depth.
- Confirm the `buildTree eager-loads owner` query-count test in `PageBuilderApiTest` drops back toward the 208 baseline (≤ 29 queries for the 15-widget scenario; current ceiling is `< 32`). If it does, tighten the test's threshold.

Tests: add a unit test showing `values()` is called once per distinct `$page->id` across N calls. Re-run the `buildTree` query-count test to confirm the improvement and tighten its threshold if it drops meaningfully.

## Phase 4 — Document the richtext-consumer pattern

Name the "declare the union" pattern so Phase 2's widget migrations follow it without relitigation.

- Update the docblock on `WidgetDefinition::dataContract()` with a short note: richtext/text config fields that accept user-authored tokens should declare the full `PageContextTokens::TOKENS` union in their `SOURCE_PAGE_CONTEXT` contract, because the widget cannot know at declaration time which tokens the user will write. Cite Text Block as the precedent.
- Update `widget-primitive-migration.md` with a one-paragraph entry under "Design decisions locked in" capturing the same observation: the fail-closed property applies to *which tokens can substitute*, not to *which tokens the user's markup might reference*.
- No code change beyond the docblock.
