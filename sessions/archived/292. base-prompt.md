## Process rules for every session

Base-prompt process rules apply in full. Session-specific notes:

- **The shared Demo User account is a load-bearing security surface — treat it like the 291 notification invariants.** A public, unauthenticated button that auto-logs anyone into the admin panel as a shared account must be tightly permission-scoped: a dedicated demo role, **not** `super_admin`; no access to settings/secrets/integration keys, no real-PII surfaces, no destructive fleet/backup actions; demo-data only. The session prompt carries these as binding constraints **and resolves the role/permission shape** (the dedicated `demo` role + deny-list under "Resolved decisions") — implement to it; surface only if that decided shape proves genuinely unworkable.
- **Form-less by construction.** Click → server creates/reuses the shared "Demo User" → `Auth::login()` → redirect into the admin panel. No invitation token, no password, no email, no `.env` mode gate beyond the existing demo-server identity check. Mirror existing `Auth::login()` usage; do not invent a new auth path.

- **Anti-abuse is rate-limit per IP, no captcha** (per `[[demo-page-architecture]]`). Mean-actor data is mitigated by the 24h wipe cadence, not by gating. Use the established `throttle:` middleware pattern; pick sane limits and note them.
