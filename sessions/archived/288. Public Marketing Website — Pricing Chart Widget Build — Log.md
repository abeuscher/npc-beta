### Phase 1 — `pricing_chart` widget end-to-end

New widget folder at `app/Widgets/PricingChart/` (definition + template + SCSS + thumbnail). Registered in `WidgetServiceProvider::boot()` and synced into the `widget_types` table by `WidgetTypeSeeder` via the registry's `sync()` call. Schema:

- **Widget-level:** `eyebrow_label` (text), `heading` (text), `subheading` (richtext), `columns` (repeater), `footnote` (richtext), `heading_alignment` (select), `gap` (text). Subheading and footnote are content-group rich text; heading and eyebrow_label are content-group text. The two appearance fields land under `group: 'appearance'` per the existing convention.
- **Per-column (inside the `columns` repeater):** `emphasize` (toggle, default false), `eyebrow` (text), `title` (text), `price` (richtext), `lead_content` (richtext), `attribute_rows` (nested repeater of `label` text + `value` richtext), `ctas` (existing `buttons` field type, full primary / secondary / secondary-dark / text style options).

`defaults()` returns concrete values for every schema key per the project's "concrete values" rule — `columns` defaults to `[]` (empty array), every text/richtext to `''`, the toggle to `false`. `validate()` round-trips clean. `defaultAppearanceConfig()` ships background `#ffffff`, text `#000000`, padding 100/0/100/0 — matches the band-rhythm convention from About / Home.

Demo path: `demoConfig()` returns the marketing-site three-tier configuration verbatim (Instant Demo / 7-Day Trial / Monthly with Monthly emphasized + Recommended eyebrow). The same data is exposed as a public `marketingSiteTiers()` helper method so session 289 can paste it straight into `pricing.json`'s widget config without duplicating the array literal. No `DemoSeeder` (the widget needs no collection); no `demoImages()` (no image fields).

Blade template at `app/Widgets/PricingChart/template.blade.php`:

- Pre-computes the column count and the max attribute-row count across all columns; emits both as CSS custom properties on the root element (`--pc-columns`, `--pc-attr-rows`) so the SCSS can size the parent grid's row tracks deterministically.
- Each card renders six rows in source order: eyebrow / title / price / lead_content / N attribute rows / CTAs. Empty slots (e.g. blank eyebrow on a non-emphasized column) render an empty placeholder div so the subgrid row tracks line up across cards.
- Cards that have fewer attribute rows than the max get padded with empty `pricing-chart__attribute-row--empty` divs so subgrid placement matches across columns and the CTA lands at the same row.
- `data-config-key` hooks omitted — the inspector edits via the Vue repeater field, not in-canvas.

SCSS at `app/Widgets/PricingChart/styles.scss`:

- Parent grid: `display: grid; grid-template-columns: repeat(var(--pc-columns), 1fr); grid-template-rows: auto auto auto auto repeat(var(--pc-attr-rows), auto) auto`.
- Each card: `display: grid; grid-template-rows: subgrid; grid-row: 1 / -1` so it spans all parent rows and inherits row-track sizing.
- Emphasis treatment: background tint `#f8f9fa` on `.pricing-chart__column--emphasized`; border slot built via CSS custom properties (`--pc-border-color`, `--pc-border-width`, `--pc-border-color-emphasized`) all defaulting to transparent / 0 — v1 ships unfilled, v1.1 lights up when `appearance_config` gains a border knob (G-pricing-1 / G19 path).
- Subgrid fallback via `@supports not (grid-template-rows: subgrid)` — older browsers drop to flex-column with `margin-top: auto` on the CTA. 2026 baseline doesn't need it but it's there.
- Mobile breakpoint at `$bp-md`: parent grid collapses to single column, cards stack as block elements, empty placeholder rows hide via `display: none`.

Public asset bundle rebuilt via `docker compose exec app php artisan build:public` — `widget-pricing-chart` styles compile cleanly into `public-widgets-15a956d6.css` (45.7 KB total bundle, +~1 KB from this widget). Manifest at `public/build/widgets/manifest.json` regenerated.

### Phase 2 — Generic `repeater` field type (system extension)

The widget needed a way to express the columns repeater in the inspector. The only existing repeater (the `buttons` field type) is specialized for `{text, url, style}` triples. Decision at session start (logged here as the load-bearing system addition): introduce a **generic `repeater` field type** that any future widget can use for arrays of structured objects.

PHP-side: no `widget_types` schema change. The new field type lives entirely in widget `config_schema` JSON. A repeater field declares its nested-field shape via a `fields: [...]` array (same shape as the top-level schema array — recursive). The lint test `WidgetDefaultsLintTest` was extended with a `case 'repeater':` allowing array defaults; everything else is implicit through the existing `WidgetDefinition::validate()` shape contract.

Vue-side: two new components.

- **`RepeaterField.vue`** at `resources/js/page-builder-vue/components/fields/RepeaterField.vue`. Renders an array of cards, each labeled `{item_label} N`, with up / down / remove icon buttons in the header and an "Add {item_label}" button at the bottom. Composes the standard ButtonListField visual language (same border / padding / button styles) so the inspector reads as one repeater family. Mutation flow mirrors ButtonListField — the field emits the full new array via `update:modelValue` on every change; the parent (the Pinia store via InspectorField) handles the debounced REST save.
- **`RepeaterRowField.vue`** at the same path. One row's worth of fields, dispatching nested-field types to existing field components: `TextField` (text + url), `TextareaField`, `RichTextField` (Quill, full image-upload + heroicon support, scoped to the parent widget), `ToggleField`, `ButtonListField` (so per-column CTAs use the existing buttons field type without re-implementing the buttons UI), and `RepeaterField` itself via `defineAsyncComponent` (lazy import) so nested repeaters work without a circular import.

Registered the new type in `InspectorField.vue`'s `componentMap` as `repeater: RepeaterField`. Widget-development.md not updated this session — it's the canonical reference and the new type belongs in its config-fields table eventually, but a doc edit landed in the user-judgment-not-needed pile and the widget itself is the primary deliverable.

### Phase 4 — Decisions resolved during the build

- **`presets()` not used for the marketing-site tier configuration.** The release-plan entry called for a v1 "Marketing site tiers" preset. `WidgetManifestTest`'s preset rule forbids content-group keys in `preset.config` — and `columns` is content-group. The preset framing collapses cleanly to "use `marketingSiteTiers()` as the column-data source." Same one-click outcome at session 289; different mechanism. Surfaced in the 289 base prompt so the next session knows what to expect.
- **CTA composition reuses the existing `buttons` field type.** Per the open-question section in the 288 prompt — the per-column `ctas` field is declared with `type: 'buttons'`, the existing `ButtonListField.vue` renders it without per-row plumbing. Saved a per-column triplet field in favor of a uniform UX (matches the "uniform UX over per-type plumbing" rule).
- **Nested repeater for `attribute_rows`.** Per the open-question section — `attribute_rows` is itself a `repeater` field nested inside the `columns` repeater. The Vue `RepeaterField` recurses into itself via `defineAsyncComponent`. Cleaner data shape than the alternative (storing rows as structured rich text or a single Quill blob); makes 289's per-row label / value editing trivial.
- **No `widget_types` schema additions.** Per the open-question section — the new `repeater` field type lives in `config_schema` JSON and the Vue inspector's component map. Zero column-level changes to `widget_types`.
- **`columns` defaults to `[]`, not a placeholder column.** Per the "concrete values" rule the empty array IS the concrete value. No magical string sentinels.

## Security considerations

None beyond standard convention. Widget templates render `{!! !!}` only on richtext fields that pass through the existing Quill / HTML sanitizer pipeline (subheading, price, lead_content, attribute value, footnote — all match the existing TextBlock pattern). The new `repeater` field type stores arrays of objects in `config` JSON; the inspector mutates locally and persists via the existing REST endpoint (`PUT /admin/api/page-builder/widgets/{id}`) which already enforces `update_page` permission. No new endpoints, no new file storage paths, no new sanitization surface.

The CTA URL on each column passes through `widget-shared/buttons.blade.php` which already escapes the `href` and applies the external-link `target="_blank" rel="noopener noreferrer"` decision based on host parsing — same path TextBlock CTAs and Hero CTAs use. No new URL handling.

No data migration. No template / chrome-page edits. No site-setting changes. The widget is opt-in — placing it on a page is the only way it appears anywhere.
