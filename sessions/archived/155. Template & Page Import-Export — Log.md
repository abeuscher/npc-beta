## What was built

### Phase 1 — Audit existing code

- Read `TemplateResource` and its `Pages/` directory: confirmed no existing import/export code, no hidden commands, no API endpoints.
- Confirmed `PageWidget::serializeStack()` already encodes column layouts (added in session 153) — no extension required.
- Confirmed the reverse path `CreatePage::hydrateWidgets()` already exists and accepts the same array shape, used as the basis for the importer's hydration code.
- Confirmed Spatie media library exposes `addMediaFromDisk()` and `preservingOriginal()` for path-based reattachment.
- Confirmed `widget_types.handle` is the stable cross-install identifier (used by both `serializeStack` and `hydrateWidgets`).
- Reported findings and paused. User confirmed all design questions:
  - Use existing `update_page` permission for templates too (no new permission, no migration).
  - Drop Phase 7 entirely (server-side exports directory + Saved Exports UI).
  - Filename scheme: `{Ymd-His}-{kind}-{slug-or-N}.json` — timestamp first for natural sort.
  - Preserve chrome page UUIDs across import (round-trip same id).
  - Keep existing `author_id` on overwrite; resolve to first user if no auth context on create.
  - Drop `source_install_id` from envelope (no install id concept yet).

### Phase 2 — Serializer service

- New `App\Services\ImportExport\ContentExporter` (read-only). Methods: `exportPages()`, `exportTemplates()`, `exportBundle()`.
- Walks the page widget+layout tree directly so each widget can carry media descriptors alongside its config.
- Page templates automatically pull their `_header` and `_footer` system pages into the bundle's `pages[]`.
- New `App\Services\ImportExport\ImportLog` — in-memory warning collection (no persistence per session direction).
- New `App\Services\ImportExport\InvalidImportBundleException`.
- Bundle envelope: `{format_version: '1.0.0', exported_at, payload: {templates, pages}}`.

### Phase 3 — Importer service + single-record UI

- New `App\Services\ImportExport\ContentImporter`. Wraps templates-first / pages-second / chrome-relink-third in a single transaction.
- Validates `format_version` and rejects non-`1.x.y` bundles with **zero** DB writes (validation happens before the transaction body).
- Slug-collision overwrite preserves the existing page id and `author_id`. Soft-deleted pages are restored on collision so the import is visible in the default scope (regression caught during manual testing — covered by test).
- New `App\Filament\Actions\ImportBundleAction` — reusable header action with file-upload modal. Used on `ListPages`, `ListPosts`, and `ListTemplates`.
- "Export Page" action on `EditPage`, "Export Post" on `EditPost`, "Export Template" on `EditPageTemplate` and `EditContentTemplate`. (`EditContentTemplate` was added on top of the original Phase 3 spec for round-trip symmetry — content templates had no other UI export path.)

### Phase 4 — Bulk export

- `Export selected` bulk action added inside the existing bulk-action group on `ListPages` (`PageResource::table`) and `ListPosts` (`PostResource::table`).
- Filename: `{Ymd-His}-pages-{N}.json` / `-posts-{N}.json`.

### Phase 5 — Page template chrome export/import

- Already covered by Phase 2 + Phase 3: `ContentExporter::collectChromePageIds()` pulls header/footer system pages into the bundle; `ContentImporter::relinkTemplateChrome()` re-links `header_page_id` / `footer_page_id` after pages are imported. No additional UI work was needed beyond the export action already in place.

### Phase 6 — Media reference rewiring (Tier 1)

- Implemented inline in `ContentImporter::rewireWidgetMedia()` rather than as a separate phase.
- For each media descriptor, looks up the file at `{disk}/{path}`, calls `addMediaFromDisk()->preservingOriginal()->toMediaCollection()`, and patches the widget config to point at the new media id.
- Falls back to a logged warning + null media when the source file is missing (widget renders its empty state on the public site).
- Defence in depth: refuses descriptors containing `..` or absolute paths.

### Phase 7 — DROPPED

- Per user direction during Phase 1 audit. The Saved Exports UI / `storage/app/exports/` directory is not built. The user accepts manual file shuffle for the round-trip workflow.

### Phase 8 — Tests

- New `tests/Feature/ContentImportExportTest.php` with 17 cases:
  - Round-trip: text widgets, column layout, media reference (with `Storage::fake('public')`), bulk page bundle, page template with chrome and header/footer system pages.
  - Slug collision: overwrite preserves id; soft-deleted page is restored on import.
  - Fallbacks: missing collection_handle cleared + warning, missing template_name falls back to default + warning, missing media file leaves null + warning.
  - Format version: rejects future major (`2.0.0`), accepts minor bump (`1.5.3`), rejects malformed.
  - Authorization gate via `ImportBundleAction::isVisible()`.
  - Author resolution: authenticated user on create, preserved on overwrite.

## Security checklist — verified

- [x] Import/export gated on `update_page` (action visibility + `abort_unless` inside callbacks).
- [x] File upload validates JSON content types and a 50 MB cap.
- [x] Bundle envelope is parsed and validated before any DB writes; rejection produces zero partial state (transaction-wrapped after validation).
- [x] No raw SQL — Eloquent / query builder throughout.
- [x] No XSS introduced — `head_snippet` / `body_snippet` are stored as-is (already raw HTML by design) and only writeable by users with `update_page`.
- [x] Path-traversal defence on media descriptors (`..` and absolute paths refused).
- [x] Importer never reads files outside the configured media disk root.
- [x] Phase 7 dropped — no server-side exports directory was created, so the public-disk concern is moot.
