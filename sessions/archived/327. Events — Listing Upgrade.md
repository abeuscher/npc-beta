# 327. Events — Listing Upgrade

Bring the events index up to its v1 shape: a day-grouped list mode alongside the existing grid, a filter row, a featured-event hero, and an art-fallback chain — additive to the existing `EventsListing` widget, with rows linking to the landing page every event now has.

## Session-specific deltas

- **Additive to `EventsListing`.** Everything here is a new **preset / mode** on the existing widget, selectable in the inspector. The current grid/swiper layout stays the default and keeps rendering unchanged on existing pages.
- **Every event has a landing page now (326).** Rows link to `events.landingPage->slug`; there is no no-LP branch to handle. This is the dependency that moved this session after 326.
- **The "Event Type" filter and the "event-type default" art tier assume a taxonomy the app does not have yet.** Events carry no `event_type`/`category` column today — only `recurrence_type` and the tag system. Resolving how to categorise is the central open question (see below); it may or may not add a column.

---

## What ships (per §1a v1 cut)

- **Day-grouped list mode** — the "luma" vertical shape: events grouped under day headings, each row showing time / title / host / location / thumbnail (host + location optional). A new layout preset beside the existing grid/swiper.
- **Filter row** — two controls: **Event Type** + **Date Range**. No free-text search.
- **Featured-event hero** — one featured event surfaced above the list.
- **Art-fallback chain** — event image → (event-type default, *pending the taxonomy decision*) → **site default thumbnail** → solid colour. **No generated text-tile** (the demo owns its seed art).
- **Status badges** — Members Only / Waitlist / Sold Out / Free / Registered (partly modelled already; surface what exists).
- **Empty / sparse states** — the demo will hit these; render them deliberately.
- **`ItemList` / `Event` schema + canonical + OG** — low cost, high credibility.
- **`data-tour` anchors** — `events-index.list`, `events-index.featured`, `events-index.filters` (§2c). No tour code this session.

## Open questions to resolve at session start

These genuinely change the build — settle at the open gate (recommendations in the base prompt):

1. **Event Type filter source** — reuse **tags** (no schema) vs a first-class `event_type` column (migration). *Lean: tags.*
2. **Art-fallback "event-type default" tier** — no per-type default images exist; drop that tier (event image → site default → solid colour) or tie it to the tag/type decision. *Lean: drop the tier for v1.*
3. **"Featured" selection** — an operator flag (migration) vs an implicit rule (next upcoming / first in sort). *Lean: implicit rule.*
4. **Date Range filter shape** — preset ranges (Upcoming / This month / Past) vs a date picker. *Lean: presets.*

The Event Type + Date Range controls (per the resolved taxonomy + range decisions), wired through the widget's existing query plumbing. The featured-event hero above the list (per the resolved featured rule). `data-tour` anchors for both.

### Phase 3 — Art-fallback + schema + states

The art-fallback chain (per the resolved tiers), the `ItemList`/`Event` schema + canonical + OG, status badges, and the empty/sparse states.
