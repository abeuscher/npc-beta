# 330 — Widget Styling Boundary Audit (Phase 1 findings)

> **Status:** Phase 1 audit. No application code written. This is the alignment
> artifact for the Phase 1 → Phase 2 pause. Findings are surfaced *inclusively*
> per the base prompt — "won't-fix" is decided in the walkthrough below, not by
> filtering here.

Surveyed: **all** `app/Widgets/*/*.scss` (14 files), the host seam
(`AppearanceStyleComposer`, `page-widgets.blade.php`, `PageBlockRenderer`), the
host SCSS partials (`_variables`/`_base`/`_layout`), and the bundle
concatenation order (`AssetBuildService::collectSources`).

**Headline:** the leak surface is genuinely small and bounded — smaller than the
spec's completionist framing implies. There is **no** type-token leak and **no**
host-var leak beyond `$gutter` (one widget) and `$bp-*` (the breakpoint media
queries). The only non-trivial decision is the `@media → @container` migration,
which is **not purely behavior-preserving** for column-placed widgets — see
Matrix 3 and the Key Decisions section.

---

## How the boundary works today (confirmed against live code)

- Every widget renders inside a host-owned wrapper `<div class="widget widget--{handle}" id="widget-{id}" style="{composed}">`, emitted by `page-widgets.blade.php` (top-level) and `PageBlockRenderer` (column children). The inline style is composed entirely by `AppearanceStyleComposer` from `appearance_config` (margin/padding/border/bg/text-colour/full-width). That seam is clean and matches the spec's diagram.
- Bundle order (`collectSources`): host partials (`_variables` → `_base` → `_layout` → `_grid` → `_forms` → …) → `theme/public.scss` → **each widget's SCSS last**, with `_variables.scss` re-inlined ahead of every widget file (this is how `$bp-*`/`$gutter` resolve inside widget SCSS — Sass literals, not runtime).
- **Precedence today** is therefore: a widget rule beats a host rule of *equal* specificity (source order), but **loses** to any host base rule of *higher* specificity. That asymmetry is the cascade fragility (Matrix 4).
- `.page-layout` already establishes `container-type: inline-size` and collapses columns via `@container (max-width: $bp-md)` (`_layout.scss:57,65`). The widget-interior model extends that one level down.

## Matrix 1 — Spacing leaks (`$gutter`)

| Widget | Lines | What exists | What's wrong | Lightest fix | Behavior-preserving? |
|---|---|---|---|---|---|
| `ThreeBuckets` | `styles.scss:6,10` | `gap: var(--bucket-gap, $gutter)` and `calc(... var(--bucket-gap, $gutter) ...)` | Reads host Sass `$gutter` as the fallback — the one real spacing leak in the corpus | Replace `$gutter` with the literal `1.5rem` (its compiled value). Widget owns its own fallback; `--bucket-gap` instance override path unchanged. | **Yes** — `$gutter` compiles to `1.5rem` today, so output is byte-identical. |

**No `--np-space-*` scale.** Only `$gutter: 1.5rem` exists in the host; exactly one widget reaches it; the fallback is already a `var()`. Building a spacing-token family would be inventing a system nothing needs (per the base-prompt rule). Recommended: widget owns the literal. The enforcement test then bans bare `$gutter` in widget SCSS.

## Matrix 2 — Type leaks

**Empty.** No widget SCSS reaches `$font-*`, `$radius`, `$shadow`, `$input-*`, `$container-*`, or any host type Sass var (grep-confirmed across all 14 files). Widgets carry their own `font-size`/`line-height`/`font-weight` literals on their *interior* elements — that is the author's playground, not a boundary crossing. `TypographyCompiler` applies type as compiled `.np-site` rules that widgets inherit; nothing leaks.

**Recommendation: no type-token vocabulary this session.** The matrix proves the negative; that is its whole value. The enforcement test *may* still add a guard banning `$font-*`/host type vars in widget SCSS (cheap, prevents future leaks), but no migration work exists.

## Matrix 3 — Breakpoints (viewport `@media` in widget interiors)

The spec flagged 6 widgets reaching `$bp-*`. **Confirmed, plus the survey found 3 more `@media` users the spec's snapshot missed.**

| Widget | Lines | Query | Placement (full-width vs column-capable) |
|---|---|---|---|
| `ThreeBuckets` | `:38` | `@media (max-width: $bp-md)` | typically full-width section |
| `ProductCarousel` | `:135` | `@media (max-width: $bp-sm)` | typically full-width section |
| `PricingChart` | `:169` | `@media (max-width: $bp-md)` | typically full-width section |
| `MapEmbed` | `:46` | `@media (max-width: $bp-sm)` | column-capable |
| `LogoGarden` | `:42,46,84,88` | `@media (max-width: $bp-md / $bp-sm)` (×4) | column-capable |
| `BoardMembers` | `:127,138` | `@media (max-width: $bp-md / $bp-sm)` (×2) | column-capable |
| **`EventsListing`** *(not in spec)* | `:147` | `@media (max-width: 48rem)` — **hardcoded literal**, not `$bp-*` | column-capable |
| **`EventMiniCalendar`** *(not in spec)* | `:15` | `@media (max-width: 1023px)` — **hardcoded literal**; hides the widget on narrow viewports (rail collapse) | right-rail only |
| **`Nav`** *(not in spec; special case)* | `template.blade.php:146,150` | inline `<style>` `@media` against a **runtime** `$mobileBreakpoint` (operator-configured, interpolated into Blade) | site chrome |

**What's wrong:** widget responsive behavior keys off the *viewport*, not the widget's own width. A widget dropped into a narrow column on a wide screen never collapses (it squishes / overflows — the exact `.layout-column { min-width: 0 }` blowout class). `EventMiniCalendar`/`EventsListing` additionally hardcode breakpoint literals (`1023px`, `48rem`) instead of the published `$bp-*` constants — host/widget drift.

**Lightest fix (architecture):** host wrapper `.widget` gets a named container (`container-type: inline-size; container-name: <name>`); each widget's `@media (max-width: X)` becomes `@container <name> (max-width: X)`.

**Behavior-preserving? — This is the one place the answer is NOT a clean yes:**

- For a widget that is **always full-width**, container width ≈ viewport width, so collapse fires at ~the same point → effectively behavior-preserving (modulo container padding).
- For a **column-capable** widget, `@container` deliberately **changes when collapse fires** (it now collapses based on its own narrow width, not the viewport). That is an *improvement* — and it is exactly the responsive/mobile-appearance behavior the session prompt scopes to the **later design pass**, not this one.
- `Nav` is a genuinely different case: its breakpoint is a **runtime operator value** interpolated into an inline `<style>`, and a media/container *condition* cannot read a `var()`. It is chrome, not a column-placed interior. **Recommend leaving Nav out of the `@container` rule entirely** and documenting it as the explicit exception.
- `EventMiniCalendar:15` is `display:none` on viewport collapse (it hides when the *rail* would collapse). That is genuinely a *viewport/layout* concern, not an own-width concern — converting it to `@container` would change its meaning. **Recommend leaving it as `@media`** (or, if banned, carving it as a reviewed exception).

**Behavior-preservation risk on the host side too:** adding `container-type: inline-size` to a wrapper applies `contain: layout style inline-size`, which establishes a **new containing block for absolutely-positioned descendants and a new stacking context**. Any widget with `position:absolute`/`z-index` resolving against an *ancestor above the wrapper* could shift. Lowest-risk approach: apply the named container **only to wrappers of widgets that declare interior `@container` queries** (a marker class), not universally — uniform-but-risky vs targeted-but-safe is an alignment call.

## Matrix 4 — Specificity / cascade (the `@layer` question)

**What exists:** widget precedence over the host relies on (a) source order (widget SCSS concatenated last) for equal-specificity ties, and (b) deeper host base selectors winning where they're more specific. The fragility is visible in live code:

- `.np-site ul:not(nav ul) li` (host base, `_base.scss`, specificity 0,2,2) forces a disc marker on every `<li>` and **out-specifies** a widget's plain `list-style: none`. `EventMiniCalendar` (`:183`) and `EventsListing` (`:255`) both work around this by dropping rows to `display:block` — explicit, commented acknowledgements of fighting host base.
- The global form rule `select { width: 100% }` out-specifies `width:auto`; `EventsListing` (`:42`) works around it with `flex: 0 0 14rem`.
- `_base.scss` itself fights specificity internally (the `button.btn` / `a.btn` carve-outs) — the same well.

**Why it matters for widget autonomy:** all of this holds only because we control every widget and the concatenation order. A third-party widget author writing a deeper selector, or loaded in a different order, breaks unpredictably.

**`@layer` assessment:** putting host reset/base in a lower layer and widget interiors in a `widgets` layer makes author precedence depend on **layer order, not selector depth** — the `display:block` and `flex-basis` workarounds could retire, and the deliberate inherited-text-colour reach-in is preserved.

**Blast radius (why I recommend deferring the *ship*, not the assessment):** any **unlayered** rule beats **every layered** rule. So `@layer` is only safe if *all* host CSS adopts layers in one coordinated pass (`_base`, `_layout`, `_grid`, `_forms`, `_buttons`, `theme/public.scss`, the typography/colour-token emit, and per-template `custom_scss`). A partial rollout silently flips precedence. That is not a behavior-preserving change verifiable by a quick visual pass, and it is the single highest-risk item in the whole arc.

**Recommendation:** include the assessment (this section) in the audit; **defer shipping `@layer` to its own dedicated session** with its own before/after byte-diff and visual gate. It does not block the spacing/breakpoint/doc/enforcement work.

## Matrix 5 — General host-var leaks

Beyond Matrices 1–3: **none.** Grep across all widget SCSS for `$(font|radius|shadow|input|color-|container-)*` returns zero hits. `$color-*` leaks are already closed and gated by `WidgetColorTokenConsumptionTest`. `PricingChart`'s `--pc-*`-over-`--np-color-*` namespace (`styles.scss:1-14`) is the reference-clean target shape — local CSS vars that read published `--np-color-*` tokens, no Sass-var reach-in. `LogoGarden`/`BoardMembers` already follow it (`--logo-*`, `--bm-*`).

**B. Architecturally right but NOT cleanly behavior-preserving — needs an explicit call:**
4. `@media → @container` for the column-capable widgets (`MapEmbed`, `LogoGarden`, `BoardMembers`, `EventsListing`) + host named-container on their wrappers. **Changes collapse behavior in columns** → arguably belongs to the mobile-appearance design pass. The full-width-only widgets (`ThreeBuckets`, `ProductCarousel`, `PricingChart`) convert with near-zero visible change and could go in this session behind a visual check.
5. `EventsListing:147` / `EventMiniCalendar:15` hardcoded literals → at minimum normalize to `$bp-*` constants (behavior-preserving); the `@container` conversion is the same Matrix-3 call.

**C. Defer to its own session — recommend OUT of 330:**
6. `@layer` cascade isolation (Matrix 4). Highest blast radius; coordinated all-host rollout; not cheaply verifiable.
7. `Nav` runtime-breakpoint inline `<style>` — leave as-is; document as the explicit exception.

## Key decisions for the pause

1. **`@media → @container` scope.** Three honest options: (a) **full-width widgets only** this session (near-zero visual change, low risk) + defer column-capable ones to the design pass; (b) **all 6–8** this session, accepting the column-case collapse change as a deliberate improvement gated behind your visual check; (c) **defer the entire `@container` migration** to the mobile-appearance pass and ship only fix-list A this session. My lean: **(a)** — it keeps 330 behavior-preserving and still proves the container-query mechanism end-to-end.
2. **Named-container application.** Targeted (only wrappers of widgets with interior queries, via a marker) vs universal (`container-type` on every `.widget`). My lean: **targeted** — universal carries the containing-block/stacking blast radius for zero benefit on widgets that don't query.
3. **`@layer` now or its own session.** My lean: **its own session** (assessment delivered here).
4. **Enforcement breadth.** Ban `$gutter` for sure; also ban viewport `@media` in widget SCSS *only for the families we actually migrate* (a baseline-list exception for `Nav`/`EventMiniCalendar` if they stay `@media`), mirroring how the colour test carries 5 reviewed exceptions.
5. **Contract doc.** Refresh `widget-development.md` for sure; create `docs/widget-styling-contract.md` only if you want the published-contract artifact now vs at the `@layer`/external-author session.
