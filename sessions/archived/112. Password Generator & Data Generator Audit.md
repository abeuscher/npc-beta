# 112. Password Generator & Data Generator Audit

Two small, independent improvements: a secure password generator for the admin user form, and a cleanup pass on the dev dashboard's random data generator to align it with current content types.

---

## Design decisions (resolved before starting)

- Password generator is Alpine.js only — no server round-trip. Crypto-random string, fills both password fields, copies to clipboard.
- Financial record types (donations, purchases, transactions) are removed from the data generator entirely. Financial data should only enter the system through the real Stripe checkout/webhook flow or manual admin entry.
- Products remain in the generator — they are catalog items, not financial records.

---

## Phase 1 — Secure password generator on user create/edit form

Add a "Generate password" button adjacent to the password field on the admin user create/edit form (`UserResource`).

### Behaviour

1. On click, generate a cryptographically random password (20 characters, mixed case + digits + symbols).
2. Fill both the **Password** and **Confirm Password** fields with the generated value.
3. Copy the generated password to the clipboard.
4. Show a brief "Copied to clipboard" notification (Filament toast or inline flash — whichever is simpler).

### Implementation notes

- Use `crypto.getRandomValues()` in Alpine.js — no PHP or Livewire round-trip.
- Use the Clipboard API (`navigator.clipboard.writeText()`).
- The button should appear on both create and edit forms, next to or below the password field.
- Read the existing `UserResource` form before implementing.

## Phase 2 — Data generator audit

Update `DashboardDebugGeneratorWidget` and its Blade view to reflect the current state of content types.

### 2a — Remove financial record types

Remove the following from the generator and its wipe function:

- **Donations** — remove `generateDonations()`, `wipeDonations()`, and the `donations` option from the type dropdown.
- **Purchases** — remove `generatePurchases()`, `wipePurchases()`, and the `purchases` option from the type dropdown.
- Remove the **Tax Year** input from the Blade view (it was only used by donations).
- Remove unused model imports (`Donation`, `Transaction`, `Purchase`, `ProductPrice`).
- **Products stay** — they are catalog data, not financial records.

### 2b — Improve event generation

Update `generateEvents()` so that generated events vary in price and capacity:

- ~50% of events should be free (`price = 0`), ~50% should have a random price ($5–$100, rounded to whole dollars).
- ~60% of events should have unlimited capacity (`capacity = null`), ~40% should have a random capacity (20–200).
- For each **free** event, generate 0–N random registrations (where N is a random number up to the lesser of 30 or the event's capacity if set). Use existing contacts if available; fall back to inline name/email from Faker.
- Paid events should not get registrations (those require Stripe checkout).

### 2c — Improve blog post generation

Update `generateBlogPosts()` to create posts with two page widgets:

1. **`text_block`** widget — with `config.content` set to 2–4 paragraphs of lorem ipsum HTML (wrapped in `<p>` tags).
2. **`blog_pager`** widget — with empty config (navigation between posts).

Look up the `widget_type_id` for each handle from the `widget_types` table at generation time. Insert `page_widgets` rows with correct `page_id`, `widget_type_id`, `config`, and `sort_order` (text_block first, blog_pager second).

### 2d — Wipe cleanup

Update the wipe functions to match:

- Remove `wipeDonations()` and `wipePurchases()`.
- `wipeGeneric()` for events should also delete associated `event_registrations` before deleting events.
- `wipeBlogPosts()` should continue to work — `page_widgets` will cascade-delete when the page is force-deleted (FK cascade rule).
