# 341. Widget Schema Sync in the Deploy Path

Add an idempotent `widgets:sync` artisan command and run it in every deploy/upgrade path after migrations, so pure-schema widget changes that ship in the image actually land on already-installed servers. The deploy-pipeline half of the ⭐ owner-designated pair (the chrome-render half shipped at 340).

---

## Stub reference

Not a `release-plan.md` entry. Executes the ⭐ `[0.324.01]` inbox item. Root cause (verify it still holds at session start):

The page builder's control list **and** the public render both read a widget's fields from the `widget_types.config_schema` DB column (`PageBuilderApiController::widgetTypes` → `WidgetType::forPicker`; `WidgetResource`; `WidgetRenderer`) — **not** from the live `WidgetDefinition` class at request time. That column is (re)written only by `WidgetRegistry::sync()`, which is called only inside `WidgetTypeSeeder::run()`. An FM upgrade / `deploy.yml` / the documented droplet upgrade runs `migrate --force` + `build:public` and **explicitly never runs the seeder**. Net: any *pure-schema* widget change (no migration — that's why it's confusing; there genuinely isn't one) ships in the image but stays invisible on installed servers. First bites: s322's Nav "Layout"/`orientation` control (E16) and the s322 seeded columns footer.

## The work

### 1. The command

Add `app/Console/Commands/SyncWidgets.php` (or the established naming) — signature `widgets:sync` — that calls `app(\App\Services\WidgetRegistry::class)->sync()` and prints a one-line summary (count synced). Idempotent by construction (`sync()` is `updateOrCreate` by handle). Mirror an existing console command's structure/output style.

### 2. Wire it into every deploy path, after `migrate --force`

- `.github/workflows/deploy.yml` — add the `widgets:sync` step right after the migrate step, before `build:public`.
- `.github/workflows/deploy-demo.yml` — same insertion (SHA-tagged / VERSION-exempt, but the sync gap applies equally).
- `docs/deployment/*` — the droplet / FM upgrade procedure (the one documenting "the workflow does NOT run the seeder"): add `widgets:sync` to the documented upgrade steps so a hand-run upgrade picks it up too.
- Confirm whether the FM-side upgrade procedure needs a coordinating pointer (the inbox note says findings were already passed to the FM-side agent — a pointer in `sessions/session-outlines.md`'s Cross-Repo block may suffice; no contract field changes).

### 3. Test

A small Pest case proving the command path syncs definitions into `widget_types` — e.g. mutate/remove a row's `config_schema`, run the command, assert the row matches the live definition again (idempotent `updateOrCreate`). Service-layer, deterministic, fast.

### 4. Document the manual unstick

Record (log + the deployment doc) the immediate fix for an already-affected node, so an operator isn't stuck waiting for the next deploy:
`docker exec nonprofitcrm_app php artisan db:seed --class=WidgetTypeSeeder --force`
