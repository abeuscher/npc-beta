## What was built

### Phase 1 — Data model

- Migration: `event_registrations` gains `ticket_type`, `ticket_fee`, `payment_state`, `transaction_id` FK, `import_session_id` FK, `custom_fields` (jsonb).
- Migration: `transactions` gains `import_source_id` FK, `import_session_id` FK, `external_id`, `payment_method`, `payment_channel`, plus composite index on `(import_source_id, external_id)`.
- Migration: `import_sources` gains events-scoped parallel columns (`events_field_map`, `events_custom_field_map`, `events_match_key`, `events_match_key_column`, `events_contact_match_key`) — per-content-type preset storage without touching the contacts preset.
- Migration: `import_logs` gains `contact_match_key` for the events importer's separate contact-lookup key.
- Migration: `events` gains `import_session_id` FK for rollback cascading.
- Models updated: Event, EventRegistration, Transaction, ImportSource, ImportLog — new fillable, casts, and relationships.

### Phase 2 — Field registries

- `EventFieldRegistry` — derived from Event::$fillable with excludes + type overrides, mirrors ContactFieldRegistry.
- `RegistrationFieldRegistry` — hand-curated: ticket_type, ticket_fee, status, payment_state, registered_at, notes.
- `TransactionFieldRegistry` — external_id (Transaction ID), amount, payment_state, payment_method, payment_channel, occurred_at (Paid At).
- `EventImportFieldRegistry` — aggregates all four registries into namespaced grouped Filament options (`event:*`, `registration:*`, `contact:*`, `transaction:*`) plus relational sentinels.

### Phase 3 — ImportEventsPage wizard

- Three-step wizard (Upload → Map Columns → Preview & Confirm), sibling of ImportContactsPage.
- Source presets use `events_*` columns on ImportSource, fully independent of contacts preset.
- Two match-key selects: "Match events by" (event:external_id default) and "Match contacts by" (smart default derived from mapped contact:* columns).
- Heuristic guesser for WCG / Wild Apricot events headers.
- Grouped options in per-column dropdowns with Event fields, Registration fields, Contact match, Transaction fields, and Other (custom event/registration, tag event/contact, note contact, org contact).

### Phase 4 — ImportEventsProgressPage row processor

- Full dry-run → awaiting-decision → commit lifecycle, same pattern as contacts.
- Per-row pipeline: resolve Event (by external ID via ImportIdMap, create if not found) → resolve Contact (email or external_id, case-insensitive email) → create Registration → upsert Transaction (fill-blanks-only on match) → timeline note → per-row tags/notes → contact org (fill-blanks-only).
- Entity preview in dry-run report: events (new/reused), registrations (new), transactions (new/reused).
- Skip-reason tracking with "Why rows would skip" panel: blank_event_id, blank_contact_key, contact_not_found.
- PII pre-scan runs before dry-run.
- Save-mapping persists events-scoped preset to ImportSource without touching contacts columns.

### Phase 5 — UI surface

- Importer landing page: "Import Events" card wired up (was a stub), blocked when a pending events session exists.
- ImporterPage review queue: event-session branch in preview modal (events + registrations + transactions tables), approve (flips status), rollback (cascades: registrations → transactions → events → idmaps).
- Delete action on ImporterPage: works for any session status (pending/reviewing/approved). Same cascade as rollback. Status and type filters added to the table.
- Import History: already handles `model_type='event'` via existing badge column.
- Events review preview blade with events, registrations, and transactions tables.

### Phase 6 — UAT polish (surfaced during testing)

- **Case-insensitive email lookup**: `resolveContact` uses `LOWER(email) = LOWER(?)` to handle casing divergence between source CSVs.
- **Event tags**: `__tag_event__` sentinel with optional delimiter. Applies tags (type='event') to the Event via the existing morphToMany taggable relation. WCG "Event tags" column works directly.
- **Contact organization (fill-blanks-only)**: `__org_contact__` sentinel with auto_create/match_only/as_custom strategy. Sets `Contact.organization_id` only when null — never overwrites. Org creation skipped entirely for contacts already linked.
- **Null-filter on contact creation**: filtered null values from `Contact::create()` attributes so NOT NULL columns with DB defaults (e.g. `mailing_list_opt_in`) use their defaults instead of crashing on explicit null.
- **Smart contact match key default**: `deriveContactMatchKey()` picks from actually-mapped `contact:*` columns instead of a static `'contact:email'` default that could mismatch the available options.
- **Organizations overhaul stub**: added to `session-outlines.md` under pre-Beta 1 — defines the four org-role cases (employer, sponsor, host, registrant) and the need for a proper model session.
- **Update-existing strategy for events**: documented in session 191's stub as a hard requirement — every client has overlapping import cycles during migration, re-import-to-update is the default customer cadence.
- **Close gate template updated**: `template-base-prompt.md` now has Phase 1 (attenuate + next session prompt) and Phase 2 (close) as separate, explicit steps.
