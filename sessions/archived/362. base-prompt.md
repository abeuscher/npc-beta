## Process rules for every session

Base-prompt process rules apply in full. Session-specific:

- **Surface the architectural choices at the Open Gate, build after.** The **navigation model for the CRM/CMS tours** (single-page vs contained navigation — the load-bearing one), **the source of the three tour scripts** (owner-provided vs authored in-session), **the launcher shape**, **whether the persistent bar stays**, **how much of the multi-page engine / resume / URL map to remove**, and **role-gating / showcase retention** all propagate or need owner taste — confirm the recommended leans (in the session prompt's Open-Questions section) before going deep, don't commit-and-unwind.
- **The owner drives step content; the session drives mechanics + verification.** The step set and copy come from the owner's three outlines/scripts. Bring a **Playwright storyboard** (a screenshot per step) and get copy approval before editing copy.
- **Anchor every step on eager, stable, in-view content (the 361 lesson).** A tour anchor must resolve to something in the DOM and geometrically settled when the step fires. Lazy relation-managers, deferred widgets, and post-hydration growth are the enemy — prefer eager, high-on-the-page anchors (apply `$isLazy = false` where needed) over reactive scroll-chasing.
- **Keep the retirement clean.** Removing the multi-page walkthrough means no dead render hooks, orphaned showcase pages/routes, or `data-tour` markers left injected for anchors no tour uses. Keep the reusable primitives (`anchors.js`, the driver wrapper, the `data-np-tour-start` launcher).
- **Anchors stay owned `data-tour` attributes**, never Filament `.fi-*` classes (the session-249 selector-fragility mitigation, reaffirmed at 361 when the Memberships table was marked via a scoped `RESOURCE_RELATION_MANAGER_BEFORE` render hook, not its `.fi-*` class).
- **Front-end build is mandatory before announcing completion.** `npm run build` (WSL-path npm per agent memory `reference-frontend-build-command`); rebuild + push the bundle into the e2e stack's `public/build` volume for Playwright, or the spec runs against a stale tour.
- **Pest and Playwright run sequentially, never in parallel** (agent memory `feedback-test-runs-not-parallel`).
- **If the owner signals a deploy-to-test,** bump `VERSION` to `0.362.<iteration>` proactively (agent memory `feedback-deploy-version-bump`) and flag it.
