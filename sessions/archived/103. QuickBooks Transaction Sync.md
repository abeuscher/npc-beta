# 103. QuickBooks Transaction Sync

One-way push of completed transactions from the CRM to QuickBooks Online as sales receipts. Core sync path, error surfacing, and admin visibility only ‚Äî advanced account mapping and reconciliation views are deferred.

---

## Design decisions (resolved before starting)

- **Sync direction:** one-way push only ‚Äî CRM is the source of truth. No reads from QuickBooks.
- **QuickBooks entity type:** Sales Receipts (not journal entries). Sales Receipts are the natural fit for completed payments received via Stripe. Each synced transaction becomes one Sales Receipt in QuickBooks.
- **Account mapping:** fixed mapping for Beta 1. All synced transactions post to a single QuickBooks income account. The account is selected by the admin on the Finance Settings page (a dropdown populated from the QuickBooks Chart of Accounts API). Store the selected account ID as a site setting (`qb_income_account_id`). If no account is selected, sync is disabled with a clear message.
- **Sync trigger:** automatic, via a queued job dispatched when a transaction reaches `completed` status. The Stripe webhook handler already creates/updates transactions ‚Äî hook into that flow.
- **Idempotency:** the `transactions.quickbooks_id` column (already exists) stores the QuickBooks Sales Receipt ID after successful sync. A transaction with a non-null `quickbooks_id` is never synced again.
- **Retry & error handling:** failed syncs are logged but do not block the webhook response. A retry mechanism is not needed for Beta 1 ‚Äî the admin can trigger a manual re-sync from the UI.
- **Refunds:** when a `refund.created` webhook fires and creates an outbound transaction, sync a Refund Receipt to QuickBooks. Same idempotency rule ‚Äî check `quickbooks_id` before syncing.
- **QuickBooks SDK:** the `quickbooks/v3-php-sdk` package is already installed (session 102). Use it for API calls.
- **Token refresh:** `QuickBooksAuth::refreshTokenIfNeeded()` is already built (session 102). Call it before every API request.

## Phase 1 ‚Äî QuickBooks Sync Service

### QuickBooks API client

Create `app/Services/QuickBooks/QuickBooksClient.php`:

- Constructor accepts no arguments ‚Äî reads credentials from `QuickBooksAuth` and site settings.
- Before every API call, call `QuickBooksAuth::refreshTokenIfNeeded()`. If it returns false, throw a descriptive exception (token expired / not connected).
- `createSalesReceipt(Transaction $transaction): string` ‚Äî creates a Sales Receipt in QuickBooks for the given transaction. Returns the QuickBooks Sales Receipt ID. The receipt should include: amount, date (`occurred_at`), a line item referencing the income account from `qb_income_account_id`, and a memo with the transaction type and subject description (e.g. "Donation ‚Äî $50.00" or "Product Purchase ‚Äî Widget Kit").
- `createRefundReceipt(Transaction $transaction): string` ‚Äî creates a Refund Receipt in QuickBooks for outbound (refund) transactions. Returns the QuickBooks Refund Receipt ID.
- `getIncomeAccounts(): array` ‚Äî fetches the Chart of Accounts filtered to Income type. Returns an array of `['id' => 'name']` pairs for the settings dropdown.

### Sync job

Create `app/Jobs/SyncTransactionToQuickBooks.php`:

- Queued job (uses the default Redis queue).
- Accepts a `Transaction` model.
- Skips silently if: QuickBooks is not connected, no income account is configured, or the transaction already has a `quickbooks_id`.
- For `direction = 'in'` (payments): calls `createSalesReceipt()`.
- For `direction = 'out'` (refunds): calls `createRefundReceipt()`.
- On success: stores the returned QuickBooks ID in `transactions.quickbooks_id`.
- On failure: logs the error. Sets a `qb_sync_error` column on the transaction (see Phase 2 migration). Does not retry automatically.

## Phase 2 ‚Äî Schema & Webhook Integration

### Migration

Add to the `transactions` table:

- `qb_sync_error` ‚Äî nullable text column. Stores the error message from the last failed sync attempt. Cleared on successful sync.
- `qb_synced_at` ‚Äî nullable timestamp. Set when sync succeeds.

Update `docs/schema.md` after writing the migration.

### Webhook integration

In `StripeWebhookController`, after a transaction is created with `status = 'completed'`:

- Dispatch `SyncTransactionToQuickBooks` for the newly created transaction.
- This applies to: `handleDonationCheckout`, `handleInvoicePaymentSucceeded`, `handleProductPurchase`, and `handleRefundCreated`.

Do not dispatch for failed transactions.

## Phase 3 ‚Äî Finance Settings UI

### Income account selector

Add a "QuickBooks Sync" section to the Finance Settings page, below the QuickBooks Connection section. Only visible when QuickBooks is connected.

- A dropdown populated by calling `QuickBooksClient::getIncomeAccounts()`. Cached for 1 hour to avoid repeated API calls during page loads.
- The selected account is stored as site setting `qb_income_account_id`.
- A "Refresh accounts" button clears the cache and reloads the dropdown.
- Helper text: "All synced transactions will be posted to this income account in QuickBooks."

### Sync status visibility

Add a "QuickBooks" column to the Transactions table in the admin (if a transactions list view exists) or add sync status to the transaction detail. Show one of:

- Synced (with timestamp)
- Error (with error message, truncated)
- Pending (no `quickbooks_id`, no error)
- N/A (QuickBooks not connected)

### Manual re-sync action

Add a table action or button that dispatches `SyncTransactionToQuickBooks` for a specific transaction. Only available when `quickbooks_id` is null. Gated to `manage_financial_settings` permission.

### Help doc update

Update `resources/docs/settings-finance.md` with:

- What transaction sync does (one-way push of completed payments and refunds).
- How to select the income account.
- What the sync statuses mean.
- Note that sync is automatic ‚Äî no manual action needed for normal operation.

## Out of scope

- Two-way sync or reading data from QuickBooks (beyond Chart of Accounts for account selection)
- Admin-configurable account mapping per transaction type or fund (post-Beta 1)
- Reconciliation view comparing CRM and QuickBooks totals
- Batch / historical sync of pre-existing transactions
- QuickBooks customer matching or creation (transactions post without a QB customer reference for now)
- Automatic retry of failed syncs
