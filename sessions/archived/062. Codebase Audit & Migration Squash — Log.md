**Clean.** All models added or modified since session 049 are fully covered:

- `PageWidget`: `is_active` in `$fillable` and `$casts` ✓. No `is_required` DB column — required widget mechanism is an in-memory static map on `WidgetType::$requiredFor`, not a DB column. Correct by design.
- `PortalAccount`: `contact_id`, `email`, `password`, `email_verified_at`, `is_active` all in `$fillable` ✓; `email_verified_at` datetime and `is_active` boolean in `$casts` ✓.
- `GeneralSettingsPage`: `portal_prefix`, `system_prefix`, `system_page_content_reset_password`, `system_page_content_email_verify` all mounted and surfaced in the form ✓.
- `WidgetType`: `is_pinned` is implemented as `WidgetType::isPinned()` static method — no DB column needed ✓.

### WS2 — DB schema doc accuracy
**One gap fixed.**

`custom_field_defs.is_filterable` was added in migration `2026_03_20_200001` but never documented in `docs/schema.md`. Row added. Header updated from "session 060" to "session 062" with a migration squash note.

All other tables added or modified since session 049 (`pages` type constraint, `portal_accounts`, `portal_password_reset_tokens`) were already correctly documented.

### WS3 — Permission gate audit
**Two gaps fixed.**

1. **`manage_routing_prefixes` not seeded.** The permission is used in `GeneralSettingsPage::canAccess()` and listed in `RoleResource::standalonePermissions()` so it appears in the roles UI — but it was never in `PermissionSeeder`, meaning the permission didn't exist in the DB and could never actually be assigned. Added to `PermissionSeeder`.

2. **`PageResource` had no `canDelete()` gate.** System page delete suppression was UI-only (hidden action buttons). A user with `delete_page` permission could bypass the UI and hit the delete endpoint directly. Added `canDelete()` that blocks deletion of any page with `type === 'system'` at the gate level.

Gate table for new resources/pages — all checked and correct:
| Resource / Page | Gate method | Permission checked |
|---|---|---|
| `PageResource` | `canViewAny()` | `view_any_page` ✓ |
| `PageResource` | `canDelete()` | `delete_page` + type ≠ system ✓ (fixed) |
| `PageResource` | `canRestore()` | `delete_page` ✓ |
| `PageResource` | `canForceDelete()` | `delete_page` ✓ |
| `WidgetTypeResource` | `canViewAny()` | `view_any_widget_type` ✓ |
| `GeneralSettingsPage` | `canAccess()` | super_admin OR `manage_routing_prefixes` ✓ |
| `RoleResource` | `canAccess()` | `isSuperAdmin()` ✓ |

### WS4 — Soft deletes consistency
**Documented decisions, no changes.**

- `PortalAccount` — **hard delete kept.** Model extends `Authenticatable`; adding soft deletes requires custom auth guard integration. Portal accounts are the only auth credential for a member — if accidentally deleted, the member loses access permanently. Acceptable risk at current scale; soft delete is a future improvement if needed.
- `PageWidget` — **hard delete kept.** Transient layout records with no independent meaning. Required widgets on system pages are protected at the UI level (cannot be removed from system pages, widget types cannot be deleted if pinned). Hard delete risk is low.

### WS5 — Fillable vs. casts consistency
**Clean.** All models checked. No gaps.

### WS6 — Foreign key index audit
**Clean.** Only new FK since session 049 is `portal_accounts.contact_id`, which has an explicit `.index()` in the migration ✓.

### WS7 — Orphaned documents and help coverage
**One gap fixed.**

`portal-pages.md` listed `portal.account` in its `routes:` array. That is a public portal route — not a Filament admin route name. The help system only resolves admin route names for contextual help; this route would never fire. Replaced with `filament.admin.resources.pages.index/create/edit` — the admin surfaces where member pages are actually managed.

All other help doc route entries verified clean. No admin surfaces added since session 049 are missing help articles.

### WS8 — Factory coverage
**One factory created.**

- `PortalAccountFactory` — minimal factory with `unverified()` and `inactive()` states. `HasFactory` trait added to `PortalAccount` model.

All other models added since session 049 already had factories or no factory is warranted.

### WS9 — Migration squash
**Completed.**

The app container did not have PostgreSQL client tools (`pg_dump`, `psql`) installed, which blocked `php artisan schema:dump`. Fix: added `postgresql-client` to the Dockerfile apt-get install block and rebuilt the image.

Steps completed:
1. `pg_dump --schema-only` run from the postgres container → `database/schema/pgsql-schema.sql` (2546 lines).
2. All 91 migration files in `database/migrations/` deleted.
3. `php artisan migrate:fresh --seed` — schema dump loads cleanly, no migrations to run, all 5 seeders pass.

The dump includes all tables: `pages`, `page_widgets`, `portal_accounts`, `portal_password_reset_tokens`, `widget_types`, `custom_field_defs`, and every other table in the schema.

### WS10 — Route list audit
**Clean.** Portal GET routes correctly use `$systemBase` (computed from `system_prefix` SiteSetting), resolving to e.g. `/system/login`, `/system/signup`. POST routes remain at root paths. `portal.account` GET route builds the slug dynamically and serves via `PageController`. No retired routes still registered.
