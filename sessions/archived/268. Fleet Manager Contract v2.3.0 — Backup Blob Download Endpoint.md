# 268. Fleet Manager Contract v2.3.0 — Backup Blob Download Endpoint

Add a single mTLS-gated HTTP endpoint (`GET /api/backup/blob`) that streams the most recent backup zip from the configured backup destination disk to the FM caller. Additive contract bump v2.2.0 → v2.3.0. CRM-side prerequisite for FM 021 (`BackupBlobClient`) + FM 022 (restore-to-fresh-node operator affordance) — together they satisfy A2(c) of Track A's success criterion.

## Design decisions (resolved before starting)

These are agreed-going-in. Confirm at session start during the read-through.

- **Streams the freshest blob from a single source disk.** No historical enumeration; FM gets the latest only. Source-disk preference: `local` first if it's in `BACKUP_DISKS`, else fall through to whatever else is configured (e.g., `spaces`). The contract spec documents the rule so FM consumers know which disk's blob they're getting. `BACKUP_DISKS` value enumeration happens at session start (verify-at-start question 2) to confirm the ordering matches actual dev/prod configurations.

- **Same mTLS gate as `/api/health` / `/api/logs` / `/api/backup/trigger`.** Location-scoped `if ($ssl_client_verify != "SUCCESS") { return 403; }` in both `default.conf` and `prod.conf`. The existing `fm-client.crt` operator-pasted material covers this endpoint without any client-side cert reissue.

- **Throttle: 60 requests per minute per source IP** (`throttle:60,1`). Matching `/api/logs` since blob fetches are I/O-bound and cheap relative to the backup-trigger's expensive synchronous artisan call. The FM-side consumer's burst pattern is "one fetch after a successful trigger, with maybe 1-2 retries on transient failure" — 60/min is generous for that. Confirm at session start during verify-at-start.

- **Method: `GET`.** Read-only — does not mutate state. No request body. Cacheability is irrelevant because the contract names the endpoint as "fetch the latest" with no version pinning, but `Cache-Control: no-store` is emitted defensively to prevent intermediate caching surprises.

- **Streaming mechanism: `Storage::disk($diskName)->download($path, $filename)`.** Laravel's wrapper handles `Content-Type`, `Content-Disposition`, and `Content-Length` automatically when the underlying disk implements the `Readable` interface (both `local` and `spaces` do). This avoids manual byte-shoveling and avoids the `StreamedResponse` boilerplate. Confirm at session start that `Backup::path()` from the spatie BackupDestination returns a path string that the `local` disk's `download()` accepts directly. For `spaces`, the underlying S3 client streams the blob through Laravel's response without buffering the whole file in PHP memory — verify under a small (~100KB) blob during dev curl smoke testing.

- **Filename in `Content-Disposition`.** Pass through the spatie blob filename verbatim (looks like `nonprofitcrm-2026-05-08-12-30-00.zip` per the `<backup-name>-Y-m-d-H-i-s.zip` default pattern). FM-side stores blobs by that filename and round-trips it for restore. No sanitisation needed (no operator input in the filename); confirm the exact pattern at session start by triggering a backup once and inspecting the resulting filename.

- **404 envelope on no-backup-available.** When no blob exists on any of the configured disks (fresh install before the first scheduled or triggered backup), return HTTP 404 with JSON envelope matching the `/api/logs` 404 shape (no `contract_version` field — this session does not change error-envelope shape across the contract; if a future revision wants `contract_version` on errors, it lands as a deliberate cross-endpoint sweep):
  ```json
  {
    "error": "no_backup_available",
    "message": "No backup found on disk \"<diskName>\" for backup name \"<backupName>\""
  }
  ```
  HTTP 404 instead of the 200-with-failure-status pattern that `/api/backup/trigger` uses, because the resource itself is absent — REST-cleaner here. The contract spec documents the distinction (mTLS gate already returned 403; 404 is for resource-absence, not authorization).

- **500 envelope on server misconfiguration.** When `BACKUP_DISKS` is empty or otherwise unconfigured, return HTTP 500 (not 404) with JSON envelope `{"error": "backup_destinations_not_configured", "message": "..."}`. The FM-side classification is meaningfully different — 404 is "trigger one and retry" (recoverable), while 500 is "operator configuration action required." 500 also covers unforeseen disk-driver errors (S3 client throwing, filesystem permission errors), routed through the same sanitisation pipeline as `BackupController::trigger`'s `message` field — path-prefix strip, newline collapse, 500-char cap.

- **No request parameters.** No `?disk=` or `?date=` query params. The disk-fallback rule is server-side; FM gets whatever the rule resolves to. Future extensions (date pinning, explicit-disk selection, blob enumeration) live post-Beta-1.

- **Controller shape: sibling method on `BackupController`.** Default to adding `blob(Request $request): \Symfony\Component\HttpFoundation\StreamedResponse|\Illuminate\Http\JsonResponse` next to `trigger()`. If the audit at session start surfaces enough divergence in helpers/state to warrant a split, surface before committing to a separate `BackupBlobController.php`.

- **`CONTRACT_VERSION` constant.** Bump `BackupController::CONTRACT_VERSION` from `'2.2.0'` to `'2.3.0'` (the trigger envelope updates automatically via `self::CONTRACT_VERSION`). Bump `HealthController::CONTRACT_VERSION` to `'2.3.0'` — this is the canonical field FM polls. Per the v2.1.0 changelog, `LogsController` does not carry its own constant (envelopes reuse the Health value); confirm at session start and skip if no constant is present.

- **No new artisan command, no new service, no new model.** This is a thin HTTP wrapper around `Spatie\Backup\BackupDestination\BackupDestination::create($disk, $backupName)->backups()->newest()` and `Storage::disk($disk)->download(...)`.

- **Out of scope** (full list under § Out of scope below): historical blob enumeration, per-disk targeting via query, progress streaming, Range / resume support, schema changes, help-doc updates, FM-side restore primitives.

## Phase 1 — Nginx conf + route + controller

**Goal:** new `GET /api/backup/blob` endpoint live behind mTLS + throttle, streaming the freshest blob from the resolved source disk.

**Steps:**

1. **`docker/nginx/default.conf`** (single-file bind-mount — use `docker compose up -d --force-recreate nginx` post-edit, not `restart`): add a fourth mTLS-gated location block immediately after `location = /api/backup/trigger { ... }`:
   ```nginx
   location = /api/backup/blob {
       if ($ssl_client_verify != "SUCCESS") { return 403; }
       fastcgi_read_timeout 600;
       try_files $uri /index.php?$query_string;
   }
   ```
   The per-location `fastcgi_read_timeout 600;` mirrors the `/api/backup/trigger` defense. The default 300s ceiling is the *between-packets* timeout, not total request time — multi-GB blobs streaming to a slow FM-side network with PHP buffer backpressure can plausibly stall a single read window past 300s. 600s is cheap insurance with no downside on the success path. Update the existing comment ("strict on /api/health, /api/logs, and /api/backup/trigger") to enumerate all four endpoints.

2. **`docker/nginx/prod.conf`** — mirror the same block (including `fastcgi_read_timeout 600;`) immediately after `/api/backup/trigger`. Update the inline comment in step 31 of `prod.conf` ("strict on …") to enumerate all four endpoints. Same nginx-restart caveat applies.

3. **`routes/api.php`** — add the new route alongside the existing three:
   ```php
   Route::middleware(['throttle:60,1'])
       ->get('/backup/blob', [BackupController::class, 'blob']);
   ```

4. **`app/Http/Controllers/Api/Fleet/BackupController.php`** — add a sibling method `blob(Request $request)`. Implementation shape:
   1. Resolve the disk-resolution order via the **two-layer fallback rule** (see § Design Decisions for the full statement):
      - **Layer A — preference:** read `BACKUP_DISKS` (already CSV-parsed in `config/backup.php` line 153). If `local` is present anywhere in the list, it goes first regardless of authored order; the remaining disks follow in their authored order.
      - If the disks list is empty (server misconfiguration), return **HTTP 500** with the sanitised envelope `{"error": "backup_destinations_not_configured", "message": "BACKUP_DISKS env var resolves to an empty disk list"}`. Do NOT return 404 here — 404 is reserved for "configured disks have no backups yet"; 500 distinguishes "feature is not configured" from "feature is configured but has produced no output." FM-side classification cleanly splits operator actions: 500 → fix config; 404 → trigger a backup.
   2. Iterate the resolved disk list (Layer B — fallthrough-on-empty). For each disk in order:
      ```php
      use Spatie\Backup\BackupDestination\BackupDestination;
      $destination = BackupDestination::create($diskName, config('backup.backup.name'));
      $newest = $destination->backups()->newest();
      ```
      - **If `$newest` is non-null:** stream from this disk (step 4 below). Stop iterating.
      - **If `$newest` is null:** continue to the next disk in the resolved list.
   3. **If all configured disks are exhausted with no blob found:** return HTTP 404 with the no-backup-available envelope.
   4. **If found:** return `Storage::disk($diskName)->download($newest->path(), basename($newest->path()), [
         'Cache-Control' => 'no-store',
      ])`. Laravel sets `Content-Type: application/zip`, `Content-Disposition: attachment; filename="..."`, and `Content-Length: <bytes>` automatically based on the file metadata. (Verify the auto-detection produces `application/zip` rather than `application/octet-stream` — if the latter, override explicitly.)
   5. Wrap the lookup in `try { ... } catch (\Throwable $e) { ... }`. On exception, log via the existing logger (no new channel) and return a 500 with a sanitised JSON envelope (same sanitisation pipeline as `BackupController::trigger`'s `message` field — path-prefix strip, newline collapse, 500-char cap). Covers unforeseen disk-driver errors (S3 client throwing, filesystem permission errors).

5. **Bump `HealthController::CONTRACT_VERSION` and `BackupController::CONTRACT_VERSION` to `'2.3.0'`.** This propagates through both controllers' envelopes automatically.

**Pass criterion:** the new route is reachable via `curl --cert ... --key ... https://<domain>/api/backup/blob -o blob.zip` and yields a valid zip; mTLS rejection returns 403; throttle returns 429 on the 61st request within a minute; 404 envelope renders when no blob exists.

## Phase 2 — Contract spec doc bump

**Goal:** `docs/fleet-manager-agent-contract.md` documents the new endpoint at v2.3.0.

**Steps:**

1. **Top-of-doc:** `Contract Version: 2.2.0 → 2.3.0`. `Status: active`.
2. **New endpoint section** after the existing `/api/backup/trigger` section. Document `GET /api/backup/blob`:
   - Request shape: GET, no body, no query params, mTLS-gated, throttled 60/min.
   - Success response: HTTP 200, `Content-Type: application/zip`, `Content-Disposition: attachment; filename="<spatie-blob-filename>"`, `Content-Length: <bytes>`, body is the zip stream. Note the `Cache-Control: no-store` defense.
   - 404 response example (verbatim from § Design Decisions, with a real-shaped disk + backup-name). Shape matches `/api/logs` 404 (no `contract_version` field).
   - 500 response — two cases, both sanitised: (a) `backup_destinations_not_configured` (server misconfiguration — `BACKUP_DISKS` empty), (b) generic disk-driver errors routed through the same sanitisation pipeline as `/api/backup/trigger`'s `message` field.
   - **Status-code semantics for FM consumers:** 200 = blob streamed; 404 = configured disks all empty (recoverable — trigger a backup and retry); 500 = server misconfiguration or driver error (operator action required, not retry-able). FM error-classification leans on this distinction.
   - **Disk-fallback rule — two layers, both load-bearing:**
     - **Layer A — preference:** if `local` is present anywhere in `BACKUP_DISKS`, it is tried first regardless of authored order. Remaining disks follow in their authored order.
     - **Layer B — fallthrough-on-empty:** if the preferred disk has no blobs (`->newest()` returns null), the resolution falls through to the next disk in the resolved order. The 404 envelope is only returned when all configured disks are exhausted.
   - Note that the response does not expose which disk the blob came from — FM treats the blob as opaque-by-source.
   - Filename pattern: `<backup-name>-Y-m-d-H-i-s.zip` per spatie defaults (e.g., `nonprofitcrm-2026-05-08-12-30-00.zip`). FM-side stores by this filename for round-trip parity.
3. **CHANGELOG** — add v2.3.0 entry above v2.2.0:
   - Additive only — new endpoint `GET /api/backup/blob`.
   - Forward-compat framing: v2.2.0 consumers don't see this endpoint and continue working unchanged.
   - Disk-fallback rule and 404-vs-500 distinction noted.
4. **Security Posture:** no change. mTLS gate inherits from v2.0.0; throttle is endpoint-specific and noted in the endpoint section.
5. **Recovery posture sub-section (added at 253):** add a brief note that v2.3.0's new endpoint is the FM-side-readable half of the restore primitive (FM 021 + 022 build on top), but restore execution remains manual `pg_restore` operator-side per the existing posture.

**Pass criterion:** spec doc renders cleanly; once the branch merges to main, the canonical raw URL returns the v2.3.0 doc.

## Phase 3 — Tests

**Goal:** controller-level test coverage per the new endpoint's behavior matrix.

**Test-pattern risk — verify before writing the full matrix.** Spatie's `BackupCollection::newest()` filters disk contents by filename pattern (`<backup-name>-Y-m-d-H-i-s.zip`) and timestamp. Pre-writing arbitrary bytes via `Storage::fake('local')->put(...)` may not register as a "backup" — `->newest()` returns null even though the file is on disk. **Before committing to the ~10-test matrix below, write test #1 against a single stub and run it.** If `Storage::fake` doesn't register the pre-written file as a recognized backup, switch to mocking `BackupDestination::create()` directly via `app()->bind(...)` or pull the resolution logic into a thin service that the controller delegates to and mock the service. Whichever path the audit lands on, document the choice in the session log.

**New file: `tests/Feature/Api/Fleet/BackupBlobEndpointTest.php`.** Test cases (estimate ~7–12):

1. **Successful download from `local` disk.** Use the test-pattern resolution from the verify-step above. Assert response is HTTP 200 with `Content-Type: application/zip`, `Content-Disposition` containing the filename, `Content-Length` set, body equals the zip bytes.
2. **Successful download from fallback disk.** Configure `BACKUP_DISKS=local,spaces`; `local` empty; `spaces` has the blob. Assert the response streams from `spaces`.
3. **404 when no backup exists on any configured disk.** All disks empty. Assert HTTP 404 with JSON envelope `{error: 'no_backup_available', message: ...}` (no `contract_version` — matches `/api/logs` 404 shape).
4. **500 when `BACKUP_DISKS` resolves to an empty list.** Server-misconfiguration case. Assert HTTP 500 with JSON envelope `{error: 'backup_destinations_not_configured', message: ...}`. Distinct from #3.
5. **500 when the disk driver throws.** Mock the storage layer to throw mid-stream; assert HTTP 500 with sanitised envelope (path-prefix strip, newline collapse, 500-char cap mirroring `/api/backup/trigger`).
6. **`local` preference rule.** Configure `BACKUP_DISKS=spaces,local` (explicit non-local-first); confirm `local` still wins when both have blobs.
7. **mTLS gate.** Match the pattern from `BackupTriggerEndpointTest` / `LogsEndpointTest` — likely deferred to deployment-time verification rather than unit-level mocking. If the precedent tests *do* exercise the gate at unit level, mirror that pattern.
8. **Throttle.** Dispatch 61 requests within 60s; assert the 61st returns 429. Match the throttle-test pattern from existing endpoint tests.
9. **Method enforcement.** `POST /api/backup/blob` returns 405.
10. **Filename pass-through.** Assert the response's `Content-Disposition` filename matches the spatie blob filename pattern verbatim (e.g., regex `/^attachment; filename="nonprofitcrm-\d{4}-\d{2}-\d{2}-\d{2}-\d{2}-\d{2}\.zip"$/`).

12. **`Content-Length` accuracy.** Pre-write a known-byte-count zip; assert the response's `Content-Length` header matches.

## Phase 4 — Manual verification

**Manual verification (the actual end-to-end drill, judgment-required):**

- Trigger a backup against the dev container so a blob exists:
  ```
  docker compose exec -T app php artisan backup:run
  ```
- Fetch the blob via curl with the existing FM-side cert:
  ```
  curl --cert <fm-client.crt> --key <fm-client.key> -o /tmp/blob.zip https://<domain>/api/backup/blob
  unzip -l /tmp/blob.zip
  ```
- Assert: the file is a valid zip; the response headers (via `curl -I` separately or via `--write-out`) include `Content-Type: application/zip`, `Content-Disposition: attachment; filename="..."`, `Content-Length: <expected bytes>`, `Cache-Control: no-store`.
- **No-backup drill:** delete all backups and refetch — confirm the 404 envelope renders correctly and doesn't expose disk paths or stack traces.
- **Empty-config drill:** unset `BACKUP_DISKS` (or set it to an empty string) in a temporary `.env` and refetch — confirm HTTP 500 with the `backup_destinations_not_configured` envelope. Restore the env afterwards.
- **Throttle drill:** run 61 requests in a tight loop with the cert; assert the 61st returns 429.
- **Disk-fallback drill** (if both `local` and `spaces` are configured in dev): verify `local` is preferred when it has the blob; verify the fallback to `spaces` works when `local` is empty.

## Security checklist

Before closing:

- [ ] **mTLS gate enforced at nginx, not application layer.** Same pattern as `/api/health`, `/api/logs`, `/api/backup/trigger`. Bypassing the cert returns 403 before reaching PHP-FPM.
- [ ] **Throttle prevents trivial DoS.** 60/min per source IP caps incidental burst patterns. Higher than `/api/backup/trigger`'s 6/min because blob downloads are cheap.
- [ ] **Response payload contains no stack traces, no absolute filesystem paths, no secret material.** 404 envelope's `message` field reveals only the configured disk name + backup name (both already public via `BACKUP_DISKS` / `config/backup.php`). 500 envelopes (both server-misconfig and disk-driver-error variants) route through the same sanitisation pipeline as `/api/backup/trigger`.
- [ ] **No new env vars introduced.** `BACKUP_DISKS` / `SPACES_*` env vars from CRM 242 cover the storage configuration.
- [ ] **GET-only / no state mutation.** Idempotent. Throttle handles burst-protection. `Cache-Control: no-store` defends against intermediate caching exposing the blob to unintended consumers.
- [ ] **Spec doc accurately documents the 404-vs-500 distinction.** FM consumers (FM 021) need to know that 404 means "no blob yet on any configured disk" (recoverable; trigger one and retry), 500 with `backup_destinations_not_configured` means "feature unconfigured" (operator must set `BACKUP_DISKS`), and a generic 500 means a disk-driver error (likely operator action required).
- [ ] **Disk-fallback rule documented in spec — both layers.** Layer A (preference: `local` first if present) and Layer B (fallthrough-on-empty across the resolved order) both need to be visible to FM consumers so they can trace which disk's blob they're reading.
- [ ] **Error-envelope shape parity with `/api/logs`.** 404 envelope shape matches `/api/logs` 404 (no `contract_version`). If the contract revisits adding `contract_version` to error envelopes, that's a deliberate cross-endpoint sweep — not this session's scope.

## Out of scope

- **Historical blob enumeration.** v2.3.0 ships "fetch latest"; enumerating all blobs (`GET /api/backup/blobs` list endpoint) is post-Beta-1 if it ever proves needed. Restore-from-an-older-blob is a manual operator drill in the meantime.
- **Per-disk targeting via query param.** `?disk=spaces` or similar. The disk-fallback rule is server-authoritative; if FM needs a specific disk's blob for a specific reason, that's a future contract bump.
- **Progress streaming during long downloads.** Operators see a download-bar in their browser if they fetch directly; FM 022 surfaces its own progress UX over HTTP-level transfer state.
- **Range / resume support (HTTP 206 partial content).** Multi-GB blobs that need resumable transfers are post-Beta-1.
- **CRM-side restore primitives.** Restore stays manual `pg_restore` per A2(c) framing. Any FM-driven restore is FM-side scope (FM 022 is the planning slot).
- **`BackupHasFailed` event listener CRM-side.** Already documented out of scope per CRM 242.
- **Schema changes / migrations.**
- **Help-doc updates.** This is API surface, not admin UI — no operator-facing CRM admin page exists.
