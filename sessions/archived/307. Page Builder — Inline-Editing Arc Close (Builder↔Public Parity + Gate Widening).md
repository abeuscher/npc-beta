## Two deliverables, both objective

### Phase 1 — Builder↔public render-parity tests (standing infrastructure)

A new Pest test in the `design` group asserts that for a representative inline-eligible widget set, the HTML produced by `WidgetRenderer::render($pw, ..., $inlineEditing: false)` (the public path) is **appearance-equivalent** to the HTML produced by `WidgetRenderer::render($pw, ..., $inlineEditing: true)` (the builder canvas path), modulo a documented expected-diff set:

- **Expected on builder, not on public:** `data-config-placeholder` attributes (per `inline-prose.blade.php` builder-only emit); empty inline-prose wrappers around blank non-`always` fields. The 305 conflation split made these the *only* expected differences.
- **Expected identical:** every other attribute, every class, every text node, every structural element. Composed inline styles (`AppearanceStyleComposer`) are not part of the render output per-se, so the test exercises the bare `WidgetRenderer::render()` output, not the wrapped block from `PageBlockRenderer` / `WidgetPreviewRenderer`.

Representative widget set (at minimum): TextBlock, Hero, ThreeBuckets (one bucket filled, one empty), PricingChart (the canonical stress test from 304 — populated and empty configs), plus each widget added by Phase 2's widening.

The test is **standing**: it runs on every push as part of the `design` group; the `DesignGroupIntegrityTest.php` pinned list is updated in the same reviewed pass. **Coordinate with the stale-stylesheet drift guard from session 296** by cross-reference in the file header — both live in the `design` group, both guard composition correctness, but they cover different layers (this = render-layer equivalence; 296 = served-bundle freshness). No duplication.

### Phase 2 — Widen the code-declared inline-eligibility gate

Extend the inline-editable widget set from the 304/305 list (TextBlock / Hero / ThreeBuckets / PricingChart) to every remaining widget whose display-prose nodes are **Tier-A-clear** per the canonical 40-widget safety pass (recorded in `sessions/archived/304. Page Builder — Inline Editing Foundation.md`):

- Each candidate widget gets `inlineEditable(): bool { return true; }` on its Definition class.
- Each genuine display-prose node in its template gets the `@include('widget-shared.inline-prose', [...])` annotation with the right tag/key/type/label.
- **No Tier-B field is ever annotated.** The 304 Tier-B guard test catches this; re-run it after every widening edit and after the widening pass as a whole.
- **No data-driven `{{item.*}}` template gets a widget-level `inlineEditable()=true`.** The widget-level gate excludes these by design (304 owner decision); preserve it.

Candidates from the 40-widget pass (the spec listed examples; do a fresh per-widget read to confirm Tier-A clarity, do not just paste a list): DonationForm heading, EventsListing static heading, plus any other widget with at least one genuine display-prose node that does not double as CSS / `alt` / chart-config / token-bearing data.

For each widening edit: re-run the Tier-B guard; round-trip the new inline path through the inline-edit save flow (Pest); run the Phase 1 parity test for the widget (its builder render === its public render modulo the documented diffs).
