### Phase 5 — Membership-import ledger fix (resolves Decision 4)

Previously imported memberships did not write a sibling `Transaction` row even when `amount_paid > 0`, breaking ledger symmetry with donation/event imports. New private `createTransactionForMembership(Membership): ?Transaction` method on [ImportMembershipsProgressPage.php](app/Filament/Pages/ImportMembershipsProgressPage.php):

- Returns null when `amount_paid` is null/zero.
- Returns null when status not in `['active', 'expired']` (no money moved historically).
- Otherwise creates a Transaction with `subject_type = Membership::class`, `subject_id = $membership->id`, `type = 'payment'`, `direction = 'in'`, `status = 'completed'`, `source = Source::IMPORT`, amount and date copied from the membership row, plus `import_source_id` / `import_session_id` / `external_id` mirrored for cascade and dedupe.

`dryRunReport.entities` extended with a `transactions: ['would_create' => 0]` slot in both the inline declaration and `emptyDryRunReport()`. `accumulateOutcome()` aggregates the new entity counter. The success-return path adds `transactions: ['would_create' => 1]` only when the helper returned a Transaction.

Mirrors the donation/event import pattern of "one row per parent + conditional Transaction child when paid." Comp / lifetime / unpaid / pending memberships skip the transaction write.

### Bundled hotfix unrelated to session scope

User reported during manual testing that adding a donation form to a public page caused every link on the public site to close the browser tab. Diagnosis traced to a missing Alpine component registration:

- `<x-custom-select>` Blade component (used by the donation form's fund picker and `<x-state-select>`) declares `x-data="customSelect({...})"`.
- `customSelect` was registered in `resources/js/admin.js` but **not** in `resources/js/public.js`.
- Public bundle had Alpine without the registration → Alpine's expression evaluator fell through to global window scope → `@click.outside="close()"` on the wrapper resolved `close` to native `window.close` → every off-component click closed the tab.
- Two collisions made the failure mode this severe: `open` and `close` are real native `window` methods (`window.open`, `window.close`), so Alpine's fallback resolved to working but wrong functions instead of throwing `is not defined`. The fund-picker dropdown silently failed to render while the document-level `@click.outside` listener was already armed.

Fix in [resources/js/public.js](resources/js/public.js): added `import customSelect from './admin/custom-select.js'` and registered via `Alpine.data('customSelect', customSelect)` inside an `alpine:init` listener (mirror of admin.js's pattern). `npm run build` rebuilt the public bundle (`public-DRTEyvxm.js` → `public-ukqgkFle.js`). User confirmed fixed.

Latent bug, not session-introduced — the deploy server didn't show it because it didn't have a published page with a donation form. Bundled in this branch since manual testing of session 233's source-column work was blocked until it was fixed.

## Production deployment notes

- Run `php artisan migrate` to apply the schema migration. Backfill UPDATEs run inside the migration's `up()`. Existing rows are populated from predicates with no row-data loss; rows that match neither IMPORT nor STRIPE predicate fall to the column default (`stripe_webhook` for donations, `human` for the others).
- **Surface backfill SQL before running on production.** The dev install ran cleanly and the predicates are conservative (only-and-no-overlap categorization), but production rows have meaningful financial history; a pre-run `EXPLAIN` or row-count preview is the responsible move. The migration is reversible via `down()` (drops columns; existing rows survive).
- No seeders to re-run; this session ships no new widget types or Views.
- No public-bundle rebuild required by 233's source-column work itself. The Alpine hotfix in `resources/js/public.js` requires `npm run build` (the new bundle is `public-ukqgkFle.js`); deploy pulls the new `public/build/manifest.json` and the public layout reads the manifest at request time, so no in-app cache invalidation needed.

## Deferred decisions

- **Phase B — Admin gating generalization.** Lifting the `! $record->stripe_id` action-visibility pattern from `TransactionResource` into a shared trait keyed off `source` and applying to Donation/Membership Resources where action surfaces exist or are about to exist. Not in this session because Refund/Cancel/Sync actions don't yet exist on Donation or Membership; gating is unneeded until those actions land. Phase B lands when an action surface that needs it is imminent.
- **Phase C — QuickBooks sync + email-trigger origin-awareness.** Deferred until those features arrive.
- **DataSink-as-mandatory-boundary.** This session adds `source` to fillable + payload at every write site, but does not require all writes to flow through `DataSink::write()`. Centralizing financial writes through DataSink is a future cleanup; the persisted column is the prerequisite, not the consequence.
- **DEMO source on financial models.** `Source::DEMO` not added to any financial model's `ACCEPTED_SOURCES`. The dashboard debug generator was reworked to use `Source::HUMAN` per the Phase-6 sub-decision; if DEMO becomes load-bearing later (e.g., a stronger demo-vs-real separation in marketing-prospect installs) it can be added then.
- **Source for non-financial models.** Contact, Page, Collection, etc. have their own origin questions; outside the financial-data-criticality scope of this session.
- **Read-side projector / widget changes.** Recent Donations (5d-3) ships in session 234 and consumes session 233's `source` column via a `donation_origin` projector field (decision locked in during 234 prompt drafting at this session's close).
- **Membership currency field.** Phase 5's transaction-creation logic assumes `currency = 'usd'` since memberships don't currently carry a currency column. If multi-currency support is added later, this needs revisiting.
- **Stripe-path manual smoke-testing.** Local dev install can't exercise real Stripe checkouts; the three Stripe paths (donation, paid event, paid membership) require deploy-server validation. The Pest tests cover the source assignment via `creating`-event capture; deploy-server smoke is the integration confirmation.
