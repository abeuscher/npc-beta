## What changed FM-side

A conversation at FM session 013 close surfaced that FM is currently a **single point of catastrophic compromise** — FM holds the encrypted private keys for every install on its filesystem under `APP_KEY`, with no documented recovery path beyond rebuilding from scratch. Full FM-filesystem exfiltration unlocks the entire fleet.

The FM-side pivot replaces that with single-operator-shaped controls that collectively make FM compromise recoverable on a documented timeline. Five components, all in Beta-1 scope:

1. **Hardware-bound FM admin auth** (WebAuthn / YubiKey, no password fallback) — FM-side only; sequences alongside CRM A5.
2. **External append-only audit sink** (`audit_events` table mirrored to a write-only Spaces bucket with object-lock) — FM-side only; reuses existing CRM-side Spaces credentials infrastructure.
3. **Off-filesystem encryption key for `tls_client_key`** (operator-entered passphrase at boot, derived key held in cache, never on disk) — FM-side only. Load-bearing architectural piece.
4. **Break-glass recovery cert + runbook** — pre-generated recovery cert kept offline; recovery procedure: SSH to each CRM, swap cert file, reload nginx. **Mostly FM-side, but the script and runbook live CRM-side.**
5. **Calendar-driven cert rotation discipline** (30-day reminders, manual rotation via existing UI) — FM-side only.

Full stub: FM repo `sessions/session-outlines.md § FM Security Posture Pivot`. The Cross-Repo block in the same file names the CRM-side dependencies. The post-Beta-1 GNAP entry was retired in the same edit (superseded by v2.0.0 mTLS + this pivot).

## CRM-side deliverables (three artifacts)

### 1. `bin/rotate-fm-cert.sh`

Operator-facing script that takes a new cert PEM as input and:

- **writes** it to the trusted-cert path (`/etc/nginx/certs/fm-client.crt` in-container; bind-mounted from `/opt/nonprofitcrm/nginx-certs/fm-client.crt` in prod);
- **validates** the input parses as a real X.509 cert (don't brick nginx with malformed input — `openssl x509 -in <file> -noout -text` is the natural validation; non-zero exit = bail);
- **reloads** nginx without restarting the container (so polling barely blips — see open question #2 for the reload mechanism);
- **logs** the rotation timestamp to a CRM-local audit trail (path TBD — likely `storage/logs/fm-cert-rotations.log` or similar).

Used during break-glass recovery: operator SSHs to each CRM, runs this script with the new cert PEM, moves on. **Ships in CRM repo so it's available on every install by default.**

### 2. `docs/runbooks/fm-compromise-recovery.md`

Canonical operator runbook authored CRM-side (recovery is executed against CRM installs, so the runbook lives where the recovery happens). Mirrors the existing `docs/runbooks/fleet-manager-cert-paste.md` style. Covers:

- **detection signals** — what triggers a recovery (FM database exfiltrated, FM admin session hijacked, lost laptop with active session, etc.);
- **pre-recovery** — provision a clean FM, generate a new client keypair offline, distribute the new cert via the recovery channel;
- **per-CRM steps** — SSH in, run `bin/rotate-fm-cert.sh`, verify the next FM poll succeeds with the new cert;
- **post-recovery** — re-enroll a fresh break-glass recovery cert offline, post-mortem the compromise.

**Tested-once gate:** rehearsed against the test/deploy node before first customer onboarding. (The rehearsal itself is not in scope for 253 — the runbook ships, the rehearsal happens against the demo or test node when A3 surfaces a clean target.)

### 3. Agent contract spec update — additive language

In `docs/fleet-manager-agent-contract.md`'s **Security Posture** section, add language naming:

- **the break-glass recovery path** — CRM trusts FM's cert; if FM is compromised, recovery is operator SSH + cert swap, not a contract-level rotation flow;
- **FM's off-filesystem-key posture** — FM does not store the master encryption key on disk; CRM-side trust assumption is "FM cannot self-decrypt without an operator at the console";
- **audit-sink discipline** — FM emits an external audit log of every action against the CRM contract surface; CRM-side nginx access logs are the CRM-side complement.

**No HTTP contract surface change. No version bump.** This is a CHANGELOG-noted documentation revision in the v2.1.0 line, not a v2.2 or v3.0 cut. Sync back to FM at next boundary-touching FM session via the standing protocol.

## Design decisions (resolved at session start)

Four open questions, all decide before any code is written.

### A. Sequencing relative to `release-plan.md`

Three options for where the CRM-side artifacts land in the existing 32-step plan:

- **(A)** Fold into A2 (Fleet Manager — node operations parity). The rotation script and runbook are operations-shaped; A2 already covers install / backup-trigger / restore. Natural fit but lengthens A2.
- **(B)** New entry — **A1c** or similar. Dedicated small session for the CRM-side dependencies. Discrete and reviewable but adds a step.
- **(C)** Spread across existing entries. Script + runbook fold into A2; contract spec update folds into whichever boundary-touching CRM session lands next.

**Recommended: C.** But worth confirming against the current state of A2 scoping. **Resolve at session start.**

### B. Reload mechanism in `bin/rotate-fm-cert.sh`

The cert rotation needs nginx to pick up the new file without a full container restart. Three candidates:

- **(B1)** `docker exec nonprofitcrm_nginx nginx -s reload` — sends `SIGHUP` inside the container. Cleanest; nginx handles reload natively; preserves connection state.
- **(B2)** `docker compose -f /opt/nonprofitcrm/docker-compose.prod.yml restart nginx` — full container restart. ~5–10s polling blip; more disruptive but unambiguous.
- **(B3)** `kill -HUP <nginx_pid>` discovered via `docker top` — same effect as B1 but more brittle (PID lookup, container introspection).

The choice affects polling-blip duration during rotation. **Recommended: B1** unless B1 has a known issue with bind-mounted cert files (the `feedback_bind_mount_edit_breaks_container.md` memory suggests bind-mounted single-file edits can break the mount; worth verifying that writing to the cert file via `cat > path` is safe — atomic `mv` from a temp file is the defensive shape).

### C. Spec update — own session or absorbed

Should the contract spec update be its own session, or absorbed into the next CRM-side boundary-touching session opportunistically?

- **(C1)** Land in 253 alongside the script + runbook. Single coherent absorption.
- **(C2)** Defer to the next naturally-occurring boundary-touching session (e.g., A2 itself). Lighter 253 scope; risk of forgetting.

**Recommended: C1.** The doc edit is small and the cross-repo coordination is cleanest if everything lands together. Picking C2 means session 253 is purely planning and the doc edit becomes a follow-on todo, which is fine if the user prefers smaller sessions.

### D. CRM-side super-admin audit sink — scope now or defer?

The pivot brief flags this as out-of-scope but a worthwhile follow-on conversation: a CRM-side equivalent audit sink for super-admin actions on the CRM itself. Different problem (FM's audit sink covers FM-driven actions; this would cover human-operator actions on CRM admin).

- **(D1)** Defer entirely — capture as a stub in `session-outlines.md`, revisit post-Beta-1.
- **(D2)** Scope it now during this planning session (since the audit-sink machinery is being thought through anyway), draft a stub or even a session prompt for it, slot into release-plan.

**Recommended: D1.** The CRM-side audit sink is a different problem with different consumers; conflating it with the FM pivot dilutes both. A clean stub is enough.
