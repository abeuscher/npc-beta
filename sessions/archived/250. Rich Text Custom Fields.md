# 250. Rich Text Custom Fields

A new `rich_text` custom-field type alongside `text` / `number` / `date` / `boolean` / `select`, backed by the existing QuillEditor primitive on admin forms, rendered as HTML on detail views and widget output. Pre-Beta-1 entry E3 in `sessions/release-plan.md`. B2 prerequisite (the other half is E2 Importer Mapping Page UX, queued for 251).

---

## Design decisions (resolved before starting)

- **Editor primitive.** Use the existing `App\Forms\Components\QuillEditor` Filament form component. Already loaded via `panels::head.end` on every admin page (Quill v2 from CDN). No new editor library, no per-form bootstrap, no toolbar customization in this session — accept the existing system-wide Quill toolbar.
- **Storage shape.** Store the Quill-produced HTML as a plain string in `custom_fields.{handle}` (the existing JSON-cast column on Contact / Event / Page). No separate column, no separate storage shape. The `rich_text` type is a presentation distinction, not a storage distinction.
- **HTML sanitization shape — RESOLVE AT PHASE 1 KICKOFF.** Three options, surfaced at session start before code is written:
  - **(A) Trust Quill's output, render with `{!! !!}`.** Quill v2 strips `<script>` tags by default. Lowest cost; meaningful XSS risk if a malicious importer uses the `rich_text` field as a vector (importer is admin-only but vendored CSV files still warrant care).
  - **(B) Sanitize on save.** Run output through an allow-list HTML sanitizer (e.g., `mews/purifier` or a hand-rolled tag-allow-list) before writing to `custom_fields`. Sanitized HTML stored canonically; render is `{!! !!}` of already-clean content. Adds a Composer dep (or hand-rolled util) but the discipline is durable.
  - **(C) Sanitize on render.** Same sanitizer but applied at every render point. More render-time cost; same security posture as (B) until something forgets to call the sanitizer at a new render site.
  - **Recommended path: (B) with a hand-rolled tag-allow-list utility under `App\Support\` (no new Composer dep), allowing the tags Quill emits by default (`p`, `br`, `strong`, `em`, `u`, `s`, `ol`, `ul`, `li`, `blockquote`, `pre`, `code`, `h1`–`h6`, `a` with `href` only, plus the heroicon `<span class="ql-heroicon" data-heroicon="…"><svg…/></span>` shape if heroicons in custom-field rich text is wanted). Pause and confirm at session start.**
- **Importer treatment.** Rich-text cells from CSVs are stored as plain strings, no HTML parsing — same as the existing `text` type. If the operator imports a Wild Apricot bio that's already HTML, it round-trips as-is and renders correctly. If they import plain text, it renders as plain text inside a `<p>` wrap (or however the renderer handles non-HTML — confirm at Phase 4). No bidirectional conversion; the field is "string-shaped data presented through a rich editor."
- **No migration needed.** `custom_field_defs.field_type` is a string column with no CHECK constraint. The `rich_text` value is a new allowed value, not a column. Verified before drafting; surface and decide if Phase 1 finds otherwise.
- **No new admin page.** The CustomFieldDef admin form already exists at `/admin/custom-field-defs`. The session extends the type Select + table BadgeColumn; no new resource or page.
- **Random Data Generator already supports rich_text.** Added at session 245 in anticipation; see `app/Services/RandomDataGenerator.php::fakeCustomFields()`. No generator change in scope.

## Manual testing

Visual + flow:

1. **Create a rich-text custom field on Contacts** at `/admin/custom-field-defs/create` — verify the type Select offers `Rich Text`, the form saves, and the list page shows `Rich Text` in the badge column.
2. **Edit a contact** — confirm the rich-text editor mounts at the configured field's slot. Apply bold + italic + a heading + a list. Save.
3. **Reload the contact** — confirm the formatting persists exactly. The editor's serialization is the same on reload.
4. **Render the value on a detail view** — confirm the HTML renders as styled content, not as raw markup.
5. **Render through the public widget surface (if any)** — same check.
6. **Import a CSV** with HTML in a rich-text-typed column — confirm the imported contact's value renders correctly through the editor on the next admin edit.
7. **XSS spot check** — manually plant a `<script>alert(1)</script>` in the editor (or via direct DB write if Quill strips it), save, reload, render. Confirm no script execution. Confirm whichever sanitizer layer was chosen does its job.

The visual outcomes (typography polish on detail views, editor chrome integration with Filament) are user-judgment surfaces — pull the user in for those after the objective outcomes (round-trip, importer pass-through, sanitizer correctness) are confirmed.

## Out of scope

- **Toolbar customization on the rich-text editor.** Use the existing system-wide Quill toolbar; this is not the session for per-field toolbar config.
- **Image upload from the rich-text editor.** The system-wide Quill is text-only today. Image-in-rich-text lands as a separate UX session if wanted; the heroicon picker shipped at 246 is the precedent for "what fits inside Quill toolbars."
- **Bidirectional plain-text ↔ HTML conversion.** Rich-text values are stored as HTML strings; plain-text imports remain plain-text strings (or wrapped in `<p>` by Quill on first edit). No magical conversion.
- **A separate `textarea` field type.** Today's `text` type uses `TextInput`. Rich-text replaces neither — operators pick `rich_text` when they want formatting, `text` when they want a single-line entry. A future "long_text / textarea" type can land separately if demand surfaces.
- **Custom field grouping / layout.** Roadmap stub `Custom Field Grouping & Layout` in post-Beta-1 covers presentation grouping; out of scope here.
