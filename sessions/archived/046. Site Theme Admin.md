# 046. Site Theme Admin

Build the admin UI for managing the public-facing theme. Two distinct audiences are served from a single CMS › Site Theme page: non-technical admins who set brand colours, fonts, logo, and site chrome through simple form controls, and developers who need direct access to the SCSS theme file. Session 045 must be complete before this session begins — the SCSS file structure and Vite build pipeline it established are prerequisites.

Settings are stored using the `SiteSetting` key-value model (`app/Models/SiteSetting.php`), consistent with the rest of the application. There is no `GeneralSettings` class.

## Part 3 — Appearance tab

A standard Filament form. On save, all values are written via `SiteSetting::set()`. Changes to appearance fields (colour, fonts) take effect on the next public page request — no build step required. Logo and site chrome changes also take effect immediately.

### Brand colour

A `ColorPicker` field. Stored as `SiteSetting::set('public_primary_color', ...)`. The inline `<style>` block in the public layout already reads this key — no layout changes needed.

### Fonts

Two `Select` fields: heading font and body font. Stored as `SiteSetting::set('public_heading_font', ...)` and `SiteSetting::set('public_body_font', ...)`. The inline `<style>` block already reads these keys.

Options for each (system stacks first, then a small curated Google Fonts list):

```php
[
    ''                                          => '— Default (Pico) —',
    'system-ui, sans-serif'                     => 'System UI',
    'Georgia, serif'                            => 'Georgia (serif)',
    "'Inter', system-ui, sans-serif"            => 'Inter',
    "'Lato', system-ui, sans-serif"             => 'Lato',
    "'Merriweather', Georgia, serif"            => 'Merriweather',
    "'Montserrat', system-ui, sans-serif"       => 'Montserrat',
    "'Open Sans', system-ui, sans-serif"        => 'Open Sans',
    "'Playfair Display', Georgia, serif"        => 'Playfair Display',
    "'Raleway', system-ui, sans-serif"          => 'Raleway',
    "'Source Sans 3', system-ui, sans-serif"    => 'Source Sans 3',
]
```

**Google Fonts link tag:** When a heading or body font value is one of the Google Fonts options above (Inter, Lato, Merriweather, Montserrat, Open Sans, Playfair Display, Raleway, Source Sans 3), the public layout must render a `<link>` tag to load it from Google Fonts. Update `resources/views/layouts/public.blade.php` to read `SiteSetting::get('public_heading_font')` and `SiteSetting::get('public_body_font')`, extract any Google Font names from those values, and render the appropriate `<link rel="stylesheet">` tag in `<head>`. System stack values do not require any markup.

### Logo

A `FileUpload` field. Stored via `SiteSetting::set('logo_path', 'storage/' . $uploadedPath)`. Read back with `SiteSetting::get('logo_path')`.

Update `resources/views/components/site-header.blade.php` to render the logo if `logo_path` is set: display an `<img>` tag using `asset(SiteSetting::get('logo_path'))` within the header-left div, before any header content.

Settings:

```php
->disk('public')
->directory('site')
->visibility('public')
->acceptedFileTypes(['image/png', 'image/jpeg', 'image/svg+xml'])
->nullable()
```

### Nav & Header Colours

Four colour controls for the site header and navigation. Stored as `SiteSetting` keys and injected into the public layout's inline `<style>` block as **scoped CSS on specific selectors** — not Pico root overrides — so they apply only to the header and do not bleed into page content.

- `header_bg_color` — `ColorPicker`, label "Header background". Injected as `header { background: {val}; }`.
- `nav_link_color` — `ColorPicker`, label "Nav link colour". Injected as `header nav a { color: {val}; }`.
- `nav_hover_color` — `ColorPicker`, label "Nav hover colour". Injected as `header nav a:hover { color: {val}; }`.
- `nav_active_color` — `ColorPicker`, label "Nav active colour". Injected as `header nav a[aria-current="page"] { color: {val}; }`.

Update `resources/views/layouts/public.blade.php` to add a second scoped block to the inline style output — separate from the `:root` block — that renders these four rules only when their values are set.

Update `resources/views/components/site-header.blade.php` to set `aria-current="page"` on nav links where the item URL matches the current request:

```php
@php $currentUrl = url()->current(); @endphp
{{-- on each <a>: --}}
{{ $currentUrl === $href ? 'aria-current="page"' : '' }}
```

This change to `site-header.blade.php` also covers the logo addition — do both in the same pass.

### Site Chrome

Three fields for controlling the site header and footer. Values are stored via `SiteSetting::set()` and read directly in `site-header.blade.php` and `site-footer.blade.php` (those components already call `SiteSetting::get()` — no changes to them needed beyond the logo addition above).

- `TextInput` for `header_nav_handle` — label "Header nav handle", default `'primary'`, helper text: "Menu handle to render in the site header."
- `TextInput` for `footer_nav_handle` — label "Footer nav handle", default `'footer'`, helper text: "Menu handle to render in the site footer."
- `RichEditor` for `header_content` — label "Header content", nullable, `columnSpanFull`, helper text: "Rendered on the left side of the site header, beside the logo."

Also include the two read-only status placeholders that were in CMS Settings:

```php
Forms\Components\Placeholder::make('custom_header_status')
    ->label('Custom header override')
    ->content(fn () => view()->exists('custom.header') ? 'custom/header.blade.php is present on this deployment.' : 'Not present — default site-header component is used.'),

Forms\Components\Placeholder::make('custom_footer_status')
    ->label('Custom footer override')
    ->content(fn () => view()->exists('custom.footer') ? 'custom/footer.blade.php is present on this deployment.' : 'Not present — default site-footer component is used.'),
```

### Mount and save

Mount the form by reading all keys from `SiteSetting::get()`. On save, write all values via `SiteSetting::set()`. Show a success notification. No build is triggered.

## Part 4 — SCSS Editor tab

Only visible when `auth()->user()?->can('edit_theme_scss')` is true. If the viewing user lacks the permission, the tab is hidden entirely (not shown as disabled).

### Editor field

A `Textarea` component:
- Loads its content from `resources/scss/_theme.scss` at mount
- `->rows(30)` minimum
- `->extraAttributes(['style' => 'font-family: monospace; font-size: 0.85rem;'])`
- `->label('Theme SCSS')`
- No character limit

The textarea is not a Filament form field backed by a model — it is a raw Livewire public property (`$themeScss`) populated in `mount()` and submitted via a dedicated save action.

### Save and build action

A Filament page action (not a form action) labelled **"Save & Build"** with a spinner/loading state.

On execute:

1. **Validate SCSS** — use `scssphp/scssphp` (check if it is available as a Composer dependency; if not, require it: `composer require scssphp/scssphp`). Attempt to compile the submitted SCSS content. If compilation throws, catch the exception, display a danger notification with the error message, and abort — do not write to disk.

2. **Write to file** — write the validated SCSS content to `resources/scss/_theme.scss`.

3. **Run Vite build** — shell out to `npm run build` from the project root using `Process::run()` or `shell_exec()`. Capture stdout and stderr.

4. **Show result** — if the build exits 0, show a success notification: "Theme built successfully." If it exits non-zero, show a danger notification with the first 500 characters of stderr so the developer can diagnose the failure. In either case, display the full build output in a collapsible section below the editor.

Build output is stored in Livewire properties (`$buildOutput`, `$buildSuccess`) and rendered in the blade view after the editor. The user waits on the page while the build runs — this is intentional and acceptable for a developer-only tool.

### Stored build output

Keep the last build output visible on the page even after the action completes (not just in a notification). Render it in a `<pre>` block styled monospace beneath the editor with a clear heading ("Last build output").

## Part 7 — Heroicons + light/dark toggle

### Heroicons

Heroicons SVGs are already present in `vendor/blade-ui-kit/blade-heroicons/`. Make them available on the public side by creating a simple Blade component at `resources/views/components/icon.blade.php` that accepts a `name` attribute and renders the appropriate SVG inline from the vendor directory. This avoids any CDN dependency or npm package.

```blade
{{-- Usage: <x-icon name="cog-6-tooth" class="..." /> --}}
```

### Light/dark toggle

Add a gear icon to the right of the main nav in `site-header.blade.php`. It should sit outside the `<nav role="group">` but still inside the container. On click it opens a small dropdown containing a light/dark mode toggle.

Use an Alpine store for theme state, initialised from `localStorage` with a fallback to `'auto'` (which lets Pico respect the OS preference). Toggling writes back to `localStorage` and updates `data-theme` on `<html>`.

Add to `resources/js/public.js`:

```js
Alpine.store('theme', {
    current: localStorage.getItem('theme') ?? 'auto',
    toggle() {
        this.current = this.current === 'dark' ? 'light' : 'dark'
        localStorage.setItem('theme', this.current)
        document.documentElement.setAttribute('data-theme', this.current)
    },
    init() {
        if (this.current !== 'auto') {
            document.documentElement.setAttribute('data-theme', this.current)
        }
    }
})
```

The gear dropdown in the header:

```html
<span x-data="{ open: false }" class="nav-dropdown" @mouseenter="open = true" @mouseleave="open = false">
    <button type="button" @click.stop="open = !open" aria-label="Site settings">
        <x-icon name="cog-6-tooth" />
    </button>
    <ul x-show="open" x-cloak>
        <li>
            <button type="button" @click="$store.theme.toggle()">
                <span x-text="$store.theme.current === 'dark' ? 'Light mode' : 'Dark mode'"></span>
            </button>
        </li>
    </ul>
</span>
```

Add any necessary CSS for the icon button to `_layout.scss` — it should sit flush with the nav items and not carry default button styling.

Rebuild assets after modifying `public.js`.
