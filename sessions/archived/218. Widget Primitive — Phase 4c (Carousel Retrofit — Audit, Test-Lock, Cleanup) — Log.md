### Phase 3 — Cleanup (outcome a)

- `app/Widgets/Carousel/CarouselDefinition.php` — `collections(): array { return ['slides']; }` removed. The parent `WidgetDefinition::collections()` returns `[]` by default, so `widgetType->collections` becomes `[]` for Carousel. The renderer's `foreach ($widgetType->collections ?? [])` loop iterates 0 times for Carousel; `WidgetPreviewRenderer::demoCollectionData()` short-circuits to `[]`. **System rule established this session: contract-locked widgets do not declare `collections()`.** Carousel joins BlogListing and EventsListing under that rule.
- The shape of demo data flow for contract-locked widgets is now uniform: live data via the contract; empty state when no live data is present; demo data only via dedicated `DemoSeeder` paths (which seed real Collection rows that the contract resolver finds directly). The legacy `WidgetPreviewRenderer` slot-name fallback path is no longer exercised by any retrofit-locked widget.

### Notable findings

- **The cleanup decision was empirically driven, not docs-driven.** The session prompt's outcome (b) framing assumed `WidgetPreviewRenderer::demoCollectionData()` was producing user-visible value for empty-Collection cases. User's manual-testing observation revealed the path doesn't fire in any user-facing scenario today: empty/unconfigured Carousel = unmounted (correct), real Collection with items = live data via contract (correct), real Collection with zero published items = also unmounted (also correct UX per user). Outcome (b) was overcautious — `collections()` is genuinely vestigial. Carry-forward lesson for future Phase 4 sessions: don't preserve legacy declarations for hypothetical edge cases without empirical UX confirmation that they're load-bearing.
- **System rule established: contract-locked widgets do not declare `collections()`.** Demo data for contract-locked widgets flows only through `DemoSeeder` (which writes real Collection rows the resolver finds directly). The legacy `WidgetPreviewRenderer` slot-name fallback path is dead for retrofit-locked widgets and will be retired across the board once every collection-using widget is locked.
- **Sibling-test retargeting becomes a recurring mechanical pattern.** Every retrofit-lock that removes `collections()` strips the widget from the legacy `widget_types.collections` JSON column — which is exactly what `PageBuilderApiController::filterQueryConfig()` reads. The two `PageBuilderApiTest` tests will keep migrating to whichever widget is still declaring slots until none remain. 218 swapped from `'carousel'`/`'slides'` to `'bar_chart'`/`'data'`; 219 will swap again to BoardMembers or LogoGarden. The eventual end-state (every collection-using widget retrofit-locked) lets the slot-strip behavior itself be retired.
- **Future architectural direction surfaced (deferred): widget-owns-demo-data via the contract's ContentType.** Today, demo data shape comes from the *Collection's* `fields` schema (`DemoDataService::generateCustomItems()` reads it), not from the widget's contract — a dependency direction inversion from the contract architecture. The natural future-session shift: extend `DemoDataService::generateFieldValue()` into a first-class registry (e.g. closed-set `DemoType` constants like `Source` from session 213, with closure fallback), and let widgets declare per-field demo hints inside their `dataContract()`'s ContentType. The widget owns its demo shape; the demo system is the registry. Examples that motivated the discussion: BoardMembers should produce plausible nonprofit-board job titles; a hypothetical 20-page-document field needs a stress-test generator no current path produces. Logged in the deferred decisions section; not in scope for 219 (BarChart retrofit).

## Deferred decisions

- **Widget-owns-demo-data architecture.** Surfaced during this session as a natural future move once the contract pattern is widely applied. Today, demo data shape comes from the Collection's `fields` schema; the contract architecture inverts this — the widget should declare its own demo shape via its ContentType, with a closed-set `DemoType` registry (Faker-backed named generators like `PERSON_NAME`, `EMAIL`, `ADDRESS`) plus closure fallback for genuinely custom shapes. Right time to formalize is when 5–6+ widgets are contract-locked and the inversion has clear payoff. Not in scope for 219.
- **Eventual retirement of the legacy `WidgetPreviewRenderer` slot-name fallback path.** Will become possible once every collection-using widget is retrofit-locked (i.e. every widget with `collections()` is converted to `dataContract()`). At that point the entire slot-keyed fallback mechanism in `WidgetRenderer::render()` lines 45–51 and `WidgetPreviewRenderer::demoCollectionData()` becomes dead code and can be removed in one cross-cutting cleanup.
- **Sibling-test retargeting will recur each Phase 4 session** until the slot-strip behavior itself can be retired. 218 swapped from `'carousel'`/`'slides'` to `'bar_chart'`/`'data'`; 219 will swap to BoardMembers or LogoGarden; the pattern continues until the last slot-declaring widget is retrofit-locked.
- **Default `vendor/bin/pest` memory-limit override.** Third consecutive session using `-d memory_limit=2G` to sidestep the latent mid-suite OOM (Spatie GD image processing). Whether to bake into project tooling (PHP ini, composer script, `tests/Pest.php` setup) remains a separate question.
- **`FakeImportCsvGeneratorTest` full-suite-only flakiness.** Did not surface this session — fast suite ran clean single-pass. Pre-existing latent flakiness flagged in 217 still applies.
- **Remaining ~17 page-builder widgets after 218.** Phase 4 retrofit campaign continues. 219 drafted as BarChart retrofit (first true `SOURCE_WIDGET_CONTENT_TYPE` retrofit, first widget where the contract's `fields` list is dynamic per-instance based on user config).
