# 269. Page Builder Focus-Scroll Clamp

Lock the page-builder canvas scroll position so the focused widget cannot be scrolled out of view, with a tall-widget exception that turns the focused widget itself into the only scrollable region while focus holds. Picks up the stretch piece descoped from session 204.

This is **E11** from `release-plan.md` Track E, scheduled at execution-order position 24. Successor to session 204's must-have scroll-to-centre-on-selection (which shipped). 204 documented the descoping rationale: wheel/touch interception fights browser inertial scrolling and the column-widget nested-scroll case is adversarial — the simple clamp shape needs both concerns addressed before it ships responsibly.

---

## Design decisions (resolved before starting)

These are agreed-going-in. Confirm at session start during the verify-at-start audits.

- **Listener target: the canvas pane (`paneEl`), not the document.** Scroll happens on the canvas pane (`overflow-y: auto` per the existing SCSS); listening at that level scopes the clamp to the page-builder editor without affecting any other admin page. Tear down on component unmount via `onUnmounted`.

- **Event scope: `wheel` + `touchmove` + `keydown`.** Wheel and touch are the obvious vectors. Keyboard inclusion (ArrowUp / ArrowDown / PageUp / PageDown / Space / Home / End) is recommended at session start unless the verify-at-start audit surfaces unexpected complexity. Without keyboard coverage the lock has a trivial hole — focus-trap-style behaviour wants to be complete on a real interaction surface.

- **Clamp math (single function for all event types):**
  ```
  given the focused widget's bounding box relative to the canvas pane:
    widget_top    = widget.offsetTop
    widget_height = widget.offsetHeight
    pane_height   = pane.clientHeight
    scroll_min    = max(0, widget_top + widget_height - pane_height)
    scroll_max    = widget_top
  if widget_height <= pane_height:
    // short widget — lock to a single position keeping the widget centred
    target = (widget_top + widget_height/2) - (pane_height/2)
    clamp scroll to [target, target]   // i.e. no scroll allowed
  else:
    // tall widget — bound scroll to the widget's vertical extent
    clamp scroll to [scroll_min, scroll_max]
  ```
  Compute fresh each event; widget bounds may shift as preview HTML re-renders. `getBoundingClientRect()` per event is cheap enough at human-scale interaction frequency.

- **Defensive nested-scroll defer.** Walk from `event.target` up to `paneEl`. For each ancestor with `overflow-y` of `auto` / `scroll` AND remaining scroll capacity in the wheel direction (`scrollTop > 0` for upward wheel, `scrollTop < scrollHeight - clientHeight` for downward wheel), let the event pass without preventDefault — the inner element absorbs it. This is the v204-flagged "nested-scroll case inside column widgets" defended structurally rather than with a per-widget allow-list. Whether any current widget actually has internal scroll is the verify-at-start audit; if none today, the defer logic is a forward-looking guard that costs little.

- **Listeners registered with `{ passive: false }`.** Required for `preventDefault()` to actually fire on `wheel` / `touchmove` (modern browsers default to passive on these events). The non-passive registration is locally scoped to the canvas pane during admin builder use; no global perf impact.

- **De-selection un-clamps automatically.** The clamp's `selectedItemId` watcher tears down the listeners (or no-ops them) when selection becomes null. No explicit "unlock" UI element is added; clicking the canvas background is the existing release mechanism. Escape-key release is a small in-session absorption candidate — flag at session start, lift if ergonomically obvious, otherwise defer.

- **Drag-drop is not affected.** During a drag-from-library-into-canvas drag, `selectedItemId` does not change (selection happens on drop, via the existing `widget-created` event handler in `App.vue`). The clamp's selection-watch hooks naturally — no drag-state guard needed unless the verify-at-start audit surfaces a mid-drag selection flow that didn't exist at 204.

- **Reduced-motion respect.** The existing focus-scroll utility checks `prefers-reduced-motion` for the smooth-scroll behaviour. The clamp itself is non-animated (it's a preventDefault, not an animation), so no parallel check is needed — but the focused-widget-centring on selection still goes through the existing utility, which already respects the preference.

- **No public-site / record-detail-view scope.** This lives in `resources/js/page-builder-vue/` only. The DashboardBuilder (no column layouts) and RecordDetailViewBuilder share some infrastructure with PageBuilder; if the clamp helper extracts cleanly into a composable that all three could use, that's fine — but lifting it into RDV / DashboardView is **out of scope** unless the user surfaces a UX request.

- **No persistent state.** The clamp is conversation-local; no per-user setting, no localStorage. Per the 204-time deferred decision pattern (inspector tab persistence, Timeline view mode), graduate to user-level setting only if feedback demands it.

## Phase 1 — Clamp implementation

**Goal:** scroll-lock + tall-widget exception live behind selection in the page-builder canvas.

**Steps:**

1. **New utility / composable** at [resources/js/page-builder-vue/composables/useFocusScrollClamp.ts](resources/js/page-builder-vue/composables/useFocusScrollClamp.ts) (composable shape preferred over a plain function — it owns lifecycle, listener registration, and the watcher). Inputs:
   - `paneRef: Ref<HTMLElement | null>` — the canvas pane that scrolls.
   - `selectedItemId: Ref<string | null>` — store-backed.
   - `selectedItemType: Ref<'widget' | 'layout' | null>` — store-backed.
   Internal:
   - On selection change, find the focused element via the same `[data-widget-id]` / `[data-layout-id]` selector pattern as `focusScroll.ts`; cache the resolved element.
   - Attach `wheel`, `touchmove`, `keydown` listeners on `paneRef.value` with `{ passive: false }` when selection is non-null.
   - Each handler:
     1. Resolve effective scroll delta (wheel: `event.deltaY`; touch: tracked via `touchstart` + current `touchmove` Y; keydown: per-key constant — Arrow ±line, PageUp/Down ±page, Home/End ±max, Space ±page).
     2. Walk from `event.target` up to `paneRef.value` looking for an ancestor that can absorb the scroll in the requested direction. If found, return without preventDefault.
     3. Compute proposed `paneRef.value.scrollTop + delta`.
     4. Compute `[scroll_min, scroll_max]` from the cached focused element + the pane's `clientHeight` per the math above.
     5. If proposed value is outside the bounds, `event.preventDefault()` and set `paneRef.value.scrollTop = clamp(proposed, scroll_min, scroll_max)` directly. (The direct set produces the natural "snap to the boundary" feel rather than just blocking the move.)
   - On selection clear, detach the listeners.
   - On component unmount, detach unconditionally.

2. **Hook into [PreviewCanvas.vue](resources/js/page-builder-vue/components/PreviewCanvas.vue)** alongside the existing focus-scroll-on-selection watcher. One call to `useFocusScrollClamp(paneEl, computed(() => store.selectedItemId), computed(() => store.selectedItemType))`. Composable owns lifecycle; no inline cleanup needed in PreviewCanvas.

3. **Optional in-session absorption — Escape-key release.** If the verify-at-start audit on the release signal confirms that Escape is not currently wired and the user agrees, add an Escape-key handler on the canvas pane that calls `store.clearSelection()` (or whatever the existing de-select action is). One-line change. Lift only if it costs <10 minutes; otherwise defer.

**Pass criterion:** focusing a short widget locks the canvas scroll position; focusing a tall widget bounds scroll to the widget's vertical extent; clicking the canvas background un-clamps; no console errors during scroll attempts; nested scroll inside any internal-scroll widget (if any exist per the audit) still works.

## Phase 2 — Tests

**Goal:** mechanical regression coverage on the clamp math + an end-to-end Playwright spec for felt behaviour.

**Steps:**

1. **Extract the clamp math as a pure function** so it's unit-testable in isolation. Probably:
   ```ts
   export function clampScrollPosition(
     proposed: number,
     widget: { offsetTop: number; offsetHeight: number },
     paneHeight: number,
   ): number
   ```
   Pure input → output; no DOM access. Lives next to or inside `useFocusScrollClamp.ts`.

2. **Vitest / Pest unit tests** on `clampScrollPosition`:
   - Short widget → returns the centred-target regardless of proposed value.
   - Tall widget, proposed within bounds → returns proposed unchanged.
   - Tall widget, proposed below `scroll_min` → returns `scroll_min`.
   - Tall widget, proposed above `scroll_max` → returns `scroll_max`.
   - Edge: widget exactly equal to viewport height → tall-widget branch with degenerate `[scroll_min, scroll_max]` of equal values.
   - Edge: widget at top of canvas (`offsetTop = 0`) — `scroll_max = 0`, no upward room.
   The project's JS tests live where? — Pest covers PHP; Vue / TS unit tests may not have a Vitest harness yet. **Verify at session start** — if no Vitest, write the math against existing test infrastructure or skip unit tests in favour of fuller Playwright coverage.

3. **New Playwright spec** at [tests/e2e/page-builder/focus-scroll-clamp.spec.ts](tests/e2e/page-builder/focus-scroll-clamp.spec.ts). Three tests:
   - **Short widget locks scroll.** Create a page with a tall stack of short widgets (10× a Text Block, say). Focus widget 5. Attempt to wheel down; assert canvas `scrollTop` does not move past widget 5's position. Wheel up; same assertion.
   - **Tall widget bounds scroll.** Create a page with one tall widget (e.g. a Carousel with content forcing height > viewport, or a Hero with explicit min-height). Focus the tall widget. Wheel within its bounds; assert `scrollTop` follows. Wheel past its top / bottom; assert clamp.
   - **De-select un-clamps.** Focus a short widget (locked). Click the canvas background. Wheel; assert `scrollTop` moves freely.
   Run via `npx playwright test tests/e2e/page-builder/focus-scroll-clamp.spec.ts`. Note: Playwright's synthetic wheel events behave slightly differently from real input (no inertia); tests should assert the scroll-position invariant, not the felt behaviour.

4. **Existing tests should not regress.** [tests/e2e/page-builder/full-width-matrix.spec.ts](tests/e2e/page-builder/full-width-matrix.spec.ts) (267) and [tests/e2e/page-builder/layout-inspector.spec.ts](tests/e2e/page-builder/layout-inspector.spec.ts) both navigate the page builder and select widgets; the clamp must not interfere with their existing scroll-into-view behaviour.

**Pass criterion:** ~3–6 new fast tests pass (count depends on whether unit tests land or only Playwright coverage); baseline 2163 → ~2166–2169 fast. Slow group unchanged at 16. New Playwright spec passes; existing two specs continue passing.

## Phase 3 — Manual verification

**Goal:** the felt-quality judgment call only the user can make.

The Playwright spec confirms the scroll-position invariant. The trackpad-inertia / mouse-wheel feel / mobile-touch behaviour are subjective.

The user should drive (in `npm run dev` page-builder mode):

- **Trackpad** (Mac, if available) — wheel scroll on a focused short widget; confirm the lock feels like "the page won't scroll past this widget" rather than "the page is broken." Inertia at the boundary should be visually quiet (no jitter / flicker).
- **Mouse wheel** — same drill on a non-trackpad surface.
- **Tall widget** — focus a Carousel or Hero with explicit tall content. Scroll within its bounds; confirm the felt behaviour is "scroll inside this widget, but you can't escape it."
- **Keyboard** — focus a short widget; press ArrowDown / PageDown / Space / End. Each should clamp to the widget bounds (no document scroll).
- **Drag-from-library** — drag a new widget from the library to the canvas; confirm the canvas auto-scroll during drag works (clamp does NOT engage during drag — drag-into-edge-to-scroll pattern preserved).
- **De-select** — click the canvas background after focusing; confirm scroll un-locks immediately.
- **Escape key** (if lifted in-session) — confirm pressing Escape while focused un-clamps.

If any of these feel visibly broken, surface to the assistant for in-session iteration before close. The 204 escape hatch is the precedent: better to ship the must-have alone than a fragile half-clamp.
