# 194. CRM Importer — Update-Existing Strategy for All Content Types

Generalise the contacts importer's `skip / update` duplicate-handling strategy to events, event registrations, donations, memberships, and transactions (invoice details) — so repeat imports during a migration can enrich previously-imported records instead of silently reusing the stale ones.

---

## Design decisions (resolved before starting)

- **Two strategies, not three.** Contacts exposes `skip / update / duplicate`. For the other four importers, `duplicate` doesn't make sense — external IDs on events/donations/transactions are unique by design, and forcing-duplicate would break the ImportIdMap invariant. Each non-contact wizard gets a two-option picker: **Skip existing** (current behaviour — reuse match, don't modify) or **Stage updates** (new — non-blank CSV values become an `ImportStagedUpdate` row awaiting reviewer approval).
- **Staging is polymorphic, not per-type.** `import_staged_updates` gains `subject_type` (string) + `subject_id` (uuid) columns; the existing `contact_id` column is dropped in the same migration (contacts' staged-update rows migrate to `subject_type=contact, subject_id=<uuid>`). One table, one approval path, five entity types. Keeps the review/approve/rollback UI uniform and matches the polymorphic patterns already used elsewhere (notes, taggables).
- **"Stage" always means review-required.** Staged updates never apply immediately at commit time — only at reviewer approval. This matches contacts. Rollback = delete the staged rows without applying. The existing `ImporterPage` approve/rollback actions are extended to dispatch per-subject-type.
- **Update semantics: CSV non-blank overwrites existing.** If a CSV column has a value and the corresponding column is mapped, that value replaces whatever's on the existing record (even if the existing value is populated). Blank CSV cells mean "no change," not "blank out the field." This matches what contacts does today via `array_filter($attributes, fn ($v) => $v !== null)`; the stub's "fill-blanks-only" phrasing is being corrected to this more accurate description.
- **Only wizard-mapped fields participate.** No field gets touched unless it's mapped to a destination in the Map Columns step. No implicit field expansion (the model's full fillable array is not the surface).
- **Children of updated parents are untouched.** Updating an Event's `starts_at` does not cascade to its Registrations. Updating a Transaction does not modify linked Donation/Membership records. Relationships set at create time are preserved.
- **Match keys per importer stay as they are today.**
  - Events: `external_id` via ImportIdMap.
  - Event Registrations: matched implicitly by `(event_external_id + contact_match_key)`. Kept write-once; no registration "update" path in this session — registrations are rarer to re-import and the semantics are murkier. Deferred to a follow-up.
  - Donations: `external_id` if present; if absent, always create (no match).
  - Memberships: `external_id` if present; if absent, fall back to `(contact + tier + starts_at)` like today.
  - Invoice Details (transactions): `invoice_number` (already used to collapse rows into line items).
- **Scope of this session: four importers × one new strategy.** Events, Donations, Memberships, Invoice Details get the full path. Event Registrations get a stub note in the log explaining why they're out. Contacts is unchanged; this session generalises, it does not rewrite.

## Phase 1 — Schema: polymorphic `import_staged_updates`

Add a migration that:

1. Adds nullable `subject_type` (string) + `subject_id` (uuid) to `import_staged_updates`.
2. Backfills existing rows: every row gets `subject_type='contact'` and `subject_id` copied from the existing `contact_id` column.
3. Drops the `contact_id` column (and its FK / index).
4. Makes `subject_type` + `subject_id` NOT NULL and indexes them together.

Update `docs/schema/import_staged_updates.md` to reflect the final shape. Update `app/Models/ImportStagedUpdate.php` to expose a `subject()` `morphTo` relation; remove the `contact` relation or alias it for compatibility (prefer remove; contacts' code paths update to use `subject`). Update `ImportContactsPage::processRow`'s staging call and `ImporterPage`'s approve/rollback closures to the new shape.

No data is lost; no contacts tests should need behavioural changes, only field-name renames.

## Phase 2 — Wizard: add the strategy picker to the four non-contact wizards

In each of `ImportEventsPage`, `ImportDonationsPage`, `ImportMembershipsPage`, `ImportInvoiceDetailsPage`, add a Radio field to the Map Columns step's schema (after the existing match-key/identifier controls) named `duplicate_strategy` with options `skip` / `update`, default `skip`. The field is **required**. Its value flows into `ImportLog::duplicate_strategy` the same way the contacts importer does today.

Factor the Radio into a shared helper on `InteractsWithImportWizard` if the four call sites are near-identical. Otherwise copy-paste is fine — the call sites are small.

## Phase 3 — Progress trait: staging path per importer

In `InteractsWithImportProgress`, generalise what's available for non-contact pages. Add a protected helper `stageSubjectUpdate(Model $subject, array $attributes): void` that writes a polymorphic `ImportStagedUpdate` row scoped to the current `importSessionId`. Called by each non-contact progress page's `processOneRow()` when `duplicate_strategy === 'update'` and a match was found.

In each of the four non-contact progress pages, extend `processOneRow()`:

- When an existing record is matched (events via ImportIdMap, donations via external_id, memberships via external_id or composite, invoice-detail transactions via invoice_number) AND `duplicate_strategy === 'update'`: build the `$attributes` map of non-blank CSV values mapped to fields, call `stageSubjectUpdate()`, and return outcome `'updated'`.
- When matched AND `duplicate_strategy === 'skip'`: return outcome `'skipped'` with `skipReason='duplicate_skipped'`, same as today.
- When no match: create as new, return `'imported'`, same as today.

The dry-run path already accumulates `'updated'` in the report — confirm each page's `accumulateOutcome()` routes to the right counters and preview blocks. The blade templates for the four non-contact progress pages need to render the "Would update" count (currently only contacts does; the other four show only "Would import / Would skip / Errors").

## Phase 4 — Approval path: apply staged updates polymorphically

In `ImporterPage::approve` action, replace the contacts-only staged-update loop with a polymorphic one that iterates every `ImportStagedUpdate` scoped to the session, resolves `$update->subject()`, and calls `$subject->fill($update->attributes)->save()`. The contacts-specific sync of `tag_ids` stays — only contacts carry `tag_ids`; for other types it's null and the existing `if (! empty($update->tag_ids))` check is enough.

The "Changes applied from import" note currently written per contact is contacts-only. For the four new types, no note is written — they're either audit-logged elsewhere (Events/Registrations have their own audit) or silent (Donations/Memberships/Transactions are financial records where an approval note doesn't match the model). Confirm or revisit in iteration.

`rollback` and `delete` actions already call `$staged->each->delete()` — polymorphic rows already get deleted correctly; no change needed beyond the earlier refactor to remove contacts-only assumptions.

## Out of scope

- **Event Registrations update path.** Re-import semantics for a registration row are ambiguous (what makes a registration "the same"? the same contact+event combo, but also: same ticket tier? same registered-at timestamp?) — deferred to a follow-up session.
- **Collision UI** on the Map Columns step (contacts' side-by-side resolver) for the non-contact wizards. Their schemas are simpler; most overlap gets caught by mapping UX itself.
- **`duplicate` strategy** for non-contact types. Breaks ID-map invariants; not in scope.
- **Audit log / history per staged update.** The `reviewing` banner + approve/rollback is the audit surface today; a proper audit trail is a separate feature.
- **Fields the user *cannot* update** (e.g. `payment_status` after a donation has been receipted). Caveats belong in a follow-up session if clients push back during testing.
