## What 338 built (the system you're fixing)

A driver.js guided walkthrough that runs **inside the Filament admin** on the demo node, launched manually from a dashboard "Tour the Product" button (`data-np-tour-start`):

- **11 steps** (`resources/js/admin/tour/tours/product-tour.js`) across multiple pages: Welcome → Contacts (click a contact) → the contact record's **membership** panel → "View transactions" → Transactions → Mailing lists → Events/website → Importer → Roles → help search → help flyout.
- A **multi-page engine** (`controller.js`) that drives one contiguous page-group per driver instance, persists the active step index to `localStorage` (`state.js`), hard-navigates across page boundaries, and resumes on load.
- **Owned `data-tour` anchors** (injected via render hooks / Blade, never Filament `.fi-*` classes), an `anchors.js` `waitForElement` primitive (8 s / 80 ms poll, resolves on element **existence**), and **Livewire-morph robustness** (self-target anchors handed a live CSS selector so driver re-queries a fresh element each highlight).
- A **role-gated URL map** (`window.__npTour`, emitted from `AdminPanelProvider`'s `HEAD_END` hook) so steps for pages the viewer can't reach are skipped; **demo-safe showcases** stand in for the locked Roles / Import pages.
- Controls live **only on each driver popover** (`Back` / `Next` / `Exit Tour` + a "Step X of Y" progress span). **There is no persistent control bar** — so when a popover is mispositioned (the membership bug), the user has no other way to advance.
- Standing regression spec `tests/e2e/tour/product-tour.spec.ts` walks all 11 steps + interactivity + cross-page resume.

## Open questions to resolve at session start

These propagate or need owner taste — settle them at the Open Gate, don't commit-and-unwind.

1. **Iteration split.** Recommended: a cohesive session — **the membership async-framing fix + Playwright verification**, then **the persistent controls + whole-tour flow review + step rework** — landing together at the normal session cadence. The membership fix does **not** need to ship ahead of the rest (the owner confirmed this at planning). Run as two iterations or one, as the work shakes out; confirm.
2. **Membership-race fix mechanism (load-bearing).** The likely root cause (confirm against code + a Playwright repro): `waitForElement` resolves the moment the `[data-tour="record.membership"]` marker **exists** in the DOM, but the membership panel's **layout geometry hasn't settled** (other record-detail widgets / the Livewire form above it are still laying out, or the panel's own content grows after the marker appears), so driver.js positions the spotlight/popover against stale bounds and lands "too low." `RecordDetailViewWidget` is already `$isLazy = false` (eager) — necessary but not sufficient. **Recommended lean:** after resolving the element, wait for **settled geometry** (e.g., poll `getBoundingClientRect()` until stable across two animation frames, or gate on a render-complete signal), then `driver.refresh()` (or re-highlight) to reposition — generalised so it protects every step, not just membership. Confirm the mechanism before building.
3. **Persistent-controls shape (needs taste).** A **second set of fixed controls**, always visible regardless of popover position, so the user always has a clear way to advance/back/exit. **Recommended lean:** a fixed bar (bottom or top) with `← Back` / `Next →` (or `Done`) / `Exit` and a "Step X of Y" indicator, wired to the same `controller.js` advance/back/stop paths the popover buttons use (single source of truth — no divergent state). Confirm: bar position; whether it **supplements** the popover buttons (recommended — redundancy is the point) or **replaces** them; styling that reads on the dark tour overlay; and that it must not break cross-page resume or leak past tour exit.
4. **Step add / rework scope (owner judges).** The owner reports the tour is "confusing as is" and wants "logical steps between each item," possibly adding a few or reworking what's there. **Method:** a Playwright walkthrough that screenshots each of the 11 steps into a storyboard, flags illogical transitions / gaps, and proposes concrete add/reword changes — the owner approves the copy/step set before any edit. Don't rewrite tour copy unilaterally.
5. **Auto-start?** A 338 deferred decision (auto-launch on demo arrival with a seen-suppress). **Recommended: out of scope** unless the owner asks — keep the manual launcher; this session is framing + controls + flow.
