## Phase 1 — PII / Sensitive data rejection

### Goal

Before processing any row, the importer must scan the CSV for data that should never enter the system — credit card numbers, SSNs, and bank routing numbers. If any match is found, the entire import is rejected immediately with a clear error message. No partial imports.

### What to detect

Two layers of detection, both enforced:

**Layer 1 — Column name blocklist**

If any CSV column header matches a blocked name (case-insensitive, ignoring spaces and underscores), reject the upload before any row processing. Blocked names:

```
ssn, social_security, social_security_number, credit_card, credit_card_number,
card_number, pan, routing_number, account_number, bank_account, aba_routing,
driver_license, drivers_license, dl_number
```

**Layer 2 — Cell content patterns**

Scan every cell value in every row. Reject if any cell matches:

- **Credit card PAN** — 13–19 digit sequences (with optional spaces or dashes) that pass Luhn algorithm check
- **SSN** — pattern `\d{3}-\d{2}-\d{4}` or `\d{9}` (9 consecutive digits)
- **ABA routing number** — 9-digit number whose first two digits fall in the ranges 01–12 or 21–32 (Federal Reserve routing range)

### Where to enforce

Enforce at the job level inside `ImportProgressPage` (or whatever job/service processes the file), **not** in the form step. This ensures the check cannot be bypassed by altering the UI flow.

The check runs on the full file before any rows are written to the database. If any violation is found:

1. Set the `import_session` status to `failed` (add this status if not already present)
2. Write the violation detail to the import log or session record (which column/row triggered it)
3. Surface a clear error notification to the user: "This import was rejected because it contains data that may include sensitive financial or personal identifiers. Remove the flagged data and try again."
4. Do not create any contact records

### Bypass flag

Add an `.env` variable `IMPORTER_SKIP_PII_CHECK=true`. When set, skip both layers. This flag is for developer use only — document it in `.env.example` with a prominent warning comment. Do not expose this setting in the admin UI.

### Error display

The ImportProgressPage already shows status messages. Add a "rejected" state that:
- Shows the rejection reason
- Does not show a "contacts imported" count
- Offers a link back to start a new import

## Phase 3 — Custom field expression indexes

### Goal

When a custom field is created with `is_filterable = true` (or when an existing field is marked filterable), create a PostgreSQL expression index on the `contacts.custom_fields` JSONB column for that field's handle. This makes mailing list filter queries using custom fields fast at scale.

### Schema change

Add `is_filterable` boolean column to `custom_field_defs` (default `false`). Write the migration.

Update `CustomFieldDefResource` to include an `is_filterable` toggle in the create/edit form.

Update `CustomFieldDef::$fillable` and `$casts` for the new column.

### Index creation

When a `CustomFieldDef` is saved with `is_filterable = true`, create a PostgreSQL expression index:

```sql
CREATE INDEX IF NOT EXISTS idx_contacts_cf_{handle}
ON contacts ((custom_fields->>'{handle}'));
```

When `is_filterable` is set to `false` on an existing field, drop the index:

```sql
DROP INDEX IF EXISTS idx_contacts_cf_{handle};
```

Implement this via an Eloquent observer (`CustomFieldDefObserver`) on the `saved` event. The observer compares the old and new values of `is_filterable` and creates or drops the index accordingly.

**Important constraints:**
- Index names must be safe: only use the field `handle` (which is already validated as alphanumeric + underscores) in the index name
- Use `DB::statement()` to run the raw SQL — do not use schema builder for this, as Laravel's schema builder does not support expression indexes
- Wrap the statement in a try/catch — if the index already exists (e.g. from a previous run), `IF NOT EXISTS` prevents an error
- Do not create indexes for non-contact fields in the first version — scope this to `model_type = 'contact'` only

### Existing fields

Existing `CustomFieldDef` records that had no `is_filterable` column default to `false` — no indexes created retroactively. An admin must explicitly enable filterability on each field if they want the index.
