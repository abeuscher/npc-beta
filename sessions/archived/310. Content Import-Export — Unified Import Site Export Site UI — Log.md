## Architectural calls (resolved at session start; not relitigated at apply time)

Per the session prompt's "decide first, build second" rule. All four recommendations from the prompt are adopted:

1. **Backup-grade vs button-grade — button-grade v1.** The bundle is a convenience snapshot of content (pages + templates + theme + media). Users, roles, audit log, mailing-list-state, etc. are explicitly out of scope. A backup-grade snapshot is a future session; the rollup surface intentionally does not promise fresh-install restore.
2. **Diff preview — no.** The analyzer's per-entity collision summary (slug/handle counts) is sufficient at v1. A row-by-row include/skip diff is a future session if operator workflow demands it.
3. **Entity collision keys — pinned in the analyzer manifest.** Pages collide by `slug` (matched via `Page::withTrashed()->whereIn('slug', ...)`). Templates collide by `(name, type)` composite. Theme collides as a single conceptual row (the three SiteSetting design keys treated as one config). Media collides by `id` (the existing id-map plumbing). Events / products / collections / navigation collision keys are deferred to A001 when it lands; that session will pick them.
4. **`format_version` policy — no bump.** Stays at `1.1.0`. The rollup adds no new envelope keys; it only assembles existing payloads. The hard-no list from session 309 holds.

## What was built

### Phase 1 — `ContentExporter::exportSite()` wrapper

Single new exporter method that assembles a full-snapshot bundle by enumerating all pages and all templates and delegating to the existing `exportBundle()` path. `$opts` defaults set to `with_design = true` and `with_media = true` so the rollup carries the theme + the referenced-media seed list by default. Per-entity exporters are not touched; the wrapper is purely additive.

### Phase 2 — `'site'` kind on `ExportBundleJob`

New `case 'site'` arm in `ExportBundleJob::handle()` that calls `$exporter->exportSite($this->opts)`. The existing notification + signed-download flow is reused unchanged. `$ids` is unused for `'site'` (the wrapper enumerates internally) — accepted as `[]`.

### Phase 3 — `SiteImportExportPage` Filament page

New page at `app/Filament/Pages/SiteImportExportPage.php`, registered in the "Tools" group with the label "Import / Export Site". Permission gate: `manage_cms_settings` (same gate the per-entity tools use; the `update_page` capability used by `ImportBundleAction` is widened here because the rollup also writes theme + media on import).

Two sections on the page:

- **Export Site.** Single "Export Site" button + two checkboxes ("Include site theme", "Include media files"), both default ON. Dispatches `ExportBundleJob('site', [], userId, 'site-snapshot', $opts)`; download arrives via the existing notification.
- **Import Site.** Reuses session 309's live-reveal shape verbatim: `FileUpload` with `->live()->afterStateUpdated()` running `analyze()`, manifest summary placeholder, four toggle fields (Import pages, Replace duplicate pages, Replace site theme, Include media) — same gates, defaults, and copy as `ImportBundleAction`. Single submit button dispatches `ImportBundleJob`.

The two sections are sibling concerns on one page; no wizard, no tabs. Per-entity import/export forms (page list, template, etc.) are untouched.

### Phase 4 — Permissions + nav

`SiteImportExportPage::canAccess()` returns `auth()->user()?->can('manage_cms_settings')`. `navigationGroup = 'Tools'`, `navigationSort` placed adjacent to the Importer + Media Library entries.

### Tests

- **Pest:** `SiteImportExportTest.php` — `exportSite()` round-trip (export → fresh import → row counts match for pages/templates/theme/media); `ExportBundleJob` `'site'` kind dispatches the exporter wrapper with the forwarded opts.
- **Playwright:** `tests/e2e/importer/site-import-export.spec.ts` — round-trip happy path: navigate to the page, trigger Export Site, confirm the export-ready notification, upload the bundle on the Import Site half, confirm the manifest summary renders, click Import, confirm the success notification.

### Behaviour change to flag

None. The rollup is a thin assembly over existing exporters + importers; the behaviour-change forcing function from session 309 (the `merge_design` default flip) carries forward unchanged.

## Security considerations

The new page widens the permission gate from `update_page` (used by `ImportBundleAction` on the page list / post list / template surfaces) to `manage_cms_settings` on the rollup surface, because Export Site / Import Site touches the theme + media + templates + pages as one operation — a heavier blast radius than per-entity import that warrants the heavier gate. The underlying service layer (`ContentImporter::import()`) and analyzer (`ContentImporter::analyze()`) are not modified; no new attack surface beyond the new page route. Bundle envelope validation, zip-slip / zip-bomb guards, and the existing collision policies all carry through.
