## What was built

### Phase 1 — Retire dead `$widgetAssets` collection

A read-only deduplicated path list — `['css' => [], 'js' => [], 'scss' => []]` — was being populated in four places via `WidgetRenderer::collectAssets()` and threaded through `PageBlockRenderer::renderLayoutBlock()` + `ChromeRenderer::renderLayoutBlock()` + `PageController::renderPage()` + `PostController::renderPage()` + a return-shape key on `ChromeRenderer::render()/renderById()`. **No view ever consumed it.** Grep across `app/`, `resources/`, `tests/` returned zero readers. ~20 lines of inert plumbing.

Removed:

- `app/Services/WidgetRenderer.php` — `collectAssets()` method, plus the now-unused `App\Models\WidgetType` import.
- `app/Services/PageBlockRenderer.php` — `&$widgetAssets` reference parameter on `renderLayoutBlock()`; the per-block `WidgetRenderer::collectAssets()` call.
- `app/Services/Media/ChromeRenderer.php` — same reference-parameter removal on the private `renderLayoutBlock()`; the two `collectAssets()` calls (widget loop in `renderPage`, layout loop in `renderLayoutBlock`); the `$assets = ['css' => [], 'js' => [], 'scss' => []]` local in `renderPage`; the `'assets' => $assets` key on the return; the matching `assets:` clause in the PHPDoc return-shape annotation on both public render entrypoints.
- `app/Http/Controllers/PageController.php` — `$widgetAssets` local; the call-site arg to `renderLayoutBlock`; the `'widgetAssets'` key in `compact(...)`; the unused `App\Services\WidgetRenderer` import.
- `app/Http/Controllers/PostController.php` — same set of removals.

**Guard test:** `tests/Feature/WidgetAssetsCollectionRetiredGuardTest.php`. Pure source-grep — `expect($src)->not->toContain(...)` against the five edited files. Cheap regression guard against the surface getting re-introduced as a copy-paste from older patterns.

### Phase 2 — Retire per-instance inline widget CSS/JS emission

The public layout was emitting two catch-all `<style>` and `<script>` blocks fed by accumulators threaded back from `WidgetRenderer::render()`, which (at lines 161-174 of the pre-A002 file) concatenated `widgetType->css` and `widgetType->js` into them for every rendered widget in both the `server` and `client` render-mode branches. Meanwhile `AssetBuildService::build()` already walks `WidgetType::whereNotNull('css')->orWhereNotNull('js')` at lines 477-495 and ships that exact content via `public/build/widgets/manifest.json` → `<link>` + `<script>` in the layout. So the two paths shipped overlapping content: a no-op for seeded widgets (no DB css/js) and a per-instance double-emit footgun the moment any operator filled the Filament `WidgetTypeResource` css/js textareas (a five-instance page would emit the same custom CSS five times inline plus once in the bundle).

Removed:

- `app/Services/WidgetRenderer.php` — the three `if ($widgetType->css)` / `if ($widgetType->js)` accumulator appends (two in the `server` branch, one in the `client` branch). The function still returns the `['html', 'styles', 'scripts']` shape; `styles` and `scripts` are now always the empty string because no callee or template writes to them.
- `resources/views/layouts/public.blade.php` — the `@php $__allInlineStyles = ...; @endphp` + `<style>$__allInlineStyles</style>` block and the matching `@php $__allInlineScripts = ...; @endphp` + `<script>$__allInlineScripts</script>` block. These also folded in `$__chromeHeader['styles']` / `$__chromeFooter['styles']` / `[scripts]`, which become empty alongside `WidgetRenderer`'s emission and so come along for the ride.

**Close-gate regression:** `tests/Feature/PublicLayoutBundleConsolidationTest.php`. Seeds two widget types with distinctive marker CSS + JS strings, attaches them to a published Page, writes a fake `public/build/widgets/manifest.json` (snapshot+restore in `beforeEach`/`afterEach` so a real bundle is never polluted), GETs the page through the real routing + `PageController` stack, asserts:

1. exactly one `<link [...] href="/build/widgets/...">` is emitted, sourced from the manifest;
2. exactly one `<script [...] src="/build/widgets/...">` is emitted;
3. neither widget-type CSS marker string appears anywhere in the response body;
4. neither widget-type JS marker string appears anywhere in the response body.

(3) and (4) are the load-bearing assertions — they fail loudly if the inline path is re-introduced for any reason.

### Phase 3 — Template `custom_scss` to bundle

**Decision: Option B (defer).** The session prompt's recommendation defaults to B unless Phases 1 + 2 land in under three iterations *with headroom*, where headroom includes a working signal on the full test suite. Phases 1 + 2 landed in a single iteration but the remote-execution environment for this agentic session has no Docker daemon and so cannot run the canonical `./dev test` PostgreSQL-backed Pest suite locally — there is no "headroom" in the prompt's sense. Option A would introduce a `body.page-using-template-{handle}` class contract that ripples through the layout and possibly the page-builder canvas; landing it blind to the suite is not a trade I'm taking when the cleanup goal (one bundled `<link>`, one bundled `<script>`, no per-instance widget CSS/JS leaks) is already substantially met by Phases 1 + 2. The template SCSS inline `<style>` block at `public.blade.php:177-191` is a single bounded artifact (not per-instance), still backed by the in-code "moves to build server post-beta" comment, and reasonable to defer to a numeric session that can verify against the full suite.

None beyond standard convention. The session removes redundant emission paths; nothing newly accepts user/operator input, and the surviving paths are unchanged. Operator-customised `widget_types.css` / `widget_types.js` content now ships *only* through the build-server bundle pipeline, which has not been altered — so the existing trust boundary on that field (operator-only Filament field) carries forward unchanged. No new XSS surface; the inline `<style>` / `<script>` block that *did* exist has been removed (smaller attack surface, not larger).

## Production deployment notes

None beyond standard `migrate`. After deploy: the rendered `<head>` of public pages will shed the per-widget-instance `<style>` and `<script>` blocks. If any operator had filled the Filament `WidgetTypeResource` css/js textareas (the latent-footgun case the diagnosis flagged), that content continues to ship via the build-server bundle as before — no behavioural change *for that operator*, only a stop on the duplicate inline emission. On first deploy after merge, worth a quick spot-check that public pages still render with the bundled widget styles applied (i.e. `public/build/widgets/manifest.json` is present and the layout's bundle `<link>` is firing). If the manifest is missing for any reason (build-server hiccup), pages will render without the bundled widget CSS — but the previous inline path would not have rescued that either, because the bundled content is sourced from the same DB rows the inline path read; so this is not a regression versus pre-A002 behaviour for any seeded-widget install.

## Deferred decisions

- **Phase 3 (template `custom_scss` → bundle):** deferred per the rationale above. Numeric-session work whenever it makes sense; the inline `<style>` block at `public.blade.php:177-191` plus the in-code "moves to build server post-beta" comment is the current state.
- **`string &$inlineStyles, string &$inlineScripts` reference parameters on `PageBlockRenderer::renderLayoutBlock()` + `ChromeRenderer::renderLayoutBlock()` + the matching locals on the controllers + `compact()` keys:** explicitly out-of-scope per the session prompt's "do not expand scope" rule. After Phase 2 they are always-empty accumulators (nothing writes to them — the only writer was the per-instance widget css/js append in `WidgetRenderer::render`, which Phase 2 removed). A follow-up cleanup pass can drop them once it is clear nothing else relies on the shape; this session leaves them intact.
- **`resources/views/layouts/portal.blade.php`** still carries an `@if (!empty($inlineStyles)) <style>{!! $inlineStyles !!}</style> @endif` block (mirrored for scripts). The portal layout is a separate render path and not in scope this session; the `@if` conditional gracefully no-ops when the accumulator is empty, so no broken HTML — but a follow-up cleanup pass could remove it for the same "always-empty" reason as the public layout's removed blocks.
