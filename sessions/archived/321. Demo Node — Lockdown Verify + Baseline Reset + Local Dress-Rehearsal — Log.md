## What was built

The CRM-side half of the demo-node work: verify the public auto-login is genuinely walled off, widen the interior as far as is safely possible, build a clean baseline-reset, and dress-rehearse the whole visitor path locally. The FM-side half (node identity, version pin, egress firewall) is handed off separately — see Production deployment notes.

### Phase 1 — Standing lockdown guard (`tests/Feature/DemoRoleLockdownTest.php`)

- Asserts a `demo`-role user is bounced at every real Filament `canAccess()` wall — settings (mail / finance / routing / CMS, which is where the Stripe/QuickBooks/Mailchimp/Resend API keys live), user + role management, the data-import tools, and site export + Media Finder (bulk real-data exfiltration + media delete).
- Asserts the scrub-data wipe / data-generator routes return **403** (super_admin-only) for the demo user.
- Asserts the binding deny-list permissions are denied and the user is never `super_admin`.
- Asserts the *intended granted width* (Phase 2) so a future narrowing is also caught.
- Guards against `canAccess()`, not just the permission vocabulary — so a future permission swap on any of those pages can't silently widen the public demo.

### Phase 2 — Demo-role width tuned (`database/seeders/PermissionSeeder.php`)

- **Events and donations widened to full CRUD** (were view-only) so a prospect can drive those flows. Backstopped by the droplet egress firewall (Stripe/email fail closed) and the daily reset.
- Transactions / funds / campaigns stay view-only (financial ledger + config). Still an allow-list with the binding deny-list intact.

### Phase 3 — `demo:reset` baseline command + seeder

- `app/Console/Commands/DemoResetCommand.php` — `demo:reset` (full: `migrate:fresh --seed` + baseline) with a `--soft` flag that re-seeds the CRM baseline only. **Refuses to run unless `isDemoMode()`**, so a misconfigured cron can never wipe a production database.
- `database/seeders/DemoBaselineSeeder.php` — the curated, idempotent baseline (≈40 contacts, 6 events, 25 registrations, 30 donations, 15 memberships, 5 posts, 6 products). Runs `RandomDataGenerator` under a resolved-or-minted super_admin acting context (events/pages carry a NOT NULL `author_id` and the generator gates on super_admin).
- `tests/Feature/DemoResetCommandTest.php` — the non-demo guard (wipes nothing) + soft-reset baseline counts + idempotency (re-running returns the same baseline, not a doubled one).

### Phase 4 — Local dress-rehearsal (isolated stack, demo mode)

- Ran the isolated e2e stack flipped to `APP_ENV=demo` (a transient, uncommitted compose override) so the dev database was never touched. Confirmed end-to-end: `demo:reset` rebuilds + seeds cleanly; baseline counts match exactly (40/6/25/30/15/5/6 + 37 generated transactions); `/demo/enter` → 302 → `/admin`; mail forced to `log`; and a **live authenticated lockdown check** — as the demo user, `/admin/users` and `/admin/roles` return 403 while `/admin/contacts` and the newly-widened `/admin/events` return 200. Stack torn down (`down -v`) and the override removed afterward.
- No demo-mode Playwright spec was added: the wall is permission-gated (`canAccess`), so the Pest guard is the faithful durable check and runs in the normal suite; a demo-mode spec would need an `APP_ENV=demo` stack that standard CI doesn't run.

## Security considerations

This session is squarely a security surface — a public, internet-facing auto-login. Reviewed and confirmed: the `demo` deny-list holds in practice at every `canAccess()` wall and the wipe routes 403 (the new standing Pest guard is the regression lock). The Phase 2 widening (events/donations CRUD) is deliberate and bounded — it adds product-feel write flows whose abuse class is covered by the FM-side egress firewall (outbound fails closed) and the daily reset, and it does not touch the binding deny-list (settings, users/roles, API keys, integrations, exfiltration, never super_admin). `demo:reset` is guarded on `isDemoMode()` so it cannot wipe a non-demo database. `DemoBaselineSeeder` resolves an existing super_admin or mints a system one with an unguessable random password and no auto-login route pointing at it — consistent with the existing shared demo-user creation pattern; it is never exposed via `/demo/enter`.

## Production deployment notes

- **Daily reset:** the demo node runs `php artisan demo:reset` on a cron (local reseed, no egress — the confirmed mechanism over FM `RestoreFromBlob`). Requires `APP_ENV=demo` in the FM-generated `.env`.
- **⚠ Faker / `--no-dev` image blocker (must resolve before the demo node goes live).** `demo:reset`'s baseline step drives `RandomDataGenerator`, which uses model factories → `fakerphp/faker`, which sits in **`require-dev`**. The production image is built `--no-dev`, so on the real demo droplet `demo:reset` will fail the baseline step with `Class "Faker\Factory" not found` (surfaced during the dress-rehearsal — worked around by `composer install` in the disposable stack). Resolution is a deliberate decision (see Deferred) — likely either build the demo node from a dev-deps image (the `public-dev` build the e2e stack uses) or move `fakerphp/faker` to `require`. The `migrate:fresh --seed` half is unaffected; only the curated CRM baseline needs faker.
- **FM-side, handed off** (`/home/al/fleetmanager/sessions/demo-node-handoff-from-crm.md`): demo node identity, `IMAGE_TAG` version pin, and the egress firewall (deny outbound except DNS/NTP/GHCR; block SMTP) — the one security control that caps the abuse blast radius. None of these are CRM-side.
