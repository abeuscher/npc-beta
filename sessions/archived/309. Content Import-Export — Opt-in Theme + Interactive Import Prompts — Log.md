## What was built

### Phase 1 — Exporter opt flags (`with_design`, `with_media`)

- `ContentExporter::exportPages($ids, array $opts = [])`, `exportTemplates($ids, $opts)`, and `exportBundle($pageIds, $templateIds, $opts)` all accept an opts array. Flags: `with_design` (attach `payload.design` from the three SiteSetting design rows), `with_media` (attach a posture-B `payload.media` seed list collected from the descriptors of the serialized pages + template widgets). Both default false — opt-in.
- Shared helper `attachOptIns()` mutates the payload; new `collectReferencedMedia()` walks the serialized page/template trees, extracts `{id}` from each media descriptor path, and emits Media rows in posture-B descriptor form (same shape as `exportMedia()`).
- `ExportBundleJob` constructor gained `public array $opts = []`; `handle()` forwards opts to `exportPages` / `exportTemplates`. The other kinds (`design`, `media`, `all_media`) ignore opts — they're already explicit.

### Phase 2 — Importer `analyze()` + opt flags

- New `ContentImporter::analyze(array $bundle): array` — pure inspection, validates envelope first, then returns:
  - `has_design: bool`, `design_keys: ['theme_colors'|'typography'|'button_styles', ...]` (intersected with present)
  - `has_media: bool`, `media_count: int`
  - `pages: [{slug, exists_locally: bool}]` (single batched `Page::withTrashed()->whereIn('slug', ...)` lookup)
  - `templates: [{name, type, exists_locally: bool}]` (single batched `Template` lookup)
  - Never writes; the UI uses it pre-flight to decide which questions to ask.
- New `ContentImporter::defaultImportOpts()` static + `import($bundle, $log, $opts = [], ?$mediaRoot = null)` signature. Opts: `merge_design`, `import_media`, `import_pages`, `replace_duplicate_pages`. **`merge_design` default flipped to FALSE** — the behaviour-change the session was scoped around. `import_media` defaults true. `import_pages` defaults true. `replace_duplicate_pages` defaults true at the service layer (UI defaults the checkbox OFF to be conservative).
- Each existing import branch is now gated on its flag. Skip paths log an `info` entry into the `ImportLog` so the operator sees why a payload was ignored.
- `ImportBundleJob` constructor gained `public array $opts = []`; `handle()` forwards opts to `import()`.

### Phase 3 — Importer UI (live-reveal, single-page)

- `ImportBundleAction` rewritten. Started as a two-step `Filament\Forms\Components\Wizard`; collapsed to a single-page **live-reveal** at the user's request after the Wizard's internal "Next" button contradicted the modal's "Import" submit. Final shape:
  - `FileUpload` with `->live()->afterStateUpdated(...)` runs `analyze()` on the temp upload the moment FilePond finalises, seeds the manifest into form state, pre-populates toggle defaults. If the bundle is invalid, sends a danger notification and resets manifest (no throw — operator can swap files without re-opening the modal).
  - `Placeholder` summary + four `Toggle` fields render conditionally below the upload, each visibility-gated on its corresponding manifest field. No fields appear until the manifest is populated; clearing the file hides them again.
  - Four toggles, all with `data-testid`:
    1. **Import N pages** (default ON, gated on `pages.length > 0`; `->live()`)
    2. **Replace duplicate pages** (default OFF, gated on `hasDuplicatePages`, `->disabled()` when "Import pages" is off)
    3. **Replace site theme** (default OFF, gated on `has_design` — the session's forcing function)
    4. **Include N media files** (default ON, gated on `has_media`)
  - Single modal submit "Import" dispatches `ImportBundleJob` with the four-key opts dict derived from the toggle states. `import_pages = false` is enforced at the service layer (skips ALL page entries, both fresh-slug and slug-collision; can result in a 0-page import per the user's explicit design intent).
- Helper `resolveUploadedAbsPath()` handles both shapes Filament's FileUpload state can be in: `Livewire\Features\SupportFileUploads\TemporaryUploadedFile` during the live analyze (in-flight, not yet moved to `imports/bundles`), and the relative-path string on the final action() callback.

### Phase 4 — Export UI (four options on all six surfaces)

Page list bulk-action, post list bulk-action, page edit, post edit, page template edit, content template edit — each gains two new entries alongside the existing two:

1. Export selected / Export Page / Post / Template (JSON, content only) — existing
2. Export with media (zip) — existing, now dispatches `['with_media' => true]`
3. **Export with theme (JSON)** — NEW, streams via `exportPages($ids, ['with_design' => true])`
4. **Export with theme & media (zip)** — NEW, dispatches `ExportBundleJob` with both opts

Both new entries `->requiresConfirmation()` with an explicit modal warning about the inbound-theme override the bundle will produce.

### Behaviour change to flag

Any caller that previously imported a bundle containing `payload.design` and expected the silent merge will now see the design payload ignored unless `['merge_design' => true]` is passed. This was the session's forcing function — surfaced at the UI layer as the "Replace site theme" toggle, default OFF.

## Security considerations

The integration-seam fix tightens an existing implicit-write path: bundle imports can no longer silently overwrite Theme editor settings without an explicit operator opt-in. `analyze()` is pure inspection — no DB writes, no storage writes, no bytes-on-disk side effects; the zip-extraction path it uses for envelope reading immediately tears down its temp dir. The `replaceDuplicatePages` and `importPages` opts default to safe-skip in the UI flow even though the service-layer default preserves the existing overwrite semantics for direct callers. No new attack surface; no permission gate changes (still `update_page` for both export and import actions). `BundleArchive`'s zip-slip / zip-bomb guards are unchanged and still active on the analyze + import paths.

## Production deployment notes

None beyond standard `migrate`.

**Worker restart required:** `ExportBundleJob` and `ImportBundleJob` both gained a new `$opts` constructor property. Long-running queue workers cache class definitions at boot — they MUST be restarted (`php artisan queue:restart` or container restart) after this iteration is deployed, or jobs dispatched by the new web container will be deserialized against a stale class definition on the worker side and the `with_design` / `merge_design` flags will be silently dropped. (This surfaced mid-session: the user's first end-to-end test produced a bundle with `payload.media` but no `payload.design` because the worker was holding pre-`$opts` class state.)
