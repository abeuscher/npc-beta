## The anchor — mailing-list simple filters save `value = NULL` *(diagnosed + reproduced at 362)*

**Symptoms (owner-reported, walking the deployed tour):** an "opt-in equals Yes" filter shows 0 matches despite known opted-in contacts, and the filter's value select resets to "Select an option" after save.

**Root cause (one bug, both symptoms):** the Simple Filters repeater row in `app/Filament/Resources/MailingListResource.php` declares **three components all named `value`** — a text input, the Yes/No select, and a tag picker — exactly one visible per field type. They share one state path; at dehydration the inapplicable hidden ones clobber the chosen value, so the DB row lands `value: NULL`. Downstream, `MailingListQueryBuilder::applyFilter` compiles `equals NULL` into `WHERE field IS NULL`, which can never match a NOT-NULL column → 0 matches. The refilled form faithfully shows the NULL.

**Blast radius:** likely **all value-carrying simple filters** (text fields and tags too, same mechanism), i.e. the feature has probably been inert since session 041 — git history shows the schema essentially unchanged since. No regression hunt needed.

**Repro recipe (from 362, reusable as the verification skeleton):** seed opted-in contacts on the e2e stack; drive the real create form (Name → Add to filters → field "Mailing List Opt-In" → operator "equals" → value "Yes" → Create); read `mailing_list_filters` (observed `value: null`) and `MailingListQueryBuilder::build($list)->count()` (observed 0); reload and read the select (observed empty).

**Fix shape (confirm at gate):** let only the applicable component dehydrate the shared path (`->dehydrated(fn (Get $get) => …valueTypeFor($get('field')) === '<type>')` per component), or split the state paths (`value_text` / `value_select` / `value_tag`) and map onto the single `value` column in the repeater's mutate hooks. Either way, pin it with a **round-trip test**: form save → DB value → query match count → form refill. Verify text and tag filter types too, not just the opt-in select.

**Owner context worth keeping:** an "all opted-in contacts" list is a real, non-redundant use case — there is no default mailing list, so creating one with this filter is currently the only way to get that segment (even though `buildSimple` already applies `mailing_list_opt_in = true` as a baseline, the *list entity* must exist to be synced/used).
