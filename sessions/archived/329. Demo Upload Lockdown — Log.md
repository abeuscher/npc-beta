# Session 329 — Demo Upload Lockdown — Log

**Date:** 2026-05-30
**Branch:** session-329/1 … /2

## What was built

A who-can-upload gate that blocks **new file uploads for the `demo` role** at every server endpoint that introduces a new file, while leaving existing-media reuse and all other demo CRUD intact. The shared, anonymous, daily-wiped demo account can no longer introduce arbitrary files onto the public demo domain.

### Phase 1 — Endpoint gates (the security boundary)

- **New middleware `App\Http\Middleware\BlockDemoUploads`** — returns **403** when `$request->user()?->hasRole('demo')`. One definition, applied at all three new-file chokepoints. Scoped to the `demo` role, not demo mode, so an admin / super_admin maintaining the node still uploads (super_admin never holds the `demo` role).
- **Chokepoint audit found three new-file paths, not the two the prompt named:**
  1. **Livewire temporary-file-upload endpoint** — gated via `config/livewire.php` `temporary_file_upload.middleware` (`['throttle:60,1', BlockDemoUploads::class]` — the existing throttle default is preserved explicitly, since setting the key overrides Livewire's `?: 'throttle:60,1'` fallback). Every Filament `FileUpload` field funnels through this one endpoint, so this single gate covers all admin-form uploads (events, products, pages/posts, etc.) at once. `FileUploadController implements HasMiddleware` and reads `FileUploadConfiguration::middleware()` from this config key, so the gate attaches to the route — verified by a test asserting `BlockDemoUploads` is in the route's gathered middleware.
  2. **Page-builder upload routes** (`routes/admin-api.php`) — `->middleware(BlockDemoUploads::class)` on `uploadImage` + `uploadAppearanceImage`. The `useExistingImage` / `useExistingAppearanceImage` reuse routes are left open (no new file).
  3. **Rich-text inline-image upload** (`AdminPanelProvider.php`, route `inline-image-upload`) — the third chokepoint the "two chokepoints" framing missed. It previously had **no permission gate at all**, so adding `BlockDemoUploads` to its middleware array also closes a small pre-existing hole.

### Phase 2 — UX + tests

- **`FileUpload::configureUsing` in `AppServiceProvider`** — when the current user holds the `demo` role, every Filament `FileUpload` field renders `->disabled()->helperText('File uploads are disabled in the demo.')`. Mirrors the existing `DatePicker`/`DateTimePicker::configureUsing` precedent in the same method. This is the UX layer matching the server gate; the Vue page-builder + inline-image surfaces are not Filament fields, so they fall back to the endpoint 403.
- **`DemoRoleLockdownTest` extended** with five cases: demo blocked at the page-builder image + appearance-image endpoints; demo blocked at the inline-image endpoint; the Livewire upload route carries the gate (proves the whole admin-form surface is covered); demo can still set an image via the existing-media (`useExistingImage`) reuse path; a non-demo user upload still succeeds (proves the gate is role-scoped, not demo-mode-scoped).

## Security considerations

This session adds an authorization boundary, so the security axis is the point of the work.

- **Chokepoints gated:** all three new-file upload paths — the Livewire temp-upload endpoint (covers every Filament `FileUpload` field), the two page-builder upload routes, and the rich-text inline-image route. The inline-image route had no prior permission gate; this session closes that gap.
- **What stays allowed:** existing-media reuse (`useExistingImage` / `useExistingAppearanceImage` and any media-library picker) — these introduce no new file, so the demo can still set images from the seeded library. All other demo CRUD (events, donations, contacts, etc.) is untouched.
- **Role-vs-permission basis:** a plain `hasRole('demo')` check, not a new `upload_media` permission. Rationale: `super_admin` holds zero explicit permissions (it passes via the `Gate::before` bypass), so a permission would have to be granted across all eight non-demo roles and special-cased for super_admin — cost with no benefit now. The role check matches `DemoRoleLockdownTest`'s role-based assertions and super_admin never holds `demo`, so it passes naturally.
- **Server-side is the boundary; the field-disable is only UX.** A demo user is unable to upload even if the form field were re-enabled client-side — the endpoint middleware is the protection.
- **Layered posture:** the disguised-executable vector was already closed before this session (uploads are MIME-gated via `acceptedFileTypes`, re-encoded via `registerMediaConversions`, SVGs run through `SvgSanitizer`). This session adds the missing *who-can-upload* layer on top. The full demo-node posture is now: pipeline validation/re-encode + this upload gate + the 328 page-edit lock + daily `demo:reset` wipe + the FM-side egress firewall.
  - **[Correction — session 345, Flag 344-A]** The "SVGs run through `SvgSanitizer`" claim above was **false at the time**: `SvgSanitizer` existed but had **zero callers**, so uploaded SVGs were stored raw (and `registerMediaConversions` deliberately skips SVG). The 344 Cycle-3 audit surfaced the wiring gap; session 345 wired `SvgSanitizer` into the media-add pipeline (`MediaSvgSanitizer` on `MediaHasBeenAddedEvent`). The claim holds from session 345 onward — not 329.
