# 205. Code Review & Cleanup (Audit)

Post-importer, post-typography, post-notes-interactions codebase sweep. Scope: artifacts from sessions 180–204 (~25 sessions since the last broad review in 178/179). Follow the procedures established in sessions 101, 116, 141, and 178/179. This is the **audit half** of a two-session pair — session 206 applies the refactors approved from this session's findings.

The 178/179 pair is the canonical precedent — the W6 / W7 table shapes and the apply-session iteration strategy should mirror that pair's conventions. This audit is broader in scope than 178 was: the CRM Importer subsystem grew under roughly 11 sessions of layered work and deserves a dedicated deep pass; the Typography, Notes interactions, and Playwright e2e harness are all net-new surfaces since 179.

## Design decisions (resolved before starting)

- **Audit with small-fix in place.** Clearly safe fixes — dead imports, stale references, confirmed dead code, doc drift, obvious trivial duplications or framework-alignment wins — are applied during the audit branch. Anything that needs strategic input (architectural calls, cross-cutting refactors, retire-or-repurpose decisions) is captured on the W6 (duplicated logic) or W7 (framework alignment) tables and left for session 206.
- **Flag large issues explicitly.** If the audit surfaces something bigger than a W6/W7 entry — a subsystem-level divergence that needs real strategy — capture it as an "Open Flag" with enough context (risk, scope, proposed paths) and surface it to the user before continuing the sweep. Do not decide architecture during the audit.
- **Deep importer pass.** The importer is the dominant growth area since 179 (sessions 188–194, 201–203). Treat Workstream 2 as a dedicated structural review, not just a duplication/alignment grep. Real optimization opportunities are expected here.
- **Playwright sweep is full.** Dead specs, auth-setup drift, assertion convention drift, flaky patterns, hard-coded selectors that would break trivially. Covered in Workstream 10.
- **Migration squash is not in this session.** The squash follows the apply session as its own session (207) using the procedure from 062 / 082 / 108 / 142 / 181. Different risk profile (schema vs code), distinct checklist (verify `migrate:fresh --seed` identity, archive old migrations), and every historical squash has been its own session.
- **Bug fixes are out of scope.** Catching a real bug during the audit triggers an open flag, not a fix — unless the fix is one line and obviously correct.
- **Long runtime expected.** Scope is wide. Split into multiple iteration branches (`session-205/1`, `session-205/2`, …) if the audit cannot complete in one sitting. Each iteration ships a self-contained slice plus the safe fixes found in it.

## Pre-loaded findings (validate during the audit)

These observations surfaced during session 204 and should be validated, refined, and placed on the W6 / W7 tables — not discovered cold.

1. **LayoutInspectorPanel architectural divergence (W6 / open flag candidate).** `resources/js/page-builder-vue/components/LayoutInspectorPanel.vue` is a bespoke one-pane scrollable inspector with its own Background, Padding, and Margin controls, while widget inspectors use the 6-panel shell (`InspectorPanel.vue` + `BackgroundPanel.vue` / `TextPanel.vue` / `SectionLayoutPanel.vue`). Layout visual config lives on `layout_config`; widget visual config on `appearance_config`. Consequences: Background gradient, background image, and the unified spacing grid aren't reachable on a column layout, and the two inspector trees drift independently. Two possible paths: (a) widen the three universal appearance panels to target layouts as well as widgets (single extraction, dual contract), or (b) give layouts a parallel `appearance_config` column so the existing panels just work. Record on W6 and flag as an Open Flag for strategic input — this is one of the bigger architectural calls in scope for session 206.

2. **`preserveKeys` audit across `JsonResource` subclasses.** Session 204 discovered Laravel's `JsonResource::removeMissingValues()` silently re-indexes any array whose keys are all numeric, which broke the `events_listing` columns dropdown (numeric-keyed options like `['1' => '1', '2' => '2', …]` came out as `['1', '2', …]`). The fix (`public $preserveKeys = true`) landed on `WidgetResource`. Grep every `extends JsonResource` in `app/Http/Resources/` and identify any other resource that carries nested options arrays or enum-like maps with numeric or numeric-string keys. Fix in place if confirmed safe (add `preserveKeys = true` on the resource class); otherwise record on W6.

3. **`InteractsWithSectionedSettings` trait adoption audit.** Session 204 added per-section Save buttons to GeneralSettingsPage, CmsSettingsPage, FinanceSettingsPage, and MailSettingsPage via a new trait (`app/Filament/Pages/Concerns/InteractsWithSectionedSettings.php`). Audit any other multi-Section Filament pages (`app/Filament/Pages/DesignSystemPage.php`, any Settings-style page not yet touched, and any content-template-editor page with multi-section forms) for whether they should adopt the same pattern. Record on W7 if adoption gaps found.

## Workstream 2 — CRM Importer subsystem (deep pass)

The importer has roughly 11 sessions of layered work; this workstream is where the audit spends most of its time. Don't just grep for duplicates — walk the lifecycle end-to-end and look for real optimization opportunities.

**Structural consistency across content types:**

- Every `Import{Type}Page.php` follows the same shape: field-map selection → dry run → staged session → review → commit. Flag any type that diverges from the shared shape.
- Every `{Type}FieldRegistry.php` follows one documented pattern — either `$fillable`-derived with `$excluded` list + appended `external_id`, or fully hand-curated. Flag any registry that mixes idioms.
- Aggregator registries (`ContactImportFieldRegistry`, `EventImportFieldRegistry`, `DonationImportFieldRegistry`, `MembershipImportFieldRegistry`, `InvoiceImportFieldRegistry`, `NoteImportFieldRegistry`) namespace correctly (prefix with `{type}:*`) and have no collisions across buckets.
- `CsvTemplateService` header lists, `FakeCsvComposer::compose*` output, and `{Type}FieldRegistry::options()` agree on canonical columns. Session 202 locked this contract; session 204 clipped the `Event Import Session Id` drift — verify no other drift exists.

**Staged-update lifecycle consistency:**

- Where does each type create `ImportStagedUpdate` rows? Are the fields consistent (`subject_type`, `subject_id`, `attributes`, `tag_ids`, `import_session_id`)?
- Approve path: every type applies `$update->attributes` via `fill()->save()` and appends an activity note tracing the source. Flag any type that skips the activity note or the tag sync.
- Rollback path: every type deletes staged updates and creates a discard-note on rollback. Flag any divergence.
- Approve/rollback logic is currently threaded through `ImporterPage::approve` / `rollback` — is the per-type logic duplicated there, or has it been consolidated into a shared path?

**Match key resolution:**

- Is there one `ContactMatchKeyService` (or equivalent) entry point, or do multiple pages implement their own match-key lookup?
- Case sensitivity, whitespace normalisation, email canonicalisation — consistent?
- Dry-run vs commit behaviour — does one path reuse the other, or are they parallel implementations?

**Progress tracking & commit orchestration (session 193):**

- Do all six per-type progress surfaces use the same mechanism — the shared trait `InteractsWithImportProgress` plus one service and one blade partial — or did each grow its own progress path?
- Chunked-commit behaviour for large imports — consistent chunk size, consistent back-pressure, consistent session-state transitions?

**Update-existing strategy (session 194):**

- Each type chose between "skip", "update", or "merge" on conflict. Verify the strategy selection UI, the commit code path, and the staged-update UI all agree on the enum values and on the branching behaviour.

**Auto-contact creation paths:**

- Events, donations, memberships, and invoices importers all auto-create contacts when a `__org_contact__` sentinel or similar is mapped. Confirm one shared creation path; flag any type with its own copy.
- `import_session_id` set on auto-created contacts so rollback cascades — confirm across all auto-create paths.

**Dead code & orphaned importer files:**

- Any `Import*Page.php` or `app/Importers/*FieldRegistry.php` that is no longer registered in Filament navigation or referenced anywhere.
- Commented-out importer code left over from the mid-arc iterations.

**Tests:**

- Pest coverage per type is consistent in shape (per-type page test + registry test + update-strategy test + staged-update test). Flag test-coverage gaps.

Output: structured finding block with sub-tables keyed by the areas above. Small things (a missing exclusion, a rename-only stale ref) are fixed in place. Structural findings (e.g. "all five per-type progress pages should share a single trait" or "staged-update approval logic should extract to an `ApprovalPipeline` service") go on W6 or the Open Flags block for session 206 and the user to strategise on.

## Workstream 8 — Framework alignment (W7 table)

Same diagnostic question as 116, 141, 178:

> "Is this code doing something the framework already does? If so, are we working with the framework or around it?"

Focus areas:

- **Laravel / Eloquent.** Manual queries or array manipulations that should use scopes, accessors, `JsonResource`, `FormRequest`. Importer code especially — 11 sessions of rapid iteration leaves hand-rolled patterns the framework could absorb.
- **Vue / Pinia.** Ad-hoc reactivity that should use `computed` / `watch` / `storeToRefs`. `@vueuse/core` was adopted in session 179 — check if any new hand-rolled debounce/abort/focus code since then should also migrate.
- **Filament.** Admin pages that duplicate form component / action / table behaviour. Importer pages especially — they are the newest crop and may have reinvented Filament primitives.
- **Blade components.** Repeated HTML patterns across importer review blades that should be anonymous Blade components.
- **Pre-loaded finding #3** — `InteractsWithSectionedSettings` adoption gaps.
- **Pre-loaded finding #2** — `preserveKeys` audit on JsonResource subclasses. If any other resource is affected, fixing it in place is a small fix (single property addition); the audit is the framework-alignment work.

Record findings in the W7 table: File, Line(s), What the code does, What the framework provides, Risk if left as-is, Suggested approach.

Do not fix anything other than trivial single-line cases (e.g. adding `preserveKeys = true` on a resource class once confirmed). Present the table for session 206.
