## Workstream 1 — Quill Editor Overflow Fix

The Quill rich-text editor fields are overflowing their containing panels. This is a CSS/layout issue, not a data issue. Quill is a custom component (not the default Filament Trix editor).

### Approach

1. Read `app/Forms/Components/QuillEditor.php` and its associated view file(s) to understand how the component is rendered.
2. Inspect the rendered HTML structure to understand the container hierarchy.
3. If the issue is not immediately obvious from the code, pause and ask the user to use the browser's element inspector to identify the overflowing element and the constraint that is missing. Report your findings and proposed fix before making changes.
4. Apply the minimum CSS/markup change needed to keep the Quill editor within its panel bounds at all viewport widths.

## Workstream 2 — Page & Event Edit Layout: Settings Column

Restructure both the Page and Event edit forms to match the two-column layout used on the Contact edit form.

### Reference

The Contact edit form (`app/Filament/Resources/ContactResource.php`) uses:
- An outer `Group` with `->columns(3)`
- A left content group at `->columnSpan(2)`
- A right `Section::make('Settings')` at `->columnSpan(1)`

Apply the same pattern to `app/Filament/Resources/PageResource.php`.

### Requirements

1. The left column (`->columnSpan(2)`) contains, in order:
   - The existing title/slug/public-URL section (unchanged)
   - The Page Builder section (unchanged)
   - The SEO section (unchanged)
   - The Custom Fields section (unchanged)

2. The right column is a new `Section::make('Settings')` at `->columnSpan(1)` containing:
   - The **Published** toggle (currently in the `Publication` section)
   - The **Publish Date** date-time picker (currently in the `Publication` section)
   - The **Tags** field (currently in the `Publication` section)

3. Remove the now-empty `Publication` section.

4. The outer wrapper must be `->columnSpanFull()` so the two-column group stretches across the full form width.

### Event edit form layout

Apply the same `columns(3)` / `columnSpan(2)` / `columnSpan(1)` pattern to `app/Filament/Resources/EventResource.php`. The current form uses a `Split` component with `->grow(false)` on the right column, which makes the right column too wide.

1. The left column (`->columnSpan(2)`) contains, in order:
   - The Description section (unchanged)
   - The Location Info section (unchanged)
   - The Online Meeting Info section (unchanged)
   - The Custom Fields section (unchanged)

2. The right column (`->columnSpan(1)`) contains two sections stacked:
   - `Section::make('Settings')` at the top, containing: **Status** select, the **Landing page** placeholder with its edit/view hint actions, and **Tags**.
   - `Section::make('Date')` below it, containing the `starts_at` and `ends_at` date-time pickers (added in Workstream 3 — implement the layout placeholder here; the fields themselves are wired up in Workstream 3).

3. Remove the old separate `Settings` section and `Dates` repeater — they are replaced by the right column sections above.

4. The outer wrapper must be `->columnSpanFull()`.

> **Note:** Workstream 2 (layout) and Workstream 3 (date simplification) both touch `EventResource.php`. Complete Workstream 3 after Workstream 2 so the form is already restructured when the date fields are wired up.

## Workstream 3 — Event Date Simplification

Events currently support multiple dates via a related `event_dates` table and `EventDate` model. This feature is being removed. Each event will have exactly one start time and one (optional) end time, stored directly on the `events` table. There is no backwards-compatibility requirement — the schema and data can be wiped as needed.

### Read these files in full before making any changes

- `app/Models/Event.php`
- `app/Models/EventDate.php`
- `app/Filament/Resources/EventResource.php`
- `app/Filament/Resources/EventResource/Pages/EditEvent.php`
- `app/Mail/EventReminder.php`
- `resources/views/widgets/event-dates.blade.php`
- `resources/views/widgets/event-description.blade.php`
- `resources/views/widgets/event-registration.blade.php`

### Migration

1. Add `starts_at` (timestamp, nullable) and `ends_at` (timestamp, nullable) to the `events` table.
2. Drop the `event_dates` table entirely.
3. Run `migrate` (not `migrate:fresh`) — there is no existing event date data worth preserving, but other tables must not be affected.

### Model changes

In `app/Models/Event.php`:

1. Add `starts_at` and `ends_at` to `$fillable`.
2. Add `'starts_at' => 'datetime'` and `'ends_at' => 'datetime'` to `$casts`.
3. Remove the `eventDates()` `HasMany` relationship.
4. Remove the `nextDate()` method.
5. Update `scopeUpcoming()` to use `starts_at` directly: `$query->where('starts_at', '>=', now())`.
6. Update `scopeOpenForRegistration()` — it calls `upcoming()`, which will now work correctly once the scope above is updated.
7. The `generateDatesFromRule()` method is now dead code — remove it.

Delete `app/Models/EventDate.php`.

### Form changes (`app/Filament/Resources/EventResource.php`)

Workstream 2 restructured the form and placed a `Section::make('Date')` placeholder in the right column. In this workstream, wire up the two date-time pickers inside that section:

```
DateTimePicker::make('starts_at')  — label 'Start', required, seconds(false)
DateTimePicker::make('ends_at')    — label 'End', nullable, seconds(false), after('starts_at')
```

### EditEvent.php changes

**`sendRemindersNow` visibility:**
Replace the `eventDates()` upcoming check with a direct attribute check:
```php
$hasUpcoming = $event->starts_at && $event->starts_at >= now();
```

**`sendRemindersNow` action:**
The action currently loops over `$upcomingDates` then `$registrations`. Simplify: there is exactly one date (the event itself). Remove the outer `foreach ($upcomingDates ...)` loop. Pass `$event` (or construct the date context from it) to `EventReminder` — update the mailable accordingly (see below).

**`deleteRegistrantContacts` visibility:**
Replace the `eventDates()` existence and date checks with:
```php
if ($event->status === 'cancelled') { return true; }
return $event->starts_at && $event->starts_at < now();
```

### Mailable changes (`app/Mail/EventReminder.php`)

Read the mailable in full. It currently receives an `EventDate` as its second constructor argument. Change it to receive an `Event` directly and update any references to `$eventDate->starts_at`, `$eventDate->ends_at`, etc. to come from the `Event` model instead.

### Widget changes

Update `resources/views/widgets/event-dates.blade.php` (and any other event widget views that reference `$eventDate` or load event dates) to read `$event->starts_at` and `$event->ends_at` directly from the event record.

### Cleanup

- Delete the `EventDate` migration file only if it has been superseded by the drop migration above and there is no reference to it in test factories or seeders. If a factory references it, remove the factory as well.
- Search the entire codebase for references to `eventDates`, `EventDate`, and `event_dates` and resolve every one.

### Update schema.md

After running the migration:
- Remove the `event_dates` table section.
- Add `starts_at` and `ends_at` to the `events` table section.
