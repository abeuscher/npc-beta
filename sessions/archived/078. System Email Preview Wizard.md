# 078. System Email Preview Wizard

Add a multi-step confirmation and preview modal to admin-initiated system email sends, so that the admin can review a merged message before it goes out.

---

## Design decisions (resolved before starting)

**Which sends get the wizard:**
The wizard is retrofitted to all admin-UI-initiated bulk or consequential sends. For this session that means: donor tax receipts (both "Send to Pending" and "Force Re-send All" actions on the Donors page), admin user invitations, and event cancellation. Event reminders sent by the console command are out of scope — the wizard applies only to UI-initiated actions.

**Wizard structure — three steps in a multi-step modal:**
- **Step 1 — Confirm:** "You are about to send [N] system email(s) of type [Name]. Recipients: [list or count]. Do you want to proceed?"
- **Step 2 — Preview:** The email template rendered with real merged data for the first recipient in the set. The admin can see exactly what will be sent. For single-recipient sends, this is the one message. For bulk sends, this is the first recipient — paging through the full series is a future nice-to-have and is not built in this session.
- **Step 3 — Send:** A single "Send Now" button. No scheduled send in this session (see Out of Scope).

**Preview rendering:**
The preview uses the existing `EmailTemplate::forHandle()` → `render()` → `resolveWrapper()` chain. The output is rendered into the modal as an `<iframe srcdoc="...">` so that the email's own HTML and inline styles are isolated from the Filament UI.

**Reusability:**
The wizard is implemented as a reusable Filament action (or trait) so that adding it to future send points requires minimal code. Read the Filament action and modal APIs carefully before designing the interface.

---

## Phase 1 — Wizard infrastructure

1. Design and build the reusable wizard. A Filament `Action` subclass or a composable trait — choose the approach that fits cleanest with how Filament multi-step modals work. Read the Filament documentation pattern in the codebase before deciding.
2. The wizard accepts: a display name for the email type, a recipient count, a callable that returns the preview HTML for the first recipient, and a callable that executes the send.
3. Step 1 shows the type name, recipient count, and a clear confirmation prompt.
4. Step 2 renders the preview HTML in an `<iframe srcdoc>` block sized appropriately within the modal.
5. Step 3 executes the send callable and closes the modal on completion, showing a success notification.

---

## Phase 2 — Retrofit existing send points

1. Replace the existing confirmation modals on the Donors page ("Send to Pending", "Force Re-send All") with the new wizard.
2. Replace the confirmation on the admin user invitation send action with the wizard.
3. Replace the confirmation on the event cancellation bulk send with the wizard.
4. For each: supply the correct type label, recipient count, first-recipient preview, and send callable.
5. Update help docs for any affected pages.

---

## Security checklist

Before closing:

- [ ] Preview rendering uses the same server-side template chain as actual sending — no separate render path that could diverge.
- [ ] The send callable is executed server-side only — the modal does not accept or transmit the rendered HTML back to the server.
- [ ] Wizard actions are admin-only — no public route.

---

## Out of scope

- Scheduled send / send-at picker (deferred — noted in roadmap for a future Infrastructure session).
- Paging through all recipients in the preview (deferred — nice-to-have, not built here).
- Console-command-initiated sends (event reminders).
- PDF or attachment preview.
