# 095. Test Coverage — Portal, Stripe & Integrations

Add test coverage for the remaining major untested areas of the application: member portal authentication, donation/Stripe checkout, products and purchases, tax receipts, mailing list filter resolution, the import review workflow, and the activity log.

---

## Design decisions (resolved before starting)

- Stripe tests mock the Stripe SDK — no live API calls. Use Laravel's `Http::fake()` or mock the Cashier/Stripe client as appropriate.
- Portal auth tests use the `portal` guard and `portal_accounts` table, not the admin `web` guard.
- Mailing list filter resolution tests must verify that `raw_where` clauses are safely handled (parameterised or sanitised).
- Activity log tests verify that observers write correct entries — they do not test the UI viewer.

---

## Phase 1 — Portal authentication

Test the portal auth flow end-to-end:

- Login with valid credentials via the portal guard.
- Login rejected for inactive accounts (`is_active = false`).
- Login rejected for invalid credentials.
- Signup creates a `portal_accounts` record and links it to the correct contact.
- Password reset flow issues and consumes a token correctly.
- Portal routes are inaccessible without authentication.
- Portal queries are scoped to the authenticated user's `contact_id` — verify no cross-contact data leakage.

---

## Phase 2 — Donations & Stripe checkout

Test the donation checkout and webhook flow:

- `DonationCheckoutController` creates a pending donation record on checkout initiation.
- One-off vs recurring donation types are stored correctly.
- Stripe webhook handler updates donation status on payment completion.
- Stripe webhook handler creates a transaction record.
- Webhook rejects invalid signatures.
- Donation amount validation (min $1, max $10,000).

---

## Phase 3 — Products & purchases

Test the product checkout flow:

- `ProductCheckoutController` initiates a Stripe Checkout session.
- Capacity enforcement — checkout blocked when product is sold out.
- Purchase record created only on `checkout.session.completed` webhook (never pre-created).
- `ProductPriceObserver` behaviour — Stripe price creation on save (mocked).
- Waitlist entry creation and status transitions.

---

## Phase 4 — Tax receipts

Test the tax receipt generation flow:

- Receipt generated with correct total and breakdown for a given contact and tax year.
- Duplicate prevention — re-running for the same contact/year creates a new row (re-send), not a duplicate send.
- Receipt includes only completed/active donations.

---

## Phase 5 — Mailing list filter resolution

Test the filter-to-query pipeline:

- Single filter with each operator (equals, contains, is_true, etc.) returns correct contacts.
- Conjunction logic — `and` vs `or` combinations.
- `raw_where` clauses are parameterised / safe from injection.
- Empty filter set returns all contacts.

---

## Phase 6 — Import review workflow

Test the staged import approve/rollback flow:

- Approving an import session applies `import_staged_updates` to contacts.
- Rolling back an import session reverts staged changes.
- Tag application on approval.
- Status transitions: pending → reviewing → approved / rolled_back.

---

## Phase 7 — Activity log

Test that observers write correct activity log entries:

- Creating a contact writes a `created` event.
- Updating a contact writes an `updated` event.
- Deleting a contact writes a `deleted` event.
- Actor type and actor ID are set correctly for admin and portal actions.
- System-initiated actions record `actor_type = 'system'` with null `actor_id`.
