## What was built

### Phase 1 — Schema: five nullable transactional FKs

Combined migration `2026_05_01_120000_add_organization_id_to_transactional_tables.php` adds `organization_id` (or `sponsor_organization_id` on events) to five tables, all `ON DELETE SET NULL`, all indexed:

- `donations.organization_id`
- `memberships.organization_id`
- `event_registrations.organization_id`
- `events.sponsor_organization_id`
- `transactions.organization_id`

The fifth FK on `transactions` was the load-bearing decision surfaced at session start: invoices are grouped Transactions (no top-level `Invoice` model), so `__org_invoice_party__` writes to `transactions.organization_id` to match the existing `transactions.contact_id` shape. Schema docs updated for all five tables.

### Phase 2 — Eloquent relationships

- Each transactional model gains `organization()` belongsTo.
- `Event` gets `sponsorOrganization()` belongsTo (named to disambiguate from a future host org).
- `Organization::donations()` (which was broken — referenced a column that didn't exist) becomes functional. Added `memberships()`, `eventRegistrations()`, `eventsSponsored()`, `transactions()` HasMany helpers.

### Phase 3 — Importer wiring (four sentinels)

Added `__org_donor__` / `__org_member__` / `__org_sponsor__` / `__org_invoice_party__` to the four namespaced importers' `*ImportFieldRegistry::otherOptions()`. Each renders the same `auto_create` / `match_only` / `as_custom` strategy radio that `__org_contact__` already used.

Trait extensions in [`InteractsWithImportWizard`](../app/Filament/Pages/Concerns/InteractsWithImportWizard.php):
- New `ORG_SENTINEL_TYPES` const map (sentinel → relational-map type).
- `buildNamespacedMappingRow` org-strategy visibility predicate generalized to all five Org sentinels.
- `serializeColumnMaps` `__org_contact__` branch generalized via the const map; covers all five sentinels with appropriate `target` routing on `as_custom`.

Shared lookup helper `resolveOrganizationByName` lifted to [`InteractsWithImportProgress`](../app/Filament/Pages/Concerns/InteractsWithImportProgress.php). The existing `applyContactOrganization` now delegates to it. Each per-importer progress page captures the relevant sentinel value and resolves the org before creating the transactional record.

`EventFieldRegistry::$excluded` adds `sponsor_organization_id` so the column doesn't surface in the destination dropdown as a raw event field — it's reachable only via the `__org_sponsor__` sentinel.

### Phase 4 — Admin UI on Organization

**Form rebuilt** (after mid-session direction): single `Section`, no Address subsection header. Layout:
- Row 1: `name` (6) + `type` (6).
- Row 2: `website` (4) + `phone` (4) + `email` (4).
- Address fields flow inline below: `address_line_1` (full) + `address_line_2` (full) + `city` (3) + `state` (3) + `postal_code` (3) + `country` (3).

**Type values** changed from `foundation/corporate/government/other` to `nonprofit/for_profit/government/other`. Migration `2026_05_01_140000_add_email_and_normalize_types_on_organizations.php` adds the `email` column AND backfills any legacy type values (`foundation → nonprofit`, `corporate → for_profit`). Table column badge + filter updated to match.

**Bottom-of-record panels reduced.** Original spec called for four read-only RelationManagers (Members / Donations Received / Memberships Held / Events Sponsored); session shipped five (added Invoices Billed for the Transactions FK). After mid-session direction — *financial transactions are referenced via filtered links to Finance, not displayed inside records* — four of the five panels were removed. **Final shape: Events Sponsored panel only.** No Notes panel either (replaced by the Timeline page below).

**Sponsor field on Event admin form.** New row in the `extraTitleFields` of `EventResource::form()` carries a Select for `sponsor_organization_id` with `relationship('sponsorOrganization', 'name')`, searchable, preloaded, with inline `createOptionForm` so users can pick existing or create on the fly.

**Ellipsis menu (first on Org admin).** `EditOrganization` gains a header `ActionGroup` containing "View affiliated contacts →", deep-linking to the Contacts list filtered by `?tableFilters[organization_id][value]=…`. Mirrors the `view_transactions` pattern from EditContact / EditProduct.

**ContactResource** gains a `SelectFilter::make('organization_id')` (label "Organization", searchable, preloaded) so the URL filter actually filters the list.

### Phase 5 — Block-with-counts deletion guard

`OrganizationResource::guardDeletion()` refuses delete if any of the five buckets (affiliated contacts / donations / memberships / sponsored events / invoices) are non-empty. Surfaces a notification with per-bucket counts. ForceDelete branch lets the operator override; DB cascades `SET NULL` automatically. Wired into table DeleteAction, table DeleteBulkAction, and the EditOrganization header DeleteAction.

### Phase 6 — Notes lift to a shared Timeline pattern

Surfaced mid-session: `OrganizationResource` was carrying a free-text `Forms\Components\Textarea::make('notes')` writing to a separate `organizations.notes` column AND a polymorphic `NotesRelationManager` panel — two parallel notes systems. Contact's pattern (a dedicated `/contacts/{record}/notes` page with a custom timeline merging Note records + `ActivityLog` entries, modal create/edit/delete actions, source/type filters) is the canonical shape.

Lift-and-reuse:
- New abstract base [`app/Filament/Concerns/RecordTimelinePage.php`](../app/Filament/Concerns/RecordTimelinePage.php) — owns `getTimeline()`, filter state, header actions, `editNoteAction()` / `deleteNoteAction()`, breadcrumbs. Three abstract hooks: `notableType()` / `recordResourceClass()` / `getRecordTitle()`.
- Blade view renamed `contact-notes.blade.php` → `record-timeline.blade.php` (already model-agnostic).
- `ContactNotes` reduced from 308 lines to a 34-line shell extending the base. All public properties + method signatures preserved; the 15 existing `NotesStructuredFieldsTest` cases pass unchanged.
- New `OrganizationNotes` (34 lines) extending the same base.
- Migration `2026_05_01_160000_drop_notes_column_from_organizations.php` drops `organizations.notes` outright (no live data per user direction).
- `notes` removed from `Organization::$fillable`. Notes Section removed from form.
- `NotesRelationManager` removed from `OrganizationResource::getRelations()`. New `'notes' => Pages\OrganizationNotes::route('/{record}/notes')` route added to `getPages()`. EditOrganization header gains "Timeline" button mirroring EditContact.
- Back-button label parameterized: `← Back to ` + `Str::lower($resource::getModelLabel())` → "Back to contact" / "Back to organization".
