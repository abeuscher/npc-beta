# 089. Image & Carousel Widgets

Two new page builder widget types: a static picture widget for single images and a carousel widget powered by Swiper.js for multi-image slides backed by collections. Both exercise the image optimization pipeline built in session 088.

---

## Design decisions (resolved before starting)

- **Swiper.js** is the carousel library. MIT licensed, tree-shakable. Install the npm package; import only the modules we need (core, Navigation, Pagination, Autoplay, EffectFade).
- The **carousel widget** pulls slides from a collection via the existing `WidgetDataResolver` and collection query system. Each collection item provides an image (background) and text fields (caption). Caption content is composed using curly-bracket token syntax (`{{field_key}}`), matching the pattern used in system email templates.
- The **picture widget** is standalone — a single image uploaded via the inspector's `image` config field (built in session 088), rendered with the `<x-picture>` component.
- No video support in v1. No vertical carousel. No per-slide interval. No WYSIWYG inline image insert (deferred to a future session).
- The Docker image must be rebuilt before testing — session 088 updated the Dockerfile to add JPEG/WebP/FreeType support to GD. Confirm GD capabilities at the start of the session if the container has not been rebuilt.

## Phase 2 — Static picture widget

Create a new widget type `image` (label: "Image") in the `WidgetTypeSeeder`.

Config schema:

| Key | Type | Label | Default |
|-----|------|-------|---------|
| `image` | image | Image | *(null)* |
| `alt_text` | text | Alt text | |
| `object_fit` | select | Image fit | `cover` |
| `link_url` | url | Link URL | |

Options for `object_fit`: `cover`, `contain`.

The widget template renders the `<x-picture>` component from session 088 using the `$configMedia['image']` media object. If `link_url` is set, wrap in an `<a>` tag. Apply `object-fit` via inline style.

Handle the case where no image is uploaded — render nothing (not a broken image placeholder).

## Phase 3 — Carousel widget type

Create a new widget type `carousel` (label: "Carousel") in the `WidgetTypeSeeder`.

### Config schema

**Primary fields** (always visible in inspector):

| Key | Type | Label | Default |
|-----|------|-------|---------|
| `caption_template` | text | Caption template | `{{title}}` |
| `object_fit` | select | Image fit | `cover` |

The carousel widget declares a collection in its `collections` array so the existing query settings panel is available in the inspector. The user picks which collection to pull slides from and can filter by tags, limit, and sort order — all via existing infrastructure.

**Advanced fields** (behind the collapsible Advanced toggle):

| Key | Type | Label | Default |
|-----|------|-------|---------|
| `autoplay` | toggle | Autoplay | true |
| `interval` | number | Interval (ms) | 5000 |
| `loop` | toggle | Loop | true |
| `pagination` | toggle | Pagination dots | true |
| `navigation` | toggle | Navigation arrows | true |
| `slides_per_view` | number | Slides per view | 1 |
| `effect` | select | Transition effect | `slide` |
| `speed` | number | Speed (ms) | 300 |
| `link_color` | text | Link color | |
| `text_color` | text | Text color | |

Options for `effect`: `slide`, `fade`. Note: `fade` only works correctly with `slides_per_view: 1` — the template should enforce this.

### Inspector grouping

The inspector already has a collapsible "Advanced (Spacing)" panel for `style_config`. The carousel's advanced config fields should appear in a similar collapsible section within the Config panel, labelled "Carousel Settings". The primary fields (`caption_template`, `object_fit`) and the collection query settings remain always visible.

This requires a small extension to the inspector: config schema fields need a way to declare themselves as "advanced". Add an optional `advanced: true` flag to the config schema field definition. The inspector renders non-advanced fields first, then advanced fields inside a collapsible toggle — following the same expand/collapse pattern as the spacing panel.

Update the inspector blade template to support this grouping. This is a generic enhancement that benefits all widget types, not carousel-specific.

## Phase 4 — Carousel template & rendering

The carousel widget template:

1. Receives `$config` (config values) and collection data via the existing `WidgetDataResolver` pipeline.
2. For each collection item, renders a Swiper slide:
   - The item's image field becomes the slide background (using `<x-picture>` or a background-image style, whichever works better within Swiper's slide markup).
   - The `caption_template` is processed with token replacement: `{{field_key}}` is replaced with the item's corresponding field value.
   - Caption text is rendered as an overlay, styled with the configured `text_color`.
3. Initializes Swiper via Alpine.js `x-init` with the config values passed as data attributes on the container element.

The template CSS should handle:
- Full-width slides by default.
- Caption overlay positioning (bottom of slide, semi-transparent background).
- Responsive behaviour — slides should work at all viewport widths without horizontal overflow.

Read existing widget templates and the Swiper documentation before implementing. Follow the same Blade rendering pattern used by other server-rendered widgets.

## Security checklist

Before closing:

- [ ] Caption template token replacement escapes output values via Blade `{{ }}` — no raw HTML injection from collection item data.
- [ ] Image URLs in the carousel come from the media library (no user-supplied paths in rendered HTML).
- [ ] Swiper initialization uses data attributes or Alpine state — no inline `<script>` with unescaped config values.
- [ ] The `link_url` field in the picture widget is rendered via `{{ }}` escaping.
