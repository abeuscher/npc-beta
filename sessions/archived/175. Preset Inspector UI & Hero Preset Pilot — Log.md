### Scope refinement mid-session — presets are appearance-only

The session prompt originally specified replace-all semantics for both `config` and `appearance_config`. During testing, a scope refinement was agreed: **presets must not touch content**. The underlying observation is that the backend doesn't cleanly separate content from appearance — `page_widgets.config` holds both groups indiscriminately, and the split lives only in the schema's `group` attribute (which the inspector's Content/Appearance tabs already honor). Option B was chosen: keep `config` as the storage bag, but require every preset's `config` keys to live under an appearance-group schema field, and change apply semantics to overlay (preserving content-group keys) instead of replace.

This meant three changes:

- Hero presets were rewritten to drop all `content` keys (they had been set on every preset in the initial draft).
- `applyPreset` store action changed from replace to overlay for `config`; `appearance_config` still replaces wholesale since that bag is 100 % appearance.
- `InspectorPresetGallery.vue` synthetic "Blank" card now filters `resolved_defaults` to appearance-group keys only, so "reset" preserves content too.
- `tests/Feature/WidgetManifestTest.php` preset-shape test gained a second assertion per preset config key: the key must live under a schema field whose `group` is `appearance`, else the test fails naming the widget, preset handle, and offending key.

## Security checklist — verified

- [x] Presets carry no file paths, URLs, or content that could inject into preview HTML beyond what `config_schema`-declared widget fields already accept. The appearance-only rule further tightens this: preset config keys are constrained to appearance-group schema fields (no rich-text, no media, no buttons), and `appearance_config` is a closed-vocabulary jsonb bag the renderer applies as inline style only.
- [x] The bootstrap payload's preset data reaches only authenticated admin users via the existing page-builder route — no public exposure of manifest content beyond what already happens. `PageBuilderApiController::widgetTypes()` already gates on `view_page`, and the Livewire bootstrap lives behind the Filament admin auth pipeline.
- [x] Apply-preset action is CSRF-guarded via the existing Livewire/API pipeline, same as every other inspector edit. `applyPreset` in the store routes mutations through `flushDebouncedSave`, which calls the same `api.updateWidget` endpoint the rest of the inspector uses — no new unguarded endpoints.
