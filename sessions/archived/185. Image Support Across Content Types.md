# 185. Image Support Across Content Types

Surface image fields that already exist on content records through to the public side, and formalize a widget-facing pattern for pulling from the sample image pool so widget demo mode and the dashboard random-data generator share one mechanism. Downstream effect: better Playwright thumbnails for widgets that currently render empty frames.

---

## Goals

1. **Per-page context bridge for widgets.** Any widget can substitute tokens like `{{title}}` / `{{author}}` / `{{date}}` / `{{starts_at}}` in its text fields, and any widget's appearance panel can opt to use the current page/post/event's header image as its background. One override toggle and one token service replace every otherwise-needed "content header" widget duplicate.
2. **Pool access pattern for widgets.** A clean, documented API by which a widget declares "I want up to N images from category X" (X matches a [sample-images folder](resources/sample-images/) name). Shared by widget demo mode and the dashboard random-data generator.
3. **Better demo-mode thumbnails.** Widgets whose `/dev/widgets/{handle}` capture currently renders empty or uninformative frames use the pool pattern to populate baseline content.

Contact portraits are **out of scope**. Hero uses the shared `appearance_background_image` slot — no new Hero-specific config key.

## Phase 1 — Audit (done)

- **Page** (incl. `type='post'`): `post_thumbnail` exposed + read; `post_header` neither exposed nor read; `og_image` exposed + read (SEO only).
- **Event**: `event_thumbnail` exposed + read; `event_header` exposed but never read publicly; `event_og_image` exposed.
- **Product**: `product_image` exposed + read via `WidgetDataResolver` → `ProductCarousel` (needs live verification).
- **Architecture:** All page types render via widget composition ([pages/show.blade.php](resources/views/pages/show.blade.php)); no template-level title/header slot. [PageContext](app/Services/PageContext.php) already exposes `$currentPage` to every widget template.

## Phase 2 — Per-page context bridge

### 2a — Expose `post_header` in the admin form

Add a `SpatieMediaLibraryFileUpload` for `post_header` to [PageResource](app/Filament/Resources/PageResource.php) alongside the existing `post_thumbnail` / `og_image` fields, so users can actually upload a post header image.

### 2b — `PageContextTokens` service

New service `App\Services\PageContextTokens` exposing a method like `substitute(string $text, Page $currentPage): string` that replaces supported tokens from the current page/post/event context. Initial token set:

- `{{title}}` — `$currentPage->title`
- `{{date}}` — `$currentPage->published_at` formatted
- `{{excerpt}}` — `$currentPage->meta_description`
- `{{author}}` — author name via `$currentPage->author->name` (verify relation; fall back to empty string if author is null)
- `{{starts_at}}` — event start datetime (requires `Page::event()` reverse relation, one line)
- `{{location}}` — event location (same)

Event tokens resolve only when `currentPage` has an associated `Event` via a new `Page::event(): HasOne` accessor on `landing_page_id`.

**Integration:** widgets opt in by calling the service from their template. Start by migrating the existing inline substituters ([BlogListing](app/Widgets/BlogListing/template.blade.php) / [EventsListing](app/Widgets/EventsListing/template.blade.php) / [Carousel](app/Widgets/Carousel/template.blade.php) / [Nav](app/Widgets/Nav/template.blade.php)) as a cleanup pass, but only if the migration is trivial — otherwise defer to the next session. Hero (and any widget with a heading/richtext field) can now accept `{{title}}` in its config.

**Unknown tokens** pass through untouched (no silent replacement to empty). This makes missing-token bugs visible without forcing an escape convention.

### 2c — Background-image override toggle

Add a shared appearance-panel toggle (e.g. "Use current page's header image as background"). When on, [AppearanceStyleComposer::composeBackgroundImage](app/Services/AppearanceStyleComposer.php#L89) picks its URL from:

1. `$currentPage->getFirstMediaUrl('post_header')` or `getFirstMediaUrl('event_header')` (based on page type)
2. Fall back to `post_thumbnail` / `event_thumbnail` if no header
3. Fall back to the widget's own `appearance_background_image` if neither exists

Gradient layer still composes independently. Graceful throughout — never throws, never 500s a public page. If the admin wants a "can't resolve" signal, surface it in the inspector only (nice-to-have; skip if it adds scope).

### 2d — ProductCarousel end-to-end verification

With the generator populating real `product_image` media (Phase 3 below), place a `ProductCarousel` on a page and confirm images render from seeded products.

## Phase 3 — Widget pool-access pattern

Define the canonical API for widget-declared pool usage:

- A method on the widget definition base (e.g. `demoImages(): array`) returning a list of entries. Each entry names a pool category, a count, and an injection target — either a widget config key (`config.image`) or the shared appearance slot (`appearance.background_image`).
- [WidgetDemoController](app/Http/Controllers/Dev/WidgetDemoController.php) reads the declaration at render time and injects URLs from [SampleImageLibrary](app/Services/SampleImageLibrary.php) at the declared targets.
- Pool returns `min(requested, available)`; empty pool is a no-op; widget renders without images and without crashing.

**Dashboard random-data generator** ([DashboardDebugGeneratorWidget](app/Filament/Widgets/DashboardDebugGeneratorWidget.php)) consumes `SampleImageLibrary` directly to attach pool images to generated `Product.product_image`, `Event.event_thumbnail`, and `Page.post_thumbnail` media collections. Category mapping: products → `product-photos`, events/posts → `still-photos`.

Document the pattern in [widget-development.md](resources/docs/widget-development.md).

---

## Phase 4 — Apply pattern to under-rendering widgets

Wire the pool pattern into widgets whose demo-mode captures currently look empty:

- **Hero** — declare `appearance.background_image` ← `still-photos` (uses the shared appearance slot).
- **Image** — declare `config.image` ← `still-photos`.
- **ProductDisplay** — declare `config.product_image` (or whichever key the template reads) ← `product-photos`.
- Any other widget surfaced during implementation whose thumbnail is empty.

Regenerate affected thumbnails via `scripts/generate-thumbnails.js`.
