# 061. System Page Type — Infrastructure

Establishes the `system` page type and all governing rules. No existing pages are migrated in this session — that is session 062.

---

## Architecture decisions (agreed pre-session)

- `system` is a first-class page type alongside `default`, `post`, `event`, and `member`.
- System pages: type field locked read-only in the editor; slug completely locked (no editing of any segment — not even the portion after a prefix); not deletable through any UI or bulk action; not offered in the New Page type selector.
- Required widgets are declared by a static definition on the widget type, keyed by page slug. Widget cards for required widgets on system pages render without a remove button. Widget types that appear in any required-widget definition are pinned and cannot be deleted from the Widget Types admin.
- `system_prefix` SiteSetting added to the Routing fieldset in General Settings. Proposed default: empty string — system pages live at their root slug (e.g. `/login`, `/account`) so no URL migration is needed on a fresh install. Users who want namespacing can set it to `system` or anything else. Prefix changes migrate all system page slugs with the same safeguards already in place for `blog_prefix`, `events_prefix`, and `portal_prefix`.
- Editor heading for system pages: "Edit System Page".
- No create flow — system pages are seeded only. The type is excluded from the New Page type selector entirely.

|-----|-------|-------|------|
| `system_prefix` | `` (empty string) | `routing` | `string` |

---

## Phase 2 — Routing fieldset

Add a `system_prefix` field to the Routing fieldset in General Settings alongside `blog_prefix`, `events_prefix`, and `portal_prefix`.

Apply the same validation, slug migration logic, and warning UI already in place for the other prefix fields. When the prefix changes, all pages with `type = 'system'` have their slugs updated in bulk.

---

## Phase 3 — Editor UI

When `type = 'system'`, apply the following to the page editor:

- **Type field**: read-only Placeholder — same pattern as `event`, `post`, `member`.
- **Slug field**: completely locked. Display the full current slug as static text. No editable segment, no prefix display — just the full path shown as a read-only value.
- **Editor heading**: "Edit System Page".
- **Delete action**: suppressed — no delete button in the editor toolbar, excluded from bulk delete on the pages list.

---

## Phase 4 — Required widgets

Add a static required-widget definition to the widget type system. The definition is keyed by page slug and lists the widget handles that are required on that page. The cleanest location is a static property or method on the `WidgetType` model, populated from the widget seeder.

Example shape:

```php
// A static map consulted by the page builder:
public static function requiredForPage(string $slug): array
{
    return static::$requiredFor[$slug] ?? [];
}

protected static array $requiredFor = [
    'login'            => ['portal_login'],
    'signup'           => ['portal_signup'],
    'forgot-password'  => ['portal_forgot_password'],
    'account'          => ['portal_account_dashboard'],
];
```

In the page builder:

- When rendering a widget card on a system page, check whether that widget's handle appears in `requiredForPage($page->slug)`. If yes, hide the remove button.
- Widget types whose handle appears in any `$requiredFor` entry are pinned: the delete action is suppressed for them in the Widget Types admin.

The slug key used here is the segment after any prefix (i.e. the bare slug as stored, without the `system_prefix`). Confirm the right key format before building.

---

## Phase 5 — New Page type selector

Exclude `system` from the type options in the New Page form. It must not appear in any dropdown, radio group, or type picker available to admin users.

---

## Security checklist

Before closing:
- [ ] System pages cannot be deleted from the list, bulk action, or edit page
- [ ] System page slugs cannot be modified in the editor
- [ ] System page type cannot be changed in the editor
- [ ] Required widget remove buttons are suppressed on system pages
- [ ] Pinned widget types cannot be deleted from the Widget Types admin
- [ ] `system` is not offered as a type when creating a new page
