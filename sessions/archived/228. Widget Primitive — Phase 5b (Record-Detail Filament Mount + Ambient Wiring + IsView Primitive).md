# 228. Widget Primitive — Phase 5b (Record-Detail Filament Mount + Ambient Wiring + IsView Primitive)

Open the record-detail slot end-to-end: introduce the `IsView` interface (the shared widget-composition primitive locked in during 227 close-gate), give `RecordDetailAmbientContext` its record payload, open `RecordDetailSidebarSlot::ambientContext($record)`'s body, add `SOURCE_RECORD_CONTEXT` + projector + closed token registry (mirror of the page-context trio), build a Filament hosting mechanism on `EditContact`, and ship one placeholder widget that proves the full pipeline flows. Visible verification: opening any Contact's edit page renders "Record detail sidebar — Contact #N" through the contract resolver. The named risk for the entire Phase 5 arc — *"low probability the contract layer cannot coexist with Filament's Livewire model"* — gets answered here. Concrete starter widgets (Recent Notes, Recent Donations, Membership Status, etc.), the `record_detail_views` table, and the sub-nav rendering primitive all wait for 5c+.

## Design decisions (resolved before starting)

- **`IsView` is a PHP interface, not an abstract class or a base model.** Top-level in the `App\WidgetPrimitive` namespace, sibling to `Slot` / `DataContract` / `AmbientContext`. Declares the widget-composition contract: `handle(): string`, `slotHandle(): string`, `widgets(): array<PageWidget>`, `layoutConfig(): array`. Concrete implementations are free to be Eloquent models (Page eventually, DashboardView when 215's surface gets brought into the View vocabulary) or plain PHP classes (RecordDetailView this session). The interface is the contract; the storage shape is the implementer's concern. Per the **"a+" View framing** locked in during 227 close-gate (see `sessions/tracks/widget-primitive.md`'s "Design decisions locked in" section).
- **`RecordDetailView` is a plain PHP class this session, not Eloquent.** Constructor takes `handle`, `recordType` (FQCN, e.g. `Contact::class`), and `widgets` (array of `PageWidget` instances). The DB-backed `record_detail_views` table lands in 5c; this session's hardcoded inline construction is the byte-equivalent of session 214's hardcoded dashboard array, and gets replaced when 5c lands the table the same way 215 replaced 214's array with `dashboard_configs`.
- **`RecordDetailAmbientContext` carries `public readonly Model $record`.** Generic Eloquent base — the resolver/projector type-narrows where it cares. Per-type subclasses (RecordDetailContactAmbientContext, etc.) are over-specification before there's a second record type to compare against; defer that decision to 5c when Event/Org enter scope.
- **`RecordDetailSidebarSlot::ambientContext(Model $record): SlotContext`.** The signature changes from the no-arg form (which throws after 5a) to a one-arg form taking the record. Body returns `new SlotContext(new RecordDetailAmbientContext($record), publicSurface: false)`. `publicSurface: false` matches the dashboard's gate — admin-only surface, allows `is_public=false` collections to resolve for any future SWCT widget mounted here.
- **`SOURCE_RECORD_CONTEXT` is the source-as-capability mirror of `SOURCE_PAGE_CONTEXT`.** Per the migration doc's "capability grain varies by source" decision (locked in session 210): contracts on this source declare no `fields`; the resolver returns the full closed token map from `RecordContextTokens::TOKENS`. Adding a token is a grep-visible edit to that registry.
- **`RecordContextTokens::TOKENS = ['record_id', 'record_type']`.** Deliberately minimal. `record_id` is the model's `getKey()`. `record_type` is `class_basename($record)` — e.g. `'Contact'`, `'Event'`. **No `record_label` this session** — every model has a different label-derivation pattern (Contact's synthesized `name()`, Event's `title`, Organization's `name`, etc.) and committing to a label-resolution mechanism is its own architectural decision that should wait for the first widget that actually needs to display a label. Unblocks 5b without prejudicing 5d's design space.
- **`RecordContextProjector` is a direct mirror of `PageContextProjector`.** Receives a contract + an optional record. When the record is null (ambient is not `RecordDetailAmbientContext`), every token returns empty string — fail-closed. Per-call no caching needed; the token derivation is trivial (id + class_basename), and the resolver's existing per-render caching covers the contract-level dedup.
- **The Filament hosting mechanism is `EditContact::getFooterWidgets(): array` returning `[RecordDetailViewWidget::class]`.** Footer placement (below the form), not header, not a custom right-rail. Footer is the cheapest mount — no view override, no custom layout, just Filament's standard `getFooterWidgets()` machinery. Right-rail "real sidebar" layout is deferred until 5c's sub-nav primitive surfaces a need for it (sub-nav arrangement may make a right-rail layout obvious or unnecessary).
- **`RecordDetailViewWidget` is a Filament Widget class, mirror of `DashboardSlotGridWidget`.** It receives `$this->record` (a Contact, in this session's hardcoded mount) from Filament's standard EditRecord parent-page mechanism. Inside, it constructs the hardcoded `RecordDetailView` for `Contact::class`, calls `RecordDetailSidebarSlot::ambientContext($contact)` via the SlotRegistry, then loops `WidgetRenderer::render($pw, [], [], 'record_detail_sidebar')` per widget. The Blade view stacks the rendered fragments vertically.
- **The placeholder widget is hardcoded into the View's widget array, not registered against a slot at the registry level.** A single `PageWidget` instance is constructed in-memory inside `RecordDetailViewWidget::view()` (or wherever the View is built) — same shape as session 214's hardcoded dashboard widgets. The placeholder is deliberately disposable: once 5d ships a real widget, the placeholder can be deleted (or repurposed as a dev-mode debug widget gated to `app()->environment() !== 'production'`, but that's optional in 5b).
- **`RecordDetailPlaceholderDefinition` declares `allowedSlots(): ['record_detail_sidebar']`** — so the widget cannot be inserted on the page builder or dashboard via existing UI paths. Belt-and-braces: it's hardcoded in code anyway, but the gate makes the intent grep-visible.
- **No widget contract changes for any existing widget.** No retrofits, no `formatHints` additions, no schema edits. The session is additive only.
- **No `getBreadcrumbs()` override needed.** This session adds no new Filament page — only a Filament Widget mounted on the existing `EditContact` page, which already has its breadcrumb story.

## Phase 3 — `SOURCE_RECORD_CONTEXT` + projector + tokens

1. `app/WidgetPrimitive/DataContract.php`:
   - Add the constant `public const SOURCE_RECORD_CONTEXT = 'record_context';` alongside the existing source constants.
2. Create `app/WidgetPrimitive/RecordContextTokens.php`:
   - Mirror the **shape** of `app/Services/PageContextTokens.php` (the legacy location is `App\Services` because PageContextTokens predates the `App\WidgetPrimitive` namespace organization that solidified during Phase 2 — booked as a deferred-decisions cleanup item, see "Out of scope" below). The new file lives at the prescribed `App\WidgetPrimitive\RecordContextTokens` location, sibling to `Slot` / `DataContract` / `AmbientContext` / `Source`.
   - Public constant `TOKENS = ['record_id', 'record_type']`.
   - Public method `values(?Model $record): array<string, string>` — returns `['record_id' => (string) $record?->getKey(), 'record_type' => $record ? class_basename($record) : '']` (each token always returns a string; null record → empty strings, fail-closed).
3. Create `app/WidgetPrimitive/Projectors/RecordContextProjector.php`:
   - Mirror the shape of `app/WidgetPrimitive/Projectors/PageContextProjector.php`.
   - Constructor takes the `RecordContextTokens` registry.
   - `project(DataContract $contract, ?Model $record): array<string, string>` — calls `$this->recordContextTokens->values($record)` and returns the full token map (per source-as-capability, contract `fields` are ignored on this source).
4. `app/WidgetPrimitive/ContractResolver.php`:
   - Constructor adds `RecordContextProjector $recordContextProjector`.
   - The `match ($contract->source)` block in `resolve()` adds an arm: `DataContract::SOURCE_RECORD_CONTEXT => $this->recordContextProjector->project($contract, $this->recordFromContext($context)),`.
   - New private method `recordFromContext(SlotContext $context): ?Model` — returns `$context->ambient instanceof RecordDetailAmbientContext ? $context->ambient->record : null`. Fail-closed: any other ambient yields null and every token renders empty.
5. `app/Providers/WidgetServiceProvider.php`:
   - Register `RecordContextTokens::class` and `RecordContextProjector::class` as singletons in `register()`, alongside the existing `PageContextTokens` / `PageContextProjector` bindings.

## Phase 4 — Wire the record through `WidgetRenderer`

The slot dispatch at [`app/Services/WidgetRenderer.php:73–76`](app/Services/WidgetRenderer.php#L73-L76) currently calls `->ambientContext()` with no args for every non-canvas slot. After 5b, `RecordDetailSidebarSlot::ambientContext(Model $record)` requires a non-null Model, so the renderer needs a way to receive the record from its caller.

Mirror of how session 212 added the `$slotHandle` positional parameter — same shape, same precedent.

1. `app/Services/WidgetRenderer.php`:
   - Add `use Illuminate\Database\Eloquent\Model;` at the top.
   - Add a 5th positional optional parameter to `render()`: `?Model $record = null`. Final signature:
     ```php
     public static function render(
         PageWidget $pw,
         array $columnChildren = [],
         array $fallbackCollectionData = [],
         string $slotHandle = 'page_builder_canvas',
         ?Model $record = null,
     ): array
     ```
   - Replace the two-way slot dispatch at lines 73–76 with a three-way branch:
     ```php
     if ($isCanvas) {
         $slot = app(SlotRegistry::class)->find('page_builder_canvas')->ambientContext($pageContext, $tokenPage);
     } elseif ($slotHandle === 'record_detail_sidebar') {
         if ($record === null) {
             return ['html' => null, 'styles' => '', 'scripts' => ''];
         }
         $slot = app(SlotRegistry::class)->find('record_detail_sidebar')->ambientContext($record);
     } else {
         $slot = app(SlotRegistry::class)->find($slotHandle)->ambientContext();
     }
     ```
   - **Fail-closed when record is null** on the record-detail path: short-circuit to the empty render shape (mirror of the `widgetType === null` guard at line 26–28). The Filament widget always passes a record when mounted on a real `EditRecord` page, so this is purely defensive against accidental null-record calls (e.g., a dev mounting the widget on a page without a record context).
2. **No other call sites need updating.** Existing callers (`PageController`, `DashboardSlotGridWidget`'s blade, public-page render paths) all leave the new 5th param at its default `null` — byte-equivalent for canvas and dashboard rendering.

## Out of scope

- **`record_detail_views` table + admin UI for authoring per-record-type View sets.** Phase 5c. The hardcoded `RecordDetailView` array in `RecordDetailViewWidget` is the byte-equivalent of session 214's hardcoded dashboard widgets, and gets replaced when 5c lands the table the same way 215 replaced 214's array.
- **Sub-nav rendering primitive (Filament `getRecordSubNavigation()` driven by Views).** Phase 5c. The Templates/Themes refactor folds in as a consumer of the same primitive.
- **Multiple record types** (Event, Org, etc.). 5c (when the View registry binds Views to record types polymorphically) or 5d+ (when concrete Event/Org-bound widgets ship).
- **Concrete record-detail widgets** — Recent Notes for current contact, Recent Donations, Membership Status, Recent Activity. Phase 5d+. Each is a contract design + projector + test-lock pair, mirror of Phase 4a–4h.
- **`record_label` token** and any per-record-type token expansions. Deferred until the first widget needs to display a label — different record types have different label-derivation patterns and committing to a mechanism in 5b would prejudice 5d's design space.
- **Right-rail layout for the slot.** Footer placement via `getFooterWidgets()` is the cheapest mount. A real right-rail layout (form + sidebar grid) would require a custom Filament view override and is deferred until 5c's sub-nav primitive surfaces a layout need — sub-nav arrangement may make a right-rail layout obvious or unnecessary.
- **`DashboardConfig` retrofit to `DashboardView` implementing `IsView`.** Cheap follow-up session post-5d, scheduled ad-hoc. Not load-bearing for any Phase 5 milestone.
- **`Page` implementing `IsView`.** Per the "a+" framing, Page-as-View is the elective bottom of the View stack. Lands only when "walk every widget-mounting surface" becomes load-bearing. Could be never. Documented in `sessions/tracks/widget-primitive.md`'s "Forward hooks".
- **Template-side migration from `$pageContext` to slot-ambient lookups.** Templates keep reading `$pageContext` as today; the typed-ambient migration is a separate Phase 5 session bundled with whatever surface forces it.
- **`PageContextTokens` namespace migration from `App\Services` to `App\WidgetPrimitive`.** Pre-existing inconsistency — `PageContextTokens` predates the `App\WidgetPrimitive` namespace organization that solidified during Phase 2 (session 211). New `RecordContextTokens` lands at the prescribed `App\WidgetPrimitive` location this session; co-locating `PageContextTokens` is a standalone mechanical cleanup pass (rename + namespace + ripple through `WidgetServiceProvider`, `PageContextProjector`, and any test imports). Load-bearing for nothing; book it for a convenient cleanup session, not 5b.
- **Schema migrations.** None this session.
