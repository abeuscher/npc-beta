## What was built

### Phase 1 (session-338/1) — Tour framework + demo-safe showcases

- **Reusable multi-page engine** under `resources/js/admin/tour/`, deliberately split so a later contextual-help mode can reuse the primitives:
  - `state.js` — localStorage resume token (a tour id + step index); the tour walks real pages, so each transition is a full reload the active step must survive.
  - `anchors.js` — `resolveAnchor()` / `waitForElement()` / `ensureNavVisible()`. **Every selector is one we own** — `data-tour` markers we inject, or sidebar links matched by our own route URLs — never a Filament-generated class (the standing session-249 selector-fragility mitigation).
  - `controller.js` — drives one *page-group* (the maximal run of consecutive steps on the current page) per driver instance and owns every cross-page transition: persist the step → hard-navigate → resume on load.
  - `tours/product-tour.js` — the ordered step script (copy + anchor + page).
  - `index.js` — registers the tour, a delegated launcher click, and resume-on-load hooks.
- **Owned anchors + role-gated URL map** via render hooks in `AdminPanelProvider`: `data-tour` markers on chrome (`CONTENT_START`, the resource-table hook, the topbar View-site link), and a `HEAD_END` script emitting each tour-target page's URL **only when the current viewer can reach it** — so the tour skips (never 404s) inaccessible steps.
- **Demo-safe showcases** for the two features the public `demo` role is deliberately walled off from (the `DemoRoleLockdownTest` guard — Stripe/QuickBooks credentials, user/role management, bulk import — left **unchanged**):
  - **Roles** → `TourRolesShowcasePage` renders the **real permission matrix component fed sample data** (no real roles/users; the permission names shown are the app's feature vocabulary, not secrets; nothing toggled persists). Achieved by extracting the matrix markup into a shared partial (`components/admin/permission-matrix.blade.php`) used by both the showcase **and** the real role form — no drift.
  - **Import** → `TourImportShowcasePage` renders a **captured screenshot** of the real importer (committed PNG, the same model as widget thumbnails; an `@on-demand` Playwright spec regenerates it).
  - **Financial** → points at the demo-viewable **Transactions ledger** (real Stripe rows), not the locked Finance Settings page.
- **Launcher** in `DashboardWelcomeWidget`; manual launch, visible to all admins (the tour doubles as onboarding). No auto-start in v1.
- `TourShowcasePagesTest` (Pest) pins the showcases are reachable by the `demo` role while the real locked pages stay shut.

### Phase 2 (session-338/3) — Owner feedback: interactive contact→record→ledger flow + polish

Reworked the tour from list-page beats into a guided walk through a real supporter, and folded in the owner's feedback round:

- **Interactive steps** — the spotlighted element is the real link/button and the user's own click carries the tour forward (the contact row → the record; "View transactions" → the ledger). Engine additions: click-to-advance (a one-shot listener persists the next step before the native navigation), a `navRow` anchor (highlight the table row whose record link matches a URL-map page), **live-selector re-query** for self-target anchors (immune to the Livewire DOM morph that detaches a captured element between same-page steps), and a stale-spotlight-ring sweep on each highlight.
- **New flow:** Welcome → click a seeded **hero contact** → Membership Status (on the record) → click **View transactions** → that supporter's Stripe/QuickBooks ledger → Mailing lists → Events → Import → Roles → a dashboard finale highlighting the **help search** and the **`?` flyout**. Collapses the standalone Donations + Memberships list beats into the record flow.
- **EditContact:** "View transactions" promoted from the ellipsis to a **visible header button** (the tour's anchor, via `extraAttributes`), Timeline demoted into the ellipsis — in both action methods.
- **`RecordDetailViewWidget` set `$isLazy = false`** so its membership/giving panel renders in the initial paint (Filament's default lazy-on-scroll left it out of the DOM when the tour looked).
- **`DemoBaselineSeeder` designates a "hero" contact** (an active membership + a run of donations, reusing the generator's correct transactions) and floats it to the top of the list.
- **Fixes:** importer no longer scroll-hops (centered step); the Roles popover no longer falls beneath (bounded anchor); dark-mode spotlight gained a bright amber ring (Playwright-verified contrast); popover type +30%; a distinct **"Exit Tour"** button; launcher reads **"Tour the Product"**; mailing-list copy says **Mailchimp only**.

## Security considerations

The whole point of the showcase mechanism was to sell two features **without weakening the demo lockdown**, and the lockdown is intact: `DemoRoleLockdownTest` still asserts the `demo` role cannot reach Finance Settings / Roles / the Importer, and `TourShowcasePagesTest` adds that the demo-safe showcases (which load no real data and no secrets) are reachable while those real pages stay shut. driver.js adds no server surface (no routes/schema/contract change); it ships in the **admin** bundle only (authenticated sessions). One forward caveat recorded for the later in-app-help reuse: driver renders popover copy as HTML, so if tour text ever becomes operator-authored it must be sanitized — in this tour every word is static and ours, so there is no exposure.

## Production deployment notes

- Front-end change only at runtime — the deploy image build compiles the tour into the admin bundle from the committed source (`public/build` stays gitignored). No migration, no seeder run required for the tour itself.
- The richer demo experience depends on `DemoBaselineSeeder` designating the hero contact; it flows to the demo node via the existing `demo:restore` blob (built in the Faker-equipped authoring env). The tour degrades gracefully without a hero (opens the newest contact; panels render empty).
