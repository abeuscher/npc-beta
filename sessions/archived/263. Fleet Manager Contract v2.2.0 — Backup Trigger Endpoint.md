# 263. Fleet Manager Contract v2.2.0 — Backup Trigger Endpoint

Add a single mTLS-gated HTTP endpoint (`POST /api/backup/trigger`) that synchronously runs the existing `backup:run` artisan command and returns a JSON envelope reporting the new `last_backup_at` timestamp. Additive contract bump v2.1.0 → v2.2.0. Unblocks FM session 020 (operator-facing "Trigger backup now" affordance in the FM admin UI).

This is **A1d** from `release-plan.md` Track A, lifted as a new working-set entry at session 262 close (parallel framing to A1b/A1c — each contract bump is its own session entry).

## Design decisions (resolved before starting)

These are agreed-going-in. Confirm at session start during the read-through.

- **Sync execution, not async dispatch.** The endpoint runs `Artisan::call('backup:run')` synchronously inside the request lifecycle and returns once the backup completes (or fails). Pre-Beta-1 scope rationale: single-operator, low frequency, simpler to implement; if a multi-tenant production driver appears post-Beta-1 (large data volumes, concurrent operator triggers, queue worker pressure), revisit and split into queued-dispatch + status-polling.

- **HTTP timeout: 600 seconds (10 minutes).** Backup duration scales with data volume; pre-Beta-1 droplets are small. 10-minute ceiling covers the realistic case with margin. Per-location `fastcgi_read_timeout 600;` in nginx; `set_time_limit(600)` defensively at the top of the controller action; `request_terminate_timeout` confirmed-or-bumped per Phase 1 step 2.

- **Same mTLS gate as `/api/health` and `/api/logs`.** Location-scoped `if ($ssl_client_verify != "SUCCESS") { return 403; }`. The existing `fm-client.crt` operator-pasted material covers this endpoint without any client-side cert reissue.

- **Throttle: 6 requests per minute per source IP** (`throttle:6,1`). Higher than `/api/health`'s 60/min because backups are expensive; lower than 1/min because operator might re-trigger after a transient failure. Per-IP because mTLS doesn't cleanly identify "which FM" (single-FM today; cert-array path in v2.x is post-Beta-1).

- **Method: `POST`.** Backup is state-mutating (writes to backup destination — local disk or DO Spaces). No request body required; ignore any body sent.

- **Success-record mtime cross-check (the integrity guard).** When `Artisan::call('backup:run')` returns 0, the controller cross-checks the `storage/app/private/fleet/last-backup-at` file's contents against the request's start time. If the recorded timestamp is null OR older than the request start, the response downgrades to `status: failed` with `message: 'backup:run exited cleanly but success record was not updated'`. Rationale: the whole point of A2(b) is "trigger and **verify** a backup against a node"; cross-checking the success record is the verification half. Costs ~3 lines, prevents a misleading "backup just succeeded" panel on the FM side when the success listener silently failed.

- **Response shape:** JSON envelope mirroring `/api/health`'s shape conventions. Always HTTP 200 (matches `/api/health`'s pattern of 200 + degraded-status envelopes; nginx 5xx reserved for transport-level failures).
  - **Success:**
    ```json
    {
      "contract_version": "2.2.0",
      "status": "success",
      "last_backup_at": "<ISO 8601>",
      "duration_ms": <int>,
      "message": null
    }
    ```
  - **Failed** (non-zero exit OR exception OR success-record mtime didn't move):
    ```json
    {
      "contract_version": "2.2.0",
      "status": "failed",
      "last_backup_at": <null OR previous timestamp if the success-record file was readable>,
      "duration_ms": <int>,
      "message": "<sanitised operator-readable error per below>"
    }
    ```

- **Error-message sanitisation.** Two error sources feed the `message` field:
  - On non-zero `Artisan::call` return: source is `Artisan::output()` (potentially multi-line / verbose).
  - On caught exception: source is `\Throwable::getMessage()` (typically single-line already).

  Sanitisation pipeline (same for both):
  1. **Strip absolute application-root prefix** (e.g. `/var/www/html/`) so paths read app-root-relative (`app/Services/Foo.php` rather than `/var/www/html/app/Services/Foo.php`). Open-source application root means relative paths are public information; absolute prefix strip is cosmetic, not security-critical (per 262 close-time sign-off — paths from app root downward acceptable).
  2. **Collapse newlines** to ` | ` so the message is single-line for envelope hygiene.
  3. **Cap at 500 characters** (truncate with trailing `…` if longer).
  4. **No stack traces.** Exceptions: take only the topmost `getMessage()`. Artisan output: as-emitted (the command's own logging is what reaches the operator).

- **No new artisan command, no new service.** This is a thin HTTP wrapper. The existing `backup:run` artisan + `RecordBackupSuccess` listener + `last-backup-at` success-record file do all the actual work.

- **Out of scope** (full list under § Out of scope below): async / queued execution, backup blob download, per-bucket targeting, `BackupHasFailed` listener, status streaming, schema changes, help-doc updates.

## Phase 1 — Nginx conf + PHP-FPM timeout + route + controller

**Goal:** new `POST /api/backup/trigger` endpoint live behind mTLS + throttle, returning the response envelope shape per Design Decisions.

**Steps:**

1. **`docker/nginx/prod.conf`** (single-file bind-mount — use `docker compose up -d --force-recreate nginx` post-edit, not `restart`): add a third mTLS-gated location block immediately after `location = /api/logs { ... }`:
   ```nginx
   location = /api/backup/trigger {
       if ($ssl_client_verify != "SUCCESS") { return 403; }
       fastcgi_read_timeout 600;
       try_files $uri /index.php?$query_string;
   }
   ```
   The per-location `fastcgi_read_timeout 600;` is load-bearing — without it, nginx kills the upstream connection at the default 300s while the backup is still running. Verify the existing `php` location block's 300s timeout stays unchanged (other endpoints don't need 10 minutes).

2. **PHP-FPM `request_terminate_timeout` confirmation/bump.** Resolve at session start where this currently lives (likely `docker/php/local.ini` if set, or default if not). If it's lower than 600s, bump to 600s in the appropriate file (single-file bind-mount caveat applies). Document the resolution in the session log either way.

3. **`routes/api.php`**: add the new route alongside the existing two:
   ```php
   Route::middleware(['throttle:6,1'])
       ->post('/backup/trigger', [BackupController::class, 'trigger']);
   ```

4. **New `app/Http/Controllers/Api/Fleet/BackupController.php`** with a single `trigger(Request $request): JsonResponse` action. Implementation shape:
   1. `set_time_limit(600);` defensively at the top of the action.
   2. Capture `$startTime = microtime(true);`.
   3. Capture the **pre-call success-record timestamp** (read `storage/app/private/fleet/last-backup-at` via `Storage::disk('local')`; null if missing) — used both for the post-call mtime cross-check AND as the `last_backup_at` value in failed-envelope responses (so the operator sees the last *known* successful backup rather than just `null` when the trigger fails).
   4. Run `Artisan::call('backup:run')` inside a `try { ... } catch (\Throwable $e) { ... }` block.
   5. **On `Artisan::call` returning 0:**
      - Re-read the success-record file.
      - **mtime cross-check:** if the recorded timestamp is null OR older than `$startTime` (Carbon-parse the timestamp), downgrade to `status: failed` with `message: 'backup:run exited cleanly but success record was not updated'`. Otherwise return the success envelope with the new timestamp.
   6. **On non-zero `Artisan::call` return:** return the failed envelope. Source for `message`: `Artisan::output()`. Sanitise via the pipeline in Design Decisions.
   7. **On caught exception:** return the failed envelope. Source for `message`: `\Throwable::getMessage()`. Same sanitisation pipeline.
   8. `duration_ms` is `(int) ((microtime(true) - $startTime) * 1000)` regardless of branch.
   9. All branches return HTTP 200 (per response-shape decision).

**Pass criterion:** the new route is reachable via `curl -X POST --cert ... --key ... https://<domain>/api/backup/trigger` and returns the JSON envelope; mTLS rejection returns 403; throttle returns 429 on the 7th request within a minute; `migrate:status` clean (no schema drift).

3. **Failed trigger via thrown exception.** Mock `Artisan::call` to throw `new \RuntimeException('pg_dump failed: connection refused')`. Assert HTTP 200 with `status: 'failed'`, `message` contains the exception message, no stack trace.
4. **Success-record mtime cross-check (integrity guard).** Pre-write the success-record file with a stale timestamp. Mock `Artisan::call` to return 0 but do NOT update the file. Assert HTTP 200 with `status: 'failed'`, `message: 'backup:run exited cleanly but success record was not updated'`, `last_backup_at` reports the stale (pre-call) timestamp.
5. **mTLS rejection.** Match the pattern from `HealthEndpointTest` / `LogsEndpointTest` — likely deferred to deployment-time verification rather than unit-level mocking. If those tests do exercise the gate at unit level, mirror that pattern.
6. **Throttle.** Dispatch 7 requests within 60s; assert the 7th returns 429. Match the throttle-test pattern from existing endpoint tests.
7. **Method enforcement.** `GET /api/backup/trigger` returns 405.
8. **Response sanitisation — absolute path strip.** Mock `Artisan::call` to throw an exception whose `getMessage()` contains `/var/www/html/app/Services/Foo.php`. Assert the response `message` contains `app/Services/Foo.php` but not `/var/www/html/`.

10. **Response sanitisation — 500-char cap.** Mock with a very long message (e.g. 1000 chars); assert the response `message` is ≤ 500 chars (truncated with trailing `…`).
11. **`Artisan::output()` source on non-zero exit.** Distinct from the exception-source test: confirm the controller pulls from `Artisan::output()` (not from a hypothetical thrown exception) when `call` returns non-zero.

## Security checklist

Before closing:

- [ ] **mTLS gate enforced at nginx, not application layer.** Same pattern as `/api/health` and `/api/logs`. Bypassing the cert returns 403 before reaching PHP-FPM.
- [ ] **Throttle prevents trivial DoS.** 6/min per source IP caps operator-driven re-triggers and prevents accidental loops.
- [ ] **Response payload contains no stack traces, no absolute filesystem paths, no secret material.** Relative paths from app root downward are acceptable per 262 close-time sign-off (open-source app, paths reveal nothing). Error messages are sanitised via the documented pipeline (path-prefix strip + newline collapse + 500-char cap).
- [ ] **`request_terminate_timeout` accommodates 10-minute backups.** Documented or bumped; resolution captured in session log.
- [ ] **No new env vars introduced.** Backup destination + credentials read from existing `BACKUP_DISKS` / `SPACES_*` env vars from CRM 242.
- [ ] **No GET / no idempotency-required semantics.** POST-only; throttle handles burst-protection.
- [ ] **Success-record mtime cross-check in place.** The integrity guard prevents the misleading "artisan exits 0 but listener silently failed" path from reporting success.
- [ ] **Spec doc accurately documents the cross-check semantics.** Consumers (FM 020) need to know that `status: failed` with a stale `last_backup_at` is a distinct failure mode.
