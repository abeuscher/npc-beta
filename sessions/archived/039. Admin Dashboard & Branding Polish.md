# 039. Admin Dashboard & Branding Polish

Improve the first thing an admin sees on login. The dashboard currently looks too generic. This session restructures it into a clean, useful layout and exposes the primary accent colour as a configurable setting.

---

## Goals

1. **General Settings — primary colour picker.** Add a colour picker field to the Admin Panel section of General Settings. Store the value in `site_settings`, load it via AppServiceProvider, and apply it to the Filament panel's primary colour at boot. The admin can change the colour, save, and see it reflected immediately on next page load.

2. **General Settings — Stripe and QuickBooks API key fields.** Add two empty text input fields (Stripe API Key, QuickBooks API Key) to a new "Integrations" section of General Settings. Save to `site_settings`, load via AppServiceProvider. No functionality wired up — these exist so the vendor status widget has something to check.

3. **Dashboard restructure.** Replace the current default Filament widgets with four custom widgets arranged in a 2×2 grid:

   | Left column | Right column |
   |---|---|
   | Welcome message | Help (placeholder) |
   | Quick actions | Integration status |

## Primary colour implementation notes

- `ColorPicker::make('admin_primary_color')` in Filament returns a hex string (e.g. `#f59e0b`)
- Store and load via the existing AppServiceProvider pattern:
  `'site.admin_primary_color' => $settings->get('admin_primary_color')?->value ?? '#f59e0b'`
- In `AdminPanelProvider::panel()`, replace `Color::Amber` with:
  `Color::hex(config('site.admin_primary_color', '#f59e0b'))`
- `Color::hex()` is the Filament v3 method for generating a full shade palette from a hex value. Verify the import is `use Filament\Support\Colors\Color;`
- Default `#f59e0b` preserves the current amber colour if the setting is not yet saved

## Stripe / QuickBooks settings notes

- Add a `Forms\Components\Section::make('Integrations')` section to GeneralSettingsPage
- Two `TextInput` fields: `stripe_api_key`, `quickbooks_api_key` — nullable, no validation
- In `AppServiceProvider::boot()`, add:
  ```php
  'services.stripe.key'       => $settings->get('stripe_api_key')?->value ?? '',
  'services.quickbooks.key'   => $settings->get('quickbooks_api_key')?->value ?? '',
  ```
- No migration needed — `site_settings` is a key/value table and accepts new keys automatically
