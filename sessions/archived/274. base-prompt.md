## Process rules for every session

The base-prompt rules (process, close gate, style) in `sessions/template-base-prompt.md` apply. Session-specific notes:

- **Pre-beta license applies throughout.** No backward-compat shims, no deprecation periods, no column-rename-with-data-migration-then-drop-later dance. If a cleanup pick calls for removing a column, renaming a method, or breaking an internal API, just do it. The squash absorbs whatever new migrations this session commits, so "drop unused column" = one migration that the squash immediately collapses into the baseline. No live installations exist; the wipe-and-reinstall path is safe.
- **No agent contract change.** No `docs/fleet-manager-agent-contract.md` body edit, no version bump. The 273 W8 #3 parity test continues to gate code-vs-spec drift; do not touch the spec doc this session.
- **Run migrations via Docker** after writing them: `docker compose exec app php artisan migrate` (or `migrate:fresh --seed` when appropriate). Do not pause to ask first.
- **Update `docs/schema/`** after writing any new migration during the optimization phase. The squash itself does not require per-table doc edits beyond the squash-note block at the top of `docs/schema/README.md`.
- **Tests for new behaviour:** this session is mostly zero-behaviour-change (the squash collapses history; the optimization pass is net-reduction). Any pick that changes user-observable behaviour needs a test. Perf-optimization picks specifically need test coverage of the before/after query counts or payload shapes if the change is non-trivial — see 208's `PageBuilderApiTest` query-count regression-guard pattern.
- **When implementation is complete on each iteration**, run the fast test suite: `docker compose exec app php artisan test --exclude-group=slow`. Target: **273 close baseline ± a small delta** (≈2169 ± any test deltas from optimization-phase picks).
- **Front-end builds — only if a safe-fix touches front-end source.** `npm run build` if any `resources/js/**` or `resources/scss/**` change lands; `build:public` if any widget asset surface changes. Optimization-phase picks may or may not touch front-end source.
- **Playwright is required before pausing for manual test** — the squash forces a full `migrate:fresh --seed` which several e2e specs depend on as a baseline. Any seed breakage will surface there. If the residual cumulative-load FilePond flake reappears, retry once on a clean stack before treating as a regression.
- **Pause for manual testing.** Announce that testing is ready. Then stop. Do not suggest closing the session or take any further action.
- **Long runtime expected.** Five phases. Inventory + user picks happen first (no code changes). Then each pick is its own logical commit. The squash itself is the final commit before test + manual-test announcement. If scope balloons mid-session, the user may bless a `/2` carve-out — but the squash must still land in this session, not be deferred.
