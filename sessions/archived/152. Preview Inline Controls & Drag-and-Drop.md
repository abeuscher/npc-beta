# 152. Preview Inline Controls & Drag-and-Drop

Add overlay controls to each preview region, drag-and-drop reordering of root widgets, and a config requirements system so the editor can warn users when a widget needs setup before it will render.

**Reference document:** `docs/editor-migration-plan.md` — read this file in full before writing any code. The component tree (Part 3, section 3.2) and store design (Part 3, section 3.1) are relevant.

**Prior sessions:** Sessions 147–151 built the full Vue editor stack: API endpoints, Pinia store, preview canvas, toolbar, inspector panel with all field types, widget deletion, and the Livewire teardown. The editor is now fully Vue-driven (except the widget picker and save-as-template modals on Livewire). `PreviewRegion.vue` currently renders preview HTML with a transparent click-to-select overlay.

---

## Design decisions (resolved before starting)

- **Overlay controls:** Each `PreviewRegion` gets a hover toolbar pinned to the top-right corner. The toolbar shows a drag handle (grip icon) and an ellipsis menu button. Controls appear on hover, always visible when selected.
- **Ellipsis menu actions:** Add above, Add below, Copy, Move up, Move down, Delete. Delete shows the existing `ConfirmDeleteModal`. "Add above/below" dispatches `open-widget-picker` with the correct `insertPosition`. Copy calls `store.copyWidget()`. Move up/down calls `store.reorderWidgets()` with swapped sort orders. All actions use existing store actions and API endpoints.
- **Drag-and-drop:** Use `vuedraggable` (SortableJS wrapper for Vue 3). Root-level reordering only — column children are out of scope. The draggable list wraps the PreviewRegion components inside PreviewCanvas. On drag end, call `store.reorderWidgets()` with the new order.
- **Config requirements on widget types:** New `required_config` JSONB column on `widget_types` (default: `null`). Shape: `{ "keys": ["collection_handle"], "message": "Select a collection and map its image field to display logos." }`. When `keys` is present and any listed config key is empty/null on a widget instance, `PreviewRegion` shows the notice instead of preview HTML. First line: widget type label (from store). Second line: the `message` from `required_config`.
- **Placeholder removal from Blade:** Remove all `@include('widgets.components.widget-placeholder', ...)` calls from widget templates. The `@else` branches render nothing. The placeholder partial file stays but is no longer referenced. The public site shows nothing for unconfigured widgets. The admin editor shows the Vue-driven notice.
- **No mode toggle removal this session.** Edit/handles mode toggle stays. That removal is a separate future session.
- **Column widgets out of scope.** No drag-and-drop for column children, no inline controls for nested widgets. That is session 153.

## Phase 1 — Config requirements system

### 1a. Migration

Add `required_config` JSONB column to `widget_types`, nullable, default `null`.

### 1b. Seeder update

Update the widget type seeder to populate `required_config` for widgets that need it. Derive the values from the existing `@else` placeholder includes — each one tells you exactly which widget needs what:

| Widget handle | required_config |
|---|---|
| `carousel` | `{ "keys": ["collection_handle", "image_field"], "message": "Select a collection and map its image field to display slides." }` |
| `logo_garden` | `{ "keys": ["collection_handle", "image_field"], "message": "Select a collection and map its image field to display logos." }` |
| `board_members` | `{ "keys": ["collection_handle"], "message": "Select a collection and map its fields to display team members." }` |
| `video_embed` | `{ "keys": ["video_url"], "message": "Enter a YouTube or Vimeo URL." }` |
| `map_embed` | `{ "keys": ["embed_code"], "message": "Paste a Google Maps share link or embed code." }` |
| `web_form` | `{ "keys": ["form_id"], "message": "Select a form to embed." }` |
| `event_registration` | `{ "keys": ["event_id"], "message": "Select an event to display its registration form." }` |
| `event_description` | `{ "keys": ["event_id"], "message": "Select an event to display its details." }` |

Read each widget template's `@else` branch to verify the correct config keys before seeding. The table above is a starting point — the actual key names must match what the template checks.

### 1c. Bootstrap data & API

Add `required_config` to the widget type data in:
- `PageBuilder::getBootstrapData()` — include in each widget's `widget_type_*` fields
- `PageBuilderApiController::buildTree()` — include in the widget tree response
- The widget types lookup endpoint — include in the registry

Add it to the TypeScript `Widget` interface and `WidgetType` interface in `types.ts`.

### 1d. PreviewRegion shows config notice

When the selected widget has `required_config` with keys, and any of those keys are empty/null in `widget.config`:
- Instead of rendering `preview_html`, show a styled notice
- First line: widget type label (large, bold)
- Second line: the message from `required_config`
- Styling uses the `.widget-preview-notice` class already in `admin.css` (adjust as needed)

### 1e. Remove placeholder includes from Blade templates

Remove all `@include('widgets.components.widget-placeholder', ...)` calls from widget templates. The `@else` branches should render nothing (empty string or remove the `@else` block entirely if the `@if` is the last thing in the template).

Widget templates to update:
- `carousel.blade.php`
- `logo-garden.blade.php`
- `board-members.blade.php`
- `video-embed.blade.php`
- `map-embed.blade.php`
- `web-form.blade.php`
- `event-registration.blade.php`

## Phase 2 — Drag-and-drop reordering

### 2a. Install vuedraggable

```bash
npm install vuedraggable@next
```

### 2b. Drag-and-drop in PreviewCanvas

Wrap the PreviewRegion list in PreviewCanvas with a `draggable` component from `vuedraggable`:
- Entire widget region is the drag handle for now (session 153 introduces the quadrant-based grab handle)
- On drag end, build the new order array and call `store.reorderWidgets()`
- Smooth drag animation (SortableJS defaults are fine)
- The empty state ("No blocks yet") remains outside the draggable wrapper
