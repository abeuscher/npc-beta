# 139. Unified Preview & Edit Layout

Replace the page builder's separate edit/preview modes with a single two-pane layout: a persistent stacked widget preview on the left, and a persistent inspector panel on the right. Clicking a widget in the preview selects it and populates the inspector ‚Äî no mode switch, no re-render.

---

## Design decisions (resolved before starting)

- **Edit / Handles toggle replaces Edit / Preview.** The toggle switches between two left-pane views: **Edit** (the new unified preview ‚Äî all widgets rendered with public CSS, clickable for selection, inspector always visible on the right) and **Handles** (the existing block card list with drag handles, ellipsis menus, and reorder ‚Äî needed until drag-to-reorder is built into the preview pane). The preview iframe is removed. The "Handles" view retains all current block card functionality unchanged.
- **Selection by click.** Clicking a widget in the left pane selects it: adds a visible selection ring, populates the inspector with that widget's config fields. Clicking outside any widget (or pressing Escape) deselects. Clicking a different widget transfers selection directly.
- **Persistent inspector.** The right pane is always visible. When nothing is selected it shows an empty state ("Click a widget to edit"). It does not collapse or overlap the preview. This pane will eventually become a tabbed Adobe-style toolbar with multiple stacked panels ‚Äî for now, the inspector is the sole occupant.
- **Libraries load once.** Session 138 built per-library bundles and a manifest. In this layout, all required library scripts/CSS are loaded once when the page builder mounts (based on the union of all widget types present on the page). Swiper, Chart.js, etc. initialize naturally as their Alpine `x-data` components mount ‚Äî no destroy/reinit cycle, no timing hacks.
- **Block management stays.** Drag-to-reorder, add/delete/copy, column widget child management all continue to work. The structural controls (drag handle, ellipsis menu) are overlaid on each widget region in the preview, not in a separate card list.
- **Zoom and viewport presets carry over.** The viewport width toggle (Desktop/Tablet/Mobile) from session 138 applies to the entire left-pane preview, not per-widget. The zoom factor scales the preview to fit the available panel width.
- **Inline text editing carries over.** The `data-config-key` / contenteditable system from session 137 continues to work inside the preview pane. It applies to the selected widget.
- **Column widgets.** Column widgets render their children inline in the preview (as the public site does). Clicking a child widget inside a column selects that child. The column widget itself is selectable for its own config (num_columns, grid_template_columns).

## Phase 1 ‚Äî Unified layout shell

Replace the edit/preview mode toggle with an Edit/Handles toggle. The default view is Edit.

**Edit mode ‚Äî left pane (preview):**
- Renders all page widgets in order, using `WidgetRenderer`, wrapped in `.np-site` so public CSS applies.
- Viewport width toggle (Desktop 1920 / Tablet 1024 / Mobile 375) at the top of the pane, controlling zoom factor for the entire preview.
- Each widget region is wrapped in a clickable container with a data attribute (`data-widget-id`) for selection targeting.
- On click, dispatch `block-select-requested` with the widget ID.
- Selected widget gets a visible selection ring. No blur/zoom on siblings ‚Äî just a ring and subtle highlight.

**Handles mode ‚Äî left pane:**
- The existing block card list (current `page-builder-block` components) with drag handles, ellipsis menus, and reorder. Unchanged from current behavior except the per-block preview rendering is removed (no more inline preview when selecting a block in handles mode).

**Right pane (inspector) ‚Äî shared by both modes:**
- Always visible, sticky-positioned.
- Shows the inspector for the selected widget, or an empty state when nothing is selected.
- No changes to inspector internals ‚Äî it already works via Livewire events.
- Selecting a block in handles mode also populates the inspector.

**Rename:**
- `switchToEdit` ‚Üí handles the Edit view (new unified preview)
- `switchToPreview` ‚Üí becomes `switchToHandles` (the block card list)
- `$mode` values: `'edit'` and `'handles'` (replaces `'edit'` and `'preview'`)

**Remove:**
- The preview iframe.
- The per-block preview rendering inside `page-builder-block.blade.php` (the `x-show="selected"` preview div, zoom logic, `x-ignore`, `reinitWidgetAlpine`).

## Phase 2 ‚Äî Block controls overlay

Add structural controls to each widget region in the preview:

- **On hover:** show a subtle top bar with the widget label and a drag handle.
- **On select:** show the top bar persistently, plus an ellipsis menu (reorder, add above/below, copy, delete ‚Äî same actions as the current block menu).
- These controls float above the widget content (absolute/sticky positioned within the widget wrapper), not in a separate panel.
- Column widgets show their slot structure with "Add Block" buttons inside each column.

---

## Phase 3 ‚Äî Library loading (simplified)

Move library loading from per-widget to page-level:

1. On mount, scan all widget types present on the page and collect the union of all `libs` arrays.
2. Load all required library scripts and CSS once, before the preview HTML is rendered.
3. Alpine `x-data` components in widget templates initialize naturally ‚Äî Swiper sees correct container dimensions because the preview is already laid out.
4. No `x-ignore` / `reinitWidgetAlpine` / `requestAnimationFrame` timing workarounds needed.

---

## Phase 4 ‚Äî Inline editing integration

Wire up the inline text editing system (session 137) to work in the unified preview:

1. When a widget is selected, activate `contenteditable` on its `data-config-key` elements.
2. When deselected, deactivate and flush pending saves.
3. The inspector reflects inline changes via the existing `inline-config-updated` event.
