## Why this, why now

Session 330 proved the mechanism on the safe cases (always-full-width widgets, where container width ≈ viewport width, so the change was effectively behavior-preserving). The remaining four widgets are **column-capable** — they can land full-width *or* in a narrow layout column. Today their viewport `@media` queries never collapse them when they're squeezed into a narrow column on a wide screen (the `.layout-column { min-width: 0 }` blowout class). Migrating them to `@container np-widget` makes them collapse off their *own* width, which composes correctly with the host's column-collapse. This is the mobile-appearance improvement the 330 groundwork enabled; it is deliberately a **behavior change**, gated behind a per-widget visual check rather than a byte-diff.

`@layer` cascade isolation — the other 330 deferral — is **not** this session; it is session 332 (own session, coordinated all-host rollout, highest blast radius).

## Out of scope

- **`@layer` cascade isolation** — session 332.
- **Restyling the widgets** — this is a `@media`→`@container` mechanism swap only. Changing how a widget *looks* (beyond the collapse-threshold shift the swap inherently causes) belongs to the broader mobile-appearance design pass, not here.
- **`EventMiniCalendar` internal margin-collapse** (a `logbug` capture) — internal spacing as the widget narrows; not a `@media`→`@container` boundary item. Touch only if the migration crosses it and the owner greenlights.
- **New spacing/type token vocabulary** — de-scoped per 330; nothing here needs it.
- **Editable / runtime breakpoints** — killed per spec; `$bp-*` stay build-time constants usable inside `@container` conditions.
