# 217. Widget Primitive — Phase 4b (EventsListing Retrofit)

Retrofit EventsListing onto the contract-bound data path. Unlike session 216's BlogListing work — which was audit + test-lock + cleanup on an already-substantially-landed retrofit — this session lands the retrofit end-to-end: declare the contract, extend the Event projector to cover the fields the template composes today, rewrite the template to read from `$widgetData['items']`, delete the now-dead `PageContext::upcomingEvents()` helper, and add the Phase 4 test-lock pair. Second widget in the Phase 4 campaign.

---

## Design decisions (resolved before starting)

- **Contract shape.** `EventsListingDefinition::dataContract()` returns `SOURCE_SYSTEM_MODEL`, `model: 'event'`, and the field whitelist:
  `['title', 'slug', 'url', 'date', 'date_iso', 'ends_at', 'location', 'is_free', 'image']`.
  Rationale: mirrors BlogListing's shape (title/slug/url/date/date_iso/image) and adds the three event-specific fields the template uses (`ends_at` as a human-formatted end time, `location` as a composed string, `is_free` as a boolean). `price_badge` stays out of the contract — it is presentational HTML and belongs in the template, not the projector.
- **Filters.** The contract passes `filters: ['date_range' => ['from' => 'now']]` to get "upcoming" semantics via the existing `ContractResolver::resolveEvent()` filter handler. Default `order_by: 'starts_at asc'` already matches the template's `sort_default: 'soonest'`. No new filter types added.
- **Projector extensions.** `SystemModelProjector::projectEvent()` is extended to produce the derived fields the template currently composes:
  - `url` — `$event->landingPage ? url('/' . $event->landingPage->slug) : url('/' . config('site.events_prefix', 'events'))`. Requires `landingPage` eager loading on the resolver's query.
  - `date` — `$event->starts_at?->format('D, F j, Y \a\t g:i A') ?? ''` (the human-formatted pattern the current template uses).
  - `date_iso` — `$event->starts_at?->toIso8601String() ?? ''`. **Renames today's `starts_at` ISO field** — the existing key `starts_at` is removed from the projector's map to avoid two keys carrying the same value. Any callers depending on `starts_at` (confirmed via grep before the rename) must be reviewed; `ThisWeeksEventsDashboardRenderTest` is the known consumer.
  - `ends_at` — `$event->ends_at?->format('g:i A') ?? ''` (human-formatted; replaces today's ISO `ends_at` in the projector map for the same two-keys-same-value reason). ISO-format `ends_at` is not currently consumed anywhere; grep confirms.
  - `location` — composed from `address_line_1`, `city`, `state` (non-empty parts joined by `', '`). Consistency with BlogListing's `excerpt`-from-`meta_description` pattern: projector does the composition, contract whitelists the derived field. Raw `address_line_1` / `city` / `state` stay in the projector map so ThisWeeksEvents' existing contract continues to resolve unchanged; the map grows, it does not shrink.
  - `is_free` — `(bool) $event->is_free`. A boolean in the DTO.
  - `image` — `$event->getFirstMediaUrl('event_thumbnail', 'webp') ?: $event->getFirstMediaUrl('event_thumbnail')`. Mirrors BlogListing's `image` derivation.
- **`starts_at` / `ends_at` key migration.** The projector's current `starts_at` and `ends_at` both emit ISO strings. That was appropriate when `ThisWeeksEventsDashboardRenderTest` was the only consumer. EventsListing needs human-formatted `date` and `ends_at`. To avoid duplicated keys, `starts_at` is removed from the projector map (replaced by `date_iso`), and `ends_at` becomes the human-formatted string. `ThisWeeksEvents` is the only other consumer — its contract is updated if it depends on the ISO `starts_at` / `ends_at` keys (grep during the session determines the actual change set). If ThisWeeksEvents' contract declares `starts_at`, it becomes `date_iso`; if it declares `ends_at`, the semantics change and the widget's template must be inspected.
- **Resolver eager loads.** `ContractResolver::resolveEvent()` adds `landingPage` to the `with()` call: `Event::published()->with(['media', 'landingPage'])`. This is needed for the projector's `url` composition without triggering N+1 on the per-event landing page lookup.
- **Template rewrite.** `app/Widgets/EventsListing/template.blade.php` is rewritten to read from `$widgetData['items']` — mirroring BlogListing's template. The imperative `$events = $pageContext->upcomingEvents()` block (lines 2–42 today) is replaced by `$items = $widgetData['items'] ?? [];`. The token-replacement card loop below stays structurally the same, but the `price_badge` composition is moved there (since the projector no longer emits `price_badge` HTML — it emits `is_free` bool, and the template composes `<span class="content-card__badge">Free</span>` conditionally).
- **Delete `PageContext::upcomingEvents()`.** Grep confirms EventsListing is the only caller. The method is removed. The `eventsCache` private property it populated becomes unused and is removed too. `resources/docs/widget-types.md`'s context-methods table entry for `upcomingEvents()` is removed, and the closing note that mentions "`$pageContext->upcomingEvents()`" is updated to remove that reference.
- **Scope excludes search/pagination/sort server-side.** The Alpine-driven client-side search/pagination/sort continues to work exactly as today. The contract does not declare a `limit` filter because EventsListing renders all upcoming events and pages client-side. Same posture as BlogListing.
- **No new schema.** No migration, no `docs/schema/` updates.
- **Front-end builds.** The template is a server-rendered Blade file — not bundled into `build:public` or `npm run build`. `styles.scss` and `script.js` are untouched by this session, so no bundle rebuild is required. If the session surfaces a need to touch either file, the appropriate build runs per the base prompt's decision rule.

1. Add `dataContract(array $config): ?DataContract` to `app/Widgets/EventsListing/EventsListingDefinition.php` with the resolved field whitelist and the `date_range: ['from' => 'now']` filter. Import `App\WidgetPrimitive\DataContract`.
2. Extend `SystemModelProjector::projectEvent()` with the derived fields listed in "Design decisions". Maintain the raw `address_line_1`, `city`, `state`, `meeting_label` keys in the map — the projector's map is the superset; per-widget contract declarations filter.
3. Update `ContractResolver::resolveEvent()`'s `->with('media')` to `->with(['media', 'landingPage'])`.
4. Review `ThisWeeksEvents` (dashboard widget): if its contract fields reference `starts_at` / `ends_at`, update to `date_iso` (or add those fields explicitly per the widget's actual template usage). Run `ThisWeeksEventsDashboardRenderTest` after the change — any failure is a direct signal about what key-migration work is still outstanding.

## Phase 2 — Template rewrite

1. Replace lines 1–42 of `app/Widgets/EventsListing/template.blade.php` with the contract-bound shape:
   - `$items = $widgetData['items'] ?? [];`
   - Remove `$eventsPrefix` (no longer needed — `url` comes pre-composed in the DTO).
   - Remove the `$events->map(...)` imperative serialization.
2. Update the token-replacement loop to compose `price_badge` in the template based on `$item['is_free']` — this is the one presentational composition that does not belong in the projector. Pattern: after the per-key `str_replace` loop, if `$item['is_free']` is true, replace `{{item.price_badge}}` with `<span class="content-card__badge">Free</span>`; otherwise replace with empty string. The current template already handles `{{item.price_badge}}` as a special case (lines 52–53); keep the structure, drive it from `is_free`.
3. Verify the JSON `items` payload embedded in `<script x-ref="listingData">` now contains exactly the contract-declared keys. Alpine's client-side search/sort code in `script.js` reads `title`, `date` (for search substring), and `date_iso` (for ordering) — these are all in the new contract.

## Phase 4 — Test-lock

Create `tests/Feature/EventsListingContractRetrofitTest.php` mirroring `tests/Feature/BlogListingContractRetrofitTest.php`. Two tests:

### Test A — Fail-closed field whitelist

Create an `Event` with rich attributes including fields the projector exposes but the contract does not declare (`address_line_1`, `city`, `state`, `meeting_label` — they stay in the projector for ThisWeeksEvents but EventsListing's contract should not surface them). Mount an EventsListing widget with a minimal `content_template` referencing whitelisted tokens (`{{item.title}}`, `{{item.location}}`, `{{item.is_free}}`) and non-whitelisted tokens (`{{item.address_line_1}}`, `{{item.meeting_label}}`, `{{item.id}}`). Assert:

- The HTML contains the event's title and composed `location` string.
- The HTML has empty paragraphs where non-whitelisted tokens were referenced (`<p class="addr"></p>`, `<p class="meeting"></p>`, `<p class="internal"></p>`).
- The HTML does not contain the `meeting_label` sentinel value (non-whitelisted, non-derived).
- The HTML does not contain the event's `id` (UUID, high-entropy).
- Decode the embedded `listingData` JSON and assert `items[0]` keys match exactly the contract's nine declared fields. This is the load-bearing invariant.

### Test B — Resolver is the only path

Create 3 published upcoming events, 1 draft upcoming event, and 1 published past event. Mount an EventsListing widget. Enable query logging; render. Assert:

- The HTML contains exactly 3 `<article class="card">` entries (draft and past events excluded).
- Exactly one `select` against `events` with a `starts_at` filter (the `date_range: from now` clause) and a `status = 'published'` filter.
- Exactly one eager load each for `media` and `landing_page` (or `pages` via the `landingPage` relation — match whichever table-name shape the executed query emits).
- No N+1 lookups per event (no per-event `media` selects, no per-event `pages` selects for landing pages).

Both tests go in the fast suite (under 5s each). Pest syntax + `RefreshDatabase`, factory-based setup.
