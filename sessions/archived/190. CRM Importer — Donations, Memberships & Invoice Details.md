# 190. CRM Importer — Donations, Memberships & Invoice Details

Extend the importer to the three remaining WCG export sheets: Donations, Members, and Invoice Details. Each gets its own wizard page (sibling of `ImportEventsPage`). Contacts must pre-exist by default, with an opt-in toggle to auto-create minimal contacts for historical data. Transaction deduplication uses the `(import_source_id, external_id)` pattern established in session 189, now supplemented by a dedicated `invoice_number` field on transactions.

---

## Design decisions (resolved before starting)

- **Three new wizard pages.** `ImportDonationsPage`, `ImportMembershipsPage`, `ImportInvoiceDetailsPage` — each is a duplicate-then-diverge sibling of `ImportEventsPage`. Session 191 extracts a shared base once all five importers exist.
- **Contact resolution toggle.** Every financial import has an Upload-step radio: **"Error and skip the row"** (default, same as events) / **"Auto-create a minimal contact record"**. Auto-create writes a bare-bones Contact (name + email from the CSV row, `source='import'`, `import_session_id` set) so the contact enters the review queue. The toggle is per-import-session, not per-source.
- **Invoice # is a dedicated field.** Add `invoice_number` (string, nullable) to `transactions`. Distinct from `external_id` (technical record ID for source-system dedupe). A real-world transaction can have both — e.g. Wild Apricot internal ID as `external_id` and a human-readable invoice number. The donations "Number" column and invoice details "Invoice #" both map here.
- **Membership tier auto-create.** Resolve by name (case-insensitive); create a new `MembershipTier` if no match. Same pattern as Organization auto-create in the contacts importer.
- **Invoice Details are line-item level.** Multiple CSV rows share the same `Invoice #`. The importer collapses them to one Transaction per invoice: total amount = sum of line-item amounts. Item descriptions are stored as a JSON array in a `line_items` column on the Transaction (new, jsonb, nullable). If a Transaction already exists for this invoice (created by events import or a prior run), enrich it with fill-blanks-only semantics.
- **Donations create both a Donation record and a Transaction.** The Donation model today is Stripe-only; extend it to accept imported donations. `stripe_*` fields stay null. `status` gains a `completed` value for historical imports where payment is already settled. Each Donation gets a linked Transaction (`subject_type = Donation`).
- **Funds and Campaigns are out of scope.** Not present in the WCG data. Both need dedicated data-model sessions before they're import-ready. Stubs added to `session-outlines.md` under post-Beta 1.
- **Duplicate-then-diverge.** All three pages are siblings, not subclasses. Session 191's polish pass extracts commonality across all five importers.

## Phase 2 — Field registries

- `DonationFieldRegistry` — amount, currency, donation_date (→ `created_at` or a new `donated_at`), type, status, external_id, invoice_number, comment/notes.
- `MembershipFieldRegistry` — membership_level (→ tier resolution), membership_status (→ `status`), member_since (→ `starts_on`), renewal_due (→ `expires_on`), amount_paid, balance, total_donated, notes, external_id.

- Wild Apricot heuristic guesses for each CSV shape (same `guessDestination` pattern as events).

## Phase 3 — `ImportDonationsPage` wizard

Three-step wizard (Upload → Map Columns → Preview & Confirm), sibling of `ImportEventsPage`. Key differences:

- **Contact toggle**: radio at Upload step for auto-create vs error-on-missing.
- **Mapping**: donation-specific grouped options. "Number" column → `donation:invoice_number`. "Donation date" → `donation:donated_at`. "Amount" → `donation:amount`. "Payment state" / "Payment type" / "Online/Offline" → transaction fields.
- **Match key**: contact match only (no "match donations by" — each row is a new donation, deduped via `(import_source_id, external_id)` on the Transaction side).

---

## Phase 4 — `ImportDonationsProgressPage` row processor

Per-row pipeline:
1. **Resolve Contact** — same as events (email or external_id lookup, case-insensitive email). If missing and toggle = auto-create, create a minimal Contact. If missing and toggle = error, skip row.
2. **Create Donation** — `Donation::create(...)` with `type='one_off'`, `status='completed'` (or mapped from CSV), `amount`, `import_session_id`, `import_source_id`.
3. **Create/upsert Transaction** — linked to the Donation via polymorphic `subject`. Deduped on `(import_source_id, external_id)` using the "Number" column. `invoice_number` populated from the same column. Fill-blanks-only on matched existing.
4. **Timeline note** on the contact: "Donation of $X imported from {source} (session: {label})".

Dry-run report extends with Donations preview (would create N donations, M transactions).

---

## Phase 5 — `ImportMembershipsPage` wizard + `ImportMembershipsProgressPage` processor

Same three-step wizard pattern. Key differences:

- **Tier resolution**: "Membership level" column maps to `membership:tier`. Resolved by name (case-insensitive), auto-created if missing.
- **Status mapping**: WA's "Lapsed" / "Active" / "Pending - Renewal" → normalise to `active`, `expired`, `pending`, `cancelled`.
- **Date parsing**: "Member since" → `starts_on`, "Renewal due" → `expires_on`. Same `parseDate` helper from events.
- **Balance / Total donated**: these are contact-level aggregates in Wild Apricot, not membership fields. Map to contact custom fields (or skip — they can be recomputed from transaction data). Decision during implementation.

Per-row pipeline:
1. **Resolve Contact** — same toggle.
2. **Resolve Tier** — by name, auto-create if missing.
3. **Create Membership** — `Membership::create(...)` with `contact_id`, `tier_id`, `status`, `starts_on`, `expires_on`, `amount_paid`, `import_session_id`.
4. **Timeline note** on the contact: "Membership ({tier name}) imported from {source} (session: {label})".

---

## Phase 6 — `ImportInvoiceDetailsPage` wizard + `ImportInvoiceDetailsProgressPage` processor

Same three-step wizard pattern. This is the richest transaction source — each row is a line item on an invoice.

Per-row pipeline:
1. **Resolve Contact** — same toggle.
2. **Upsert Transaction** on `(import_source_id, invoice_number)`. If the Transaction already exists (created by events or donations import), enrich with fill-blanks-only. If new, create.
3. **Accumulate line items.** Multiple rows with the same Invoice # → append to the Transaction's `line_items` JSON array. Each entry: `{item, quantity, price, amount}`. Total transaction amount = sum of line-item amounts (set on creation, left alone on enrichment if already populated).
4. **Invoice metadata**: invoice_date → `occurred_at`, status → `status`, payment_date, payment_type, origin, origin_details, currency.
