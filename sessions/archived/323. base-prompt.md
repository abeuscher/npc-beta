## Process rules for every session

Base-prompt process rules apply in full (see `sessions/template-base-prompt.md`). Session-specific notes:

- **One value, one control, several bind targets.** The border value model and the control are universal; only the mount point differs. v1 mounts it in the appearance panel (all widgets) and ships the concrete default. The in-widget `border` field-type registration (the `alignment`-field precedent) is a *later* bind target — design the primitive so it's reusable, but don't build the field-type path in v1 unless it falls out for free.
- **Default off = no regression.** The border default is all-sides-off, a concrete no-op. Shipping the control to every widget changes nothing visually until an operator turns a side on. Confirm `box-sizing: border-box` is in force so an enabled border doesn't shift layout.
- **Parity is the real cost.** The bulk of the work is emitting borders identically across `AppearanceStyleComposer`, `ChromeRenderer::buildInlineStyle`, and the preview renderer — not the Vue control. Budget accordingly.
- **Front-end builds:** appearance UI is Vite-compiled page-builder Vue → `npm run build`. The border emitter is server-side PHP (no build). Per the build decision rule, only run a build if `resources/js/**` or `resources/scss/**` changed.
- **The tool is the priority, not single-session completion.** If v1 (control + universal outer-border + Text-widget verification) doesn't all land in one session, that's fine — land a coherent slice and continue next session. Do not compress scope to force a one-session close.
- **Verify objective outcomes yourself** (computed `border-*` via a render test or Playwright); **pull the owner in for the visual judgment** (does the control feel like the definitive tool). `feedback_questions_are_not_instructions` and `feedback_pm_level_tone` apply.
