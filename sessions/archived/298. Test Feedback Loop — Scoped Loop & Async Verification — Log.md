## What was built

### Phase 1 — Scoped inner loop (`->group('design')`)

- **Open Q1 resolved to form (b)** — a real `->group('design')` Pest tag across a one-time reviewed cluster, plus an explicit-list integrity guard. Rationale (carried from the prompt, confirmed in code): the cluster has no uniform naming (`TypographyCompiler/Resolver`, `AssetBuildSession123`, `BuildServerSettings125`, `WidgetJsLibs138`, `DesignSystemButtons`, `GradientComposer`, `Appearance*`, `PageBlockRendererLayout`, `AssetBundleDriftGuard`, …), so any naming/namespace heuristic would over/under-match forever; an explicit reviewed marker is the only rot-proof option and is idiomatic + greppable.
- **17 files tagged** `->group('design')`: 15 via `uses(TestCase::class, RefreshDatabase::class)->group('design')`; 2 with no `uses()` (`TypographyCompilerTest`, `GradientComposerTest`) via a standalone `uses()->group('design')`. List: TypographyCompiler, TypographyResolver, AppearanceStyleComposer (Unit); TypographyMigration, ThemeTypographyController, ThemeColorRelocationByteFaithful, AssetBuildSession123, AssetBundleDriftGuard, BuildServerSettingsSession125, WidgetJsLibsSession138, DesignSystemButtons, GradientComposer, PageBlockRendererLayout, AppearanceBackgroundRender, AppearanceConfigRename, AppearanceImageUpload, AppearanceLayerRegression (Feature).
- **Guard test** `tests/Feature/DesignGroupIntegrityTest.php` — scans `tests/` for the `design` tag and asserts the discovered set equals the explicit 17-file list; fails both rot directions (untracked tag added / listed file lost its tag). Self-excludes (its own message text mentions the tag).
- **`dev` wrapper:** added `./dev test` (`php artisan test --exclude-group=slow`) and `./dev test:design` (`php artisan test --group=design`), plus usage lines.
- **Documented** in `CLAUDE.md` (new "Testing — scoped inner loop" section).
- **Verified (objective, self):** `--group=design` = **213 tests green in ~64s** vs ~11 min full fast suite (~10× faster). Guard green.

### Phase 2 — Hardened CI + additive close-gate shift

- **`.github/workflows/tests.yml` hardened (not duplicated):** single `test` job split into **`fast` (php artisan test --exclude-group=slow, first signal) → `slow` (--group=slow, `needs: fast`) → aggregate `tests` job named `Tests` (`needs: [fast, slow]`)**. Triggers unchanged (`push: ["**"]` + PR→`main`). No image build/publish, no `VERSION` read, no deploy steps — cannot race `deploy.yml`/`deploy-demo.yml` (validated by YAML parse + grep).
- **Additive close-gate wording** added to `sessions/template-base-prompt.md` and (shorter, consistent) `CLAUDE.md`. Verbatim addition:
  > - **CI is the authoritative full-suite signal (additive, staged — does not replace the line above).** Every push runs the full suite (fast → slow) asynchronously via `.github/workflows/tests.yml`; the aggregate **`Tests`** status check is green only when both the fast and slow suites pass. Once `Tests` is observed green on the pushed branch, that — not a local full run — is the authoritative pass signal, and the human need not block-wait on a local full run. This is **additive**: the local "run the fast test suite … fix failures before proceeding" step above **remains the documented fallback and is not retired**. Retiring block-and-wait is a later, explicit step, gated on (a) the hardened CI proving out over a few sessions and (b) confirming where CI status surfaces in the user's working flow. **Main-green depends on a repo-admin action that cannot be committed:** the async model only guarantees `main` is green if GitHub branch protection on `main` requires the `Tests` status check — a repo admin must enable this; the session-298 log surfaces it as a blocking handoff.
  The original block-and-wait sentence is retained verbatim immediately above it. User approved the wording as-is.
- **Branch-protection handoff (Open Q3):** surfaced explicitly as a required, non-committable repo-admin action. **User actioned it during the session** — GitHub branch protection on `main` requiring the `Tests` check is now enabled.
