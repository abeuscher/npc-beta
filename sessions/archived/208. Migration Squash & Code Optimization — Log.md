### Phase 3 — Picks applied

**Pre-flight verifications (all five data-phase migrations clean):**
- Typography: `TypographyResolver::load()` falls back to `defaults()` when the site_setting row is absent — no seeder addition required.
- `use_current_page_header`: `WidgetDefinition::defaultAppearance()` emits the key concretely on every new widget.
- Polymorphic widget owner: `templates.definition` column is unreferenced in `app/`; drop is safe.
- Polymorphic staged-update subject: both creation sites (`ImportProgressPage`, `InteractsWithImportProgress`) write `subject_type` + `subject_id` directly.
- Listing token rewrite: EventsListing, BlogListing, and Carousel definitions all emit `{{item.<key>}}` form in their defaults.

**B1 — Root `tailwind.config.js` replaced with inert stub.** Original config carried customizations (primary color, heading/body fonts, `@tailwindcss/forms` plugin) aimed at a Tailwind-based public site that was retired long ago. Public SCSS has no `@tailwind` directives; Filament's theme.css pins its own config via `@config` directive pointing at `tailwind.config.filament.js`. Root config is now a three-line `content: []` stub with a one-line comment explaining why it exists. `npm run build` clean; warning about missing Tailwind content config is silenced.

**B2 — `PageBuilderApiController::buildTree` eager-loads `owner`.** The polymorphic `owner` morph on `page_widgets` / `page_layouts` is accessed by `WidgetRenderer::render` and `AppearanceStyleComposer::compose`, which are called per-widget during preview rendering in `buildTree`. Added `->with(['widgetType', 'owner'])` on both the root-widgets query (line 522) and the layouts-with-children query (line 530). Measured delta on a 14-widget test page: **40 queries → 28 queries (12 saved)**. New query-count test locks this in at `< 32` to catch future regressions.

**B3 — `ImportSessionActions::approve()` / `rollback()` preload staged subjects.** Previously each staged-update loop called `resolveStagedSubject($update)` per row, firing a `find` query against the subject model per iteration. Replaced with a `preloadStagedSubjects(Collection $staged): array` helper that groups staged rows by `subject_type`, fires one `whereIn` per type, and returns a keyed map `[type][id] => Model`. Foreach loop now does a hash lookup instead of a DB call. Measured delta on 25 staged updates: **128 queries → under 110 queries (~18 saved)**. `resolveStagedSubject` method deleted; was only used within this file. Unused `Model` import dropped. New query-count test locks the improvement in.

**C1 — `events.registrations` route registered on `events.md`.** Single line added to the `routes:` frontmatter array. The Event Registrations sub-page (`EventResource/Pages/ViewRegistrations.php`, route `filament.admin.resources.events.registrations`) now resolves to the Events help doc via the in-admin help affordance.

**D1 — Playwright importer teardown.** Instead of 8 table-specific delete helpers, went with a single artisan command + one TS helper that covers every importer spec uniformly:
- New: `app/Console/Commands/ImporterCleanupSessionCommand.php` — signature `importer:cleanup-session {session-id}`, calls `ImportSessionActions::delete()` which already carries the full cascade logic for every model type.
- New: `cleanupImportSession(sessionId)` in `tests/e2e/helpers/db.ts` — execs the artisan command.
- New: `cleanupAllImportSessionsOfType(modelType)` — queries all sessions of a given model_type and cleans each, covering update-strategy specs that create two sessions per run.
- New: `deleteContactsByEmails(emails[])` — for the `donations-error-report` spec's pre-inserted ambiguous contacts, which don't belong to any import session.
- `afterAll` hooks added to all 14 non-clean importer specs. `contacts-pii-rejection` left as-is (rejection never persists).

**Deferred picks and reasons:**
- **B4** (buildTree returns both `widgets` and `items` arrays): deferred — requires confirming no Vue code reads `response.data.widgets` independently from `response.data.items`. Small potential win, needs a Vue audit the squash session wasn't going to make time for.
- **B5** (`WidgetResource` ships full `config_schema` per instance): deferred — potentially a meaningful payload reduction but requires Vue code audit and possibly a new "load schema on-demand" endpoint. Focused perf session rather than squash-ride-along.
- **B6** (batch `count()` queries in approve/rollback/delete descriptions): deferred — low impact, moderate effort.
- **B7** (`PageBuilderApiController::fresh()` redundancy with immediate `buildTree()`): deferred — redundant work rather than a bug, moderate refactor for small gain.
