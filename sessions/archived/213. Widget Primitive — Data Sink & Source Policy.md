# 213. Widget Primitive — Data Sink & Source Policy

Establish the write-side paradigm complementary to the read-side data contract: a typed `Source` identity primitive, a **model-local** `HasSourcePolicy` trait with a `ACCEPTED_SOURCES` constant convention, a fail-closed `DataSink` service singleton, and an `accepted_sources` column on `collections` with explicit source checkboxes at creation time. Migrate the two existing demo data generators (`DemoDataService` and `DebugGenerator`) to route their writes through `DataSink` with `SOURCE_DEMO`. This ends demo data's existing status as "exception to the paradigm" and folds it into the paradigm as a first-class declared source, with fail-closed gating at every target.

Scope is deliberately narrow: **primitive + declarations on three system models + collection column/form + demo-generator migration + tests.** The importer, Google Docs ingestion, LLM-synthesis paths, Stripe webhook tagging, and migration of non-demo synthetic data paths are **out of scope** — they use the primitive when they land, they do not drive its shape this session.

---

## Design decisions (resolved before starting — do not relitigate)

Settled across the Phase 1–2 track (sessions 209–211) and the scoping for this session. Several were explicitly agreed in session 211's close — see `feedback_security_over_usability` memory for the general principle these operationalise.

- **Source identity is a class of well-known constants.** `App\WidgetPrimitive\Source` exposes public-const strings: `Source::HUMAN`, `Source::DEMO`, `Source::IMPORT`, `Source::GOOGLE_DOCS`, `Source::LLM_SYNTHESIS`, `Source::STRIPE_WEBHOOK`. Adding a source is a grep-visible edit to one file, reviewable as a boundary change.
- **Source policy is model-local.** `HasSourcePolicy` trait exposes `acceptsSource(string $source): bool` and expects the using class to declare `public const ACCEPTED_SOURCES = [...]`. **No central registry.** Reasons: adding a new model without declaring a policy fails closed (empty constant → every source rejected); additive and destructive schema changes keep their policy co-located with the model; central registries bifurcate policy from the thing being policed. If a derived "show me every policy" view is ever needed, it's a helper that iterates the model classes — not a source of truth.
- **`SOURCE_HUMAN` is a universal pass.** All models accept `Source::HUMAN` implicitly. Filament forms continue writing through Eloquent mass-assignment unchanged — their writes are `Source::HUMAN` by default. `HasSourcePolicy::acceptsSource(Source::HUMAN)` returns `true` regardless of `ACCEPTED_SOURCES`. The universal-pass rule is a trait-level default, not a per-model repetition.
- **`DataSink` is a singleton service.** `DataSink::write(string $modelClass, string $source, array $payload): Model`. Fail-closed: unknown source → throw; target's `acceptsSource()` returns false → throw. Mirrors `ContractResolver` as the single hardened boundary — one auditable thing is better than fifty remembered callsites.
- **Scope is narrower than full write contracts.** The mutation shape (which fields, what values, validation, authorization-by-field) still goes through Filament's mass-assignment + policy story. `DataSink` gates **who** is writing, not **how** or **which fields**. Per-field write authorization is the 2.0 mutation-contract story and stays deferred.
- **Collection source policy is fail-closed at creation.** New collections default to `accepted_sources = [Source::HUMAN]`. The Create Collection Filament form surfaces this explicitly: a checkbox group labelled "This collection accepts data from:" with the full set of `Source::*` constants as options, only `Human` checked by default. Admin widens by explicit consent. Existing collections backfill with `[Source::HUMAN]`.
- **`WidgetDefinition::acceptedSources(): array` default is `[Source::HUMAN]`.** Pattern-parallel to `allowedSlots()` from 211. Override per widget when the widget wants to consume synthetic sources (demo data for preview, LLM synthesis, etc.).
- **`ContentType` gains an `accepts` field** in its constructor, declaring which sources can write items into a widget-declared content type. Defaults to `[Source::HUMAN]` when omitted.
- **System models touched this session: `Page`, `Contact`, `Donation`.** These are the widest-blast-radius models where demo data currently writes. Other system models (`Event`, `Transaction`, `Organization`, `Membership`, etc.) get `HasSourcePolicy` applied later — not this session, to keep scope honest.
- **No audit log of source writes yet.** `DataSink` throws on rejection; the throw is observable in logs but no dedicated source-write log is written. Deferred.
- **No per-field source gating.** As above: `DataSink` is a write-boundary gate, not a field-authorization service.
- **Security over usability is the deciding principle** where the two genuinely conflict. Captured in memory; this session operationalises it.

## Phase 2 — `HasSourcePolicy` trait

`app/WidgetPrimitive/HasSourcePolicy.php`:

- Trait.
- Provides `acceptsSource(string $source): bool`:
  - Returns `true` when `$source === Source::HUMAN` (universal pass).
  - Returns `true` when `$source` is in the using class's `ACCEPTED_SOURCES` constant.
  - Returns `false` otherwise (including when the constant is absent — fail-closed on missing declaration).
- Uses `defined(static::class . '::ACCEPTED_SOURCES')` guard so a trait-user that forgets the constant still gets fail-closed (only `Source::HUMAN` passes).

Tests at `tests/Unit/WidgetPrimitive/HasSourcePolicyTest.php`:

- Class with no `ACCEPTED_SOURCES` accepts `HUMAN` only.
- Class with `ACCEPTED_SOURCES = [Source::DEMO]` accepts `HUMAN` and `DEMO`, rejects others.
- Unknown source rejected regardless of declaration.

Apply trait to:

- `App\Models\Page` — `ACCEPTED_SOURCES = [Source::DEMO, Source::GOOGLE_DOCS, Source::LLM_SYNTHESIS]`. (Pages accept blog-post ingestion and LLM authoring in addition to human and demo.)
- `App\Models\Contact` — `ACCEPTED_SOURCES = [Source::IMPORT, Source::DEMO]`. (Contacts come primarily from imports; demo gen for dev fixtures.)
- `App\Models\Donation` — `ACCEPTED_SOURCES = [Source::IMPORT, Source::STRIPE_WEBHOOK]`. **No `Source::DEMO`** — this is a finance model and demo data must never write real donation rows. This is the sharpest security win of the session: grep `Donation.*ACCEPTED_SOURCES` surfaces the policy immediately.

## Phase 3 — `DataSink` service

`app/WidgetPrimitive/DataSink.php`:

- Final class.
- `write(string $modelClass, string $source, array $payload): Model`:
  - Guard: `Source::isKnown($source)` — throw `InvalidArgumentException` with a clear message naming the unknown source.
  - Guard: model class has `HasSourcePolicy` trait. If absent, throw `LogicException` — every write target must declare a policy.
  - Guard: `$modelClass::acceptsSource($source)` — if false, throw `App\WidgetPrimitive\Exceptions\SourceRejectedException` (new) with message naming the target class and rejected source.
  - On pass: `$modelClass::create($payload)` and return the model.
- Bind as a singleton in `WidgetServiceProvider::register()` alongside `WidgetRegistry`, `SlotRegistry`, etc.

Tests at `tests/Unit/WidgetPrimitive/DataSinkTest.php`:

- `write` creates a `Contact` when source is `Source::IMPORT` (accepted).
- `write` creates a `Contact` when source is `Source::HUMAN` (universal).
- `write` throws `SourceRejectedException` when source is `Source::GOOGLE_DOCS` on `Contact` (not in Contact's `ACCEPTED_SOURCES`).
- `write` throws `InvalidArgumentException` when source is the literal string `'nonsense'`.
- `write` throws `LogicException` when the target class lacks `HasSourcePolicy` — use a temporary fake model class for this test, or pick a still-unpolicied system model and assert.
- `write` never creates `Donation` with source `Source::DEMO` — the critical security assertion.

## Phase 4 — `collections.accepted_sources` column + model + form

Migration: new file `database/migrations/YYYY_MM_DD_HHMMSS_add_accepted_sources_to_collections.php`:

- Adds `accepted_sources` JSONB column to `collections`, not-nullable, default `'["human"]'`.
- Uses `json` or `jsonb` consistent with existing collection columns (`fields` is JSONB — match that).
- Down: drop the column.

Run the migration via `docker compose exec app php artisan migrate` per the process rules.

Update `app/Models/Collection.php`:

- Cast `accepted_sources` to `array`.
- Add to `$fillable` if the model uses an explicit fillable list.
- Apply `HasSourcePolicy` trait. **`ACCEPTED_SOURCES` constant is not used for Collection** — Collection's policy is per-row, not per-class. Override `acceptsSource()` on the model to read from `$this->accepted_sources` instead of the constant.
  - Edge case: `Source::HUMAN` still passes universally — the trait's default handles this, but the override needs to preserve that behaviour.
- Update `docs/schema/collections.md` with the new column per the process rule.

Update the Collection create form (locate via `CollectionResource.php` and its Create page):

- Add a `CheckboxList` or equivalent Filament primitive labelled "This collection accepts data from:".
- Options drawn from `Source::KNOWN` minus `Source::HUMAN` (human is always implicit; don't show a user-unchecked-able option).
- Default: empty (no sources beyond human). Visual hint: "Human authors are always accepted. Check additional sources as needed."
- On save, the selected sources plus `Source::HUMAN` are stored in `accepted_sources`.

**Items are not gated individually** — `CollectionItem` writes inherit their parent collection's policy. `DataSink::write(CollectionItem::class, $source, $payload)` must check the parent collection's `acceptsSource()`. Implement this by either (a) overriding `CollectionItem::acceptsSource()` to delegate to its collection, or (b) special-casing `CollectionItem` in `DataSink::write`. Prefer (a) for locality.

Tests:

- Collection form renders the checkbox list with `Source::HUMAN` not visible.
- New Collection defaults `accepted_sources` to `["human"]`.
- `DataSink::write(CollectionItem::class, Source::DEMO, ...)` rejects when the parent collection doesn't accept `Source::DEMO`; accepts when it does.

- `ContentType` constructed with `accepts: [Source::HUMAN, Source::DEMO]` carries that value.

## Phase 6 — Demo generator migration

Migrate the two existing demo paths to route writes through `DataSink` with `Source::DEMO`:

1. **`app/Services/DemoDataService.php`** — reads and returns demo data for widget previews. Review its write surface: if it writes to DB at all (vs. producing in-memory arrays), those writes route through `DataSink::write(..., Source::DEMO, ...)`. If it's read-only, document that in the log — nothing to migrate.

2. **`DebugGenerator`** — locate via `grep -rn 'DebugGenerator' app/`. Session 079 built donation/product/purchase generation; sessions 112 and 064 are also relevant predecessors. Migrate every `Model::create(...)` / factory call that writes to a `HasSourcePolicy`-participating table to route through `DataSink::write(..., Source::DEMO, ...)`. Factories that write to tables not yet carrying `HasSourcePolicy` (Event, Transaction, Membership, Organization, etc.) stay as direct writes — those tables' policies land in later sessions.

Any collection whose demo data lands via these paths must have `Source::DEMO` in its `accepted_sources`. The two demo collections most likely affected:

- The Carousel slides demo collection (check `DemoDataService` and any demo seeder).
- Logo garden / board members demo collections.

**Seeder decision:** update the collection seeder (or the demo data seeder that creates these collections) to write `accepted_sources = [Source::HUMAN, Source::DEMO]` for collections that are demo-intended. New collections created by the admin still default to human-only.

Tests:

- DebugGenerator writing a Contact succeeds (Contact accepts `Source::DEMO`).
- DebugGenerator writing a Donation **fails with `SourceRejectedException`** — this is the security assertion. If an existing debug flow writes donations, update it to either (a) not write donations during demo (if donations aren't actually needed for demo), or (b) route through a different mechanism that is explicitly not synthetic (e.g. a fixture file). Document the decision in the log.
- A demo collection item write succeeds when the collection has `Source::DEMO` in its accepted list; fails when it doesn't.
