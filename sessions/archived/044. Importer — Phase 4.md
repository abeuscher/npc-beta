# 044. Importer — Phase 4: Staged Updates & Queue Control

Architectural completion of the import review workflow. Three problems identified in session 043 are resolved together because they share a root cause: proposed changes to existing contacts must be staged and held for reviewer approval, the same way new contacts already are. This session also enforces a one-at-a-time import queue per content type to eliminate data collision risk, and merges the Importer and Review Queue into a single unified page.

## Part 1 — Staging table

### Migration

Create `import_staged_updates`:

```
id              — bigint, primary
import_session_id — uuid, foreign → import_sessions.id, cascade delete
contact_id      — uuid, foreign → contacts.id, cascade delete
attributes      — jsonb, nullable — proposed field value changes
tag_ids         — jsonb, nullable — tag UUIDs to sync onto the contact on approval
timestamps
```

Index on `import_session_id`. Index on `contact_id`.

Update `docs/schema.md` to include the new table.

### Model: `app/Models/ImportStagedUpdate.php`

Simple Eloquent model. `$fillable`: `import_session_id`, `contact_id`, `attributes`, `tag_ids`. Cast `attributes` and `tag_ids` as `array`. `belongsTo` relationships to `ImportSession` and `Contact`.

## Part 2 — Update processRow() to stage instead of apply

In `ImportProgressPage::processRow()`, both update paths currently call `$contact->fill($nonNull)->save()` and do not apply tags. Replace both with staging logic.

**For the External ID match path and the email match path:**

1. Create an `ImportStagedUpdate` record:
   - `import_session_id` — `$this->importSessionId`
   - `contact_id` — `$contact->id`
   - `attributes` — the `$nonNull` array that would have been applied
   - `tag_ids` — `$this->tagIds` (may be empty array)

2. Write a `Note` to the contact:
   - body: `"Match found from import in session {$this->sessionLabel} — field changes are staged and awaiting reviewer approval"`
   - `author_id`: `$this->importerUserId ?: null`
   - `occurred_at`: `now()`

3. Return `'updated'` as before (the counter is used for the progress display).

Do **not** call `$contact->fill()->save()`. Do **not** call `$contact->tags()->sync()`. The existing contact is untouched until approval.

**The new-contact creation path is unchanged** — contacts are still created with `import_session_id` and held by the global scope. Tags are still synced immediately (the contact is hidden anyway, and tags will travel with it when approved). The note written at creation is unchanged.

## Part 4 — One-at-a-time queue lock per content type

### Hard guard in `ImportContactsPage::runImport()`

Before creating the `ImportSession`, check for an existing blocking session:

```php
$blocking = ImportSession::where('model_type', 'contact')
    ->whereIn('status', ['pending', 'reviewing'])
    ->exists();
```

If true, send a `Notification::make()->danger()` and call `$this->halt()`. Do not create any records.

### Soft lock on the Importer page

In `ImporterPage.php`, add a method `getBlockedTypes(): array` that returns the model types currently locked (those with a pending or reviewing session). Pass this to the blade.

In `importer.blade.php`, when a content type is in the blocked list, render it as a disabled card (same style as the "coming soon" stubs) with an explanation beneath it: "A previous contact import is awaiting review. Approve or roll it back before starting a new one." If the viewing user has `review_imports`, append a link directly to the review queue section of the same page.

## Part 5 — Unified Importer + Review Queue page

The `ImporterPage` becomes the single entry point for both importers and reviewers.

### Permission change

`ImporterPage::canAccess()` — change to:

```php
return auth()->user()?->can('import_data')
    || auth()->user()?->can('review_imports');
```

All existing canAccess() checks on ImportContactsPage, ImportProgressPage, ImportHistoryPage remain as-is (import_data only).

### Absorb the review queue into ImporterPage

`ImporterPage` must implement `HasTable` and use `InteractsWithTable`. Add a `table()` method that mirrors the current `ImportReviewPage::table()` exactly — same columns, same preview/approve/rollback actions, same query.

**Approve action — extended:**

After setting `status = 'approved'`, apply all staged updates for this session:

```php
$staged = ImportStagedUpdate::where('import_session_id', $record->id)->get();

foreach ($staged as $update) {
    $contact = Contact::withoutGlobalScopes()->find($update->contact_id);

    if (! $contact) {
        continue;
    }

    if (! empty($update->attributes)) {
        $contact->fill($update->attributes)->save();
    }

    if (! empty($update->tag_ids)) {
        $contact->tags()->syncWithoutDetaching($update->tag_ids);
    }

    Note::create([
        'notable_type' => Contact::class,
        'notable_id'   => $contact->id,
        'author_id'    => auth()->id(),
        'body'         => "Changes applied from import session {$record->importSource?->name ?? $record->filename} — approved by " . auth()->user()->name,
        'occurred_at'  => now(),
    ]);
}

$staged->each->delete();
```

**Rollback action — extended:**

After deleting new contacts (existing behaviour), also:

```php
$staged = ImportStagedUpdate::where('import_session_id', $record->id)->get();

foreach ($staged as $update) {
    $contact = Contact::withoutGlobalScopes()->find($update->contact_id);

    if ($contact) {
        Note::create([
            'notable_type' => Contact::class,
            'notable_id'   => $contact->id,
            'author_id'    => auth()->id(),
            'body'         => "Staged changes from import session {$record->importSource?->name ?? $record->filename} were discarded during rollback.",
            'occurred_at'  => now(),
        ]);
    }
}

$staged->each->delete();
```
