## What was built

### Phase 1 — `app:reset` artisan command

- New file [app/Console/Commands/AppResetCommand.php](app/Console/Commands/AppResetCommand.php). Signature `app:reset` (no `--force`, no confirmation prompt — see "Scope reduction" below).
- Production-refusal gate: `$this->laravel->environment('production')` → `error()` + `self::FAILURE`. Absolute, no escape hatch.
- Order of operations on dev: `Artisan::call('migrate:fresh', ['--seed' => true, '--force' => true])` → wipe `storage/app/public/*` via `Storage::disk('public')` directory + file iteration → wipe `storage/media-library/temp/*` via `File::cleanDirectory()`. Summary line on success.
- Storage wipe uses the `Storage` facade for the public disk (so `Storage::fake('public')` works in tests) and `File::cleanDirectory()` for the temp path (which doesn't have a Laravel disk).

### Phase 2 — Scheduled `media-library:clean`

- Added to the existing `withSchedule()` block in [bootstrap/app.php](bootstrap/app.php#L33), alongside the 242 backup entries:
  ```
  $schedule->command('media-library:clean')->daily()->onOneServer()->withoutOverlapping();
  ```
- Registers at the `0 0 * * *` slot. Verified via `php artisan schedule:list`.
- Spatie's `media-library:clean` removes orphan `media` rows (where the polymorphic owner is gone) and orphan conversion files. Does NOT walk the public tree for files-with-no-`media`-row — that gap is exactly what `app:reset` covers for the wholesale-wipe case.

### Scope reduction (mid-session)

The original session prompt called for a confirmation prompt requiring literal `RESET`, a `--force` flag to skip it, and a Console-Kernel-level filter to hide the command from `php artisan list` on production. The user pushed back on this framing — the actual problem is just "media isn't cleaned up when we wipe and reseed," and the prod-refusal gate already provides the safety. The confirmation prompt and the auto-discovery filter were dropped as gold-plating that didn't earn their weight; the `--force` flag went with them since there was no longer a prompt to skip. Final shipped surface is the minimal version that solves the actual problem.

## Production deployment notes

- **`app:reset`:** dev-only command. Refuses to run on production via `app()->environment('production')`. No production action needed.
- **Scheduled `media-library:clean`:** the new schedule entry does NOT fire automatically until a scheduler-runner is in place — the `worker` service runs `queue:work` only, no `schedule:work` or cron-driven `schedule:run`. This is the same gap that prevents the 242 backup commands from firing automatically; listed as a Beta-1-blocking carry-forward in `sessions/tracks/fleet-manager-agent.md`. Manual invocation works today: `docker exec nonprofitcrm_app php artisan media-library:clean`.
