### Iteration /4 — Remaining framework-alignment cleanups

- **W6 #6:** `WidgetType::forPicker()` now carries `assets`, `full_width`, and `default_open`. `PageBuilderApiController::widgetTypes()` delegates to `forPicker()` — the ~25-line duplicate payload is gone.
- **W7 #3:** `PresetController::update()` uses `$model->fill($request->only([...]))->save()` with a non-null filter instead of a `foreach`/`array_key_exists` loop.
- **W7 #4:** `WidgetType::getDefaultConfig()` consults the `WidgetRegistry` first (same path as `WidgetConfigResolver::baseDefaults`), so defaults cannot drift between call sites. The schema-walking fallback remains as a safety net for handles not in the registry.
- **W7 #8:** new `App\Http\Requests\ReorderWidgetsRequest` wraps the reorder endpoint's rules plus widget/layout ownership checks. Controller action shrinks to a 4-line body. Error shape changes from `{"error": "..."}` to standard Laravel `ValidationException` (`{message, errors}`) — no client or test depended on the old shape.

### Iteration /5 — Column-preview bug fix + remaining extractions

- **Bug fix (surfaced mid-session):** `WidgetPreviewRenderer` wrapped every non-full-width widget in `<div class="site-container">`, including column children. The public `PageController` path never wraps column children (the enclosing `.layout-column` handles layout). Aligned the preview: column children skip the wrap. Regression test added to `PageBuilderApiTest` asserts a root widget's preview contains `site-container` and a column child's preview does not.
- **W6 #8:** extracted `App\Services\PageBlockRenderer` from `PageController`. The two private methods (`renderWidgetBlock`, `renderLayoutBlock`) move to the service unchanged. `PostController` was intentionally left untouched — it has a separate, simpler render path that does not share the layout-aware pipeline, and merging the two was out of scope for a cleanup session.
- **W7 #5 (partial, decided mid-session):** added `@vueuse/core` and replaced the layout-save debounce with `useDebounceFn`. The config-save debounce, per-widget blur/spinner timers, and `AbortController` map were left alone — their integration points (manual flush from `refreshPreview`, per-id Maps, abort semantics) do not map cleanly to `useDebounceFn` and the existing code is correct.

## Deferred decisions

- **W7 #5 (partial):** `useDebounceFn` applied only to the layout-save debounce. The config-save debounce, per-widget indicator timers, and `AbortController` map were deliberately not migrated because the useDebounceFn shape doesn't fit their integration constraints (synchronous flush-on-demand, per-id Maps, abort semantics). These stay as hand-rolled code. Revisit only if a specific bug surfaces.
- **W6 #8 (partial):** `PageBlockRenderer` extracted from `PageController` but `PostController` retains its own inline render path. PostController's pipeline predates the layout-aware renderer and has its own quirks (inline image processing, no layout support). Migrating it would be a feature-level change, not a cleanup.
- **Form widget consolidation:** scoped during the session at user request. Decision: keep all form widgets separate. The portal auth trio (login / signup / forgot-password) is the only candidate where consolidation would reduce duplication, but the pinning contract and conditional schema cost outweighed the saving.
