### Issue 1 (in-session absorption) — editor parity for layout backgrounds

- Reported at first manual review: the page-builder editor's column-layout did not render a configured background at all. Root cause was a pre-existing gap in the editor's data flow combined with a post-split structural mismatch:
  - `formatLayout()` in both `PageBuilderApiController` and `RecordDetailViewBuilderApiController` did not include the layout's composed inline style. Added an `inline_style` field populated via `app(AppearanceStyleComposer::class)->composeForLayout($layout)` so the Vue editor reaches gradient/image parity with the public site without duplicating `GradientComposer` in JS.
  - The Livewire bootstrap path (`PageBuilder.php`, `RecordDetailViewBuilder.php`) built the layout item *inline* and never called `formatLayout()`. `PageBuilder` was missing `appearance_config` entirely; `RecordDetailViewBuilder` was missing `inline_style`. Both bootstrap sites now compose the inline style and ship `appearance_config` on first load — so the editor renders correctly without requiring an API roundtrip to "wake up" the appearance.
  - `LayoutRegion.vue` rewired: the inline style on the container now concatenates the API-baked appearance string first (gradients/images) and the local JS-computed declarations last (color/padding/margin), so live editing on simple properties stays instant while gradient/image edits round-trip through the API.

### Issue 2 (in-session absorption) — editor toggle-independence for columns

- Reported at second manual review: in the editor, the "Background fills page width" toggle was actually constraining the column tracks (which is the content toggle's job) and the "Content fills page width" toggle had no visible effect. Public side was correct.
- Root cause: the editor's `.layout-region__container` carried both the bg style AND `display: grid` for column tracks. The `--contained` class (the editor's `.site-container` equivalent) only had one element to act on, so toggling bg constrained everything including the grid; content had no separate element to constrain.
- Fix: mirrored the public-side three-element structure in the editor.
  - `.layout-region__container` now carries appearance only (bg + padding + margin); `--contained` keyed off `bg_full_width`.
  - New inner `.layout-region__grid` carries `display: grid` + column tracks; `--contained` keyed off `content_full_width`.
  - The two `--contained` rules share a CSS selector so the box-bound math is identical.

## Findings register

- **Bypass-audit clean.** All 38 widgets respect the `.site-container` wrapper; no escape patterns. The structural-enforcement half of E10 was satisfied entirely by the read-path convergence with no per-template CSS work needed. Future widget contributors should know the contract is enforced at the wrapper level — escape attempts in widget CSS will not produce edge-to-edge rendering.
- **Per-type defaults landscape narrower than the prompt anticipated.** Only Hero / Nav / BlogListing / EventsListing carried `fullWidth(): true`; no `banner` / `cta` / `spacer` widget types exist. Per user direction, all per-type defaults flipped to `(bg:true, content:false)` uniformly.
- **`widget_presets` carried the JSON key.** The verify-at-start audit on out-of-scope #5 confirmed `widget_presets.appearance_config` carries `layout.full_width` and folded the rewrite into the migration. (Empty in dev DB; the migration's effect on widget_presets is forward-looking — preset-spawned new widgets land at the post-split shape.)
- **Editor-side bg rendering had a pre-existing parity gap.** The Vue's `LayoutRegion` containerStyle handled solid color via `appearance.background.color` but never handled gradient or image. Surfaced when the user tested the new bg toggle behavior. Fixed by adding `inline_style` to `formatLayout()` and the Livewire bootstrap so the editor's container picks up the API-composed gradient/image style. Live editing on simple properties (color/padding/margin) stays instant via the JS-side path.
- **Livewire bootstrap layout-item shape was inconsistent across the three builder paths.** `PageBuilder.php` was missing `appearance_config` entirely on layout items; `RecordDetailViewBuilder.php` had `appearance_config` but no `inline_style`; `DashboardBuilder.php` doesn't use column-layouts at all. Both gaps closed in this session. Future Livewire-builder paths that add layouts should call `formatLayout()` (or its inline equivalent) end-to-end.
- **Editor structural mismatch surfaced as a bug, not a missing feature.** The original `LayoutRegion` had one element doing both bg and grid display. The split into outer `.layout-region__container` (appearance) + inner `.layout-region__grid` (display) is a structural change — the public side already did this implicitly via `.page-layout` → `.layout-grid`. Worth noting for future editor-vs-public-rendering parity work: structural mirroring beats inline-style-merging.
- **Spatie's `BackupCollection::newest()` filters by filename pattern.** Surfaced in the 268 prompt review (FM-side feedback): pre-writing arbitrary bytes via `Storage::fake('local')` may not register as a recognized backup. Not a 267 finding per se — flagged here so 268's test-pattern audit doesn't lose the context.
