# 223. Widget Primitive — Phase 4g (Event Single-Row Pair: EventDescription + EventRegistration + PageContext::event Retirement)

Retrofit the Event single-row pair (EventDescription + EventRegistration) onto the contract layer and retire their sole upstream dependency — `PageContext::event()` — in the same session. After this lands, the only remaining widgets walking direct Eloquent are the Product pair (ProductCarousel, ProductDisplay), and `PageContext` retains only `collection()`, `product()`, and `form()` (all three on track to retire in 224 and a follow-up cleanup vertical).

This is the **first non-list-shaped contract** in the system. The single-row DTO shape, the `cardinality` declaration on `DataContract`, and the aggregate-derived field pattern (`is_at_capacity` via `withCount`) all originate here and propagate to ProductDisplay (224) and every Phase 5 record-detail-sidebar widget afterward. Get the shape right.

---

## Design decisions (resolved before starting)

- **Single-row DTO shape: `['item' => row | null]`.** Symmetric with the existing `['items' => [...]]` plural shape; nullable on miss (slug not found, slug empty, or `null` config). Templates check `isset($widgetData['item'])` before reading. This shape locks in for ProductDisplay (224), Forms (when it retrofits), and every record-detail-sidebar widget in Phase 5. Pick once, apply everywhere.
- **`cardinality` declaration on `DataContract`.** New constructor field `cardinality: 'one' | 'many'`, default `'many'`. Constants `DataContract::CARDINALITY_MANY` and `DataContract::CARDINALITY_ONE`. Resolver dispatches on `cardinality` first, then `model`. Filter shape is independent: a single-row contract carries `filters: ['slug' => $slug]`; a list-shaped contract does not.
- **Aggregate strategy: `withCount` baked into the resolver.** `Event::isAtCapacity()` runs `COUNT(*)` on `event_registrations` filtered by status. The resolver's `resolveEventOne()` adds `withCount(['registrations as registered_count' => fn ($q) => $q->whereIn('status', ['pending','registered','waitlisted','attended'])])` to the base query; the projector compares `$event->registered_count >= $event->capacity` (with a null-capacity guard returning `false`, matching the model method's behaviour). One coordinated query, no N+1 risk if a list-shaped event widget ever wants this field. This is the chosen pattern for any future "derived field that needs an aggregate query" case.
- **Per-request slug-keyed cache in `resolveEventOne()`.** Cache key `'event:one:' . $slug`. EventDescription + EventRegistration on the same event landing page share one query. Mirror of `resolvePost()`'s filter-hash cache key shape, scoped to the unique-key lookup.
- **`Event::published()` scope is the gate.** Only published events resolve. Draft, cancelled, and soft-deleted (no — events have no `deleted_at`; status='cancelled' is the tombstone) events return `['item' => null]`. The existing `Event::published()` scope handles this.
- **Field whitelist** (the canonical contract for both widgets):
  - **EventDescription**: `['title', 'starts_at', 'ends_at', 'description', 'is_in_person', 'is_virtual', 'city', 'state']` — 8 fields. The template composes its own date string from these; no `_label` derived presentation field is exposed (the conditional formatting requires Carbon::parse template-side).
  - **EventRegistration**: `['slug', 'title', 'status', 'registration_mode', 'is_free', 'is_in_person', 'mailing_list_opt_in_enabled', 'external_registration_url', 'price', 'is_at_capacity']` — 10 fields. `slug` is on the whitelist because route helpers (`route('portal.events.checkout', $event->slug)`) need it. `is_at_capacity` is the aggregate-derived field.
  - The two whitelists overlap (both declare `title`, `is_in_person`); per-field fail-closed means each widget receives only what it declared. The projector emits the superset.
- **Eight new fields on the projector.** `SystemModelProjector::projectEvent()` currently emits 15. Add `description`, `is_in_person`, `is_virtual`, `is_at_capacity`, `registration_mode`, `mailing_list_opt_in_enabled`, `external_registration_url`, `price`. Final field count: 23. Per-field fail-closed: existing list-shaped consumers (EventsListing, ThisWeeksEvents) ignore the new fields because they did not declare them. No regression on those widgets — verified by their existing tests staying byte-equivalent.
- **`description` carries raw HTML in the DTO.** EventDescription's template runs `\App\Services\Media\InlineImageRenderer::process($item['description'])` — same as today, just on the DTO field instead of the model attribute. `description` is rendered with `{!! ... !!}` (raw); the InlineImageRenderer is the security boundary on inline images.
- **Conditional date formatting stays template-side.** EventDescription needs `'g:ia'` for "5:30pm" and `'ga'` for "5pm" (when minute === 0). Projector emits `starts_at` and `ends_at` as ISO 8601 (already does); template uses `\Illuminate\Support\Carbon::parse($item['starts_at'])->format(...)` with the conditional. Centralizing this format vocabulary is **out of scope** for this session — it's the subject of the upcoming "Date Handling — Centralized Vocabulary + Schema Audit" stub.
- **No `querySettings()` declaration.** Single-row contracts have no panel — there is no admin-meaningful "limit" or "order_by" for a unique-slug lookup. Both widgets' `dataContract()` returns a contract with `querySettings: null` (the parent default). The QuerySettings panel does not render for either widget. Verified against the existing 221 test pattern: `query_settings === null` in the per-widget JSON.
- **`PageContext::event()` retires this session.** EventDescription + EventRegistration are the only production callers (verified in 222 scoping; re-verify mid-session). Mirror of 217's `upcomingEvents()` retirement and 222's `posts()` retirement: after both Event widgets retrofit, delete `event(?string $slug)` (lines ~66–79) and the `array $eventCache = []` property. Update `widget-types.md` and `widget-development.md` prose references.
- **Auth/session/request access stays in templates.** `auth('portal')->user()`, `$portalUser?->contact`, `session('registration_success')`, `request()->query('registration')`, `$errors->has('register')` — all session/request state, not data shape. The contract layer does not project these. Templates continue to access them directly.
- **Slug uniqueness gates the resolver shape.** `events.slug` has a UNIQUE constraint (verified in `database/schema/pgsql-schema.sql`). The resolver uses `Event::published()->where('slug', $slug)->first()` — at most one row.
- **Contract-locked widgets do not declare `collections()`.** System rule established in 218; neither EventDescription nor EventRegistration currently declares `collections()` (verified) — no removal needed. No `WidgetTypeSeeder` re-run required either.

## Phase 2 — Contract layer extension

1. Add `cardinality` field to `DataContract`:
   - Constants `CARDINALITY_MANY = 'many'` and `CARDINALITY_ONE = 'one'`.
   - New constructor field `public readonly string $cardinality = self::CARDINALITY_MANY` — default many for back-compat with every existing list-shaped contract.
   - Update `orderByDefault()` only if needed (probably not — single-row contracts do not order).
2. Extend `SystemModelProjector::projectEvent()` with 8 new fields. Source columns and casts:
   - `'description' => (string) ($event->description ?? '')` — raw HTML, ready for InlineImageRenderer.
   - `'is_in_person' => (bool) $event->is_in_person` — accessor (computed from `address_line_1` presence).
   - `'is_virtual' => (bool) $event->is_virtual` — accessor (computed from `meeting_url` presence).
   - `'is_at_capacity' => $event->capacity !== null && (int) ($event->registered_count ?? 0) >= (int) $event->capacity` — relies on `withCount` aliasing `registered_count` onto the model. Null capacity → false (matches `Event::isAtCapacity()`'s behaviour).
   - `'registration_mode' => (string) ($event->registration_mode ?? '')`.
   - `'mailing_list_opt_in_enabled' => (bool) $event->mailing_list_opt_in_enabled`.
   - `'external_registration_url' => (string) ($event->external_registration_url ?? '')`.
   - `'price' => (string) ($event->price ?? '0.00')` — decimal cast on the model emits a string already; preserve.
3. Add `projectOne(DataContract, ?Model)` to `SystemModelProjector`. Returns `['item' => row | null]`. Reuse the same per-model dispatch as `project()`. Field-filter through `$contract->fields` exactly as `project()` does.
4. Extend `ContractResolver`:
   - `resolveSystemModel()`: dispatch on cardinality first. If `CARDINALITY_ONE`, dispatch by model to `resolveEventOne()` (and a future `resolveProductOne()` in 224, default `['item' => null]`). If `CARDINALITY_MANY`, dispatch as today.
   - New private `resolveEventOne(DataContract, &$cache)`:
     - Read `$slug = $contract->filters['slug'] ?? ''`. If empty/null, return `['item' => null]` without hitting the database.
     - Cache key `'event:one:' . $slug`. On cache miss, run `Event::published()->where('slug', $slug)->withCount(['registrations as registered_count' => fn ($q) => $q->whereIn('status', ['pending','registered','waitlisted','attended'])])->with(['media', 'landingPage'])->first()`. Cache the result (model or null).
     - Return `$this->systemModelProjector->projectOne($contract, $cache[$key])`.
5. The contract's `resolve()` outer return shape is unchanged: each contract resolves to its own DTO array; the per-contract shape (`['items' => [...]]` vs `['item' => row | null]`) is what differs.
