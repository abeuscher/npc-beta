# 201. Widget Template Tokens — Per-Item Namespace Split

Separate the per-item token grammar used by listing-widget `content_template` fields from the page-context token grammar handled by `PageContextTokens`, so `{{title}}` inside a per-event (or per-post, per-product, etc.) card template no longer gets hijacked by the owning page's title.

---

## Context

Surfaced during Session 200 manual testing. The Events Listing widget in both the page-builder preview and the public page rendered **every** event card with the title "Events" — which is the title of the hosting page — while slugs (`demo-event-1/2/3`) rendered correctly per event.

Root cause: token-name collision between two namespaces that share the `{{key}}` grammar.

- `PageContextTokens::TOKENS` = `['title', 'date', 'excerpt', 'author', 'starts_at', 'location']`. These are resolved against the current (or owning) `Page` in `app/Services/PageContextTokens.php`.
- `EventsListingDefinition::defaults()['content_template']` uses `{{title}}`, `{{date}}`, `{{location}}` as **per-event** placeholders, filled by the per-item loop in `app/Widgets/EventsListing/template.blade.php`.
- `app/Services/WidgetRenderer.php` (lines ~66–77) iterates every `richtext`/`text` config field and calls `PageContextTokens::substitute` on it **before** the widget's own template runs. The colliding keys (`title`, `date`, `location`) are replaced with the page's values for every card at once. The non-colliding keys (`slug`, `url`, `date_iso`, `ends_at`, `price_badge`, `image`) survive intact for the per-item loop to fill.

A field like `heading` on the same widget using `{{title}}` to mean "the page title" is **legitimate** — that is the intended PageContextTokens behaviour. Only the per-item loop's private vocabulary needs to move to a different namespace.

---

## Design decisions (resolved before starting)

- **Syntax for per-item tokens.** Use `{{item.key}}` — dot-delimited namespace. The bare `{{key}}` form continues to mean "page context". Alternatives considered and rejected:
  - `{{@title}}` — `@` is reserved in Blade contexts and would read ambiguously.
  - `[[title]]` — introducing a second delimiter grammar for no gain.
  - `{{:title}}` — the colon prefix reads as a pseudoclass when editing in CSS-aware highlighters.
  - `{{item.title}}` has precedent in Laravel/Livewire user-surfaces and is self-explanatory to a designer reading a template string.
- **Which widgets are in scope.** Any widget whose blade template performs per-item token substitution against a user-editable `content_template` (or equivalent) field. At minimum: `EventsListing`, plus whichever other listing widgets the Phase 1 audit surfaces (Blog listing, product listing/carousel, content-listing successors). Non-listing widgets that use `{{title}}` to mean the page title (hero, text block, etc.) are **NOT** in scope — their usage is correct.
- **`PageContextTokens` is unchanged.** No change to `TOKENS`, no change to substitution logic, no widget-awareness added. The fix lives entirely in listing-widget defaults, listing-widget blade templates, and a data migration.
- **Back-compat via migration.** A one-time data migration rewrites `{{key}}` → `{{item.key}}` inside `page_widgets.config.content_template` for rows belonging to in-scope widget types. The rewrite is keyed per-widget (using that widget's per-item vocabulary), so unrelated tokens in a user's customized template are left alone.
- **Idempotency and reversibility.** The migration detects already-migrated rows (presence of `{{item.` substrings for the expected keys) and is a no-op on second run. A `down()` direction restores the old grammar for the same keys. Rows outside the in-scope widget types are never touched.
- **Substring safety.** `str_replace('{{title}}', …)` does not match inside `{{item.title}}` because `{{title}}` is not a substring of `{{item.title}}`. This means `PageContextTokens::substitute` naturally skips the namespaced tokens with no changes required to the substitution logic itself. Confirm this invariant during Phase 1 before proceeding.

## Phase 1 — Scope audit

Before editing anything, produce a punch list. Record it in the session log as part of closing.

1. Grep `app/Widgets/*/*Definition.php` for `content_template` default strings. List every widget that defines one.
2. Grep `app/Widgets/*/template.blade.php` for per-item token substitution loops. Typical patterns:
   - `str_replace('{{' . $key . '}}', …)`
   - `preg_replace('/\{\{[^}]+\}\}/', '', $card)` (the trailing cleanup after the loop)
3. For each matched widget, record:
   - The widget's handle.
   - The keys its per-item loop actually substitutes (the vocabulary).
   - Which of those keys collide with `PageContextTokens::TOKENS` (the collisions are what the bug manifests on; the non-collisions are already working).
4. Confirm the substring-safety invariant by spot-checking a `{{item.title}}` string against `str_replace('{{title}}', …)` — include a one-line note or a test asserting no match.

Do not edit code in Phase 1. The punch list is the deliverable; it drives Phases 2 and 3.

---

## Phase 2 — Code changes per widget

For each listing widget on the punch list:

1. Update `defaults()` and `schema()` in the Definition class — rewrite the default `content_template` string so every per-item token uses `{{item.key}}`.
2. Update the widget's `template.blade.php` — the per-item loop must replace `{{item.key}}` instead of `{{key}}`. Keep every other rendering detail (escape handling, raw-HTML exceptions for `image` / `price_badge`, the `preg_replace` cleanup pass) identical.
3. Add a focused Pest test per widget covering the collision case: a hosting page whose title collides with a per-item field (e.g. a page titled "Events" rendering an EventsListing widget against seeded events). Assert each rendered card shows the event's own title, not the page's title. Keep the test in the fast suite.

Non-listing widgets, `PageContextTokens`, and `WidgetRenderer` are **not** modified in Phase 2.

---

## Phase 3 — Data migration

Write a migration that rewrites existing `page_widgets.config.content_template` values for rows belonging to in-scope widget types.

Requirements:

- **Widget-aware.** The rewrite uses each widget's own per-item vocabulary from the Phase 1 audit — only those keys are rewritten. Incidental `{{...}}` strings a user added for other reasons are left alone.
- **Idempotent.** Detect already-migrated rows (the expected `{{item.key}}` forms are already present) and skip them. Running the migration twice changes nothing the second time.
- **Reversible.** `down()` restores `{{item.key}}` → `{{key}}` for the same per-widget key list, so a rollback returns the DB to pre-migration grammar.
- **Audit trail.** Log a short per-widget summary to `storage/logs/laravel.log` at info level: widget handle, row count scanned, row count rewritten, sample before/after for one row per widget. No PII concerns here — `content_template` is widget config, not user content.
- **Scope.** Touches only `page_widgets.config->>'content_template'`. Does not touch any other column, table, or JSON path. Does not touch `widget_types` (the default `content_template` lives in `defaults()` on the Definition class, not in the `widget_types` row for sovereign widgets).

Run the migration locally via `docker compose exec app php artisan migrate`. Do **not** run `migrate:fresh --seed` — the migration needs to exercise against the accumulated local DB state, not a fresh one.

Update `docs/schema/page_widgets.md` with a short note under the `config` column describing the token-grammar convention for per-item `content_template` values (`{{item.key}}` for per-row, `{{key}}` for page context). No schema columns change.

## Testing

- **Slow test groups to run this session:** none.
- **New tests expected:** yes. One Pest test per in-scope listing widget (Phase 2, step 3). Plus one migration test that:
  - seeds a `page_widgets` row with the old grammar,
  - runs the migration,
  - asserts the row now uses the new grammar,
  - runs the migration again (idempotency),
  - asserts nothing changed the second time,
  - runs `down()`, asserts the old grammar is back.
- Run `docker compose exec app php artisan test --exclude-group=slow` at the end.
- Run `npm run build` — the inspector field-hint surfaces may reference the token grammar (`{{title}}` in help copy on the content template field). If any Vue strings need updating to reflect the new grammar, update them; otherwise build is a belt-and-braces check.
- **Manual verification.** Open the page builder on a page titled "Events" (or anything colliding with `PageContextTokens::TOKENS`) with an Events Listing widget. Confirm each event card renders its own title, not the page title. Repeat in the public page view. Spot-check any other in-scope widgets on seed-data-backed pages.

## Out of scope

- **Rebuilding `PageContextTokens`** into a general-purpose templating engine.
- **Per-widget expression evaluation** (filters, pipes, formatting like `{{item.title | upper}}`). If surfaced as tempting during implementation, stop and park it.
- **Non-listing widgets.** Hero, text block, navigation, etc. continue using `{{title}}` to mean the page title. That behaviour is correct.
- **Inspector UX / designer-facing docs for the new grammar.** A follow-up session can add field-hint strings, an inspector helper that lists available `{{item.*}}` keys per widget, or documentation in `resources/docs/`. This session is mechanical — the grammar change and the migration — not the designer-education layer.
- **Renaming keys in `PageContextTokens::TOKENS`** (e.g. `starts_at` → `event.starts_at`). The page-level grammar stays as-is.
- **Widget Toolset Tightening** (session-outlines stub). That's the per-widget appearance audit. Not this session.
