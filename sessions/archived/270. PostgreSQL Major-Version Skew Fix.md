# 270. PostgreSQL Major-Version Skew Fix

Addendum to 269 — PostgreSQL major-version skew between app container and DB server, surfaced by FM 021 manual testing on 2026-05-08.

Context: FM-side ran the new `fleet:restore-from-blob` artisan command (FM session 021) against a freshly-installed destination droplet. The command successfully fetched a backup blob from the source CRM via `GET /api/backup/blob`, SFTPed it to the destination, ran `DROP SCHEMA public CASCADE` cleanly, then failed at the dump-load step with:

  ERROR: unrecognized configuration parameter "transaction_timeout"

Diagnosis: `transaction_timeout` is a PostgreSQL 17 directive (added June 2024). `pg_dump` from PostgreSQL 17+ writes `SET transaction_timeout = 0;` into every dump's preamble. Both source and destination droplets run `postgres:16-alpine` per `docker-compose.prod.yml`, but the app container's `pg_dump` binary comes from Debian's `postgresql-client` meta-package (line 60 of `Dockerfile`), which on the current `php:8.4-fpm` Trixie base resolves to client 17. So spatie's backup pipeline runs `pg_dump 17` against the `postgres 16` server, producing a dump postgres 16 cannot ingest. The source CRM literally cannot restore from its own backups today.

Fix scope (no shortcuts — pin both ends explicitly and align them):

1. `Dockerfile` line 60: change `postgresql-client` (unpinned meta-package) to `postgresql-client-17` (explicit major). Add a comment naming the coupling: this MUST match the postgres image major in both docker-compose files. Use Debian Trixie's native packaging if it ships postgresql-client-17 (it does); otherwise add the apt.postgresql.org repo.

2. `docker-compose.yml` line 49 and `docker-compose.prod.yml` line 50: bump `postgres:16-alpine` to `postgres:17-alpine`. Add the same coupling comment above each `image:` line.

3. Add a fast-group test (`tests/Feature/Infrastructure/PostgresVersionSkewTest.php`) that runs inside the built app container and asserts `pg_dump --version` major matches `SHOW server_version` major. Skips cleanly if postgres isn't reachable. Catches future drift mechanically.

4. Existing-install migration runbook at `docs/runbooks/postgres-major-upgrade.md`: dump-and-restore procedure (pg_dumpall → docker compose down -v → pull + up on PG17 → load dump). pg_upgrade is the alternative but more moving parts; not recommended for our pre-Beta droplet count. PG16 dumps don't contain `transaction_timeout`, so dumpall on the source succeeds and forward-loads into PG17 cleanly.

5. After implementation: locally run `php artisan backup:run` against the new PG17 stack, then verify the produced zip's dump re-loads cleanly via `unzip -p .../db-dumps/postgresql-nonprofitcrm.sql.gz | gunzip | psql` against a fresh PG17 instance.

Out of scope: bumping the Fleet Manager agent contract (this is CRM-internal infrastructure, contract surface unchanged). Pinning postgres minor versions (17-alpine accepts whatever minor Docker Hub publishes). Spatie backup config tuning. pg_upgrade-based migration.

FM 021 manual testing is blocked on this. Once it ships, the operator wipes + reinstalls existing droplets (or runs the migration runbook), then FM-side runs `fleet:restore-from-blob` end-to-end.
