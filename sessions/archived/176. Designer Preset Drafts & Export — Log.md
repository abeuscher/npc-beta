## What was built

### Phase 1 — Database & model

- New `widget_presets` migration: uuid pk, `widget_type_id` FK (cascade on delete), `handle` string, `label` string, `description` text nullable, `config` jsonb default `{}`, `appearance_config` jsonb default `{}`, timestamps. Composite unique index on `(widget_type_id, handle)`.
- `App\Models\WidgetPreset` model with `HasUuids`, jsonb casts for `config` + `appearance_config`, `belongsTo(WidgetType)`.
- `draftPresets()` `hasMany` relation on `WidgetType` (named distinctly from the registry-sourced code `presets` key to avoid semantic overlap).
- New schema doc `docs/schema/widget_presets.md` documenting columns, indexes, and the appearance-group rule; index entry added to `docs/schema/README.md`.

### Phase 2 — Backend: save / update / delete, bootstrap payload, tests

- New `App\Http\Controllers\Admin\PresetController` with `store`, `update`, `destroy`. All methods gate on `update_page`.
- Routes added to `routes/admin-api.php` under the existing Filament middleware group: `POST widget-presets`, `PATCH widget-presets/{preset}`, `DELETE widget-presets/{preset}`. Went with a dedicated controller rather than folding into `PageBuilderApiController` — preset CRUD has no dependency on the page-widget tree.
- **Save materializes a fat snapshot.** The initial implementation captured only the sparse keys that were present in `$widget->config`, which produced a bug: applying a preset after another preset had set e.g. `min_height` left the higher value sticking around because the saved preset had no key for it. Fix: overlay `$widget->config` onto `WidgetConfigResolver::resolvedDefaults($widget)`, filtered to appearance-group keys. Presets are fat; instances remain sparse. Tradeoff: exported PHP literals are chattier than hand-written ones — designer prunes default-valued keys by hand after paste. Decision discussed and accepted mid-session.
- Auto-naming on save: `handle` = `draft-N` where N is the next unused integer for that widget_type_id; `label` = `Draft N`. After a delete, the next save may re-use a freed slot (e.g. save draft-1, delete, save again → new row is also `draft-1`). Documented as expected behaviour.
- PATCH validates `handle` slug shape (`^[a-z0-9-]+$`) and uniqueness per widget_type_id server-side. No edits to `config` / `appearance_config` via PATCH — rename/describe only.
- Server-side appearance-group guard: on save, keys outside the widget type's appearance-group schema are stripped (content-group keys never reach the preset row). DB rows that violate the rule are caught by the mirror test in `PresetControllerTest`.
- Bootstrap payload extended in two places — `WidgetType::forPicker()` (Livewire mount path) and `PageBuilderApiController::widgetTypes()` (admin API path). Each widget_types entry now carries `draft_presets[]` alongside the existing `presets[]`. Draft entries include `id`, `handle`, `label`, `description`, `config`, `appearance_config`, and `is_draft: true`.
- Feature tests in `tests/Feature/PresetControllerTest.php` (9 tests, 48 assertions): save with appearance-group filtering, default materialization, auto-increment handles, auth/permission gates, rename + handle-uniqueness, delete, DB preset shape mirror (appearance-group keys only), bootstrap payload surfacing.

### Phase 3 — Frontend: save / rename / delete / export + store + types

- `resources/js/page-builder-vue/types.ts` — new `WidgetDraftPreset extends WidgetPreset` with `id: string` + `is_draft: true`. `WidgetType.draft_presets?: WidgetDraftPreset[]` added.
- `resources/js/page-builder-vue/api.ts` — `createDraftPreset`, `updateDraftPreset`, `deleteDraftPreset` helpers routing to the new admin-api endpoints through the existing CSRF-authenticated `request()` helper.
- `resources/js/page-builder-vue/stores/editor.ts` — `saveDraftPreset(widgetId)`, `renameDraftPreset(presetId, payload)`, `deleteDraftPreset(presetId)` actions that mutate the widget type's `draft_presets` array in place after the server round-trip completes.
- `resources/js/page-builder-vue/components/InspectorPresetGallery.vue` — rewritten:
  - Gallery iterates Blank → `widgetType.presets` (code) → `widgetType.draft_presets` (DB).
  - "Save current appearance as preset" button at the top of the tab.
  - Draft cards carry a small "Draft" badge and an actions row: Rename (inline label + description edit, commit on Enter/Save, revert on Escape), Export (copies PHP literal to clipboard), Delete (with confirm). Code-authored cards and the Blank card remain apply-only.
  - `navigator.clipboard.writeText` call with a 2-second "Copied!" toast on the Export button. No server round-trip for export.
  - PHP array literal generator produces pretty-printed, 4-space-indented output with a trailing comma and key alignment — paste-ready for dropping into a `presets()` return list.

## Deferred decisions

- **Fat exports include default-valued keys.** The fix for the default-value bug made saved presets fat (every appearance-group key materialized). Exported PHP literals carry those default values. Pruning defaults out of the exported literal was rejected — it would mean reconstructing sparse runtime behaviour at export time, which is the bug we just fixed going in reverse. Designers prune by hand after paste.
- **Slot re-use on delete+save.** A new save after deleting an earlier draft re-uses the freed `draft-N` slot. Kept as-is; renumbering would be client-observable churn for no gain.
- **Handle collisions between code and DB.** The gallery tolerates identical handles across sources — one code card, one DB card distinguished by the Draft badge. A real conflict only matters at paste time, where CI rejects a duplicate on promotion.
- **Preset thumbnails.** Not in this session. Session 177 base + session prompts written this session and committed; dev preset route + script extension + serving controller + card image slot + Hero pilot capture all live there.
