# 255. Organizations Model Overhaul (Min)

Lift Organization from "named entity that groups Contacts" to a peer of Contact that can hold transactional records — corporate donations, corporate memberships, sponsored events. Implementation is the **minimum** of B1: nullable `organization_id` FKs on the four transactional tables, four new importer sentinels mirroring the existing `__org_contact__` pattern, four read-only related-records panels on the Organization edit page, and block-with-counts deletion. **No affiliation junction this session** — that is B1b, queued post-B2.

---

## Design decisions (resolved before starting)

- **Schema shape — four new nullable FKs.**
  - `donations.organization_id` (UUID FK to `organizations.id`, `ON DELETE SET NULL`).
  - `memberships.organization_id` (same shape).
  - `event_registrations.organization_id` (same shape).
  - `events.sponsor_organization_id` (same shape — the explicit-name disambiguates from a future host org).

- **Coexistence with `contact_id` — non-exclusive.** A donation, membership, or event_registration may carry both `contact_id` (the human handler) and `organization_id` (the corporate party-of-record). At least one of the two must be non-null in application logic, but the database does not enforce XOR — leaving room for B1b's soft-credit work to attribute multi-party donations without a schema change. Validation happens at create-time in the importer and on Filament forms; existing records (contact-only) require no migration.

- **`Contact.organization_id` stays exactly as it is.** The "Contact's primary employer" link is unchanged, both schema and behavior. The existing `__org_contact__` importer sentinel keeps its current semantics (fill `Contact.organization_id`).

- **Four new importer sentinels — namespaced per importer.** Each writes to a different transactional model's `organization_id`:
  - `__org_donor__` (donations) — sets `donations.organization_id`.
  - `__org_member__` (memberships) — sets `memberships.organization_id`.
  - `__org_sponsor__` (events, event-level row) — sets `events.sponsor_organization_id` from the first row introducing each event (consistent with how event-level columns are read today).
  - `__org_invoice_party__` (invoice details) — *target depends on a verify-at-start check.* If invoices are a top-level `Invoice` model in `app/Models/`, the sentinel writes to `invoices.organization_id`. If invoices are grouped Transactions, decide at session start whether the FK lives on `Transaction` or on a derived "Invoice" record; if neither path is clean, defer this sentinel to B1b. The other three sentinels are unblocked regardless.

- **`event_registrations.organization_id` schema only — no importer sentinel.** The column ships in the migration but the importer does not gain a `__org_registrant__` sentinel. Org-as-registrant is a rare data shape; admin UI can set the FK manually if needed. If session 255 verifies a real CSV with this column, surface and add the sentinel; default is to skip.

- **Sentinel UX mirrors `__org_contact__` exactly.** Each new sentinel renders the same "create or match by name" Radio sub-form (`auto_create` / `match_only` / `as_custom`) that already exists for `__org_contact__`. The strategy field name is `org_strategy_{$n}` (already in use). The match logic is the same `whereRaw('LOWER(TRIM(name)) = ?')` pattern from [`InteractsWithImportProgress.php`](../app/Filament/Pages/Concerns/InteractsWithImportProgress.php) lines 800–807, lifted into a shared helper if reuse is clean, otherwise paste-and-adapt per importer.

- **Admin UI — four read-only related-records panels on Organization edit.** Filament `RelationManager` per panel: Members (existing `Contact.organization_id` reverse), Donations Received, Memberships Held, Events Sponsored. Each shows row count, the most recent N entries, and a deep-link to the underlying resource. No edit affordances on these panels — set the FK from the source record's edit page or via the importer.

- **Deletion — block-with-counts, force-delete with explicit confirmation.** Mirror the pattern that Contact deletion already uses: refuse delete if the Org has any of (members, donations, memberships, sponsored events) and surface the counts; offer a "Force delete" branch that the operator must explicitly confirm. The force-delete `ON DELETE SET NULL` cascade nulls the FK on every transactional record but does not delete those records.

- **B1b is parked, not added back as a stub.** The release-plan entry for B1b is the canonical record. Do not re-stub the affiliation junction or soft-credit work in `session-outlines.md` outside of the existing release-plan reference.

## Phase 2 — Schema migrations

Write four (or one combined) migration(s) under `database/migrations/`:

- `donations.organization_id` — nullable UUID, FK to `organizations.id`, `ON DELETE SET NULL`, index.
- `memberships.organization_id` — same shape.
- `event_registrations.organization_id` — same shape.
- `events.sponsor_organization_id` — same shape.
- `invoices.organization_id` — only if the verify-at-start in Phase 1 surfaced an `Invoice` model; otherwise skip with a note.

Run via `docker compose exec app php artisan migrate`. Update each affected `docs/schema/*.md` file in the same commit (`donations.md`, `memberships.md`, `event_registrations.md`, `events.md`, possibly `invoices.md`).

Add the matching Eloquent relationships:

- `Donation::organization()` — belongsTo.
- `Membership::organization()` — belongsTo.
- `EventRegistration::organization()` — belongsTo.
- `Event::sponsorOrganization()` — belongsTo.
- `Organization::membershipsHeld()`, `Organization::eventsSponsored()`, `Organization::eventRegistrations()` — hasMany. (`Organization::donations()` already declared; just becomes functional.)

## Phase 3 — Importer destinations and progress-page wiring

For each of donations / memberships / events / invoice-details (if Phase 1 cleared invoices), add the new sentinel to its `*FieldRegistry::groupedOptions()`. Use the same human-facing label convention as `__org_contact__`:

- `__org_donor__` → "— Link to Organization (Donor) —"
- `__org_member__` → "— Link to Organization (Member) —"
- `__org_sponsor__` → "— Link to Organization (Sponsor) —"
- `__org_invoice_party__` → "— Link to Organization (Bill-To) —"

Update [`InteractsWithImportWizard::buildNamespacedMappingRow`](../app/Filament/Pages/Concerns/InteractsWithImportWizard.php) — the `$orgSubForm` Radio is currently visible only when destination = `__org_contact__`. Extend the visibility predicate to include the four new sentinels (each renders the same `auto_create` / `match_only` / `as_custom` choice).

Update [`InteractsWithImportWizard::serializeColumnMaps`](../app/Filament/Pages/Concerns/InteractsWithImportWizard.php) — the `__org_contact__` branch (line ~927) handles the existing case. Add four new branches mirroring it; each writes the relational map entry with a different `type` key (`donation_organization`, `membership_organization`, `event_sponsor_organization`, `invoice_organization`) plus the strategy.

Update each of the four progress-page classes (`ImportDonationsProgressPage`, `ImportMembershipsProgressPage`, `ImportEventsProgressPage`, `ImportInvoiceDetailsProgressPage`) — add a new helper that handles the relational map entry for its sentinel. The helper resolves the org by name (using the existing `whereRaw('LOWER(TRIM(name)) = ?')` pattern), creates if `auto_create` and missing, sets the FK on the transactional record. The contacts importer's existing `__org_contact__` handling stays untouched.

For each new sentinel, the strategy `as_custom` semantics: ignore the column for the org link entirely; if the importer page also offers a custom-field path, the column gets demoted to a custom field on the transactional record (mirrors how `__org_contact__` + `as_custom` works today).

## Phase 4 — Admin UI: four related-records panels on Organization edit

In [`OrganizationResource`](../app/Filament/Resources/OrganizationResource.php) — declare four `RelationManagers`:

- `MembersRelationManager` — drives off `Contact.organization_id` reverse (existing relationship). Read-only table; columns = Name, Email, Phone, last activity. Deep-link to `ContactResource::edit`.
- `DonationsReceivedRelationManager` — `donations.organization_id` reverse. Columns = Date, Amount, Fund, Status. Deep-link to the Donation edit / view.
- `MembershipsHeldRelationManager` — `memberships.organization_id` reverse. Columns = Tier, Status, Start Date, End Date. Deep-link to Membership edit.
- `EventsSponsoredRelationManager` — `events.sponsor_organization_id` reverse. Columns = Title, Date, Status. Deep-link to Event edit.

All four are display-only — no inline create / edit / delete actions. The user manages org links from the source record's edit page or via the importer.

## Phase 5 — Block-with-counts deletion

In `OrganizationResource` — override the delete action (header + bulk + table actions) to:

1. Count rows in each of the four reverse-relation tables.
2. If any count is > 0: refuse the delete and surface a notification listing the four counts ("This organization has 47 members, 12 donations, 3 memberships, 1 sponsored event. Reassign or remove these before deleting.").
3. Offer a "Force delete" follow-up action that the operator must confirm. Force-delete proceeds with `ON DELETE SET NULL` cascading on the four FKs (no records are deleted; their org link is nulled).

Mirror the existing pattern from `ContactResource`'s deletion if a `force-delete` precedent exists there; otherwise lift the simplest viable Filament Action shape.

## Phase 6 — Tests

- **Migration tests** — schema asserts that each new column exists, is nullable, has FK + index. Standard Pest pattern.
- **Importer round-trip tests** — for each of the four new sentinels, a Pest feature test: build an `ImportLog` with a column mapped to the sentinel, run the import, assert the resulting transactional record has `organization_id` correctly populated. Cover all three strategies (`auto_create` creates new org, `match_only` skips when org not found, `as_custom` falls back to custom field). Reuse fixtures under `tests/e2e/fixtures/` where shape allows.
- **Admin panel render tests** — Pest feature test mounting `OrganizationResource::edit` and asserting the four `RelationManager` tables render. One covers the "Org with rows" case (counts > 0), one covers "Org with no rows" (empty states).
- **Deletion-block test** — Pest feature test: create Org with associated rows, attempt delete, assert blocked + notification surfaces; force-delete, assert FKs nulled and rows preserved.

No Playwright spec required this session unless the four panels reveal a visual-feel concern only the browser exposes (per the verify-objective-outcomes rule, structural assertions through Pest are sufficient).

## Security checklist

Before closing:

- [ ] Each new FK is nullable and uses `ON DELETE SET NULL` — no cascade-deletes silently nuke transactional history.
- [ ] The block-with-counts logic refuses deletion until the operator force-confirms; the force-delete path doesn't bypass the count-display warning.
- [ ] No widening of `Organization` permissions — existing `OrganizationPolicy` gates apply unchanged.
- [ ] Importer sentinel handling validates the source-name string before passing to SQL (existing `whereRaw('LOWER(TRIM(name)) = ?')` pattern uses parameter binding — verify the new branches use the same pattern).
- [ ] PII surface unchanged — Org-as-source records expose only the same fields already public on Contact-as-source records.
