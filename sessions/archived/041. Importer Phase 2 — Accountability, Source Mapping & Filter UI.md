# 041. Importer Phase 2 — Accountability, Source Mapping & Filter UI

This session rebuilds the contact importer into a production-ready, accountable data pipeline, consolidates it under a new unified **Importer** tool, and improves the mailing list filter builder's field-awareness. All patterns are built to extend cleanly to future content types (events, financial data) without rework.

---

## Goals

1. **Importer tool** — replace the current importer page with a dedicated "Importer" navigation item. An intro screen offers three content type buttons: Import Contacts (built this session), Import Events, and Import Financial Data (both are stubs — visible but disabled, linking to a placeholder screen).
2. **Import accountability** — every import runs as a session with states (`pending → reviewing → approved`). Pending records are invisible to all users without the `review_imports` permission. A reviewer approves or rolls back the full session. Rollback is surgical and complete.
3. **Import source mapping** — named import sources persist the `source_id → uuid` mapping across sessions, enabling re-import matching and future cross-type relationship resolution.
4. **Contact source field** — lock the `source` field on contact records: always required, never editable by any user in the UI.
5. **Mailing list filter UI** — field-aware operator lists and value inputs; layout changed to show existing rules inline with delete buttons, new rule fields at the bottom.

## Phase 1 — Contact source field lock

The `source` column on `contacts` tracks how the record entered the system. Valid values: `manual`, `import`, `api`, `web_form`. This field must be:

- **Required** — every contact must have a source. Add a database-level default of `manual` if not already present.
- **Not editable** — remove the `source` field from the contact edit form entirely. It should remain visible as read-only information. Display it as a `TextEntry` (or `Placeholder`) in the edit form, not an editable `Select`.
- **Set automatically** — the importer sets `source = 'import'` on every record it creates. Manual creation defaults to `manual`. API and web form paths set their values at the point of creation in those respective flows.
- **No user at any permission level can change this field through the UI.** Document this in `resources/docs/contacts.md`.

## Phase 2 — Data model: import sessions and source mapping

### `import_sources` table

Stores named external systems that contacts (and later other records) are imported from.

| Column | Type | Notes |
|---|---|---|
| `id` | uuid | primary key |
| `name` | string | user-given name, e.g. "Old CRM", "Salesforce 2024" |
| `notes` | text, nullable | optional description |
| `timestamps` | — | |

### `import_sessions` table

One row per import run.

| Column | Type | Notes |
|---|---|---|
| `id` | uuid | primary key |
| `import_source_id` | uuid, nullable, FK | which named source this import came from |
| `model_type` | string | e.g. `contact` |
| `status` | string | `pending`, `reviewing`, `approved` |
| `filename` | string, nullable | original uploaded filename |
| `row_count` | integer, nullable | total rows processed |
| `imported_by` | uuid, FK → users | who ran the import |
| `approved_by` | uuid, nullable, FK → users | who approved |
| `approved_at` | timestamp, nullable | |
| `timestamps` | — | |

### `import_id_maps` table

Persists the source-ID-to-UUID mapping for each import source, enabling re-import matching and cross-record relationship resolution.

| Column | Type | Notes |
|---|---|---|
| `id` | uuid | primary key |
| `import_source_id` | uuid, FK | which source system |
| `model_type` | string | e.g. `contact` |
| `source_id` | string | the original ID from the external system |
| `model_uuid` | uuid | our internal UUID |
| `timestamps` | — | |

Add a unique index on `(import_source_id, model_type, source_id)`.

### `contacts` table changes

Add `import_session_id` (uuid, nullable, FK → import_sessions). This column links each contact to the import session that created it, enabling surgical rollback and scope filtering.

## Phase 3 — Import accountability: states, visibility, rollback

### Global scope on Contact

Add a global scope to the `Contact` model that excludes records where `import_session_id` is set and the linked session's `status` is not `approved`. This scope must be automatically bypassed for users who have the `review_imports` permission.

The net effect: pending imported contacts are completely invisible in all normal Filament views, mailing list queries, exports, and any other query that uses the Contact model. They do not exist until approved.

### Permissions

Two new permissions, added to the `PermissionSeeder`:

- `import_data` — can access the Importer tool, upload files, and run imports
- `review_imports` — can view pending import sessions, approve them, and trigger rollback. Cannot themselves import.

Assign `import_data` to the `admin` and `data_entry` roles (or equivalent). Assign `review_imports` to `admin` only. These are separate — a user can have one without the other.

### Rollback

The rollback action on an import session must:
1. Delete all records in `contacts` where `import_session_id = <session_id>`
2. The cascade on `taggables` and other relationship tables handles cleanup automatically — verify FK cascades are in place
3. Delete the session row itself
4. **Do not** touch `import_id_maps` — preserve the source mapping for future reference

Rollback is only available while the session status is `pending` or `reviewing`. Once `approved`, the button is gone.

### Review interface

A dedicated **Import Review** page (not mixed into the contact list) accessible only to users with `review_imports`. Shows:

- Session metadata (source, filename, row count, imported by, imported at)
- A paginated read-only table of the imported contacts
- An **Approve** button (moves status to `approved`, records become globally visible)
- A **Rollback** button with a confirmation dialog ("This will permanently delete all N contacts from this import. This cannot be undone.")

The review page is read-only. Reviewers cannot edit individual imported records during review.

## Phase 4 — Importer tool rebuild

### Navigation

Replace the existing importer page with a single **Importer** item in the Tools navigation group. Remove the old entry from wherever it currently lives.

### Intro screen

The importer landing page shows three large buttons:

- **Import Contacts** — navigates to the contact import flow
- **Import Events** — disabled, shows "Coming soon" tooltip or greyed state
- **Import Financial Data** — disabled, shows "Coming soon" tooltip or greyed state

### Contact import flow

Preserve the existing 3-panel step-by-step pattern. Add or modify the following:

**Step 0 — Source selection (new)**

Before the file upload step, ask the user to either:
- Select an existing named import source from a dropdown, or
- Create a new one by typing a name

This selection is stored on the `import_session` record. It determines which `import_source_id` is used for ID mapping.

**Step 1 — File upload**

Unchanged from current implementation, except: after upload and parse, create the `import_session` record in `pending` status immediately.

**Step 2 — Field mapping**

The field mapping step must be field-type-aware. Each mappable field in the system has a definition that specifies:

- Its display label
- Its data type (text, email, phone, date, boolean, select, tags, external_id)
- Any special handling on import (e.g. tags require relationship sync, not column write)

Define this registry as a PHP class or config file — it will be reused by future importers for other content types. For this session, implement the contacts field registry only.

A special field type: **External ID**. If the user maps a CSV column to "External ID", the values in that column are stored in `import_id_maps` as `source_id` values keyed to the newly created contact UUIDs. On re-import from the same source, the importer checks `import_id_maps` before creating a record — if `source_id` already exists for this source, it updates the existing contact rather than creating a duplicate.

**Step 3 — Review and import**

Show a preview of the first N rows as they will be imported. Run PII validation (existing logic). On confirm, process the import, set `import_session.status = 'reviewing'`, and redirect the user to a confirmation screen that says the import is pending review and shows a link to the review queue (visible to reviewers only).

## Phase 5 — Mailing list filter UI

### Field type awareness

The filter builder currently shows the same operator list and a plain text input for every field. Change this so each field drives its own available operators and value input.

Define a `MailingListFieldRegistry` (or equivalent) that maps each field name to:

- An array of valid operators
- A value input type (`text`, `select`, `tag_picker`, `none`)

Minimum definitions for this session:

| Field | Operators | Value input |
|---|---|---|
| first_name, last_name, email, phone, city, state, postal_code | equals, not_equals, contains, not_contains, is_empty, is_not_empty | text |
| tags | includes, not_includes | tag picker (contact tags only) |
| mailing_list_opt_in | equals, not_equals | select (Yes / No) |

When the user selects a field in the filter row, the operator dropdown updates to show only valid operators for that field, and the value input renders the appropriate control.

### Layout changes

Restructure the filter repeater so that:

- Each existing rule occupies a single horizontal row: Field | Operator | Value | (trash icon)
- The "new rule" input row sits at the bottom of the list, always visible, with an **Add** button below it (full width, consistent with row width)
- No collapsed/hidden state — all existing rules and the new-rule row are visible simultaneously
