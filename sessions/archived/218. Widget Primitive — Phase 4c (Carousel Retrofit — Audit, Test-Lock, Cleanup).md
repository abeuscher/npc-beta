# 218. Widget Primitive — Phase 4c (Carousel Retrofit — Audit, Test-Lock, Cleanup)

Retrofit-lock Carousel: audit the live render path, add the Phase 4 test-lock pair, and clean up whatever the audit surfaces. Same playbook as session 216 (BlogListing), applied to the first `SOURCE_WIDGET_CONTENT_TYPE` retrofit candidate. Carousel's contract is already declared and its template already reads `$widgetData['items']`, so the work is verification + test-lock + targeted cleanup, not a true retrofit.

---

## Design decisions (resolved before starting)

- **Audit scope.** Three things must be verified before the test-lock lands:
  1. `CarouselDefinition::dataContract()` returns a `SOURCE_WIDGET_CONTENT_TYPE` contract whose `resourceHandle` resolves correctly through `ContractResolver::resolveWidgetContentType()`. Confirmed live by template line 2 reading `$widgetData['items']`. The audit re-confirms.
  2. `CarouselDefinition::collections()` (legacy slot declaration returning `['slides']`) and the renderer's legacy `$collectionData` loop in `WidgetRenderer::render()` lines 45–51 — establish whether the loop's output is dead under the contract path or whether the `$fallbackCollectionData` half is still load-bearing for demo data when a Carousel instance has an empty live collection.
  3. No direct Eloquent access remains in `app/Widgets/Carousel/` outside the dataContract declaration. Grep `App\\Models\\|::find\(|::where\(|->slides\(|getFirstMedia` inside the widget folder — expect zero matches.
- **Cleanup posture.** The cleanup half of this session is **scoped by the audit**, not pre-decided. Three possible outcomes:
  - (a) `collections()` and the renderer's `$collectionData` half are fully dead → remove `collections()` from `CarouselDefinition`, no renderer change (the loop is generic across widgets, and other widgets may still need it).
  - (b) The fallback path through `$fallbackCollectionData` is still the only way demo data reaches a Carousel with an empty live collection → keep `collections()`, document the dependency in a one-line comment, ensure the test-lock covers the fallback projection shape.
  - (c) Both halves are dead AND the contract path's `resolveWidgetContentType()` already routes through the demo seeder via `Collection`-handle resolution → remove `collections()` and tighten the test-lock to assert no fallback path runs. Less likely but possible if the demo seeder seeds a real `Collection` row that the resolver finds directly.
  - The session must not pre-commit to which outcome lands. Audit first; cleanup second.
- **Test-lock fields.** Carousel's contract declares `fields: ['title', 'description']` plus the implicit `_media` map keyed by the content type's image fields (`image` is the only one). Test A asserts that undeclared content-data keys do not surface in the rendered HTML or in the per-slide row shape. Test B asserts a single `select` against `collections` (resolving the handle), a single `select` against `collection_items`, and a single eager-load `select` against `media`.
- **Test-lock — JSON-key assertion shape.** Carousel does **not** embed a `<script x-ref="listingData">` JSON blob the way BlogListing/EventsListing do — it renders Swiper slides directly into the DOM. Therefore Test A cannot decode an in-template JSON payload. Two acceptable shapes for the key-level invariant:
  1. Drive the test through `ContractResolver::resolve()` directly (separate from the `WidgetRenderer::render()` HTML assertion) and assert `array_keys($dto['items'][0])` canonically equal `['title', 'description', '_media']`. This is the cleanest direct contract assertion.
  2. Assert the rendered HTML shape: declared fields render where their tokens appear, undeclared field tokens render as empty (the template's regex sweep at line 80 strips unmatched tokens). Sentinel-based assertion against the caption rendering path.
  - Land **both** assertions — the resolver-level assertion is the architectural invariant, the HTML-level assertion is the user-visible regression guard.
- **Sentinel choice.** Per the 217 carry-forward lesson: choose sentinels in collection-item `data` keys that are not derived into any whitelisted field. Carousel's content-type fields are user-authored (`title`, `description`, `image`) — there are no derivations like BlogListing's `excerpt` from `meta_description`. Any non-declared `data` key (e.g. `internal_notes`, `legacy_id`) is a clean sentinel.
- **Demo-seeder data.** Carousel's `DemoSeeder` is read but not modified. If the audit reveals the seeder is the test-lock's source of demo fallback data, the test creates a real `Collection` + `CollectionItem` rows directly via factory rather than running the seeder, so the test is self-contained and seeder-change-resilient.
- **Public-surface gate.** `ContractResolver::resolveWidgetContentType()` filters by `is_public = true` when `$context->publicSurface` is true (the canvas slot's default). Test B exercises the canvas-slot path — the test collection must be created with `is_public => true, is_active => true`. Document this in the test fixture for future-readers.
- **No new schema.** No migration, no `docs/schema/` updates.
- **Front-end builds.** No widget asset files (`assets()` only declares `swiper` lib) are modified by this session unless cleanup surfaces an unexpected need. Default expectation: no `npm run build`, no `build:public`. If either becomes necessary, run per the base prompt's decision rule.
- **Out of scope: presentational HTML in the projector.** Carousel's `caption_template` is rendered token-by-token inside the template. There is no projector-side composition to add (unlike EventsListing's `location` and `is_free`). The `_media` map is the only projector-side computation, and it already exists in `WidgetContentTypeProjector::projectRow()`. Do not add anything.

## Phase 1 — Audit

1. Re-read `app/Widgets/Carousel/CarouselDefinition.php` and confirm the `dataContract()` declaration. Compare its content-type `image` field key against the template's `$slide['_media'][$imageField]` access at template line 66 — same key.
2. Read `app/Widgets/Carousel/template.blade.php` end-to-end. Confirm zero direct Eloquent access. Confirm the only data path is `$widgetData['items']`. Confirm the `_media` lookup pattern at lines 65–71 matches what `WidgetContentTypeProjector::projectRow()` produces.
3. Read `app/Widgets/Carousel/DemoSeeder.php`. Determine: does it write to a `collections` row + `collection_items` rows that the contract resolver will find on its own? Or does it stage data for the legacy `WidgetDataResolver::resolve('slides', ...)` path the renderer's `collections` loop uses?
4. Trace one Carousel render through `WidgetRenderer::render()`:
   - Line 46 enters `foreach ($widgetType->collections ?? [])` — Carousel's `collections()` returns `['slides']`, so this loop iterates once.
   - Line 49 calls `WidgetDataResolver::resolve('slides', $queryConfig)` (or whatever `collection_handle` resolves to). Establish what this returns and where the result flows.
   - Line 50 stores `$collectionData['slides'] = $resolved ?: ($fallbackCollectionData['slides'] ?? [])`.
   - Line 84 calls `app(ContractResolver::class)->resolve([$contract], $slot, $fallbackCollectionData)`. The `$collectionData` half is **not** passed — only `$fallbackCollectionData` is. This is the key clarity: `$collectionData` is computed but not used by Carousel under the contract path.
   - The contract resolver's `resolveWidgetContentType()` looks up the live collection via `resourceHandle` (the `collection_handle` config value), pulls items, projects them. If items are non-empty, `$fallback` is unused. If empty, fallback rows are pulled by handle from `$fallback`.
5. Grep `App\\Models\\|::find\(|::where\(|->slides\(|getFirstMedia` inside `app/Widgets/Carousel/` — expect zero matches except possibly in `DemoSeeder.php` (which is allowed to use Eloquent — it's a seeder, not the live render path).
6. Grep `Carousel|use App\\Widgets\\Carousel` across the whole repo — log every callsite to confirm nothing outside the widget folder reaches into Carousel-specific helpers.
7. Record findings in the session log under "Audit results" — the cleanup phase pivots on what this audit produces.

## Phase 2 — Test-lock

Create `tests/Feature/CarouselContractRetrofitTest.php` mirroring the BlogListing/EventsListing shape. Two tests:

### Test A — Fail-closed field whitelist + key-level invariant

Set up:
- Create a `Collection` row with `is_public: true, is_active: true, source_type: 'manual'`, handle `'audit-slides'`.
- Create three `CollectionItem` rows under it with `data => ['title' => 'Slide N', 'description' => 'Caption N', 'internal_notes' => 'NOTES_NOTLEAKED_SENTINEL', 'legacy_id' => 'LEGACY_ID_SENTINEL']`. No media attached for this test (the `_media` map will project as `['image' => null]`).
- Mount a Carousel widget with `config => ['collection_handle' => 'audit-slides', 'image_field' => 'image', 'caption_template' => '{{item.title}} | {{item.description}} | {{item.internal_notes}} | {{item.legacy_id}}']`.

Assertions:
- Direct `ContractResolver::resolve()` call: `array_keys($dto['items'][0])` canonically equals `['title', 'description', '_media']`. Items 0–2 each contain those three keys and no others.
- Rendered HTML contains the title and description for each slide.
- Rendered HTML does NOT contain `'NOTES_NOTLEAKED_SENTINEL'` or `'LEGACY_ID_SENTINEL'`.
- Rendered HTML's caption block contains the title and description tokens substituted, but the sentinel-token positions render as empty (the `preg_replace('/\{\{[^}]+\}\}/', '', $caption)` sweep at template line 80 strips unmatched tokens).

### Test B — Resolver is the only path, with eager-load query-count pin

Set up:
- Same fixture as Test A but with media attached to each `CollectionItem`'s `image` collection (use `addMedia()` against a fixture image — or, if media setup is expensive, skip media and assert the eager-load `select` count as 0 instead of 1, which is honest about the no-media case).
- Three live published items + one unpublished item in the same collection — the unpublished item must not render.

Assertions:
- HTML contains exactly 3 `<div class="swiper-slide">` entries.
- HTML does not contain the unpublished item's title.
- `DB::enableQueryLog()` shows: exactly one `select` against `collections` (the `where handle = 'audit-slides' and is_active = true and is_public = true` query), exactly one `select` against `collection_items` (the `where collection_id = X and is_published = true order by sort_order` query), and the eager `media` select count matches the media-attached items count (1 if present, 0 if absent).

Both tests go in the fast suite (under 5s each).

## Phase 3 — Cleanup

Driven by the audit's findings.

- **If the audit confirms `$collectionData` is dead and `$fallbackCollectionData` is still the live demo-data path:** keep `CarouselDefinition::collections()`, document the rationale with a one-line comment in the definition, and add a test-lock assertion in Test A that confirms the fallback projection round-trips through `WidgetContentTypeProjector::projectFallback()` shaped identically to the live-item projection.
- **If the audit confirms both halves are dead** (the demo seeder seeds a real `Collection` the resolver finds directly): remove `CarouselDefinition::collections()`. Run the fast suite to confirm nothing depends on the slot key.
- **If the audit reveals an unexpected dependency** (e.g. another widget or service references Carousel's `'slides'` slot key): stop and ask the user before changing anything.
- Empty diff is a valid outcome — same as 216's BlogListing cleanup.

## Out of scope

- **Adding new fields to Carousel's content type.** The current `title` / `description` / `image` triple is the contract; widening it is a feature session, not a retrofit.
- **Migrating Carousel's caption-template substitution into the projector.** Caption rendering is presentational HTML composed in the template — same posture as EventsListing's `price_badge`. Stays in the template.
- **Changing the legacy `WidgetDataResolver` loop in `WidgetRenderer::render()`** (lines 45–51) or any service-layer architecture. The cleanup is widget-local. The legacy loop's eventual removal is a separate session, conditional on every collection-using widget being retrofit-locked first.
- **Retrofitting other widgets** — BarChart, BoardMembers, LogoGarden are the next genuine `SOURCE_WIDGET_CONTENT_TYPE` retrofit candidates. Each gets its own Phase 4 session.
