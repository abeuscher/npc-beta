## What was built

### Phase 1 — Routing prefix consolidation

- `blog_prefix` moved out of `CmsSettingsPage` into a new **Routing** fieldset in `GeneralSettingsPage`, alongside the existing `portal_prefix` and a new `events_prefix` field.
- All three prefix fields are now in one place with a section-level warning: *"Changing these prefixes will break existing bookmarked links and any external links published elsewhere. Old URLs will return 404 unless redirects are configured."*
- `CmsSettingsPage`: `blog_prefix` field and all associated slug-migration logic removed. `Artisan::call('config:clear')` removed (now called from `GeneralSettingsPage`).
- `config/site.php`: both `blog_prefix` and `events_prefix` now read from `SiteSetting` at runtime (with a `try/catch` fallback to env vars), so route prefixes stay in sync without requiring a config cache rebuild after each change.
- `GeneralSettingsPage::save()` migrates page slugs automatically when either prefix changes — same logic as previously lived in `CmsSettingsPage` for `blog_prefix`, extended to cover `events` type pages for `events_prefix`.
- `BlogPrefixValidationTest` updated to target `GeneralSettingsPage`; all 6 tests pass.

### Phase 1 addition — `manage_routing_prefixes` permission

- A new standalone permission `manage_routing_prefixes` added to `RoleResource::standalonePermissions()`.
- `GeneralSettingsPage::canAccess()` updated to allow users with this permission to access the page (not just `super_admin`).
- The Site, Admin Panel, and Integrations sections are hidden for non-super-admin users; only the Routing fieldset is visible to permission holders.
- `save()` guards non-routing settings behind an `isSuperAdmin` check so partial-access users cannot overwrite values they cannot see.

### Phase 2 — Page type locking and editor labeling

- **Slug field** in `PageResource`: extended to show the system-assigned prefix as a locked read-only prefix for `event` and `post` types (in addition to the existing `member` handling). All three prefix values are read from `SiteSetting` at render time. The user edits only the slug segment after the prefix; `dehydrateStateUsing` reconstructs the full slug on save. Uniqueness is validated against the full reconstructed slug via a custom rule closure.
- **Type field on edit**: the `type` Select was previously hidden on edit with `->hiddenOn('edit')`. This session originally added a `type_label` Placeholder to show the type on edit — but this was removed mid-session as redundant (the page title already announces the type). The underlying locking behaviour (type cannot be changed after creation for event/post/member) is enforced by the hidden Select on create and no type-change UI on edit.
- **Editor headings** updated per spec:
  - `EditPage::getTitle()`: "Edit Page" / "Edit Post" / "Edit Event Landing Page" / "Edit Member Page"
  - `CreatePage::getTitle()` added: "New Page" / "New Member Page" (event/post not creatable through this UI)

### Phase 3 — Signup and login widgets

- Two new widget types seeded via `WidgetTypeSeeder`:
  - `portal_signup` (label: "Member Signup Form") — `render_mode: server`, no config schema
  - `portal_login` (label: "Member Login Form") — `render_mode: server`, no config schema
- `resources/views/widgets/portal-signup.blade.php`: inline signup form with honeypot (`_hp_name`) and timing (`_form_start`) fields; posts to `portal.signup.post`; on success redirects to `portal.verification.notice` as normal.
- `resources/views/widgets/portal-login.blade.php`: inline login form with remember-me; posts to `portal.login.post`; on success redirects via `redirect()->intended()` to `portal.account`.
- Field IDs namespaced (`sw_*`, `lw_*`) to avoid collisions when both widgets appear on the same page.

### Phase 4 — Help docs

- `resources/docs/settings-general.md`: updated to note the Routing Prefixes section and the `manage_routing_prefixes` permission.

## Mid-session corrections and bug fixes

### PostResource layout and slug field

`PostResource` previously had a flat single-column layout with no sidebar. Restructured to match the `PageResource` / `ContactResource` 3-column pattern:
- Left area (`columnSpan(2)`): title/slug section, Page Builder, SEO, Custom Fields.
- Right sidebar (`columnSpan(1)`): "Publication" section (published toggle, publish date, tags).

The slug field was also updated to show the `/{blog_prefix}/` prefix as a locked read-only prefix (same pattern as PageResource for `post` type), with `formatStateUsing` / `dehydrateStateUsing` / custom uniqueness rule.

The blank `_slug_placeholder` hack (used to maintain grid alignment on create) was removed.

### EventResource layout

- Slug field moved from the Settings sidebar into the Main Info section, to the right of the title field. `->columns(2)` added to Main Info; `description` (QuillEditor) given `->columnSpanFull()` so it remains full-width; Start/End dates pair naturally in the two columns. No prefix is shown — the event slug is the event's own identifier, not a page slug.
- The `landing_page_status` Placeholder (which showed "Landing page" label, the page title as content, and two icon hint-actions) replaced with a `Forms\Components\Actions` component containing just the two action buttons ("Edit landing page", "View public page") — no surrounding label or content text.

### Page builder drag-to-reorder bug fixed

Drag-to-reorder had silently failed since the sort system was written. Root cause: the `x-sort` callback in `page-builder.blade.php` used `getAttribute('x-sort:item')` to read block IDs, but that attribute holds the raw Alpine expression `'uuid'` (with surrounding single quotes), not the bare UUID. The `PageWidget::where('id', $id)` query never matched.

Fix:
- Added `data-block-id="{{ $block['id'] }}"` to the block root element in `page-builder-block.blade.php`.
- Updated the `x-sort` callback in `page-builder.blade.php` to query `[data-block-id]` instead of `[x-sort:item]` — this attribute contains the bare UUID with no wrapping characters.
