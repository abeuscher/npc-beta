# 309. Content Import/Export — Opt-in Theme + Interactive Import Prompts

> **Architectural fix** to a write-conflict between the Theme/Typography editor and the page-import flow, surfaced during session 308 typography debugging. The current bundle envelope can carry a `design` payload (`theme_colors` / `typography` / `button_styles`) that `ContentImporter::importDesign()` silently deep-merges into `SiteSetting` rows on every import — a second write path into the same DB rows the Theme editor writes to. The user has no way to see or refuse it. Same applies, less acutely, to media inclusion and to duplicate-slug pages: existing branches in `ContentImporter::import()` that fire implicitly with no user-visible confirmation.

One sentence: turn the implicit `design` / `media` / duplicate-page passes into user-controlled options on both the export form and the import form, with a pre-flight analyzer step so the import UI knows which questions to ask.

This is the first session of a five-piece arc to clean up the content import/export surface end-to-end. After this session lands: session 310 unifies the UI under a single "Import Site / Export Site" affordance; session A001 (agentic) extends the same envelope to events, products, navigation, and collections.

---

## What's wrong today

- `ContentExporter::exportBundle()` produces a combined `design + templates + pages` payload; whether design ships in a page-focused export is not user-controlled at the export surface.
- `ContentImporter::import()` runs `importDesign()` first whenever `payload.design` is present (lines 53–58). Deep-merges into `SiteSetting::{theme_colors, typography, button_styles}` and triggers `AssetBuildService::build()` on the way out. No user gate.
- `payload.media` import behaves the same — runs implicitly when present.
- Duplicate-slug pages: overwrite or duplicate depending on importer branch, no warning either way.

Net symptom (observed 308): a user who configured their theme via the Theme editor and later imported a stale bundle lost their theme edits silently. The Theme editor and the import path both write to the same SiteSetting rows; import wins.

## What we want

### Export side

Page-export form gains THREE options instead of the current two:

1. **Content + layout only** (DEFAULT — no `design`, no `media` in the envelope).
2. **Content + layout + media** (existing "with media" preset).
3. **Content + layout + theme** (NEW — includes `design`, surfaces a "will override inbound site theme on import" warning in the form copy and in the bundle's envelope header).

Optionally a fourth option: **Full snapshot** (1 + 2 + 3). Defer if scope tight; session 310 will need it for the unified rollup anyway.

`exportDesign()` (theme-only export) stays as-is — it's already an explicit opt-in surface.

### Import side

Two-step flow:

**Step 1 — Upload + analyze.** New method `ContentImporter::analyze(array $bundle): array` returns a manifest the UI uses to decide which questions to ask. Pure inspection, no DB writes:

```
{
  has_design: bool,
  design_keys: [theme_colors, typography, button_styles],   // present subset
  has_media: bool,
  media_count: int,
  pages: [{slug, exists_locally: bool}, ...],
  templates: [{handle, exists_locally: bool}, ...]
}
```

**Step 2 — Confirm + run.** UI surfaces a confirmation form with checkboxes, each gated on whether the corresponding payload is present:

- **(a) Replace duplicate pages** — only shown if `pages[]` includes any `exists_locally: true`. Default OFF. Copy: "N pages in this bundle already exist on this site by slug. Replace them?"
- **(b) Replace site theme** — only shown if `has_design`. Default OFF. Copy: "This bundle includes a theme payload. Importing will merge it over your current Theme settings. Replace?"
- **(c) Include media** — only shown if `has_media`. Default ON. Copy: "Bundle includes N media files. Import?"

Each option becomes an `$opts` flag passed into `ContentImporter::import($bundle, $log, $opts = [])`. Existing branches stay — they become gated on the flag.

## Architecture

- `ContentExporter::exportPages($pageIds, array $opts = [])` — add `with_design`, `with_media` flags. Default both false.
- `ContentExporter::exportBundle(…, array $opts)` — same flags, plus templates. The existing implicit-include behaviour flips to opt-in.
- `ContentImporter::analyze(array $bundle): array` — new method. No DB writes; pure inspection.
- `ContentImporter::import($bundle, $log, array $opts = [])` — `merge_design`, `replace_duplicate_pages`, `import_media` flags. Each gates an existing branch. **Default behaviour flips: `merge_design` defaults to FALSE** so old callers that import bundles with design no longer silently overwrite theme.
- Filament-side: a two-step import page mirroring the bulk-contact importer's "preview then apply" pattern. The export form gains the third radio option.

## Out of scope

- Extending the bundle to events, products, navigation, or collections (session A001, agentic).
- The unified "Import Site / Export Site" UI rollup (session 310).
- Changes to the `deepMergeOverDefaults` semantics inside `importDesign()`. The *how* of the design merge is correct; only the *whether* changes here.
- Bundle envelope `format_version` bump. Keep 1.1.0 — every change is additive (new opt flags) or restrictive (default flips to off).
- Multi-tenant snapshot considerations.

## Phases

1. **Exporter splitter.** Add `$opts` to `exportPages()` / `exportBundle()`. Flip defaults to no-design / no-media. Extend `ContentExporterTest` to cover the three combinations.
2. **Importer analyzer.** New `ContentImporter::analyze()` method. Extend `ContentImporterTest` to cover bundles with/without design, with/without media, and with duplicate-slug pages.
3. **Importer opt flags.** Add `$opts` to `import()`. Each existing branch becomes gated. Default `merge_design` to FALSE. Add Pest cases for each flag combination.
4. **Export UI.** Add the third radio option (and the fourth if scope holds) to the page-export form. Update copy.
5. **Import UI.** Two-step Filament page: upload → call `analyze()` → render confirmation form with gated checkboxes → submit calls `import()` with the chosen `$opts`. One Playwright spec covers the happy path.
