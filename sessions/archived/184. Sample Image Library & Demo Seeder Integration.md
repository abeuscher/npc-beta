# 184. Sample Image Library & Demo Seeder Integration

Formalize a reusable pool of demo imagery in the Spatie media library — sourced from four canonical folders on disk — and wire existing demo seeders and the dev-mode random data generator to draw from it instead of ad-hoc file paths and loremflickr URLs.

---

## Design decisions (resolved before starting)

- **Four canonical source folders** under `resources/sample-images/`, using long-form names:
  - `portraits/`
  - `still-photos/`
  - `logos/`
  - `product-photos/`
- **Pool lives in the Spatie media library.** A new `SampleImage` model (HasMedia) owns four media collections matching the folder names. A bootstrap seeder ingests files from disk into those collections.
- **Bootstrap seeder is idempotent and sync-style.** Re-running it brings the pool into line with the folder contents (adds new files, removes entries for files no longer on disk). This is how the user swaps image sets: drop new files in, re-seed.
- **Demo seeders copy from the pool.** Spatie media rows cannot be shared across owner models, so each widget's CollectionItem still gets its own Media row — sourced from the pool's on-disk path rather than from `resources/sample-images/` directly.
- **DemoDataService returns pool URLs.** The `generateImageUrl()` path in `app/Services/DemoDataService.php` stops returning loremflickr URLs and instead returns a random pool item's `getUrl()`, with a graceful fallback (e.g. a single bundled placeholder or an empty string) if the pool is empty.
- **ProductCarousel does NOT get a demo seeder.** Like EventsListing and BlogListing, ProductCarousel relies on built-in content (the `products` source type). Demo product imagery comes through the random data generator, not an external seeder.
- **Carousel does get a seeder update.** The existing `app/Widgets/Carousel/DemoSeeder.php` currently has no images at all — this session adds still-photo attachments to its slides, drawn from the pool.
- **Empty folders are acceptable to start.** Folders are created with `.gitkeep`. The user will deliver image sets afterward; the pipeline degrades gracefully when a folder is empty.

## Phase 2 — `SampleImage` model & migration

- Migration: `sample_images` table — minimal, just an id, a `category` string, and timestamps. One row per category (portraits, still-photos, logos, product-photos).
- Model `App\Models\SampleImage` implements `HasMedia`, registers four media collections matching the folder names. Images for each category attach to the matching row's matching collection (e.g. the `portraits` row holds its media in the `portraits` collection).
- No admin UI, no Filament resource — this is infrastructure, not user-managed content. The existing Media Library page ([app/Filament/Pages/MediaLibraryPage.php](app/Filament/Pages/MediaLibraryPage.php)) will surface these rows automatically via the Media query; optionally add an "owner type" filter entry for `App\Models\SampleImage` so they can be browsed there.
- Update `docs/schema/README.md` and add `docs/schema/sample_images.md`.

---

## Phase 3 — `SampleImageLibrarySeeder`

- New seeder at `database/seeders/SampleImageLibrarySeeder.php`.
- For each of the four categories: ensure the `SampleImage` row exists, then sync the matching media collection with the files on disk at `resources/sample-images/{category}/`.
  - Files present on disk but not in the collection → attach (preservingOriginal).
  - Files in the collection but no longer on disk → delete that Media row.
  - Files present in both → leave alone.
- Idempotent. Safe to run on `migrate:fresh --seed` and on demand.
- Register in `DatabaseSeeder` so `migrate:fresh --seed` populates the pool before any widget demo seeders run.
- Empty folders are a no-op — seeder logs a note and moves on.

---

## Phase 4 — `SampleImageLibrary` service

- New service `App\Services\SampleImageLibrary` with a minimal API:
  - `random(string $category, int $count = 1): \Illuminate\Support\Collection` — returns Media rows from the matching pool collection, in random order. Returns an empty collection if the pool is empty.
  - `randomUrl(string $category): ?string` — convenience for the DemoDataService: returns one pool Media's public URL, or `null` if empty.
- Categories are constants on the service to keep call sites consistent.

---

## Phase 5 — Wire existing demo seeders to the pool

Update the three image-using demo seeders to pull from `SampleImageLibrary` instead of `glob()` / hardcoded filenames:

- `app/Widgets/LogoGarden/DemoSeeder.php` — replace the `logo-adidas.png` / `logo-amazon.png` / etc. hardcoded list with `SampleImageLibrary::random('logos', $count)`. Keep the existing name list (Adidas / Amazon / …) as the item `data['name']` values, but image attachment no longer depends on specific filenames. If the pool returns fewer images than names, attach what we have and leave the rest without images.
- `app/Widgets/BoardMembers/DemoSeeder.php` — replace the `glob($portraitDir . '/*.jpg')` call with `SampleImageLibrary::random('portraits', 6)`. Same graceful degradation if pool is short.
- `app/Widgets/Carousel/DemoSeeder.php` — add image attachment to each slide using `SampleImageLibrary::random('still-photos', 4)`. Images were not previously attached; this session adds them.

Do not add or modify seeders for ProductCarousel, EventsListing, or BlogListing.

---

## Phase 6 — Wire `DemoDataService` to the pool

- In `app/Services/DemoDataService.php`, update `generateImageUrl()` (currently returns loremflickr URLs) to map the field-key keyword to a pool category and call `SampleImageLibrary::randomUrl($category)`:
  - `portrait` / `headshot` / `photo` / `member` → `portraits`
  - `logo` → `logos`
  - `product` / `item` → `product-photos`
  - anything else (`hero`, `background`, `thumbnail`, default) → `still-photos`
- If `randomUrl()` returns `null` (empty pool), fall back to a bundled static placeholder shipped in `public/` (e.g. `public/images/sample-placeholder.png`) so templates always get a valid URL. Add the placeholder file if one isn't already present.
- Update `generateEventItems()`, `generateBlogPostItems()`, and `generateProductItems()` to use the service too so their `thumbnail_url` / `image_url` fields draw from the pool (`still-photos`, `still-photos`, `product-photos` respectively).
