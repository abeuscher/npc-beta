## What was built

The session opened on three carry-forward bugs from the 332 container-query rollback (C ‚Äî editor Swiper load race; B ‚Äî uneven grid tracks; E ‚Äî possible layout-column drop) plus a standing parity harness. During live diagnosis the owner pointed at a **fourth, concrete editor‚Üîpublic divergence** ‚Äî the page-builder pager rendering completely differently from the public site ‚Äî which turned out to be the real WYSIWYG bug worth fixing. ### B ‚Äî Uneven grid tracks (fixed + verified)

`.content-card` grid items lacked `min-width: 0`, so `grid-template-columns: repeat(N, 1fr)` distorted to content min-width at narrow widths. Reproduced in the browser at viewport 1024 (three "equal" tracks measured `111 / 133 / 100px`). Added `min-width: 0` to `.content-card` in both `app/Widgets/BlogListing/styles.scss` and `app/Widgets/EventsListing/styles.scss`. Re-measured through the real `build:public` pipeline: tracks now even at every narrow width (`103.3 / 103.3 / 103.3px` at 1024; even at 480/375 too).

### The real fix ‚Äî editor‚Üîpublic pager parity (root-caused + fixed + verified)

The owner observed the page-builder pager looking "totally different" from public. Driving both surfaces: the editor rendered **Swiper's default pagination** (tiny round blue dots, numbers invisible) while public rendered the **designed numbered pill pager** (`1 2 3 4 5`, brand-blue active). Confirmed by computed style (editor bullet: 8px tall, 50% radius, `#007AFF`, `display:block` vs public: ~27px, 6px radius, brand `#0172AD`, `display:flex`) and the live CSSOM.

**Root cause:** session 332 put the public bundle's vendor (swiper) CSS into `@layer reset` and widget CSS into `@layer widgets`, so the widget pager wins by layer order. The **editor** loaded swiper's CSS as a **bare unlayered `<link>`** (emitted by `AdminPanelProvider`'s `panels::head.end` hook), and **unlayered CSS beats every layer regardless of specificity** ‚Äî so Swiper's defaults won in the editor only. Verified the mechanism via the CSSOM (editor swiper rules `(unlayered)`, public `[reset]`) and proved the fix live by re-injecting swiper CSS wrapped in `@layer reset` (bullet flipped to the designed pager).

**Fix:** emit the vendor lib CSS into `@layer reset`, mirroring public ‚Äî
- `app/Providers/Filament/AdminPanelProvider.php`: the head hook now emits `<style data-widget-lib="‚Ä¶">@import url("‚Ä¶") layer(reset);</style>` instead of a bare `<link>`.
- `resources/js/page-builder-vue/composables/useLibraryLoader.ts`: `loadLibs` now fetches the vendor CSS and injects it wrapped in `@layer reset` (consistent dedup marker).

Re-driven through the real `npm run build`: the editor now shows the same numbered pager as public, the unlayered `<link>` is gone. One residual divergence was surfaced and **owner-accepted as out of scope**: the pager nav-arrow border differs (editor `0px` vs public `1px`) because Filament's **unlayered Tailwind preflight** (`*{border-width:0}`) leaks into the canvas and beats `@layer widgets` ‚Äî a host-cascade problem that can't be fixed from inside a layer (logged as the architecture stub, below).

### C ‚Äî Editor Swiper load race (dispositioned: stale-build artifact; defensive guards added)

Could **not** reproduce the intermittent throw despite throttling the swiper lib 0‚Üí3000ms across full-reload **and** `wire:navigate` SPA entry, on both a guarded widget (blog_listing) and an unguarded one (a throwaway logo_garden carousel test page) ‚Äî the editor initialized Swiper on every run. The current load path is correctly gated on `loadLibs`. The owner agreed the 332 observation was a **stale-build artifact** and also could no longer recreate it. Kept the cheap **defense-in-depth guards** anyway (the unguarded `window.SwiperModules.*` accesses are a genuine latent crash): the five Swiper widgets now bail if `window.Swiper`/`window.SwiperModules` are absent ‚Äî
- `BlogListing/script.js` + `EventsListing/script.js`: `init()` and the x-effect-driven `rebuildSlides()` (which read `SwiperModules` unguarded).
- `ProductCarousel/template.blade.php`, `LogoGarden/template.blade.php` (carousel + smooth), `Carousel/template.blade.php`: `init()` (previously no guard at all).

No behavior change on the happy path (public re-verified: blog_listing initializes, 5 bullets, zero errors).

### E ‚Äî Possible layout-column drop (did not reproduce; reported)

One bounded empirical attempt: a 3-column layout with a blog_listing in the middle column, change the widget's `config.columns` 3‚Üí2 through the real editor save endpoint. Result: `page_layouts.columns` stayed **3** ‚Äî no bleed. Confirmed by code: the widget `update` controller writes only widget fields; layout columns are a separate inspector + endpoint with no shared key. The Welcome-page "3 Column Layout / columns=2" mismatch is a **stale free-text label** (the label is stamped once at creation as `${n} Column Layout`; reducing the count later updates `columns` but not the label) ‚Äî cosmetic, not a dropped column. No chase, per owner guidance.

### Standing parity harness (Phase 3+4)

A Playwright editor-vs-public behaviour/computed-style harness is **not viable on the isolated e2e stack**: that stack deliberately does not run the widget build pipeline (no build server in CI), so the swiper lib + widget CSS the bug depends on are never produced there (and 308's harness was torn down for related reasons). Instead added a **build-pipeline-independent Pest guard** that locks the exact regression at the render layer, in `tests/Feature/WidgetJsLibsSession138Test.php` (already in the `design` group, so no `DesignGroupIntegrityTest` churn): it binds a fixture manifest, renders the real admin head, and asserts the vendor CSS is emitted into `@layer reset` (`@import ‚Ä¶ layer(reset)`) and **never** as an unlayered `<link>`. Rationale documented in the test.
