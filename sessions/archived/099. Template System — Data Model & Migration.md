# 099. Template System — Data Model & Migration

Build the data model for the template system and migrate existing Site Theme functionality into it. Templates unify page framing (header, footer, colors, fonts, SCSS) into a single portable record. One template is marked as the default; non-default templates inherit from the default on a per-field basis.

---

## Design decisions (resolved before starting)

- A `templates` table holds everything the current Site Theme manages: colors, fonts, custom SCSS, header system page, footer system page. One record is marked `is_default`.
- Every overridable field on a non-default template is independently inheritable. Colors, fonts, SCSS, header, footer — each can be "inherit from default" (null) or overridden with a custom value. The default template has no inherit concept — it is the source of truth.
- The existing Site Theme page (`SiteThemePage`) is retired. Its functionality moves into the Templates view built in session 100. This session migrates the data; session 100 builds the UI.
- Content templates (saved widget stacks applied at page creation) are a separate concept stored in the same `templates` table with a `type` discriminator: `page` for header/footer/styling templates, `content` for widget stack presets.
- Event landing pages become mandatory — the opt-out is removed. EventObserver always creates a landing page on event creation.
- The `pages` table gains a nullable `template_id` FK pointing to the `templates` table. Null means "use the default template." Pages created going forward are assigned the default template explicitly.

## Phase 1 — Templates table and migration

### Migration: `create_templates_table`

| Column | Type | Notes |
|---|---|---|
| id | uuid | PK |
| name | string | e.g. "Default", "Landing Page", "Minimal" |
| type | string | `page` or `content` |
| description | text, nullable | |
| is_default | boolean | default: false; exactly one `page`-type template may be default |
| definition | jsonb | default: {}; for `content` templates: array of widget specs `[{handle, config, sort_order}]`. For `page` templates: unused (header/footer are system pages) |
| primary_color | string, nullable | null = inherit from default |
| heading_font | string, nullable | null = inherit |
| body_font | string, nullable | null = inherit |
| header_bg_color | string, nullable | null = inherit |
| footer_bg_color | string, nullable | null = inherit |
| nav_link_color | string, nullable | null = inherit |
| nav_hover_color | string, nullable | null = inherit |
| nav_active_color | string, nullable | null = inherit |
| custom_scss | text, nullable | null = inherit |
| header_page_id | uuid, nullable | FK → pages (system page); null = inherit default's header |
| footer_page_id | uuid, nullable | FK → pages (system page); null = inherit default's footer |
| created_by | bigint unsigned, nullable | FK → users |
| created_at | timestamp | |
| updated_at | timestamp | |

### Model: `App\Models\Template`

- Casts: `definition` as array, `is_default` as boolean.
- Method: `resolved(string $field)` — returns the template's own value if non-null, otherwise falls back to the default template's value. Used by the renderer to resolve inherited fields.
- Scope: `scopeDefault` — returns the template where `is_default = true` and `type = 'page'`.
- Scope: `scopePage` — filters to `type = 'page'`.
- Scope: `scopeContent` — filters to `type = 'content'`.

### Migration: add `template_id` to `pages`

Add a nullable `template_id` (uuid, FK → templates, nullOnDelete) to the `pages` table.

### Data migration: seed default template from current site settings

Write a seeder (`TemplateSeeder`) that:

1. Creates a "Default" template (`type = 'page'`, `is_default = true`) populated with current `site_settings` values: `public_primary_color`, `public_heading_font`, `public_body_font`, `header_bg_color`, `footer_bg_color`, `nav_link_color`, `nav_hover_color`, `nav_active_color`.
2. Links the existing `_header` and `_footer` system pages as the default template's header and footer.
3. Copies the current `theme_scss` site setting value into the default template's `custom_scss`.

The site_settings keys remain in place for now (they still work as fallbacks during the transition). Session 100 will clean them up after the UI is fully migrated.

## Phase 2 — Content template presets

Seed built-in content templates (`type = 'content'`, `is_default = false`) with widget stack definitions. These use the `definition` JSONB field — an array of `{handle, config, sort_order}` objects referencing widget type handles.

### Built-in content templates to seed:

- **Contact Page** — text_block (heading: "Contact Us") + web_form
- **About Page** — text_block (heading: "About Us") + image + text_block
- **Event Landing Page** — event_description + event_registration (formalizes what EventObserver currently creates)
- **Blog Post** — text_block (main content area)
- **Blank** — no widgets (already the implicit default, but make it explicit as a named option)

---

## Phase 3 — Mandatory event landing pages

- Remove the landing page opt-out. EventObserver always creates a landing page when an event is created.
- Refactor EventObserver to use the "Event Landing Page" content template from Phase 2 instead of hardcoded widget creation. Look up the template by a known handle or name, read its `definition`, and hydrate the widgets from it.
- If the content template doesn't exist (e.g. unseeded database), fall back to the current hardcoded behavior so nothing breaks.

---

## Phase 4 — Renderer integration

Update `PageController` and `ChromeRenderer` to resolve template values for the current page:

1. When rendering a page, check `$page->template_id`. If set, load the template. If null, load the default template.
2. The template's `resolved()` method provides colors, fonts, SCSS, header page, and footer page — falling back to the default template for any null fields.
3. Replace the current `SiteSetting::get()` calls in `public.blade.php` for colors/fonts with template-resolved values.
4. Replace the hardcoded `_header`/`_footer` slugs in the layout with the template's resolved header/footer page IDs.
5. SCSS compilation: the template's resolved `custom_scss` replaces the current `theme_scss` site setting in the compilation flow.

This phase makes the rendering pipeline template-aware without changing any UI. The existing Site Theme page continues to work (it still writes to site_settings, which the seeder copied into the default template). Session 100 will retire it.

## Security checklist

Before closing:

- [ ] Template `resolved()` never exposes another template's data — only the page's own template or the default.
- [ ] Content template definitions reference widget type handles (strings), never IDs — no information leakage.
- [ ] `template_id` on pages is not mass-assignable from public routes.
- [ ] Event landing page creation is gated to authenticated admin actions (EventObserver context), not public endpoints.
