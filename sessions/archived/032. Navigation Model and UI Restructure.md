# 032. Navigation Model and UI Restructure

Rebuild the navigation management UI from a flat item list to a menu-first interface with drag-and-drop ordering. The current `NavigationItemResource` is replaced entirely by a `NavigationMenuResource` that presents menus as first-class objects, with their links managed inline via a reorderable Repeater.

---

## Background & decisions

Session 031 added `menu_handle` as a freeform string on `navigation_items` and introduced `NavigationItemResource` with a standard Filament table. During testing it became clear that:

- A flat list of items with a numeric `sort_order` field is not usable for editors. They cannot visualise order or hierarchy from a table.
- The freeform handle approach has no referential integrity — a typo in a handle silently creates an orphan set of items.
- The right mental model for editors is *menu → links*, not *link → menu assignment*.

The architecture decision for this session:

- **Introduce `NavigationMenu` as a first-class model.** A menu has a `label` (human name) and a `handle` (machine name, unique). Items belong to a menu via FK.
- **Replace `NavigationItemResource` entirely.** Items are managed inline within the menu edit form. They are never edited in a standalone table.
- **Use a Filament Repeater with `reorderable()` for ordering.** Editors drag to reorder. `sort_order` is maintained by the system from DOM position on save — editors never see or type it.
- **One level of children only.** Each top-level link may have child links. Children are also reorderable within their parent. No grandchildren. This is enforced by the UI (the child Repeater has no nested Repeater of its own).
- **No multiple-menu assignment on a single link.** A `NavigationItem` belongs to exactly one `NavigationMenu` via FK. This is structural, not a UI rule.

## Data model changes

### New table: `navigation_menus`

| column | type | notes |
|--------|------|-------|
| `id` | uuid, PK | |
| `label` | string | Human name, e.g. "Primary Navigation" |
| `handle` | string, unique | Machine name, e.g. `primary` |
| `timestamps` | | |

### Changes to `navigation_items`

- Add `navigation_menu_id` (uuid, FK → `navigation_menus.id`, cascade delete)
- Drop `menu_handle` string column
- `parent_id`, `sort_order`, and all other columns unchanged

### Migration strategy (single migration)

1. Create `navigation_menus` table.
2. Insert one `NavigationMenu` row for each distinct `menu_handle` value currently in `navigation_items`, using the handle as both `handle` and `label` (editors can rename from the UI afterward).
3. Add `navigation_menu_id` (nullable initially) to `navigation_items`.
4. Populate `navigation_menu_id` by joining on `menu_handle`.
5. Make `navigation_menu_id` non-nullable.
6. Drop `menu_handle` from `navigation_items`.

## What to build

### 1. `NavigationMenu` model

- `HasUuids`
- `$fillable`: `label`, `handle`
- Relationship: `hasMany(NavigationItem::class)`

### 2. Update `NavigationItem` model

- Replace `menu_handle` in `$fillable` with `navigation_menu_id`
- Add `belongsTo(NavigationMenu::class)` relationship

### 3. Remove `NavigationItemResource`

Delete `app/Filament/Resources/NavigationItemResource.php` and its `Pages/` subdirectory entirely. It is superseded by the inline Repeater inside `NavigationMenuResource`.

### 4. `NavigationMenuResource`

**List page** — standard Filament table with columns:
- `label` (sortable, searchable)
- `handle` (sortable)
- Item count (computed: `items_count` via `withCount`)

Actions: Edit, Delete (with confirmation). Bulk delete.

**Create / Edit form** — two sections:

**Section 1 — Menu details** (columns: 2):
- `TextInput::make('label')` — required
- `TextInput::make('handle')` — required, unique (ignoring current record on edit), alpha-dash, helper text explaining it is used in templates and settings

**Section 2 — Links** — a `Repeater` of top-level navigation items, reorderable via drag handle.

Each top-level item schema:
- `Select::make('link_type')` → `['page' => 'Internal page', 'url' => 'URL']`, default `page`, `->live()`
- `Select::make('internal_page')` — same grouped options as the current `internalPageOptions()` (Pages / Events / Blog), visible when `link_type === 'page'`, searchable
- `TextInput::make('url')` — visible when `link_type === 'url'`
- `TextInput::make('label')` — required
- `Toggle::make('open_in_new_window')` — default false
- `Toggle::make('is_visible')` — default true

Within each top-level item, a nested `Repeater::make('children')` with the same schema as above (minus the children Repeater itself — no grandchildren). Also `reorderable()`. Collapsed by default, with an `addActionLabel` of "Add child link".

**State management:** The Repeater cannot use `->relationship()` for this nested structure because children require `parent_id` to be set to the parent item's ID, which is not yet known for new items at form-fill time. Instead:

- In the resource's `Edit` and `Create` page `mutateFormDataBeforeFill()` / `mutateFormDataBeforeSave()` hooks, manually marshal the data:
  - **Fill:** load the menu's top-level items (ordered by `sort_order`), each with their children (ordered by `sort_order`), and map to the Repeater array structure.
  - **Save:** delete all existing `NavigationItem` rows for this menu, then re-insert from the Repeater array: top-level items first (with `sort_order` from array index), then children (with `parent_id` set to the freshly-inserted parent's ID and `sort_order` from child array index).

The link_type / internal_page virtual fields follow the same resolution logic as the existing `resolveFormData()` in `NavigationItemResource` — decode the `page:UUID` key to `page_id`, set `url` to null for internal pages and vice versa. Move this logic into a private static method on `NavigationMenuResource`.

### 5. Update Blade components

`resources/views/components/site-header.blade.php` and `site-footer.blade.php` currently query `NavigationItem::where('menu_handle', $handle)`. Change to:

```php
$menu = \App\Models\NavigationMenu::where('handle', $handle)->first();
$navItems = $menu
    ? $menu->items()
        ->where('is_visible', true)
        ->whereNull('parent_id')
        ->orderBy('sort_order')
        ->with([...])
        ->get()
    : collect();
```

The rest of each component is unchanged.

### 6. CMS Settings — no change required

`header_nav_handle` and `footer_nav_handle` are stored as handle strings in `site_settings`. The lookup now goes through `NavigationMenu::where('handle', ...)` rather than directly on items. The settings page form and save logic are unchanged.

## Out of scope

- More than one level of children
- Drag-and-drop between menus
- Preview of the rendered nav within the admin
- Any change to how handles are used in templates
