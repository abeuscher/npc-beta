## What was built

Demo-site build session **B**. Additive upgrade to the `EventsListing` widget plus a `sold_out` event flag. The scope evolved materially through owner iteration: the single "day-grouped list mode" the prompt described became **two orthogonal layout toggles**, and the featured hero + event-type filter became **layout-independent** widget features.

### Phase 1 ‚Äî Layout: two orthogonal toggles (not one "list mode")

- **`side_by_side_rows`** (toggle) ‚Äî renders each event as a thumbnail-left / details-right row instead of the grid/carousel cards.
- **`group_by_day`** (toggle) ‚Äî groups events under day headings.
- Independent: rows work without grouping, grouping works with cards or rows, both off = the existing grid/carousel (unchanged on every existing page). Implemented as a single static-render path (one "group" with no heading when not grouping) beside the untouched Alpine/Swiper carousel.
- **`columns`** now drives layout in *every* mode ‚Äî the card grids *and* the side-by-side row grid (inline `grid-template-columns`).
- **Styleable day headings** ‚Äî a rich-text `day_heading_template` with **namespaced `{{day.*}}` tokens** (`{{day.weekday}}`, `{{day.weekday_short}}`, `{{day.month}}`, `{{day.number}}`, `{{day.year}}`, `{{day.date}}`). Namespacing was required: bare tokens like `{{date}}` collide with `WidgetRenderer`'s page-context token substitution (which runs over richtext config) and get consumed before the widget sees them ‚Äî `{{day.*}}` survives, like the card template's `{{item.*}}`.

### Phase 2 ‚Äî Featured hero + event-type filter (layout-independent)

- **Featured event** picker (`featured_event_slug`, `options_from: events`) ‚Äî renders an image-left / details-right hero in **all** layouts (carousel included), matched from the already-fetched items (no extra query) and pulled out of the listing so it isn't shown twice.
- **Event-type filter** ‚Äî a dropdown (`show_event_type_filter` toggle) in the same controls row as search, built as a client-side facet from the **tags** on the listed events (no schema; uses the existing tag system). Carousel filtering rides the Alpine component (`typeFilter` added to `getFilteredIndices`); static layouts use a vanilla `<select>` handler that hides non-matching events and collapses empty day groups. Hidden automatically when no listed event is tagged.

### Phase 3 ‚Äî Art fallback, badges, schema, states; honest controls

- **Art-fallback chain** ‚Äî event thumbnail ‚Üí header image ‚Üí solid-colour date tile (token background). The generated text-tile tier was cut per ¬ß1a (the demo owns its seed art).
- **Status badges** ‚Äî Free + Sold Out (the cheaply-available ones).
- **`ItemList` / `Event` JSON-LD** emitted inline; canonical + OG are page-level (the public layout already sets them from page SEO).
- **Empty / sparse states** rendered deliberately; `data-tour` anchors `events-index.list` / `events-index.featured` / `events-index.filters`.
- **No exposed dead controls** ‚Äî the carousel-only fields (transition, slide spacing, items-per-page, default sort, show-search) now hide whenever a layout toggle is on, and the card template hides in row layout. This needed a small, general inspector enhancement: `shown_when`/`hidden_when` now accept an **array** ("hide if any of these is on") in `InspectorField.vue`.
- **Two layout bugs fixed during browser verification:** list rows were getting bullets from the global `.np-site ul:not(nav ul) li` prose-marker rule (fixed by dropping the rows out of `display: list-item`, mirroring the mini-calendar); and the filter dropdown was wrapping below search because the global `select { width: 100% }` out-specified it (fixed with an explicit `flex-basis`). Also: the featured hero now carries a bottom margin in all layouts (it only had one in list mode, so it collided with the search box in the carousel layout).

### `sold_out` event flag

- New **`events.sold_out`** boolean column (default false); model fillable + boolean cast; operator **toggle on the event form** (Registration Details).
- **Closes registration server-side, not just visually:** the `EventRegistration` widget shows "This event is sold out. Registration is closed.", and **both** registration controllers (`EventController` + `Portal\EventRegistrationController`) reject the POST with "This event is sold out." A demo of the close-registrations workflow without per-event capacity bookkeeping.
- `sold_out` + a projected `tags` array added to `SystemModelProjector::projectEvent`; the listing query eager-loads `tags`. Excluded from the events importer (`EventFieldRegistry`, alongside `registration_mode`) ‚Äî it's an operational toggle, not bulk-import data.

### Enabling / infra

- **`InspectorField.vue`** ‚Äî array-form `shown_when`/`hidden_when` (general; load-bearing for the EventsListing field gating). Rebuilt the Vite admin bundle.
- **`docker-compose.e2e.yml`** ‚Äî isolated `bootstrap/cache` volumes for the e2e `app` + `worker`. A local checkout's dev-built package manifest (lists Debugbar) was being bind-mounted into the `--no-dev` e2e image, breaking every artisan command in the e2e stack; the named volumes let each container regenerate the manifest against its own vendor. Makes local Playwright runs viable again; harmless in CI (fresh checkout ‚Üí empty volume ‚Üí regenerated).

## Security considerations

The `sold_out` registration block is enforced **server-side in both the public and portal registration controllers**, not only in the widget template ‚Äî a sold-out event rejects the POST regardless of the rendered form. The flag is **operator-set only**; there is no automatic capacity-driven flip (deliberately deferred ‚Äî see Deferred decisions). The listing work added no routes, policies, or auth surfaces (JSON-LD is inline, OG/canonical are the page's existing SEO). Non-boundary: no Fleet Manager surface, CRM contract unchanged at v2.3.0.

## Production deployment notes

- **Run `build:public`** on deploy ‚Äî widget SCSS/JS changed, regenerating the public widget bundle + `public/build/widgets/manifest.json`.

A deploy must run the seeder/sync or the new controls won't appear in the builder. (This is the standing deploy-time widget-schema-sync gap already noted in the housekeeping inbox; it bit during this session's manual testing.)
