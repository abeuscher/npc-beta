# 192. CRM Importer — Data Review Step

Insert a dedicated wizard step between Upload and Map Columns that assesses the shape of the uploaded CSV and flags issues for the user to resolve *before* mapping begins — starting with duplicate/similar column headers, and designed to accumulate more data-health checks over time.

---

## Design decisions (resolved before starting)

- **Distinct wizard step, not an inline banner.** The new step ("Review Data") is its own page in the Filament wizard, placed between Upload and Map Columns. It is a first-class gate — the user reviews and resolves flagged issues here, then proceeds to mapping with a cleaner state. We will add more logic to this step over time as other data-quality landmines surface; the separate-step framing keeps that growth clean.
- **Rules are pluggable, not monolithic.** The step runs a pipeline of rule checks. Each rule is its own class/method producing a list of findings with a standard shape (severity, title, affected columns, suggested action). The step renders the findings; adding a new rule later means adding a class, not rewriting the UI. First rule out of the gate: `DuplicateHeaderDetector`.
- **Duplicate-header detection is heuristic, not authoritative.** Heuristics catch common cases; the user always has final say. Rules for v1:
  - **Exact match after normalization** (lowercase + strip spaces/underscores/punctuation). Catches `First name` / `FirstName` / `first_name`.
  - **Word-subset overlap** on normalized tokens. Catches `Email` / `Email address`, `Phone` / `Phone number`, `Address` / `Address Line 1`.
  - **Trailing-digit / suffix patterns**: `Address` / `Address 2`, `Name_1` / `Name_2`, `Email` / `Email (alt)`.
- **Non-blocking by default.** Findings warn but do not prevent advancing. The user can acknowledge and proceed. Later rules may upgrade to blocking if warranted.
- **User's "ignore" choices flow forward into mapping.** When a user marks a column as "ignore this" on the review step, it arrives at the Map Columns step pre-set to `— ignore —`. The user can still manually un-ignore it if they change their mind; no state is destroyed.
- **Trait-first.** The new step and its rule-runner live in `InteractsWithImportWizard`. Each of the five wizard pages picks it up automatically. This is exactly the payoff from session 191's trait extraction.
- **Same UX for all five importers.** Contacts, Events, Donations, Memberships, Invoice Details — all get the step. No per-importer divergence unless a future rule requires it.

## Phase 1 — `DuplicateHeaderDetector` service

`App\Services\Import\DuplicateHeaderDetector`. Given the parsed CSV headers, returns a list of finding groups, each containing the column indices and the rule that matched.

```php
DuplicateHeaderDetector::detect(array $headers): array
// returns: [
//   [
//     'rule'    => 'exact_match_normalized',
//     'headers' => ['First name', 'FirstName', 'first_name'],
//     'indices' => [1, 19, 23],
//     'summary' => 'Three columns normalize to the same name.',
//   ],
//   [
//     'rule'    => 'word_subset',
//     'headers' => ['Email', 'Email address'],
//     'indices' => [4, 21],
//     'summary' => 'Both columns appear to hold email-like data.',
//   ],
//   ...
// ]
```

Rules are separate private methods on the service (`detectExactMatches()`, `detectWordSubsets()`, `detectTrailingDigitSuffixes()`). Each emits zero or more finding groups. Public `detect()` concatenates them.

Deliberate out of scope for this service: looking at cell values. Header text alone is enough for v1; value-based detection is the `NoiseDetector`'s territory.

## Phase 2 — New wizard step: "Review Data"

In `InteractsWithImportWizard`, add a `buildReviewStep()` method that returns a `Wizard\Step`. Each wizard page's `form()` method inserts this step between Upload and Map Columns.

Step contents:

- Top nav (Back / Next) via the existing `topNav()` helper.
- A banner at the top: "N findings to review" or "No issues detected — click Next to proceed."
- One section per finding, showing:
  - The rule description (`summary` from the detector).
  - Each affected column header as a list row, with the first 5 non-blank sample values from that column shown inline.
  - A per-column action: **Keep (map later)** / **Ignore this column** (radio or checkbox). Default: ignore all but the first column in the group (heuristic: the first column is usually the primary).
- A dismissable helper text: "When multiple columns hold the same kind of data, dropping the duplicates is generally safer than trying to merge them. You can always revisit this by re-uploading."
- An explicit "Acknowledge and continue" behaviour: clicking Next applies the chosen ignores and advances.

The step's `afterValidation` hook:
1. Reads the user's per-column decisions from form state.
2. Stores ignored column indices into a new `$data['ignored_columns']` array.
3. Falls through to the mapping step.

## Phase 3 — Mapping step integration

Update `processUploadedFileNamespaced()` (and the contacts-specific `processUploadedFile()`) so that columns whose indices appear in `$data['ignored_columns']` are pre-set to `null` (ignore) in `column_map`, and any saved-mapping or `guessDestination()` defaults for those columns are suppressed.

The user can still manually un-ignore a column in the mapping step — this is a default, not a lock.

No changes to `runImport()` or the progress pages are needed; an ignored column produces a null destination in `column_map`, which the existing row processors already treat as "skip this column."

---

## Phase 4 — UI polish and guidance

- On the review step, if zero findings are detected, show a reassuring empty-state: "No obvious column issues detected. Click Next to continue to mapping." (Don't skip the step — the user should still see it so they understand the process.)
- Severity colouring: findings render in amber (advisory). Reserve red for future blocking rules.
- Column sample values: render up to 5 non-blank cells per column, truncated at 80 characters each.
- When the same source file is re-uploaded with the same resolved source, remember the user's previous ignore choices. (Store on `ImportSource` alongside the field_map? TBD during implementation — could defer if it adds scope.)

## Phase 5 — Tests

New test file `tests/Feature/ImportSession192Test.php`:

- `DuplicateHeaderDetector` unit tests per rule:
  - Exact match finds `First name` / `FirstName` / `first_name` as one group.
  - Word-subset finds `Email` / `Email address` as a group but does NOT flag `Email` / `Phone`.
  - Trailing-digit finds `Address` / `Address 2` but NOT `Address` / `City`.
  - Empty array in, empty array out.
  - Case and whitespace insensitivity in all rules.
- State flow test: an ignored column from the review step arrives at mapping with `column_map[col_N] = null`.
- Regression: existing import tests must continue to pass unchanged; the new step is transparent when there are no findings (user just clicks Next).

## Out of scope

- Value-based duplicate detection (beyond headers) — stays with `NoiseDetector` until/unless we find a concrete case the two services should share.
- Automatic column dropping without user consent — the whole point of the step is that the user decides.
- Cross-content-type rules (e.g. "this looks like a donations CSV, not a contacts CSV") — interesting future rule, not scoped here.
- Saving per-source ignore preferences to `ImportSource` — may defer depending on scope; flag in log if deferred.
- Fully rewriting the contacts collision UI. Collision detection currently lives on the mapping step *after* the user has already started mapping; this session adds a *pre-mapping* layer. The two coexist; harmonizing them is a follow-up.
