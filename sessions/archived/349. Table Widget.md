# 349. Table Widget (Embedded ProseMirror)

A new Page-Builder widget called **Table** that embeds a self-contained ProseMirror + `prosemirror-tables` editor to author a real WYSIWYG table — merge/split cells, row/column add-delete, header rows, cell alignment, and **paste fidelity** from Word / Google Docs / the web — stored as clean semantic HTML in the widget config and rendered through the existing `HtmlSanitizer` pipeline. The **block-level launch stopgap** for tabular content, and the team's **first ProseMirror integration** — a deliberate down-payment on the deferred [Rich-Text Editor (ProseMirror) migration track](tracks/rich-text-editor-prosemirror.md).

---

## What this is — and what it is NOT

**IS:** a standalone, block-level page-builder widget whose editing surface is one *isolated* ProseMirror editor configured with only a table schema. It touches nothing else — Quill, the 347 inline-format toolbar, and the three existing rich-text surfaces are all untouched.

**Is NOT:**

- **Not the inline-in-prose table.** Authoring a table *inside a flowing paragraph of a Text widget's rich text* requires migrating the rich-text editor off Quill — Quill structurally cannot host a table (it strips any non-blot markup on its normalization pass; see [useInlineEdit.ts:194-208](../resources/js/page-builder-vue/composables/useInlineEdit.ts#L194-L208)). That migration is the deferred [ProseMirror track](tracks/rich-text-editor-prosemirror.md) and is **out of scope here**. This widget delivers a block-level table (a widget between paragraphs), which covers ~90% of real table use — blog/legal tables sit *between* paragraphs, not mid-sentence.
- **Not the PricingChart clone.** The earlier 311/349 framing (a count-picker + repeater widget with inline-edited cells) was the wrong shape — 311 built it and it was rewound. It can't merge cells, can't paste a real table, and isn't a real table editor. This is a *real* table editor (`prosemirror-tables`), not a grid of repeater rows.
- **Not an app-wide editor swap.** The embedded ProseMirror is fully scoped to this widget. The rest of the app stays on Quill until the deferred track runs.

---

## Why ProseMirror, embedded

`prosemirror-tables` is the mature, MIT-licensed engine for content-table editing — cell-range selection, merge/split, column resize, HTML-table paste-normalization: the exact hard 80% that rolling-our-own or forking a basic table tool would force us to rebuild. We embed a *minimal* ProseMirror editor (table schema + basic cell marks only) in this one widget: real tables now, ~2 sessions not ~7, and the same engine + patterns the deferred full migration will reuse — a down-payment, not throwaway. The compromise we knowingly accept: block-level, not mid-paragraph inline.

## Design decisions (resolved before starting)

- **Storage = clean semantic HTML in the widget config** (e.g. `config.table_html`), sanitized via the existing `HtmlSanitizer` on save and rendered `{!! HtmlSanitizer::sanitize($config['table_html']) !!}`. **No ProseMirror-JSON in storage** — HTML keeps it non-destructive, keeps the export bundle clean (it carries the table as an HTML string like all rich text), and sidesteps any content-format migration. ProseMirror parses the stored HTML back when the editor opens. **No DB schema, no migration, no `format_version` bump.**
- **Editor = a minimal, isolated ProseMirror instance.** Schema: `table` / `table_row` / `table_cell` / `table_header_cell` (from `prosemirror-tables`) + `paragraph` + basic cell-content marks (`strong`, `em`, `link`). Plugins: `prosemirror-tables` (table editing + commands), `prosemirror-keymap`, `prosemirror-history`, base input rules. **Deliberately NOT the full prose feature set** — no color, heroicon, media-insert, or headings inside cells (that's the full editor's job, deferred).
- **ProseMirror ships via npm + Vite**, bundled into the page-builder admin bundle (`npm run build`). It is **NOT loaded on the public site** — the public render is plain sanitized HTML, zero editor JS, so **no public-bundle size impact**. (Quill's CDN load is untouched.)
- **Sanitizer extension (the one back-end change).** `HtmlSanitizer` already allow-lists the table *tags* ([HtmlSanitizer.php:20](../app/Support/HtmlSanitizer.php#L20)) but strips all attributes on `td`/`th`/`table`/`tr`. Extend `ATTRS_PER_TAG` so `td`/`th` carry `colspan`/`rowspan` (validated as bounded positive integers) and a constrained alignment class (`np-table-cell--{left,center,right}`), and `table`/`tr` carry their structural class. Strip everything else (style / script / event-handlers) as today. This is **untrusted-on-paste input** → a security-reviewed allow-list, not permit-anything.
- **Appearance is widget-level config applied at render — not per-cell, not baked into the stored HTML.** The PM editor owns the table *structure* + cell content; the widget config owns its *appearance*, applied by `template.blade.php` + `styles.scss` (CSS custom props / utility classes). The required controls are spelled out in **Table styling** below and they **reuse the existing appearance tools** (the E17 border helper, the alignment field, the color field) — do not hand-roll. Consequence: background color, zebra striping, and row-group alignment need **no sanitizer change** — only the structural attributes (`colspan`/`rowspan`) pass through the sanitizer.
- **Not inline-eligible.** `inlineEditable()` returns `false` — the table is authored in the embedded editor surface, not cell-by-cell on the canvas. The canvas shows a read-only render. (A deliberate departure from PricingChart's inline-cell pattern — simpler and fully isolated.)

## Table styling — reuse the built appearance tools (required, must ship in v1)

All four are **widget-level** inspector controls applied at render — the owner explicitly wants them in this widget. Reuse what already exists; do not build bespoke versions:

1. **Border — the E17 border tool.** Carry a `border` appearance block (mirror `TextBlock`'s `appearance` border config in `app/Widgets/TextBlock/TextBlockDefinition.php`) and run it through `App\Services\AppearanceStyleComposer::composeBorderProps()` in `template.blade.php`, exactly as the Text widget / `ChromeRenderer` do. Same width / style / color / radius vocabulary operators already know — applied to the table's outer box. No bespoke `border_style` enum.
2. **Alignment — the `alignment` inspector field** (the built alignment tool), at minimum for **header row** and **content rows** as two separate controls (`header_align`, `body_align`). Applied as `text-align` on the header cells (`thead th`) and body cells (`tbody td`) via the template's CSS. The header row itself is **structural** (set in the editor via the header-row toggle → `<th>` cells); these controls style whatever the editor produced. (Per-column / per-cell alignment is a later refinement — out of scope for v1.)
3. **Background color — the `color` inspector field.** A table background color (`bg_color`), applied as a CSS custom prop on the `.np-table` wrapper. Config → template → CSS; not baked into cell HTML, so no sanitizer involvement.
4. **Zebra striping — a toggle that doubles the color options.** `zebra` (bool); when ON it **reveals a second `color` field** (`zebra_color`, the alternate-row background) and the template stripes via `tbody tr:nth-child(even)`, with the header row excluded. The second color control is conditional — it appears only when `zebra` is on. ("Doubles the color options" = base `bg_color` + the zebra `zebra_color`.)

Everything here is config → `template.blade.php` + `styles.scss` (CSS custom props / utility classes); nothing is per-cell and nothing changes the stored `table_html`. Defaults must satisfy the concrete-values rule (every knob carries a concrete default — e.g. a sane border, left alignment, a transparent/`--np-*` base bg, zebra off).

## Open questions to settle at session start

1. **Editor surface placement.** The table editor needs room. An Inspector custom field (`TableEditorField.vue`, a new `field.type` in `InspectorField.vue`'s `componentMap`, mirroring the existing field components) is the default — but a table may want an expandable / full-width / modal editing surface rather than the narrow inspector rail. Settle inspector-field vs. expandable-surface first; either way it's a bounded new field component.
2. **The table toolbar.** `prosemirror-tables` supplies the *commands* (insertTable / addRow / addColumn / deleteRow / deleteColumn / mergeCells / splitCell / toggleHeaderRow); this session wires the *buttons*. Confirm the v1 button set: insert (with a Google-Docs-style grid picker for initial dimensions), add/delete row & column, merge/split, header-row toggle. Cell-content marks: bold / italic / link only. **Alignment, border, background, and zebra are widget-level appearance controls** (see Table styling) — not editor-toolbar commands in v1.
3. **Storage key + paste scope.** Confirm `config.table_html` (HTML-only, no JSON cache). Confirm paste handling: a pasted `<table>` (Word / Docs / web) maps into the schema via `prosemirror-tables` + the schema's `parseDOM`; verify a real cookie-policy-style table round-trips (a Playwright case).

## Phases

Build after each step. The widget render + sanitizer are Pest-covered; the authoring round-trip is Playwright-covered.

1. **Decide the open questions** (editor surface; toolbar set; storage key). Log at session start.
2. **ProseMirror build integration + the minimal table schema.** Add the npm deps; a Vite-bundled module exporting the editor factory (schema + plugins). Smoke that the editor mounts, a table inserts, and `editor → HTML → sanitizer → editor` round-trips clean.
3. **The editor surface + table toolbar.** The embedded editor as the chosen surface (`TableEditorField.vue` or expandable); the button row wired to the `prosemirror-tables` commands + the grid-picker insert. `npm run build`.
4. **`TableDefinition.php` + `template.blade.php` + `styles.scss` + `WidgetTypeSeeder` append.** Config schema = `table_html` + the **Table styling** controls (the `border` appearance block, `header_align` / `body_align`, `bg_color`, `zebra` + conditional `zebra_color`), concrete defaults, public render (`.np-table` overflow-x wrapper + sanitized HTML + `composeBorderProps` border + the alignment / bg / zebra CSS). `build:public` for the widget CSS. Pest: definition load + seeder registration + default config + render-shape (sanitized HTML in the wrapper, the border CSS props from `composeBorderProps`, header/body alignment, base bg + zebra `nth-child` striping, sanitizer preserves colspan/rowspan + strips disallowed). Mirror `PricingChart`'s definition/seeder test shape for those parts.
5. **Sanitizer extension + its Pest.** Extend `HtmlSanitizer` for the table cell attributes; extend `tests/Feature/Support/HtmlSanitizerTest.php` (allow colspan/rowspan/alignment-class, strip style/script).
6. **Help doc + `help:sync`.** `resources/docs/widget-table.md` mirroring an existing widget doc.
7. **Playwright spec** `tests/e2e/page-builder/table-widget.spec.ts`: add the widget, build a table (insert 3×3, add a row, merge two cells, toggle header, type cell content), paste a table, save, navigate to the public page, assert the table structure + content survive the round-trip.

## Out of scope (fence hard)

- **The inline-in-prose table** (a table mid-paragraph in a Text widget's rich text) — the deferred [ProseMirror migration track](tracks/rich-text-editor-prosemirror.md). If the work starts wanting to put this editor inside the inline-format toolbar or migrate any Quill surface, scope has drifted — stop.
- Per-cell color / shading / borders (widget-level appearance only).
- Full prose features inside cells (color, heroicon, media-insert, headings) — basic marks only.
- Collection-bound tables, CSV-import-to-table, cross-table style presets.
- DB schema / migration, CRM contract bump, bundle `format_version` bump, any Fleet Manager surface.

## Drift signals (stop and surface)

- Work reaches into Quill, `useInlineEdit`, the 347 inline-format toolbar, or any existing rich-text surface — this widget's editor is isolated; that's the deferred track, not here.
- The embedded editor grows toward the full prose feature set (color / heroicon / media inside cells) — scope creep toward the migration.
- ProseMirror ends up loaded on the public site — the public render is sanitized HTML only.
