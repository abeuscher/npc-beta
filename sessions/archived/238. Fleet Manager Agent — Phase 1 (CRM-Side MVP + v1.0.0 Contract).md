# 238. Fleet Manager Agent — Phase 1 (CRM-Side MVP + v1.0.0 Contract)

Land the CRM-side surface that Fleet Manager polls — `/api/health` route, bearer-token auth middleware, six subcheck implementations, version reporting via Docker build-arg, and the v1.0.0 contract spec doc that unblocks the parallel Fleet Manager repo workstream.

---

## Design decisions (resolved before starting)

All locked in by `sessions/fleet-manager-planning-spec.md` ("Resolved leans") and confirmed during session 237's prompt-drafting conversation:

- **Endpoint path:** `/api/health` (single route, GET, JSON response)
- **Auth:** bearer token in `Authorization` header. Single per-install API key in `.env` as `FLEET_MANAGER_AGENT_KEY`. Comparison is timing-safe (`hash_equals`).
- **Rate limit:** Laravel `throttle:60,1` (60 requests/minute per IP). Defends against an FM bug polling every second.
- **Version reporting:** `APP_VERSION` Docker build-arg sourced from `${GITHUB_SHA::7}` in CI; written by the Dockerfile to `/var/cache/app/VERSION` (outside the local bind-mounted tree). Read at boot by Laravel into a cached config value via `config('fleet.agent.app_version')`. Local dev defaults the build-arg to `dev`.
- **`last_backup_at`:** returns `null` until Phase 2 (Backup Pipeline) lands. Fleet Manager treats `null` as "unknown — don't alarm" per the planning spec.
- **SSL expiry:** **NOT in this session.** Fleet Manager checks externally per the planning spec lean.
- **API key bootstrapping:** manual env-var on first deploy. Operator generates a value (e.g., `Str::random(64)`) and sets it on the droplet. Automation deferred.
- **Contract spec doc location:** `docs/fleet-manager-agent-contract.md` (already exists at v0.0.0). This session bumps it to v1.0.0.
- **Subcheck status grammar:** each subcheck returns `green` / `yellow` / `red`; overall status is the worst of all subchecks (green if all green; yellow if any yellow and no red; red if any red).
- **Endpoint behavior on subcheck failure:** the endpoint returns HTTP 200 with the failing subcheck marked `red` in the body. The endpoint itself does not return 503 unless the entire app is unable to compose a response (genuinely unreachable). This is so Fleet Manager always gets structured data to act on.

## Phase 1 — Routes, middleware, controller skeleton

Land the request-path scaffolding before any subcheck logic:

- **New route** in `routes/api.php`:
  ```php
  Route::middleware(['throttle:60,1', 'fleet.agent'])
      ->get('/health', [\App\Http\Controllers\Api\Fleet\HealthController::class, 'index']);
  ```
- **New middleware** at `app/Http/Middleware/AuthenticateFleetManagerAgent.php`:
  - Reads the expected token via `config('fleet.agent.api_key')`
  - Reads the request token via `$request->bearerToken()`
  - Compares via `hash_equals($expected, $actual)` (timing-safe)
  - Fails closed: missing token → 401, wrong token → 401, missing config → 500 (misconfiguration)
  - 401 response body: `{"error": "unauthorized"}` (no leakage of which side is wrong, no key echoed back)
- **Register the alias** `'fleet.agent' => AuthenticateFleetManagerAgent::class` in the appropriate Laravel 11 location (`bootstrap/app.php` per current Laravel 11 conventions; verify in the actual codebase).
- **New config file** at `config/fleet.php`:
  ```php
  return [
      'agent' => [
          'api_key'     => env('FLEET_MANAGER_AGENT_KEY'),
          'app_version' => null, // populated at boot — see Phase 3
      ],
  ];
  ```
- **`.env.example`** gains `FLEET_MANAGER_AGENT_KEY=` (no default value — operator must set).
- **New controller** at `app/Http/Controllers/Api/Fleet/HealthController.php` — skeleton only at this phase, returning a stub response:
  ```php
  public function index(): JsonResponse
  {
      return response()->json([
          'status' => 'green',
          'version' => 'dev',
          'timestamp' => now()->toIso8601String(),
          'contract_version' => '1.0.0',
          'subchecks' => [],
      ]);
  }
  ```

Smoke-test at end of Phase 1: `curl -H "Authorization: Bearer <key>" http://localhost/api/health` returns 200 with the stub shape.

## Phase 2 — Subcheck implementations

Six subchecks, each returning the canonical shape `{'status' => 'green'|'yellow'|'red', 'value' => mixed, 'threshold' => mixed|null, 'message' => string|null}`.

Inline as private methods on `HealthController` for v1 (six tightly-scoped checks don't warrant a class hierarchy yet):

1. **`checkApp()`** — always green if the request reaches the controller. `value` is `'responding'`, no threshold. (The fact that this method runs IS the check.)
2. **`checkDatabase()`** — `DB::connection()->getPdo()` inside a try/catch. Green on success, red on `PDOException`. `value` is `'reachable'` / `'unreachable'`. `message` carries the exception class name on failure (not the full exception message — leak risk).
3. **`checkRedis()`** — `Redis::ping()` inside a try/catch. Green on success, red on any exception. `value` / `message` shape mirrors `checkDatabase()`.
4. **`checkDisk()`** — `disk_free_space('/') / disk_total_space('/')` → percent-used integer. `threshold` is `80` (yellow) and `95` (red). `value` is the percent-used integer. Green if <80, yellow if 80–94, red if ≥95.
5. **`checkLastBackupAt()`** — returns `{'status' => 'green', 'value' => null, 'message' => 'backup pipeline not yet implemented'}` for v1. The shape is in place; Phase 2 of the track populates it. Status is **green not yellow** — "unknown" without a backup pipeline is not an alarming state; this changes when Phase 2 lands and a missing recent backup *does* mean something is wrong.
6. **`checkVersion()`** — returns `{'status' => 'green', 'value' => $appVersion}` where `$appVersion` is `config('fleet.agent.app_version')`. Green regardless of value (a `dev` build is not a failure state).

`HealthController::index()` calls all six, composes the overall status via the worst-of rule, returns the full JSON.

## Phase 3 — Version reporting via Docker build-arg

Wire the build-arg through Dockerfile, both compose files, and the deploy workflow:

- **`Dockerfile` (app target):**
  ```dockerfile
  ARG APP_VERSION=dev
  RUN mkdir -p /var/cache/app && echo "$APP_VERSION" > /var/cache/app/VERSION
  ```
  Verify the insertion point — should be late in the `app` target so it doesn't bust earlier layer caching for code-only changes. Read the current Dockerfile to confirm structure.
- **`docker-compose.yml` (local):** add to the `app` service's `build:` block:
  ```yaml
  args:
    APP_VERSION: ${APP_VERSION:-dev}
  ```
- **`docker-compose.prod.yml`:** **no change.** It pulls pre-built images from GHCR; the version is baked into the image at CI time.
- **`.github/workflows/deploy.yml`:** add `APP_VERSION=${{ steps.tag.outputs.IMAGE_TAG }}` to the existing `build-args:` list under the "Build and push app image" step (alongside the existing `BUILD_ENV` arg).
- **Laravel-side reader** — populate `config('fleet.agent.app_version')` at boot. Cleanest path: a small service provider that reads `/var/cache/app/VERSION` once at boot and writes the value into the config repository. Or simpler: use a closure in the config file:
  ```php
  // config/fleet.php
  'app_version' => is_readable('/var/cache/app/VERSION')
      ? trim((string) file_get_contents('/var/cache/app/VERSION'))
      : 'dev',
  ```
  The closure form is fine here — config files are evaluated once and cached via `config:cache` in production.

After Phase 3: `cat /var/cache/app/VERSION` inside any running container returns the version string. `config('fleet.agent.app_version')` matches it. `/api/health`'s top-level `version` field reflects it.

## Phase 4 — Author `docs/fleet-manager-agent-contract.md` v1.0.0

Replace the placeholder body with the actual contract surface. Final shape:

- **Endpoint:** `GET /api/health`
- **Auth:** `Authorization: Bearer <token>` header. Token is the per-install API key. Validation is timing-safe.
- **Request:** GET, no body, no query parameters.
- **Response 200 (success — including subcheck failures):**
  ```json
  {
    "status": "green|yellow|red",
    "version": "string",
    "timestamp": "ISO-8601 string",
    "contract_version": "1.0.0",
    "subchecks": {
      "app":             { "status": "...", "value": "...", "threshold": null,    "message": null },
      "database":        { "status": "...", "value": "...", "threshold": null,    "message": "..." },
      "redis":           { "status": "...", "value": "...", "threshold": null,    "message": "..." },
      "disk":            { "status": "...", "value": 42,    "threshold": [80,95], "message": null },
      "last_backup_at":  { "status": "...", "value": null,  "threshold": null,    "message": "..." },
      "version":         { "status": "...", "value": "...", "threshold": null,    "message": null }
    }
  }
  ```
- **Response 401 (auth failure):** `{"error": "unauthorized"}`. No body data leaks.
- **Response 429 (rate limited):** Laravel default rate-limiter response. FM should back off and retry.
- **Response 503:** reserved for future use (endpoint-disabled state). Not emitted by v1.0.0.
- **Subcheck status grammar:** documented with the worst-of rule for overall status.
- **Version negotiation:** the response always includes `contract_version`. FM reads this on every poll. If the FM-side cached version differs from what it fetches via WebFetch from the canonical URL, FM logs a drift warning. Forward-compat: FM can poll a CRM running v1.X (X > 0) using its v1.0 understanding as long as the response includes the v1.0 fields it needs; new fields are additive.
- **CHANGELOG entry:**
  ```markdown
  ### `1.0.0` — 2026-04-XX (session 238)

  Initial contract. Endpoint, auth, response schema, version negotiation, error envelope all defined. CRM-side `/api/health` ships in this session; Fleet Manager repo can begin building its HTTP client against this contract from the canonical URL.
  ```
- **Bump `Contract Version: 0.0.0` to `Contract Version: 1.0.0`** at the top of the doc.

## Phase 6 — Tests

New tests in `tests/Feature/Api/Fleet/HealthEndpointTest.php` (or wherever the existing API tests live — verify before placing). Coverage:

**Auth:**
- Missing `Authorization` header → 401
- Wrong bearer token → 401
- Empty bearer token (`Authorization: Bearer ` with no value) → 401
- Correct bearer token → 200
- Comparison is timing-safe (assert via reflection or by inspecting middleware code that `hash_equals` is the comparator — direct timing-attack tests are out of scope)

**Response shape:**
- Top-level keys exactly `['status', 'version', 'timestamp', 'contract_version', 'subchecks']`
- `contract_version` equals `'1.0.0'`
- `subchecks` keys exactly `['app', 'database', 'redis', 'disk', 'last_backup_at', 'version']`
- Each subcheck has keys `['status', 'value', 'threshold', 'message']`
- `status` is one of `green|yellow|red`

**Subcheck happy paths (test environment, Postgres + Redis up):**
- All six subchecks return `green`
- Overall status is `green`
- `version` subcheck's `value` matches `config('fleet.agent.app_version')`

**Subcheck failure paths (mockable):**
- DB facade swap → `database` subcheck is `red`, overall status is `red`, endpoint still returns 200
- Redis facade swap → `redis` subcheck is `red`, overall status is `red`, endpoint still returns 200
- Disk failure simulation: out of scope for v1 (global function `disk_free_space` not easily mockable; integration-test only)

**Rate limit:**
- 60 requests inside a minute return 200; the 61st returns 429

**Version reader:**
- Helper test: when `/var/cache/app/VERSION` exists with content `abc1234`, `config('fleet.agent.app_version')` returns `abc1234`
- When file is absent, returns `dev`

## Security checklist

Before closing:

- [ ] Bearer-token comparison uses `hash_equals()`, not `===` or `==`.
- [ ] Auth middleware fails closed on every misconfiguration scenario (missing config, empty token, wrong token).
- [ ] 401 response body does not echo the submitted token or the expected token.
- [ ] Rate limit middleware (`throttle:60,1`) is on the route group.
- [ ] No DB data, user info, session info, or stack traces appear in the `/api/health` response — only the documented subcheck shapes.
- [ ] The route is in the API middleware group (stateless), NOT the web group (no CSRF, no session cookie).
- [ ] Subcheck failure messages do not leak internal paths, exception messages, or stack details — only the exception class name where useful.
- [ ] `FLEET_MANAGER_AGENT_KEY` is in `.env.example` with no default value (operator must set).
- [ ] Contract spec doc explicitly forbids reusing the API key across multiple installs in v1 (each CRM install has its own).
