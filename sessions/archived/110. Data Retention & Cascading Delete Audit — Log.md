## What was built

### Phase 1 — FK cascade audit and fixes

Audited all 50+ foreign keys in the database (schema dump + post-dump migrations). Wrote a single migration fixing 10 FK constraints:

**Critical fixes (prevented data loss on force-delete):**
- `donation_receipts.contact_id`: CASCADE → RESTRICT — tax receipts no longer destroyed
- `memberships.contact_id`: CASCADE → RESTRICT — membership history preserved
- `import_sessions.imported_by`: CASCADE → SET NULL (column made nullable) — import audit trail survives user deletion
- `page_widgets.widget_type_id`: CASCADE → RESTRICT — DB-level enforcement of session 109 UI guard

**Audit trail fixes:**
- `contact_duplicate_dismissals.dismissed_by`: CASCADE → SET NULL (column made nullable)

**Explicit intent (was implicit NO ACTION):**
- `purchases.product_id` and `purchases.product_price_id` → explicit RESTRICT
- `allocations.product_id` and `allocations.product_price_id` → explicit RESTRICT (legacy table)
- `waitlist_entries.product_id` → CASCADE (non-financial, meaningless without product)

### Phase 2 — Soft-delete purge policy

Defined retention periods for all 10 soft-deletable models, purge prerequisites per model, and super_admin-only access rule. Documented in the retention policy (Phase 3). No automated purge implemented (per prompt — deferred).

### Phase 3 — Retention policy document

Wrote `docs/data-retention-policy.md` covering:
- Soft-delete retention periods per model
- Complete cascade rules for every parent type (Contact, User, Event, Page, Collection, Form, Product, Fund, Tier, Widget Type, Nav Menu, Template, Import Session, Tag, Mailing List)
- Purge policy and prerequisites
- Financial record immutability rules
- Contact PII handling on deletion
