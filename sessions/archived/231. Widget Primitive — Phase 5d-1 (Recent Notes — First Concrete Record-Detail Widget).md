# 231. Widget Primitive — Phase 5d-1 (Recent Notes — First Concrete Record-Detail Widget)

Land the first concrete widget that consumes the record-detail primitive: **Recent Notes for the current contact**. Mirror of Phase 4a — contract design + projector + test-lock pair. Resolves the "first View wins on EditContact" gap surfaced during 5c.5 manual testing by wiring `EditContact` to a concrete `RecordDetailViewPage` subclass. Forces a minimal Phase 5e decision (resource-level permission gate at the resolver) without committing to the full perm-in-contract architecture.

---

## Design decisions (resolved before starting)

- **`SOURCE_SYSTEM_MODEL` with `model: 'note'`.** Notes is an Eloquent model; the contract source is `SOURCE_SYSTEM_MODEL`, the resolver gains a `note` arm, the projector gains a `projectNote()` method. Mirror of 4f's BlogPager retrofit (which added `'post'` to the system-model dispatch).
- **Ambient record id flows into the resolver via `SlotContext`, not the contract.** The widget's contract declares `model: 'note'`, `fields: [...]`, and `filters: ['limit' => N, 'order_by' => 'occurred_at', 'direction' => 'desc']` — but **does not** declare a per-instance contact filter. The resolver, when running a `note` contract under a record-detail slot, reads the ambient record from `SlotContext::ambient()` (which is `RecordDetailAmbientContext`) and applies `where('notable_type', $record::class)->where('notable_id', $record->getKey())` automatically. **Rationale:** the contract describes capability ("I read notes; here are my fields and limit"); the slot binding describes **scope** ("attach to whatever record I'm rendered for"). Same shape as how `SOURCE_PAGE_CONTEXT` reads `currentPage` from ambient, not from the contract.
- **Projector field-naming follows the 226 concept-named convention.** Display fields are named after concepts the template will read, not after raw column names. Initial set: `note_id`, `note_subject`, `note_body_excerpt`, `note_type`, `note_occurred_at` (formatted), `note_author_name`. Raw `body` is excerpted to ~140 chars by the projector — full-body access is not exposed.
- **`view_note` permission gate at the resolver, fail-closed (the minimal Phase 5e decision).** Before the resolver dispatches a `note` contract, it checks `auth()->user()?->can('view_note')`. If false, the resolver short-circuits and returns `['items' => []]` — the widget renders empty rather than throwing. The widget itself does not gate; concentrated enforcement at the resolver is the principle. **This is the minimal Phase 5e shape — a literal one-line `Gate::denies` check on the `note` arm, not a contract-level perm declaration.** The full perm-in-contract architecture (where every contract declares its required permissions and the resolver gates uniformly) stays carved out as a future Phase 5e session; this session ships a real gate alongside the first sensitive widget so prod is safe today, and the architecture catches up later.
- **`EditContact::recordDetailViewSubPageClass()` returns a concrete `EditContactView` page.** Resolves the "first View wins" gap from 5c.5 manual testing. The new page extends `RecordDetailViewPage` (the abstract base from 5c) and is registered in `ContactResource::getPages()` under route `contacts/{record}/edit/view/{view}`. EditContact's footer-widget mount stays — it renders the View whose handle matches the entry-page's default. Sub-nav surfaces all Views for Contact when more than one exists.
- **The seeded `contact_overview` View replaces its placeholder widget with Recent Notes.** Idempotency rule: the seeder's "widgets already exist on this view" early-return stays — existing installs that already have the placeholder keep it. New installs (and `migrate:fresh --seed`) seed Recent Notes instead. No data migration; the placeholder widget definition stays on disk for now (deletion is a separate cleanup pass once all installs have re-seeded).
- **No widget assets (CSS/JS).** Recent Notes is a server-rendered Blade widget that uses only the existing public typography. No `assets()` declaration, no SCSS partial. `npm run build` is sufficient (Vue/Vite path is unaffected — actually no Vue source is touched at all this session). `build:public` not required.
- **Cardinality: many.** Recent Notes is list-shaped. The contract declares `cardinality: 'many'` (or omits, since 'many' is the default). The DTO shape is `['items' => array<int, array<string, mixed>>]`. Empty state: "No notes yet."
- **Test-lock mirrors 4a.** Fail-closed field whitelist (a contract requesting `body`, `meta`, `external_id` returns empty strings for those keys). Resolver-is-only-path query-count pin (rendering the widget hits the resolver exactly once per page mount). Ambient-record scoping pin (widget configured under contact A renders A's notes, not B's — the IDOR analogue at the data layer).

## Phase 1 — Resolver `note` arm + projector + ambient-record scoping

1. Update `app/WidgetPrimitive/ContractResolver.php`:
   - Add a `'note'` case in `resolveSystemModel()`'s match dispatch.
   - New private `resolveNote(DataContract $contract, array &$cache, SlotContext $context): array` method. Permission gate: `if (! auth()->user()?->can('view_note')) { return ['items' => []]; }`. Reads ambient record from `$context->ambient()`; if not a `RecordDetailAmbientContext` or `record()` is null, return `['items' => []]`. Otherwise: `Note::query()->where('notable_type', $record::class)->where('notable_id', $record->getKey())->orderBy(...)->take($limit)->get()` with whitelisted `order_by` (`occurred_at`, `created_at`) and `limit` clamped to ≤ 50. Cache key: `'note:' . $recordType . ':' . $recordId . ':' . sha1(serialize($contract->filters))`.
   - **Note**: `resolveSystemModel()` currently does not receive `SlotContext` — it reads from a captured `$cache` only. Verify the existing signature; if `SlotContext` is not threaded through, a small upstream refactor is required (one-liner — pass `$context` from `resolve()` down through `resolveSystemModel()`). This is a real refactor that affects post/event/product arms; do it carefully and run the existing 4a–4h test suite to confirm zero regression.
2. Update `app/WidgetPrimitive/Projectors/SystemModelProjector.php`:
   - Add `'note'` to `projectorFor()`'s match.
   - New private `projectNote(Note $note, array $formatHints): array` method. Returns the full field map: `note_id`, `note_subject`, `note_body_excerpt` (Str::limit body to ~140 chars), `note_type`, `note_occurred_at` (formatted via `DateFormat::format($note->occurred_at, $formatHints['note_occurred_at'] ?? DateFormat::SHORT)`), `note_author_name` (`$note->author?->name ?? '—'`).
3. **Tests** at `tests/Unit/WidgetPrimitive/ContractResolverNoteTest.php` and `tests/Unit/WidgetPrimitive/Projectors/SystemModelProjectorNoteTest.php`:
   - Resolver returns empty for user without `view_note` (super_admin and a role with `view_note` both pass; cms_editor without it gets empty).
   - Resolver returns only notes attached to the ambient record (other contacts' notes do not leak).
   - Resolver respects `limit` (default 5, configurable up to 50) and `order_by` (`occurred_at desc` default; `created_at` allowed; anything else falls back to default).
   - Projector returns only declared fields; `body`, `meta`, `external_id` are undeclared and return empty strings.
   - Projector excerpts body at ~140 chars with ellipsis.

## Phase 2 — `RecentNotes` widget (definition + Blade template)

1. Create `app/Widgets/RecentNotes/RecentNotesDefinition.php`:
   - `handle: 'recent_notes'`
   - `label: 'Recent Notes'`
   - `description: 'Lists notes attached to the current contact, ordered by occurred_at descending.'`
   - `category: ['admin']`
   - `allowedSlots(): ['record_detail_sidebar']`
   - `acceptedSources(): [Source::HUMAN, Source::IMPORT]` (Notes are imported from CSVs in addition to being human-authored)
   - `schema()`: one `limit` field (number, default 5, min 1, max 50)
   - `defaults(): ['limit' => 5]`
   - `dataContract(array $config): DataContract` — `SOURCE_SYSTEM_MODEL`, `model: 'note'`, `cardinality: 'many'`, fields `['note_id', 'note_subject', 'note_body_excerpt', 'note_type', 'note_occurred_at', 'note_author_name']`, filters `['limit' => $config['limit'] ?? 5, 'order_by' => 'occurred_at', 'direction' => 'desc']`, formatHints with `note_occurred_at` set to `DateFormat::SHORT_WITH_RELATIVE` (or whatever is canonical for "Mar 5, 2026 (3 days ago)" rendering).
2. Create `app/Widgets/RecentNotes/template.blade.php`:
   - Iterates `$widgetData['items']`; per-item renders subject (bold), occurred_at (muted), author_name (italic), body excerpt.
   - Empty state: "No notes yet."
3. Register in `app/Providers/WidgetServiceProvider.php` alongside other widgets.
4. Run `db:seed --class=WidgetTypeSeeder` to register the new `widget_types` row.
5. **Tests** at `tests/Unit/Widgets/RecentNotesDefinitionTest.php`:
   - allowedSlots returns `['record_detail_sidebar']`
   - acceptedSources returns `[Source::HUMAN, Source::IMPORT]`
   - dataContract shape (model, fields list, default limit, formatHints)
   - config-driven limit propagates to the contract's filter

## Out of scope

- **Full Phase 5e (perm-in-contract architecture).** This session lands a literal `Gate::denies('view_note')` check at the resolver's `note` arm. Generalizing to "every contract declares required perms; resolver gates uniformly" stays carved out for a dedicated Phase 5e session.
- **Note authoring from the widget.** Recent Notes is read-only. The "compose a new note" affordance lives in the existing NoteResource; adding it to the widget is a separate session.
- **Placeholder widget removal.** `RecordDetailPlaceholderDefinition` stays on disk and registered. Removing it is a follow-up cleanup pass once all installs have re-seeded.
- **Recent Donations / Membership Status / Recent Activity.** Phases 5d-2+ — one widget per session, each may force its own design questions.
- **Notes pagination / "View all" link.** Limit-only for now. Pagination, jump-to-NoteResource link, etc. are widget-config additions for a later session.
- **Filtering by note type / outcome.** Initial Recent Notes shows everything. Filters (subtype, outcome, author) are config additions for a later session.
- **`DashboardConfig` retrofit to `DashboardView` implementing `IsView`.** Cheap follow-up post-5d, scheduled ad-hoc.
