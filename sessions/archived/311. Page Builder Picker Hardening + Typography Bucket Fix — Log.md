## Summary

Three pre-existing bugs landed across two iteration commits, plus a small housekeeping iteration that removed a footer padding rule and a stale planning doc.

**Iteration 1 — Page Builder picker slot filter (4 lines, 2 files):** the Page Builder picker was the only `WidgetType::forPicker()` caller that did not pass a slot handle, so the eight backend-only widgets (`recent_donations`, `membership_status`, `recent_notes`, `this_weeks_events`, `memos`, `quick_actions`, `random_data_generator`, `setup_checklist` — declared with `allowedSlots()` of `dashboard_grid` or `record_detail_sidebar`) leaked into the picker and could be attached to public Pages. `DashboardBuilder` and `RecordDetailViewBuilder` have always passed their slot handles. Fix: `app/Livewire/PageBuilder.php` and `app/Http/Controllers/Admin/PageBuilderApiController.php` now pass `'page_builder_canvas'`. Defense-in-depth in `WidgetRenderer::render()` (the "even if attached, don't render off-slot" guard) was investigated and deliberately deferred — it has its own regression surface and the picker fix closes the leak at the source.

**Iteration 2 — typography bucket → element cascade + nav font CSS path (5 files):** two pre-existing bugs in the Theme typography pipeline.

*Bug 1 (heading + body):* `TypographyPanel.vue`'s `onBucketChange` cascade was gated on `previous && next`, so a fresh install (`buckets.heading_family = null`, elements at the shipped `'Inter', system-ui, sans-serif` default) skipped the cascade entirely on the first bucket change. Per-element `font.family` values stayed at Inter; the compiler (which reads `elements.<key>.font.family`, not the bucket) kept emitting `font-family: 'Inter'`; the literal element rule then beat the `--font-family-heading` `:root` CSS variable the layout dumps in. Fix: the cascade now runs on any null→value transition, and elements that match either the previous bucket value OR the shipped default family follow the bucket. The default family is sourced from the bootstrap (`DesignSystemPage::getTypographyBootstrap()` now emits `defaultFamily`) to avoid PHP/JS drift.

*Bug 2 (nav):* `buckets.nav_family` was saved to the DB and used by `googleFontsUsed()` to load the Google Font, but no CSS path consumed it. The compiler's per-element output excludes nav (`h1:not(nav h1)`, etc.), `_base.scss` has no nav rule, and the layout didn't emit `--font-family-nav`. The picker UI was dead code from the moment it shipped. Fix: `public.blade.php` now emits a `--font-family-nav` variable alongside heading/body and an inline `.np-site nav, .np-site nav a { font-family: <stack> }` rule — direct, no SCSS rebuild required.

**Iteration 3 — small footer cleanup + stale doc removal (2 files, mostly deletions):** the `.site-footer { padding: 1.5rem 0 }` rule in `_custom.scss` overrode the layout-tools control over footer spacing — removed. The A001/4 cloud session left a planning doc at `sessions/theme-typography-bucket-rendering-gap.md` flagging the same typography bug iteration 2 fixed; deleted as stale.

**Merge from main:** A001/4 (cloud session, the fourth A001 iteration) landed in main during the session — extends `exportSite()` to enumerate all A001 entities and rewires the nav widget's `navigation_menu_id` round-trip. Merged in with one VERSION conflict (resolved to `0.311.02`, keeping the 311 track; A001 is on the 901 track per the new dual-numbering policy).

## Drift

- The original Table-widget session prompt referenced `database/seeders/WidgetTypeSeeder.php` as the registration site for new widgets. The actual registration site is `app/Providers/WidgetServiceProvider.php` — `WidgetTypeSeeder` is just a stub that wipes legacy handles and calls `WidgetRegistry::sync()`. This was noted during the Table-widget implementation before the rewind; recording here in case the prompt is rewritten.
- The user surfaced that this is the first session in a while that prompted a `db:seed --class=WidgetTypeSeeder` step. Reason: prior sessions touched existing widgets — the `widget_types` row already existed and `WidgetRegistry::sync()` was a no-op. The Table widget was the first new-handle addition in a while; nothing in the request path auto-syncs, so a new handle requires the seeder run. Worth knowing for the next new-widget session.
