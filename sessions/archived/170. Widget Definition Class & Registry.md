# 170. Widget Definition Class & Registry

First step in the multi-session Sovereign Widget arc: introduce a per-widget code model (a `WidgetDefinition` class) and a `WidgetRegistry` service. Convert the nav widget as proof-of-concept. Other widgets continue working unchanged via a coexistence path. This session is entirely additive ‚Äî no destruction of existing UI, render pipeline, or seeder entries for other widgets.

---

## Design decisions (resolved before starting)

- **Definition class is abstract, not an interface.** Use `abstract class WidgetDefinition` so shared helper methods (e.g. `template()` path resolution, default `fullWidth()` / `category()` returns) live once.
- **Namespace:** `App\Widgets\Contracts\WidgetDefinition` for the base class. Concrete widget classes live at `App\Widgets\{Name}\{Name}Definition` ‚Äî e.g. `App\Widgets\Nav\NavDefinition`. The `Definition` suffix makes the purpose unambiguous and leaves room for companion classes (demo data providers, interaction handlers) under the same namespace later.
- **Registry is a singleton.** `App\Services\WidgetRegistry` is registered as a singleton in a service provider. Widget definitions self-register via the provider's `boot()` method ‚Äî no auto-discovery magic in Stage 1.
- **DB remains the runtime source of truth.** The registry is a *code* layer that syncs *to* the `widget_types` table. Renderer, inspector, WidgetType::forPicker() all continue reading from the DB. The registry doesn't change runtime behavior ‚Äî only the authoring surface.
- **File colocation deferred.** The nav widget's Blade template stays at `resources/views/widgets/nav.blade.php` for this session. Same for `_navigation.scss`. The definition class references those existing paths. Physical file relocation happens in Stage 3.
- **Seeder coexistence.** The nav widget's entry is removed from `WidgetTypeSeeder`. Remaining widgets stay in the seeder unchanged. A new step runs *after* the seeder that calls `app(WidgetRegistry::class)->sync()` to write registry-declared widgets to the DB. This way a fresh `migrate:fresh --seed` produces the same widget_types rows as before.
- **No new DB columns.** Everything the definition class emits maps to existing `widget_types` columns. This keeps the migration story clean.
- **Tests stay working.** `ChromeWidgetTest`'s "seeds logo and nav widget types" and nav rendering tests must still pass. `WidgetType::where('handle', 'nav')->firstOrFail()` still resolves.

## Phase 2 ‚Äî WidgetDefinition Base Class

Create `app/Widgets/Contracts/WidgetDefinition.php`. Abstract class with:

**Required abstract methods** (concrete widgets must implement):

- `handle(): string` ‚Äî unique widget handle, e.g. `'nav'`
- `label(): string` ‚Äî display label, e.g. `'Navigation'`
- `description(): string` ‚Äî short description for picker
- `schema(): array` ‚Äî field definitions, same shape as today's `config_schema` array
- `defaults(): array` ‚Äî key/value map of concrete defaults for every field declared in `schema()`. **Every schema field must have a corresponding defaults entry.** Stage 2 (defaults-binding session) will enforce and leverage this; in Stage 1 we just establish the contract.
- `template(): string` ‚Äî the Blade template string, e.g. `"@include('widgets.nav')"`

**Optional methods** (defaulted in base class):

- `category(): array` ‚Äî default `['content']`
- `collections(): array` ‚Äî default `[]`
- `assets(): array` ‚Äî default `[]`
- `fullWidth(): bool` ‚Äî default `false`
- `defaultOpen(): bool` ‚Äî default `false`
- `allowedPageTypes(): ?array` ‚Äî default `null` (meaning "all")
- `renderMode(): string` ‚Äî default `'server'`
- `requiredConfig(): ?array` ‚Äî default `null`

**Introspection methods**:

- `toRow(): array` ‚Äî returns the array shape that `WidgetType::updateOrCreate()` expects. Builds from the other methods.

**Validation hook:**

- `validate(): void` ‚Äî called during sync; throws if `defaults()` is missing keys declared in `schema()`. This catches authoring errors at sync time.

## Phase 3 ‚Äî WidgetRegistry Service

Create `app/Services/WidgetRegistry.php`. Responsibilities:

- Holds an array of registered `WidgetDefinition` instances, keyed by handle.
- `register(WidgetDefinition $def): void` ‚Äî add a definition.
- `all(): array` ‚Äî return all registered definitions.
- `find(string $handle): ?WidgetDefinition` ‚Äî lookup.
- `sync(): void` ‚Äî iterate all definitions, call `validate()`, then call `WidgetType::updateOrCreate(['handle' => $def->handle()], $def->toRow())` for each. This is the bridge from code to DB.

Register as a singleton in a dedicated service provider: `app/Providers/WidgetServiceProvider.php`. The provider's `register()` creates the singleton; `boot()` registers concrete definitions. Add the provider to `config/app.php` (or `bootstrap/providers.php` in Laravel 11+).

## Phase 4 ‚Äî Nav Widget Definition

Create `app/Widgets/Nav/NavDefinition.php` extending `WidgetDefinition`. Port the current nav widget's seeder entry one-to-one:

- `handle()` ‚Üí `'nav'`
- `label()` ‚Üí `'Navigation'`
- `description()` ‚Üí the current description string
- `category()` ‚Üí `['layout']`
- `fullWidth()` ‚Üí `true`
- `assets()` ‚Üí `['scss' => ['resources/scss/widgets/_navigation.scss']]`
- `schema()` ‚Üí the full array currently in the seeder (all 16+ fields, including the `heading` notice and all groupings)
- `defaults()` ‚Üí extracted from the `default` keys in each schema field. This may reveal fields with no concrete default (e.g. `'default' => ''` on colors) ‚Äî for Stage 1, keep those as-is (empty strings). Stage 2 will tighten them.
- `template()` ‚Üí `"@include('widgets.nav')"`
- `requiredConfig()` ‚Üí the current `['keys' => ['navigation_menu_id'], 'message' => ‚Ä¶]`

Register it in `WidgetServiceProvider::boot()`:

```php
public function boot(): void
{
    $registry = $this->app->make(WidgetRegistry::class);
    $registry->register(new \App\Widgets\Nav\NavDefinition());
}
```

## Phase 5 ‚Äî Seeder Integration

Update `database/seeders/WidgetTypeSeeder.php`:

- Remove the `WidgetType::updateOrCreate(['handle' => 'nav'], ‚Ä¶)` block entirely.
- At the end of `run()`, call `app(\App\Services\WidgetRegistry::class)->sync()`. This writes all registry-declared widgets to `widget_types` after the seeder-declared ones.

Order matters: seeder entries run first, then registry sync. This matches the "additive coexistence" model ‚Äî registry wins on handle collision (since `updateOrCreate` on the same handle overwrites).
