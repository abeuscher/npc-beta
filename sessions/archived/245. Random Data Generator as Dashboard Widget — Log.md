## What was built

### Phase 1 — `Source::SCRUB_DATA` taxonomy

- New `Source::SCRUB_DATA = 'scrub_data'` constant added to `app/WidgetPrimitive/Source.php`; included in `Source::KNOWN`.
- `ACCEPTED_SOURCES` extended on six models — Contact, Donation, EventRegistration, Membership, Transaction, Page — to accept the new value.
- Schema docs updated for contacts, donations, event_registrations, memberships, transactions (events.md and pages.md updated later when their `source` columns landed via in-session scope expansions).
- `SourceTest` extended to assert the seventh constant + KNOWN list shape; `HasSourcePolicyTest` extended with paired positive `Model/SCRUB_DATA accepted invariant` tests next to each existing `Model/DEMO security invariant` test. The session 213 DEMO/Donation invariant stays unchanged and still passes.

### Phase 2 — `EnforcesScrubInheritance` trait

- New trait `app/WidgetPrimitive/EnforcesScrubInheritance.php` enforcing three rules: (1) infectious downward via FK, (2) sticky source on update, (3) any-scrub-parent-wins precedence on multi-parent rows.
- New exception `app/WidgetPrimitive/Exceptions/ScrubSourceLocked.php` extending `LogicException` (mutation attempts on locked source = programmer error).
- Trait applied to Contact, Donation, EventRegistration, Membership, Transaction. Contact uses the default empty `scrubInheritsFrom()`; Donation/Membership/EventRegistration declare `['contact_id' => Contact::class]`; Transaction declares both polymorphic subject + contact_id.
- New test file `tests/Unit/WidgetPrimitive/EnforcesScrubInheritanceTest.php` — 12 tests covering all three rules, both no-op cases (already-scrub source skip; non-scrub-parent skip), polymorphic resolution including the non-existent-subject edge, both directions of multi-parent precedence, and cross-model inheritance through Membership and EventRegistration.

### Phase 3 — Generation service

- New file `app/Services/RandomDataGenerator.php` — pure-backend service with `generate(array $counts): array` orchestrator (super-admin assertion at entry, `DB::transaction` wrap, per-type method dispatch, unified summary aggregation).
- Per-type methods: `generateContacts`, `generateEvents`, `generateRegistrations`, `generateDonations`, `generateMemberships`, `generatePosts`, `generateProducts` (the last two arriving in later iterations).
- Custom-field type respect via `fakeCustomFields()`: text/rich_text → fake string/paragraph; number → 1–100; date → ISO date; boolean → bool; select → random from declared options.
- Ledger discipline: active donations and active+paid memberships write matching `Transaction` rows (per session 233's discipline). Pending and zero-amount records do not.
- `EventRegistration::withoutEvents()` wraps the registration writes — surfaced during testing that `EventRegistrationObserver::created` was firing per-registration `Mail::to(...)->send(new RegistrationConfirmation(...))` AND auto-creating real-source-tagged Contacts via `Contact::firstOrCreate(['email' => ...])`. Suppressing the observer avoids both side-effects in the generator's flow.
- New test file `tests/Feature/RandomDataGeneratorServiceTest.php` — 25 tests by session end (started at 13 for Phase 3 alone; expanded as Phase 5 / blog-posts / products / images landed).

### Phase 4 — Contract widget + admin route + controller

- New folder `app/Widgets/RandomDataGenerator/` with `RandomDataGeneratorDefinition.php` and `template.blade.php`. Definition declares `allowedSlots(): ['dashboard_grid']`, `category(): ['admin']`, no schema/config, no dataContract.
- Template gates rendering at the top with `auth()->user()?->isSuperAdmin()`. Below the gate: an Alpine-driven generation form, an Alpine modal showing dynamic totals on the form's count inputs, classic-HTML POST to `route('filament.admin.dev-tools.random-data.store')` with `@csrf`. Pattern reference: `app/Widgets/PortalContactEdit/template.blade.php` — proves the contract widget surface supports admin-interactive widgets without contract-system changes.
- New controller `app/Http/Controllers/Admin/RandomDataGeneratorController.php` with `store()` method. Three-layer permission gate: widget template, Filament `Authenticate` middleware on the route, controller method body abort. Validation: `counts.{type}` integer min:0 max:1000.
- Route registration in `AdminPanelProvider`'s `->routes()` callback under panel prefix `dev-tools/`. Final route name auto-prefixed by Filament to `filament.admin.dev-tools.random-data.store`.
- Widget registered in `WidgetServiceProvider`. Widget_types row created via `db:seed --class=WidgetTypeSeeder`.
- CSS rules added to `public/css/admin.css` (hand-edited per the admin CSS rule) for `np-random-data-generator__*` classes — BEM-pattern matching the existing dashboard-widget styles.
- Cross-cutting test surface deferred to Phase 6c. Phase 4 itself shipped without dedicated controller tests.

### Phase 5 — Wipe service + scrubCounts + wipe form

- `RandomDataGenerator::wipe()` method — super-admin gated, FK-safe ordered deletes inside `DB::transaction`. Order: transactions → registrations → donations → memberships → events → posts → products → contacts. `withTrashed()->forceDelete()` on Contact and Membership (both use SoftDeletes); plain `delete()` on others; per-row `forceDelete()` for Pages so `PageObserver::deleting` fires and cascades widgets/layouts (bulk query-builder delete bypasses observers).
- `RandomDataGenerator::scrubCounts()` method — per-table `WHERE source = 'scrub_data'` counts, `withTrashed` on Contact/Membership.
- Widget template extended: Wipe section with pre-rendered scrubCounts, Alpine modal showing per-table totals, danger-styled confirm button, danger-button CSS variant.
- Controller `wipe()` method + new route `random-data.wipe`. Same three-layer permission gate.
- 6 new wipe tests (auth gate, no-op on clean install, generate-then-wipe round-trip preserves real data, inheritance + wipe interaction for downstream rehearsal-flow rows, scrubCounts accuracy, soft-deleted Contact rows hard-deleted via withTrashed+forceDelete chain).

- **Blog post generation** (originally not in scope at all). New `generateBlogPosts($n)` method. Posts are `Page` rows with `type='post'`, `status='published'`, `published_at` randomized in last 2 years, `author_id = auth()->id()`, slug under blog prefix. - **Image attachments** via Spatie Media Library on events / blog posts / products. New `attachPoolImage($model, $category, $collection)` helper lifted from the old generator's pattern. Each model's `registerMediaConversions()` fires automatically (responsive WebP variants per the model's profile). Silent try/catch.
- **Product generation** (lifted from out of scope). New migration `2026_04_29_140000_add_source_to_products.php`. ## Production deployment notes

1. **Three migrations** to run via standard `php artisan migrate`: `add_source_to_events`, `add_source_to_pages`, `add_source_to_products`. Existing rows on each table backfill to `source = 'human'` (or `'import'` for events with import_session_id). No data loss.
2. **Widget registration**. Run `php artisan db:seed --class=WidgetTypeSeeder` to write the `random_data_generator` row to the `widget_types` table. Required for the widget to appear in the picker.
3. **Super_admin DashboardView**. Existing installs where the super_admin DashboardView already has rows will NOT auto-receive the new widget (`DashboardViewSeeder` is idempotent — only seeds when the view has zero pageWidgets). Operator must manually add the Random Data Generator widget via Tools → Dashboard View.
4. **APP_DEBUG_TOOLS env var path retired**. Remove `APP_DEBUG_TOOLS=...` lines from production `.env` if present. The variable is no longer read by any code.
5. **Wipe in production**. Source-scoped delete is safe (no real-data row carries `source = 'scrub_data'` unless an operator explicitly set it). Recommend operators run `SELECT COUNT(*) FROM contacts WHERE source = 'scrub_data'` (and equivalent for donations / memberships / event_registrations / transactions / events / pages / products) before clicking Confirm Wipe on any shared install.
6. **Scrub data DOES NOT exist on production by default**. The generator widget renders only for super-admin role users; the routes are super-admin-gated. The widget is harmless on installs that aren't actively rehearsing.
7. **Stripe production safety reminder**. Per the new release-blocker stub: a production install configured against a real (non-test) Stripe account could see an operator generate scrub donations that create real Stripe activity. The Stripe Test-Mode Detection guard ships in a future session before Beta-1.
8. **Image attachments leak orphan media**. Generated content (events, blog posts, products) attaches sample images via Spatie Media Library. These conversion files persist on disk after `wipe`. The pair-tooling for cleanup is the `app:reset` artisan command (session 247) + scheduled `media-library:clean` (also session 247).
