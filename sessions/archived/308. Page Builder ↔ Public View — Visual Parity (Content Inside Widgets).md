# 308. Page Builder ↔ Public View — Visual Parity (Content Inside Widgets)

> **Critical-path, Beta-1-blocking polish.** Drives the page-builder canvas and the public render of the same widget to visual parity for content inside widgets — the canvas should look like the public site once the zoom transform and tooling overlays are subtracted. The forcing function is concrete: rich-text editing in the builder is currently producing visual divergences (double bullets on `<ul><li>` items being the named example) that the user can see at the canvas and that the HTML-layer parity guard from session 307 will not catch.

One sentence: stand up a real-browser per-element computed-style diff between the canvas-rendered widget DOM and the public-rendered widget DOM for the inline-editable widget set, root-cause the double-bullet bug first, sweep the resulting diff set, fix what's a CSS-scope or Quill-injected leak, and flag the residual unfixable / intentional-divergence list to the user.

## The forcing-function bug, by name

`<ul><li>` items in a TextBlock widget render **one** browser-native bullet on the public site and **two** on the builder canvas. The second bullet is visible on the canvas but never on the public render of the same widget. This is the canonical example of the bug class 308 catches; root-causing it is the first deliverable, and how it root-causes determines whether the rest of the audit is "find siblings of one shared cause" or "find N independent bugs."

Two hypothesis families to probe in Phase 1:

- **Quill snow-theme leak.** The Quill `.ql-snow` chrome was stripped by session 304's WYSIWYG fix, but the strip may not have covered the list-marker rule (`.ql-snow .ql-editor ol li::before` / `ul li::before` — Quill renders its own bullet via `content: '\\2022'`). If the strip missed those, browser-native markers stack on top of Quill's pseudo-element bullets.
- **CSS namespace leak.** A rule scoped only to `.np-site ul li` (public) doesn't apply in the builder, or vice versa — a rule scoped under `.widget-preview-scope` runs in the builder but not the public site. Either direction can introduce or hide a marker.

Both are testable against the live builder. The harness built in Phase 2 will distinguish them.

## Phases

### Phase 1 — Root-cause the double-bullet bug

Reproduce the bug in a clean test page (TextBlock widget, a `<ul>` with two `<li>` items, no other formatting). Inspect the rendered DOM in both surfaces, identify the source of the second marker, and decide which hypothesis family it belongs to. The output of this phase is one or two sentences describing the cause + a one-line fix. Apply the fix in this phase (it is the smallest piece of the session).

If the root cause is shared (one rule scoped wrong, one Quill chrome strip incomplete), Phase 3 becomes "verify the fix removed the rest of the diff set" rather than "audit dozens of distinct bugs." If the root cause is local (just this one bug), Phase 3 becomes a real audit. Decide which after Phase 1, before drafting the harness.

### Phase 2 — Standing visual-parity harness

A new Playwright spec at `tests/e2e/page-builder/widget-visual-parity.spec.ts` exercises the inline-editable widget set (the 13 widgets opted in at session 307: TextBlock, Hero, ThreeBuckets, PricingChart, BarChart, BlogListing, BoardMembers, DonationForm, EventCalendar, EventsListing, MapEmbed, ProductCarousel, SocialSharing — re-confirm the list from `WidgetDefinition::inlineEditable()`).

For each widget:

- Create a page with that widget configured to a known-content state (the same Pest factories used by `BuilderPublicRenderParityTest`, lifted into the e2e helper).
- Open the page-builder canvas at the widget's natural zoom level. Subtract the zoom transform (read it from the canvas wrapper, divide computed dimensions through it, or set the zoom to 100% for the duration of the spec — whichever is cleaner against the existing canvas plumbing).
- Open the public render of the same page in a second browser context.
- Walk the widget DOM in both surfaces in lockstep (same selector path). For each visible element, capture the relevant subset of `getComputedStyle` — font-family, font-size, font-weight, line-height, color, background-color, margin, padding, border, list-style-type, list-style-position, text-decoration, text-align, white-space, display, position. Diff the two snapshots; fail the spec on any divergence not in the documented expected-diff set.

Documented expected-diff set, locked into the harness once:

- The zoom transform (subtracted before comparison).
- Drag-handle / "Edit" affordance / inline toolbar / selection outline — these sit outside the widget subtree; the harness compares only inside `[data-widget-id]` (or the equivalent selector the builder uses for widget root) so they are naturally excluded.
- Contenteditable caret + `::selection` artifacts on the active editor.
- The session-307 HTML-layer differences (placeholder attribute, empty editable wrappers) — those don't carry computed style, but mention them in the harness preamble for the next reader.

Sits in the e2e suite (now un-parked as of 2026-05-19) and runs on every CI push, non-gating like the rest of the Playwright suite.

### Phase 3 — Sweep and fix

Run the harness. For each surfaced diff:

- **Fix** if the cause is a CSS-scope leak that can be cleanly resolved by tightening the selector (`.np-site` → `.np-site, .widget-preview-scope` or vice versa), or a Quill chrome rule that the 304 strip missed, or a stale rule from an earlier era.
- **Flag as intentional divergence** if the diff is between the builder's editing affordance and the public render — e.g., contenteditable caret styling, focused-element outline, the inline toolbar's positional anchor.
- **Flag as unfixable** if the diff is intrinsic to Quill's contenteditable rendering and cannot be cleanly scoped out without breaking the editor. Each unfixable gets a one-line note in the close-log for the user to disposition.

Default-to-fix per the existing audit-findings rule in memory — surface inclusively, the user dispositions the "won't-fix" set during the apply walkthrough at session close.

### Phase 4 — Lock the exclusion set into the harness

Whatever diffs survive Phase 3 as "intentional" get encoded in the harness as an explicit exclusion list — selector + computed-style key + a one-line rationale. The harness fails on anything outside this list. This makes the next regression cheap to catch: future Quill or toolbar work that introduces a new divergence trips the spec on the next CI push.

`DesignGroupIntegrityTest`'s pinned list is unchanged (this is a Playwright spec, not Pest); no design-group integrity bookkeeping this session.

## Out of scope

- Anything outside the widget DOM in the canvas — drag handles, edit affordance, inline toolbar, selection outlines, the canvas chrome itself. Intentional divergence by design.
- The zoom transform. Subtracted, not removed.
- Pixel-perfect cross-browser rendering. The harness runs against Chromium (the e2e default); other browsers are out of scope.
- Any change to the toolbar shipped at 306, the eligibility gate landed at 307, or the HTML-layer parity guard from 307.
- Any Fleet Manager / `/api/health` / response-schema / DB-schema touch. CRM contract stays at v2.3.0.
