# 072. Activity Log

Build an auditable activity stream that records CUD operations on key records, attributed to the actor who caused them. On the contact record, the existing Notes sub-page becomes a unified Timeline showing both human-authored notes and system activity entries in a single chronological stream. Admin users' activity is also surfaced on their own edit page.

---

## Design decisions (resolved before starting)

**Storage:**
- New `activity_logs` table — separate from `notes`, polymorphic, covers all model types.
- Schema: `id` (bigint PK), `subject_type` (string), `subject_id` (uuid string), `actor_type` (string: admin/system/portal), `actor_id` (bigint, nullable), `event` (string: created/updated/deleted/restored), `description` (string, nullable — for context), `created_at` (timestamp). No `updated_at` — records are immutable.
- Index on `(subject_type, subject_id)` for per-record queries. Index on `(actor_type, actor_id)` for admin user activity queries.

**What is logged:**
- CUD only — Create, Update, Delete (and Restore for soft-delete models). No read events, no browsing behaviour.
- Models in scope: Contact, Membership, Donation, Page, Event.
- For `updated` events: record that the model changed and who changed it — no field-level diff.
- For Page and Event: add a human-readable `description` when `is_published` or `status` changes (e.g. "Published", "Unpublished", "Status changed to cancelled").

**Actor attribution:**
- `admin` — authenticated user on the `web` guard. `actor_id` = `users.id`.
- `portal` — authenticated user on the `portal` guard. `actor_id` = `portal_accounts.id`.
- `system` — no authenticated user (import, API, web form, auto-creation). `actor_id` = null.

**Implementation:**
- Laravel model Observers — one per loggable model, registered in `AppServiceProvider`.
- No third-party auditing package.
- A shared `ActivityLogger` service resolves the current actor from auth guards and writes the entry. If logging fails for any reason it must fail silently — never break the primary operation.

**Timeline UI (contacts):**
- The existing ContactNotes sub-page becomes the Timeline: same page, same route, title changes to "Timeline".
- Shows notes and activity log entries merged in descending chronological order.
- Notes retain their edit/delete actions. Activity log entries are read-only.
- The "Notes" button label on the `EditContact` header is updated to "Timeline".
- The merge is done in PHP (fetch both collections, merge, sort) — not a SQL UNION. Cap the combined result at 200 entries.

**Admin user activity:**
- Same `activity_logs` table. A read-only collapsible section (collapsed by default) on the User edit page showing that user's most recent 50 activity entries (`actor_type = admin`, `actor_id = user.id`), sorted newest first.

## Phase 1 — Schema and infrastructure

1. Write a migration creating the `activity_logs` table with the columns and indexes described above.
2. Create `App\Models\ActivityLog` — fillable, `timestamps = false`, `created_at` cast as a Carbon datetime, `CREATED_AT` constant set (no `UPDATED_AT`).
3. Create `App\Services\ActivityLogger` with a single static method:
   `log(Model $subject, string $event, ?string $description = null): void`
   — resolves the current actor by checking the `web` guard then the `portal` guard, writes to `activity_logs`, wraps in a try/catch so failures are silent.

Update `docs/schema.md`.

---

## Phase 2 — Observers

Read the current `AppServiceProvider` and each target model before writing observers.

Create one observer class for each model: `ContactObserver`, `MembershipObserver`, `DonationObserver`, `PageObserver`, `EventObserver`.

Each observer implements `created`, `updated`, `deleted`, and `restored` (where the model uses SoftDeletes), calling `ActivityLogger::log()` in each.

`PageObserver::updated` and `EventObserver::updated`: check `$model->wasChanged('is_published')` / `$model->wasChanged('status')` and set a meaningful `description` if so (e.g. "Published", "Unpublished", "Status changed to cancelled").

Register all five observers in `AppServiceProvider::boot()`.

---

## Phase 3 — Contact Timeline

Read `ContactNotes.php` and its Blade view before modifying either.

- Update the page title from "Notes" to "Timeline".
- Update the `Notes` button label in `EditContact::getHeaderActions()` to "Timeline".
- Extend the page to fetch both the contact's notes and its activity log entries, merge them into a single collection sorted by `created_at` descending (cap at 200), and pass the merged collection to the view.
- Render the merged list. Notes display as before with edit/delete actions. Activity log entries display as read-only rows showing: event label, actor attribution (e.g. "by Admin Name", "by system", "by portal member"), description if present, and timestamp.

---

## Phase 4 — Admin user activity

Read the User edit page before modifying it.

Add a collapsible "Activity" section (collapsed by default) to the User edit form's right column. Query the 50 most recent `activity_logs` entries where `actor_type = 'admin'` and `actor_id = user.id`. Display as a read-only list: event, subject type (human-readable), description if present, timestamp.

## Security checklist

Before closing:

- [ ] `ActivityLogger::log()` is wrapped in try/catch — logging failures never surface to the user or break the primary operation.
- [ ] No route, action, or UI allows editing or deleting activity log entries.
- [ ] Portal actor attribution uses the authenticated portal guard user only — no cross-contact attribution possible.
- [ ] Admin activity section on the User edit page is gated by existing user-view permissions.
