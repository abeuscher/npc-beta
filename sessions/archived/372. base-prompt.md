## Process rules for every session

Base-prompt process rules apply in full (`sessions/template-base-prompt.md`). Session-specific:

- **Boundary-touching — the Two-Repo Coordination Protocol is mandatory, not optional.** Before implementing the app-layer gate, update `docs/fleet-manager-agent-contract.md` (body describing the new app-layer auth requirement + a CHANGELOG entry + bump `Contract Version` to `2.7.0`) and the "Cross-Repo: Fleet Manager / CRM" block in `sessions/session-outlines.md`. The contract is the source of truth both repos build against; it changes first.
- **Additive, with a rollout window — no flag day.** Fleet Manager must keep polling and operating while it absorbs the new credential. A change that hard-rejects every FM request the instant it deploys (before FM sends the new credential) is a breaking change, not additive. Decide the rollout posture at the Open Gate — e.g. gate-enforced-but-env-flippable, or accept-either-during-a-window — so FM absorbs on its own cadence.
- **Worst-first.** `/api/backup/blob` (the whole donor DB) and `/api/admin/recover` (admin credential reset) get the app-layer gate first and are proven first; the other three (`/health`, `/logs`, `/backup/trigger`) follow the same pattern. The gate is defense-in-depth *behind* the mTLS termination — it does not replace nginx's `ssl_client_verify` check, it backstops it.
- **The mechanism is an in-session design call** (surface at the Open Gate, don't pre-commit): a shared-secret header FM already possesses / can be provisioned at install time, vs. forwarding the verified client-cert fingerprint from nginx to the app for independent re-verification. Both are additive; they differ in what survives an nginx misconfig and in what FM must send. Recommendation formed after the code read, confirmed at the Open Gate.
- **`/api/logs` PII tension is in scope** (surface-map low-risk note folded into S2): the endpoint ships raw application log lines (possible PII / stack traces) against the counts-only privacy boundary the fleet design holds elsewhere. Resolve it — redaction, scoping, or a documented conscious acceptance on the Gate-3 register — as part of this session.
- **Security over usability** (standing project rule): when the two conflict — e.g. the gate adds friction to FM's polling — choose security and move the friction to the coordination/rollout layer, not by weakening the gate.
- **Do not touch:** the CSP/headers policy from S1 or the XSS/exposure work from S3 (only rely on them); the `/api/health` response schema beyond the additive auth note (FM polls it every minute — a schema change ripples).
