# 216. Widget Primitive — Phase 4a (BlogListing Retrofit — Audit, Test-Lock, Cleanup)

Begin the Phase 4 widget retrofit campaign by auditing BlogListing — the smallest-scope page-builder widget with an already-declared contract — locking its contract-bound data path in with explicit load-bearing tests, and deleting any now-dead direct-model code in its neighborhood. This session is intentionally narrow: prove the retrofit pattern is mostly audit + test + cleanup before committing to the remaining ~19 widgets.

---

## Design decisions (resolved before starting)

- **BlogListing is already substantially retrofitted** — its `BlogListingDefinition::dataContract()` declares `SOURCE_SYSTEM_MODEL` with `model: 'post'` and a seven-field whitelist; `ContractResolver::resolvePost()` fetches Posts through `Page::where('type', 'post')->published()`; `SystemModelProjector::projectPost()` constructs the DTO; and the template already reads `$widgetData['items']`. This session does **not** redesign the contract. It audits, test-locks, and deletes dead code.
- **Scope is strict.** No new contract filters (`order_by`, `limit`, pagination). No widening the projector's field map. No client-side behavior changes. If the audit surfaces a legitimately missing feature (e.g. the contract ignores `sort_default` config), it is noted as a deferred decision in the log and left for a follow-on session. The point of this session is to prove the Phase 4 retrofit pattern works and is testable, not to extend any one widget.
- **Include the dead-code cleanup.** Any direct-Post / Page-post Eloquent access that lives inside the BlogListing widget folder, or any import/trait/helper that was only used by BlogListing's pre-contract data path, is removed. "Dead" means: the reference has no caller after the retrofit, as confirmed by grep. No backwards-compatibility shims.
- **Add one explicit test that proves the assumption.** Create `tests/Feature/BlogListingContractRetrofitTest.php` with two assertions per the user's direction:
  - (a) BlogListing's rendered HTML for a given Post reflects only the projector's field map — fields outside the contract declaration (e.g. `meta_description`, `author_id`, `id`) do not appear in `$widgetData['items']` rows.
  - (b) The render path produces zero direct Eloquent queries against `posts`-as-Post-type outside of `ContractResolver::resolvePost()`. Use `DB::listen()` or `DB::getQueryLog()` to capture queries; assert the query shape matches the resolver's `Page::where('type', 'post')->published()->with('media')->orderByRaw(...)` pattern. One query through the resolver, no others.
- **The existing "renders a blog listing widget end-to-end through the contract resolver" test in `tests/Feature/WidgetPrimitiveIntegrationTest.php` stays as-is.** The new retrofit-lock test is strictly additive — it asserts the invariant the existing test relies on but doesn't explicitly enforce.
- **No changes to asset bundles.** `styles.scss`, `script.js`, and the `libs: ['swiper']` declaration are unchanged. `build:public` is not required unless a file listed in `BlogListingDefinition::assets()` changes.
- **No schema changes.** No migration, no `docs/schema/` updates.

- Confirm `BlogListingDefinition` does not declare `collections()` (i.e., does not route through the legacy `WidgetDataResolver::resolve($collHandle, …)` path in `WidgetRenderer`).

Its current map (`id`, `title`, `slug`, `url`, `date`, `date_iso`, `excerpt`, `image`) is the allowed surface; the contract requests a subset of these (no `id`), so `id` is projected but not exposed by the contract.

Specifically the `WidgetDataResolver::resolve()` call at line ~49 — since BlogListing declares no `collections()`, the `$widgetType->collections ?? []` loop is empty and that path is skipped. Record the confirmation in the log.

The audit is a deliverable — write it into the session log, don't just run the greps and move on. The audit is what makes the retrofit pattern reproducible for the remaining widgets.

## Phase 2 — Test-lock

Create `tests/Feature/BlogListingContractRetrofitTest.php` using Pest + `RefreshDatabase`. Two tests at minimum:

### Test A — Fail-closed field whitelist

Create a `Page` of `type: 'post'` with rich attributes (`title`, `slug`, `published_at`, `meta_description`, `author_id`, etc., including fields the projector's map omits). Mount a BlogListing widget with a minimal `content_template` that tries to render both whitelisted and non-whitelisted fields:

```
<p class="title">{{item.title}}</p>
<p class="meta">{{item.meta_description}}</p>
<p class="author">{{item.author_id}}</p>
<p class="internal">{{item.id}}</p>
```

Render the widget via `WidgetRenderer::render($pw)`. Assert:

- The HTML contains the post's title (whitelisted, present in contract `fields`).
- The HTML does NOT contain the meta_description text, the author_id value, or the post id — the template's `{{item.meta_description}}`, `{{item.author_id}}`, `{{item.id}}` tokens are substituted with empty strings because those fields are not declared on the contract.
- The HTML's `.meta`, `.author`, and `.internal` paragraphs exist but are empty (not absent — the template's structure is preserved; the projection just feeds no value).

This is the load-bearing security assertion: a widget template author cannot accidentally leak a non-whitelisted field by referencing it as a token.

### Test B — Resolver is the only path

Create 3 published posts and one draft post. Mount a BlogListing widget on a host page. Enable query logging via `DB::enableQueryLog()`. Render the widget. Assert:

- `$widgetData['items']` contains exactly 3 rows (drafts excluded by `published()`).
- The executed SQL includes exactly one `select` against `pages` filtered by `type = 'post'` and a `published` guard — i.e., the query `ContractResolver::resolvePost()` would emit.
- No additional `pages` selects happen during the render pass (no N+1 per-post lookup, no second "get all posts" for anything else).
- The executed SQL for `media` is one eager load (the resolver's `->with('media')`), not per-post.

Record the exact observed query count in the test for regression pinning — if a future change doubles the queries, the test fails loudly.

Both tests go in the fast suite (individual runtime well under 5s). No `@group('slow')` annotation.

## Phase 3 — Cleanup

Based on Phase 1's audit, delete anything that is demonstrably dead:

- Unused `use` statements in `BlogListingDefinition.php` or the template.
- Any helper class/method/scope that was only used by BlogListing's pre-contract data path. Grep before deleting — if it has another live caller, leave it.
- Dead comments that describe the old data path. Example: comments like "// fetched via WidgetDataResolver" next to code that now goes through the contract. Rewrite or remove.

**Do not delete:**
- `SystemModelProjector::projectPost()` — actively used.
- `ContractResolver::resolvePost()` — actively used.
- The `WidgetDataResolver::resolve()` path in `WidgetRenderer::render()` — still used by every widget that declares `collections()`. Not BlogListing's concern.
- Any code that is used by other widgets even tangentially.

This phase may produce an empty diff if the audit confirms everything is already clean. That is a valid outcome — the session log records it and moves on.

## Out of scope

- **Adding `order_by`, `limit`, pagination, or search filters to the BlogListing contract.** Real feature work, not retrofit. Deferred.
- **Widening `SystemModelProjector::projectPost()`'s field map.** Each added field is a fail-closed boundary loosening; do it deliberately, not as part of this audit.
- **Retrofitting other widgets** — EventsListing, Carousel, Faq, etc. Each gets its own Phase 4 session.
- **Client-side behavior changes** — the Alpine.js pagination/search/sort all stays as-is.
- **Asset bundle touch-ups** — no SCSS, JS, or `assets()` changes.
- **Public-page-builder convergence (Phase 6)** — not triggered by this session. Happens later once more widgets are retrofit-locked.
- **Memos composer widget** — orthogonal write-side work, separate session.
- **Dashboard grid visual previewing in the builder (the 215 "builder stacks vs dashboard two-across" observation)** — still out of scope, still not implementing per-widget `dimensions`.
