## Process rules for every session

- **Before writing any code**, read every file you intend to modify.
- **Before implementing any integration** (email, file storage, authentication, payments, queues, or any other infrastructure), ask the user what existing system handles it. This project has accumulated significant infrastructure — assume something already exists before building new. Do not implement from scratch until you have confirmed no existing system covers the use case.
- **Pause and ask** any time a decision point arises that is not covered by the agreed prompt.
- **If any external service is unavailable**, stop and ask — do not attempt to troubleshoot.
- **Run migrations via Docker** after writing them: `docker compose exec app php artisan migrate`. Do not pause to ask first. **This session writes one migration adding `source` columns and backfilling rows across four tables.**
- **Update `docs/schema/`** after writing any migration — this session updates `donations.md`, `memberships.md`, `event_registrations.md`, and `transactions.md`.
- **If the PostgreSQL container becomes unresponsive** (500/exec errors), ask the user to restart it rather than retrying.
- **When implementation is complete**, run the fast test suite: `docker compose exec app php -d memory_limit=2G vendor/bin/pest --exclude-group=slow`. The 2G memory override avoids the pre-existing mid-suite OOM fatal documented since session 212. Fix any failures before proceeding.
- **Front-end builds — when to run which:** Pest does not catch Vue/TS errors or SCSS compile failures, so any change to front-end source must be built before announcing completion. Decision rule: *if this change could affect what ends up in the public-site bundle at runtime, run `build:public`; if it's Vite-compiled admin/public/page-builder source, run `npm run build`; if neither, no build.* **This session touches no front-end source — backend-only data-layer + write-path session. Neither `npm run build` nor `build:public` should be required.**
- **Tests for new behaviour:** every session that changes application behaviour should include tests. This session lands a schema migration, model-trait additions, and write-path updates across ~10 controllers/pages — every one gets a test confirming `source` is set correctly on creation.
- **Fast vs slow classification:** tests that individually take >5 seconds belong in a Pest `->group('slow')`. Everything else stays in the default (fast) suite.
- **Playwright artifact hygiene:** `resetDatabase()` fires once per spec in `beforeAll`; anything a test creates after that point persists into the next manual-test session. Delete out-of-band rows in `afterAll`.
- **Pause for manual testing.** Announce that testing is ready. Then stop. Do not suggest closing the session or take any further action.
- **Financial-data session caveat — extra care on backfill.** This session writes to four production tables (donations, memberships, event_registrations, transactions). The backfill SQL must be reversible (the migration's `down()` simply drops the column; existing rows survive). **Before running the migration on the dev install, surface the SQL the migration generates and pause for user inspection.** This is one step of paranoia warranted by the data-criticality of the affected tables.

## Style rules

- Follow existing code conventions exactly.
- Do not add features, refactor, or improve anything beyond what the session specifies.
- Prefer editing existing files to creating new ones.
- No docstrings, comments, or type annotations on code you did not write.
- Simple, correct, well-considered solutions over fast or clever ones.

- **Portal security rule:** every portal route and query that reads contact data must be scoped strictly to the authenticated portal user's own `contact_id`. No exceptions without explicit design review.
- **Security over usability:** when they directly conflict, choose security. Fail-closed beats fail-open; explicit consent beats silent default; narrowly-scoped beats broadly-permissive.
- **No collapse-to-summary panels:** inspector/settings panels display fields directly; do not hide behind chevron disclosure toggles.
