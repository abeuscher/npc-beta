# 311. Table Widget

> **Standard new-widget session.** Adds a Table widget to the Page Builder — rows + columns + header-row toggle + per-cell alignment + optional border style, with rich-text cell content and responsive horizontal overflow on narrow viewports. The architecture is fixed; this session is pattern-application, not pattern-invention. Picked for this slot specifically because it lands entirely outside A001's blast radius.

One sentence: a new widget called Table that produces an HTML `<table>` from operator-authored rows and columns, with rich-text cells, header-row + alignment + border affordances, and mobile-friendly overflow scrolling.

---

## What we want

### Admin authoring

The widget's Inspector form gives the operator:

- **Rows.** A repeater of rows. Each row carries an ordered list of cell objects. Default: 2 rows on a fresh widget.
- **Columns.** A column count picker (2–8). Changing the column count adds/removes cells per row consistently. Default: 3 columns.
- **Header row toggle.** A boolean — when on, row 1 renders as `<thead>` with `<th>` cells; when off, all rows render as `<tbody>` `<td>` cells.
- **Cell alignment.** A per-column alignment picker (left / center / right). Cell-level override is **out of scope** for v1 — column-level is enough for the public-website use case and avoids a per-cell appearance-config explosion.
- **Border style.** A small picker: `none` / `outer` / `all` (outer = table outline only; all = every cell). Default: `all`.
- **Cell content.** Uses the existing rich-text primitive (`RichTextField` shape — same as `PricingChart`'s `attribute_rows.value` cell). Links / emphasis / lists work consistently with the rest of the CMS via the existing Quill + HtmlSanitizer pipeline.

### Public render

The widget renders a semantic `<table>` element:

- Wrapped in a `.np-table-widget` container with `overflow-x: auto` and `max-width: 100%` so narrow viewports get horizontal scroll rather than overflow-out-of-page.
- `<thead>` + `<tbody>` reflect the header-row toggle.
- Column alignment is applied via `text-align` on the `<th>`/`<td>` elements.
- Border style is applied via a single utility class on the wrapper (`np-table-widget--border-{none|outer|all}`) so SCSS owns the visual; no inline borders.
- Cell rich-text content is rendered through the same sanitizer path every other rich-text cell uses (`HtmlSanitizer::sanitize`).

- New widget definition class at `app/Widgets/Table/TableDefinition.php`.
- New blade template at `app/Widgets/Table/template.blade.php`.
- New SCSS partial at `app/Widgets/Table/styles.scss`.

## Architecture

- The widget is a standard `app/Widgets/<Name>/` directory, mirroring `app/Widgets/PricingChart/` exactly.
- Config schema: `rows` (repeater) carrying an array of `cells: [{content: string, …}]` per row; `columns` (int 2–8); `column_alignment` (array of N strings, one per column); `header_row` (bool); `border_style` (string enum).
- The render template walks `rows`, emits `<thead>` for row 0 if `header_row`, `<tbody>` for the rest. Each cell prints sanitized rich-text via `{!! HtmlSanitizer::sanitize($cell['content']) !!}`.
- Column-count change behaviour: when the operator increases columns, each row's cell array is padded with empty cells; when they decrease, surplus cells are trimmed from the right. The trim is non-destructive at the admin layer (a confirmation modal warns if non-empty cells will be dropped).

## Out of scope

- Per-cell alignment override (column-level only — see "Admin authoring").
- Per-cell border / shading (cell appearance config) — visual customisation is at the table level only.
- Collection-bound tables (pulling rows from a Collection). Future-session candidate if a use case surfaces; v1 is operator-authored only.
- CSV import for table content. Not a widget concern; the Page Builder authoring tools are enough at this size.
- Cross-table styling presets / shared themes. The widget's appearance is driven by site theme variables; per-instance styling beyond border-style + alignment is out of scope.
