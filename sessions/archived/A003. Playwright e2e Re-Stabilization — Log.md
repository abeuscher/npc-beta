### Phase 2 — Classify the residuals

One unexpected; the rest of the surface was green-with-retry. The single failing spec was `tests/e2e/importer/site-import-export.spec.ts` ("Site import / export rollup UI (session 310)"). Walking the failure:

- `data-testid="site-export-section"` exists in `resources/views/filament/pages/site-import-export.blade.php`. The element should be in the DOM. The page wasn't loading at all.
- Compared the spec's `page.goto('/admin/site-import-export')` against the Filament page-class naming pattern. Every other Filament `*Page` class in the codebase routes at `/admin/{kebab-case-class-name-including-Page-suffix}` — `ImporterPage` → `/admin/importer-page`; `GeneralSettingsPage` → `/admin/general-settings-page`; `ImportContactsPage` → `/admin/import-contacts-page`. No slug overrides anywhere.
- Verdict (decision-tree (a)): the s310 spec used the wrong URL. The actual route for `SiteImportExportPage` is `/admin/site-import-export-page`. The spec couldn't have passed on any environment — the author committed without running it.

### Phase 3 — Apply fixes

**Iteration #1 — URL fix** (`0936647`). Two `page.goto` calls in `tests/e2e/importer/site-import-export.spec.ts` updated to the correct route. Pushed; CI verification showed the first assertions (section visibility) now passed but the test failed further at the modal-open step: the Export Site button's `requiresConfirmation()` modal was present in the DOM but Alpine's `isOpen` stayed `false` across 16 retries.

**Iteration #2 — Modal-selector disambiguation** (`fd468df`). Two hypotheses tested in one push: (a) `getByRole('button', { name: 'Export Site', exact: true })` was ambiguous because `modalSubmitActionLabel('Export Site')` on the same action labels the submit button identically; (b) the spec's `.first()` on `[role="dialog"]` was picking the hidden pre-rendered placeholder. Switched the click to `[data-testid="site-export-action"]` (existing on the action via `extraAttributes`), tightened the modal selector to `[role="dialog"]:visible`, scoped Cancel inside the modal. Push; still failed at the visible-modal assertion.

**Iteration #3 — Diagnostic instrumentation** (`3155bbb`). Added a temporary in-spec probe: logged the header HTML, the button element, the dialog count + per-dialog visibility + `aria-labelledby`, plus captured browser console / pageerror / requestfailed / 4xx-5xx response events. The captured DOM resolved the mystery:

- The button is correctly located: `<button ... wire:click="mountAction('exportSite')" data-testid="site-export-action">` — single match, correct attribute, click is hitting the right element.
- After click, 4 dialogs exist; 0 are visible.
- The action's modal (dialog[1]) is **fully populated server-side** — innerText contains the modal's heading "Export Site", the description with live snapshot counts ("15 pages, 6 templates, theme (2 design keys), 44 media files"), and the Cancel button text.
- Alpine's `isOpen` state stays `false`. The server-side mount succeeded; the client-side toggle didn't fire.
- No browser console errors, no failed network requests.

**Decision point surfaced to user:** the failure is a Filament behaviour gap specific to **header-action `requiresConfirmation()` modals on a custom `Filament\Pages\Page`** — the same `requiresConfirmation()` pattern works for table actions (importer approve in `ImporterPage`), and `->form([...])` modals work for Resource ListPage header actions (`Import Bundle` on `/admin/pages`). Only this specific combination fails in CI. User picked decision-tree (b) — delete the failing test; document the coverage tradeoff.
