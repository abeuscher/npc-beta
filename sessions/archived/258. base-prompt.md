## Process rules for every session

- **Rule 1 (stub-then-rehearsal):** B2 does NOT do feature-implementation work. It exercises existing surfaces and writes the runbook.
- **Rule 2 (audit absorption):** B2 is **feature-style**, not audit-style. Small in-session fixes are NOT permitted. Findings get logged in the playbook + lifted to follow-on tickets in `release-plan.md`'s Out-of-Gate register or as new release-plan entries.
- **Execution mechanism — Playwright steering, observation-only.** Phases 2–5 drive the admin UI through Playwright (page.goto, page.fill, page.click, screenshots, DOM/console capture) rather than predicting UI behavior from code-reading. **Hard rule:** no spec files persist after the session — `tests/e2e/` does not gain entries from B2. F-track sessions exist for durable Playwright authoring; B2 does not author within them. See the session prompt's design decisions for full rationale.
- **Honest measurement framing.** Claude-driven Playwright is system-savvy, not operator-naïve. Manual-cleanup % and similar measurements should be reported in the playbook as system-savvy lower bounds, not as honest operator measurements. The honest operator measurement is the planned User-Testing Plans pass (separate stub in `session-outlines.md`).
- **Rule 3 (success criterion is the gate):** The four sub-scenarios are the gate. If new evidence emerges during rehearsal that the success criterion needs adjustment, re-sign-off with the user before closing.
- **Rule 4 (per-rehearsal artifact):** The Onboarding Playbook at `docs/runbooks/onboarding-playbook.md` is the artifact. "Green check" alone does not close the session.
- **Before writing any code**, read every file you intend to modify. Code writes this session should be minimal — primarily documentation and possibly a small Pest assertion or two if a sub-scenario warrants automation.
- **Pause and ask** any time a decision point arises that is not covered by the agreed prompt — especially if a finding is large enough that you're tempted to fix it in-session. Per Rule 2, the answer is "log it, don't fix it."
- **If the PostgreSQL container becomes unresponsive**, ask the user to restart it.
- **When implementation is complete**, run the fast test suite: `docker compose exec app php -d memory_limit=2G vendor/bin/pest --exclude-group=slow`. Baseline is 1921 passed (257 close). Slow group baseline: 7 stress tests passing in ~110s.
- **Front-end builds — when to run which:**
  - Public-collection rendering (sub-scenario 3) may touch SCSS or widget assets — if so, run the appropriate build per the base-template rules.
  - If no SCSS / Vue / widget asset changes — no `npm run build` or `build:public`.
- **Cross-repo coordination — Fleet Manager agent contract.** This session is **not** boundary-touching. The standing cross-repo discipline applies as a passive check.
- **Verify objective outcomes yourself; pull the user in for judgment.**
  - **Objective**: import row counts, export file row counts, public-collection page HTTP status + DOM presence, fast-suite pass/fail, manifest tally vs runner outcome (already automated by 257). Close these loops yourself.
  - **Judgment-required**: whether a "messy" import flow *feels* tractable for a real operator, whether the playbook's edge-case section reads as useful, whether the Migration-out timing meets the implicit "feels-fast-enough" bar at 5 minutes for 10K records. Pull the user in at Phase 10 manual verification for these.
