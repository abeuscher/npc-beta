# 207. Column Layout Inspector — Appearance Unification

Give column layouts a parallel `appearance_config` column and a three-tab inspector built from the widget Background and Margin & Padding panels, so layouts gain gradients, background images, and the unified spacing grid — without turning layouts into widgets.

Carved out of session 206's **Open Flag A** (LayoutInspectorPanel architectural divergence) once the scope proved too large to absorb into the 206 apply session. Session 208 (migration squash) was displaced one slot to make room.

---

## Architectural invariant — load-bearing

**A column layout is not a widget.** This has been true since the two constructs were split, and it stays true after this session.

- `page_layouts` and `page_widgets` are separate tables with separate contracts, separate model classes, separate API endpoints, and separate rendering paths.
- The goal of this session is **narrow**: share two of the three universal appearance panels (Background and Margin & Padding) across the two inspectors, by making the panels data-driven. Nothing else is unified.
- Specifically *not* in scope: promoting layouts into widgets, demoting widgets into layouts, a common base table, polymorphic relationships, shared schemas, a shared inspector root component, presets for layouts, text controls for layouts.

If implementation pressure during the session starts to suggest "we should just make layouts widgets," stop and pause — that is not this session.

---

## Design decisions (resolved before starting)

- **Tabs on the new layout inspector, in order:** Column Settings | Margin & Padding | Background. Three tabs, same chrome (`InspectorHeader` + `InspectorTabs` + inspector pane) as the widget inspector. **No Text tab. No Presets tab.**
- **Full-width toggle placement.** Full-width is the most important layout-behavior control. It lives on the **Column Settings** tab, at the top, above the display toggle. It is explicitly **not** inside Margin & Padding. This is a user directive and non-negotiable for this session.
- **Storage shape.** New column `page_layouts.appearance_config jsonb NOT NULL DEFAULT '{}'`. Appearance-key whitelist for layouts is `['background', 'layout']` — **no `text` key**. `layout_config` retains only layout-behavior keys.
- **Key split between `layout_config` and `appearance_config` on layouts:**
  - `layout_config` keeps: `full_width`, `display`, `columns`, `gap`, `grid_template_columns`, `grid_auto_rows`, `align_items`, `justify_items`, `justify_content`, `flex_wrap`, `flex_basis[]`.
  - `appearance_config.background.*` gets: what was previously `layout_config.background_color`, plus anything the widget BackgroundPanel writes (gradient, background image).
  - `appearance_config.layout.*` gets: what was previously `layout_config.padding_{top,right,bottom,left}` and `margin_{top,right,bottom,left}`, matching the widget SectionLayoutPanel's write shape.
- **Data migration.** In-place data migration moves existing layout rows' `background_color`, `padding_*`, `margin_*` keys from `layout_config` into `appearance_config` under the shape map above, then removes those keys from `layout_config`. `full_width` stays on `layout_config`. Rows with no prior appearance values get `appearance_config = '{}'`.
- **No presets for layouts.** Discussed during the 206 walkthrough and declined: layouts are internal containers with no standalone visual (an empty column layout renders as nothing until widgets are dropped in), so preview thumbnails don't work the way they do for widgets. Revisit only if demand emerges.
- **Concrete values rule applies.** Layout `appearance_config` is a first-class JSONB column with a concrete empty-object default, not a null-proxy. Every key the panels read has a declared default in the panel code — no "missing key means default" idioms.
- **Iteration strategy.** Phases sequence within a single branch `session-207/1`. The phases are tightly coupled (the migration must land before the renderer and inspector can read the new column; the panel genericization must land before the layout inspector can mount them). If a clean split emerges during implementation, the user may bless a `/2` carve-out.

## Phase 1 — Schema & data migration

- Write a migration adding `page_layouts.appearance_config jsonb NOT NULL DEFAULT '{}'`.
- In the same migration (or an immediately-adjacent one), move existing key values:
  - For every `page_layouts` row: read `layout_config`, extract `background_color` → `appearance_config.background.color`; extract `padding_{top,right,bottom,left}` and `margin_{top,right,bottom,left}` → `appearance_config.layout.{padding,margin}_{…}`; write back both columns with the moved keys removed from `layout_config`.
  - Rows with no relevant keys get untouched `layout_config` and default-empty `appearance_config`.
- Update `app/Models/PageLayout.php`: add `appearance_config` to `$fillable` and `$casts => ['appearance_config' => 'array', ...existing casts]`.
- Update `docs/schema/page_layouts.md`: document the new column and the key split.
- Run `docker compose exec app php artisan migrate` on the dev DB (which has layout rows with populated `layout_config`) to verify the data migration does what it should. Then run `migrate:fresh --seed` to verify the clean-bootstrap path.

---

## Phase 2 — Panel genericization

Refactor `BackgroundPanel.vue` and `SectionLayoutPanel.vue` so they operate on a generic `appearance_config`-shaped blob plus an update callback, instead of reading `widget.appearance_config` directly.

- New prop shape (both panels): `:config="Record<string, any>"` (reads the relevant subtree — `config.background` for BackgroundPanel, `config.layout` for SectionLayoutPanel) and `@update="(path: string, value: any) => void"` where `path` is a dot-path relative to the panel's subtree root (e.g. `color`, `image.url`, `padding_top`).
- Widget inspector (`InspectorPanel.vue`) is updated to pass `widget.appearance_config` and wire `@update` to the existing store method `updateLocalAppearanceConfig(widgetId, fullPath, value)`. The widget user-facing behavior is identical.
- `TextPanel.vue` is untouched — only widgets use it, and the widget inspector keeps calling it as it does today.

This is structurally a pure refactor on the widget side — no behaviour change. Rely on the existing Pest + Playwright coverage for regression.

---

## Phase 3 — Layout inspector rewrite

Rewrite `LayoutInspectorPanel.vue` to match the widget inspector chrome, with three tabs.

- Header: `InspectorHeader`-style markup — type badge "Column Layout", label input (preserves the current debounced save via `updateLocalLayout`), delete button (preserves the current confirmation dialog).
- Tab strip: `InspectorTabs` with three entries — `column-settings` | `margin-padding` | `background`. Default tab is `column-settings`.
- **Column Settings tab:**
  - **Full-width checkbox at the top.** Writes `layout_config.full_width`.
  - Display toggle (flex/grid). Writes `layout.display`.
  - Columns count. Writes `layout.columns` plus resizes `layout_config.flex_basis[]` / `layout_config.grid_template_columns` as today.
  - Grid block (conditional on `display === 'grid'`): `grid_template_columns` preset/manual, `gap`, `grid_auto_rows`, `align_items`, `justify_items` — all writing to `layout_config.*`.
  - Flex block (conditional on `display === 'flex'`): `justify_content`, `align_items`, `gap`, `flex_wrap`, per-column `flex_basis[]` — all writing to `layout_config.*`.
- **Margin & Padding tab:** mount `SectionLayoutPanel` with `:config="layout.appearance_config"` and wire `@update` to a new store method.
- **Background tab:** mount `BackgroundPanel` with `:config="layout.appearance_config"` and wire `@update` to the same new store method.
- Store addition: `updateLocalLayoutAppearance(layoutId: string, path: string, value: any)` mirroring `updateLocalAppearanceConfig` but targeting the layout's `appearance_config` and triggering the layout save-debounce.
- `InspectorPanel.vue` keeps its `v-if="layout"` early-return mounting `LayoutInspectorPanel`, unchanged.
- Delete the bespoke Container / Background Color / Padding / Margin markup from the old panel — those responsibilities now live in the reused panels.

## Phase 4 — PHP rendering path

- `AppearanceStyleComposer::composeForLayout(PageLayout $layout)` (new method) composes the CSS equivalent of the widget appearance path, reading `appearance_config.background.*` and `appearance_config.layout.{padding,margin}_*`. If a corresponding widget-side method already exists and is reusable, delegate to it; otherwise mirror the pattern.
- `PageBlockRenderer` (and/or `PageController::renderLayoutBlock`) switches layout CSS composition from `layout_config.background_color` / `padding_*` / `margin_*` to `composeForLayout`. The `full_width` check on `layout_config` stays put — `full_width` did not move.
- Any serialization path that exposes `PageLayout` to the Vue app (e.g. `PageBuilderApiController::getBootstrapData`, any layout JsonResource) adds `appearance_config` to the payload.

Manually verify a seeded page with a styled column layout renders identical CSS before and after the phase — the migration round-trip test (Phase 6) covers this formally.

---

## Phase 5 — API schema-key filter

Mirror the session-179 Flag C pattern for widgets, scoped to layouts:

- The layout update endpoint (`PageBuilderApiController::updateLayout` or whichever action handles `PATCH /admin/api/page-builder/layouts/{id}`) validates `appearance_config` and filters unknown top-level keys. Whitelist: `['background', 'layout']`. **Reject or silently strip `text`** and anything else not in the whitelist.
- Preserve the existing `update_page` permission gate.
- Preserve any existing filtering on `layout_config` — if that column has no filter today, leave that decision alone (out of scope for this session).

## Phase 6 — Tests

- **Pest — migration round-trip.** Seed a page with a column layout that has `layout_config.background_color`, `padding_*`, and `margin_*` set; run the migration; assert the composed CSS (via `AppearanceStyleComposer::composeForLayout`) is byte-identical to the pre-migration output produced by the old code path. This is the load-bearing test.
- **Pest — API schema filter.** POST to the layout update endpoint with `appearance_config.text.color = '#fff'` and confirm the `text` key is stripped / rejected. Mirror the widget-side test shape from session 179.
- **Pest — existing layout CRUD.** Confirm existing tests continue to pass after `$fillable` / `$casts` changes on `PageLayout`.
- **Playwright — layout inspector drive-through.** Open a page with a column layout, click each of the three tabs, confirm the right panel mounts, change a Background key and a Padding key, save, reload, confirm persistence. This can be scoped small — one spec file under `tests/e2e/page-builder/` is sufficient.
- Run `docker compose exec app php artisan test --exclude-group=slow`. Run `npm run build`. Run `npm run test:e2e`. Fix failures before pausing for manual test.
