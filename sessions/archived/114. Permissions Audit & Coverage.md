# 114. Permissions Audit & Coverage

Audit and strengthen permission enforcement across the entire admin panel. This session does not introduce any new roles or demo-specific behaviour — it fixes gaps in the existing permission system so that every role (current and future) is correctly constrained by the permissions it holds.

---

## Design decisions (resolved before starting)

- No new roles are created. The existing role vocabulary (super_admin, cms_editor, crm_editor, event_manager, volunteer_coordinator, treasurer, blogger) is unchanged.
- No new permission names are created unless a model is missing a policy entirely.
- The fix for every gap is the same: make the existing permission system work correctly. If a button shows for a user who lacks the permission, fix the button. If a write method can be called without the right permission, add the check.
- Custom actions (promote to member, portal access, merge duplicates, cancel event, etc.) are grouped under the nearest existing permission — typically `update_*` or `create_*` on the parent or child resource. No new fine-grained permissions for individual controls.

---

## Phase 1 — Missing policies

Create policies for models that lack them. Register in AuthServiceProvider. Each policy follows the same pattern as ContactPolicy: viewAny, view, create, update, delete, deleteAny, plus restore/forceDelete where the model uses SoftDeletes.

Models to audit: Event, Form, Product, MailingList, FormSubmission.

---

## Phase 2 — Custom action permission checks

Audit every custom action across all Filament resources and pages. For each action that writes data, add:
1. A `->hidden()` or `->visible()` check tied to the relevant permission (UI layer)
2. An `abort_unless()` inside the action callback (server-side layer)

Work through every resource listed in `docs/app-reference.md` systematically.

---

## Phase 3 — Livewire method hardening

Audit all public Livewire methods that write data in PageBuilder, PageBuilderBlock, and PageBuilderInspector. Each write method must check `update_page` permission server-side. A user with browser devtools can call any public Livewire method directly — the method itself must refuse the operation.

---

## Phase 4 — canDelete permission coverage

Several resources override `canDelete()` with business logic only (e.g. "block delete if registrations exist") without also checking the delete permission. Fix these to check the permission alongside the business logic.

---

## Testing

- **Slow test groups to run this session:** none
- **New tests expected:** no — existing permission tests and deletion guard tests should continue to pass. Run the full fast suite to verify no regressions.

## Out of scope

- Read-only edit views
- UI changes to form layouts or navigation
- Settings page changes
