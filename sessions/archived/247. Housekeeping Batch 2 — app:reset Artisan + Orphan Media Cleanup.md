# 247. Housekeeping Batch 2 — app:reset Artisan + Orphan Media Cleanup

Lands the operator-facing `app:reset` artisan command and the scheduled orphan-media sweep. Forcing function: the 14.6 GB backup incident at session 245 UAT — the bulk of the data was orphan media files left behind by repeated `migrate:fresh --seed` cycles, because raw-SQL truncation bypasses Spatie Media Library's delete observer. This session closes that loop at the operator-tooling layer plus pairs it with Spatie's built-in `media-library:clean` on the scheduler.

The build-trigger audit (Batch 2 item 2) defers to a later sub-session.

---

## Background

Two distinct media-leak failure modes surface in this codebase:

1. **Wholesale wipe leak.** `php artisan migrate:fresh --seed` truncates tables via raw SQL DDL. Eloquent model events do not fire on raw SQL. Spatie Media Library's "delete file when model deleted" observer is registered on each `HasMedia` model; raw truncation bypasses it. Result: every `migrate:fresh` cycle leaves the previous run's media files behind in `storage/app/public/`. Session 245's image-attaching generator multiplied this by ~6 conversion variants per attached image. Effectively unbounded growth across dev cycles.
2. **Slow-drip orphan-conversion leak.** Normal admin usage (operator deletes a record via Filament) DOES fire the observer, which deletes the canonical original. But Spatie's conversion files (responsive WebP variants, thumbnails) are tracked separately and can drift out of sync. Plus: orphan `media` rows can persist if a parent model row is deleted via raw query. Spatie ships `php artisan media-library:clean` to handle exactly this.

This session addresses both:
- `app:reset` solves failure mode 1 (a wholesale wipe of `storage/app/public/` paired with `migrate:fresh --seed`).
- Scheduling `media-library:clean` solves failure mode 2 (a slow-drip cleanup of orphans that accumulate during normal usage).

## Phase 1 — `app:reset` artisan command

**New file:** `app/Console/Commands/AppResetCommand.php`.

- **Signature:** `app:reset {--force : Skip the confirmation prompt}`
- **Description:** Wipes the database and the on-disk media tree, then runs `migrate:fresh --seed` to restore a clean dev state.
- **Production-refusal gate.** First action in `handle()`: if `app()->environment('production')`, print a clear error message ("`app:reset` cannot run in production. This is a dev-only command.") and exit non-zero. The check uses `app()->environment()`, not raw `env('APP_ENV')` — env reads can be cached differently. No `--force-on-production` escape hatch — refusal is absolute.
- **Confirmation prompt.** When not in production AND `--force` is NOT passed, prompt the operator: "This will hard-delete every row, every uploaded file, and every cached conversion. The current install will be reset to a freshly-seeded state. Type `RESET` to continue." Only literal `RESET` confirms; anything else aborts non-zero. The all-caps confirm-string makes accidental enter-key acceptance unlikely.
- **Order of operations:**
  1. `Artisan::call('migrate:fresh', ['--seed' => true, '--force' => true])` — runs first because the database is the canonical record of what media files SHOULD exist; once it's wiped, the file tree is unambiguously orphan.
  2. Wipe the storage tree:
     - `rm -rf storage/app/public/*` (Spatie Media Library's primary disk)
     - `rm -rf storage/media-library/temp/*` (Spatie's temp upload area; smaller but still leaks)
  3. Print a summary line: "Reset complete. N migrations applied, M seeders ran, storage cleared."
- **Important: command must NOT be auto-discoverable in production.** Even though the production-refusal gate stops execution, the command itself appearing in `php artisan list` on a prod box is undesirable surface area. Filter out the command at the Console Kernel level when `app()->environment('production')` — pattern reference: any existing dev-only artisan command in the codebase. If none exists, the simplest move is `if (! $this->laravel->environment('production')) { Artisan::starting(fn ($artisan) => $artisan->resolve(AppResetCommand::class)); }` in the appropriate provider; confirm at session start which provider is the right home.

**Tests** (`tests/Feature/AppResetCommandTest.php` — new):

- Production-refusal gate fires when `APP_ENV=production` (mock the environment).
- Confirmation prompt appears in interactive mode when `--force` is not passed.
- `--force` skips the prompt.
- Happy path on dev: command runs `migrate:fresh --seed` and clears the storage tree. Use `Storage::fake('public')` to avoid touching real disk.

## Phase 2 — Scheduled `media-library:clean`

**Updates to `bootstrap/app.php` `withSchedule()` block.**

Spatie's `media-library:clean` command exists out of the box (provided by `spatie/laravel-medialibrary`). It walks the `media` table + on-disk conversions and removes:

- Orphan `media` rows (where the polymorphic owner row is gone).
- Orphan conversion files (where the parent `media` row is gone but the conversion file lingers).

It does NOT walk the entire `storage/app/public/` tree looking for files with no `media` row — that's the gap `app:reset` addresses for the wholesale-wipe case.

**Schedule entry:** add to `withSchedule()` block (which already contains `backup:run` + `backup:clean` from session 242):

```php
$schedule->command('media-library:clean')
    ->daily()
    ->onOneServer()
    ->withoutOverlapping();
```

Cadence rationale: daily is enough for the slow-drip case; orphans don't grow fast enough to need hourly. `onOneServer` is defensive in case multi-instance ever lands. `withoutOverlapping` prevents pile-up if a previous run is slow.

**Tests** (extend `tests/Feature/ScheduleConfigurationTest.php` if it exists; otherwise create):

- `php artisan schedule:list` includes `media-library:clean`.
- The schedule entry has the daily cadence.

**Production deployment caveat (LOG IN CLOSE-GATE).** Per `docs/app-reference.md`'s "Scheduler runner — known gap" section, the `worker` container runs `queue:work` only — there's no `schedule:run` runner in production. Adding `media-library:clean` to the schedule list does NOT fire it automatically until a scheduler-runner is added (separate worker variant, sidecar container, or host-level cron). The schedule entry is correct and ready; the runner work is a separate Beta-1-blocking carry-forward listed in `sessions/tracks/fleet-manager-agent.md`. Document this in the 247 log so the dependency is visible.
