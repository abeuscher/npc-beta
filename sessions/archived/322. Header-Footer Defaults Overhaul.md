# 322. Header / Footer Defaults Overhaul

Make a fresh install's site chrome look production-ready out of the box: centered (not full-width) header chrome, a vertically-stacked footer nav option that the default footer ships with, and a bounded review of chrome defaults.

---

## Plan reference

This session executes `sessions/release-plan.md` § E16 (Header / footer defaults overhaul, execution-order position 47). Read that entry first; it is canonical for scope, success criterion, and artifact.

**Success criterion (from the plan):** Header default changes — no longer full-width; reasonable centered chrome. Footer gains a stacking nav option (vertically stacked links instead of horizontal) and the default footer template includes the new stacking nav out of the box. Site chrome defaults reviewed for sensible production-ready appearance on a fresh install.

---

## Session-specific deltas

- **Chrome is built from `_header` / `_footer` page-builder pages + template chrome config**, not a single Blade file. Templates point at them via `header_page_id` / `footer_page_id` (`TemplateSeeder`); the pages themselves are seeded (`BasePageSeeder`). "The default footer includes the stacking nav out of the box" therefore means the seeded `_footer` page ships a Nav widget configured to the stacked layout. `ChromeRenderer` renders these; `resources/views/components/site-header.blade.php` / `site-footer.blade.php` are the fallback partials.
- **The Nav widget already has a `layout` config** (`NavDefinition`). The footer stacking nav is most likely a new value/option on that config rather than a new widget — confirm at the open gate.
- **Session 284 already redesigned the footer** for the marketing site (centered nav + copyright, `#d4d4f2` band) and fixed the `public.blade.php` chrome-wrapper bug. This session makes the sensible shape the **seeded default**; it does not undo 284. Read the 284 log first.
- **Session 294 left the duplicate header/footer `renderLayoutBlock` non-collapsing on purpose, tied to this entry** — if the mobile behavior of chrome comes up, that's the thread; keep any change bounded to the E16 success criterion.
- **Pairs with A006 (demo-site design, in cloud).** The chrome defaults set here are what the demo install and every fresh install render. Don't design defaults that fight the demo-site direction.

## Open questions to resolve at session start

- **Footer stacking nav — Nav `layout` option vs separate variant.** Lean: extend the Nav widget's existing `layout` config with a stacked/vertical option (keeps one widget, one config surface). Confirm it round-trips through the chrome editor and export/import.
- **Header "centered" width.** Which max-width / container token to standardize on — reuse the marketing-site content width rather than inventing a new one. Settle the value once.
- **How the seeded default ships the stacked footer nav.** Edit the seeded `_footer` page-builder page, the fallback partial, or both — so a genuinely fresh `migrate:fresh --seed` install renders the stacked footer without operator action.
- **Standing visual guard.** Whether to land a small Playwright spec asserting centered-header (not full-width) + stacked-footer render, as a durable regression guard. Lean: yes, one small spec — chrome is exactly the kind of visual default that silently regresses.
