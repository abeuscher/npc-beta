### Iteration 1 — Hero widget: button-group alignment + text-block max-width

Two Hero widget controls, both touching the same widget definition so bundled into one iteration:

- `button_alignment` (select; default `auto`) — explicit left/center/right override for the call-to-action button group. `auto` preserves the prior behavior of deriving alignment from the text position.
- `text_max_width` (select; default `42rem`) — exposes the hero text block's width (Narrow / Medium / Wide / Full). Bound via a new `--hero-text-max-width` CSS variable on the wrapper; `.hero-inner` consumes it with the 42rem fallback preserving prior renders.

Schema-keys lock test updated to the new key order; four new tests cover the defaults, the CSS-variable emission, and the auto/explicit alignment paths.

### Iteration 2 — Logo widget: fresh-install placeholder image

Added `public/images/default-logo.svg` (a dashed-border "Your Logo" placeholder) and rendered it in the Logo widget when neither a logo image nor wordmark text is configured — the fresh-install posture the system-page seeder leaves on the header. Operators who set text-only or upload their own logo are unaffected. Two tests cover the placeholder render and its suppression when text is present.

The companion inbox item ("default link = site home") was caught as drift: the Logo definition already declares the link default as `/` in both schema and defaults. No change needed.

### Iteration 3 — Logo widget: respond to a full-bleed hero's text colour

When a hero with full-bleed (overlap-nav) leads the page, the chrome sits on top of it and the operator's hero text colour is the one they want for the logo wordmark too. Plumbed the hero's text colour through the page and post controllers as a new shared view value, emitted as a `--overlay-text-color` CSS variable on the overlay nav wrapper, and had the logo wordmark consume it inside the overlay scope (mirrors the existing nav-link-colour reach-over). The placeholder SVG keeps its static colours and user-uploaded raster logos are untouched — only the wordmark recolours. Two integration tests cover the variable's presence on a full-bleed hero page and its absence otherwise.

### Process tooling — mid-session bug-capture flow *(added at user direction; retcon note)*

Not an inbox item — added mid-session to close a process gap the dropped items exposed (bugs were tracked in an out-of-band doc and went stale before being scheduled). Folded into this session under the housekeeping umbrella per user direction rather than split to its own session.

- `npm run logbug -- "…"` (`scripts/logbug.mjs`) appends a VERSION+date-stamped line to a new `sessions/housekeeping-incoming.md` buffer. Pure node — fires anytime, including while an agent is mid-loop, and never touches the canonical inbox mid-session.
- The session close gate (`template-base-prompt.md`) now digests the buffer: Phase 1 surfaces a validity-checked review to the user before close; Phase 2 folds survivors into `housekeeping-inbox.md` (stamp intact) and clears the buffer.
- `housekeeping-inbox.md` documents the flow and adds the **verify-the-premise-before-scheduling** staleness guard — re-check each candidate against current code at the session-start walk before committing it to an iteration. This is the actual fix for the staleness that produced the three dead items below; decay happens between capture and execution, not at capture.

The buffer's digest was exercised this session (a user-planted test entry was caught and removed at close), confirming the path end to end.

## Security considerations

The logbug script writes only to a fixed in-repo path and does not execute or interpolate its input into anything but a markdown line.
