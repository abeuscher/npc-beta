## What was built

### Phase 1 — Table widget, embedded ProseMirror (iteration 1)

The block-level **Table** page-builder widget — the launch stopgap for tabular content and the team's first ProseMirror integration (a down-payment on the deferred Quill→ProseMirror track). A real WYSIWYG table authored in a self-contained `prosemirror-tables` editor, stored as clean semantic HTML in `page_widgets.config.table_html`.

- **ProseMirror module** (`resources/js/page-builder-vue/prosemirror-table/`): `schema.ts` (a deliberately minimal schema — the `prosemirror-tables` node set + paragraph/text + `strong`/`em`/`link` marks; cells constrained to `paragraph+`), `editor.ts` (the editor factory — HTML→doc on open, doc→HTML on change; `tableEditing` + history + keymaps), `commands.ts` (the toolbar commands: a hand-built `createTable` since `prosemirror-tables` ships no insert, plus the re-exported structural commands + bold/italic/link).
- **`TableEditorField.vue`** — a new `table` inspector field type opening an **expandable modal** editing surface (a table needs more room than the inspector rail): the embedded editor + a toolbar (Google-Docs-style grid-picker insert, add/delete row & column, merge/split, header-row toggle, bold/italic/link) wired to the commands.
- **`InspectorField.vue`** registration of two new field types: `table` → `TableEditorField`, and `border` → the shared `BorderInput` primitive.
- **Widget shell:** `app/Widgets/Table/TableDefinition.php` + `template.blade.php` + `styles.scss`, registered in `WidgetServiceProvider` (the registry sync — not the seeder — is the real registration step). `inlineEditable()` is false; the canvas shows a read-only render; the public render carries **no editor JS**.
- **The one back-end change — `HtmlSanitizer` extension:** `td`/`th` now carry `colspan`/`rowspan` (validated as bounded positive integers via `isBoundedSpan`, ≤ `SPAN_MAX`) + `class`; `table`/`tr` carry `class`. Style/script/event-handlers still stripped. Wired into the two save/import seams that gate rich-text — `PageWidget::sanitizeAgainstSchema` and `WidgetTreeHydrator` now treat `'table'` like `'richtext'`. The template re-sanitises on render (defence in depth).
- **Help doc** `resources/docs/widget-table.md` (+ `help:sync`); **standing Playwright spec** `tests/e2e/page-builder/table-widget.spec.ts` (the full authoring round-trip → public render).

### Phase 2 — Table appearance overhaul (iteration 2, owner feedback)

After the first iteration the owner asked for richer, fully operator-controlled styling. Shipped:

- **Interior gridlines on the shared border tool.** `BorderInput.vue` gained an opt-in `allowInterior` mode — two interior line toggles (horizontal/vertical) alongside the four outer edges, sharing the border's width + colour; radius hidden in that mode. Opt-in via a prop, so every other widget's border control is unchanged. The table maps outer edges to its box (`composeBorderProps`) and the interior toggles to cell gridlines (`np-table--inner-h` / `--inner-v`).
- **Six user-controlled colours:** header background + header text, cell background + cell text, and (gated on the zebra toggle) stripe background + stripe text. Replaced the single `bg_color`. All applied at render via `--np-table-*` custom properties; the widget SCSS carries **no literal colours** (passes the colour-token guard).
- **WYSIWYG editor modal:** the editor surface now renders the table with the configured colours, alignment, and gridlines (the same `--np-table-*` props + modifier classes, mirrored in scoped CSS), so what's built matches what publishes; a faint dashed guide keeps borderless cells editable.
- **Per-column percentage widths:** a percentage box above each column in the editor; widths stored in `config.column_widths` and rendered via a `<colgroup>` (merge-safe). In the editor a custom **table NodeView** renders the same colgroup live; on the public side the template injects the colgroup into the sanitised table HTML after sanitisation (trusted, template-generated).

### Event-orphan landing-page fix (unrelated to the widget — kept a separate commit)

Surfaced mid-session: a local DB carrying **252 orphaned `type=event` landing pages** (admin showed 19; the export pushed ~270) with the `events` table empty. Root cause: scrub events were mass-deleted by `RandomDataGenerator::wipe` (`Event::where('source',scrub)->delete()` bypasses the observer), and their landing pages were `source='human'` so `wipeScrubPages` (matches `source=scrub`) never cleaned them.

- **`EventResource::createLandingPageForEvent`** now sets the landing page's `source` from the event (`?? Source::HUMAN` for a source-less event, since `pages.source` is NOT NULL) — so scrub events make scrub pages the wipe catches.
- **`EventObserver::deleted`** force-deletes the landing page on a per-event delete (`PageObserver::deleting` cascades its widgets/layouts), closing the UI/single-delete path.
- **`pages:prune-orphan-events`** artisan command (dry-run default, `--force`) for the existing orphans; ran `--force` on the dev DB — **252 cleared**, pages back to 21.
- 3 lifecycle Pest tests (source inheritance, delete cascade, prune).

## Security considerations

The Table widget's security surface is the `HtmlSanitizer` table-attribute extension, which handles untrusted (pasted) HTML: `colspan`/`rowspan` are strictly validated as bounded positive integers (non-integer, zero, and over-`SPAN_MAX` values are stripped), and style / script / event-handlers continue to be stripped; classes pass through the existing arbitrary-class allowance (non-executable). `table_html` is sanitised at the save boundary (`PageWidget`), the import seam (`WidgetTreeHydrator`), and again on render. The public render carries no editor JS. The per-column `<colgroup>` is template-generated (trusted) and injected *after* sanitisation, so it needs no sanitiser allowance. The event-orphan prune force-deletes (cascading widgets via the existing `PageObserver` path); it is operator-run, dry-run by default. JSON-LD / blockquote untrusted-input handling is deferred to session 351.
