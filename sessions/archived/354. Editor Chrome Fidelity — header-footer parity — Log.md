## What was built

### Phase 1 — Editor chrome fidelity (the reported bug)

**Cause (confirmed against code).** The page-builder preview and the public site render the *same* server-produced chrome HTML (`ChromeRenderer`), but the editor wrapped it in `.preview-chrome > .preview-chrome__html` and **omitted the public layout's wrapper structure** — `div.site-nav-wrapper[.--overlay] > header.site-header` (header) and `footer.site-footer` (footer) from `resources/views/layouts/public.blade.php`. Every `.np-site`-scoped chrome style keyed to those selectors therefore failed to attach, so the editor chrome drifted from the front end. (The editor already loads the editor-variant of the public bundle via `AdminPanelProvider`'s `widgetEditorCss()`, so the rules were present — only the markup that activates them was missing.)

**Fix (`resources/js/page-builder-vue/components/PreviewCanvas.vue`).** Reproduced the public wrapper structure inside the chrome bands: header HTML now nests as `.site-nav-wrapper[.site-nav-wrapper--overlay] > header.site-header > .preview-chrome__html`; footer as `footer.site-footer > .preview-chrome__html`. The `.preview-chrome` hover outline + "Edit header/footer ↗" deep-link affordance is preserved as the **outer** layer (not deleted).

**Overlay-nav vars — done editor-side, not via `chromeBands()`.** The session prompt anticipated threading the `--overlay-*` nav-colour vars through `app/Livewire/PageBuilder.php::chromeBands()`. On inspection the source data already rides the first page-widget in the Vue store, so this was implemented **reactively in the editor** instead: two computeds (`navOverlay` / `navOverlayStyle`) mirror `PageController::show`'s overlap-hero detection off `store.pageItems[0]` and apply the conditional `.site-nav-wrapper--overlay` class + the three `--overlay-*` custom properties. No backend change — cleaner, and the overlay state now tracks the `overlap_nav` toggle live.

**`styles` blob — not a cause; left as-is.** Verified by inspection: the editor already emits the chrome `styles` blob (`resources/views/livewire/page-builder.blade.php:86–91`), and the public layout emits only `html` (widget CSS rides the prebuilt bundle on both surfaces). Not load-bearing for this fix.

**nav-family parity.** Added the public layout's nav font-family rule to the editor blade — an `@layer host` rule scoped to `.widget-preview-scope nav` (`page-builder.blade.php`), mirroring `public.blade.php`'s `@layer host { .np-site nav { … } }`, so the canvas nav font matches the front end instead of falling back to the body family.

**Link-colour leak — root-caused and fixed (the session's most consequential change).** The editor chrome's nav/footer links rendered grey (`#1F2937`) where the public site renders them blue (`#1D4ED8`). Traced to ground via throwaway Playwright diagnostics: the link colour is set by the widget rule `.widget-nav__column-link { color }` in `@layer widgets`; on the public site that rule wins, but in the editor **Filament admin's *unlayered* Tailwind preflight `a { color: inherit }` beat the layered widget rule** inside the `.np-site` preview (unlayered always beats layered, regardless of specificity), so the link collapsed to the inherited admin grey. Fixed with a one-line change in `resources/css/filament/admin/theme.css` — `@import '…filament…/theme.css' layer(filament);` — wrapping Filament's whole admin CSS (preflight included) in `@layer filament`, declared below `@layer widgets`, so layered widget rules win again. This is exactly the stub's **Track-2 "manually layer the preflight"** hack (owner-approved as the interim; the clean Tailwind v3→v4 path stays as carried debt). It fixes the *entire* unlayered-preflight-beats-widgets class for the preview (borders, colours — the stub's pager-arrow example included), not just chrome link colour. Owner verified no admin-UI regressions.

**Verification.** Standing guard `tests/e2e/page-builder/chrome-parity.spec.ts` (3 tests): editor reproduces the `.site-nav-wrapper`/`.site-header`/`.site-footer` structure + the band background attaches + no uncaught page errors; the public render exposes the source-of-truth chrome; and **chrome nav link colour matches the public render** (editor link === public link, and ≠ the inherited grey — locks in the layer fix). All 3 green. Mid-session diagnostic specs were torn down at close; `chrome-parity.spec.ts` is kept as a standing regression guard. (e2e lesson recorded: the isolated stack serves an image-baked `public/build` named volume, so a host `npm run build` must be `docker cp`'d into `nonprofitcrm_e2e_app` to be exercised.)
