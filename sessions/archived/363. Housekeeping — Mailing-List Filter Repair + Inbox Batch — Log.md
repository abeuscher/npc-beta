## What was built

### Phase 1 — Anchor: mailing-list simple filters save `value = NULL` (session-363/1)

- **Root cause pinned deeper than the 362 diagnosis.** Filament hides ⇒ doesn't dehydrate; a hidden field's dehydration `Arr::forget`s its state path unless the root container finds another *dehydrated* component at the same path — but that guard **prunes its search at any non-dehydrated component**, and `Repeater::relationship()` marks the repeater itself `dehydrated(false)`. So the last-declared hidden `value` control erased what the visible one wrote. Consequence: the gate-leaned `->dehydrated(condition)` shape was mechanically impossible; the agreed fallback shipped instead.
- **Fix:** split state paths — `value_text` / `value_select` / `value_tag`, one per field type, mapped onto the single `value` column by the repeater's `mutateRelationshipDataBefore{Fill,Create,Save}Using` hooks. Changing a filter's Field now also resets the operator and all three value paths. No schema change; same UX.
- **Verification:** `MailingListFilterFormRoundTripTest` (4 cases) drives the real create/edit Livewire forms — form save → DB `value` → `MailingListQueryBuilder` match count → form refill — for select, text, and tag types, plus a change-type-on-edit case. Red before the fix (reproducing the NULL), green after. Confirmed nuance: **tag filters had always worked** (the tag picker is declared last, so nothing clobbered it); text and select filters were the inert ones.
