## Workstream 1 ‚Äî Stale reference audit

Grep the codebase for references to classes, methods, variables, or patterns that were renamed, moved, or deleted during sessions 135‚Äì140.

- **Deleted or renamed Livewire events** ‚Äî sessions 136‚Äì139 restructured how the builder communicates between components (preview refresh, widget selection, inline config sync). Grep for any event names that were used in earlier iterations but are no longer dispatched or listened for.
- **Removed blade sections** ‚Äî session 139 replaced the edit/preview toggle with edit/handles toggle and removed the preview iframe. Grep for any references to the old preview iframe, mode toggle, or retired blade variables.
- **Retired Alpine state** ‚Äî sessions 136‚Äì139 changed the Alpine state shape on the builder. Grep for any `x-data`, `x-on`, or `$wire` references that target properties or methods that no longer exist on the Livewire components.
- **General sweep** ‚Äî grep for any `use` statement that imports a class which does not exist on disk.

Fix confirmed stale references in place.

## Workstream 2 ‚Äî Widget copy & serialization consistency

Session 116 flagged two gaps (deferred to a builder session):

- `PageBuilder::copyBlock()` drops `style_config`, `column_index`, and nested children when duplicating a widget
- `PageWidget::serializeStack()` omits `is_active`

Verify whether these are still present. If so, fix them. Also audit any new copy/serialize paths added in sessions 135‚Äì140 for the same field coverage: `label`, `config`, `query_config`, `style_config`, `column_index`, `sort_order`, `is_active`.

## Workstream 3 ‚Äî Dead code and orphaned files

- **Dead imports** ‚Äî `use` statements that import classes not referenced in the file. Focus on Livewire components and services touched in sessions 135‚Äì140.
- **Unreachable methods** ‚Äî public methods on PageBuilder, PageBuilderBlock, or PageBuilderInspector that are no longer called from any blade or event listener.
- **Orphaned blade partials** ‚Äî any partial in `resources/views/livewire/partials/` that is no longer `@include`d from anywhere.
- **Orphaned JS/CSS** ‚Äî any asset file in `resources/js/` or `resources/css/` that was created during the builder overhaul but is no longer referenced.

Fix confirmed dead code in place. Flag uncertain cases for discussion.

---

## Workstream 4 ‚Äî Permission and security audit

- Verify that `PageBuilderInspector::assertCanEdit()` is called on every mutating method (config updates, image uploads, label changes, style changes).
- Verify that `validateBlockOwnership()` prevents cross-page widget injection on every write path.
- Verify the preview rendering path does not expose unpublished page content to unauthenticated users.
- Verify `wire:model` bindings on the inspector cannot be used to write arbitrary keys into `config`, `query_config`, or `style_config` beyond what `config_schema` defines.

Flag any gaps.

## Workstream 5 ‚Äî Duplicated logic

Scan the page builder files (PHP and Blade) for blocks of 10+ lines that appear in substantially similar form in multiple files. For each:

- Describe what the repeated code does
- List where it appears
- Propose an extraction target (model method, trait, service, blade partial, Alpine component)

Do not refactor. Present findings for discussion.

---

## Workstream 6 ‚Äî Framework alignment audit

Examine the page builder codebase for places where custom code does something the framework already provides. Same diagnostic question as session 116:

> "Is this code doing something the framework already does? If so, are we working with the framework or around it?"

Focus areas:

- **Livewire lifecycle** ‚Äî are there manual `$this->dispatch()` calls that could use Livewire's built-in reactive properties or computed properties instead?
- **Alpine.js** ‚Äî are there inline `x-data` blocks that reinvent Alpine plugins or utilities that already exist?
- **Filament integration** ‚Äî does the inspector duplicate any Filament form component behaviour that could be leveraged?
- **Blade components** ‚Äî are there repeated HTML patterns that should be anonymous Blade components?

For each finding, record: File, Line(s), What the code does, What the framework provides, Risk if left as-is, Suggested approach.

Do not fix. Present the table for discussion.
