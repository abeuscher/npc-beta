## Phase 1 — Blog Post Pager — page context not resolving correctly

**Symptom (from session 245 UAT).** The `blog_pager` widget renders on individual blog post detail pages and is supposed to expose previous/next post navigation by reading the current post's page context. During 245 manual testing (generating scrub blog posts and clicking through them), the pager renders but isn't receiving correct page context. Either the wrong neighbors are shown, the title/links are wrong, or the surface is empty when it shouldn't be — the precise failure mode surfaces during diagnosis.

**Likely area.** The widget's `dataContract()` returns a contract that the resolver walks against the active `SlotContext`. For posts (blog detail pages), the active record context is the current `Page` row of `type = 'post'`. The token resolution flows through `RecordContextProjector` reading the per-record-type `RecordContextTokens::TOKENS` map. The bug is somewhere in this chain — most plausibly:

- `BlogPagerDefinition::dataContract()` declares the wrong `model` or wrong field set.
- The `RecordContextProjector` doesn't have a projection registered for `blog_pager` against the post record type.
- The widget's template reads tokens that aren't being resolved by the projector.
- A neighbor-resolution query (previous post / next post by `published_at`) is filtering against the wrong scope.

**Phase deliverable:**

1. Reproduce the bug. Walk through a scrub blog post on `/news/{slug}` (with `news` being the configured `blog_prefix`) and capture the exact failure mode.
2. Identify the seam. Walk: widget definition → contract → resolver → projector → template. The session-241 mutation-testing slice (`tests/Unit/WidgetPrimitive/ContractResolverTest.php`) is a useful pattern reference for how the resolver behaves.
3. Fix at the smallest seam that resolves the symptom. Avoid expanding scope into adjacent record-aware widgets unless they share the bug.
4. Add a regression test. Pattern: walk a record-aware widget against a Page-of-type-post, assert the pager surfaces the right neighbor titles/links. Test file likely: `tests/Feature/BlogPagerTest.php` (new, or extend an existing pager test if one exists).

## Phase 2 — Text Widget Vertical Alignment

**Premise (from the stub).** The `text_block` widget currently renders with text aligned to the vertical middle of its container, which doesn't fit all column-layout use cases (e.g., a text block sitting next to an image where the operator wants the text top-aligned to the image's top edge).

**Design decision in this session.** Two shapes were flagged in the stub:

- **(a) Per-widget control.** Add a `vertical_align` field to TextBlock's schema (top / middle / bottom), expose in the inspector, render via `align-self` on the wrapper.
- **(b) Generalized control.** Lift to the universal Appearance layer — add `layout.vertical_align` to `AppearanceStyleComposer::defaultAppearanceConfig()`. Every widget inherits.

Recommendation: **(b) generalized**, in the long term. Vertical alignment is a layout concern that applies to all widgets, not just text blocks. But (b) is a bigger surface change — every widget's appearance composition path needs to honor it, the inspector needs the new field, and existing widgets need a default value backfill.

**For this session, ship (a)** — narrow, scoped, gets the felt friction off the user's path immediately. Capture (b) as a roadmap stub for later (or fold into the existing universal Appearance layer's evolution path).

**Phase deliverable:**

1. Add `vertical_align` to `TextBlockDefinition::schema()` as a `select` field with options `top` / `middle` / `bottom`, default `middle` (preserves current behavior).
2. Add a sensible default to `TextBlockDefinition::defaults()`.
3. Update `app/Widgets/TextBlock/template.blade.php` to read `$config['vertical_align']` and emit `align-self: flex-start | center | flex-end` (or equivalent CSS) on the wrapper.
4. Add a render test asserting the wrapper class/style reflects the chosen alignment value.
5. Log a roadmap stub in `session-outlines.md` for the generalized version: "Universal Appearance — Vertical Alignment" *(post-Beta-1 if not surfaced as forced; lift if multiple widgets need it)*.

**Out of scope:** generalized inheritance across all widgets; the universal Appearance layer's `defaultAppearanceConfig()` shape change.

## Phase 3 — Heroicon Picker in Quill Editor

**Premise (from the stub).** Add a Quill custom format that lets users insert a heroicon inline in rich text. Heroicons are the assumed icon library — confirm at session start by grepping the codebase for `heroicon-o-*` usage in widget definitions / Filament resources.

**Design decision in this session.** Quill v2 (the version Filament's RichEditor wraps) supports custom formats via its `register()` API. The picker can be implemented as:

- **Toolbar button** that opens a modal or popover with the icon set, click an icon → inserts at cursor.
- **Type-ahead trigger** (e.g., `:icon-name:`) that auto-replaces with the icon SVG.

Recommendation: **toolbar button + popover**. More discoverable, doesn't require operator memorization of icon names. Type-ahead is power-user-friendly but invisible to first-time users.

**Heroicon set choice.** Heroicons ships in three styles: outline (24px), solid (24px), mini (20px). Default to **outline** — matches Filament's default usage. Allow per-icon style selection in the popover if it doesn't blow up the UI; otherwise outline-only and stub the other styles for later.

**Phase deliverable:**

1. Confirm the heroicon set is available — check `package.json` for `@heroicons/vue` or similar. If not present, install.
2. Author a Quill custom blot/format under `resources/js/admin/` (or wherever existing Quill customizations live — confirm at session start). The blot wraps an inline `<svg>` element.
3. Author a toolbar button + popover in the admin Quill setup. Triggers blot insertion on icon click.
4. Style the popover via `public/css/admin.css` (hand-edited per the admin CSS rule).
5. Smoke-test: open any rich-text-bearing admin form (event description, page-builder text block), insert an icon, save, verify it round-trips through the database and renders on the public side.
6. Update help-doc stub at `resources/docs/quill-heroicon-picker.md` (small frontmatter-only stub — body deferred to E13 per the help-docs roadmap).

**Out of scope:** custom icon sets beyond heroicons; an icon picker that lets users upload their own SVGs (separate feature); inline icon styling beyond the default size.
