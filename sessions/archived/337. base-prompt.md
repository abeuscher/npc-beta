## Process rules for every session

Base-prompt process rules apply in full (`sessions/template-base-prompt.md`). Session-specific notes:

- **Presentation, not plumbing.** The portal forms, validation, controllers, and page-builder rendering already work. Style the widgets; do not rebuild flows. If a flow is visually broken, fix the markup/SCSS — don't touch the form logic.
- **One visual language.** All three views share a hero, spacing, typography, and button style. Resist per-view divergence — unlike event listings, the portal is deliberately uniform.
- **Front-end builds are required.** Portal widget templates/SCSS feed the public bundle. Run `build:public` for widget-asset changes and `npm run build` if `public.scss` (or a partial it imports) changes — Pest does not catch SCSS compile failures. Match the touched path to its build command per the base-prompt "Front-end builds" rule.
- **Dashboard data is illustrative, not member-scoped.** The v1 dashboard uses demo content (`BarChart` + `TextBlock`), not data wired to the logged-in `contact_id`. If you find yourself querying the member's own records, stop — that's the later lift, and it triggers the portal-security scoping rule.
- **Reserve, don't write, tour anchors.** Add the `portal.*` `data-tour` anchors to the styled widgets per the demo-site-plan convention; write no tour code (that's the next session).
