## What was built

### Phase 1 — Schema & data migration

- New migration `2026_04_21_120000_add_appearance_config_to_page_layouts.php` adds `page_layouts.appearance_config jsonb NOT NULL DEFAULT '{}'` and in the same `up()` moves pre-existing keys out of `layout_config`:
  - `layout_config.background_color` → `appearance_config.background.color`
  - `layout_config.padding_{top,right,bottom,left}` → `appearance_config.layout.padding.{top,right,bottom,left}`
  - `layout_config.margin_{top,right,bottom,left}` → `appearance_config.layout.margin.{top,right,bottom,left}`
  - The moved keys are stripped from `layout_config` after transfer. `full_width` intentionally stays on `layout_config` — it's a layout-behavior toggle, not an appearance override.
- `PageLayout` model: `appearance_config` added to `$fillable` and `$casts => 'array'`.
- `SystemPageSeeder` updated: the seeded Header layout now writes padding under `appearance_config.layout.padding` directly, so fresh-install does not rely on the migration's data phase.
- `docs/schema/page_layouts.md` rewritten to document the new column, the key split, and the `full_width`-stays-on-`layout_config` rule.
- Verified on dev DB: pre-existing rows transformed correctly (empty `appearance_config = '{}'` for layouts without relevant keys; nested shape populated otherwise). `migrate:fresh --seed` verified as a clean-bootstrap path.

### Phase 2 — Panel genericization

- `BackgroundPanel.vue` refactored. Old prop shape `widget: Widget` replaced with:
  - `config: Record<string, any>` — the full appearance_config bag
  - `imageUrl: string | null` — URL of the current background image (null if none)
  - `idPrefix: string` — used for the file input element id
  - `showImage: boolean` (default `true`) — hides the Image sub-tab when false; layouts pass `false` per design decision (no image support for layouts this session)
  - Events: `@update(path, value)` (path relative to the `background` subtree, e.g. `color`, `gradient`), `@uploadImage(file)`, `@removeImage()`
- `SectionLayoutPanel.vue` refactored. Old prop shape replaced with:
  - `config: Record<string, any>`
  - `showFullWidth: boolean` (default `true`) — layouts pass `false` because full-width lives on Column Settings for layouts, not on Margin & Padding
  - `fullWidthDisabled: boolean` + `fullWidthDisabledReason: string | null` — widgets use this to grey out the toggle for column-child widgets
  - Events: `@update(path, value)` (path relative to the `layout` subtree, e.g. `full_width`, `padding.top`)
- `InspectorPanel.vue` (widget inspector) updated to pass the new props and wire emits to `store.updateLocalAppearanceConfig(widgetId, 'background.' + path, value)` / `store.updateLocalAppearanceConfig(widgetId, 'layout.' + path, value)` respectively. Widget-visible behavior unchanged — pure structural refactor.
- `TextPanel.vue` untouched. Only widgets render it.

### Phase 3 — Layout inspector rewrite

- `LayoutInspectorPanel.vue` rewritten from a single bespoke panel into a three-tabbed inspector mirroring widget-inspector chrome:
  - Header row with type badge "Column Layout", debounced label input, delete button (preserves the confirmation dialog from the old implementation).
  - Tab strip via shared `InspectorTabs` component: **Column Settings** (default) | **Margin & Padding** | **Background**.
  - **Column Settings** tab: full-width checkbox at the top (user directive, non-negotiable), display toggle (flex/grid), columns count, grid-block controls (template-columns preset/manual, gap, auto-rows, align/justify-items), flex-block controls (justify-content, align-items, gap, flex-wrap, per-column flex-basis).
  - **Margin & Padding** tab: mounts the reused `SectionLayoutPanel` with `showFullWidth=false`, wired to the new store method.
  - **Background** tab: mounts the reused `BackgroundPanel` with `showImage=false`, wired to the new store method.
- Store additions (`editor.ts`):
  - `layoutInspectorTab` ref — persists the active layout tab across selection changes.
  - `updateLocalLayoutAppearance(layoutId, path, value)` — mirrors `updateLocalAppearanceConfig` but for layouts, writes through the debounced layout save.
- Types (`types.ts`): `PageLayout.appearance_config` added, plus a narrow `LayoutAppearanceConfig` type that omits the `text` subtree. `UpdateLayoutPayload.appearance_config` added.

### Phase 4 — PHP rendering path

- `AppearanceStyleComposer::composeForLayout(PageLayout $layout): string` — new method. Reads `appearance_config.background.{color, gradient, alignment, fit}` and `appearance_config.layout.{padding,margin}.{side}`. Emits `background-color`, `background-image` (gradient), `background-position`, `background-size`, `background-repeat`, `padding-{side}`, `margin-{side}`. Does **not** emit text color / text shadow / `full_width` override — those are widget-only concerns.
- `PageBlockRenderer::renderLayoutBlock` switched from reading `layout_config.background_color` / `padding_*` / `margin_*` to calling `composeForLayout`. The `full_width` check against `layout_config.full_width` stays put.
- `PageBuilderApiController::formatLayout()` now includes `appearance_config` in the serialized payload so the Vue app can read it.

### Phase 5 — API schema-key filter

- `PageBuilderApiController::updateLayout` validates `appearance_config` as array and filters via new private `filterLayoutAppearanceConfig()` with whitelist `['background', 'layout']`. The `text` key is stripped even if sent.
- The `layout_config` whitelist on the same endpoint no longer accepts `background_color`, `padding_*`, or `margin_*` — those moved to `appearance_config`. `full_width` stays accepted.
- `update_page` permission gate preserved.

## Mid-session bug fixes absorbed into the branch

The session's manual-test cycle surfaced several bugs tangential to the appearance-unification scope but immediate enough to fix on the same branch:

- **Column content overflow (public + preview).** `.layout-region__slot` (preview) and `.layout-column` (public) are grid items with default `min-width: auto`, which forbids shrinking below content min-content width. Wide widgets (carousel) blew out the grid track. Added `min-width: 0` in both places. Also applied to `.layout-region__slot-list` in the preview for flex-inside-grid safety.
- **Preview does not read appearance_config.** Found immediately after the migration landed: `LayoutRegion.vue`'s `containerStyle` computed still read `config.background_color` / `config.padding_*` / `config.margin_*` from `layout_config`, which the migration emptied. Rewrote to read from `appearance_config.background.color` and `appearance_config.layout.{padding,margin}.{side}` — parity with the PHP `composeForLayout`.
- **Selection outline shape.** User direction: selection outline should be a hard-edged rectangle that always spans the full width of the layout's horizontal claim, regardless of whether the content is contained (90% width). Moved the outline from `.layout-region__container` to `.layout-region` itself. The outline is now full-width with no border-radius (`.layout-region` has none set); contained-ness is now conveyed by the inner dashed container shrinking to 90% inside the outline.
- **Handle visible when selected.** The layout handle was opacity-0 except on hover; made it also opacity-1 when `.layout-region--selected` is on the parent, so selected layouts stay visually anchored.
- **Preview-scope scroll room.** Added `padding-bottom: 50vh` to `.widget-preview-scope.np-site` in `admin.css` (user applied directly) so widgets at the bottom of a page can scroll into view and stay positioned when the inspector is open.
- **Tablet/mobile handle sizing.** Attempted a container-query-based shrinking of the handle chrome at narrow preview widths. Change was scoped incorrectly — touched the desktop-default sizes — and had to be reverted. The underlying concern (handle dominates the column at mobile preset) is real but unresolved. Captured as a deferred decision; see below.

### Post-session-scope refactor — `api.ts` client factory

Manual test surfaced a reproducible "header shows footer content" bug on `EditPageTemplate`: add a column layout to the header tab → delete it → the header's page-builder now displays the footer's content. Root cause: `api.ts` held `baseUrl` / `lookupUrl` / `csrfToken` as module globals, written by `configure()` on each Vue-app mount. `EditPageTemplate` mounts two page-builder Vue apps simultaneously (header tab + footer tab); the second `configure()` call silently overwrote the first's `baseUrl`, so the header app's `createLayout` call actually posted to the footer's owner-scoped endpoint.

Fix: refactored `api.ts` into a `createApiClient(csrfToken, baseUrl, lookupUrl)` factory. The factory returns an `ApiClient` object with all the endpoint methods as closures over the per-instance config. Each Vue app creates its own client in `App.vue` via a new `store.configureApi(bootstrap)` action and looks it up via `store.requireApi()`. Module globals are gone. Every consumer (editor.ts store, `InspectorPresetGallery.vue`, `SelectField.vue`, `App.vue`) updated to use the per-instance client.

The remaining trigger — two Vue apps on one page — is captured as a deferred concern: the Template Editor Record Sub-Navigation stub (paired with the mobile-collapse work) switches `EditPageTemplate` to Filament record-sub-navigation, which renders only one sub-page per request, eliminating the two-apps-per-page pattern entirely. Hardening `api.ts` is no longer load-bearing once that session lands; today's fix just defuses the landmine in the meantime.

## Deferred decisions

- **Handle sizing at mobile / tablet preview presets.** The layout-region handle chrome (`.layout-region__handle`, `.layout-region__selector`) uses content-scale sizing (2rem text, 3.5rem height) dating to session 153 when layouts were introduced. At the Mobile (375px) preset, this dominates the column. An attempt to scope the fix via container queries touched the desktop-default sizing and was reverted at user direction. The underlying bug remains. Captured implicitly under the new "Column-Layout Mobile Collapse" stub — once columns collapse at narrow widths, the handle-sizing concern largely dissolves (the handle is less visually prominent on a single-column layout anyway).
- **Module-global `api.ts` hardening beyond the factory refactor.** The factory pattern solves the current bug. A deeper refactor (e.g. injecting the client via `provide`/`inject` rather than via the store) could be cleaner but isn't necessary for any known need. Captured as latent tech debt; the sub-navigation refactor eliminates the only known trigger case, at which point this concern becomes moot.
