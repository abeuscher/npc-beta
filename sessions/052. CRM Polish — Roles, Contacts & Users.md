# 052. CRM Polish — Roles, Contacts & Users

Two workstreams. Read all relevant files before touching anything.

**Architectural note:** `Contact` and `User` are intentionally kept as separate, unlinked models. Users are admin/staff accounts; Contacts are constituents. The concerns have different lifecycles and the privacy-first philosophy of the product is easier to honour with clean separation. No foreign key link will be added. This decision is recorded in `docs/decisions/contact-user-separation.md` — create that file at the start of the session (one paragraph, no code).

## Workstream 1 — Role Deletion Safety

Read `app/Filament/Resources/RoleResource.php` in full before making any changes.

Currently, deleting a role silently removes access from any users who held that role. The fix must gate the delete action behind a check, and surface the impact to the admin before they confirm.

### Requirements

1. Before a role can be deleted, check how many active users currently hold that role.
2. If the count is zero, allow deletion normally (no change to UX).
3. If the count is one or more:
   - Show the affected user count clearly.
   - Offer a **reassign** option: a dropdown of all other roles. Selecting a role and confirming reassigns all affected users to that role, then deletes the old role.
   - Offer an **acknowledge and delete** option: a separate confirmation that explicitly warns the admin that the affected users will lose all access. Require them to type or confirm something before proceeding — do not make this easy to click past accidentally.
4. The bulk delete action on the roles list should apply the same logic. If any of the selected roles have assigned users, block the bulk delete entirely and show an error notification listing the affected roles by name. Do not attempt reassignment in bulk — it is too ambiguous.

### Implementation notes

- Prefer overriding the delete action on the resource rather than adding a custom page. Read how the existing delete/force-delete actions are defined first.
- The user count check must use the role–user pivot, not a raw count that would miss soft-deleted users with active role assignments.
- `super_admin` is already protected from deletion at the resource level. Do not change that behaviour.

## Workstream 2 — Birthday & Age Fields

Read `app/Filament/Resources/ContactResource.php` and the contacts migration(s) before writing anything.

### Requirements

1. Add a `date_of_birth` column (`date`, nullable) to the `contacts` table. Write and run the migration.
2. Update `docs/schema.md` to reflect the new column.
3. Add `date_of_birth` to the contact edit form in an appropriate position within the Contact Information section.
4. On the contact **view/edit** page, display a read-only calculated **Age** field adjacent to `date_of_birth`. Age is calculated from `date_of_birth` to today — do not store it.
5. Add `date_of_birth` to the contact CSV export in `ListContacts.php`.
6. Do not add age verification logic or threshold checks in this session — that belongs in a volunteer management session where the business rules are defined.
