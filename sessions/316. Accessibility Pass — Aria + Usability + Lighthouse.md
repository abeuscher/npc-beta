# 316. Accessibility Pass — Aria + Usability + Lighthouse

Three accessibility-adjacent sweeps consolidated into one session. Surfaces inventoried across the public-render, admin, and (where applicable) portal surfaces. Findings triage at session-middle decides what lands this session vs. carves out.

---

## Plan reference

This session executes `sessions/release-plan.md` § E19 (execution-order position 45). The plan entry is canonical for scope and milestone-block positioning. This prompt covers session-shape detail: how the three sweeps run, how findings get triaged, what the carve-out criterion is.

## Session-specific deltas

- **Three sweeps in one session.** The user combined what could have been three separate sessions (Aria pass, usability pass, Lighthouse pass) into one consolidated effort because the findings overlap and triaging them together is more efficient than running three audits in series.
- **Audit-then-remediate shape.** First half of the session is the three audits, surfacing findings inclusively. Second half walks the findings with the user and applies the agreed subset iteration-sliced.
- **Text-toolbar `title` attributes fold in.** The housekeeping-inbox item for text-editor toolbar button `title` attributes is part of the Aria sweep — it ships in this session, not in a separate housekeeping batch.

## Phases

### Phase 1 — Aria sweep

Mechanical sweep over the markup surfaces:

- **Admin / Filament surfaces** — pages, actions, modals, form fields, toolbar buttons. Aria labels on icon-only buttons, `aria-describedby` on form-field hint text, `aria-current` on nav active states, focus management on modals.
- **Public render** — header, footer, nav, hero, all widgets. Landmarks (`<main>`, `<nav>`, `<header>`, `<footer>`), heading hierarchy, alt text on `<img>`, `aria-label` on icon-only social-share links.
- **Page builder canvas** — the Vue editor's own controls (widget chrome, inspector panels, inline toolbar). Aria labels on the inline-formatting toolbar buttons (the inbox `title` attribute item), keyboard reachability of all controls.
- **Portal** (if in scope per the demo) — login, dashboard, record views.

Output: a findings table grouped by surface. Each finding is one issue or one cluster (e.g. "every icon-only button in the inline toolbar is missing an aria-label" is one cluster, not 8 individual findings).

### Phase 2 — Usability sweep

Judgment-heavy review of operator-facing controls. The user is the primary tester here — the agent surfaces candidates, the user calls which are real findings.

- **Confusing labels** — controls that say one thing but do another, or that require domain knowledge to interpret. The columns-primitive alignment-tool labels (already in the inbox) belongs to this sweep too.
- **Hidden affordances** — features that exist but operators don't find. The page-builder ellipsis-menu (inbox item — visible on edit-detail but not on main edit view) belongs here.
- **Inconsistent patterns** — same shape of action presented differently across resources, same kind of decision asked differently in different places.
- **Friction points** — multi-step flows that should be one step, confirmation modals that ask for confirmation on safe actions, error messages that don't say what to do next.

Output: a findings list with severity (operator-blocking / operator-confusing / cosmetic) and surface citation.

### Phase 3 — Lighthouse audit

Tool-driven audit using the locally-installed Lighthouse app. Run against:

- The public-render demo site at the dev URL.
- A couple of key admin pages (login, dashboard, the page builder editor).

Capture the JSON reports. Focus on accessibility + best-practices categories; performance numbers are diagnostic, not gating (the dev stack isn't representative of production timing).

Output: Lighthouse category scores + per-category specific findings. Compare against any prior baseline if one exists.

### Phase 4 — Findings triage (walkthrough with user)

Combine the three sweeps' outputs. Walk with the user, classify each finding:

- **Apply this session** — small, low-risk, clearly scoped fixes.
- **Carve out to a follow-on session** — findings that have grown past one-iteration size or that require a design decision the session can't carry.
- **Defer to housekeeping inbox** — small but not-blocking; absorb in a future Batch session.
- **Won't-fix** — explicitly intentional, with citation.

### Phase 5+ — Remediation iterations

Apply the "apply this session" subset, iteration-sliced. Six-iteration ceiling on the remediation half.

## Open questions to resolve at session start

- **Surface scope.** Confirm whether the portal is in scope for this pass (it's part of the public-facing demo to varying degrees) or deferred.
- **Lighthouse invocation mechanism.** Direct app invocation, Chrome DevTools CLI, or another path?
- **Carve-out threshold.** Default leaning is "fix if one-iteration-sized, carve out if larger." Confirm the user wants that default or a tighter / looser threshold.
- **`@axe-core/playwright` integration.** Adding axe-core to the Playwright suite would catch a class of Aria regressions automatically. Worth picking up as a side-task this session, or defer to its own session?

---

## Out of scope

- WCAG-compliance certification — not the goal. The goal is "good-enough that the demo doesn't surface obvious accessibility failures and operators don't hit usability dead ends."
- Visual-design accessibility (color contrast, font sizes) — pairs with the Design System Editor track when that lands; not this session.
- Internationalization / RTL support — separate concern.
- Schema or behavioral changes — accessibility work is markup + attributes + tab order + label text.

---

## Testing

- **Slow test groups to run this session:** none expected.
- **New tests expected:** possibly — if axe-core is added to Playwright, the integration test itself is the new artifact. Aria additions typically don't get per-attribute tests (the attribute is the test); a smoke spec confirming the inline toolbar is reachable by keyboard counts.
- **Playwright:** isolated stack via `npm run test:e2e`. Any new accessibility spec lands as a standing regression artifact.
