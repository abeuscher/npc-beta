# Track: Fleet Manager Agent

The CRM-side companion to the Fleet Manager operational tool. Fleet Manager is a separate Laravel application in a separate repository that polls every CRM install on a schedule and surfaces health, version, and backup state in a single dashboard. This track owns the *CRM half* of that arrangement: the authenticated `/api/health` endpoint Fleet Manager polls, the version pipeline that feeds the response, and the backup mechanism whose success-record `last_backup_at` reports.

The canonical product spec for the entire two-repo system — purpose, scope, two-repo coordination protocol, local development scheme, resolved leans — lives at [`sessions/fleet-manager-planning-spec.md`](../fleet-manager-planning-spec.md). That doc is the seed for the Fleet Manager repo and the reference for both workstreams. This track doc carries CRM-side history and forward plan only.

This doc carries three things:

- **Status snapshot** — where the track is right now, what's next.
- **Phase Retrospectives** — compressed history of closed phases (sessions list, outcomes, key decisions, carry-forwards).
- **Forward plan** — the remaining phase, with design decisions, security posture, and known risks.

When a phase closes, its retrospective lands in this doc and its entry in the roadmap (`session-outlines.md`) collapses to a one-liner.

**v2.1.0 documentation revision — FM Security Posture Pivot absorption (session 253).** Sub-phase under Phase 3; not a new full phase, not a contract bump. Drop-in planning session triggered by FM session 013 close, where the conversation surfaced that FM was a single point of catastrophic compromise — encrypted client keypairs on FM's filesystem under `APP_KEY` with no documented recovery path. FM's pivot has five components; three have CRM-side dependencies that landed here:

- **`bin/rotate-fm-cert.sh`** — operator-facing rotation script. Validates input PEM via `openssl x509`, replaces `/opt/nonprofitcrm/nginx-certs/fm-client.crt` via temp-file + atomic `mv`, reloads nginx via `docker exec nonprofitcrm_nginx nginx -s reload` (graceful, near-zero polling blip), appends a host-side audit record to `/opt/nonprofitcrm/logs/fm-cert-rotations.log` (timestamp + SHA-256 fingerprint + cert subject).
- **`docs/runbooks/fm-compromise-recovery.md`** — fleet-wide compromise-recovery runbook. Covers pre-installation break-glass cert generation (offline keypair, cold-storage discipline, fingerprint-recording), at-recovery-time procedure (triage, per-CRM cert swap via the rotation script, FM-side clean-provision step, re-establish the cold-storage invariant), post-recovery audit-log review and fleet-inventory check.
- **`docs/fleet-manager-agent-contract.md` Security Posture revision** — new "Recovery posture and FM-side trust assumptions" sub-section + v2.1.0 revision CHANGELOG entry. Section-header preface labels item-by-item shipped (item 1: break-glass recovery path) vs FM-side intended-posture, Beta-1 scope (item 2: off-filesystem-key posture; item 3: audit-sink discipline). Future revisions promote intended items to shipped without a contract bump.

Load-bearing decisions: (1) Sequencing — new A1c entry in `release-plan.md` (FM compromise recovery infrastructure) over folding into A2 or spreading across entries. The compromise-recovery infrastructure is a discrete operational discipline; A1b (mTLS auth surface) + A1c (mTLS recoverability) keeps the thread coherent. (2) Reload mechanism — `nginx -s reload` over container restart or `kill -HUP` discovery. The bind-mount-edit-breaks-container concern (captured at 251 close in `feedback_bind_mount_edit_breaks_container.md`) was specific to single-file mounts; writing into a directory mount (`/opt/nonprofitcrm/nginx-certs`) is normal filesystem behavior and propagates cleanly. The rotation event is a host-level operator action; putting the audit trail inside `/api/logs`-exposed application logs would defeat the compromise-recovery defensive intent (the audit trail must live outside the surface that recovery is defending). **v2.1.0 sub-phase — additive log-fetch endpoint (session 251).** Sub-phase under Phase 3 (the v2 major); not a new full phase. Adds `GET /api/logs` so FM 013+ can fetch the Laravel application log from a CRM install without operator SSH. Single `?lines=N` parameter (default 500, max 10000 clipped silently); JSON envelope `{ lines, lines_returned, lines_truncated, source }`; ~850 KB raw byte cap under a 1 MB encoded body ceiling. Reuses the existing nginx-terminated mTLS gate via a per-location `if`-return-403 block on `/api/logs` mirroring `/api/health`'s exactly. New `LogsController` at `app/Http/Controllers/Api/Fleet/LogsController.php` implements a backward-read tail (`fseek(SEEK_END)`, 8 KB chunks growing backward, long-line guard for chunks containing zero `\n` to avoid silent truncation of >8 KB log lines). New `logs` disk in `config/filesystems.php` rooted at `storage_path('logs')` so tests use `Storage::fake('logs')` cleanly. `HealthController::CONTRACT_VERSION` flipped `'2.0.0' → '2.1.0'` (single constant covers both endpoints; `/api/logs` does not carry its own `contract_version` field — FM reads version from `/api/health`). Spec doc gained an Endpoints section (plural), full `/api/logs` Request/Response/Error documentation including line semantics, multi-container unification, scope clarification ("what `/api/logs` is NOT"), and v2.1.0 CHANGELOG entry. 17 new feature tests at `LogsControllerTest`, 1801 fast tests at close (+17 from 1784 baseline). Forward-compatible with v2.0.0 consumers; FM-side consumers wanting the new endpoint upgrade their HTTP client.

Load-bearing decisions: (1) Tail-only `?lines=N` (over `?date=YYYY-MM-DD` or `?since=ISO8601`) — today's `LOG_CHANNEL=stack` / `LOG_STACK=single` posture writes one ever-growing `laravel.log`, so date-partitioned options can't work without a channel migration; tail-only has the smallest implementation surface and the smallest spec footprint. Date-based selection becomes a future v2.x addition pairing with the channel migration if/when needed. (2) JSON envelope (over `text/plain`) for symmetry with `/api/health`'s JSON shape — keeps consumer-side parsing uniform; the truncation hint (`lines_truncated`) is operationally useful (FM warns the operator they hit the cap). (3) Newline-delimited rows count as "lines," **not** logical log entries — Laravel writes a multi-line stack trace across many `\n`-delimited rows, so `?lines=500` during a heavy-error period might surface as few as 3 logical entries. Logical-entry parsing is out of scope at v2.1; documented explicitly in the spec so FM-side UX assumptions don't drift. (4) Backward-read with explicit long-line guard (over `file($path)` or `tail -n` shell-out) — `laravel.log` can grow to multi-GB at scale; `file()` OOMs and shell-outs introduce a binary dependency. The 8 KB chunk loop with growing-buffer remainder when a chunk contains zero `\n` is the load-bearing operational concern at scale. (5) Threshold-derivation deferral held — v2.1.0 stays narrowly the log-fetch addition, matching the v2.0.0 retrospective's discipline of one major boundary change per cutover.

After editing `docker/nginx/default.conf`, the running container's `stat` reported `Links: 0` on the file; `docker compose restart nginx` failed with an OCI mount error citing a no-longer-existent Docker Desktop bind-mount cache path; recovery required `docker compose up -d --force-recreate nginx`. **Load-bearing decisions that survived Phase 1:**

- **Bearer token in `Authorization` header, timing-safe `hash_equals` comparison.** Simpler than signed-request schemes; meets v1 needs.
- **Per-install API key, manual env-var bootstrap.** Each CRM install has its own key. Reuse across installs is forbidden by the contract. Automation deferred.
- **Endpoint returns 200 even when subchecks are red.** Fleet Manager always gets structured data to act on. 503 is reserved for a future "endpoint disabled" state, never emitted by v1.
- **Docker build-arg + image-baked VERSION file** (not git-rev-parse-at-boot, not env-var). Build-arg is sourced from `${GITHUB_SHA::7}` in CI; `/var/cache/app/VERSION` lives outside the bind-mount tree so local dev's `.:/var/www/html` doesn't shadow it. Local dev defaults the build-arg to `dev`.
- **Subcheck failure messages carry exception class names only, never `$e->getMessage()`.** Leak risk on unhandled exceptions.
- **`last_backup_at` is green-with-null in v1, not yellow.** "Unknown without a pipeline" is not an alarming state; the Backup Pipeline (Phase 2) flips this to threshold-driven status.
- **Spec doc is the canonical communication surface, not the live response.** The CRM emits the spec; the FM repo consumes it via WebFetch against the canonical raw-GitHub URL. No automatic notification — FM-side discipline is "fetch + diff against local cache on every boundary-touching session."

## Stance

- **Single operator, single product, no multi-tenancy.** Fleet Manager is not client-facing and is not a product. It is internal tooling. The contract surface is correspondingly tight: one key per install, one operator polling, one dashboard.
- **The CRM is the source of truth for the contract; FM is a consumer.** The locality-of-policy-to-surface decision: the side that defines the endpoint owns the spec doc. FM reads via WebFetch; FM never proposes contract changes directly — if FM needs a new subcheck, the request lands as a CRM-side issue and the CRM authors the change.
- **Boundary-touching sessions update the spec doc, the Cross-Repo block, and the CHANGELOG before the next boundary-touching session in either repo.** The protocol is in the planning spec ("Two-Repo Coordination Protocol" section); the prompt template's standing cross-repo flag enforces it on every session in either repo.
- **Operational simplicity beats sophistication for v1.** Manual env-var bootstrap; one key per install; no rotation flow; no admin UI. Each of those is a deferred-but-named carry-forward; landing them requires a forcing function the current scope does not have.
