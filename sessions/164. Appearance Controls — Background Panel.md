# 164. Appearance Controls â€” Background Panel

Phase 3a of the Appearance Controls project. Builds the Background panel of the universal Appearance tab in the Vue inspector, extracts a shared `AppearanceStyleComposer` service so the public and admin renderers stay in lockstep, and extends `GradientPicker` / `GradientComposer` to support per-stop alpha so that gradients can act as image tints.

This session is the first of a two-iteration split of the original 164 outline. Session 165 picks up the Text panel, the Section Layout panel, the retirement of `WidgetAppearanceControls.vue` and `SpacingControl.vue`, and the regression smoke tests for the widgets that lost fields in session 162.

## Design decisions (resolved before starting)

### Storage shape â€” `appearance_config.background`

```
background: {
  color: '#hex',
  gradient: {
    gradients: [
      { type, from, from_alpha, to, to_alpha, angle, css_override },
      ...
    ]
  },
  alignment: 'top-left' | 'top-center' | 'top-right'
             | 'middle-left' | 'center' | 'middle-right'
             | 'bottom-left' | 'bottom-center' | 'bottom-right',
  fit: 'cover' | 'contain'
}
```

The background image itself is stored exclusively in the Spatie media collection `appearance_background_image` on `PageWidget` (single-file). **No `image_id` is written into the jsonb bag** â€” Spatie's collection is the single source of truth (Option B, decided 2026-04-11). The renderer detects "image present" via `$pw->getFirstMedia('appearance_background_image')`. The Vue inspector detects "image present" via the new `appearance_image_url` field added to the `formatWidget` response.

`background.gradient.gradients[i].from_alpha` and `to_alpha` are integers in `[0, 100]`. Default `100` (fully opaque) when missing. The composer divides by `100` to produce the `0..1` rgba alpha channel.

There is **no** `background.overlay` key. The overlay concept has been removed from the design â€” gradients with rgba alpha *are* the overlay. To produce a "dark tint over a hero photo" the user creates a gradient with a dark color and `from_alpha`/`to_alpha` < 100.

### Layer order â€” single CSS shorthand

A widget's outer div emits a single combined background style:

```css
background-color: COLOR;
background-image: GRADIENT_CSS, url(IMAGE_URL);
background-position: ALIGNMENT_CSS;
background-size: FIT_CSS;
```

In a CSS `background-image` comma stack, the **first** comma item paints on top. So `gradient, url(image)` paints the gradient over the image; the image paints over `background-color`. This collapses the entire layered-background concept to a single inline style with no markup change to the widget outer div â€” no layer divs, no `widget__content` wrapper, no `position: relative; overflow: hidden`, no new SCSS partial. The widget outer markup stays exactly as it is today.

When only one of gradient/image is present, the shorthand uses just that one term. When neither is present, none of the `background-image`/`background-position`/`background-size` props are emitted at all.

### Service extraction â€” `App\Services\AppearanceStyleComposer`

Today, two places compute inline styles from `appearance_config`:

- [page-widgets.blade.php](resources/views/components/page-widgets.blade.php) â€” public renderer.
- [PageBuilderApiController::buildInlineStyles()](app/Http/Controllers/Admin/PageBuilderApiController.php#L768) â€” admin preview renderer.

Both currently emit color / text / padding / margin / full-width. Both need to expand for background image, gradient, alignment, and fit. **Duplicating the expansion is unsafe** â€” any safety regression in one place silently diverges from the other.

This session extracts the logic to `App\Services\AppearanceStyleComposer`. Single class, one main entry:

```php
public function compose(PageWidget $pw): array
```

Returns a flat associative array with documented keys (the project leans on plain arrays for small structs; follow that convention):

```
[
  'inline_style'  => string,  // full inline style for the outer widget div
  'is_full_width' => bool,    // resolved from appearance_config.layout.full_width with fallback
                              // to widget_type default; forced false when widget.layout_id is non-null
]
```

All hex/alpha/int safety lives in the composer. Both renderers become thin wrappers that ask the composer for the struct and emit the markup.

The composer:

- Validates colors against the existing hex pattern (`/^#(?:[0-9a-fA-F]{3}|[0-9a-fA-F]{6})$/`); rejects anything else.
- Casts spacing values to int.
- Calls `App\Services\GradientComposer::compose()` (extended in this session â€” see below) to produce the gradient CSS.
- Resolves the background image via `$pw->getFirstMedia('appearance_background_image')?->getUrl('webp')` â€” uses the optimized webp variant. Returns null if no media.
- Maps the 9-point alignment value (`top-left`, `top-center`, `top-right`, `middle-left`, `center`, `middle-right`, `bottom-left`, `bottom-center`, `bottom-right`) to a CSS `background-position` string (`0% 0%`, `50% 0%`, `100% 0%`, `0% 50%`, `50% 50%`, `100% 50%`, `0% 100%`, `50% 100%`, `100% 100%`). Use a small lookup table inside the composer. Defaults to `center` (`50% 50%`) when missing or malformed.
- Validates `fit` against the allowlist `{'cover', 'contain'}` and falls back to `cover`.
- Resolves `is_full_width`: if `appearance_config.layout.full_width` is non-null, use that; else fall back to `widget_types.full_width`. **If `widget.layout_id` is non-null, force `false`** â€” column-child widgets cannot be full-width, the parent column controls width.
- Composes the final `inline_style` string by joining only the props that are present and valid. Skips the entire `background-image` / `background-position` / `background-size` set if neither a gradient nor an image is present.

### `GradientComposer` and `GradientPicker` extensions â€” per-stop alpha

`GradientComposer` ([app/Services/GradientComposer.php](app/Services/GradientComposer.php)) currently enforces hex-only colors via `HEX_PATTERN`. To support transparency:

- Each gradient layer in the input array gains optional `from_alpha` and `to_alpha` integer fields. Both default to `100` when missing.
- The composer reads each color as hex (existing validation unchanged), reads its alpha as an int, clamps to `[0, 100]`, divides by `100` to get a `0..1` float, and emits `rgba(r, g, b, a)` instead of `#hex` when alpha < 100. When alpha === 100, the existing `#hex` output is preserved (cleaner CSS for the common case, and keeps existing `GradientComposerTest` cases passing untouched).
- The hexâ†’rgb conversion supports both 3-digit and 6-digit hex (parse to integer triplets).
- `CSS_OVERRIDE_PATTERN` stays hex-only â€” `from_alpha`/`to_alpha` apply to non-override gradients only. The override path is rarely used and stays strict.

`GradientPicker.vue` ([resources/js/page-builder-vue/components/primitives/GradientPicker.vue](resources/js/page-builder-vue/components/primitives/GradientPicker.vue)) adds per-stop alpha sliders:

- Each layer's editor gains a "From opacity" and "To opacity" slider, integer 0..100, sitting under the existing color pickers for that layer.
- The 8 built-in presets all emit `from_alpha: 100, to_alpha: 100` â€” solid gradients stay the default look.
- The `GradientLayer` TypeScript type in [resources/js/page-builder-vue/helpers/gradient.ts](resources/js/page-builder-vue/helpers/gradient.ts) gains the two alpha fields (optional, default 100).
- The TypeScript `composeGradientCss` helper mirrors the PHP composer's rgba emission logic so the inspector preview matches the rendered output exactly.

### Background panel UI â€” `BackgroundPanel.vue`

New file at `resources/js/page-builder-vue/components/appearance/BackgroundPanel.vue`. Sections within the panel (each a small subgroup with its own label):

**Color & Gradient**

- `ColorPicker` bound to `appearance_config.background.color`.
- `GradientPicker` bound to `appearance_config.background.gradient`.
- A small helper line under the gradient picker: "Use opacity sliders on each gradient stop to tint a background image."

**Image**

- Image upload tile (drag-drop or click to upload). Uses the new `POST widgets/{widget}/appearance-image` endpoint. Shows a thumbnail when an image is present (sourced from `widget.appearance_image_url`).
- A "Remove image" link below the thumbnail when an image is present, calling `DELETE widgets/{widget}/appearance-image`.
- `NinePointAlignment` primitive bound to `appearance_config.background.alignment`. Disabled (gray + tooltip "Upload an image to set alignment") when no image is present.
- A `<select>` for `appearance_config.background.fit` with options `Cover` and `Contain`. Disabled when no image is present.

The panel header reads "Background", with a subtle divider above it. The panel mounts as the first child of the Appearance tab â€” above the existing `WidgetAppearanceControls`, which still exists in this session and retires in 165.

Every binding for `appearance_config.background.*` flows through `store.updateLocalAppearanceConfig(widget.id, path, value)` â€” the existing dotted-path API. No store changes for the appearance bag itself; only the new image upload/remove actions are added (see below).

### New API endpoints

**`POST /admin/api/page-builder/widgets/{widget}/appearance-image`**

- Permission: `update_page` (same as the existing image endpoint).
- Validation: `file` is required, `mimes:png,jpg,jpeg,gif,webp,svg`, `max:51200` (50 MB).
- Clears the existing `appearance_background_image` collection on the widget (single-file).
- Adds the new file to the collection.
- Returns `{ "url": "...webp variant url..." }` so the client can immediately render the thumbnail.
- Does **not** mutate `widget.appearance_config`.

**`DELETE /admin/api/page-builder/widgets/{widget}/appearance-image`**

- Permission: `update_page`.
- Clears the `appearance_background_image` collection.
- Returns `{ "removed": true }`.

Both routes registered in [routes/admin-api.php](routes/admin-api.php) next to the existing `widgets/{widget}/image` routes.

The store gains two new actions in [editor.ts](resources/js/page-builder-vue/stores/editor.ts):

- `uploadAppearanceImage(widgetId, file): Promise<string|null>` â€” calls the POST endpoint, on success sets `widgets[id].appearance_image_url` to the returned URL, marks the widget dirty, and triggers `refreshPreview`.
- `removeAppearanceImage(widgetId): Promise<void>` â€” calls the DELETE endpoint, on success sets `widgets[id].appearance_image_url` to `null`, marks dirty, refreshes preview.

Both modelled on the existing `uploadImage` / `removeImage` actions.

### `formatWidget` / `formatWidgetWithPreview` â€” new field

Both `formatWidget` and `formatWidgetWithPreview` add a top-level field:

```
'appearance_image_url' => $pw->getFirstMedia('appearance_background_image')?->getUrl('webp'),
```

Sibling to the existing `image_urls` map (not nested inside it â€” `image_urls` is keyed by widget config field name and the appearance image lives in a different namespace; mixing them would muddy that contract).

The field is in the "always merge" set under the session 163 selective merge contract â€” it's server-authoritative, the JS doesn't mutate it locally. The selective-merge skip list (`config`, `appearance_config`, `query_config`, `label`) is unchanged.

### `PageWidget::registerMediaCollections()`

The method does not exist on `PageWidget` today â€” [app/Models/PageWidget.php](app/Models/PageWidget.php) has `registerMediaConversions` but no `registerMediaCollections`. Add the method and register a single-file `appearance_background_image` collection:

```php
public function registerMediaCollections(): void
{
    $this->addMediaCollection('appearance_background_image')->singleFile();
}
```

The existing `registerMediaConversions` already generates the webp variant + responsive sizes for any image media added to any collection on this model, so no conversion changes are needed.

## Security checklist

Before closing:

- [ ] All color values pass through the hex pattern validator before being written to inline styles or rgba conversions.
- [ ] All spacing values are cast to int before being written to inline styles.
- [ ] Gradient alpha values are cast to int and clamped to `[0, 100]` before division.
- [ ] Image URLs come exclusively from `Media::getUrl('webp')` â€” never from user-controllable strings.
- [ ] Background `fit` value is validated against the `{cover, contain}` allowlist.
- [ ] 9-point `alignment` value is validated against the alignment allowlist.
- [ ] Gradient CSS is composed exclusively via `App\Services\GradientComposer::compose()`.
- [ ] The new upload endpoint requires `update_page` permission and validates `mimes` + `max` on the file.
- [ ] The new delete endpoint requires `update_page` permission.
- [ ] Column-child widgets (`widget.layout_id !== null`) cannot bypass the parent column's width by setting `appearance_config.layout.full_width = true`. The composer enforces this defensively.
- [ ] `formatWidget`'s new `appearance_image_url` field does not leak any URL outside the `Media::getUrl('webp')` path.
- [ ] The `AppearanceStyleComposer` refactor does not regress any of the existing safety checks in `PageBuilderApiController::buildInlineStyles` or `page-widgets.blade.php`.
