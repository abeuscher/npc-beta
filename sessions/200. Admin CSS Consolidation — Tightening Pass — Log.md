### Phase 2 — Token substitution for exact RGB matches

Where a `--np-control-*` token's value is an exact match for a raw RGB value already in `admin.css`, the RGB was replaced with the token. Scope: only the matches explicitly called out in the prompt (plus the Trix toolbar's gray-300 pair, also called out).

Swapped to `var(--np-control-chip-border)`:

- Table row dividers — `html:not(.dark) .fi-ta-row { border-color: … }`.
- Table header cell bottom border — `html:not(.dark) .fi-ta-header-cell { border-bottom-color: … }`.
- Trix toolbar separator — `.fi-fo-rich-editor-toolbar { border-bottom-color: … }`.
- Trix button border — `.fi-fo-rich-editor-toolbar-btn { border: 1px solid … }`.

**Not swapped**, flagged for future decisions:

- Filament `fi-*` ring overrides use `rgb(156 163 175)` (gray-400). That IS an exact match for `--np-control-border-hover` (`#9ca3af`), but the prompt explicitly instructed to leave these alone and to flag the gray-300-vs-gray-400 question. The design question is: should form / section / input / fieldset outlines tighten from gray-400 down to `--np-control-chip-border` (`#d1d5db`), loosen the other way, or stay where they are? No colour change made without explicit approval.
- Trix button hover `rgb(229 231 235)` (gray-200). The session prompt stated "no exact token match, leave as raw RGB or note for future token addition." In fact `--np-control-active-bg` = `#e5e7eb` = `rgb(229 231 235)` is an exact match — the prompt was mistaken. Left as raw RGB per the explicit prompt instruction, but noting the exact match here so a follow-up can swap it if desired.
- Trix `.trix-active` border `rgb(107 114 128)` (gray-500). Exact match for `--np-control-border-active`. Not swapped — the prompt did not call it out, and its current rule is inside the Phase 3 `!important`-removal block, which was touched separately. A semantically-correct swap is available if the user wants it in a follow-up.

**Prompt-vs-reality note on token names.** The prompt's Phase 2 block quoted `--np-control-border: #d1d5db`. The actual value in `admin.css` is `--np-control-border: #d9d9d9` and the `#d1d5db` value lives on `--np-control-chip-border`. Substitutions were made to the actual exact-match token (`--np-control-chip-border`) in keeping with the prompt's "exact match" rule. This semantic awkwardness (a table row divider routed through a "chip-border" token) is worth revisiting when any future tightening pass reviews the token vocabulary.

### Phase 3 — Trix `!important` removal

Specificity raised by prepending `.fi-fo-rich-editor-toolbar` to each button/icon/hover/active selector, and the `!important` flags dropped. Specifically:

- `.fi-fo-rich-editor-toolbar-btn` → `.fi-fo-rich-editor-toolbar .fi-fo-rich-editor-toolbar-btn` (height / min-width / border / border-radius — all `!important` removed).
- `.fi-fo-rich-editor-toolbar-btn svg` → `.fi-fo-rich-editor-toolbar .fi-fo-rich-editor-toolbar-btn svg` (height / width — `!important` removed).
- `.fi-fo-rich-editor-toolbar-btn:hover` → `.fi-fo-rich-editor-toolbar .fi-fo-rich-editor-toolbar-btn:hover` (background-color — `!important` removed).
- `.fi-fo-rich-editor-toolbar-btn.trix-active` → `.fi-fo-rich-editor-toolbar .fi-fo-rich-editor-toolbar-btn.trix-active` (border-color / background-color — `!important` removed).

The separator rule `.fi-fo-rich-editor-toolbar { border-bottom-color: … }` simply had the `!important` dropped without extra specificity — admin.css loads after Filament's Vite theme via the `panels::head.end` render hook, so source order carries equal-specificity ties.

**No `!important` flags remained in the Trix block after the pass.** The block-header comment was updated to document the specificity-raise approach instead of the old "`!important` is required" note.
