## Architectural calls (resolved at session start; not relitigated at apply time)

Per the session prompt's hard-no list + acceptance criteria. The two non-trivial calls below are documented adjustments to the prompt's assumptions, not deviations from its intent:

1. **Navigation storage is relational, not SiteSetting JSON.** The prompt's draft assumed a `SiteSetting` row holding the navigation tree. Confirmed at session start by reading `NavigationMenuResource` and the migration schema: navigation lives in two real tables — `navigation_menus` (handle + label) and `navigation_items` (label, url, page_id, parent_id, sort_order, target, is_visible, navigation_menu_id). Per the prompt's own instruction ("adjust the export shape to match"), the bundle key is **`payload.navigation_menus`** (plural list, not singleton object) and each entry is `{menu: {handle, label}, items: [{label, url, page_slug, target, is_visible, sort_order, children: [...]}]}`. Two-deep nesting matches the Filament editor's repeater shape (`saveItems()` parent-then-children); the importer mirrors that two-pass write path exactly.
2. **No `import_id_maps` plumbing exists in the 309 baseline.** The prompt referenced an `import_id_maps` / `findByExternalId` pattern as the established cross-entity reference plumbing. Verified absent: 309's importer resolves templates by `(name, type)` and pages by `slug`, full stop. The A001 entities follow the same shape — handle / slug as identity, denormalised name/email/slug as cross-entity references with graceful fallback (warn + leave FK null) when the target row is absent. No new abstraction extracted; four entity methods written explicitly per the prompt's "extend, don't reinvent" rule.

## What was built

Four entity exporters + importers + analyzer extensions + round-trip Pest tests, one phase per entity. All work on `session-A001/1` as four atomic commits. Diff scope: `app/Services/ImportExport/ContentExporter.php`, `app/Services/ImportExport/ContentImporter.php`, `tests/Feature/ContentImportExportTest.php`, `VERSION`, plus this log + the `completed-sessions.md` index entry. ### Phase 1 — Navigation

- **Exporter:** `ContentExporter::exportNavigation(array $menuIds): array` — emits `payload.navigation_menus: [{menu, items}]`. Per-item page references travel as `page_slug` (resolved to id on import, warn-on-miss). Two-deep tree (parent + children), matching the editor.
- **Importer:** `ContentImporter::importNavigationMenu($data, $log)` — upsert menu by `handle`, wipe existing items, re-insert parents then children. Items with `page_slug` resolve to `page_id` via `Page::where('slug', ...)->value('id')`; absent slug leaves `page_id` null with a warning and keeps the `url` fallback if present.
- **Import-loop wiring:** runs after `importPage()` so freshly imported pages are visible to the slug resolver.
- **Opt-flag:** `import_navigation` (default true) added to `defaultImportOpts()`.
- **Analyzer:** new `navigation_menus[{handle, label, items_count, exists_locally}]` key; `exists_locally` batched via `NavigationMenu::whereIn('handle', ...)`.
- **Round-trip Pest:** create a menu with 2 roots (one URL-only, one page-linked) + 2 children (one `_blank`, one `is_visible=false`), export, wipe items + mutate menu label, import, assert the menu and four items round-trip with parent_id wiring + page reference intact. Plus three guard tests: missing-slug warning, analyze() reporting, and `import_navigation=false` skip.

### Phase 2 — Products

- **Exporter:** `ContentExporter::exportProducts(array $productIds): array` — emits `payload.products: [{product, prices, media}]`. The product's single `product_image` media collection rides along as a descriptor (mirroring `serializePageMedia()`); the standalone `payload.media` flag isn't needed because per-entity bundles already carry their own media descriptors in 309's design.
- **Importer:** `ContentImporter::importProduct($data, $log)` — `updateOrCreate` by `slug`, wipe + re-insert prices (canonical-from-bundle, mirrors the page widget pattern), `rewireProductMedia()` for the `product_image` collection.
- **Opt-flag:** `import_products` (default true).
- **Analyzer:** new `products[{slug, name, prices_count, exists_locally}]` key.
- **Round-trip Pest:** create a product with two prices + a `product_image` media row, export, DB-level wipe of prices + media (bypasses the Spatie observer so the file stays on the fake disk for import to find), import, assert id-preserved overwrite + price list + media re-attachment. Plus three guards: no-media single-price case, analyze() reporting, `import_products=false` skip.

### Phase 3 — Events

- **Exporter:** `ContentExporter::exportEvents(array $eventIds, array $opts = []): array` — emits `payload.events: [{event, tiers, media, registrations?}]` + `payload.pages: [...landing pages...]` (landing pages travel alongside via the existing page serializer so `landing_page_slug` resolves on re-import). Registrations are operator data: the `registrations` key is **omitted entirely** when `with_registrations=false` (default) — a re-import of an event without that key leaves any existing registrations on the target untouched; presence of the key (even when empty) means "registrations are canonical from bundle, wipe + replace." Tier back-references on registrations travel as `ticket_tier_name` so a re-import with regenerated tier UUIDs still re-resolves correctly. Contact / organization back-references on registrations travel as `contact_email` / `organization_name`.
- **Importer:** `ContentImporter::importEvent($data, $log)` — `updateOrCreate` by `slug`, wipe + re-insert tiers, conditionally wipe + re-insert registrations (only when the key is present), `rewireEventMedia()` for the three event-level collections. Landing-page link resolves via slug-lookup, sponsor org via name-lookup, both warn-on-miss + leave FK null.
- **Import-loop wiring:** runs after `importPage()` so landing-page slug resolves against pages we just imported.
- **Opt-flag:** `import_events` (default true).
- **Analyzer:** new `events[{slug, title, tiers_count, registrations_count, exists_locally}]` key.
- **Round-trip Pest:** create an event with a landing page + two tiers (VIP + General) + three registrations (one per tier + one tier-less attended comp), export with `with_registrations=true`, wipe registrations + tiers + clear landing_page_id, import, assert id-preserved overwrite + tier list + registration list with `ticket_tier_name` re-resolved correctly + landing_page_id reattached. Plus three guards: default-off registrations leave existing untouched, analyze() reporting, `import_events=false` skip.

### Phase 4 — Collections

- **Exporter:** `ContentExporter::exportCollections(array $collectionIds): array` — emits `payload.collections: [{collection, items}]`. Item `data` JSON travels verbatim (the collection's `fields` config is the shape's source of truth — the exporter never policies field shape). Item tags travel as `[{name, type}]` so any tag type (`collection_item`, `contact`, etc.) round-trips. Item-level media is **out of scope for the first pass** — collection items can implement `HasMedia` with dynamic collections driven by the `fields` config, but the prompt's test scope was minimum-viable (three items, one tagged) so the media surface is deferred (see Deferred decisions).
- **Importer:** `ContentImporter::importCollection($data, $log)` — `updateOrCreate` collection by `handle`, wipe + re-insert items, per-item tag sync via `Tag::firstOrCreate(['name', 'type'])`.
- **Opt-flag:** `import_collections` (default true).
- **Analyzer:** new `collections[{handle, name, items_count, exists_locally}]` key.
- **Round-trip Pest:** create a collection with three fields (text + textarea + image) and three items (one tagged "Featured"), export, wipe items + mutate name + `is_active`, import, assert id-preserved overwrite + field config + all three items with verbatim `data` JSON + tag attachment preserved + `is_published` flag preserved. Plus two guards: analyze() reporting, `import_collections=false` skip.

## Security considerations

The new entity importers mirror 309's existing safety conventions:

- All four `importX()` methods run **inside the same `DB::transaction()`** the existing page / template / design / media passes use — a partial-fail rolls back wholesale, no half-state.
- Cross-entity references that don't resolve **degrade to null + warning** rather than throw or accept arbitrary input (navigation `page_slug` → page lookup; event `landing_page_slug` / `sponsor_organization_name` / registration `contact_email` / registration `organization_name` / registration `ticket_tier_name`).
- No new attack surface. Bundle envelope validation (309), zip-slip / zip-bomb guards (308), and the `format_version` major-bump rejection (1.x.y only) all carry through unchanged.
- Tag creation on collection import uses `Tag::firstOrCreate(['name', 'type'])`, which can land new tag rows on a target install. This is intended behaviour for round-trip fidelity (the alternative — refusing to create tags — would silently drop tag references on first import); the operator-side guard is the import opt-flag (`import_collections=false` skips the whole collections pass).

## Deferred decisions / carry-forward

- **Item-level media for collections.** `CollectionItem` implements `HasMedia` with dynamic media collections driven by the collection's `fields` config (`image`-type fields produce a `{key}` media collection). The first-pass collection exporter/importer does **not** carry item media; only the `data` JSON + tags round-trip. A future session can extend `serializeCollection()` to walk the collection's `fields` for `image` types and emit per-item media descriptors keyed by field, mirroring the widget-config-schema-driven pattern in `serializeWidgetMedia()`.
- **No `exportSite()` extension.** Session 310's `exportSite()` rollup enumerates `Page` + `Template` only. A001 deliberately does **not** extend the rollup to enumerate events / products / navigation / collections — the question of "what does the full site mean?" is a separate product call (button-grade vs backup-grade) and was out of scope for this agentic pass per the prompt's hard-no list (`app/Filament/Pages/`). When the operator-side decision is made, the additions are a one-line `pluck` per entity in `exportSite()` plus a small UI copy update on `SiteImportExportPage`.
- **UI surface for the four new entity exports.** A001 ships service-layer exporters only. The per-entity export action surface (a "Export as bundle" action on `EventResource`, `ProductResource`, `NavigationMenuResource`, `CollectionResource`) is a follow-up — none of those Filament Resources is in this session's diff scope.
- **Analyzer UI summary.** `SiteImportExportPage`'s manifest summary still reads "X pages, Y templates, theme, Z media" — a one-line addition per new entity (`+ N events, + N products, + N menus, + N collections`) is the follow-up touch on the UI side, deferred until the operator-side rollup-scope decision lands.
