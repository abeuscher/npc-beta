## What was built

### Phase 1 — Defaults export endpoint + UI button

- New `App\Http\Controllers\Admin\WidgetDefaultsController@export` at `POST /admin/api/page-builder/widget-defaults/export`. Accepts `{ widget_id: uuid }`, returns `{ php: string }` containing a paste-ready `public function defaults(): array { ... }` method body. Gated on `update_page`.
- Values emitted are the resolved config (`WidgetConfigResolver::resolvedDefaults` merged with `$pw->config`, instance overrides win). Keys emitted in `config_schema` declaration order. PHP formatter inlined in the controller, using the same shape the Vue `buildPhpLiteral` helper uses for draft-preset export.
- New `api.exportDefaults(widgetId)` in `resources/js/page-builder-vue/api.ts`.
- New "Copy as defaults()" button in `resources/js/page-builder-vue/components/InspectorPresetGallery.vue` next to the existing "Save current appearance as preset" button. On click, copies the returned PHP string to the clipboard and shows a transient "Copied!" affordance.
- Feature test `tests/Feature/WidgetDefaultsControllerTest.php`: shape + schema-order assertion, round-trip `eval()` assertion that the emitted PHP parses and returns the expected resolved values, and two permission tests (401 unauthenticated, 403 without `update_page`).

### Phase 2 — Widget lint command

- New `App\Console\Commands\WidgetsLintCommand` (`widgets:lint`) — shells out to `artisan test --group=widget-lint` and prints a one-line pass/fail summary; on failure, prints the underlying Pest output.
- Tagged every existing test in `WidgetManifestTest` and `WidgetRegistryTest` with `->group('widget-lint')` (the one test that ends in `->throws()` was left untagged because it asserts error behavior, not a lint-worthy invariant).
- New `tests/Feature/WidgetDefaultsLintTest.php` with three new assertions:
  - **Type alignment.** For each schema field: `toggle → bool`, `number → int|float|null`, `image|video → null|int|string`, `checkboxes → array`, `gradient → null|array`, `buttons → array|string`, everything else → `string`.
  - **Select options honoured.** Every `select` field's `default` must be in the declared `options` (loose-compared against stringified keys to handle PHP's integer-key coercion).
  - **No orphan defaults.** Every key in `defaults()` must appear in `schema()` — complements the inverse check in `WidgetDefinition::validate()`.
- Current widget set passes all assertions cleanly; no pre-existing drift surfaced.
- Documented in `docs/widget-system.md` under the existing "CI validation" heading.

### Phase 3 — Hero widget link-colour override fix

- Root cause: the Nav widget sets its own `--nav-link-color` inline on the `<nav>` element when `link_color` is configured on the Nav widget. That DOM position is a closer ancestor than `.site-nav-wrapper`, so the hero's `nav_link_color` (plumbed through `__navOverlayLinkColor` → `.site-nav-wrapper` inline `style=""`) was being shadowed rather than winning.
- Fix: use a distinct CSS custom property for the hero override so the two don't collide.
  - `resources/views/layouts/public.blade.php`: hero override now writes `--overlay-nav-link-color` / `--overlay-nav-hover-color` to `.site-nav-wrapper`.
  - `resources/scss/_layout.scss`: the `.site-nav-wrapper--overlay nav a` rule now reads `var(--overlay-nav-link-color, var(--nav-link-color, inherit))`, with specificity (0,1,2) beating `.widget-nav__link` (0,1,0) regardless of whether the nav widget sets its own link color.
- No `!important`. User confirmed the fix working in the browser.

### Phase 4 — phpunit.xml DB override fix

- Added `force="true"` to every `<env>` tag in `phpunit.xml`. Without it, phpunit only sets vars that aren't already in the process environment, so `.env`'s `DB_DATABASE=nonprofitcrm` leaked through and `RefreshDatabase` was running against the dev DB.
- This caused the dev DB wipe observed during this session. Restored via `migrate:fresh --seed` against `nonprofitcrm`.
- Verified: running `artisan test --filter=WidgetDefaultsControllerTest` leaves the dev DB user count unchanged.

### Phase 5 — Color picker dropdown close

- `resources/js/page-builder-vue/components/primitives/ColorPicker.vue`: `selectColor()` (theme swatches + saved swatches) and `clearColor()` (no-color button) now call `closePopover()` after emitting the new value. The color-wheel and hex-text inputs intentionally do not close on every keystroke — continuous editing would be broken otherwise; click-outside and Escape still close for those inputs.
