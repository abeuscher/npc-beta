# 193. CRM Importer — Commit Progress, Loading States & Large-Import Handling

Close three related UX gaps in the importer's commit-side flow: buttons that do work without showing loading feedback, the Importer landing page rendering slowly after commit, and an intermittent Chrome JS error observed around ~1k rows on the commit → done phase transition.

---

## Design decisions (resolved before starting)

- **Per-button loading indicators, not a page-wide overlay.** The admin already uses per-button loading patterns in other wire:click handlers; continue that convention. Each affected button gets `wire:loading.attr="disabled"` + an inline spinner while its action runs. The user explicitly confirmed this direction.
- **All five importers get the fix in a single pass.** The commit/tick/finaliseSession plumbing is in `InteractsWithImportProgress`; per-page blade overrides are thin. Fix once in the trait + shared pieces, visit all five blades for the button markup.
- **Keep the current same-page architecture.** An earlier scoping sketch suggested redirecting from dry-run to a separate commit-progress page; after inspection that was a misread of the code. The page already transitions through `awaitingDecision` → `committing` → `done` without a redirect, using `wire:poll.500ms="tick"` during the committing phase. The architecture is sound; the polish is missing.
- **ImporterPage is in scope.** The "Go to review queue" action is a plain `<a href>` to `ImporterPage`, which runs several queries per render to compose the review queue and blocked-type indicators. Adding a loading indicator to the link itself (browser-native `a[href]` doesn't have one) requires either a small JS snippet or a switch to a wire:click + redirect. Prefer the lightweight browser-native approach: add a click handler that disables the link and shows a spinner before the navigation completes. Parallel: audit the queries that run on ImporterPage render and see whether any are avoidable or cache-able without architectural change.
- **Investigate the ~1k-row JS error empirically.** User reports the error three times now, around the transition from `committing` to `done`. Reproduce with a 1k-row CSV (synthetic — no real client data in tests), collect the Chrome console payload and the Livewire request that failed, and fix the root cause. Leading candidates: (a) chunk-size tuning so no single tick exceeds the Livewire request timeout; (b) race condition between a late-firing poll and the phase transition re-render; (c) Livewire payload size on the `done` phase re-render. Do not guess — reproduce first.
- **No schema changes.** This session is entirely in the Blade + trait + Filament page layer.

## Phase 1 — Audit existing loading-state patterns

Before writing any new CSS/JS, catalog how the rest of the admin surfaces loading feedback. Grep the codebase for `wire:loading`, `wire:target`, and Filament `->action` patterns to establish a consistent primitive. Pick the cleanest pattern and document it briefly in the session log. Goal: one consistent loading indicator used everywhere in this session, not one per surface.

---

## Phase 2 — Per-button loading indicators on the progress pages

For every `wire:click` that does meaningful work on a progress page, add a visible busy state:

- **Commit button** (`runCommit`) in each of the five progress blades — disabled during request, inline spinner swapped in for the button icon, button text optionally changed ("Committing…").
- **Cancel button** (`cancel`) — same pattern. It's a short action but still worth consistency.
- **Save Mapping button** (`saveMapping`) — same pattern, in the `done` phase markup of each progress blade.
- **Approve / Reject buttons** on `ImporterPage`'s review queue (they also do meaningful work and currently have no loading state).
- **Error-report download buttons** (`downloadPiiErrors`, `downloadErrors`) — same pattern where they exist.

Factor the spinner markup into a small Blade partial (e.g. `components/loading-spinner.blade.php` or an inline `@include` fragment) so each call site is a one-liner. Do not invent custom CSS — use existing Tailwind utilities and the existing primary-500 colour.

---

## Phase 3 — Navigation loading indicators

"Go to review queue" and similar plain `<a href>` links: add a tiny Alpine handler (or a shared Blade partial) that:

- On click, disables further clicks,
- Shows an inline spinner next to the link text,
- Lets the browser navigation proceed normally.

This is purely a UX affordance — no architectural change. Apply to every action-style link where the destination is a heavy page: "Go to review queue" from the progress page's `done` phase, and any equivalent link out of the wizard's final step.

---

## Phase 4 — ImporterPage render performance

Review the queries that run when `ImporterPage` renders. Expected finds:

- Per-content-type "blocked" checks (pending/reviewing ImportSessions for each model_type).
- Review-queue table: ImportSessions with their counts of imported vs staged records.

For each, decide whether:
- The query can be cheaper (e.g. aggregate in one pass rather than one-per-type),
- Or deferred (rendered empty on first paint and filled in via a second Livewire tick),
- Or left alone because it's already fine.

Do not rebuild the page architecturally. Goal: the page renders in a reasonable time for a CRM with hundreds of sessions in history. If a specific query is the long pole, fix that and move on.

---

## Phase 5 — Reproduce and fix the ~1k-row JS error

Generate a synthetic 1 000-row contacts CSV (any plausible schema will do — first name + email + a couple of custom columns). Run the import end-to-end with Chrome DevTools open and capture:

- Which Livewire request fails (URL, payload size, duration).
- The exact JS error text and where it's thrown.
- Whether the page recovers, gets stuck in `committing`, or transitions to `done` despite the error.

Then pick the fix based on the finding. Likely candidates, in decreasing order of probability:

1. **Chunk too large for the final tick.** The final tick includes `finaliseSession` and transitions `phase` to `done`, which also triggers a full blade re-render of the heavier `done` markup. Livewire's response includes the whole DOM diff. Lowering the chunk size (or splitting the final tick into "process last chunk" and "finalize" ticks) may resolve it.
2. **Poll race.** A poll fires before the previous one has returned, and the two step on each other at the phase-transition boundary. Adding a `wire:poll.keep-alive` or guard flag on `tick` may fix it.
3. **Payload size on `done` render.** The `done` phase shows counts and source-name info, all from existing state. Unlikely but confirmable via DevTools.

Document the cause and fix in the session log. Regression test: a Pest test that commits a 1 000-row CSV end-to-end and confirms it reaches `done` without error. Put this in a `->group('slow')` — it's a full commit flow with real DB writes.

## Out of scope

- Restructuring the commit flow into a separate URL / page. The same-page phase-transition architecture is working; polish around it is the scope.
- Rate-limiting or queueing commits. If per-row work becomes slow enough that even chunked tick can't keep up, that's a post-Beta problem — address via Laravel queue, not via inline Livewire changes.
- Rewriting ImporterPage's review queue UI. Query tuning only — no visual rework.
- Dry-run phase polish. The user reports dry-run is fine; don't touch it.
