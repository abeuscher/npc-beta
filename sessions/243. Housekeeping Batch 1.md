- **Order of phases is ascending risk / scope** so the operator can pause and ship after any phase if priorities shift. Phase 1 is a one-line change; Phase 6 is the largest (theme SCSS).

Resolve at session time per the schema docs and the existing CMS-settings storage pattern; if a migration lands, update `docs/schema/` to reflect the final column state.

**Acceptance:**

- Editing an existing Memo does not silently update the date.
- Pest passes; +1 case net.

## Phase 2 — Backup pipeline tweaks

Two small `config/backup.php` adjustments surfaced during the 242 manual restore drill.

**Files touched:**

- `config/backup.php`
- `docs/app-reference.md` (Backups section, Restoration step 5)

**Implementation:**

- (a) **Compression.** Set `database_dump_compressor` to `\Spatie\DbDumper\Compressors\GzipCompressor::class` (currently `null`). The dump file extension automatically changes — Spatie appends `.gz` when a compressor is set. The restore command becomes `gunzip -c <dump>.sql.gz | psql ...` instead of `psql -f <dump>.sql`. Update the Restoration doc accordingly.
- (b) **Relative path.** Set `relative_path` (or whichever key the published config exposes — confirm at session time; may be `relative_path` under `source.files` or a top-level key in newer spatie versions) to `base_path()`. Today the key is implicitly `null`, which causes Spatie to store media at the absolute container path `var/www/html/storage/app/public/...` inside the zip. With `base_path()`, media stores at `storage/app/public/...` — flatter, easier to manually `cp -r` during restoration.
- Update `docs/app-reference.md` Backups → Restoration step 5: change `cp -r /tmp/restore/var/www/html/storage/app/public/* /var/www/html/storage/app/public/` to `cp -r /tmp/restore/storage/app/public/* /var/www/html/storage/app/public/`. Also update step 4 to reference the `.sql.gz` extension and the `gunzip` pipe.

**Verification (manual, in-flight):**

- Run `docker compose exec app php artisan backup:run` once after the config changes. Inspect the resulting zip:
  - `unzip -l <new>.zip | grep db-dumps` should show `db-dumps/postgresql-nonprofitcrm.sql.gz` (note the `.gz`).
  - The media tree paths inside the zip should now read `storage/app/public/...` not `var/www/html/storage/app/public/...`.
  - Total zip size should be smaller than the corresponding 242 baseline (the 9 MB clean-state zip should drop a bit; the 99 MB admin-data zip should drop more).

**Test:**

- No automated test required. Both changes are config-only; verification is the manual inspection above.

**Acceptance:**

- Backup zips contain a gzipped SQL dump.
- Backup zips store media at `storage/app/public/...` (relative).
- The Restoration doc's commands match the new shape.
- An end-to-end restore drill against the new zip (extract → `gunzip -c | psql` → `cp -r`) succeeds — same recipe as 242 manual testing, adjusted for the new file extension and path shape.

## Phase 3 — SEO/AEO additions to CMS settings

Three small items bundled because they all live in the same CMS-settings + SEO surface.

**Files touched:**

- The settings storage layer (a column on a settings table OR a key in a JSON-bag settings entry — confirm the existing CMS-settings storage pattern at session time and follow it).

- A migration if the noindex toggle warrants a column add; `docs/schema/` update if so.

**Implementation:**

**(a) Global noindex/nofollow toggle.**

- Add a `noindex_global` boolean field to the CMS settings storage. Default `false`. Distinct from the per-page noindex delivered in session 096 — this one is site-wide.
- Add a Filament `Toggle` (or `Checkbox`) field to `CmsSettingsPage` with a clear help label: "Block search engines from indexing this site (use during staging or pre-launch)."
- In the public layout, after the existing per-page noindex check, add a global override: if `noindex_global` is true, emit `<meta name="robots" content="noindex,nofollow">`. If both per-page and global are evaluated, the global takes precedence (or the most-restrictive wins — pick one; "most restrictive" is the safer default).

**(b) Sitemap audit.**

- Read `SitemapController` and confirm it includes the current page + post catalog. Hit `/sitemap.xml` against the current dev install and verify the output contains the seeded pages plus any user-created pages.

**(c) AEO file at `/llms.txt`.**

- Add a route + controller following the `SitemapController` / `RobotsController` shape. Returns `text/plain` content.

- The content is generated dynamically from CMS settings + the site's published-page catalog at request time, not a static file.

**Acceptance:**

- Public-page view-source includes the meta tag when toggle is on, excludes it when off.
- `/sitemap.xml` returns the expected page+post catalog.

- Pest passes; +1 to +2 cases net.

- Possibly `app/Services/AppearanceStyleComposer.php` if alignment is supposed to be a wrapper-level concern.

**Implementation:**

- Audit step: identify which widgets expose horizontal alignment in their inspector. Hero is the working baseline.
- Compare hero's template with a broken widget's template. Find the difference — likely Hero's template applies a `text-align` class or inline style to the wrapper while other widgets apply it to the inner element (or don't apply it at all).
- Fix uniformly: pick the cleanest implementation pattern (hero's, ideally) and apply it to every text-editor-bearing widget. If the AppearanceStyleComposer should be the canonical source, route alignment through it instead of per-widget templates.

**Test:**

- No automated test — this is rendering CSS. Visual verification via the page builder preview and a public-page render.

**Acceptance:**

- Setting horizontal alignment in the inspector of any text-editor-bearing widget produces the expected alignment in both the page-builder preview and the public render.
- The fix is applied at one architectural layer (template wrapper, AppearanceStyleComposer, or shared partial), not duplicated per widget.

## Phase 5 — List rendering + indent fixes in admin Quill editor

Two paired bugs, admin only — public site renders correctly. Track down the offending CSS rule and scope it.

**Files touched:**

- The admin SCSS for the Filament Quill editor — likely `resources/scss/admin.scss` or a Filament-overrides partial (grep for `ql-editor` or `quill` in `resources/scss/`).
- Possibly the Filament editor blade template if the bug is structural rather than CSS.

**Implementation:**

- (a) **Both bullet and number markers showing.** Inspect element on a bullet list and a numbered list in the admin. Find the CSS rule that's setting `list-style-type` on both `<ul>` and `<ol>` (or applying both a `::marker` and a CSS-based bullet). Scope the rule to apply correctly per list type.
- (b) **Markers outside the padded text area.** Likely a CSS reset (`modern-normalize` or a Filament reset) sets `padding-left: 0` on `<ul>` / `<ol>` and forces markers to render outside their parent. Restore the standard list padding (`padding-inline-start: 1.5em` or equivalent) for lists *inside* the Quill editor, scoped to `.ql-editor ul, .ql-editor ol` so the rule doesn't leak into other admin lists.

**Test:**

- No automated test — this is rendering CSS. Visual verification: open a Quill editor (any rich-text field on any admin form), create both a bulleted and numbered list, verify markers render correctly and indents correctly.

**Acceptance:**

- Bullet lists show only bullet markers; numbered lists show only numeric markers.
- Both list types render with markers inside the padded text area, indented from the list's starting position.
- No regression on the public-site render (which already worked correctly).

## Phase 6 — Mobile type scaling

Page text doesn't shrink correctly on narrow viewports — typography stays at desktop sizes when the layout collapses. Add responsive type-scale rules to the theme SCSS.

**Files touched:**

- The public-site theme SCSS — likely `resources/scss/public.scss` or a typography partial it imports (grep for `font-size` declarations on body / heading selectors).
- Possibly `resources/scss/_variables.scss` (or wherever theme variables live) if type-scale tokens make sense as variables.

**Implementation:**

- Pick the scaling approach: `clamp()` for fluid type (preferred — single declaration handles all viewport widths) or breakpoint-based media queries (more explicit but more rules).
- Apply to the major text scales: `body`, `h1`, `h2`, `h3`, `h4`, `h5`, `h6`, lead paragraphs, captions if styled separately. Verify which selectors are actually used by inspecting the public layout and a representative widget render.
- The lower bound of `clamp()` should produce mobile-readable type at 320px viewport width (the standard "smallest realistic phone" baseline). The upper bound is the current desktop size. The middle term scales with `vw`.
- Don't change the appearance at desktop widths — only the small-viewport rendering. Verify by comparing before/after at 1280px viewport.

**Test:**

- No automated test — this is responsive CSS. Visual verification via DevTools device emulation across 320px, 480px, 768px, 1024px, 1280px viewport widths.

**Acceptance:**

- Headings and body type scale down at narrow viewports without manual user-zoom.
- Desktop appearance unchanged at 1024px+.
- Type at 320px viewport is readable (not awkwardly small, not awkwardly large).

If the fix needs more than `clamp()` rules and a couple of breakpoint media queries — for example, if it surfaces deeper theme-architecture concerns about how type tokens flow from theme settings into output CSS — **stop and lift it into its own stub**. Don't let Phase 6 absorb a Theme Colors Refactor or Site Theme Builder scope creep.
