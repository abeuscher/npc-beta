## What was built

### Phase 1 — `Slot` abstract class and `SlotRegistry`

- `app/WidgetPrimitive/Slot.php` — abstract base declaring `handle()`, `label()`, `layoutConstraints()`, `configSurface()`. `ambientContext()` is intentionally **not** declared on the abstract — per-slot signatures vary (the canvas slot takes `PageContext` + `?Page`, the dashboard and record-detail slots take different shapes entirely) and PHP 8's LSP enforcement rejects compatible-signature overrides across those variants. Class-level docblock documents the convention; each slot declares its own concrete `ambientContext(...)` signature.
- `app/WidgetPrimitive/SlotRegistry.php` — singleton service mirroring `WidgetRegistry`'s shape exactly: `register(Slot)`, `find(string): ?Slot`, `all(): array<string, Slot>`. Duplicate-handle registration follows last-wins semantics (same as `WidgetRegistry`, via associative-array assignment).

### Phase 2 — The three slot declarations

- `app/WidgetPrimitive/Slots/PageBuilderCanvasSlot.php` — live `ambientContext(PageContext $pageContext, ?Page $currentPageOverride = null): SlotContext` returning the exact `SlotContext` shape `WidgetRenderer` previously constructed inline. Layout constraints are permissive (`allowed_appearance_fields: '*'`, `dimensions: null`, `column_stackable: true`, `full_width_allowed: true`). `configSurface: 'page_builder_vue'`.
- `app/WidgetPrimitive/Slots/DashboardGridSlot.php` — `ambientContext(): object` body throws `RuntimeException('Slot ambient context not yet wired — lands with Phase 3')`. Layout constraints: grid-cell dimensions (`['width' => 'int', 'height' => 'int']`), bounded appearance fields (`['background', 'text']`), no full-width, not column-stackable. `configSurface: null`.
- `app/WidgetPrimitive/Slots/RecordDetailSidebarSlot.php` — `ambientContext(): object` body throws `RuntimeException('Slot ambient context not yet wired — lands with Phase 5')`. Layout constraints: compact, column-stackable, bounded appearance, no full-width. `configSurface: null`.
- All three registered in `WidgetServiceProvider::boot()` alongside the existing `WidgetRegistry` registrations.

### Phase 3 — `WidgetDefinition::allowedSlots()` default method

- New base method `public function allowedSlots(): array { return ['page_builder_canvas']; }`. Pattern-parallel to other low-churn defaults on `WidgetDefinition` (`collections()`, `assets()`, `fullWidth()`, `allowedPageTypes()`).
- No widget definitions overridden — the default preserves every migrated widget's current compatibility unchanged.

### Phase 4 — `WidgetRenderer` retrofit

- Single substitution in `app/Services/WidgetRenderer.php` (line 74): `new SlotContext($pageContext, $tokenPage)` replaced with `app(SlotRegistry::class)->find('page_builder_canvas')->ambientContext($pageContext, $tokenPage)`. Byte-equivalent output — the canvas slot's `ambientContext` body is `return new SlotContext($pageContext, $currentPageOverride);`, so the retrofit routes through the slot but produces the same `SlotContext` instance shape.
- Import statement swapped: `use App\WidgetPrimitive\SlotContext;` → `use App\WidgetPrimitive\SlotRegistry;`. `WidgetRenderer` no longer references `SlotContext` directly.
- `$tokenPage` computation stays in the renderer (it uses `$pageContext->currentPage` and `$pw->owner` — caller-side concerns). The slot only receives the resolved page override.

### Phase 5 — `SlotRegistry` singleton binding

- `WidgetServiceProvider::register()` gained `$this->app->singleton(SlotRegistry::class, fn () => new SlotRegistry());` alongside `WidgetRegistry`, `WidgetConfigResolver`, `PageContextTokens`.
- Three slot instances registered in `boot()` before the widget-registry block (one-time per request; singletons share the registry across every `WidgetRenderer::render()` call).

### Out-of-session admin logo fix (user-requested)

- `app/Providers/Filament/AdminPanelProvider.php` — the admin brand `<img>` tag was rendering as `<img src="http://localhost/storage/">` (broken image) when `admin_logo_path` was stored as `null` or whitespace, because the existing ternary treated only the literal empty string as "unset." Gated emission of the `<img>` tag on a new `$hasLogo = is_string($logoPath) && trim($logoPath) !== ''` check; the asset-fallback branch of the `$logoSrc` ternary was unreachable under the new gate and was removed. Pure PHP change, no build. Not in the Phase 2 mandate — pulled in as a small quality fix while the session was open.

## Deferred decisions

- **Typed `SlotContext` subtypes per slot** (e.g. `PageAmbientContext`, `DashboardAmbientContext`, `RecordDetailAmbientContext`). Deferred to **Phase 5** per `sessions/widget-primitive-migration.md`. Session 212 (Phase 3a) will exercise the dashboard slot with a minimum-viable null-page `SlotContext` — intentionally under-building to let shape problems surface organically before committing to subtypes.
- **Runtime layout-constraint enforcement.** Each slot's `layoutConstraints()` returns declarative metadata only; nothing consumes it yet. Enforcement is deferred, likely bundled with the record-detail IDOR / ownership work in Phase 5.
- **IDOR / context-ownership in the resolver.** Unchanged from 210's log — still deferred to Phase 5 (record-detail slot + `SlotContext` generalisation). Phase 2 did not touch it; Phase 3a won't either.
- **Full-suite Pest memory pressure** (`Spatie\Image\Drivers\Gd\GdDriver::imagewebp` OOM inside `SeoTest` mid full-suite run). Pre-existing from session 210; `SeoTest` runs clean in isolation (22 passed). Widget-adjacent surface fully green. Flagged for the "Test Suite Audit — Cost, Coverage, and Shape" stub when that session lands.
- **`firstSampleLogo()` test-helper ordering bug** (`ContentImportExportTest.php:160` calls the function before `Session156Test.php` declares it alphabetically later). **Pre-existing on `main`** — verified by stashing session work and running the same test against origin/main, same 1-failed / 16-passed result. Not a regression from Phase 2. Cheap fix whenever the next cleanup pass touches that file.
- **Phase 3 sequence, revised.** During 211's close-gate scoping, the previously-planned sequence (212 = dashboard plumbing, 213 = dashboard-native widgets, 214 = config UI) was **re-ordered** to insert a write-side paradigm session. New shape: 212 = plumbing + hosted render (existing widgets), 213 = `DataSink` + `HasSourcePolicy` primitive, 214 = dashboard-native widgets consuming the sink, 215+ = storage model and config UI. 213's prompt was fully drafted in this session ahead of time. Rationale: data generation was a recurring "exception to the paradigm" and folding it in as a first-class source with fail-closed per-target policies is more load-bearing than any individual dashboard widget.
