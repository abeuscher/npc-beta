## The model (confirm in one real round-trip first)

The demo node resets on a schedule by restoring a curated baseline blob:

- **FM (other repo, parallel):** provisions the node, generates its `.env` (`APP_ENV=demo`, `base_url`, unpinned `IMAGE_TAG`), writes the reset cron, and **pushes** the baseline blob onto the node through its provisioning channel. The blob is *pushed*, never pulled — the node needs no outbound egress, keeping the demo egress firewall (deny outbound except DNS/NTP/GHCR, block SMTP) as tight as possible.
- **CRM (this session):** a **`demo:restore`** artisan command that the reset cron calls — restores the pushed blob (database + media), then sets env-specific values (`base_url`, etc.) from the node's own `.env` so the authoring environment's values never leak onto the node. **Hard-gated to `isDemoMode()`** at the top of the command: it refuses to run anywhere else, because a restore-from-arbitrary-blob primitive must never touch a real customer node.

The owner already backs up and restores blobs to nodes and reports the transport is well-tested — this session is the **wiring** (the demo-gated command + the env fix-up) plus **proving the round-trip is faithful**.

## The fidelity problem (the load-bearing precondition)

The whole model rests on the demo backup faithfully round-tripping the demo's own content. The owner has flagged that **background images don't carry** on the current "site export," which breaks a page that uses one — and that a handful of **general settings** have to be re-entered by hand every rebuild, because they're not in the site files: **site name, site URL (`base_url`, defaults to `http://localhost`), the backend logo, the dashboard welcome message** (all `SiteSetting` rows; the logo is a file on the public disk referenced by `admin_logo_path`).

The first task is to **establish which export path the demo actually uses** and round-trip it once:

- If the demo is delivered via a **full spatie backup** (database + `storage/app/public` + media), then the `SiteSetting`s, the logo file, and the background-image media all carry **for free** — a full backup copies the DB and files wholesale, so the background-image bug (likely in the *selective* session-303 portability-bundle path) may not apply at all. This is the expected, simplest outcome.
- Only fix what genuinely breaks the **full-backup** round-trip. Do not build a settings import/export feature — the DB path (full backup) was chosen precisely so settings carry without one; a real settings export stays a post-Beta feature candidate.

Confirm the logo **file** rides along (the setting row alone is not enough — the file under `storage/app/public/site/` must be in the backup) and that background images (Spatie `appearance_background_image` media) restore intact.
