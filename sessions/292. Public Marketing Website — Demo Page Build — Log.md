### Phase 2 — Form-less auto-login wiring

- `app/Http/Controllers/DemoLoginController.php` — invokable. `abort_unless(isDemoMode(), 404)` first; `User::firstOrCreate(['email' => 'demo@demo.local'], …)` (create-or-reuse, one shared row); re-activates if needed; `syncRoles(['demo'])` every entry (re-pins to exactly the demo role — cannot accumulate privilege); `Auth::login()`; redirect to `filament()->getPanel('admin')->getUrl()` (mirrors the existing `InvitationController` path — no new auth path invented).
- `routes/web.php` — `GET /demo/enter` → `DemoLoginController`, name `demo.enter`, `middleware('throttle:10,1')`, registered before the slug catch-all.
- **Shared-account role/permission shape.** New `demo` role in `PermissionSeeder` (label "Demo"). Granted product-feel surfaces: full CMS (page/post/form/collection/collection_item/navigation_menu/product) + full CRM (contact/organization/household/membership/note/tag/mailing_list) + read-only finance (donation/transaction/fund/campaign) + read-only event + `view_any_member` / `use_advanced_list_filters` / `view_any_form_submission` / `view_form_submission`. **Binding deny-list — never granted:** all `user` resource perms (user/role management), `manage_mail_settings`, `manage_email_templates`, `manage_financial_settings`, `edit_theme_scss`, `manage_routing_prefixes`, `manage_cms_settings`, widget_type management, and anything secret/integration/instance-config/real-data-exfiltration. **Never `super_admin`** (which would bypass every policy via `Gate::before`).
- **Shared "Demo User" account.** Seeded in `DatabaseSeeder` **only under `isDemoMode()`** — a production DB never carries a public auto-login account even latently; the controller's `firstOrCreate` is the runtime create-or-reuse safety net.
- **Rate limit:** `throttle:10,1` — 10 requests/min/IP, the established sensitive-route limit (matches the form-submission and event-registration routes). A real visitor needs one request; deeper abuse is bounded by the 24h wipe per the architecture, not by gating.
- **Demo-server guard:** `abort_unless(isDemoMode(), 404)` — `isDemoMode()` keys on `app()->environment('demo')`, the same signal `AppServiceProvider` / `GeneralSettingsPage` already gate on. Inert (404, no auth, no row) on any non-demo install.
- `docs/security-demo.md` — threat model pre-registered the way `docs/security-forms.md` did for notifications: what's protected (demo-server guard, scoped role, per-IP throttle, shared-session/wipe model), known limitations + accepted risks (shared-session cross-visitor visibility, proxy-IP throttle caveat, no-CAPTCHA, distributed exhaustion), the Fleet-Manager cross-session arc explicitly out of scope, and a 6-row Finding register. Disposition-tagged for the pre-GA audit.
- `VERSION` bumped `0.291.1` → `0.292.1` (291 pseudo-versioning discipline; mechanically enforced by the deploy pipeline).
- Pest: `tests/Feature/DemoLoginTest.php` — 6 cases: demo.json imports with zero warnings + publishes the split; demo-server auto-login creates/authenticates the shared user and redirects into admin; second visit reuses the one row; per-IP throttle returns 429 past the window; off-demo route is inert (404, guest, no row); the `demo` role grants product-feel surfaces but denies the settings/secrets/user-management abilities and is never `super_admin`.

## Security considerations

This session's load-bearing surface is a public, unauthenticated route that authenticates an anonymous visitor into the admin panel. Controls reviewed and in place: (1) demo-server-only — `abort_unless(isDemoMode())` plus demo-only seeding, so production never exposes or even stores the account (Pest-asserted); (2) tightly-scoped dedicated `demo` role, never `super_admin`, with a binding deny-list on settings/secrets/integration/user-management/destructive surfaces (representative denied abilities Pest-asserted); (3) `syncRoles(['demo'])` on every entry so the shared row cannot accumulate privilege; (4) per-IP `throttle:10,1`; (5) shared-session cross-visitor exposure accepted by design, bounded by the 24h wipe + demo-data-only scope per `[[demo-page-architecture]]`. Threat model pre-registered in `docs/security-demo.md`. The Fleet-Manager abuse-alerting / wipe-coordination arc is explicitly out of scope (cross-session). No real-PII surface, no instance-config mutation, no secret reachable by the demo role.
