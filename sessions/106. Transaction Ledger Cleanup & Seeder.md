# 106. Transaction Ledger Cleanup & Seeder

Clean up the manual transaction form so its fields match reality, lock down fields that should not be user-editable, and add a transaction seeder so the system has representative test data for QB sync, reporting, and UI testing.

---

## Design decisions (resolved before starting)

- **Manual transactions are off-system entries only.** Stripe-originated transactions are already gated off from edit/delete. The form should make it clear that manual entry is for grants, expenses, adjustments — not payments or refunds, which are system-managed.
- **Direction is derived from type, not a separate field.** Grants and adjustments are `in`. Expenses are `out`. Users should not choose direction independently.
- **QuickBooks reference is read-only.** The `quickbooks_id` field is set by the sync job. Users should never manually enter or edit it. Show it as a placeholder when present, hide it on create.
- **Status options for manual transactions:** `pending` and `cleared`. Stripe statuses (`completed`, `failed`, `refunded`) should not appear as options in the manual form.
- **Type dropdown must not break on edit.** If a record has a type not in the manual options list (e.g. `payment`, `refund` from Stripe), the edit button is already hidden. But as a safety measure, the form should handle unknown types gracefully — display the stored value as disabled text rather than a blank dropdown.
- **Contact selector on manual transactions.** Manual transactions should optionally allow linking to a contact. This enables QB customer matching for off-system entries like grant cheques from a known donor.
- **Seeder creates a representative mix.** The seeder should create transactions that cover the scenarios needed for testing: donations (Stripe-originated), manual grants, refunds, transactions with and without contacts, transactions with and without QB sync. It should use existing contacts from the contact factory and create realistic subject records (donations, purchases) where needed.

## Phase 1 — Manual Transaction Form Cleanup

### Type field

Replace the current type select with options that reflect manual entry use cases:

| Type value | Label | Direction (auto-set) |
|---|---|---|
| `grant` | Grant | `in` |
| `expense` | Expense | `out` |
| `adjustment` | Adjustment | `in` |

### Direction field

Remove the direction dropdown from the form. Set direction automatically based on the selected type using `afterStateUpdated` on the type field.

### Contact selector

Add an optional contact select field (searchable, nullable) so manual transactions can be linked to a contact for QB customer matching and reporting.

### QuickBooks reference

- On **create**: hide the field entirely.
- On **edit**: show as a read-only placeholder if `quickbooks_id` is set. Hide if null.

### Status field

Keep as-is: `pending` and `cleared`. These are correct for manual transactions.

### Safety on edit with unknown types

If the record's `type` is not in the manual options list (defensive case — edit is already hidden for Stripe records), display the type as a disabled text input showing the stored value instead of a blank select.

## Phase 2 — Transaction Seeder

Create `database/seeders/TransactionSeeder.php` and add it to the `DatabaseSeeder` call chain.

The seeder should create a representative set of transactions that exercises the full range of the system. All data should be PII-free — use contacts created by the contact factory.

### Seeder records to create

1. **Contacts** — create 8–10 contacts via factory. Ensure a mix: some with email, some without, some with `quickbooks_customer_id` already set.
2. **Donations** — create 4–6 donation records linked to contacts. Mix of one-off and recurring, various amounts.
3. **Purchases** — create 2–3 purchase records linked to contacts and products (create minimal product records if needed).
4. **Transactions from donations** — create completed `in` transactions with `subject_type = Donation`, linked to contacts. Some with `quickbooks_id` set (simulating already-synced), some without.
5. **Transactions from purchases** — create completed `in` transactions with `subject_type = Purchase`, linked to contacts.
6. **Refund transactions** — create 1–2 `out` / `refund` transactions with `stripe_id` set.
7. **Manual transactions** — create 2–3 manual transactions: a grant (`in`), an expense (`out`), an adjustment (`in`). Some linked to contacts, some not.
8. **Error state transaction** — create 1 transaction with `qb_sync_error` set and no `quickbooks_id`, to test the error badge display.

The seeder should be idempotent — safe to run on an existing database without creating duplicates. Use a sentinel check (e.g. skip if transactions already exist).
