# 275. Rich-Text Surface Sanitization Hardening

Implement the standing 250-time stub. Today every rich-text surface in the app stores Quill output (or Trix output, in the Memos collection case) verbatim and renders via `{!! !!}`. `ContentImporter::sanitizeWidgetConfig` does collection-handle remap + media-id clearing but **not** HTML sanitization. Quill v2 strips `<script>` tags client-side but that's a UX nicety, not a server-side guarantee — direct DB writes, importer payloads, and any future bypass of the editor would land unsanitized HTML. This session closes that gap end-to-end.

This session is the carve-out from Flag W4c/A in the 271+272 audit, ratified at 273-close per the user's "accept all recommendations" direction. 207 is the procedural precedent (a single-concern carved-out session emerging from a code-review apply half).

---

## Design decisions (resolved before starting)

- **One sanitiser, one allow-list, no Composer dep.** `App\Support\HtmlSanitizer` ships as a single PHP class implementing a Quill-shape allow-list directly via `DOMDocument` + per-element walking. No `ezyang/htmlpurifier` or similar — the allow-list is small and tightly defined; a vendored library carries surface area we don't need.
- **Allow-list scope follows the 250-time stub verbatim:**
  - **Tags:** `p`, `br`, `strong`, `em`, `u`, `s`, `ol`, `ul`, `li`, `blockquote`, `pre`, `code`, `h1`, `h2`, `h3`, `h4`, `h5`, `h6`, `a`, `span`, plus the heroicon-shape `<svg>` children.
  - **Class attribute:** restricted to `ql-*` regex (covers `ql-align-{center,right,justify}`, `ql-indent-N`, `ql-direction-rtl`, `ql-size-*`, `ql-font-*`, `ql-heroicon`).
  - **Other attributes:** `data-heroicon` on `span.ql-heroicon`. `href` on `a` allow-listed to `http(s)://`, `mailto:`, or relative (no `javascript:`, no `data:`). `title` on `a` allowed.
  - **Stripped on every element:** all `on*` event handlers, `style` attribute, `id`, `name`, anything not in the allow-list above.
  - **Element disposition:** disallowed tags drop the tag but keep the inner text content (`<script>alert(1)</script>` → empty string; `<div>hello</div>` → `hello`).
- **Apply on save at every model boundary that stores rich-text HTML.** The eight surfaces from the 272 W4c inventory:
  1. `PageWidget.config` — rich-text-typed schema fields (sanitise during the existing `WidgetConfigResolver` save path or a new model-level mutator)
  2. `Collection.data.*` rich_text fields (post-Memos-convergence: all paths Quill-shape)
  3. `Note.body`
  4. `Event.description` + `Event.meeting_details`
  5. `EmailTemplate.body`
  6. `SiteSetting` rich-text keys: `dashboard_welcome`, `system_page_content_reset_password`, `system_page_content_email_verify`
  7. Any model's `custom_fields` jsonb where the matching `CustomFieldDef` has `field_type='rich_text'`
  8. (Implicitly) any future rich-text surface — the sanitiser becomes the single seam every save path runs through
- **Apply at the import boundary too.** `ContentImporter::sanitizeWidgetConfig` walks each widget's schema and re-sanitises any `richtext`-typed field. Each `Import*ProgressPage` (contacts/events/etc.) sanitises any `custom_fields` `rich_text` cell at row-process time — same boundary that today writes the value via `processOneRow`.
- **Memos Trix→Quill convergence ships in the same session.** Filament's `RichEditor` (Trix) at `Collection.php:getFormSchema()` swaps to `App\Forms\Components\QuillEditor`, converging on the dominant convention. One-time data migration converts existing Trix-shape values (typically wrapped in `<div>` blocks with Trix-specific classes) to Quill-shape (`<p>` blocks with `ql-*` classes where applicable). Migration is row-level on the `collection_items.data` jsonb — operates only on rows where the parent `Collection` has a `rich_text`-typed key. Idempotent: re-running the migration on already-converted data is a no-op.
- **Pre-beta license applies.** No deprecation period for the Trix shape. The data migration runs once on `php artisan migrate`; the next squash (T1 terminal) absorbs it. Operators with existing data deploy the migration once and the old shape disappears.
- **The allow-list test suite is the security gate.** No surface is "applied" until its companion regression-guard test passes. The test surface covers: every Quill default-toolbar shape round-trips clean, every disallowed tag/attribute is stripped, representative XSS payloads (script tags, `javascript:` href, `on*` handlers, embedded SVG with script, base64 data: URLs, HTML-entity-encoded payloads) are neutralised.
- **Iteration strategy.** Five iterations on `session-275/1`:
  - `/1` — `App\Support\HtmlSanitizer` utility + allow-list test suite (the security gate ships first; subsequent iterations apply it)
  - `/2` — Apply at the seven save boundaries (one commit per boundary, each with its companion regression-guard test); except Memos which lands at `/3`
  - `/3` — Memos Trix→Quill convergence + one-time data migration + the matching boundary application + Playwright spec for the Memos edit flow
  - `/4` — `ContentImporter::sanitizeWidgetConfig` extension + per-importer `custom_fields` sanitisation + import-boundary regression tests
  - `/5` — Final verification + manual smoke test announcement

## Phase 1 — `App\Support\HtmlSanitizer` utility + allow-list test suite (`session-275/1`)

Build the sanitiser first, with comprehensive test coverage, before applying it anywhere. This is the security gate — no save path uses it until the test surface confirms it behaves correctly.

### Implementation shape

`app/Support/HtmlSanitizer.php` exposes a single static method:

```php
public static function sanitize(string $html): string
```

Internal walker uses `DOMDocument` (available in stock PHP, no Composer dep). Pseudocode:

1. Wrap input in a synthetic `<root>` element so the parser doesn't auto-add `<html><body>` wrappers (use `LIBXML_HTML_NOIMPLIED | LIBXML_HTML_NODEFDTD` flags).
2. Recursively walk every node:
   - Element nodes: if tag is in allow-list, retain; check each attribute — drop any not in the per-tag allow-list; for `class`, filter the list to `ql-*` only; for `href`, validate scheme (`http`/`https`/`mailto`/relative).
   - Element nodes not in allow-list: replace with their concatenated child text content (preserves human-readable text from `<div>`, `<font>`, etc.).
   - Text nodes: pass through unchanged.
   - Comment / processing-instruction / doctype nodes: drop.
3. Serialise back to HTML, strip the synthetic root wrapper, return.

## Phase 2 — Apply at seven model-save boundaries (`session-275/2`)

Each boundary application lands as its own commit with companion regression-guard test. The pattern: apply `HtmlSanitizer::sanitize()` at the model's save path (Eloquent mutator, observer, or controller-level transform — pick the lowest-coupling site that runs on every write). The regression-guard test asserts that an XSS-shaped payload written through that surface gets sanitised at rest.

The seven boundaries (Memos lands at /3 alongside the Trix→Quill swap):

1. **`PageWidget.config` rich-text-typed schema fields.** Add a `setConfigAttribute` mutator on `PageWidget` that walks the attached `WidgetType.config_schema`, finds every field with `type=richtext`, sanitises the corresponding value in `config`, and stores the result. Test: write a PageWidget with `<script>` in a richtext field, assert the stored value is sanitised.
2. **`Note.body`.** Add `setBodyAttribute` mutator on `Note`. Test: same pattern.
3. **`Event.description` + `Event.meeting_details`.** Two mutators on `Event`. Test: same pattern × 2 fields.
4. **`EmailTemplate.body`.** Mutator on `EmailTemplate`. Test: same pattern.
5. **`SiteSetting` rich-text keys.** `dashboard_welcome`, `system_page_content_reset_password`, `system_page_content_email_verify`. The sanitisation runs at the `SiteSetting::set($key, $value)` boundary with a hard-coded list of rich-text keys (the same list that drives the `GeneralSettingsPage` form's QuillEditor renders). Test: write each key with a payload, assert sanitised.
6. **Any model's `custom_fields` jsonb where the matching `CustomFieldDef` has `field_type='rich_text'`.** This is the cross-cutting one — every model that uses `HasCustomFields` (or whatever the trait is called; confirm at impl time) gets a save-path sanitisation that consults `CustomFieldDef::where('model_type', get_class($this))->where('field_type', 'rich_text')->pluck('handle')` and sanitises matching keys in the `custom_fields` array. Test: write a Contact (or any custom-field-using model) with a rich-text custom field carrying a payload, assert sanitised.
7. **Reserved for any surface that surfaces during impl that the 272 inventory missed** — there shouldn't be any, but the inventory was a snapshot. If a fresh grep for `QuillEditor::make` or `RichEditor::make` (post-Memos-swap) turns up something the inventory missed, add the boundary here.

Run fast Pest after each commit. Run `npm run build` only if any of the boundaries surface a Vue/SCSS change (unlikely — most are model-mutator changes).

## Phase 3 — Memos Trix→Quill convergence + data migration (`session-275/3`)

The Memos collection is the only Trix surface left in the app. Two parts:

### 3a. Editor swap

In `app/Models/Collection.php::getFormSchema()` (around line 126), replace `Forms\Components\RichEditor::make(...)` with `App\Forms\Components\QuillEditor::make(...)` for any field with `type=rich_text`. Match the field config (label, helper, required, etc.) verbatim. Confirm via `php artisan tinker` or a small Pest test that the Memos collection's form schema now declares Quill instead of Trix.

### 3b. Data migration

New migration: `2026_MM_DD_HHMMSS_convert_memos_collection_items_trix_to_quill.php`. The `up()` method:

1. Find every `Collection` with at least one schema field of `type=rich_text`.
2. For each such collection, find every `CollectionItem` belonging to it.
3. For each item's `data` jsonb: walk every key whose corresponding schema field is `rich_text`-typed. Convert the stored Trix HTML to Quill HTML using a small `App\Support\TrixToQuillConverter::convert(string $trixHtml): string` helper.
4. Write the converted `data` back; sanitise via the new `HtmlSanitizer` on the way (defence-in-depth — every save path including the migration runs through the sanitiser).

Trix → Quill conversion mapping (the dominant differences in stored shape):

- Trix wraps blocks in `<div>` with no class; Quill wraps in `<p>` with optional `ql-*` class. Convert top-level `<div>` → `<p>` unless the div carries a class indicating a different block type.
- Trix uses `<strong>` and `<em>` for bold/italic — Quill same. No conversion.
- Trix uses `<a>` for links — Quill same. No conversion.
- Trix uses `<ol>` / `<ul>` / `<li>` for lists — Quill same. No conversion.
- Trix uses `<blockquote>` — Quill same. No conversion.
- Trix-specific shapes (`<figure data-trix-attachment=...>` for embeds) drop to plain text or get stripped — confirm during impl whether Memos uses these (probably not; Memos was simple text).

The conversion is best-effort. The data-migration log records any items whose conversion produced a non-trivial diff vs the literal source so an operator can sanity-check. Idempotency: running the migration twice on already-Quill-shaped data is a no-op (the converter detects existing `<p>` blocks and skips).

### 3c. Memos boundary application

Add the `setData` mutator (or whatever the right hook is) on `Collection` / `CollectionItem` so future Memos writes also run through `HtmlSanitizer`. Same pattern as Phase 2 surfaces.

### 3d. Playwright spec

New `tests/e2e/memos/memos-trix-to-quill.spec.ts`:

- Seed a Memo with stored Trix-shape HTML before the spec runs (or rely on the seeder if Memos seeds are still Trix-shape pre-migration).
- Run the migration.
- Open the Memo in admin; confirm the Quill editor renders the converted content correctly.
- Edit the Memo; confirm the saved form-state is sanitised (write a payload with `<script>` and assert it's stripped at rest).

Run `npm run build` after the editor swap (Vue/JS surface may shift). Run `npm run test:e2e` to confirm the Memos spec passes.


Companion regression test: write a Pest test for one importer (Contacts is sufficient — the others use the same trait shape) that drives a CSV with a rich-text custom field carrying `<script>`; assert the imported Contact's `custom_fields` has the script stripped.

## Out of scope

- Net-new rich-text editing capabilities (e.g. embedded media, tables, mentions) — this session locks down the existing surface, doesn't extend it.
- Changing the Quill toolbar configuration — out of scope unless a sanitiser allow-list decision forces a toolbar shape change (unlikely).
- Migrating any other editor surface — Quill is the canonical convention; this session converges Memos onto it. There's no other surface to converge.
- `data:` URI image uploads — Quill's image-upload path uses Media Library + a temp URL flow; `data:` URIs are blocked by the allow-list at the `<img>`-tag level (img isn't in the allow-list at all; images in rich-text content surface as Media Library references, not inline `data:`).
- Sanitising plain-text-typed surfaces (`<input>`-shape custom fields, contact names, etc.) — those don't accept HTML; they're already plain-text-cast at the model level.
- Rich-text rendering hardening (CSP headers, `iframe` sandboxing on previews) — server-side allow-list sanitisation at save is the load-bearing fix; rendering-side defence-in-depth is a separate post-Beta-1 concern.
