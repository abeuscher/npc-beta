# 310. Content Import/Export — Unified Import Site / Export Site UI

> **The rollup.** After session 309 lands the opt-in theme + analyzer infrastructure, this session adds the single-button "Import Site" / "Export Site" affordance that exercises the full envelope: pages + templates + theme + media (and, once session A001 lands, events + products + navigation + collections). This is the architectural-shape session — most of the calls live here, not the implementation. Keep the surface small and the questions answered.

One sentence: one button somewhere in admin chrome, opens a panel, lets the operator export the whole site as a bundle or import a bundle as the whole site, both flows backed by the analyzer + opt-flag plumbing from session 309.

---

## What we want

### Export Site

A single action — **Export Site** — that produces a full-snapshot bundle of the live site state. Includes by default:
- All published + draft pages
- All templates
- The current theme (`design` payload)
- All referenced media (`media` payload, reference-mode by default; embedded-mode as a checkbox)
- Whatever entity coverage session A001 has shipped (events, products, navigation, collections)

The action emits a single `.json` file the operator can store or hand off.

Confirmation is light: a summary of what's in the snapshot (counts) before download. No per-entity opt-out at this surface — that's what the per-entity export forms are for (page export, theme export, etc.). The whole point of Export Site is "the whole thing."

### Import Site

The corresponding action — **Import Site** — accepts a bundle and runs the analyzer pre-flight, surfacing a manifest panel:

- Counts per entity type (pages, templates, theme keys, media files, events, products, collections, navigation).
- Per-entity collision summary: how many existing rows on this site would be touched, by slug / handle / id-map.
- Three checkboxes mirroring session 309's import shape (replace duplicates, replace theme, include media), expanded to whichever entities A001 has added.
- A single **Import** button at the bottom.

This is intentionally NOT a guided merge — it's a confirm-and-run flow. Anything more nuanced lives in the per-entity import forms.

### Surface placement

A new admin page under the existing Filament admin: **Tools → Import / Export Site**. Single page, two sections (Export Site / Import Site). Reuses the per-entity import UI patterns (the bulk-contact importer's two-step shape, the session-309 import UI).

## Architectural calls to make this session

These are the decisions the session needs to resolve — flag them at session start, not at apply time:

1. **Backup-grade or button-grade?** Backup-grade implies the snapshot can fully restore the site to its current state (every relevant row, every referenced file, every site-wide setting beyond `design`). Button-grade implies the bundle is a convenience snapshot — most things are covered, but a fresh-install restore isn't the guarantee. **Recommended default: button-grade for v1**, with a roadmap slot for backup-grade once entity coverage is complete. Backup-grade adds users, roles, audit log, mailing-list-state — out of scope for this session even if technically reachable.
2. **Diff preview yes/no?** Diff preview means the analyzer compares incoming bundle row-by-row against current state and shows which rows would change, with a per-row "include / skip" checkbox. **Recommended default: no — out of scope for this session.** The collision-count summary in the analyzer manifest is sufficient. Diff preview is a future session if the operator workflow demands it.
3. **What's an "entity" for collision purposes?** Pages and templates collide by `slug` / `handle`. Theme collides as a single row (it's three SiteSetting rows but treated as one config). Media collides by `id` (the id-map plumbing handles this already). Events collide by `slug` (or `id` if A001 picks differently — defer to that session's call). Spell this out in the analyzer's manifest shape so the UI knows what counts as a duplicate.
4. **Format version policy.** No bump. Session 309 keeps `format_version: 1.1.0` for additive opt flags; this session keeps it too. The envelope's outer shape doesn't change — only what's inside `payload` grows.

## Architecture

- `ContentExporter::exportSite(array $opts = []): array` — new wrapper. Calls the existing per-entity exporters internally, assembles a full snapshot. `$opts` carries the same `with_design` / `with_media` flags from session 309 (with full-snapshot defaults set to true here).
- `ContentImporter::analyze()` (already exists, session 309) — extended to cover the additional entity types as A001 ships them. No change in this session for entities not yet covered.
- `ContentImporter::import()` — same. The opt flags from session 309 cover the rollup's needs.
- Filament page: `app/Filament/Pages/SiteImportExportPage.php`. Two sections, no widget machinery beyond Filament Form + an action button per section.

## Out of scope

- Backup-grade snapshot (users, roles, audit log, mailing-list state, etc.) — future session.
- Diff preview / per-row include-skip checkboxes — future session if workflow demands.
- Scheduled / automated exports — future session.
- Multi-tenant snapshot or cross-instance migration — future session, possibly Fleet Manager surface.
- Any change to the per-entity import/export forms — those stay as-is, this is an additive rollup surface.
