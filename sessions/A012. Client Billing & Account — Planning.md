# A012. Client Billing & Account Track — Planning (FM-Owned Stripe Billing + Node Account Page)

Design the missing commercial layer of the product — how client organizations see and manage their account (plan, billing contact, invoices), how the owner bills them (variable monthly rates, discounts, hourly project work), how personalized demo instances convert to paying clients or suspend un-deleted at trial end, and how Fleet Manager surfaces delinquency — and decompose it into buildable sessions. **Planning only; the deliverable is a track doc, not code.**

## Settled direction — LOCKED (agreed with the owner; design within these)

1. **Fleet Manager owns billing.** Subscriptions, per-client pricing, discounts, delinquency state, and trial/conversion state are managed centrally in FM, keyed to FM's existing per-client records. No billing engine on the nodes.
2. **Stripe is the risk holder.** One vendor Stripe account, one Stripe Customer per client org. Card data never touches owner infrastructure — Stripe holds cards, runs subscriptions, retries failed payments (Smart Retries / dunning), and issues receipts. Per-client pricing and discounts use Stripe's native per-customer prices and coupons — **do not design a bespoke pricing engine.**
3. **Polling, not webhooks.** FM currently runs on the owner's local machine — outbound-only, deliberately unreachable from the internet, and staying that way for now. FM learns of payment events by polling Stripe's API on its existing polling cadence (billing status fresh-to-the-day is sufficient for delinquency). **The design must not require any inbound endpoint on FM.** Note in the track doc: if FM later moves to a hosted box, polling swaps to webhooks with no other design change.
4. **Restricted Stripe API key, in FM's existing vault.** FM holds a restricted key (customers + subscriptions + invoices scope, nothing more), stored under FM's established off-filesystem vault encryption like the node secrets it already keeps. Never the full-access key.
5. **The node's "My Account" page is a window, not an engine.** It displays billing state (plan, status, next invoice, billing contact) that **FM pushes to the node** over FM's existing provisioning/config channel — the node never calls the vendor Stripe API and holds zero vendor-Stripe credentials. Card updates and receipt history go through **Stripe's hosted customer portal** (email magic-link login), linked from the account page.
6. **Two Stripes, strictly separate.** The app's existing Stripe integration is the *client org's own* account for *their donors* (donations, event tickets). Vendor billing is the *owner's* account. No shared keys, no shared config keys, no shared code paths, no shared webhooks. The plan must name the concrete separation (distinct config namespaces, distinct services) and add a guard against future co-mingling.
7. **Suspension policy (working default — owner may adjust at review):** payment fails → Stripe's retry/dunning window runs its course (~2–3 weeks) → a **grace period** → then **admin access locks but the public site and donation processing stay up** (a nonprofit's donation page never goes dark over the owner's invoice) → full shutoff is a **manual operator decision in FM, never automatic**. Nothing is ever auto-deleted. Suspension and un-suspension are reversible state flips pushed from FM.
8. **Demo → paying conversion rides existing machinery.** FM already provisions demo-profile nodes and pushes config to running nodes. Conversion = attach a Stripe customer + subscription in FM, flip the node's profile out of demo mode, push the config change. Trial-expiry suspension uses the same suspension mechanism as delinquency (different reason code).
9. **Hourly project work bills through the same Stripe customer** as one-off invoice items, so subscription + hours appear on one invoice. Where hours get *logged* (an FM surface vs. directly in Stripe with FM read-only) is for this session to design — present both shapes with a recommendation.
10. **Billing contact is a durable identity** — an email address on the account, deliberately not "whoever is currently super-admin," so it survives admin turnover on the node.
11. **CRM-side permission:** a single "manage account" ability restricted to owner-level admins gates the account page's editable parts (billing contact, portal access). Not a new role system.
12. **Offboarding ties into the existing full-site export.** Cancellation offers the client their data via the export-bundle machinery that already exists. Design the flow, don't rebuild the exporter.
13. **Timing:** this track slots around the multi-node / first-customer work after the Beta-1 core, not ahead of it. Propose concrete slotting; the owner decides.

## FM ground-truth context *(the cloud environment cannot read the FM repo — this section is your FM knowledge; do not invent beyond it)*

Fleet Manager is a separate Laravel + Filament application, single operator (the owner), running locally on the owner's machine (WSL), **48 sessions complete**. It is the other half of the two-repo contract in `docs/fleet-manager-agent-contract.md` (v2.5.0). What exists FM-side that this plan builds on:

- **Client registry:** a `clients` table, one row per provisioned CRM node — display name, host, connection state, `is_demo` (write-once) and `is_public_website` flags, plus **vault-encrypted per-node secrets** (mTLS client key, node `APP_KEY`, DB password, read-only DB credentials). The vault is off-filesystem-key encryption (libsodium), the established home for any new secret (→ the restricted Stripe key).
- **Polling loop:** FM polls every node's `/api/health` on a schedule (mTLS client-cert auth), interprets status, records incidents, sends email alerts. This is the cadence billing polling joins.
- **Node operations, all shipped:** end-to-end droplet provisioning (including a demo profile with an egress firewall, and an adopt-existing-server flow); per-client upgrades with release gates (registry, canary verification, auto-rollback on boot failure); backup trigger + blob download; same-node restore and cross-node restore; log fetching; on-demand config push to a running node (`NodeConfigPush` — sets/updates a single `.env` key over SSH and recreates containers; shipped FM 047 for the public-website flag — **this is the channel billing state and suspension flags ride**).
- **Demo machinery:** demo nodes reset on cron by restoring a curated baseline blob FM pushes onto the node; demo nodes have `APP_ENV=demo` and code-level `isDemoMode()` hard gates CRM-side.
- **Audit ledger:** FM records an audit event for every client-touching action (provision, upgrade, restore, config push...) — billing actions (subscription attach, suspension, conversion) must join this vocabulary.
- **Privacy boundary (standing, load-bearing):** FM is never *built* to read raw node data — counts/metadata only over the wire. Billing data flows FM→node (push), which respects this; nothing in the plan may add a node endpoint that returns client-org record data to FM.
- **Contract discipline:** any new FM↔node surface (e.g. the node receiving/serving billing state, a suspension flag, an account-page data shape) is contract surface → additive version bump (v2.5.0 → v2.6.0), documented in `docs/fleet-manager-agent-contract.md` with CHANGELOG, per the Two-Repo Coordination Protocol. The *build* session authors that bump; this planning session only sketches its shape.
- **FM-side UI grain:** Filament resources/pages; per-client view page with header actions (trigger backup, push baseline, reclaim disk...) — billing affordances (attach subscription, record conversion, suspend/unsuspend, delinquency badge) follow that pattern; a fleet-wide list view already shows per-client status columns (version, demo, public-site) — a billing-status column joins it.

## What this session designs (the wrinkles, made concrete)

1. **The billing architecture** end to end under the locked decisions: FM's Stripe integration (restricted key, polling sync, what FM persists locally vs. reads live), the per-client billing model (price, coupon, trial state, status), and the FM operator UI.
2. **The node account page**: what it shows, where it lives in the admin IA, what data shape FM pushes, how staleness is communicated ("as of this morning"), the customer-portal handoff, and the "manage account" permission.
3. **The suspension lifecycle** as an explicit state machine (active → past-due → grace → admin-locked → manually-off; plus trial → trial-expired) with reason codes, what each state means concretely on the node (which routes/surfaces lock), how FM pushes state flips, and how un-suspension works. Working default per locked decision 7.
4. **Demo conversion flow**, operator-driven in FM: personalize-demo → trial terms → convert (customer + subscription + profile flip + config push) → or expire to suspension. Includes what happens to the demo reset cron on conversion.
5. **Hourly work billing**: two shapes (hours logged in FM and pushed to Stripe as invoice items, vs. hours managed in Stripe directly with FM read-only display), a recommendation, and how either appears on the client's account page.
6. **The contract-surface sketch**: what new FM↔node surface the build needs (likely: billing-state push semantics + the suspension flag + whatever the account page reads), shaped additively for a future v2.6.0 — sketched, not authored.
7. **The two-Stripes separation**, concretely: config namespace, service class boundaries, and a proposed guard (test or convention check) that fails if vendor-billing code ever reaches for the donation-Stripe config or vice versa.
8. **Session decomposition + slotting**: an ordered list of buildable sessions (FM-side and CRM-side, named descriptively, with rough size), which repo each lands in, the dependency order (contract bump session sits where?), and a slotting recommendation relative to Beta 1 and the multi-node/first-customer work.
