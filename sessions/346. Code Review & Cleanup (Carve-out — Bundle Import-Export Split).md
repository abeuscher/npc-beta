## Phases

Suggested order ‚Äî Exporter first (lower risk, near-mechanical), Importer second.

### Phase 1 ‚Äî ContentExporter split
The Exporter is almost entirely `exportX()` + `serializeX()` pairs per entity (Pages, Templates, Events, Products, Navigation, Collections, Media, Design, SiteSettings) plus the shared `serializeWidgetTree`/`serializeWidget`/`serializeLayout`/`serializeWidgetConfigReferences`/`serializeWidgetMedia` cluster (used by pages + templates). Extract per-entity serializers + the shared widget-tree serializer; the orchestrator keeps `exportPages`/`exportTemplates`/`exportSite`/`exportBundle` + `envelope`/`buildManifest`/`attachOptIns`/`collectReferencedMedia`. Round-trip after.

### Phase 2 ‚Äî ContentImporter split
Extract the page/widget hydration engine (`importPage`‚Üí`hydrateWidgets`/`hydrateLayout*`/`hydrateRootWidget*`/`resolveWidgetType`/`sanitizeWidgetConfig`/`rewireWidgetMedia`) ‚Äî the biggest cohesive cluster ‚Äî and the per-entity import+media-rewire pairs (Products/Events/Navigation/Collections/Templates) + Design/SiteSettings/Media. The orchestrator keeps `analyze`/`verifyManifest`/`import`/`validateEnvelope`/opts + the deny-list/author-resolution helpers. Preserve the 345/2 EventRegistration `withoutEvents()` wrap through the move. Round-trip after.
