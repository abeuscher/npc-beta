# 329. Demo Upload Lockdown

Block **new media uploads for the `demo` role** across every surface, so the shared, anonymous, daily-wiped demo account can't introduce arbitrary files onto the public demo domain. Demo keeps CRM CRUD and existing-media reuse; it just can't upload new files.

## Why this is worth doing (and what's already covered)

The demo's other edits are low-risk: spoofed data, undone by the daily `demo:reset`. **Uploads are the exception** — a file served from `demo.nphelper.com` could host abusive/illegal *content* during the window between wipes, and automated bots find open upload forms regardless of traffic. The classic disguised-executable vector is **already closed** (uploads are MIME-gated via `acceptedFileTypes` and re-encoded via `registerMediaConversions`; SVGs run through `SvgSanitizer`). What's missing is a **who-can-upload** gate. This session adds it: the demo role simply can't introduce new files. (Owner decision: do this "the right way cheaply" rather than accept the low-probability hosting risk.)

---

## What ships

- **Gate the new-file upload chokepoints against the `demo` role:**
  1. **The Filament/Livewire temporary-file-upload path** — every Filament `FileUpload` field funnels through Livewire's upload endpoint, so one gate there covers all admin-form uploads (events, products, pages/posts, etc.) at once.
  2. **The page-builder upload routes** — `PageBuilderApiController::uploadImage()` and `uploadAppearanceImage()` (`routes/admin-api.php`). Reject when the user holds the `demo` role.
- **Leave existing-media reuse working** — `useExistingImage` / `useExistingAppearanceImage` and any media-library picker introduce no new file; demo can still set images from the seeded library.
- **UX layer** — a demo user shouldn't hit a raw error: disable/message the upload affordances (a shared Filament hook if one's clean; otherwise a clear "uploads are disabled in the demo" message). This is polish on top of the server-side gate.
- **Extend `DemoRoleLockdownTest`** — demo upload rejected at each chokepoint; a non-demo (admin) upload still succeeds; existing-media reuse still works for demo.

---

## Open questions to resolve at session start

1. **Gate basis** — `hasRole('demo')` check vs an `upload_media` permission granted to all non-demo roles. *Lean: role check* (simplest, matches the existing role-based lockdown guard); permission only if it earns its keep.
2. **Blocked-upload UX** — endpoint hard-reject only vs also disabling/messaging the fields. *Lean: endpoint gate for security + a single shared field-disable if available; clear message otherwise.*
3. **Chokepoint completeness** — confirm Livewire temp-upload + the two page-builder routes are the only new-file paths; audit for any bespoke upload controller and surface anything else.

---

## Phases

### Phase 1 — Endpoint gates (the security boundary)
Gate the Livewire temp-upload path and the two page-builder upload routes against the `demo` role. Confirm existing-media-reuse paths remain open. This alone closes the surface server-side.

### Phase 2 — UX + tests
Disable/message the upload affordances for demo so there's no raw error. Extend `DemoRoleLockdownTest` with the upload-denied / reuse-allowed / admin-allowed cases.
