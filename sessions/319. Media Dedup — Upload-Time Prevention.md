# 319. Media Dedup — Upload-Time Prevention

Stop the media library from filling with re-uploads of the same asset: add a content hash to media, and make the upload surfaces warn-and-offer-reuse when a file already exists. Disk reclamation (one physical file for identical bytes) is the *next* session, not this one.

## The problem this solves

Operators re-upload the same image repeatedly because the existing library is hard to search — so they take the path of least resistance and upload again. This compounds: the more bloated and unsearchable the library gets, the more often people re-upload, which bloats it further. A typical install ends up ~3× its necessary size, mostly duplicate logos and stock images (each multiplied ~6× again by responsive conversions).

This session breaks the loop at the source. It does **not** reclaim the disk already wasted — that is content-addressed storage (the next session). The two are sequenced deliberately: prevention is low-risk and high-leverage; reclamation rewrites how every media file is stored and deleted.

## Phases

### Phase 1 — `content_hash` column + backfill

- Migration: add `content_hash` (string, SHA-256 hex, indexed, nullable) to `media`. Update `docs/schema/media.md`.
- Populate on upload: a Spatie media hook / `Media` model event that hashes the stored original once, at add time. Verify the hook fires for every `addMedia*` path (uploads, `copy()`, importer seeding).
- Backfill: hash existing rows (in the migration if the library is small, or a one-shot `media:backfill-hashes` command if large — judge against the ~178-row local library and document the choice).
- Point session 318's `MediaFinderService::scanDuplicates()` content-hash pass at the column when present, falling back to reading bytes only for un-backfilled rows. This makes the duplicate scan instant.

### Phase 2 — Upload-time dedup check (all surfaces)

On each upload entry point, after the file is received but before/around storing it: detect a likely match against the existing library and, if found, **prompt the operator with a three-way choice** rather than silently storing a copy:

- **Use existing** — reference the asset already in the library; do not keep the new upload. Best when the bytes are identical.
- **Replace existing** — swap the matched asset's file with this upload. Clean under Spatie: replacing the file on the existing `media` row keeps the same id and `/storage/{id}/` URL, so **every owner and embed that references it updates at once** — this is the path forward for a user iterating on a graphic (logo v1 → v2). Regenerate conversions on replace.
- **Keep as new / upload anyway** — the two are genuinely separate assets.

**Match detection must cover two signals, because they catch different cases:**

- **Content hash** (identical bytes) — the exact-duplicate / re-upload case. Here *Use existing* is the headline action; *Replace* is a no-op (same bytes).
- **Filename** (same name, possibly different bytes) — the *iteration* case. An edited graphic has a different hash, so a hash-only check would miss it entirely; the filename match is what surfaces the *Replace existing* path. Decide at session start how to scope filename matching (exact `file_name`, or name within the same owner/collection) to avoid noisy false matches on generic names.

**When more than one existing asset matches, do not auto-pick — disambiguate.** Treat the signals as a ladder: **content hash = identity** (use it to confirm and to collapse), **size = a cheap pre-filter** (good for "are these even plausibly the same?", never a confirmation — two distinct graphics can share a byte count), **filename = the fuzzy net** that surfaces the iteration case. Resolution:

- If the incoming hash matches an existing row, that is the unambiguous *Use existing* target regardless of how many same-named rows exist — the other identical-hash rows are the dupe pile.
- If the filename matches several but the hash matches none (genuinely different versions sharing a name), no automatic signal can decide which the user means. Show a short **disambiguation list** — thumbnail, size, dimensions, "used in N places," upload date — sorted **most-referenced first** (the one the site actually uses is the likely target), with the use / replace / keep-as-new choice per row.
- **Collapse identical-hash entries in that list** so the picker is not itself a wall of duplicates.

This ambiguity largely evaporates once the content-hash collapsing of the CAS session lands — at that point the only same-name multiples left are genuinely-different versions, which is exactly when human choice is wanted.

Surfaces (confirmed at 318 close — cover all, or document exclusions):

- `InlineImageUploadController` (rich-text inline images).
- `PageBuilderApiController` — config image fields + appearance background.
- `DashboardBuilderApiController`, `RecordDetailViewBuilderApiController` — appearance backgrounds.
- Filament `FileUpload` fields: `EventResource`, `PostResource`, `ProductResource`, `WidgetTypeResource`, `PageResource/Pages/EditPageDetails`.

**Reuse semantics (be honest in the log):** under the current `DefaultPathGenerator`, attaching the existing asset to a new owner still copies bytes (`Media::copy()` writes a fresh file). So "reuse" this session means *the operator picks the known asset instead of re-uploading a possibly-different one* — it keeps the library coherent and sets up CAS, but does not yet save disk. Do not implement byte-sharing here; that is the next session.

### Phase 3 — Make the library findable (reduce the incentive to re-upload)

The behavioral half: a browsable/searchable existing-media picker so reuse is the easy path. Evaluate at session start whether to extend the page-builder Vue picker (`useLibraryLoader.ts`) and/or add a "choose from library" affordance on the Filament upload surfaces, vs. scope this phase to a minimal "you already have this — use it?" prompt only. Keep v1 tight; a full DAM browser is out of scope.

## Open questions to resolve at session start

- **Backfill mechanism.** Migration-time backfill vs. a one-shot artisan command — decide against the actual library size and whether prod libraries are large enough to need batching.
- **Hash hook reliability.** Confirm a single hook covers every `addMedia*` path (including importer seeding and `copy()`), or enumerate the paths that need it explicitly.
- **Phase 3 scope.** Minimal "you already have this" prompt vs. an actual library picker on the upload surfaces. Lean minimal for v1; confirm.
- **Reuse mechanics.** When the operator chooses **Use existing**, what happens for a collection-owned field (copy the existing asset to the new owner via `Media::copy()`?) vs. an inline-image embed (just insert the existing URL — no new row at all, the cleanest case)? Inline images are the one surface where reuse needs no copy.
- **Replace scope.** When the operator chooses **Replace existing**, confirm the blast radius is acceptable — replacing the file on a shared media row updates *every* surface referencing it. That is the intended behavior for an iterated asset, but the prompt should make the propagation clear (e.g. "this image is used in N places — all will update"). Decide whether replace is offered on every surface or only where a filename match is found.
- **Filename-match noise.** Generic upload names (`image.png`, hashed names from `->hashName()`) could produce false "you already have this" matches. Note that the page-builder upload already stores files via `->hashName()`, so filename matching there is near-useless — lean on content hash for those surfaces and reserve filename matching for surfaces that preserve the original name.
- **Multi-match disambiguation columns.** The disambiguation list needs a "used in N places" count per candidate (drive it from session 318's `MediaReferenceInventory`) and a thumbnail/dimensions read. Confirm those are cheap enough to compute inline on the upload path, or precompute the reference count. Decide the sort tie-breakers (most-referenced first, then most-recent).

## Out of scope

- **Content-addressed storage / byte-level dedup** — the next session. No path-generator change here.
- **Refcounted deletion** — pairs with CAS; not this session.
- Retroactive dedup of the existing library beyond what 318's finder already surfaces.
- A full digital-asset-management browser; Phase 3 is a findability nudge, not a DAM.
