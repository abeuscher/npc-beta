- `routes/api.php` — `Route::middleware(['throttle:60,1'])->get('/backup/blob', [BackupController::class, 'blob'])`. Same throttle as `/api/health` and `/api/logs`; intentionally generous compared to `/api/backup/trigger`'s `6,1` because blob fetches are I/O-bound and cheap relative to a synchronous backup pipeline.
- New sibling method `BackupController::blob(Request $request): StreamedResponse|JsonResponse` — sibling to `trigger()` rather than a new controller (default decision; no audit pressure to split). Implementation:
  - Resolves the disk order via `resolveBackupDiskOrder()` private helper. Layer A: if `local` is in `BACKUP_DISKS`, move it to the front regardless of authored position. Layer B: iterate the resolved order in the foreach.
  - 500 with `backup_destinations_not_configured` envelope upfront if the resolved list is empty.
  - For each disk: `BackupDestination::create($diskName, config('backup.backup.name'))`. If `connectionError !== null`, `continue` to next disk. Else `$destination->backups()->newest()`.
  - On non-null newest: `Storage::disk($diskName)->download($newest->path(), basename($newest->path()), ['Content-Type' => 'application/zip', 'Cache-Control' => 'no-store'])`. Returns immediately.
  - On null newest: continue to next disk.
  - On exhaustion: 404 `no_backup_available` envelope.
  - Both BackupDestination resolution and `Storage::disk()->download()` invocation wrapped in try/catch returning 500 `backup_disk_error` with sanitised message via the existing `sanitise()` helper.
- `HealthController::CONTRACT_VERSION` and `BackupController::CONTRACT_VERSION` bumped `'2.2.0' → '2.3.0'`. The trigger envelope's `contract_version` field moves automatically via `self::CONTRACT_VERSION`.
- Existing `HealthEndpointTest` + `BackupTriggerEndpointTest` version-string assertions updated to `'2.3.0'`.

### In-session absorptions

- **Filename pattern correction** vs the session prompt's stated assumption. The prompt's design-decisions block claimed spatie blobs look like `nonprofitcrm-2026-05-08-12-30-00.zip` (default `<backup-name>-Y-m-d-H-i-s.zip`). Reading `BackupJob::FILENAME_FORMAT` directly confirmed the actual default is `Y-m-d-H-i-s.zip` — no `<backup-name>-` prefix. The path-within-disk is `<backup_name>/<filename>` (so `NonProfitCRM/2026-05-08-00-07-27.zip`), but `basename($newest->path())` strips the directory, leaving just the timestamp + `.zip` for `Content-Disposition`. Triggered a real `php artisan backup:run` during verify-at-start which produced exactly that filename pattern. Spec doc + CHANGELOG explicitly call this out so FM consumers don't expect the prefix. FM-side prefixing for multi-install storage is FM's concern.
- **`Cache-Control` header value drift.** Test #1 against `Storage::fake('local')` exercised the streaming end-to-end and surfaced that Laravel's API middleware appends `private` to the Cache-Control header, producing `no-store, private` rather than the bare `no-store` the prompt wrote. The defensive intent (no caching) holds regardless. Test relaxed from exact-match to substring assertion (`->toContain('no-store')`); spec doc table notes the appended `private` directive is harmless.
- **`Storage::fake('local')->put()` registers as a recognized backup.** The 267 close-time concern (carried in the prompt's verify-at-start question 6) was that `BackupCollection::newest()` filters by spatie's `(new File)->isZipFile()` check and might not register pre-written bytes. Reading the spatie source clarified `isZipFile` returns true on `.zip` extension alone (no body read); `Storage::fake` produces a real local-driver disk; pre-written files at `<backup_name>/<Y-m-d-H-i-s>.zip` register cleanly. Test #1 confirmed before committing to the full matrix; no need to drop into container-binding mocks for `BackupDestination::create()`.

## Findings register

- **Spatie filename format correction.** `BackupJob::FILENAME_FORMAT = 'Y-m-d-H-i-s.\z\i\p'`. No `<backup-name>-` prefix. The session prompt's example (`nonprofitcrm-2026-05-08-12-30-00.zip`) was wrong; actual is `2026-05-08-00-07-27.zip`. Spec doc + CHANGELOG explicit. FM-side prefixing for multi-install storage is FM's concern.
- **`isZipFile` filters by extension, not body.** Spatie's `BackupCollection::createFromFiles` calls `(new File)->isZipFile($disk, $path)` to filter; `isZipFile` returns true on `.zip` extension match alone (no MIME / body read). This unblocked the test pattern — `Storage::fake('local')->put('NonProfitCRM/<filename>.zip', $bytes)` registers fine without dropping into container-binding swaps.
- **`BackupDestination::create()` swallows disk-instantiation errors into `connectionError`.** A misconfigured disk doesn't throw — it returns a destination with `disk = null` and a populated `connectionError`. The blob controller checks for this and skips silently (continues to the next disk in the resolved order), which is the right behaviour for the fallthrough case but means a misconfigured-but-listed disk is invisible from the response. Documented as part of the disk-fallback rule's behaviour.
- **`BackupDestination::backups()` swallows `allFiles()` exceptions internally** (spatie source: `try { $files = $this->disk->allFiles($this->backupName); } catch (Exception) {}`). Asynchronous storage errors during enumeration manifest as 404 with the no-backup-available envelope, not as a 500. The 500 `backup_disk_error` envelope is reachable only via synchronous download-time exceptions (`mimeType()` / `size()` failures). Spec doc names this distinction.
- **Laravel API middleware appends `private` to Cache-Control.** The bare `Cache-Control: no-store` returned from `Storage::download()`'s headers parameter ends up as `no-store, private` after middleware processing. Defensive intent (no caching) holds; spec doc table notes the appended directive is harmless. Tests use substring-match (`toContain('no-store')`) rather than exact-match.
- **Manual smoke against a 4.5GB blob worked through the streaming pipeline cleanly.** No memory issues on the streaming side; nginx's `fastcgi_read_timeout 600;` held; `Content-Length` matched the on-disk file size. The 600s ceiling concern (multi-GB blobs to slow FM-side networks) remains as forward-looking defense; the dev success path doesn't exercise the slow-network case.
