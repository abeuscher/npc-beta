# 209. Widget Primitive — Data Contract Prototype

The go/no-go decision session for the widget primitive architectural track. Build a minimal `DataContract` + `ContractResolver`, migrate three existing widgets onto it, and produce a findings document that says either "the premise survives, proceed to Phase 2" or "the premise did not survive, here is specifically why."

This session is Phase 1 of the plan in `sessions/widget-primitive-migration.md`. Scope is deliberately small. The rest of the plan does not execute until this one produces a clear go signal.

---

## Design decisions (resolved before starting — do not relitigate)

Settled in `widget-primitive-migration.md`:

- **Pragmatic, not purist.** Filament stays. Not in scope this session in any case — this is a prototype, not an integration.
- **One resolver, single service.** All contract resolution flows through one class. No per-widget resolver, no per-model resolver.
- **Contracts are typed structured declarations.** PHP value objects, not stringly-typed scopes. The declaration surface is a first-class artifact.
- **Contracts carry a version field from day one.** Even with one version shipping, the field exists.
- **DTOs are narrow and short-lived.** Each render produces a fresh payload. Widgets do not hold Eloquent references.
- **Batching designed in.** The resolver signature accepts a list of contracts, not one at a time, even if the first consumer passes one.
- **Pre-beta license.** No backward-compat shims. If migrating a widget requires changing its config shape, change it. The squash in 208 absorbs whatever new migrations land.

Settled in scoping conversations for this session:

- **Three widgets to migrate:** Carousel, Blog Listing, Text block. Each exercises a structurally different contract shape:
  - **Blog Listing** — typed system model, known fields at compile time.
  - **Carousel** — items from a widget-declared content type. The widget's structural type declaration *is* the content type's schema.
  - **Text block** — no model data; only page-context tokens.
- **Content shapes are widget-declared.** Carousel's content type is owned by the widget. The existing admin-side UI for creating arbitrary Collections is not touched this session — its retirement is Phase 4 work — but the Carousel contract is written as if widget-declared content types are the adopted direction. No runtime "validate Collection against widget expectation" indirection; the schema comes from the widget.
- **Page context is a typed data source.** The Text block contract declares which page-context fields it consumes (`site_name`, `title`, etc.). "Widget that only reads page context" becomes a concrete contract case, not a template-render bypass.
- **DTOs are flat-with-row-sets.** Scalar fields for aggregates; named row-sets for listings; sub-DTOs inside row-sets are themselves flat. No walkable relationships in either direction. Strong prior, not a hard rule — if it breaks on one of the three widgets, the adjustment is a finding, not a failure.

To be decided inside this session:

- The exact contract syntax: path expressions, filter expressions, aggregate expressions.
- How filters compose. The existing `query_config` semantics are the starting point, not the ceiling.
- How the Carousel content-type declaration is represented — inline in the Definition class, a dedicated types file alongside the widget, or a new declaration method on `WidgetDefinition`. Pick one and document the precedent; the pattern set here applies to every widget-declared content type.
- The findings doc's verdict: proceed, proceed with refinements, or stop.

## Phase 1 — Scaffold the contract and resolver

Minimum viable machinery. No widgets migrated yet.

- New namespace: `app/WidgetPrimitive/` for all prototype code.
- `app/WidgetPrimitive/DataContract.php` — value object. Minimum shape:
  - Version field.
  - Requested fields (path expressions).
  - Optional filter expressions.
  - Optional content-type declaration for widget-owned collections (the Carousel case).
- `app/WidgetPrimitive/ContractResolver.php` — single service.
  - Signature: `resolve(array $contracts, SlotContext $context): array` — list in, list out, matched by index.
  - `SlotContext` wraps the existing `PageContext` for this session. Do not generalize slot taxonomy — Phase 2 work.
  - Batching shape present even if not fully optimized: when multiple contracts resolve against the same model, do not fire independent queries.
- `app/WidgetPrimitive/SlotContext.php` — thin wrapper over `PageContext` for this session. The type exists so the signature can evolve; the contents are passthrough.
- Skeleton tests in `tests/Unit/WidgetPrimitive/`:
  - Contract construction with version + field list.
  - Resolver returns DTOs indexed to match input contract list.
  - Fields not declared in the contract do not appear in the DTO.

## Phase 2 — Migrate Blog Listing

The typed system-model case. Easiest of the three. If this doesn't migrate cleanly, the premise is in doubt — stop and raise for user consultation before attempting the other two.

- Write the Blog Listing `DataContract` in the widget's Definition class (or its declared companion file if that convention emerges in Phase 4).
- Replace the template's direct model access with DTO field access. `{{item.title}}` etc. continues to work but resolves against the DTO, not a live Page model.
- Byte-equivalent output on the rendered side. Existing Pest + Playwright coverage for Blog Listing must continue to pass without test changes — if a test has to change, the migration introduced a behaviour delta and that's a finding.

---

## Phase 3 — Migrate Text block

The no-model case. Establishes page context as a typed data source.

- Write the Text block `DataContract` declaring which `SlotContext` fields it consumes.
- `{{site_name}}` / `{{title}}` resolution routes through the contract, not a template-render bypass.
- Add a minimal render test asserting page-context tokens substitute as expected under the new path. Text block has thinner existing coverage; backfilling one test here earns its keep.

---

## Phase 4 — Migrate Carousel

The decisive case.

- Write the Carousel `DataContract` including a content-type declaration for the items shape — required slots (image field, caption template inputs) plus declared passthrough fields that items may carry for template-token addressing (`{{item.title}}`, `{{item.date}}`, etc.).
- Pick the representation for the content-type declaration. Options to weigh in-session:
  - Inline in the Definition class as a PHP array method (`contentType(): array`).
  - A dedicated companion file alongside the widget (e.g. `app/Widgets/Carousel/ContentType.php` returning a schema).
  - A typed method on the `WidgetDefinition` base contract so every widget either declares a content type or declares "no content type consumed."
- Document the choice in the findings doc. The precedent set here will apply to every widget-declared content type.
- Preserve existing behaviour. Carousel has Pest + Playwright coverage; must pass unchanged.
- **Explicitly not in scope:** migrating the existing admin-side Collection creation UI, migrating existing `collection_items` rows to widget-owned ownership, or any user-visible change to how Carousel is configured today. Carousel continues to read from `collection_items` as it does now; only the contract path between the fetch and the template changes. The admin-UI tightening and data-ownership migration are Phase 4 of the broader plan — this session only proves the contract shape works.

## Phase 5 — Findings doc

The load-bearing artifact of the session. Write `sessions/209. Widget Primitive — Data Contract Prototype — Findings.md`. Structure:

- **Verdict.** One of: "Proceed to Phase 2", "Proceed with refinements (listed)", or "Stop — premise did not survive, specifics below."
- **What worked.** Specific observations on what the contract shape handled cleanly.
- **What did not work.** Specific observations on awkward cases, places the contract language bent toward being a second ORM, where a declaration felt forced.
- **Proposed refinements before Phase 2, if any.** Concrete list.
- **Precedents set for future phases.** What patterns this session establishes — content-type declaration shape, DTO shape conventions, page-context-as-data-source, slot-context boundary, etc.
- **Open questions for future phases.** Things the prototype raised but did not resolve — mutation contracts, batching at scale, configuration UI for contracts, etc.

Committed alongside the session log at close.

## Testing

- **Slow test groups to run this session:** none.
- **New tests expected:** yes. Resolver behavior (field scope enforcement, batching shape, version-field round-trip). One render test per migrated widget asserting the output is byte-equivalent to the pre-migration render for the same inputs.
- **Playwright:** required — `npm run test:e2e` must pass before pausing for manual test. Carousel and Blog Listing are exercised by the page-builder specs; regressions surface there. Text block is exercised implicitly wherever it's rendered in test pages.
- **Widget asset rebuild:** not expected — no widget SCSS/CSS/JS changes.

## Security checklist

Before closing:

- [ ] Three migrated widgets enforce the same permission scope post-migration as pre-migration. No expansion of what any of them can read.
- [ ] Resolver does not expose fields outside what the contract declares.
- [ ] `SlotContext` wraps the existing `PageContext` without widening its ambient fields.
- [ ] No new untrusted-input → filesystem or database write paths.
- [ ] Fast Pest suite passes.
- [ ] `npm run build` passes.
- [ ] `npm run test:e2e` passes.
- [ ] Portal `contact_id` scoping preserved — no portal code touched this session, but verify by grep.

## Out of scope

- Migrating any widget other than Carousel, Blog Listing, Text block.
- Slot taxonomy beyond wrapping `PageContext` as `SlotContext`.
- Dashboard or record-detail slot surfaces.
- Filament integration.
- Admin UI for creating / managing widget-declared content types.
- Retiring the existing arbitrary-Collection admin UI — Phase 4 work.
- Migrating existing `collection_items` rows to widget-owned ownership — Phase 4 work.
- LLM-assisted widget-authoring pathway — separate track.
- Write / mutation contracts — this prototype is read-only.
- Any performance work beyond "batching shape is designed in."
- Public extension API.
