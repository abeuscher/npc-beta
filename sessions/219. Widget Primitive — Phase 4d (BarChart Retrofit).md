# 219. Widget Primitive — Phase 4d (BarChart Retrofit)

Retrofit BarChart onto a `SOURCE_WIDGET_CONTENT_TYPE` data contract: add `dataContract()` (per-instance field whitelist driven by user-configured `x_field` and `y_field`), rewire the template to read `$widgetData['items']`, drop the legacy `collections()` slot declaration, swap the two sibling slot-strip tests off BarChart, and land the Phase 4 test-lock pair. First true `SOURCE_WIDGET_CONTENT_TYPE` retrofit (Carousel was an audit), and the first widget where the contract's `fields` list is dynamic rather than fixed.

## Design decisions (resolved before starting)

- **Contract shape: per-instance, dynamic fields.** Unlike Carousel's static `['title', 'description']`, BarChart's contract `fields` list is built from `$config['x_field']` and `$config['y_field']` at `dataContract()` call time. The contract whitelists exactly the two Collection fields the admin mapped — no more, no less. When either field is empty (unconfigured widget), the corresponding entry is omitted from the contract's `fields` array, and the projector returns rows with whatever subset is set. The widget's existing `requiredConfig` already gates rendering on both fields being populated, so the empty-fields case is the unmounted-state path.
- **Inline ContentType.** Same pattern as Carousel: declare an inline `ContentType` in `dataContract()` with two `text`-typed fields keyed by `$xField` and `$yField`. No image fields (BarChart has none), so the projector's `_media` map will be `[]` — that's fine, the template never reads `_media`. The ContentType handle is `'bar_chart.data'`.
- **Source: `SOURCE_WIDGET_CONTENT_TYPE`.** `resourceHandle` = `$config['collection_handle']`. Same source as Carousel; the resolver already handles this case (`ContractResolver::resolveWidgetContentType()`).
- **Template change.** Line 9 `$collectionData['data'] ?? []` becomes `$widgetData['items'] ?? []`. The downstream `foreach ($items as $item) { $item[$xField], $item[$yField] }` access pattern is unchanged — the projected DTO row still has those keys (because we declared them on the contract). No other template edits.
- **`collections()` removal.** Per the system rule established in 218: contract-locked widgets do not declare slots. `BarChartDefinition::collections()` is removed. Admin preview shows the unmounted state for empty/unconfigured BarCharts (correct UX). Demo data flows through `DemoSeeder` for dev/seeded paths.
- **Sibling-test retargeting.** Two tests in `tests/Feature/PageBuilderApiTest.php` ("strips query_config slots that are not declared on the widget type" and "update endpoint contract that the JS selective merge depends on") use `'bar_chart'` and slot key `'data'` to exercise the legacy slot-strip behavior. After 219 removes `BarChartDefinition::collections()`, both must swap to either BoardMembers (slot key `'members'`) or LogoGarden (slot key `'logos'`). Pick one and apply consistently. Same pattern 218 used to swap from `'carousel'`/`'slides'` to `'bar_chart'`/`'data'`.
- **Test-lock fields.** Test A asserts the projector emits exactly two declared keys (`$xField` and `$yField`) plus `_media`, with sentinels in undeclared Collection-item `data` keys not leaking. Sentinels go in keys that have no derived whitelisted counterpart — for BarChart, this is anything other than the user-mapped X/Y fields. Use sentinel keys `internal_notes` and `legacy_id` (same pattern as Carousel). Test B asserts the contract path emits exactly one collections, one collection_items, and one media select — `WidgetDataResolver` loop no longer fires for BarChart after `collections()` removal.
- **Public-surface gate.** `ContractResolver::resolveWidgetContentType()` filters by `is_public = true` when `$context->publicSurface` is true (canvas-slot default). Test fixture's Collection must be `is_public => true, is_active => true`.
- **Demo seeder unchanged.** `BarChart/DemoSeeder.php` already creates a real `chart-demo` Collection — no fallback path involved. The dev demo route renders against live data via the contract. No seeder edits.
- **No new schema.** No migration, no `docs/schema/` updates.
- **Front-end builds.** `script.js` and `assets()` declarations are untouched. Default expectation: no `npm run build`, no `build:public`. Run only if cleanup unexpectedly touches widget asset files.
- **Out of scope: changing how `x_field` / `y_field` are validated or persisted.** They remain free-text-from-Collection-fields-list as today.

## Phase 2 — Contract + projector

`WidgetContentTypeProjector` already handles SWCT projection generically — no projector edits needed. The projection map is the contract's `fields` list (which the widget controls per-instance).

Add `dataContract(array $config): ?DataContract` to `BarChartDefinition.php`:

```php
public function dataContract(array $config): ?DataContract
{
    $xField = (string) ($config['x_field'] ?? '');
    $yField = (string) ($config['y_field'] ?? '');
    $fields = array_values(array_filter([$xField, $yField], fn ($f) => $f !== ''));

    $contentTypeFields = array_map(
        fn ($f) => ['key' => $f, 'type' => 'text'],
        $fields,
    );

    $contentType = new ContentType(
        handle: 'bar_chart.data',
        fields: $contentTypeFields,
    );

    return new DataContract(
        version: '1.0.0',
        source: DataContract::SOURCE_WIDGET_CONTENT_TYPE,
        fields: $fields,
        resourceHandle: $config['collection_handle'] ?? null,
        contentType: $contentType,
    );
}
```

Add the `use App\WidgetPrimitive\ContentType;` and `use App\WidgetPrimitive\DataContract;` imports.
