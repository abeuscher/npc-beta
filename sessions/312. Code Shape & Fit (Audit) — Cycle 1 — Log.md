### Flag W-R1/A — `buildTree()` returns a triple-key superset; each endpoint destructures a different slice

- **What was found:** `PageBuilderApiController::buildTree()` (at line 510+) returns `['widgets' => ..., 'items' => ..., 'required_libs' => ...]`. Seven different endpoints (`index`, `store`, `update`, `destroy`, `copy`, `storeLayout`, `destroyLayout`) each call `buildTree()` and destructure a different subset of the three keys for their response. Within `buildTree()` itself, `WidgetType::requiredForPage($owner->bareSlug())` is computed at line 527 and again at line 605 in a sibling private method; `WidgetPreviewRenderer::collectLibs()` is called in two parallel loops (root widgets at line 553, layout-nested widgets at line 567) accumulating into the same `$allLibs` array.
- **Risk + scope:** The cardinality finding is real and the surface is the buildTree contract. A clean fix introduces a `PageTreeBuilder` service exposing `tree()` / `items()` / `requiredLibs()` as separate memoized computed properties, so each endpoint asks for what it needs. That's a W-R5-shape change (seam move) wearing W-R1 clothing — too big for a single 313 iteration. **Risk class:** medium (touches the page-builder API surface that the Vue editor depends on; the Playwright page-builder spec set will need to confirm green).
- **Proposed paths:**
  - **(a) Instance fix.** Pick the most-wasteful caller (`reorder` — touches many widgets but only returns tree structure; or `destroy` — derives `requiredHandles` once for permission check then again via buildTree) and introduce a `buildTreeMinimal()` variant that skips preview-renderer collectLibs for that path. ~20 LOC, 1 file, low risk. Leaves the other 6 endpoints unchanged.
  - **(b) Pattern fix.** Lift a `PageTreeBuilder` service that exposes `tree()`, `items()`, `requiredLibs()` as separate computed properties with memoization. Each endpoint becomes thin. ~80 LOC across 2 files, medium risk. Better long-term seam.
  - **Default leaning:** (a). 313 walkthrough decides.

### `app/Http/Controllers/Admin/PageBuilderApiController.php` — `WidgetType::requiredForPage()` re-derivation across `destroy()` + `buildTree()`

- **Surface pattern:** W-R1 (Cardinality). The same `WidgetType::requiredForPage($owner->bareSlug())` call appears at line 156 (inside `destroy()` for the permission check) and again at line 527 (inside `buildTree()` which `destroy()` calls immediately after at line 164). - **What's searched / what's missing:** No docstring on `WidgetType::requiredForPage()`; no docstring on `PageBuilderApiController::destroy()`; no test names that pin the cardinality (`grep -rn "requiredForPage" tests/` returns only call-site tests, none that assert single-derivation); no session-log reference for the required-widget feature's intent at this controller seam; no memory entry. The structural duplication is real and easy to verify, but there is no recoverable Intent for what `requiredForPage()` is trying to do *at this controller seam* — only the model-level concept of "a Page has required widget handles that cannot be deleted" is implicit in the call name.
- **Why this matters:** Inlining the second derivation looks like a safe four-line change, but without recoverable Intent the apply session can't confirm whether the second derivation is a slip or whether some not-yet-discovered code path expects fresh evaluation per controller method. The cost of a wrong fix is small (the call is pure); the cost of *documenting before fixing* is also small and serves the broader documentation-gap.
- **Suggested next step:** Document the `requiredForPage` seam's intent in either the controller method docstring or in a session log before lifting the duplication. Walkthrough discussion: does the user recall whether the original design called for fresh derivation per endpoint?

### `app/Http/Controllers/Admin/PageBuilderApiController.php` — `WidgetPreviewRenderer::collectLibs()` parallel loops at lines 553 + 567

- **Surface pattern:** W-R1 (Cardinality). Two foreach loops (root widgets, then layout-nested widgets) each call `app(WidgetPreviewRenderer::class)->collectLibs($widget, $allLibs)` accumulating into the same `$allLibs` array. The loops have the same shape and the same effect on the accumulator.
- **What's searched / what's missing:** No docstring at the loop site; no docstring on `WidgetPreviewRenderer::collectLibs()`; no test names pinning the dual-loop accumulation pattern; no session log explaining why the iteration is two passes instead of one over `array_merge($rootWidgets, $layoutChildren)`. Function names (`collectLibs`, `$allLibs`) express *what* but not *why* the work is structured this way.
- **Why this matters:** The straightforward consolidation (one loop over a merged collection, one `collectLibs` call per widget) looks safe and saves a structural duplicate. But the dual-loop shape may reflect ordering invariants the lib accumulator depends on, or a downstream rendering invariant tied to the order widgets appear in the response items list. Without a recoverable Intent the apply session cannot confirm safety.
- **Suggested next step:** Document the loop-order intent (or confirm the order doesn't matter) before consolidation. Walkthrough discussion: does the user recall whether the original design depended on root-before-layout ordering of accumulated libs?

### `PageBuilderApiController` endpoint shape — per-action persistence, not bulk save

- **What it looks like:** A W-R5 (Seam quality) finding. Twenty-two HTTP endpoints, most of them per-action CRUD-shape (widgets store/update/destroy/copy/reorder/preview; layouts store/update/destroy; image upload/remove; appearance image upload/remove; updateColorSwatches; data lookups). Mirrors a 1:1 mapping with Vue store actions.
- **Why it's intentional:** The shape supports optimistic UI, per-action revert / retry, per-action undo, and aligns with the rest of Filament's admin patterns (per-action REST). Collapsing to one bulk endpoint per save transaction would be a regression in those dimensions.
- **Source / citation:** 312 audit log § W-R5; 273 apply log § /6 editor.ts composable extraction (establishes the Pinia store's per-action dispatch pattern); the `api.ts` thin-facade design at `resources/js/page-builder-vue/api.ts:82-90` documents the owner-scoped baseUrl pattern for per-action calls.
- **Proposed workstream coverage:** W-R5 (Seam quality), W-R1 (Cardinality — the surface pattern that triggers the consideration).
