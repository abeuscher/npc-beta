## What was built

### Phase 1 — Read-through, audit, sign-off

- Confirmed all 7 CF model_type keys (`contact`, `event`, `event_registration`, `donation`, `organization`, `transaction`, `membership`) are already established in code via the namespaced importer auto-create paths in [`InteractsWithImportProgress::resolveCustomFieldDefs()`](app/Filament/Pages/Concerns/InteractsWithImportProgress.php). No new CF wiring needed for the export feature. Pre-existing gap noted: `CustomFieldDefResource` admin form's `model_type` select only exposes `contact / event / page`; the other four CF model types exist in the data but aren't surfaced for admin browsing. Deliberately out of scope.
- Architectural shape signed off: shared `App\Services\ListExportService::stream()` with a single format flag, called from each ListXxx page's action.
- EventRegistration export confirmed event-scoped (lives on `ViewRegistrations`, query already filtered by `event_id`).
- ActionGroup inventory: ListContacts already had one (with Review Duplicates / Import / Export CSV). ListEvents had one (Show/Hide Past Events). The other 8 needed new ones.

### Phase 2 — Build the shared writer service

- Created [`app/Services/ListExportService.php`](app/Services/ListExportService.php) with a single public `stream()` method. Inputs: `Builder $query`, `array $columnSpec`, `string $format`, `string $filename`, `?string $cfModelKey`. Output: `Symfony\Component\HttpFoundation\StreamedResponse`.
- Private `streamCsv()` mirrors the existing Contacts CSV exporter shape — `fputcsv` per row, headers + per-column values + per-CF values, iterates via `$query->each()`.
- Private `streamJson()` writes a streaming top-level array — opens `[`, comma-separated `json_encode` per row, closes `]`. Custom fields nested as a `custom_fields` sub-object, empty CFs omitted (not present rather than `null`-or-empty-string).
- CSV `Content-Type: text/csv`; JSON `Content-Type: application/json`.
- Refactored [`ContactResource\Pages\ListContacts::exportContacts`](app/Filament/Resources/ContactResource/Pages/ListContacts.php) to call into the service. The Contacts CSV exporter is now a 7-line action body that delegates to `ListExportService::stream()` with `ContactResource::exportColumnSpec()`. Behavior is bit-for-bit identical on the user-visible side.

### Phase 3 — `exportColumnSpec()` per Resource + per-page wireup

- Added `exportColumnSpec()` static method to each Resource (Contact, Organization, Donation, Event, Membership, Transaction, Fund, Campaign, Note) declaring the column shape — `[['key', 'header', 'value' => Closure], ...]`.
- Added `EventResource::registrationExportColumnSpec()` for the `ViewRegistrations` event-scoped export.
- Wired `Actions\Action::make('exportCsv')` and `Actions\Action::make('exportJson')` into ellipsis ActionGroups across 9 non-Contact list pages, plus a new `exportContactsJson` action on `ListContacts` alongside the existing CSV action. `ViewRegistrations` (a `Page implements HasTable`, not `ListRecords`) gained a new `getHeaderActions()` method.

### In-session absorptions

- **DuplicateHeaderDetector address-line carve-out** *(B2b finding, surfaced during round-trip-readiness check)* — `address_line_1`/`address_line_2`, `address_1`/`address_2`, `street_1`/`street_2`, `address-line-1`/`address-line-2` were getting flagged as duplicates by the importer's Review Data step despite living in distinct database columns. Root cause: [`isKnownLegitimateGroup()`](app/Services/Import/DuplicateHeaderDetector.php) regex used `\s` as the line/digit separator (matched `Address 1`, `Address Line 2`) but missed `_` and `-` separators. Surfaced when round-tripping the new exporter (which emits `address_line_1`/`_2` as canonical headers) back through the Organizations importer. Fix: widened to `[\s_\-]`. 6 new regression tests + 1 still-flagged-non-address guard (`phone_1`/`phone_2` still surfaces as duplicates) added to [`tests/Feature/ImportSession192Test.php`](tests/Feature/ImportSession192Test.php) under a new "Carve-out: separator-agnostic address-line variants" section. The section comment explicitly notes the carve-out is the fragile bit of the detector ("has been wiped out before by optimization passes") so future maintainers understand why the coverage is intentionally exhaustive across separator forms.
- **Import Contacts link removed from contacts ellipsis menu** — operator-test feedback noted that linking the importer from the contacts ellipsis menu makes per-importer permission isolation harder to enforce. Removed the action and its `ImporterPage` import. Operators reach the importer through Tools → Importer where the per-importer permission gating lives.

## Security checklist — verified

- [x] Permission gate enforced on every export action via `->hidden(fn () => ! auth()->user()?->can('view_any_<resource>'))` — exact pattern match with the existing Contacts CSV action. Capability-gate tests cover all 10 resources.
- [x] No PII leakage in JSON: column shape mirrors importer `flatFields()` output (where a registry exists) or `$fillable` (where one doesn't); no admin-only audit columns surface.
- [x] Custom-field flattening doesn't leak across model types — `CustomFieldDef::forModel($cfModelKey)` is scoped per resource.
- [x] EventRegistration export is event-scoped — the query is `EventRegistration::query()->where('event_id', $this->record->id)`; cross-event URL fishing is impossible.
- [x] Filter+sort state honored via `getFilteredSortedTableQuery()` (the existing Filament helper); a service-level test exercises a filtered query end-to-end.
- [x] DuplicateHeaderDetector regression-guard coverage: non-address numbered pairs (e.g. `phone_1` / `phone_2`) still flag as duplicates; the address-line carve-out doesn't over-reach.
