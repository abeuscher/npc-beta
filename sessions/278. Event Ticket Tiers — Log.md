## What was built

Promoted event ticket pricing from a single `events.price` field into a tiered
structure — `ticket_tiers` table, FK on `event_registrations.ticket_tier_id`,
admin tier repeater on EventResource, public tier picker in the registration
widget. Shape **(A)** chosen at session start: tier-canonical, with
`events.price` and `events.capacity` dropped; price + capacity live on
`ticket_tiers` only. Truly free + uncapped events have zero tiers.

### Iteration /1 — TicketTier model + migration + General-tier backfill (937ef1c)

- New `ticket_tiers` table (uuid id, event_id FK cascade, name, price
  decimal(8,2), capacity nullable int, sort_order, timestamps; indexed on
  `(event_id, sort_order)`).
- `event_registrations.ticket_tier_id` — nullable uuid FK, set-null-on-delete.
- One atomic migration covers: table create, FK add, General-tier backfill,
  retroactive importer linkage, drop of `events.price` and `events.capacity`.
- Backfill rule: every event with `price > 0` OR `capacity IS NOT NULL` gets
  one `{name: 'General', price: events.price, capacity: events.capacity,
  sort_order: 0}` tier. Truly-free-uncapped events get no tier.
- Retroactive linkage: every `event_registrations` row with non-null
  `ticket_type` resolves to a tier — case-insensitive name match reuses,
  otherwise creates a new tier on that event with the importer's
  `ticket_fee` and `capacity: null`.
- **Backfill counts (dev DB at migrate time):** 0 General tiers from real data
  (current dev DB had only free-uncapped scrub events). Synthetic-data
  verification round (rolled back, seeded 4 priced + free-capped + retro
  events, re-ran): 4/4 events backfilled, 2/2 retroactive linkage (1 matched
  General by name, 1 created VIP). Documented the migration's behavior on
  populated data.
- `Event::is_free` and `Event::isAtCapacity` rewired to walk tiers (loaded-
  relation fast path for the resolver, lazy-load fallback elsewhere). New
  `TicketTier` model + factory; `EventFactory` rewired with `paid($price,
  $capacity?)` and `withCapacity($n)` states that use `afterCreating` to
  attach a tier.
- Controllers (free + paid + portal-free + portal-paid) read the chosen tier
  for price/capacity; widget contract drops `price`; resolver eager-loads
  `ticketTiers` with `withCount('registrations')`; projector reads
  `is_at_capacity` from the loaded collection.
- EventResource admin form: price + capacity controls removed (replaced by
  tier repeater in /2). Export columns scrubbed.
- Schema docs: new `ticket_tiers.md`; `events.md` notes the migration;
  `event_registrations.md` documents the new FK + the snapshot/canonical
  relationship between `ticket_type`/`ticket_fee` and `ticket_tier_id`.
- 14 new Pest tests in `TicketTierTest.php` (relations, per-tier capacity,
  is_free across tier shapes, cascade + null-on-delete).

### Iteration /2 — Admin tier repeater on EventResource (8eb801c)

- Filament `Repeater::make('ticketTiers')->relationship('ticketTiers')` in
  EventResource — name (required, max 255), price (required, decimal, ≥ 0,
  $-prefixed), capacity (optional int, ≥ 1, "leave blank for unlimited").
- Drag-handle reorder via `->reorderable('sort_order')->orderColumn('sort_order')`.
- Section hidden when `registration_mode` is `external` or `none`.
- Cross-row uniqueness rule rejects duplicate tier names within an event
  (case-insensitive) — same matcher rule the importer's retroactive linkage
  uses, so admin-UX and importer agree on what counts as "same tier".
- Collapsed-row label: `Name — $price`.
- 4 new Pest feature tests (create, name-required, negative-price-rejected,
  duplicate-name-rejected). Filament's repeater UUID-keyed state doesn't

### Iteration /3 — Public tier picker + per-tier capacity + Playwright spec (5d885c3)

- Three rendering modes:
  - **0 tiers** → no tier UI, single "Register for this event" button.
  - **1 tier** → hidden `ticket_tier_id` input + "Register & pay" / "Register"
    button per the tier's price; price displayed above the form for paid.
  - **2+ tiers** → radio picker with name + price + "(sold out)" labels;
    first non-sold-out tier pre-selected; neutral "Register" button
    (server-side routes based on chosen tier's price).
- Form action collapses to a single endpoint per surface (`events.register`
  / `portal.events.register`); the controller reads the chosen tier's price
  and either lands the registration directly (free) or hands off to checkout
  (paid). Cross-event tier_id rejected by `Rule::exists(...)->where('event_id',
  ...)`.
- Per-tier capacity replaces event-level — "this ticket tier is at capacity"
  fires when only the chosen tier is full; the event-level "registration
  full" notice only fires when *every* tier is at capacity. A non-full tier
  remains registerable when a sibling tier is full.
- Stripe Checkout line-item product name now reads `"{event title} — {tier
  name}"`; amount uses the chosen tier's price.
- `resources/docs/widget-event-registration.md` rewritten — the "tiers are
  not on this widget today" gotcha replaced with the tier-aware behaviour
  walkthrough (no-tier / single-tier / multi-tier behaviour, per-tier
  capacity semantics, mixed-price-set note). `help:sync` re-run.
- 10 new Pest tests in `TicketTierPublicPickerTest.php`. Existing
  EventCheckoutController / FinancialSourceWritePathTest / PaidCheckoutTest /
  EventTest tests updated to pass `ticket_tier_id` on paid-event flows.
- Playwright spec at `tests/e2e/event-registration/ticket-tier-picker.spec.ts`
  walks the rendered DOM for all three tier cases. **Playwright spec count
  delta: +1 spec, +3 scenarios** (46 → 47 specs, scenarios 49 → 52). New TS
  helper `createPublishedEventWithTiers` builds the event + tiers + landing
  page + widget in a single SQL pass.

### Iteration /4 — Notes field + drop email-uniqueness dedup (34b38d6)

Operator feedback from /3 manual testing surfaced two affordances:

- **Notes field** on the public registration form (textarea, ≤ 2000 chars,
  optional). Persists to the existing `event_registrations.notes` column —
  no schema change. Serves as the near-term workaround for missing per-
  attendee data: "bringing a +1; vegetarian; wheelchair accessible please"
  lives here until the post-beta sub-row work lifts attendee data into
  proper structured form.
- **Drop absolute email-uniqueness silent-success dedup.** Prior guard
  ("if a registration with this event_id + email exists, silently redirect
  to success") was too restrictive — it locked legitimate repeat
  registrations (buyer returning for another ticket) and would lock the
  upcoming multi-quantity flow the same way. Replaced with client-side
  double-submit protection (`onsubmit="if(this._busy)return false; ..."`
  + deferred button-disable). Stripe's session-id idempotency continues to
  handle the multi-charge case.
- 2 new Pest tests (`notes` passthrough + 2000-char cap); 1 existing test
  recast (`silently succeeds for duplicate event checkout` →
  `allows free-path repeat registrations from the same email`).

## Security considerations

- The `notes` field is operator-visible attendee free text. Like the
  existing `notes` column it lands as plain text (no rich-text); no
  sanitization layer needed beyond Laravel's escape-on-output. The 2000-
  char cap is the only validation gate.
- Dropping the email-uniqueness silent-success dedup widens the path to
  "same email registers twice for the same event." This was already
  possible in practice via different browsers / cleared sessions; the old
  guard was a UX nicety that became a UX blocker as soon as repeat
  legitimate purchases entered scope.
- `ticket_tier_id` request input is validated against `Rule::exists` with
  a per-event scope on all four registration controllers; cross-event
  tier-id submission is rejected before any registration row is created.
- The migration drops `events.price` and `events.capacity` columns; pre-
  beta license applies. No backward-compat shim retained.
- Pre-beta license also covers the absence of a deferred-cancellation /
  partial-refund affordance (flagged in deferred decisions below).

## Deferred decisions

- **Multi-quantity ticket purchase (option A).** Next-session work. Buyer
  picks quantities per tier (e.g. 2 General + 1 Senior) and pays in one
  Stripe session with multi-line-items. Schema: add `quantity` smallint
  (default 1) to `event_registrations`; capacity aggregate switches from
  `withCount` to `withSum`. Widget moves from radio picker to per-tier
  spinners. Importer unchanged (existing one-row-per-attendee semantics
  map to quantity=1). New stub added to `sessions/session-outlines.md`
  and Track C of `sessions/release-plan.md` (C2a). See those for
  full design notes.
- **Per-attendee data inside the registration flow.** Post-1.0. The notes
  field is the current workaround. Folded into the "Event Registration
  Depth — post-1.0" stub.
- **Partial-refund handling.** Post-1.0. Until then operators do "full
  refund + repurchase remaining." Folded into the same post-1.0 stub.
- **`EventRegistration` semantic disambiguation.** Post-1.0. Under (A) a
  row will represent a purchase line (tier × quantity), not a single
  attendee — the class name keeps fitting (still "registered for the
  event") but a rename to something purchase-shaped is worth thinking
  through once per-attendee data lands. Folded into the same post-1.0
  stub.
