# 234. Widget Primitive — Phase 5d-3 (Recent Donations — Third Concrete Record-Detail Widget)

Third concrete consumer of the record-detail Slot primitive. Adds a `donation` arm to the resolver's cardinality-many match (sibling to `note`); ships a `RecentDonations` widget that displays a contact's recent donation history on the EditContact record-detail surface; consumes session 233's new `source` column via a `donation_origin` projector field. Mirror of 5d-1 (Recent Notes) at the contract / projector / per-arm-gate / test-lock layers, with one extension: this is the first record-detail widget that surfaces an origin-aware affordance, validating the Phase A foundation.

This session also produces the **Phase 5e decision artifact** at close-gate: with three resolver arms in place (`note`, `membership`, `donation`), the per-arm `Gate::denies` pattern has enough evidence points to evaluate whether to centralize permission enforcement into the `DataContract` declaration or continue per-arm-inline.

## Design decisions to confirm before starting

The 5d-1/5d-2 design decisions (single-row vs list-shaped DTO, concept-named field map, ambient-record scoping, server-rendered Blade) all carry forward unchanged. Four new decisions specific to Recent Donations.

### Decision 1 — `donation_origin` projector field (consumes session 233's `source` column)

The Recent Donations projector includes a `donation_origin` field surfacing each row's origin as a friendly label. This is the first read-side consumer of session 233's foundation work and provides display-side evidence that origin is queryable.

| `source` raw value | `donation_origin` label |
|---|---|
| `Source::STRIPE_WEBHOOK` | `'Stripe'` |
| `Source::IMPORT` | `'Imported'` |
| `Source::HUMAN` | `'Manual'` |

Donation's `ACCEPTED_SOURCES` is `[IMPORT, STRIPE_WEBHOOK]` and HUMAN passes universally via the trait. The label map covers all three values that can land on a row; any unknown value (defensive) projects to empty string per the 226 fail-closed convention.

The template surfaces the label as a small caption next to each donation amount (or similar — exact placement is a template-level call during implementation). Confirm placement during the session.

### Decision 2 — Status filter

Donation `status` values are `pending` / `active` / `cancelled` / `past_due`. The widget needs a filter posture — show all status values, or restrict to a meaningful subset.

**Locked: filter to `status IN ('active', 'cancelled', 'past_due')`** — exclude only `pending`. Pending donations are checkout-in-flight (the public `DonationCheckoutController` creates the row before the Stripe redirect, and the row is either deleted on Stripe failure or transitions to `active` on webhook). Pending rows showing on a contact's record-detail page would be confusing UX (a half-completed donation attempt that may or may not exist as a real contribution). Cancelled and past_due remain visible because they're meaningful history.

This differs from Membership Status (5d-2) which filtered strictly to `active` because that widget shows the *current* state. Recent Donations is *history*, so the wider filter is correct.

### Decision 3 — Ordering and date column

Default order: `started_at DESC` with `created_at` as tiebreak. Whitelist `started_at` and `created_at` as user-selectable values for `order_by` (matches 5d-1's pattern of whitelisting two date-shaped columns).

`started_at` is set when status transitions to `active` and is the natural "when did this donation actually happen" timestamp. `created_at` is the row-creation timestamp (essentially the checkout-initiation time for Stripe-driven donations, or the import-batch time for imported ones). Filtering excluded `pending` rows means `started_at` is always non-null on the result set, so the COALESCE complication doesn't arise.

### Decision 4 — Field set on the projector

Seven concept-named fields:

| Field | Type | Notes |
|---|---|---|
| `donation_id` | string (UUID) | Row identity |
| `donation_amount` | string (`"50.00"`) | `(string) $d->amount` for canonical decimal shape; matches Membership Status's `membership_amount_paid` precedent |
| `donation_date` | string | `started_at` formatted per `DateFormat::MEDIUM_DATE` |
| `donation_fund_name` | string | `?->fund?->name ?? ''` — defensive null-fund handling for unrestricted donations (where `fund_id` is null). Loaded via eager-load. |
| `donation_type` | string | `'one_off'` or `'recurring'` raw value (could grow a label map later if needed; raw is fine for V1 since both labels are already user-friendly) |
| `donation_status` | string | Raw status value (`active` / `cancelled` / `past_due`) — template can stylize later |
| `donation_origin` | string | Per Decision 1 |

No `donation_frequency` (`monthly` / `annual` for recurring) in this first cut — recurring donation history is interesting but secondary to the at-a-glance "amount + date + fund + origin" surface. Defer to a future iteration if user feedback wants it.

## Phase 1 — Resolver `donation` arm + projector + ambient-record scoping + per-arm gate

1. `app/WidgetPrimitive/ContractResolver.php` — add `'donation'` arm to the cardinality-many match in `resolveSystemModel()`, sibling to `'note'`. New private `resolveDonationList(DataContract, &$cache, SlotContext)`:
   - Per-arm `Gate::denies('view_donation')` at the top → return `['items' => []]` fail-closed.
   - Ambient gate: when not `RecordDetailAmbientContext` carrying a `Contact`, return `['items' => []]`.
   - Query: `Donation::query()->where('contact_id', $contact->getKey())->whereIn('status', ['active', 'cancelled', 'past_due'])->orderBy($orderBy, $direction)->limit($limit)->with('fund')->get()`.
   - `$orderBy` whitelisted to `'started_at'` (default) or `'created_at'`. Unknown values fall back to `'started_at'`.
   - `$limit` clamped 1-50, default 5 (mirror RecentNotes).
   - Contact-scoped cache key `'donation:list:' . $contact->getKey() . ':' . $orderBy . ':' . $direction . ':' . $limit`.
   - Pass-through to `$this->systemModelProjector->projectList($contract, $cache[$key])`.
2. `app/WidgetPrimitive/Projectors/SystemModelProjector.php` — add `'donation' => fn (Donation $d) => $this->projectDonation($d)` to `projectorFor()`'s match. New private `projectDonation(Donation): array` with the seven-field map per Decision 4. The `donation_origin` mapping uses a small private helper or inline match for the `Source::*` → label translation; defensive `''` for any unknown value.
3. **Tests:**
   - `tests/Unit/WidgetPrimitive/ContractResolverDonationTest.php` — mirror of `ContractResolverNoteTest` shape:
     - empty `['items' => []]` for user lacking `view_donation`
     - empty when ambient is not record-detail (PageAmbientContext)
     - empty when ambient record is not a Contact
     - status filter (pending excluded; active/cancelled/past_due included)
     - ambient-record-scoped (Contact A's donations don't leak into Contact B's render)
     - default ordering (`started_at desc`)
     - explicit `order_by: created_at` honors the whitelist
     - unknown `order_by` value falls back to `started_at` (not SQL-injection point)
     - limit clamping (request 100 → 50, request 0 → 1)
     - resolver-is-only-path query-count pin (one donations select + one funds eager-load)
   - `tests/Unit/WidgetPrimitive/Projectors/SystemModelProjectorDonationTest.php` — mirror of the Note projector test:
     - declared-fields-only with concept-named map
     - fail-closed empty strings for undeclared fields (e.g. `stripe_subscription_id`, `external_id`, `custom_fields`)
     - `donation_origin` produces correct label for each of the three `Source::*` values
     - unknown `source` value (defensive) → empty string
     - null `fund_id` → empty string for `donation_fund_name`
     - `donation_amount` produces canonical `"50.00"` shape (string-cast decimal)

## Phase 2 — `RecentDonations` widget (definition + Blade template)

1. `app/Widgets/RecentDonations/RecentDonationsDefinition.php`:
   - `handle: 'recent_donations'`
   - `label: 'Recent Donations'`
   - `category: ['admin']`
   - `allowedSlots: ['record_detail_sidebar']`
   - `acceptedSources: [Source::HUMAN, Source::IMPORT]` — matches the RecentNotes/MembershipStatus precedent for record-detail widget instances.
   - Empty schema/defaults — no per-instance config knobs in this first cut. (A `limit` knob could land later; defer until requested.)
   - `dataContract($config)` returns SOURCE_SYSTEM_MODEL / `model: 'donation'` / `cardinality: CARDINALITY_MANY` / the seven-field whitelist / empty `filters` (the resolver derives the filter from the ambient record).
2. `app/Widgets/RecentDonations/template.blade.php` — server-rendered Blade. Mirror of RecentNotes layout:
   - Heading "Recent Donations" rendered unconditionally.
   - Empty state "No recent donations." when `$widgetData['items']` is empty.
   - For each row: amount (prefixed `$`), date, fund name (or "Unrestricted" when empty), origin label, status (small caption only when not `'active'` — i.e., highlight cancelled/past_due rows). Exact visual ordering is a template-level call during implementation; the seven fields are all available for the template to use.
   - No SCSS, no JS — public typography handles styling.
3. `app/Providers/WidgetServiceProvider.php` — register `RecentDonationsDefinition` alongside other record-detail widgets.
4. WidgetTypeSeeder auto-discovery picks up the new widget — re-run `db:seed --class=WidgetTypeSeeder` to register the `recent_donations` row.
5. **Tests:**
   - `tests/Unit/Widgets/RecentDonationsDefinitionTest.php` — 4 cases mirroring `RecentNotesDefinitionTest`:
     - allowedSlots returns `['record_detail_sidebar']`
     - acceptedSources returns the expected list
     - dataContract shape (cardinality:'many', model:'donation', fields list, empty filters)
     - empty schema/defaults

## Out of scope

- **Phase 5e — Permissions in the contract layer.** This session ships the third arm; the *decision* on whether to centralize lands at this session's close-gate as a drafting deliverable (per the base prompt's Phase 1). The actual centralization refactor is its own session if pursued.
- **Donation history filters / pagination.** This widget shows the most recent N rows (default 5, max 50). A separate "Donation History" full-pagination widget is a possible future widget; not in scope.
- **`limit` / `order_by` config knob in the widget admin UI.** The contract layer supports these as resolver inputs but the widget definition doesn't expose them as configurable per-instance yet (matches 5d-1 RecentNotes which also defers config knobs). Defer until a concrete use case justifies the config surface.
- **`donation_frequency` field on the projector** (monthly/annual labels for recurring donations). Defer; raw `donation_type` (one_off/recurring) is sufficient for V1.
- **Refund display.** Refunds live as Transaction rows with `direction: out` against the parent Donation. Not surfaced by this widget; would be its own design surface.
- **Action affordances** (refund, cancel, sync) on individual rows. Not in scope; this is read-only display. Origin-aware action gating is Phase B of the Financial Data Origin stub, deferred until those actions exist.
- **Placeholder widget removal.** Carry-forward from 5d-1 / 5d-2.
- **DashboardConfig→DashboardView retrofit.** Carry-forward, ad-hoc post-5d.
- **Themes/Templates nav merge.** Carve-out from 5c.
- **Widget help system integration.** Track-doc carry-forward; the rollup gap may become more visible with three widgets in play but lifting it out is its own design surface.
