# 167. Page Builder as Main Edit View

Restructure the page edit experience so the page builder is the primary view when editing a page, and page metadata (title, slug, status, images, SEO) lives on a separate details sub-page. Same pattern for posts.

---

## Design decisions (resolved before starting)

- The page builder already auto-saves via REST (debounced Pinia store → `PUT /widgets/{id}`). No "Save" button is needed on the builder view.
- Status is **read-only** in the builder header. To change status (draft/published), the user navigates to Edit Page Details.
- The `+ Add Block` dropdown currently in the viewport bar is removed. Adding widgets and columns happens exclusively via the `+ Widget` and `+ Columns` buttons below the preview canvas.
- The `Save changes` and `Cancel` buttons (currently rendered by `ApplyChangesButton.vue` and the Filament form footer) are removed from the builder view. `Save as Template` takes their place.
- `EditPage.php` becomes the builder view. A new `EditPageDetails.php` sub-page holds the form fields. The Filament resource registers the new route.
- Same pattern applies to `EditPost.php` / `PostResource`.

## Phase 2 — New "Edit Page Details" sub-page

Create a new Filament page for page metadata. This is the form that currently lives on `EditPage`.

### `EditPageDetails.php`

A new file at `app/Filament/Resources/PageResource/Pages/EditPageDetails.php`:

- Extends `ReadOnlyAwareEditRecord` (same base as current `EditPage`).
- Contains the full form schema currently assembled by `HasPageBuilderForm::pageBuilderFormSchema()` — the unified header fieldset (title, slug, author, status, published_at, tags, template, page type), extra title fields, images section, custom fields section, SEO section. **Does not** include the Page Builder section.
- Standard Filament save button for the metadata form.
- Header actions: the existing export page, save-as-template, and edit-snippets actions currently on `EditPage` move here.
- Delete action moves here.
- Breadcrumb: `[ListPages::getUrl() => 'Pages', EditPage::getUrl(['record' => $record]) => 'Edit Page', 'Page Details']`.

### Register the route

In `PageResource::getPages()`, add:

```php
'details' => Pages\EditPageDetails::route('/{record}/details'),
```

### Same for PostResource

Create `EditPostDetails.php` mirroring the pattern. Register in `PostResource::getPages()`.

## Phase 3 — Restructure EditPage as builder-primary view

### Strip the form from EditPage

`EditPage` no longer renders the Filament form. It becomes a thin wrapper around the page builder Livewire component. The Filament form schema on `PageResource::form()` is split:

- The page builder section moves to `EditPage` (it is the only thing `EditPage` renders).
- Everything else moves to `EditPageDetails`.

The `HasPageBuilderForm` trait may need refactoring into two methods — one for the metadata form, one for the builder section — or the resource form can be split directly. Choose the cleanest approach during the audit.

### Remove the Filament form chrome from EditPage

`EditPage` should not render the standard Filament form save/cancel footer. The page builder is the entire page content. Investigate using Filament's custom page approach or overriding the view to render only the Livewire component.

## Phase 4 — Builder header redesign

### Bootstrap data — pass page metadata to Vue

Add the following to `PageBuilder::getBootstrapData()`:

```php
'page_title'  => $page->title,
'page_author' => $page->author,  // or however the author name is resolved
'page_status' => $page->status,
'page_url'    => $pageUrl,        // the public URL, same as currently computed in EditPage header actions
'page_tags'   => $pageTags,       // array of tag names
'details_url' => $detailsUrl,     // URL to the new EditPageDetails page
```

Update `BootstrapData` in `types.ts` to include these fields. Update the Pinia store to expose them.

### EditorToolbar.vue — new header layout

Replace the current toolbar content with:

**Row 1 (left):** Page title in bold, "by" author in plain text, status in italics.
**Row 2 (left):** Page URL as a clickable link (opens in new tab, dimmed if draft), tags listed horizontally as small badges.
**Right side:** "Edit Page Details" button linking to the details sub-page URL.

Remove the block count and "Save as Template" from the toolbar — both move elsewhere (see below).

### PreviewCanvas.vue — bottom area changes

1. **Remove** the `+ Add Block` dropdown from the viewport bar (top of preview). The viewport controls (device icons, px readout) stay.
2. **Below the preview canvas**, change the existing `+ Add widget` button area to a row with two buttons:
   - `+ Widget` — opens the widget picker (same behaviour as the current `+ Add widget` button).
   - `+ Columns ▾` — dropdown with column options (2, 3, 4 columns). Same logic as the current column section in the `+ Add Block` dropdown.
3. **Block count** — add the "N block(s) on this page" text to the right of the `+ Widget` / `+ Columns` row, right-aligned.

### Footer — Save as Template

Remove `ApplyChangesButton.vue` from `InspectorPanel.vue`. In its place (or at the bottom of the builder view), render a "Save as Template" button. This dispatches the existing `open-save-template-modal` event that the Livewire component already handles. Only visible when the page has at least one widget (same condition as the current toolbar's Save as Template).
