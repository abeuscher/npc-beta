## What was built

The CRM-side surface that the Fleet Manager workstream consumes. This was the inaugural fire of the cross-repo coordination flag introduced at 237 close — by design, every artifact the flag is meant to govern lands here.

### Phase 1 — Routes, middleware, controller skeleton, config, env

- **`bootstrap/app.php`** — `withRouting()` extended with `api: __DIR__.'/../routes/api.php'` (Laravel 11's `api:` route file is opt-in; previously only `web` and `commands` were wired). The middleware-alias block grew a `'fleet.agent'` entry alongside the existing `'portal.auth'`.
- **`app/Http/Middleware/AuthenticateFleetManagerAgent.php`** — new middleware. Reads `config('fleet.agent.api_key')`, reads `$request->bearerToken()`, compares via `hash_equals()` (timing-safe). Fails closed: missing config → 500 `{"error":"misconfigured"}`; missing/empty/wrong token → 401 `{"error":"unauthorized"}`. No leakage of submitted or expected token in any response body.
- **`app/Http/Controllers/Api/Fleet/HealthController.php`** — new controller. Single `index()` action composes the JSON response. Six private subcheck methods (see Phase 2). `CONTRACT_VERSION = '1.0.0'` exposed as a class constant for cross-referencing in tests and the spec doc.
- **`config/fleet.php`** — new config file. `agent.api_key` reads `FLEET_MANAGER_AGENT_KEY` env. `agent.app_version` reads `/var/cache/app/VERSION` via `is_readable + file_get_contents + trim`, with `'dev'` fallback. The reader is a closure in the config file itself (config files are evaluated once and cached via `config:cache` in production — no service provider needed).
- **`routes/api.php`** — new route. `Route::middleware(['throttle:60,1', 'fleet.agent'])->get('/health', [HealthController::class, 'index'])`. The 60-rpm cap defends against an FM-side bug polling every second.
- **`.env.example`** — gained `FLEET_MANAGER_AGENT_KEY=` under a new "FLEET MANAGER AGENT" section, with explanatory comment per the file's existing convention. No default value — operator must set.

### Phase 2 — Six subcheck implementations

All inline as private methods on `HealthController`. Each returns the canonical four-key shape `{status, value, threshold, message}`:

- **`checkApp`** — always green; `value: 'responding'`. The fact that the method runs IS the check.
- **`checkDatabase`** — `DB::connection()->getPdo()` in try/catch. Green/`'reachable'` on success; red/`'unreachable'` on `Throwable`. `message` carries `class_basename($e)` only — never the exception message (leak risk).
- **`checkRedis`** — `Redis::ping()` in try/catch. Same shape as `checkDatabase`.
- **`checkDisk`** — `disk_free_space('/') / disk_total_space('/')` → integer percent-used. Yellow ≥80, red ≥95. Returns red with `value: null` if the read fails (suppressed via `@`-prefix to avoid noisy warnings on platforms that disallow root reads).
- **`checkLastBackupAt`** — green/null/`'backup pipeline not yet implemented'` for v1. Status is **green not yellow** — "unknown" without a backup pipeline is not an alarming state. The shape is in place; Phase 2 of the track populates it.
- **`checkVersion`** — green/`config('fleet.agent.app_version')`. Always green (a `dev` build is not a failure state).

`HealthController::index()` calls all six, composes `overallStatus` via the worst-of rule (`red` if any red; else `yellow` if any yellow; else `green`), wraps in the documented envelope.

The Phase 1 / Phase 2 split in the prompt was internal sequencing — the controller landed with full subcheck implementations in one pass since the skeleton-then-fill split was a workflow split, not a code-state split.

### Phase 3 — Version reporting via Docker build-arg

- **`Dockerfile` (`app` target)** — `ARG APP_VERSION=dev` declared at the top of the stage alongside `ARG BUILD_ENV`. Late in the stage (after the chown of storage / bootstrap_cache / scss / public_build), a single `RUN mkdir -p /var/cache/app && echo "$APP_VERSION" > /var/cache/app/VERSION`. Placement is late so `COPY . .` invalidations don't bust the version write's layer position; `/var/cache/app` is outside `/var/www/html` so the local bind mount (`.:/var/www/html` in `docker-compose.yml`) does not shadow the file at runtime.
- **`docker-compose.yml`** — both `app` and `worker` services' `build:` blocks gain `args: APP_VERSION: ${APP_VERSION:-dev}`. Local dev defaults to `dev`; `APP_VERSION=feat-x docker compose build app worker` lets manual testing exercise the non-default branch.
- **`docker-compose.prod.yml`** — untouched, as planned. It pulls pre-built images from GHCR; the version is baked into the image at CI build time.
- **`.github/workflows/deploy.yml`** — the existing "Build and push app image" step's `build-args:` list grew `APP_VERSION=${{ steps.tag.outputs.IMAGE_TAG }}` alongside the existing `BUILD_ENV=...` arg. The `IMAGE_TAG` step already extracts `${GITHUB_SHA::7}`, so the SHA-7 is what gets baked into `/var/cache/app/VERSION` on every deploy.

## Security checklist — verified

- [x] Bearer-token comparison uses `hash_equals()`, not `===` or `==`. Verified by reading `AuthenticateFleetManagerAgent::handle()` and by an automated test that greps the middleware source for `hash_equals(`.
- [x] Auth middleware fails closed on every misconfiguration scenario (missing config, empty token, wrong token, blank `Authorization` header). All four paths covered by tests.
- [x] 401 response body does not echo the submitted token or the expected token. Body is exactly `{"error":"unauthorized"}` — confirmed by `assertExactJson` in three auth-failure tests.
- [x] Rate limit middleware (`throttle:60,1`) is on the route group. Confirmed by `routes/api.php` source assertion + manual verification of `X-RateLimit-*` response headers.
- [x] No DB data, user info, session info, or stack traces appear in the `/api/health` response. The response shape is closed: only the documented top-level fields and the documented subcheck shapes, all of which are non-PII metadata.
- [x] The route is in the API middleware group (stateless), NOT the web group (no CSRF, no session cookie). Confirmed by `bootstrap/app.php` — `routes/api.php` is registered via `withRouting(api: ...)`, which Laravel 11 wires through its stateless API middleware stack.
- [x] Subcheck failure messages do not leak internal paths, exception messages, or stack details. `checkDatabase` and `checkRedis` carry `class_basename($e)` only — never `$e->getMessage()`. Disk-failure message is the static string `'unable to read disk usage'`.
- [x] `FLEET_MANAGER_AGENT_KEY` is in `.env.example` with no default value (operator must set). Empty-value semantics in dotenv mean Laravel sees `null` for the config key, and the middleware returns 500 `misconfigured` rather than allowing access.
- [x] Contract spec doc explicitly forbids reusing the API key across multiple installs in v1. The "Auth" section of `docs/fleet-manager-agent-contract.md` includes the rule and the rationale.
- [x] Portal `contact_id` scoping preserved — zero changes to portal read paths. Confirmed by `git diff` (no files under `app/Http/Controllers/Portal/` or any portal-scoped query touched).

## Production deployment notes

**Required operator step before the next deploy completes:** set `FLEET_MANAGER_AGENT_KEY` in the droplet's `.env` (`/opt/nonprofitcrm/.env`) to a fresh `Str::random(64)` value (or any 64-character random hex generated via `php -r 'echo bin2hex(random_bytes(32));'`). Without it, the auth middleware will fail-closed on every poll once the Fleet Manager workstream begins polling. The droplet's `.env` is not in source control, so this setting does not propagate via merge — it must be set by hand on the server.

**Automatic:** the next deploy after this branch merges will be the first one to bake `APP_VERSION=${GITHUB_SHA::7}` into the image via the workflow's new build-arg. `/var/cache/app/VERSION` will reflect that SHA-7 from that deploy forward; the agent endpoint's `version` field starts emitting real version strings the moment that deploy completes.

No migration. No `view:clear` or `config:clear` step beyond what the existing deploy workflow already runs.
