# 088. Image Optimization & SVG Support

Wire up the existing `spatie/laravel-medialibrary` package to handle image uploads with eager optimization, WebP conversion, responsive breakpoints, and context-aware sizing. Add SVG support as both inline markup and `<img>` source. No media browser UI — this session is pipeline only.

---

## Design decisions (resolved before starting)

- `spatie/laravel-medialibrary` is already installed (`composer.json`) along with `spatie/image` and `spatie/image-optimizer`. The config file exists at `config/media-library.php`. No models currently use it. This session wires it up — no new packages.
- Optimization happens eagerly on upload, not on first request. Variants are generated immediately and stored alongside the original.
- WebP is the target optimized format. The original file is always kept as a fallback. AVIF is out of scope for now.
- SVGs are never processed through the image optimization pipeline — they are resolution-independent and pass through as-is.
- No CDN. All images served from local `public` disk.
- No media library browser UI. Uploads continue to use Filament's `FileUpload` component. The media library operates behind the scenes.

---

## Phase 1 — Size profiles

Define a set of named size profiles that control how uploaded images are resized. Each profile specifies a max width, max height (maintain aspect ratio), and a set of responsive breakpoints for `srcset` generation.

Profiles:

| Name | Max width | Max height | Breakpoints (widths) | Use case |
|------|-----------|------------|---------------------|----------|
| `photo` | 1920 | 1080 | 1920, 1280, 960, 640, 320 | Hero images, content images, full-width photos |
| `thumbnail` | 480 | 480 | 480, 240 | Cards, list items, collection thumbnails |
| `icon` | 128 | 128 | *(none)* | Logos, icons — single size, no responsive variants |

Profiles live in a config file or a dedicated class in `app/Services/` — whichever fits the codebase conventions better. Read existing service classes before deciding.

The `icon` profile skips responsive breakpoint generation entirely. Logos and icons are small enough that a single optimized file is sufficient.

## Phase 2 — Media model integration

Add the `InteractsWithMedia` trait and `HasMedia` interface to models that have image fields. Read every model that currently uses a `FileUpload` or stores an image path before making changes.

For each model, register media collections with the appropriate size profile. Register media conversions that:

1. Resize to the profile's max dimensions (maintain aspect ratio, do not upscale).
2. Generate a WebP variant.
3. Generate responsive breakpoint variants at each width in the profile.

Conversions run eagerly (not queued) for simplicity in this first pass. If performance is a concern on large uploads, queued conversions can be revisited later.

Update the corresponding Filament form fields to use Spatie's media library integration (`SpatieMediaLibraryFileUpload`) instead of the plain `FileUpload` component. Read the Filament Spatie media library plugin documentation pattern before implementing — Filament has a first-party integration package. Check whether `filament/spatie-laravel-media-library-plugin` is already in `composer.json`; if not, install it.

---

## Phase 3 — Picture element helper

Create a Blade component or helper that renders a `<picture>` element from a Spatie media object. The component should:

- Emit a `<source>` tag for the WebP variant with `srcset` containing all responsive breakpoints and a `sizes` attribute.
- Emit an `<img>` fallback with the original format `srcset` and breakpoints.
- Accept optional `alt`, `class`, `loading` (default: `lazy`), and `width`/`height` attributes.
- Handle the case where no WebP conversion exists (e.g. SVGs, GIFs) gracefully — fall back to `<img>` only.

Use this component in every public-facing view that renders an uploaded image. Admin-side previews can continue using simple `<img>` tags.

---

## Phase 4 — SVG support

SVGs require two rendering modes:

**Inline SVG** — read the file contents and emit them directly in the HTML. This allows CSS styling of SVG internals (fill, stroke, dimensions). Used for icons, logos, and any SVG where the user or theme needs to control appearance.

- Sanitize SVG content before inlining: strip `<script>` tags, `on*` event attributes, `<foreignObject>`, `xlink:href` pointing to `javascript:` or `data:` URIs, and any `<use>` elements referencing external URLs.
- Create a sanitization utility in `app/Services/` (or add to an existing service) that processes the raw SVG string. Do not trust uploaded SVGs blindly.
- Inline SVGs skip the image optimization pipeline entirely.

**Image source SVG** — render as `<img src="path.svg">`. No `<picture>` wrapper needed — SVGs are resolution-independent. Used when inline styling is not required and the SVG is purely decorative or complex.

Update the `FileUpload` fields that accept SVGs (site logo, admin logo, email template header) to detect SVG uploads and skip the optimization pipeline. SVGs should be stored as-is without conversion.

The picture element helper from Phase 3 should detect SVG media and render a plain `<img>` tag instead of a `<picture>` element.

---

## Phase 5 — Existing upload migration

Audit every existing image upload field and storage path in the application. For each:

1. Determine the correct size profile (`photo`, `thumbnail`, or `icon`).
2. Wire it into the media library with the appropriate collection and conversions.
3. Update the corresponding public-facing views to use the picture element helper.

Existing uploaded files on disk will not have generated variants. The picture element helper must handle this gracefully — if no WebP or responsive variants exist for a given image, render a plain `<img>` tag with the original file. Do not fail or show broken images.

Write an artisan command (`php artisan media:regenerate` or similar — check if Spatie provides one) that can be run manually to generate variants for existing uploads. Document this in the deployment notes.

---

## Phase 6 — Widget image fields

Add an `image` field type to the page builder `config_schema` system, following the same pattern as the existing `select` type added in session 087. An image config field uses Spatie's media library upload with the appropriate size profile.

Update widget templates that render config images to use the picture element helper.

This phase depends on Phase 2 and Phase 3 being complete. Read `PageBuilderInspector.php` and the config schema rendering logic before implementing.

## Security checklist

Before closing:

- [ ] SVG sanitization strips all executable content: `<script>`, `on*` attributes, `javascript:` URIs, `<foreignObject>`, external `<use>` references.
- [ ] Uploaded SVGs are validated as valid XML before storage — reject malformed files.
- [ ] Image uploads validate MIME type server-side (not just extension) — reject files that claim to be images but are not.
- [ ] No user-supplied filenames are used in rendered HTML — all paths come from the media library's generated names.
- [ ] The picture element helper escapes all attribute values via Blade `{{ }}`.
