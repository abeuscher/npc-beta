# 373. C3b — Auto Tax Receipt Email

Wire an **automatic per-gift acknowledgment email** into the successful-donation path so every donation is provably acknowledged the moment it clears Stripe — the money-path piece that C4's donation-to-acknowledgment rehearsal depends on. This is a **wiring-an-existing-seam** session, not a greenfield build: the receipt template, Mailable, `DonationReceipt` model, and a manual annual-statement sender already exist; today the donation webhook records the gift and sends nothing.

## Session-specific deltas — the drift the plan entry doesn't capture

The plan calls C3b a "feature-half prerequisite stub." The code is **further along than that framing**, so the real work is smaller and more precise than "build a receipt email." Verify all of this at session start (drift rule), but this is the map as of 372:

- **`StripeWebhookController::handleDonationCheckout()` is the seam.** On `checkout.session.completed` with a `donation_id`, it updates the `Donation`, records a `Transaction`, and returns — **it sends no email.** That is the gap C3b fills. (Recurring donation renewals land in `handleInvoicePaymentSucceeded()`; membership/product/event checkouts are sibling branches — out of scope here.)
- **A receipt Mailable + template already exist, but for a *different document*.** `app/Mail/DonationReceipt.php` + the `donation_receipt` `EmailTemplate` handle + `app/Filament/Pages/DonorsPage.php` ("Giving Summary") together are an **annual, aggregated, year-end tax statement** — a donor's whole year of giving, grouped by fund, keyed on `(contact_id, tax_year)`, sent manually by an admin. C3b's target is a **per-gift acknowledgment** (one donation: amount, date, transaction id, fund, EIN, IRS contemporaneous-acknowledgment language) sent **automatically** on each successful charge. These are two different artifacts — do not conflate them.
- **The manual annual path stays.** The plan says today's manual "Send Receipts" admin action "stays as a backfill / one-off resend affordance." Confirmed — the `DonorsPage` annual flow is untouched by this session; C3b **adds** the automatic per-gift path alongside it.
- **Idempotency is load-bearing here.** Stripe retries webhooks; the handler already guards `Transaction` writes by `stripe_id`. The auto-acknowledgment needs its **own** per-gift "already sent" guard so a webhook replay never double-emails a donor. The existing `DonationReceipt` model is keyed on `(contact_id, tax_year)` — the wrong grain for a per-gift marker.

Framing: this is the `feedback_audit_integration_seams` pattern — an existing seam (`handleDonationCheckout`) wired one step short. Default hypothesis is "add the acknowledgment dispatch + a per-gift idempotency marker + a per-gift template," not "build a receipt system."

## Open questions to resolve at session start (Open Gate)

These are the architectural calls — surface a recommendation and confirm before implementing:

1. **Per-gift acknowledgment vs. the annual statement — new template or adapt the existing one.** Recommendation: a **new per-gift path** — a new `EmailTemplate` handle (e.g. `donation_acknowledgment`) with single-gift tokens (donor name, amount, date, transaction id, fund, org EIN, IRS "no goods or services were provided…" language), dispatched from the webhook, reusing the existing `EmailTemplate` render/wrapper machinery. Leave the annual `donation_receipt` statement + `DonorsPage` flow as-is (different document, different purpose). Confirm the handle name and whether a new Mailable or a parameterized one.
2. **Idempotency key.** Recommendation: a per-donation marker — an `acknowledged_at` (nullable timestamp) column on `donations`, checked-and-set inside the webhook transaction, so a replay is a no-op. (Alternative: a per-gift `DonationReceipt`-style row keyed on `donation_id`/`stripe_id`.) **This is likely the session's one migration** — confirm the approach before writing it.
3. **Recurring donations in scope?** The initial gift is `checkout.session.completed`; renewals are `invoice.payment_succeeded`. Each successful charge is a separate deductible contribution. Recommendation: acknowledge **both** the initial gift and each renewal (same template, per-charge idempotency) — but if that widens the session past one context, scope to the **initial gift** now and note renewals as a fast-follow. Confirm the cut.
4. **EIN / org tax-id source + IRS language.** Recommendation: read the org EIN from a Finance `SiteSetting` (verify whether one exists; if not, add it to the Finance settings surface). The IRS contemporaneous-acknowledgment language ships as the new template's default body copy, admin-editable via `manage_email_templates`. Confirm.
5. **Dispatch mechanism.** The Mailable is `Queueable` and a `worker` container runs. Recommendation: queue the send (`Mail::to(...)->queue(...)`) so a mail-provider hiccup never fails the webhook (which must return 200 fast). Confirm sync-vs-queue.
