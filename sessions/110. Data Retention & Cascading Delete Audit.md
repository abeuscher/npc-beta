# 110. Data Retention & Cascading Delete Audit

Audit every deletion path and FK cascade rule in the database. Define what happens when parent records are deleted, fix any incorrect or missing onDelete rules, and produce a written retention policy document. This session builds on the deletion guards from Session 109.

---

## Design decisions (resolved before starting)

- This is a forensic audit session — read the schema dump and every migration before writing code.
- Cascade rules should be explicit and intentional. No `CASCADE` on delete unless the child record is meaningless without the parent.
- Soft-deleted records are not purged automatically. Define a purge policy but do not implement automated purging — purge is a manual super_admin action for now.
- The retention policy document is a developer-facing reference, not a legal document.

---

## Before writing any code

Read the following files:

- `docs/schema.md` — the full schema reference
- `database/schema/pgsql-schema.sql` — the actual schema dump, which has the definitive FK constraints
- All migration files in `database/migrations/` written after the last schema dump
- `app/Models/` — scan every model for `SoftDeletes`, relationships, and `boot` / observer hooks that affect deletion
- Session 109 log — understand which deletion guards are already in place

---

## Phase 1 — FK cascade audit

For every foreign key in the database, document and verify the `onDelete` behaviour. Produce a table of every FK with:

- Parent table → child table
- Current onDelete rule (CASCADE, SET NULL, RESTRICT, NO ACTION)
- Correct onDelete rule (with rationale)
- Action needed (none, migration to change)

Write migrations to fix any incorrect rules. Categories:

### When a **User** is deleted
- Notes authored by the user — what happens?
- Tags created by the user — what happens?
- Import sessions owned by the user — what happens?
- Pages authored by the user — what happens?
- Activity log entries with the user as actor — what happens?
- Templates created by the user — what happens?

### When a **Contact** is deleted (soft)
- Event registrations linked to the contact
- Memberships linked to the contact
- Mailing list memberships
- Donations linked to the contact
- Portal account linked to the contact
- Household linkage (household_id self-reference)
- Activity log entries for the contact

### When an **Import Session** is deleted
- Contacts created by that import session (import_session_id FK)
- Import review records

### Other parent deletions to audit
- Page deleted → page widgets
- Event deleted → event registrations
- Collection deleted → collection items
- Form deleted → form submissions
- Widget Type deleted → page widgets referencing it
- Template deleted → pages referencing it
- Navigation Menu deleted → navigation items
- Fund deleted → donations referencing it
- Membership Tier deleted → memberships referencing it

---

## Phase 2 — Soft-delete purge policy

Define the rules for when soft-deleted records can be permanently purged:

- Which models support soft delete?
- What is the minimum retention period before purge is allowed?
- What dependencies must be cleared before purge?
- Who can trigger a purge? (super_admin only)

Do not implement automated purging. Document the policy and add a note about future implementation (artisan command or admin action).

---

## Phase 3 — Retention policy document

Write `docs/data-retention-policy.md` covering:

- Soft-delete behaviour and retention periods per model
- Cascade rules summary (what happens when each parent type is deleted)
- Purge policy and prerequisites
- Financial record immutability rules (donations, transactions never deleted)
- Contact PII handling on deletion

- GDPR/CCPA-specific compliance features
- UI for viewing or managing soft-deleted records beyond what Filament's TrashedFilter provides
