## Production deployment notes

- **No migration to run.** Standard deploy pipeline covers the change without `php artisan migrate` doing anything new.
- **No env-var changes.**
- **The `installation_completed_at` SiteSetting key is auto-created on first write.** Existing installs that never run `db:seed` post-deploy will see the row appear the first time the operator clicks "Mark setup complete." `SiteSetting::get('installation_completed_at')` returns null for missing rows, which is the correct first-run semantic.
- **Existing super-admin DashboardViews will NOT auto-pick-up the widget.** `DashboardViewSeeder`'s "skip if any widgets exist" guard preserves operator customizations, which means an operator who already has a configured dashboard sees no change post-deploy. Three remediation paths: (a) operator adds the widget manually via Tools → Dashboard View; (b) operator runs `app:reset` to wipe and re-seed; (c) operator runs `db:seed --class=DashboardViewSeeder` after first deleting their super-admin view's pageWidgets. Surfaced to the user during local visual testing; agreed to defer the production-rollout fix (idempotent backfill in seeder, OR one-off data migration) to a future session — there are no live customer installs yet (A3 hasn't shipped). Worth fixing before A3.
