## Session-specific deltas

The seam was mapped at 358 close prep. Confirm against the code, but the starting facts:

- **Greenfield 2FA.** No `laravel/fortify`, no `pragmarx/google2fa`, no 2FA columns on `users` (the table has only `id, name, email, email_verified_at, password, remember_token, is_active, timestamps`). Filament **3.2** (`composer.json`) — its stock `->login()` has **no native 2FA** (that's a Filament v4 feature). So this is a build, not a config.
- **Admin auth:** Filament panel at `App\Providers\Filament\AdminPanelProvider.php` (stock login, `web` guard, `Authenticate` middleware). `User` model uses Spatie `HasRoles`; `canAccessPanel()` gates only on `is_active`. There is **no existing post-login enforcement gate** to mirror — the invitation flow is token-based, not a post-login redirect — so the enrollment gate is a new middleware in the panel stack.
- **FM API path confirmed unaffected.** `routes/api.php` (`/api/health`, `/api/logs`, `/api/backup/*`) carries **only `throttle`** middleware — no session/web guard. 2FA is a session-state, post-login concern, so it **structurally cannot touch** these stateless endpoints. The success-criterion clause is satisfied by construction; add a guard test asserting the API routes stay reachable, and do not add 2FA middleware to the API group.

## Open questions to resolve at session start

These are the architectural choices that propagate — surface them at the Open Gate, don't commit-and-unwind.

1. **2FA backend (the load-bearing dependency choice).** Candidates:
   - **(a) Reuse Laravel Fortify's 2FA *primitives* without mounting its routes/views** — the `two_factor_secret` / `two_factor_recovery_codes` / `two_factor_confirmed_at` column shape, `TwoFactorAuthenticationProvider`, recovery-code generation — wiring enrollment + challenge into Filament's flow ourselves. Battle-tested crypto + column shapes, minimal new surface.
   - **(b) A Filament-v3-specific 2FA package** (e.g. a community plugin). Faster UI, but adds a dependency whose lifecycle we don't control.
   - **(c) Hand-rolled with `pragmarx/google2fa` + `bacon/bacon-qr-code`.** Most control, most code.
   - **Recommended lean: (a)** — standard column shapes and recovery logic, no framework-fight, since we drive the Filament UI anyway. Confirm with owner; it's a composer dependency + auth-flow decision.
2. **Enforcement-gate mechanism.** Recommended: **a single post-login middleware** on the admin panel stack (after `AuthenticateSession`) that redirects un-enrolled / un-challenged users to the enrollment / challenge page. One place, covers every panel entry point. (Alternative: per-page `canAccess()` overrides — rejected, leaks coverage.)
3. **Scope of "admin users."** The success criterion says login requires a second factor for admin users with one-time enrollment on next login. Recommended: **uniform mandatory for all real panel users** (it's the item an external auditor flags first). Per-role granularity (e.g. "only super_admin must") is a later refinement, out of scope.

## The demo-mode exemption — hard constraint (do not ship without it)

The owner flagged this explicitly at 358 close: the demo server's frictionless single-button entry must not get trapped behind a 2FA gate it can't answer. Spell it into the design from the start.

- **Predicate:** `isDemoMode()` — defined in `app/helpers.php` as `app()->environment('demo')`. The demo server runs `APP_ENV=demo`; a real install never does, so this is the **clean, non-leaky** exemption (no risk of weakening 2FA on a production install).
- **Where it bites:** the demo auto-login (`App\Http\Controllers\DemoLoginController` → `/demo/enter`) does `Auth::login()` on the shared `demo@demo.local` user (role `demo`) and redirects straight to the admin panel — it **bypasses the password login form**, so it never hits a *login-form* challenge. The danger is the **post-login enrollment-enforcement middleware**: it would catch the demo user on the redirect. **That gate must short-circuit when `isDemoMode()` is true** — no enrollment required, no challenge, demo entry stays one click.
- **Defense in depth:** optionally also skip the gate for the `demo` role, but `isDemoMode()` is the load-bearing check (the demo role is only seeded when `isDemoMode()`).
- **No state accumulation:** the demo user is recreated daily by `demo:reset`; it must never acquire 2FA columns/secret. The `isDemoMode()` short-circuit in the gate handles this — don't write 2FA state for demo logins.
- **Verify it:** an explicit test (below) that, with `isDemoMode()` true, `/demo/enter` lands on the dashboard with no enrollment redirect and the demo user carries no 2FA secret.

## The test-suite interaction — plan it, don't discover it

A *mandatory* post-login gate will redirect **every existing Filament feature test that does `actingAs($adminUser)` and expects a dashboard/resource page** straight to the enrollment screen — silently reds a large slice of the suite. This must be designed for, not patched after:

- Decide the **test-context posture** at session start (it's a sibling of the demo exemption — another context where the gate must not fire). Options: (i) exempt the gate in the `testing` environment by default and add *targeted* tests that exercise the gate explicitly with the testing-exemption lifted; (ii) make factories/`actingAs` helpers mint a **2FA-confirmed** admin by default so the gate is already satisfied; (iii) a `withTwoFactorConfirmed()` test helper applied where needed. Recommended: **(i) + targeted opt-in** — least churn, keeps the existing suite green, and the 2FA behaviour is proven by the dedicated tests rather than by every unrelated test.
- Whatever the choice, the **full fast suite must stay green** at session end; treat a wave of `actingAs`-redirect failures as expected-and-handled, not as a surprise.

## Phases

### Phase 1 — 2FA backend + schema
Add the 2FA columns to `users` via a migration (`two_factor_secret`, `two_factor_recovery_codes`, `two_factor_confirmed_at` — encrypted secret/codes per the chosen backend). Wire the TOTP provider + recovery-code generation + verification. Update `docs/schema/` for the `users` table. Run the migration via Docker.

### Phase 2 — Enrollment + challenge wired into Filament
A one-time enrollment page (QR + secret, recovery codes shown once, confirm with a live TOTP code before `two_factor_confirmed_at` is set) and a login-challenge step (TOTP or recovery code) integrated with Filament's auth flow.

### Phase 3 — Enforcement gate + demo exemption + FM-path guard
The post-login middleware on the admin panel stack: un-enrolled users → enrollment, enrolled-but-unverified-session → challenge, **`isDemoMode()` → bypass entirely**, and the chosen testing-context posture. Confirm (test) the FM `/api/*` routes carry no 2FA and stay reachable.

### Phase 4 — Help doc + tests + suite-green
Help-doc entry on enrollment (`resources/docs/…` + route registration per convention). Full test coverage (below). Bring the full fast suite back to green.

## Drift signals (stop and surface)

- The chosen 2FA backend forces mounting Fortify's full web routes/views (colliding with Filament's stock login) — surface before fighting the framework; pick the package or hand-rolled path instead.
- `isDemoMode()` turns out **not** to be the sole demo predicate (e.g. an install runs demo data without `APP_ENV=demo`) — surface before relying on it as the exemption.
- Enforcing 2FA on all roles breaks a seeded/automated login path beyond the planned `actingAs` test interaction (e.g. the invitation-accept flow, or a console path) — surface; the gate's exemption set may need one more entry.
- The migration can't add encrypted columns cleanly against the post-squash schema baseline — surface before improvising schema shape.
