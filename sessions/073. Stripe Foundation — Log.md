## What was built

### Phase 1 — API key management

- Added `encrypt`/`decrypt` support to `SiteSetting`: rows with `type = 'encrypted'` are encrypted on write via `Crypt::encryptString()` and decrypted on read via `Crypt::decryptString()` (AES-256-CBC, keyed from `APP_KEY`). Legacy plaintext values are handled gracefully.
- Migration: renamed `stripe_api_key` → `stripe_secret_key` (type: encrypted); inserted `stripe_publishable_key` (type: string) and `stripe_webhook_secret` (type: encrypted); set `quickbooks_api_key` type to encrypted.
- Updated `AppServiceProvider` to load all three Stripe keys and the QuickBooks key from `SiteSetting` into `config('services.stripe.*')` and `config('services.quickbooks.*')`.
- Built `FinanceSettingsPage` (gated to `super_admin` and `manage_financial_settings` permission):
  - Publishable key: standard text input, saved via the Save button.
  - Secret key and webhook secret: each has a read-only status placeholder (••••) and a gated action button:
    - **Set Key** (first time): modal with non-retrievability warning, password input, `autocomplete="new-password"`.
    - **Change Key** (rotation): danger modal with system-impact warning, password input, and a required confirmation checkbox.
  - Key set/rotation events written to the activity log.
  - `preservePublishableKey()` prevents the publishable key from being lost when a secret key action triggers a redirect.
- Added `manage_financial_settings` permission to `PermissionSeeder`; assigned to `treasurer` role.
- Removed Stripe and QuickBooks key fields from `GeneralSettingsPage`.
- Added `autocomplete="new-password"` to Resend API key, Mailchimp API key, and Mailchimp webhook secret fields in `MailSettingsPage`.

### Phase 2 — Transaction table migration

- Migration: dropped `donation_id` FK and column; added `subject_type` (string, nullable) and `subject_id` (string, nullable) with a composite index.
- Updated `Transaction` model: replaced `donation()` BelongsTo with `subject()` MorphTo; updated `$fillable`.
- Updated `Donation` model: replaced `HasMany transactions()` with `MorphMany`.
- Updated `TransactionFactory` to use polymorphic fields.

### Phase 3 — Webhook receiver

- Created `StripeWebhookController` handling three event types:
  - `checkout.session.completed` → creates a `payment/in/completed` Transaction.
  - `payment_intent.payment_failed` → creates a `payment/in/failed` Transaction.
  - `refund.created` → creates a `refund/out/completed` Transaction; inherits subject from the original payment transaction by matching `payment_intent` ID.
- Signature verification via `\Stripe\Webhook::constructEvent()` — returns 400 on failure.
- Idempotency check on `stripe_id` before writing any record.
- Outer try/catch returns 200 on any unhandled exception to prevent Stripe retry storms.
- Registered `POST /webhooks/stripe` in `routes/web.php`; covered by existing `/webhooks/*` CSRF exemption.
- Stripe PHP SDK already available via `laravel/cashier` (v16.6.0, PHP 8.4 compatible) — no separate install needed.

### Phase 4 — Transaction resource scoping

- Scoped the `TransactionResource` manual entry form to off-system transactions only: Grant, Expense, Adjustment types.
- Removed Stripe fields from the form; status options reduced to Pending/Cleared; status default set to `pending`.
- Source badge added to the table (Stripe / Manual based on presence of `stripe_id`).
- Edit/Delete actions hidden on Stripe-originated rows.
- Rewritten help article at `resources/docs/transactions.md`.

### Phase 5 — Session 074 prompt and admin help

- Wrote `sessions/074. Products & Checkout.md` — full prompt for the Products & Checkout session.
- Rewrote `resources/docs/settings-finance.md` to document Stripe key setup, restricted key permissions, webhook event selection, and the gated key entry UX.
- Updated `docs/schema.md` for both migrations.

## Security checklist — verified

- [x] Webhook signature verified on every request before any processing — unsigned requests return 400.
- [x] Stripe secret key and webhook secret are encrypted at rest (AES-256-CBC); never stored or logged in plaintext.
- [x] No card data, billing address, or other payment instrument PII from the Stripe payload is stored.
- [x] The application holds no Stripe permission that allows it to initiate a refund, payout, or balance transfer (enforced at the restricted key scope level).
- [x] Idempotency check prevents duplicate transaction records on repeated webhook delivery.
- [x] Webhook endpoint is CSRF-exempt and does not require authentication, but is verified by signature.
- [x] All API key / secret text inputs use `autocomplete="new-password"` to suppress browser credential-save prompts.

## Stripe restricted key permissions

The live restricted key (`rk_live_...`) is scoped to:

| Section | Resource | Permission |
|---|---|---|
| Core | Charges and Refunds | Read |
| Core | Customers | Write |
| Core | Payment Intents | Read |
| Core | Products | Write |
| Billing | Invoices | Read |
| Billing | Prices | Write |
| Billing | Subscriptions | Write |
| Checkout | Checkout Sessions | Write |

All other permissions are set to None. Payouts, Refunds (write), Transfers, and Balance are explicitly excluded.
