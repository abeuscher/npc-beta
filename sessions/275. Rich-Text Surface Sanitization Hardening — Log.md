### Iteration `/1` — `App\Support\HtmlSanitizer` utility + allow-list test suite (`8696ee0`)

DOMDocument-based walker, no Composer dep. Quill-shape allow-list covers every default-toolbar shape plus the heroicon blot. **Architectural deviation from spec at session start** (ratified before any code shipped): `<img>` retained in the allow-list because Quill's inline-image flow stores plain `<img src="/storage/...">` in DB and `InlineImageRenderer::process()` upgrades to `<picture>` at render time on three surfaces (PageWidget config richtext, Event description/meeting_details, EmailTemplate body). Stripping `<img>` would silently destroy inline images on next-edit. `src` attribute scheme-validated to http(s)/relative; `data:`/`javascript:`/`vbscript:` rejected.

Allow-list:
- **Tags:** p, br, strong, em, u, s, ol, ul, li, blockquote, pre, code, h1–h6, a, span, img + svg/path inside `span.ql-heroicon` context only.
- **Classes:** initially restricted to `ql-*`; relaxed at iteration `/2` close (see below).
- **href on a:** http(s)/mailto/relative; control-char strip defeats tab/newline-encoded `javascript:` schemes.
- **src on img:** http(s)/relative; same control-char strip.
- **style:** retained on img only (InlineImageRenderer reads it for user-applied resize CSS).
- **on\* event handlers, id, name:** stripped on every element.
- **Disposition:** disallowed tags drop tag, preserve inner text. script/style/iframe/object/embed/noscript drop tag AND content (void-drop).

Test surface: **71 cases** at `tests/Feature/Support/HtmlSanitizerTest.php` across the three spec categories (round-trip-clean / strip-disallowed / XSS-payload neutralisation) plus a few extras (Unicode round-trip, idempotency, multi-vector single-document neutralisation).

### Iteration `/2` — Apply at six save boundaries + sanitiser relax (`58fc5b7` … `6e0f142`)

Six commits, one per boundary, each landing the apply + companion regression-guard test:

1. **`Note.body`** (`58fc5b7`) — `setBodyAttribute` mutator.
2. **`Event.description` + `Event.meeting_details`** (`420b898`) — two mutators on Event.
3. **`EmailTemplate.body`** (`36c3c61`) — `setBodyAttribute` mutator + token-restoration post-process pass that decodes `%7B%7B...%7D%7D` back to `{{...}}` inside the sanitiser output (DOMDocument percent-encodes braces in URL attribute values; the seeded password-reset / admin-invitation templates rely on `<a href="{{reset_url}}">` patterns surviving for `replaceTokens` at render time).
4. **`SiteSetting` rich-text keys** (`4a86887`) — three keys (`dashboard_welcome`, `system_page_content_reset_password`, `system_page_content_email_verify`) declared via `SiteSetting::RICH_TEXT_KEYS` and sanitised at the `SiteSetting::set()` boundary.
5. **`PageWidget.config` richtext fields** (`8ae554c`) — `PageWidget::booted()` saving hook walks the related `WidgetType.config_schema`, finds every field with `type=richtext`, and sanitises the matching key in config.
6. **Cross-cutting `custom_fields` rich_text** (`c50b691`) — new `App\Models\Concerns\SanitisesRichTextCustomFields` trait hooks the saving event, queries `CustomFieldDef` for `rich_text`-typed fields matching the model class, runs `HtmlSanitizer` over each matching key in `custom_fields`. Applied to all 8 models with a `custom_fields` column: Contact, Donation, Event, EventRegistration, Membership, Organization, Page, Transaction.

Then **iteration `/2` relax commit** (`6e0f142`) — surfaced when the full fast suite hit 8 failures across BlogListing / EventsListing / QuerySettings / WidgetTokenNamespace tests. Operator-authored widget `content_template` fields ship structural HTML5 tags (`<article class="card">`, `<section>`) and arbitrary classes (`class="meta"`, `class="card"`) that the strict spec'd allow-list stripped, breaking widget rendering. **Architectural call ratified mid-iteration:** relax the class rule to allow any class token (still strip when zero tokens) and broaden the tag allow-list to include common structural HTML5 tags (div, section, article, header, footer, nav, aside, main, figure/figcaption, hr, table family, dl/dt/dd, semantic inline tags, details/summary). None introduce JS-execution surface; the JS-execution defence (`script`/`on*`/`style`/`javascript:`/`iframe` strip) stays unchanged.

### Iteration `/3` — Memos Trix→Quill convergence + data migration (`e21134d`)

Four-part landing for the only Trix surface left in the app:

1. **Editor swap.** `Collection::getFormSchema()` at the `rich_text` branch now returns `App\Forms\Components\QuillEditor::make(...)` instead of Filament's native `RichEditor` (Trix). Every custom collection with a `rich_text`-typed field — Memos and any future custom collection — now uses Quill.
2. **Boundary application.** `CollectionItem::booted()` registers a saving hook that walks the parent Collection's `rich_text`-typed fields and runs `HtmlSanitizer` over each matching key in `data`.
3. **Data migration.** New `App\Support\TrixToQuillConverter` utility (DOMDocument-based, idempotent fast-path on already-Quill data) plus migration `2026_05_10_120000_convert_collection_items_trix_to_quill.php` that walks every Collection with `rich_text`-typed schema fields, converts each `CollectionItem.data[key]` from Trix shape to Quill shape, and defence-in-depths the result through `HtmlSanitizer` on save.
4. **Playwright spec.** `tests/e2e/memos/memos-trix-to-quill.spec.ts` covers the create-modal mounting `QuillEditor` (proves the swap; Trix not present) and the end-to-end save round-trip via the admin UI (proves the boundary wired up correctly through the live form).

Tests added: 19 new fast Pest (12 converter + 7 boundary/migration); 2 new Playwright. New helpers in `tests/e2e/helpers/db.ts`: `findCollectionIdByHandle`, `findCollectionItemByTitle`, `deleteCollectionItem`.

### Iteration `/4` — Import boundary sanitisation + custom_fields trait fix (`4a69adb`)

Two surfaces:

1. **`ContentImporter::sanitizeWidgetConfig` extension.** The method now walks every `richtext`-typed schema field (alongside its existing image/video media-id clearing pass) and runs `HtmlSanitizer` over each matching value before the `PageWidget` create. Companion test at `tests/Feature/ContentImporterRichTextSanitizationTest.php` exercises a bundle with both script-injection and `javascript:` href / `iframe` payloads and asserts both stripped cleanly.
2. **Bug fix:** `SanitisesRichTextCustomFields` trait was querying with FQCN (`$model::class` = `App\Models\Contact`) but the codebase convention for `CustomFieldDef.model_type` is lowercase short names (`'contact'`, `'event'`, `'page'` per `CustomFieldDefResource:39-42`). The trait's saving hook silently no-op'd in production. Fixed to use `Str::snake(class_basename($model::class))`. The /2 test that constructed defs with FQCN was passing despite the production gap; updated to use the production form (`'contact'`).
3. **Companion end-to-end Contacts importer test** at `tests/Feature/ContactsImporterRichTextSanitizationTest.php` (representative of all 7 importers per spec) drives a CSV row with a `rich_text` custom field carrying a script payload through `ImportProgressPage::tick()` and asserts the imported Contact's `custom_fields[bio]` has the script + `javascript:` href stripped.

**Skipped per architectural reasoning:** the per-importer pre-save helper described in the spec (4b shared trait method on `InteractsWithImportProgress`). The model-layer `SanitisesRichTextCustomFields` trait covers all 7 importers transitively — every `Contact::create` / `Model::update` path through the importers fires the trait's saving hook (no importer uses `DB::insert()` or `saveQuietly()` to bypass model events). Adding redundant pre-save calls in 7 importer pages would not change the security posture. Documented in the `4a69adb` commit body.
