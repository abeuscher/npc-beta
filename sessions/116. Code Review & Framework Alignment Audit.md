# 116. Code Review & Framework Alignment Audit

Codebase-wide review with two goals: (1) standard cleanup — orphaned code, dead imports, duplicated logic, inconsistent patterns; (2) a systematic audit for places where custom code reimplements or overrides something the framework already handles, and where doing so displaces the framework's built-in behaviour rather than extending it.

---

## Design decisions (resolved before starting)

- This is an audit-only session. No new features.
- The framework alignment audit produces a findings table, not fixes. Every flagged item is presented for discussion before any code is changed.
- Standard cleanup items (dead imports, orphaned files, duplicated blocks) can be fixed in place if the fix is clearly safe. Anything uncertain gets flagged for discussion.
- "Framework" means Laravel, Filament 3, Livewire, and the Spatie packages this project depends on (permissions, media-library, activitylog, sluggable). All are in scope.

## Workstream 1 — Framework alignment audit

Examine the codebase for places where custom code does something the framework already provides, and where the custom implementation displaces or conflicts with the framework's built-in behaviour. The diagnostic question for each candidate is:

> "Is this code doing something the framework already does? If so, are we working with the framework or around it?"

Not every custom implementation is wrong. Business logic the framework cannot know about (e.g. hiding system pages from deletion, protecting certain record types) is legitimate. The audit targets cases where:

- A **policy method exists** but the same authorization check is also expressed as a manual `->visible()`, `->hidden()`, `->disabled()`, or static method override on the resource — displacing the framework's automatic policy consultation.
- A **Filament component's built-in behaviour** (e.g. action visibility tied to record state, automatic validation, relationship handling) is overridden with manual logic that reimplements the same thing less completely.
- A **Laravel feature** (form requests, middleware, model events, route model binding, built-in validation rules) is bypassed in favour of hand-rolled equivalents in controllers or services.
- A **Spatie package feature** (permission wildcards, media conversions config, slug generation) is reimplemented rather than configured.
- **Custom query scopes** duplicate what a Filament filter, Laravel scope, or Eloquent global scope already provides.

For each finding, record:

| File | Line(s) | What the code does | What the framework provides | Risk if left as-is | Suggested approach |
|------|---------|-------------------|---------------------------|--------------------|--------------------|

Do not fix anything in this workstream. Present the table for discussion.

### What this audit is NOT

This is not a search for code that is merely unconventional, verbose, or could be written differently. It specifically targets cases where custom code **displaces** framework behaviour — where removing the custom code and relying on the framework would produce the same or better result. If the custom code adds behaviour the framework cannot provide, it is not a finding.

## Workstream 2 — Stale reference audit

Grep the codebase for references to classes, files, settings keys, or routes that have been renamed, moved, or deleted. These accumulate across sessions and cause runtime errors or silent failures when hit.

- **Moved classes** — session 115 moved `ChromeRenderer`, `InlineImageRenderer`, `SvgSanitizer`, and `ImageSizeProfile` from `App\Services` to `App\Services\Media`. Grep for any remaining `App\Services\ChromeRenderer` (etc.) references outside `storage/framework/views/` (cached views will recompile).
- **Deleted schema path** — session 115 replaced `docs/schema.md` with `docs/schema/`. Grep for `docs/schema.md` in live code and config (session logs are exempt — they're historical).
- **Retired SiteSetting keys** — the template system (sessions 099–100) retired several site setting keys (`public_primary_color`, `public_heading_font`, `public_body_font`, `header_bg_color`, `footer_bg_color`, `nav_link_color`, `nav_hover_color`, `nav_active_color`, `theme_scss`). Grep for any remaining reads or writes in live code.
- **Deleted classes** — grep for `SiteThemePage` in live code. The class was deleted in the template migration.
- **General sweep** — grep for any `use` statement that imports a class which does not exist on disk.

Fix confirmed stale references in place.

## Workstream 3 — Widget copy & serialization consistency

Multiple places serialize or copy widget stacks:

1. `PageBuilder::serializeWidgetStack()` — Save as Template from the page builder
2. `EditPage::serializeWidgetStack()` — Save Block Layout as Template from the edit page menu
3. `CreatePage::hydrateWidgets()` — Hydrate widgets from a content template definition
4. `ListTemplates::copyWidgets()` / `EditPageTemplate::copyWidgets()` — Copy widgets for "New From Default" and custom header/footer creation

Audit these for:

- **Field coverage** — do all copy/serialize paths handle the same set of `PageWidget` fields? Specifically check: `label`, `config`, `query_config`, `style_config`, `column_index`, `sort_order`, `is_active`. Flag any path that drops a field.
- **Nested widget handling** — do all paths correctly handle column widget children (recursive copy)?
- **Code duplication** — if the same logic appears in 3+ places, flag it for extraction into a shared method. Do not refactor — present findings for discussion.

## Workstream 4 — Permission gate completeness

For every Filament resource and admin page, verify that permission gates are present and consistent:

- Every resource with a `canViewAny()` override should have a corresponding policy registered in `AuthServiceProvider`.
- Every policy should cover the full set of actions the resource exposes (view, create, update, delete, and where applicable: restore, forceDelete).
- Filament pages (non-resource pages like settings, importers, donors) should have an access gate — either `canAccess()` or middleware. Flag any admin page accessible without an authorization check.
- Verify no resource has a `canDelete` / `canEdit` / `canCreate` override that duplicates what its policy already provides (this is a Workstream 1 finding if so, but check here for completeness).

Fix missing gates in place. Flag ambiguous cases for discussion.

## Workstream 5 — Help doc & route audit

- Run `php artisan route:list` and cross-reference against `docs/app-reference.md`. Flag any routes missing from the reference or reference entries pointing to non-existent routes.
- For every file in `resources/docs/`, verify the `routes:` frontmatter array points to valid Filament route names. Run `php artisan route:list --name=filament` to get the canonical list.
- Flag any help doc stub that has no `routes:` mapping (unreachable from the contextual help system).
- Verify `docs/app-reference.md` accurately reflects the current admin panel structure (all resources, pages, and settings).

Fix confirmed broken route mappings in place.

## Workstream 6 — Dead code and orphaned files

- **Dead imports** — `use` statements that import classes not referenced in the file.
- **Unreachable code** — classes, methods, or constants defined but never called from anywhere.
- **Orphaned views** — Blade views not rendered by any controller, Livewire component, or `@include`.
- **Dead routes** — route definitions pointing to missing controllers or methods.

Fix confirmed dead code in place. Flag uncertain cases for discussion.

### Known cleanup items

- **Orphaned `WidgetRegistry` import** — `PageWidgetsRelationManager.php` imports `App\Widgets\WidgetRegistry`, which does not exist. The relation manager itself may be unused (widget editing goes through the Livewire PageBuilder path). Confirm whether the relation manager is dead code. Remove the import and delete the file if it is no longer referenced anywhere.

Do not refactor. Present findings for discussion.
