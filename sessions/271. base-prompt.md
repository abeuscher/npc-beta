## Process rules for every session

The base-prompt rules (process, close gate, style) in `sessions/template-base-prompt.md` apply. Session-specific notes:

- **No schema changes; no Composer changes; no migrations.** This is an audit; safe-fix-in-place is the only code change permitted, and only for confirmed-trivial cases (dead imports, stale references, single-line framework-alignment wins like `preserveKeys = true`). - **No agent contract change.** No `docs/fleet-manager-agent-contract.md` edit, no version bump. The Fleet Manager API surface review (deferred to 271's W4) is read-only here.
- **Front-end builds — only if a safe-fix touches front-end source.** `npm run build` if any `resources/js/**` or `resources/scss/**` change lands; `build:public` if any widget asset surface changes. Most of this session's safe fixes are PHP / blade / docs — front-end touches should be rare.
- **Pause and ask** if a workstream surfaces something architecturally bigger than a "safe in-place fix" — those are W7 / W8 / Open Flags entries, not in-session decisions. Specifically: do not refactor an extraction during the audit; capture it on a table for 272.
- **Bug fixes are out of scope.** Catching a real bug during the audit triggers an Open Flag, not a fix — unless the fix is one line and obviously correct.
- **Long runtime expected.** Scope is wide. Split into multiple iteration branches (`session-270/1`, `session-270/2`, …) if the audit cannot complete in one sitting. Each iteration ships a self-contained slice plus the safe fixes found in it.
- **Tests for new behaviour.** Pure structural extractions or rename-only safe fixes don't need new tests. Anything that *adds* a new seam (won't happen in this audit per scope) would need coverage.
- **When implementation is complete**, run the fast test suite: `docker compose exec app php artisan test --exclude-group=slow`. Target: **2163 ± a small delta** (safe-fix-in-place may delete dead-code tests or remove dead imports; small drops are fine, document any drop in the log). Slow group baseline 16 (unchanged expected — no new slow-grouped tests this session).
- **Manual verification not expected.** Audit output is text (W7/W8 tables, Open Flags block, per-workstream findings); the user reviews the log directly. Manual UI testing only if a safe-fix touched a user-visible surface.

## Style rules

The style rules in `sessions/template-base-prompt.md` apply. Session-specific:

- **Audit output is the deliverable.** Write to-the-point findings; no narrative padding. Use file:line citations everywhere; the apply session and the migration squash both consume this log directly.
- **Safe in-place fixes carry their own commit message describing the fix.** Don't bundle unrelated safe fixes into one commit; each kind of fix (dead import sweep, stale reference rename, etc.) gets its own commit so the reviewer can bisect.
- **W7 / W8 tables follow the 205/206 column shape exactly.** W7 columns: # / What it does / Where (file:line list) / Proposed extraction. W8 columns: # / File:lines / Current approach / Framework alternative / Risk.
- **W11 (file-length) output format:** per-language summary block (`PHP: avg N LOC, M files >2× avg`) followed by a numbered list of outliers (`#1 app/Filament/Pages/ImportContactsPage.php — 976 LOC (3.4× avg)`). The apply session targets the outliers for extraction.
- **W12 (inline-code-in-markup) output format:** table with columns # / File:lines / Block size (LOC) / Block kind (`<script>` / `<style>` / inline JS handler / multi-line CSS string in PHP) / Suggested target file or asset path.
- No docstrings, comments, or type annotations on code you did not write.
- Simple, correct, well-considered solutions over fast or clever ones.
