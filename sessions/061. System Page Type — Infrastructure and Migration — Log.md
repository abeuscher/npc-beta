## What was built

### system_prefix routing

- Added `system_prefix` SiteSetting to `DatabaseSeeder` with default value `'system'` (changed from earlier plan of empty string — avoids clashing with root-slug portal routes on fresh installs).
- Added `system_prefix` to `config/site.php` following the same IIFE/try-catch pattern as `blog_prefix` and `events_prefix`.
- Added `system_prefix` Routing field to `GeneralSettingsPage` alongside the other prefix fields.
- All portal auth **GET** routes are now prefixed by `system_prefix`: `/system/login`, `/system/signup`, `/system/forgot-password`, `/system/forgot-password/sent`, `/system/reset-password/{token}`, `/system/email/verify`, `/system/email/verify/{id}/{hash}`, `/system/account` and its sub-routes.
- All portal auth **POST** routes remain at root (no prefix) for form action simplicity: `/login`, `/signup`, `/logout`, `/forgot-password`, `/reset-password`.
- Updated `system_prefix` fallback from `''` to `'system'` in `LoginController`, `SignupController`, `ForgotPasswordController`, and `PageBuilder::computeBareSlug()`.

## Decisions made during the session

- **`system_prefix` default is `'system'`**, not empty string. Rationale: avoids `/login` and `/account` appearing at root, where they would conflict with common expectations and potentially shadow other routes. A fresh install gets `/system/login` etc. out of the box.
- **POST routes stay at root**, not under the system prefix. Rationale: HTML form actions cannot use named routes, and hardcoding a prefix into form templates creates a maintenance burden. Root POST routes are invisible to end users.
- **Filament's pre-compiled CSS** does not include arbitrary Tailwind values. Solution: use inline styles for arbitrary values (`max-height: 50vh`) and Filament's established CSS variable grid pattern for column layouts.
- **`@teleport('body')`** required for any `fixed`-positioned modal inside a Filament form section, because Filament sections use CSS `transform` which creates a new stacking context and anchors `fixed` children.

## Deferred decisions

- System pages carved out as Blade-view exceptions (`reset-password`, `email/verify`) remain as Blade views with settings-driven content. Full CMS migration is impractical until the page builder supports token/dynamic-URL patterns.
- `account` sub-routes (`/account/address`, `/account/membership`, etc.) remain at root (no `system_prefix`) — they belong logically under `portal_prefix` and will be addressed in a dedicated routing consolidation pass.
- The unsubscribe page is not yet built — deferred to a future session.
- "Change password" member self-service — deferred.
