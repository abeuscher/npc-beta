## Session-specific deltas

- **The token list is fixed (formalization, not design).** Tier-1 user-tunable (`--np-color-*`): `brand`, `bg` (page/body — a genuine gap today, new), `surface`, `text`, `heading`, `text-muted`, `link` (in-content; gap today), `border`, plus the chrome set `header-bg`, `footer-bg`, `nav-link`, `nav-hover`, `nav-active`. Tier-2 published-but-not-user-tunable (widget-dev contract surface is wider than the user-facing knob set): `brand-contrast` (readable on brand fill — may be derived), `success` / `error` / `warning`, `focus-ring`. Names are `--np-color-*`. Do not re-derive or expand the set.
- **Delivery is the 295-corrected path, verbatim pattern.** Theme defaults emit as `.np-site`-scoped custom properties into the public bundle via `AssetBuildService::collectSources()`, exactly like the now-corrected typography/buttons; `TypographyCompiler::compileScoped(['.np-site'])` is the mirror. SCSS `$color-*` demote to build-time fallbacks; alias `--color-primary` / reconcile `--btn-*` brand refs — do not break consumers.
- **296 is a hard prerequisite.** The drift guard must be green at start and close; it is the objective oracle that the bundle matches saved settings. 297 is the large CSS system 296 exists to protect.
- **Session-295 lessons are rules here:** verify in a real browser by computed values across public + admin + preview (never from emitted CSS/`curl`); Theme defaults go in the bundle (drift-prone, guard-covered); the per-template override layer is request-time inline and is **299's** concern, not built here.
- **Interim consequence, by design:** relocating colors off `templates` means there is **no per-template color between 297 and 299** (schemes restore deliberate deviation in 299). Most installs run one effective template; the default template's colors become the single Theme default. This is acceptable and is Open Question 1 — confirm, don't silently re-add per-template color.

## Open questions to resolve at session start

1. **Interim no-per-template-color tradeoff.** Confirm acceptable that 297→299 has Theme-only color (no per-template deviation) until schemes land. Default: acceptable (one effective template per install; deviation returns as 299 schemes). If a non-default template carries divergent colors today, **capture and log the values + count before the column drop** and surface them — they have no Theme home until 299; do not silently discard without a recorded decision.
2. **SiteSetting shape for the palette.** One `theme_colors` JSON key in group `design` (mirrors `typography` / `button_styles`) vs per-token keys. Recommendation: single JSON key, mirroring the existing design-settings convention and keeping the 296 drift-guard's design-group reconciliation simple. Decide and note.
3. **Tier-2 derived vs stored.** `brand-contrast` and `focus-ring`: auto-derived from `brand` at emit time, or stored concrete values? Recommendation: derived (one fewer knob; consistent with "author worries about content, not appearance"), but confirm contrast math is cheap and deterministic; fall back to stored concrete if not.
4. **Data-migration mapping.** The default template's six color columns → which tier-1 tokens (primary_color→brand; header_bg_color→header-bg; footer_bg_color→footer-bg; nav_*→nav-*). Confirm 1:1 and byte-faithful; this is the data-preservation guard (write first).

## Phases

### Phase 1 — Token contract + SiteSetting shape + `.np-site` bundle emission

Define the tier-1/tier-2 `--np-color-*` set with concrete defaults (every token a concrete value — the concrete-values rule). Persist as a `design`-group `SiteSetting` (per Open Q2). Emit the defaults as `.np-site`-scoped custom properties into the bundle via `AssetBuildService::collectSources()`, mirroring `TypographyCompiler::compileScoped(['.np-site'])`. Demote `$color-*` SCSS to `var(--np-color-x, #{$color-x})` fallbacks; alias `--color-primary`/`--btn-*` brand refs. Regression: bundle contains only `.np-site`-scoped color tokens, zero bare/`:root` (the 295 scoping discipline); 296 drift guard still green.

### Phase 2 — Colors editor on the Theme page

Add a Colors surface to `DesignSystemPage` (sibling to Text Styles + Buttons), with the same save → `AssetBuildService::build()` rebuild path (mirror the typography/buttons pattern + the explicit-rebuild affordance). Tier-1 tokens are editable; tier-2 are not surfaced (published to widget-devs only). Publish/extend the **widget-dev token contract doc** listing the guaranteed `--np-color-*` variables (the "publish it somewhere" deliverable).

### Phase 3 — Relocation + kill default-template-as-brand

One-shot data migration: the default template's six color columns → the Theme defaults, **byte-faithful** (data-preservation guard written first). Repoint `public.blade.php`'s `$__tpl?->resolved('primary_color' / header/footer/nav)` injection at the Theme tokens. Drop the six `templates` columns (DB migration) + update `docs/schema/`. Remove the "default template carries brand" concept. Non-default divergent colors: log + surface per Open Q1.

### Phase 4 — Verify (objective) + palette judgment (human)

Real-browser computed-value verification across **three surfaces**: public (`.np-site` → tokens resolve), admin (no `.np-site` → no leak), page-builder preview (`.np-site` → faithful). Drift guard green. Capture before/after multi-surface. Then **stop for the one human-judgment call**: surface the chosen default palette values + captures for the user's visual review (brand/colors are judgment, not objective) — do not auto-pick.
