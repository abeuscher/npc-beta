## Process rules for every session

The base-prompt rules (process, close gate, style) in `sessions/template-base-prompt.md` apply. Session-specific notes:

- **Audit walks the surface; doesn't restructure it.** Findings produce a permission matrix entry, an in-session fix (if small), or a follow-on flag. The session does NOT redesign the role vocabulary, add new roles, or restructure `PermissionSeeder.php` unless an obvious bug requires it.
- **Audit findings default to "fix"** *(per the user's standing rule — see `memory/feedback_audit_findings_default_to_fix.md`)*: surface inclusively in W7/W8/W11/W12/Open Flags shaped sections of the session log; "won't-fix" comes from the apply walkthrough, not from audit-side filtering.
- **Persona-vs-role mismatch is an open question.** The release-plan C3 entry names four personas: volunteer / board-read-only / staff-admin / public-visitor. The shipped roles are `super_admin`, `cms_editor`, `crm_editor`, `event_manager`, `volunteer_coordinator`, `treasurer`, `blogger`, `developer`. Resolve the mapping at session start before walking the surface; the audit walks the actual shipped roles, with persona names as a cross-reference for the release-plan success criterion.
- **No migration expected.** If audit findings require a schema-level fix (unlikely but possible — e.g. a missing permission column), that's a finding to flag, not a session-line implementation. Carry to a follow-on or fix in-session per Rule 2 if small.
- **Tests for new behaviour.** Any in-session fix gets a Pest test (assertion: role X cannot reach route Y, or role X can reach route Y). If no behaviour changes, no new tests — but a "permission matrix Pest test" that codifies the matrix as executable spec is in scope as the artifact's runtime guard.
- **When implementation is complete on each iteration**, run the fast test suite: `docker compose exec app php artisan test --exclude-group=slow`. Target: **279 close baseline (2356 sequential) + any new audit tests**. Acknowledge the carried parallel-runner flake.
- **Front-end builds.** Audit-only sessions typically don't touch front-end assets. If a permission-related Filament component change touches SCSS or widget assets, run `build:public` or `npm run build` per the build matrix in the template.
- **Playwright run** likely not warranted unless an audit fix changes a public-form surface. Judgment at impl time.
- **Pause for manual testing** is judgment-driven this session. If audit findings require role-membership verification ("does cms_editor actually see this page?"), the user is the one who can log in as each role and confirm — announce when the audit is ready for manual cross-check. The permission matrix doc itself is objectively verifiable (the matrix exists and is internally consistent); the lived-experience verification is what needs the user.

## Style rules

The style rules in `sessions/template-base-prompt.md` apply. Session-specific:

- **Iteration commit messages describe the deliverable** — e.g. "Iteration /1 — permission matrix doc + walk of Filament resources A–M" or "Iteration /2 — walk of pages + N–Z resources + in-session fixes".
- **Audit findings format in the matrix doc.** One row per (resource × persona) cell with: UI gate status (gated / open / missing), controller gate status (gated / open / missing — checks for direct-URL bypass), notes (e.g. "missing controller gate but UI gate hides it; flag for 32b's exposure work"), and one of {OK / fixed in-session NNN/N / deferred to 32b / open flag for follow-on}.
- **Findings inclusively.** Default to surfacing — `feedback_audit_findings_default_to_fix.md` says don't filter on the audit side; let the apply walkthrough decide won't-fix.
- **Super_admin bypass is in scope** — verify `Gate::before` works as expected and isn't accidentally bypassed elsewhere. Document it in the matrix.
- **No docstrings, comments, or type annotations on code you did not write.**
- **Simple, correct, well-considered solutions over fast or clever ones.**
