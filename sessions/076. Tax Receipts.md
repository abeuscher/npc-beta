# 076. Tax Receipts

Build annual donation receipt generation — fund designation on donations, a receipt email template, and an admin action to generate and send year-end receipts to all eligible donors.

---

## Design decisions (resolved before starting)

**Fund designation:**
The `funds` table already exists in the database and a `Fund` model likely exists. Read both before writing anything. Add `fund_id` (nullable FK → funds, nullOnDelete) back to the `donations` table. `null` means unrestricted / general fund. The donation widget shows an optional fund picker only if at least one active fund exists — otherwise no fund field is rendered and all donations are unrestricted by default.

**Restriction type on funds:**
Add a `restriction_type` column to `funds` if not already present: values `unrestricted`, `temporarily_restricted`, `permanently_restricted`. This maps to the three IRS/GAAP revenue classification buckets. Displayed on the tax receipt alongside the fund name.

**Controller and webhook:**
`fund_id` is accepted as an optional POST field on `DonationCheckoutController`, validated as a UUID that exists in `funds` if provided. Stored on the `Donation` at creation (pending state). The webhook does not modify `fund_id` — it is already set when the checkout session was created.

**What a receipt covers:**
All `Donation` records with `status = 'active'` for a given contact in the selected tax year (matched on `started_at`). Both one-off and recurring. Purchases are not charitable contributions and are excluded. No goods or services are assumed to have been exchanged — the standard IRS acknowledgement line is included unconditionally.

**Receipt record:**
A `donation_receipts` table stores one row per send: `id`, `contact_id` (FK → contacts, cascade), `tax_year` (integer), `sent_at` (timestamp), `total_amount` (decimal 10,2), `breakdown` (JSON: array of `{fund_label, restriction_type, amount}`). Prevents double-send and gives the admin a send history. If a receipt has already been sent for a contact/year, the admin sees a warning and must explicitly re-send (a forced re-send creates a second record — it does not overwrite).

**Email delivery:**
Uses the existing email system (Resend). A new seeded email template with handle `donation_receipt` is created. Template variables: `contact_name`, `org_name` (from site settings), `tax_year`, `donations` (array of fund-grouped line items), `total`. Follow the existing email template seeding pattern exactly.

**Admin receipt action:**
A custom Filament page under Finance — `GenerateTaxReceiptsPage`. Admin selects a tax year (integer input, defaults to prior calendar year). The page shows a count of eligible contacts (those with at least one active donation in that year) and a count of contacts already receipted for that year. A "Send Receipts" button sends only to un-receipted contacts. A "Force Re-send All" button sends to everyone. Both are gated behind a confirmation modal.

**Funds resource:**
If a `FundResource` does not already exist, create one under the Finance navigation group. Minimal: name, description, restriction_type, is_active. Read-only restriction_type after creation.

## Phase 1 — Fund designation

1. Read the existing `Fund` model, `funds` table schema, and any existing `FundResource` before writing anything.
2. Migration: add `restriction_type` to `funds` if not present (string, default `'unrestricted'`); add `fund_id` (nullable FK → funds, nullOnDelete) to `donations`.
3. Update `Donation` model: add `fund_id` to `$fillable`; add `fund()` BelongsTo.
4. Update `DonationCheckoutController`: accept optional `fund_id`, validate `exists:funds,id` if provided, store on the Donation.
5. Update `DonationResource` view: show fund name (placeholder `—` if null/unrestricted).
6. Update `donation_form` widget: if active funds exist, render an optional fund select below the amount section. Pass `fund_id` in the fetch payload. If no active funds exist, render nothing.
7. Update `WidgetTypeSeeder` if needed. Update `docs/schema.md`.

---

## Phase 2 — Receipt infrastructure

1. Migration: create `donation_receipts` table (`id` bigint PK, `contact_id` UUID FK cascade, `tax_year` integer, `sent_at` timestamp, `total_amount` decimal 10,2, `breakdown` JSON, `created_at` timestamp, no `updated_at`).
2. `DonationReceipt` model: `$fillable`, casts (`total_amount` decimal:2, `breakdown` array, `sent_at` datetime, `created_at` datetime). `const UPDATED_AT = null`. `contact()` BelongsTo.
3. Seed `donation_receipt` email template. Subject: `Your {{tax_year}} donation receipt — {{org_name}}`. Body: donor greeting, fund-grouped line items, grand total, standard IRS acknowledgement line, org name footer. Follow the exact seeding pattern of existing templates.
4. Update `docs/schema.md`.

---

## Phase 3 — Admin receipt page

1. `GenerateTaxReceiptsPage` under Finance navigation group. Breadcrumbs: `Finance › Generate Tax Receipts`.
2. Form: tax year input (integer, required, default prior calendar year). On submit, shows a summary panel:
   - Total eligible contacts (active donations in year)
   - Already receipted (have a `donation_receipts` row for this year)
   - Pending (eligible but not yet receipted)
3. "Send to Pending" action: iterates pending contacts, builds receipt data, sends email, writes `DonationReceipt` record. Runs synchronously for now (no queue). Shows success notification with count sent.
4. "Force Re-send All" action: same as above but includes already-receipted contacts. Requires confirmation modal with explicit warning.
5. Register route and create help doc stub at `resources/docs/tax-receipts.md`.

## Security checklist

Before closing:

- [ ] Receipt sending is admin-only — no public route.
- [ ] Each receipt query is scoped strictly to the target contact — no cross-contact data leakage.
- [ ] `fund_id` validated server-side against the `funds` table — never accepted raw.
- [ ] Re-send creates a new record rather than silently overwriting — audit trail preserved.
- [ ] Email template renders only variables explicitly passed — no arbitrary data injection.

## Out of scope

- PDF generation (future Finance session)
- Automatic year-end sending
- In-kind or non-cash donation receipts
- Partial receipts (goods or services received disclosures)
- Non-US tax formats
- Donor-initiated receipt download via portal
- Receipt preview before sending
