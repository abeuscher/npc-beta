## The two defects (confirm both live before fixing)

There are **two distinct nav surfaces** on the public site; the first task is establishing which renders the live mobile header/footer and where each defect lives:

- The CMS **Nav widget** — `app/Widgets/Nav/template.blade.php`. The `horizontal` branch already ships a full mobile implementation (a `widget-nav__hamburger` button, `mobileOpen` Alpine state, a **duplicated** `widget-nav__mobile` list with `x-collapse` sub-menus, and an inline `<style>` block that swaps `widget-nav__menu` ↔ `widget-nav__hamburger` at the 768 breakpoint). The `columns` branch (the footer preset) has **no mobile collapse at all**.
- The hardcoded **site-header chrome** — `resources/views/components/site-header.blade.php` (its own `site-nav__toggle` hamburger, `x-data="{ open: false }"`, `.site-nav.is-open`) + `resources/views/components/site-footer.blade.php`.

### 1 — Hamburger broken *(priority)*

The mobile navigation hamburger does not work. Root-cause it **in the browser at a mobile viewport** before touching code — likely candidates to rule in/out: Alpine not initializing the nav subtree, `x-cloak` never lifting (menu stays hidden), the inline `<style>` display-toggle not firing at the breakpoint, the toggle button not wired, or the two nav surfaces conflicting. Name which surface and which link in the chain fails; the smallest fix that makes open/close deterministic is the bar.

### 2 — Footer columns collapse poorly

The footer column layout renders poorly on mobile. The Nav widget's `widget-nav--columns` preset (and/or the footer's page-layout columns) does not stack cleanly at narrow widths. Confirm in-browser how it currently renders, then add the mobile collapse so footer columns stack to a single readable column at the 768 breakpoint (consistent with the brief's fixed breakpoints; viewport `@media`, matching how the column collapse already works elsewhere).

## Phases

### Phase 1 — Reproduce + root-cause (real browser, mobile viewport)
Drive the live local public site at a phone viewport (read-only Playwright / standalone script — never the DB-resetting specs). Establish which nav surface is live, reproduce both defects, and pin each root cause. Output: a one-paragraph cause + fix-plan per defect, surfaced at the Open Gate.

### Phase 2 — Fix
- **Hamburger:** the chosen smallest fix; verify by re-driving the mobile viewport — the menu opens and closes reliably, sub-menus expand, Escape/again closes.
- **Footer columns:** add the mobile collapse; verify columns stack cleanly at narrow width in the browser.

### Phase 3 — Verify + guard
Re-drive both at mobile viewport for the proof. Add a regression guard if one fits cheaply (e.g. a Pest assertion that the footer/columns preset emits its collapse rule, or a Playwright mobile-viewport check in the e2e suite — but only if it runs without the widget build pipeline, per the 333 finding; otherwise note why a standing guard isn't viable and rely on the render-layer assertion).
