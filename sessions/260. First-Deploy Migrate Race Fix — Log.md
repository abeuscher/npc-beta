### Phase 5 — In-session absorptions and findings

- **Operator-paste-friction findings during Phase 4** (folded forward into operator-checklist authoring guidance, not absorbed as code changes):
  - **Markdown fence interaction with bash paste.** Triple-backtick fences (` ```bash ... ``` `) in operator-paste blocks silently break bash output: bash interprets the opening fence as command substitution and swallows the inner commands' output. Operator hit this on step 2 (UFW); commands ran (state mutated as expected) but no output was visible. Recovery: re-paste only the inner lines (no fence markers) or close + reopen the SSH session. Pattern note for future operator checklists: avoid markdown fences around bash command blocks, or include a clear "skip the fence lines when copying" disclaimer at the top.
  - **SSH key non-default-path friction.** Operator generated their personal SSH key at `~/.ssh/fresh_droplet_test` (named-file form) rather than the default `~/.ssh/id_ed25519`. SSH searches default paths only — `scp` returned `Permission denied (publickey)` until the operator added `-i ~/.ssh/fresh_droplet_test` to the command. Resolved in-session; pattern note for future operator checklists: when generating a key with a non-default name, the checklist should pre-fill the `-i` flag in subsequent ssh / scp commands.
- **Per-droplet operator-checklist pattern lift** — the `sessions/260. nphelper-verification.md` artifact's success at making a multi-substitution install tractable for a non-detail-oriented operator suggests a forward pattern: **generic install runbooks stay generic; every actual install gets a generated project-specific checklist as its execution artifact, with all values pre-filled.** Not formalized this session — the verification checklist served its purpose and ages out via session archival. Naming the pattern for future-us.
