## What was built

A planning-only cloud design session. Two deliverables, no application code. Designed the public website the demo install (session 321) ships on top of its `demo:reset` CRM baseline, and drafted the build session's prompt.

### Phase 1 — Org identity + page list

- Chose the fictional org: **Mill Creek Conservancy** — a small community watershed/restoration nonprofit. Mission, warm-civic voice, and a `still_photos`/`product_photos` visual register that matches what `RandomDataGenerator` attaches to its records. Identity maps each generated record type onto a believable site purpose (events → stewardship days, donations → watershed fund, memberships → Friends tiers, posts → field notes, products → store).
- Finalized the **seven-page set** (within 5–7): Home `/`, Events `/events`, Donate `/donate`, About `/about`, Contact `/contact`, News `/news`, plus the reused `/demo` gate. Each page's exact existing-widget handles named.

### Phase 2 — Per-page composition + copy

- Band-by-band layout for the six authored pages, following the marketing-site section-band / appearance-config / Text+Image-cell conventions (reuses the standing gap workarounds rather than re-deriving). Realistic on-brand copy written for headlines, body, CTAs, the three giving levels, and a curated six-person board.
- Per-page data alignment recorded: which widgets read the generated baseline (`events_listing`, `event_calendar`, `recent_donations`, `blog_listing`) and what content the build session must seed *beyond* the generator — chiefly the **board collection** (`board_members` reads a collection, the generator makes none).

### Phase 3 — Nav, `/demo` integration, reset wiring

- Header/footer nav specified by link set; explicitly defers chrome styling to whatever E16 (session 322) lands as production defaults — no hard-coding around current chrome.
- `/demo` brought in as just one more bundle on the import path (no clipping logic, per the user's "mechanism doesn't matter").
- **Reset wiring designed precisely:** a page-import step added to `DemoBaselineSeeder::run()` before the existing wipe+generate, iterating `database/seeders/demo_pages/*.json` through `ContentImporter::import(..., ['replace_duplicate_pages' => true, 'import_media' => true])`, inside the seeder's existing super_admin acting context. Placed in `run()` so `--soft` also restores defaced pages; idempotent by slug. New reset path: rebuild → seed → import curated pages → generate CRM baseline.

### Phase 4 — Deliverables

- **`sessions/public website/demo-site-pages-spec.md`** — the build-ready design spec (org identity, page list, per-page composition+copy, data alignment, nav, reset wiring, gaps).
- **`sessions/demo-site-pages-build.PROMPT-DRAFT.md`** — the local build session's prompt (no session number; user slots it). Builds the pages in the CMS, exports JSON, wires the reset, resolves the Faker blocker, and dress-rehearses on the `APP_ENV=demo` isolated stack.
