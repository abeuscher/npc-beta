## What was built

### Phase 1 — `RecordDetailViewBuilderApiController` + routes

- [app/Http/Controllers/Admin/RecordDetailViewBuilderApiController.php](app/Http/Controllers/Admin/RecordDetailViewBuilderApiController.php) — mirror of `DashboardBuilderApiController` with two named differences over the dashboard controller: the slot is `record_detail_sidebar` (not `dashboard_grid`); record-detail allows column layouts, so the controller gains layout CRUD methods (`storeLayout` / `updateLayout` / `destroyLayout`) and a `buildTree` that interleaves layouts with root widgets — mirror of `PageBuilderApiController`'s shape, IDOR-gated. Every endpoint gates on `manage_record_detail_views`. Widget-keyed endpoints assert `$widget->owner_type === RecordDetailView::class && $widget->owner_id === $view->getKey()`; layout-keyed endpoints assert the same on `PageLayout`. `appearance_config` writes filtered to `RecordDetailSidebarSlot::layoutConstraints()['allowed_appearance_fields']` (currently `['background', 'text']`). Slot-allowedSlots check on create returns 422 for handles whose definition doesn't include `record_detail_sidebar`. The `reorder` endpoint accepts both `widget` and `layout` item types and dispatches per-row.
- [routes/admin-record-detail-view-api.php](routes/admin-record-detail-view-api.php) — owner-scoped under `views/{view}/...` so the IDOR boundary is automatic at the route binding level. URL parameter named `view` (not `recordDetailView`) per the prompt's `views/{viewId}` API surface convention; implicit Laravel route model binding via the controller's `RecordDetailView $view` type-hint resolves cleanly without needing a global `Route::bind` side-effect.
- [app/Providers/Filament/AdminPanelProvider.php](app/Providers/Filament/AdminPanelProvider.php) — registered the route file at `/admin/api/record-detail-view-builder` alongside the existing `/admin/api/dashboard-builder` and `/admin/api/page-builder` registrations.

### Phase 3 — `RecordDetailViewBuilder` Livewire shell

- [app/Livewire/RecordDetailViewBuilder.php](app/Livewire/RecordDetailViewBuilder.php) — mirror of `DashboardBuilder.php` shape, owns a `RecordDetailView` rather than a `DashboardConfig`. 403-fail-closed on `manage_record_detail_views`. `getBootstrapData()` emits `mode: 'record_detail'`, `owner_type: 'record_detail_view'`, `api_base_url == api_lookup_url == /admin/api/record-detail-view-builder/views/{viewId}`, slot-filtered `widget_types` (via `WidgetType::forPicker(null, 'record_detail_sidebar')`), `allowed_appearance_fields` from the slot's constraints, `view_label` from the model, `record_type_label` via `class_basename($view->record_type)`. The bootstrap also emits any column layouts owned by the View (mirror of PageBuilder's tree shape) since record-detail allows them. `createBlock(widgetTypeId)` Livewire handler accepts an optional layout-id + column-index and runs the same allowedSlots check as the API controller.
- [resources/views/livewire/record-detail-view-builder.blade.php](resources/views/livewire/record-detail-view-builder.blade.php) — mirror of `dashboard-builder.blade.php`. Mounts the Vue app via `data-page-builder-app` with `data-bootstrap` JSON. Reuses the existing `x-widget-picker-modal` component with a pre-filtered `$widgetTypes` array.

### Surfaced during manual testing

The user created two Views bound to Contact, each containing the placeholder widget, and observed only one rendering on `EditContact`. Investigation confirmed this is expected pre-5d-1 behavior, not a 5c.5 bug:

- The `HasRecordDetailSubNavigation` trait's `recordDetailViewSubPageClass()` returns `null` on `EditContact` — no concrete `RecordDetailViewPage` subclass is registered — so the trait surfaces zero Views in the sub-nav.
- `RecordDetailViewWidget::resolveView()` falls back to `ViewRegistry::forRecordType($record::class)->first()` when no `detailView` is explicitly set, so the second-and-onward Views are silently dropped at the footer-widget mount.
- The Templates/Themes refactor (5c) demonstrates the multi-View case working correctly: `EditPageTemplate` has `recordDetailViewSubPageClass()` set to `EditPageTemplateChrome::class`, and Header / Footer Views surface as proper sub-nav entries.

This is a **forward gap** to be resolved in 5d-1: when the first concrete record-detail widget arrives, `EditContact` gains a concrete `EditContactView` page extending `RecordDetailViewPage` and overrides `recordDetailViewSubPageClass()` to return it. Multi-View on Contact becomes first-class at that point. Carved out of 5c.5 because the wiring requires a concrete sub-page class, which itself requires a real widget to mount — premature without a concrete consumer.

## Deferred decisions

- **EditContact sub-nav wiring (forward gap surfaced during manual testing).** When a Contact has more than one View bound to it, the second-and-onward Views silently drop because `EditContact::recordDetailViewSubPageClass()` returns null and the footer widget falls back to `forRecordType()->first()`. Resolved in 5d-1 when the first concrete record-detail widget lands and EditContact gains a `EditContactView` page extending `RecordDetailViewPage`. Carved out here because the wiring requires a concrete sub-page class, which requires a real widget to mount.
- **Phase 5e — Permissions in the contract layer.** Forcing function: first sensitive concrete widget in 5d+. Recent Notes (5d-1) is the natural one — Notes carry a `view_note` permission gate that the contract layer has no story for today. The 231 prompt drafts a "minimal 5e" approach: a literal `Gate::denies('view_note')` check at the resolver's `note` arm, fail-closed. Full perm-in-contract architecture (where every contract declares required perms; resolver gates uniformly) stays carved out for a dedicated future Phase 5e session.
- **Widget help system integration.** Forcing function: second or third concrete widget in 5d+ when the rollup gap becomes visible.
- **Concrete record-detail widgets** — Recent Notes (5d-1, next session), Recent Donations (5d-2 candidate, forces full Phase 5e), Membership Status (5d-2 alternative, lighter), Recent Activity. Phase 5d arc spans ~4 sessions, one widget per session.
- **Multiple record types beyond Contact in `getRecordTypeOptions()`.** Event, Organization, etc. — added by editing the static method when a concrete widget arc forces them. No dynamic discovery in 5c.5; the surface stays auditable.
- **Admin UI for chrome Views.** The `page_template_header` / `page_template_footer` rows are excluded from the Resource. Chrome editing remains in the Templates editor (per 5c).
- **`DashboardConfig` retrofit to `DashboardView` implementing `IsView`.** Cheap follow-up post-5d, scheduled ad-hoc. Brings dashboards into the unified View vocabulary.
- **`Page` implementing `IsView`.** Per the "a+" framing, Page-as-View is the elective bottom of the View stack. Could be never.
- **Themes/Templates top-level nav merge.** Carved out of 5c per the original prompt scope. Separate follow-up session — a structural decision about navigation taxonomy that doesn't fit cleanly inside any current phase.
- **Placeholder widget removal.** `RecordDetailPlaceholderDefinition` stays on disk and registered. Removing it is a follow-up cleanup pass once 5d-1 lands and all installs have re-seeded.
