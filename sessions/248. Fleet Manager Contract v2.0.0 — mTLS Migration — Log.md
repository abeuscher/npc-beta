## What was built

### Phase 1 — Nginx mTLS configuration

- [docker/nginx/default.conf](../docker/nginx/default.conf) (local) and [docker/nginx/prod.conf](../docker/nginx/prod.conf) (production) gained the mTLS gate on `/api/health`. Server-level `ssl_verify_client optional` keeps public routes reachable; per-location `if ($ssl_client_verify != "SUCCESS") { return 403; }` enforces strict cert presence on the agent endpoint only.
- The single-server-block + per-location-`if` shape is correct for TLS 1.3 (the location-scoped `ssl_verify_client on` shape is silently broken on modern nginx because TLS 1.3 disables renegotiation post-CVE-2009-3555).
- Local dev gained a 443 listener (TLS) alongside the existing 80 listener. Server cert for `localhost` is generated via [bin/dev-certs.sh](../bin/dev-certs.sh) — gitignored (`nginx-certs/localhost.{crt,key}`); operator runs once per fresh checkout.
- Placeholder client trust cert committed at [nginx-certs/fm-client.crt](../nginx-certs/fm-client.crt) with `/CN=placeholder-do-not-trust` and the matching private key discarded at generation time. Harmless (no client can authenticate against it) but lets nginx load on a fresh checkout — `ssl_client_certificate` requires a real file.
- Defense-in-depth: [.gitignore](../.gitignore) gained `/nginx-certs/*.key` and `/nginx-certs/localhost.crt` so private keys can never be accidentally committed even if an operator copies an FM-supplied keypair into the directory.
- [docker-compose.yml](../docker-compose.yml) publishes 443 alongside 80 and bind-mounts `./nginx-certs` to `/etc/nginx/certs:ro`. [docker-compose.prod.yml](../docker-compose.prod.yml) bind-mounts `/opt/nonprofitcrm/nginx-certs` to the same in-container path — pattern mirrors the existing `.env` bind-mount discipline.

## Production deployment notes

The operator must populate the bind-mount target on the droplet **before** the next nginx start, or nginx fails to load the SSL config (the `ssl_client_certificate` directive references a file that does not exist under the new bind-mount). The committed placeholder at `nginx-certs/fm-client.crt` is a **local-dev convenience only** — it lets a fresh repo checkout `docker compose up` work without first generating a cert. Production does NOT receive the placeholder via deployment because the bind-mount source `/opt/nonprofitcrm/nginx-certs/` is on the droplet host, not part of any docker image artifact.

Exact commands the operator runs on the droplet, in order:

```bash
sudo mkdir -p /opt/nonprofitcrm/nginx-certs
# paste the FM-supplied PEM cert (from FM admin UI → Client → "View cert")
# into /opt/nonprofitcrm/nginx-certs/fm-client.crt
sudo chmod 644 /opt/nonprofitcrm/nginx-certs/fm-client.crt
sudo docker compose -f /opt/nonprofitcrm/docker-compose.prod.yml up -d
```

Notes:

- The cert file must be world-readable through the bind-mount so the nginx container's UID can read it. The cert is a public artifact — world-read does not leak anything sensitive.
- Private keys are never stored on the CRM side. FM holds the keypair; CRM holds only the public cert.
- FM 012 cannot poll successfully until **both** the cert is in place on the droplet **and** main has merged (the canonical raw-GitHub URL must return v2.0.0 before FM 012 can pass its Phase 0 pre-flight).
- Migration: none. No `php artisan migrate` step required this session.

## Deferred decisions

- **`last_backup_at` threshold-derivation ownership.** FM-raised question of whether the CRM's emitted derived `status` should be canonical (FM trusts the string, drops re-derivation, surrenders per-client overrides on this subcheck) or FM-canonical (CRM emits only the timestamp + threshold, FM derives). Considered for v2.0.0 and explicitly deferred to a future additive v2.x bump — keeps v2.0.0's retrospective focused on the auth swap. FM continues to re-derive locally against its own threshold matching the CRM's `[24, 36]` by configuration. Resolution lands when there's natural impetus (per-client override pressure, drift event, or a v2.x bump that already touches the subcheck).
- **Log-fetch endpoint.** Additive CRM-side endpoint that A2's log-reading session blocks on. Out of scope for v2.0.0 by design — lands as part of A2's CRM-side companion work whenever A2's log-reading half is scheduled.
- **Cert rotation automation.** Manual paste-and-restart is the v2 discipline (per the new runbook). FM-orchestrated cert distribution (FM SSHes the new cert into place, or hits a CRM-side endpoint that updates the trust file) lands when fleet shape forces it. Carry-forward.
- **Cert expiry tracking.** Self-signed certs default to long lifetimes (the runbook generates 3650-day certs); no FM-side surfacing in v2. Revisit when a real expiry approaches. Carry-forward.
- **Multi-FM-instance support.** Each install trusts exactly one FM-side cert under v2.0.0. If multi-FM scenarios emerge (one CRM serving multiple FM instances for failover, geographic distribution, etc.), the surface change is additive — `ssl_client_certificate` accepts a directory of trusted certs, or the contract supports a cert-array shape — and would land as a v2.x bump within the major.
- **Local-dev cert-bootstrap friction.** A fresh checkout requires running `bin/dev-certs.sh` once before `docker compose up` will succeed; without it, nginx fails to start because `nginx-certs/localhost.crt` is missing. Documented in the script header, in `app-reference.md`, and in this log. Not a blocker — but if it bites repeatedly, options range from "commit a self-signed localhost cert+key (option (c) from the architectural decision)" to "auto-generate via the docker entrypoint."
