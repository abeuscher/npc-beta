# 053. Duplicate Contact Detection

A focused session on detecting and resolving duplicate contacts. Two surfaces: the import preview step (before records are saved) and the contacts list (for reviewing existing duplicates). Read all relevant files before touching anything.

---

## Workstream 1 — Duplicate Detection Service

Before touching the import wizard or the contact list, build a reusable service class that encapsulates the matching logic. Both workstreams 2 and 3 will call this service.

Read the `Contact` model and any existing email/name uniqueness logic before writing the service.

### Matching strategy

Two tiers:

- **Hard duplicate** — exact email match (case-insensitive). These are definite duplicates.
- **Probable duplicate** — same `last_name` (case-insensitive) AND same `postal_code` (normalised — strip spaces, uppercase). These are candidates that require human review.

The service should accept either a `Contact` model instance or a plain data array (for use in the import preview, where records don't exist yet) and return a result indicating whether a hard duplicate exists, a list of probable duplicate IDs, or neither.

### Scope note

Do not implement phonetic matching, Levenshtein distance, or any fuzzy name algorithm in this session. The `last_name + postal_code` strategy is deliberately simple. It can be tightened in a future session once real data has been run against it.

---

## Workstream 2 — Import-Time Duplicate Flagging

Read `app/Filament/Pages/ImportContactsPage.php` and the preview step schema in full before making changes.

### Requirements

1. After column mapping and before the final confirm step, run each preview row through the duplicate detection service.
2. In the preview table, add a **Duplicate** column. For each row:
   - If a hard duplicate is found: flag the row clearly (e.g. red badge — "Exact match").
   - If a probable duplicate is found: flag with a softer indicator (e.g. amber badge — "Probable match") and show the matching contact's name.
   - If no match: no indicator.
3. The preview is read-only — do not add merge UI here. The purpose is to inform the admin before they commit.
4. The existing duplicate handling strategy (skip / update) already exists on the Map Columns step and continues to apply. The flagging in the preview is additional visibility, not a replacement for that mechanism.
5. Only run detection against rows that have a mappable email, last name, or postal code. Rows with none of those fields mapped are skipped silently.

## Workstream 3 — Contact List "Review Duplicates" Action

Read `app/Filament/Resources/ContactResource/Pages/ListContacts.php` and `ContactResource.php` before making changes.

### Requirements

1. Add a **Review Duplicates** header action to the contacts list page.
2. The action opens a slide-over or modal that presents probable duplicate pairs — one pair at a time (or paginated in a simple list).
3. For each pair, show: name, email, phone, created date for both records. Make it easy to tell them apart.
4. For each pair, offer three options:
   - **Merge** — keep one record, pull any non-blank fields from the other, reassign all related records (notes, tags, donations, memberships, event registrations) to the surviving contact, then soft-delete the discarded contact.
   - **Not a duplicate** — dismiss this pair from the review queue permanently (persist the dismissal so it doesn't re-appear).
   - **Skip** — skip this pair for now without dismissing.
5. The "surviving" record in a merge should default to the older contact (lower `id`), but allow the admin to choose.

### Merge field resolution

When merging, apply the following rule per field: if the surviving record has a value, keep it. If the surviving record's field is blank/null and the other record has a value, copy it over. Do not overwrite non-blank values on the surviving record.

### Persistence for dismissed pairs

Add a `contact_duplicate_dismissals` table:

| column | type | notes |
|--------|------|-------|
| `id` | uuid pk | |
| `contact_id_a` | uuid fk → contacts | lower of the two IDs |
| `contact_id_b` | uuid fk → contacts | higher of the two IDs |
| `dismissed_by` | uuid fk → users | |
| `dismissed_at` | timestamp | |

The detection service should exclude dismissed pairs from its results.

### Detection scope

The "Review Duplicates" action runs the detection across all non-deleted contacts. For large datasets this could be slow — run it synchronously for now (this is a staff-only admin action, not a public endpoint). Note in the log if the query needs pagination or a queued job for scale.

---

## Out of scope

- Automatic merging without admin review
- Three-way or multi-record merges (handle two at a time only)
- Public-facing duplicate prevention (e.g. at web form submission)
- Phonetic / fuzzy name matching
