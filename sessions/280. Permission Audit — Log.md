Working as intended; pinned by `PermissionMatrixTest::it hard-denies DonationResource create...`. **OK.**
7. **`household` permission family in seeder has no admin surface.** Permissions exist; no HouseholdResource. **Open flag** — either delete vocabulary or add a resource.

## Security considerations

The audit's foundational question — "can someone reach an admin URL by typing it directly even though the UI hides the link?" — is empirically answered: **no**, the `Resource::canAccess()` override is a universal URL gate via Filament's `mountCanAuthorizeResourceAccess` Livewire hook. This was the bypass theory I initially flagged as a possible finding; verifying it via test showed the gate works as designed.

Three super-admin-only behaviors are intentional and load-bearing:
- The `Gate::before` bypass at `AuthServiceProvider.php:40` — every `$user->can()` call returns true for super_admin.
- `RoleResource::canAccess` and `canEdit` for the super_admin role itself — only super_admin can reach Role management; the super_admin role record is non-editable even for super_admin (ReadOnlyAwareEditRecord lets it open in read-only mode).
- `UserResource` requires `view_any_user`, granted to no other role — user management is super-admin-exclusive.

Two belt-and-suspenders patterns documented:
- DonationResource — canCreate/canEdit/canDelete return literal `false` AND the create/edit routes aren't registered in `getPages()`. Even super_admin gets 404.
- EmailTemplateResource — canCreate / canDelete return literal `false` (templates are seed-data only).

The accidental-public-exposure work (deferred to 32c / session 282) is the next security-axis session — extends the matrix doc with the data-classification section.

## Test handoff sketch

Per the writer-side discipline started this session (per `sessions/test-runner-brief.md` § 2):

The session adds one test file: `tests/Feature/PermissionMatrixTest.php`. Sixteen tests covering three concern areas:
- Bypass-guard probes proving the canAccess gate enforces at URL level for the resources with permissive CRUD defaults (MembershipTier, CustomFieldDef, EmailTemplate).
- Cross-role probes pinning the seeder grants against drift (cms_editor cannot reach DonationResource list; treasurer cannot reach PageResource create; blogger cannot reach ContactResource list; treasurer can reach DonationResource list).
- super_admin Gate::before bypass + RoleResource hard-coded super-admin gate + RoleResource canEdit guard on the super_admin role record itself.

All 16 tests pass at session close. Failure modes a future runner should flag:
- Any test returning 200 instead of 403 → a permission gate has been weakened.
- Any test returning 404 → a Filament resource route registration changed (e.g. someone added a create page to DonationResource).
- Any test asserting `canEdit returns false` failing → RoleResource's super_admin-protection logic changed.

The tests do not exercise actual data mutation; they probe URL accessibility only. The matrix doc at `docs/runbooks/permission-matrix.md` is the authoritative documentation; the test file is the regression guard.
