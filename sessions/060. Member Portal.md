# 060. Member Portal

Builds the member-facing portal dashboard and the admin surface for managing portal access. Extends the portal foundation from sessions 056–059.

---

## Architecture decisions (agreed pre-session)

- No new tables. Member data is read from the existing `contacts`, `memberships`, and `event_registrations` tables via `contact_id`.
- `portal_accounts` gains an `is_active` boolean (default `true`). Setting it `false` immediately blocks login without deleting any data — the primary security mitigation tool for staff.
- **What members can edit**: email (triggers re-verification — see Phase 3), mailing address fields, password.
- **What members see but cannot edit**: all other contact fields — membership tier/status/dates/amount paid, volunteer status, event registrations, mailing list opt-in, custom fields, date of birth.
- **Hidden entirely from members**: the notes field (internal staff log).
- **Admin surface**: a read-only Portal Access panel on the Contact edit page showing account status and an `is_active` toggle. No other contact data is affected when portal access is disabled.
- **Form collision path**: when an unauthenticated web form submission includes an email that belongs to an active portal account, the submission is silently blocked (neutral response to the submitter) and a notification email is sent to the portal account holder asking them to log in and resubmit. Rate-limited to one notification per email address per hour to prevent abuse.

## Phase 1 — Migration

Add `is_active` to `portal_accounts`:

| column | type | default | notes |
|--------|------|---------|-------|
| `is_active` | boolean | `true` | `false` = portal access suspended |

Update `PortalAccount` model: add `is_active` to `$fillable` and `$casts`. Update the `portal` guard authentication logic to reject login attempts where `is_active = false` — return the same generic "Invalid email or password" error, no signal that the account is suspended.

## Phase 2 — Member dashboard (`/account`)

The existing `/account` route returns `view('portal.account')`. Build out that view.

**Layout**: use `layouts.portal` (authenticated header). The page is divided into sections:

### Contact information (read-only)
Display all contact fields except notes. Group them naturally:
- Personal: first name, last name, date of birth
- Contact: email, phone, phone_secondary, email_secondary
- Address: city, state, postal_code, country
- Preferences: mailing_list_opt_in

Each section has an "Edit" link/button where editing is permitted (address and password — email edit is Phase 3).

### Membership status (read-only)
If one or more `memberships` records exist for this contact, display them: tier, status, starts_on, expires_on, amount_paid. If none, display "No membership on file."

### Event registrations (read-only)
List `event_registrations` for this contact (via the contact's `contact_id`), showing event title, date, and registration status. Most recent first. If none, display "No event registrations on file."

**Portal security rule applies throughout**: every query must be scoped to `auth()->guard('portal')->user()->contact_id`. No exceptions.

## Phase 3 — Member edit flows

### Edit address
A simple form allowing the member to update: city, state, postal_code, country. Updates `contacts` scoped to the authenticated user's `contact_id`. Redirect back to `/account` with a flash success message.

### Change password
A form with three fields: current password (validated against stored hash), new password (min 12 chars), confirm new password. On success: update the hash, invalidate all other sessions via `Auth::guard('portal')->logoutOtherDevices()`, redirect back with flash success.

### Change email
Security-sensitive — the email is both the contact email and the portal login identifier. Flow:

1. Member submits a new email address.
2. Validate: required, valid email format, not already in use by another `portal_account`.
3. Do not update immediately. Store the pending email in a signed, time-limited token (60 minutes) — same pattern as the password reset broker.
4. Send a verification email to the **new** address containing a confirmation link.
5. On confirmation: update `portal_accounts.email`, update `contacts.email`, mark `email_verified_at = now()`.
6. Until confirmation, the old email remains active.

> **Note:** If the signed-token approach for email change adds excessive scope to this session, defer it to a patch and implement a simpler "submit new email → re-verify at next login" flow as a placeholder.

## Phase 4 — Admin: Portal Access panel on Contact edit page

Add a read-only collapsible panel to the Contact edit page in Filament, visible only when a `portal_account` record exists for this contact's email.

Panel displays:
- Status: Active / Suspended / Unverified (derived from `is_active` and `email_verified_at`)
- Email on the portal account
- Created at date

Actions:
- **Suspend access** (when `is_active = true`): sets `is_active = false`. Requires `manage_contacts` permission.
- **Restore access** (when `is_active = false`): sets `is_active = true`. Requires `manage_contacts` permission.

No delete action. Disabling/enabling portal access does not affect the contact record, memberships, registrations, or any other data.

## Phase 5 — Form collision path

Modify `FormSubmissionController::store()`. After the honeypot and timing checks, before any contact creation or update logic:

1. Check if the submitted email matches an `is_active` portal account.
2. If yes and the request is **not** authenticated as that portal account:
   a. Do not create or update any contact record.
   b. Return the standard neutral success response to the submitter (no signal that anything was blocked).
   c. Check a cache key `portal_collision_notified:{email}` — if set, skip the email. If not set, send the collision notification email and set the cache key with a 60-minute TTL.
3. If yes and the request **is** authenticated as that portal account: process normally.
4. If no match: existing logic unchanged.

### Collision notification email

Add `portal_form_collision` to `EmailTemplate` defaults and the `DatabaseSeeder`:

| field | value |
|-------|-------|
| handle | `portal_form_collision` |
| subject | `Someone submitted a form using your email address` |
| body | `<p>Hi {{first_name}},</p><p>A form on our website was submitted using your email address. If this was you, please <a href="{{login_url}}">log in to your account</a> and complete the action while signed in.</p><p>If this was not you, no action is needed — the submission was not processed.</p>` |

Available tokens: `{{first_name}}`, `{{login_url}}`.

## Security checklist

Before closing:
- [ ] `is_active = false` accounts cannot log in — the rejection is indistinguishable from wrong credentials
- [ ] All dashboard queries are scoped to the authenticated user's `contact_id`
- [ ] Notes field is not rendered anywhere on the portal dashboard
- [ ] Change password invalidates other sessions
- [ ] Change email does not take effect until the new address is verified
- [ ] Form collision blocks silently — no signal to the submitter
- [ ] Collision notification is rate-limited to one per email per hour
- [ ] Admin portal access panel only appears when a portal account exists for the contact
