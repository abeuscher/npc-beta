### Notable findings

- **Naming convention agreed mid-session.** The session prompt's original projector-key proposal — `date` (human start), `date_iso` (ISO start), `ends_at` (re-purposed as human end-time) — was rejected during design discussion as semantically opaque (`date_iso` conflates *which* date with *what format*; `ends_at`'s silent re-purposing from ISO timestamp to human-formatted time-of-day breaks both the existing key's contract and consistency with the start side). The convention adopted: source-column name carries the canonical timestamp; `_label` suffix carries the presentation companion. Applied uniformly to BlogListing in the same session — the field-rename was cheap (no live-data regression, app pre-Beta 1) and the alternative (BlogListing carries `date`/`date_iso` in perpetuity, EventsListing carries `starts_at`/`starts_at_label`) was strictly worse.
- **ThisWeeksEvents simpler than expected.** The session prompt anticipated possible template changes in `ThisWeeksEvents` if the `starts_at` rename to `date_iso` reached its template's `Carbon::parse($event['starts_at'])` call. Dropping `starts_at` from `ThisWeeksEvents`'s contract was unnecessary — the new naming convention keeps `starts_at` as the canonical ISO key in the projector, so `ThisWeeksEvents`'s template needed no edits and only `ends_at` (which the template never used) was dropped from its contract.
- **Sibling test fallout.** Six events-listing tests in `tests/Feature/ContentListingWidgetsTest.php` were rendering the EventsListing template directly via `Blade::render()` and passing the now-deleted `pageContext` data path. These were updated to mirror the BlogListing pattern that was already in the file: a new `eventsListingWidgetData()` helper resolves the contract via `ContractResolver::resolve()` and the template renders are passed `widgetData` directly. Six call sites updated; pattern now consistent across both widgets in this test file.
- **Eager-load query-skip caveat in Test B.** First version of Test B failed its `landingPage` query-count assertion (got 0, expected 1). Root cause: Laravel's belongsTo eager-load skips the secondary query when all parent foreign keys are null. Fix: factory-create three real `Page` rows up front and set each Event's `landing_page_id` to point to one. Carry-forward note for any future `SOURCE_SYSTEM_MODEL` retrofit whose contract uses a belongsTo eager-load.
