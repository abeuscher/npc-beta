# 253. Importer Mapping Page UX

Three focused UX improvements to the **Map Columns** step of the importer wizard, applied in two parallel-edited row-builder methods so all six wizards (contacts, donations, events, memberships, notes, invoice-details) inherit the change automatically. No refactor, no new abstractions — the shared infrastructure already exists.

---

## Design decisions (resolved before starting)

- **Status indicator shape.** Each mapping row carries a circular badge **on the left** of the row, ahead of the column header label.
  - **Incomplete** (destination is `null` / unmapped): white **×** glyph on a **red** background circle.
  - **Complete** (destination is non-null, including custom-field sentinels and relational sentinels like `__note_contact__`): white **✓** glyph on a **green** background circle.
  - "Sensitive header — always ignored" rows (where `FieldMapper::isSkipped()` returned true) render as **incomplete** — the operator gave no input, so the row hasn't been resolved by them yet. Helper-text already explains the auto-ignore; the icon stays red.
  - Implementation: pure CSS, no SVG inlining. The icon is a `::before` pseudo-element on the row container with `content: "\2715"` (×) or `content: "\2713"` (✓) and a circular background. Container class is `np-import-map-row` with state modifier `--complete` or `--incomplete`. CSS lives in [`public/css/admin.css`](../public/css/admin.css) under the existing `.np-*` BEM convention (hand-edited per `docs/app-reference.md`, no build step).
- **State-class binding.** Each row's wrapper is a `Forms\Components\Group` with a reactive `extraAttributes()` closure that reads the current `column_map.col_N` value via `Forms\Get` and emits the corresponding modifier class string. Filament re-renders the closure on Livewire roundtrips, so the icon flips automatically when the operator picks or clears a destination.
- **Reduced vertical sprawl.** Three changes, all CSS + minor Filament wrapping:
  - Wrap the per-column **Select** (the destination dropdown) in a `Forms\Components\Grid::make(N)` so the column-header label sits inline with the Select on wide viewports rather than stacked. The exact column ratio is a Phase 1 detail; the **target** is a single visual row with the indicator + label + Select on one line, sub-forms (custom-field, note-split, tag, org-strategy) appearing below only when their visibility predicate fires.
  - Tighten the visible sub-forms' padding so when a custom-field Grid expands beneath the Select, it doesn't add 80px of vertical real estate.
  - Tighten the row-to-row gap (currently default Filament Forms spacing — generous, well-suited to forms with 5–10 fields, painful at 80+).
- **Searchable dropdowns.** Add `->searchable()` to the per-column Select in both row builders. Filament's native search filters the optgroup-shaped destination list cleanly. No additional configuration — the `searchable` flag alone gives a search input atop the dropdown menu.
- **Out of scope: optional grouping by entity.** The "cluster contact-fields together / address-fields together / custom-fields together" idea from the prior stub is deliberately not in this session. Programmatic regex-shaped grouping is brittle; doing it well likely needs LLM/MCP-shape help that surfaces header rows to a model for clustering judgment. Revisit only on real-importer-use feedback. **Do not add a stub for this** — it reopens via a fresh conversation, not a stub-and-defer cycle.
- **Both row builders edited in parallel.** `buildNamespacedMappingRow()` in [`InteractsWithImportWizard`](../app/Filament/Pages/Concerns/InteractsWithImportWizard.php) covers donations / events / memberships / notes / invoice-details. `columnMappingRowSchema()` in [`ImportContactsPage`](../app/Filament/Pages/ImportContactsPage.php) covers contacts. The two are almost-identical in component shape; whatever wrapping/styling lands in one lands in the other in the same commit. Drift between them is a process failure.

## Phase 2 — Status indicator (visual completion state)

- In `InteractsWithImportWizard::buildNamespacedMappingRow()`:
  - Wrap the existing components (Select + sub-forms) in a `Forms\Components\Group::make([...])` whose `extraAttributes()` closure reads the row's current `column_map.col_N` value and emits a class string like `'np-import-map-row np-import-map-row--' . ($filled ? 'complete' : 'incomplete')`.
  - The `$filled` predicate: non-null, non-empty-string. Sentinels (`__custom_*__`, `__note_contact__`, `__tag_contact__`, `__tag_event__`, `__org_contact__`) all count as complete — they're explicit operator choices, not unmapped state.
- Mirror the same wrap in `ImportContactsPage::columnMappingRowSchema()`. Same class names, same predicate.
- Add the styling to `public/css/admin.css`:
  - `.np-import-map-row` — base flex container, sets the row's positioning context.
  - `.np-import-map-row::before` — the circular badge: `width: 1.5rem; height: 1.5rem; border-radius: 50%; display: inline-flex; align-items: center; justify-content: center; font-weight: bold; color: white;` plus `flex-shrink: 0` so it doesn't deform when the Select gets long.
  - `.np-import-map-row--incomplete::before` — `background-color: #dc2626; content: "\2715";` (Tailwind red-600 to match the admin palette).
  - `.np-import-map-row--complete::before` — `background-color: #16a34a; content: "\2713";` (Tailwind green-600).
  - Confirm contrast against the admin's dark-mode background; add `.dark .np-import-map-row--*` overrides only if the contrast suffers.

---

## Phase 3 — Reduced vertical sprawl

- Wrap the existing Select inside a `Forms\Components\Grid` so on a wide viewport the column-header label and Select sit on one line. Indicator (from Phase 2) is part of the row container's `::before`, so it doesn't take a Grid cell.
- Tighten `.np-import-map-row` padding: `padding: 0.5rem 0;` (vs Filament's default ~`1rem` per form item).
- Tighten the row-to-row gap by setting `.np-import-map-row + .np-import-map-row { margin-top: 0.25rem; }` or via the parent `space-y-*` Tailwind utility on the form's outer container if Filament exposes it.
- Sub-forms (custom-field Grid, note-split Grid, tag input, org-strategy Radio) stay vertically stacked beneath the Select when visible — their visibility predicates already gate them. Tighten **their** internal padding/spacing only if the visual stack feels heavy at this point; do not change their visibility logic.

---

## Phase 4 — Searchable dropdowns

- Add `->searchable()` to the per-column Select in `buildNamespacedMappingRow()`.
- Add `->searchable()` to the per-column Select in `columnMappingRowSchema()`.
- That's the entire change for this phase. Filament Select's native search supports optgroup-shaped option arrays (which is what the registries already produce via `groupedOptions()` / equivalent), so no additional configuration is needed. Verify in Phase 6 that the search filters across both group labels and option labels.
