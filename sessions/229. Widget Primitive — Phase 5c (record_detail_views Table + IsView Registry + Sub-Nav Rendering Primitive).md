# 229. Widget Primitive — Phase 5c (`record_detail_views` Table + IsView Registry + Sub-Nav Rendering Primitive)

Lift session 228's hardcoded `RecordDetailView` array into a real `record_detail_views` table; introduce the `ViewRegistry` that resolves Views by record type; build the sub-nav rendering primitive that drives Filament's `getRecordSubNavigation()` from registered Views; retrofit the Templates/Themes editor pages onto the same primitive (the absorbed Templates/Themes record sub-navigation refactor). Admin UI for authoring Views is **wholesale deferred** to a follow-up session — Views in 5c are PHP-seeded.

---

## Design decisions (resolved before starting)

- **`record_detail_views` is a real Eloquent-backed table.** Replaces 5b's hardcoded `RecordDetailView` PHP class array. Schema: `id` (uuid), `handle` (slug, unique per `record_type`), `record_type` (FQCN string), `label` (display name for sub-nav), `sort_order` (int), `layout_config` (jsonb, mirrors `page_layouts.layout_config` shape), timestamps. **No `is_default`, no `icon`, no `description`, no `visibility`, no `required_permission`.** Permissions are a known gap (see "Phase 5e — Permissions in the contract layer" in the track doc's "Pending paradigm questions"); explicitly out of scope for 5c.
- **Cardinality: schema allows N Views per record type, practice is usually 1.** Most record types (Contact, Event, Post) will have one View — its widgets render in the Filament edit page's footer slot, no sub-nav needed. The exception is Templates/Themes record types (`PageTemplate`, `ContentTemplate`) which have multiple Views (header, footer) that surface as sub-nav entries. Schema must support N from day one; the rendering primitive must handle "skip sub-nav when only one sub-page" as the common case.
- **`RecordDetailView` model implements `IsView`.** The PHP class introduced in 5b is replaced by an Eloquent model of the same name in `App\WidgetPrimitive\Views\RecordDetailView`. The `IsView` interface is unchanged. `widgets()` returns the `page_widgets` rows owned by this View via polymorphic relation (`owner_type='record_detail_view'`, `owner_id=view.id`). Mirror of `dashboard_configs` from session 215.
- **Polymorphic widget ownership via `page_widgets`.** Honors the polymorphic-owner discipline — no parallel `record_detail_view_widgets` table. Widgets attach with `owner_type='record_detail_view'`, `owner_id=view.id`. Identical pattern to `dashboard_configs` from 215.
- **`ViewRegistry` singleton resolves Views by record type.** New class at `App\WidgetPrimitive\ViewRegistry`. Method: `forRecordType(string $fqcn): Collection<RecordDetailView>` returns all Views bound to that FQCN, ordered by `sort_order`. Registered as a singleton in `WidgetServiceProvider::register()`. The registry is a thin Eloquent query wrapper — not a configuration cache (the table is the canonical source).
- **Sub-nav rendering primitive — Filament-native, mixed-source-aware.** The primitive is a trait (`HasRecordDetailSubNavigation` or similar) applied to the parent `EditRecord` page. Inside the trait, `getRecordSubNavigation()` returns a list of sub-nav entries assembled from two sources: (a) Views registered via `ViewRegistry::forRecordType($this->record::class)`, and (b) Filament-page sub-pages declared by the host page itself (returned from a new `additionalSubNavigationPages()` hook the trait calls — empty default). The order: Views first (by `sort_order`), then Filament-page sub-pages, then the host's "main" edit page itself. **The primitive must skip rendering sub-nav entirely when the total sub-page count is 1** (one View, no Filament pages → no sub-nav, the View's widgets render on the main edit page footer the same way 5b did).
- **View-backed sub-pages mount as a new `Filament\Resources\Pages\Page` subclass.** A new generic `RecordDetailViewPage` (extends Filament's `Page` or `EditRecord` depending on what the trait surfaces cleanly) takes a View handle as a route parameter, resolves the View, and renders its widgets through `RecordDetailViewWidget` (5b's Filament widget). One page class serves all Views — the route parameter selects which View to render. Mirror of how Filament's record sub-navigation works for related-resource pages.
- **Filament-form sub-pages stay as their own page classes — unchanged.** For `EditPageTemplate`, the existing SCSS and Colors tabs become two distinct sub-pages of `EditPageTemplate` (each its own Filament page class with its own form). The **Header** and **Footer** tabs become two `record_detail_views` rows bound to `PageTemplate::class`, rendered through the View-backed sub-page. Sub-nav primitive renders all four side by side; the kind of each sub-page is invisible to the user.
- **No widget contract changes.** No retrofits, no `formatHints` additions, no schema edits to `page_widgets`. The session is additive.
- **`Contact` keeps its 5b mount unchanged for now.** Session 228 mounted `RecordDetailViewWidget` on `EditContact` via `getFooterWidgets()`. After 5c, `EditContact` retains exactly one View (`contact_overview`) and the placeholder still renders in the footer slot — no sub-nav. The mount mechanism stays `getFooterWidgets()` for the single-View case; the sub-nav primitive coexists for the multi-View case (Templates/Themes).
- **Seeding: a small `RecordDetailViewSeeder` ships with 5c.** Seeds three Views: `contact_overview` (binding to Contact, holds the placeholder widget — replaces 5b's hardcoded array), `page_template_header` and `page_template_footer` (binding to PageTemplate). The seeder is the source of truth until the admin UI lands in a follow-up session.
- **No `getBreadcrumbs()` override needed for the View-backed sub-pages.** They get a default breadcrumb shape from Filament's standard `EditRecord` machinery; the trait sets up the sub-page name/label from the View's `label` field.

## Phase 1 — `record_detail_views` table + `RecordDetailView` Eloquent model

1. Write migration `database/migrations/YYYY_MM_DD_HHMMSS_create_record_detail_views_table.php`:
   - `id` uuid primary
   - `handle` string, indexed
   - `record_type` string, indexed
   - `label` string
   - `sort_order` integer, default 0
   - `layout_config` jsonb, nullable
   - timestamps
   - unique index on (`record_type`, `handle`)
2. Run the migration: `docker compose exec app php artisan migrate`.
3. Update `docs/schema/`: create `docs/schema/record_detail_views.md` following the existing per-table doc convention; add the entry to `docs/schema/README.md`'s table index alphabetically.
4. **Replace** `app/WidgetPrimitive/Views/RecordDetailView.php` (currently a plain PHP class introduced in 5b) with an Eloquent model:
   - Extends `Illuminate\Database\Eloquent\Model`, uses `HasUuids`, `HasFactory`
   - `$fillable`: `handle`, `record_type`, `label`, `sort_order`, `layout_config`
   - `$casts`: `layout_config => 'array'`
   - Implements `App\WidgetPrimitive\IsView` (interface declared in 5b — unchanged)
   - `slotHandle(): string` returns `'record_detail_sidebar'`
   - `widgets(): array` returns the polymorphic `page_widgets` rows: `PageWidget::query()->where('owner_type', 'record_detail_view')->where('owner_id', $this->id)->where('is_active', true)->with('widgetType')->orderBy('sort_order')->get()->all()`. Mirror of `DashboardConfig::widgets()` from 215.
   - `recordType(): string` returns the stored `record_type` field.
   - `layoutConfig(): array` returns the stored `layout_config` field (array-cast, defaults to empty array).
   - `handle(): string` returns the stored `handle` field.
   - Polymorphic relation method on the model itself (`pageWidgets(): MorphMany`) for the inverse direction.
5. Create `database/factories/RecordDetailViewFactory.php` with sensible defaults.

## Phase 2 — `ViewRegistry` singleton

1. Create `app/WidgetPrimitive/ViewRegistry.php`:
   - `forRecordType(string $fqcn): \Illuminate\Support\Collection` — returns all `RecordDetailView` rows where `record_type = $fqcn`, ordered by `sort_order` then `created_at`.
   - `findByHandle(string $recordType, string $handle): ?RecordDetailView` — returns a single View by composite key (used by the route parameter on View-backed sub-pages).
2. Register in `app/Providers/WidgetServiceProvider.php` as a singleton (mirror of `SlotRegistry` registration pattern).

## Phase 3 — Sub-nav rendering primitive

1. Create `app/Filament/Concerns/HasRecordDetailSubNavigation.php` (a trait):
   - `getRecordSubNavigation(): array` — Filament's standard method. Implementation: assemble entries from `ViewRegistry::forRecordType($this->record::class)` + `additionalSubNavigationPages()` (a hook returning an array of page classes, default empty). If total count ≤ 1, return `[]` (no sub-nav rendered, the View's widgets render in the host's footer slot).
   - `additionalSubNavigationPages(): array` — overridable hook. Default returns `[]`. `EditPageTemplate` overrides to return SCSS and Colors page classes.
2. Create `app/Filament/Resources/Pages/RecordDetailViewPage.php` — the generic Filament page that renders one View's widgets:
   - Extends an appropriate Filament base class (`Page` or `EditRecord`, whichever the existing record-detail page hierarchy supports cleanly — read first to confirm).
   - Receives the `view` handle as a route parameter.
   - Resolves the View via `ViewRegistry::findByHandle($recordType, $handle)`.
   - Renders the View's widgets via the same `RecordDetailViewWidget` mounted in 5b (passes the View instance instead of building one inline).
3. **Update** `app/Filament/Widgets/RecordDetailViewWidget.php`:
   - Remove the inline hardcoded `RecordDetailView` construction in `view()`.
   - Add a public `?RecordDetailView $view = null` property (in addition to the existing `?Model $record` property).
   - When `$view` is set, render its widgets directly. When `$view` is null (footer-mount case from 5b on `EditContact`), fall back to looking up the single View bound to `$record::class` via `ViewRegistry::forRecordType()->first()`. This preserves 5b's `EditContact` behaviour byte-equivalent (the seeded `contact_overview` View resolves to the same widget set).
4. Apply the trait to `EditContact` — adds nothing visible (Contact has 1 View, sub-nav is skipped) but ensures the trait is exercised on the canonical record type.

## Phase 4 — Templates/Themes refactor (consumer of the primitive)

1. Read the current state of `app/Filament/Resources/TemplateResource/Pages/EditPageTemplate.php` and `EditContentTemplate.php` to map the existing tabs to the new structure.
2. Convert `EditPageTemplate`:
   - Apply the `HasRecordDetailSubNavigation` trait.
   - The Header tab and Footer tab → become `record_detail_views` rows (`page_template_header`, `page_template_footer`, both bound to `PageTemplate::class`). The seeder seeds them empty (or with whatever placeholder is appropriate for first-cut).
   - The SCSS tab and Colors tab → extracted into two new Filament page classes: `EditPageTemplateScss` and `EditPageTemplateColors`, each its own page with its own form. Override `additionalSubNavigationPages()` to return `[EditPageTemplateScss::class, EditPageTemplateColors::class]`.
   - The main `EditPageTemplate` page becomes the entry point — its own form (the General fields, if any) renders on the main edit page; sub-nav surfaces the four sub-pages.
3. Same shape for `EditContentTemplate` — convert its tabs into View-backed and Filament-form-backed sub-pages per the existing tab layout. Read first to confirm the tab structure.
4. **Important — preserve existing form data.** The SCSS and Colors tabs' fields are stored on the `templates` table (or wherever they currently live). Moving them to their own Filament pages doesn't migrate any data; the form fields are just rendered on a different page. Verify nothing relies on the old tab structure for save semantics.
5. Update `EditPageTemplate`'s tests if they assert on the tab structure — replace with sub-nav assertions (the new pattern).

## Security checklist

- [ ] `record_detail_views` table has no PII or sensitive columns; the schema is configuration metadata only. Widget data flows through the existing resolver per 5b's wiring.
