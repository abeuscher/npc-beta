# 266. Backup notification hardening — FM 020 finding

Drop spatie's mail-channel notification on the **success path** so it can no longer kneecap `RecordBackupSuccess`, and bridge `SiteSetting` into the remaining `backup.notifications.mail.*` config so spatie's failure-path notifications use the same FROM/TO/transport that the rest of the application's mail uses. Closes a structural fragility surfaced during FM session 020 manual testing on 2026-05-05.

Working-set placement: lifted as **A1d'** — a follow-on to A1d (Fleet Manager Contract v2.2.0 — Backup Trigger Endpoint, closed at session 263), mirroring the B2b → B2b' pattern. This is **not** boundary-touching — no changes to the agent contract surface, no contract version bump.

---

## Background — the FM 020 finding

FM 020 ships a "Trigger backup now" Filament action that POSTs to `/api/backup/trigger`. First manual test against `beuscher.net`, the BackupController returned the `200 + status: failed` envelope with `message: "backup:run exited cleanly but success record was not updated"` — the integrity guard fired.

Tracing the failure:

1. `Artisan::call('backup:run')` succeeded — zip created, copied to local disk, "Backup completed!" logged, exit 0.
2. `BackupWasSuccessful` event dispatched.
3. **Spatie's own listener** (mail notification) ran first in the listener chain. It called `Mail::send` through Resend with FROM = `hello@example.com` (spatie's default).
4. Resend rejected: *"The example.com domain is not verified."*
5. The throw propagated out of `Illuminate\Events\Dispatcher::invokeListeners` → subsequent listeners didn't run → **`App\Listeners\RecordBackupSuccess` never fired** → success-record file at `storage/app/private/fleet/last-backup-at` not written.
6. BackupController's integrity-guard cross-check correctly caught: artisan exited clean, success record didn't move forward → returned `failed` envelope.

The integrity guard did its job. The bug is upstream of it: spatie reads `env('MAIL_FROM_ADDRESS', 'hello@example.com')` directly, which bypasses the CRM's `SiteSetting`-driven mail-config pattern. `printenv | grep MAIL_FROM` on the deployed container is empty; `mail.from.address` resolves correctly to `info@beuscher.net` only because [`AppServiceProvider::boot()`](../app/Providers/AppServiceProvider.php) line 73 overrides it from the `site_settings` table at runtime. Spatie's parallel config keys are not in that override list.

Two structural notes captured FM-side worth lifting here:

- **`'to' => 'your@example.com'` placeholder in [`config/backup.php`](../config/backup.php) line 212** — even after the FROM resolves, those notification emails would go to the unconfigured spatie default. Either bridge to a real address from `SiteSetting` or drop the relevant mail channel entries.
- **Listener-chain fragility on the success path** — any future throw in any listener that sits ahead of `RecordBackupSuccess` (spatie's notification path being the most likely culprit) silently kneecaps the integrity-record write. The integrity guard still catches it, but the operator sees a misleading "Backup failed" panel for what is actually a successful backup with a notification problem. Robust fix: drop spatie's mail channel on the **success path** entirely. FM's audit-mirror email captures the success signal via `client.backup.succeeded`; spatie's per-CRM email is duplicative.

## Design decisions (resolved before starting)

- **Drop `['mail']` from spatie's success-class notifications.** `BackupWasSuccessfulNotification`, `HealthyBackupWasFoundNotification`, `CleanupWasSuccessfulNotification` all flip from `['mail']` to `[]`. Rationale: FM-side audit mirror covers operator notification on success (`client.backup.succeeded` + `client.backup.cleanup.succeeded` if/when that lands). Per-CRM email is duplicative AND structurally fragile — any throw from spatie's mail handler kneecaps `RecordBackupSuccess` regardless of which env var or config bridge is involved. Removing the listener removes the throw-surface, not just patches the current symptom.
- **Keep `['mail']` on spatie's failure-class notifications.** `BackupHasFailedNotification`, `UnhealthyBackupWasFoundNotification`, `CleanupHasFailedNotification` stay mailed. Rationale: failure-class notifications are the FM-down redundancy layer — if FM is up, FM-side audit mirror catches the `client.backup.failed` / `transport_failed` events; if FM itself is down, the per-CRM email is the only operator signal that the daily 01:30 backup blew up. Don't lose that.
- **Bridge `SiteSetting` into the remaining failure-path mail config in `AppServiceProvider::boot()`**. Three new override lines alongside the existing `mail.from.address` / `mail.from.name`:
  - `'backup.notifications.mail.from.address' => $settings->get('mail_from_address')?->value ?: 'hello@example.com'`
  - `'backup.notifications.mail.from.name' => $settings->get('mail_from_name')?->value ?: config('app.name')`
  - `'backup.notifications.mail.to' => $settings->get('contact_email')?->value ?: 'hello@example.com'`
  This eliminates the `'your@example.com'` hardcoded placeholder for the TO recipient and makes spatie's mail layer use the same SiteSetting-driven values as everything else in the app.
- **`SiteSetting` schema is unchanged.** All three keys (`mail_from_address`, `mail_from_name`, `contact_email`) already exist in the settings table per the existing override block. No new keys, no migration.
- **No agent contract change.** This is internal to the CRM. The `/api/backup/trigger` envelope shape, the integrity-guard semantics, the `last_backup_at` mtime cross-check — all unchanged.
- **No FM-side change required.** The fix is structurally self-contained CRM-side. FM 020's `BackupTriggerClient` will see `status: success` envelopes after this session ships and the deployed CRM is updated.

## Phase 1 — `config/backup.php` notifications hygiene

**Goal:** spatie's mail channel disabled on the success path, kept on the failure path.

**Steps:**

1. Edit [`config/backup.php`](../config/backup.php) `notifications.notifications` block (lines ~196–203):
   ```php
   'notifications' => [
       \Spatie\Backup\Notifications\Notifications\BackupHasFailedNotification::class => ['mail'],
       \Spatie\Backup\Notifications\Notifications\UnhealthyBackupWasFoundNotification::class => ['mail'],
       \Spatie\Backup\Notifications\Notifications\CleanupHasFailedNotification::class => ['mail'],
       \Spatie\Backup\Notifications\Notifications\BackupWasSuccessfulNotification::class => [],   // was ['mail']
       \Spatie\Backup\Notifications\Notifications\HealthyBackupWasFoundNotification::class => [],  // was ['mail']
       \Spatie\Backup\Notifications\Notifications\CleanupWasSuccessfulNotification::class => [],   // was ['mail']
   ],
   ```
   The three success-class notifications go to empty array (no channels). The three failure-class notifications stay on `['mail']`.

2. **Do not edit** the `'to' => 'your@example.com'` placeholder in this file. Phase 2's `AppServiceProvider` bridge overrides it at runtime from `SiteSetting`. Leaving the file's literal value as-is keeps the diff against spatie's published config narrow and avoids an inert constant fighting the runtime override.

## Phase 2 — `AppServiceProvider` bridge to spatie config

**Goal:** spatie's failure-path mail uses the same SiteSetting-driven FROM/TO that the rest of the application uses.

**Steps:**

1. Edit [`app/Providers/AppServiceProvider.php`](../app/Providers/AppServiceProvider.php), inside the `try { ... } catch` block in `boot()` that sets `mail.from.address` (around line 73). Add three sibling overrides:
   ```php
   'mail.from.address' => $settings->get('mail_from_address')?->value ?: 'hello@example.com',
   'mail.from.name'    => $settings->get('mail_from_name')?->value    ?: config('app.name'),
   // New — bridge SiteSetting into spatie's parallel config keys
   'backup.notifications.mail.from.address' => $settings->get('mail_from_address')?->value ?: 'hello@example.com',
   'backup.notifications.mail.from.name'    => $settings->get('mail_from_name')?->value    ?: config('app.name'),
   'backup.notifications.mail.to'           => $settings->get('contact_email')?->value     ?: 'hello@example.com',
   ```
   Defaults match the existing override block's defaults — no behaviour change when SiteSetting is empty.

2. **Demo-mode parity.** The existing `if (isDemoMode())` block on line 93 forces `mail.default` to `log`. Spatie reads its own mail driver via the standard Laravel mailer (no separate driver config), so this already covers spatie's mail in demo mode. No additional override needed.

## Phase 3 — Tests

**Goal:** regression guard against re-introducing either fragility (success-channel mail OR unbridged from-address).

**New test file: `tests/Feature/Backup/SpatieBackupNotificationConfigTest.php`** — three tests:

1. The three success-class notifications resolve to `[]` in `config('backup.notifications.notifications')`.
2. The three failure-class notifications resolve to `['mail']` in the same config block.
3. With `SiteSetting` rows seeded for `mail_from_address` / `mail_from_name` / `contact_email`, the three new `backup.notifications.mail.*` config keys reflect those values. Trigger the bridge by calling `(new AppServiceProvider(app()))->boot()` directly after seeding — the boot block is idempotent (just `config([...])` calls). No precedent for a `config:reload` helper exists in the suite.

**New test file: `tests/Feature/Backup/RecordBackupSuccessListenerTest.php`** — one test:

4. The listener writes `storage/app/private/fleet/last-backup-at` with a current ISO-8601 timestamp when invoked. Invoke the listener directly with a Mockery stub of `BackupWasSuccessful` — the listener ignores the event payload, so a stub avoids the awkward `BackupDestination` construction that `event(new BackupWasSuccessful(...))` would require.
