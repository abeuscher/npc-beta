# 187. Content Template Editor & Per-Type Defaults

Make content templates first-class: give them a real widget editor (the existing Vue page builder, template-aware), let any page type pick one at creation, and wire a per-page-type default so new pages/posts/events can hydrate their starter widgets from a template instead of hardcoded fallbacks.

---

## Background

Most of the plumbing already exists:

- `templates.definition` (JSONB) already stores a full widget stack — handles, configs, appearance, query config, sort order.
- "Save as Template" (in `app/Livewire/PageBuilder.php` and `EditPageDetails`) already serialises a live page's widget stack via `PageWidget::serializeStack()`.
- `CreatePage::hydrateWidgets()` already deserialises a template's definition into `PageWidget` + `PageLayout` records linked to the new page.
- Token substitution (`{{title}}`, `{{author}}`, `{{date}}`, etc.) already runs at render time against the widget's owning page — so a template Hero with tokens Just Works once hydrated onto a post.

What's missing:

1. **No editor for a content template's widget stack.** `EditContentTemplate.php` is a bare name + description form. The only way to produce a definition today is to design on a scratch page and invoke "Save as Template."
2. **Blog posts can't pick a template at creation.** Currently excluded — only page-type pages and some others expose the `content_template_id` selector. This session removes that exclusion.
3. **No per-page-type default.** Each creation form requires the user to manually pick a template (or get nothing). We want a fallback: if the user doesn't pick, the per-page-type default hydrates.

The user still has total control after creation. A template is a *starting point*, not a contract — once widgets are on the page, they're owned by the page and can be added, removed, or reconfigured freely. Changing the template later (or editing it) does **not** retroactively touch pages already hydrated from it.

---

## Design decisions (resolved before starting)

- **Editor approach: template-aware page builder API** (not the scratch-page pattern). Add routes that read/write `Template.definition` directly and teach the Vue store that it can run in "template mode." More invasive up front than the scratch-page trick, but cleaner long-term and avoids a hidden-system-page indirection for every content template.
- **Template scope stays flat.** No `applicable_to` field, no per-page-type filtering of the template dropdown. The full list shows at every page-type's creation form. (Tagging / scoping deferred — can be added later if the list grows unwieldy.)
- **Per-page-type defaults live on `site_settings`.** One setting per page type: `default_content_template_page`, `default_content_template_post`, `default_content_template_event`, etc. No new table. Empty string = no default (user starts blank unless they pick one).
- **Default applies only when the user does not make a selection.** "None" must be a first-class choice on the creation form. Selecting it produces a blank page regardless of the default.
- **No re-apply on template edit.** Editing a content template never touches pages that already hydrated from it. Confirmed by the existing one-shot copy flow in `hydrateWidgets()` — no back-reference from `PageWidget` to the source template.
- **Blog posts get the selector.** Remove the current exclusion on `post` type; the template dropdown + default lookup applies to `type = post` identically to other page types.
- **Tokens for template widgets.** Already wired (session 186). No work needed this session — template widgets hydrated onto a post render with `{{title}}` / `{{author}}` / `{{date}}` resolving to the post itself.

## Phase 1 — Template-aware page builder API

Extend the page-builder API so it can operate on a `Template.definition` instead of a page's `PageWidget` records.

- Add routes (likely under `/admin/template-builder/{template}` mirroring the shape of `/admin/page-builder/{page}`) for: widget create / update / delete / reorder, layout create / update / delete, widget preview HTML, widget picker data, appearance updates. Map each to `Template.definition` read/write instead of DB rows.
- Introduce a small abstraction that represents "the widget stack being edited" so the controller can dispatch to either `PageWidget`-backed storage or `Template.definition`-backed storage without the Vue app caring which it's hitting. Keep as thin as possible — avoid rewriting the whole editor pipeline.
- Preview rendering: template widgets need preview HTML too. Since there's no page, build a synthetic `PageWidget`-like shape from the definition entry and feed it through `WidgetPreviewRenderer`. Token substitution in preview falls back gracefully when there's no owning page (renders tokens literally or with a sentinel — decide during implementation, keep it obvious to the editor).

---

## Phase 2 — Editor UI for content templates

Replace the bare form in `EditContentTemplate.php` with a full editor.

- Add a tab structure on `EditContentTemplate` (match the pattern in `EditPageTemplate`): "Details" (name, description) + "Widgets" (embeds the Vue page-builder in template mode).
- Bootstrap the Vue app with the template ID and a flag that routes its API calls to the template endpoints from Phase 1.
- "Save as Template" from a page continues to work — it's a one-shot serialise. After that, the template is edited via the new editor; the scratch-page flow becomes optional.

---

## Phase 3 — Creation-time template selection + per-type defaults

- Remove the "blog posts can't have templates" exclusion in `PostResource` (and/or `PageResource` where post type is filtered). The `content_template_id` selector appears on every page-type's create form, defaulted to the per-type default, with "None" as a selectable option.
- Add per-page-type default settings in `site_settings`: `default_content_template_{type}` for each page type. Surface these in an admin settings page (likely CMS Settings, or a new sub-page — decide during implementation based on where it sits naturally).
- `CreatePage::afterCreate()` (and any equivalent on other page-type create flows — `CreateEvent`, etc.) looks up the default when the form's `content_template_id` is unset, and hydrates from it. If the user picked "None," hydrate nothing.
- Leave existing hardcoded page-type fallback widgets in place *only* until a default template is configured — once a default exists for a page type, it supersedes the hardcoded fallback. Consider whether to remove the hardcoded fallbacks entirely; if they're kept as a last-resort, document why.
