## What was built

### Phase 1 — Admin IA (items 1–4)

- **CMS-Settings surface relabel (`CmsSettingsPage.php`).** Nav label `CMS` → **`Site`**, page title `CMS Settings` → **`Site Settings`**. Kills the "CMS" double-meaning (there is a separate `CMS` *nav group* holding Theme/Templates/Pages) and names the bucket's real job — site identity & defaults — consistent with the arc's three-bucket model. Class name, route, view, and all `SiteSetting` keys unchanged. Kept in the `Settings` nav group with General/Mail/Finance (no wholesale settings-IA redesign).
- **"Header & Footer Code Snippets" → two accurate sections (same file).** The mislabeled section actually held the favicon + default OG image + three raw code snippets. Split into **"Site Icon & Social Image"** (favicon upload + preview, default OG image upload + preview) and **"Custom Code Snippets"** (head / body-open / body snippets). Same fields, same `SiteSetting` keys; the single `header-footer` section-save id became two internal ids (`site-imagery`, `code-snippets`) with matching `persistSection` arms — behaviour-preserving (identical keys written).
- **Stripe-checkout branding → Finance.** The whole `Stripe Checkout — Branding` section moved verbatim from `CmsSettingsPage` to `FinanceSettingsPage` as a self-contained **"Checkout & Statement Branding"** section (after Payment Methods, before QuickBooks). Moved with it: the `mount()` fill keys, the `stripeBrandingImageBlock()` helper, the `persistStripeCheckoutBranding()` method, the `save()` call, and a `stripe-checkout-branding` `persistSection` arm. Identical `SiteSetting` keys, upload collections, and validation. Consequence (intended, more-correct, recorded as a deferred-decision note below): the section is now behind Finance's `manage_financial_settings` gate instead of `manage_cms_settings` — a financial setting behind the financial permission.
- **Page-vs-content template IA clarity (`TemplateResource.php`).** One resource kept; the existing `type` badge column surfaced by default (was `isToggledHiddenByDefault: true`) with clearer labels ("Page template" / "Content template") and made sortable, plus a new Page/Content **Type** `SelectFilter`. Routes and the existing `EditPageTemplate` / `EditContentTemplate` edit pages unchanged — labelling/surfacing only, not a second resource.

### Out-of-session-scope work folded in at user request

- **CI/deploy fix — Playwright e2e `vendor/` mount.** `docker-compose.e2e.yml` bind-mounts `.:/var/www/html`, which shadows the image-baked, fully-autoloaded `vendor/`; `node_modules` was already protected with the `app_node_modules_e2e` named volume but `vendor/` was overlooked. On a dev machine the host has `vendor/` so it's masked; in CI the Playwright job never runs `composer install`, so the empty bind-mount overlaid the image vendor → `artisan` could not load `vendor/autoload.php` → `migrate:fresh` fatal in Playwright `globalSetup`. Fix: added a named volume `app_vendor_e2e:/var/www/html/vendor` to the `app` and `worker` services and to the top-level `volumes:`, exactly mirroring `app_node_modules_e2e` (nginx excluded — it executes no PHP). Compose config validated; volume wiring confirmed. Separate, clearly-scoped commit (`b0fdb26`). The companion Postgres-host CI fix was confirmed already on `main` (`bca2914`) — no action.

## Security considerations

The functional change with a security axis is the Stripe-branding relocation: it now sits behind `FinanceSettingsPage`'s `manage_financial_settings` permission instead of `CmsSettingsPage`'s `manage_cms_settings`. This is the intended, more-correct posture (a financial setting governed by the financial-settings permission) — a tightening that aligns with the catalogued IA mismatch, not a regression; same `SiteSetting` keys, no data-path or value change. No auth, routes, queries, or response shape touched. CRM contract unchanged at v2.3.0; no Fleet / `/api/health` surface. The folded e2e fix is a CI-only Docker named-volume change (no secret, no app code, no production surface).
