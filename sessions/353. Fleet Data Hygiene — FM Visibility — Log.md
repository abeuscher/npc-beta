## What was built

Additive contract bump **v2.3.0 → v2.4.0**.

### The `data_hygiene` subcheck (`HealthController`)

- New seventh subcheck, slotted after `version`. Source is the build-once 352 seam — `DataHygieneAudit::counts()` — consumed, not re-derived.
- **`value` = the four-category breakdown object** `{orphan_event_pages, scrub_records, orphan_media_dirs, dead_owner_media}` — all non-PII aggregate integers. Richer than a single total and still count-only; FM can render a useful per-node line.
- **Status: informational, never red.** `green` while the total across the four categories is under a soft `threshold` (a single integer, default **100**, adjustable), soft **`yellow`** at/above it so a heavily-crufted node self-surfaces for attention. There is no path to `red` — a cruft pile is not a node-health emergency. `message` is `null` when green, a **count-only** summary (`"<n> items of accumulated cruft (counts only)"`) when yellow — never a record identifier.
- **Excluded from the worst-of overall status.** `overallStatus()` now derives over every subcheck *except* `data_hygiene` (`array_diff_key`), so benign cruft never drags a node's top-level `status` to yellow. Every other subcheck rolls in unchanged. This is the one structural change to the health semantics; documented in the contract's worst-of rule.

### Count-only guarantee (the governing privacy boundary)

Only aggregate integers ever cross the FM wire. No record, slug, title, id, email, file path, or content is surfaced. The deep records mode (`app:data-hygiene --deep`) stays node-local + consent-gated and is **not** exposed here. The surface is shaped so it *can't* read data, not policied not to — `value` is literally the four ints from `counts()`.

### Caching / compute strategy (a frequently-polled endpoint)

`counts()` walks the media filesystem (`orphanMediaDirectories()`) and scans the media table (`deadOwnerMedia()`), and `/api/health` is polled ~once a minute per node. The subcheck reads the counts through `Cache::remember` with a **~10-minute TTL** (`HealthController::DATA_HYGIENE_CACHE_KEY`). Cruft accumulates over hours/days, so a 10-minute-stale count is operationally identical to a live one, while the expensive path runs at most a few times an hour regardless of poll rate. A scheduled precompute was rejected — the worker runs no `schedule:work` (the standing scheduler-runner gap, `docs/app-reference.md`), so a scheduled job would never fire. No schema, no new dependency.

## Security considerations

The whole session is a deliberate exercise of the data-access privacy boundary: Fleet Manager must never be *built* to read raw node data. The subcheck's `value` is four integers from `counts()`; the design makes raw-record exposure structurally impossible rather than policy-gated. A Pest case asserts the count-only guarantee (value is named integers; the only non-integer field is the uniform envelope's status/threshold/message, and `message` carries only the numeric total). No new endpoint, no auth change, no migration. The `--deep` records mode stays node-local. mTLS gate, throttles, and the rest of the contract surface are untouched.
