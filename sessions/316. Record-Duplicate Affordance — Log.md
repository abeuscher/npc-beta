## What was built

A single-record **Duplicate** action on Events, Pages, Posts, and Products at two surfaces each (list-row action + record-page header action) — eight affordances. Each duplicate lands as a **draft**, titled "Copy of …" with a `-copy` slug (uniqueness-incremented), authored by the current user, and redirects into the editor. Pattern: a `duplicate()` method on each model encapsulates the per-resource semantics; the Filament actions are thin wrappers (confirmation modal → `$record->duplicate()` → notification → redirect).

### Phase 1 — Pages (pattern source)

- `Page::duplicate()` — slug `-copy`, status draft, `published_at` cleared, `source` reset to `human`, author = current user, child widgets/layouts cloned via `PageWidget::copyOwnedStack`, tags re-pointed to the same rows, own media collections copied. Hidden for `system` pages.
- List-row action in `PageResource`; record action added to the shared `HasPageSecondaryActions` ellipsis group (so EditPage and EditPageDetails both expose it).
- **Two bugs found and fixed in `copyOwnedStack` (surfaced by the user testing Pages):**
  - `copyMediaTo` iterated `getMedia()` (default Spatie collection only) — widget media lives in `appearance_background_image` and `config_{key}` collections, so every widget background image / video / config image was silently dropped on duplicate (and on the existing "Save as Template" / content-template flows). Fixed to iterate `$source->media` (all collections).
  - The layout-clone branch copied `layout_config` but omitted `appearance_config` — column/section backgrounds (color, gradient, alignment, fit) were dropped. Fixed to carry `appearance_config`.
  - Verified on the isolated e2e stack: a duplicated Hero page renders pixel-identical to the original (background + CTA buttons intact). Buttons were never lost — they live in the `config` jsonb; the missing-buttons symptom was the Hero rendering collapsed without its background.

### Phase 2 — Posts

- Reuses `Page::duplicate()` (posts are `Page` records scoped `type='post'`); copy keeps `type='post'`.
- List-row action in `PostResource`; record action via the new `HasPostSecondaryActions` trait (see Phase 5).

### Phase 3 — Products

- `Product::duplicate()` — name "Copy of …", slug `-copy`, draft, `published_at` cleared, `source` reset to human, `is_archived` reset to false. Price tiers cloned and re-pointed with `stripe_price_id` omitted so `ProductPriceObserver` re-mints Stripe prices on save (no-op when Stripe is unconfigured). Product image copied.
- List-row action in `ProductResource`; record action in the `EditProduct` ellipsis group. Gated on `create_product`.

### Phase 4 — Events

- `Event::duplicate()` — title "Copy of …", slug `-copy`, draft, `published_at` cleared, `source` reset to human, author = current user. `starts_at`/`ends_at` **carry forward** (duplicate-a-recurring-event use case, per user sign-off). Ticket tiers cloned and re-pointed; tags re-pointed; event media copied. `landing_page_id`, `registrants_deleted_at`, `import_session_id`, and registrations are intentionally **not** copied — the copy has no landing page (created on demand) and no attendees.
- List-row action in `EventResource`; record action in the `EditEvent` ellipsis group. Gated on `create_event`.

### Phase 5 — Builder edit-view UX pass (user-requested at close)

- Replaced the floating "Save as Template" footer button in the page-builder Vue app with a primary **"Save changes"** button (bottom-right on both page and post builder edit screens). It fires a cosmetic "Saved" Filament toast via a Vue→Livewire bridge (`PageBuilder::saveChanges`); the builder still auto-saves live edits — the button gives operators the save affordance every other record has and seats the UX for a future explicit-save model. (User noted the old placement caused accidental "Save as Template" clicks while editing.)
- Save-as-Template now lives only in the header ellipsis (no longer duplicated as a footer button).
- Added `HasPostSecondaryActions` (mirrors `HasPageSecondaryActions`: Duplicate, exports, Save as Template, snippets, Delete) applied to both `EditPost` and `EditPostDetails`, so the post edit view gained the ellipsis it lacked and both post screens match the page screens item-for-item. Confirmed on the isolated stack.

## Security considerations

Each duplicate action is permission-gated (`create_page` / `update_page` for posts / `create_product` / `create_event`) with an `abort_unless` inside the handler, matching the surrounding resource conventions. Duplication produces drafts (never auto-published) and resets `source` to `human`, so a copied scrub/import record does not inherit scrub-inheritance constraints. No portal/contact-data surfaces touched. The `copyMediaTo` widening copies more media than before but only between records the operator already controls.

## Deferred decisions

- **Background-image CSS escaping (pre-existing, flagged to housekeeping):** the renderer emits `background-image:url(...)` unquoted, so filenames containing parentheses or spaces (e.g. `photo (1).jpg`) break the CSS on the public site generally — affects originals as well as copies, unrelated to duplication. Folded to the housekeeping inbox.
- **Duplicate for other resources** (Donations, Memberships, Notes, Invoices) out of scope; the `duplicate()`-method + thin-action pattern established here ports cleanly if wanted later.
- **"Save changes" is cosmetic for now** — wired through a real save hook so an explicit-save model is a small change rather than a new feature.
