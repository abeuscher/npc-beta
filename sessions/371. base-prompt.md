## Process rules for every session

Base-prompt process rules apply in full (`sessions/template-base-prompt.md`). Session-specific:

- **Audit findings default to fix, surfaced inclusively** (standing project rule): surface every finding in the audit writeup (W7/W8/W11/W12/Open Flags shape); a "won't-fix" is decided at the apply walkthrough with the owner, not filtered out on the audit side.
- **Integration seams first** (standing project rule): most issues here are two existing systems wired together slightly wrong, not greenfield gaps. The default hypothesis for a missing-sanitizer path is "a write path that skips the existing model mutator," not "the sanitizer is broken." The sanitizer itself was rated genuinely solid at 368 — audit the *paths into it*, don't re-implement it.
- **Security over usability** (standing project rule): when the two directly conflict — e.g. the exposure gate blocks a convenient publish — choose security and move the friction to the UX layer (a warning gate, an indicator), surfacing the conflict rather than silently weakening protection.
- **CSV formula-injection technique is a design call** — the neutralization method (prefix `=`/`+`/`-`/`@` cells so they don't execute on re-open in Excel/Sheets) is decided at the Open Gate; apply it uniformly across the list-export family, not per-resource.
- **Member-page scoping is the portal security rule applied to CMS** — every data widget on a `type=member` page must scope to the authenticated portal user's own `contact_id`, no exceptions without design review.
