## Binding security constraints (not advisory)

A public, unauthenticated button that auto-logs an anonymous visitor into the admin panel as a shared account is the load-bearing security surface of this session. These are acceptance criteria:

- **Dedicated demo role, never `super_admin`.** The shared "Demo User" gets a purpose-built, tightly-scoped role. No access to: mail/integration/secret settings, API keys, fleet/backup/destructive actions, user/role management, or any surface that can exfiltrate real data or change instance configuration. Demo-data only.
- **Demo-server-only.** The auto-login route is inert on non-demo installs — gate on the existing demo-server/demo-mode identity signal (the same one `isDemoMode()` keys on). A production install must never expose a public auto-login. Pest must assert the off-demo guard returns 404/403 and does not authenticate.
- **No privilege bleed through the shared session.** Because the account is shared, one visitor's session must not expose another's in-flight state beyond what the 24h wipe already accepts. Don't add per-visitor identity (that would reintroduce the email/form the architecture deliberately removed) — scope risk via role + wipe cadence, per `[[demo-page-architecture]]`.
- **Rate-limited per IP, no captcha** (per `[[demo-page-architecture]]`). Use the established `throttle:` middleware pattern. Pick and document sane limits; assert the throttle in Pest.
- Pre-register the threat model the way 291 did for notifications: add a short, disposition-tagged section to the relevant security reference doc (the `docs/security-*.md` analog) covering the shared-account surface, the demo-server guard, and the accepted residual (shared-session + wipe-cadence model). "Document as we go" — the pre-GA audit consumes it.

## Resolved decisions

Settled — implement to these; don't re-litigate at session start. Surface only if implementation reveals a decided shape is genuinely unworkable.

- **Shared-account role/permission shape.** A new `demo` role, seeded alongside the existing roles, granted the read/most-write CMS + records surfaces a prospect needs to feel the product, and explicitly **denied**: mail/integration/secret settings, API keys, fleet/backup/destructive actions, user/role management, and any real-data-exfiltration surface. **Never `super_admin`.** This deny-list defines the role seeder and is the binding security constraint stated above.
- **Auto-login route + href.** A single GET route on the demo server, `/demo/enter`, guarded by the demo-server identity check + per-IP throttle; the page JSON button's href points at it.
- **Shared account create-or-reuse.** One persistent seeded `users` row, reused idempotently across visitors (matches "shared sandbox"); the route `Auth::login()`s that row and redirects to the admin-panel landing.
- **Page-structure gaps** (from `demo-page-spec.md`, expected, non-blocking): G-demo-1 (no larger-than-standard `primary` size variant), viewport-height-aware layout sizing, full-bleed cells in a 2-column layout. Use the spec's named workarounds and log each gap; none blocks the page.
