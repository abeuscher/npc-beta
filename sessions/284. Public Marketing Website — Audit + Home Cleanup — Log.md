### Phase 3 — Mid-session: bug fix (chrome wrapper missing `<header>` / `<footer>`)

- Surfaced when the footer's `#d4d4f2` background wasn't rendering despite the inline CSS rule. Diagnosis: `public.blade.php` had two branches that dropped chrome HTML directly into the body without wrapping. The fallback partials (`components.site-header.blade.php`, `components.site-footer.blade.php`) emit `<header>` / `<footer>` elements; the chrome-render branches did not. The template's `header { background: ... }` and `footer { background: ... }` rules had nothing to target on installs using chrome pages.
- Two-line fix: chrome HTML now wrapped in `<header class="site-header">` and `<footer class="site-footer">` to match the fallback partials. The `.site-footer` class also gives SCSS rules a hook.

### Phase 4 — Mid-session: bug fix (Quill bullet markers + sanitizer + public CSS)

- User observed bullet lists were rendering with both a bullet and a number overlapping. Diagnosis: Quill v2 emits every list as `<ol>` and distinguishes bullet vs ordered via `data-list="bullet"` on each `<li>`. `App\Support\HtmlSanitizer::ATTRS_PER_TAG` had no entry for `<li>`, so it defaulted to allowing only `class` — the `data-list` attribute was stripped on every save. Public SCSS had no rules targeting `data-list`, so the bare `<ol>` rendered as a numbered list.
- Fix applied:
  - `HtmlSanitizer`: added `'li' => ['class', 'data-list']` to `ATTRS_PER_TAG`. `data-list="bullet"` and `data-list="ordered"` now survive round-trip.
  - `resources/scss/_base.scss`: added a `data-list`-aware rendering block inside `.np-site`. `<ol>` containing `data-list`-marked `<li>`s gets `list-style: none` + `counter-reset: quill-list`; markers drawn via `::before` per `data-list` value. Lists without `data-list` fall through to native browser markers.
  - Rebuilt: `npm run build` for Vite, `php artisan build:public` for the widget bundle.
- Home-side fix (separate from the bug): converted the six bullet `<ol>` blocks in `home.json` to `<ul>` and stripped the now-redundant `data-list` attributes (script-applied). The home renders 6 `<ul>` content blocks + 1 `<ol>` (Small Data). The sanitizer/CSS fix is for future Quill-authored content; the home.json conversion was the user's explicit immediate ask.

## Security considerations

`HtmlSanitizer::ATTRS_PER_TAG` allow-list extended to permit `data-list` on `<li>`. The attribute is part of the Quill rich-text convention and carries one of two enumerated string values (`bullet` or `ordered`); no script execution, URI, or style-vector risk. Surrounding rule that strips `on*` event handlers stays in force. No other security-relevant change this session.
