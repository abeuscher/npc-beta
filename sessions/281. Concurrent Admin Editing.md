# 281. Concurrent Admin Editing

Lands a record-locking system for admin-editable resources: pessimistic lock on edit-page mount + optimistic version-check safety net + super-admin takeover + non-super-admin takeover after 5-min idle + list-view "locked by X" indicator + JS hook for proactive release on tab close. Centralized polymorphic table shared across resources.

### Polymorphic lock table shape

```
record_locks
  id              bigint, pk
  lockable_type   varchar (model class — Contact, Page, Post, ...)
  lockable_id     bigint
  user_id         bigint, fk users
  acquired_at     timestamp
  last_saved_at   timestamp, nullable (null until first save; drives the 5-min idle window)
  expires_at      timestamp (acquired_at + idle window; refreshed on each save)
  created_at      timestamp
  updated_at      timestamp

  unique (lockable_type, lockable_id)  -- only one active lock per record
  index  (user_id)
  index  (expires_at)  -- for sweep-expired queries
```

Confirm at session start that `unique (lockable_type, lockable_id)` is the right enforcement (vs allowing multiple rows with one-active-at-a-time semantics). The unique constraint is simpler — acquire is an upsert that handles the takeover atomically.

### `last_saved_at` vs idle definition

The 5-min idle window measures **time since the last save on the locked record**. On first acquire `last_saved_at` is null and `expires_at = acquired_at + 5min`. On each save by the lock-holder, `last_saved_at = now()` and `expires_at = now() + 5min`. Takeover is permitted whenever `now() > expires_at`.

Super-admin override is unconditional (ignores `expires_at`).

### Optimistic safety net

A `version` column is NOT added to model tables. The optimistic check uses the lock row's `acquired_at` timestamp embedded in the form (as a hidden field). On save, compare submitted `acquired_at` to the current lock's `acquired_at` — if they differ, the lock was taken over while the editor was working. Show "this record changed while you were editing; please reload" and reject the save.

This avoids per-table schema changes and keeps the safety net inside the lock subsystem.

### Resources to opt in on day 1

- ContactResource

- EventResource

Other resources (NoteResource, TagResource, MembershipResource, MailingListResource, etc.) opt in later as the trait is well-bedded. ResourceList shows the lock indicator only for opted-in resources.

Notes on specifics:
- **PostResource uses the Page model** (per the 280 audit finding). The lock keys on the model + id, so PostResource locks compose with PageResource locks correctly — same Page row, same lock row.
- **MemberResource is a filtered Contact view.** Edits redirect to ContactResource; locks acquired there. MemberResource doesn't need its own opt-in.
- **DonationResource** is view-only (per 280 finding — hard-deny on CRUD). No edit page, no lock.

## Out of scope

- **Field-level locking** (lock individual form fields, not the whole record). Defer; the use case is unclear at the CRM scale.
- **Cross-resource locks** (e.g. locking a Contact also locks all its Notes). Defer; the use case can be revisited if findings surface.
- **Real-time displaced-editor notification** (websocket pushing "your lock was taken"). Defer per the 280-close conversation — the save-time warning gets 95% of the value at 5% of the cost.
- **Portal user locks.** Locks live in the admin panel only.
- **Lock acquisition on the list view's row-action edit shortcuts.** Filament's inline `EditAction` opens a modal — different lifecycle from the full edit page. Decide at session start whether to lock modal edits or leave them at last-write-wins. Recommendation: leave modal edits at last-write-wins for v1 (modals are quick edits; lock churn outweighs collision risk).
- **Deploy-time admin behavior** — separate roadmap discussion item (see `sessions/session-outlines.md` § "Deploy-Time Admin Behavior — Discussion").
