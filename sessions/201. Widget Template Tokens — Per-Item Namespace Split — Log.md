## What was built

### Phase 1 — Scope audit

Confirmed three in-scope listing widgets and recorded their per-item vocabularies. The collision between page-context and per-item grammars was the root cause of the bug surfaced during session 200 manual testing (every Events Listing card showing the hosting page's title).

| Widget handle | Field | Field type | Per-item vocabulary | Collisions with `PageContextTokens::TOKENS` |
|---|---|---|---|---|
| `events_listing` | `content_template` | richtext | title, slug, url, date, date_iso, ends_at, location, is_free, price_badge, image | **title, date, location** |
| `blog_listing` | `content_template` | richtext | title, slug, url, date, date_iso, excerpt, image | **title, date, excerpt** |
| `carousel` | `caption_template` | text | iterates `$slide` keys from collection_item shape (data-driven) | **title** (and any other collection-item key that happens to match `PageContextTokens::TOKENS`) |

Substring-safety invariant verified empirically: `str_replace('{{title}}', 'PAGE', '{{item.title}} and {{title}}')` returns `'{{item.title}} and PAGE'`. `PageContextTokens::substitute` naturally passes `{{item.*}}` tokens through unchanged with no service-level change required.

### Phase 2 — Code changes per widget

Per-item grammar migrated from `{{key}}` to `{{item.key}}` across the definition defaults, the schema default strings, and the per-item substitution loops in Blade. The trailing `preg_replace('/\{\{[^}]+\}\}/', '', ...)` cleanup pass was preserved verbatim (retains existing behaviour of stripping unsubstituted tokens, including unknown page-level tokens).

**Before → After (default `content_template` / `caption_template` strings):**

- **EventsListing** — `<p>{{image}}</p><h3><a href="{{url}}">{{title}}</a></h3><h4>{{date}}</h4><p>Ends: {{ends_at}}</p><p>{{location}}</p><p>{{price_badge}}</p><p>{{slug}}</p><p>{{date_iso}}</p>` → `<p>{{item.image}}</p><h3><a href="{{item.url}}">{{item.title}}</a></h3><h4>{{item.date}}</h4><p>Ends: {{item.ends_at}}</p><p>{{item.location}}</p><p>{{item.price_badge}}</p><p>{{item.slug}}</p><p>{{item.date_iso}}</p>` — applied to both `schema()` and `defaults()`.
- **BlogListing** — `<p>{{image}}</p><h3><a href="{{url}}">{{title}}</a></h3><h4>{{date}}</h4><p>{{excerpt}}</p><p>{{slug}}</p><p>{{date_iso}}</p>` → `<p>{{item.image}}</p><h3><a href="{{item.url}}">{{item.title}}</a></h3><h4>{{item.date}}</h4><p>{{item.excerpt}}</p><p>{{item.slug}}</p><p>{{item.date_iso}}</p>` — applied to both `schema()` and `defaults()`.
- **Carousel** — `{{title}}` → `{{item.title}}` on the default `caption_template`.

Blade templates updated accordingly: the per-item loops now substitute `{{item.key}}` keys; raw-HTML exceptions for `image` and `price_badge` preserved; the `$caption = str_replace('{{' . $key . '}}', e($value), $caption)` in Carousel became `str_replace('{{item.' . $key . '}}', e($value), $caption)`.

Existing `tests/Feature/ContentListingWidgetsTest.php` fixtures were updated to the new grammar (they test the per-item substitution directly via `Blade::render`, and therefore must use `{{item.*}}` to exercise the loop).

New file: `tests/Feature/WidgetTokenNamespaceTest.php` — four assertions exercising the full `WidgetRenderer::render()` pipeline to prove the collision is gone:

1. The substring-safety invariant at the unit level (one-line sanity test).
2. Events Listing with a page titled "Events" renders each event card with the per-event title, not the page title.
3. Blog Listing with a page titled "News" renders each post card with the per-post title, not the page title.
4. Carousel with a page titled "Gallery" renders each slide's own caption, not the page title.

### Phase 3 — Data migration

Shipped `database/migrations/2026_04_20_120000_rewrite_listing_widget_per_item_tokens.php`.

- **Widget-aware.** A `WIDGET_SPECS` map declares, per in-scope widget handle, which config field to rewrite and which keys are rewritten from `{{key}}` to `{{item.key}}`. Keys outside the declared list are left untouched.
- **Carousel narrow set.** Per the Phase 1 scope decision, Carousel rewrites only the six `PageContextTokens::TOKENS` keys (`title`, `date`, `excerpt`, `author`, `starts_at`, `location`) — the keys that were actually being hijacked. Any other `{{key}}` a user authored in a `caption_template` is left alone.
- **Idempotent.** Because `str_replace('{{title}}', '{{item.title}}', s)` on a string that already contains `{{item.title}}` and no `{{title}}` is a no-op, the migration is naturally idempotent. The update-write is gated on `$updated !== $original` so unchanged rows don't incur a DB write.
- **Reversible.** `down()` runs the inverse str_replace for the same per-widget key list.
- **Audit-logged.** Emits `[widget-token-migration] handle=X direction=up|down scanned=N rewritten=M sample_id=... before=... after=...` lines to `storage/logs/laravel.log` at info level.

Local migration run on the dev DB: three widget types scanned (1 row each), 0 rows rewritten. The local `page_widgets` rows all rely on Definition-class defaults rather than storing their own `content_template` / `caption_template` overrides, so nothing needed rewriting. Verified by direct DB inspection.

Migration test file `tests/Feature/WidgetTokenMigrationTest.php` covers: up-rewrite per widget (3 tests), idempotency on repeat up-run, down-reversal restores original grammar, non-listing widgets (hero) untouched, and Carousel's narrow-set scope (confirms `{{slug}}` is NOT rewritten in a Carousel `caption_template` even though it's a typical collection-item key).

`docs/schema/page_widgets.md` updated with a short grammar note on the `config` column documenting the two-namespace convention and referencing the migration by filename.

## Production deployment notes

Beyond standard `migrate`, worth noting:

- The migration will log per-widget `scanned` / `rewritten` counts to `storage/logs/laravel.log` when it runs on prod. Tail the log during deploy to confirm the rewrite counts match expectations — any production widget whose stored `content_template` / `caption_template` contains the old grammar will be rewritten in place.
- Carousel rows where a user authored a `caption_template` referencing a collection-item key *outside* the narrow set (`title`, `date`, `excerpt`, `author`, `starts_at`, `location`) will NOT be rewritten — by design. If the prod log reveals users using e.g. `{{slug}}` in Carousel captions, that template will effectively stop working after deploy (the per-slide loop now only substitutes `{{item.*}}`). Recommended check post-deploy: grep rendered Carousel widgets on prod pages for any surviving `{{` sequences in captions.
- No seeder or config change required.

## Deferred decisions

- **Inspector UX / designer-facing docs for the new grammar.** Explicitly out of scope this session. A follow-up can add field-hint strings on the `content_template` / `caption_template` inputs, an inspector helper that lists available `{{item.*}}` keys per widget, or a docs stub under `resources/docs/`. Not urgent — the defaults are correct out of the box, so designers won't encounter the grammar unless they hand-edit the template string.
- **Renaming keys in `PageContextTokens::TOKENS`** (e.g. `starts_at` → `event.starts_at`). Page-level grammar stays as-is; the session prompt explicitly deferred.
- **Carousel broad-set rewrite.** If prod post-deploy surfaces user templates that were hijacked on Carousel tokens *outside* `PageContextTokens::TOKENS` (unlikely but possible), a follow-up migration could widen the Carousel rewrite. Deferred pending real-world evidence — no point broadening the rewrite pre-emptively and risking legitimate page-context references.
