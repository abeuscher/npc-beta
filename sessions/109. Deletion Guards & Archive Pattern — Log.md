## What was built

### Phase 1 — Deletion guards

- **Hard blocks (verified):** Donation (`canDelete=false`), Email Template (`canDelete=false`) — already in place, confirmed.
- **Event:** blocked when registrations exist; delete action hidden, bulk delete skips protected events.
- **MembershipTier:** blocked when active memberships exist.
- **Fund:** delete action hidden when donations reference it; safe bulk delete added (relational guard already existed).
- **Product:** delete action hidden when purchases exist; safe bulk delete added (relational guard already existed).
- **WidgetType:** extended existing pinned-widget guard to also block when page_widgets reference it.
- **Form:** soft-delete warning shows submission count; force-delete warns about permanent loss.
- **Collection:** delete/force-delete warnings show item count for custom collections; system collections still fully blocked.
- **NavigationMenu:** general warning: "This menu may be in use on your site."
- **Page:** member-type pages get loud portal-impact warning; system pages still fully blocked.
- **Contact:** force-delete restricted to `super_admin` only (was any user with `delete_contact` permission).
- **Event Registration:** relation manager made non-read-only; delete action with Stripe refund warning for paid registrations.

### Phase 2 — Archive pattern

- Migration adds `is_archived` (boolean, default false) to `products`, `membership_tiers`, `funds`, `forms`.
- `Archivable` trait with `withoutArchived` and `onlyArchived` scopes.
- All four resources: archive/unarchive toggle action, archived filter, default list excludes archived.
- Archived records excluded from: tier dropdowns (3 admin locations + portal signup widget), fund dropdown in donation form widget, product queries in WidgetDataResolver, form submission controller (returns 404).
- **Mailing List** — evaluated, no archive needed (existing `is_active` suffices).
- **Collection** — evaluated, no archive needed (soft delete + `is_active` covers the use case).

### Event edit page restructure

- **Dedicated ViewRegistrations page** at `/admin/events/{record}/registrations` — table with name, email, status, paid badge, registered date; "View Contact Record" link per row; delete action with Stripe refund warning for paid registrations.
- **Ellipsis menu** on edit event page: View registrants, View ticket purchases (paid events), Export registrants, Send reminders, Delete registrant contacts.
- Removed EventRegistrationsRelationManager from edit page — eliminates registration-count load problem.
- Cancel event wizard submit button changed from "Cancel Event" to "Send cancellation emails".
- "Send reminders" label shortened (removed "now").

### Transaction resource enhancements

- Added visible `Event` SelectFilter — filters transactions by event through `event_registrations` join.
- Added hidden `id` SelectFilter for direct-link use.
- "View ticket purchases" link from event edit ellipsis menu opens transactions list filtered to that event.

### Roadmap update

- API Key Pattern Validation session stub updated to include environment mismatch hard gate (Stripe live + QB sandbox or vice versa — refuse to save).
