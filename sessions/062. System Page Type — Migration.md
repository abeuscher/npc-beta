# 062. System Page Type — Migration

Migrates existing portal routes to CMS-backed system pages, using the infrastructure built in session 061. Two routes are exceptions and handled via a settings-based rich text editor instead of the page builder.

---

## Architecture decisions (agreed pre-session)

Every publicly viewable page falls under one of four types: `default`, `post`, `event`, `member`, or `system`. No public-facing route renders outside the CMS. Admin and Filament routes are not affected.

**Pages migrated to full CMS system pages** (GET route renders through the page builder):

| current route | slug | required widget |
|---|---|---|
| `GET /login` | `login` | `portal_login` |
| `GET /signup` | `signup` | `portal_signup` |
| `GET /forgot-password` | `forgot-password` | `portal_forgot_password` |
| `GET /account` | `account` | `portal_account_dashboard` |

Slugs above assume `system_prefix` is empty (the proposed default). If a prefix is set, slugs are prefixed accordingly at seed time.

**Exception pages** — settings-based rich text, Blade views continue to render:

| route | reason | settings key |
|---|---|---|
| `GET /reset-password/{token}` | Token in URL path makes slug-based CMS lookup impractical | `system_page_content_reset_password` |
| `GET /email/verify` | Verification notice — grouped with reset-password for consistency | `system_page_content_email_verify` |

**Carved out entirely** (action endpoints, not pages):

| route | reason |
|---|---|
| `GET /email/verify/{id}/{hash}` | Signed redirect URL — no rendered view, not a page |

---

## Phase 1 — Missing widget types

Verify which widget types already exist. Create any that are missing:

- `portal_login` — added in session 058, verify present
- `portal_signup` — added in session 058, verify present
- `portal_forgot_password` — renders the forgot-password email form
- `portal_account_dashboard` — renders the member account view

For each missing type, follow the existing widget type seeder pattern. Config schema may be empty at this stage — the widgets render their forms from the existing Blade partials.

---

## Phase 2 — Seed system pages

Seed the four system pages in `DatabaseSeeder` (or a dedicated `SystemPageSeeder`). Each record:

- `type = 'system'`
- `is_published = true`
- Slug per the table above (honoring `system_prefix` SiteSetting at seed time)
- A seeded `PageWidget` record for the required widget, `sort_order = 1`, `is_active = true`

Use `firstOrCreate` so the seeder is idempotent.

---

## Phase 3 — Wire GET routes to CMS rendering

For each of the four migrated pages, update the GET route handler to look up the system page by slug and render it through the page builder. Keep the named routes intact (`portal.login`, `portal.signup`, etc.) so all existing `route()` calls continue to work.

Pattern:

```php
// Before:
Route::get('/login', [LoginController::class, 'show'])->name('portal.login');

// After: handler looks up the system page and delegates to the page builder renderer.
// The page builder renders widgets; the portal_login widget delivers the form HTML.
```

The POST routes (`portal.login.post`, `portal.signup.post`, etc.) are unchanged — they stay as controller handlers.

Once each GET route is wired and tested, retire the corresponding Blade view:

- `resources/views/portal/login.blade.php`
- `resources/views/portal/signup.blade.php`
- `resources/views/portal/forgot-password.blade.php`
- `resources/views/portal/account.blade.php`

---

## Phase 4 — Settings rich text for exception pages

Add a "System Page Content" section to General Settings with two rich text fields:

| label | settings key | used by |
|---|---|---|
| Password reset page content | `system_page_content_reset_password` | `ResetPasswordController::show()` |
| Email verification notice content | `system_page_content_email_verify` | `EmailVerificationController::notice()` |

Default values should reproduce the current content of the Blade views so the pages are unchanged for existing installations.

The Blade views for these two routes read the relevant SiteSetting and render it as HTML. The form fields and controller logic are unchanged.

## Security checklist

Before closing:

- [ ] POST routes are untouched and continue to work
- [ ] The `portal.auth` and `verified` middleware chains are still applied correctly after the route handler change
