# 338. Guided Product Tour — driver.js framework + demo CRM walkthrough

Wire driver.js into the Filament admin and author a short, honest guided tour of the demo CRM — launched from the dashboard — so a prospect who lands on the demo node understands the product without a salesperson.

## Session-specific deltas

- **The tour runs in the Filament admin, not on public pages.** §2c of the demo plan assumed the tour would drive stable public-CMS markup (and used that to wave off session 249's "Filament DOM is selector-fragile" concern). The owner's tour is a CRM product walkthrough — dashboard, topbar, left nav, records — i.e. the admin DOM. **Mitigation:** anchor every step on a `data-tour` attribute *we* inject (render hooks for chrome; widget Blade for dashboard widgets), so we own the selectors and Filament version churn can't silently break a step.
- **Public view is a hand-off, not a tour leg.** The CMS step starts on an event, clicks through to its landing page to show the page builder, and uses the page's **"View Site"** button — which opens the live site in a **new tab** while the tour keeps running on the admin tab. No driver.js continuation across tabs/contexts (owner-decided).
- **Launch from the dashboard.** A "Take the tour" affordance on the dashboard (a button in an existing dashboard widget, or a small new launcher widget). Packaging is an open question (below); the owner is open to whatever is cleanest.
- **Honesty bar:** every claimed feature is live (Stripe + QuickBooks both confirmed real during 337-close planning).

## Open questions to resolve at session start

1. **Packaging / delivery.** Lean: `npm i driver.js`, import it + a tour-definition module into `resources/js/admin.js`; launcher = a button in `DashboardWelcomeWidget` (or `DashboardQuickActionsWidget`) that fires a JS event the module listens for; `data-tour` anchors injected via `AdminPanelProvider` render hooks (`TOPBAR_START`, the sidebar hooks) + widget Blade. Confirm before building — this is the one architectural choice.
2. **Anchor inventory.** The exact chrome elements each step targets and where each `data-tour` attribute is injected.
3. **Demo-mode behavior.** Manual launch via the dashboard button (lean), or also auto-start once per prospect on demo arrival (gated to `isDemoMode()`, suppress-after-seen via localStorage so it never re-nags). v1 can ship manual-only and leave auto-start as a fast follow.
4. **Step boundaries / count.** The outline below is ~9 beats; confirm whether events + page-builder stay one combined beat or split, and whether welcome/wrap are full steps or lightweight modals. Hard cap 12.

## Phases

### Phase 1 — Framework
Add driver.js to the admin bundle; build the tour-definition module (one ordered list of steps: `data-tour` anchor + title + body + side/align); inject the `data-tour` anchors onto the admin chrome via render hooks + dashboard-widget Blade; wire the dashboard launcher button; gate launcher/auto-start to the demo as decided at the gate. A replay path (the button) always works.

### Phase 2 — Author the walkthrough
Write the step copy against the anchors — ~1 point each, 2 for the heavy features. Draft order:

1. **Welcome** (centered modal, no anchor) — what this CRM is; "click through, or close anytime and explore."
2. **Contacts** — every supporter is one rich record (profile + activity together). *Invite: click into a contact.*
3. **Donations** — track gifts and see each donor's giving history. *(The nonprofit headline — up to 2 points.)*
4. **Mailing list** — sync segments to your email tool (Mailchimp).
5. **Memberships** — tiered memberships with renewals.
6. **Events → page builder** — create/manage an event, click through to its landing page to show the page builder, then **View Site** opens it live in a new tab (your public website is built in). *(Up to 2 points; the all-in-one differentiator.)*
7. **Financial** — Stripe payments across events, products, memberships, and donations + QuickBooks sync for the books. *(Up to 2 points.)*
8. **Import / Export** — bring your existing data in (and get it out) with guided importers.
9. **Roles & Permissions** — control who can see and do what — then a short wrap inviting free exploration.

### Phase 3 — Verify
A Playwright spec (standing regression artifact): launch the tour from the dashboard, advance through every step, assert each step's anchor resolves to a visible element, and that it ends cleanly. Confirm the "View Site" step opens a new tab and the tour survives on the admin tab.

## Out of scope

- **Cross-tab / front-end↔back-end tour continuation** — explicitly deprioritized.
- **A separate public-site tour** — the reserved `events-index.*` / `event-lp.*` / `portal.*` anchors stay for a later session.
- **A general customer-onboarding tour system** — this is the demo/marketing tour; reuse for onboarding is a later lift.
- **Schema / HTTP-contract changes.** CRM stays v2.3.0.
