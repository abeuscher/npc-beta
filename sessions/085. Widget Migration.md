# 085. Widget Migration

Complete the migration of all remaining server-side widgets to use `$pageContext`, resolving the five broken widget types left over from session 084, removing three orphaned blade files, and adding a per-widget-type default open/closed state for the page builder.

---

## Design decisions (resolved before starting)

- `event_description`, `event_dates`, and `event_registration` each assign `$event = $pageContext->event($config['event_slug'] ?? null)` at the top of their template. The existing `@isset($event)` guards stay in place — no further structural change needed.
- `events_listing` uses `$pageContext->collection('events', $limit)` rather than `$pageContext->upcomingEvents()`. This keeps the array data format the template already uses and avoids a template restructure.
- `product_display` uses `$pageContext->product($config['product_slug'] ?? null)`. `PageContext::product()` gains `->with('prices')` eager loading, since the template always iterates prices.
- The `collections` array on the `events_listing` widget type is cleared to `[]` in the seeder, consistent with the cleanup applied to `blog_listing` in session 084.
- The three orphaned blade files — `rich-text.blade.php`, `blog-roll.blade.php`, `collection-list.blade.php` — are deleted. They reference a `$data` variable that was never populated by the controller and have no corresponding widget type in the seeder.
- No portal widgets, `web_form`, or `donation_form` require any changes — they never used the old resolution system.

---

## Phase 1 — Update PageContext

In `App\Services\PageContext::product()`, add `->with('prices')` to the query so the product display template does not trigger a lazy load on every price iteration.

---

## Phase 2 — Migrate the five broken widgets

**`event_description`, `event_dates`, `event_registration`**

At the top of each template, add:

```
@php $event = $pageContext->event($config['event_slug'] ?? null); @endphp
```

No other changes to these templates are needed — the existing `@isset($event)` block structure already handles the null case correctly.

**`events_listing`**

Replace the `$events` variable reference with `$pageContext->collection('events', $limit)`. The limit should be cast from `$config['limit']` the same way `blog_listing` handles it (cast to int, treat empty string as null). The array data format returned by `collection()` is identical to what the template already uses — no further template changes are needed.

**`product_display`**

Replace the `@isset($product)` guard with a `@php` assignment and null check:

```
@php $product = $pageContext->product($config['product_slug'] ?? null); @endphp
```

The `@isset($product)` guard can remain as-is after the assignment.

---

## Phase 3 — Seeder update, reseed, and cleanup

- In `WidgetTypeSeeder`, set `'collections' => []` on the `events_listing` widget type.
- Run `php artisan db:seed --class=WidgetTypeSeeder` to push the change to the live DB.
- Delete the three orphaned blade files: `resources/views/widgets/rich-text.blade.php`, `blog-roll.blade.php`, `collection-list.blade.php`.

---

## Phase 4 — Widget type default open/closed state

Add a `default_open` boolean flag to the `widget_types` table so each widget type can declare whether it starts expanded or collapsed in the page builder.

**Migration**

Add a `default_open` boolean column, default `false`, to `widget_types`. Run the migration via Docker.

**Model**

Add `default_open` to `WidgetType::$fillable` and cast it as `boolean`.

**PageBuilderBlock**

In `loadBlock()`, add `'widget_type_default_open'` to the `$block` array, reading from `$pw->widgetType?->default_open ?? false`.

**Block view**

In `resources/views/livewire/page-builder-block.blade.php`, change the Alpine initialiser from:

```
x-data="{ open: false, menuOpen: false, confirmDelete: false }"
```

to:

```
x-data="{ open: {{ $block['widget_type_default_open'] ? 'true' : 'false' }}, menuOpen: false, confirmDelete: false }"
```

This applies to both newly placed blocks (which initialise from the widget type's default) and existing blocks (which re-initialise on each page load — open state is not persisted per instance).

**WidgetType resource form**

Add a `Toggle` field for `default_open` with label "Start open in page builder" to the WidgetType Filament resource edit form.

**Seeder**

All existing widget types default to `false` (preserving current closed behaviour). No seeder changes are required — the column default handles it. If any widget type warrants a different default, set it explicitly in `WidgetTypeSeeder` at that time.

**Update `docs/schema.md`** after the migration.

## Security checklist

Before closing:

- [ ] All five migrated templates access event/product data only through `$pageContext` — no raw model queries in widget templates.
- [ ] `$pageContext->event()` and `$pageContext->product()` return only published records — a widget cannot surface a draft event or unpublished product by passing an arbitrary slug.
- [ ] No widget template has been given access to unscoped data outside `$pageContext` and `$config`.

---

## Out of scope

- Livewire widget context — Livewire components are mounted differently and are a separate concern.
- Persisting the open/closed state per widget instance — the default is a widget-type-level setting only; instance state is not saved.
- Any changes to widget config schemas or the widget picker UI.
- New widget types.
- Adding a `limit` config field to `events_listing` (the collection resolver already supports it if the key is present — exposing it in the schema is a future task).
