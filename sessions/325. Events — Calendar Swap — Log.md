### Phase 2 — `event_mini_calendar` (server-rendered, vanilla, self-contained)

- Handle `event_mini_calendar`, label "Events Mini-Calendar", category `events`; `data-tour="events-index.mini-calendar"`.
- Carbon month grid, pre-rendered **±1-month window** toggled client-side (no JS date math, no Vue). Density dots (low/med/high) from a windowed published-events `DataContract` query (no limit).
- Two back-end **list modes** (`list_mode`): **Selected day's events** (click a day → its events; today, else soonest-upcoming, selected on load; month-nav clears the list) and **Visible month's events** (whole displayed month, follows month-nav, per-month empty state).
- **Self-contained expandable event detail** (caret): date/time, location, Free / In-person / Virtual / Sold-out pills, optional description (`show_description` toggle), an **"Event page →"** link only when the event has a landing page, and **"Register →"** when an external registration URL exists — so a no-LP event is never a dead end. Added `has_landing_page` to `projectEvent`; widened the widget's contract `fields` to carry the detail.
- Heading is an **inline-editable rich-text** field (inspector-hidden; edited on the canvas).
- **Theme-driven accents:** dots, today-outline, and the selected day (brand ring + bold number) all follow `--np-color-brand`; text / link / background respect the standard appearance controls. No hardcoded colours — the colour-token guard is clean.
- Width capped (21rem, centred) + square day cells (`aspect-ratio`) so it never morphs full-width; optional **"Event List Max Height"** (`events_max_height`) opts into a scroll; event rows dropped out of `list-item` display so the global prose bullet can't render.

## Decisions

- **widget_types removal:** seeder edit, not a migration (open-gate call).
- **month-nav:** pre-rendered ±1-month window (open-gate call).
- **LP-model pivot (settled at close):** the "optional / no-LP" model is retired — building the mini-calendar showed it was collecting compensating complexity (dead-end fallback, `has_landing_page` conditionals, an inline-detail surface per widget). Every event will get a **Simple / Standard landing page** at creation instead. The demo-site plan was reshaped + reordered (A ✅ → **C event landing pages = 326** → **B listing upgrade = 327** → D → E → F); the 326 + 327 base + session prompts are drafted; `demo-site-plan.md` §3 carries the decision note.

## Security considerations

The expandable detail renders the event description through the same path as the `EventDescription` widget — `HtmlSanitizer`-sanitized on save, `InlineImageRenderer` at display — so no new unsanitized-HTML surface. The inline-editable rich-text heading is a standard rich-text config field, handled exactly like `TextBlock` content. The widened `DataContract` exposes only published-event public fields already surfaced by `EventsListing`. No auth, permission, or boundary surface was touched.
