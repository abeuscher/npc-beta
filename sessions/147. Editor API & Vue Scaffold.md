# 147. Editor API & Vue Scaffold

Build the REST API endpoints for editor widget CRUD and mount a Vue 3 application inside the Filament EditPage shell. This is the first implementation session of the Vue editor migration — it lays the foundation that sessions 148–151 build on.

**Reference document:** `docs/editor-migration-plan.md` — read this file in full before writing any code. It contains the API endpoint specifications (Part 2), the Pinia store design (Part 3, section 3.1), the component tree (Part 3, section 3.2), the bootstrap data shape (Part 3, section 3.4), and the session 147 acceptance criteria (Part 4).

---

## Design decisions (resolved before starting)

- **API controller:** Single controller `PageBuilderApiController` with methods for each endpoint. Routes in a dedicated `routes/admin-api.php` file, included from `routes/web.php`.
- **Middleware:** All API routes use Filament's admin panel middleware stack (web session + CSRF). No token auth.
- **Vue entry point:** Mounts inside the existing `page-builder.blade.php` as `<div id="page-builder-app">`. Bootstrap data passed as a JSON attribute on the div.
- **Parallel operation:** Both the Livewire editor and Vue app coexist during this session. The Vue proof-of-concept renders below the Livewire editor as a temporary dev-time section. No toggle or feature flag — just both visible on the page so the Vue app can be verified while the Livewire editor stays functional. The temporary section is removed in session 151 when the Livewire components are torn down.
- **TypeScript:** Use `.ts` files for the store, API client, and types. Vue SFCs use `<script setup lang="ts">`.
- **File location:** Vue source files live in `resources/js/page-builder-vue/`.

## Phase 1 — API endpoints

Build the REST API controller and routes. The migration plan (Part 2) specifies each endpoint's URL, request shape, and response shape. Implement all of these:

### Widget CRUD
1. `GET /admin/api/page-builder/{page}/widgets` — load full widget tree with preview HTML
2. `POST /admin/api/page-builder/{page}/widgets` — create widget
3. `PUT /admin/api/page-builder/widgets/{widget}` — update config/style/query/label
4. `DELETE /admin/api/page-builder/widgets/{widget}` — delete widget and children
5. `POST /admin/api/page-builder/widgets/{widget}/copy` — duplicate widget and children
6. `PUT /admin/api/page-builder/{page}/widgets/reorder` — batch reorder

### Preview
7. `GET /admin/api/page-builder/widgets/{widget}/preview` — render single widget preview HTML

### Lookups
8. `GET /admin/api/page-builder/widget-types` — widget type registry (accepts `page_type` query param)
9. `GET /admin/api/page-builder/collections` — collection list
10. `GET /admin/api/page-builder/collections/{handle}/fields` — collection field definitions
11. `GET /admin/api/page-builder/tags` — tag list
12. `GET /admin/api/page-builder/pages` — page list for link pickers
13. `GET /admin/api/page-builder/events` — event list
14. `GET /admin/api/page-builder/data-sources/{source}` — named data source options

### Image upload
15. `POST /admin/api/page-builder/widgets/{widget}/image` — upload image for config field
16. `DELETE /admin/api/page-builder/widgets/{widget}/image/{key}` — remove image

**Implementation notes:**
- Extract the logic from the existing Livewire components — don't rewrite from scratch. The API formalises what `PageBuilder.php`, `PageBuilderBlock.php`, and `PageBuilderInspector.php` already do.
- The `GET widgets` endpoint reuses the preview rendering logic from `PageBuilder::renderWidgetForPreview()`. Extract it into a shared method or service so both the Livewire component and API can call it.
- Permission checks: read endpoints require `view_page`, write endpoints require `update_page`. Match the existing `assertCanEdit()` pattern.
- Widget operations must validate that the widget belongs to the specified page.

## Phase 2 — Vue scaffold

### 2a. Install dependencies

Add to `package.json`:
- `vue` (^3.x)
- `pinia`
- `@vitejs/plugin-vue`

### 2b. Vite configuration

Update `vite.config.js` to:
- Add the Vue plugin
- Add a new entry point for the page builder Vue app (`resources/js/page-builder-vue/main.ts`)
- Ensure the existing Filament theme build is unaffected

### 2c. Vue application files

Create the Vue application structure:

- `resources/js/page-builder-vue/main.ts` — `createApp()` + `createPinia()`, mount to `#page-builder-app`
- `resources/js/page-builder-vue/App.vue` — root component. Reads bootstrap data from the mount element's `data-bootstrap` attribute, initialises the store.
- `resources/js/page-builder-vue/stores/editor.ts` — Pinia store following the design in the migration plan (Part 3, section 3.1). Implement the core state shape, `loadTree` action (populates from bootstrap data), `selectBlock` action, and `setMode` action. Stub the remaining actions (createWidget, updateWidget, deleteWidget, etc.) — they should call the API client and update state, but don't need to be fully wired to UI yet.
- `resources/js/page-builder-vue/api.ts` — thin fetch wrapper with CSRF token handling. Base URL from bootstrap data. Methods for each API endpoint.
- `resources/js/page-builder-vue/types.ts` — TypeScript interfaces for Widget, WidgetType, EditorState, and API response shapes.

### 2d. Minimal proof-of-concept UI

Create just enough Vue UI to prove the pattern works:

- A simple block list that renders widget labels from the store
- Click a block to select it (updates `selectedBlockId` in the store, highlights the selected block)
- Display the selected widget's config as a read-only JSON dump
- An "Update Label" text input that calls the `PUT widgets/{id}` endpoint and updates the store
- No styling needed beyond basic readability — this gets replaced by real components in sessions 148–150

### 2e. Mount point in Blade

Modify `resources/views/livewire/page-builder.blade.php` to:
- Add a `<div id="page-builder-app" data-bootstrap='@json($bootstrapData)'></div>` below the existing Livewire UI
- Add the Vite tag for the Vue entry point
- Add `getBootstrapData()` to `PageBuilder.php` that assembles the bootstrap JSON (page ID, widget tree, widget types, required handles, lookups, CSRF token, API base URL)

The Vue proof-of-concept section renders below the Livewire editor — both are visible on the page so you can verify the Vue app is loading data correctly. The Livewire editor remains the primary working editor. This temporary arrangement is removed in session 151.

## Testing

- **Slow test groups to run this session:** none
- **New tests expected:** yes — API endpoint tests:
  - `GET widgets` returns the correct tree structure for a page
  - `POST widgets` creates a widget and returns updated tree
  - `PUT widgets/{id}` updates config and returns updated widget
  - `DELETE widgets/{id}` deletes widget and children, returns updated tree
  - `POST widgets/{id}/copy` duplicates widget and children
  - `PUT widgets/reorder` persists new sort order and parent/column assignments
  - `GET widgets/{id}/preview` returns rendered HTML
  - Permission checks: unauthenticated requests return 401/403, users without `update_page` cannot write
  - Widget ownership: cannot manipulate widgets belonging to a different page
  - Lookup endpoints return expected data shapes

## Security checklist

Before closing:

- [ ] All API endpoints require Filament session auth — no unauthenticated access
- [ ] Write endpoints check `update_page` permission; read endpoints check `view_page`
- [ ] Widget CRUD validates `page_id` ownership — cannot manipulate widgets on other pages
- [ ] No sensitive data (API keys, user credentials) in API responses or bootstrap JSON
- [ ] Image upload validates file type and size
- [ ] CSRF token required on all mutating requests
- [ ] Lookup endpoints return only data the authenticated user may view
