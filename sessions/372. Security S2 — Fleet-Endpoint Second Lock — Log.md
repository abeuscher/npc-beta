## What was built

Security Hardening track **S2** (Gate 3), launch-schedule position 3 (schedule order S1 ‚Üí S3 ‚Üí S2). The track's **one** boundary-touching session. Acts on Appendix A risk #1 ‚Äî the worst concentration: all five Fleet Manager `/api/*` endpoints hung on a single per-location nginx `if ($ssl_client_verify != "SUCCESS")`, with the app layer deliberately trusting the connection. A dropped / shadowed / misconfigured `if` on the blob or recover location silently exposed the whole donor DB or an admin credential reset. S2 adds an independent app-layer second lock behind the mTLS termination so two things must both hold, not one.

### The gate

- **`app/Http/Middleware/VerifyFleetAgent.php`** ‚Äî reads `config('fleet.gate.secret')`; **empty ‚áí inert** (`$next` unchanged, mTLS the only lock, exactly v2.6.0); set ‚áí requires the `X-Fleet-Gate-Key` header, compared `hash_equals` (constant-time), rejecting a missing/invalid credential with `401 {error: "fleet_gate_unauthorized", message}`. The secret is **not cert-derived** ‚Äî a separate credential on a separate channel ‚Äî so an nginx regression / misconfig / location-shadow that lets an un-certed request reach PHP still fails here. Cert-fingerprint forwarding was considered and rejected at the Open Gate (it re-trusts the nginx-populated value the lock exists to backstop, and adds nginx surface).
- **`routes/api.php`** ‚Äî the five routes wrapped in a `Route::middleware([VerifyFleetAgent::class])->group(...)`; per-route throttle buckets preserved inside.
- **`config/fleet.php`** ‚Äî new `gate.secret` from `env('FLEET_GATE_SECRET')` (absent ‚áí inert, same additive shape as `SUSPENSION_STATE` absent = none).
- **`app/Http/Controllers/Api/Fleet/Concerns/HasContractVersion.php`** ‚Äî `CONTRACT_VERSION` `2.6.0 ‚Üí 2.7.0` (Health / Backup / Recovery `contract_version` fields follow automatically).
- **`.env.example`** ‚Äî documented `FLEET_GATE_SECRET` (unset = inert; per-install, never shared; rotate like the cert; `openssl rand -hex 32`).

### `/api/logs` PII tension ‚Äî resolved as a conscious acceptance

The raw-log-tail endpoint keeps returning verbatim Laravel log lines. Rationale, recorded for the Gate-3 findings register: it is now behind **two** independent locks, the FM operator is the node's own data controller with full node access regardless, and the correct control for "PII in logs" is logging hygiene at the write site ‚Äî not lossy read-time redaction that would gut the endpoint's debugging value. No code change; documented in the contract's Security posture.

### Rollout posture (Open Gate decision)

Additive, no flag day: absent node secret ‚áí gate inert ‚áí a v2.7.0 image is a no-op on every running node until Fleet Manager provisions a secret. FM absorbs by sending the header first (harmless while inert), then pushing `FLEET_GATE_SECRET`; the reverse order would lock FM out for one poll cycle. No stored state beyond env/config (no migration).

## Security considerations

This session is a net security *increase* and its whole subject is the security axis. The five FM endpoints move from single-lock (nginx mTLS only) to two independent locks; the second lock is deliberately TLS-independent so it survives the exact failure mode S2 targets (an nginx regression collapsing the only defense). Worst-first: `/api/backup/blob` (whole donor DB) and `/api/admin/recover` (admin credential reset) proven first, then all five. The gate uses a constant-time comparison and never echoes or hints at the expected value; the `401` envelope carries no PII. The mTLS layer, the S1 headers/CSP, and the S3 XSS/exposure work are unchanged and relied upon, not revisited. One consciously-accepted residual recorded for the Gate-3 register: `/api/logs` returns raw log lines (now double-locked; logging hygiene is the real control).

## Production deployment notes

None beyond standard `migrate` ‚Äî and the deploy is a **no-op on every running node**: the gate is inert until `FLEET_GATE_SECRET` is set. To activate the second lock on a node (a launch-checklist step, not required to deploy): (1) Fleet Manager ships the change that sends `X-Fleet-Gate-Key` on every request to all five endpoints (harmless while the node secret is unset); (2) generate a unique-per-install secret and push it to the node's `.env` over FM's config-push channel; (3) restart `app` + `worker` (env_file is read at container start ‚Äî see `docs/app-reference.md` `.env` propagation note). Order matters ‚Äî pushing the secret before FM sends the header locks FM out for one poll cycle.
