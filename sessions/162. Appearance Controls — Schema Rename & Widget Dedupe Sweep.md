# 162. Appearance Controls — Schema Rename & Widget Dedupe Sweep

Phase 2 of the Appearance Controls project. A destructive schema change and a coordinated widget config cleanup that prepares the ground for the universal Appearance panels in session 163. There is no live data on this column — the migration is irreversible by design.

This session does not build any new UI. Every change is plumbing: rename the storage column, restructure the persisted shape, scrub deprecated and ambiguously-named widget config fields, and remove dead template references.

---

## Design decisions (resolved before starting)

### Schema migration

- Rename `page_widgets.style_config` → `page_widgets.appearance_config` (jsonb, default `{}`).
- The migration is a single column rename, applied via `Schema::table(...)->renameColumn(...)` plus a default reset if needed. **No data migration**, no down rewrite — there is no live data.
- After the migration runs, run `php artisan migrate:fresh --seed` to reset the database and reseed demo data so the page builder has working content to test against.
- The new shape is **nested**, not flat:

  ```
  {
    background: {
      color,
      gradient,
      image_id,
      alignment,
      fit,
      overlay: { enabled, color, opacity }
    },
    text: {
      color
    },
    layout: {
      full_width,
      padding: { top, right, bottom, left },
      margin:  { top, right, bottom, left }
    }
  }
  ```

  Session 163 builds the panels that populate the `background.gradient`, `background.image_id`, `background.alignment`, `background.fit`, and `background.overlay.*` fields. This session only needs to migrate the keys that *are* in use today: `background.color`, `text.color`, `layout.full_width`, `layout.padding.{top,right,bottom,left}`, `layout.margin.{top,right,bottom,left}`. The remaining nested keys will be written for the first time in session 163.

- Update `docs/schema/page_widgets.md` to reflect the rename and the new default shape after the migration is written.

### Renderer migration

- Update `resources/views/components/page-widgets.blade.php` to read from the new column and the new nested paths.
- The padding/margin render path is preserved exactly — same int casting, same `px` suffix, same per-key skipping. Only the read path changes (`$sc['padding_top']` → `$sc['layout']['padding']['top']`, etc.).
- Full-width resolution is preserved: `appearance_config.layout.full_width` overrides the WidgetType default, falling back to `widget_type->full_width` when unset.
- **Inherits the bg/text color emission landed in `session-161/2`.** Session 161/2 will have added background-color and text-color emission to the public renderer reading from `style_config.background_color` / `style_config.text_color`. Migrate those reads to `appearance_config.background.color` / `appearance_config.text.color`, preserving the emission behaviour. The Pest test 161/2 added must be updated to write the new shape; otherwise the existing assertion semantics are unchanged.
- The editor preview path in `App\Livewire\PageBuilder::buildInlineStyles()` follows the same migration: read from the new nested paths, same casting and sanitization rules.

### Vue / Pinia migration

- Update `resources/js/page-builder-vue/types.ts` — the `Widget` interface's `style_config` field is renamed to `appearance_config` with the new nested type. Same change to `UpdateWidgetPayload`.
- Update `resources/js/page-builder-vue/stores/editor.ts` — `updateLocalStyleConfig` is renamed and rewritten to operate on the nested shape (it currently does a shallow `{ ...style_config, [key]: value }` merge — the new helper needs to do a path-aware merge into `appearance_config`). The existing call sites in `WidgetAppearanceControls.vue` and `SpacingControl.vue` are updated to use the new helper signature.
- `WidgetAppearanceControls.vue` continues to write `background.color`, `text.color`, and `layout.full_width` to the new shape. This component is **not** retired in this session — that happens in session 163 when the universal panels replace it. In this session it simply repoints its three writes at the new nested paths.
- `SpacingControl.vue` continues to write padding/margin to the new `layout.padding` and `layout.margin` sub-objects. It is **not** retired in this session — that also happens in session 163. In this session it simply repoints its writes.
- `LayoutInspectorPanel.vue` operates on `layout_config` (the `page_layouts` row's separate jsonb), **not** on `appearance_config`. It needs no migration. Verify this in the audit but do not touch the file unless it actually references `style_config`.
- The `PageBuilder::createBlock()` method writes `style_config` defaults at widget creation (`background_color: '#ffffff'`, `text_color: '#000000'`). Migrate those writes to the new nested shape under `appearance_config.background.color` and `appearance_config.text.color`.

### Widget dedupe sweep — REMOVE

The following widget-level config fields are absorbed by the universal Appearance layer in session 163. Remove them from each widget's `config_schema` in `database/seeders/WidgetTypeSeeder.php` and remove every reference from the corresponding Blade template:

| Widget | Field | Notes |
|---|---|---|
| `product_carousel` | `background_color` | Absorbed by universal Background panel. |
| `product_carousel` | `text_color` | Absorbed by universal Text panel. |
| `product_carousel` | `full_width` | Dead code today — absorbed by universal Layout panel. |
| `hero` | `background_color` | Absorbed by universal Background panel. |
| `hero` | `text_color` | Absorbed by universal Text panel. |
| `hero` | `background_image` | Absorbed by universal Background panel. |
| `hero` | `full_width` | Dead code today — absorbed by universal Layout panel. |

After removing each field from the schema, audit `resources/views/widgets/product-carousel.blade.php` and `resources/views/widgets/hero.blade.php` for every reference to the removed keys (CSS variables, inline styles, conditional renders) and remove all of them. The widgets must still render cleanly with no `undefined variable` errors after the sweep.

### Widget dedupe sweep — LEAVE ALONE (do not touch under any circumstances)

These fields look superficially similar to the removed ones but are deliberately preserved:

- **`hero.nav_link_color`** and **`hero.nav_hover_color`** — critical for hero full-bleed mode where the site nav overlays the hero. The consumer wiring may live inside `hero.blade.php` itself or in a partial referenced from the chrome render path. **Investigate before any future touch.** Do not remove or rename in this session.
- **`hero.background_video`** — a hero-only feature, not absorbed into the universal Background panel (which covers colour, gradient, and image only). Stays as a hero config field.
- **`board_members.pane_color`** and **`board_members.border_color`** — already well-disambiguated by their names. No change.

If the audit surfaces any other widget config field that smells like it might be absorbed, **stop and ask** before removing it.

### Widget rename sweep — sub-region disambiguation

| Widget | Old key | New key |
|---|---|---|
| `carousel` | `slide_text_color` | `caption_text_color` |
| `carousel` | `slide_link_color` | `caption_link_color` |
| `bar_chart` | `bar_color` | `bar_fill_color` |
| `logo_garden` | `background_color` | `container_background_color` |
| `board_members` | `background_color` | `grid_background_color` |
| `hero` | `overlay_opacity` | `background_overlay_opacity` |

For each rename, three things must change in lockstep:

1. **Seeder schema** — update the field key in the widget's `config_schema` array in `database/seeders/WidgetTypeSeeder.php`.
2. **Blade template** — replace every `$config['old_key']` with `$config['new_key']` in `resources/views/widgets/{widget}.blade.php`.
3. **Stylesheet/JS consumers** — see special cases below.

**Special case — `bar_chart.bar_color → bar_fill_color`:** the chart instantiation must work in **both** the public Chart.js init path and the editor preview-mode Chart.js init path. Grep for `bar_color` across `resources/views/widgets/bar-chart.blade.php`, `resources/js/public.js`, the widget type's inline `js` column in the seeder, and any preview shim before declaring done. Verify both render paths after the rename.

**Special case — `logo_garden.background_color → container_background_color`:** also rename the corresponding CSS custom property (`--logo-bg` if present) in the SCSS partial `resources/scss/widgets/_logo-garden.scss` and any other consuming partial. Audit before edit.

**Special case — `board_members.background_color → grid_background_color`:** also rename the corresponding CSS custom property (`--bm-bg` if present) in `resources/scss/widgets/_board-members.scss` and any other consuming partial. Audit before edit.

### Schema bug fix — `blog_listing` and `events_listing`

Both Blade templates currently read `$config['background_color']` and `$config['text_color']`, but neither widget's `config_schema` defines those fields — the reads always return null and the styles never apply. **Remove the dead template references entirely.** Do **not** add the missing schema fields. The universal Background and Text panels delivered in session 163 will provide these controls instead.

The audit should pinpoint the exact lines in `resources/views/widgets/blog-listing.blade.php` and `resources/views/widgets/events-listing.blade.php` that reference these dead config keys. Remove those lines (and any container divs / style blocks that exist solely to apply them) cleanly.

### Inheritance from session 161/2

Session 161/2 will have shipped a fix to `resources/views/components/page-widgets.blade.php` adding `background-color` and `color` emission from `style_config.background_color` / `style_config.text_color`, plus a Pest test asserting the wrapper carries those styles. Both must be migrated:

- The renderer's read paths move from `$sc['background_color']` / `$sc['text_color']` to `$sc['background']['color']` / `$sc['text']['color']` (where `$sc = $block['appearance_config']`). The emission logic is unchanged.
- The Pest test from 161/2 must be updated to write the new nested shape. Its assertion (the wrapper carries `background-color: #hex` in its inline style attribute) is unchanged.

If 161/2 has not merged before this session starts, **stop and ask** — the order matters.
