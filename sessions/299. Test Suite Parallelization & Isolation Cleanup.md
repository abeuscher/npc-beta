# 299. Test Suite Parallelization & Isolation — incl. Isolated e2e Stack (Playwright Parallel + CI)

> **Session 2 of the test-cost split — scope expanded by user decision (session-298/1).** Pure speed, zero coverage loss. Two coupled goals: (1) make `php artisan test --parallel` the green default by root-causing the isolation defects that block it; (2) **stand up a second, isolated Docker stack so Playwright runs collision-free — locally in parallel with Pest *and* in CI.** Goal 2 was folded in deliberately: after parallelization Pest drops to ~3 min and **Playwright (~7–10 min, single-threaded, not in CI today) becomes the long pole** — the isolated stack is the critical-path fix, not creep. Testing has grown into the majority of dev-loop time (full corpus ≈20–30 min today); this session is the dig-out. **Session 1 (`298`) landed first** (hardened CI + scoped loop); this is code-independent of it. § D4 parallelization slice + the D4-anticipated test-isolation cleanup, lifted early per the sanctioned Rule-11 exception; D4's mutation/coverage-shape stays release-gated. **May split per Rule 11** if the combined surface is too wide for one clean session — decide *with* the user, never balloon silently.

One sentence: get `php artisan test --parallel` green and default by root-causing test-isolation defects, **and** give Playwright its own isolated Docker stack + throwaway DB so it runs parallel-safe locally and in CI — driving a ~20–30 min full-corpus pass down toward ~5 min with the identical set of tests.

**Playwright isolation + CI (folded in — user decision, session-298/1).** There is **no Playwright in CI today** — a real coverage gap (memory `project_playwright_ci_cost`). The e2e harness is hard-coupled to the one shared dev stack: `tests/e2e/helpers/db.ts` `resetDatabase()` shells `docker compose exec -T app php artisan migrate:fresh --seed` (default project, service `app`), `withClient()` connects pg at a fixed `localhost:5432`, `playwright.config.ts` `APP_URL` defaults `http://localhost`, and `workers:1`/`fullyParallel:false` exist *because* of that shared stack. The chosen fix: a **second, isolated stack** (separate compose project, distinct container names + host ports, its own throwaway DB volume, reusing the repo bind-mount + the existing built image — not a separate checkout, no second image build). The crux is small but real — **parameterise the three hardcoded targets**: the `resetDatabase()` exec target, the `db.ts` pg `host:port`, and `APP_URL`. The *same* parameterisation makes Playwright-in-CI work (point it at CI service containers), and isolating the infra **dissolves the `feedback_test_runs_not_parallel` hazard for the cross-stack case** (Pest on stack A, Playwright on stack B — no collision) and lets Playwright itself go multi-worker.

## Open questions to resolve at session start

1. **Residual isolation surface size.** The release-plan-named defects (shared `storage/app/private/` paths; the `seedWidgetCollections` flake) are the *known* list. The first `--parallel` run reveals the rest. If that surface is larger than one session can root-cause cleanly, decide with the user: land parallelization with a small, explicitly-logged serial-only carve-out for the stragglers + a carry-forward, vs. extend. Do not silently serial-carve-out a large set to force green.
2. **Build-pipeline isolation is already done (296, verified on `main`) — confirm, don't re-solve.** `AssetBuildService` is constructor-injectable; `BuildServerSettingsSession125Test`/`WidgetJsLibsSession138Test`/`AssetBundleDriftGuardTest` already build into temp dirs with guarded cleanup and no real `public/build/**` writes. The residual isolation surface this session owns is **whatever `--parallel` newly surfaces beyond that** (shared static/time/env state, non-isolated seeders like `seedWidgetCollections`, other shared-FS paths). Reuse the existing injectable-constructor pattern for any such offenders; do not invent a second approach or duplicate 296's work.
3. **Isolated-stack shape — resolve at session start (recommendation: faithful duplicate).** Two viable shapes: **(a) faithful** — a second compose project bringing up app + nginx + postgres + redis, port-shifted, prod-like serving (matches how the site really runs; heavier — roughly 1.5–2× the dev-stack footprint while active, mostly RAM, ~0 idle CPU); **(b) minimal** — a separate test Postgres only + Playwright driving `php artisan serve` (much lighter, but `artisan serve` ≠ nginx+fpm so less faithful, and it still needs the identical harness parameterisation). Recommend **(a)**: the entire point is trustworthy e2e, fidelity outweighs the modest extra RAM, and it generalises cleanly to CI. This is the one expensive-to-reverse choice — confirm with the user before building.

## Phases

### Phase 1 — Install & wire paratest; surface the failure set

Add Laravel's first-party parallel testing (paratest) as a dev dependency, wire `--parallel` as the default fast-suite invocation (`dev`, CI from Session 1, `CLAUDE.md`'s inner-loop guidance). Run it. Capture the full set of tests that fail/flke only under parallelism — this is the isolation defect list (framework handles per-process DBs; the residue is shared filesystem / static / time / env state).

### Phase 2 — Root-cause the isolation defects

Fix at the source, defect by defect: tests own their own temp paths/fixtures instead of sharing; non-isolated seeders (`seedWidgetCollections`) made per-process-safe; static caches/time/env reset between tests where leaked. The build-pipeline real-artifact writers (`BuildServerSettingsSession125Test`, `WidgetJsLibsSession138Test`, `AssetBundleDriftGuardTest`) are **already temp-dir-isolated by 296** — confirm they hold under `--parallel`, don't re-solve (Open Q2); reuse 296's injectable-constructor pattern as the template for any newly-surfaced shared-FS offender. Prefer ownership over serialization; a serial-only carve-out is allowed only with a logged one-line rationale and user awareness (Open Q1).

### Phase 3 — Prove order-independence & make it the default

Run `--parallel` ≥2× and shuffled; the **identical pass count** must hold every run (this is the zero-coverage-loss proof). Make `--parallel` the default fast invocation everywhere it's invoked (`dev`, CI, docs). Record before/after wall time.

### Phase 4 — Isolated e2e stack

Stand up the second stack (Open Q3 shape): a separate compose project/override with distinct container names + host ports and its own throwaway DB volume, reusing the repo bind-mount and the existing built image (no separate checkout, no second image build). It must come up alongside the dev stack without colliding. **Parameterise the three hardcoded e2e targets** so the harness can be pointed at it: `resetDatabase()`'s `docker compose exec` target (project/service), `db.ts`'s pg `host:port`, and `APP_URL`. Defaults stay the dev stack so existing local usage is unaffected; the isolated stack is opt-in via env. This is the crux deliverable — keep the change at the harness boundary; do not rewrite specs.

### Phase 5 — Playwright parallel-safe + into CI

On the isolated infra: relax `workers:1`/`fullyParallel:false` only as far as order-independence provably holds (per-worker DB or serial-reset semantics — the same zero-coverage-loss / shuffled-run discipline as the Pest side; never trade correctness for speed). Wire a **Playwright CI job** using the parameterised reset against CI service containers — sequenced after Pest within a single CI run (`feedback_test_runs_not_parallel` still holds *within* one stack), as its own stage off the human critical path, gated into the existing `Tests` aggregate so branch protection still means "everything green." Record the **full-corpus** before/after wall time — total time for everything that runs, the number that actually matters.
