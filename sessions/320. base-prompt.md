**Session shape:** migration-heavy infrastructure. Two coupled halves: (1) switch Spatie's path generator so files with identical `content_hash` resolve to **one physical file** (content-addressed storage), with a one-time relocation migration that moves existing files and collapses duplicates; (2) make deletion **refcounted** so removing one owner's `media` row never deletes bytes another row still references — the physical file is removed only when the last referencing row goes. It rewrites how every file is stored and deleted and touches every file on disk, so the open gate below settles the design choices up front. There's no production data anywhere in this project and the user keeps their own backups of meaningful public-site work, so a botched relocation just means a reseed — treat the design decisions as "pick the right shape once so we don't rebuild," not as risk management.

## Process rules for every session

Base-prompt process rules apply in full. Session-specific notes:

- **The relocation migration touches every file on disk.** Make it idempotent and chunked by id, so re-running is a no-op for already-relocated files — that's ordinary good practice for a bulk migration, not risk management; there's no production library to protect, and a bad run just reseeds. Decide migration-time vs. one-shot `media:relocate-cas` command against the actual library size, exactly as 319 did for the hash backfill, and document the choice.
- **Refcount design before code.** Deleting a `media` row must remove the physical file **only when no other row references the same content-addressed path**. Decide at the open gate whether the refcount is **computed** (count rows sharing `content_hash` / CAS path at delete time — no schema change, slightly slower delete) or **stored** (a counter column maintained on add/delete — schema change, faster, must stay consistent). Either way it goes through a custom `file_remover_class`; do not delete files outside that path.
- **Settle URL stability at the open gate, not mid-build** — purely so the approach isn't rebuilt halfway. See reading-list item 6. Whichever way you go (id-stable URLs via a decoupled `UrlGenerator`, or a migration that rewrites embedded `/storage/{id}/` URLs), `MediaReferenceInventory` Rule B and the 318 finder must still resolve references correctly afterward, and existing rich-text embeds must not 404. Add a regression test that proves an embedded inline image still renders and is still classified `live` after relocation.
- **Conversions are the real multiplier — state the scope explicitly.** Each media original carries responsive/webp conversions (~6× the originals). CAS dedup of *originals* does not dedup *conversions* unless conversions are also content-addressed. Decide at the open gate whether v1 dedupes originals only (simpler, still a large win) or conversions too (bigger), and be honest in the log about which.
- **Make the 319 reuse path real.** Once CAS lands, the page-builder `use-existing-*` endpoints (which currently `Media::copy()` fresh bytes) should resolve to the shared physical file rather than writing a copy. Update them or document explicitly why they still create a distinct `media` row (row-per-owner is fine; it's the *bytes* that must dedupe).
- **Backup/restore + portability.** The spatie backup stores the media tree by path and the 303 portability work preserves media ids; a path-generator change interacts with both, so check them and update the restore runbook in `docs/app-reference.md` if the on-disk layout changes. Not a blocker — just keep the docs honest.
- **Run migrations via Docker** after writing them, then re-run the full media classification (live / dead / orphan) and a duplicate scan to prove the relocation collapsed duplicates without orphaning or breaking any referenced file.
- **PM-level tone for status summaries** per `feedback_pm_level_tone`. **Questions are not instructions** per `feedback_questions_are_not_instructions`.
