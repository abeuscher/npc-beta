# 240. Fleet Manager Agent — Contract v1.1.0 (`unknown` Status for `last_backup_at`)

Bump the agent contract from `v1.0.0` to `v1.1.0`: introduce `unknown` as a fourth valid subcheck-level status value, flip `HealthController::checkLastBackupAt()` to emit `unknown` (not `green`) when no backup pipeline exists yet, and treat `unknown` as equivalent to `yellow` in the overall-status worst-of derivation. Sequenced before FM-side session 004 so the FM HTTP client and contract validator build against the final shape with no retrofit.

---

## Design decisions (resolved before starting)

- **Bump is additive — `1.0.0 → 1.1.0`, not `2.0.0`.** No existing field shape changes, no field is removed, and any existing FM consumer still reading the v1.0.0 surface continues to function (the only response value that changes for FM is the `last_backup_at.status` literal — everything else is byte-equivalent). Per the version-negotiation discipline already documented in the spec doc: minor bumps are forward-compatible within a major.
- **`unknown` is a subcheck-level status only — top-level `status` stays `green | yellow | red`.** The worst-of overall-status derivation maps `unknown` to `yellow`-tier ranking. FM reads the top-level `status` field and gets a value in the existing v1.0.0-compatible enum; FM reads the per-subcheck `status` and may now see `unknown`.
- **`unknown` ≡ `yellow` in the worst-of rule for v1.1.0.** Concretely: overall is `red` if any subcheck is `red`; else overall is `yellow` if any subcheck is `yellow` **or** `unknown`; else overall is `green`. This keeps the top-level enum stable while honoring the FM-requested semantic at the subcheck layer. Ranking may evolve if operational experience shows missing data is more (or less) concerning than degraded data — flagged in the spec doc for future revisit.
- **Only `last_backup_at` emits `unknown` in v1.1.0.** Other subchecks (`app`, `database`, `redis`, `disk`, `version`) continue to emit `green | yellow | red`. The `unknown` value is reserved for "we genuinely don't know yet" — currently only `last_backup_at` (no pipeline). When Phase 2 (Backup Pipeline) lands, `last_backup_at` flips from `unknown` to threshold-driven (green/yellow/red against a real timestamp).
- **`message` field on `last_backup_at` stays as-is.** Currently `'backup pipeline not yet implemented'`; that string remains accurate. The `unknown` status reframes its meaning operationally without changing the human-readable message.
- **No migration.** Pure-PHP contract evolution; no database changes.
- **No FM-side cache update from this session.** The CRM emits the canonical spec doc; FM refreshes its local cache (`docs/imported/fleet-manager-agent-contract.md`) on its next boundary-touching session per the Two-Repo Coordination Protocol. The CRM does not cross repo boundaries.

## Phase 1 — Controller surface

Edit `app/Http/Controllers/Api/Fleet/HealthController.php`:

1. **Bump the version constant.** `CONTRACT_VERSION = '1.0.0'` → `'1.1.0'`.

2. **Flip `checkLastBackupAt()`.** Currently emits `['status' => 'green', 'value' => null, 'threshold' => null, 'message' => 'backup pipeline not yet implemented']`. Change `'status' => 'green'` to `'status' => 'unknown'`. All other fields unchanged.

3. **Update the worst-of derivation in `index()`.** The current rule emits overall `red` if any subcheck is red; else `yellow` if any subcheck is yellow; else `green`. Adjust so the yellow-tier branch fires when any subcheck is `yellow` **or** `unknown`. Concretely: the existing yellow-detection condition becomes `if (in_array($status, ['yellow', 'unknown'], true))` (or equivalent — match the existing code's idiom). Top-level emissions remain in the `green | yellow | red` enum — `unknown` never appears as a top-level value.

No other changes to the controller or its dependencies. The auth middleware, route, config, and version pipeline are all unchanged.
