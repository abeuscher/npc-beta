## What was built

The reclamation leg of the media trilogy (318 visibility → 319 prevention → **320 reclamation**). Identical media bytes now share one physical file, and deletion is refcounted so removing one owner's `media` row never deletes bytes another row still references.

### Open-gate decisions (settled with the user before any code)

- **URL stability:** URLs follow the content-addressed path (`/storage/cas/{shard}/{hash}/{file_name}`); embedded `/storage/{id}/` URLs are rewritten in the relocation. *Chosen over the prompt's "id-stable decoupled UrlGenerator" lean* because media is served as static files through the `public/storage` symlink — the public URL **is** the physical path, so an id-stable URL decoupled from the hash path would 404 without either a PHP streaming route (kills static serving) or per-id symlinks (not S3-portable).
- **Refcount:** **computed** at delete time over the `content_hash` index — **no schema column added**.
- **Conversions:** deduped as a consequence of directory-sharing — content-addressing keys the whole media *directory* on the original's hash, so identical originals share their `conversions/` and `responsive-images/` too (no separate conversion-hashing pass).
- **Relocation:** a one-shot idempotent `media:relocate-cas` command invoked inline by the migration; `down()` is intentionally irreversible (the project's reset is a reseed, not a rollback).

### Phase 1 — Content-addressed path generator

- **`App\Services\Media\ContentAddressedPathGenerator`** (registered as `path_generator`) — derives the base directory from `content_hash`: `cas/{hash[0:2]}/{hash}/`. Falls back to the legacy id path when the hash is not yet known (the brief window before `MediaHasBeenAddedEvent` fires on initial add).
- **`App\Services\Media\MediaRelocator`** — moves a media's files from the legacy id path to the content-addressed path, deleting the source when an identical file already sits at the target (dedup). Idempotent/resumable. Used by both the add-time listener and the relocation command.
- **Relocate-on-add:** the existing `MediaHasBeenAddedEvent` listener now hashes **then** relocates. Because the event fires after the original is copied but before conversions are generated, the original lands at its hash path and all subsequent conversions follow. A consequence verified in tests: `Media::copy()` (the 319 "use-existing" reuse path) now resolves to the source's shared physical file — **the 319 reuse path became real byte-sharing with no endpoint change.**

### Phase 2 — Relocation migration + embed rewrite

- **`media:relocate-cas`** (`RelocateMediaToCasCommand`) — chunked by id; hashes any un-hashed row, relocates files (moved / deduped / noop / no-hash counts reported), then rewrites embedded `/storage/{id}/` URLs across the five rich-text surfaces (`page_widgets.config`, `collection_items.data`, `events.description`/`meeting_details`, `email_templates.body`) to the content-addressed form. Idempotent.
- **Migration** `2026_05_22_130000_relocate_media_to_content_addressed_storage.php` — runs the command inline; no schema change.
- **`MediaReferenceInventory` Rule B** — now resolves identity from `/storage/cas/{shard}/{hash}/` tokens (hash → all rows carrying it) as well as the legacy numeric `/storage/{id}/` form. Conservative (an embedded hash marks every row with that hash referenced); never falsely flags an embedded asset unused.

### Phase 3 — Refcounted deletion

- **`App\Services\Media\ContentAddressedFileRemover`** (registered as `file_remover_class`) — extends Spatie's `DefaultFileRemover`; removes physical files only when no other `media` row shares the `content_hash`. Otherwise the DB row is deleted and the shared bytes stay. Runs in the existing `MediaObserver::deleted()` path, so the 318 Media Finder's per-row delete is now refcount-safe.

### CAS ↔ portability interaction (fixed in-session)

The s303 self-contained-bundle / id-preserving media seed hardcoded `{id}/{file_name}` as the byte read/write path and parsed the media id out of that path — all broken by the layout change. Fixed: `ContentExporter` records each descriptor's real `getPathRelativeToRoot()` and carries `content_hash`; `collectReferencedMedia` reads an explicit `id` field (legacy path-regex kept as a fallback); the importer's explicit-id seed writes bytes to the descriptor's path and seeds `content_hash` so the row resolves at its CAS path. Reconciled the seven 303/156/import-export tests that asserted the old id-based layout.

## Security considerations

No new external surface, no contract change, no portal/public auth change. The refcounted remover reads only the `content_hash` column to decide deletion; the relocation rewrites rich-text URLs in place (no rendering/execution) over the same surfaces the 318 inventory already scans. The embed rewrite and importer byte-write both reject path traversal (`..` / leading `/`) before touching disk. The content-addressed directory name is the SHA-256 of the bytes, so an attacker cannot steer a file to a chosen path without producing the matching content.

## Schema changes

**None.** Refcounted deletion is computed over the existing `media.content_hash` index (319). The relocation is a one-time data/file migration with no DDL. `docs/schema/media.md` documents the content-addressed layout + refcounted-delete model.

## Production deployment notes

Standard `migrate` runs `media:relocate-cas` inline (idempotent — a no-op on already-relocated files, and a no-op on a fresh-seed install with no media yet). For a large library the command can be re-run by hand. The on-disk layout changed (`{id}/…` → `cas/{shard}/{hash}/…`); the spatie backup/restore copies the tree by path so it is transparent (restore runbook in `docs/app-reference.md` updated). `media-finder.md` runbook updated (deletion is refcounted; duplicate clusters now share bytes).
