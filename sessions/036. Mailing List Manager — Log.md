## What was built

- Migration: `mailing_list_opt_in` boolean (default `false`) added to the `contacts` table

### Service
- `app/Services/MailingListQueryBuilder.php` — static `build(MailingList $list)` with mutually exclusive modes:
  - **Simple mode**: applies repeater filters with AND/OR conjunction using `where`/`orWhere` closures
  - **Advanced mode**: validates a blocklist of dangerous SQL keywords, executes `SELECT id FROM contacts WHERE (...)` against the `pgsql_readonly` connection with a 5-second statement timeout, returns `Contact::query()->whereIn('id', $ids)`
  - Simple and advanced are mutually exclusive — `raw_where` presence determines which path runs

## Decisions made

- Simple filters and advanced raw WHERE are **mutually exclusive** — not combined. If `raw_where` is set the simple filters are ignored entirely.
- The `pgsql_readonly` user should only have `SELECT` on the `contacts` table — not CMS, financial, or other tables. SQL to create the user is documented in the help article.
- If `DB_READONLY_USERNAME`/`DB_READONLY_PASSWORD` are not set, the main credentials are used as fallback — advanced mode still works but without the extra isolation layer.
- `is_member` and `is_donor` were considered as Settings flags and rejected — both are already derived from real relational data (`memberships` and `donations` tables). Boolean flags would contradict live data.
- `is_volunteer` was also considered and deferred — no volunteer system exists yet; will be addressed in the Volunteer Management session using the same relational pattern.
