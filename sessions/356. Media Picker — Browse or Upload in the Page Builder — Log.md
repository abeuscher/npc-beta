## What was built

A page-builder **media picker**: authors can now visually browse the site's existing images as a thumbnail grid and pick one, or upload — in one modal — wired into both the widget config image field and the appearance background-image field. Two pieces over the existing media stack: a browsable-media **list endpoint** (none existed) and a **Vue browser modal**. The attach-existing-media-by-id backend (`useExistingImage` / `useExistingAppearanceImage`) already existed and was built on, not rebuilt.

### Phase 1 — Browsable-media list endpoint

- **`PageBuilderApiController::mediaList()`** + route `GET admin/api/page-builder/media` (slotted with the other lookups in `routes/admin-api.php`; **no** `BlockDemoUploads` — read-only, `view_page`-gated, so the demo role can browse).
- Contract: `{ data: [{ media_id, url, file_name, size, mime_type }], has_more }`. Image media only (`mime_type LIKE 'image/%'`); **excludes** `WidgetType`-owned chrome thumbnails (`model_type != WidgetType::class`); filename `search`; paginated (60/page, `has_more` via fetch-one-extra).
- **Deduped by `content_hash` (Q1):** one representative row per distinct image — `MAX(id)` per `GROUP BY COALESCE(content_hash, id::text)`, ordered most-recent-first; null-hash rows stand alone via the `id::text` fallback (never collapsed together).
- **Thumbnail URL (Q5):** the generated `webp` conversion when present, else the original — `browseThumbUrl()`, no new conversion introduced (mirrors `MediaLibraryPage` / `MediaDedupService::previewUrl`).
- **Pest:** new `tests/Feature/PageBuilderMediaListTest.php` (6 cases — picker shape + non-image excluded, dedup-by-hash representative, WidgetType chrome excluded, filename search, 403 without `view_page`, demo can browse) + a "demo can browse the media list" guard added to `tests/Feature/DemoRoleLockdownTest.php` (sits with the session-329 upload-lockdown block — browse open, upload still gated).

### Phase 2 — Media-browser modal + field wiring

- **`MediaBrowserModal.vue`** (new, mounted at app root in `App.vue` next to `DedupPromptModal`): thumbnail grid + search box (debounced) + "Upload new" + "Load more", with loading/empty/error states, Escape + click-outside to close. Driven by store state.
- **Store (`stores/editor.ts`):** `mediaBrowser` ref + `openMediaBrowser(target)` / `closeMediaBrowser()`, where `target` is `{ kind: 'config', widgetId, key }` or `{ kind: 'appearance', widgetId }`. Exposes `useExistingMedia`.
- **`useUploadActions.ts`:** new `useExistingMedia(target, mediaId)` — mirrors the use-existing branch of the existing upload actions (CAS copy, no dedup gate) **without touching** `uploadImage` / `uploadAppearanceImage` / `removeImage` / `removeAppearanceImage`.
- **`api.ts`:** `listMedia({ search, page })` + `MediaBrowserItem` / `MediaBrowserTarget` types (`types.ts`).
- **Entry points:** a "Browse library" button in `ImageUploadField.vue` (image fields only — video fields keep the bare file input, Q4) and in the appearance image sub-panel (`BackgroundPanel.vue` emits `browseLibrary` → `InspectorPanel.vue` `onWidgetBrowseImage` → `openMediaBrowser({ kind: 'appearance', … })`). The modal's "Upload new" reuses the existing demo-gated upload path; selecting reuses `useExistingMedia`.
- **Modal home (Q3):** a new component; `DedupPromptModal` kept separate (not folded in) — different trigger (post-upload hash nudge vs proactive browse), and keeping it separate left the working upload+dedup path untouched.
- `npm run build` clean.

### Phase 3 — Verify the browse-and-pick flow

- **`tests/e2e/page-builder/media-picker.spec.ts`** (new standing guard, 2 tests, both green): (1) upload a PNG on one Image widget (upload still works) → on a second widget open the library → the uploaded image shows as a thumbnail → click to select → the widget's image id is set + persists across a reload; (2) the in-modal search box empties the grid on a miss and restores on clear. Self-contained — seeds selectable placeholders (empty Image widgets render zero-height/unclickable) and builds a valid PNG (CRC + zlib IDAT) for the upload.
- Built bundle `docker cp`'d into `nonprofitcrm_e2e_app` (image-baked `public/build` volume); run via `npm run test:e2e` (sequential with Pest).

### Q1–Q5 decisions (as built)

1. **Dedup by `content_hash`** — yes; representative = `MAX(id)` per hash group; null-hash rows stand alone.
2. **Browsable scope** — all `image/*` except `WidgetType` chrome thumbnails; seeded `SampleImage` placeholders kept in (owner-confirmed at the gate).
3. **Modal home** — new `MediaBrowserModal`; `DedupPromptModal` left separate.
4. **Video fields** — images-first; Browse shows only on image fields; video unchanged.
5. **Thumbnail source** — existing `webp` conversion if present, else original; no new conversion.

## Security considerations

The new list endpoint is read-only and `view_page`-gated, matching the other page-builder lookups. It deliberately carries **no** `BlockDemoUploads` middleware because browsing/selecting existing media introduces no new file — preserving the established demo split (new-file upload stays 403 for the `demo` role; existing-media reuse was already ungated). A new Pest case in `DemoRoleLockdownTest` locks that split in. The endpoint exposes only image-media metadata + URLs from the already-public media disk (no PII). The dedup grouping uses a fixed `DB::raw` expression (`COALESCE(content_hash, id::text)`) with no user input in the raw SQL; `search` is a bound `where … like`. No new write path; `useExistingMedia` calls the pre-existing `useExistingImage` controller actions unchanged.
