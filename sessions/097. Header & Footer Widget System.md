# 097. Header & Footer Widget System

Replace the hard-coded header and footer Blade partials with widget-driven content areas using the existing page builder system, so that header and footer are editable the same way as any page.

---

## Design decisions (resolved before starting)

- Header and footer each become a system Page record (type `system`, slugs `_header` and `_footer`, status `published`). These are non-routable — the underscore prefix signals internal-only pages. The `PageController` catch-all route should reject slugs starting with `_` (404).
- The two system pages are created by a seeder (`SystemPageSeeder` or a dedicated `ChromePageSeeder`), following the same pattern as the existing portal system pages.
- The Site Theme page gains a third tab, **Header & Footer**, to the right of SCSS Editor. The tab contains two stacked PageBuilder instances — one for the header page, one for the footer page. Each shows its own widget picker, sort/drag, inspector, etc.
- A new Spatie permission `edit_site_chrome` gates visibility of the Header & Footer tab. Not assigned to any role by default — power-user feature. Register in `PermissionSeeder`.
- Two new purpose-built widget types are seeded:
  - `site_header` — config fields: logo (image), nav menu handle (text, default `primary`), header content (richtext). Its template replicates the current `components/site-header.blade.php` output.
  - `site_footer` — config fields: nav menu handle (text, default `footer`), show theme toggle (boolean, default true), copyright text (text, optional). Its template replicates the current `components/site-footer.blade.php` output.
- No widget region restrictions. Any widget type can be placed in the header, footer, or any page. The purpose-built widget types are simply designed for those areas.
- The public layout rendering logic: if the header system page has active widgets, render them via the widget engine; otherwise fall back to the current `components/site-header.blade.php` partial. Same for footer. This preserves backward compatibility for existing installs and custom overrides.
- The existing Site Theme "Header & Footer" section (nav handles, logo upload, header content rich editor, custom header/footer status placeholders) is removed from the Appearance tab. These controls are superseded by the widget config fields. The `header_nav_handle`, `footer_nav_handle`, `header_content`, and `logo_path` site settings remain in the database for fallback rendering but are no longer editable through the Appearance form.
- The `custom.header` / `custom.footer` Blade override check (`view()->exists('custom.header')`) is preserved — if a custom override exists, it takes priority over everything (widget-driven or fallback). Order of precedence: custom Blade override > widget-driven > legacy fallback partial.

---

## Phase 1 — System pages for header and footer

- Create the two system Page records (`_header`, `_footer`) in the seeder. Both are type `system`, status `published`, author is the first admin user.
- Guard the `_header` and `_footer` slugs from public routing — return 404 for any slug starting with `_` in `PageController::show()`.
- Register the `edit_site_chrome` permission in `PermissionSeeder`. Do not assign to any role by default.

---

## Phase 2 — Header & Footer tab on Site Theme

- Add a third tab **"Header & Footer"** to the Site Theme page Blade view, visible only to users with `edit_site_chrome` permission.
- The tab renders two labelled sections, each containing a Livewire `PageBuilder` component instance pointed at the respective system page (`_header` / `_footer`).
- Remove the existing "Header & Footer" section from the Appearance tab form schema. Remove the corresponding fields from `mount()` and `save()`: `header_nav_handle`, `footer_nav_handle`, `logo_upload`, `header_content`, `custom_header_status`, `custom_footer_status`. Leave the site settings in the database — the fallback partials still read them.

---

## Phase 3 — Purpose-built widget types

Seed two new widget types in `WidgetTypeSeeder`:

### `site_header`

- **Config schema:**
  - `logo` — image field
  - `nav_handle` — text, default `primary`, helper: "Navigation menu handle to display"
  - `header_content` — richtext, optional, helper: "Content beside the logo"
- **Template:** renders the same HTML structure as the current `components/site-header.blade.php` — responsive nav with mobile hamburger menu, logo, optional content area, dropdown support for nested nav items.
- **CSS/JS:** move any header-specific styles into the widget's CSS field if needed.

### `site_footer`

- **Config schema:**
  - `nav_handle` — text, default `footer`, helper: "Navigation menu handle to display"
  - `show_theme_toggle` — boolean, default true, helper: "Show the light/dark mode toggle"
  - `copyright_text` — text, optional, helper: "Copyright line (e.g. '2026 My Org. All rights reserved.')"
- **Template:** renders the same HTML structure as the current `components/site-footer.blade.php` — nav links, optional theme toggle, optional copyright text.

---

## Phase 4 — Public layout rendering

Update `resources/views/layouts/public.blade.php`:

- **Header rendering order of precedence:**
  1. If `custom.header` Blade view exists → use it (existing behaviour, unchanged)
  2. Else if the `_header` system page has active widgets → render them through the widget engine (same rendering path as page widgets: resolve widget types, run Blade templates, collect inline styles/scripts)
  3. Else → include `components/site-header.blade.php` (legacy fallback)

- **Footer rendering order of precedence:**
  1. If `custom.footer` Blade view exists → use it
  2. Else if the `_footer` system page has active widgets → render them through the widget engine
  3. Else → include `components/site-footer.blade.php` (legacy fallback)

- Inline styles and scripts from header/footer widgets should be collected and injected alongside the page widget styles/scripts (not duplicated).

Create a service or helper method (e.g. on `PageController` or a new `ChromeRenderer` service) that renders a system page's widgets to HTML + styles + scripts, reusing the existing widget rendering logic. This avoids duplicating the render loop in the layout Blade file.
