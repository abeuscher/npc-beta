## Process rules for every session

Base-prompt process rules apply in full (see `sessions/template-base-prompt.md`). Session-specific notes:

- **Reuse, don't reinvent.** The LP is a `Page` (page-builder page). Deletion protection plugs into the existing `PageObserver` guard. The presets seed *existing* widgets (Hero / EventDescription / EventRegistration). No new page type, no new rendering pipeline, no new route — an LP at slug `events/{event-slug}` renders through the normal `/{slug}` page path.
- **The Simple/Standard choice is a creation-time preset, not persisted state.** Once created, the page's widgets *are* the state; the operator can edit it like any page. Do not add a column to remember the choice unless a concrete need surfaces at the gate.
- **Free-gating.** Simple is offered only when the event has no ticket tiers (is free). The moment ticketing exists, Standard is forced. A free Simple event that later gains ticketing needs no special migration — the operator adds the registration widget to the page like any other widget (decided at scoping).
- **Front-end build** if any widget asset / `widget_types` row / Vite source changes (`build:public` and/or `npm run build` per the path→command rule). Relinking the mini-calendar touches its assets → `build:public`.
- **Seeders:** event-creating seeders (e.g. `DemoBaselineSeeder`) must give their events LPs so the demo's calendar/listing links resolve — update them in step with the new creation flow.
