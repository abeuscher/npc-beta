# Session 335 — Mobile Vertical Spacing — Reduce Section Whitespace — Log

**Date:** 2026-06-01
**Branch:** session-335/1

## What was built

The mobile public site carried desktop-sized vertical section spacing on phones — operator padding/margins were written as **literal inline px** (`padding-bottom:80px`), and inline always beats a stylesheet, so nothing could compress them at narrow widths. This session built the **section-spacing primitive (vertical axis only)**: emit the operator's top/bottom padding/margin as `--np-*` custom properties, consume them in a host-layer rule, and scale them down at tablet/mobile via one tunable ratio. Desktop is byte-for-byte unchanged.

### Phase 1 — Reproduced the whitespace in a real browser

Drove the live local site (read-only standalone Playwright, never the DB-resetting specs) at desktop / tablet / mobile. The dominant offender was a **chrome footer text-block** set to `padding-bottom:150px` — identical on a 375px phone and a 1280px desktop. Confirmed the wrapper elements (`.widget`, `.page-layout`, and the chrome wrappers) carry **no intrinsic padding/margin** in any SCSS — the only vertical spacing on them is the operator's appearance values. That fact decided the "0 = no override" approach.

### Phase 2 — Built the primitive

**The "concrete 0 = no override" approach (the one real design call):** keep the existing non-zero emit gate, but emit the **vertical** axis as `--np-pad-top` / `--np-pad-bottom` / `--np-mar-top` / `--np-mar-bottom` instead of literal `padding-*`/`margin-*`. The host rule consumes each with a **`var(--np-pad-top, 0px)` fallback** — the `0px` reproduces today's "no override" exactly, because the wrappers' natural value is already 0, so an unset widget is unchanged. Horizontal (left/right) stays a literal declaration, untouched. Tradeoff: this leans on the verified "wrappers have no intrinsic padding" fact; a comment by the rule names the constraint.

- **Shared helper** `AppearanceStyleComposer::composeVerticalSpacingVars(array $padding, array $margin)` — the single source emitting the four vertical custom properties (only for non-zero sides). Called by all three render surfaces so they can't drift.
- **Surfaces moved, in lockstep:**
  - **Widgets** — `AppearanceStyleComposer::compose()` (also flows to the page-builder preview via `WidgetPreviewRenderer`, keeping editor↔public consistent).
  - **Column layouts** — `AppearanceStyleComposer::composeForLayout()`.
  - **Chrome** — `ChromeRenderer::buildInlineStyle()` (header/footer widgets) and the flat-key layout path in `renderLayoutBlock()`. Chrome **root** widgets render as a bare `<div style>` with no `.widget` class, so they gained a **`.np-chrome-section`** hook class to consume the properties.
- **Host rule + scaling** in `resources/scss/_layout.scss` (`@layer host`, ships in both the Vite bundle and the build-server widget bundle per the 332 dual-compile): `.widget, .page-layout, .np-chrome-section` consume the properties, then two cascading viewport steps scale the vertical axis — `@media (max-width: $bp-md)` ×ratio, `@media (max-width: $bp-sm)` ×ratio². **Single-sourced ratio:** `$section-spacing-ratio: 0.8` (mobile step is the Sass-computed square = 0.64) — tune the whole ruleset by editing one number.

**Before/after evidence (live, re-driven):** the chrome footer `padding-bottom` went **150px desktop → 120px tablet (≤768) → 96px mobile (≤576)**; `padding-top` 25 → 20 → 16. Desktop identical before and after. Screenshots + measurements captured under `/tmp/drive-335/{before,after}/`.

*(Gotcha worth recording: an early docblock contained `padding-*/margin-*`, and the `*/` closed the PHP comment, throwing a 500. Fixed; lint-clean after.)*
