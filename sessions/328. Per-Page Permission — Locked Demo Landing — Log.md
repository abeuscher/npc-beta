## What was built

A per-page **edit lock** for the `Page` model (pages, posts, event landing pages, and the header/footer system pages — all one model), enforced server-side and used to protect the demo site's content from the shared `demo` account.

### Phase 1 — Column + permission + enforcement

- **Migration** `2026_05_30_130000_add_locked_to_pages_table` — `pages.locked` boolean, default `false`, after `status`. `Page` model: `locked` added to `$fillable`, concrete `false` in `$attributes`, boolean `$casts`. `docs/schema/pages.md` updated in the same pass.
- **`edit_locked_pages` permission** seeded in `PermissionSeeder`; granted to the three non-demo roles that already hold full page CRUD — `cms_editor`, `blogger`, `developer`. `super_admin` is covered by the existing `Gate::before` bypass. `demo` is **excluded**. (The session prompt said "admin + super_admin"; there is no literal `admin` role, so this maps to the real page-editing roles — flagged at the open gate, owner-confirmed.)
- **One rule, two enforcement points** — `locked && ! user->can('edit_locked_pages')` ⇒ deny:
  - `PagePolicy::update()` **and** `delete()` (the policy covers the Filament edit + single-delete actions on both PageResource and PostResource, since posts are the same model).
  - The **page-builder save API** — a private `PageBuilderApiController::assertOwnerEditable()` guard wired into every write method: `store`, `update`, `destroy`, `copy`, `reorder`, all six image-upload/remove endpoints, `storeLayout`, `updateLayout`, `destroyLayout`. It covers both route shapes — the owner-scoped `pages/{owner}/...` routes (owner resolved by `ResolvePageBuilderOwner`) and the widget/layout-keyed routes (owner via `$widget->owner` / `$layout->owner`). Templates are unaffected. Returns **403** on a locked page.

### Phase 2 — Admin control

- Permission-gated **"Lock editing (Published & Locked)"** toggle on the page edit-details form (`EditPageDetails`), the page create form (`PageResource::form`), and — added when the owner asked to extend it to blog posts — the post edit-details form (`EditPostDetails`) and post create form (`PostResource::form`). Visible only to holders of `edit_locked_pages`.
- **Combined lock indicator** on the status badge in both the page list and the post list: a `heroicon-m-lock-closed` icon + tooltip when `locked` (not a second status column — confirmed at the open gate).

### Phase 3 — Demo protection

- **`demo:reset` locks every page** — `Page::query()->update(['locked' => true])` runs after the baseline seeder inside the already-`isDemoMode()`-guarded `DemoResetCommand`. Idempotent and re-applied on every reset, so the lock survives the daily wipe and automatically covers the curated showcase pages session **E** will seed. Pages a demo user creates mid-session stay editable by them and are wiped on the next reset.
- **Demo role `navigation_menu` trimmed to view-only** in `PermissionSeeder` (create/update/delete revoked) — the header/footer *link structure* lives in `NavigationMenu`/`NavigationItem`, a model the page lock can't reach.

### Beyond the prompt's two named points (deletion hardening)

The `demo` role holds `delete_page`, and Filament's resource-level delete paths bypass the policy, so two gaps were closed to make the lock real:
- `PageResource::canDelete()` (a static override standing in for `PagePolicy::delete()`) is now lock-aware.
- The **bulk-delete** closures on both `PageResource` and `PostResource` skip locked records (mirroring the existing silent system-page skip), since Filament's default `DeleteBulkAction` authorizes the action, not each record.

## Security considerations

This session adds a real authorization boundary, so the security axis is load-bearing.

- **What the boundary covers:** editing of any `Page`-model record (page, post, event LP, header/footer system page) that is flagged `locked`. A user without `edit_locked_pages` cannot edit it via the Filament resource form, cannot delete it (single or bulk, either resource), and cannot mutate its widgets/layouts/media through the page-builder save API.
- **Where it's enforced — server-side only.** `PagePolicy::update()`/`delete()` + the `PageBuilderApiController` guard are the protection; the Filament toggle is UX for *setting* the value, never the gate. The save API is the real edit surface and previously did **no** per-page check — that wiring is the genuinely new bit, and it covers every write route (both the owner-scoped and the widget/layout-keyed shapes), not just the `pages/{owner}` group the prompt named.
- **Lock is orthogonal to `status`.** A locked page stays `status='published'` and publicly visible — `Page::scopePublished()` is untouched, so the demo landing remains live; only editing is barred. Deliberately not a `status` enum value (that would drop locked pages out of every `published()` query) and not the superseded §2a `restricted_roles` deny-list.
- **`super_admin`** is never blocked — `can('edit_locked_pages')` returns true via the existing `Gate::before` bypass, so all enforcement points pass it through.
- **Demo node:** every page is locked on each `demo:reset`, and the demo role lacks both `edit_locked_pages` and nav-menu write — closing the page-content and the header/footer-link edit surfaces. CRM/event *record* edits stay allowed (the demo's purpose; the daily wipe is the mitigation).
- **Intentionally not covered this session** (documented for the 329 / pen-test pass): `form` and `collection` edits that surface in public widgets (same "site content" category — view-only vs lock to be decided later); and the **demo media-upload surface**, which is **session 329**'s separate axis. Mid-session demo-created pages are editable by their creator until the next reset wipes them.
