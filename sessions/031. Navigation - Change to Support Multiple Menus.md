# 031. Site Chrome — Named Menus, Header & Footer

Build the public-facing site chrome (header and footer) as managed, configurable components while preserving a clean developer escape hatch for per-client customisation. This session does not introduce a page template system — that is session 059.

---

## Background & decisions

Navigation currently renders from a single hard-coded query in `public.blade.php`. There is no concept of named menus. The header and footer are not separately configurable.

The architecture decision reached in session 030:

- **Widgets are for page content.** Header and footer are site-level chrome — they do not live on a page and do not belong to the widget system.
- **Navigation stays as its own model.** Merging nav into Collections was considered and rejected: `NavigationItem` has a live `page_id` foreign key that resolves slugs at render time. Storing nav items as Collection JSONB would break that link silently when page slugs change.
- **The escape hatch pattern:** the layout renders a default Blade component; if a developer has dropped a `custom/header.blade.php` into the deployment, that file is used instead. No UI needed for this — it is a developer feature.

---

## What to build

### 1. Named navigation menus

Add a `menu_handle` string column to `navigation_items` (e.g. `primary`, `footer`). Default all existing rows to `primary`.

Update `NavigationItemResource`:
- Add a `menu_handle` field to the form (text input, required, default `primary`)
- Add a `menu_handle` column to the table
- Add a `SelectFilter` for `menu_handle` so editors can filter by menu

No separate `NavigationMenu` model is needed at this stage. Handles are freeform strings managed by convention.

### 2. Site Chrome settings

Extend the existing CMS Settings page with a new "Site Chrome" section containing:

- **Header nav handle** — which menu handle to render in the header (default: `primary`)
- **Footer nav handle** — which menu handle to render in the footer (default: `footer`)
- **Header content** — rich text block for the left side of the header (stored in `site_settings`)
- **Custom override indicator** — two read-only fields showing whether `custom/header.blade.php` and `custom/footer.blade.php` exist on this deployment (`view()->exists()` check, no edit, just informational)

### 3. Default header and footer components

Create `resources/views/components/site-header.blade.php`:
- 40/60 layout (left: header content rich text, right: primary nav)
- Collapses to stacked single-column on mobile
- Alpine.js handles the mobile nav toggle (hamburger)
- Renders the nav tree from `NavigationItem` filtered by the configured handle, with child items supported one level deep

Create `resources/views/components/site-footer.blade.php`:
- Renders the footer nav (flat list, no children needed) from the configured handle
- Simple, minimal structure — styled by the client's custom CSS

### 4. Update `public.blade.php`

Replace the hard-coded nav query and nav markup with:

```blade
@include(view()->exists('custom.header') ? 'custom.header' : 'components.site-header')
```

```blade
@include(view()->exists('custom.footer') ? 'custom.footer' : 'components.site-footer')
```

Remove the existing `@php` nav query block and `<nav>` markup entirely.

### 5. Migration

- Add `menu_handle` string column to `navigation_items`, nullable, default `primary`
- Seed existing rows to `menu_handle = 'primary'`

---

## The escape hatch

A developer customising a client deployment creates `resources/views/custom/header.blade.php` (and/or `custom/footer.blade.php`). These files are not tracked in the main git repository. They have full access to all Laravel view data, Alpine.js, site settings, and can still `@include` the managed nav partial if desired:

```blade
@include('components.site-nav', ['handle' => 'primary'])
```

The `custom/` directory should be added to `.gitignore` so it is never committed to the base app.
