# 068. Roadmap Planning & Help Content

A housekeeping and planning session — no new features. Two workstreams, both aimed at improving clarity before the next phase of development.

---

## Workstream 1 — Session outlines audit

**File:** `sessions/session-outlines.md`

Compare the outlines document against:
- The completed sessions table (cross-reference log files in `sessions/` where available)
- The current codebase (routes, models, Filament resources, pages)

For each item in the upcoming sections of the outlines doc, determine one of:

- **Done** — the work described is clearly already built. Move it to the Completed Sessions table (or remove it entirely if it has no meaningful log value).
- **Partial** — some but not all of the described work is built. Note what remains and leave it in place with a clear status line.
- **Unclear** — cannot determine from code alone whether it is complete. Flag it for the user to decide.
- **Not started** — clearly not built yet. Leave in place.

**Do not rewrite the roadmap.** Only update the status of existing items. Flag anything unclear to the user as a numbered list of decisions needed — do not resolve ambiguities unilaterally.

After the audit, review the order of remaining sessions and flag any sequencing concerns (dependencies out of order, blockers not yet resolved). Again, present as questions for the user — do not reorder without agreement.

---

## Workstream 2 — Help page content

**Directory:** `resources/docs/`

Every admin view that is currently built and accessible should have a help doc with real initial content — not just a frontmatter stub.

**Step 1:** Inventory all existing help doc files and compare against all currently registered Filament resources and pages. Identify any views that are missing a help doc entirely and create stubs.

**Step 2:** For each existing help doc, read the current content. If it is a stub (frontmatter only, or a single placeholder line), write real initial content:
- What the view is for
- Key fields or columns and what they mean (where non-obvious)
- Common workflows (e.g. "To add a tag to a contact…")
- Any important constraints or gotchas (e.g. age gating on contacts, portal vs. member distinction)

Keep the tone practical and direct — these are internal staff docs, not marketing copy. Two to five paragraphs per page is the right length. Do not pad.

**Step 3:** For any help doc that covers a feature that has been meaningfully changed since the stub was written (e.g. notes moved to a sub-page in session 067, portal controls moved to the ellipsis menu), update the content to reflect current behaviour.
