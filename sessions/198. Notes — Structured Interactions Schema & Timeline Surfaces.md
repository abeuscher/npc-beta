# 198. Notes — Structured Interactions Schema & Timeline Surfaces

Expand the current bare-bones `notes` record (polymorphic body + author + occurred_at) into a 10-column import-compatible schema so the Contact Timeline can distinguish calls, meetings, emails, tasks, and plain notes, and so imports from external nonprofit CRMs (CiviCRM, Raiser's Edge, DonorPerfect, Bloomerang, Little Green Light, Salesforce NPSP, etc.) land in a shape that preserves their structure.

---

## Design decisions (resolved before starting)

- **Table name stays `notes`.** No rename. UI labels can evolve independently of the data layer; minimising churn across the existing polymorphic relationship and Timeline page is worth more than semantic purity.
- **10-column MVP schema.** Existing columns (`notable_type`, `notable_id`, `author_id`, `body`, `occurred_at`, `import_source_id`, timestamps, soft-delete) stay. Six new columns added: `type`, `subject`, `status`, `follow_up_at`, `outcome`, `duration_minutes`, `meta`. This matches what exports from CiviCRM / Salesforce / Raiser's Edge / HubSpot / DonorPerfect / Bloomerang consistently carry — the minimum shape that makes "import most things from anywhere" possible.
- **`type` and `status` are strings, not Postgres enums.** Every source system uses different vocabulary ("Phone Call" vs "Call", "Left Message" vs "Voicemail", "Completed" vs "Done"). Strings preserve source vocabulary through import with zero mapping config. UI treats a canonical set as known-with-icons (`call` / `meeting` / `email` / `note` / `task` / `letter` / `sms` for type; `completed` / `scheduled` / `cancelled` / `left_message` / `no_show` for status) and renders anything else as a generic badge. Canonical consolidation (merge/rename types like the Tag Manager) is a later session.
- **`meta` jsonb is the catch-all.** Source-specific fields the 10 columns don't capture (priority, location, campaign linkage, solicitor ID, multiple participants, custom CSV columns) land here keyed by the source field name. Never dropped during import. Not user-authorable from the UI in this session — it's write-once from the importer path, read-displayed on the Timeline for any keys present. Later sessions may promote frequently-used `meta` keys into real columns.
- **Tasks are notes with `type='task'` + `status='scheduled'` + `follow_up_at`.** No separate `tasks` table. A task is just a note whose date is in the future and whose status hasn't flipped to `completed`.
- **Polymorphic stays — Contact and Organization both get the new surface.** All ten columns apply equally to both parents (you can have a call with either). Audit every surface that currently edits notes and update each to match.
- **`activity_logs` untouched.** That table remains the automatic CUD audit stream. Notes and activity_logs continue to coexist in the Timeline, distinguished by the existing `_type` discriminator.
- **Participants, assignees, campaign links, moves-management stage** — all deferred to later sessions. If imported data has them, they go into `meta` for now.
- **Importer CSV mapping for the new fields is out of scope.** Stubbed as its own pre-Beta-1 session in `session-outlines.md` under "Importer — Custom Field Mapping for Notes / Interactions." The `meta` column lands in this session; the wizard that writes to it lands later.

## Phase 1 — Migration, model, factory

- Write `database/migrations/{timestamp}_add_structured_fields_to_notes.php`:
  - `type` string, not-null, default `'note'`
  - `subject` string, nullable
  - `status` string, not-null, default `'completed'`
  - `follow_up_at` timestamp, nullable
  - `outcome` text, nullable
  - `duration_minutes` integer, nullable
  - `meta` jsonb, nullable
  - Indexes: `(notable_type, notable_id, occurred_at desc)` for Timeline queries (if not already present); `(type)` for type-filter queries; `(follow_up_at)` partial index where not null for the follow-up dashboard later.
- Run `docker compose exec app php artisan migrate`.
- Update `app/Models/Note.php`: add the six new keys to `$fillable`; add casts (`follow_up_at` → datetime, `meta` → array, `duration_minutes` → int).
- Update `database/factories/NoteFactory.php` (create if missing — check first). Defaults should produce realistic values: `type` randomised over the canonical set, `subject` short faker sentence, `status` default `'completed'`, occasional `follow_up_at` in the future, occasional `outcome` + `duration_minutes` for call/meeting types, `meta` usually `null`.
- Update `docs/schema/notes.md` to document all new columns.

## Phase 2 — Create / edit form surfaces

Every surface where a note is authored needs the new fields added. Audit and update each:

- `app/Filament/Resources/NoteResource.php` — the standalone resource form.
- `app/Filament/Resources/ContactResource/RelationManagers/NotesRelationManager.php` — the inline panel on the Contact edit page.
- `app/Filament/Resources/ContactResource/Pages/ContactNotes.php` — the `create_note` action and `editNoteAction` modal forms.
- Any Organization-side equivalent (grep `NotesRelationManager` and `OrganizationResource`).

### Form layout

Wrap the fields in **two Filament `Section` blocks**:

- **Section 1 — main fields** (always visible, two-column grid for the top row, full-width below):
  - Row 1: `Type` (left) · `Occurred at` (right)
  - Row 2: `Subject` (full-width)
  - Row 3: `Body` (full-width textarea, required)

- **Section 2 — "Additional details"** (Filament `Section::make('Additional details')->collapsed()`, so simple note-taking stays fast):
  - Row 1: `Status` (left) · `Follow-up at` (right)
  - Row 2: `Outcome` (left) · `Duration (minutes)` (right)

### Input components

- **Type** — Filament `Select` with `options()` pre-seeded with the canonical set (`call`, `meeting`, `email`, `note`, `task`, `letter`, `sms`), `searchable()`, `native(false)`. If the Filament version in use supports free-text entry into a Select cleanly, enable that. If not, fall back to `TextInput` with a datalist-style helper text listing the canonical options. Default `'note'`. Imported records may carry other strings; the form just needs to accept and display them without error when editing.
- **Subject** — `TextInput`, max 255, nullable, `placeholder('Optional short title')`.
- **Occurred at** — existing `DateTimePicker`, unchanged. Default `now()`.
- **Body** — existing `Textarea`, `required()`, `rows(4)`.
- **Status** — same pattern as Type: `Select` seeded with (`completed`, `scheduled`, `cancelled`, `left_message`, `no_show`), free-text tolerant. Default `'completed'`.
- **Follow-up at** — `DateTimePicker`, nullable, labelled `"Next action"`.
- **Outcome** — `TextInput`, nullable, labelled `"Result / outcome"`, `placeholder('Short summary of what happened')`.
- **Duration (minutes)** — `TextInput::make('duration_minutes')->numeric()->suffix('min')`, nullable.
- **Meta** — omit from the form entirely. Written only via the importer path (future session).

### Styling

Use existing Filament form component defaults. No custom CSS. The shared `--np-control-*` tokens from session 197 apply to the Timeline rendering (Phase 3), not the form — forms continue to use Filament's native field chrome.

## Phase 3 — Timeline rendering

Update `resources/views/filament/pages/contact-notes.blade.php` and the Timeline data assembly in `ContactNotes::getTimeline()`.

### Data projection

Add `type`, `subject`, `status`, `follow_up_at`, `outcome`, `duration_minutes`, `meta` to the `$noteItems->map()` projection so the Blade can reach them.

### Two visual treatments in one stream

Notes and activity_logs continue to be queried separately and merged at read time, then rendered in the same vertical list, sorted by date descending. They get **two distinct visual treatments** so a reader can tell them apart at a glance:

- **Note entries → structured card** (rich chrome, ellipsis menu).
- **Activity-log entries → compact one-liner** (no card chrome, no menu, system-event styling).

Adjacent entries separate with a light horizontal rule (or equivalent spacing). Use the session-197 control tokens (`--np-control-border`, `--np-control-chip-bg`, `--np-control-bar-bg`, `--np-control-icon-default`) so the Timeline matches the Inspector / Quill visual language.

### Note card layout

Header row — **title leads** because that's the scannable content. Type badge sits to the right, paired with the timestamp block:

```
{subject or body-preview}            {type badge}  {timestamp}   [⋮]
  {body text}
  {outcome, if present}
  {follow-up pill, if present}
  {meta disclosure, if present}
```

Per-element rules:

- **Scannable title (leftmost):** if `subject` is present, render it as semibold (no truncation up to ~60 chars, then ellipsis). If `subject` is null, render the first line of `body` truncated to ~60 chars as the scannable title — so bare notes still have a left-anchored label.
- **Type badge (right-aligned, before timestamp):** icon + label pill. Canonical icon mapping:
  - `call` → phone (heroicon `phone`)
  - `meeting` → users (heroicon `users`)
  - `email` → envelope (heroicon `envelope`)
  - `note` → chat-bubble (heroicon `chat-bubble-left`)
  - `task` → clipboard-check (heroicon `clipboard-document-check`)
  - `letter` → envelope-open (heroicon `envelope-open`)
  - `sms` → chat-bubble-oval (heroicon `chat-bubble-oval-left`)
  - Unknown → generic tag icon + the raw type string.
  - For `call` / `meeting` types, append ` · {N} min` to the badge when `duration_minutes` is set.
- **Timestamp (rightmost):** occurred_at, relative format ("2pm today", "Tuesday", "Apr 3"). Same format as current Timeline.
- **Ellipsis menu:** existing edit/delete actions.
- **Body:** rendered below the header row as today — plain text, line breaks preserved. If `subject` is null (the scannable title came from body preview), the body still renders in full below so the reader sees everything.
- **Outcome (if set):** rendered below body, italic, muted colour (`var(--np-control-chip-text)`).
- **Follow-up pill (if set):** small pill below outcome.
  - Future `follow_up_at` AND status not `completed` → neutral tone pill, label `"Follow up: {relative date}"`.
  - Past `follow_up_at` AND status not `completed` → warning tone (amber/red border), label `"Overdue: {relative date}"`.
  - Past `follow_up_at` AND status is `completed` → don't render the pill.
- **Status badge:** hide when `status === 'completed'` (common case, reduces noise). Any other status → small muted badge next to the type badge.
- **Meta disclosure (if `meta` has any keys):** small collapsed row at the bottom of the card, label `"Source fields ▸"`; clicking expands a key/value list. Not editable. Purely informational.

### Activity-log one-liner

Not a card. Single horizontal line with:

- Small system icon (heroicon `cog-6-tooth` or `bolt`) on the left.
- `{description}` (e.g. `Email changed`, `Record created`).
- `by {actor}` (existing logic).
- Relative timestamp on the right.
- Muted colour throughout — visually subordinate to the note cards. No ellipsis menu, no interactive affordances beyond what exists today.

### Filter bar

Two filters in the page header, side by side:

- **Source filter (existing):** `All / Notes only / Activity only`.
- **Type filter (new):** dropdown, `All types / Call / Meeting / Email / Note / Task / Letter / SMS / More…`. The "More…" entry reveals any non-canonical types present in this contact's notes (pulled from a `distinct type` query). Type filter only applies to notes — when the Source filter is `Activity only`, the Type filter disables itself (greyed, tooltip: "Type filter applies to notes only").

Use a dropdown, not chips — chips would overflow in the narrow admin column once imported source-vocabulary types appear.

### Empty-state graceful fallback

A bare `type='note'` with no subject, outcome, follow-up, duration, or meta should render almost identically to today — just a note bubble with a date and a body. Structure appears only when fields are filled. This protects the quick-note-taking use case.
