# 056. Secure Public Signup Flows — Log

## What was built

### Phase 1 — Database & Model

**Migration `2026_03_21_220001_create_portal_accounts_table`** — creates `portal_accounts` with: `id` (bigint PK), `contact_id` (uuid FK → contacts, nullOnDelete), `email` (string, unique), `password` (string), `email_verified_at` (timestamp nullable), `remember_token`, timestamps. Contact FK uses `uuid` — matching the `contacts` table primary key type, correcting the bigint specified in the original prompt.

**`App\Models\PortalAccount`** — extends `Illuminate\Foundation\Auth\User as Authenticatable`, implements `MustVerifyEmail`. Uses the `MustVerifyEmail` trait and `Notifiable`. Password cast to `hashed`. `email_verified_at` cast to `datetime`. Belongs to `Contact` via `contact_id`. `sendEmailVerificationNotification()` overridden to send `PortalEmailVerification` mailable directly (bypasses the default notification).

### Phase 2 — Auth Guard

**`config/auth.php`** — added `portal` guard (`driver: session, provider: portal_accounts`) and `portal_accounts` provider (`driver: eloquent, model: PortalAccount`).

**`app/Http/Middleware/PortalAuthenticate.php`** — extends Laravel's `Authenticate` middleware. Overrides `redirectTo()` to point unauthenticated requests to `portal.login`. Overrides `authenticate()` to always use the `portal` guard, ignoring any others.

**`bootstrap/app.php`** — registered `portal.auth` as a middleware alias pointing to `PortalAuthenticate`.

### Phase 3 — Routes

Eight routes added to `routes/web.php` before the catch-all `/{slug}` pattern (critical — slug route uses `.*` and would otherwise swallow `/signup`, `/login`, etc.):

| method | URI | name |
|--------|-----|------|
| GET | `/signup` | `portal.signup` |
| POST | `/signup` | `portal.signup.post` |
| GET | `/login` | `portal.login` |
| POST | `/login` | `portal.login.post` |
| POST | `/logout` | `portal.logout` |
| GET | `/email/verify` | `portal.verification.notice` |
| GET | `/email/verify/{id}/{hash}` | `portal.verification.verify` |
| GET | `/account` | `portal.account` |

`/account` is protected by `portal.auth` + `verified:portal.verification.notice`. The verification routes are behind `portal.auth`. Login/signup POSTs are throttled at `10,1`.

### Phase 4 — Controllers

**`App\Http\Controllers\Portal\SignupController`**
- `GET /signup` — renders signup form.
- `POST /signup` — honeypot + timing check first (silent redirect on bot detection). Validates first_name, last_name, email, password (min 12, confirmed). If a portal account already exists for the email: silent redirect to verification notice — no error, no signal. Looks up Contact by email (using `withoutGlobalScopes()` to find contacts in pending import batches). If found: updates first_name and last_name from the form (form values always win). If not found: creates a new Contact with `source = 'member_signup'`. Creates `PortalAccount`, sends `PortalEmailVerification` mailable, logs in via `Auth::guard('portal')->login()`, redirects to `portal.verification.notice`.

**`App\Http\Controllers\Portal\LoginController`**
- `GET /login` — redirects to `/account` if already authenticated via portal guard.
- `POST /login` — `Auth::guard('portal')->attempt()`. On success: `session()->regenerate()`, redirect to intended or `/account`. On failure: identical error message ("Invalid email or password.") for both wrong email and wrong password — no enumeration.
- `POST /logout` — logs out, invalidates session, regenerates CSRF token, redirects to `/login`.

**`App\Http\Controllers\Portal\EmailVerificationController`**
- `GET /email/verify` — renders notice. Redirects to `/account` if already verified.
- `GET /email/verify/{id}/{hash}` — loads `PortalAccount` by id, validates `hash_equals(sha1($account->email), $hash)`, marks verified, redirects to `/account` with `verified` flash. The route is protected by the `signed` middleware ensuring the URL has not been tampered with.

### Phase 5 — Views

All views in `resources/views/portal/`, all extending `layouts.public`.

- `portal/signup.blade.php` — form with honeypot div + `_form_start` hidden input (matching event registration pattern). Error list at top. First name, last name, email, password, password confirmation.
- `portal/login.blade.php` — email, password, remember-me checkbox. Link to signup.
- `portal/verify-email.blade.php` — shows authenticated user's email address. Logout button (user is in limbo state: authenticated but unverified, cannot access protected content).
- `portal/account.blade.php` — placeholder. Shows `verified` flash on first visit post-verification. "Welcome, [first name]" heading. Logout button.

### Phase 6 — Email

**`App\Mail\PortalEmailVerification`** — accepts `PortalAccount`. Generates a `URL::temporarySignedRoute` to `portal.verification.verify` with 60-minute expiry. Subject includes site name from `SiteSetting`. Sends both HTML (`emails/portal-email-verification.blade.php`) and plain-text (`emails/portal-email-verification-text.blade.php`) parts.

The email template is **not** wired into the `EmailTemplate` system — it is a standalone mailable with a dedicated blade view. Making it editable in the system email editor is deferred to a future session.

### Additional — ADMIN_PATH env variable

`AdminPanelProvider` updated: `.path(env('ADMIN_PATH', 'admin'))`. The admin panel path is now configurable via `.env` without a code change. Default is `admin`. Noted: this is a deployment concern; runtime DB-driven path changes are not feasible (Filament builds routes at boot time).

## Security decisions

- **Duplicate portal account on signup** — no error, no confirmation, no email. Silent redirect to verification notice. Prevents external actors from probing whether an email address has a portal account.
- **Login error message** — identical for wrong email and wrong password: "Invalid email or password." No enumeration.
- **Verification URL** — `URL::temporarySignedRoute`, 60-minute expiry, validated with both `signed` middleware and `hash_equals(sha1($email), $hash)`.
- **Session fixation on login** — mitigated with `session()->regenerate()`. Note: not called on signup (low risk — session was just created, but worth tidying in a future pass).
- **Contact merge on signup** — form values always win for first_name and last_name when merging into an existing contact record. All other contact fields are left untouched.
- **Contact lookup bypasses import scope** — `withoutGlobalScopes()` used so contacts in pending import batches are found and not duplicated.

## Deferred

- Portal verification email not editable in system email editor — future session
- Session fixation hardening on signup path (minor, low risk)
- Rate limiting / lockout after N failed login attempts (current throttle is 10/min, no lockout)
- "Remember me" session persistence beyond standard session lifetime — explicitly out of scope
- Password reset — session 059
- Admin UI for viewing/revoking portal access from the Contact record — later session
- Portal security notification email (send email to existing account holder when someone attempts to register with their email) — deliberately omitted; no external signals about account existence
