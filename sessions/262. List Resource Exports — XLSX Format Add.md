# 262. List Resource Exports — XLSX Format Add

Add an operator-facing **Export Excel** admin action to every list resource that already got the 261 CSV + JSON export pass, mirroring the existing pattern. The shared [`ListExportService`](../app/Services/ListExportService.php) writer abstraction grows a third format flag (`'xlsx'`); per-page wireup adds one `Actions\Action::make('exportXlsx')` entry to the existing ellipsis ActionGroup. CF flattening matches the CSV shape (one column per `CustomFieldDef`).

This is **B2b'** from `release-plan.md` Track B, lifted from the Out-of-Gate register at 261 close.

---

## Design decisions (resolved before starting)

These are the agreed-going-in shape from session 261 close + the Out-of-Gate stub. Confirm at session start during the read-through.

- **Resource list — same 10 list pages 261 covered:**
  - `ListContacts`
  - `ListOrganizations`
  - `ListDonations`
  - `ListEvents`
  - `ViewRegistrations` (event-scoped)
  - `ListMemberships`
  - `ListTransactions`
  - `ListFunds`
  - `ListCampaigns`
  - `ListNotes`

- **Column shape per resource:** identical to 261. Every resource's `exportColumnSpec()` static method is the single source of truth; the XLSX branch consumes it verbatim.

- **CF flattening:** XLSX matches CSV exactly — one column per `CustomFieldDef::forModel($cfModelKey)` row, header is the CF's `label`. Not the JSON-style nested object.

- **Filename / MIME:**
  - `<resource>-YYYY-MM-DD.xlsx`
  - `Content-Type: application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`
  - For `ViewRegistrations`: `event-<id>-registrations-YYYY-MM-DD.xlsx` (event-scoped naming, parallel to CSV/JSON).

- **Action placement:** the third entry in each page's existing ellipsis ActionGroup. Order in the menu: CSV, JSON, Excel — chronological by introduction. Don't reshuffle 261's pattern.

- **Action label / icon:** `'Export Excel'` label (operator-facing language; matches what they expect to see). Icon: same `heroicon-o-arrow-down-tray` as CSV/JSON for visual consistency.

- **Permission gate:** matches the existing per-resource `view_any_<resource>` gate (Contacts: `view_any_contact`, Memberships: `view_any_membership`, etc.). Inherits the 261 pattern verbatim.

- **Architectural shape:** extend `ListExportService::stream()`. Format flag becomes `'csv' | 'json' | 'xlsx'`. The XLSX branch streams via a temp file (OpenSpout writes to disk, then `readfile` inside the `streamDownload` callback) — distinct from the CSV/JSON `php://output` streaming since OpenSpout cannot target stdout directly.

- **Library:** OpenSpout (`openspout/openspout`), already a transitive dependency via `filament/actions v3.3.49`. No Composer change.

- **Cell-type semantics — the headline decision to confirm at session start.** The 261 column specs return pre-stringified values from their value callables (`$contact->date_of_birth?->toDateString()` → `"1990-05-15"`, `$donation->amount` → string-cast decimal). With the existing spec passed verbatim to OpenSpout, every XLSX cell becomes a `StringCell` — Excel renders dates and numbers as text, losing native sortability, date pickers, and numeric formatting. The headline decision is which cell-type approach to ship:

  - **Option A — strings everywhere.** Pass column-spec value-callable returns directly to `Cell::fromValue()`; everything that's already a string stays a `StringCell`. Simplest. Loses Excel-native UX. Operators can re-format in Excel manually. Zero changes to 261's column specs.
  - **Option B — additive `'type'` hint per column-spec entry.** Each column-spec entry optionally declares `'type' => 'date' | 'datetime' | 'number' | 'boolean'`; the XLSX branch coerces the value-callable's return to the native type before passing to OpenSpout (Carbon parse for dates, `(float)` for numerics, `(bool)` for booleans). CSV path unchanged. Each Resource's `exportColumnSpec()` grows optional `'type'` keys on date/number/boolean columns. Most operator-friendly without breaking 261's CSV/JSON contract.
  - **Option C — invert the column-spec semantics.** Value-callables return native types (Carbon, float, bool); CSV stringifies on emit. Most invasive, breaks the 261 CSV contract, requires re-running the 20 page-level tests with new expected values.

  **Decision: Option B** — agreed at 262 prompt-write. Small additive change, preserves 261's byte-for-byte CSV/JSON output (tests pass unchanged), gives Excel users the experience they expect from a `.xlsx` file (native date sortability, numeric formatting, boolean rendering). Sanity-check the inventory of `'type'` hints needed across the 10 resources at session start, but the cell-type-semantics question itself is settled.

- **Out of scope:**
  - **JSON-Lines (`.jsonl`)** — still deferred from 261; not opened here.
  - **TSV / ODS / XML / Parquet** — same.
  - **Multi-sheet XLSX** — single sheet per export, named after the resource. No tabs-per-CF or tabs-per-relation experiments.
  - **Cell formatting beyond type detection** — no bold headers, no conditional formatting, no column widths, no frozen panes, no styles. The OpenSpout `Style` API exists but operator-pleasing visual polish is post-Beta-1 territory.
  - **Formulas** — no `=SUM(...)`, no totals row. Operators can add their own.
  - **Bulk-action exports** (per-row checkbox-driven export of selected rows). Operator filters via the existing table filters; no per-row selection needed.
  - **Importer-side adjustments.** The 261 mapper services already recognize the standard column shape; XLSX uses the same headers as CSV.
  - **Schema changes.** No new columns, no new migrations.

## Phase 2 — Extend the writer service

**Goal:** `App\Services\ListExportService::stream()` accepts `'xlsx'` as a third format value and emits a valid `.xlsx` `StreamedResponse`.

**Steps:**

1. Extend the `$format` parameter validation (or implicit dispatch) in `stream()` to recognize `'xlsx'`. Existing CSV/JSON branches remain unchanged.
2. Add a private `streamXlsx(Builder $query, array $columnSpec, Collection $customDefs): void` method parallel to the existing `streamCsv` and `streamJson`. Implementation shape:
   - `tempnam(sys_get_temp_dir(), 'xlsx-')` for the writer's target file.
   - **Wrap the entire write-and-stream sequence in `try { ... } finally { @unlink($tempPath); }`.** This is load-bearing: OpenSpout's `Writer` cannot target `php://output`, so the temp file is the only output sink, and any exception between `openToFile` and `readfile` (database disconnect mid-iteration, OOB kill, malformed Carbon, Storage error) leaves the file behind. The `finally` block guarantees cleanup runs even when control leaves the function abnormally. Use `@unlink` (silenced) so a missing-file at finally-time (e.g. `tempnam` failed and threw before assigning) doesn't itself raise. Repeat: this is a hard requirement, not an optional polish.
   - `Writer` from `OpenSpout\Writer\XLSX\Writer`. `openToFile($tempPath)`.
   - Header row: `Row::fromValues(array_merge(array_column($columnSpec, 'header'), $customDefs->pluck('label')->toArray()))`. `addRow($row)`.
   - Per-row body via `$query->each(...)` (same chunking as CSV). Each row builds a flat values array — column-spec values per column, then CF values per CF def. If Option B lands, the value-coercion-by-`'type'`-hint happens here before `Cell::fromValue()` is called inside `Row::fromValues`.
   - `close()` to finalize the workbook (must run before `readfile` — OpenSpout's footer XML is written on close).
   - `readfile($tempPath)` to stream to client.
   - `@unlink($tempPath)` runs in the `finally` block.
3. Update the Content-Type header dispatch in `stream()` to emit `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet` for `'xlsx'`.
4. If Option B is chosen: add a small private helper `coerceForXlsx(mixed $value, ?string $type): bool|float|int|string|\DateTimeInterface|null` that takes the column-spec value-callable's raw return and the optional `'type'` hint, returns the OpenSpout-native form. Used only inside `streamXlsx`; CSV path unchanged.
5. Add unit tests at `tests/Feature/ListExportServiceTest.php` covering: XLSX response shape (status / content-type / filename); OpenSpout `Reader` round-trip parses cleanly; CF flattening one-column-per-CFDef; empty-query behavior (header-only XLSX); cell-type semantics per the chosen option (StringCell-only for A; native types per `'type'` hint for B). Estimate ~5–7 tests.

6. **Cleanup-on-exception test (load-bearing, not optional).** Add a service-level test asserting the temp file is removed even when row iteration throws mid-stream. Inject a column-spec entry whose value-callable throws on the second row, snapshot `glob(sys_get_temp_dir() . '/xlsx-*')` before and after the (caught) export call, assert no `xlsx-*` files persist past the exception. This is the test that protects against the disk-leak scenario the user has flagged historically (the resolved media-removal incident is in this same family); without it, future refactors of the streaming shape can silently regress cleanup.

**Pass criterion:** Existing CSV + JSON behavior unchanged (no regression on the 38 tests 261 added). New XLSX tests pass. The OpenSpout `Reader` can round-trip every emitted file.

## Phase 3 — Wire actions into each ListXxx page

**Goal:** All 10 list pages get `Actions\Action::make('exportXlsx')` added to the existing ellipsis ActionGroup.

**Per-page change pattern** (mechanical, mirrors 261):

```php
Actions\Action::make('exportXlsx')
    ->label('Export Excel')
    ->icon('heroicon-o-arrow-down-tray')
    ->hidden(fn () => ! auth()->user()?->can('view_any_X'))
    ->action(function (HasTable $livewire): StreamedResponse {
        return app(ListExportService::class)->stream(
            query: $livewire->getFilteredSortedTableQuery()->orderBy('created_at'),
            columnSpec: XResource::exportColumnSpec(),
            format: 'xlsx',
            filename: 'X-' . now()->format('Y-m-d') . '.xlsx',
            cfModelKey: 'X' or null,
        );
    }),
```

For `ViewRegistrations`: filename pattern is `event-{$eventId}-registrations-...xlsx`, column spec is `EventResource::registrationExportColumnSpec()`, CF model key is `'event_registration'` — same shape as the CSV/JSON actions there.

If Option B landed in Phase 2, also touch each Resource's `exportColumnSpec()` (and `registrationExportColumnSpec()` for events) to add `'type'` hints on date / number / boolean columns. This is the bulk of the per-resource work in Option B; Option A skips it entirely.

**Pass criterion:** Manual smoke test on each of the 10 list pages confirms the ellipsis menu shows three export actions in the order CSV → JSON → Excel, and clicking Excel downloads a valid `.xlsx` that opens in Excel / LibreOffice / Google Sheets cleanly.

## Security checklist

Before closing:

- [ ] Permission gate enforced on every XLSX export action — anonymous users + role-mismatched users get the action hidden, identical to the 261 CSV/JSON gating. (Inherited verbatim from 261; verify the new third action carries the same `->hidden(fn () => ! auth()->user()?->can('view_any_X'))` callback.)
- [ ] No PII leakage in XLSX: confirm sensitive fields (e.g. internal-only contact notes, admin-only audit columns) don't appear in the XLSX shape unless they're in the importer's `flatFields()` output. Same scope as 261 since the column spec is identical.
- [ ] Custom-field flattening doesn't expose unintended fields — CFs scoped to one model don't leak into another's export via shared `CustomFieldDef::forModel` lookups.
- [ ] EventRegistration export is event-scoped — a user viewing event A's registrations cannot URL-fish their way to event B's registrations XLSX.
- [ ] Filter+sort state in the URL is honored on the server side. Same `getFilteredSortedTableQuery()` plumbing as 261.
- [ ] Temp-file cleanup is `try/finally`-wrapped per Phase 2 step 2 (load-bearing requirement), and the cleanup-on-exception test from Phase 2 step 6 is in the suite. Verified by running the new test in isolation and confirming no stragglers in `sys_get_temp_dir()` post-suite.
