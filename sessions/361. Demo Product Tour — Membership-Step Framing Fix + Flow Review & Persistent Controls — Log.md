## What was built

### Phase 1 — Membership-step framing fix (root cause found; fixed cleanly)

- **Reproduced the misposition objectively with Playwright** under CPU throttling (bounding-box measurement, not eyeballing): the membership step's spotlight landed on an element that had scrolled below the fold, popover stranded at the viewport bottom.
- **Real root cause (the key finding):** the step anchored on the low **membership-status summary widget**, and the important content — the **Memberships relation-manager table** — sits higher up but is **lazy** (`CanBeLazy::$isLazy = true` by default). Filament defers it, so it loads *after* the rest of the record paints; that late pop-in is the layout shift that threw driver.js's one-shot positioning off, and it's also why the marker wasn't in the DOM when the tour looked.
- **Fix:** made `MembershipsRelationManager` **eager** (`$isLazy = false`) and **re-anchored the membership step onto that table** via an owned `data-tour="record.memberships"` marker, injected with a `RESOURCE_RELATION_MANAGER_BEFORE` render hook **scoped to that one relation manager** (not the Notes manager, and not a `.fi-*` class — the 249/338 selector-fragility rule). Removed the now-dead `data-tour="record.membership"` from the status-widget cell.
- **Added `waitForStableRect`** to `anchors.js` — a settle-before-position primitive (poll `getBoundingClientRect()` until stable) so the driver positions against final geometry. With a high, stable, eager target this is sufficient; a **reactive scroll-chasing "watcher" was built and discarded** (it fought fast advances and was fragile — the wrong tool; the anchor, not the layout, was the problem).
- **Verified:** the membership step now spotlights the real Memberships table (tier/status/dates/edit), centered and in view, popover in view — under 2× and 4× CPU throttle.

### Phase 1 prerequisite — 2FA e2e auth regression (found + fixed)

- Discovered that **session 359's mandatory-2FA enforcement had silently broken the entire admin Playwright suite**: the gate redirects every authenticated admin request to `/admin/two-factor-setup`, and the isolated e2e stack runs as env **`dev`**, which 2FA does not exempt (only `demo` / `testing` do). The real demo node is unaffected; Playwright CI has been red since 359.
- **Fixed the e2e auth helper:** `tests/e2e/helpers/db.ts` now enrolls the seeded admin in 2FA with a fixed recovery code on every reset; `helpers/auth.ts` clears the challenge with it (recovery code — no TOTP time-window). Restores the whole admin e2e suite.

### Phase 2 — Persistent controls + flow review (built; superseded by the pivot)

- **Persistent control bar** (`Exit · ← Back · Step X of Y · Next →`) in `controller.js`, wired to extracted `advanceTour` / `retreatTour` / `stopTour` paths shared with the popover buttons (single source of truth); mounts on drive, torn down with the driver, removed on exit. Verified driving all 11 steps including the two interactive ones.
- **Bar/popover overlap resolved spatially, not with a z-index war** (an important lesson — the owner flagged an escalation to `1e9+1` as the wrong approach): driver's overlay is z `10000`, its popover z `1000000000`; the bar sits at **`10001`** — above the overlay so it's clickable, below the popover so the popover's own buttons win where they overlap and the bar is the visible fallback elsewhere.
- **Playwright storyboard** (screenshot per step) produced for flow review; surfaced that step 7 (Events) copy instructed actions the non-interactive tour doesn't perform. Flow/copy rework was **not** applied — the owner pivoted the whole architecture before copy edits.
- Standing spec `tests/e2e/tour/product-tour.spec.ts` kept green at **4/4** throughout (2FA fix + framing fix + bar). Throwaway repro/storyboard specs torn down at close per Playwright hygiene.
