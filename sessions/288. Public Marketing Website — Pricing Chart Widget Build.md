## Decisions resolved at planning time

Four decisions were settled during 287-close planning. They are no longer open questions for the agent; build directly to them.

- **Columns are an in-widget config repeater, not a collection.** Per-tier data lives attached to the widget instance, edited in the inspector. Rationale: tiers are widget-specific, not customer-managed indefinite-N content. This requires the system to support a generic "repeater of objects" field type (the only existing repeater is the specialized `buttons` field on `hero` / `text_block`) — a small system extension is part of this session's scope. Surface the extension's shape in the work log; it's load-bearing for future widgets that want similar per-instance structured arrays.
- **Per-column "Emphasize this column" toggle**, not a widget-level emphasized_column_index. Each column's repeater entry carries its own `emphasize` boolean (default false). Multiple columns may be emphasized simultaneously — rare usage, but supported and simpler to implement than mutual-exclusion logic. The "Recommended" / "Most popular" eyebrow text remains a separate per-column `eyebrow` field; the two are independent (a column can be emphasized without an eyebrow, or carry an eyebrow without emphasis).
- **Cross-card row alignment via CSS subgrid.** The widget renders cards as a parent CSS Grid (one column per pricing tier); each card is itself a subgrid that inherits the parent's row tracks. Result: the Nth row of each card lines up with the Nth row of the others automatically; cards with extra rows extend below the others; CTAs land at consistent vertical positions when card content lengths match and at honest-extra-tall positions when they don't. No widget-side label-union computation, no blank-cell logic. Customer convention: type attribute rows in matching order across cards. Browser support is fine for the 2026 baseline (Chrome 117+, Firefox 71+, Safari 16+). Mobile: parent grid switches to `1fr` at the relevant breakpoint, cards stack as block elements, subgrid context collapses naturally.
- **Ship one v1 preset: "Marketing site tiers."** Three columns matching the Pricing page's Band 2 configuration verbatim (Instant Demo, 7-Day Trial, Monthly — with Monthly emphasized and carrying the "Recommended" eyebrow). The preset doubles as session 289's one-click Band 2 setup AND as the agent's correctness check that the subgrid alignment, the emphasis treatment, and the columns repeater all work end-to-end against a realistic configuration. Shape and content match the pricing-page spec's Band 2 section verbatim — column titles, prices, attribute rows, CTAs, eyebrow.

## Open questions to resolve at session start

These are smaller surfaces the agent should think about briefly before committing, but they aren't blockers — defaults below are reasonable.

- **Repeater field type implementation shape.** The system has no generic "repeater of objects" field type today — `buttons` is the only existing repeater and it's specialized. The columns repeater needs a new field type (call it `repeater` or `object_list` or similar). Surface the field type's shape (config schema, Vue inspector component, value validation) before building it; this is the load-bearing system extension this session adds. Sub-question: does the new field type also handle the per-column attribute_rows nested repeater, or is nesting a v2 concern (and v1 stores attribute_rows as a structured rich-text or similar)? The v2 path would mean more config-time thinking but cleaner per-row schema. Pick at session start with rationale in the work log.
- **CTAs per column — reuse the existing `buttons` field type at the column level, or roll a smaller per-column CTA shape.** The existing `buttons` field handles arrays of `{text, url, style}` objects cleanly and the inspector already renders it. Reusing it inside the new repeater field type is the cleaner path if the new repeater can compose it; otherwise a per-column `cta_text` / `cta_url` / `cta_style` field triplet is the workable fallback.
- **Whether to update `widget_types` schema or hold extensions to `config_schema` JSON.** The new `repeater` field type lives in the config_schema JSON and the Vue inspector's component map — no `widget_types` table change needed. Confirm there's nothing that needs a column-level addition.
