# 119. Widget Picker — Categories, Page-Type Filtering & Search

Add `category` and `allowed_page_types` columns to `widget_types`, categorise all existing widgets, filter the picker modal by page type, group widgets by category, and add an Alpine.js title filter to the picker header.

---

## Design decisions (resolved before starting)

- **`category`** is a required string on `widget_types`. Values: `content`, `layout`, `media`, `portal`, `commerce`, `form`, `data`. This groups widgets visually in the picker — nothing more.
- **`allowed_page_types`** is a nullable JSON array on `widget_types`. Values are drawn from the `pages.type` column: `default`, `post`, `member`, `system`. `null` means "allowed everywhere" — this is the default for most widgets. Only widgets that genuinely cannot function outside a specific page type get a restriction. Widgets that are merely unusual on a page type (e.g. a donation form on a blog post) are **not** restricted — the system does not enforce taste.
- The widget picker modal groups widgets into collapsible `<x-filament::section>` blocks, one per category. Widgets whose `allowed_page_types` excludes the current page's type are simply omitted from the modal — no disabled state, no explanation.
- The Alpine.js title filter is a text input in the modal header. It filters widget buttons by label, client-side only. Category sections with no visible widgets after filtering are hidden.
- **Widget contract principle:** every widget interacts with the system through the same columns and the same picker logic. No special-casing. If a new widget needs a behaviour the schema doesn't support, the schema is extended for all widgets — not bypassed for one.

## Phase 3 — Filter picker by page type

### 3a. PageBuilder.php

In `mount()`, after loading `widgetTypes`, filter the array: if a widget's `allowed_page_types` is not null, it must contain the current page's `type`. Also pass the page type to the view so the child block picker can filter too.

Store the page type as a public property so the blade template has access.

### 3b. PageBuilderBlock.php

Apply the same `allowed_page_types` filter in the child widget picker. The page type must be available from the parent — investigate how the block component accesses the parent page. If it doesn't currently, pass it as a mount parameter.

## Phase 4 — Group picker by category

### 4a. Root picker modal (page-builder.blade.php)

Replace the flat grid of widget buttons with grouped sections:

```
@foreach ($widgetsByCategory as $category => $widgets)
    <x-filament::section :heading="$categoryLabel" collapsible>
        <!-- same button grid, scoped to this category -->
    </x-filament::section>
@endforeach
```

Group labels should be title-cased: "Content", "Layout", "Media", "Portal", "Commerce", "Form", "Data". Empty groups (after page-type filtering) are hidden.

### 4b. Child picker modal (page-builder-block.blade.php)

Apply the same grouped layout to the child widget picker modal inside column widgets.

## Phase 5 — Alpine.js title filter

Add a text input at the top of both picker modals (root and child):

- `x-data="{ filter: '' }"` on the modal wrapper
- `<input x-model="filter" placeholder="Filter widgets…">` in the modal header
- Each widget button gets `x-show="'{{ strtolower($widget['label']) }}'.includes(filter.toLowerCase())"`
- Each category section gets `x-show` that checks if any child button is visible (use a ref or count approach)
- Clearing the input (or Escape) resets the filter

Style the filter input consistently with the existing modal UI.
