# 249. Onboarding/Install Dashboard Widget

A super-admin-gated dashboard widget that surfaces a checklist of setup items the operator works through to bring a fresh install to a usable state, deep-linking into existing admin pages for each item. The widget doubles as an ongoing health-check surface post-completion. Pre-Beta-1 entry E1 in `sessions/release-plan.md`.

---

## Design decisions (resolved before starting)

- **Shape: checklist with deep links, not a wizard or DOM-anchored tour.** Each item carries: title, one-line description, status pill (done / incomplete / warning), category band (required to boot / required for specific features / optional), and a "Configure →" button that routes to the existing admin page where the item lives. The widget itself is the guide — there is no overlay, no tour library, no separate wizard sequence. Each integration's per-page setup affordances (Stripe Connect OAuth, Resend domain verification, etc.) live inside their own admin pages and are not in scope for this session.
- **Two display modes, same widget.** "First-run mode" — full-prominence, top-of-grid, every item visible — when `site_settings.installation_completed_at` is null. "Health-check mode" — compact, items default-collapsed, surfaces only items with non-green status — when the flag is set. Operator flips between modes with two affordances inside the widget: "Mark setup complete" (sets the flag to `now()`) and "Reset install state" (nulls the flag, behind a confirmation). The reset is intentionally lighter than `app:reset` — that wipes the database; this just nulls one flag so the operator can re-walk the checklist.
- **Required vs optional categorization (initial set, refine during Phase 1's inventory pass):**

  | Required to boot at all | Required for specific features | Optional |
  |---|---|---|
  | Admin user (≥1 user with super-admin role) | Tax-receipt info (EIN, 501c3 status, return address) — for any donation work | Logo |
  | Organization legal + display name | DKIM/SPF/DMARC DNS verification — for mail at any volume | Theme colors |
  | URL / primary domain | Default fund — for donation imports | Stripe / QuickBooks / Resend / Mailchimp keys |
  | Timezone | | Custom fields |
  | Mail "from" address | | Import data |

  The middle column is the right home for *most* items — the left column is the truly-can't-boot set.

- **Source of truth: a `SetupChecklist` service, not data on the widget.** The service returns an array of items, each with `key`, `title`, `description`, `category`, `status`, `configure_url`, `message?`. The widget renders what the service returns. Service stays decoupled so future surfaces (CLI command, public health endpoint, FM-side polling, audit reports) can read the same state without re-implementing detection.
- **Per-item status detection — deliberately simple checks**, not deep validation. Examples: "admin user exists" = `User::where('role', 'super_admin')->exists()`; "default fund exists" = `Fund::exists()`; "Stripe configured" = `! empty(config('services.stripe.key'))`. Status interpretation:
  - `done` — the configured-detection check passes.
  - `incomplete` — the check fails AND the item is in the required-to-boot or required-for-feature category.
  - `optional` — the check fails AND the item is in the optional category. (Rendered as a soft hint, not a warning.)
  - `warning` — a configured item is in a known-bad state (e.g., Stripe is configured against a `sk_test_` key, or DKIM was verified once and the DNS record is now missing). Warning detection is opportunistic — only land checks for cases you can detect cleanly. Don't gold-plate.
- **No DOM-anchored tour library**. driver.js / shepherd / introjs / custom overlay are all out of scope. The cost (Livewire DOM-coupling, stable selectors, navigation-mid-tour edge cases, ongoing maintenance against Filament version bumps) does not earn its weight against the lighter checklist-with-deep-links shape.
- **No new admin surfaces this session unless an inventory gap forces one.** The widget links to existing admin pages. If Phase 1's inventory shows an item lacks an admin surface (e.g., mail "from" address might live only in `.env` today), the checklist item describes env-var status and tells the operator to set on the droplet. Building a new admin page lands as its own session if needed.
- **First-run flag is operator-flippable in this session; FM-side automation is post-A2 concern.** The widget has the "Mark setup complete" button to flip the flag manually. When A2's FM-side install path lands, FM can flip the flag via whatever mechanism A2 chooses (additive contract endpoint, SSH, cloud-init, etc.) — out of scope for E1.
- **Per-widget outline CSS change is in scope as a small lift.** Currently the dashboard widget grid is wrapped in a single outline; each individual widget should be wrapped in its own outlined container instead. SCSS-only change.

## Phase 1 — Inventory pass

Before writing any widget code, walk each candidate checklist item and document where it lives.

For each item, capture:
- **Configured-detection check** — the smallest possible code expression that returns true when the item is configured.
- **Configure URL** — the existing admin page that lets the operator change the item. If none exists, note "no admin surface; lives in `.env`" and skip building one (defer to a later session).
- **Description** — one-line operator-facing copy.
- **Category** — required-to-boot / required-for-feature / optional, per the table in Design Decisions (refine if inventory surfaces something).

Output: a matrix at the top of the session log (or as a comment in the service class — operator's call) that drives Phase 2's checklist content.

Likely-existing surfaces to verify (not exhaustive — the inventory may surface more or fewer):

- Admin users: Filament's user resource (likely `app/Filament/Resources/UserResource/`).
- Organization name + URL + timezone: `SiteSetting` admin page (find via `grep -r "SiteSetting" app/Filament/`).
- Mail "from" address: may be `MAIL_FROM_ADDRESS` env-var only — confirm.
- Tax-receipt info: `resources/docs/generate-tax-receipts.md` already exists; the admin surface backing it should too.
- Default fund: `app/Filament/Resources/FundResource/` likely.
- Logo + theme colors: Design System editor (per memory; lives under Tools group).
- Integrations: each may or may not have an admin page today — Stripe likely does, QuickBooks likely does, Resend may not, Mailchimp may not. Document per-integration shape.

Pause and surface to the user if the inventory turns up significant gaps (e.g., 3+ items lack any admin surface) — that changes scope.

## Phase 3 — `SetupChecklist` service

- New class `app/Services/Setup/SetupChecklist.php`.
- Constructor takes the dependencies it needs (typically just `Illuminate\Contracts\Config\Repository`).
- Public method `items(): array` returns an ordered array of checklist items. Shape per item:
  ```
  [
      'key' => 'admin_user',
      'title' => 'Create at least one admin user',
      'description' => 'You need a super-admin login to manage the CRM.',
      'category' => 'required_to_boot',
      'status' => 'done|incomplete|optional|warning',
      'configure_url' => '/admin/users',
      'message' => null,  // populated for warning state
  ]
  ```
- Private method per item — `checkAdminUser()`, `checkOrgName()`, `checkTimezone()`, etc. Each returns the per-item array.
- Items are ordered: required-to-boot first, then required-for-feature, then optional.
- The service is super-admin-only at the *consumer* layer (the widget); the service itself has no auth check (any caller has its results — defense in depth lives at the surface).

## Phase 4 — Dashboard widget

- New widget under `app/Filament/Widgets/SetupChecklistWidget.php` (or whatever directory the Random Data Generator widget lives in — match the convention).
- Widget gating: same super-admin gate the Random Data Generator uses.
- Two render modes:
  - **First-run mode** when `installation_completed_at` is null: full checklist visible, prominent at top of grid, no collapse.
  - **Health-check mode** when set: compact rendering, default-collapsed, lists only items with `status != done`.
- Blade view at `resources/views/filament/widgets/setup-checklist.blade.php`.
- View renders three sections (one per category band) with a section heading. Each item shows status pill + title + description + "Configure →" button that opens the `configure_url` in the same tab.
- "Mark setup complete" button at widget bottom (only visible in first-run mode) — Filament action that updates `site_settings.installation_completed_at = now()`.
- "Reset install state" button at widget bottom (only visible in health-check mode) — Filament action with confirmation, nulls the flag.

## Manual testing

Visual + flow:

1. **First-run mode:** with `installation_completed_at` null, log in as super-admin → dashboard shows the widget prominently at top of grid → all items rendered, grouped by category band, status pills accurate against the install's actual state.
2. **Mark complete flow:** click "Mark setup complete" → flag is set, widget collapses into health-check mode → only non-green items remain visible.
3. **Reset flow:** click "Reset install state" → confirmation dialog → confirm → flag nulls, widget returns to first-run mode.
4. **Deep-link flow:** click "Configure →" on the org-name item → routes to the SiteSetting admin page; check the same for two other items (default fund, theme colors).
5. **Per-widget outline:** dashboard renders with each widget in its own outlined container, no outer outline around the whole grid. Compare against pre-change screenshot if possible.
6. **Non-super-admin spot check:** log in as a non-super-admin and confirm the widget is not visible.

The visual outcomes (per-widget outline shape, widget prominence in first-run mode, status-pill clarity) are user-judgment surfaces — pull the user in for those after the objective outcomes (mode-switching, deep-link routing, gating) are confirmed.

## Security checklist

Before closing:

- [ ] Widget is super-admin gated. Non-super-admins cannot see or interact with it.
- [ ] "Mark setup complete" + "Reset install state" actions are also super-admin-gated at the action layer (defense in depth — not just at the widget visibility layer).
- [ ] `SetupChecklist` service does not leak sensitive values (no echoing of API keys, env-var values, etc.) — status flags only.
- [ ] No checklist item's `message` field carries internal paths, stack traces, or secrets — same discipline as the agent contract's subcheck `message` field.
- [ ] The new `installation_completed_at` setting is not portal-readable or otherwise exposed outside the admin surface.

---

## Out of scope

- **DOM-anchored tour libraries** (driver.js / shepherd / intro.js / custom overlay). The checklist-with-deep-links *is* the guide; no overlay layer.
- **Separate wizard pages** that duplicate fields already present in existing admin pages. Each item links to its existing admin home; if an admin home is missing, the checklist item describes the env-var or surfaces the gap.
- **B1 Organizations Model Overhaul fixes**. The checklist may surface "organization model is incomplete" as a known limitation, but does not author the fix.
- **Importer rework / Magellan-style guided import.** The "Import data" checklist item deep-links into the existing importer, which already has good in-page help.
- **Per-integration setup wizards (Stripe Connect OAuth, Resend domain verification, etc.).** Those live inside each integration's own admin page; this session does not author them. The checklist surfaces status only.
- **Admin user provisioning UI / wizard.** Admin users are still provisioned via `DatabaseSeeder` against env vars on first install; no admin-create flow lands here.
- **FM-side automated flag-flipping.** Post-A2 concern — FM's install path may flip `installation_completed_at` automatically once A2 lands; out of scope for E1.
- **Stripe Test-Mode Detection** (the separate release-blocker stub from session 245). Lands as its own session.
- **Custom field configuration walkthroughs.** "Custom fields" is one of the optional checklist items; the deep-link routes to the existing custom-fields admin page.
