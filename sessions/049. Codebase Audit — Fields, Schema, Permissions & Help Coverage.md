# 049. Codebase Audit — Fields, Schema, Permissions & Help Coverage

A repeat of session 042. Enough has been added since that audit — web forms, form submissions, the public frontend build system, site theming, importer staged updates, and the JSON import round-trip — to warrant a clean sweep. No new features. The output of this session is corrections and stubs only.

---

## Before writing any code

Read the following files:

- Every file in `app/Filament/Resources/` — all resource edit forms
- Every file in `app/Filament/Pages/` — all custom admin pages
- Every file in `app/Models/` — all models, their `$fillable`, `$casts`, and soft delete status
- Every migration added since session 042 (`database/migrations/` — sort by date, focus on anything after `2026_03_20`)
- Every file in `resources/docs/` — all help articles and their registered routes
- `vite.config.js`, `package.json`, `resources/js/public.js`, and `resources/scss/` — the public frontend build

Take notes as you read. Cross-reference against the session 042 log to avoid re-auditing what was already resolved.

## Workstream 1 — Edit page field audit

Same methodology as session 042. For every Filament resource and admin page that has an edit form, compare the fields shown against the database columns and the model's `$fillable`.

Focus on what is new since session 042:

- `Form` model and `FormResource` — form builder fields, settings JSON, is_active flag
- `FormSubmission` model and `FormSubmissionsRelationManager` — data, ip_address, contact_id
- `ImportStagedUpdate` model (added in session 044) — check its resource or relation manager if one exists
- `SiteTheme` or any model introduced in session 046 — confirm all columns are reachable or deliberately omitted

For each discrepancy: deliberate omission (document with a comment) or gap to fix.

## Workstream 4 — Soft deletes consistency

Produce the same model table as session 042, updated to include new models. Specifically decide:

- `Form` — submitted forms accumulate a history; accidental deletion of a form definition would be hard to recover. Recommend soft delete. Implement if agreed.
- `FormSubmission` — submissions are audit records; hard delete loses data permanently. Recommend soft delete. Implement if agreed.
- `ImportStagedUpdate` — review and decide.

Document the decision for each with a one-line justification.

## Workstream 5 — Fillable vs. casts consistency

Scan all models added or modified since session 042. Every column in `$fillable` that is a date, boolean, or JSON/array type must appear in `$casts`. Focus on:

- `Form` — `fields` and `settings` should cast to `array`; `is_active` to `boolean`
- `FormSubmission` — `data` should cast to `array`; `created_at` needs the `UPDATED_AT = null` pattern confirmed
- `ImportStagedUpdate` — check whatever JSON or boolean columns it has

## Workstream 6 — Foreign key index audit

Review all migrations added since `2026_03_20`. For every `foreignId` / `foreignUuid` / `foreign()` call, confirm there is an explicit index on the column (PostgreSQL does not auto-index FK columns). New tables to check:

- `form_submissions.form_id` — index present?
- `form_submissions.contact_id` — index present?

Write a migration for any missing indexes.

List every Eloquent model in `app/Models/`. Note whether a factory exists. Models added since session 042 that likely need factories:

Create minimal factories for any model that lacks one and could reasonably appear in tests.

## Workstream 9 — Frontend build system audit

This workstream is new — the public frontend build (Vite, SCSS, Alpine.js) was introduced in session 045 and has not been audited.

**Build output in version control**: confirm that compiled files (`public/build/`, `public/css/`, `public/js/`) are covered by `.gitignore` and are not checked into the repository. If any compiled output is tracked, remove it and add the correct ignore rules.

**Dead imports and unused variables**: read `resources/scss/public.scss`, `_theme.scss`, and `_layout.scss`. Flag any `@use`, `@import`, or variable declarations that are never referenced. Remove or comment them.

**`public.js` review**: read `resources/js/public.js`. Confirm all imports and registered behaviours (Alpine components, event listeners) correspond to elements that still exist in the public templates. Flag anything that looks orphaned.

**Build health check**: run `npm run build` inside the container and confirm it exits cleanly with no warnings. Record the output. If warnings are present, assess whether they need fixing.
