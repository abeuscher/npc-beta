## What was built

### Phase 1 — Polymorphic `import_staged_updates`

- Migration `2026_04_18_120000_polymorphic_subject_on_import_staged_updates.php` adds `subject_type` (string) and `subject_id` (uuid) columns, backfills existing rows as `subject_type='App\Models\Contact'` + `subject_id=contact_id`, drops the `contact_id` column and its FK/index, and indexes `(subject_type, subject_id)`. Reversible `down()` recreates `contact_id`, backfills from contact-typed subjects, and drops the polymorphic columns.
- `ImportStagedUpdate` model now exposes a `subject()` `morphTo` relation; removed the `contact()` belongsTo.
- Updated contact call sites: `ImportProgressPage::processRow()` writes `subject_type => Contact::class` instead of `contact_id`; `ImportContactsCollisionTest`, `ImportContactsMatchKeyTest`, `ImportReviewTest` updated to query by `subject_type` + `subject_id`; `import-review-preview.blade.php` reads `$update->subject` instead of `$update->contact`.
- `docs/schema/import_staged_updates.md` rewritten to reflect the polymorphic shape.

### Phase 2 — Wizard strategy picker on non-contact importers

- Shared helper `duplicateStrategyRadio($entityLabel)` added to `InteractsWithImportWizard`: a two-option Radio (`skip` / `update`) with per-entity copy.
- `ImportEventsPage`, `ImportDonationsPage`, `ImportMembershipsPage`, and `ImportInvoiceDetailsPage` each render the Radio after the match-key controls on the Map Columns step.
- Each page's `createSessionAndLog()` call passes `duplicate_strategy` from form state into `ImportLog::duplicate_strategy` (default `skip`).

### Phase 3 — Progress trait staging path + per-page branching

- `InteractsWithImportProgress` trait gained a `stageSubjectUpdate(Model $subject, array $attributes): void` helper that writes a polymorphic `ImportStagedUpdate` row scoped to `$this->importSessionId`, filtering out null attributes.
- `buildRowContext()` on each of the four non-contact progress pages now includes `duplicateStrategy` from `$log->duplicate_strategy`.
- **Events**: matched Event + `update` strategy stages non-blank `$eventAttrs` (with parsed dates and merged custom_fields), returns `'updated'`, and skips registration/transaction creation for that row. Default `skip` strategy preserves today's reuse-event / create-registration behaviour.
- **Donations**: introduces a match-by-`external_id` lookup when both a mapped `donation:external_id` value and the import source are present. Matched + `skip` returns `'skipped'` with `skipReason='duplicate_skipped'`; matched + `update` stages attrs and returns `'updated'`; no match falls through to today's create path.
- **Memberships**: introduces match-by-`external_id`, and a composite fallback `(contact_id + tier_id + starts_on)` when `external_id` is absent. Same skip/update/create branching as donations.
- **Invoice Details**: `processInvoiceGroup()` honours the strategy at the group level. `update` stages non-blank group attrs (line_items, payment_method, status, merged custom_fields) against the existing transaction. Default `skip` preserves today's fill-blanks-only enrichment. New transactions are created unchanged.
- Each page's `accumulateOutcome()` updated to recognise the `'updated'` outcome and increment new `would_update` entity counters.
- `InteractsWithImportProgress::tick()` already increments `$this->updated` for outcome `'updated'`; no change needed there.

### Phase 4 — Polymorphic approval path

- `ImporterPage::approve` action replaces the contacts-only staged-update loop with a polymorphic one. A private helper `resolveStagedSubject(ImportStagedUpdate)` returns the underlying model: `class::query()` + `withTrashed()` for soft-delete models + `withoutGlobalScopes()` for Contact. Approval fills and saves attributes on any subject type. Contacts-only side effects (tag sync, approval Note creation) are gated on `$subject instanceof Contact`.
- `rollback` action applies the same helper for its discarded-staged-changes Note, also Contact-gated. Non-contact rollback paths continue to delete the session and let cascade-on-delete clear the staged rows.
- Preview modal (`ImporterPage::actions[]`) eager-loads `with('subject')` instead of `with('contact')`.

### Phase 5 — Blade progress-page polish

- All four non-contact progress blades (`import-events-progress`, `import-donations-progress`, `import-memberships-progress`, `import-invoice-details-progress`) expanded from 3-up to 4-up grids in dry-run summaries (added "Would update") and committing/done grids (added "Staged"). Invoice-details done grid also got the Staged cell; its committing view shows only a spinner today and was left as-is.

## Deferred decisions

- **Event Registrations update path.** Re-import semantics for registrations are ambiguous (matching by event+contact pair vs. including ticket tier / registered-at), and this session explicitly deferred it. A follow-up session will need to decide whether registrations are write-once or gain their own match+stage path once the semantics are agreed with a client use case.
- **Financial-audit guardrails.** Fields like `payment_status` on a receipted donation are mutable via `update` strategy today. No guardrails added. If client testing surfaces this, a follow-up session can add a per-field immutability list.
- **Progress-bar batched-poll bug.** Diagnosed during UAT but intentionally out of scope for this session. Root cause: `wire:poll.500ms` on a slow tick queues missed polls into a single follow-up Livewire request, causing the server to run `tick()` N times back-to-back and skip intermediate UI updates. Session 195 (Playwright) will lock in regression coverage; a separate follow-up is needed to address the transport bug itself.
