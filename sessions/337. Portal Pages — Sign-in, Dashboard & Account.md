Turn three already-built-but-unstyled portal surfaces — the sign-in page, a member dashboard, and the account-edit page — into presentable, cohesive pages with a consistent hero + look-and-feel, the same widget-styling/packaging pass the event landing pages got.

## Session-specific deltas

The portal is built on the same page-builder + widget stack as the public site, so this is the **event-landing-page styling work applied to the portal**, not new functionality. What exists today:

- **Seeded system pages** (`SystemPageSeeder`): `system/login` → `portal_login` widget, `system/signup` → `portal_signup`, `system/forgot-password` → `portal_forgot_password`, `system/account` → `portal_account_dashboard`.
- **Portal widgets** under `app/Widgets/`: `PortalLogin`, `PortalContactEdit`, `PortalChangePassword`, `PortalAccountDashboard`, plus `PortalSignup` / `PortalForgotPassword` / `PortalEventRegistrations` / `MembershipStatus`. They render working forms but ship **without dedicated styling** — raw `template.blade.php` leaning on shared SCSS component classes (`_forms.scss`, `_buttons.scss`, `_controls.scss`). For the dashboard's illustrative content, `BarChart` and `TextBlock` already exist.
- **Post-login redirect** currently targets `portal.account`.

**The look-and-feel is low-ambiguity** — unlike the event listings, there is one cohesive portal style. The owner's references (charmhealth sign-in card; Atlassian/Nexus profile-edit forms; a lightly-customized member dashboard — *not* an icon grid) may be dropped into the gitignored `sessions/demo website/scrap/` folder; if absent, the brief below stands on its own.

**The brief, in the owner's words:** *"looks right in nav, add a hero to each view with a consistent look and feel, and that's about it."* Three views, one visual language: consistent hero, spacing, typography, and buttons across all three.

## Open questions to resolve at session start

1. **Page structure.** Today `system/account` renders the `portal_account_dashboard` widget — one page. The owner wants a **dashboard view** *and* a separate **account-edit view**. Decide: split into a dashboard landing page (the post-login target) + a distinct account-edit page (`PortalContactEdit` + `PortalChangePassword`), reached from nav? Lean: yes — add/compose a `system/dashboard` page as the post-login landing, keep `system/account` as the edit page. Settle at the Open Gate; it drives the seeder + redirect change.
2. **Post-login redirect.** Follows from #1 — repoint `portal.account` → the dashboard route if a dashboard landing is added.
3. **Dashboard data (confirmed, not open — stated for the record).** The dashboard's widgets are **illustrative/demo content** (a stock `BarChart`, a `TextBlock` welcome message), **not** wired to the logged-in member's own records. This keeps v1 simple and avoids the portal-security per-`contact_id` scoping rule. Real member-scoped data is a later lift.

## Phases

### Phase 1 — Sign-in
Style the `PortalLogin` widget into a clean, centered sign-in card (logo, heading, fields, primary button, the secondary forgot-password link) on a calm, branded background — the charmhealth shot is the model.

### Phase 2 — Account edit
Make the existing `PortalContactEdit` (+ `PortalChangePassword`) form *look nice*: sectioned, aligned, readable — the Atlassian/Nexus profile-form shape. Form behaviour is unchanged; this is presentation.

### Phase 3 — Dashboard
Compose the member dashboard as the post-login landing: a consistent hero, then a couple of presentable widgets (`BarChart` + `TextBlock`) so it reads as a real dashboard. No icon grid. Wire the page-structure + post-login redirect decided at the gate; extend `SystemPageSeeder` if a new page is added.

### Phase 4 — Cross-cutting + verify
Ensure all three views sit correctly in the site nav for the logged-in portal state, share the same hero/visual language, and build cleanly (`build:public` / `npm run build` per the touched path). Announce ready for the owner's visual judgment.
