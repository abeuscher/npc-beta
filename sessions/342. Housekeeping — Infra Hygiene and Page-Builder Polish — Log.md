## What was built

A design-independent housekeeping batch draining nine verified inbox items across three clusters. All nine shipped; none turned out stale at the walk. Two inbox stretch items were deliberately held out (deferred to session 343, drafted at close).

### Phase 1 — Dependency & config hygiene

- **Removed Laravel Debugbar.** Pulled `barryvdh/laravel-debugbar` (dev dependency) and its transitive `php-debugbar/php-debugbar` via `composer remove --dev`, and removed the now-dead `class_exists(...)`/`Debugbar::disable()` guard in `WidgetDemoController::resolveBaseline`. Proven public-site outcome: the homepage renders HTTP 200 with `APP_DEBUG=true` and zero debugbar markers in the HTML, and `php artisan about` boots clean. The host-side capture scripts' defensive `.phpdebugbar` strip selectors (`scripts/*.js`) were left alone — they are inert once the package is gone and are not app runtime.
- **`.env.example` audit.** Pruned the dead broadcasting scaffold (`BROADCAST_*`, the entire `PUSHER_*` block, the `VITE_*` block — there is no `config/broadcasting.php` and no frontend Echo/Pusher consumption) and the env-based `STRIPE_*` keys (`config/services.php` has no stripe block; `StripeCheckoutService` reads `services.stripe.secret`, set only from `SiteSetting`; no model uses Cashier's `Billable`). Added a header note distinguishing the FM-provisioned vs standalone/self-host paradigms and a "CONTROL-PANEL entered" integrations block (Stripe / Mailchimp / Resend / QuickBooks credentials are entered in the admin UI; only `QB_ENVIRONMENT` stays an env var). Drift recorded: the inbox's five named dead vars (`APP_DEBUG_TOOLS`, `FLEET_MANAGER_AGENT_KEY`, etc.) were **not** present in `.env.example` — they were deploy-`.env`/`temp-config` findings; the real prune targets in the file were the framework scaffold plus the lingering `STRIPE_*` (Mailchimp/Resend env keys had already been removed in a prior pass).
- **Branded error pages.** Added `resources/views/errors/{layout,404,403,500}.blade.php`. `layout.blade.php` is a self-contained shell (inline CSS, no dependency on the public layout's DB queries / ChromeRenderer / runtime SCSS / widget manifest on the hard path) that reads the brand colour (`ColorTokenResolver::load()['brand']` → the live #dd6a18 orange) and site name defensively in a try/catch, falling back to neutral defaults if the DB is unavailable during a hard 500. Verified: a missing URL returns HTTP 404 rendering the branded page on-brand; `view:cache` compiles all three child views cleanly.

### Phase 2 — Page-builder operator-safety + copy

- **Confirm before hiding a populated column.** `LayoutColumnSettingsTab.vue` `onColumnsChange` now checks, on a column-count *decrease*, whether the dropped slots (indices ≥ the new count) hold widgets; if so it asks `window.confirm` before applying (mirrors the sibling delete-layout confirm in `LayoutInspectorPanel`), and reverts the input on cancel. Wording is accurate to the mechanism: a decrease **hides** widgets (they persist at their `column_index` and return if the count is increased again — `updateLayout` never touches `PageWidget` rows), it does not delete them.
- **Inline disabled-checkbox hint.** The "Background fills page width" checkbox's disabled reason now renders as a visible hint under the faded checkbox (previously only in a hover `title` tooltip).
- **Page-form terminology pass.** Clarified the `Page Type` / `Page Template` / `Content Template` helper texts in `PageResource` + `EditPageDetails` (de-vagued the "applied once at creation" copy). Labels were kept aligned with the Templates resource's own vocabulary and differentiated via helpers rather than renamed — a copy pass, no mechanism change.
- **Duplicate action on four resources** (owner-settled: all four). Organizations / Templates / Navigation / Mailing Lists each gained a Duplicate table action plus a tailored model `duplicate()`: Organization copies its own fields + tags only (no donations/memberships/affiliations/notes; `source` + import provenance reset to a hand-made copy); Template copies its owned widget/layout stack via `PageWidget::copyOwnedStack` and is never the default; NavigationMenu gets a uniquified handle + parent-remapped items; MailingList copies its filter rules. Each action is gated on the resource's create permission (`create_organization` / `update_page` / `create_navigation_menu` / `create_mailing_list`) via both `->visible()` and a server-side `abort_unless`. `ResourceDuplicateTest` (5 cases, 26 assertions). `npm run build` clean.

## Security considerations

Debugbar removal narrows the public information-disclosure surface — the toolbar (queries, environment, stack frames) can no longer render on any public page in any environment; verified objectively on the live homepage with debug mode on. The four new Duplicate actions are write surfaces, each gated by the resource's create permission through both the table action's `->visible()` and a server-side `abort_unless` (the same shape as the existing Page/Product duplicate actions); Organization duplicate deliberately drops import provenance and resets `source` so a copy cannot masquerade as imported or scrub data. The branded error views carry `noindex`, render no exception detail or sensitive data, and never widen what an error already exposes. The `.env.example` change is documentation only (no secrets), and the Stripe mock is test-only. Nothing on the security axis was weakened.
