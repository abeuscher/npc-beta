### Iteration 2 — Sub-nav threshold + primary-View deletion protection

Surfaced during user manual testing of iteration 1: with one seeded View on Contact, `EditContact` rendered an "Edit Contact / Overview" sub-nav even though Overview just shows the same widgets that already render in EditContact's footer. Compounding bug: the user could delete the seeded primary View through `/admin/record-detail-views`, leaving Contact with no foundation View and no UI path to recreate one.

- [app/Filament/Concerns/HasRecordDetailSubNavigation.php](app/Filament/Concerns/HasRecordDetailSubNavigation.php) — threshold gates on `$views->count() + count($additionalPages)` only; the entry page is the implicit primary destination and no longer counts toward the threshold. Sub-nav now renders iff there are at least 2 alternate destinations. Result on Contact: 1 View → no sub-nav (Recent Notes renders in EditContact footer via the resolveView fallback); 2+ Views → sub-nav surfaces with the user's chosen alternates as a feature, not a side-effect. Templates/Themes still has 3+ navigable destinations (Header / Footer Views + SCSS additional page) so its sub-nav is unaffected.
- [app/Filament/Resources/RecordDetailViewResource.php](app/Filament/Resources/RecordDetailViewResource.php) — added `primaryHandles(): array<class-string, array<int, string>>` and `isPrimary(RecordDetailView): bool` static helpers. `(Contact, contact_overview)` is the only registered primary today; new record types append entries here. `canDelete($record)` returns false for primaries before falling through to the existing permission gate. `DeleteBulkAction::before()` hook scans selected rows and cancels the action with a danger notification if any selected row is primary.
- [tests/Feature/RecordDetailViewSubNavigationTest.php](tests/Feature/RecordDetailViewSubNavigationTest.php) — added 2 cases for the new threshold semantic: entry + 1 View suppresses sub-nav (the regression catch); entry + 2 Views renders all 3 nav items.
- [tests/Feature/EditContactSubNavigationTest.php](tests/Feature/EditContactSubNavigationTest.php) — added 2 integration cases on `EditContact` directly: getSubNavigation returns [] with the seeded primary alone; returns 3 items (entry + Overview + custom) once a second View is bound.
- [tests/Feature/RecordDetailViewResourceTest.php](tests/Feature/RecordDetailViewResourceTest.php) — added 2 cases for the canDelete gate: primary `contact_overview` is flagged primary and cannot be deleted; user-created Views are not flagged and remain deletable.
- Re-seed restored `contact_overview` on the dev install (the user had deleted it during iteration-1 manual testing); the seeder's `firstOrCreate` shape made the recovery one-command.

### Surfaced during manual testing

Two user-driven findings during iteration-1 manual testing reshaped the session:

1. **Sub-nav over-triggered on Contact.** The 5d-1 prompt framed the EditContact wiring as "resolving the first-View-wins gap." That reading conflated two distinct concerns: (a) supporting multi-View on Contact when the user opts in, (b) auto-rendering sub-nav whenever any View exists. The right behavior is (a) without (b) — sub-nav should appear only when the user creates a second View, not as a side-effect of the seeded primary. Iteration-2 made the threshold honor the user's mental model.

2. **Primary View was deletable through the admin UI.** Compounding (1), the user's natural exploration ("delete the View to suppress the sub-nav") wiped the foundation that EditContact's footer auto-resolves. With no View bound to Contact, the Resource list at `/admin/record-detail-views` showed nothing for Contact and the only path forward was to *create* a new View (which then re-triggered sub-nav). Iteration-2 protects seeded primaries via the Resource's canDelete gate. The list still shows them so widgets can be added; the delete action just doesn't appear.

These weren't bugs in any single piece of code — they were emergent UX from how the trait's threshold and the Resource's canDelete behaviors composed. Manual testing surfaced them; the fix shape (one-line threshold change + a `primaryHandles` registry) was small once named correctly.

### Architectural alignment surfaced

A late-session conversation re-confirmed the multi-View affordance's purpose. Multi-View per record type is for cases like Templates/Themes where each View represents a genuinely different sub-area of the record (Header / Footer / SCSS / Colors do unrelated things). For Contact, the affordance exists but isn't load-bearing — the expected shape is **one View per record type, with widgets composed into it via the page-builder canvas at `/admin/record-detail-views`**. Adding a hypothetical "Hours Worked" widget to a Contact's front page is achieved by attaching it to `contact_overview` via the admin UI, not by creating a second View. The sub-nav primitive remains available for cases that genuinely want it (Templates, future record types with multiple sub-areas), but Contact-style record types stay flat by default.

This isn't a new architectural decision — it's what 5c's track-doc intent already named ("Most record types have one [View]; the exception is template record types..."). Iteration-2 brought the implementation in line with the documented intent.
