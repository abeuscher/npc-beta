# 172. Per-Widget File Colocation

Stage 3 of the Sovereign Widget arc. Every widget becomes a self-contained folder under `app/Widgets/{Name}/` holding its definition class, Blade template, and (if any) SCSS. The legacy `resources/views/widgets/` and `resources/scss/widgets/` directories are retired. This completes the structural half of the arc — after this, a widget is a *place*, not a scatter of files joined by naming conventions.

Session 171 made the resolver route every widget's config through `WidgetDefinition::defaults()` (for registered widgets) or `WidgetType::getDefaultConfig()` (for seeder-only widgets). Session 172 eliminates the second branch by promoting every seeder-declared widget to a definition class.

---

## Design decisions (resolved before starting)

- **Blade namespace.** Register a new `widgets` Blade namespace pointing at `app/Widgets/`: `View::addNamespace('widgets', app_path('Widgets'))` in `WidgetServiceProvider::boot()`. After registration, `@include('widgets::Nav.template')` resolves to `app/Widgets/Nav/template.blade.php`. No custom view finder, no per-widget namespace registration — one namespace, folder-based lookup.
- **Folder naming.** Each widget folder is `app/Widgets/{PascalName}/` — PascalCase, matching the PSR-4 location of its definition class (e.g. `NavDefinition` lives at `app/Widgets/Nav/NavDefinition.php`). The widget's handle (which may be snake_case, e.g. `bar_chart`) is a separate thing that stays in `defaults()` / the DB row.
- **File layout per widget.** Inside each folder:
  - `{PascalName}Definition.php` — the definition class
  - `template.blade.php` — the Blade template (required for `render_mode = server`)
  - `styles.scss` — optional; present only if the widget had a SCSS partial under `resources/scss/widgets/`
- **Default `template()` implementation.** Move the logic onto the base class: `WidgetDefinition::template()` returns `"@include('widgets::" . class_basename(static::class) . ".template')"` by default. Widgets that need inline templates can override, but none should after this session.
- **Default `assets()` SCSS path.** Keep the existing `assets(): array` contract. By convention, when a widget has a `styles.scss` alongside it, the definition returns `['scss' => ['app/Widgets/{PascalName}/styles.scss']]`. The `AssetBuildService` collector is extended to honour these paths (it already resolves paths via `base_path()`, so the change is just where the paths point).
- **Shared Blade partials.** The current `resources/views/widgets/components/` and `resources/views/widgets/partials/` directories hold Blade fragments shared across multiple widgets (icons, picture components, etc.). These are **not** widgets and don't belong inside any single widget's folder. Move them to `resources/views/widget-components/` and `resources/views/widget-partials/` (or a single `resources/views/widget-shared/` — decide in Phase 1 audit based on their count and semantics). Update every `@include('widgets.components.X')` / `@include('widgets.partials.X')` site-wide to the new path.
- **Seeder becomes trivial.** After this session, `WidgetTypeSeeder::run()` does nothing but call `app(WidgetRegistry::class)->sync()`. The giant `WidgetType::updateOrCreate([...])` blocks are gone because every widget is now a registered definition. Keep the file (it's still the seeder entry point) but strip its body to the sync call plus any supporting setup (e.g. thumbnail media).
- **Thumbnails stay where they are.** Thumbnail media (`thumbnail`, `thumbnail_hover` collections on `WidgetType`) are database-attached and untouched by this session. Thumbnail source files and the Playwright workflow are Stage 4.5 scope.
- **Inline CSS / JS columns.** `widget_types.css` and `widget_types.js` stay in the DB for now. They get populated by each widget's definition via `toRow()`. (If a widget has inline CSS/JS, the definition returns it through overridable `css(): ?string` / `js(): ?string` methods if those aren't already on the base class — audit Phase 1 decides.) The goal of colocating *all* of a widget's source into its folder is a Stage-N aspiration; this session stops at templates and SCSS.
- **No DB migration.** `widget_types` table shape is unchanged. Every row keeps being sourced from `WidgetRegistry::sync()`.
- **No user-visible behaviour change.** Everything should render byte-identically to the pre-session state. This is a refactor.

## Phase 4 — AssetBuildService update

The build pipeline currently walks `WidgetType::assets.scss` paths and resolves them via `base_path()`. With new paths pointing at `app/Widgets/{Name}/styles.scss`, this should already work — the service is path-agnostic. But confirm:

- Open `app/Services/AssetBuildService.php` and trace how asset paths are resolved. Confirm it reads files by absolute path under `base_path()` and doesn't hard-code `resources/scss/`.
- If it does hard-code the `resources/scss/widgets/` prefix anywhere (e.g. in a source-order list, in a partial-import resolver), update it to walk both the site-level `resources/scss/` partials AND the per-widget `app/Widgets/*/styles.scss` files.
- No change to the build server itself; the app just sends different source paths.

## Phase 5 — Promote every remaining widget to a Definition

For each widget currently in `WidgetTypeSeeder`, create an `App\Widgets\{PascalName}\{PascalName}Definition` class. This is the repetitive core of the session. Per widget:

1. Create `app/Widgets/{PascalName}/` folder.
2. Create `{PascalName}Definition.php` extending `WidgetDefinition`. Port every field from the seeder entry:
   - `handle()` returns the snake_case handle from the seeder
   - `label()`, `description()`, `category()`, `allowedPageTypes()`, `renderMode()`, `collections()`, `defaultOpen()`, `fullWidth()`, `requiredConfig()` — mirror the seeder
   - `schema()` returns the `config_schema` array verbatim
   - `defaults()` returns a key→value map derived from the schema's `default` entries (follow the rule `WidgetType::getDefaultConfig()` uses: schema `default` if present, otherwise the type-based fallback — `''` for text, `false` for toggle, `[]` for checkboxes, `null` for number/image). The base class's `validate()` will enforce defaults-schema parity during `sync()`.
   - `assets()` — copy from the seeder, updating any `resources/scss/widgets/_X.scss` path to `app/Widgets/{PascalName}/styles.scss` (if a partial exists for this widget)
   - `css()` / `js()` — for widgets with inline CSS/JS in the seeder, return the value (these need new optional overridable methods on the base class; see below)
3. If the widget has a Blade template file under `resources/views/widgets/{handle}.blade.php`, move it verbatim to `app/Widgets/{PascalName}/template.blade.php`. The base-class default `template()` will find it.
4. If the widget has a SCSS partial under `resources/scss/widgets/_{handle}.scss`, move it verbatim to `app/Widgets/{PascalName}/styles.scss`.
5. If the widget's seeder entry used an **inline** Blade body in the `template` column (not `@include`), put that body into `template.blade.php` verbatim.
6. Register it in `WidgetServiceProvider::boot()`: `$registry->register(new \App\Widgets\{PascalName}\{PascalName}Definition());`
7. Delete the widget's `WidgetType::updateOrCreate(...)` block from `WidgetTypeSeeder`.

### Base-class additions (inline CSS/JS)

To keep the `css` / `js` / `code` / `variable_name` columns populated for widgets that use them, add overridable methods to `WidgetDefinition`:

- `css(): ?string` — default `null`
- `js(): ?string` — default `null`
- `code(): ?string` — default `null` (for `render_mode = client` widgets)
- `variableName(): ?string` — default `null`

Extend `toRow()` to include `css`, `js`, `code`, `variable_name` keys sourced from these methods.

### Per-widget PR-style discipline

Because Phase 5 is a repetitive mass migration, commit after each widget is promoted and its tests pass — or at least run the fast suite after each widget to catch per-widget surprises early. Don't batch all 28 widgets into one commit.

## Phase 7 — Retire the old directories

Once Phases 3–6 are complete and all tests pass:

- `resources/views/widgets/` — should contain only the (now-migrated) `components/` and `partials/` subdirs which are empty after Phase 6. Delete the whole directory.
- `resources/scss/widgets/` — should be empty after every partial migrated. Delete the whole directory.
- Confirm nothing in the app, tests, docs, or build configs references either path.

## Phase 8 — Seeder trivialisation & coexistence cleanup

- `WidgetTypeSeeder::run()` becomes: call `app(WidgetRegistry::class)->sync()` and whatever thumbnail-seeding logic it currently does (if any). Strip the rest.
- In `App\Services\WidgetConfigResolver`, the `baseDefaults()` fallback to `$pw->widgetType?->getDefaultConfig()` is now unreachable under normal operation (every widget has a definition). Decide:
  - **Keep** as a safety net for DB rows whose widget definition was removed at runtime (e.g. a widget was uninstalled but its page_widgets rows remain).
  - **Remove** to simplify.
  - Recommendation: keep. The coexistence branch costs nothing and guards against handle drift.
- `WidgetType::getDefaultConfig()` itself — keep; it's consumed by the coexistence branch and may be useful for introspection.
