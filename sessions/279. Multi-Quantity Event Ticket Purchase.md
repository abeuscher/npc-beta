# 279. Multi-Quantity Event Ticket Purchase

Promote the 278 tier-canonical registration shape from "one ticket per submission" into "N tickets across one or more tiers per submission." Buyer picks quantities per tier on the public form; checkout assembles one Stripe session with multi-line-items; webhook fans out the N pending registrations on session.completed. Per-tier capacity switches from `withCount('registrations')` to `withSum('registrations', 'quantity')`. Admin "View Registrants" surfaces a `tickets` column.

## Session-specific deltas

The plan entry pins the "what" (`quantity` column, `withSum` capacity aggregate, multi-line-item Stripe session, webhook fan-out, quantity-spinner widget, admin `tickets` column, importer unchanged). 278-close pre-chose data shape **(A)** over a parent-orders shape after explicit weighing — that decision stands and doesn't need re-litigation.

The remaining design choices this session pins at impl time:

- **Quantity-spinner UI.** Three reasonable shapes: bare `<input type="number">` per tier; +/- step buttons around a read-only display; native number input with custom +/- buttons stacked alongside. Native `<input type="number">` is the simplest and accessibility-correct; custom buttons read more deliberate at the cost of more markup + JS. Lean toward the native input with `min=0`, `max={remaining-capacity-or-99}`, `step=1`; confirm at impl time if there's a strong reason to deviate.
- **Subtotal display.** Live-updating subtotal line above the submit button, summing each tier's `quantity × price` as the buyer adjusts. Implement with a small Alpine `x-data` or vanilla event listener — judgment at impl time.
- **Max-quantity per tier on the form.** Cap to remaining capacity when the tier has finite capacity; cap to a high constant (e.g. 99) when capacity is null. Server-side validation re-checks against actual current registrations at submit time.
- **Mixed free + paid tier purchases.** Allowed — the form sums quantities across all chosen tiers; if any tier has price > 0, the flow routes to Stripe with a multi-line-item session including the free-tier rows as zero-amount lines (Stripe accepts $0 line items). If every chosen tier has price 0, the flow stays on the free path and creates N rows directly. Cleanest is "the chosen tier's prices, summed, decide the path" rather than "every tier must be the same path" — confirm at impl time.
- **Free-path mixed-tier grouping.** Free purchases produce N rows without a `stripe_session_id` to group by. Use a `created_at` ± 1 second window + shared `email` for the admin "View Registrants" grouping. Not a strong relational join; sufficient for the UI surface.

## Open questions to resolve at session start

- **Quantity-spinner UI shape (native input vs custom +/- buttons).** Lean native; confirm or push for custom.
- **Max-quantity policy.** "Cap to remaining capacity / fallback to 99" is the proposal. Confirm or pin a different cap.
- **Mixed-price purchase routing.** "Any tier > 0 → Stripe; all zero → free path" is the proposal. Confirm.
- **Form-level minimum.** Cleanest is "submit requires sum(quantities) ≥ 1." Confirm — vs e.g. "≥ 1 per chosen tier."

## Implementation shape

Two iterations (plus optional polish):

### Iteration /1 — Schema + controllers + webhook + per-tier capacity-via-withSum

- Migration `add_quantity_to_event_registrations` — `smallint default 1`. Existing rows get 1 (the default fills naturally for existing data).
- `EventRegistration::$fillable` adds `quantity`; `$casts` adds `quantity => 'integer'`.
- `EventRegistration::factory()->definition()` keeps default 1.
- Per-tier capacity:
  - `TicketTier::isAtCapacity()` swaps `count()` for `sum('quantity')` over the same status filter.
  - `ContractResolver::resolveEventOne()` swaps `withCount(['registrations as registered_count' => ...])` for `withSum(['registrations as registered_count' => ...], 'quantity')`. Eloquent supports this directly.
  - `SystemModelProjector::projectTiers()` and `projectEvent()` unchanged at the surface; the underlying aggregate just means a sum instead of a count.
- Controllers:
  - `EventController::register` reads a `quantities` map: `request('quantities')` is `{<tier_id>: <int>}` for selected tiers (rows with `0` quantity are dropped or absent). Validates each tier_id belongs to the event (`Rule::exists` with `where('event_id', ...)`). Validates each quantity is a positive integer. Validates sum ≥ 1.
  - For each `(tier_id, quantity)` pair: check per-tier remaining capacity (`tier.capacity - tier.registered_sum >= quantity`); if any tier fails, error with "This ticket tier doesn't have enough remaining capacity" (or per-tier specific messaging — judgment).
  - For the free path (all chosen tiers have price 0 OR no tiers): create N rows directly, one per tier with its quantity. Source remains `Source::HUMAN`.
  - For the paid path (any chosen tier has price > 0): create N pending rows (one per tier with its quantity), all sharing the eventual `stripe_session_id`. Build the Stripe session with N line items (one per tier — `price_data.unit_amount` = tier.price in cents, `quantity` = quantity). Stripe accepts $0 line items if a free tier is included alongside paid ones.
  - `EventCheckoutController::store` does the same as the paid path above (consolidate: the free→paid handoff in `EventController` produces the same payload either way).
  - Portal variants mirror the guest variants.
- Webhook `StripeWebhookController::handleEventRegistrationCheckout`:
  - Metadata now carries `event_stripe_session_id` (or just use the Stripe `session.id` directly — judgment) instead of a single `event_registration_id`.
  - Find all pending registrations sharing the stripe_session_id (`->where('stripe_session_id', $session->id)->where('status', 'pending')->get()`); promote them all to `registered`; record one `Transaction` at `amount_total / 100` against the *first* registration (subject linkage stays single-valued; the others share the transaction implicitly via shared stripe_session_id — adequate for v1 + revisitable in the post-1.0 stub).
- Tests:
  - `quantity` column persists; default 1 for missing.
  - Per-tier capacity via `withSum` (tier with capacity 5, two registrations qty 2 + qty 3 → at capacity; one more registration qty 1 → rejected).
  - Multi-tier purchase produces N rows; all share `stripe_session_id` for paid path; all carry the buyer email for free path.
  - Webhook fan-out promotes all N rows on session.completed; one Transaction recorded.
  - Free-path multi-tier purchase rejected if sum < 1.
  - Mixed-price purchase: free tier line at $0 + paid tier line at $X passes through Stripe correctly.
  - Cross-tier capacity respect (one tier full, another open → only the open tier accepts the quantity).
  - Updated existing tests that pass a single `ticket_tier_id` continue to work via a defaulted-to-1 quantity (the existing controllers can accept the legacy single-tier shape OR the new quantities map — confirm shape decision at impl time).

### Iteration /2 — Quantity-spinner widget + Playwright spec + admin tickets column

- `app/Widgets/EventRegistration/template.blade.php`:
  - Replace the radio picker with per-tier quantity spinners. Each tier row gets `<label>{{ tier.name }} — ${{ tier.price }} <input type="number" name="quantities[{{ tier.id }}]" min="0" max="{{ remaining-or-99 }}" value="0" step="1"></label>`.
  - Live subtotal: small Alpine `x-data="{ subtotal: 0, update() { ... } }"` listener that sums the spinners' values × prices on input. Update a `<strong>` element near the submit button.
  - "Sold-out" tier renders disabled at `max=0` with a "(sold out)" label.
  - The single-tier case (the common 278 shape) still works — a single spinner.
  - The zero-tier case (free uncapped) stays as today: no quantity UI, no `quantities` payload, controller treats as one registration.
- `app/WidgetPrimitive/Projectors/SystemModelProjector.php` — `projectTiers()` adds a `remaining_capacity` field per tier (capacity minus registered sum, or null when unlimited). Template reads this for the spinner `max`.
- `app/Filament/Resources/EventResource/Pages/ViewRegistrations.php` — table gets a `tickets` column (`TextColumn::make('quantity')`). The "View Registrants" admin surface now reads "Jane Doe — 3 tickets" as one row.
- Playwright spec at `tests/e2e/event-registration/quantity-spinner.spec.ts` (or add scenarios to the existing `ticket-tier-picker.spec.ts`):
  - Single-tier event: spinner increments, subtotal updates, submit with quantity 3 lands one registration with quantity=3.
  - Multi-tier event: two spinners; bumping General to 2 + VIP to 1 → subtotal sums correctly; submit lands two registrations (qty 2 + qty 1) sharing stripe_session_id on paid, or both at $0 on free.
  - Capacity-cap: a tier with remaining capacity 2 has spinner max=2; trying to set 3 client-side caps OR server rejects.
  - Sold-out tier: spinner disabled at 0.
- Update `resources/docs/widget-event-registration.md` for the quantity-spinner reality; re-run `help:sync`.

### Iteration /3 — *(judgment, only if needed)*

If admin "View Registrants" needs more than a simple column (e.g. operators want the multi-row purchases grouped visually), carve as /3. Otherwise fold into /2.

## Out of scope

- **Per-attendee data inside the registration flow.** Post-1.0 — folded into the "Event Registration Depth" stub in `session-outlines.md`. The 278 `notes` field is the current workaround.
- **Partial cancellation / partial refunds.** Post-1.0 — same stub. Until then operators do "full refund + repurchase remaining" if a buyer wants to drop tickets from a purchase.
- **`EventRegistration` semantic rename.** Post-1.0 — same stub. Under (A) a row represents a purchase line (tier × quantity); the class name still reads coherently in shorthand and a rename is cosmetic churn until per-attendee sub-rows land.
- **Stripe subscription / recurring quantity.** Not in scope. Event tickets are one-shot purchases.
- **Tier-level quantity caps independent of capacity** (e.g. "limit 4 per buyer"). Not in scope — capacity is the only quantity constraint this session ships.
- **Promo codes / discount layered on quantity.** Not in scope; deferred to the broader Stripe Checkout Branding work (E4 in the release plan).
