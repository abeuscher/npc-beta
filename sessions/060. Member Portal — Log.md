# 060. Member Portal — Log

## What was built

### Phase 1 — Account suspension

- Migration: `add_is_active_to_portal_accounts` — adds `is_active boolean default true`
- `PortalAccount`: added `is_active` to `$fillable` and `$casts`
- `LoginController`: after successful `attempt()`, checks `is_active`; if false, logs out and returns the same generic credential error (no signal that the account is suspended)

### Phase 2 — Portal page/widget architecture

The initial approach (standalone controller views with GET routes) was discarded mid-session. The correct approach uses the existing widget/page system: member pages are pages of `type = 'member'` with portal widget types rendered by PageController via the catch-all slug route. No separate portal page controller needed.

Three new widget types added to `WidgetTypeSeeder`:
- `portal_contact_edit` — address + email change forms
- `portal_change_password` — password change form with client-side match validation
- `portal_event_registrations` — table of the member's event registrations

Four seeded member pages via `PortalPageSeeder`:
- **Dashboard** (`{prefix}`) — seeded with `saveQuietly()` to bypass PageObserver double-prefix behaviour
- **Edit Account** (`{prefix}/edit-account`)
- **Change Password** (`{prefix}/change-password`)
- **Event Registrations** (`{prefix}/event-registrations`)

Portal navigation menu (handle: `portal`) seeded with four items pointing to the above pages.

`layouts/portal.blade.php` updated to read the portal nav menu from the database and render items dynamically; member name and logout button remain.

### Phase 3 — AccountController (POST routes only)

`app/Http/Controllers/Portal/AccountController.php` — four actions:
- `updateAddress()` — validates city/state/postal_code/country, updates `contact`, redirects `back()` with success flash
- `updatePassword()` — verifies current password, updates, calls `logoutOtherDevices()`, redirects `back()`
- `requestEmailChange()` — validates uniqueness, sends `PortalEmailChange` mailable to new address via `URL::temporarySignedRoute()` (60-minute expiry)
- `confirmEmailChange()` — validates signature and ownership, updates both `portal_account` and `contact` email, marks `email_verified_at`

`routes/web.php`: GET `/account` now redirects to dashboard. GET routes for address/password/email removed (served as widgets). POST/PATCH routes kept: `PATCH /account/address`, `PATCH /account/password`, `POST /account/email`, `GET /account/email/confirm`.

### Phase 4 — Admin Portal Access panel

`ContactResource` gains a collapsible "Portal Access" section in the sidebar with:
- Status, email, and created-at placeholders
- Suspend / Restore toggle actions (gated on `is_active` state + `manage_contacts` permission)
- Mark Verified / Mark Unverified actions (gated on `email_verified_at` state + `manage_contacts` permission)

### Phase 5 — Form collision path

`FormSubmissionController`: before processing a public form submission, checks whether the submitted email belongs to an active portal account. If it does and the submitter is not authenticated as that account, returns a neutral success response (no error shown) and sends a rate-limited notification to the account holder via `PortalFormCollision` mailable (one email per hour per address, keyed in Cache). Staff are not notified of the collision.

`portal_form_collision` email template added to `DatabaseSeeder`.

### Contact model

- Added `portalAccount()` hasOne relationship
- Uncommented and enabled `eventRegistrations()` relationship

### Orphaned files removed

Four standalone portal blade views created in the first pass (`portal/account.blade.php`, `portal/edit-address.blade.php`, `portal/change-password.blade.php`, `portal/change-email.blade.php`) were deleted after the architecture correction.

## Key decisions

- **Portal page architecture**: member pages + widget types served by PageController — not standalone controller views. Keeps everything inside the existing CMS.
- **Dashboard slug (`saveQuietly()`)**: PageObserver auto-prefixes slugs on save. The dashboard slug is the bare prefix with no sub-segment, so it must be saved quietly to avoid double-prefixing.
- **Form collision — neutral response**: A portal account holder who submits a public form gets a silent success. This prevents enumeration — the submitter cannot tell whether the block happened.
- **Email change race condition**: If an admin changes a contact's email between the member initiating and confirming a change, the member's confirmation overwrites it. Accepted as correct — in realistic use both parties would be assigning the same value.
- **`back()` redirects**: Post-submit redirects stay on the current page (`back()->with('success')`) rather than going to the dashboard. This keeps the success flash visible on the widget page where the action was taken.
- **Household architecture (discussed, not built)**: Separate `households` table (not self-referential FK on contacts). Inviter address always canonical. Auto-create on invite acceptance. Sync warning UI when addresses diverge. Invite to unknown email gives neutral response + logs a note on the inviting contact's record.
