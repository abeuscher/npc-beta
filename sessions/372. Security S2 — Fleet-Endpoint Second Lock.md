## Session-specific deltas

- **The problem, precisely.** Today `docker/nginx/prod.conf` sets `ssl_verify_client optional` and each of the five `location = /api/*` blocks carries a bare `if ($ssl_client_verify != "SUCCESS") { return 403; }`. That per-location `if` is the **only** defense — the app layer (`routes/api.php` → the four `app/Http/Controllers/Api/Fleet/*` controllers) deliberately trusts the connection and re-verifies nothing. A dropped/shadowed/misconfigured `if` on the blob or recover location silently exposes the whole donor DB or the admin-credential reset. S2 adds an independent app-layer gate so two things must both hold, not one.
- **Additive contract bump v2.6.0 → v2.7.0.** The gate is a *new requirement behind* the existing mTLS, not a schema change. Update the contract doc + CHANGELOG + `Contract Version`, and the Cross-Repo block, **before** implementing. The rollout must not break a Fleet Manager that hasn't yet started sending the new credential — decide the rollout posture (additive-with-window vs env-flippable-enforcement) at the Open Gate.
- **Worst-first, same pattern across the five.** Land and prove the gate on `/api/backup/blob` and `/api/admin/recover` first (worst blast radius), then apply the identical mechanism to `/health`, `/logs`, `/backup/trigger`. Model the gate on the S1 security-headers middleware pattern (`app/Http/Middleware/*`, session 370) — a route-middleware applied to the fleet route group is the natural shape, but confirm against the existing route wiring.
- **`/api/logs` PII tension resolved here.** The endpoint returns raw `storage/logs/laravel.log` lines — possible PII / stack traces — which sits against the counts-only privacy boundary the fleet design holds elsewhere (`data_hygiene`, `suspension` are count/state-only by design). Resolve it: redact at the tail, scope what is returned, or consciously accept it with a documented rationale on the Gate-3 findings register. Decide the approach at the Open Gate.
- **The mechanism is a design call at session start** (see Open questions). Do not pre-commit in the prompt.

## Open questions to resolve at session start

1. **Gate mechanism** — a **shared-secret header** FM already possesses / is provisioned at install time (simple, survives an nginx misconfig entirely because it's TLS-independent, but adds a secret FM must store and send), vs. **client-cert fingerprint forwarding** (nginx forwards the verified client-cert fingerprint/DN to the app via a proxy header, the app re-verifies against a pinned value — reuses the existing cert, no new secret, but is only as independent as the header nginx sets). Recommendation formed after the code read, confirmed at the Open Gate.
2. **Rollout posture** — how does Fleet Manager keep working while it absorbs the new credential? Enforced-but-env-flippable (ship gate off, flip on after FM sends it), or accept-either during a documented window? Whichever avoids a flag day.
3. **Stored state** — does the chosen mechanism need a per-install secret in env/config, or nothing new stored? (A migration is unlikely; confirm.)
4. **`/api/logs` resolution** — redact / scope / consciously-accept, and if redact, where in the tail path.
