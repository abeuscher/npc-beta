For driving over our own public CMS pages on the demo node, those concerns mostly don't apply — different DOM, our markup, our selectors. - **vue-cal is the picked calendar library.** It is a Vue 3 component; our public widgets are server-rendered Blade with JS assets, so mounting vue-cal in a public widget means standing up a small Vue runtime on the public side (or imperatively mounting per instance). Settle the mount pattern at the open gate. If the architectural cost reads as too high once examined, surface to the owner before committing — a vanilla-JS calendar lib is the fallback.

The plan references them by relative path with an explicit "(local-only, not committed)" annotation. **Add `sessions/demo website/scrap/` to `.gitignore` if it isn't already** before saving any reference there.
- **Per-page permission, not exception.** The locked landing page (the demo arrival page) and any portal-style read-only demo pages should be locked via a **generalizable per-page permission mechanism** — not a one-off exception for the demo landing's slug. Owner preference is explicit: try to make it not an exception. Settle the design at 324; the build session implements it.
- **Portal pages are tour-only.** The product's portal is a feature prospects should see but can't be live on the demo node (anonymous arrival + port lockdowns). Include portal pages as **tour-driven, read-only** demo content so driver.js can walk a prospect through them. No portal login on the demo node.
