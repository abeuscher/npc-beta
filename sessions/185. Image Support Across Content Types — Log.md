## What was built

### Phase 1 — Audit

Mapped the gap between admin form exposure and public rendering for image fields on Page (incl. `type='post'`), Event, and Product:

- Page: `post_thumbnail` (exposed + read), `post_header` (neither exposed nor read), `og_image` (SEO only).
- Event: `event_thumbnail` (exposed + read), `event_header` (exposed but never rendered publicly), `event_og_image`.
- Product: `product_image` (exposed + read via WidgetDataResolver).
- Architecture: pages, posts, and events all render via widget composition (`pages/show.blade.php`); no template-level title/header slot. `PageContext` already exposes `$currentPage` to every widget template.

### Phase 2 — Per-page context bridge

- Added `post_header` Filament field to `PageResource` so users can upload it.
- New `App\Services\PageContextTokens` service. Substitutes `{{title}}`, `{{date}}`, `{{excerpt}}`, `{{author}}`, `{{starts_at}}`, `{{location}}` from the current page (and its associated event when applicable). Unknown tokens pass through untouched.
- Added `Page::event(): HasOne` reverse relation against `Event.landing_page_id` to support event-specific tokens.
- Made `$pageContextTokens` available alongside `$pageContext` in every widget Blade template via `WidgetRenderer`.
- Background-image override toggle on the shared appearance panel: when on, `AppearanceStyleComposer::composeBackgroundImage` reads `post_header` for posts/default pages or `event_header` for event landing pages; when no header is attached, no background image renders. Widget-owned background image is bypassed while the override is on. Gradient layer still composes independently.
- New `BackgroundPanel.vue` UI for the toggle; concrete `false` default added to `WidgetDefinition::defaultAppearanceConfig()`.
- Backfill migration `2026_04_15_040103_backfill_page_widget_use_current_page_header` populates the new key on existing rows.

### Phase 3 — Widget pool-access pattern

- Added `WidgetDefinition::demoImages(): array` hook. Returns entries with `category`, `count`, `target` (`appearance.background_image` or `config.<key>`).
- `WidgetDemoController` injects pool URLs at render time via `SampleImageLibrary`. Pool returns `min(count, available)`; empty pool no-ops.
- For `appearance.background_image` target: writes the URL into `appearance_config.background.image_url`; `AppearanceStyleComposer` reads it as a fallback when no `appearance_background_image` media is attached.
- For `config.<key>` target: writes URL string (or array) into `config[key]`; widgets opt in to read it.
- Preset rendering (`WidgetDemoController::showPreset`) skips widget-level `demoImages()` injection — presets define their own complete look and may opt in via their own `demo_images` key.
- `DashboardDebugGeneratorWidget` now attaches pool images to generated records: Products → `product_image` (`product-photos`), Events → both `event_thumbnail` and `event_header` (`still-photos`), blog Posts → both `post_thumbnail` and `post_header` (`still-photos`). Empty pool no-ops gracefully.
- Pattern documented in `resources/docs/widget-development.md`.

### Phase 4 — Applied to under-rendering widgets

- **Hero**: declared `appearance.background_image ← still-photos` for baseline demo. Original opaque-gradient demo look preserved (image is covered by the gradient at baseline; presets unaffected because preset rendering skips widget-level demoImages).
- **Image widget**: declared `config.image ← still-photos`; template falls back to a plain `<img>` when no uploaded media is present so demo capture renders something.
- **BoardMembers**: added `demoConfig()` mapping `image_field → photo` and other field bindings so demo mode resolves the seeded demo collection's portraits (root cause of the empty-thumbnail symptom — `image_field` was unset).
- **EventCalendar DemoSeeder**: rewrote the three hardcoded events ("Bedrock Community Dig", "Moose Lodge Annual Meeting", "Annual Anvil Drop Gala") to match the SampleLibrary tone; each gets a `still-photos` pool image attached to both `event_thumbnail` and `event_header`. Stable slugs and day offsets preserved for deterministic capture.
- ProductDisplay deferred — needs broader template work to render without a real Product row.

### Polish surfaced during the session

- BlogPager: thumbnails now render above the prev/next links (`.pager-link__thumb`); SCSS added in `resources/scss/_custom.scss`.
- Default `items_per_page` for BlogListing and EventsListing changed from 10 to 6 (schema + defaults).
- BoardMembers: `grid_background_color` default changed to `transparent` (concrete CSS keyword).
- `<x-picture>` component fix: when only the base `webp` conversion exists (no `responsive-*`), the component now emits a single-URL srcset rather than dropping the source entirely and falling back to the original JPG.
- New `app/Widgets/Image/styles.scss` constrains `.widget-image` to `max-width: 100%; height: auto` (registered via `assets()`); fixes the Image widget overflowing its container at intrinsic pixel size.
- User-side category edits to several widget definitions (TextBlock, Carousel, BarChart, Hero) — added `most_used` membership.
- User regenerated thumbnails for the affected widgets (BlogListing, BoardMembers, Carousel, EventCalendar, EventsListing, Image, LogoGarden, ProductCarousel, BarChart).

## Production deployment notes

- Run `php artisan migrate` to apply the backfill.
- Run `php artisan db:seed --class=WidgetTypeSeeder` to pick up widget definition changes (categories, BoardMembers `demoConfig`, Image widget `assets`, etc.).
- Run `php artisan build:public` to ship the updated public widget bundle (Image widget styles, BlogPager pager-link styles, etc.).
- After deploy, the dashboard random-data generator now attaches pool images to records when the pool is populated. To populate the pool on a fresh install, run `php artisan db:seed --class=Database\\Seeders\\SampleImageLibrarySeeder`.

## Deferred decisions

- **ProductDisplay demo-mode improvement.** Needs a path for the template to render without a real Product (either accept a stubbed product object or add a fallback render mode). Out of scope this session.
- **Single-record header rendering.** Originally proposed as `ContentHeader` widget or auto-prepend in the controller; deferred in favour of the broader per-page context bridge approach delivered here. Future widgets can opt into rendering current-page header media via the appearance toggle and `{{title}}`-style tokens.
- **TextBlock (richtext) embedded image overflow.** Surfaced during testing — a plain `<img>` inside richtext content is unconstrained. Intentional non-fix this session; will be addressed locally to richtext rendering rather than via a global rule.
