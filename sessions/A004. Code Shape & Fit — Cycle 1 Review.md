## Phase 1 — Pre-discussion evaluation

After the reading list completes (and after the Open Gate is cleared), the agent prepares a structured evaluation against the track's stated goals. The evaluation is what gets surfaced to the user to anchor the discussion.

### Evaluation axes

For each, the agent assesses based on 312 + 313 evidence and proposes a verdict (worked / partially worked / didn't work / inconclusive) with one-paragraph rationale:

- **Anti-performative-change discipline.** The track's central claim is that requiring cited Intent prevents the agent from inventing plausible reasons to refactor. Did it? 312 flagged three fabricated citations — two reclassified to intent-unrecoverable, one salvaged. Is that the discipline working, or the discipline barely working?
- **Citation-verbatim-with-line-numbers calibration.** Landed in the Cycle 1 → 2 standing improvements as the response to the fabricated-citation incidents. Is it the right fix, or is the deeper issue that audit agents are reaching for citations they don't have? What would change for Cycle 2 to actually catch this earlier?
- **Five-workstream taxonomy.** W-R1 Cardinality / W-R2 Locality / W-R3 Directness / W-R4 Necessary vs accidental / W-R5 Seam quality. W-R3 and W-R5 produced zero landing-grade findings; W-R1 / W-R2 / W-R4 produced 2 inline + 3 flags. Is the taxonomy carrying its weight, or are some workstreams structurally low-yield in this codebase?
- **Intent-unrecoverable shape.** Two findings carried forward unmodified. Is this the right outcome — a deliberate documentation-gap surfacing — or a sign that the discipline is too restrictive and is leaving real cardinality fixes on the floor?
- **Six-iteration ceiling.** 313 used two iterations. The ceiling didn't bind; nothing to test. Calibration data is null for Cycle 1. Worth noting for Cycle 2.
- **Class-finding pattern-fix vs instance-fix default.** Default is instance-fix; user opted into pattern-fix for Flag W-R1/A (the `PageTreeBuilder` lift). Was the pattern fix the right call in hindsight? Anything the agent observed during the lift that the instance-fix path would have avoided or that the pattern-fix specifically improved?
- **Pulled-forward calibration.** Cycle 1 ran at 312/313 instead of the originally-scheduled 327+ (trailing pair after cleanup Cycle 3). One W-R2 outlier was stale by the time the audit walked it. Was the pull-forward worth it, or should Cycle 2 be more strictly sequenced after the next cleanup cycle?
- **Cycle-shape (2 sessions: audit / apply, no squash).** Compare to the cleanup track's three-session shape (audit / apply / squash). For Cycle 1 the no-squash call held — nothing landed that would have generated migration churn. Will that always be the case, or does the track need a squash slot for cycles where a seam move ends up dropping a column or renaming a table?
- **Permanent inter-cycle artifacts.** The finding-template (`sessions/tracks/code-shape-and-fit-finding-template.md`) carries the row format and worked examples. Did the audit agents actually use it? If they did, where did it help; if not, what made it skippable?
- **PM-level tone re-coaching (process incident).** Surfaced mid-session in 313, not in the original track design. The memory entry `feedback_pm_level_tone` was widened to call out project-internal jargon. Is this a track-track issue or a general agent-behavior issue? Should the track doc carry a note that audit-track bookkeeping codes belong in artifacts, not in user-facing prose?

### Evaluation output

A document the agent surfaces in the discussion, structured per the axes above. One paragraph per axis: verdict + rationale + (if applicable) a proposed track-doc edit that would encode the lesson.

---

## Phase 2 — Discussion with user

The agent walks the evaluation with the user, one axis at a time. For each:

- **Agent surfaces** its verdict + proposed track-doc edit (if any).
- **User responds** — agree / disagree / refine.
- **Outcome captured** — what conclusion the discussion landed on, whether a track-doc edit follows, and if so, the specific edit shape.

The agent should expect refinements. The user may surface observations the agent didn't see (most cycles will have at least one such observation — the user has cross-cycle context the agent doesn't). New observations from the user that don't fit the pre-loaded axes get their own discussion section.

The discussion ends when the user signals alignment. The agent should not push to wrap — let the discussion run as long as it needs to.

---

## Phase 3 — Track-document audit

Once alignment is reached, the agent updates the track artifacts to reflect the discussion's conclusions. Possible writes:

- **`sessions/tracks/code-shape-and-fit.md`** — additions to the Cycle 1 retrospective (if the discussion surfaced lessons that should be added), additions to the standing-improvements list (Cycle 1 → 2 carries), tightening of workstream definitions if a discussion topic revealed ambiguity, calibration adjustments to the cadence trigger or pulled-forward rule.
- **`sessions/tracks/code-shape-and-fit-finding-template.md`** — refinements to row format, intent-citation discipline language, the W-R5 fifth-column rule, or the worked examples per workstream. The template is meant to evolve cycle over cycle as audits surface what works.
- **`sessions/refactor-track-deliberately-kept.md`** — refinement to existing carve-out language or addition of a new register entry if the discussion identified one that wasn't surfaced at 313 close.
- **`sessions/housekeeping-inbox.md`** — promotion of process-observation items that don't belong in the track doc.
- **`sessions/release-plan.md`** — addition of a stub entry if the discussion surfaced an own-session-shaped follow-on.

Each edit lands as a discrete diff-able block in the relevant file; no restructuring of the doc for minor changes. Edits cite the A004 discussion as their origin via a session-citation note.
