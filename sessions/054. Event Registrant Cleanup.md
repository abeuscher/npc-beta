# 054. Event Registrant Cleanup

A focused data-hygiene action on the Event module. Adds a manual staff action to the Event edit page that removes contacts who were auto-created solely by registering for a specific event, along with all registration records for that event. Read all relevant files before touching anything.

---

## Workstream 1 — Delete Event-Originated Contacts Action

Read `app/Filament/Resources/EventResource.php` and `app/Filament/Resources/EventResource/Pages/EditEvent.php` in full before making any changes.

### Requirements

1. Add a **Delete Registrant Contacts** header action to the Event edit page.

2. The action is only shown when the event is eligible for cleanup — either all of its associated dates have passed, or the event is cancelled. Hide (not disable) it for upcoming events.

3. When triggered, the action first runs the identification query (see below) and presents a confirmation modal containing:
   - The count of contacts that will be deleted.
   - The count of registration records that will be removed.
   - The message: *"This will permanently remove these contacts and all registration records for this event. Contacts with memberships, donations, or registrations at other events are not affected."*
   - A note that the operation may take a moment to complete.
   - Confirm and Cancel buttons.

4. On confirm:
   - Soft-delete the identified contacts (use `$contact->delete()`).
   - Delete all `event_registration` rows for this event, regardless of whether the associated contact was identified for deletion.

5. Display a summary notification: *"X contacts removed, Y registration records deleted."*

### Contact identification logic

A contact is eligible for deletion if ALL of the following conditions are true:

- Their `id` appears in `event_registrations.contact_id` for this event.
- `contacts.source = 'web_form'` — they were created via the public registration form, not manually added or imported.
- They have no `event_registration` records linking them to any other event.
- They have no `membership` records (any status).
- They have no `donation` records.

Contacts that fail any condition are silently skipped. They remain untouched; only their registration record for this event is removed in step 4.

### Notes

- Do not add a scheduled or automatic trigger. This is a deliberate, one-off staff action.
- The eligibility query may require multiple joins. This is acceptable — it is a low-frequency operation and the admin can be told to wait.
- Do not hard-delete contacts. Soft-delete only.
- If zero contacts are identified as eligible, the confirmation modal should still appear and note that no contacts will be deleted, but registration records will still be removed.

---

## Out of scope

- Automatic cleanup on event close or date expiry
- Bulk cleanup across multiple events at once
- Hard deletion of any record
- Any changes to the MINIMUM DATA concept or global purge policies
