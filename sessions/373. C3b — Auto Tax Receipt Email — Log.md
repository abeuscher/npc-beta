## What was built

An **automatic per-gift tax-acknowledgment email** wired into the successful-donation Stripe webhook path — C3b, launch-schedule position 4, the first Gate-2 money-path session and the "→ receipt" leg of the **Stripe → CRM records → receipt** path. Prerequisite for the C4 rehearsal. This was **wiring an existing seam**, not a greenfield build (`feedback_audit_integration_seams`).

- **The seam.** `StripeWebhookController::handleDonationCheckout` recorded the gift + a `Transaction` and sent nothing. It now captures the created transaction and queues a per-gift acknowledgment via a new `sendDonationAcknowledgment()` helper right after `Transaction::recordStripe(...)`.
- **New template handle.** `donation_acknowledgment` added to `EmailTemplate::defaults()`, the `DatabaseSeeder` template list, and the `EmailTemplateResource` token hint — single-gift IRS contemporaneous-acknowledgment copy (donor name, amount, date, transaction reference, fund, org name from `site_name`), admin-editable under **System Emails**. Body is clean prose (`<p>/<strong>/<br>` only, no inline styles) so it survives `HtmlSanitizer` on save.
- **New Mailable.** `App\Mail\DonationAcknowledgment` mirrors the annual `DonationReceipt` — reuses the `EmailTemplate` render/wrapper machinery + the `mail.system-email` view. Dispatched **queued** (`Mail::to()->queue()`) so a mail-provider hiccup never fails the webhook (which must return 200 fast).
- **Idempotency.** New `acknowledged_at` nullable timestamp on `donations` (migration `2026_07_17_000001`), checked-and-set in the helper. Layered over the handler's existing `Transaction`-exists guard, so a webhook replay never double-emails the donor. Proven by a test that replays the same signed event.
- **Scope cut.** Initial gift only (`checkout.session.completed`). Recurring renewals (`invoice.payment_succeeded`) are a deliberate **per-charge fast-follow** — one recurring donation row spans many charges, the wrong grain for a per-donation `acknowledged_at` column.
- **Owner data-minimization decision — no EIN stored.** Declined a structured `org_ein` setting. The IRS-required elements ship (org name, amount, date, "no goods or services" statement); an org that wants its EIN can type it into the editable template body. Rationale saved to memory (`feedback_dont_store_sensitive_ids_when_self_serviceable`).
- **Untouched.** The annual `donation_receipt` year-end statement + the manual `DonorsPage` "Send Receipts" flow — a different document — stay as the backfill/resend path.

## Security considerations

Payment-webhook-adjacent, no new public surface. The acknowledgment dispatch sits on the Stripe-signature-verified webhook path; the email is queued, isolating mail failures from the 200 response. No new PII leaves the system — the acknowledgment goes only to the donor's own email and carries only data the donor already provided (name, amount, date). No EIN / org tax-id is stored (data-minimization). The template body is sanitized by the existing `HtmlSanitizer` on save. Contract unchanged (v2.7.0); non-boundary.

## Production deployment notes

None beyond standard `migrate`. The `worker` container must be running for the queued acknowledgment to send (it already runs on every install). The `donation_acknowledgment` template row is seeded on fresh installs and lazily created via `EmailTemplate::forHandle()` on the first donation for existing installs, so it appears under System Emails for editing either way.
