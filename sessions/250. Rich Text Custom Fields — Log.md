## What was built

### Phase 1 — Sanitization decision

Resolved at session kickoff against existing rich-text discipline in the codebase. The de facto pattern across [TextBlock](app/Widgets/TextBlock/template.blade.php), [Hero](app/Widgets/Hero/template.blade.php), [BoardMembers](app/Widgets/BoardMembers/template.blade.php) (via Collection rich_text), [Memos](app/Widgets/Memos/template.blade.php) (via Collection rich_text), [NoteResource](app/Filament/Resources/NoteResource.php), [EventResource](app/Filament/Resources/EventResource.php) (description + meeting_details), [EmailTemplateResource](app/Filament/Resources/EmailTemplateResource.php), and [GeneralSettingsPage](app/Filament/Pages/Settings/GeneralSettingsPage.php) is path A: trust Quill's output, render with `{!! !!}`. The page-export/import path at [ContentImporter::sanitizeWidgetConfig](app/Services/ImportExport/ContentImporter.php#L420) does collection-handle remap + media-id clearing only — no HTML sanitization on `richtext` config values. HTML round-trips byte-for-byte across page export/import today.

The session prompt's Design decisions had recommended path B (sanitize on save with hand-rolled allow-list under `App\Support\`); during read-through I surfaced that custom_fields would be the only sanitized rich-text surface in the app, with TextBlock / Hero / BoardMembers / etc. all staying raw. User confirmed path A for E3 — match existing convention exactly. The broader hardening question lifts to a separate stub at close.

### Phase 3 — Form-component bridge

### Phase 4 — Event + Page admin form wiring

Initial read-through suggested a gap: `toFilamentFormComponent()` was only invoked from [ContactResource:97-102](app/Filament/Resources/ContactResource.php#L97-L102), so Event and Page admin forms wouldn't render their custom_fields. Verification overturned the finding — [HasPageBuilderForm::metadataFormSchema:80-89](app/Traits/HasPageBuilderForm.php#L80-L89) already injects a "Custom Fields" section into every resource that uses the trait. EventResource (via `pageBuilderFormSchema`) and PageResource (directly via `metadataFormSchema`) both flow through it. The grep against the resource files alone missed the trait-injected wiring. **No code change needed** — Event and Page admin forms already render their custom_fields sections, including the new rich_text type.

### Phase 5 — Importer auto-create-custom-fields

The importer's auto-create path lets operators map a CSV column to a brand-new CustomFieldDef on the fly with a chosen `field_type`. The field-type Select offers `text` / `number` / `date` / `boolean` / `select` today; `rich_text` was missing. Two locations both updated to add `'rich_text' => 'Rich Text'`:

- [ImportContactsPage.php:565-575](app/Filament/Pages/ImportContactsPage.php#L565-L575) (the contacts import wizard).
- [InteractsWithImportWizard.php:573-583](app/Filament/Pages/Concerns/InteractsWithImportWizard.php#L573-L583) (the shared trait used by donations / events / memberships / invoices / notes import wizards).

Without this, an operator importing a Wild Apricot HTML `bio` column via auto-create would silently land it as plain `text`. The CSV value passes through verbatim into `custom_fields.{handle}` either way ([ImportProgressPage:326,433,493](app/Filament/Pages/ImportProgressPage.php#L326-L493)) — what changes is whether the CustomFieldDef row records `field_type='rich_text'` (so the next admin edit mounts QuillEditor instead of TextInput).

## Deferred decisions

- **Rich-Text Surface Sanitization Hardening** — surfaced during E3 read-through. Today every rich-text surface in the app stores Quill output (or Trix output, in the [Collection RichEditor](app/Models/Collection.php#L126) case) verbatim and renders via `{!! !!}`. [ContentImporter::sanitizeWidgetConfig](app/Services/ImportExport/ContentImporter.php#L420) does collection/media remapping but not HTML sanitization, so HTML in widget config and (post-E3) `custom_fields` round-trips byte-for-byte through page export/import. Quill v2 strips `<script>` tags client-side but that is a UX nicety, not a server-side guarantee. Captured as a new stub in `sessions/session-outlines.md` (Housekeeping & Review — Beta 1 Scope) folding in the previously-floated "Collection Rich-Text Editor Convergence" stub. Scope outlined: one `App\Support\HtmlSanitizer` utility with a Quill-shape allow-list, applied at every model save boundary that stores rich-text HTML, plus the Memos `body` Trix→Quill swap (with one-time data-migration) so the convergence and hardening land together.
- **Daily channel migration** — surfaced while drafting session 251 (Fleet Manager Contract v2.1.0 — Log-Fetch Endpoint). The CRM runs `LOG_CHANNEL=stack` / `LOG_STACK=single` today, which means 251 ships path A (`?lines=N` tail-only) against the unbounded `laravel.log`. Date-based selection (`?date=YYYY-MM-DD`) requires flipping to `daily` first; that channel-migration plus the additive `?date=` parameter is queued as a future v2.x bump if/when FM-side UX needs it.
