# 163. Page Builder Bug Fixes

Bug-fixing session covering issues surfaced during session 162 manual testing and any other page builder rough edges that have accumulated. Sequenced ahead of session 164's universal Appearance panels because the inspector/RTE bugs below actively block productive testing of the existing inspector, and 164 lands a large new inspector surface on top.

## Design decisions (resolved before starting)

### Bug 1 — RTE typing race on the page builder inspector

**Symptoms:** typing into a Quill-backed `richtext` field (e.g. hero `content`, text_block `content`) at moderate speed (~60–80 WPM) causes characters to be dropped or the editor contents to be wiped back to a stale state. Easiest to reproduce on a fresh page load — clicking a widget for the first time and typing immediately. Becomes much harder to reproduce after the first widget interaction in a given page-load.

**Mechanism — full trace:**

1. **Bulk default-write on first widget click.** [InspectorField.vue:29-37](resources/js/page-builder-vue/components/InspectorField.vue#L29-L37) writes default config values into the store on mount whenever `widget.config[field.key] === undefined`. On a freshly-loaded page (or after `migrate:fresh --seed`), most config keys are missing because the seeders only populate required values. Clicking a widget mounts the inspector, every `InspectorField` fires its `onMounted`, and a stack of `updateLocalConfig` calls land in the same global 350 ms `debounceSaveTimer` bucket *before the user has typed anything*. The user's first keystroke arrives inside that already-armed window; the timer fires before the field-level Quill debounce has flushed the user's typing.

2. **Save response overwrites in-flight typing.** [editor.ts:215-228](resources/js/page-builder-vue/stores/editor.ts#L215-L228) does `Object.assign(widgets.value[id], updated)` after every save, copying the *whole* server response (including `config`) onto the local widget. Anything the user typed between the moment the save fired and the moment the response arrived is silently overwritten by the older server snapshot. This race is structural and pre-dates session 162 — it has been latent since the Vue migration in session 149. Session 162 made it more visible by changing the conditions under which it fires (the `migrate:fresh --seed` baseline state), not by introducing it.

3. **Quill watch wipes the editor on stale config.** [RichTextField.vue:72-83](resources/js/page-builder-vue/components/fields/RichTextField.vue#L72-L83) watches `props.modelValue` and, on any change where `quillInstance.root.innerHTML !== newVal`, directly assigns the new value to Quill's root. Combined with #2, this is the visible "garble": the editor contents jump back to whatever the server returned. The string-equality check is also fragile — Quill normalizes whitespace and attribute order, so even a "no-op" round trip can trip the watch.

4. **Redundant 300 ms field-level Quill debounce.** [RichTextField.vue:57-63](resources/js/page-builder-vue/components/fields/RichTextField.vue#L57-L63) buffers `text-change` events for 300 ms before emitting `update:modelValue`. The store's [editor.ts:481-502](resources/js/page-builder-vue/stores/editor.ts#L481-L502) `flushDebouncedSave` already provides 350 ms of debounce — this field-level layer is duplicate work and widens the total race window.

**Fix plan — apply all three, ordered by impact:**

#### Fix 1.1 — Delete the InspectorField onMounted defaults block

[InspectorField.vue:29-37](resources/js/page-builder-vue/components/InspectorField.vue#L29-L37) currently runs:

```ts
onMounted(() => {
  if (
    props.widget.config[props.field.key] === undefined &&
    props.field.default !== undefined
  ) {
    store.updateLocalConfig(props.widget.id, props.field.key, props.field.default)
  }
})
```

Delete it entirely. The render-time `fieldValue` computed at [InspectorField.vue:24-27](resources/js/page-builder-vue/components/InspectorField.vue#L24-L27) already falls back to `field.default` when the config value is undefined:

```ts
const fieldValue = computed(() => {
  const val = props.widget.config[props.field.key]
  return val !== undefined ? val : props.field.default
})
```

So the visible behaviour is unchanged — fields display their defaults — but the value is no longer auto-persisted on first mount. The renderer also treats missing config values as "no inline style emitted." The only behaviour change is that a widget that has never been touched will *not* have its defaults written into the database; that is fine, because the renderer applies the same defaults.

This is the highest-impact fix because it eliminates the bulk-defaults-on-mount race entirely. Mirrors the surgical fix already applied to `WidgetAppearanceControls.vue` in session 162.

#### Fix 1.2 — Make updateWidget skip config/appearance_config/query_config when the widget is dirty

[editor.ts:215-228](resources/js/page-builder-vue/stores/editor.ts#L215-L228) currently does:

```ts
async function updateWidget(id, changes) {
  saving.value = true
  try {
    const res = await api.updateWidget(id, changes)
    const updated = res.widget
    if (widgets.value[id]) {
      Object.assign(widgets.value[id], updated)
    }
    dirtyWidgets.value.add(id)
    return updated
  } finally {
    saving.value = false
  }
}
```

Replace the unconditional `Object.assign` with a selective merge that:

- **Always merges** server-authoritative fields: `id`, `widget_type_id`, `widget_type_handle`, `widget_type_label`, `widget_type_collections`, `widget_type_config_schema`, `widget_type_assets`, `widget_type_default_open`, `widget_type_required_config`, `layout_id`, `column_index`, `sort_order`, `is_active`, `is_required`, `image_urls`, `preview_html`. These never change as a side effect of local typing.
- **Skips merging** the user-mutable fields `config`, `appearance_config`, `query_config`, and `label` *if and only if* the widget is currently in `dirtyWidgets`. The local copy is by definition newer than the server's response — the server is just confirming "I received what you sent." If the widget is *not* dirty (e.g. the local copy has been cleaned up by a successful preview refresh, or the merge is happening from `populateFromItems`), the response is authoritative and a normal merge happens.

Subtlety to handle carefully: `dirtyWidgets` is currently reset only by `populateFromItems` (which wholesale replaces the local map) and by `refreshPreview` on a successful preview fetch (which calls `dirtyWidgets.delete(id)`). After this fix, the dirty bit must remain set until we know the local and server copies match. The cleanest interpretation: a widget is dirty from the moment the user mutates it until a clean preview round trip completes. Do not clear the dirty bit inside `updateWidget` — only `refreshPreview` should clear it. Audit the existing code paths to confirm this is consistent.

This is the structural fix that closes the race window even in the rarer "user types faster than the round trip" mid-session case.

#### Fix 1.3 — Remove the field-level Quill debounce

[RichTextField.vue:57-63](resources/js/page-builder-vue/components/fields/RichTextField.vue#L57-L63) currently does:

```ts
quillInstance.on('text-change', () => {
  if (suppressChange) return
  if (emitTimeout) clearTimeout(emitTimeout)
  emitTimeout = setTimeout(() => {
    emit('update:modelValue', quillInstance.root.innerHTML)
  }, 300)
})
```

Replace with a synchronous emit:

```ts
quillInstance.on('text-change', () => {
  if (suppressChange) return
  emit('update:modelValue', quillInstance.root.innerHTML)
})
```

Also delete the `emitTimeout` variable, the `clearTimeout` in `onUnmounted`, and any other lingering field-level debounce machinery. The store's 350 ms `flushDebouncedSave` is the single source of truth for save batching. This removes one buffering layer where keystrokes can sit invisibly during a save, makes the timing easier to reason about, and shortens the total wall clock between typing and save firing.

### Bug 1 — optional polish (only if time)

The `RichTextField.vue:76` string-equality check on innerHTML is fragile because Quill normalizes whitespace and attribute order. Two cleaner options:

- **Option A:** in the watch, skip the assignment entirely if the widget is in `dirtyWidgets`. The local Quill state is by definition newer in that case, so there is nothing to sync from props.
- **Option B:** compare a normalized form of both strings (e.g. parse to DOM and compare), accepting that this is slower.

Option A is simpler and consistent with Fix 1.2's "dirty means local wins" rule. Take Option A if the polish window is open.

## Phase 3 — Tests

- **Manual verification is the primary surface for the RTE race itself**, since it is a Vue/timing issue. Hard-refresh the browser, click a freshly-seeded widget, immediately type a paragraph at moderate speed (~60–80 WPM) into the RTE. Confirm no characters dropped or content wiped. Repeat across hero, text_block, and any column-child rich text widget.

- **Pest test for Fix 1.2 — selective merge:** add a Pest test that constructs a `PageWidget`, simulates the local-mutation-then-server-update flow at the API layer (PUT a partial config update and assert the server response shape), and asserts the relevant invariants. The test cannot reach into Pinia from PHP, but it can lock in the API contract that the JS-side fix relies on. If a JS test runner is needed, this is the moment to either skip (deferred to the post-Beta JS test runner stub) or add a single targeted test under `tests/Feature/`.

- **Pest test for Fix 1.1 — defaults render at the renderer not the store:** add a test that creates a widget with `config: []` (empty), renders the widget through the public renderer, and asserts the rendered HTML reflects the schema-default values. This locks in the contract that renderer-time fallbacks are sufficient and the on-mount writes are not required.

- **Run the full fast suite** to make sure none of the fixes regress existing API/contract tests: `docker compose exec app php artisan test --exclude-group=slow`.

- **Run `npm run build`** to confirm the JS compiles cleanly. Vue/JS build failures are silent to the Pest suite.

## Security checklist

Before closing:

- [ ] No new permissions, routes, or fillable fields introduced.
- [ ] Selective merge in `updateWidget` does not silently bypass server-side validation. The server response is still the source of truth for `is_required`, `is_active`, and the widget type's `config_schema`; only user-mutable content fields are conditionally skipped.
- [ ] Removing the InspectorField defaults block does not allow any field to render as `undefined` in a way that could be exploited (e.g. a type confusion in the Blade renderer). The renderer's `?? field.default` fallback was already handling this case.
- [ ] No raw user input is rendered into the editor preview or public output without sanitization. The renderer continues to cast spacing values to int and continues to skip non-hex colour values.
