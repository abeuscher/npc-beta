# 100. Template Manager UI & Page Creation Flow

Build the Template Manager admin page and integrate templates into the page creation and editing flows. Retire the Site Theme page.

---

## Design decisions (resolved before starting)

- The Template Manager lives under the CMS nav group, labeled "Templates". It replaces the Site Theme page.
- The view opens with two tabs: **Content Templates** (tab 1, default) and **Page Templates** (tab 2).
- Page template editing uses the same tabbed layout as the former Site Theme: colors/fonts, SCSS, header, footer. Non-default templates get an "inherit from default" toggle per section (colors, fonts, SCSS, header, footer — each independently inheritable).
- The default page template does not show inherit toggles — it is the source of truth.
- Content templates are listed with name and description. Clicking one opens a detail/edit view showing the widget stack definition. Users can rename, edit description, and delete non-system content templates.
- "Save as Content Template" is an action on the page builder that serializes the current page's widget stack (handles, configs, sort orders) into a new content template.
- The page creation flow (Create Page) shows a template picker: first pick a page template (header/footer/styling — defaults to "Default"), then pick a content template (widget preset — defaults to "Blank"). Content template is applied once at creation and locked. Page template can be changed later.
- The page edit view gains a "Page Template" dropdown in a settings section, allowing the user to change which page template (header/footer/styling) is applied. No content template change after creation.
- The Site Theme page is retired: its nav entry is removed, its route is removed or redirected. All functionality now lives in the Template Manager.
- Clean up site_settings keys that were migrated to the templates table in session 099. The renderer should no longer read from site_settings for colors/fonts/SCSS — only from the template.

## Phase 1 — Template Manager page

### Filament page: `TemplateManagerPage`

- Navigation group: CMS
- Navigation label: Templates
- Route: `/admin/templates`
- Two tabs using the same pattern as Site Theme (Alpine tab switching).
- Breadcrumbs: `[Templates]`

### Tab 1 — Content Templates

- List all templates where `type = 'content'`.
- Each row shows: name, description, widget count (from definition), system badge if applicable.
- Actions per row: Edit (name, description), Delete (non-system only, with confirmation).
- "Save as Content Template" action is available from the page builder (Phase 3), not from this list.

### Tab 2 — Page Templates

- List all templates where `type = 'page'`.
- Each row shows: name, description, default badge if applicable.
- Click a row to open the editor (same view, below the list or as a slide-over).
- Editor has sub-tabs matching the former Site Theme: Colors & Fonts, SCSS, Header, Footer.
- Non-default templates: each sub-tab has an "Inherit from Default" toggle at the top. When on (field is null), the section is read-only showing the inherited value. When toggled off, the fields become editable.
- Default template: no inherit toggles, all fields always editable.
- Actions: Create new page template, Delete (non-default only), Set as Default.

## Phase 2 — Page creation flow

### Update `CreatePage` (PageResource)

- After the existing title/slug fields, add two selection steps:
  1. **Page Template** — select dropdown or card grid of `type = 'page'` templates. Defaults to the default template. This sets `template_id` on the new page.
  2. **Content Template** — select dropdown or card grid of `type = 'content'` templates. Defaults to "Blank". This determines which widgets are created on the page.
- On page creation: set `template_id` from the page template choice. If a content template other than "Blank" was selected, hydrate its `definition` into `page_widgets` for the new page.
- Content template choice is one-time — it does not persist on the page record. It's just used to prepopulate widgets.

### Update `EditPage` (PageResource)

- Add a "Page Template" dropdown in the page settings section (alongside existing meta fields).
- Changing the page template updates `template_id` on the page. This is non-destructive — it only changes the header/footer/styling wrapper.
- No content template selector on edit — the widget stack is already created and user-editable.

---

## Phase 3 — Save as Content Template

### Page builder action

- Add a "Save as Template" button to the page builder header (alongside the existing "+ Add Block" button).
- Clicking it opens a modal: template name (required), description (optional), save button.
- On save: serialize the page's current widget stack into the content template `definition` format — array of `{handle, config, sort_order}` objects, using widget type handles. Column widget children are nested. Save as a new `Template` record with `type = 'content'`, `is_default = false`.
- Success notification with the template name.

---

## Phase 4 — Retire Site Theme

- Remove `SiteThemePage` from the Filament panel (delete the class or remove its registration).
- Remove or redirect the `/admin/site-theme` route.
- Clean up site_settings keys that are now stored on the template: `public_primary_color`, `public_heading_font`, `public_body_font`, `header_bg_color`, `footer_bg_color`, `nav_link_color`, `nav_hover_color`, `nav_active_color`, `theme_scss`.
- Update any remaining code that reads these settings to use the template system instead.
- The `_header` and `_footer` system pages remain — they are now owned by the default page template.

## Security checklist

Before closing:

- [ ] Template Manager respects existing CMS permissions — only users with page management permissions can access.
- [ ] "Save as Template" does not expose widget configs from pages the user cannot access.
- [ ] Content template definitions only store widget type handles and config — no sensitive data.
- [ ] Deleting a page template does not break pages that reference it (nullOnDelete on FK — pages fall back to default).
