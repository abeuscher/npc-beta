# 259. Importer Auto-Mapping Pattern Lift (B2a)

Lift the Contacts importer's `App\Services\Import\FieldMapper` pattern (a service-level alias dictionary with rich preset coverage and `isSkipped` predicates) to the five namespaced importers — Events / Donations / Memberships / Invoice Details / Notes — so a non-preset-supported CSV's column headers auto-map at parity with the Contacts surface. Audit the existing Contacts FieldMapper for missing common header aliases and add them in the same pass.

This is the architectural follow-on to **B2** (Onboarding rehearsal cluster, closed at 258). The B2 rehearsal measured the auto-mapping coverage gap: 5 namespaced importers' per-page `guessDestination($normalizedHeader)` match statements recognize ~10% of header conventions under non-preset shapes, vs the Contacts importer's substantially higher coverage. This session shrinks that gap.

---

## Design decisions (resolved before starting)

These need to be confirmed at session start during the read-through audit. Drafts of intent below; the read-through may reframe.

- **Architectural shape — per-importer mapper services parallel to the existing `FieldMapper`.** Three options surfaced from the prior session's reading: (a) extend each `*ImportFieldRegistry` with mapping methods; (b) create per-importer mapper services parallel to `FieldMapper`; (c) unified service with importer-keyed dispatch. **Lean: option (b)** — per-importer mapper services mirror the existing pattern (`FieldMapper` is for Contacts; `EventFieldMapper` would be for Events). Each preserves a single class's worth of cognitive load when reading or editing one importer's mapping. The trade-off is duplicated structure across five files; the unified-service alternative inverts that. Surface architectural recommendation on session start; final shape is the user's call.

- **Preset coverage scope — `generic` required, `wild_apricot` and `bloomerang` populated where source systems actually export the record type.** `generic` is the universal floor — all five lifted importers gain rich `generic` coverage. `wild_apricot` and `bloomerang` are populated based on what those source systems actually export for each record type; not every importer × preset cell is meaningful (e.g., Wild Apricot doesn't have a separate "Invoice Details" export, so that cell may stay empty or fall back to `generic`). Audit each importer's source-system reality before committing.

- **`SOURCE_SKIPPED_HEADERS` parity.** Contacts has per-preset skip-header arrays (Wild Apricot's `password`, `archived`, `subscribed to emails`, etc.). The lifted importers gain analogous per-preset skip lists *only where source-system exports actually carry sensitive or never-relevant headers* for that record type. Don't pre-populate; expand only when a known header needs filtering.

- **Floor-vs-ceiling discipline.** Every header→destination pair the existing `guessDestination()` match statements recognize must continue to map after the lift. Test coverage for the existing pairs is the *floor*; new aliases are the *ceiling*. The session does NOT silently drop existing coverage, even if a particular alias looks redundant or weird.

- **Sentinel pattern — out of scope.** Each importer has slightly different sentinel coverage (`__custom_*__`, `__note_*__`, `__tag_*__`, `__org_*__`). Standardizing sentinel coverage is *not* this session's work. Flag any sentinel-parity gap noticed during the read-through as an Out-of-Gate or follow-on candidate; do not absorb into 259.

- **Value-normalization findings — out of scope.** The events `status` case-normalization and the email-whitespace-trim findings (both lifted from B2 to the Out-of-Gate register) touch a *different* code path (`processOneRow` row-processing, not column mapping). They are not absorbed by this session. Schedule them in their own session or batch.

- **Help-doc revision in scope.** The playbook's "Some column names won't auto-recognize" edge-case entry needs a revision post-lift to reflect the new coverage. Light edit; one paragraph at most.

- **Out of scope this session:**
  - Export CSV actions for non-Contact list resources (B2b — separate working-set entry).
  - Sentinel coverage parity (per the design decision above).
  - Value-normalization fixes (events status, email whitespace).
  - Any boundary changes to the Fleet Manager agent contract.
  - The `SOURCE_SKIPPED_HEADERS` per-preset skip lists for the lifted importers — populate only where there's a known specific header to skip; don't pre-populate speculatively.

## Phase 2 — FieldMapper alias-gap audit (Contacts surface)

**Goal:** Add the missing common aliases to the existing `FieldMapper` so the Contacts surface itself starts the lift in good shape (not just the namespaced importers).

**Steps:**
1. Walk the audit list from Phase 1.
2. Add each missing alias to `FieldMapper::presetMap('generic')` (and per-preset arrays where appropriate — `wild_apricot` may not have `Postal Code` if it uses `Zip` consistently).
3. Add unit tests to `FieldMapperTest` for each new alias — assert `map($header, $preset)` returns the expected destination.
4. Confirm `auto_create_custom_fields` behavior still kicks in for unmapped headers (regression check).

**Pass criterion:** every alias added has a test that maps it correctly, and the existing FieldMapper test suite continues to pass.

## Phase 3 — Per-importer mapper service skeletons

**Goal:** Create the five per-importer mapper services (or whatever shape Phase 1's architectural decision lands on) with the existing `guessDestination()` floor coverage migrated in.

**Per importer (one phase iteration each, but bundled here):**
1. Create `App\Services\Import\<Type>FieldMapper` (or equivalent per the architectural decision).
2. Migrate the page class's `guessDestination()` match statement into the new service's `presetMap('generic')` array.
3. Update the page class's `guessDestination()` method to delegate to the new service.
4. Update the trait's `processUploadedFile()` integration if needed (the seam should stay, but the call shape may change).
5. Ship a parallel `<Type>FieldMapperTest` mirroring `FieldMapperTest`'s test patterns. Floor coverage = every existing match-statement entry has a test.

**Pass criterion:** the five new services exist; the page classes delegate; floor coverage is fully tested; existing import end-to-end specs still pass.

## Phase 4 — Per-importer alias enrichment

**Goal:** Expand each importer's `presetMap('generic')` to recognize substantially more common header conventions, plus populate `wild_apricot` and `bloomerang` presets where the source systems actually export that record type per Phase 1's audit.

**Per importer:**
1. Catalog 15–30 additional `generic` aliases per importer based on common naming conventions across CRM exports. Drawn from operator-realistic header forms (e.g., events: `Title` / `Event Name` / `Name`; `Start` / `Start Date` / `Start Time` / `Begins At`; etc.).
2. Populate `wild_apricot` preset entries where Wild Apricot's exports actually carry that record type. Same for `bloomerang`.
3. Add tests for each new alias.

**Pass criterion:** the regenerated G1 messy-shape fixtures auto-map a substantially higher percentage of columns under the lifted importers — target shift from ~10% (at 258) to ~70%+ for the `generic` preset. Re-run the Phase 2 portion of the B2 rehearsal (the messy-fixture import flow) and observe coverage improvement directly.
