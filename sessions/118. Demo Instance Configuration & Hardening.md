# 118. Demo Instance Configuration & Hardening

Final lockdown session for demo mode. Configure custom route prefixes to obscure defaults, audit every exposed surface for security, and build the definitive demo mode test suite that verifies every write path is blocked.

---

## Design decisions (resolved before starting)

- Route prefixes on the demo instance differ from defaults to add a layer of obfuscation: `/blog` instead of `/news`, `/portal` instead of `/members`, `/calendar` instead of `/events` (or similar — confirm during implementation). These are set via site settings, not code changes.
- The demo mode test suite is the canonical verification that the entire demo permission layer works. It lives in the `demo` slow test group and is designed to be run before any demo instance deployment.
- This session is the final audit — anything missed in sessions 114–117 gets caught and fixed here.
- The test suite should be thorough enough that after a period of non-demo development, running it reveals any regressions before updating the demo instance.

---

## Phase 1 — Route prefix configuration

1. **Set custom route prefixes** via the site settings seeder or the `DemoSeeder`:
   - Blog prefix: `blog` (default is `news`)
   - Portal prefix: `portal` (default is `members`)
   - Events prefix: `events` (or `calendar` — confirm with user)
2. **Verify** that all internal links, breadcrumbs, and navigation items resolve correctly with the new prefixes.
3. **Verify** that the CMS export/import (session 116) handles the prefixes correctly — page slugs that include the prefix should import cleanly.

## Phase 2 — Comprehensive demo mode test suite

Build a thorough test suite in `tests/Feature/DemoModeTest.php` (or a directory `tests/Feature/Demo/`) in the `demo` slow group. This suite boots the app with `APP_ENV=demo` and a demo-role user, then verifies:

### Resource write paths (for every Filament resource)
- [ ] Cannot access create routes (403 or redirect)
- [ ] Cannot submit edit forms (403 or no-op)
- [ ] Cannot trigger delete actions
- [ ] Cannot trigger bulk actions
- [ ] Cannot trigger custom table actions (invite, archive, etc.)

### Settings pages
- [ ] Cannot submit any settings form
- [ ] Routing fields are not present in the rendered form
- [ ] API key fields show no stored values

### Livewire write methods
- [ ] Page builder save, reorder, delete, add-block methods refuse
- [ ] Any other Livewire component write methods refuse

### Import & tools
- [ ] Import pages return 403
- [ ] Tax receipt generation returns 403
- [ ] Debug generator widget is not visible

### Infrastructure
- [ ] Mail driver is `log` (or scoped exception for auth emails — per session 115 decision)
- [ ] Horizon returns 403
- [ ] No unexpected routes are accessible (dump route list and verify)

### Data isolation
- [ ] Users list does not include demo-role users
- [ ] Demo user cleanup command deletes only demo users
- [ ] Demo data refresh does not touch marketing pages or navigation

## Phase 3 — Security audit

1. **Route audit** — dump all registered routes (`artisan route:list`) and verify every POST/PUT/PATCH/DELETE route is either:
   - Behind authentication + non-demo permission, or
   - A public route that is harmless in demo mode (e.g. login form submit)
2. **Port audit** — verify Docker exposes only ports 80/443 (nginx). No direct database, Redis, or Horizon ports exposed.
3. **Debug surface** — verify `APP_DEBUG=false`, `APP_DEBUG_TOOLS=false`, `TELESCOPE_ENABLED=false` (if applicable), no `phpinfo()` routes, no debug bar.
4. **Headers** — verify no server version headers leak (nginx version, PHP version).
5. **Public file audit** — verify `storage/` symlink doesn't expose anything beyond intended public files. No `.env`, no `composer.json`, no git directory accessible via HTTP.
