# 125. Build Server Settings & CI Pipeline

Move build server configuration from `.env` into the CMS Settings admin page. Add a build server health check to the dashboard integrations widget. Move asset compilation into the GitHub Actions deploy pipeline so CI handles all builds. Remove Node.js from the production Docker image.

---

## Design decisions (resolved before starting)

- Build server URL and API key move from `config/services.php` env vars to `site_settings` rows, managed in the CMS Settings page ŌĆö same pattern as other site-level settings.
- The new "Build Server" section goes on the CMS Settings page between the "General" section and the "Header & Footer Code Snippets" section.
- The dashboard integration status widget gains a build server entry that checks connectivity (HTTP ping to the build server health endpoint).
- The deploy workflow runs `npm run build` (Vite admin theme) and `php artisan build:public` (widget bundle) as CI steps. The production Docker image no longer needs Node.js.
- `AssetBuildService` reads the build server URL and API key from `site_settings` (via `SiteSetting::get()`), falling back to the existing env vars for backwards compatibility during migration.

## Phase 1 ŌĆö Build server settings in CMS Settings page

Add a "Build Server" section to `CmsSettingsPage` between "General" and "Header & Footer Code Snippets":

- **Build Server URL** ŌĆö text input, nullable, placeholder `http://bundleserver:8080`
- **Build Server API Key** ŌĆö password input (masked), nullable
- **Test Connection** button ŌĆö calls the build server's health endpoint, shows success/failure notification
- Save both values to `site_settings` as `build_server_url` and `build_server_api_key`

Update `AssetBuildService` to read from `SiteSetting::get()` first, falling back to the existing `config('services.build_server.*')` env vars. This keeps local dev working via `.env` while production uses the admin-configured values.

---

## Phase 2 ŌĆö Dashboard integration health check

Add a "Build Server" row to `DashboardIntegrationStatusWidget`:

- Show as configured when a URL is set (either via site settings or env)
- Health check: HTTP GET to `{build_server_url}/health` with Bearer auth ŌĆö green if 200, amber if configured but unreachable, grey if not configured

---

## Phase 3 ŌĆö CI build pipeline

Update `.github/workflows/deploy.yml`:

1. Add a build step after checkout that runs `npm ci && npm run build` (Vite admin theme + public assets)
2. Add a step that runs `php artisan build:public` inside the build context (requires the build server to be accessible from CI ŌĆö use the production build server URL and API key from GitHub secrets: `BUILD_SERVER_URL`, `BUILD_SERVER_API_KEY`)
3. The compiled assets (`public/build/`) are baked into the Docker image

Update the `Dockerfile`:

- Remove Node.js installation from the production app stage (lines installing `nodejs` via nodesource)
- Keep the `node-builder` multi-stage build for CI asset compilation
- The final production image should have no Node.js runtime
