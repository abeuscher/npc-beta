# 070. Admin User Invitations

Staff invite new admin users by supplying a name, email address, and role. The system creates the user record, sends an invitation email containing a secure one-time token link, and forces the invited user to set their own password before gaining access to the admin panel.

---

## Design decisions (resolved before starting)

**Invitation flow:** Click link in email → set-password page → auto-login → land in admin dashboard. No separate login step after setting the password.

**Token expiry:** 48 hours from the time the invitation is sent, AND invalidated immediately on first use. Both conditions apply.

**Resend / revoke:** Yes — a Resend action and a Revoke action, both on the User edit page. Resending generates a fresh token and resets the expiry clock. Revoking deletes the pending token without deactivating the user.

---

## Database

Add an `invitation_tokens` table:

| Column | Type | Notes |
|--------|------|-------|
| `id` | UUID PK | |
| `user_id` | UUID FK → users | cascade delete |
| `token` | string, unique | cryptographically random, hashed in DB |
| `expires_at` | timestamp | 48 hours from creation |
| `accepted_at` | timestamp, nullable | set on use; token is invalid once non-null |
| `created_at` / `updated_at` | timestamps | |

One pending token per user at a time — creating a new invitation deletes any existing unaccepted token for that user before inserting the new one.

Update `docs/schema.md` after writing the migration.

## Invitation action

Add an **Invite User** action to the `UserResource` list and edit pages. The action opens a modal to confirm (or optionally adjust) the role, then:

1. Creates or updates the `User` record (name, email, role, `is_active = false`).
2. Deletes any existing unaccepted token for this user.
3. Generates a cryptographically random token, stores a hashed copy in `invitation_tokens`.
4. Sends the invitation email.
5. Shows a success notification.

The action should not be available on a user record that already has an active (accepted, non-expired) session — i.e. a user who has already logged in at least once. Read how to detect this before implementing.

---

## Invitation email

Use the existing mail system (Resend). The email contains:

- A greeting using the invitee's name.
- A brief explanation of what the link does.
- The invitation URL: `/admin/invitation/{plain-text-token}` — the plain token is passed in the URL; the hashed version is stored in the DB.
- An expiry notice ("This link expires in 48 hours.").

Read the existing system email templates and mail classes before writing anything to match the conventions already in place.

---

## Set-password page

A public route (no auth required): `GET /admin/invitation/{token}`.

On load:
- Look up the hashed token in `invitation_tokens`.
- If not found, expired, or already accepted: render an "invalid or expired invitation" error page. Do not reveal which condition applies.
- If valid: render a simple set-password form (password + confirm password).

On submit:
- Validate the password (minimum 12 characters, confirmed).
- Hash and save to the user record.
- Mark `invitation_tokens.accepted_at = now()`.
- Set `users.is_active = true`.
- Log the user in (create session).
- Redirect to the admin dashboard.

The set-password page should not use the Filament panel — render it as a plain Laravel Blade view consistent with the admin login page style (or as close as reasonable without over-engineering).

---

## Resend and Revoke actions

Both live on the `UserResource` edit page, visible only when a pending (unaccepted, non-expired) invitation token exists for that user.

- **Resend**: deletes the old token, generates a new one, sends the email again, shows confirmation.
- **Revoke**: deletes the token, shows confirmation. Does not deactivate the user.
