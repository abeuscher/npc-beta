## What was built

### Feature 1 — Custom JSON-LD slot (augmenting `SeoMetaGenerator`)

- `SeoMetaGenerator::forPage()` now returns a `custom_json_ld` array (0–2 breakout-safe JSON strings) alongside the auto graph. New private `customGraphs()` resolves the **site-wide** `custom_json_ld` `SiteSetting` + the **per-page** `custom_fields['json_ld']`; `safeJsonLd()` validates each (`json_decode` → reject if not a JSON object/array) and re-encodes with `JSON_HEX_TAG | JSON_UNESCAPED_SLASHES | JSON_UNESCAPED_UNICODE`.
- **Injection safety:** `JSON_HEX_TAG` escapes every `<`/`>`, so a `</script>` smuggled into the value becomes inert (`</script>`) and cannot break out of the `<script>` element. Invalid JSON is dropped at render, never emitted raw. **Not** routed through `HtmlSanitizer` (it strips `<script>` — the wrong tool here).
- **Render:** `layouts.public` emits each custom graph as a **separate** `<script type="application/ld+json">` after the existing auto block.
- **Form fields:** site-wide field in CmsSettingsPage → *Search Engine Visibility* (mount fill + both `save()` and `persistSection('search-visibility')` paths); per-page field in the page *SEO* section (`HasPageBuilderForm`, `custom_fields.json_ld`). Both gated by a new `App\Rules\ValidJsonLd` rule (blank OK; else must be a JSON object/array — mirrors the render gate so the operator gets immediate feedback). Site-wide `custom_json_ld` added to the `SiteSettingsBundlePolicy` allow-list (SEO group) for export/import round-trip.

### Feature 2 — Blockquote + citation (verification, control already shipped)

- No source change: the native blockquote toolbar control is already present in both editors (`resources/js/admin/quill-editor.js`, `resources/js/page-builder-vue/components/fields/RichTextField.vue`), and `HtmlSanitizer::ALLOWED_TAGS` already carries `blockquote` + `cite`. Confirmed and locked in with tests: a sanitizer `blockquote with cite` round-trip + an end-to-end render-shape test (a blockquote with `<cite>` authored in a richtext widget survives save-sanitize + render). No `<cite>` toolbar control in v1.

### Feature 3 — Responsive `sizes` from column layout

- `PageBlockRenderer::deriveColumnSizes()` parses the layout's `grid_template_columns` into per-column `fr` fractions and emits a `sizes` string per column (e.g. `(max-width: 768px) 100vw, 60vw` for the 3fr column of a `2fr 3fr` grid; the mobile clause mirrors the `.layout-grid` collapse container query at `$bp-md` = 768px, omitted when `collapse_mobile` is false). Non-`fr` tracks (px/%/auto/repeat/minmax) and single-column layouts return nothing → the partial's `100vw` default.
- Threaded through: `renderWidgetBlock($pw, ?string $columnSizes)` → `WidgetRenderer::render(..., ?string $columnSizes = null)` (added to `$templateVars`) → the Image widget template, which passes a **concrete** `:sizes="$columnSizes ?? null) ?: '100vw'"` to `x-picture`. The derived value ignores grid gaps and the site-container cap, which only makes it err slightly large — a safe direction for `sizes`.

### Off-topic fix (owner-reported mid-session) — Hero "Small" min-height

- The Hero widget's "Small" (16rem) min-height didn't take effect. Root cause: the generic block wrapper (`page-widgets.blade.php`) carries `widget--hero` (it's `widget--{handle}`, handle = `hero`), the **same** class the Hero template's own root uses — so the base `.widget--hero { min-height: 24rem }` in `_custom.scss` pinned the **outer wrapper** at 24rem while the `hero--height-*` modifiers only sit on the inner div. "Small" shrank the inner div but the wrapper stayed ≥24rem. Fix: removed the bare base `min-height` (the inner div always carries either a `hero--height-*` class or `hero--fullscreen`, so coverage stays complete). SCSS-only + comment; `npm run build`.

## Security considerations

The one security-sensitive surface is the operator-authored JSON-LD. It is treated as **untrusted-but-not-HTML**: validated as JSON (object/array only; anything else dropped) and re-encoded with `JSON_HEX_TAG` so the emitted string carries no raw `<`/`>` — a `</script>` breakout attempt cannot terminate the surrounding `<script>` element, and the value still round-trips when decoded. Covered by Pest (site-wide + per-page render, a `</script>` breakout-escape assertion proving no raw angle brackets survive, invalid-JSON and bare-scalar rejection, and the `ValidJsonLd` rule's accept/reject cases). It is deliberately **not** routed through `HtmlSanitizer` (which would strip `<script>`). The blockquote work introduces no new tags or attributes (already in the allow-list); the responsive-`sizes` work is render-only computed output. Net no new untrusted-input path beyond the JSON-LD slot, which is gated.

## Deferred decisions

- **Custom `<cite>` Quill blot / blockquote attribution control** — declined for v1 (marginal SEO payoff, no native Quill blot). The toolbar blockquote control and sanitizer `<cite>` allowance already exist if a future session wants to build the attribution UI.
- **Responsive `sizes` container-cap refinement** — the derived `{fraction}vw` ignores the `.site-container` max-width, so on ultra-wide viewports it over-estimates (downloads a larger source than strictly needed — safe, just not optimal). A `(min-width: …) {fraction × container}px` clause would tighten it; deferred as a nicety.
- **`sizes` derivation scope** — applied to the Image widget on the public layout-grid render path only. The page-builder preview and record-detail/dashboard slots keep `100vw` (no column context plumbed there); invisible divergence (`sizes` only affects srcset source choice). Extend later if wanted.
