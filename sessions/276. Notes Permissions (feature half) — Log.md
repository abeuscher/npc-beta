### Iteration `/1` — `edit_others_note` permission registration (`b8da8a3`)

Single `PermissionSeeder.php` edit landed the standalone `edit_others_note` permission alongside the existing standalone block. Granted to `developer` by default; `super_admin` gets it free via the `Gate::before` bypass at [AuthServiceProvider.php:40](app/Providers/AuthServiceProvider.php#L40). `crm_editor`, `volunteer_coordinator`, `cms_editor`, `event_manager`, `treasurer`, `blogger`, and no-role users do *not* get it.

**Role-assignment design call surfaced at session start.** The session prompt had said "grant to super_admin and admin", but no `admin` role exists in the codebase — confirmed prompt drift, not a missing role. The roles with full `note` permissions today are `crm_editor`, `volunteer_coordinator`, `developer`. User direction: only `developer` gets the override; the toggle is meaningful precisely because operators with full note caps (crm_editor / volunteer_coordinator) should be restricted to their own notes when it's on.

Companion test surface at `tests/Feature/NotesPermissionsTest.php` (6 tests at /1; extended to 21 at /2): permission exists, developer + super_admin pass, crm_editor + volunteer_coordinator + cms_editor + no-role deny.

Also lifted into the Code Review track at `sessions/tracks/code-review-and-cleanup.md` (Cycle 3 deliverable): boolean SiteSetting / config-shape consistency audit, surfaced when scoping the toggle's storage shape. The codebase has both `'true'`/`'false'` strings (`horizon_enabled`) and `'1'`/`'0'` patterns floating around; codification is a track item.

### Iteration `/2` — `NotePolicy::update` + `::delete` gates + Timeline rewire (`83b3f4d`)

Three-step shape on both `NotePolicy::update` and `NotePolicy::delete`:

1. Outer `update_note` / `delete_note` capability — if missing, deny (today's check).
2. Read `notes_edit_only_by_creator` SiteSetting; if off, fall through to allow (today's behaviour).
3. If on, require `auth()->id() === $note->author_id` OR `auth()->user()->can('edit_others_note')`.

Storage shape follows the `horizon_enabled` local convention — string `'true'` / `'false'`, default `'false'`, compared with `=== 'true'` in the policy. The session prompt's `'1'`/`'0'` shape was set aside in favor of the local convention.

**Timeline composability fix.** [`app/Filament/Concerns/RecordTimelinePage.php`](app/Filament/Concerns/RecordTimelinePage.php) had four call sites in `editNoteAction()` and `deleteNoteAction()` that checked `auth()->user()->can('update_note')` / `'delete_note'` directly — capability-only, bypassing the policy. All four (the `->hidden()` callbacks and the `->action()`'s `abort_unless`) now invoke the policy via `$user->can('update', $note)` / `'delete'`. Without this rewire the new SiteSetting gate would not have enforced on the Contact / Organization Timeline — the dominant edit surface for notes.

Test surface: 15 new tests on `NotesPermissionsTest`. Four-cell update matrix × override + four-cell delete matrix × override = 10 matrix tests, plus 3 outer-cap baseline tests, plus 1 super_admin Gate-bypass test, plus 1 default-toggle-state preservation test.

### Iteration `/3` — Settings toggle + UI gating + Playwright + help doc (`74b4aa2`)

**`GeneralSettingsPage`** gained a "Notes" section (super-admin gated, matches the rest of the page's super-admin sections) with one `Toggle` — "Restrict note edits to author" — wired to `SiteSetting::set('notes_edit_only_by_creator', 'true' | 'false')` via both the bulk `save()` path and the per-section `saveSection('notes', 'Notes')` path.

**Timeline UI gating.** `RecordTimelinePage::getTimeline()` decorates each note item with `can_edit` and `can_delete` booleans computed via `$user->can('update' | 'delete', $note)`. The `record-timeline.blade.php` template's Edit and Delete buttons now wrap in `@if ($item->can_edit ?? false)` / `@if ($item->can_delete ?? false)` guards. UX layer matches the protection layer per the standing security-over-usability rule — no more inert-but-visible buttons.

**Help doc.** [`resources/docs/settings-general.md`](resources/docs/settings-general.md) gained a new "Notes" section under "System Page Content" describing the toggle, default, and override-permission interaction.

The `Bash(git merge:*)` deny entry in `.claude/settings.json` was blocking *every* `git merge` invocation, including legitimate `git merge origin/main` on a feature branch (the standard pattern for keeping a long-running iteration branch caught up with main). ## SiteSetting added

- **Key:** `notes_edit_only_by_creator`
- **Type:** stored as `'string'` (the DB-default for `SiteSetting::set` which doesn't pass a type argument); values are `'true'` / `'false'` string literals.
- **Default:** `'false'` (preserves pre-276 behaviour — no row created until the operator flips the toggle on for the first time).
- **Admin-form location:** `GeneralSettingsPage` → "Notes" section (super-admin gated).
- **Read shape:** `SiteSetting::get('notes_edit_only_by_creator', 'false') === 'true'` (matches the `horizon_enabled` precedent on the same page).

## Security considerations

The protection layer is `NotePolicy::update` / `::delete`. Three composing layers were verified end-to-end:

- **Policy-level** — extended both methods with the toggle + override gate. 21 Pest tests at `NotesPermissionsTest` cover the full matrix (toggle off / on × author / non-author / non-author-with-override / no-update_note-cap × update / delete), plus a baseline preservation test that the default toggle state matches today's behaviour, plus a super_admin Gate-bypass test for both methods. Outer `update_note` / `delete_note` capability still fires regardless of toggle state.
- **Admin UI** — Timeline edit/delete affordances rewired to call the policy (`->can('update', $note)` instead of `->can('update_note')`). Confirmed via the new Playwright spec that operator B can edit operator A's note when the toggle is off but cannot when it's on, and that granting `edit_others_note` reopens the edit path. Filament's `EditAction` / `DeleteAction` on `NoteResource` itself already routed through the policy (resource-level row actions read the resource's `model` policy) — no change needed there.
- **Action layer** — `RecordTimelinePage::editNoteAction()` and `deleteNoteAction()` both `abort_unless(... ->can('update'|'delete', $note), 403)` inside the action callback. Even if the policy-gated `->hidden()` were bypassed (e.g. by a hand-crafted `mountAction` call from the browser console), the action would 403.

**Intentionally unchanged:** Filament `EditAction::make()` / `DeleteAction::make()` on `NoteResource.php` — those already inherit `NotePolicy::update` / `::delete` via Filament's default resource-action policy hook. The 273 W8 #3 FM contract version-stamp parity test continues to fire; no FM controller paths touched. Portal `contact_id` scoping preserved — no portal code touched this session.

**Storage-shape audit lifted to Code Review track Cycle 3** — there's no benefit to two boolean SiteSetting shapes (`'1'/'0'` vs. `'true'/'false'`) coexisting; the track doc carries the audit item for codification across the codebase.
