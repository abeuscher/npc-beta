## What was built

### Phase 1 — Baseline reconnaissance

- Captured baseline screenshots of the Quill toolbar on both surfaces (Filament form field on `EmailTemplateResource`, Vue `RichTextField` in the page builder) in light + dark admin modes, via a throwaway Playwright script at `/tmp/quill-shots/capture.mjs`.
- Reference image (`/mnt/e/Clients/nonprofitcrm/test_import_data/wysiwyg toolbar.jpg`) held as the visual target: two-row compact toolbar, light-grey chrome, grouped buttons with frames, chip-shaped dropdowns.

### Phase 2 — CSS target surface decision

- Chose Option A (append to `public/css/admin.css`). Kept all admin-side chrome in one file.
- Reordered `renderHook('panels::head.end', …)` calls in `AdminPanelProvider.php` so `admin.css` loads **after** the Quill CDN stylesheet — eliminates equal-specificity cascade ties where Quill's defaults would otherwise win.

### Phase 3 — Light-mode styling pass

- Rewrote the Quill toolbar section in `public/css/admin.css` to match the reference:
  - Toolbar strip chrome (`--np-control-bar-bg`), rounded corners, flex-wrap grouping.
  - Group frames via `.ql-formats:not(:has(> .ql-header, > .ql-font, > .ql-size))` — lighter fill (`--np-control-group-bg`) with thin border, covers multi-button groups (B/I/U/S, lists, link+image), single-button groups (blockquote, clean), and icon-picker groups (colour, background, align). Text-picker groups (header / font / size) keep their own chip chrome instead.
  - Compact button chrome: 28×24 rect, `3px` radius, transparent at rest, subtle `rgba(17,24,39,0.06)` hover tint, `#e5e7eb` + inset shadow on active.
  - Text-picker chip (the "Normal" / Format dropdown): white fill, bordered, `26px` tall, custom CSS `::after` chevron using `currentColor` so it tracks the chip's text colour across rest / hover / expanded. Default two-triangle Quill SVG hidden for these pickers only.
  - Icon-picker labels (align, colour, background) flattened to button-like behaviour so they inherit the framed-group chrome.

### Phase 4 — Dark-mode variant

- Initial dark block rewritten inline with the light block (same selectors, colour tokens via `.dark { --np-control-* }`). Later collapsed to token-driven behaviour — see Phase 6 extraction.
- Residual dark-mode rules kept for editor-body colour and Quill tooltip (`.ql-tooltip` + input inside it) which sit outside the control-bar vocabulary.

### Phase 5 — Cross-surface verification

- Verified identical rendering across: `EmailTemplateResource` (existing editor in use), `GeneralSettingsPage` (Dashboard welcome + system-page content editors), `EventResource/create` (description + meeting details), and Vue `RichTextField` in the page-builder widget inspector.
- Fixed `RichTextField.vue`: removed the outer container border so the "Edit site styles →" link is naked above the Quill box, leaving only Quill's own chrome wrapping the editor.

### Phase 6 — Pilot-to-pattern extraction (Option A: CSS custom properties)

User agreed to lift the inline values into a shared token layer. Created `:root` + `.dark` blocks at the top of the Quill section with 19 tokens:

- `--np-control-bar-bg`, `--np-control-group-bg`, `--np-control-border`, `--np-control-chip-border`
- `--np-control-border-hover`, `--np-control-border-active`
- `--np-control-hover-tint`, `--np-control-active-bg`, `--np-control-active-shadow`
- `--np-control-icon-default`, `--np-control-icon-active`
- `--np-control-chip-bg`, `--np-control-chip-text`, `--np-control-chip-text-active`
- `--np-control-menu-bg`, `--np-control-menu-shadow`
- `--np-control-radius`, `--np-control-radius-sm`, `--np-control-transition`

Quill CSS refactored to consume the tokens throughout. Dark-mode block collapsed to ~12 lines of residuals that don't flow through the vocabulary.

### Scope creep — user-directed in-session work

Several adjacent surfaces were brought into the session as the Quill pilot exposed their gaps:

1. **ColorPicker primitive** (`resources/js/page-builder-vue/components/primitives/ColorPicker.vue`):
   - Consumes `--np-control-*` tokens throughout; dark-mode support appeared for free (previously hardcoded light colours only).
   - Popover converted from `position: absolute` to inline flow — opening the picker now pushes content below down the column (matching `GradientPicker` behaviour).
   - Trigger border stepped up to `--np-control-border-hover` at rest so it reads against the white inspector body.
   - New `panelOnly` prop: when true, hides the built-in trigger + disables the click-outside listener + renders the panel unconditionally when mounted. Parent fully controls visibility via `v-if`. Enables the split-trigger pattern in BackgroundPanel.

2. **GradientPicker primitive** — same click-outside-disable treatment when `compact` is set, so it can live as a tab in a parent-controlled pattern.

3. **BackgroundPanel three-tab architecture** (`resources/js/page-builder-vue/components/appearance/BackgroundPanel.vue`):
   - Three sibling swatch handles in one row: Color, Gradient, Image.
   - `openPanel` ref drives which of the three corresponding panels renders below the row. Initial value by presence priority: `image > gradient > color` (color default when nothing is set).
   - Clicking an active handle collapses the tab to `null`; clicking another switches. Active handle gets `--np-control-border-active` border + 1px ring so it visually connects to its panel.
   - The "Use current page's header image" checkbox moved from the main view into the Image panel body.
   - New Image panel layout: Fit dropdown + 9-point alignment on the top row, override checkbox on the second row, full-width image preview or "Click to upload" tile on the third.

4. **GradientPicker — removed per-layer `css_override` input.** Layer-level CSS can't be resolved predictably against other background layers. The `GradientLayer.css_override` field + `GradientComposer` sanitisation path remain for backward compatibility with existing stored data; only the authoring surface is gone. Deferred-decision note added to the "CMS Style System — Full Widget Styling" outline stub: when scoped, decide whether widget-scoped `[data-widget="{uuid}"]` CSS should replace all per-primitive escape hatches including the orphan field.

5. **Inspector dual-pane architecture** (`resources/js/page-builder-vue/components/InspectorPanel.vue`):
   - Two stacked `<section>` panes inside a new `.inspector-panel__panes` wrapper. Wrapper has `flex: 1 1 0` against the header, `background: var(--np-control-bar-bg)` so empty space below a collapsed pane matches the toolbar strip.
   - Top pane tabs: **Content** / **Presets** / **Widget Settings** (widget-specific).
   - Bottom pane tabs: **Background** / **Text** / **Margin & Padding** (universal appearance).
   - `InspectorTabs` generalised: takes a `tabs: {id, label}[]` prop + exposes a right-side toolbar slot for the expand/collapse button.
   - Pane state: two independent `topCollapsed` / `bottomCollapsed` booleans. Each pane's expand button only toggles its own collapse — sibling is not coupled. Each pane has `max-height: 50%` so collapsing one leaves empty space rather than feeding the sibling (earlier three-state model that force-coupled the panes was replaced on user feedback).
   - Clicking a tab in a collapsed pane auto-uncollapses it.
   - Tab labels truncate with ellipsis + `title` tooltip in the narrow column.
   - Tabs and pane bodies adopt the `--np-control-*` vocabulary — tab strip bg, tab borders, active-tab chrome, pane body bg/border all route through the tokens. Matches the Quill visual language.
   - `App.vue` inspector container changed from `max-height` to `height: calc(100vh - 2rem)` so the flex-based pane layout has a definite height to divide.
   - Orphaned `html.dark .inspector-panel__body` override removed from `admin.css` (selector no longer exists after the rename to `.inspector-pane__body`; dark mode now flows via tokens).
