### Phase 2 тАФ Create Mailing List (built then removed)

- Replaced the stub `createMailingList` action with a working modal (name, year, threshold тАФ pre-filled from page state)
- On submit: queried eligible donor contact IDs using `HAVING SUM(amount) >= ?`, generated a server-side `id IN (...)` raw_where clause, created a `MailingList` record, and redirected to the new list's edit page
- During testing discovered a fundamental design conflict: `MailingListQueryBuilder` universally gates all list results on `mailing_list_opt_in = true`, which is correct for marketing/MailChimp use but silently drops donors who haven't opted in to marketing тАФ making the list unreliable for tax receipt purposes
- Decision: feature removed entirely. A treasurer generating a donor mailing list for receipts would have no indication that opted-out donors were missing, creating a legal compliance risk. The existing receipt send workflow handles the obligation correctly without this feature. Donors who also happen to be opted in can be targeted via the standard Mailing Lists filter UI
- Note: this session ran a full circle on this feature тАФ designed, built, tested, and correctly pulled back
