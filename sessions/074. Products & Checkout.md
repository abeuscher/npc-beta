# 074. Products & Checkout

Build a lightweight product catalogue with fixed inventory, variable price tiers, and a waitlist — backed by Stripe Checkout. This is a resource allocation system, not a commerce system: "products" are named entitlements (plots, slots, memberships, packages) with no shipping, no SKUs, and no physical fulfilment. Payment is handled by Stripe; the app records who holds what.

---

## Design decisions (resolved before starting)

**What a product is:**
A named thing an organisation offers in finite quantity. Examples: garden plots, parking spaces, naming rights packages, CSA shares. Not physical goods requiring shipping or inventory tracking beyond a simple count.

**Price tiers:**
One product can have multiple price tiers (e.g. Full / Reduced / Free). Each tier is a separate row in a `product_prices` table with a label and an amount. Each tier produces its own Stripe Checkout path. All tiers draw from the same inventory pool. A zero-amount tier bypasses Stripe entirely and creates an allocation directly.

**Allocation timing — Option A:**
No pending/pre-allocation state. An `allocation` record is created only on `checkout.session.completed` (or immediately for zero-price tiers). This means inventory is not held during an active Checkout session. For the scale of organisations this serves, the theoretical double-booking race condition is acceptable and handled manually if it occurs.

**Waitlist:**
When a product's active allocation count reaches its capacity, the purchase UI is replaced with a "Join Waitlist" button. This creates a `waitlist_entries` row — no payment, no Checkout session. Waitlist processing is entirely manual: an admin sees the list, contacts the next person, and processes their allocation through the normal flow. Automated notification and slot-holding are out of scope.

**Contact record visibility:**
Allocations appear in the contact activity log as a timeline entry (same pattern as other logged events). No separate allocations tab on the contact record is needed for v1.

**Inventory view:**
Each product's admin detail page shows a relation manager listing all allocations (contact, price tier, date, status) and a separate tab for the waitlist. This is the canonical place to see who holds what.

**Stripe metadata:**
When creating a Checkout session, include `metadata: { subject_type: 'App\\Models\\Allocation', subject_id: '<uuid>' }` — consistent with the convention established in session 073. The allocation record is created *after* the webhook confirms payment; the subject_id in metadata is a pre-generated UUID written to the allocation at creation time via the webhook.

Wait — this conflicts with Option A (no pre-allocation). Resolve as follows: the Checkout session metadata carries `product_price_id` and `contact_id` instead of a subject. The webhook handler creates the allocation using those values. No allocation UUID is needed in metadata.

**Public-facing:**
Products with a `published` flag are visible on the public portal. Each published product gets a simple page showing description, available price tiers, and current availability (or waitlist status). Portal authentication is required to purchase or join the waitlist.

## Phase 1 — Data model and admin UI

1. Create migrations:
   - `products`: `id` (uuid), `name`, `description` (text, nullable), `capacity` (integer), `status` (string: draft/published), `sort_order` (integer, default 0), timestamps
   - `product_prices`: `id` (uuid), `product_id` (FK→products, cascade), `label` (string, e.g. "Full price", "Reduced", "Free"), `amount` (decimal 10,2), `stripe_price_id` (string, nullable), `sort_order` (integer, default 0), timestamps
   - `allocations`: `id` (uuid), `product_id` (FK→products), `product_price_id` (FK→product_prices), `contact_id` (FK→contacts, nullOnDelete), `status` (string: active/cancelled), `occurred_at` (timestamp), timestamps
   - `waitlist_entries`: `id` (uuid), `product_id` (FK→products), `contact_id` (FK→contacts, nullOnDelete), `status` (string: waiting/notified/converted/cancelled), `created_at` (timestamp)

2. Create Filament resources:
   - `ProductResource` — list, create, edit. Fields: name, description, capacity, status, sort_order. Relation manager for price tiers (inline, sorted by sort_order). Relation manager for allocations (read-only: contact, price tier, occurred_at, status). Relation manager for waitlist (contact, status, created_at).
   - No standalone `AllocationResource` or `WaitlistResource` — these are managed through the product.

3. Add `MorphMany allocations()` to relevant models if needed. Add `AllocationObserver` that writes to the activity log on `created` and `cancelled`.

4. Update `docs/schema.md`.

## Phase 2 — Stripe price sync

Each `ProductPrice` with a non-zero amount needs a corresponding Price object in Stripe. When a price is created or its amount changes, sync to Stripe and store the returned `price.id` in `stripe_price_id`.

1. On `ProductPrice` save (observer or model event): if `amount > 0` and `stripe_price_id` is empty, create the Stripe Price via the API and store the ID. If the amount changes on an existing price, archive the old Stripe Price and create a new one (Stripe prices are immutable).
2. If `amount = 0`, `stripe_price_id` stays null — zero-price allocations bypass Stripe entirely.
3. Read the existing Stripe integration (AppServiceProvider, config) before writing any API calls.

## Phase 3 — Checkout flow

1. Add a portal route: `GET /{portal_prefix}/products/{product:slug}` — shows the product page with available price tiers or waitlist status.
2. Add a portal route: `POST /{portal_prefix}/products/{product:slug}/checkout` — creates a Stripe Checkout session for the selected price tier and redirects the user to Stripe. Middleware: portal auth required.
   - Check capacity before creating the session. If full, return the user to the product page with a message.
   - Checkout session `success_url` and `cancel_url` return to the portal product page.
   - Metadata: `{ product_price_id: '...', contact_id: '...' }`
3. Add a portal route: `POST /{portal_prefix}/products/{product:slug}/waitlist` — creates a `waitlist_entries` row if capacity is full and the contact is not already on the list. Middleware: portal auth required.
4. Update `StripeWebhookController` to handle `checkout.session.completed` for product purchases: read `product_price_id` and `contact_id` from metadata, create the `Allocation` record, mark status active.
5. Zero-price tier: submit handler creates the allocation directly without a Stripe session.

---

## Phase 4 — Public product page

Add a public-facing product page to the portal. Keep it minimal:
- Product name, description
- If capacity available: list price tiers with labels and amounts, each with its own checkout button
- If at capacity: waitlist signup form (portal auth required)
- If the contact already holds an allocation: show a confirmation message, no purchase option
- If the contact is already on the waitlist: show their position, no duplicate entry

---

## Security checklist

Before closing:

- [ ] All checkout and waitlist routes require portal authentication — no anonymous purchases.
- [ ] Checkout session is created server-side — price and product are never taken from form input, only from the database.
- [ ] Contact ID in metadata is taken from the authenticated portal session, never from user input.
- [ ] Capacity check is performed server-side immediately before Checkout session creation.
- [ ] Zero-price allocation path creates the record server-side, not via any client-submitted amount.
- [ ] Stripe Price sync does not expose the secret key or any API response beyond the price ID.

---

## Out of scope

- Automated waitlist notification and slot-holding (manual admin process for v1)
- Product expiry / term-based allocations (membership handles subscriptions)
- Shipping, fulfilment, or physical inventory tracking
- Admin-initiated allocations without payment (can be added later if needed)
- Stripe product/price archival UI in the admin panel
