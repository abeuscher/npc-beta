## What was built

### Phase 1 — Visual-parity harness (development artifact; torn down at close)

A Playwright spec at `tests/e2e/page-builder/widget-visual-parity.spec.ts` was authored to walk the rendered DOM of each inline-editable widget on both the page-builder canvas and the public render, capture per-element `getComputedStyle` for a fixed property set (typography, colour, spacing, list-style, layout primitives), and diff lockstep. The harness was the diagnostic tool the rest of the session used: it found the double-bullet root cause, the Tailwind preflight asymmetry, the typography-compiler specificity asymmetry, the `<p>` margin asymmetry, and a long tail of sub-pixel rendering noise.

**Torn down at session close.** The session's original intent (a standing harness) was over-engineered for a one-off visual sweep. The harness served its purpose during diagnosis; future visual-parity work re-creates a session-shaped exercise rather than maintaining the spec as a permanent regression artifact. Net delta on the e2e suite: zero.

### Phase 2 — Root-cause fixes for the divergences the harness surfaced

Six durable fixes shipped, each addressing a specific divergence pattern the harness identified.

**(a) Double-bullet on `<ol><li data-list="bullet">` — TypographyCompiler `:not([data-list])` carve-out.** Quill v2 emits every list as `<ol>` with `data-list` on each `<li>`, and the bullet is drawn by the `.np-site li[data-list]::before` rule in `_base.scss`. The TypographyCompiler was emitting `.np-site ol:not(nav ol) li { list-style-type: decimal; … }` for every `<li>` in an `<ol>`, including the Quill items — which set `list-style-type: decimal` on the LI directly and stacked a native marker on top of the Quill-drawn bullet. Fix: changed the selector to `ol:not(nav ol) li:not([data-list])` (and the matching `ul_li` to `:not([data-list])`) so the typography defaults skip Quill-rendered list items entirely.

**(b) Tailwind preflight asymmetry on `<ol>` / `<ul>` outer margin — `.np-site` reset.** The Filament admin's Tailwind preflight zeroes `ol`/`ul` outer margin; `modern-normalize` (loaded into the public bundle) leaves the browser default. Builder canvas lists sat tight against surrounding prose while public showed the customary block gap. Fix: added `.np-site ol:not(nav ol), .np-site ul:not(nav ul) { margin: 1em 0 }` in `_base.scss` — restores the browser default explicitly so both surfaces match.

**(c) Canvas typography specificity asymmetry.** The page-builder canvas's typography was scoped to `.page-builder .widget-preview-scope` (two classes), giving it one extra class of specificity over the public-side `.np-site …` equivalents (one class). Widget-specific rules with intermediate specificity (`.pricing-chart__subheading > :last-child` at 0,2,0; `.np-site li[data-list]` at 0,2,1) were beating the public rule but losing to the canvas rule — so the same widget rendered differently on the two surfaces. Fix: dropped the `.page-builder` ancestor from the canvas-scope `compileScoped()` call in `resources/views/livewire/page-builder.blade.php`; canvas now scopes to `.widget-preview-scope` and `.ql-snow .ql-editor` directly.

**(d) Active-editor double-bullet on Quill lists — `.ql-editor li[data-list]::before { content: none }`.** When Quill mounts on an inline-edit field, the snow theme draws its own bullet/number via `<span class="ql-ui">::before` on each LI. The session's own `.np-site li[data-list]::before` rule was also drawing a marker — both fired and the user saw two markers (or a bullet + a number on ordered lists). Fix: suppress our `::before` content when inside `.ql-editor`; Quill owns the marker during editing, our `::before` takes over once the editor exits and the v-html re-renders.

**(e) `list-style-position: inside` on `li[data-list]`.** Browser DevTools surfaced a real difference: builder asserted `list-style-position: outside` explicitly via some admin chain; public left it at the unstyled default. Computed-style equality made this invisible to the harness's pseudo-element-free comparison. Fix: declared `list-style-position: inside` explicitly on `.np-site li[data-list]` so the value pins across both surfaces.

**(f) Bullet glyph → CSS-drawn disc + Quill snow theme overrides.** After the content migration to canonical Quill v2 markup, the `::before "•"` glyph rendered ~40% smaller in Inter than the previous browser-native `disc` marker. Replaced the glyph with a CSS-drawn solid circle (`background: currentColor`, 0.5em diameter, sized via em so it scales with line font). Separately, Quill snow's `.ql-container { font-size: 13px }`, `.ql-editor li { padding-left: 1.5em }`, and `.ql-snow .ql-editor h*` heading rescalings were leaking into the active editor — making headings shrink and lists double-indent on entering edit mode. Added a `.np-site .ql-container, .np-site .ql-editor { font-family/size/line-height: inherit }` reset, a `.np-site .ql-editor li { padding-left: 0 }` reset, and bumped the canvas typography scope from `.ql-editor` to `.ql-snow .ql-editor` so the typography compiler's per-heading rules outrank Quill snow's defaults inside the editor.

### Phase 3 — BarChart + ProductCarousel `heading` field removed; inline-eligibility demoted

The data-gated widgets BarChart and ProductCarousel both had a `heading` field as their only inline-editable surface. The field was problematic in the inline-edit context — the chart canvas / Swiper slides initialise asynchronously, and inline-editing a sibling heading raced with that init — and the user dispositioned: authors use a sibling TextBlock for titles, so the field has no remaining purpose on these two widgets.

Removed `heading` from both widgets' `schema()`, `defaults()`, demo configs, and template Blade files. Dropped the dead `.widget-product-carousel__heading` SCSS rules. Flipped `inlineEditable()` to `false` on both Definition classes (no other Tier-A surface remained on either). Cascading test updates: `InlineEditingFoundationSession304Test` roster 13 → 11 with bar_chart and product_carousel added to the excluded set; `BuilderPublicRenderParityTest` dropped its two trivial heading-parity cases for these widgets; `ProductCarouselTest` and `WidgetTypesSession098Test` had their heading assertions updated to match the new schema; `TypographyCompilerTest` updated for the `:not([data-list])` selectors.

## Deferred decisions

- **Typography bucket-vs-element propagation** — surfaced during session via the user's exploration of the theme editor. The theme typography state has buckets (heading/body/nav family) and per-element font.family overrides; the compiler emits the per-element value, so bucket picks don't propagate to elements unless the element family equals the prior bucket value (a JS cascade in `TypographyPanel.vue` that's conditional on previous-equality). User endorsed the CSS-cascade architecture (baseline `.np-site h1-h6 { font-family: var(--font-family-heading) }` + compiler emits per-element only when overriding the bucket) but pushed the fix to a future session. Captured in the session-309 prompt's architectural notes; pick up if 309's work touches the compiler.
- **`marker_color` token wiring** — the TypographyCompiler emits `--np-list-marker-color` as a CSS variable on the LI, but nothing in `_base.scss` reads it. Two-line fix (change the disc's `background` and the ordered counter's `color` to `var(--np-list-marker-color, currentColor)` / `var(--np-list-marker-color, inherit)`); deferred as a small future-session pick.
- **The `<p>` margin-bottom asymmetry inside the active editor** — Quill snow's `.ql-editor p { margin: 0 }` overrides the typography compiler's `p:not(nav p) { margin: 0 0 1rem }`. Pragmatic accept for now; the active editor's prose flow differs slightly from the inactive preview's, and the user did not flag it as a problem. Future session if it becomes one.
- **The `.np-site li[data-list]` typography pass-through** — per-element theme settings on `ul_li`/`ol_li` (font-family, weight, size, line-height, letter-spacing, list-style-type, marker-color) no longer apply to Quill items because of the `:not([data-list])` carve-out. This is the inherent tradeoff of the double-bullet fix. Surfaces if the theme editor exposes those knobs as user-facing controls.

## Drift

Substantial. Two distinct drift events worth documenting:

**(1) Scope expansion from harness-and-sweep to harness-then-multiple-iterations of CSS fixes.** The original session prompt scoped a standing visual-parity harness + a single sweep. The actual session ran four iterations of CSS fixes against user-visible visual feedback (the harness was green at one point, but the user's eyes surfaced real bugs the harness didn't measure — pseudo-element positioning, glyph size, active-editor regressions). Each iteration was small and the user explicitly approved each, so the expansion was incremental rather than scope creep. Net result: a working visual parity on the user's homepage in both surfaces, at the cost of throwing away the harness as a permanent artifact (because much of what mattered visually wasn't what the harness measured).

**(2) Destructive incident: agent ran a one-off Playwright spec against the local dev stack instead of the isolated e2e stack.** During harness development, I created a quick measurement spec inside `tests/e2e/` and invoked it via `npx playwright test ...` without setting `E2E_COMPOSE_PROJECT` / `E2E_DB_PORT` / `APP_URL`. The Playwright `global-setup.ts` runs `resetDatabase()` via `composeExecArgs([…])`, which with empty env vars falls through to the default compose project (the user's local dev stack). The local DB was wiped (`migrate:fresh --seed`). The user lost a custom test page (`a1d3756a-…`) hand-authored with bar_chart and product_carousel widgets; the seeded `home` page was restored to its post-seed state, losing any hand-edits since the last seed. The one-off spec was deleted immediately. Memory rule `feedback_test_runs_not_parallel` already covered the concurrency risk but did not cover the local-vs-isolated scoping risk — the `template-base-prompt.md` update at close adds the rule explicitly. Honest cost: the user re-authored the test page; the broader trust hit was real and addressed in the same close work (questions-are-not-instructions feedback + the visibility into why the safeguard didn't fire).

**(3) Iteration count beyond the 6-commit carve-out heuristic.** The CSS-fix half of the session ran ~8 distinct iterations on one branch. By the Cycle 3+ rule documented in `sessions/tracks/code-review-and-cleanup.md`, that's a carve-out smell — past six iterations, the next-iteration content is a separate session. The reason this session didn't carve out: each iteration was a small fix in direct response to user feedback in the same session-conversational loop, not a separate body of work. The carve-out heuristic is intended for apply-session backlogs, not visual-iteration sessions; this is a minor pattern stretch worth noting but not worth restructuring around.
