### Phase 2 — Contract-layer surface

- New `App\WidgetPrimitive\QuerySettings` value object — `readonly bool $hasPanel`, `readonly array $orderByOptions`, `readonly bool $supportsTags`. Static factory `QuerySettings::swctOrderByOptions(array $textFieldKeys)` builds the SOURCE_WIDGET_CONTENT_TYPE allowlist (user-mapped text fields plus `sort_order` / `created_at` / `updated_at` system columns).

- `App\Widgets\Contracts\WidgetDefinition` gains `querySettings(array $config): ?QuerySettings` returning `null` by default. Eight widgets override:
  - **Carousel, BoardMembers, LogoGarden, BarChart, Memos**: `swctOrderByOptions()` of contract-text-field union, `supportsTags: true`. Per-instance dynamic shape mirrors `dataContract()`'s field union for BarChart/BoardMembers/LogoGarden.

- `App\WidgetPrimitive\ContractResolver` extended:
  - `resolvePost()` honors user `order_by` / `direction` / `include_tags` / `exclude_tags`. Preserves the legacy `orderByRaw('COALESCE(published_at, created_at) DESC')` default ordering when neither user nor contract supplies an `order_by` (BlogListing test pin survives byte-equivalent).
  - `resolveEvent()` honors the same set; existing per-source allowlist replaced by the contract's `querySettings->orderByOptions` re-validation.
  - `resolveWidgetContentType()` honors the same set with one extra wrinkle: a new private `applySwctOrderBy()` distinguishes system columns (`sort_order`/`created_at`/`updated_at`, ordered via `orderBy`) from JSONB-keyed content-type fields (ordered via `orderByRaw("data->>'$col' $dir")` after a regex sanity gate). This unblocks Memos' `posted_at desc` default — a JSONB-keyed sort that was a no-op under the legacy resolver and would have errored under a naive `orderBy` port.
  - SWCT cache key now includes `sha1(serialize($contract->filters))` — two BoardMembers widgets pointing at the same Collection but with different `order_by` no longer share a cached query.
  - Tag filtering is uniform across all five taggable models via `whereHas('tags', fn ($q) => $q->whereIn('slug', $slugs))` and the `whereDoesntHave` mirror. Gated on `$contract->querySettings?->supportsTags`.
- `WidgetRenderer::render()` merges `$pw->query_config` into a copy of `$contract->filters` via a new private `mergeUserQueryConfig()` helper. The merge is restricted to the closed five-knob set via `array_intersect_key`; non-honored keys (e.g. EventsListing's `date_range`) remain immutable contract defaults. The merge replaces the contract object with a freshly-constructed copy carrying the merged filters; everything else (source, fields, model, resourceHandle, contentType, querySettings) is preserved verbatim.

### Notable findings (carried into 222's design decisions)

- **Memos exposed a JSONB-key vs system-column distinction in the SWCT order_by path.** The legacy resolver hardcoded `orderBy('sort_order')` and silently ignored Memos' `'order_by' => 'posted_at desc'` contract default. Honoring it under the new code triggered a Postgres "column does not exist" error — `posted_at` is a JSONB key in `collection_items.data`, not a real column. Fix landed inline as `applySwctOrderBy()`: system columns sort via `orderBy`, anything else (after a `[A-Za-z_][A-Za-z0-9_]*` regex sanity gate on top of the querySettings allowlist) sorts via `orderByRaw("data->>'$col' $dir")`. Carry-forward note: any future SWCT widget declaring a JSONB-keyed default order works automatically; allowlist values are the gate, not the storage shape.
- **`DashboardBuilderApiController` had a parallel `filterQueryConfig`** introduced by session 215 that the prompt didn't anticipate. Same legacy slot-strip mechanism, same fix. The session-outlines stub for 221 mentioned only `PageBuilderApiController::filterQueryConfig`. Discovered during the Phase 9 grep audit, fixed in the same session for symmetry. Carry-forward note: future cross-cutting cleanups in this area should grep both controllers, not just `PageBuilderApiController`.
- **Laravel's `validate()` filters out top-level keys whose only sub-keys don't match nested rules.** The legacy slot-keyed PUT-payload test failed initially because `validated['query_config']` was absent when the input was `{slides: {limit: 4}}` — none of the nested `query_config.limit`/`query_config.order_by` rules matched, so the `array_key_exists('query_config', $validated)` check was false. Switched to `$request->has('query_config')` for presence detection while keeping the nested validation rules for the security gate on per-knob values.
