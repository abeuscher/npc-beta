## The seam as it stands — drift the plan doesn't capture

Verify all of this at session start (drift rule), but this is the map as of 373:

- **The skip-Stripe-on-zero short-circuit already exists.** The public registration handler is `EventController::register` (`app/Http/Controllers/EventController.php:15`) — note there is **no** `EventCheckoutController` (the `docs/app-reference.md` "Event checkout (POST)" row is **stale**; a regression test even asserts the old split routes are gone). The pay-vs-confirm decision is `EventRegistrationQuantities::isPaid()` (`app/Services/EventRegistrationQuantities.php:23`), which is just `totalCents > 0`. A pure-$0 order creates the `EventRegistration` rows **directly as `status = 'registered'`, `source = HUMAN`, with no Stripe session** (`EventController.php:90-106`); a tier-less event does the same (`:68-86`). Tests already assert it (`MultiQuantityRegistrationTest.php:227`). → **This session locks it in with a regression test; it does not rebuild it.**

- **There is NO `is_complimentary` flag.** `price = 0` is the *only* signal for "free." The `ticket_tiers` table has `name, price (numeric, default 0), capacity, sort_order` — no comp column (`docs/schema/ticket_tiers.md`). The admin edits tiers via a Filament `Repeater` on `EventResource` (`app/Filament/Resources/EventResource.php:421`), fields `name / price / capacity` only. → **This session adds the label.**

- **Mixed comp+paid orders already behave as the plan wants.** A mixed order (one $0 comp tier + one paid tier) has `totalCents > 0`, so the whole order routes through Stripe with the $0 comp line included alongside the paid line (`EventRegistrationQuantities.php:87-114`; test `MultiQuantityRegistrationTest.php:207`). The plan says mixed orders "continue through Stripe with the $0 line item as today (no change)." → **No change.**

- **The confirmation-email timing bug — this is the substantive fix.** The thank-you (`App\Mail\RegistrationConfirmation`) is dispatched by `EventRegistrationObserver::created` (`app/Observers/EventRegistrationObserver.php:12-27`) with a **single guard — `! empty($registration->email)` — and NO status guard**. So today:
  - **Free/comp path** (rows created `registered`): email fires — but **once per tier line**, so a multi-tier order sends **N emails**.
  - **Paid path** (rows created `pending` before payment): the observer emails **at pending-creation, before Stripe confirms**. Abandon checkout and the buyer keeps a "confirmation" for a registration that stays `pending` forever (rows are deleted only on a Stripe *throw*, not on abandonment). The later webhook `pending → registered` flip is an *update*, and the observer has no `updated` hook — so no correcting email.
  - **Caution:** the same observer also **auto-creates the contact** and handles **mailing-list opt-in** on create (`ContactAutoCreationTest`, `MailingListOptInTest`). The fix must relocate *only* the email dispatch and preserve those side-effects. → **This session fixes the timing.**

Framing: `feedback_audit_integration_seams` taken to its conclusion — the headline behaviour is already wired; the value is a small named label plus a real bug the seam carries.

## What this session ships

Three things, all in scope:

1. **Comp-tier label.** Add `is_complimentary` (boolean, default false) to `ticket_tiers` (the session's one migration), a toggle in the `EventResource` ticket-tier repeater, and a "Complimentary" label in the public ticket picker. Behaviour stays **price-driven** — the flag is display-only; it does not itself make a tier free.

2. **Lock in the free/comp path.** A regression test proving a pure-$0 order (comp-only or free) still skips Stripe, creates rows `registered`, and sends exactly one thank-you. No behaviour change — a guard against future drift.

3. **Fix the confirmation-email timing** so the thank-you fires **once per order, at the right moment**:
   - **Free/comp path:** one email the moment the order is confirmed server-side.
   - **Paid path:** one email when the webhook flips the order `pending → registered` — **not** at pending-creation.
   - **Recommended approach:** move the `RegistrationConfirmation` dispatch **out of `EventRegistrationObserver::created`** (which fires per-row on *every* create path) and into the two confirm-points — the free-path registration controllers (`EventController::register` + the portal twin `Portal\EventRegistrationController::store`) and `StripeWebhookController::handleEventRegistrationCheckout` for the paid path — sending **one** email per order. Keep the observer's contact auto-create + mailing-list opt-in intact.
   - **Idempotency (paid path):** the webhook promotes only `status = 'pending'` rows, so a Stripe replay finds none and won't re-email — the fix inherits idempotency from the existing pending-filter; no new marker is strictly required (confirm at Open Gate).

## Implementation approach to confirm at the Open Gate

Scope is settled (the three items above). Surface the **approach** and confirm before going deep — the email relocation touches multiple create paths and is expensive to unwind (`feedback_verify_objective_outcomes`):

1. **Comp-flag data model.** `is_complimentary` column + repeater toggle + picker label (recommended), vs. driving the label purely off `price == 0` with no migration. Confirm — and whether the migration lands.
2. **Email-dispatch relocation.** Moving the send out of the observer into the free-path controllers + the paid webhook, one per order. **Audit the create paths this changes:** importer-created and admin-created registrations currently *also* email via the observer — a bulk import emailing every attendee is almost certainly unwanted, and relocating dispatch to the public entry-points fixes that too, but verify no path silently loses a *needed* email. This is the load-bearing call.
3. **"One email per order" anchor.** Paid rows share a `stripe_session_id`; free rows currently share no order key. Confirm how the free path identifies "the order" for a single send (e.g. send once after the create loop using the primary row — the `registration_confirmation` template is event-level, so any row of the order carries the right tokens).
4. **Idempotency.** Rely on the existing pending-row filter (recommended) vs. an explicit per-order marker. Confirm.
