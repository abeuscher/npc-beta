## The design contract is settled — do not relitigate

The spec resolved every open question during the 305 close-out:

- **Theme**: always dark, never follows the admin's light/dark mode (spec §4.1 / §M10).
- **Geometry, color tokens, icon set (Lucide), per-control state visuals**: §4.2 / §4.3 / §4.4 / §4.5 — fully pinned.
- **Controls (18 buttons + 7 separators in 8 groups, left-to-right by popularity)**: §4.6 / §4.7. Text style (dropdown) · Inline marks (B/I/U/S) · Block structure (bullet, numbered, blockquote) · Link · Alignment (4) · Color & highlight (2) · Insertions (image, heroicon) · Clear formatting.
- **Text-style dropdown**: §4.8 — closed and open states, each row in the theme's actual heading family + weight, size capped at 22px; no `text-transform` / `letter-spacing` propagation (decided for menu scannability).
- **Color / highlight popovers**: §4.9 — reuse the existing `ColorPicker.vue` in `panelOnly` mode (already exists); spec §6.1 / §K18 specify the arrow-key grid navigation enhancement on top.
- **Link popover**: §4.10 / §G — full field stack, page picker, mutual-exclusion rule, save/cancel/remove, focus capture/restore, Cmd/Ctrl+K shortcut.
- **Image + heroicon**: §4.11 / §4.12 / §F7 — reuse existing inline-image upload endpoint + existing `heroicon-picker.js`.
- **Clear formatting**: §4.13 / §F8 — inline marks only; block format untouched; disabled when collapsed.
- **Overflow menu + responsive collapse ladder**: §4.14 / §J — Step 1 (clear) → Step 2 (insertions) → Step 3 (color/highlight) → Step 4 (alignment); the floor is text-style + inline marks + lists/blockquote + link + overflow `⋯`.
- **Lifecycle and the active-editor handle**: §A — the single shared reactive value. §A4 fields. §A13 (centralised refresh suppression — already true via 305). §A14 (MutationObserver fallback on host removal — 306 implements). §A11 (re-target = full teardown then mount; B11 — flush previous editor's debounced save first).
- **Appear / disappear / re-target**: §B — `pointerdown` outside both editor and toolbar is the single teardown trigger (§B4); same-widget re-target snaps instantly (§B10); cross-widget re-target fades out/in over 80ms (§B9); the bar never disappears on a toolbar/popover click (§B12).
- **Placement and tracking**: §C — anchor is the field's host element; default above; flip below near canvas top; horizontal clamping; pin to viewport top while editing the bottom of a tall field; widget's drag-grip + Edit affordance hidden while editing inside (§C14); fixed-position layer at app root, never inside `v-html` (§C15).
- **Selection preservation and focus**: §D — every control `mousedown.preventDefault()`; pointer interactions never transfer focus from the editor (§D4); the link popover's URL field is the one exception under pointer (§D4a); keyboard entry via Alt+F10 is a separate, intentional focus path (§D4b / §K9).
- **Format-state reflection**: §E — synchronous on `editor-change`; active/mixed/disabled per control; the toolbar must not visibly lag the caret.
- **Data flow**: §L — every change through Quill's API with source `'user'`; debounced raw-config save flow from 304 unchanged; sanitisation unchanged.
- **Accessibility**: §K — `role="toolbar"`, `aria-pressed`, roving-tabindex, Alt+F10 entry, Escape exit, WCAG AA contrast verified for the active-pill colours, `prefers-reduced-motion` honoured.
- **Reconciliation with reference images**: §7 — HTML source toggle, word count, Google-Docs Title/Subtitle, paperclip-as-attach, light theme — all out, with rationale.

Do not propose alternatives to settled decisions. If something in the spec is ambiguous, stop and ask the user.

## Phases

### Phase 1 — Lifecycle foundation: handle refactor + MutationObserver fallback

Extend the store's active-editor handle from the stub `{quill, widgetId, path}` to the spec §A4 shape `{quill, hostEl, widgetId, configPath, getRect}` (rename `path` → `configPath` to match the spec's naming exactly). Update `useInlineEdit.activateRich` / `activatePlain` to publish the extended record on mount (plain-text fields still do not publish per §I4 — only rich-text). Update the type. Add the §A14 MutationObserver: while the handle is non-null, observe the parent subtree of `handle.hostEl` for `childList` mutations; if `document.contains(handle.hostEl)` becomes false, clear the handle synchronously within the same microtask. Pest verifies the handle shape and the MutationObserver clears on host removal; Playwright (parked) verifies the corner-ghost bug is impossible by triggering an out-of-band refresh while editing.

### Phase 2 — ColorPicker arrow-key grid navigation (spec §6.1)

The one remaining §6 companion change. Add 2-D arrow-key grid navigation to `ColorPicker.vue`'s swatches: Left/Right/Up/Down move focus across swatches within a group; Enter/Space applies; Tab moves between groups (Theme colors → My swatches → Custom hex); Escape closes. The existing `panelOnly` mode and click behaviour are unchanged. Pest verifies the navigation contract; Playwright verifies the panel is operable via keyboard.

### Phase 3 — Toolbar component build (the centerpiece — §4 visuals + §5 behaviour)

Build `resources/js/page-builder-vue/components/InlineFormatToolbar.vue` (a fresh implementation; the 305 first cut is gone). Mount once in `App.vue` at the page-builder Vue app root, into a fixed-position layer (§C15). All 18 controls per §4.6 / §4.7, in the order specified, with Lucide glyphs from `lucide-vue-next`. The text-style dropdown, link popover, colour / highlight popovers, heroicon picker reuse, and image upload flow all per §4.8–§4.12. Format-state reflection per §E. Selection preservation per §D. Responsive collapse ladder + overflow menu per §J. Accessibility per §K. Position tracking + scroll pinning per §C. Re-target transitions per §B9 / §B10. The toolbar holds no direct Quill reference (§A7 / §M3); every operation reads through `store.activeInlineEditor.value.quill` at call time. The link popover's "Open in new tab" persists target+rel per the 305 sanitiser allow-list. The text-style menu's row preview uses `store.themeHeadingFamily` / `store.themeBodyFamily` per the 305 bootstrap additions. The link popover's page picker uses `store.pages[].url` per the 305 bootstrap additions. **Mid-build verification is against the spec rules in §5 (each independently testable). Do not pause for look/feel judgment — the spec resolved those choices.**

### Phase 4 — Verification against the spec's §4.15 mockup-state checklist (user-facing pause)

Walk through each of the 10 named states from §4.15:

1. Bar at rest, full width
2. Bar with active toggles + one mixed state
3. Text-style menu open, theme-style rows, H2 highlighted
4. Link popover empty state (insert)
5. Link popover edit state (Remove visible)
6. Color popover open
7. Overflow menu open at Step 3 collapse
8. Bar at narrow widths (Step 1 → floor → wrapped two-row)
9. Bar pinned to viewport top while editing a tall field
10. Suppressed corner affordances (drag-grip + Edit hidden during active edit)

Verify each in the running builder. **Pause for the user to confirm the build matches the spec.** Do not suggest closing.
