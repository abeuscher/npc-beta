# 122. Widget System — Build Server Integration

Connect the app to the external build server, migrate public-facing CSS from Tailwind to a custom SCSS framework, and repackage all widgets into the installable folder structure. This is a multi-session effort — session 122 handles the foundation, subsequent sessions handle widget-by-widget migration and testing.

**Prerequisite:** Build server is running and accepting requests (built separately).

---

## General requirements

### 1. App-side build service

- Laravel service class that collects installed widget sources (SCSS, CSS, JS) and posts to the build server
- Receives compiled bundles, writes to `public/build/`
- Updates the asset manifest so the layout references the new filenames with cache-busting hashes
- Artisan command to trigger a build manually (`php artisan build:public`)
- Graceful failure: if build server is unreachable, log error, keep serving last known good bundle, surface warning in admin

### 2. Widget folder structure

- Create `widgets/` directory at project root
- Each widget gets its own folder: `widgets/{handle}/`
- Each folder contains: `widget.json` (manifest), `template.blade.php`, optional `style.scss`, optional `script.js`
- Shared field types live in `widgets/_fields/{type}/`
- Register `widgets::` Blade view namespace so templates resolve from the new location

### 3. `widget:sync` artisan command

- Scans `widgets/*/widget.json` manifests
- Creates or updates `widget_types` DB rows from manifests
- Clears the `css`, `js`, `template` inline columns (everything comes from files now)
- Triggers a build via the build service
- Replaces the `WidgetTypeSeeder` entirely

### 4. SCSS framework (replaces Tailwind on public side)

- Custom SCSS foundation: variables, breakpoint mixins, base typography, grid/layout helpers, component classes
- CSS custom properties for theming (already partially in place: `--color-primary`, `--font-family-body`, etc.)
- Base styles restore what Tailwind Preflight currently strips (headings, links, form elements)
- Dark mode via SCSS mixins instead of Tailwind `dark:` variants
- Button classes (`btn-primary`, `btn-secondary`, `btn-text`) as SCSS components
- Form input styles to replace `@tailwindcss/forms`

### 5. Layout and renderer changes

- Remove Tailwind from the public SCSS entry (`public.scss`)
- Remove the on-the-fly ScssPhp compilation from the layout
- Remove inline `<style>` and `<script>` output from the layout
- Layout references the build server's compiled bundles via manifest
- Keep Tailwind on the admin side — Filament theme build is unchanged
- Renderer resolves templates from `widgets::` namespace instead of inline DB `template` column

### 6. Widget migration (each widget)

For each of the ~25 existing widgets:
- Create folder in `widgets/{handle}/`
- Write `widget.json` manifest from the current seeder entry
- Move Blade template from `resources/views/widgets/` to widget folder
- Replace Tailwind utility classes in the template with semantic SCSS classes
- Write `style.scss` with the widget's styles
- Move any JS from the DB `js` column to `script.js`
- Test rendering through the build server pipeline

### 7. Cleanup

- Delete `WidgetTypeSeeder.php`
- Delete `resources/views/widgets/` (templates moved to `widgets/`)
- Delete `resources/scss/widgets/` (styles moved to widget folders)
- Remove ScssPhp dependency from composer if no longer used elsewhere
- Remove Tailwind + PostCSS from the public build entry in `vite.config.js`
- Drop or null the `css`, `js`, `code` columns on `widget_types` (migration)

## Suggested order of operations

1. **Build service class + artisan command** — get the app talking to the build server with a test payload
2. **SCSS framework foundations** — variables, mixins, base typography, layout helpers. Get a working public stylesheet that doesn't use Tailwind.
3. **Widget folder structure + view namespace + widget:sync** — the infrastructure for installable widgets
4. **Migrate 2-3 simple widgets** (text_block, image, video_embed) — prove the full pipeline works end to end
5. **Migrate remaining widgets** in batches, testing each
6. **Remove Tailwind from public side** — delete the dependency, clean up config
7. **Cleanup** — remove old files, seeder, inline compilation

Each step is independently deployable and testable. Steps 1-3 can happen without breaking existing widgets (they continue to work through the old pipeline until migrated). Step 4 is the proof point — if those three widgets work, the rest is mechanical.

---

## This will break things

This migration involves:
- Wiping the site's public CSS and rebuilding from scratch
- Every public-facing template will be touched
- The seeder is deleted — `widget:sync` replaces it
- The layout's asset loading changes completely

There is no backwards compatibility path. The site will be broken during migration and rebuilt widget by widget.

- NPM dependency auto-merging from manifests
- Admin panel CSS changes (Filament stays on Vite + Tailwind)
