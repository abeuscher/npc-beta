# 248. Fleet Manager Contract v2.0.0 ‚Äî mTLS Migration

The CRM-side companion to FM 012's mTLS absorption. Replaces bearer authentication with mTLS at the TLS layer, terminated by nginx with one client cert trusted per install. PHP doesn't see auth ‚Äî if the request reached the application, the cert handshake succeeded. Bumps the agent contract from v1.2.0 to v2.0.0 (non-additive).

The FM-side `last_backup_at` threshold-derivation ownership question is **explicitly deferred** out of v2.0.0 ‚Äî it stays pending in FM's coordination-requests block and lands at a future additive v2.x bump. v2.0.0 stays narrowly the auth-handshake change so the retrospective is clean.

---

## Design decisions (resolved before starting)

- **Auth scheme: mTLS at the TLS layer.** Nginx-terminated. The application has no auth code path for the FM agent endpoint after this session ‚Äî `/api/health` is reachable only if nginx accepted the client cert. PHP doesn't authenticate; the application discipline collapses to "trust the connection."
- **Cert trust shape: one specific cert per install.** Each CRM install trusts exactly one FM-side cert configured at nginx via `ssl_client_certificate`. No CA, no PKI tooling, no chain. FM holds N keypairs for N installs ‚Äî same per-install credential discipline as the v1.x bearer.
- **Cert paste mechanism: bind-mounted file on the host.** Operator pastes the FM-supplied cert into `/opt/nonprofitcrm/nginx-certs/fm-client.crt` on the droplet, which is bind-mounted into the nginx container at `/etc/nginx/certs/fm-client.crt`. Local dev uses the same in-container path mounted from `~/nonprofitcrm/nginx-certs/`. No CRM-side cert generation; no app-layer cert handling. Mirrors the existing `.env` bind-mount pattern.
- **No backward compat.** The v1.x bearer path is removed in the same migration. `AuthenticateFleetManagerAgent` middleware deletes; the `'fleet.agent'` alias drops from `bootstrap/app.php`; `config/fleet.php` loses `agent.api_key`; `FLEET_MANAGER_AGENT_KEY` removes from `.env.example`. Justification: pre-Beta-1, no live clients, FM-side hasn't entered production; backward-compat would be empty discipline.
- **Threshold-derivation ownership for `last_backup_at`: deferred out of v2.0.0.** The FM-raised question (CRM-canonical vs FM-canonical for the derived `status`) stays pending in FM's coordination-requests block. v2.0.0 ships only the auth-handshake change; the threshold question lands at a future additive v2.x bump. Bundling would inflate the v2.0.0 retrospective and grow FM 012's surface area beyond the auth swap; better to settle the question separately when there's natural impetus or per-client-override pressure on the FM side.
- **Response shape unchanged in v2.0.0.** Top-level fields, subchecks, status enum ‚Äî all identical to v1.2.0. Only the auth handshake changes. FM 012's Phase 4 explicitly verifies this and aborts if response shape drifts; this session must hold that line.
- **`/api/health` route stays at the same path.** FM 012 polls the same URL; only the auth surrounding it flips.

## Phase 1 ‚Äî Nginx mTLS configuration

Nginx is the auth gate. The application gets no auth signal beyond "this request arrived."

- **Confirm nginx config layout at session start.** Likely candidates: `docker/nginx/`, `nginx/conf.d/`, or inline in compose. Surface to user if not obvious.
- **Local dev (`docker-compose.yml`):** the nginx service mounts `./nginx-certs/fm-client.crt` to `/etc/nginx/certs/fm-client.crt` (read-only). The directory is committed to git carrying the placeholder cert (see below); only `*.key` files are gitignored.
- **Production (`docker-compose.prod.yml`):** the nginx service mounts `/opt/nonprofitcrm/nginx-certs/fm-client.crt` to the same in-container path. Operator drops the FM-supplied cert there; `docker compose -f docker-compose.prod.yml restart nginx` picks it up.
- **nginx config ‚Äî server-level optional, location-level strict.** The naive shape (`ssl_verify_client on` inside a location block) is silently broken on modern nginx: it triggers TLS renegotiation, which is disabled by default since CVE-2009-3555 and is not supported at all in TLS 1.3. The location-scoped directive fails to challenge for a cert and the gate is bypassed. The pattern that actually works on a single server block:
  ```
  server {
      listen 443 ssl;
      ssl_verify_client optional;
      ssl_client_certificate /etc/nginx/certs/fm-client.crt;
      ssl_verify_depth 1;

      location = /api/health {
          if ($ssl_client_verify != "SUCCESS") { return 403; }
          # ... existing fastcgi/proxy directives mirrored from the catch-all location
      }

      location / { ... }   # unconditional ‚Äî no cert required
  }
  ```
  `ssl_verify_client optional` requests a client cert during the TLS handshake but does not reject the connection if one is absent ‚Äî public routes stay reachable. The per-location `if ($ssl_client_verify != "SUCCESS") { return 403; }` then enforces the gate strictly for `/api/health`. The `if` directive is generally discouraged in nginx for content-selection logic, but it's correct here because the action is a `return` (not a request modifier) ‚Äî this usage is on the [If is Evil] page's allow-list. The alternative (separate server block on a dedicated port with `ssl_verify_client on` at server level) also works but requires opening a second port and is more intrusive; only fall back to it if the single-server-block shape surfaces issues at session time.
- **Committed placeholder cert.** nginx refuses to load if `ssl_client_certificate` references a missing file, so a fresh checkout (or a pre-paste production bring-up) would fail to start. Commit a throwaway self-signed cert at `nginx-certs/fm-client.crt` whose private key is discarded at generation time. Initial state is "the install trusts this placeholder" ‚Äî harmless because nobody has the matching private key, so no client can authenticate. The operator overwrites the placeholder with the real FM-supplied cert during install (per the runbook in Phase 5). Generation recipe (run once at session implementation, commit the resulting `.crt`):
  ```
  openssl req -x509 -newkey rsa:2048 -days 3650 -nodes \
    -keyout /tmp/placeholder.key -out nginx-certs/fm-client.crt \
    -subj '/CN=placeholder-do-not-trust'
  rm /tmp/placeholder.key
  ```
  Defense-in-depth: add `nginx-certs/*.key` to repo `.gitignore` so private keys can never be accidentally committed even if someone copies their FM-supplied keypair into the directory by mistake. The `.crt` files are public by nature and stay in git.

**Tests:** nginx config can't be unit-tested directly. Validation lives in the manual-testing curl scenarios.

## Manual testing

Three curl scenarios on local. Generate a test keypair first (RSA-2048 chosen for universal nginx/OpenSSL compatibility ‚Äî the trust file is algorithm-agnostic so there's no benefit to Ed25519 at the test level):
```
mkdir -p nginx-certs
openssl req -x509 -newkey rsa:2048 -days 3650 -nodes \
  -keyout nginx-certs/fm-client.key \
  -out   nginx-certs/fm-client.crt \
  -subj '/CN=fm-client'
docker compose restart nginx
```

(If session-time experiments with Ed25519 are wanted, swap in `-newkey ed25519` ‚Äî requires nginx ‚â• 1.18 + OpenSSL ‚â• 1.1.1 on the host, which may or may not be present.)

Then:

1. **No cert presented:** `curl -k https://localhost/api/health` ‚Üí TLS handshake fails (`error:14094412:SSL routines::sslv3 alert bad certificate` or similar). Application emits no JSON envelope.
2. **Wrong cert presented:** generate a second keypair, present that. `curl --cert wrong.crt --key wrong.key https://localhost/api/health` ‚Üí same handshake failure.
3. **Correct cert presented:** `curl --cert nginx-certs/fm-client.crt --key nginx-certs/fm-client.key https://localhost/api/health` ‚Üí 200 with the v2.0.0 JSON envelope (`contract_version: "2.0.0"`).

Spot-check that public routes (the home page, the admin login, the portal login) still serve normally without any client cert ‚Äî mTLS must be `/api/health`-scoped, not site-wide.

## Out of scope

- **A2's FM-side capability work** (install / backup-trigger / restore / log-fetch UI). Resequenced to FM 013+ after FM 012 absorbs v2.0.0.
- **Cert rotation flow.** Manual rotation is the v1 discipline (operator pastes a new cert, restarts nginx). Carry-forward.
- **Cert distribution automation.** Operator-pasted manually for v1; FM never touches the CRM filesystem. Carry-forward.
- **Cert expiry tracking.** Self-signed certs default to long lifetimes; no FM-side surfacing in v1 ‚Äî carry-forward, FM-side concern.
- **CA-signed certs / PKI tooling.** Self-signed per-install is sufficient for the v1 fleet shape.
- **Multi-FM-instance support.** Each install trusts exactly one FM-side cert. Multi-instance is a contract change of its own (additive ‚Äî would land as a v2.x bump, not v3.0.0).
- **App-layer cert generation.** FM holds the keypairs; CRM only holds the public cert. Generation is FM's responsibility.
- **Application-layer cert subject parsing / cert-derived identity.** The application sees no cert at all; identity is collapsed to "the request reached the controller." Per-cert role/scope mapping is a future concern, additive to a future contract version.
