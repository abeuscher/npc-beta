## The split — read this first

There are **two public-facing surfaces and they are deliberately separate**:

- **The public marketing site** (`nonprofitcrm.com`) — the on-brand product site. **This session is only about this site.**
- **The demo node** (separate server) — example CRM pages a prospect lands in. **Off-brand on purpose** (examples, likely branded differently per set to emphasise that). **The demo is entirely out of scope here** — none of these colour/spacing changes touch it, and the demo `data-tour` / tour work from session 338 is unrelated.

## Workflow — this shapes everything

The **"before" home page was exported off the public site and imported into the local CMS.** All work happens **locally**, against that imported page and the local theme colours. The deliverable is the **export the owner pushes back up** — so:

- **No seeder changes.** Edit the live/imported CMS + theme state, never `DesignSettingsSeeder` (or any seeder). The seeded defaults stay as-is.
- **Ideally no code changes.** This should be theme-colour **settings** + page-builder **appearance** edits — both exportable. If a change genuinely can't be expressed as content/theme settings and would need code or a deploy, **flag it at the open gate** — it breaks the clean export round-trip and is a decision for the owner.
- **The owner pushes the result.** Your job is to land the local CMS/theme in the target state and record exactly what changed (which tokens, which page widgets) so the owner knows what the export must carry.

## The change spec (from the comp)

Source files in `sessions/public website/revisions/`:

- `Main-before.svg` — current desktop home. `Main.svg` / `Fullscreen.png` — revised desktop (`Fullscreen.png` is a raster of `Main.svg`; the two PNGs are identical). `Mobile.svg` — revised mobile. `Header.svg` / `Footer.svg` — reference, **unchanged**.
- **SVG text is outlined (not extractable) — render the SVGs in a browser to read them** (a headless-Chromium screenshot at the SVG's intrinsic size works; the files are large). The **green text down the right margin is the designer's change spec**; the design itself shows the rest.

What the comp asks for:

- **Brand accent → orange, site-wide.** The teal/blue brand colour (`--np-color-brand`, currently `#0172ad`) becomes **orange** across the whole site — every button, the "Try it **free!**" CTA. **Replace** the value in Theme → Colors; don't add a new token. **Sample the exact orange off the comp** (the owner doesn't have hexes).
- **Backgrounds → cooler.** Light sections → a **"cool off-white"** (replace `--np-color-bg` / the relevant surface token, currently `#ffffff`). The dark "not a SaaS company" section → **"slightly darker."** Sample both off the comp; **replace, don't add.**
- **CTA copy:** "Try it" → "**Try it free!**" (the hero/CTA buttons pick up the orange).
- **Spacing:** **80px between content blocks** (mobile note). Use the operator **section-spacing primitive** (session 335 — vertical-axis spacing, scales on mobile), not new CSS. Confirm whether 80px is mobile-only or desktop too.
- **Mobile = desktop.** The comp says "All recommended changes to desktop apply to mobile too" — no mobile-only design beyond the 80px spacing.

It's a **colour / spacing / layout sweep**, not a re-layout — section order and copy are otherwise unchanged.

## Open questions to resolve at the open gate

1. **Pure-data feasibility.** Confirm the whole sweep is achievable as theme-colour settings + page-builder appearance (exportable, no code). Flag anything that isn't.
2. **What the export carries.** Confirm the owner's export round-trips both the **page edits** and the **theme-colour settings** (or the owner applies the theme colours separately). The colours are settings compiled into the public bundle by `ColorTokenCompiler` via `build:public`.
3. **Sampled values.** Surface the sampled hexes (orange / off-white / slightly-darker) for owner sign-off **before** applying them.
4. **Spacing scope** — 80px mobile-only, or desktop too.

## Phases

### Phase 1 — Sample + map
Render the comps. Sample the orange / cool-off-white / slightly-darker values. Map each to the **token it replaces** (`--np-color-brand`, `--np-color-bg`, the dark surface token / the dark section's background). Surface the mapping + hexes for sign-off.

### Phase 2 — Theme colours (site-wide)
Replace the token values via Theme → Colors (settings, not seeder, not the in-code defaults). `build:public` to recompile the tokens into the public bundle. Verify the accent / buttons / backgrounds change everywhere.

### Phase 3 — Home page sweep
On the imported home page, via the page builder appearance only: per-section backgrounds, 80px block spacing, the "Try it free!" copy, and any layout tweaks the comp shows. No code.

### Phase 4 — Verify
Render the local home page (desktop + mobile) and compare against the comp side-by-side (Playwright screenshots). Confirm parity and that all changes sit in exportable state. Record the exact token + widget changes for the owner's export.
