# 111. Trash Management UI

Wire up Filament's built-in soft-delete support on every resource that uses `SoftDeletes`. Minimal custom code — use the standard Filament pattern throughout.

---

## Design decisions (resolved before starting)

- Use the standard Filament `TrashedFilter` + action pattern. No separate Trash page.
- Force-delete (purge) is gated to `super_admin` role only.
- Restore is available to any user who has the resource's `delete` permission (same permission that soft-deleted the record).
- Read `docs/data-retention-policy.md` before starting — the cascade rules and RESTRICT constraints defined there inform which force-delete paths are safe.

## Phase 1 — Add soft-delete support to each resource

For each of the following resources, apply the standard Filament soft-delete pattern:

### Models with SoftDeletes

| Model | Resource | Notes |
|-------|----------|-------|
| Contact | ContactResource | Force-delete already restricted to super_admin (session 109) |
| Page | PageResource | Includes posts (type=post) — PostResource shares the pages table |
| Organization | OrganizationResource | |
| Membership | MembershipResource | contact_id FK is RESTRICT — force-delete blocked if contact exists |
| Form | FormResource | Has Archivable trait; soft-delete is separate from archive |
| FormSubmission | FormSubmissionResource (if exists, or relation manager) | |
| Collection | CollectionResource | System collections already blocked from deletion |
| CollectionItem | CollectionItemResource (if exists, or relation manager) | |
| Note | NoteResource (if exists, or relation manager) | |
| Campaign | CampaignResource | |

### Per-resource checklist

For each resource:

1. **`getEloquentQuery()`** — override to remove `SoftDeletingScope` so trashed records are included in the query
2. **`TrashedFilter::make()`** — add to the table's `filters()` array
3. **`RestoreAction`** — add to table row actions (visible only on trashed records)
4. **`ForceDeleteAction`** — add to table row actions, gated to `super_admin`
5. **`RestoreBulkAction`** — add to bulk actions
6. **`ForceDeleteBulkAction`** — add to bulk actions, gated to `super_admin`

If a model's records are only accessible via a relation manager (e.g. FormSubmission, CollectionItem, Note), apply the same pattern there instead.

### PostResource

Posts are pages with `type = 'post'`. The PostResource may need the same treatment as PageResource. Check whether it has its own query scope and apply accordingly.

## Phase 2 — Policy gates

If model policies exist, add `forceDelete`, `forceDeleteAny`, `restore`, and `restoreAny` methods gated appropriately:

- `restore` / `restoreAny` — same permission as `delete` for that resource
- `forceDelete` / `forceDeleteAny` — `super_admin` only

If no policies exist for a resource, use inline action authorization (`->authorize()` or `->visible()`) instead.
