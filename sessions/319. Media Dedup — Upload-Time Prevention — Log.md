## What was built

The prevention leg of the media trilogy (318 visibility ‚Üí **319 prevention** ‚Üí 320 reclamation). Two halves: a persisted content hash on every media file, and an upload-time warn-and-offer-reuse flow on the page-builder and inline-image surfaces.

### Phase 1 ‚Äî `content_hash` column + hash-on-add + backfill

- **Migration** `2026_05_22_120000_add_content_hash_to_media.php` ‚Äî adds `media.content_hash` (string, nullable, indexed) after `size`, then runs the backfill inline.
- **`App\Services\Media\MediaContentHasher`** ‚Äî `hashFor(Media)` streams the stored original off its disk and returns the SHA-256 hex (null if unreadable, so callers skip cleanly); `persist(Media)` computes-and-`saveQuietly`s the hash for rows that lack one.
- **Hash-on-add hook** ‚Äî a single `MediaHasBeenAddedEvent` listener registered in `AppServiceProvider::boot()`. That event fires *after* the file is copied to the library on **every** `addMedia*` path (uploads, `Media::copy()`, importer seeding), so one hook covers all of them. Verified against the Spatie `Filesystem::add` source (the event is dispatched post-copy for local and remote alike).
- **Backfill** ‚Äî `media:backfill-hashes` (`App\Console\Commands\BackfillMediaHashesCommand`), chunked by id, skips rows that already carry a hash. Invoked from the migration; re-runnable for larger libraries. Local run: all 178 rows hashed, 0 null.
- **Made the 318 duplicate scan instant** ‚Äî `MediaFinderService::scanDuplicates()` now prefers the `content_hash` column and only reads bytes for un-backfilled rows.

### Phase 2 ‚Äî Upload-time dedup detection + reuse endpoints

- **`App\Services\Media\MediaDedupService::findMatches(hash, fileName?)`** ‚Äî surface-agnostic detection. Content-hash matches (identity) collapse to one representative candidate carrying a `duplicate_count`; filename matches with a *different* hash surface the iteration case, but only when the name isn't a randomised `->hashName()` (40-hex) basename. Candidates sort referenced-first (drives most-referenced-first disambiguation) via `MediaReferenceInventory::classify`.
- **`media-dedup-check` endpoint** (`MediaDedupController`) ‚Äî registered twice against the same controller so both front-ends can reach it: under the page-builder lookup base (`/admin/api/page-builder/media-dedup-check`, for the Vue client) and at the admin path (`filament.admin.media-dedup-check`, for the Quill handler). Validates a 64-char hex hash; auth-gated.
- **Reuse endpoints on `PageBuilderApiController`** ‚Äî `useExistingImage` (config image field) and `useExistingAppearanceImage` (appearance background). Both **copy-first-then-evict**: copy the source into the target collection, then remove any prior media in that single-file collection except the new copy ‚Äî so reusing the widget's *own* current image can't delete the source before it's read (the bug the tests caught). Shared `evictOtherMedia` helper.

### Phase 2 ‚Äî front end

- **Vue page builder** ‚Äî the dedup gate lives in `useUploadActions` so call sites (`ImageUploadField`, `BackgroundPanel`) are unchanged. Before storing, it hashes the file client-side (`composables/hashFile.ts`, Web Crypto; degrades to a plain upload if unavailable), calls `dedupCheck`, and on a match opens a shared modal state in the editor store (`dedupPrompt` / `resolveDedup`). New `DedupPromptModal.vue` (mounted at app root, mirrors the existing confirm-modal pattern) offers **Use this** / **Upload as new** / **Cancel**, with thumbnail, size, "in use on your site," and collapsed-duplicate count. `api.ts` gained `dedupCheck` / `useExistingImage` / `useExistingAppearanceImage`.
- **Inline rich-text images** (`resources/js/admin/quill-editor.js`) ‚Äî before uploading, hashes the file and checks the library; an exact (content-hash) match offers reuse via a `confirm()` and, on accept, inserts the existing `/storage/{id}/` URL with **no new upload at all** (the one zero-copy surface). Blade passes `data-dedup-check-url`.

### Honest scope notes

- **Reuse copies bytes (for now).** Under the current `DefaultPathGenerator`, `useExisting*` still writes a physical copy for the new owner ‚Äî the win this session is behavioural (a coherent, findable library; far fewer fresh re-uploads), not disk reclamation. Inline-image reuse is the exception (no new row/file). True byte-sharing is **session 320** (content-addressed storage + refcounted delete).
- **Surfaces covered:** page-builder widget image fields + appearance backgrounds, and inline rich-text images. **Deferred to a post-Beta stub:** the Filament `SpatieMediaLibraryFileUpload` resource fields (post/event/product thumbnails + headers + OG, WidgetType thumbnails, Page details) ‚Äî packaged form components with no idiomatic hook to interrupt a form-save with a three-way choice, and low-frequency single-purpose slots, not the bloat driver. Dashboard/record-detail-builder appearance backgrounds fall through to a normal upload (the check 404s silently where the route isn't mounted). The backend `MediaDedupService` + endpoint are surface-agnostic, so wiring those later is front-end-only.

### Prompt drift noted

- The base prompt named `useLibraryLoader.ts` as "the page-builder Vue picker"; it is actually a widget-JS-lib (Swiper/Chart) loader. The real upload path is `useUploadActions.ts` + `ImageUploadField.vue` / `BackgroundPanel.vue`. Adapted silently.
- All three JSON upload controllers use `->hashName()`, so filename matching is dead there ‚Äî those surfaces match on content hash only, as the prompt itself anticipated.

### Post-session library cleanup (UAT prep, dev DB only)

To make the feature testable without built-in confusion, traced the dev library's bloat: the **Random Data Generator** admin tool (`RandomDataGenerator::attachPoolImage`) copies a pool image's bytes onto every fake event/post/product it creates ‚Äî that, not the base seed, produced the ~178 rows over ~45 distinct files. After the user's scrub-data wipe (178 ‚Üí 128), swept 63 owner-orphaned rows that the bulk-delete left behind (mass `->delete()` skips Spatie's media cascade; `media-library:clean` only removes stray *files*, not owner-orphaned *rows*). Final dev state: **65 referenced rows / 45 distinct files, 0 unused, 0 missing, 0 orphans.** Remaining duplicate clusters are all referenced (each a distinct owner's copy) ‚Äî the disk session 320 reclaims.

## Security considerations

The `media-dedup-check` endpoint is auth-gated (`abort_unless(auth()->check())`) and only reveals existence/metadata of library assets to an authenticated admin; it stores nothing and validates the hash shape strictly. The reuse endpoints reuse the existing `update_page` gate that already governs the page-builder image uploads, verified by a 403 test. The hash is computed server-side from stored bytes; the client-side hash is only a lookup key (a forged hash returns no match or an unrelated one, never another user's private data, since all media here is admin-owned site content). No new file is written on the dedup-check path. Nothing changed in the portal or public surfaces.

Standard `migrate` adds the column and backfills inline.
