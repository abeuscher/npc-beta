### Surfaced during manual testing — importer↔widget status mismatch

Donations importer's `mapDonationStatus()` ([ImportDonationsProgressPage.php:569-584](app/Filament/Pages/ImportDonationsProgressPage.php#L569-L584)) writes `'completed'` for any `active` / `paid` / `succeeded` / blank input. Three downstream consequences:

1. The canonical `donations.status` enum (per `docs/schema/donations.md`) is `pending` / `active` / `cancelled` / `past_due`. `'completed'` is outside the enum.
2. Recent Donations resolver filter is `whereIn('status', ['active', 'cancelled', 'past_due'])`. Imported "paid" rows land at `'completed'` and are invisible to the widget.
3. Donations list-page Status badge color map ([DonationResource.php:112-118](app/Filament/Resources/DonationResource.php#L112-L118)) only knows the four canonical values — `'completed'` rows render uncolored.

**Decision:** out of scope for 234. Documented as a 235-or-Phase-B carry-forward (Financial Data Origin Phase B is the natural home, alongside admin gating generalization). The fix is a one-line change in `mapDonationStatus()` to emit `'active'` instead of `'completed'`, plus a one-time backfill of any installs that have already run the importer post-233.

### Phase 5e decision artifact (drafted in close-gate Phase 1)

Comparison of the three resolver arms at their `Gate::denies` lines:

| Arm | Permission gate (line 1) | Empty shape on deny |
|---|---|---|
| `resolveNote` | `! auth()->user()?->can('view_note')` | `['items' => []]` |
| `resolveMembershipOne` | `! auth()->user()?->can('view_membership')` | `['item' => null]` |
| `resolveDonationList` | `! auth()->user()?->can('view_donation')` | `['items' => []]` |

All three arms compress identically: single permission name, single `?->can()` call, fail-closed return that's purely a function of cardinality. **Clean centralization is possible** — no wrinkle. Phase 5e (session 235) lifts the pattern into a `requiredPermission` field on `DataContract` enforced at the resolver dispatch boundary; ambient-type gating stays per-arm; instance-bound `Gate::denies('view', $model)` is a forward hook for a later arm if needed.
