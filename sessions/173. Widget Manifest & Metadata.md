# 173. Widget Manifest & Metadata

Stage 4 of the Sovereign Widget arc. Every widget declares human-facing metadata (version, author, license, screenshots, keywords, presets) via its definition class. Metadata is code-only — not persisted to `widget_types` — and consumed by the Stage 5 widget browser UI and the Stage 6 install/uninstall mechanics. This session defines the manifest schema, adds the base-class methods, wires a CI validation test, and documents the contract. No widget's user-visible behaviour changes.

---

## Design decisions (resolved before starting)

- **Storage: code-only.** Metadata lives on `WidgetDefinition` methods, accessed at runtime via `$registry->find($handle)`. No new columns on `widget_types`, no migration, no `toRow()` changes. The Stage 5 browser will call `->manifest()` on each registered definition.
- **Fields included this session:**
  - `version(): string` — semver, default `'1.0.0'`
  - `author(): string` — default `'Nonprofit CRM'`
  - `license(): string` — default `'MIT'`
  - `screenshots(): array` — list of paths relative to the widget folder, default `[]`
  - `keywords(): array` — lowercase tags for browser search, default `[]`
  - `presets(): array` — list of named config bundles, default `[]`
- **Fields deferred:** `requiredCapabilities()` (unclear scope until Stage 5 clarifies browser needs) and `minAppVersion()` (app has no version string yet — Stage 6 introduces one when install/uninstall needs version checks).
- **Base-class uniform defaults.** Because every widget is first-party, `author()` and `license()` will be identical across all 30. Don't override per-widget until a widget has a genuine reason to differ (e.g. a future third-party widget). `version()` stays `'1.0.0'` for every widget; future bumps are per-widget.
- **Screenshots are declared, not authored.** Every widget's `screenshots()` returns `[]` this session. Authoring actual marketing screenshots is a later content-pass task. The `screenshots()` method and its CI validation nonetheless land now so the contract is stable for Stage 7 third-party widgets.
- **Presets are declared, not authored.** Every widget's `presets()` returns `[]` this session. Preset shape is documented and CI-validated. Actual preset authoring and the inspector preset picker become their own future session.
- **Aggregator method.** Add `manifest(): array` on the base class that bundles every manifest field — plus the already-existing `handle`, `label`, `description`, `category` — into a single array. The Stage 5 browser calls one method per widget instead of eight.
- **Preset shape (canonical, CI-enforced):**
  ```php
  [
      'handle'            => 'dark-hero',           // slug, unique within the widget
      'label'             => 'Dark Hero',           // human-readable
      'description'       => 'Short sentence.',     // optional, may be null
      'config'            => [...],                 // widget config overrides (subset of schema keys)
      'appearance_config' => [...],                 // appearance jsonb bag subset
  ]
  ```
  Each preset's `config` keys must all exist in the widget's `schema()`. `appearance_config` shape is not strictly validated this session (too many nested paths) beyond "it's an array."
- **CI validation test.** New test file `tests/Feature/WidgetManifestTest.php` iterates every registered definition and asserts:
  - `version()` matches `/^\d+\.\d+\.\d+$/`
  - `license()` is one of `['MIT', 'Apache-2.0', 'GPL-3.0', 'BSD-3-Clause', 'proprietary']`
  - Every path in `screenshots()` exists on disk under the widget folder
  - Every item in `keywords()` matches `/^[a-z0-9-]+$/`
  - Every preset in `presets()` has the required keys with correct types, and every key in `preset.config` appears in `schema()`
- **No DB migration.** `widget_types` table is unchanged. Nothing about the renderer, inspector, or builder changes.
- **No user-visible behaviour change.** This is a contract-and-documentation session. The fast-test suite should still pass with the new test added.

## Phase 1 — Manifest methods on the base class

Add six new optional methods to `App\Widgets\Contracts\WidgetDefinition`:

```php
public function version(): string      { return '1.0.0'; }
public function author(): string       { return 'Nonprofit CRM'; }
public function license(): string      { return 'MIT'; }
public function screenshots(): array   { return []; }
public function keywords(): array      { return []; }
public function presets(): array       { return []; }
```

None are abstract — every existing widget inherits defaults and keeps working unchanged. Do **not** include these in `toRow()` — manifest is code-only.

---

## Phase 2 — Aggregator `manifest()` method

Add on the base class:

```php
public function manifest(): array
{
    return [
        'handle'      => $this->handle(),
        'label'       => $this->label(),
        'description' => $this->description(),
        'category'    => $this->category(),
        'version'     => $this->version(),
        'author'      => $this->author(),
        'license'     => $this->license(),
        'screenshots' => $this->screenshots(),
        'keywords'    => $this->keywords(),
        'presets'     => $this->presets(),
    ];
}
```

This is the surface the Stage 5 browser UI will consume. Keep the shape stable — Stage 7 third-party widgets will depend on it.

---

## Phase 3 — Extend `WidgetRegistry` with manifest accessor

Add a convenience method on `App\Services\WidgetRegistry`:

```php
/** @return array<string, array> handle → manifest */
public function manifests(): array
{
    return array_map(
        fn (WidgetDefinition $def) => $def->manifest(),
        $this->definitions
    );
}
```

This is what the Stage 5 browser will call to populate its grid in one pass.

---

## Phase 4 — CI validation test

Create `tests/Feature/WidgetManifestTest.php`. One test per rule, each iterating every definition registered on the `WidgetRegistry` (via `app(WidgetRegistry::class)->all()`). Rules:

1. `version()` matches `/^\d+\.\d+\.\d+$/`.
2. `license()` is in the allow-list.
3. `screenshots()` — every path exists on disk when resolved against `base_path()`.
4. `keywords()` — each item is a lowercase slug.
5. `presets()` — each entry has `handle` (string, slug), `label` (string, non-empty), `description` (string or null), `config` (array), `appearance_config` (array). Every key in `preset.config` must exist in the widget's `schema()[].key`.
6. `manifest()` — returns an array with exactly the expected keys (prevents accidental shape drift).

Report tests one-per-rule with a helpful failure message identifying the offending widget handle.
