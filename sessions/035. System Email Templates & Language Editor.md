## Background & decisions

Three transactional mailables exist today:

| Mailable | Trigger |
|----------|---------|
| `RegistrationConfirmation` | Contact submits event registration form |
| `EventCancellation` | Event is marked cancelled in admin |
| `EventReminder` | Scheduled command or manual admin action |

All three currently use hardcoded subjects and Blade views. This session replaces those with database-driven templates.

**No preview in the UI.** Admins test by sending the test email from the Mail Settings page. Browsers lie for email HTML.

**CAN-SPAM compliance for transactional email:**
- Physical mailing address required (even for transactional)
- Accurate From / Reply-To
- Non-deceptive subject line
- "Why you received this" statement
- No unsubscribe link required for purely transactional email

**Wrapper approach:** A default Blade wrapper handles header and footer. Admins who need full control can upload a custom HTML file per email type — the system injects the rendered body into a `{{content}}` marker in the uploaded file.

## `EmailTemplate` model

`app/Models/EmailTemplate.php`

- `$fillable`: all columns
- Static `forHandle(string $handle): self` — returns the record, creating defaults if missing (safety net)
- `render(array $tokens): string` — replaces `{{token}}` placeholders in subject/body/footer_reason with provided values. Returns rendered body HTML.
- `resolveWrapper(): string` — if `custom_template_path` is set, returns the file contents; otherwise returns the default Blade wrapper rendered with this template's header/footer settings.

## Default Blade wrapper

`resources/views/mail/default-wrapper.blade.php`

The wrapper is a simple table-based HTML email layout:
- Header: background colour from `$template->header_color` (fallback `#1a56db`), logo image if set, headline text if set
- Body: `{!! $body !!}`
- Footer: sender name, reply-to, mailing address, reason line

This view is used by `EmailTemplate::resolveWrapper()` when no custom template is uploaded.

## Update the three Mailables

Update `RegistrationConfirmation`, `EventCancellation`, and `EventReminder` to:

1. Load their `EmailTemplate` record via `EmailTemplate::forHandle('...')`
2. Build the token array from the available model data
3. Call `$template->render($tokens)` to get the rendered body
4. Pass the resolved wrapper + body to a single shared Blade view: `mail.system-email`

`resources/views/mail/system-email.blade.php` — outputs `{!! $html !!}` (the fully assembled HTML string).

The subject line is also rendered through `$template->render($tokens)` and passed to `Envelope`.
