### Surfaced during prompt-drafting after manual testing

The 233 prompt-drafting conversation surfaced a structural question that the date-fallback in the original 5d-3 (Recent Donations) draft had papered over: financial parent rows (`donations`, `memberships`, `event_registrations`, `transactions`) carry their **lifecycle** cleanly via `status` but their **origin** (Stripe webhook / hand-imported / manual admin entry) is conflated under the same column. Status overload was breeding latent runtime hazards — a future "cancel donation" admin action would happily try to drive a non-Stripe imported row through a non-existent state machine.

User-led design conversation reached three locked-in conclusions:

1. **Origin and lifecycle are orthogonal.** `status` stays as-is — universal "this is real / countable" reality filter for reads, aggregates, and access predicates; no read-side changes anywhere.
2. **Origin gets its own column** carrying an `App\WidgetPrimitive\Source` constant, lifting the existing write-side primitive to read-side queryable.
3. **`status = 'imported'` is rejected** — collapses a 2-dimensional problem into 1D and would force every read site to either lose lifecycle (filter on imported) or lose origin (filter on active).

Audit (10 minutes of grepping, captured in 233 prompt's Design Decisions section) confirmed the discipline gap is half-applied: `Donation` declares `ACCEPTED_SOURCES` via `HasSourcePolicy`, but `Membership` and `Transaction` do not. `Transaction` admin already has origin-aware action gating keyed off `stripe_id IS NULL` ([TransactionResource.php:45,55,313,315](app/Filament/Resources/TransactionResource.php#L45)); Donation/Membership admins have no equivalent gating but will need it the moment Refund/Cancel/Sync actions land. Importers DO maintain a ledger invariant for donations and events (write parent + Transaction), but **membership imports skip the Transaction write** — an asymmetry to fix.
