# 371. Security S3 — XSS Containment & Exposure Audit

Prove the app's first line of XSS defense is airtight — every rich-text write path funnels through the save-time sanitizer — then neutralize CSV formula-injection on exports, prove member-page data widgets scope to the viewing member, and gate sensitive fields against accidental public exposure.

## Session-specific deltas

- **The CSP backstop is live (S1 / session 370).** The public surface enforces a strict `script-src` with no `'unsafe-inline'`; the sanitizer is the *first* layer and the CSP the *second*. That makes this audit's residual-risk statement honest — but it does **not** lower the bar: a stored-XSS that renders inside the **admin panel** still matters because admin CSP is Report-Only (not enforcing) today, and admin sessions hold full PII/money access. Audit as if the CSP weren't there.
- **Write-path audit — the convention is model mutators.** `HtmlSanitizer` is applied at save time via model mutators (`Note`, `Event`, `EmailTemplate`, `SiteSetting`, `CollectionItem`, `PageWidget`'s schema-walk, custom fields). The audit hunts paths that **write rich text while bypassing the mutator**: the importer (`app/Services/ImportExport/Import/*` + the Filament importer pages — the prime suspect per the surface map), seeders, the admin JSON API (`PageBuilderApiController` / `routes/admin-api.php`), and any bulk/mass-update operation. Default hypothesis per the standing rule: a small missing funnel on an existing seam, not a broken sanitizer.
- **CSV formula-injection on the list-export family.** `app/Services/ListExportService.php` (the 261/262/B2b CSV+JSON+XLSX exports for non-Contact list resources, plus the per-resource list exports) must neutralize cells beginning `=`, `+`, `-`, `@` so a malicious stored value can't execute when the file is re-opened in Excel/Sheets. Apply the technique uniformly, not per-resource. Confirm the technique at the Open Gate.
- **Member-page widget scoping — the surface-map open check.** `type=member` CMS pages render behind "logged-in + verified" (`PageController::show`); verify that any data widget on such a page scopes CRM reads to the **viewing member's own `contact_id`**, not to arbitrary/all data. The portal's relationship-derived scoping looked clean for the direct portal routes; this proves it for the CMS-page path too. A Pest feature test hitting a member page as two different contacts (Bob must never see Alice's data) is the cheaper first signal before Playwright.
- **#32c absorbed (Path A, no C3a prereq).** Sensitive fields — home addresses (`contacts`), donor amounts (`donations`), internal notes (`notes`) — either **cannot be flipped public** or hit a **warning gate** on the leak-capable surfaces; per-field protection is documented in the data-classification section of `docs/runbooks/permission-matrix.md`; a public-content indicator lands on the leak-capable surfaces. The accountability / audit-trail feature half (**C3a**) stays deferred — protection doesn't need the paper trail.
- **Do not touch:** the five FM `/api/*` mTLS blocks + the fleet controllers (S2's territory — this session doesn't go near them), the CSP/headers policy shipped at S1 (only *rely* on it), the admin-CSP-enforcement promotion (a separate tracked residual).
