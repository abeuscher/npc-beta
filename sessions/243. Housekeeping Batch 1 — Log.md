### Phase 4 — Horizontal text alignment uniform across all Quill richtext-bearing widgets

- `resources/scss/_custom.scss` — lifted Quill's `.ql-align-*` rules out of `.hero-content` scoping and re-scoped them to `[data-config-type="richtext"]` (the canonical "this is Quill output" data-attribute marker carried by every richtext-bearing widget template). Three widgets currently emit Quill richtext content with the data attribute on the wrapper — Hero (`.hero-content`), TextBlock (root `<div>`), ThreeBuckets (per-bucket `.three-buckets__body`) — all three render correctly under the new selector.
- Why structural over per-widget: the prompt asked for "the cleanest implementation pattern (hero's, ideally) and apply it to every text-editor-bearing widget." Hero's pattern was scoped to its own template, not generalized; extending it would mean duplicating CSS-rule blocks per widget. The data-attribute-scoped form is one rule for all current and future richtext widgets, which matches the user's "uniform UX over per-type plumbing" preference.
- Hero's `text-align: left` baseline preserved as a standalone block (independent of the alignment rules — it anchors the default text direction even when the hero's flex-positioning centers the wrapper).
- AppearanceStyleComposer was flagged in the prompt as a possible alignment seam, but examination shows the composer only handles wrapper-level concerns (background, text color, padding, margin, full-width); horizontal *paragraph* alignment lives outside its remit and belongs in the public stylesheet via Quill's class convention.
- Build: `npm run build` ran successfully — `public.scss` (which `@use`s `_custom`) and the admin/page-builder bundles all rebuilt; the public bundle hash refreshed in `public/build/manifest.json`.

### Phase 5 — Admin Quill list rendering + indent fixes

- `public/css/admin.css` — two paired CSS rules added between the existing dark-mode Quill block and the dashboard slot grid block, both scoped to `.ql-editor` so they don't leak into other admin lists (Filament's resource-page tables/lists/sidebars all use the same admin.css):
  1. `.ql-editor ol, .ql-editor ul { padding-inline-start: 1.5em; list-style: none; }` — `padding-inline-start` restores indent so Quill's `::before` markers sit inside the padded text area; `list-style: none` defensively suppresses any browser-default markers re-enabled upstream.
  2. `.ql-editor ol > li, .ql-editor ul > li { list-style: none; }` — defense-in-depth in case any rule downstream tries to re-enable per-`<li>` markers.
- Two paired bugs the user reported: (a) bullet lists rendered with both bullet *and* numeric markers simultaneously; (b) markers sat outside the editor's padded text area. Quill 1.x renders list markers via `::before` counter pseudo-elements, NOT via the browser's native `list-style-type`. Quill's own stylesheet clears list-style and padding so its `::before` markers can take over; a stray re-enabling rule somewhere upstream was reaching into the editor and (i) re-asserting `list-style: decimal` on the `<ol>` produced for bullet lists (Quill 1.x historically emits `<ol>` with `data-list="bullet"` for bullets), causing the duplicate-marker render, and (ii) clearing padding-inline-start to zero so markers ended up outside the padded boundary.
- No build (admin.css is hand-edited and served directly per `docs/app-reference.md`'s "no build required" rule).

### Phase 6 — Mobile Type Scaling — lifted out, captured as new pre-Beta-1 stub

- The session prompt explicitly carried the lift-out rule: *"If the fix needs more than `clamp()` rules and a couple of breakpoint media queries — for example, if it surfaces deeper theme-architecture concerns about how type tokens flow from theme settings into output CSS — stop and lift it into its own stub."* Investigation surfaced exactly that: TypographyCompiler emits user-configured per-element font sizes into an inline `<style>` block that loads *after* the Vite-compiled bundle, so any SCSS-only fix is overridden by the user's typography config on installs that have configured typography (the encouraged path).
- User-supplied design captured for the new stub: in each font/text-style control in the design system, add 3 fields beneath the size picker — text sizes for the font at the lg/md/sm breakpoints below xl. Default to 25%-per-step reduction, rounded to 1 decimal point. Existing breakpoint variables ($bp-md = 768px, $bp-lg = 992px, $bp-xl = 1200px in `_variables.scss`) map cleanly. Storage migration: each element's `font.size` becomes `{ xl, lg, md, sm }` instead of a flat single value.
- Compiler change is small: TypographyCompiler emits one `font-size` declaration today; in the new shape it emits the xl declaration plus three `@media (max-width: $bp-X)` blocks per element. The clamp-vs-mediaqueries question disappears (user has answered: media queries with fixed steps, user-controllable).
- Stub added to `session-outlines.md` Beta-1 scope under the title "Mobile Type Scaling — Per-Breakpoint Sizes in the Design System" (replaces the descope-flag in Phase 6 of the 243 prompt with concrete forward scope).
