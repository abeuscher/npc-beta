## Phases

The two components are independent; build/commit each naturally.

### Phase 1 — Prevention: the CI keystone test (component A)

One high-leverage, future-proof test: snapshot baseline → generate a full scrub spread across every relevant family (events, posts, products, contacts, orgs, donations, registrations, memberships, media…) via `RandomDataGenerator::generate()` → `RandomDataGenerator::wipe()` → assert **every table count returns to baseline and zero derived pages / orphan media remain.** Any future scrub-generating-but-not-cleaning code fails it automatically — it catches the whole class, not just the 349 bug. Pair it with **targeted cascade tests** extending the 349 delete-event→landing-page cascade (e.g. delete-model→owned-media). If `wipe()` leaves residue today, fix the leak rather than weakening the assertion.

### Phase 2 — Detection: the `app:data-hygiene` audit (component B)

A node-local artisan command reporting, for the box it runs on: orphan event pages (`type=event` with no backing event), any `source=scrub_data` records still present (pure cruft on a live box), orphan media (reuse `PruneOrphanMediaCommand`'s detection), and media rows whose owning model is gone. Two modes: a **count-only** default (a reusable method returning the aggregate count array — the safe signal Phase 2/C's health subcheck will later read) and a **deep mode** (`--deep`, the actual records — node-local). Read-only visibility; it reclaims nothing — cleanup stays in the prune commands. Pair the audit with the prune commands in the operator runbook so "what's piled up" → "clean it" is one documented flow.

- **Any schema/migration** — the audit computes from existing columns (`source`, `type`) and Spatie media ownership.

## Drift signals (stop and surface)

- `RandomDataGenerator::wipe()` already leaves residue, and cleaning it is larger than a contained fix (e.g. needs a new cascade across several models) — surface the scope before widening.
- The audit can't compute a category from existing data without a new column — surface before adding schema (the expectation is read-only).
- A "cruft" category turns out to be legitimate, retained data (not all `scrub_data` rows on a box are cruft if the operator deliberately seeded demo content) — surface the classification call before the audit flags false positives.
