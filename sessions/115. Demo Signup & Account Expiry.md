# 115. Demo Signup & Account Expiry

Build the demo user signup flow and automatic account cleanup. Users provide only an email address, receive a set-password link via the existing auth infrastructure, and are auto-deleted after one hour. Minimal PII collection, minimal retention.

---

## Design decisions (resolved before starting)

- Signup is email-only. The system sends a set-password link using the existing invitation/auth flow (the same `InvitationToken` + `AdminInvitation` mail pattern used for admin invites, adapted for demo).
- New demo accounts are auto-assigned the `demo` role. No other roles.
- The user's `name` field is set to a default (e.g. "Demo User") since we don't collect it. If the existing invitation flow requires a name, use this default.
- Account expiry: a scheduled artisan command runs every 10 minutes and deletes demo user accounts older than 1 hour (based on `created_at`). Deletion is hard — `users` row removed, cascading whatever the FK rules dictate.
- Resend API key: when `APP_ENV=demo`, the mail configuration reads the Resend API key from `.env` (`RESEND_API_KEY`) and ignores the value stored in `site_settings`. This ensures demo users cannot see the API key in the admin Mail Settings page (which is read-only per session 114) while still allowing the signup email to send.
- The signup page is a public route, not behind authentication. It should be visually simple and consistent with the marketing site's look.
- Rate limiting on the signup endpoint to prevent abuse (e.g. 5 requests per minute per IP).

---

## Phase 1 — Demo signup page

1. **Create a public signup route** (e.g. `/demo/signup`) and controller.
2. **Signup form** — single email input + submit button. Minimal styling consistent with the public site layout.
3. **On submit:**
   - Validate email (required, valid format).
   - Check if a demo account with this email already exists and is not expired. If so, show a message ("A demo account with this email is already active").
   - Create a `User` record with `name = 'Demo User'`, the provided email, a random password (they'll set their own via the link), and `is_active = false` (activated when they set their password).
   - Assign the `demo` role.
   - Generate an `InvitationToken` and send the set-password email.
   - Show a confirmation page ("Check your email for a link to set your password and access the demo").
4. **Rate limit** the endpoint — 5 requests per minute per IP.

---

## Phase 2 — Set-password flow adaptation

1. **Review the existing invitation acceptance flow** (`InvitationToken`, `AdminInvitation` mail, the accept/set-password route).
2. **Adapt as needed** so that demo invitations work through the same mechanism. The user clicks the link, sets a password, and lands in the admin panel.
3. **Ensure the demo user is marked `is_active = true`** upon password set.
4. **After login, demo users land on the admin dashboard** — same as any other user.

---

## Phase 3 — Account expiry

1. **Create an artisan command** (e.g. `demo:cleanup`) that:
   - Finds all users with the `demo` role where `created_at` is older than 1 hour.
   - Deletes them (hard delete). FK cascades handle dependent records.
   - Logs the count of deleted accounts.
2. **Register the command** in the scheduler to run every 10 minutes.
3. **Verify cascading behaviour** — when a demo user is deleted, confirm that sessions, activity log entries attributed to them, and any other FK-dependent records are handled cleanly (cascade or nullify as appropriate).

---

## Phase 4 — Resend .env override

1. **In the mail service provider** (or wherever mail config is resolved), when `APP_ENV=demo`, read `RESEND_API_KEY` from `.env` instead of from `site_settings`.
2. **Verify** that the Mail Settings admin page shows an empty/placeholder API key field for demo users (it should, since the DB has no key stored — the key lives only in `.env`).
3. **Verify** the signup confirmation email actually sends via Resend in demo mode. Note: session 114 forces mail to `log` driver in demo mode. The signup email is the one exception — it must actually send. Resolve this tension: either the mail driver override is scoped (suppress all mail except system auth emails) or the signup controller explicitly uses the Resend driver for this one send. Decide during implementation and document the choice.
