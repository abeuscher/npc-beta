## Session-specific deltas

- **The contract is fixed; this is a consumption migration, not a design.** 297 shipped and published the tier-1/tier-2 `--np-color-*` set in `docs/theme-color-tokens.md`. 300 migrates widgets *onto* it. Do not re-derive, rename, or expand the token set.
- **Two distinct classes, separated by the 297 demotion.** 297 demoted `$color-*` to `var(--np-color-x, #{$color-x})`. So a widget that references `$color-primary` is **already transitively token-backed** — it resolves through the demotion (lower risk; migrate to direct `var(--np-color-brand)` for contract clarity and to retire the indirection). A widget that hardcodes a raw hex (`#0172ad`, `#fff`) is **not** token-backed — it receives neither the Theme default nor any override. That second class is the silent-failure that consumed session 295 and is the real point of this session.
- **Survey is the scoping gate (Rule 11).** Size is unknown until the ~30-minute widget-hardcoding survey. The survey output — total count, the (a)/(b) breakdown, per-widget hit list — drives whether 300 is one session or splits into batches. Surface it before migrating.
- **The objective oracle is a Pest guard, written early.** A test that scans the widget SCSS corpus and fails on any raw hex literal / bare `$color-*` reference. It defines "done", measures remaining work as batches land, and becomes the permanent regression gate (same shape as the convention-drift gate the Code-Review track institutionalised).
- **Concrete-values throughline (carried from 297 close).** A hardcoded widget colour is the same defect class as the zero-margin typography default 297 fixed: a value that can never receive the system default or an override. The arc principle — *every themed value flows from the contract* — is what this session enforces at the widget layer.
- **Delivery & verification are the 295/297 rules, not guidelines.** Widget SCSS compiles into the public bundle (drift-prone → 296-guarded; green at start and close). Verify by **real-browser computed values across public + admin + preview**, never from emitted CSS/`curl`. `feedback_test_runs_not_parallel`.

## Open questions to resolve at session start

1. **Split shape (Rule 11).** Run the survey first. Decision: one session vs. N batches, and the batching axis (by widget group / by risk class / by hit count). Default: if the raw-hex class is small (≲ a dozen widgets) do it in one session; if wide, batch by widget group with the guard tightening per batch. Surface the proposed split before migrating.
2. **Fallback policy in migrated SCSS.** `var(--np-color-x)` bare vs `var(--np-color-x, <fallback>)`. Recommendation: **bare** — `.np-site` always carries every token (the bundle guarantees it), and a literal fallback reintroduces exactly the drift class this arc exists to close. Confirm; carve an exception only for a genuinely token-less surface, recorded.
3. **Tier mapping for semantic colours.** A widget's success/error/warning/focus accents map to tier-2 (`--np-color-success` / `-error` / `-warning` / `-focus-ring`) and on-brand text to `--np-color-brand-contrast` — not to tier-1 `brand`/`text`. Confirm the mapping table at survey time; semantic re-mapping is the one human-judgment surface — batch and surface ambiguous cases, don't silently decide.
4. **Non-colour hardcoded values.** rgba shadows, gradient stops, opacity tints. Recommendation: **out of scope** — not in the token contract; log to the housekeeping inbox, do not migrate. Confirm the boundary so the survey filters cleanly (colour hex / `$color-*` only).

## Phases

### Phase 1 — Survey + objective guard + split decision

Enumerate the widget SCSS corpus (`app/Widgets/*/styles.scss` and any `assets`-declared SCSS, plus `widget_types.css` rows). Classify every colour occurrence: (a) `$color-*` SCSS-var reference (transitively token-backed via the 297 demotion), (b) raw hex literal (silent-failure class), (c) non-colour (out of scope, log). Write the no-hardcoded-colour Pest guard now (initially xfail/baselined to the survey count). Surface: total, (a)/(b) breakdown, per-widget hit list, proposed Rule-11 split. **Stop for the split decision before Phase 2.**

### Phase 2 — Migrate to `var(--np-color-*)` (one or more batches)

Per batch: replace raw hex with the correct tier-1/tier-2 token; replace `$color-*` references with the direct `var(--np-color-*)` equivalent (retiring the indirection). Mirror the `_base.scss` consumption pattern. Rebuild the bundle (`build:public`); the no-hardcoded-colour guard tightens to zero for migrated widgets. Semantic-mapping judgment calls batched for the handoff.

### Phase 3 — Verify (objective) + drift + handoff

Fast suite green incl. the no-hardcoded-colour guard at zero. 296 drift guard FRESH. Real-browser computed-value verification across **three surfaces**: public (`.np-site` → widget colours resolve to tokens; flip a `theme_colors` value and confirm widgets follow), admin (no `.np-site` → no leak), page-builder preview (`.np-site` → faithful). Before/after multi-surface captures. Surface the semantic-mapping judgment batch for the user. If split, hand off the remaining batches as the next session.
