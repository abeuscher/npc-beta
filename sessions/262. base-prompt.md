8. Read OpenSpout's API surface (already a transitive dependency via `filament/actions v3.3.49` — no Composer change):
   - [`vendor/openspout/openspout/src/Writer/XLSX/Writer.php`](../vendor/openspout/openspout/src/Writer/XLSX/Writer.php) — entry point; `openToFile($path)` writes to a path, `addRow(Row $row)` per row, `close()` finalizes.
   - [`vendor/openspout/openspout/src/Common/Entity/Cell.php`](../vendor/openspout/openspout/src/Common/Entity/Cell.php) `fromValue()` — auto-detects cell type from a scalar: `bool` → BooleanCell, `int`/`float` → NumericCell, `DateTimeInterface` → DateTimeCell, `string` → StringCell, `=...` → FormulaCell, `null`/`''` → EmptyCell.
   - [`vendor/openspout/openspout/src/Common/Entity/Row.php`](../vendor/openspout/openspout/src/Common/Entity/Row.php) `fromValues($values)` — builds a Row from a flat array, calling `Cell::fromValue` per value.
9. **Verify-at-start questions** the session prompt names but the implementation should resolve before going deep:
   - **Cell-type semantics — the headline decision.** The 261 column specs emit pre-stringified values (e.g. `$contact->date_of_birth?->toDateString()` returns `"1990-05-15"`, `$donation->amount` returns a string-cast decimal from the model cast). With the existing spec passed verbatim to OpenSpout, every XLSX cell would be a `StringCell` — Excel would render dates and numbers as text, losing native sortability, date pickers, and numeric formatting. Three options: **(A)** ship strings-as-cells for parity with CSV (simplest; loses Excel-native UX); **(B)** add an optional `'type'` hint per column-spec entry that the XLSX branch consults to coerce the value back to its native type before passing to OpenSpout (Carbon parse for dates, `(float)` for decimals, `(bool)` for booleans) — CSV path unchanged; **(C)** invert the column-spec semantics so the value-callable returns the native type (Carbon, float, bool) and CSV stringifies on emit instead of in the spec — most invasive, breaks existing 261 contract. Lean: **(B)** — small additive change, preserves 261's CSV/JSON byte-for-byte, gives Excel users the native experience operators expect. Confirm before going deep.
   - **OpenSpout streaming shape.** OpenSpout's writer cannot target `php://output` directly — it needs a temp file path. Plan: `tempnam(sys_get_temp_dir(), 'xlsx-')` inside the `streamDownload` callback, `openToFile($tempPath)`, write rows, `close()`, then `readfile($tempPath)`, then `unlink($tempPath)`. Memory-bounded since OpenSpout streams rows to disk during write. Confirm.
   - **CF flattening behavior.** Mirrors CSV exactly — one column per `CustomFieldDef`, header is the CF's `label`. Not the JSON-style nested object. Operators editing XLSX in Excel expect flat columns.
   - **Filename / MIME.** `<resource>-YYYY-MM-DD.xlsx` and `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet` (the canonical Excel MIME, what OpenSpout's `openToBrowser` emits internally).
   - **Action label and icon.** Lean: `'Export Excel'` label, same `heroicon-o-arrow-down-tray` icon as the existing CSV/JSON actions for visual consistency in the ellipsis menu. Confirm.
10. Open `sessions/262. List Resource Exports — XLSX Format Add.md` and read the session prompt carefully.
11. Summarise your understanding and confirm you are ready to proceed.
