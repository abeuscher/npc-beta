## Why this iteration exists

User ran the first real end-to-end round-trip of A001/3's site-import/export pipeline (export → wipe DB → import) and surfaced two reported bugs:

1. **Nav widget unassigns itself from its menu on round-trip** — the nav widget on imported pages renders empty, as if no menu were assigned.
2. **Header / nav fonts rewrite to Georgia** — user set Lato in the Theme Editor's bucket controls, exported, wiped, re-imported, and saw the headings/nav rendering as Georgia (the body bucket value) rather than Lato.

Investigation traced bug 2 to a pre-existing **bucket-vs-element rendering gap** in the theme typography pipeline — the import/export round-trip is faithful, but the buckets only partially drive rendering (heading + nav buckets are emitted as CSS variables but never consumed by any SCSS rule; per-element families win on specificity when they're set). Filed as `sessions/theme-typography-bucket-rendering-gap.md` for a future small numeric session — not in A001's lane (the fix is in `resources/scss/_base.scss` + layout, both outside the agentic hard-no scope).

This iteration fixes bug 1 + the structural underpinning that made it manifest. Two coupled changes:

### Phase 1 — `exportSite()` enumerates all A001 entity types (commit pending)

A001/3 deliberately left `exportSite()` covering only pages + templates (310's original scope), with the rationale that "extending it is a separate UI-shaped decision." The user's first real round-trip surfaced the consequence: a full-site export bundle had no `navigation_menus` payload, so after wipe the menus weren't restored, so even a correctly-rewired widget would have nothing to reference. Extending `exportSite()` to enumerate navigation menus, products, events, and collections turns the rollup into an actually-complete site snapshot.

- `ContentExporter::exportSite($opts)` no longer delegates to `exportBundle()`. Instead it directly enumerates `Page` + `Template` + `NavigationMenu` + `Product` + `Event` + `CollectionModel` ids and assembles a payload with all six sections, plus the opt-in design + media payloads.
- New opt: `with_registrations` (default false). Mirrors the per-entity `exportEvents()` flag — registrations are operator data, not template content. Set to true to include registrations in the site rollup.
- The manifest section list now covers all six A001 entity types automatically because every payload key is reflected in the manifest at envelope-build time.

### Phase 2 — Nav widget config rewiring (commit pending)

The Nav widget is the only entity-referencing widget in the codebase that stores its reference as a **UUID** rather than a slug or handle. (Verified by grepping `options_from` across all widget definitions: `BoardMembers`, `LogoGarden`, `Carousel`, `BarChart` all use `collection_handle`; `EventDescription` + `EventRegistration` use `event_slug`; `WebForm` uses `form_handle`; `ProductDisplay` uses `product_slug`. Only `Nav` uses `navigation_menu_id`.) When a nav menu is recreated on import with a fresh UUID (because `importNavigationMenu` upserts by handle, not id), the widget's stored `navigation_menu_id` becomes a dangling reference.

The fix mirrors the existing `template_name` / `page_slug` patterns:

- **Export:** `ContentExporter::serializeWidgetConfigReferences()` walks each nav widget's config and, when `navigation_menu_id` is set, looks up the menu and adds `navigation_menu_handle` to the serialized config alongside the UUID. The original UUID stays in the bundle for legibility + legacy-importer fallback; the new handle is the portable identifier.
- **Import:** `ContentImporter::sanitizeWidgetConfig()` attempts handle resolution during widget hydration (best-effort — the menu may already exist on the target). The authoritative resolution happens in a new post-pass `relinkNavWidgetMenus()` that runs after both pages and navigation menus have landed inside the same transaction. The post-pass walks every nav widget on the install, re-resolves its handle to the current menu_id, and warns + nulls if the handle can't be matched. Mirrors the existing `relinkTemplateChrome` two-pass pattern.
- **Legacy bundles** (pre-A001/4 with only the UUID): the importer verifies the UUID still resolves against an existing menu; if it doesn't, warn + null. Old bundles continue to import cleanly when their UUIDs happen to match existing menus on the target.

### Bug 2 status — not fixed here, filed for a numeric session

Filed as `sessions/theme-typography-bucket-rendering-gap.md`. The fix is three-part (wire `--font-family-heading` / `--font-family-nav` into SCSS; bucket-to-element auto-propagation in Theme Editor; optionally promote `nav` to a first-class element in the resolver). Minimum-viable cut is the SCSS wiring — Part A in the writeup. Out of A001's hard-no scope (`resources/scss/`).

## Tests

6 new Pest cases:

- `exportSite includes navigation_menus, products, events, and collections` — verifies the rollup payload has all six section keys and the manifest section list reflects them.
- `exportSite with_registrations=true includes event registrations` — verifies the new opt.
- `exporter emits navigation_menu_handle alongside navigation_menu_id on nav widgets` — the additive serialization.
- `round-trips a nav widget through exportSite when the menu is recreated with a fresh UUID` — the end-to-end fix proof. Creates a nav widget pointing at a menu, calls `exportSite()`, deletes the menu + widget, imports the bundle, asserts the menu was recreated with a NEW UUID and the widget's config has been rewired to that new UUID.
- `warns and nulls navigation_menu_id when the referenced menu is absent from bundle and target` — the failure path: bundle has a nav widget pointing at a ghost menu (no menu in bundle, none on target). Post-pass warns + nulls.
- `handles legacy bundles without navigation_menu_handle by verifying UUID still resolves` — backwards-compat: bundle without handles, importer verifies UUID against existing menus, keeps it if found, warns + nulls if not.

## Production deployment notes

None beyond standard `migrate`. No worker restart required. Two behavior changes worth flagging in release notes:

1. **`exportSite()` bundles are now larger** because they include navigation menus, products, events, and collections in addition to pages + templates + design + media. Operators using the existing Site Export action will see bigger downloads; nothing in the operator flow needs to change.
2. **Nav widgets exported pre-A001/4 will resolve as long as the menu still exists on the target install** — legacy bundles continue to work. Nav widgets exported post-A001/4 carry both the UUID and a portable handle.

## Deferred / carry-forward

- **Bug 2 fix** — see `sessions/theme-typography-bucket-rendering-gap.md`. Three-part fix, minimum-viable is one SCSS rule + one Blade emit. Recommended as a one-iteration small session.
- **Generic widget config reference rewiring** — currently `serializeWidgetConfigReferences()` is hard-coded for the Nav widget's `navigation_menu_id`. If another widget ever joins the UUID-reference club, this helper grows a second case. A more general "look at the widget's `config_schema` for `options_from` and resolve to handle/slug" approach would generalize it — but premature now, with a single case. Note the pattern-of-one for future review.
- **Forms / widget_presets / custom_field_defs in `exportSite()`** — A001/3 listed these as next content-shaped sections. Now that the rollup has the right shape, adding them is mechanical. Probably next iteration if you want them, otherwise queue for the eventual forms-track session.
- **SiteSettings in `exportSite()`** — currently `exportSite()` does NOT include the curated SiteSettings slice that A001/3 added. Adding it is a one-line `payload['site_settings'] = $this->serializeSiteSettings(...)` change, but the slight architectural question is whether the rollup operator wants brand-identity values (site_name, etc.) to travel by default. Defer to your call.
