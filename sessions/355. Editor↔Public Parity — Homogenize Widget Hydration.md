# 355. Editor↔Public Parity — Homogenize Widget Hydration

The page-builder canvas re-implements widget **hydration** (HTML → libs loaded + correctly-layered CSS + Alpine/Swiper/Chart initialized) as a bespoke second copy of what the public site does, and the two copies drift. Consolidate the widget-owned half into **one shared `hydrate(rootEl)` routine both surfaces call** — public once over `document`, the editor over each freshly-injected canvas subtree — so the init-race and vendor-layer bug classes become structurally impossible instead of separately patched.

## Scoping (verify against code, build from it, don't re-derive)

Model + view are already single-sourced — one API / `WidgetRenderer` produces the widget HTML for both surfaces. The divergence is **client-side hydration + interaction**, and it splits cleanly:

- **Widget-owned** (HTML / CSS / libs / init) — **must** be single-sourced. This is the session's target.
- **Surface-owned** (selection / drag / inline-edit / host chrome) — legitimately separate per surface. **Out of scope** — do not unify.

**The two copies of the widget-owned half today:**

- **Public** (`resources/js/public.js`): statically imports Alpine + Swiper (+ modules) + Chart, assigns `window.Swiper`/`window.SwiperModules`/`window.Chart`/`window.Alpine`, registers the `customSelect` Alpine component + theme store on `alpine:init`, then `Alpine.start()` **once over `document`**. Libs are **bundled**; init is whole-document and one-shot.
- **Editor** (`resources/js/page-builder-vue/composables/useLibraryLoader.ts`): `loadLibs(libs)` **lazy-loads** per-lib CSS (fetched and wrapped in `@layer reset` so it doesn't beat `@layer widgets` — the exact session-333 swiper-pager fix) + per-lib JS from the `window.__widgetLibs` manifest (emitted by `AdminPanelProvider`'s `panels::head.end` hooks); `reinitAlpine(scopeEl)` destroys + re-creates Swiper/Chart/Alpine trees **over a freshly-injected canvas subtree**, with `x-ignore` lift/restore bookkeeping and a post-init `swiper.update()`. Libs are **lazy/manifest-driven**; init is per-subtree and repeatable.

Same goal, two mechanisms — they drift (the `loadLibs` comment is itself a scar from a drift bug). The third scattered piece is **`AdminPanelProvider`** (the admin head-hook lib emission: `widgetEditorCss()`, `widgetJs()`, the per-lib bundles, and `window.__widgetLibs` — all libs on every admin page today).

**The shape of the fix:** a single `hydrate(rootEl)` (widget-owned hydration: ensure the subtree's libs are present + correctly layered, then init/re-init Alpine + Swiper + Chart over it) that **both** surfaces call — public once over `document` at load, the editor over each injected canvas subtree on mutation. `WidgetAssetResolver` (`widgetEditorCss` / `widgetJs` / `allLibs` / `libsForWidgets`) is the asset-resolution seam.

## Open questions to resolve at session start

1. **Where does the shared routine live, and how do both build graphs reach it?** `public.js` is plain JS (Vite public entry); the editor is the Vue/TS app (`resources/js/page-builder-vue/**`). The shared `hydrate()` needs to be a module both import without duplicating. Propose the module location + the import seam at the gate — this is the one architectural choice worth surfacing before going deep.
2. **Bundled-libs vs lazy-libs reconciliation.** Public bundles Swiper/Chart; the editor lazy-loads them from the manifest. Does the shared routine treat "libs already present" as a precondition (public path) and lazy-load only when absent (editor path)? Or do both move to one strategy? Decide the lib-presence contract before unifying the init half.
3. **Alpine ownership.** Public calls `Alpine.start()` once (document-wide, first-init semantics); the editor uses `destroyTree`/`initTree` per subtree (re-init semantics). The shared `hydrate()` must do the right one per surface (first-init vs re-init) — settle whether that's a parameter, a "has this tree been initialized" check, or two thin entry points over one core.

## Phases

Single architectural refactor against a green parity net. Split across iterations per Rule 11 if the seam proves load-bearing; otherwise land together.

### Phase 1 — Design the `hydrate(rootEl)` contract

Map the three current pieces (`public.js` auto-init, `useLibraryLoader`'s `loadLibs`/`reinitAlpine`, `AdminPanelProvider`'s lib emission) onto one routine. Define the widget-owned/surface-owned boundary precisely, the lib-presence contract (Q2), and the first-init-vs-re-init split (Q3). Decide the shared module's home and import seam (Q1). Surface this at the Open Gate before writing it.

### Phase 2 — Implement the shared routine and wire both surfaces

Extract the widget-owned hydration into the shared `hydrate(rootEl)`. Route `public.js` (over `document`) and the editor canvas (over each injected subtree, replacing the bespoke `reinitAlpine` body) through it. Single-source the lib-load + layered-CSS injection + Alpine/Swiper/Chart init. Keep the surface-owned editor code (selection/drag/inline-edit) untouched. `npm run build` after.

### Phase 3 — Verify against the parity harness

The 333 editor↔public parity specs (`tests/e2e/page-builder/full-width-matrix.spec.ts`) + the 354 `chrome-parity.spec.ts` are the green regression net. Confirm no init-race / swiper-default-pager / uneven-grid / Alpine-double-init regressions on either surface; add a parity assertion if the refactor exposes a guardable invariant (e.g. "a freshly-dropped carousel hydrates to the designed pager, not Swiper's default"). Drive it with Playwright; pull the owner in for any visual call.

## Drift signals (stop and surface)

- The shared module can't be cleanly imported by both build graphs without duplicating Alpine/Swiper into two bundles — surface the seam problem before forcing it.
- Single-sourcing the init forces a change to the **public** hydration that risks the live site's widget behaviour — the public path is load-bearing; surface before committing.
- The widget-owned/surface-owned line turns out blurry for a specific widget (an interaction that's half hydration, half editor selection) — surface the specific case rather than guessing the boundary.
