## What was built

### Phase 1 — Database & broker config
- Migration: `portal_password_reset_tokens` table (email PK, hashed token string, nullable created_at). Entirely separate from the admin `password_reset_tokens` table.
- `config/auth.php`: added `portal_accounts` password broker pointing at the new table, 60-minute expiry, 60-second throttle.

### Phase 2 — Routes
Four new routes added to `web.php` under the portal auth block:
- `GET /forgot-password` → `portal.password.request`
- `POST /forgot-password` → `portal.password.email` (throttle: 5/min)
- `GET /reset-password/{token}` → `portal.password.reset`
- `POST /reset-password` → `portal.password.update`

Patch 001 added: `GET /forgot-password/sent` → `portal.password.sent`

### Phase 3 — Controllers
- `ForgotPasswordController` — enumeration-safe: always returns the same response regardless of whether the email exists. Delegates to `Password::broker('portal_accounts')->sendResetLink()`.
- `ResetPasswordController` — validates token via the broker, calls the broker `reset()` closure (sets new password, rotates remember token, fires `PasswordReset` event), redirects to login with flash on success.

### Phase 4 — Email
- `PortalPasswordReset` mailable — uses `EmailTemplate::forHandle('portal_password_reset')`. Tokens: `{{first_name}}`, `{{reset_url}}`. Follows the `EventReminder` pattern (renders via `mail.system-email`).
- `PortalAccount::sendPasswordResetNotification($token)` override — sends the mailable instead of Laravel's default notification.
- `portal_password_reset` handle added to `EmailTemplate::defaults()` and `DatabaseSeeder`.

### Phase 5 — Views
- `portal/forgot-password.blade.php` — email input form
- `portal/forgot-password-sent.blade.php` — confirmation page (added in patch)
- `portal/reset-password.blade.php` — token + email + new password + confirm
- `portal/login.blade.php` — added "Forgot your password?" link and flash status display

### Patch 001
- POST-redirect-GET on forgot-password: `store()` now redirects to `portal.password.sent` instead of flashing back to the form. Refreshing the confirmation page no longer risks resubmission.
- Client-side password match validation: vanilla JS `@push('scripts')` block added to `reset-password.blade.php` and `signup.blade.php`. Inline "Passwords do not match." message appears as you type, disappears when fields match. No framework dependency.

## Production deployment notes

After deploying this session, the following must be run on the production server (seeders are not automatic):

```bash
docker exec nonprofitcrm_app php artisan migrate
docker exec nonprofitcrm_app php artisan db:seed --class=DatabaseSeeder
```

`DatabaseSeeder` is safe to run against production — all inserts use `firstOrCreate`/`updateOrCreate` and the demo data block is guarded by `app()->environment('local')`.
