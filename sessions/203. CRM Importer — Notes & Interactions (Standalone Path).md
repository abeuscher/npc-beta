# 203. CRM Importer — Notes & Interactions (Standalone Path)

Build a standalone notes importer that accepts a separate activities CSV (one row = one note) and writes full structured notes — type, subject, status, body, occurred_at, follow_up_at, outcome, duration_minutes, plus a `meta` JSONB catchall for source-specific fields. Contact lookup by email / external_id / phone via the same patterns used by the events / donations / memberships importers.

This closes the importer arc: every canonical content type (contacts, events, donations, memberships, invoice details, notes) now has a dedicated importer with a consistent wizard flow, field-map persistence, and update-strategy support.

---

## Design decisions (resolved before starting)

- **Standalone importer, not an enhancement of the existing `__note_contact__` path.** The contacts-importer `__note_contact__` sentinel stays untouched — it covers the degenerate "one notes column on a contacts CSV" case (LGL-simple, hand-rolled spreadsheets). Session 203 addresses the dominant case: Raiser's Edge / Salesforce / CiviCRM / Bloomerang-style separate activities CSV with one row per note.
- **Contact not found = row errors out.** No auto-create of stub contacts. The error-table surface from session 202 (Spec C) is reused: the row lands in the error table with a specific message, and the admin is directed to run the contacts importer first. This matches the "contacts are always loaded first" workflow the events/donations/memberships importers already assume.
- **Attach target is Contact only in v1.** Organization as a `notable_type` target is deferred until the Organization Timeline surface lands (session 198 log flagged this as a gap). Event as target (e.g. notes about an event, not an attendee) is also deferred. The importer hardcodes `notable_type = App\Models\Contact`.
- **Free-text tolerance for `type` and `status`.** Session 198 already tolerates free-text in these columns ("phone call" doesn't auto-normalize to "call"). The Timeline surfaces non-canonical types via `getNonCanonicalTypes()`. The importer follows the same posture: write the source value verbatim, no normalization.
- **Unmapped columns → `meta` JSONB.** Each column the user maps to the `__custom_note__` sentinel gets its value written into `meta.{handle}` where the handle is derived from the source header (lowercased, non-alphanumerics → underscore). Unlike the contacts importer's custom fields path, there is no separate Custom Fields admin page for notes — `meta` is a schemaless catchall surfaced read-only on the Timeline via the `Source fields` `<details>` disclosure from session 198.
- **Update-strategy requires a dedupe key on notes.** Session 203 adds `external_id` to the `notes` table plus the standard `(import_source_id, external_id)` index. Matches the donations / memberships / transactions pattern. Without this column, re-imports would always create duplicates.
- **Rich body tolerance.** The `body` column stores rich HTML (Quill-authored in the admin). The importer accepts plain text and stores it as-is — no HTML escaping, no auto-wrapping in `<p>` tags. Sources that export pre-formatted HTML (rare) flow through unchanged. Admins can edit post-import to reformat if needed.
- **Duration parsing.** `duration_minutes` is stored as integer. Accept:
  - Plain integers: `30` → 30
  - Numeric strings with "min" / "minutes" suffix: `"30 min"` → 30
  - `HH:MM:SS` or `MM:SS` format: `"00:30:00"` → 30, `"45:00"` → 45 (ambiguous — assume longer format when two colons present, minutes:seconds when one)
  - Free-text: discarded (store null, log a skip reason)

## Phase 3 — Admin page + progress page

Two new Filament pages following the Donations pattern exactly:

  - `'duration', 'duration minutes', 'call duration', 'duration (minutes)'` → `note:duration_minutes`
  - `'external id', 'activity id', 'action id', 'interaction id'` → `note:external_id`
  - `'email', 'email address'` → `contact:email`
  - `'user id', 'constituent id', 'contact id'` → `contact:external_id`
  - `'phone', 'phone number'` → `contact:phone`

Wizard submit action: `runImport`, Stage Import button, testid `import-commit-button`.

### `app/Filament/Pages/ImportNotesProgressPage.php`

- Extends `Page`, uses `InteractsWithImportProgress` trait.
- `processOneRow()` implements:
  1. Build `noteAttrs` (note:* fields) and `contactLookup` (contact:* fields) from the row.
  2. Resolve contact via `resolveContactByNamespacedKey` using the configured `contact_match_key`.
  3. If contact not found → return `['outcome' => 'error', 'row' => $rowNumber, 'message' => "Contact not found for {$matchField} = {$matchValue}. Run the contacts importer first.", 'identity' => [...]]`. No auto-create.
  4. Match existing note by `(import_source_id, external_id)` if `external_id` is set.
  5. Apply duplicate strategy:
     - `skip`: return `['outcome' => 'skipped', 'skipReason' => 'duplicate_skipped']` on existing match.
     - `update`: build staged-update attrs via `buildNoteStageAttrs()`, call `stageSubjectUpdate()`, return `['outcome' => 'updated']`.
     - `duplicate`: ignore the existing match, create a new note.
  6. Create note: `Note::create([...])` with `notable_type = Contact::class`, `notable_id = $contact->id`, all the structured fields, plus `meta` built from `__custom_note__`-mapped columns.
- `buildRowContext` returns `columnMap`, `customFieldMap` (for `__custom_note__`), `duplicateStrategy`, `contactMatchKey`, `relationalMap`.
- `afterPiiScan` hook unchanged from default.

## Phase 4 — CSV template + generator + seeder

### `CsvTemplateService::noteHeaders()`

New static method. Emits:

- Each `NoteFieldRegistry` label prefixed with `'Note '` (e.g. `'Note Type'`, `'Note Subject'`, `'Note Body'`, `'Note Occurred At'`, …, `'Note External ID'`) — matches the events-template naming convention post-session-202.
- Contact match columns: `'Email'`, `'User ID'`, `'Phone'` (matches donations / memberships / invoice_details template shape).

Register `'notes'` as a fourth `stream($type)` case.

### `FakeCsvComposer::composeNotes(int $count, array $contactEmails): array`

- Takes a contact-email pool (generated earlier in the run).
- Produces `$count` rows, each representing one note on one contact.
- Cross-file references resolve: every `Email` column value appears in `contacts.csv` from the same run.
- Synthetic values:
  - Type: randomised over `['call', 'meeting', 'email', 'note', 'task', 'letter', 'sms']`.
  - Subject: short phrase from a small pool.
  - Status: usually `'completed'`; occasionally `'scheduled'` or `'no_show'`.
  - Body: one- or two-sentence synthetic text.
  - Occurred At: `dateTimeBetween('-2 years', 'now')`.
  - Follow-up At: ~20% of rows; always in the future; only set when `status != 'completed'`.
  - Outcome: present for `call` / `meeting` types only.
  - Duration: present for `call` / `meeting` types, 5–90 minutes.
  - External ID: `N-{6-digit}`.

## Out of scope

- **Organization or Event as `notable_type` targets.** Contact only in v1.
- **Custom Fields UI for notes.** Unmapped columns go into `meta` JSONB without any separate admin surface (unlike `contacts.custom_fields` which has its own Custom Fields admin page). Schemaless by design.
- **The `__note_contact__` appendage on the contacts importer.** Stays exactly as-is. Session 203 does not enhance, deprecate, or reshape it.
- **Canonical-type consolidation UI.** Session 198 log deferred this; still deferred. The importer writes whatever `type` string the CSV has.
- **Follow-up / task dashboard.** The `follow_up_at` column is populated by the importer; the dashboard surface that uses it is a separate post-Beta stub.
- **Update-strategy Playwright spec for notes.** Pest covers the service layer; Playwright happy-path is sufficient for Beta 1 confidence.
- **Additional competitor source presets for notes** (Raiser's Edge "Actions", Salesforce Task/Event, etc.). These belong in the "Importer Source Presets — Competitor Coverage" stub, not this session.
