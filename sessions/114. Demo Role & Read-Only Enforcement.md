# 114. Demo Role & Read-Only Enforcement

Create the `demo` role with view-only permissions and enforce read-only access across every surface of the admin panel. This is the foundational session for demo mode — every subsequent demo session depends on the permission layer being airtight here.

---

## Design decisions (resolved before starting)

- Demo mode is active when `APP_ENV=demo`. All demo-specific behaviour checks this value.
- The `demo` role is created via the existing Spatie permission system. It receives only `view_any_*` and `view_*` permissions — no create, update, or delete permissions on any resource.
- The Users resource list is scoped so demo users only see seeded non-demo users (staff accounts). Demo-role users are excluded from the query. This avoids exposing real PII from other demo signups.
- Settings pages are fully read-only for demo users — all form fields disabled, save buttons hidden.
- Routing-related settings (blog prefix, portal prefix, events prefix) are hidden entirely in demo mode — not just read-only, but removed from the settings UI.
- Outbound email is suppressed in demo mode by forcing the mail driver to `log`. This closes the open-relay risk on public-facing form submission and event registration endpoints.
- Horizon is blocked for demo users.
- `APP_DEBUG_TOOLS` must be `false` on any demo instance (verified, not changed).
- Permission checks must live in the methods, not just in UI visibility. A user with browser devtools can call any public Livewire method directly — the method itself must refuse the operation for demo users.

---

## Phase 1 — Demo role & permissions

1. **Create the `demo` role** in the `RoleSeeder` (or a dedicated `DemoSeeder` — decide based on existing seeder structure). Assign only `view_any_*` and `view_*` permissions for every resource.
2. **Add a `isDemoMode()` helper** — a lightweight check (`app()->environment('demo')`) available globally. Place it where the existing permission helpers live.
3. **Add a `isDemo()` convenience method** on the User model that checks `$this->hasRole('demo')`.

---

## Phase 2 — Resource-level read-only enforcement

Audit every Filament resource and enforce read-only for demo users:

1. **Table actions** — hide edit, delete, bulk delete, invite, and any custom action for demo users.
2. **Create pages** — block access entirely. Demo users should not reach create routes.
3. **Edit pages** — demo users can view the edit page (to see the form layout) but all fields must be disabled and save buttons hidden. Alternatively, redirect to a read-only view if that's simpler.
4. **Relation managers** — any attach, detach, create, or delete actions within relation managers must be hidden.

Work through every resource in the admin panel systematically. Refer to `docs/app-reference.md` for the complete list.

---

## Phase 3 — Page-level & settings enforcement

1. **Settings pages** (General, Mail, Finance, CMS) — all form inputs disabled for demo users. Save actions hidden. Routing fields (blog prefix, portal prefix, events prefix) removed entirely from the rendered form in demo mode.
2. **Import pages** (Importer, Import Contacts, Import History) — block access for demo users entirely.
3. **Generate Tax Receipts page** — block access for demo users.
4. **Giving Summary / Donors page** — allow view access (read-only data display).

---

## Phase 4 — Livewire method hardening

1. **Page builder** (`PageBuilder`, `PageBuilderBlock`) — all public methods that write data (save, reorder, delete, add block, etc.) must check permissions and refuse for demo users. A demo user can view the page builder UI but not modify anything.
2. **Any other Livewire components** with public write methods — audit and gate.

---

## Phase 5 — Infrastructure gating

1. **Mail driver override** — when `APP_ENV=demo`, force `config('mail.default')` to `log` in a service provider boot method. This ensures no outbound email regardless of DB settings.
2. **Horizon access** — add the demo role to the Horizon auth gate deny list.
3. **Verify `APP_DEBUG_TOOLS`** — confirm the debug generator widget's `canView()` check works correctly and document that demo instances must set this to `false`.

## Testing

- **Slow test groups to run this session:** `demo`
- **New tests expected:** yes — a dedicated `demo` test group that:
  - Boots the app with `APP_ENV=demo`
  - Verifies demo role has only view permissions
  - Verifies create/edit/delete routes return 403 for demo users
  - Verifies settings pages render as read-only
  - Verifies Livewire write methods refuse operations for demo users
  - Verifies mail driver is forced to `log`
  - Verifies Horizon is blocked
  - Verifies Users list excludes demo-role users
  - Verifies routing settings are hidden

These tests go in the `slow` group so they don't run in the standard fast suite. They are the smoke test for the entire demo permission layer.
