# 211. Widget Primitive ‚Äî Phase 2 (Slot Taxonomy)

Declare the Slot primitive formally ‚Äî one class per slot, carrying ambient context, layout constraints, and config surface ‚Äî then declare the v2.0 three-slot set (dashboard grid, record-detail sidebar, page-builder canvas) and retrofit the existing page-builder rendering path to resolve its ambient context through the slot declaration.

Scope is deliberately narrow: **no new UI, no live dashboard or record-detail surface**. The dashboard-grid and record-detail-sidebar slots are declaration-only this session ‚Äî their consumers land in Phases 3 and 5. The page-builder-canvas slot gets a retrofit because it is the one existing consumer, and proving the primitive holds against today's rendering path is the load-bearing test.

---

## Design decisions (resolved before starting ‚Äî do not relitigate)

Settled across the Phase 1 track (sessions 209 and 210) and the scoping for this session:

- **V1 slot set is exactly three: `page_builder_canvas`, `dashboard_grid`, `record_detail_sidebar`.** Additional slots named in the premise doc (record-detail main, form field, email block) stay deferred per `widget-primitive-migration.md`. No `record_detail_main` split in this session ‚Äî a single `record_detail_sidebar` declaration covers the v1 surface.
- **Slots are PHP classes, one per slot, under `app/WidgetPrimitive/Slots/`, extending a base `Slot` abstract class.** Decision over a registry-array shape because slots have per-slot behaviour (ambient context construction, constraint descriptions) that demands polymorphism, not just config.
- **Ambient-context generalisation is Phase 5 work, not this session.** Per `widget-primitive-migration.md` Phase 5: "`PageContext` generalizes to `SlotContext` with subtypes (page, dashboard, record-detail)." Phase 2 declares slot methods whose return types document what each slot will surface, but only `PageBuilderCanvasSlot::ambientContext()` has a live implementation body. The other two slots throw `RuntimeException('Slot ambient context not yet wired ‚Äî lands with Phase 5')` from their method bodies. This is honest about the gate between declaration and live consumers.
- **Layout constraints are declarative metadata only in this session.** Each slot's `layoutConstraints()` returns a typed description (shape TBD in Phase 1 ‚Äî a simple array or a `LayoutConstraint` value object). No runtime enforcement. Enforcement is a later phase, likely bundled with the record-detail IDOR work in Phase 5.
- **Widget-to-slot compatibility uses a new `WidgetDefinition::allowedSlots(): array` base method.** Default return is `['page_builder_canvas']` ‚Äî preserves every existing widget's current compatibility unchanged. No widget definitions need editing this session.
- **`SlotRegistry` is a singleton service.** Pattern matches `WidgetRegistry`. Bound in `WidgetServiceProvider::register()`; slot instances registered in `boot()`.
- **Retrofit scope is exactly one touch:** `WidgetRenderer` constructs its `SlotContext` by calling `app(SlotRegistry::class)->find('page_builder_canvas')->ambientContext(...)` instead of `new SlotContext(...)` inline. Byte-equivalent output maintained. Every existing integration test continues to pass.
- **No Filament work. No dashboard page. No record-detail panel. No new routes.** Those all belong to Phases 3 and 5.
- **Pre-beta license carries over.** No backward-compat shims. `WidgetRenderer`'s ambient-context construction is simply replaced ‚Äî no parallel path.

## Phase 1 ‚Äî The `Slot` primitive

`app/WidgetPrimitive/Slot.php` ‚Äî abstract class with the following shape:

- `handle(): string` ‚Äî stable identifier (e.g. `'page_builder_canvas'`).
- `label(): string` ‚Äî human-readable label for admin UI surfaces.
- `ambientContext(...)`: returns the slot's ambient-context value object. Signature varies by slot (see each slot's declaration below); the abstract parent may type it as `object` or use per-slot overrides.
- `layoutConstraints(): array` ‚Äî declarative description of the slot's layout shape. Shape convention to agree in this session ‚Äî likely an associative array with keys like `allowed_appearance_fields`, `dimensions` (for grid slots), `column_stackable`, etc. Document the shape in a docblock on the abstract method.
- `configSurface(): ?string` ‚Äî identifier or class name for the admin surface that configures widgets in this slot. `null` for slots whose surface isn't built yet.

`app/WidgetPrimitive/SlotRegistry.php` ‚Äî singleton service. Methods:

- `register(Slot $slot): void`
- `find(string $handle): ?Slot`
- `all(): array` (ordered by registration)

Bind as singleton in `WidgetServiceProvider::register()`.

Tests: construction, registry register/find round-trip, duplicate-handle behaviour (last-registered wins, matching `WidgetRegistry`'s current behaviour ‚Äî confirm during implementation).

## Phase 2 ‚Äî The three slot declarations

`app/WidgetPrimitive/Slots/PageBuilderCanvasSlot.php`:

- `handle`: `'page_builder_canvas'`
- `ambientContext(PageContext $pageContext, ?Page $currentPageOverride = null): SlotContext` ‚Äî returns today's `SlotContext` shape exactly. No behavioural change from what `WidgetRenderer` currently constructs inline.
- `layoutConstraints()`: permissive ‚Äî full appearance config allowed, no dimension constraint, full-width honoured.
- `configSurface()`: `'page_builder_vue'` (identifier only; no class coupling this session).

`app/WidgetPrimitive/Slots/DashboardGridSlot.php`:

- `handle`: `'dashboard_grid'`
- `ambientContext(...)`: signature documents "current admin user + request context, no current page." Body throws `RuntimeException('Slot ambient context not yet wired ‚Äî lands with Phase 3')`.
- `layoutConstraints()`: grid-cell dimensions (tile units ‚Äî shape TBD in this session, e.g. `['dimensions' => ['width' => 'int', 'height' => 'int']]`), no full-width, background/spacing allowed.
- `configSurface()`: `null`.

`app/WidgetPrimitive/Slots/RecordDetailSidebarSlot.php`:

- `handle`: `'record_detail_sidebar'`
- `ambientContext(...)`: signature documents "current record (Contact/Event/Organization/etc.) + current user." Body throws `RuntimeException('Slot ambient context not yet wired ‚Äî lands with Phase 5')`.
- `layoutConstraints()`: compact, column-stackable, no full-width, bounded appearance surface.
- `configSurface()`: `null`.

Register all three in `WidgetServiceProvider::boot()`.

Tests: one unit test per slot asserting handle, label, layout-constraint shape, and that `ambientContext()` does the right thing (returns `SlotContext` for page-builder-canvas; throws `RuntimeException` for the other two).

## Phase 3 ‚Äî Widget-to-slot declaration

`WidgetDefinition::allowedSlots(): array` ‚Äî new base method returning `['page_builder_canvas']`. Override per widget later when other slots come online.

No widget definitions change in this session ‚Äî the default preserves current behaviour.

Tests: one unit test on `WidgetDefinition` asserting the default; one assertion in an existing widget-definition test (pick any migrated widget) confirming it returns the default.

## Phase 4 ‚Äî Retrofit `WidgetRenderer`

`WidgetRenderer::render()` replaces `new SlotContext($pageContext, $tokenPage)` with a call through the registry: `app(SlotRegistry::class)->find('page_builder_canvas')->ambientContext($pageContext, $tokenPage)`.

Confirm byte-equivalent HTML output for all three migrated widgets (Blog Listing, Text Block, Carousel) via the existing `WidgetPrimitiveIntegrationTest`.

Query count on `PageBuilderApiTest > buildTree eager-loads owner` stays at or below the 210 ceiling (`< 29`). Bind the registry as a singleton so slot-lookup doesn't scale with widget count.
