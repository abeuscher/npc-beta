# 326. Event Landing Pages — Simple & Standard

Give every event a real landing page, created at event-creation time via a Simple / Standard choice — retiring the optional/no-LP model so the calendar and listing always have a resolvable URL to link to.

## Why this session exists (the 325-close decision)

Building the mini-calendar made it clear the "events optionally have a landing page" model was collecting compensating complexity: a dead-end `url` fallback (no-LP events linked to the bare `/events` index), `has_landing_page` conditionals, and an inline event-detail surface invented inside the calendar to avoid the dead end — with the listing widget about to need its own third treatment. The owner's resolution: **every event has a landing page.** The original goal was never "no page" — it was "don't force art and copy for a simple event." A **Simple** page (no image required) honours that better than no-page did, while killing the dead end at the root and collapsing every downstream widget to "just link."

## What ships

### The create-time choice

- On the event **create** form (`EventResource` / `CreateEvent`), a **Landing page: Simple / Standard** toggle.
- **Simple** is offered **only for free events** (no ticket tiers). **Standard** is **forced** the moment the event is ticketed — a paid event must have the registration surface.
- On save, the event's `landing_page_id` is set to a newly-created `Page` seeded with the chosen preset. The choice is **not** persisted as a flag; the page's widgets are the state.

### The two presets (both are normal pages, editable in the builder afterward)

- **Simple LP:** `hero` + event-details (`EventDescription`). Image **optional** (the hero supports one but none is required). **No** registration widget.
- **Standard LP:** the fuller v1 event page per §1b — `hero` + `EventDescription` + `EventRegistration` (registration card). About folds into the description; Share + Related-events + `Event` schema/OG are part of the standard build (see Phases; trim to fast-follow only if the gate agrees).

### Deletion protection

- The LP is a plain page, but an event must never be left LP-less. Plug into the **existing `PageObserver` deletion guard** (the block-with-counts pattern from B1a / session 255): a page that is an event's `landing_page_id` cannot be deleted while the event references it. **Reuse the existing scheme — do not invent a new one.**

### Relink the widgets + cleanup

- Point the mini-calendar and `EventsListing` event rows at the LP URL (`events.landingPage->slug`), which now always exists.
- In `SystemModelProjector::projectEvent`, the `url` no longer needs the `/events`-index dead-end fallback (every event has an LP); `has_landing_page` becomes effectively always-true. Simplify accordingly.
- **Mini-calendar inline-expand:** keep it as a deliberate **rail-peek enhancement** (it is good UX and already built), but remove the dead-end-era logic. The LP link in the expanded panel now always resolves. *(Confirm keep-vs-simplify at the open gate.)*

## Phases

### Phase 1 — Creation flow + presets

The Simple/Standard toggle on the create form, free-gating, and the page-seeding for each preset. Wire `landing_page_id`. Update event-creating seeders (`DemoBaselineSeeder`) to give their events LPs.

### Phase 2 — Deletion protection

Extend `PageObserver` so an event's LP cannot be deleted out from under it (existing block-with-counts pattern). Surface the block the same way other guarded deletions surface today.

### Phase 3 — Relink + simplify

Repoint the mini-calendar + `EventsListing` rows to the LP URL; strip the projector dead-end fallback; keep the inline-expand as a peek; `build:public`. Confirm the public render of both LP presets.
