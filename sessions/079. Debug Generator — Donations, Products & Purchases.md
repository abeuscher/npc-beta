# 079. Debug Generator — Donations, Products & Purchases

Extend the existing `DashboardDebugGeneratorWidget` to support generating products (with prices), purchases, and realistic donations with tax year control and fund/contact linkage, so the Donors page, tax receipt flow, and purchase history can be tested with minimal manual setup.

---

## Design decisions (resolved before starting)

**Environment assumption:**
The widget is only shown when `APP_DEBUG_TOOLS=true`. All generated data is understood to be scrub data. The generator may freely read existing contacts, products, and funds from the database to create realistic relationships — this is expected and safe in a dev environment.

**Donations — tax year control:**
Add a "Tax Year" input to the widget (numeric, default current year). Generated donations spread `started_at` randomly across that year. If active funds exist in the database, pick one at random per donation; otherwise `fund_id` is null.

**Donations — contact linkage:**
Pick randomly from existing contacts if any exist; otherwise create new contacts via the factory (current behaviour).

**Purchases — contact source:**
Use existing contacts if any exist; otherwise create fresh contacts.

**Purchases — product source:**
Use existing active products that have at least one price if any exist; otherwise create one new product (with 1–3 prices) and use that for all generated purchases.

**Products generation:**
Generates standalone products with 1–3 prices each. Purchases are not created alongside them — the admin generates purchases separately.

**Wipe behaviour:**
- Wipe "products" deletes all products and their prices.
- Wipe "purchases" deletes all purchases only.
- Wipe "donations" unchanged from current behaviour.

---

## Phase 1 — New factories

1. Write `ProductFactory` — realistic name, slug derived from name, status `active`, random capacity (10–200), sequential `sort_order`.
2. Write `ProductPriceFactory` — label (e.g. "General Admission", "Early Bird"), amount ($10–$500), `sort_order`.
3. Write `PurchaseFactory` — `status` randomised between `active` and `completed`, `occurred_at` within the last year, `amount_paid` matching the linked price. Requires `product_id`, `product_price_id`, and `contact_id` to be provided by the caller — the factory does not create these automatically.
4. Improve `DonationFactory` — add `started_at` defaulting to a random date in the current year.

No state methods beyond defaults are needed — the widget passes overrides directly.

---

## Phase 2 — Widget expansion

1. Add `taxYear` public property (int, default `now()->year`) and a "Tax Year" number input in the Blade view (shown for all types, meaningful only for donations).
2. Add `products` and `purchases` to the type dropdown in the Blade view.
3. Add `generateProducts(int $count)` — creates N products, each with 1–3 prices.
4. Add `generatePurchases(int $count)` — resolves contacts and product/price per the design decisions, creates N purchase records.
5. Replace the existing `'donations'` plain factory call with `generateDonations(int $count)` — applies year-spread, fund linkage, and contact linkage.
6. Extend `wipe()` to handle `products` (cascade-delete prices) and `purchases`.
7. Update the `match` in `generate()` to dispatch all five types. Update the feedback message.

---

## Out of scope

- Populating the tax year dropdown from existing donation data (plain number input is sufficient).
- Generating event registrations or memberships alongside purchases.
- Generating `Transaction` records alongside donations or purchases.
- Cascading purchase deletion when wiping contacts.
