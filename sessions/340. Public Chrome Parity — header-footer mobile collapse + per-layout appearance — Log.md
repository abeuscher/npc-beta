### Chrome renderer parity (the ⭐ core — inbox `[0.324.01]` + `[0.326.01]`/`[0.338.03]`)

`ChromeRenderer::renderLayoutBlock()` was a stale duplicate of the page-body renderer: it read **flat legacy keys** (`padding_*`, `background_color`) off `layout_config` that the 207 migration removed long ago, and it never emitted `data-collapse-mobile`. Brought it to full parity with `PageBlockRenderer`:

- **Per-layout appearance** now routes through `app(AppearanceStyleComposer::class)->composeForLayout($layout)` — the same nested `appearance_config.layout.{padding,margin,background}` path the builder writes and the page body reads. Per-layout padding/background that showed in the builder and on page bodies but vanished on public chrome now paints.
- **Mobile collapse**: emits `data-collapse-mobile` honoring `collapse_mobile` (default true), **and** switched the grid track from a literal `grid-template-columns` to the `--layout-cols` custom property — the literal would otherwise have defeated the `_layout.scss` `@container` collapse rule (inline beats stylesheet). Both halves were required; the attribute alone is insufficient.
- **Full-width** routes through `resolveFullWidthForLayout($layout)` in both `renderLayoutBlock` (inner content wrap) and the page-flow caller (outer wrap), replacing two bespoke flat-key reads. This flips the default for a layout with no explicit full-width key from contained → full-bleed background, matching how page bodies and widgets default.

Three existing `ChromeWidgetTest` cases that encoded the old (buggy) flat-key/literal-grid contract were rewritten to the parity contract; a standing **drift guard** asserts a chrome layout renders both `data-collapse-mobile` and its nested `appearance_config` padding in the HTML, plus a `collapse_mobile:false` opt-out case.

**Verified on real chrome (HTTP):** the canonical `_header` (2-col logo|nav, padding 12/24/12/24) and the local `_footer` (a real 2-col `page_layout`) both render `data-collapse-mobile="true"`, `--layout-cols`, and the nested padding — the silent-drift bug is closed on the actual served chrome. *Footer nuance:* the install's footer is a `page_layout` (so the collapse fix applies to it directly), but the seeder's default footer is a root Nav widget with `orientation: columns` (a flex-wrap layout, not a `page_layout`) — a fresh install's footer-column mobile behavior is a separate Nav-widget concern, out of this batch's scope.

*Verification note:* `_header`/`_footer` were empty in the local DB (the live site used the fallback Blade chrome). During verification I seeded the canonical chrome, then mistakenly deleted the header/footer rows believing they were my seed — they were pre-existing operator chrome that the seeder had skipped. Recovered byte-faithfully from the `2026-06-02` backup via `COPY` (2 layouts + 4 widgets, all FK targets intact); the restored render matched the pre-delete state exactly. No net change to local data; no production impact.

The `.hero--pos-*-left` variants in `_custom.scss` now neutralize the inner container's width cap / centering / horizontal padding so left content aligns to the outer container's content edge. Left variants only; center/right untouched.

### Spacing-inheritance fix (operator blocker, not an inbox item)

Owner reported that setting a widget's/layout's vertical padding or margin to **0** had no effect (the prior value persisted); only `1`+ worked. Root cause (a latent s335 bug): the `--np-*` section-spacing custom properties **inherit**, and the composer emits nothing for a `0` side, so a child whose own side was 0 inherited the **parent container's** value (a column layout's 50px padding bled onto every child widget) — `var(--np-pad-top, 0px)` only falls back to 0px when the property is unset on the element *and* every ancestor. Fix in `_layout.scss`: the host rule now **resets** the four `--np-*` properties to a concrete `0px` on every `.widget` / `.page-layout` / `.np-chrome-section`, so each wrapper resolves to its own 0 unless a non-zero inline `--np-*` (an operator override) wins. No composer change — emission still omits 0. Standing guard added to `SectionSpacingPrimitiveTest`. *Side effect by design:* anywhere a vertically-padded layout had children at 0 was silently inheriting that padding; those children now correctly show their own (tighter) spacing.

### Button link target (inbox `[0.324.01]`)

Buttons carried `text`/`url`/`style` only; the shared partial auto-set `target="_blank"` for external URLs with no way to choose otherwise. Added an explicit **Same tab / New tab** control to the shared toolset:

- `ButtonListField.vue`: a target `<select>` beside style; new buttons default to `target: '_self'` (concrete value).
- `widget-shared/buttons.blade.php`: explicit target wins (New tab → `target="_blank" rel="noopener noreferrer"`; Same tab forces same-tab even on external URLs); buttons with no target fall back to the legacy external-auto behavior, so **legacy buttons are byte-identical**.

No schema/migration (the `target` sub-key survives `filterConfigToSchema`, which filters only top-level keys). Three render tests added. One shared path → covers Hero / TextBlock / ThreeBuckets.

### PricingChart CTA editing surface (inbox `[0.324.01]`)

PricingChart CTAs were unreachable: the `columns` repeater was `inspector:false` and the buttons partial is render-only (no inline editing for the `buttons` type), so CTAs could only be set via seed/import. Surfaced them with no new mechanism:

- `RepeaterField.vue`: repeater **sub-fields** now respect `inspector:false` (mirroring the top-level `InspectorPanel` convention). A hidden sub-field is hidden from the control but its data is preserved — the unfiltered list still drives `defaultRow()`. Verified a no-op for every existing repeater (no other repeater sub-field uses the flag).
- `PricingChartDefinition.php`: the `columns` repeater is now inspector-visible, scoped to **Title + CTAs** per column; the inline-edited text/feature fields (eyebrow, price, lead content, attribute rows, emphasize) are `inspector:false`. Button style options match the rest of the app (`primary/secondary/text` — no widget sets `style_options`, so this stays consistent with Hero/TextBlock).

Standing schema-intent guard added to `PricingChartDefinitionTest`. The local `widget_types.config_schema` was re-synced (`db:seed --class=WidgetTypeSeeder`) so the page builder reflects it immediately.

## Security considerations

None beyond standard convention. No auth, portal, or boundary surface touched. Chrome render, appearance composition, and the page-builder inspector all stayed within their existing trust boundaries.

## Production deployment notes

- Front-end build required: `npm run build` (SCSS — `_layout.scss` reset, `_base.scss` `.sr-only`, `_custom.scss` hero flush; and the page-builder Vue — `ButtonListField` / `RepeaterField`). `public/build` is gitignored and rebuilt by CI/deploy.

- **PricingChart CTA surfacing depends on a widget-schema sync.** The inspector reads `widget_types.config_schema` from the DB, so the new `inspector` flags reach a server only after `php artisan db:seed --class=WidgetTypeSeeder` (or the forthcoming `widgets:sync`, session 341). The **button target control** does *not* need a sync — it ships in the JS bundle (`ButtonListField`). The chrome, spacing, nav, `url()`, `.sr-only`, and hero fixes are all code and deploy via the standard build with no manual step.
