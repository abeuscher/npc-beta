### Phase 2 — Cross-node restore portability fix

The one live defect: a CRM backup's database dump carried permission and ownership statements naming the **source node's** per-node read-only database role (a role Fleet Manager mints per node at provision time), so restoring that backup on any *other* node aborted on the missing role. This blocked FM's demo-baseline restore verification since 2026-06-05 (FM session 042) and would have failed any real cross-node restore.

- **Fix:** the pgsql connection in `config/database.php` now passes `--no-privileges --no-owner` to `pg_dump` via the backup library's per-connection dump options. One config block; the restore path needed no change (verified `demo:restore` feeds the dump to psql verbatim — removing the aborting statements is the whole fix).
- **Objective verification:** a real local `backup:run` produced a dump with **zero** grant/ownership statements; a control dump of the same database *without* the flags contained **90** of them. The flags were also confirmed to flow through the backup library onto the actual `pg_dump` command line.
- **Tests:** `tests/Feature/Backup/BackupDumpOptionsTest.php` — three cases pinning the config value, its resolution onto the built dump command, and that the fixed connection is the one the backup config actually dumps.
- **Contract doc revision (no version bump, per the 253/291/335 precedent):** new "Blob content — role-privilege-free dumps" sub-section under the backup-download endpoint, a cross-node-portability consequence bullet under the demo-node reset coordination section, and a dated CHANGELOG revision entry. Records the two operational consequences: old blobs stay non-portable (re-record baselines), and a restored node's own read-only role may need re-granting (an FM-runbook check, not CRM code).

## Security considerations

The dump flags *remove* privilege/ownership statements from backups — nothing gains access anywhere; database privileges on live nodes are untouched (they're granted node-side at provision, not carried in dumps). One consequence deliberately surfaced rather than hidden: because restores no longer re-grant anything, whether a restored node's read-only role keeps SELECT is now an explicit post-restore check (recorded in the handoff and the contract doc) instead of an implicit cross-node failure. The handoff doc references where credentials live, never values. No auth, contract-surface, or vault code touched in either repo.
