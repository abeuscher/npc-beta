# 116. CMS Content Export/Import

Build a simple artisan-based export/import tool for CMS content so the marketing site built by hand in the CMS can survive database resets. This is not the final production import/export feature — it is a developer tool to avoid re-entering data. Keep it simple and pragmatic.

---

## Design decisions (resolved before starting)

- Two artisan commands: `cms:export` and `cms:import`.
- Export format is a single JSON file written to a known path in the repo (e.g. `database/demo-content/cms-export.json`).
- Exported entities: `pages` (type `default` and `system` only — not `post` or `event`), `page_widgets`, `navigation_menus`, `navigation_items`. Blog posts and events are handled by the demo data seeder (session 117).
- Collections are seeded separately — they are not exported. Widget configs that reference collection IDs will resolve against seeded collections by handle at import time, or be left as-is if the collection exists.
- Images: media files referenced by widgets are not embedded in the JSON. Instead, the actual image files are committed to the repo in a known directory (e.g. `database/demo-content/media/`). The import command copies them into the media library storage path. The JSON stores relative file paths that the import command resolves.
- UUIDs: exported records keep their original UUIDs. On import, if a record with that UUID already exists, it is skipped (not overwritten). This makes the import idempotent and safe to re-run.
- Templates: the export includes `template_id` references. Templates are expected to exist (created by the `TemplateSeeder`). If a referenced template doesn't exist at import time, the field is set to null (use default template).
- Widget types: `widget_type_id` references are resolved by handle at import time, not by UUID. This makes the export portable across instances where UUIDs may differ.
- Author IDs: `author_id` on pages is set to the first admin user at import time, since the original author may not exist on the target instance.

---

## Phase 1 — Export command

1. **Create `app/Console/Commands/CmsExport.php`** — an artisan command `cms:export`.
2. **Query and serialize:**
   - All pages where `type` is `default` or `system` (exclude `post`, `event`, `member`).
   - All `page_widgets` belonging to those pages, with `widget_type` handle included (resolved from the FK).
   - All `navigation_menus` and their `navigation_items`.
3. **Write to** `database/demo-content/cms-export.json` (or a path argument).
4. **Output** a summary: count of pages, widgets, menus, items exported.

---

## Phase 2 — Import command

1. **Create `app/Console/Commands/CmsImport.php`** — an artisan command `cms:import`.
2. **Read the JSON file** from the default path (or a path argument).
3. **Import pages:**
   - Skip if a page with the same UUID already exists.
   - Resolve `template_id` — look up by UUID, null if not found.
   - Set `author_id` to the first admin user.
   - Insert the page record.
4. **Import page_widgets:**
   - Skip if the widget UUID already exists.
   - Resolve `widget_type_id` by looking up the handle in `widget_types`.
   - Resolve `page_id` — must exist (either just imported or pre-existing).
   - Insert the widget record.
5. **Import navigation_menus and navigation_items:**
   - Same UUID-skip logic.
   - Navigation items reference `page_id` — resolve against existing pages.
6. **Output** a summary: counts of records imported vs skipped.

---

## Phase 3 — Image handling

1. **Define the convention:** images are stored in `database/demo-content/media/` in the repo, organized by a flat or simple directory structure.
2. **On export:** identify media library records associated with exported pages/widgets. Copy the physical files to `database/demo-content/media/` and record the mapping in the JSON (original media library path → repo path).
3. **On import:** copy files from `database/demo-content/media/` into the media library storage location and create the corresponding `media` table records. Skip if the media record already exists.
4. **If media handling proves too complex**, defer it and document a manual step ("copy `database/demo-content/media/` to `storage/app/public/media/`"). The simple approach is acceptable for this use case.

## Testing

- **Slow test groups to run this session:** none
- **New tests expected:** yes —
  - Export command produces valid JSON with correct record counts
  - Import command creates pages, widgets, menus, items from JSON
  - Import is idempotent — running twice doesn't duplicate records
  - Widget type resolution by handle works
  - Missing template falls back to null

## Security checklist

Before closing:

- [ ] Export does not include any credentials, API keys, or user data
- [ ] Import does not overwrite existing records (skip-on-conflict)
- [ ] No path traversal risk in media file handling

## Out of scope

- Exporting blog posts or events (handled by demo seeder, session 117)
- A production-grade import/export UI or format versioning
- Conflict resolution or merge logic
- Exporting site settings, templates, widget types, or user accounts
