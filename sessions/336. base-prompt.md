## Process rules for every session

Base-prompt process rules apply in full (`sessions/template-base-prompt.md`). Session-specific notes:

- **`demo:restore` is destructive — gate it hard.** It wipes and replaces the database. It must refuse to run unless `isDemoMode()` (`APP_ENV=demo`), checked at the top of the command, mirroring `DemoRoleLockdownTest`'s posture. A restore-from-arbitrary-blob primitive must never be reachable or runnable on a real customer node — this is the session's load-bearing safety property (security-over-usability).
- **Reproduce the round-trip before wiring anything.** Back up and restore the demo locally once and enumerate what does and does not carry (background images, `SiteSetting`s, the logo file). The round-trip *is* the test; observe, don't assert.
- **Blob is pushed, never pulled.** `demo:restore` reads a **local** blob file (FM pushes it via its provisioning channel). Do not add an outbound fetch — the demo node's egress firewall (deny outbound except DNS/NTP/GHCR) is the one security control that matters, and the push model keeps it tight.
- **Env-specific values come from the node's `.env`, not the blob.** After restoring, set `base_url` (and any other env-specific value) from the node's own environment so the authoring environment's value never leaks onto the demo node.
- **Stay scoped to the restore primitive + fidelity.** The thin portal (F) and the driver.js tours are later sessions. FM-side provisioning / cron / blob-push is the FM repo's half — do not build it here.
