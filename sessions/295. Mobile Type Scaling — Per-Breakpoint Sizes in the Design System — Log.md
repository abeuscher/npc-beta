## What was built

### Phase 1 — Per-breakpoint type scaling (the planned E5; iteration /1, commit `472e040`)

- `TypographyResolver`: `font.size` migrated from flat `{value,unit}` to per-breakpoint `{xl,lg,md,sm}`, idempotent and **non-destructive on read** (stored `SiteSetting` row not rewritten until an explicit save). Existing hand-tuned values preserved **byte-exact at `xl`**; `lg/md/sm` derived from a calibration-driven per-class ramp (display / section / body). Restrained-vs-mean call settled: display phone = **0.60** (Attio-leaning), not the 0.54 raw 3-site mean. Body = 1.0 (no scaling). Required byte-exact back-compat regression written first (G1/285-shape guard).
- `TypographyCompiler`: emits the `xl` declaration + three `@media (max-width: 992/768/576px)` blocks per element; line-height stays a unitless multiplier (rides the size); heading bottom-margin as an em multiple.
- `DesignSystemPage` / theme-editor Vue: per-breakpoint size fields surfaced; shared `FontInput` primitive left untouched (adapter at the panel boundary so the page-builder is unaffected).
- Objective oracle: `scripts/type-oracle.js` (legibility floor, no text/page overflow, heading≥body, monotonic hierarchy) — **0 violations** across 5 public pages × 5 widths, before and after. `VERSION` 0.295.1.

### Phase 2 — Post-iteration-/1 corrections (commit `b890cb9`)

Manual testing exposed three compounding problems, fixed together:

- **Misdiagnosed em-rhythm scheme, reverted.** An iteration-/2 attempt added an `ELEMENT_MARGIN_EM` scheme + a "stale ≤1px stored margin" sweep, on the belief the stored `1px` heading margins were junk. They were not — they were a deliberate **rem** spacing scheme (h1 1.5rem … p 1rem). The scheme was reverted in full; it would have destroyed real configuration.
- **The actual bug.** `TypographyCompiler` emitted margins as `(int) $val . 'px'`, ignoring the stored `unit` — int-truncating `1.5rem` → `1px`. This predated the session and was why headings rendered with no space. Fixed to honour the stored box `unit` (no truncation).
- **`.np-site` scoping.** `TypographyCompiler::compile()` emitted bare element selectors (`h1:not(nav h1)` …) that leaked into the Filament admin once delivered globally. Typography delivery switched to `compileScoped(['.np-site'])`, and `_buttons.scss` + `generateButtonOverrideCss()` were wrapped/scoped to `.np-site` to match the documented `_base.scss` convention. Verified real-browser, computed values, three surfaces (public styled, admin clean, preview faithful).
- **Build-pipeline delivery correction.** The runtime inline `<style>` injection in `public.blade.php` (live since s182) was retired; typography now compiles into the public bundle via `AssetBuildService::collectSources()` like the button overrides, with an explicit rebuild action. The four session canon docs (session-outlines stub, release-plan E5, the 295 prompt + base-prompt) were corrected — their "runtime inline `<style>` is the only place the fix can live" premise was wrong.

### Phase 3 — Stale-bundle-artifact diagnosis

The public primary button rendered orange/pill though no saved setting configured it. Traced (verified, not assumed): not a Playwright test, not a seeder, not the DB, not the cache — a **stale compiled CSS file** under the gitignored, untracked `public/build/widgets/`, served long after the config that produced it was removed, with nothing reconciling bundle vs. settings. Rebuilding restored the correct default (blue, slightly-rounded), confirmed cache-busted. This is the forcing function for session 296.
