# Session 209 — Widget Primitive — Data Contract Prototype — Findings

**Date:** 2026-04-22
**Branch:** session-209/1

---

## Verdict

**Proceed with refinements (listed below).**

The premise survived contact with reality. Three structurally-distinct widgets (Blog Listing, Text Block, Carousel) migrated to typed data contracts without the contract language turning into a second ORM. Byte-equivalent output held on all three. Fail-closed field projection is demonstrated by the unit suite. The contract + resolver + SlotContext surface is small (five PHP files, ~220 LOC total).

The refinements are specific and bounded — none of them invalidate the architecture, but each should be resolved before Phase 2 (slot taxonomy) locks the contract shape for the remaining ~17 widgets.

---

## What worked

- **Three source types covered three structurally distinct shapes.** `SOURCE_PAGE_CONTEXT` (scalar map), `SOURCE_SYSTEM_MODEL` (row-set with compile-time-known fields), `SOURCE_WIDGET_CONTENT_TYPE` (row-set with widget-declared schema). No common super-type, no query-expression language.
- **Fail-closed field projection is real.** `ContractResolverTest::it resolves a widget_content_type contract into a row-set DTO with declared fields only` creates a collection item with `secret_internal: 'should not appear'`. The contract doesn't declare `secret_internal`. The DTO doesn't contain it. That's the entire security promise of the primitive, verified.
- **The `ContentType` value object lands cleanly inline.** Carousel's `dataContract()` method constructs a `ContentType` with three fields (title, description, image). ~15 LOC, reads naturally, doesn't require a separate file. Option A (inline) was the right call over Option B (companion file) or Option C (typed base method).
- **DTO shape convention held.** `['items' => [row, row, ...]]` for row-sets, flat scalar maps for single-record contexts. Every row is a flat array of declared fields plus an optional `_media` map for image fields. No walkable relationships in either direction.
- **Batching shape is present and minimally functional.** Per-`resolve()`-call cache keyed by source fingerprint. The `batches system_model resolution when multiple contracts share the same filter shape` test passes with `count(postQueries) < 3` for two contracts on posts.
- **Blog Listing template collapsed from ~30 lines of data-shaping to 1 line** (`$items = $widgetData['items'] ?? [];`). The difference is stark. The template is now a pure view.
- **`SlotContext` wrapping `PageContext` + optional `currentPageOverride` threaded the owner-page fallback** (admin preview, standalone render) through cleanly without widening ambient fields beyond the existing `PageContext` surface.
- **Contract path and legacy path coexist in `WidgetRenderer`** behind a single conditional (`$definition->dataContract($config)`). No globally-branching code. Unmigrated widgets are byte-identical to pre-session behaviour.

## What did not work

- **Fallback-data path bypasses contract field projection.** When the Carousel contract resolves to empty items (collection not found), `WidgetRenderer` pulls the fallback from `$collectionData[$collSlot]` (which `WidgetDataResolver` + `$fallbackCollectionData` populated). That fallback data is not run through the contract's field list — whatever shape the fallback carries flows into the DTO unchanged. This is an acceptable shim for the prototype but a clear Phase 4 cleanup: the resolver should accept fallback data per contract, or fallback semantics should move into the resolver entirely.
- **`SOURCE_PAGE_CONTEXT` contracts fire unconditionally on every widget render, regressing the buildTree query-count test.** `PageContextTokens::values()` lazily hits `$page->event` and `$page->author`. The legacy path short-circuited on `Str::contains($text, '{{')`; the contract path had no such check. Fix was a token-presence short-circuit in `WidgetRenderer` (inspect config values for `{{` before invoking the resolver). This matches legacy behaviour but is a stop-gap — Phase 2 batching should cover it structurally by resolving `SOURCE_PAGE_CONTEXT` once per page across all widgets.
- **Text Block declares the full `PageContextTokens::TOKENS` union** because the contract can't know which tokens the user will write into the richtext content. This weakens the "widgets declare what they need" property for the richtext-consumer case: Text Block effectively declares "anything page-context can expose." Honest, but the contract loses fail-closed bite here. Not a bug, a category observation.
- **Template-isolation tests needed signature updates.** Eight existing tests (seven in `ContentListingWidgetsTest`, one in `AppearanceConfigRenameTest`) render migrated templates directly via `Blade::render` with `pageContext` / `collectionData`, bypassing `WidgetRenderer`. Each needed a one-line change to inject `widgetData` instead. Per the session prompt, this counts as a behaviour delta. The shared helper `blogListingWidgetData()` at the top of `ContentListingWidgetsTest` is the natural shape for future widgets' test helpers; this pattern will repeat in Phase 4.
- **The `excerpt` page-context token is an alias over `meta_description`.** Pre-existing ambiguity, not introduced by this session — two consumers (SEO meta description output and template tokens) share one column. Blog Listing's contract exposes it as `excerpt` (truncated to 160 chars); Text Block's `PageContextTokens` path exposes it as `excerpt` (untruncated). Worth naming because Phase 2 will multiply this kind of alias.

## Proposed refinements before Phase 2

1. **Move fallback-data semantics into `ContractResolver`.** Either the `resolve()` signature grows a fallback map (`resolve(array $contracts, SlotContext $context, array $fallback = []): array`), or per-source resolvers accept a fallback hook. Goal: a widget template should never see a DTO that bypassed the field projection.
2. **Structural batching for `SOURCE_PAGE_CONTEXT`.** Bind `PageContextTokens` as a singleton with per-page-id memoization, OR make `ContractResolver` a singleton with a per-request resolution cache. Either eliminates the per-widget query regression without the `{{` short-circuit. The short-circuit can stay as belt-and-braces but shouldn't be load-bearing.
3. **Extract projection logic from `ContractResolver`.** `projectPost` is an inline method. Adding 17 more widgets means 17 more inline projections (for system models) or the resolver becomes a switchboard. Phase 2 should define a per-source projector interface — the resolver does source routing, projectors own field-level shape.
4. **Decide the richtext-consumer case explicitly.** Text Block's "declare the union" pattern will repeat for any widget that accepts arbitrary user-authored markup. Options: accept that richtext consumers declare the full page-context union; or carve a distinct source (`SOURCE_PAGE_CONTEXT_FREEFORM`) that makes this explicit.

## Precedents set for future phases

These are the patterns this session establishes. Phase 2 widgets should follow them until a concrete case forces a change.

- **Content-type declaration lives inline in the widget's `dataContract(array $config)` method**, as a `ContentType` value object constructed and attached to the returned `DataContract`. Widgets with no content type return a contract whose `contentType` is `null` (or a contract with a different source entirely). No companion file, no base-method "declare no content type" busywork.
- **`dataContract()` takes the resolved `$config`** as its sole argument. Structural shape (fields, source) is widget-level and constant; runtime shape (`resourceHandle`, `filters`) is config-driven.
- **DTO shape conventions:**
  - `SOURCE_PAGE_CONTEXT` → flat scalar map: `['title' => '...', 'date' => '...', ...]`, keyed by declared field names.
  - `SOURCE_SYSTEM_MODEL` and `SOURCE_WIDGET_CONTENT_TYPE` → `['items' => [row, row, ...]]`, where each row is a flat scalar map of declared fields, with optional `_media` when the contract's content type declares image fields.
  - No nested walkable objects. No Eloquent models leaked through.
- **`SlotContext` wraps the existing `PageContext`** and carries an optional `currentPageOverride` for callers (admin preview, standalone render) that need to bind the slot to a specific page without mutating the shared `PageContext`.
- **Contract version field ships from day one.** All three prototype contracts declare `version: '1.0.0'`. No consumer reads it yet. The field existing is the precedent.
- **Per-call fetch deduplication** via source fingerprint (`sha1(serialize($filters))` for system models, `collection:{handle}` for widget content types) is the minimum batching shape. Phase 2 decides whether this scales or needs a coordinator.
- **Contract source type `SOURCE_PAGE_CONTEXT` routes substitution through the DTO.** Widgets declaring this source have their richtext/text config fields substituted from the contract-resolved DTO, not via the direct `PageContextTokens` path.
- **Contract-driven widgets still receive `pageContext` and `collectionData` template vars** for the transition period. The legacy vars are ignored by migrated templates; they remain populated so unmigrated widgets on the same page continue to work unchanged.

## Open questions for future phases

Explicitly not resolved by this session, named here so Phase 2+ can pick them up.

- **IDOR in the resolver.** The prototype resolver doesn't enforce context ownership ("does this portal user actually own this contact?"). Fine for page/blog/carousel, load-bearing for the eventual record-detail and dashboard slots.
- **Mutation contracts.** Writes still flow through Filament. This prototype is read-only. A parallel write-contract story is a separate track.
- **Contract versioning in practice.** The field exists. No migration or deprecation mechanism yet. What happens when Carousel's `ContentType` adds a field in v1.1, and a deployed widget instance was configured against v1.0?
- **Nested DTO shapes.** Every case in this prototype fit the flat-with-row-sets model. Events with nested speakers, products with nested price rows, contacts with donation history — would these fit without subcontracts, or does Phase 2 need to introduce a recursive shape?
- **Configuration UI.** Admins configure widgets without writing code. How the configuration surface maps onto contract declarations is the second-hardest design problem after the contract itself (per the premise doc). Not touched here.
- **Asset scoping enforcement.** Widget SCSS is class-namespaced by convention, not by build-step enforcement. The contract-aware world suggests an eventual asset-bundling pass; when?
- **The "declares the union" pattern for richtext consumers.** See refinement #4. A naming/structural decision Phase 2 should make before the next batch of widgets migrates.
