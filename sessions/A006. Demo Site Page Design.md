# A006. Demo Site Page Design

Design the public website the demo install ships with — a small pretend-nonprofit site that showcases the product's widgets to a prospect. The deliverable is a design spec plus a drafted prompt for the local build session that will author the pages; no code, no page-building, in this cloud session.

## Session-specific deltas

- **The demo site is its own isolated instance.** Per the FM handoff note, the demo droplet's public site must never point at the real `nonprofitcrm.com` marketing site — demo visitors can edit pages. So these are *new* pages for the demo install, not a reuse of the live marketing pages (except the `/demo` page itself — see below).
- **Pages must align with `demo:reset` data.** The daily reset already runs `RandomDataGenerator` (≈40 contacts, 6 events, 30 donations, 15 memberships, 5 posts, 6 products — see the session 321 log). Pages that surface records (events listing, recent donations, blog index) must be designed so they render populated against that generated data.
- **The pages must survive the reset.** The build session (not this one) will wire the curated page JSON into the demo baseline (`DemoBaselineSeeder` / `demo:reset`) so a defaced or wiped demo restores the real site each cycle. This spec describes that wiring so the build prompt can carry it.
- **The `/demo` dark-side page is reused, not redesigned.** It lives in the *main* marketing site's nav (`sessions/public website/demo.json`). For the demo install it's clipped from the main-site export and dropped into the demo install. The user has said the exact execution doesn't matter — describe a simple mechanism, don't over-build it.
- **Everything composes existing widgets.** The widget inventory (`app/Widgets/`) already covers every page below — including `MapEmbed` for the contact page. No new widget is in scope; if a page seems to want one, flag a gap rather than spec it.

## Phases

### Phase 1 — Org identity + page list

- Choose the fictional nonprofit: name, one-line mission, voice/tone, a rough visual feel (which sample-image-library categories the pages will draw from). This anchors all copy.
- Finalize the **5–7 page list**. Proposed working set (refine, don't treat as fixed):
  1. **Home / landing** — `Hero` + a few showcase bands (mission, featured event, donation CTA, recent posts).
  2. **Event landing page** — `EventsListing` / `EventCalendar` against the generated events, plus `EventRegistration` / `EventDescription` on a detail.
  3. **Donation page** — `DonationForm` + `ThreeBuckets` (giving levels) + `RecentDonations` against generated donations.
  4. **About / board page** — `BoardMembers` + `TextBlock` + `Image`.
  5. **Contact page** — `WebForm` + `MapEmbed`.
  6. **Blog** — `BlogListing` index + a post detail with `BlogPager` (the prev/next pager flow the site has used).
  7. **The `/demo` dark-side page** — reused from the main site, in the demo install's nav.
- For each page, state the URL/slug, its purpose in the prospect demo, and the exact widget handles it uses.

### Phase 2 — Per-page composition + copy

- For each page, specify the widget arrangement (layout bands, columns, order) following the marketing-site's section-band / appearance-config conventions.
- Write realistic placeholder copy for the fictional org — headlines, body, CTAs, board bios, a couple of blog posts, event blurbs. "Lorem-ipsum-but-believable": a prospect should read it as a real small nonprofit, not filler.
- Note per-page data alignment: which widgets read from the `RandomDataGenerator` output, and any content the build session must seed beyond what the generator makes (e.g. specific named board members, since the generator makes contacts but not a curated board).

### Phase 3 — Nav, chrome, `/demo` integration, and reset wiring

- Specify the demo site's header/footer nav (links across the page set) and any chrome defaults the demo should use. *(Note for sequencing: E16 Header/footer overhaul may land before the build session and improve these defaults — the spec should be written to use whatever the production chrome defaults are at build time, not hard-code around the current ones.)*
- Describe how the `/demo` page is brought into the demo install (clip from main-site export → import into the demo install). Keep it simple.
- Design the **reset wiring**: how the curated page JSON gets re-established on every `demo:reset`. The likely shape — the build session exports each page to `sessions/public website/demo-*.json`, adds a step to `DemoBaselineSeeder` (or a sibling seeder it calls) that imports those bundles via the existing `ContentImporter` after the `migrate:fresh --seed`, so the reset path becomes: rebuild → seed → import curated demo pages → generate CRM baseline. Specify it precisely enough that the build session just implements it.

### Phase 4 — Produce the deliverables

- Write `sessions/public website/demo-site-pages-spec.md` capturing Phases 1–3 in build-ready detail.
- Draft the local build session's prompt as `sessions/demo-site-pages-build.PROMPT-DRAFT.md` (no session number — the user slots it). The build session: authors the pages in the CMS, exports their JSON, wires them into the demo baseline so the reset restores them, and verifies via the `APP_ENV=demo` local dress-rehearsal (session-321 isolated-stack recipe). It is a normal local session.
