### Phase 5 — PageContextTokens namespace bug fix (out-of-scope, user-approved mid-session)

The retrofit surfaced a pre-existing bug: BlogPager's `prev_template`/`next_template` fields used `{{title}}`, `{{author}}`, `{{date}}`, `{{image}}` tokens — which `WidgetRenderer`'s richtext preprocessing replaces with the host page's PageContextTokens values *before* the BlogPager template runs. The result: the prev/next links displayed the host post's metadata, not the neighbor's. BlogPager's `$resolveTokens` closure was effectively dead code — by the time it ran, the tokens had already been clobbered.

This is a pre-existing bug (BlogPager had used `{{title}}` etc. since before the retrofit). The user approved fixing it. The fix aligns BlogPager with the established `{{item.foo}}`-namespace convention used by BlogListing, EventsListing, and Carousel — `WidgetRenderer`'s `PageContextTokens::substitute` only matches unprefixed `{{title}}` etc., so `{{item.title}}` survives preprocessing.

Fix:
- `BlogPagerDefinition.php`: schema defaults and `defaults()` now use `{{item.title}}`, `{{item.author_name}}`, `{{item.published_at_label}}` (matching BlogListing/EventsListing convention).
- `template.blade.php`: `$resolveTokens` rewritten to iterate `$item` and substitute `{{item.<key>}}` per field, with `{{item.image}}` special-cased for `<img>` HTML composition.

This is a minor breaking change for any existing user customization of `prev_template`/`next_template` that used the old tokens — but those customizations were already producing wrong output (host page metadata, not neighbor metadata), so the change just acknowledges the correct token names.
