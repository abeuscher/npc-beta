## Process rules

Base-prompt process rules apply, with **cloud-design tightening:**

- **Docs only. Hard-no on application code.** Do not create, modify, or delete any file under `app/`, `database/`, `resources/`, `tests/`, or any migration / compose / CI file. The only files this session writes are the spec doc, the drafted build-session prompt, and the planning-doc bookkeeping named in the close gate.
- **Make the call, document the rationale.** This is an autonomous cloud session — where the prompt leaves a design choice open (org identity, exact page list within 5–7, per-page widget arrangement, copy voice), pick the strongest option and write it down with a one-line why. Reserve Open Questions for choices that genuinely change the build session's scope (e.g. does the supplanting touch fresh-install defaults; does the contact-page map need a real address or a placeholder).
- **Compose existing widgets; never design a new one.** If a page seems to want a capability no existing widget provides, note it as a gap in the spec — do not spec a new widget. (Per the inventory read, none should be missing.)
- **Align pages to the generated data.** Pages that surface CRM records (events listing, recent donations, blog index, board members) must be designed against the data shape `DemoBaselineSeeder` produces, so the live demo shows populated pages, not empty states.
