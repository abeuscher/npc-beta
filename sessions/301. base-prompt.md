## Process rules for every session

Base-prompt process rules (`sessions/template-base-prompt.md`) apply in full. Session-specific notes:

- **The token list is fixed; schemes select, they do not extend.** `docs/theme-color-tokens.md` is canonical and closed (297). A scheme is a *named, contrast-vetted assignment* over the existing tokens for the content region — not a new token, not a per-token editor exposed to the page author. Power-user-defined named schemes are the extensible shape; the page/template author only ever **selects** one.
- **Schemes recolour the content region only — chrome composes, it does not bleed.** The standard header/footer always render in their own vetted colours; a dark content scheme does **not** restyle them. A dark landing page that wants a dark header *supplies its own header* (the slot model) — they compose, they never bleed. Confirm this rule at session start; it resolves every ambiguous case (Edge-#1: appearance is always a Theme token; the Template makes structural choices and selects vetted wholes).
- **One resolver, two call sites, byte-identical.** The single shared resolver (Template record → inline `--np-*` string) must be called by **both** the public layout and the page-builder preview renderer with no divergent branch. That one-resolver discipline is the entire reason `.np-site` exists and is the only thing that guarantees preview fidelity. Per-template overrides are **request-time inline, never bundled** — building any of it into `AssetBuildService`/the bundle is the 295 stale-bundle regression and is scope drift.
- **Verify in a real browser, computed values, three surfaces, scheme-switched.** Public (content region follows the selected scheme; chrome unchanged), Filament admin (no `.np-site` → no leak), page-builder preview (scheme-faithful, identical to public). Flip a template's scheme and confirm the content region follows while chrome holds. Never assert "it works" from emitted CSS or `curl` — the 295/297/300 rule. 296 drift guard green at start and close (and explicitly not extended to the request-time path).
- **Scope fences — do not cross them:**
  - **No token-list change, no palette redesign, no widget-SCSS rework.** 297's contract and 300's consumption are done and guarded.
  - **No new appearance primitive exposed to the page author.** Deviation is *select a vetted scheme* / *supply or suppress chrome* — never per-token editing (the relocated footgun).
  - **No Fleet / `/api/health` / response-schema change** — non-boundary, v2.3.0.
  - **Naming/IA polish tail** — the arc's optional ≤0.5-session tail is **not** this session unless explicitly folded; default defer.
- **Run the fast suite when implementation is complete:** `docker compose exec app php artisan test --exclude-group=slow` (or `./dev test`). Schema change → `docker compose exec app php artisan migrate` + update `docs/schema/`. Front-end build per the base-prompt rule: `resources/scss/**` or the Vue preview app → `npm run build`; widget assets → `build:public`. Then verify the three surfaces scheme-switched in a real browser and confirm the drift guard green.
- **Pause for the human-judgment surface:** the Inverse scheme's contrast vetting (which token values make a legible dark content region) and whether a given deviation is "select a scheme" vs "supply chrome" vs out-of-scope are judgment calls. Batch and surface them with before/after multi-surface captures; do not silently pick scheme values.
