## What was built

### Phase 1 — Prevention: the CI keystone scrub-residue test (component A)

- **Found and fixed a real `wipe()` media leak (in-scope, not papered over).** `RandomDataGenerator::wipe()` mass-deleted scrub **events** and **products** via the query builder (`Event::where(...)->delete()`), which bypasses Spatie's per-model media teardown — orphaning the `media` rows + content-addressed files behind them. Posts were already clean (`wipeScrubPages()` force-deletes each page individually). Fix: a `wipeEach(Builder)` helper that deletes events/products one model at a time so the teardown (and `EventObserver::deleted`'s landing-page cascade) fires.
- **The keystone invariant** (`RandomDataGeneratorServiceTest`, *"KEYSTONE — scrub generate-then-wipe leaves zero residue across every table + media"*): snapshot raw `DB::table` row counts for every scrub-touched table + the `media` count → `generate()` a full spread → attach owned media to the scrub events/posts/products → `wipe()` → assert every count returns to baseline. Raw counts include soft-deleted rows, so any residue fails it. Catches the whole "generates-but-doesn't-clean" class, not just today's bug.
- **Classified fast (~0.8s, `Queue::fake()` skips conversions), not `->group('slow')`** — at that cost it belongs in the primary fast suite as the launch gate. **Revert-checked:** with the `wipe()` fix reverted it fails (*"residue left in media — 6 ≠ 0"*), proving the assertion is not vacuous; restored and green.
- **Targeted cascade tests:** delete-scrub-event→owned-media and delete-scrub-product→owned-media teardown (the per-family guard the keystone generalizes).

### Phase 2 — Detection: the `app:data-hygiene` audit (component B)

- **`App\Services\DataHygieneAudit`** — the build-once shared core. `counts()` returns the four-category aggregate (`orphan_event_pages`, `scrub_records`, `orphan_media_dirs`, `dead_owner_media`) — all non-PII ints, **shaped as the seam the Phase-2 FM-visibility subcheck consumes**. Detection primitives (`orphanEventPages()`, `orphanMediaDirectories()`, `deadOwnerMedia()`) back the `--deep` mode; `scrubBreakdown()` reuses `RandomDataGenerator::scrubCounts()`.
- **`app:data-hygiene` command** — count-only summary table by default; `--deep` lists the actual records (node-local). Read-only; it reclaims nothing.
- **Reuse, not re-derivation:** `PruneOrphanEventPagesCommand` + `PruneOrphanMediaCommand` refactored to consume the audit's detection primitives (one source of truth). `MediaPruneOrphansTest` + `EventLandingPageLifecycleTest` stayed green through the refactor.
- **`media:prune-dead-owner` (new — the owner-requested "real fix").** Spatie's `media-library:clean --delete-orphaned` **crashes on this schema** (`operator does not exist: character varying = uuid` — its orphan check joins varchar `media.model_id` against uuid owner PKs). The new command reuses `DataHygieneAudit::deadOwnerMedia()` and deletes each `Media` model so the refcounted file remover fires. Dry-run default; `--force` to act; a soft-deleted (recoverable) owner is treated as alive.
- **`deadOwnerMedia()` hardened** to pull owner-table ids and compare stringified in PHP (no SQL `model_id = id` comparison at all), so neither the audit nor the prune command can hit the varchar/uuid wall on any node. Verified locally (17 dead-owner rows, matching the `app:data-hygiene` count).

## Security considerations

The audit is read-only and node-local. Its count-only `counts()` method is deliberately shaped as the **non-PII** signal Phase 2 will put on the FM wire (counts only, never raw rows — the track's privacy boundary), while the `--deep` records mode stays node-local. No new external-input surface; no auth or contract change. The `wipe()` change only alters *how* scrub rows are deleted (per-model vs mass), not *what* is deletable (still `source = scrub_data`, still super-admin gated). None beyond standard convention otherwise.

- **404 responsive-image incident (diagnosed + fixed live, separate from the hygiene work).** 351's responsive `sizes` made browsers request small `responsive-*.webp` variants that were missing on nphelper (**1350 of 1510 conversions claimed-but-absent**). Root cause: a DB load that brought rows (`generated_conversions = true`) + originals but **not** the conversion files (DB/media-tree out of sync) — **not** our cleanup (the refcounted remover is all-or-nothing per content_hash), **not** import/export (ships originals + regenerates per-media on import), **not** a code bug. Fixed on the box with `media-library:regenerate --only-missing --force` (the box has GD + WebP). Standing remedy documented.
