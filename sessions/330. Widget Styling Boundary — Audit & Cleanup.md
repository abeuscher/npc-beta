# 330. Widget Styling Boundary — Audit & Cleanup

Audit the widget styling system for where widgets cross the host boundary or compete with the host/each other on specificity, align on the fix list, then execute the agreed behavior-preserving cleanup — so the boundary is explicit, isolated, and documented ahead of a later mobile-appearance design pass.

## Why this, why now

The styles are largely built and working, but they grew **organically** — widgets reach into host Sass internals (`$gutter`, `$bp-*`), and widget-vs-host precedence holds only because we control concatenation order and every widget. That is fine today and untenable for **widget autonomy** (a widget can't be a portable plugin while it reaches across the host boundary and relies on implicit cascade order). This session makes the boundary **explicit, isolated, documented** — the groundwork. It is **not** the design work: the **mobile appearance layer needs real improvement and will get a dedicated design pass later**; that pass wants a clean framework to land on, which is what this builds. Nothing here changes how anything looks.

## Two phases, one pause

### Phase 1 — Audit (no application code)

Survey **all** of `app/Widgets/*` (the spec's flagged list is a starting point, not a guarantee) plus the host seam, and produce a findings doc organized by matrix. For each finding: what exists, what's wrong, the **lightest** recommended fix, and whether that fix is behavior-preserving.

The matrices (the set the owner wants examined):

- **Spacing** — widgets reaching host spacing vars (`$gutter`). Note: only `$gutter: 1.5rem` exists today; the recommended fix is the lightest (widget owns the value, or reads one published token equal to 1.5rem) — **not** a new spacing scale unless a widget genuinely needs more.
- **Type** — widgets reaching host type internals instead of a readable vocabulary. Report the coupling; recommend the lightest fix. Do not propose a full type-token system unless a real widget needs it.
- **Breakpoints** — widget SCSS using viewport `@media` (against `$bp-*`) instead of `@container` against the widget's own container. The spec flags `ThreeBuckets`, `ProductCarousel`, `PricingChart`, `MapEmbed`, `LogoGarden`, `BoardMembers` — confirm against live code and find any others.
- **Specificity / cascade** — where a widget's styles can win or lose against host base styles (or another widget) by selector depth or concatenation order rather than an explicit boundary. Assess `@layer` as the isolation mechanism, including its blast radius on host base ordering (`_base.scss` / `_layout.scss`).
- **General host-var leaks** — any other place widget SCSS reaches across the host boundary (the `PricingChart` `--pc-*`-over-`--np-color-*` shape is the reference-clean target).

**Phase 1 ends at a pause.** Present findings, walk the fix list with the owner, agree what's in / deferred. Won't-fix is decided here, not by filtering during the audit.

### Phase 2 — Action (only the agreed fixes)

Execute the aligned fix list. All behavior-preserving. Likely shape (subject to the pause):

- Decouple flagged widgets from host vars (`$gutter` → owned/published value; viewport `@media` → `@container` against a named widget container, once the host wrapper carries `container-type: inline-size; container-name`).
- Introduce `@layer` cascade isolation **if** the alignment said yes (host base in a lower layer, widget interiors in a `widgets` layer), preserving the deliberate inherited-text-colour reach-in.
- Extend the enforcement gate — mirror `WidgetColorTokenConsumptionTest`: an explicit reviewed baseline in the `design` group that fails on a new leak and on a stale baseline entry; update `DesignGroupIntegrityTest` in the same pass.
- Refresh the stale SCSS section of `resources/docs/widget-development.md` (and the contract doc, shape per the pause) so the boundary is documented, not folklore.

## Out of scope

- **The mobile-appearance design pass** — the actual improvement of how widgets look/respond on mobile. This session builds the framework for it; it does not do it. Any fix that changes rendered appearance is deferred to that pass.
- **A spacing scale / type-token vocabulary** — de-scoped unless the audit finds a real widget that needs more than the existing values. Not built for symmetry.
- **Editable layout breakpoints** — killed per the spec's settled decisions; `$bp-*` stay fixed build-time constants.
- **`image_breakpoints`** (CMS Settings) — unrelated feature, untouched.
- **Package extraction** to `abeuscher/npc-widgets` and the public authoring guide — release-phase, downstream of this.
- **Shadow-DOM isolation** — `@layer` is the chosen mechanism if any.

## Testing

- **Slow test groups:** none expected (token/SCSS work is fast suite + `design` group).
- **New tests expected:** yes, in Phase 2 — extend the consumption-gate pattern for the agreed families (no host-var leak / no viewport `@media` in widget SCSS), in the `design` group; update `DesignGroupIntegrityTest` in the same reviewed pass. Phase 1 adds none.
- **Build:** `build:public` for widget asset changes (regenerates the widget manifest); `npm run build` for `resources/scss/**` changes.
- **Visual verification:** behavior-preservation is the bar — confirm nothing moved on screen. Responsive collapse under container queries is a human-judgment surface: Playwright can confirm a collapse *fires*, but "does it look right at each width" is the owner's call. Pause for manual testing on any responsive change.
