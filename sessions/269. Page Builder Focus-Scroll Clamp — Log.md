### Verify-at-start findings

Two architectural assumptions in the session prompt's design-decisions block were factually incorrect against the current code:

- **Listener target / scroll container.** The prompt claimed `paneEl` (the `.preview-canvas` element) is the scroll container with `overflow-y: auto`. Inspection of [PreviewCanvas.vue](resources/js/page-builder-vue/components/PreviewCanvas.vue) and [App.vue](resources/js/page-builder-vue/App.vue) showed `.preview-canvas` carries only `style="min-width: 0"`; its parent `.vue-editor__main` declares no overflow either. The actual scroll happens at **document/window level** — `paneEl.scrollTop` is always 0 — and the inspector panel stays visible via `position: sticky; height: calc(100vh - 2rem)`. Any clamp would need to listen on `window` rather than `paneEl`, with math reading `window.scrollY` + `window.innerHeight` instead of pane-relative scroll state.
- **Release signal.** The prompt claimed clicking the canvas background de-selects "via the existing `clearSelection` path." No such path exists. The store has no `clearSelection` action; selection only flips via overlay clicks on a different widget/layout (`PreviewRegion.vue` uses `@click.stop`, so clicks landing on `.preview-canvas` background do nothing). "De-selection un-clamps" would require adding a new background-click affordance.

The internal-scroll-widget audit and drag-drop interaction notes came back clean:

- **Internal-scroll widget audit.** All `app/Widgets/*/styles.scss` `overflow` declarations are `overflow: hidden` (clipping, not scrolling). Carousel / ProductCarousel use Swiper.js (swipe gesture, not native scroll). No widget today has a native scroll container, so the prompt's "defensive nested-scroll defer" walk would be forward-looking only.
- **Drag-drop interaction.** `selectedItemId` only changes via [App.vue:23-31](resources/js/page-builder-vue/App.vue#L23-L31)'s `widget-created` handler, which fires after drop completion. No mid-drag selection — no drag-state guard would have been needed.
- **Keyboard scope.** Recommendation in the prompt (include arrow keys / PageUp/Down / Space / Home / End via `keydown` preventDefault) was sound; no counter-evidence in code.
- **Vitest infrastructure.** Not present. `package.json` only has Playwright; `pest` covers PHP. Math-extraction-as-pure-function would have been covered via Playwright only (or a thin Vitest setup added).

### Closing-decision evaluation

Surfaced four paths to the user with honest cost/benefit per path:

- **A. Ship the prompt as written, retargeted to `window`.** preventDefault on each wheel/touch event likely produces visible jitter on Mac trackpad inertia (the 204 prediction). Same fight, no resolution.
- **B. Ship only the tall-widget half.** The tall-widget-as-scroll-container behaviour aligns with browser scroll semantics; the short-widget lock is the half that fights inertia. Drop the latter, keep the former.
- **C. Restructure first.** Turn `.preview-canvas` (or `.vue-editor__main`) into a real fixed-height `overflow-y: auto` container with `overscroll-behavior: contain`; clamp becomes a non-fight `scroll`-event handler. This is how Notion / Figma / Webflow / Framer do it. Significantly bigger than the prompt's scope — touches the Filament admin shell, the sticky inspector, the mobile breakpoint collapse, the public-style preview viewport math.
- **D. Don't ship.** Reaffirm the 204 escape hatch. The must-have (scroll-to-center on selection) already shipped at 204; the manageable UI sufficed; and the user's general dislike of scroll-jacking sealed it.

User chose **D**. Quote: *"This doesn't feel right and every tangential problem to the one we are solving has been resolved and frankly the UI is quite manageable now. I don't like the kind of problems we are running into and I don't like scroll jacking very much anyways."*

The 204 framing is reaffirmed: reopen only if user testing confirms the must-have scroll-to-centre isn't enough.
