## What was built

### Iteration /1 — Widget catalog + 5 widget help docs + cms-pages callout

- Five widget help docs at `resources/docs/widget-{bar-chart,donation-form,event-calendar,event-registration,web-form}.md`. Each frontmatter carries `title`, `description`, `tags`, `category: cms`, `standalone: true`. Body covers: what the widget does, the inspector Content + Appearance tabs with each significant config field, common configuration patterns, and codebase-grounded gotchas (Y-field must be numeric for bar_chart, capacity is registration-time-checked for event_registration, preset-amounts are dollars not cents for donation_form, etc.). Written directly from each widget's `WidgetDefinition::schema()`, not invented.
- `resources/docs/widget-catalog.md` — standalone help article rendering a sortable Alpine-driven HTML table over the 5 widgets. Columns: Label, Category, Description, Help Doc. Click a header to re-sort, click again to reverse. Default sort: Label A→Z.
- `resources/docs/cms-pages.md` callout — one-paragraph block near the top linking to `/admin/help/widget-catalog`.
- `tests/Feature/WidgetHelpIndexingTest.php` — three Pest tests asserting (a) post-`help:sync`, all 6 widget-related slugs exist as `HelpArticle` rows; (b) `HelpSearch::search('bar chart')` returns `widget-bar-chart`; (c) `HelpSearch::search('donation form')` returns `widget-donation-form`.

### CSS link-styling follow-up

- `public/css/admin.css` gained scoped anchor rules on `.help-page-content` and `.help-panel-body`: blue-600 / underline in light mode, blue-400 in dark, with hover deltas. Filament's admin theme strips default anchor styling, so internal links inside docs (e.g. the cms-pages callout) read as plain text without the rule. Lands on the standalone help-article page and on the `?` slide-over flyout. No build step — `public/css/admin.css` is hand-edited and loaded directly via the AdminPanelProvider link tag.

### Iteration /2 — Canonical Widgets page replaces widget-catalog

- `widget-catalog.md` renamed to `widgets.md`. Becomes the canonical introduction to page-builder widgets — opens with what widgets are and how the page builder uses them, then renders the same sortable table that was in the catalog. Title is "Widgets" (was "Widget Catalog"); slug is `widgets` (was `widget-catalog`); URL `/admin/help/widgets`. Lives under `category: cms` so it sits in the CMS help section, lateral to the rest of CMS help articles.
- `cms-pages.md` callout retargeted to `/admin/help/widgets` and reworded: "*Widgets. For an introduction to page-builder widgets and per-widget configuration help, see the Widgets article.*"
- `WidgetHelpIndexingTest.php`: expected slug `widget-catalog` swapped for `widgets`; the redundant `widget-%` LIKE-count assertion dropped since the explicit-slug list covers it.
- `help:sync` reported 58 synced / 1 removed (the old `widget-catalog` row) / 0 skipped.

### Iteration /3 — Help article search-weight tiebreaker

- Migration `2026_05_10_140000_add_search_weight_to_help_articles.php` — additive integer column `search_weight` on `help_articles`, default `0`. Non-destructive.
- `HelpSync` parses optional `search_weight:` frontmatter key, persists as integer (defaults to 0 when missing).
- `HelpSearch` ORDER BY chain becomes `search_rank ASC, search_weight DESC, title ASC` — rank still dominates, weight is the tiebreaker, title is the stable final order. Default-0 articles behave exactly as before.
- `HelpArticle::$fillable` extended.
- `widgets.md` gains `search_weight: 100` in frontmatter. Motivating case: searching "widget" used to return `widget-bar-chart` first (rank 1, alphabetical); now returns `widgets` first.
- `docs/schema/help_articles.md` updated.
- `WidgetHelpIndexingTest.php` gains two tests: "widget" returns `widgets` first; `search_weight` persists from frontmatter (100 on widgets.md, 0 on widget-bar-chart).

### Iteration /4 — Parent breadcrumb chain

- Migration `2026_05_10_150000_add_parent_slug_to_help_articles.php` — nullable string column `parent_slug` on `help_articles`.
- `HelpArticle::$fillable` extended with `parent_slug`; `HelpArticle::parent()` helper resolves the parent model or null.
- `HelpSync` parses optional `parent:` frontmatter key into `parent_slug`.
- `HelpArticlePage::getBreadcrumbs()` walks the parent chain depth-capped at 10, inserting each ancestor between the category crumb and the current article. Top-down display order.
- Each of the 5 widget detail docs gains `parent: widgets` in frontmatter. Resulting chain on `widget-bar-chart`: **Help > CMS > Widgets > Bar Chart Widget** instead of the previous flat **Help > CMS > Bar Chart Widget**.
- `docs/schema/help_articles.md` updated.
- `WidgetHelpIndexingTest.php` gains two tests: `parent_slug` persists and `parent()` resolves to widgets; the breadcrumb chain renders the exact 4-crumb sequence.

## Security considerations

None beyond standard convention. The help-doc pipeline is admin-only; doc body content is operator-authored markdown rendered with `Str::markdown()` which allows raw HTML by default — this is unchanged from the prior state and remained acceptable for the same reason (admin-authored content trusted at the same level as direct database writes). No new public surfaces; no new auth flows.

## Schema changes

Two additive, non-destructive migrations on `help_articles`:

- `search_weight` integer not null default 0 — tiebreaker for HelpSearch ordering.
- `parent_slug` nullable string — pointer to another `help_articles` row's slug; used by breadcrumb chain walk.

`docs/schema/help_articles.md` updated with both columns. No other schema touches.

The session 277 base prompt declared no migrations expected; the user expanded scope mid-session to add the search-weight feature (and then the breadcrumb feature as a follow-on UX ask). Both migrations are additive, both fold cleanly into the next squash. No data migration ran; both columns have safe defaults.

---

## Production deployment notes

None beyond standard `migrate` + `help:sync`. After `migrate` populates the two new columns with defaults, `php artisan help:sync` re-reads frontmatter and populates `search_weight` and `parent_slug` from the markdown files. Without `help:sync` the columns stay at defaults and the help system behaves as if the features were dormant (no broken state).

## Deferred decisions

- **Auto-generating the catalog from `widget-*.md` files at `help:sync` time** — explicitly out of scope per the session prompt. Hand-curated list is the design target. Revisit only if the curated list goes stale in practice.
- **Per-widget `?` affordances in the page-builder inspector** — out of scope; the catalog is one click from the page-builder via the `?` icon's existing cms-pages route.
- **Per-field tooltips inside the inspector** — out of scope; inspector field helpers already cover most cases.
- **Authoring help docs for the other 33 widgets** — the 5 documented this session validate the pattern; the rest land incrementally as their complexity warrants.
- **Search-ranking tweaks beyond the new `search_weight` tiebreaker** — `HelpSearch`'s rank-by-match-location logic is unchanged; only the tiebreaker is new. Subject to revisit if the simple weight integer proves insufficient.
