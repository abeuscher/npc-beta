# 212. Widget Primitive — Phase 3a (Dashboard Grid — Slot Plumbing & Hosted Render)

Replace `DashboardGridSlot::ambientContext()`'s `RuntimeException` body with a real implementation, mount a widget-primitive slot render inside the existing Filament admin dashboard, and prove the slot abstraction from session 211 survives contact with a non-page-builder consumer. First slice of Phase 3 (3–5 sessions total per `widget-primitive-migration.md`); this session is pure plumbing and an existing-widget render — **no new dashboard-native widgets, no source-policy work, no storage model for admin-arranged dashboards.**

---

## Design decisions (resolved before starting — do not relitigate)

Settled across the Phase 1–2 track (sessions 209–211) and the scoping for this session:

- **Mount point is a new Filament widget, not a new page.** `app/Filament/Widgets/DashboardSlotGridWidget.php` extends `Filament\Widgets\Widget` and renders via a blade view. It lives **alongside** the four existing Filament-native dashboard widgets (Welcome, QuickActions, IntegrationStatus, DebugGenerator); it does not replace them. The existing `app/Filament/Pages/Dashboard.php` is the host, unchanged.
- **Widgets rendered in this session are the two existing primitives whose contracts do not depend on a current `Page`:** `BlogListing` (`SOURCE_SYSTEM_MODEL`, `model=post`) and `Carousel` (`SOURCE_WIDGET_CONTENT_TYPE`). Arrangement is hardcoded in the blade view. Two widgets, not three, because the goal is minimum-viable render, not catalogue coverage.
- **`DashboardGridSlot::ambientContext()` returns a plain `SlotContext`** — the app's `PageContext` service plus no page override. This is the minimum shape that satisfies the `ContractResolver`'s single consumed surface (`$context->currentPage()` returns null on dashboard). **No new `AmbientContext` subtype, no `currentUser()` accessor, no generalisation.** Phase 5 does the typed-subtypes refactor when the record-detail slot surfaces its ownership-check need. This session intentionally under-builds to let shape problems surface.
- **Dashboard widgets are represented by unsaved, in-memory `PageWidget` instances** with `widget_type_id` and `config` set. No storage model, no `owner_type = Dashboard`, no `page_widgets` row committed in this session. The polymorphic-owner discipline from the migration doc applies when the storage model actually lands (later session in the Phase 3 batch).
- **`WidgetRenderer::render()` gains an optional `string $slotHandle = 'page_builder_canvas'` parameter.** When the caller passes `'dashboard_grid'`, the renderer looks up `DashboardGridSlot` via the `SlotRegistry` and constructs ambient context through it. The default is unchanged so every existing caller is byte-equivalent.
- **No `SOURCE_PAGE_CONTEXT` widgets on the dashboard.** TextBlock and friends depend on page metadata tokens; those tokens don't exist on the dashboard by design. The slot's `layoutConstraints()` implies this; no runtime enforcement in this session (layout enforcement is deferred per 211).
- **No admin configuration UI for arranging widgets in the dashboard slot.** Deferred to a later Phase 3 session. Arrangement is literally hardcoded in the blade.
- **No source-policy work.** That is session 213 — fully scoped and agreed during 211's close. `acceptedSources()` does not appear in this session.
- **Dashboard is admin-only already** via Filament auth. No new auth plumbing.

## Phase 2 — `WidgetRenderer` slot-handle parameter

Open `app/Services/WidgetRenderer.php`. The existing retrofit at line ~74 reads:

```php
$slot = app(SlotRegistry::class)->find('page_builder_canvas')->ambientContext($pageContext, $tokenPage);
```

Change `render()`'s signature to accept a trailing `string $slotHandle = 'page_builder_canvas'` argument. Inside `render()`:

- When `$slotHandle === 'page_builder_canvas'`, keep the existing call (`ambientContext($pageContext, $tokenPage)`).
- When `$slotHandle === 'dashboard_grid'`, call `ambientContext()` with no arguments (matching the Phase 1 signature) and do not compute `$tokenPage`.
- Ignore `SOURCE_PAGE_CONTEXT` token substitution on the dashboard path — richtext fields render their raw template without token replacement. This is consistent with "no current page" and the layout-constraints declaration.

Byte-equivalence guarantee for existing callers (no `$slotHandle` passed) must hold. Existing integration tests must stay green.

## Phase 3 — `DashboardSlotGridWidget`

New file `app/Filament/Widgets/DashboardSlotGridWidget.php`:

- Extends `Filament\Widgets\Widget`.
- `protected static string $view = 'filament.widgets.dashboard-slot-grid';`
- `protected int | string | array $columnSpan = 'full';` — full width of the Dashboard's two-column grid, so the slot render has room.
- Method to return the two widget-primitive widgets it renders, each as an unsaved `PageWidget` with `widget_type_id` and `config` set. Suggested shape: a `protected function widgets(): array` returning `['blog_listing' => PageWidget instance, 'carousel' => PageWidget instance]`. Pass to the view.

New blade `resources/views/filament/widgets/dashboard-slot-grid.blade.php`:

- Calls `App\Services\WidgetRenderer::render($widget, [], [], 'dashboard_grid')` per widget.
- Outputs the returned HTML, inline styles, and inline scripts.
- A light visual wrapper (e.g. a simple two-column grid via inline CSS or a small admin-scoped class in `public/css/admin.css`) is acceptable.

Register the widget in `app/Providers/Filament/AdminPanelProvider.php`'s `widgets([...])` array — add it alongside the existing four.

## Phase 4 — Widget selection + config

Choose hardcoded configs for the two widgets such that they render meaningful content on a fresh demo-seeded install:

- **Blog Listing** — `limit: 3`, standard layout. Relies on there being at least one published post in the dev DB; if none, the widget renders its empty-state.
- **Carousel** — `collection_handle: 'slides'` (or whichever demo collection the dev seed creates). Relies on the demo collection existing; fallback collection data is handled via `WidgetDataResolver` already.

If either widget renders poorly due to missing seed data, document the missing seed in the log and defer the seed addition to a later session — do not expand scope to seed-fixing this session.
