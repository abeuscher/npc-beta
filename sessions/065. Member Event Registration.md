# 065. Member Event Registration

Adds a "Register as member" path to event pages for portal users. Logged-in members skip the public form entirely and register with one click using their stored contact record. Non-logged-in visitors see the member button as the primary call to action alongside the public form as a secondary option. Fixes a pre-existing bug: the public form had no duplicate-registration guard, allowing the same email to register multiple times for the same event.

No schema changes required.

---

## Architecture decisions

- **`auth('portal')` in public views**: the portal guard is available globally in all Blade views. No middleware change is needed — `auth('portal')->check()` works on any page.
- **Button behaviour (not logged in)**: an `<a>` styled as a primary button linking to `route('portal.login', ['intended' => url()->current()])`. After login the user lands back on the event page and sees the one-click POST button.
- **Button behaviour (logged in)**: a minimal `<form method="POST">` with only a CSRF token and a submit button. No fields — the contact record supplies everything.
- **Public form visibility**: hidden entirely when the portal user is logged in. Shown as secondary when not logged in.
- **Silent dedup on the member route**: same pattern as the public form fix — if the contact already has a registration for this event, return `registration_success` silently without creating a duplicate.
- **Confirmation email**: fire the same confirmation mail that the public form fires, using the contact's email.
- **No new tables**: purely a new route, controller method, and template conditional.

---

## Phase 1 — New portal route and controller method

**File:** `routes/web.php`

Add inside the portal auth middleware group:

```
POST /account/events/{slug}/register → Portal\EventRegistrationController@store
```

**File:** `app/Http/Controllers/Portal/EventRegistrationController.php` (new)

The `store()` method must:

1. Resolve the event by slug — 404 if not found
2. Load `$user = auth('portal')->user()` and `$contact = $user->contact` — abort 403 if contact is missing
3. Run the same guard checks as `EventController@register`:
   - Event cancelled → redirect back with error
   - `registration_mode !== 'open'` → redirect back with error
   - `!is_free` → redirect back with error
   - `isAtCapacity()` → redirect back with error
4. Silent dedup: if `EventRegistration` already exists for this `event_id` + `contact_id`, redirect to the event page with `registration_success`
5. Create `EventRegistration`:
   - `event_id` → event id
   - `contact_id` → contact id
   - `name` → contact's full display name
   - `email` → contact's email
   - `status` → `'registered'`
   - `registered_at` → `now()`
6. Fire the confirmation mail (same mailable as the public form)
7. Redirect to the event landing page (or events prefix) with `registration_success`

The controller must **not** use `$request->validate()` for user-supplied fields — there are none. All data comes from the authenticated contact record.

## Phase 2 — Update event-registration widget template

**File:** `resources/views/widgets/event-registration.blade.php`

At the top of the `@isset($event)` block, compute:

```php
$portalUser = auth('portal')->user();
$portalContact = $portalUser?->contact;
```

### When the portal user is logged in (`$portalUser` is not null)

Replace the public form section with:

```html
<form method="POST" action="{{ route('portal.events.register', $event->slug) }}">
    @csrf
    <p>Registering as <strong>{{ $portalContact?->display_name }}</strong></p>
    <button type="submit">Register as member</button>
</form>
```

The public form must not be rendered at all when logged in.

### When the portal user is not logged in

Show the "Register as member" call to action **above** the public form:

```html
<a href="{{ route('portal.login', ['intended' => url()->current()]) }}">
    Register as member
</a>
```

Then render the public form below as-is (secondary option). Use whatever visual hierarchy the existing theme provides — primary vs secondary button styling, or a clear heading separation. Do not invent new CSS.

The existing conditional logic (`$isCancelled`, `$isAtCapacity`, mode checks) continues to gate both paths — do not render either option if the event is not open for registration.

---

## Phase 3 — Route name and registration redirect

Confirm the new route is named `portal.events.register`. Update `routes/web.php` accordingly.

The redirect after a successful member registration should resolve the event landing page URL using the same logic as `EventController` (check `$event->landingPage`, fall back to the events prefix).

## Out of scope

- Waitlist support
- Paid event registration for members
- Member registration management (cancel/edit) on the portal account page — deferred
- Any UI changes to the public event form beyond the conditional hide
