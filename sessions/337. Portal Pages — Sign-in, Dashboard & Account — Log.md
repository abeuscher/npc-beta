## What was built

### Discovery / decision (Open Gate)

- Found **two parallel portal page systems** seeded side-by-side: `system/*` (type `system`, newer — s322: `system/login`, `system/account`→`portal_account_dashboard`) and `members/*` (type `member`, older — `PortalPageSeeder` s187: a bare-text dashboard + split `edit-account` / `change-password` / `event-registrations` pages, plus the `portal` nav menu that `layouts.portal` actually renders). Post-login landed on `/system/account`, which **wasn't in the nav** → nav state resolved to nothing. Owner-confirmed resolution: **consolidate the portal onto `/members` (auth stays at `/system`)**, retire the duplicate `system/account` dashboard + the split pages, and rewire the nav. Documented the *why login routing is locked* rationale (owner recollection) at the same time.

### Phase 1 — Sign-in

- `PortalLogin` rebuilt as a centered branded auth card (brand wordmark + heading + fields + primary submit + forgot/sign-up links), floating on a shared branded band. Same card treatment applied to `PortalSignup` + `PortalForgotPassword` for a consistent auth experience. `data-tour="portal.login"` reserved.

### Phase 2 — Account edit

- `/members/account` now carries **both** account forms — `PortalContactEdit` (Mailing Address + Email) and `PortalChangePassword` (Password) — styled as sectioned profile cards (`.portal-section`). Alerts consolidated at the top (the duplicate generic alert block was dropped from `PortalChangePassword`, which is only ever rendered alongside `PortalContactEdit`). `data-tour="portal.account"` reserved.

### Phase 3 — Dashboard

- `/members` (the post-login landing) composed as a real dashboard: the shared portal hero band (page title + "Welcome back, {first_name}") + a welcome `TextBlock` + an illustrative `BarChart` (the existing `chart-demo` collection — **not** member-scoped data, per the portal-security rule). `data-tour="portal.dashboard"` reserved on the hero.

### Phase 4 — Cross-cutting

- **One visual language:** a `portal-hero` branded band defined once in `layouts.portal` (covers dashboard / account / event-registrations); the sign-in card reuses the same band/tokens in `layouts.public`. So the logged-out auth screens and the logged-in portal read as one designed surface across the two chromes.
- **`layouts.portal` now loads the widget asset bundle** (CSS + JS from the build manifest), mirroring `layouts.public` — required so page-builder widgets (the dashboard `BarChart`'s chart.js + `NPWidgets.barChart`) actually render in the portal. The portal layout had silently lacked this.
- **Routing:** `/system/account` is now a legacy redirect alias → the `/members` dashboard (keeps the `route('portal.account')` name working across 6 controllers + tests with zero churn). `MembershipCheckoutController`'s account URL repointed to `/members`. Post-login lands on the dashboard.
- **Seeders reconcile in place** (idempotent + self-correcting): `PortalPageSeeder` creates the dashboard + combined account + event-regs, force-deletes the retired split pages, rewires the `portal` nav to Dashboard / Account / Event Registrations, and ensures the `chart-demo` collection; `SystemPageSeeder` retires the `system/account` page.
- `portal.nav` `data-tour` anchor + an active-nav (`aria-current`) style added.
- Documentation: the locked-auth-routing rationale (login bound to fixed controllers; content prefixes operator-editable) added to `resources/docs/settings-general.md` + `docs/app-reference.md`, with a portal-topology note.

### Close housekeeping

- Fixed a **pre-existing flaky test** (`RecordDetailViewSeederTest` idempotency case asserted widget order from a query with no `orderBy` → nondeterministic on Postgres; surfaced once on the deploy CI run, passed on re-run). One-line `orderBy('sort_order')`, matching its sibling cases. Not a 337-introduced defect; fixed here so CI stops flaking.

## Security considerations

Portal scoping unchanged and intact. The login / sign-up / password-reset / email-verify **auth flow stays bound to its fixed controllers under `/system`** (GET routes prefixed by `system_prefix`; POST endpoints at fixed root paths) — only the URL *prefixes* are operator-editable, never the route→controller binding; this boundary is now documented. The logged-in account-edit page moved to `/members` (a `type=member` page gated by `PageController` auth + verified-email), but the *password-change operation* is unchanged — it still posts to the fixed `/account/password` endpoint behind portal auth + CSRF + current-password check. The dashboard `BarChart` uses illustrative `chart-demo` data, not the member's own records (avoids the per-`contact_id` scoping surface). No new HTTP surface; contract stays v2.3.0.

## Production deployment notes

- **Front-end build:** `npm run build` (the `_custom.scss` portal styling, via `public.scss`) **and** `build:public` (host partials feed the widget bundle too). Both run.
- **Page structure on an existing node:** the new portal topology lives in seeder-managed pages, so deploying the code alone does **not** reshape an existing node's DB. Run the two seeders once to reconcile (idempotent — creates new, retires old, rewires nav, ensures `chart-demo`):
  `php artisan db:seed --class=PortalPageSeeder --force` and `--class=SystemPageSeeder --force`.
  (Alternatively the admin Export/Import carries pages + widgets + nav menus + collections, but it's additive — it won't delete the retired pages or rewire the nav, so the seeders are the clean path.)
- Portal testing needs a verified portal account; on a `log`-mailer node, signup→verify dead-ends — create a verified `PortalAccount` directly (the "seeded portal login" follow-on, next session, removes this friction).
