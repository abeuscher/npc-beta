The IsView contract methods (`handle`, `slotHandle`, `widgets`, `layoutConfig`) coexist with Eloquent attribute access — `handle()`/`recordType()` read from columns; `widgets()` returns the `pageWidgets()->where('is_active')->with('widgetType')->orderBy('sort_order')->get()->all()` — array contract preserved. Polymorphic relation `pageWidgets(): MorphMany` declared for the inverse direction.
- [database/factories/RecordDetailViewFactory.php](database/factories/RecordDetailViewFactory.php) — sensible defaults (`record_type: Contact::class`).
- The model lives outside `App\Models\` so Laravel's default factory resolver fails. Added `protected static function newFactory(): Factory { return RecordDetailViewFactory::new(); }` to the model — explicit binding.

### Phase 3 — Sub-nav rendering primitive

- [app/Filament/Concerns/HasRecordDetailSubNavigation.php](app/Filament/Concerns/HasRecordDetailSubNavigation.php) — trait with three hooks: `subNavigationEntryPage(): ?string` (default null — host's own page class for self-listing), `recordDetailViewSubPageClass(): ?string` (default null — concrete page class that renders Views), `additionalSubNavigationPages(): array` (default empty — Filament-form-only sub-pages). `getSubNavigation()` orders entries `[entry] + [Views] + [additional pages]`. `canAccess(['record' => $this->record])` filters out sub-pages the current user cannot reach (added during verification — without it Header/Footer entries would surface for users without `edit_site_chrome` and 403 on click). The "skip when ≤ 1" threshold counts entry + Views + additional pages together.
- [app/Filament/Resources/Pages/RecordDetailViewPage.php](app/Filament/Resources/Pages/RecordDetailViewPage.php) — abstract base for the generic "render a View as a widget grid" sub-page. Resolves the View via `ViewRegistry::findByHandle($recordType, $handle)` (404 on miss). Uses `InteractsWithRecord` + `HasRecordDetailSubNavigation` with `insteadof` resolving the `getSubNavigation` collision. Mounts `RecordDetailViewWidget` via `getFooterWidgets()` and passes the View via `getWidgetData()`.
- [resources/views/filament/resources/pages/record-detail-view-page.blade.php](resources/views/filament/resources/pages/record-detail-view-page.blade.php) — minimal `<x-filament-panels::page>` wrapper; widgets render through Filament's footer-widget machinery.
- [app/Filament/Widgets/RecordDetailViewWidget.php](app/Filament/Widgets/RecordDetailViewWidget.php) — rewired. Removed the inline hardcoded RecordDetailView construction. Added `public ?RecordDetailView $detailView = null` (renamed from the prompt's `$view` to avoid collision with Filament's `protected static string $view` — PHP fatal on redeclaring static as non-static; confirmed by empirical test). When `$detailView` is set, render its widgets directly. When null and `$record` is set, fall back to `ViewRegistry::forRecordType($this->record::class)->first()` — preserves the 5b `EditContact` footer-mount behavior byte-equivalent. Also set `protected static bool $isDiscovered = false` so Filament's `discoverWidgets()` does not auto-mount the widget on the admin dashboard (an empty `<x-filament::section>` was rendering at the top of the dashboard otherwise).
- Trait applied to [app/Filament/Resources/ContactResource/Pages/EditContact.php](app/Filament/Resources/ContactResource/Pages/EditContact.php) — exercises the trait on the canonical record type. Sub-nav skipped (1 View → ≤ 1 → return []).

### Phase 4 — Templates/Themes refactor (Chrome / Colors / SCSS sub-pages, EditPageTemplate restructured)

The original prompt's "Header/Footer become empty Views" framing was clarified mid-session: Header and Footer Views are page-builder-mount **anchors** (the existing `header_page_id` / `footer_page_id` `Page` records mount inside the View-backed sub-page). The user's stated motivation: a) tabs are awkward, b) the existing Alpine-tab pattern mounts both header and footer page-builders simultaneously, surfacing a "they reference each other on save" bug — sub-nav fixes the bug at the usage site by mounting one chrome page-builder per request, and c) establishes the sub-nav pattern for any future page that needs another nesting level.

- [app/Filament/Resources/TemplateResource/Pages/EditPageTemplateChrome.php](app/Filament/Resources/TemplateResource/Pages/EditPageTemplateChrome.php) — concrete sub-page using the trait directly (does **not** extend `RecordDetailViewPage` — that abstract base stays for future record-detail widget cases). Routes via `templates/{record}/edit-page/chrome/{view}`. Mounts the existing `@livewire('page-builder', ['pageId' => …])` against the parent template's `headerPageId` (when handle = `page_template_header`) or `footerPageId` (when handle = `page_template_footer`). Migrated the chrome management methods from `EditPageTemplate`: `enableCustomChrome`/`inheritChrome` (renamed from per-position-specific names — single set of methods that branches on `$this->chromePosition`), `getHasCustomChrome`, `createChromePageFor`, `headerPageId`/`footerPageId` properties for live binding. `canAccess()` gated by `edit_site_chrome`.
- [app/Filament/Resources/TemplateResource/Pages/EditPageTemplateScss.php](app/Filament/Resources/TemplateResource/Pages/EditPageTemplateScss.php) — custom Filament `Page` (with `InteractsWithRecord`). Migrated `$themeScss`, `$buildOutput`, `$buildSuccess`, `saveAndBuildScss()`, `clearScss()`. Routes via `templates/{record}/edit-page/scss`. `canAccess()` gated by `edit_theme_scss`. `protected static ?string $title = 'SCSS'` so the nav label is "SCSS" rather than the Filament-default class-basename humanization "Edit Page Template Scss".
- [app/Filament/Resources/TemplateResource/Pages/EditPageTemplate.php](app/Filament/Resources/TemplateResource/Pages/EditPageTemplate.php) — restructured during manual testing: merged the General fields (Name + Description) with the Colors fields onto the entry page. `EditPageTemplateColors` was deleted as a separate page; the form now stacks Name + Description in a left-half-width column (via `Forms\Components\Group::make([...])->columnSpan(6)`) with the Colors section (six color pickers, 12-column grid) beneath at full width. `clearAppearance()` migrated onto this class. Title set to the template's name (so the user sees which template they're editing); breadcrumbs read `Templates > Edit Page Template > Label and Colors`. Trait applied with `subNavigationEntryPage` returning self (entry self-lists as the first sub-nav entry), `recordDetailViewSubPageClass` returning Chrome, `additionalSubNavigationPages` returning `[EditPageTemplateScss::class]`. Sub-nav order on every page in the cluster: `Label and Colors → Header → Footer → SCSS`.
- Updated [app/Filament/Resources/TemplateResource.php](app/Filament/Resources/TemplateResource.php) `getPages()` — added `edit-page.chrome` and `edit-page.scss` routes; `edit-page.colors` removed when Colors merged into the entry.
- New Blade views for all three new sub-pages; existing `partials/scss-editor.blade.php` reused by `EditPageTemplateScss`.
- `EditContentTemplate` left unchanged — the original prompt expected ContentTemplate to have similar tabs to migrate; on read, ContentTemplate has no tabs (just General form + page-builder mount). No trait application.

### Mid-session structural restructure (post-manual-testing)

After the initial four-page split landed clean, the user clarified the entry-point UX: stack Name + Description (half-width) at the top with Colors beneath on a single page, rather than splitting "General" and "Colors" into separate pages. The trait gained a fourth hook (`subNavigationEntryPage`) to support self-listing in sub-nav, the Colors page was deleted entirely (form fields and `clearAppearance` migrated onto `EditPageTemplate`), and the four-page sub-nav order became `Label and Colors → Header → Footer → SCSS` with the entry page self-listed first.

### Page-builder preview-link fix

Chrome `Page` records have `_`-prefixed slugs (`_header`, `_footer`, or `_{position}_{8-char-id}` for custom-per-template chrome pages) that are 404 in the public router by design. The page-builder's `details_url` / `page_url` was generating `localhost/_footer` etc. when mounted inside the chrome editor. Patched [app/Livewire/PageBuilder.php](app/Livewire/PageBuilder.php) to detect `$page->type === 'system' && str_starts_with($page->slug, '_')` and route the preview URL to `/` (the front page) for those rows. The `home` slug case still resolves to `/` as before; everything else unchanged.

- **Admin UI for authoring per-record-type View sets** — Phase 5c.5, scoped at close-gate. Vue page-builder mounted in a new `record_detail` mode against a Filament `RecordDetailViewResource`. Prompt drafted at `sessions/230. *.md`.

- **Phase 5e — Permissions in the contract layer.** Forcing function: first sensitive concrete widget in 5d+. Recent Donations is the natural one. The principle is concentrated enforcement at the resolver, fail-closed per source/contract — same shape as source-as-capability.
- **Widget help system integration.** Forcing function: second or third concrete widget in 5d+ when the rollup gap becomes visible.
- **Themes/Templates top-level nav merge.** Carved out of 5c per the original prompt scope. Separate follow-up session — a structural decision about navigation taxonomy that doesn't fit cleanly inside any current phase.
- **`DashboardConfig` retrofit to `DashboardView` implementing `IsView`.** Cheap follow-up post-5d, scheduled ad-hoc.
- **`Page` implementing `IsView`.** Per the "a+" framing, Page-as-View is the elective bottom of the View stack. Could be never.
- **Per-record-type token expansions on `RecordContextTokens::TOKENS`.** The token map is currently `['record_id', 'record_type']`. Different record types have different label-derivation patterns; deferred to the first concrete widget that needs to display a per-record label.
- **`PageContextTokens` namespace migration from `App\Services` to `App\WidgetPrimitive`.** Pre-existing inconsistency carried from 5b. Standalone mechanical cleanup pass for a convenient cleanup session, not load-bearing.
- **`canAccess` gate on the trait** was added during verification rather than designed in. Worth revisiting when 5e (perms in the contract layer) lands — the right shape may be to push permission gating down into the registry rather than the trait, matching the "concentrated enforcement at the resolver" discipline.
- **Discoverability of the entry page form.** Users with only `update_page` (no `edit_site_chrome`, no `edit_theme_scss`) see only "Label and Colors" as a candidate sub-nav entry → ≤ 1 → sub-nav skipped → they see the entry form unadorned (no nav back to anything). Acceptable for 5c since the entry IS the form; flagged here in case later sessions expose a usability gap.
