## What was built

A small, idempotent `widgets:sync` artisan command, wired into every deploy/upgrade path after `migrate --force`, so a pure-schema widget change (a new/changed inspector control with no migration) that ships in the image actually lands on an already-installed server. Closes the ⭐ owner-designated `[0.324.01]` inbox item — the deploy-pipeline half of the pair whose chrome-render half shipped at 340.

**Root cause (re-verified against current code at session open):** the page-builder control list *and* the public render both read a widget's fields from the `widget_types.config_schema` DB column — `WidgetType::forPicker`, `WidgetResource.php:38`, `WidgetRenderer.php:49` — not from the live `WidgetDefinition` class at request time. That column is only (re)written by `WidgetRegistry::sync()`, whose sole caller was `WidgetTypeSeeder::run()`. Every deploy path ran `migrate --force` + `build:public` and **never** the seeder, so pure-schema widget changes stayed invisible on installed servers.

### The `widgets:sync` command

- Added `app/Console/Commands/WidgetsSyncCommand.php` — signature `widgets:sync`; calls `app(WidgetRegistry::class)->sync()` (idempotent `updateOrCreate` by handle) and prints a one-line synced-count summary. Mirrors the existing `widgets:lint` / `widgets:manifest-json` command structure + output style; auto-discovered by Laravel 11 (no Kernel registration). Live run synced 39 widget definitions.

### Wired into every deploy path (after `migrate --force`, before `build:public`)

- `.github/workflows/deploy.yml` — `widgets:sync` step added immediately after the `migrate --force` step.
- `.github/workflows/deploy-demo.yml` — same insertion (SHA-tagged / VERSION-exempt path; the sync gap applies equally).

## Security considerations

`widgets:sync` is an internal artisan command that runs only in the deploy/upgrade path (operator / CI context). It takes no user input, adds no route or HTTP surface, and only re-syncs the widget definitions the image already ships — writing the same `widget_types.config_schema` the seeder already wrote. No auth, permission, or `/api/*` contract surface is touched; the Fleet Manager agent contract is unchanged at v2.3.0. None beyond standard convention.

---

## Schema changes

None. No migration. The command updates existing `widget_types` rows in place via the existing `WidgetRegistry::sync()` (`updateOrCreate` by handle); no column, table, or `docs/schema/` change.

## Production deployment notes

The `widgets:sync` step now runs automatically on every deploy after `migrate --force`, so no manual step is needed on a normal deploy going forward.

For a node already stuck on stale widget schema (deployed before this step existed, or upgraded out-of-band), the immediate manual unstick is:

```
docker exec nonprofitcrm_app php artisan widgets:sync
```

(equivalently, the seeder path that also runs the sync: `docker exec nonprofitcrm_app php artisan db:seed --class=WidgetTypeSeeder --force`).

**FM coordination (no contract change):** Fleet-Manager-orchestrated upgrades that run `migrate --force` on a node outside `deploy.yml` must add the same `widgets:sync` step. Flagged in the Cross-Repo block of `sessions/session-outlines.md`; the FM-side incorporation is a separate FM-repo task.
