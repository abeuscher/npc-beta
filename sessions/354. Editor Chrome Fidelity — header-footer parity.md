# 354. Editor Chrome Fidelity — header/footer parity with the front end

The back-end page-builder editor renders the site header/footer into the preview canvas, but they are **not styled to match the front end**. The front end (public render) is the source of truth. Bring the editor's chrome to visual parity with the public render, header/footer only — then fold in a small page-builder-editor-appearance housekeeping batch.

## Scoping (already done — verify against code, build from it, don't re-derive)

Both sides render the **same server-produced chrome HTML** — `ChromeRenderer::render('_header'|'_footer')` / `renderById()`. So this is **not a content problem**; it's structural wrapping + render context. The editor reuses the HTML but drops the public layout's wrappers and renders in an altered scope.

**Front end — the source of truth** (`resources/views/layouts/public.blade.php`, ~226–250):

```
body.np-site
  div.site-nav-wrapper[.site-nav-wrapper--overlay]  style="--overlay-nav-link-color:…; --overlay-nav-hover-color:…; --overlay-text-color:…"
    header.site-header
      {chrome html}
  main#main
  footer.site-footer
    {chrome html}
```

Plus inline `<style>@layer host { .np-site nav { font-family:… } }</style>` blocks the public layout emits (nav font-family; content scheme vars on `<main>`).

**Back-end editor** (`resources/js/page-builder-vue/components/PreviewCanvas.vue`, ~262–345):

```
div.np-site.widget-preview-scope   (zoom: <zoomFactor>; container-type: inline-size; width: <presetViewport>px)
  div.preview-chrome.preview-chrome--header
    div.preview-chrome__html   (pointer-events:none)
      {chrome html}            ← v-html="store.chromeHeader.html"
  …page widgets…
  div.preview-chrome.preview-chrome--footer
    div.preview-chrome__html
      {chrome html}
```

**The named cause(s):**

1. **Missing wrapper structure.** The editor omits `.site-nav-wrapper` (and its `--overlay--` variant class + the three `--overlay-*` nav color custom properties) and `.site-header` / `.site-footer`. Every public chrome style keyed to those selectors/vars therefore does not attach. This is almost certainly the dominant cause.
2. **Altered render context.** The editor renders inside `zoom` + `container-type: inline-size` (`np-viewport`) at a fixed preset width, so full-bleed (`100vw`-style) and `@container`-driven chrome behavior resolves differently than the real page.
3. **Possibly dropped payload.** The editor injects only `store.chromeHeader.html` — not the `styles` blob `ChromeRenderer` returns, and the inline `@layer host` rules the public layout emits aren't reproduced. **Verify whether `styles` is load-bearing** — the public layout also emits only `['html']`, so `styles` may be empty in practice (chrome widget CSS comes from the prebuilt `/build/widgets/*.css` bundle). Don't assume; check before treating it as a cause.
4. **Overlay-nav vars not threaded.** The `--overlay-*` vars are computed in `public.blade.php` from page/template context; `app/Livewire/PageBuilder.php::chromeBands()` may not pass them through to the Vue store. If the chrome uses overlay nav, the editor needs them.

**The fix has a bounded floor and a Playwright-verified tail:**

- **Floor (concrete, small):** make the editor's chrome bands reproduce the public wrapper structure — wrap the header HTML in `.site-nav-wrapper[.--overlay] > header.site-header` and the footer in `footer.site-footer`, keeping the `.preview-chrome` hover/edit affordance as an **outer/sibling** layer (don't delete the "Edit header/footer ↗" deep-link). This likely recovers most of the look.
- **Tail (needs the Playwright loop):** thread the overlay-nav vars through `chromeBands()` if the chrome uses them; confirm parity at the **preset viewports** (desktop/tablet/mobile) under the editor's zoom/container scope, not just at one width; resolve the `styles`-blob + `@layer host` question. Each is a visual confirmation against the front-end render of the same page.

## Open questions to resolve at session start

1. **`styles` blob + `@layer host`.** Is the chrome `styles` blob non-empty in practice, and do the public layout's inline `@layer host` rules (nav font-family, scheme vars) materially affect chrome? If yes, the editor must emit them; if no, leave them out. Settle by inspection before wiring.
2. **Overlay-nav scope.** Does the install's chrome actually use overlay nav (`site-nav-wrapper--overlay` + `--overlay-*` vars)? If the current header isn't overlay, threading the vars is lower priority — but reproduce the wrapper class structure regardless so parity holds if overlap is later turned on.
3. **Standing guard vs dev-only.** Should the editor↔public chrome comparison ship as a **standing Playwright parity guard** (a durable regression artifact for this exact drift class), or is the dev-loop spec torn down at close? Lean toward a small standing guard given the editor↔public path is a known recurring drift source — confirm at the gate.
4. **Housekeeping pairing + the optional guard.** Confirm the fold: `[0.344.01]` + `[0.319.01]`, with `[0.324.01]` (knockout guard) as optional stretch. Recommendation: fold the two appearance bugs; take the knockout guard only if the owner wants the durable gate this session (it's a test-infra build, not an appearance fix, and can outgrow housekeeping size).

## Phases

Single primary fix + a housekeeping fold. Split across iterations per Rule 11 if the chrome tail runs long; otherwise land together.

### Phase 1 — Editor chrome fidelity (the bug)

Reproduce the public chrome wrapper structure in `PreviewCanvas.vue`'s header/footer bands (`.site-nav-wrapper` + overlay class/vars where applicable, `.site-header` / `.site-footer`), preserving the `.preview-chrome` edit affordance. Thread any missing context (overlay-nav vars, the `styles` blob if load-bearing) through `app/Livewire/PageBuilder.php::chromeBands()` and the Vue store/`types.ts`. Verify editor-vs-front-end parity with Playwright at the preset viewports; surface the visual result for the owner. `npm run build` after Vue/SCSS changes.

### Phase 2 — Page-builder housekeeping fold

After re-verifying each at the walk:

- `[0.344.01]` **Drag-handle z-index vs admin header** — the column drag-handles render over the admin header as they scroll offscreen. Resolve the layering (the page-builder editor's z-index stack vs the Filament admin header).
- `[0.319.01]` **Resize-handle text scaling vs page zoom** — the columns-primitive resize handles are oversized to offset page zoom; scale their text down as the view approaches actual size. Best-effort ramp tuning, not an exact fix (the "even out at page size" target isn't technically reachable — per the inbox note).
- *(Optional)* `[0.324.01]` **Knockout-controllability Playwright guard** — only if folded at the gate.
