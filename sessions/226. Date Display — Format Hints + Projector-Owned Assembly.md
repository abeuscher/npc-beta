# 226. Date Display — Format Hints + Projector-Owned Assembly

Add per-widget date format selection to the contract pattern, retire the `_label` field-naming convention in favor of concept-named display fields (`event_date`, `event_time`, `post_date`), and pull every contract-locked widget's date formatting back into the projector — leaving the templates as dumb token renderers.

This session is **architectural rationalization, not new feature surface.** The user-visible delivery is one new "Date format" select in the inspector for five widgets. The deeper delivery is enforcing the rule the session 225 work surfaced: *the projector is the view-model assembler; the template is a dumb renderer.* Templates that today contain `@php` blocks with imperative `Carbon::parse + DateFormat::format` calls are template-side anti-patterns that quietly opt out of the contract pattern's fail-closed promise. This session pulls them back in.

The framing the user named for EventDescription specifically: it's the canonical "show me this event" surface — the public layer's pass-through to event description, image, and date/time. Not a list, not a card, not page-context-driven. With that role named clearly, the structural same-day vs cross-day branching is solving a problem we don't have (multi-day events are not meaningfully supported), so it goes away.

---

## Design decisions (resolved before starting)

- **`DataContract::formatHints` is an associative array on the contract**, keyed by projector field name, value is a `DateFormat` constant. Default `[]`. Optional. Sibling axis to `querySettings`: both express per-instance, widget-curated knobs that travel via the contract.
- **Hint validation happens in the projector, not on the contract.** The contract carries whatever the widget gave it; the projector validates each hint against the per-field option-set helper (`DateFormat::eventDateOptions()` / `DateFormat::postDateOptions()`) and falls back to the per-field default on unknown values. Mirror of `querySettings` `orderByOptions` allowlist enforcement.
- **Allowlist is the schema's `options` map.** Each widget declares its `date_format` schema field with an `options` value that is exactly the option-set helper's return. A future site-setting fallback becomes one more lookup layer; this session does not deliver it.
- **Field rename: retire `_label` suffix; adopt `<model>_<concept>` for derived display fields.** The naming convention from session 217 (`<col>` for ISO machine, `<col>_label` for canonical display companion) made sense when display fields were 1:1 with columns. `event_date` and `event_time` are derived display concepts that *cross* start/end columns. Naming them column-style would be misleading. The new rule:
  - **Machine fields** keep column names: `starts_at`, `ends_at`, `published_at` (raw ISO).
  - **Display fields** use `<model>_<concept>`: `event_date`, `event_time`, `post_date`.
  - **Posts retire `published_at_label`** even though there's no equivalent column-crossing — the convention is uniform across models.
- **Multi-day event support is explicitly out of scope.** The session 223 EventDescription template's same-day vs cross-day `@php` block is removed. `event_time` is `start – end` (or just `start` if no end). If multi-day support becomes a need, a future session can re-introduce the structural decision *in the projector* (where it belongs), not in the template.
- **`event_time` always uses `TIME_SMART` this session.** Format-hint only applies to `event_date` (the date portion). A future session can add a separate `time_format` hint if user feedback supports the choice. Keep the first hint mechanism focused.
- **One new vocabulary entry: `EVENT_TILE_DATE` = `'D, M j'`.** "Sat, May 2" — the abbreviated dow-prefixed compact date. Required because the existing `EVENT_FULL` carries the year + time, and the user's preferred tile shape (`Sat, May 2 / 5:30pm – 8:30pm`) has the date and time on separate lines, so the date portion alone doesn't want the year or time. `LONG_DATE` ("May 2, 2026") and `MEDIUM_DATE` ("May 2, 2026") both lack the dow prefix that visually anchors event tiles. `EVENT_TILE_DATE` fills that gap.
- **Per-widget option-set curation.** Each option-set helper returns a curated subset of the vocabulary appropriate to that surface. Initial curation:
  - `eventDateOptions()`: `EVENT_TILE_DATE` (default), `EVENT_FULL`, `EVENT_LIST_DATE`, `LONG_DATE`, `MEDIUM_DATE`.
  - `postDateOptions()`: `LONG_DATE` (default), `MEDIUM_DATE`, `EVENT_FULL` excluded (irrelevant for posts).
  Each widget can declare its own subset by overriding which helper it draws from.
- **Tile cleanup is bundled, not separate.** The session 225 conversation surfaced two debug-y default templates (EventsListing's `Ends:` / `{{item.slug}}` / `{{item.starts_at}}` lines; BlogListing's `{{item.slug}}` / `{{item.published_at}}` lines) and a structural disagreement between the Blade fallback default and the definition's schema/defaults default. The field rename forces every default template to update; the cleanup is folded in. After this session, every contract-locked widget's default template is clean (no debug tokens, no `Ends:` placeholder strings, no out-of-sync defaults).
- **Existing widget instances get no data migration.** Pre-existing `page_widgets` rows whose `config.content_template` references `{{item.starts_at_label}}` or `{{item.published_at_label}}` will silently render those tokens as empty strings after this session. The user (solo deploy, pre-Beta 1) is the only consumer; resetting a widget's template via the inspector or hand-editing the JSONB column are both acceptable repairs. This is documented in the log's production deployment note. **A migration is explicitly not written** because token rewrites against custom templates (which are the only ones that *aren't* getting reset by the default change anyway) are too easy to false-positive.
- **`DataContract::formatHints` is enforced as final on the dataclass.** Like every other field on `DataContract`, it is `public readonly`. Not mutable post-construction.

## Phase 3 — `SystemModelProjector` refactor

1. `projectorFor()` ([SystemModelProjector.php:75](app/WidgetPrimitive/Projectors/SystemModelProjector.php#L75)) threads `$contract->formatHints` into per-model closures:
   ```
   'post'  => fn (Page $post) => $this->projectPost($post, config('site.blog_prefix', 'news'), $contract->formatHints),
   'event' => fn (Event $event) => $this->projectEvent($event, $contract->formatHints),
   ```
   Product is unaffected (no date fields).
2. `projectEvent(Event $event, array $formatHints = [])`:
   - **Retire** `starts_at_label` and `ends_at_label` from the return array.
   - **Add** `event_date`: validated hint against `DateFormat::eventDateOptions()`, fallback `DateFormat::EVENT_TILE_DATE`, formatted via `DateFormat::format($event->starts_at, $resolvedFormat)`.
   - **Add** `event_time`: assembled by a new private helper `buildEventTime(Event $event): string`. Logic:
     - If `starts_at` is null → return `''`.
     - Format start as `TIME_SMART` ("5pm" or "5:30pm").
     - If `ends_at` is null → return start alone.
     - Else format end as `TIME_SMART` and return `start . ' – ' . $end` (em-dash + spaces).
     - **No same-day vs cross-day branching.** Multi-day not supported.
3. `projectPost(Page $post, string $blogPrefix, array $formatHints = [])`:
   - **Retire** `published_at_label`.
   - **Add** `post_date`: validated hint against `DateFormat::postDateOptions()`, fallback `DateFormat::LONG_DATE`, formatted via `DateFormat::format($post->published_at, $resolvedFormat)`.
4. Hint validation pattern (factor into a private helper if it appears more than once):
   ```php
   $resolved = isset($formatHints['event_date']) && array_key_exists($formatHints['event_date'], DateFormat::eventDateOptions())
       ? $formatHints['event_date']
       : DateFormat::EVENT_TILE_DATE;
   ```
   Fail-closed: unknown hint → default. No exception thrown; the user picked an invalid value, the widget renders the default. Mirror of session 221's `orderByOptions` fallback to `created_at`.

## Out of scope

- **`time_format` hint.** This session only delivers `date_format`. `event_time` always uses `TIME_SMART`. A second hint can land in a follow-on if user feedback supports the choice.
- **Site-setting "default date format."** The hint mechanism makes this a cheap one-layer extension, but it's a separate concern with its own UX (where in Settings? Per-widget-type defaults? Tenant-wide?). Defer.
- **Multi-day event support.** Out of scope as a feature; the projector's `event_time` does not distinguish same-day from cross-day. If multi-day support is later wanted, the structural decision lives in the projector — not the template.
- **Memos `posted_at` formatting.** Different code path (JSONB key on collection items, not a system-model column). The template-side formatting remains for now. Bringing it into the pattern would require either a SWCT-side hint mechanism or a per-content-type display field.
- **PortalEventRegistrations.** Direct-query widget, not contract-locked. Migrating would require a contract for portal data, which is its own architectural beat.
- **EventCalendar.** JS-driven (jcalendar lib); date formatting happens client-side. Out of scope.
- **Data migration to rewrite existing widget `content_template` strings.** Not written. Pre-Beta 1, the user resets templates to defaults manually if the field rename breaks any custom layouts; risk of false-positive token rewrites against custom templates is too high.
- **Token modifier syntax** (`{{item.event_date|long}}`). Different ergonomic — useful for power users editing the rich-text card template, but doesn't surface in widget settings. Defer.
- **`event_location` projector field migration of other location-composing logic.** This session only adds `event_location` to projectEvent because EventDescription's template needs it. ThisWeeksEvents's location composition also lives in its template — leave as-is for now (its logic differs slightly: meeting_label fallback). Move in a follow-on if uniformity becomes valuable.
