# A010. Page Editor Full-Screen & Faithful Viewports

> **Agentic session.** Four-part page-editor pass, planned interactively and executed in one cloud session on `claude/clever-johnson-cwwjrr`: (0) remove the zoom-compensated layout handle bar, (1) lift the viewport preset into the editor store + extract the canvas control bar, (2) full-screen editing mode with a collapsed-right inspector drawer, (3) media-faithful tablet/mobile preview via an editor CSS variant (`@media` → `@container np-viewport`), (4) the `activeBreakpoint` store seam for future per-breakpoint authoring. Non-boundary, CRM contract stays v2.3.0, no schema, no migration.

One sentence: the page editor gains a CSS-driven full-screen mode whose canvas re-zooms itself through the existing ResizeObserver, and the tablet/mobile presets finally render what a phone would render, because the breakpoint rules now evaluate against the simulated viewport instead of the operator's browser window.

---

## Settled design decisions (locked at the planning conversation)

1. **Full-screen is a fixed overlay, almost entirely CSS.** `.vue-editor--fullscreen { position: fixed; inset: 0; z-index: 40 }` painted over the admin chrome; the document scroll locks behind it. No teleport, no DOM moves — the existing pane ResizeObserver re-derives the canvas zoom when the pane widens. Layering: overlay 40, drawer/control bar 44–45, body-teleported modals 50, inline format toolbar 60, Livewire loading bar 200.
2. **Not persisted.** An edit page always opens windowed so the surrounding Filament form (title, slug, status) stays discoverable. (Deliberate divergence from the localStorage-persisted admin `np-fullscreen` topbar toggle, which is a different feature — a max-width uncap.)
3. **Inspector drawer overlays; it does not push.** Sliding over the canvas keeps the pane width — and with it the zoom — stable on every toggle. Starts collapsed each time full-screen is entered; expands via the edge tab only (no auto-expand on selection in v1). Save stays fixed bottom-right and shifts left while the drawer is open.
4. **The control bar carries exactly two controls in full-screen** — the full-screen toggle (which exits) and the viewport selector — pinned top-left as a floating cluster. Windowed, the same bar spans the editor top: toggle left, viewport right.
5. **The layout top handle bar is gone, not adapted.** It was deliberately oversized (3.5rem/2rem type) to compensate for canvas zoom-out, which inverts into a giant banner as zoom approaches 1 in full-screen. The always-visible side selector takes the root drag-handle duty (`PreviewCanvas` root draggable handle selector now includes `.layout-region__selector`). Owner's call: see if the side handle is enough before building anything new.
6. **Media-faithfulness is a build-time editor variant, not a runtime rewrite.** `build:public` writes `public-widgets-editor-{hash}.css` beside the public bundle with width-only `@media` preludes rewritten to `@container np-viewport (…)` (new `EditorCssTransformer`); the manifest gains `editor_css`; the admin head loads the variant via `WidgetAssetResolver::widgetEditorCss()` with **fallback to the public bundle** when the manifest predates the variant — a stale manifest degrades to the old behaviour, never to an unstyled preview. The public bundle is byte-untouched.
7. **The container is named.** Widgets declare anonymous query containers of their own (e.g. `event_mini_calendar`, session 343), so an unnamed `@container` would resolve against the nearest widget container instead of the viewport scope. The preview scope element carries `container-name: np-viewport`; transformed rules and the typography compiler's container mode target it by name. Public pages have no `np-viewport` container and never load the variant.
8. **`TypographyCompiler::compileScoped(..., containerQueries: true)`** covers the page-builder's inline-injected breakpoint font ramps (the request-time half that doesn't ride the bundle); the section-spacing compression, `.site-container` max-widths, and all widget `@media` ride the bundle transform.
9. **`activeBreakpoint` is an authoring label, not a CSS tier.** `desktop ≥1200 / tablet ≥768 / mobile` derived from the preset (so the 1024 preset is 'tablet' even though ≤768 public rules don't fire at 1024). Exposed from the store; nothing consumes it yet — it is the seam the deferred per-breakpoint authoring controls (session 335 carry-forward) will key off.

## Known limits (documented, accepted)

- A transformed rule whose selector targets the scope element itself (`.np-site` root-level rules) cannot match inside the editor — a container's queries apply to descendants only. No current rule class depends on this.
- The inspector's Quill editor sits outside the `np-viewport` container, so its container-mode typography ramps don't fire there (it renders xl sizes — on a desktop browser this matches the previous behaviour).
- Admin surfaces that render widget previews without an `np-viewport` ancestor (dashboard slot grid) no longer get width-`@media` styling from the bundle at narrow *browser* widths; previously those rules keyed off the browser window, which was its own kind of wrong.
- `position: fixed` would be defeated by a transformed/filtered ancestor between `<body>` and the editor. None exists in the current Filament/Livewire DOM; if one ever appears, the fallback is a `<Teleport to="body">` wrapper toggled with the mode.
- The empty-slot "+ Add widget" button keeps its zoom-compensated 2rem type and will read large near zoom 1 — visible but not content-overlapping; tune later if it bothers.

## Deferred — RTE longform writing mode (scoped, not built)

**Problem (owner-reported, code-verified):** writing longform copy in-page is cramped. Three separable causes: (1) inline editing mounts Quill inside the zoomed preview scope, so prose renders at ~50–70% of real size; (2) editor chrome physically overlaps the text — hover handle bars, selection outlines, the floating format toolbar; (3) authoring happens in the design's colors (white-on-dark heroes are hostile writing surfaces).

**Solution shapes, scoped for a future session (~1 session / 2 iterations):**

- **B — quiet chrome while editing (cheap floor, build first).** A state class on the editor root while an inline edit session is active (the store already tracks `beginInlineEdit`/`endInlineEdit`) suppresses hover handles and outlines via CSS. Nearly free; kills the chrome-overlap annoyance independently.
- **A — focused writing mode (the real fix).** An "expand" affordance on an active rich-text session opens the field in a full-screen writing surface: white bg / black text, ~70ch centred measure, 1:1 scale, full toolbar; Done/Escape commits back. Structurally cheap because inline editors already seed from the raw config value and commit through `store.updateLocalConfigPath` — the rendered DOM is never serialized, so the surface is just a different host for the same contract. Composes as a second level inside A010's full-screen mode.
- **C — legibility override in place** (white/black on just the active field) — folded into A rather than built standalone; it fixes color but not scale or chrome.

**Constraint to honour when built:** keep the surface at the *editor-host* level (the store's active-inline-editor seam), not the Quill-blot level, so it survives the planned ProseMirror migration (`sessions/tracks/rich-text-editor-prosemirror.md`) intact.
