## Settled at planning (357 close prep)

The two architectural choices were surfaced and decided during the 357 close prep, so they are **not** open at the gate — confirm they still hold, then build:

1. **`demoContext()` shape — decided.** The hook returns a generic descriptor; the controller stays generic (no per-widget branch):
   ```php
   public function demoContext(): ?array
   {
       return [
           'guard'  => 'portal',
           'seeder' => DemoPortalMemberSeeder::class,
           'login'  => DemoPortalMemberSeeder::ACCOUNT_EMAIL, // identifier on the guard's provider
       ];
   }
   ```
   `WidgetDemoController`: if a widget declares a context, run the seeder, resolve the guard's user provider, log the member in by the `login` identifier, render, and **log out in a `finally`**. One shared fixture, declared per-widget — joins the existing `demo*()` family.
2. **Framing override — dropped.** No per-widget capture-viewport feature. The **manual-thumbnail opt-out is the single mechanism** for widgets that frame poorly: commit a tightly-framed `static.png` (a one-off tight capture or hand-authored) and set the opt-out flag so `--all` never clobbers it. One escape hatch covers framing *and* any "generated thumb just looks bad" case, and it removes the "framing changes the real render" risk entirely.

## Phases

Stage 4.5 completion against the green Pest net. Split across iterations per Rule 11 if `demoContext()` proves load-bearing; otherwise land together.

### Phase 1 — `demoContext()` (auth-context rendering) — load-bearing

Add a `demoContext(): ?array` hook to `WidgetDefinition` (default null). In `WidgetDemoController`, when a widget declares a context, run its seeder, resolve the declared guard's user provider, log the member in by the descriptor's `login` identifier before `WidgetRenderer::render()`, and restore prior auth in a `finally`. Add a shared, idempotent `DemoPortalMemberSeeder` that mints exactly:

- a **Contact** (`first_name`, `email`, address fields, optional `household_id`);
- a **PortalAccount** (`contact_id` → the Contact, `email`, hashed `password`, `email_verified_at`) keyed on a fixed `ACCOUNT_EMAIL` constant — this is the `auth('portal')` member the controller logs in;
- one **EventRegistration** (`contact_id` + `status` + `registered_at`) against the **existing `DemoEventSeeder` event** (`demo-community-gala`, from 357) so `PortalEventRegistrations` shows a real row.

No membership is needed (`PortalAccountDashboard` shows name/email/household only; `PortalChangePassword` needs only a logged-in account). Wire the four portal widgets — `PortalAccountDashboard`, `PortalContactEdit`, `PortalEventRegistrations`, `PortalChangePassword` — to `demoContext()`. Regenerate their thumbnails; confirm non-blank.

### Phase 2 — Manual-thumbnail opt-out (the single presentation mechanism)

Add a Definition flag (e.g. `usesManualThumbnail(): bool`, default false) surfaced through `manifest()`; `generate-thumbnails.js` skips `static.png` capture for declaring widgets (presets still captured unless separately opted out). Document the flag in `docs/widget-system.md`. This is the durable home for any committed-by-hand thumbnail — the escape hatch the strip-widget framing (Phase 3) and any future hand-authored thumbnail rely on.

### Phase 3 — Strip-shaped widget thumbnails (via the opt-out)

For widgets that render a short, wide element floating in the 800×500 frame — `Nav`, `Logo`, `SocialSharing`, and any others the 357 audit / a quick re-check surface — commit a tightly-framed `static.png` (a one-off tight capture or hand-authored) and set the Phase-2 opt-out flag so `--all` never clobbers it. **No permanent per-widget viewport feature** is added to the pipeline. The framed images are a visual call — surface them for the owner. Leave widgets that already fill the canvas untouched.

### Phase 4 — Thumbnail-coverage guard

Add a Pest test (group `widget-lint`, alongside `WidgetManifestTest`) asserting every registered widget's committed `static.png` is non-blank **or** on an explicit allowlist. Detect "blank" by **fingerprint match against a reference empty-canvas capture** (hash-identical, or the deterministic blank byte-size observed in 357 — every blank widget produced a byte-identical 2350-byte PNG), **not** an arbitrary size floor — a fingerprint match never false-positives on a legitimately sparse-but-valid widget. Seed the allowlist with `random_data_generator` and `setup_checklist` (one-line reason each); pin it like the `demoSeeder` list so adding/removing an entry is a reviewed edit. If a genuine edge case ever trips the check, allowlist it with a reason — never fabricate content to clear the bar. This keeps a future `--all` regen or a new widget from silently shipping a blank thumbnail. Widgets opted into manual thumbnails (Phase 2/3) pass trivially — a committed framed PNG is non-blank.

## Out of scope

- **Stage 6 — Widget Browser UI.** The admin grid that consumes these thumbnails. Post-Beta-1; this session is its predecessor, not its delivery.
- **A per-widget thumbnail-viewport / framing feature.** Explicitly *not* built (decided at planning) — strip widgets are handled by the manual-thumbnail opt-out + a committed framed PNG. Do not add viewport plumbing to the generator.
- **Animated thumbnails (Stage 4.5 Phase 2).** Deferred until after Stage 6 per the outlines. Do not start the `--animated` capture path.
- **Placement padding.** Giving small widgets default padding when *dropped in* (not just in the thumbnail) is a widget-defaults change with real product-behavior consequences — separate decision, its own entry if pursued. `demoAppearanceConfig()` for thumbnail-only breathing room is fine; default `appearance_config` changes are not this session.
- **Inbox items.** Logo-widget rich-text controls and per-widget colour-picker schema defaults stay parked (design-gated).
- **Thumbnail-pipeline rewrite / schema.** Add hooks + a flag + a guard to the existing pipeline; don't re-architect `generate-thumbnails.js` or the controller. No migrations expected. Non-boundary; CRM stays v2.4.0.

## Drift signals (stop and surface)

- `demoContext()` can't authenticate the `portal` guard cleanly inside the dev render with a plain seed-and-login (e.g. it needs session/middleware machinery the isolated render doesn't have) — surface before expanding scope. (The shape is decided; this is about implementation reality.)
- A portal widget turns out to read data the named fixture (Contact + PortalAccount + one EventRegistration, no membership) doesn't carry — surface before enriching or forking the fixture per widget. (The four templates were read at planning; this guards against a miss.)
- A strip widget can't be framed acceptably by a one-off capture and would actually need the per-widget viewport feature we dropped — surface; don't quietly reintroduce viewport plumbing.
- The coverage guard flags a widget you believe renders fine. With fingerprint-match detection this should be near-impossible (a real render diverges from the blank fingerprint); if it happens, **allowlist the widget with a one-line reason — never fabricate content to clear the bar.**
