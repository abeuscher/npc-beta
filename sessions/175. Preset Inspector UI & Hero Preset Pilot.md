# 175. Preset Inspector UI & Hero Preset Pilot

First session of the Stage 5 preset sub-arc. The `presets()` contract shipped in session 173 and every widget definition can already declare named config bundles; CI validates shape and cross-references keys against `schema()`. This session turns that dormant contract into a shipped feature: a **third inspector tab** called "Presets" that lists every preset the widget declares as a full-panel-width card; clicking a card replaces the widget's `config` and `appearance_config` with the preset's values; one widget (Hero) is authored with 3â€“4 preset entries as the pilot so the whole round-trip can be tested.

No backend schema changes. Presets stay code-only, living on the definition class. No preset thumbnail imagery is captured this session â€” the session reserves the per-preset thumbnail path under each widget's `thumbnails/` folder but leaves it empty. User-authored / exportable presets are an explicit future concern; this session does not touch that path.

## Design decisions (resolved before starting)

- **Third inspector tab, not a collapsible section.** The existing `InspectorTabs.vue` has two buttons ("Content" and "Appearance"); this session adds a third ("Presets"). The component's `activeTab` prop type widens from `'content' | 'appearance'` to `'content' | 'appearance' | 'presets'`. Every usage site (likely `InspectorPanel.vue` and anywhere else that types that prop) updates accordingly.
- **Full-panel-width preset cards.** Each card is a `<button>` spanning the full width of the inspector panel, stacked vertically. Card contents this session: the preset's `label` as the primary line, `description` as a smaller muted line below. The thumbnail image slot is **reserved in the card layout** (a blank box with a fixed aspect ratio, same position every card) but renders empty â€” no file lookup, no fallback icon. This keeps layout stable when thumbnails are added later without reshaping the component.
- **"Blank" is always the first card.** The gallery prepends an implicit preset representing `defaults()` + empty `appearance_config`. Its card says "Blank" / "Default starting point." Clicking it replaces config and appearance_config with `defaults()` and `{}` respectively. This card is synthetic â€” not authored in `HeroDefinition::presets()`, not included in the manifest's presets array. It is rendered by the frontend when the Presets tab opens.
- **Replace-all semantics on apply.** Clicking a preset **replaces** the widget instance's `config` and `appearance_config` wholesale with the preset's values. No merge with existing instance keys. This matches the product intent ("presets are opinionated starting points") and avoids half-applied looks.
- **No "which preset is applied" state tracking.** The store does not record that a preset was applied. After apply, the widget is just a widget with that config â€” subsequent edits don't "dirty" a preset, there is no "reset to preset X" button. Re-applying the same preset card will re-overwrite.
- **No save/export of user-created presets.** Presets are code-only this session. The eventual workflow (designer iterates, saves as preset, exports) is noted for a future session and explicitly not in scope here.
- **Preset thumbnail path reservation.** Each widget's `thumbnails/` folder (introduced in session 174) gains a documented convention: per-preset thumbnails will live at `app/Widgets/{PascalName}/thumbnails/preset-{handle}.png`. No files are written this session. The path is mentioned in `docs/widget-system.md` under the preset section so future sessions don't have to invent a new convention.
- **No frontend test runner this session.** The project has no Vitest/Jest/@vue-test-utils setup yet (flagged as post-Beta 1 in session-outlines.md). Do not add one for this session. Vue changes are covered by a backend feature test that asserts the preset data reaches the builder payload correctly and by manual testing of the UI in the browser.
- **`npm run build` must succeed after any Vue changes.** The session checklist requires this.

## Phase 3 â€” Frontend: Presets tab

1. **`InspectorTabs.vue`** â€” add a third button "Presets" between Content and Appearance, or after Appearance â€” implementer's visual judgment, but it must be a third sibling button, not a dropdown or nested element. Widen the prop type.
2. **`InspectorPanel.vue`** â€” handle the third tab value. When active, render a new child component `InspectorPresetGallery.vue` (new file) that lists the widget's presets.
3. **`InspectorPresetGallery.vue`** (new) â€” component that accepts:
   - The widget's preset array (as supplied by the bootstrap payload),
   - A reference to the currently-selected widget instance,
   - An `apply` callback.

   Renders a stack of full-panel-width `<button>` cards. First card is always the synthetic "Blank" card. Each card layout:
   - An empty thumbnail slot (a `<div>` with fixed dimensions and a placeholder background) â€” no image, just reserved space.
   - Preset `label` on its own line, prominent.
   - Preset `description` below in muted text. Omit the element entirely when `description` is null/empty.
4. **Apply mechanics.** The component's `apply` callback routes through the Pinia store. Write a store action (under `resources/js/page-builder-vue/stores/editor.ts` or wherever widget mutations live) called `applyPreset(widgetId, preset)` that:
   - Replaces the widget instance's `config` with `{ ...preset.config }` (a fresh copy; do not share object identity with the manifest).
   - Replaces the widget instance's `appearance_config` with `{ ...preset.appearance_config }`.
   - Triggers the same save/preview-refresh path the existing inspector field edits use. Reuse the existing debounced-save plumbing â€” do not bypass it.
   - The synthetic "Blank" preset is applied the same way; the frontend constructs the equivalent object `{ handle: '__blank', config: resolvedDefaults, appearance_config: {} }` from data already in the payload (`resolved_defaults` ships on every widget instance per session 171).
