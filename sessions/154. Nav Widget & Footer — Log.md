## What was built

### Phase 1 — ChromeRenderer renders layouts

- `App\Services\Media\ChromeRenderer::renderPage()` now loads root widgets AND `PageLayout` records on the same system page, merges them into a single page-flow list ordered by `sort_order`, and renders each item — mirroring the merge pattern in `PageController::renderPage()`. Previously layouts on system pages were silently dropped (the unfinished business from session 153).
- New private `ChromeRenderer::renderLayoutBlock()` method emits `<div class="page-layout" style="…">` with one `<div class="layout-column">` per slot, builds the container CSS from `display`, `columns`, `layout_config`, and walks each slot's child widgets. Inline styles, scripts, and assets from layout child widgets are collected the same way root widgets do.

### Phase 2 — Logo widget

- New blade `resources/views/widgets/logo.blade.php`. Single anchor wrapping an `<x-picture>` and a `<span>`, with class names `widget-logo`, `widget-logo__link`, `widget-logo__img`, `widget-logo__text`.
- New SCSS `resources/scss/widgets/_logo.scss`. `width: fit-content` on the outer wrapper, `display: inline-flex` + `gap: 0.5rem` + `align-items: center` on the anchor. Image left, text right, anchored to the left of its container with no stretching.
- New `logo` entry in `WidgetTypeSeeder`. Category `['layout']`, server-rendered, `full_width: false`, config schema: `logo` (image), `text` (text), `link_url` (text/url, default `/`).

### Phase 3 — Nav widget

- New blade `resources/views/widgets/nav.blade.php`. Direct extraction of the nav-rendering portion of `site-header.blade.php` — same `site-nav__*` class names, same Alpine `x-data="{ open: false }"`, same hamburger toggle, same dropdown handling, same `aria-current` logic.
- New `nav` entry in `WidgetTypeSeeder`. Category `['layout']`, server-rendered, `full_width: false`, config schema: `nav_handle` (text, default `primary`). `required_config` set on `nav_handle`. CSS for nav remained in `_custom.scss` (least disruptive — the existing `site-nav__*` classes work as-is from the new widget).

### Phase 4 — Migration: rebuild `_header` and `_footer`

- New migration `database/migrations/2026_04_09_120000_rebuild_chrome_pages_with_logo_and_nav.php`. One-way (no rollback beyond best effort). Walks: find `_header` and `_footer` system pages by slug → wipe their `PageLayout` records (cascades to layout children) and root widgets → delete `site_header` and `site_footer` widget types → re-seed `_header` with a 2-column grid layout (`auto 1fr`, gap `1rem`, `align_items: center`) containing logo (col 0) + nav `primary` (col 1) → re-seed `_footer` with a 2-column grid layout (`1fr auto`, gap `1rem`, `align_items: center`) containing a `text_block` with copyright HTML (col 0) + nav `footer` (col 1). Migration self-inserts the `logo` and `nav` widget types if they're not yet seeded so it's safe to run before `db:seed`.
- `WidgetTypeSeeder` updated: removes `site_header` / `site_footer` rows (and their orphaned `page_widgets`) at the top of `run()`, then `updateOrCreate`s `logo` and `nav`. A fresh `migrate:fresh --seed` produces the same end state — `site_header` / `site_footer` are gone for good.
- `SystemPageSeeder` updated: no longer attempts to attach a `site_header` / `site_footer` widget to the chrome pages. The seeder only ensures the `_header` / `_footer` pages exist; the migration populates their content.

### Iteration 2 — User-reported issues fixed in-session

These were not in the original session prompt but were discovered during manual testing and fixed in the same session-154/1 branch.

#### 1. Dual page-builder mount in template editor

The page-builder Livewire component used a hardcoded `id="page-builder-app"` and `main.ts` mounted via `getElementById`. The template editor renders Header AND Footer page-builder instances simultaneously (Alpine `x-show` does not unmount), so two elements shared the same ID and only the first mounted. Fix:

- `resources/views/livewire/page-builder.blade.php` — replaced static ID with `data-page-builder-app` attribute.
- `resources/js/page-builder-vue/main.ts` — `querySelectorAll`s every `[data-page-builder-app]`, mounts an independent Vue + Pinia instance per element, marks each `dataset.mounted = 'true'` to guard against double-mount.
- `resources/js/page-builder-vue/App.vue` — accepts `bootstrap` as a root prop instead of looking it up from a global ID.

#### 2. Layout container appearance controls

Added container-level style controls to layouts (mirrors what individual widgets already had, minus text colour to avoid cascade confusion). New keys allowed on `layout_config` (no schema change — JSONB):

- `full_width` (bool, default false — when off, layout is contained in `.site-container`)
- `background_color` (string)
- `padding_top|right|bottom|left` (px)
- `margin_top|right|bottom|left` (px)

Renderer side:
- `PageController::renderLayoutBlock()` emits the new style keys as inline CSS on the layout container and returns `'full_width' => (bool) ($config['full_width'] ?? false)` so the existing `page-widgets.blade.php` containment wrapping handles it.
- `ChromeRenderer::renderLayoutBlock()` mirrors the same style emissions, AND each top-level chrome block (widget OR layout) is now wrapped in `.site-container` by default unless its full-width flag is on. **This fixes the chrome edge-to-edge bug introduced when the new logo + nav layout replaced `site_header.blade.php` (which had `.site-container` baked into the widget template).**

Inspector side:
- `LayoutInspectorPanel.vue` rebuilt to mirror the widget inspector chrome — type badge + label + delete button in a header pill, then a single static "Appearance" tab strip, then the existing controls in a body box matching `inspector-panel__body`. New "Container" section divider at the bottom with: Full width checkbox, Background Colour (`ColorPickerField`), Padding (px) 5-cell grid, Margin (px) 5-cell grid.

#### 3. Latent layout API merge bug

Found while wiring up the new keys: `PageBuilderApiController::updateLayout` was REPLACING `layout_config` with whatever subset the editor sent, wiping unrelated keys. The Vue editor sends partial diffs, so any incremental edit (e.g. just `gap`) silently deleted other keys (e.g. `grid_template_columns`). Fixed: the controller now merges sanitised partial updates into the existing `layout_config` rather than replacing it.

#### 4. Cross-instance event scoping

With two PageBuilder Livewire components alive in the DOM at once, `window`-dispatched events fired both listeners. Selecting a widget in the picker on one tab opened modals on both, and only one closed when its `createBlock` ran. Fix: every cross-boundary event now carries `pageId` in its `detail`, and every receiver early-returns when `detail.pageId !== this.pageId` / `store.pageId`. Affects:

- `PreviewCanvas.vue` — `open-widget-picker` from "+ Add Block"
- `LayoutRegion.vue` — `open-widget-picker` from empty column slot
- `EditorToolbar.vue` — `open-save-template-modal`
- `App\Livewire\PageBuilder` — `widget-created` and `template-saved` (PHP-side `dispatchEvent` calls now include `pageId` in detail)
- `resources/views/livewire/page-builder.blade.php` — `x-on:open-widget-picker.window` and `x-on:open-save-template-modal.window` listeners both wrapped in `if ($event.detail?.pageId === @js($pageId))` guards
- `App.vue` — `handleWidgetCreated` and `handleTemplateSaved` early-return if `detail.pageId !== store.pageId`

#### 5. Preview canvas — top-row column handle clipped

Top-row layout's hover-revealed handle (`.layout-region__handle`, `bottom: 100%`) was getting clipped by `.preview-canvas__container`'s `overflow: hidden` against the viewport toolbar above it. Fix: added `padding-top: 5rem` to `.preview-canvas__container` (handle height 3.5rem + ~25px headroom).

#### 6. Drop hit area inside empty columns

The `+ Add widget` button (`min-height: 9rem; width: 100%`) covered the entire empty column slot, leaving only the slot's outer 0.5rem padding strip as a clean Sortable drop target. Fix (option 1, smallest change): added `dragging: ref(false)` to the editor store, flipped on by `@start` / `@end` on both the root canvas draggable and each column-slot draggable. CSS rule: `.layout-region__container--dragging .layout-region__add-widget { pointer-events: none }`. The button is still painted; during a drag the entire 9rem slot becomes a clean drop target. Click-to-add at rest is unchanged.

#### 7. Preview render not honoring layout style fields

Two missing pieces in `LayoutRegion.vue`'s `containerStyle` and rendering logic:

- The new `padding_*` / `margin_*` / `background_color` keys were never emitted in the preview (only the public renderer honored them).
- No containment behaviour at all — the preview always showed layouts at 100% width regardless of `full_width`.

Fix:
- Extended `containerStyle` to emit the spacing and background-colour keys (mirrors `PageController::renderLayoutBlock`).
- Added `--contained` modifier class on `.layout-region__container` when `!full_width`. CSS rule: `width: 90%; max-width: var(--np-preview-container-max-width, 100%); margin: 0 auto`.
- `PreviewCanvas.vue` exposes the CSS variable `--np-preview-container-max-width` on `.widget-preview-scope`, computed from the active viewport preset (1920→1320px, 1024→960px, 375→100%) — same step function as the public `.site-container` breakpoints, just resolved against the editor's fixed pixel viewport.

## Deferred decisions

- **Widget JS bundle rebuild not run.** The new `logo` and `nav` widget types were added via `WidgetTypeSeeder`, but the public asset pipeline (`AssetBuildService::build()`) was not re-run because the build server isn't part of the local dev workflow. The widgets render correctly via Vite for now; production deploys should run `php artisan build:public` after applying this migration.
- **Layout selection highlight uses a different CSS technique than widget selection highlight.** Discovered while investigating a UI question — `LayoutRegion.vue` uses `outline` while `PreviewRegion.vue` uses inset `box-shadow` on a separate overlay element. Both are now visually consistent (same colour + width) but the implementations are duplicated. Not consolidated this session per user direction; a future cleanup could extract a shared `--editor-selection-color` CSS variable and unify the chrome.
- **Nav widget redesign deferred.** This session only extracted the existing nav verbatim. Modes, dropdown rebuild, hamburger collapse rebuild, mega menus, multi-level submenus, animations — all deferred to a future dedicated nav session.
- **Theme toggle from old `site_footer` was dropped.** No replacement widget exists; the user can re-add it as a future widget if needed.
