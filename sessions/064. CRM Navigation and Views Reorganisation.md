# 064. CRM Navigation & Views Reorganisation

Reorganises the admin CRM navigation for clarity: Memberships moves to the Finance group where it belongs financially, Notes is removed from the top-level nav (it is a relation manager feature, not a primary entity), and a new Members resource provides a focused lens on contacts who hold active memberships. No new tables, no migrations required.

---

## Architecture decisions

- **MemberResource** points to the `Contact` model and scopes via `isMember()` — the same pattern as `PostResource` scoping `Page` to `type = 'post'`. It is read-only: no create action (members are contacts with membership records, not a distinct creatable entity), no delete. The table Edit action redirects to `ContactResource::getUrl('edit', ['record' => $record])` so the full contact form is used, not a separate member form.
- **MembershipResource** moves to the Finance navigation group, sort 5, after Transactions. No other changes to the resource itself.
- **NoteResource** is hidden from navigation via `protected static bool $shouldRegisterNavigation = false`. The model, `NotesRelationManager`, and all relation behaviour remain completely intact.
- **HouseholdResource** is left unchanged in this session. Its nav fate will be decided in the Household & Family Grouping session.
- **ContactResource** sort and filters are unchanged. The existing "Members only" table filter stays — it remains useful for staff who want a quick filter without navigating away.
- **Permissions**: a new `view_any_member` permission is added. This allows it to be granted independently of full contact access (e.g. a volunteer coordinator may need the Members view but not the full Contacts list). `super_admin` receives it automatically via the existing wildcard grant. `cms_editor` does not.

## Phase 3 — Add MemberResource

Create `app/Filament/Resources/MemberResource.php`:

- **Model**: `Contact`
- **navigationGroup**: `'CRM'`
- **navigationSort**: `2`
- **navigationIcon**: `heroicon-o-identification`
- **canViewAny**: `auth()->user()?->can('view_any_member') ?? false`
- **getEloquentQuery()**: applies `->isMember()` scope to the base query

**Table columns** (eager-load `memberships` scoped to active):
- Display name — searchable on first/last name, sortable
- Email — searchable
- Active tier — `getStateUsing()` from eager-loaded active membership, placeholder `—`
- Status — from active membership, placeholder `—`
- Expires on — date column from active membership, placeholder `—`

**Actions**:
- Edit action: redirects to `ContactResource::getUrl('edit', ['record' => $record])`. Use a custom `Action` rather than the default `EditAction` since editing is handled by a different resource.
- No create action, no delete action, no bulk delete.

**Pages**: `ListMembers` only. No `CreateMember`, no `EditMember` pages are created.

Create a help doc stub at `resources/docs/members.md` following the frontmatter convention of existing docs. Register the list route in the `routes:` array.
