## Phase 1 — PII / Sensitive Data Rejection

### New file: `app/Services/PiiScanner.php`

Standalone, testable service. Two detection layers:

**Layer 1 — Column header blocklist** (`scanHeaders()`): normalises headers to lowercase with spaces and underscores stripped, rejects if any match the blocked list: `ssn`, `socialsecurity`, `socialsecuritynumber`, `creditcard`, `creditcardnumber`, `cardnumber`, `pan`, `routingnumber`, `accountnumber`, `bankaccount`, `abarouting`, `driverlicense`, `driverslicense`, `dlnumber`.

**Layer 2 — Cell content patterns** (`scanCell()`):
- Credit card PAN — 13–19 digit sequences (spaces/dashes stripped) that pass Luhn check
- SSN — `\d{3}-\d{2}-\d{4}` format, or any 9-digit sequence not matching ABA prefix
- ABA routing number — 9-digit sequence whose first two digits fall in 01–12 or 21–32

`scan()` runs both layers in order and returns the first violation found (with row/column detail), or null if clean.

### `ImportProgressPage.php` changes

- Added `use App\Services\PiiScanner`
- Added two public Livewire properties: `$rejected` (bool) and `$rejectionReason` (string)
- `mount()` runs `PiiScanner::scan()` after reading the CSV header row, before `resolveCustomFieldDefs()`. On violation: sets `$done = true`, `$rejected = true`, marks the import log `failed`, writes violation detail to `errors` JSON column, calls `failSession()`, returns early. No contacts are created.
- Respects `IMPORTER_SKIP_PII_CHECK=true` env flag (developer bypass only)

### `import-progress.blade.php` changes

Added a `$rejected` branch at the top of the `@if ($done)` block. When rejected:
- Red shield card with the rejection message: "This import was rejected because it contains data that may include sensitive financial or personal identifiers. Remove the flagged data and try again."
- Monospaced detail line showing which row/column triggered the rejection
- "Start a new import" link back to ImporterPage
- No import counters shown

When not rejected, the existing pending-review / complete cards and counters render as before.

### `.env.example`

Added `IMPORTER_SKIP_PII_CHECK=false` with a prominent developer-only warning comment.

## Deferred to Session 044

Three issues surfaced during manual testing that require architectural changes beyond this session's scope:

**1. No note written on contact update.**
When an existing contact is matched and updated by an import (via External ID or email), no note is written to the contact record. Notes are only written for newly created contacts. Fix deferred to session 044.

**2. Tags not applied to updated contacts.**
The `$contact->tags()->sync($this->tagIds)` call only runs in the new-contact creation branch of `processRow()`. It is never called for the update paths. Tags are silently dropped on all re-imports that match existing contacts.

**3. Architectural gap — updates bypass the review gate.**
More significantly: all field changes to existing contacts (phone, city, name, etc.) are applied immediately in `processRow()`, before any reviewer approves the session. The review queue's approval step only affects the visibility of *new* contacts (via the global scope). Updates to existing records are silent and irreversible. This violates the intent of the review workflow. Session 044 will introduce a staging table to hold all proposed changes to existing contacts pending approval.
