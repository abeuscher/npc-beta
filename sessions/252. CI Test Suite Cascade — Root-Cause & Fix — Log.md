## What was built

### Phase 1 — CI cascade root-cause + fix

The session opened against a 2202-line CI failure log dropped by the user (`sessions/test-failures.txt`): **86 failed / 1724 passed** with zero memory signatures across the entire output. Same failure pattern as the cascade session 251 had attributed to `memory_limit` (256M → 1G), but with the memory bump in place — the 251 diagnosis was incomplete. Three constraint patterns repeated across the failures:

| Table | Constraint | Failing tests |
|---|---|---|
| `site_settings` | `site_settings_key_unique` (`key = blog_prefix`) | `BlogPagerEndToEndTest`, `PostTest`, `TemplateTest`, others |
| `navigation_menus` | `navigation_menus_handle_unique` (`handle = primary`) | `ChromeWidgetTest` |
| `help_articles` | `help_articles_slug_unique` (`slug = contacts` / `donations`) | `HelpArticlePageTest`, `HelpArticleServiceTest`, `HelpSearchTest` |

All three keys are seeded by `DatabaseSeeder` (or its callees: `BasePageSeeder` for `navigation_menus`; `Artisan::call('help:sync')` for `help_articles`). All three seeders use `firstOrCreate` (idempotent). The failing tests insert via `create()` (e.g., [`tests/Feature/BlogPagerEndToEndTest.php:13`](tests/Feature/BlogPagerEndToEndTest.php#L13)) — and they all use `RefreshDatabase`. So the contradiction: with `RefreshDatabase` + no autoseed, the DB at the start of each test should be empty; yet the rows exist before `create()` is called. **Something was committing seed data outside the per-test transaction wrap.**

**Phase 0 — Reproduce locally.** Ran `docker compose exec app php artisan test` (no flags, full suite, matching CI's invocation exactly). Result: **86 failed / 1724 passed (9410 assertions), 500.80s** — matched CI (86/1724, 308s; local slower under WSL2 Docker). Cascade reproduced cleanly; investigation could proceed against a working reproduction.

**Phase 1 — Investigation.** Five hypotheses from the session prompt, ruled in/out methodically:

1. *Test uses `DatabaseMigrations` and runs `db:seed`* — `grep -rn "DatabaseMigrations\|migrate:fresh" tests/` returned zero matches in the test suite. Ruled out.
2. *Test explicitly commits via `DB::commit` / `withoutTransactions`* — `grep -rn "DB::commit\|->withoutTransactions" tests/ app/` returned zero matches. **But a variant fired** — see below.
3. *Boot-time seeding* — `AppServiceProvider::boot()` reads `SiteSetting::all()` inside a try/catch but does not write; no migrations carry data-insert steps; `bootstrap/app.php` schedule entries don't run at test boot. Ruled out.
4. *PostgreSQL connection quirk* — would have presented as local-vs-CI asymmetry; reproduction was identical, so ruled out.
5. *Test-order leak* — first failing test (`BlogPagerEndToEndTest`) is alphabetically very early; the question became "what runs before B?"

The smoking gun surfaced under hypothesis 2's variant. [`tests/Feature/AppResetCommandTest.php`](tests/Feature/AppResetCommandTest.php) (shipped at session 247) declares `uses(TestCase::class)` only — **no `RefreshDatabase`**. Its slow-grouped happy-path test calls `Artisan::call('app:reset')`, which (per [AppResetCommand.php:24-27](app/Console/Commands/AppResetCommand.php#L24-L27)) executes `migrate:fresh --seed --force`. That call runs the full `DatabaseSeeder` → `BasePageSeeder` → `help:sync` chain, **committing** every seed row at the database level. After the test exits, `RefreshDatabase`'s static `$migrated = true` is unchanged, so subsequent tests only `BEGIN`/`ROLLBACK` per-test — they don't re-migrate. Every later test that does `*::create([...'blog_prefix'...])` etc. in `beforeEach` hits the seeded row and fails with a unique-constraint violation.

**Why session 251's fast suite hid this:** the fast invocation `vendor/bin/pest --exclude-group=slow` excludes `->group('slow')`. The polluting test is `->group('slow')`, so it never ran. CI's bare `php artisan test` runs everything, including slow groups.

**Phase 2 — Fix shape decision.** Two options surfaced. The narrow fix was *add `RefreshDatabase` to `AppResetCommandTest`* (one-line trait change). The broader fix was *delete `app:reset` entirely — the function itself is the problem*, on the grounds that:

- `app:reset` bundles three trivial operations (`migrate:fresh --seed --force`, wipe `storage/app/public/*`, wipe `storage/media-library/temp/*`) plus a production-refusal gate that's mostly theater (a developer who'd run `app:reset` on prod would equally run `migrate:fresh --seed --force` directly);
- the artisan-command surface is structurally unfit for this work — any test that exercises `migrate:fresh --seed` from inside Pest fights `RefreshDatabase`'s isolation;
- the happy-path test is mostly tautological (asserts that Laravel's `migrate:fresh --seed` actually re-seeds the DB, and that `Storage::disk('public')->deleteDirectory()` actually deletes — both Laravel's job, not ours).

User picked the broader fix, with explicit instruction: *"the bug is real but this was not a good fix at all. We do need to at least put that back in the planning doc."* Delete `app:reset`, document the manual two-step recipe, reopen the underlying orphan-media-cleanup bug as a stub.

**Implementation:**

- Deleted [`app/Console/Commands/AppResetCommand.php`](app/Console/Commands/AppResetCommand.php) (49 lines).
- Deleted [`tests/Feature/AppResetCommandTest.php`](tests/Feature/AppResetCommandTest.php) (30 lines).
- [`docs/app-reference.md`](docs/app-reference.md): removed the "Dev tooling — `app:reset`" section; rewrote the "Orphan media sweep" cross-reference to point to the reopened stub instead of `app:reset`.
- [`sessions/session-outlines.md`](sessions/session-outlines.md) line 151: un-struck the orphan-media-cleanup stub. Captured the bug (`migrate:fresh` truncates via raw SQL, bypassing Spatie Media Library's `deleting` observer; session 245 UAT severity was 14.6 GB for ~20 MB of distinct content), why session 247's fix was reverted (the `->group('slow')` happy-path test ran `migrate:fresh --seed` from inside Pest, breaking `RefreshDatabase` isolation), four candidate paths forward (shell script outside Laravel, migration-lifecycle hook, Docker volume swap, accept-and-document), and the **constraint for the next attempt: no new artisan command that re-runs `migrate:fresh` from inside the application**.
- [`sessions/release-plan.md`](sessions/release-plan.md): updated E12's success-criterion mention of `app:reset` to acknowledge it was reopened; updated the deferred-decisions note about E12 shape.

**Phase 3 — Verification.** Re-ran `docker compose exec app php artisan test`. Result: **1808 passed (9508 assertions), 551s, zero failures.** Math accounts cleanly: `1724 + 86 - 2 = 1808` (the −2 is the deleted `AppResetCommandTest` tests). All 86 previously-failing tests now pass.

**Droplet-side troubleshooting (in iteration order):**

1. **Stock marketplace image had `2375/tcp` and `2376/tcp` open in UFW** (Docker daemon remote API). User verified nothing was actually listening; removed the rules. Same hardening had been done on the prod droplet but forgotten.
2. **DNS** — A records for `@` and `www` pointing to 147.182.214.147 (no CNAME needed; an A record for `www` is the alternative to a CNAME, not in addition). Verified resolution from local before certbot.
3. **Certbot timed out with "likely firewall problem."** UFW didn't have `:80` allowed. Added `ufw allow 80/tcp` and `443/tcp`; certbot then succeeded for both `rectanglemachine.com` and `www.rectanglemachine.com` in one cert.
4. **First deploy SSH handshake failed** (`unable to authenticate, attempted methods [none publickey], no supported methods remain`). Root cause: user generated a deploy keypair on the droplet at `/root/.ssh/github_key{,.pub}` and put the private half in `DEMO_DEPLOY_SSH_KEY`, but never appended `github_key.pub` to `/root/.ssh/authorized_keys`. Fix: `cat ~/.ssh/github_key.pub >> /root/.ssh/authorized_keys`.
5. **Second deploy partial-success then `dial tcp <ip>:22: i/o timeout`** at the SCP step. Diagnosed as UFW's `LIMIT` rule on `:22` (six connections per 30 seconds) silently dropping `appleboy/scp-action`'s rapid burst. User's first attempt to remove the LIMIT (`ufw delete limit 22/tcp 2>/dev/null`) silently no-op'd because `2>/dev/null` swallows the y/n confirmation prompt. Fix: `ufw status numbered` + `ufw delete <N>` with interactive `y` for the v4 rule, then `ufw allow 22/tcp` (which auto-replaced the v6 LIMIT in place — `Rule updated (v6)` in the output).
6. **nginx conf had to be manually placed.** The deploy workflow only copies `docker-compose.prod.yml` to the droplet — the bind-mounted `./docker/nginx/prod.conf` source file must already exist on the droplet host filesystem. User initially thought the workflow handled it; verified against the prod droplet (`stat /opt/nonprofitcrm/docker/nginx/prod.conf` showed mtime of `Mar 19 05:30`, six weeks old — placed manually at prod setup time, never refreshed by any pipeline). Wrote a domain-substituted version of `prod.conf` (server_name and ssl_certificate paths flipped from `beuscher.net` to `rectanglemachine.com`); user pasted to `/opt/nonprofitcrm/docker/nginx/prod.conf` on the demo droplet. Same manual-placement pattern needed for `/opt/nonprofitcrm/nginx-certs/fm-client.crt` placeholder (any valid PEM works; user used the letsencrypt fullchain as a stand-in).
7. **Login 500'd after deploy went green.** `migrate:status` returned `Migration table not found` — confirming the deploy workflow's `migrate --force` step exits 0 without actually creating tables on first install. Suspected race between `docker compose up -d` returning and postgres's first-time data-directory init; not investigated further (the manual workaround is reliable). Fix: `docker exec nonprofitcrm_app php artisan migrate:fresh --seed --force` manually after first deploy goes green. User confirmed this same problem had bitten on the prod droplet at its first install.

## Security checklist — verified

This session did not have a security-critical scope per se, but several decisions had security implications that landed correctly:

- [x] **`app:reset` deletion preserves the production-refusal posture.** Removing the artisan command does not weaken any production safety — `migrate:fresh` itself still requires `--force` outside `local` env (Laravel's default), and the manual two-step recipe documented in the docs is the same friction floor as every other destructive artisan command available to operators.
- [x] **No defensive workarounds masking the polluter.** The fix removes the polluter; it does not add `firstOrCreate` substitutions in `beforeEach` to hide the symptom. Per the session prompt's checklist.
- [x] **Test isolation not weakened elsewhere.** No `RefreshDatabase` was converted to `DatabaseMigrations` to "fix" a transaction-boundary problem. Per the session prompt's checklist.
